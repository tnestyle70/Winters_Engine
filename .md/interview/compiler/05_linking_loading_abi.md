# 링킹 · 로딩 · ABI — 컴파일러가 끝난 뒤의 세계

> "exe를 더블클릭하면 무슨 일이 일어나나요?", "가상 함수 호출은 어셈블리로 어떻게 생겼나요?", "DLL 경계에서 STL을 넘기면 왜 안 되나요?" — 컴파일러가 `.obj`를 뱉은 순간부터 `main`의 첫 줄이 실행되기까지, 그리고 두 바이너리가 서로를 "호환"이라 부를 수 있는 조건(ABI)까지를 이 챕터가 담당한다. 빌드 실무(TU·ODR·헤더·PCH)는 [cpp/02_compile_link_dll.md](../cpp/02_compile_link_dll.md)가 맡고, 여기서는 그 아래층 — 링커·로더·ABI의 내부 원리를 판다.

---

## ① 한 줄 본질

> **"컴파일러는 번역 단위마다 '주소를 아직 모르는 자리'의 목록(재배치 엔트리)과 '내가 만든 것/필요한 것'의 목록(심볼 테이블)이 붙은 미완성 기계어(.obj)를 남기고, 링커는 그 목록들을 맞춰 심볼 해석과 재배치라는 두 가지 일만 한다. 동적 링킹은 그 재배치의 일부를 프로세스 로드 시점으로 미룬 것이고(IAT를 로더가 채운다), ABI는 이렇게 따로 빌드된 바이너리들이 만나는 지점의 계약 — 호출 규약, 구조체 레이아웃, vtable 배치, 예외 전파 방식 — 이다. C++은 이 계약을 표준화하지 않았기 때문에, DLL 경계에서는 '같은 툴체인·같은 설정'이라는 빌드 규율이 곧 타입 시스템의 연장이 된다."**

## ② 기본 개념

### 오브젝트 파일 해부 — 컴파일러가 남긴 미완성 조각

컴파일러가 한 TU를 다 처리해도 완성된 기계어를 만들 수 없다. 다른 TU의 함수 주소, 전역 변수 주소, 최종 이미지에서의 자기 위치를 **아직 모르기 때문**이다. 그래서 `.obj`(COFF)는 "코드 + 모르는 것의 목록" 구조가 된다:

```
Foo.obj (COFF)
┌───────────────────────────────────────────────────────┐
│ COFF 헤더 (섹션 수, 심볼 테이블 위치, 머신 타입)          │
├───────────────────────────────────────────────────────┤
│ .text   기계어. 외부 호출 대상 주소 자리는 0으로 비워둠    │
│ .data   초기화된 전역/정적 (int g_count = 3;)           │
│ .bss    0-초기화 전역 — 크기만 기록, 파일에 내용 바이트 없음│
│ .rdata  읽기 전용: 문자열 리터럴, 상수, vtable, 점프 테이블│
├───────────────────────────────────────────────────────┤
│ 심볼 테이블                                             │
│   [정의] ?Update@CWorld@@QEAAXM@Z → .text + 0x40       │
│   [참조] ?Length@Vec3@@QEBAMXZ    → (미해결, UNDEF)     │
├───────────────────────────────────────────────────────┤
│ 재배치 테이블 (섹션별)                                   │
│   ".text+0x52 의 4바이트에 ?Length@... 의 주소를          │
│    이런 방식(REL32/ADDR64)으로 박아 넣어라"               │
└───────────────────────────────────────────────────────┘
```

섹션이 왜 이렇게 나뉘는가 — **메모리 권한과 파일 크기의 트레이드오프**다:

| 섹션 | 내용 | 페이지 권한 | 설계 이유 |
|---|---|---|---|
| `.text` | 기계어 | 읽기+실행 (쓰기 불가) | W^X 보안, 프로세스 간 공유 가능 |
| `.rdata` | 상수·리터럴·vtable | 읽기 전용 | 수정 불가 보장 + 공유 가능 |
| `.data` | 초기화된 전역 | 읽기+쓰기 | 초기값을 파일에서 복사해야 함 |
| `.bss` | 0-초기화 전역 | 읽기+쓰기 | **파일에 바이트가 없다** — 크기만 적고 로드 시 0으로 채움 |

`.bss`가 따로 있는 이유가 이 설계의 요점을 보여준다. `int g_buffer[1000000];`을 `.data`에 넣으면 파일에 4MB의 0이 실제로 기록된다. `.bss`는 "이만큼의 0-초기화 공간"이라는 메타데이터 한 줄로 대체한다. 읽기 전용 섹션들은 여러 프로세스가 같은 물리 페이지를 공유할 수 있어서(코드 페이지 공유), 같은 DLL을 쓰는 프로세스 10개가 코드 메모리를 한 벌만 쓴다 — 동적 링킹의 메모리 이점이 여기서 나온다.

**재배치(relocation) 엔트리**가 이 챕터의 첫 번째 열쇠다. "주소를 아직 모르는 자리"를 컴파일러가 목록으로 남기면, 그 자리를 채우는 것은 (a) 링커가 하거나(정적 링킹), (b) 로더가 하거나(베이스 재배치, IAT), (c) 아예 필요 없게 만들거나(RIP 상대 주소) — 셋 중 하나다. 링킹·로딩의 모든 메커니즘이 이 세 선택지의 조합이다.

### 링커의 일 (1) — 심볼 해석: strong, COMDAT, 그리고 조용한 ODR 위반

링커는 모든 `.obj`/`.lib`을 모아 두 목록을 대조한다: "누가 이 심볼을 정의했나" vs "누가 이 심볼을 참조했나". 짝이 안 맞으면 에러(LNK2019 = 정의 없음, LNK2005 = 정의 둘). 그런데 **"정의 둘"이 항상 에러는 아니다** — 이게 ODR 질문의 핵심이다.

| 정의의 종류 | 심볼 성격 | 중복 시 링커 행동 |
|---|---|---|
| non-inline 함수, 전역 변수 | strong | **LNK2005 에러** — 프로그램에 한 벌만 허용 |
| `inline` 함수, 템플릿 인스턴스, `inline` 변수, 클래스 내 정의 멤버 | COMDAT (select-any) | **아무거나 하나 골라 쓰고 나머지 버림 — 내용이 달라도 침묵** |
| ELF의 weak 심볼 (`__attribute__((weak))`) | weak | strong이 있으면 strong 채택, 없으면 weak 중 하나 |

왜 이런 이중 규칙인가. `inline` 함수와 템플릿은 **정의가 헤더에 있어야만 동작하는 언어 기능**이다(각 TU가 본문을 보고 인스턴스화해야 하므로). 그러면 같은 함수의 기계어가 TU 수만큼 `.obj`에 생긴다. 이걸 전부 에러로 처리하면 헤더에 정의를 두는 것 자체가 불가능해진다. 그래서 링커는 이런 심볼을 COMDAT 섹션(MSVC 기준 "select any")으로 받아 **중복을 병합**한다. 대가는: 모든 사본이 동일하다는 것을 링커가 검증하지 않는다는 것. 두 TU가 서로 다른 매크로 정의 아래서 같은 inline 함수를 다르게 컴파일하면(예: 한쪽만 `_DEBUG`), 링커는 아무거나 하나를 골라 조용히 쓴다 — **진단 의무 없는 미정의 동작(no diagnostic required)**. "어떤 ODR 위반은 링크 에러고 어떤 건 조용히 지나가는가"의 답이 정확히 이 표다: strong 중복은 에러, COMDAT 중복은 침묵.

심볼 해석에는 실무에서 얻어맞는 규칙이 하나 더 있다. **정적 라이브러리(.lib)에서 링커는 "지금 미해결인 심볼을 정의해 주는 .obj만" 꺼낸다.** 아무도 참조하지 않는 `.obj`는 링크에 포함되지 않는다. 전역 객체 생성자의 부수효과로 팩토리에 자기 등록(self-registration)하는 패턴이 정적 라이브러리에서 소리 없이 증발하는 이유가 이것이다 — 그 `.obj`의 심볼을 아무도 참조하지 않으니 꺼내지지도, 전역 생성자가 돌지도 않는다. (`/WHOLEARCHIVE`, `/INCLUDE:심볼` 같은 강제 포함 옵션이 존재하는 이유.)

### 링커의 일 (2) — 재배치: 주소 확정과 구멍 메우기

심볼 해석이 끝나면 링커는 (1) 모든 섹션을 최종 이미지 레이아웃으로 배치하고(같은 이름 섹션 병합, 주소 부여), (2) 재배치 테이블을 순회하며 비워둔 자리에 확정 주소를 박는다. 여기서 x64의 중요한 분화가 생긴다:

- **RIP 상대 주소(RIP-relative addressing)**: x64 코드는 "다음 명령어 주소 + 오프셋"으로 데이터·함수를 참조할 수 있다. 코드와 데이터의 **상대 거리**는 이미지가 어디에 로드되든 불변이므로, 링크 타임에 완전히 확정되고 로드 시 재배치가 필요 없다. x64 코드 섹션에 로드타임 재배치가 거의 없는 이유.
- **절대 주소**: 데이터 섹션 안의 포인터들 — 대표적으로 **vtable의 함수 포인터 배열**, 함수 포인터 전역 — 은 64비트 절대 주소라서, 이미지가 선호 베이스(preferred base)가 아닌 곳에 로드되면 전부 고쳐야 한다. 최종 PE의 `.reloc` 섹션이 "절대 주소가 박힌 자리들의 목록"이고, 로더가 (실제 베이스 − 선호 베이스)의 차이만큼 일괄 보정한다. **ASLR이 켜진 모든 모듈이 로드 때마다 이 비용을 낸다** — 재배치로 더럽혀진 페이지는 copy-on-write로 프로세스 사유가 되므로, 공유 이점도 그 페이지만큼 줄어든다.

`.obj`의 재배치(링커가 소비)와 PE의 베이스 재배치(로더가 소비)는 **같은 문제("주소를 모른다")를 다른 시점에 푸는 같은 구조**다. 이 대칭을 말할 수 있으면 링킹과 로딩을 하나의 원리로 묶은 것이다.

### name mangling — 링커는 C++을 모른다

링커는 C와 Fortran 시대에 설계된 도구다. 아는 것은 **평평한 문자열 이름 하나당 심볼 하나**뿐이고, 오버로딩·네임스페이스·템플릿을 모른다. C++ 설계 시 선택지는 둘이었다: 세상의 모든 링커를 고치거나, C++ 쪽에서 타입 정보를 문자열에 인코딩하거나. 후자가 name mangling이다 — **오버로딩은 언어 기능이 아니라 이름 인코딩으로 구현된 것**이다.

```cpp
void Foo(int);     // MSVC: ?Foo@@YAXH@Z        Itanium: _Z3Fooi
void Foo(float);   // MSVC: ?Foo@@YAXM@Z        Itanium: _Z3Foof
namespace Bar { void Foo(int); }
                   // MSVC: ?Foo@Bar@@YAXH@Z    Itanium: _ZN3Bar3FooEi
```

같은 `Foo`가 시그니처마다 다른 심볼이 되므로 링커는 구분할 필요조차 없다. 네임스페이스·클래스 스코프도 이름에 접합된다(MSVC는 `@스코프@`, Itanium은 `N...E` 블록). 인코딩 규칙은 컴파일러마다 다르다 — **MSVC와 GCC의 mangling이 다르므로, 다른 컴파일러가 만든 C++ 바이너리는 심볼 이름 수준에서부터 이미 호환되지 않는다.** ABI 비호환의 첫 관문이 링크조차 안 되는 것으로 나타나는 이유다.

`extern "C"`의 정체는 이 인코딩을 끄는 스위치다. 심볼 이름이 함수 이름 그대로가 되므로(x64 MSVC 기준 장식 없음), (1) 오버로딩이 불가능해지고(이름이 같으면 심볼이 충돌하니까), (2) C 코드·타 언어(FFI)·`GetProcAddress` 같은 **이름 문자열로 심볼을 찾는 모든 소비자**와 접점이 생긴다. "C ABI로 노출한다"는 말은 정확히는 "mangling을 끄고, C가 표현 가능한 타입만 시그니처에 쓴다"는 두 가지 규율의 합이다.

링커 에러를 읽는 실전 요령도 mangling에서 나온다. LNK2019가 뱉는 것은 사람이 쓴 이름이 아니라 mangled name이다:

```
Client.obj : error LNK2019: unresolved external symbol
  "__declspec(dllimport) public: unsigned __int64 __cdecl CWorld::CreateEntity(void)"
  (__imp_?CreateEntity@CWorld@@QEAA_KXZ)
  referenced in function "void __cdecl SpawnChampion(void)" (?SpawnChampion@@YAXXZ)
```

이 한 덩어리에 필요한 정보가 전부 있다. 읽는 순서: (1) **어느 .obj가** 참조했나(`Client.obj` — 소비자 쪽 문제인지 정의 쪽 문제인지의 첫 갈림), (2) **정확히 어떤 시그니처를** 찾았나 — MSVC는 mangled name을 사람이 읽을 형태로 풀어서 같이 보여주고, 안 풀린 이름은 `undname.exe`(MSVC)나 `c++filt`(GCC/Clang)로 푼다. "선언은 `const&`인데 정의는 값으로 받았다" 같은 불일치는 찾던 시그니처와 정의한 시그니처를 나란히 놓는 순간 보인다, (3) **`__imp_` 접두가 있나** — 있으면 컴파일러가 dllimport 경로(IAT 슬롯)로 부르도록 코드를 만들었다는 뜻이므로, 정의 .cpp가 아니라 import lib 링크 누락·dllexport 누락·stale lib부터 의심한다. 없으면 일반 심볼 해석 실패다. 에러 메시지의 접두 하나가 "링크 입력 문제인가, DLL export 문제인가"를 갈라준다.

### 정적 링킹 vs 동적 링킹 — 복사냐 참조냐

| 축 | 정적 링킹 (.lib → EXE에 복사) | 동적 링킹 (.dll + import lib) |
|---|---|---|
| 코드 존재 | 사용 모듈마다 사본 | 프로세스/시스템에 한 벌 |
| 주소 확정 | 링크 타임 | 로드 타임 (IAT/재배치) |
| 배포 | 파일 하나로 끝 | DLL 동봉 필요, 버전 관리 문제 |
| 패치 | 전체 재배포 | DLL만 교체 가능 |
| 메모리 | 프로세스마다 중복 | 코드 페이지 프로세스 간 공유 |
| 전역 상태 | 모듈마다 별도 사본 (함정) | DLL 안에 단일 사본 |
| ABI 노출 | 링크 타임에 전부 검증 | **경계의 계약이 런타임까지 살아있음** |

트레이드오프의 본질: 정적 링킹은 "모든 결정을 빌드 타임에" 내려 배포를 단순화하고, 동적 링킹은 결정을 로드 타임으로 미뤄 공유·교체 가능성을 얻는 대신 **버전·ABI 불일치라는 새 실패 모드**를 만든다. DLL 지옥도, STL 경계 금지도 전부 이 "미뤄진 결정"의 대가다.

### DLL 메커니즘 — export 테이블, import 테이블, IAT

Windows에서 DLL 심볼은 기본 비공개다. `__declspec(dllexport)`가 심볼을 DLL의 **export 테이블**(이름/서수 → RVA 매핑)에 올리고, 소비자 쪽 `__declspec(dllimport)`는 호출을 **임포트 주소 테이블(Import Address Table, IAT)** 경유 간접 호출로 컴파일하게 한다:

```asm
; __declspec(dllimport) 선언이 있을 때 — IAT 슬롯을 통한 간접 호출
call    qword ptr [__imp_?CreateEntity@CWorld@@QEAA_KXZ]

; dllimport 없이 함수 선언만 있을 때 — 직접 호출로 컴파일되고,
; 링커가 임포트 함수임을 발견하면 점프 썽크(thunk)를 합성해 한 단계 경유
call    CreateEntity_thunk
CreateEntity_thunk:
        jmp     qword ptr [__imp_...]        ; 결국 IAT로
```

**왜 간접 호출인가** — DLL이 어디에 로드될지 링크 타임에 모르기 때문이다. 여기서 재배치와의 설계 대비가 선명하다: 호출 지점마다 주소를 박으면 로더가 수천 개의 코드 자리를 고쳐야 하지만(코드 페이지가 전부 더럽혀짐), **IAT라는 데이터 슬롯 하나에 몰아넣으면 로더는 함수당 포인터 하나만 쓰면 된다.** 코드 섹션은 읽기 전용·공유 가능으로 남는다. 간접 계층 하나로 "고칠 자리"를 N에서 1로 줄인 것 — 링킹 전체에서 반복되는 패턴이다.

함수는 dllimport가 없어도 썽크 한 단계 손해로 링크가 되지만, **데이터 심볼(전역 변수)은 dllimport가 필수**다. `mov eax, [g_value]` 같은 직접 접근 코드를 링커가 간접 접근으로 바꿔줄 수 없기 때문이다(명령어 자체가 달라져야 한다). import lib(.lib)의 정체는 코드가 아니라 "이 심볼은 저 DLL에 있다"는 메타데이터 + 썽크 묶음이다 — 그래서 DLL을 새로 빌드하고 오래된 import lib에 링크하면 어긋난다.

DLL과 연결되는 시점은 하나가 아니다. 두 방식이 서로 다른 실패 모드와 유연성을 갖는다:

| 축 | 암시적 링킹 (import lib) | 명시적 링킹 (`LoadLibrary`+`GetProcAddress`) |
|---|---|---|
| 연결 시점 | 프로세스 로드 (로더가 IAT 채움) | 코드가 원하는 임의 시점 |
| DLL 부재 시 | **프로세스 기동 자체가 실패** | 반환값 검사로 코드가 대응 가능 |
| 심볼 접근 | 헤더의 타입 그대로 (컴파일러가 검증) | 이름 문자열 → 함수 포인터 캐스팅 (사람이 계약 책임) |
| 쓰임 | 항상 필요한 핵심 모듈 (엔진 DLL) | 플러그인, 선택적 기능, 버전 분기 |

명시적 링킹이 플러그인 아키텍처의 기반인 이유가 표의 마지막 두 줄이다: 어떤 DLL이 올지 빌드 타임에 모르므로 import lib이 있을 수 없고, `GetProcAddress`는 이름 문자열로 export 테이블을 검색하므로 **mangled name을 문자열로 쓸 수는 없으니** 플러그인의 진입점은 관례적으로 `extern "C"`가 된다 — mangling·export 테이블·명시적 링킹이 한 지점에서 만난다.

### 로더 — 프로세스가 태어나는 순서

```
CreateProcess("Client.exe")
 ├─ 커널: 주소 공간 생성, Client.exe + ntdll.dll 매핑
 └─ ntdll 로더 (유저 모드):
     1. Client.exe의 임포트 디렉터리 파싱 → "WintersEngine.dll 필요"
     2. WintersEngine.dll 매핑
        · 선호 베이스 충돌 or ASLR → rebase: .reloc 순회하며 절대 주소 보정
     3. WintersEngine.dll의 임포트를 재귀 처리 (d3d11.dll, ...)
        → 의존성 그래프 완성 (순환은 허용되되 초기화 순서가 불완전해짐)
     4. 각 모듈의 IAT 채우기
        · export 테이블에서 이름/서수로 검색 → 실제 주소 기록
        · 못 찾으면 이 시점에 프로세스 기동 실패 ("프로시저를 찾을 수 없습니다")
     5. 의존성 역순으로 DllMain(DLL_PROCESS_ATTACH) 호출 — 로더 락 보유 상태
 └─ CRT 엔트리(mainCRTStartup) → CRT 초기화 → 전역 생성자 → main()
```

로더가 **의존성 그래프의 잎부터** 초기화하는 이유는 자명하다 — 내가 부를 DLL이 먼저 준비되어 있어야 하니까. `DllMain`이 로더 락(loader lock) 안에서 불리므로 그 안에서 `LoadLibrary`나 스레드 동기화를 하면 교착이 나는 것도 이 구조의 직접 귀결이다.

2번 단계의 "DLL을 찾는다"에도 규칙이 있다 — 임포트 테이블에는 경로가 아니라 **파일 이름만** 있으므로, 로더는 검색 순서(대략: 이미 로드된 모듈 → EXE 디렉터리 → 시스템 디렉터리 → PATH)로 이름을 해석한다. 같은 이름의 다른 DLL이 검색 경로 앞쪽에 있으면 그것이 로드된다 — DLL 하이재킹(planting) 공격과 "내 옆의 DLL이 먼저"라는 앱 로컬 배포 전략이 같은 규칙의 양면이다.

**DLL 지옥(DLL hell)**은 이 메커니즘의 역사적 실패 사례다: 여러 앱이 시스템 디렉터리의 공유 DLL 한 벌을 쓰던 시절, 한 앱의 설치가 DLL을 다른 버전으로 갈아치우면 다른 앱이 깨졌다. 심볼은 이름으로만 찾으니 "이름은 같은데 동작·레이아웃이 다른" DLL을 로더는 구분할 수 없다. 현대적 해법은 결국 공유를 포기하는 방향 — **앱 로컬 배포(EXE 옆에 자기 DLL 동봉)**, 매니페스트/SxS, 패키지 매니저의 버전 고정 — 으로 수렴했다. Winters도 빌드 후 이벤트로 엔진 DLL을 클라이언트 출력 폴더에 복사한다(④).

### ABI (1) — 호출 규약: 레지스터에 인자를 싣는 계약

호출자와 피호출자가 따로 컴파일되는 이상, "인자를 어디에 두고 반환값을 어디서 받는가"는 사전에 못 박힌 계약이어야 한다. 이것이 호출 규약(calling convention)이고 ABI의 첫 조항이다.

| 항목 | x64 Windows | System V AMD64 (Linux/macOS) |
|---|---|---|
| 정수/포인터 인자 | RCX, RDX, R8, R9 (4개) | RDI, RSI, RDX, RCX, R8, R9 (6개) |
| 부동소수 인자 | XMM0~XMM3 | XMM0~XMM7 |
| 슬롯 계산 | 정수·부동 **합산 4슬롯, 위치 고정** (2번째 인자가 float면 XMM1) | 정수/부동 **별도 카운트** |
| 5번째 이후 | 스택 | 7번째 이후 스택 |
| shadow space | **32바이트, 호출자가 예약** | 없음 |
| red zone | 없음 | RSP 아래 128바이트 사용 가능 |
| 반환 | RAX (float는 XMM0) | RAX(+RDX), XMM0(+XMM1) |
| callee-saved | RBX, RBP, RDI, RSI, R12~R15, XMM6~XMM15 | RBX, RBP, R12~R15 |

멤버 함수의 숨은 첫 인자 `this`는 x64 Windows에서 RCX로 들어간다 — "멤버 함수 = this를 첫 인자로 받는 함수"라는 말이 ABI 수준에서 문자 그대로다. 구체 예 하나로 슬롯 배치를 못 박자:

```cpp
float ApplyDamage(CUnit* pUnit, int amount, float multiplier, bool isCrit);
```

```asm
; x64 Windows — 인자 위치는 "몇 번째 인자인가"로만 결정된다
mov     rcx, rbx            ; 1번째 pUnit      → RCX  (정수/포인터 슬롯 1)
mov     edx, 30             ; 2번째 amount     → EDX  (슬롯 2)
movss   xmm2, [rip+...]     ; 3번째 multiplier → XMM2 (슬롯 3 — XMM0이 아니라 XMM2!)
mov     r9b, 1              ; 4번째 isCrit     → R9B  (슬롯 4)
call    ApplyDamage
; 반환 float → XMM0
```

3번째 인자가 float인데 XMM0이 아니라 **XMM2**로 가는 것이 Windows 규약의 "위치 고정 슬롯" 규칙이다(System V라면 부동소수 카운트를 따로 세서 XMM0). 이 배치를 아느냐가 크래시 덤프에서 레지스터만 보고 "이 함수에 뭐가 들어왔었나"를 읽어내는 능력으로 직결된다 — 호출 규약을 외우는 실익이 디버깅에 있다.

**shadow space(그림자 공간)는 왜 있나.** 호출자가 스택에 32바이트(레지스터 인자 4개 분량)를 예약해 두면, 피호출자는 필요할 때 레지스터 인자를 **자기 프레임의 정해진 자리(home)에 내려놓을 수 있다.** 가변 인자 함수는 인자를 연속 메모리로 봐야 하고, 디버거는 인자를 메모리에서 읽고 싶어 한다 — 이 요구를 "항상 예약"이라는 균일한 규칙으로 해결했다. 대가는 인자가 없어도 호출마다 32바이트를 쓰는 것. System V는 반대로 shadow space를 없애고 red zone(리프 함수가 스택 포인터를 안 내리고 쓸 수 있는 128바이트)을 두었다 — 같은 문제 공간에서 다른 트레이드오프를 고른 것이므로, 두 규약의 차이는 암기가 아니라 설계 선택으로 설명하는 게 좋다.

caller-saved(volatile)/callee-saved(non-volatile) 구분도 계약이다: 호출을 건너 살아남아야 하는 값은 callee-saved에 두면 피호출자가 보존 책임을 지고, 수명이 짧은 값은 volatile에 두면 저장/복원 비용이 없다. 컴파일러의 레지스터 할당이 이 계약 위에서 논다.

### ABI (2) — 스택 프레임과 프롤로그/에필로그

```asm
; void Foo(int a, int b) — 전형적 프레임 (개념 형태)
push    rbp                 ; 이전 프레임 포인터 보존 (callee-saved)
mov     rbp, rsp            ; 내 프레임 기준점 설정
sub     rsp, 48             ; 지역 변수 + 내가 호출할 함수용 shadow space
;   ... 본문 ...
mov     rsp, rbp            ; 에필로그: 스택 복원
pop     rbp
ret
```

```
높은 주소
│  호출자의 프레임          │
│  [shadow space 32B]      │ ← 호출자가 예약 (내 인자 home)
│  복귀 주소                │ ← call이 push
│  저장된 RBP              │ ← 프롤로그
│  지역 변수들              │
│  [내가 부를 함수의 shadow] │
낮은 주소                     ← RSP
```

x64 Windows는 스택이 **call 직전 16바이트 정렬**이어야 한다(SSE 정렬 요구). 또 하나 중요한 현대적 사실: MSVC x64는 최적화 빌드에서 RBP 프레임 체인을 자주 생략하고, 대신 **모든 함수의 프레임 구조를 언와인드 테이블(.pdata/.xdata)에 정적으로 기술**한다. "프레임 포인터를 따라가며 스택을 푼다"는 x86-32 시대의 그림이 x64에서는 "테이블을 보고 푼다"로 바뀌었다 — 이 전환이 아래 예외 ABI의 복선이다.

### ABI (3) — 구조체 레이아웃: 정렬과 패딩

각 멤버는 자기 타입의 정렬(alignment) 배수 오프셋에 놓여야 하고, 구조체 전체 정렬은 멤버 정렬의 최댓값, 전체 크기는 그 정렬의 배수다(배열에서 연속 배치가 성립하려면 꼬리 패딩까지 포함해야 하므로). 컴파일러는 **멤버 순서를 바꿔주지 않는다** — C 호환성과 "메모리 배치는 프로그래머의 것"이라는 원칙 때문에, 선언 순서가 곧 메모리 순서다. 그래서 패딩은 프로그래머 책임이다:

```cpp
struct Bad  { char tag; /*pad 3*/ int hp; char team; /*pad 3*/ int mp; }; // 16B
struct Good { int hp; int mp; char tag; char team; /*pad 2*/ };           // 12B
```

패딩이 왜 존재하는가부터 답할 수 있어야 한다. 많은 CPU에서 정렬 안 된 접근은 느리거나(캐시 라인/워드 경계 걸침) 아예 불가하며(일부 아키텍처, SIMD 정렬 로드), atomic 연산은 정렬을 전제한다. 컴파일러는 "멤버 접근 = 단순 로드 명령 하나"라는 성능 계약을 지키기 위해 빈 공간을 삽입하는 쪽을 택했다 — 메모리를 버려서 접근 속도를 산 것이다. `alignas(64)`처럼 정렬을 올리는 것은 이 메커니즘의 의도적 활용이다(캐시 라인 분리로 false sharing 방지 — cpp/01의 `WorkStealingDeque` 사례).

이것이 ABI 조항인 이유: 두 모듈이 같은 헤더의 같은 구조체를 주고받아도, 한쪽이 `#pragma pack(1)` 영향 아래서 컴파일됐다면 **같은 선언이 다른 레이아웃**이 된다. 컴파일 에러도 링크 에러도 없이, 필드를 읽는 순간 어긋난 바이트를 읽는다. 구조체 레이아웃 불일치는 도구가 못 잡는 ABI 드리프트의 대표 사례이고, 그래서 직렬화 경계·DLL 경계·GPU 상수 버퍼 경계에서는 레이아웃 가정을 컴파일 타임 검증으로 계약서에 박는 것이 방어책이 된다:

```cpp
// 와이어 포맷/경계 구조체 옆에 상시 배치 — 레이아웃이 흔들리면 빌드가 깨진다
static_assert(sizeof(SnapshotHeader) == 24, "wire layout drifted");
static_assert(offsetof(SnapshotHeader, tick) == 8, "field moved");
```

패딩 바이트에는 값의 보장이 없다는 것도 경계에서 중요하다 — 구조체를 통째로 `memcmp`로 비교하거나 해시하면 쓰레기 패딩이 결과를 오염시킨다. 와이어로 내보내는 구조체는 패딩이 없도록 멤버를 배치하거나 명시적으로 0을 채우는 것이 규율이다.

### ABI (4) — vtable의 실제 레이아웃과 가상 호출의 어셈블리

`virtual`이 하나라도 있는 클래스의 객체는 선두에 vptr(가상 테이블 포인터) 한 개를 숨겨 갖고, vptr는 클래스당 한 벌 `.rdata`에 놓인 **함수 포인터 배열(vtable)**을 가리킨다:

```
객체 (힙/스택)              vtable (.rdata, 클래스당 1벌)
┌──────────────┐          ┌────────────────────────┐
│ vptr ─────────┼────────→ │ [0] &CBTNode::Tick     │
│ m_lastStatus  │          │ [1] &CBTNode::GetName  │
│ ...멤버들...   │          │ [2] &CBTNode::Reset    │
└──────────────┘          └────────────────────────┘
```

가상 호출의 어셈블리(x64 MSVC 개념 형태):

```asm
; pNode->Tick(ctx)   — pNode는 RCX(this)
mov     rax, qword ptr [rcx]        ; ① 객체 선두에서 vptr 로드   (메모리 접근 1)
call    qword ptr [rax+0]           ; ② vtable 슬롯에서 주소 로드 후 간접 호출 (메모리 접근 2)
```

비가상 호출은 `call ?Tick@...`(주소가 링크 타임 확정, 인라인 가능) 한 줄인 것과 대비된다. 가상 호출의 비용은 흔히 말하는 "느린 간접 호출" 이전에 **의존적 메모리 접근 2회**다 — 객체가 캐시에 없으면 vptr 로드가 미스, vtable이 캐시에 없으면 슬롯 로드가 또 미스. 그리고 진짜 비용은 대개 그다음이다: 호출 대상이 런타임에 결정되므로 **컴파일러가 인라인을 못 하고**, 인라인이 막히면 그 뒤의 모든 최적화(상수 전파, 죽은 코드 제거)가 연쇄로 막힌다.

파생 클래스는 자기 vtable 사본에서 오버라이드한 슬롯만 자기 함수로 바꾼다 — "같은 슬롯 번호 = 같은 의미"라는 **슬롯 배치 자체가 ABI**다. 그래서 공개 헤더의 클래스에서 가상 함수의 선언 순서를 바꾸거나 중간에 하나 끼워 넣으면, 재컴파일 안 한 소비자는 어긋난 슬롯을 호출한다(아래 ⑤ 드리프트 질문). 다중 상속에서는 vptr가 여러 개 생기고 this 보정 썽크가 끼는 등 레이아웃이 급격히 복잡해지며, vtable 앞쪽에 RTTI 정보를 어떻게 붙이는지도 구현마다 다르다(MSVC와 Itanium ABI가 서로 다른 방식) — 세부는 구현 의존이라고 한 단계 물러서 말하는 것이 안전하다.

### ABI (5) — 예외 ABI: 프레임 기반 vs 테이블 기반, "제로 코스트"의 의미

예외의 어려움은 던지는 쪽이 아니라 **스택을 거슬러 오르며 지역 객체 소멸자를 전부 호출해야 한다(unwinding)**는 데 있다. 두 가지 구현 계보가 있다:

- **프레임 기반 (x86-32 Windows SEH)**: 함수에 들어갈 때마다 예외 핸들러 레코드를 스레드별 연결 리스트(FS:[0])에 등록하고 나갈 때 해제한다. 구조가 단순하고 동적이지만, **예외가 안 나도 함수 진입/탈출마다 등록 비용을 낸다.**
- **테이블 기반 (x64 Windows SEH, Itanium ABI의 .eh_frame)**: 컴파일러가 "이 코드 주소 범위에 있을 때 스택을 이렇게 풀고, 이 객체들이 살아 있다"는 정보를 **정적 테이블**(x64 Windows: .pdata/.xdata, Itanium: unwind 테이블 + LSDA)로 빌드 타임에 굽는다. 정상 실행 경로에는 명령어가 하나도 추가되지 않는다.

| 축 | 프레임 기반 (x86-32 SEH) | 테이블 기반 (x64 SEH / Itanium) |
|---|---|---|
| 정상 경로 비용 | 함수마다 등록/해제 명령 | **0 명령** |
| throw 경로 비용 | 리스트 순회 — 상대적으로 단순 | 복귀 주소로 테이블 검색 + 프레임 복원 — 비쌈 |
| 바이너리 크기 | 코드에 분산 | 언와인드 테이블이 별도 섹션 차지 |
| 부가 효과 | — | 같은 테이블로 디버거·프로파일러의 스택 워킹도 해결 |

마지막 줄이 설계의 보너스다: x64 Windows는 예외가 없어도 **모든 함수의 언와인드 정보를 의무화**했고, 그 덕에 프레임 포인터 없이도 정확한 콜스택 복원이 가능해졌다(크래시 덤프 분석·샘플링 프로파일러가 이 테이블을 쓴다). 예외 기계가 관측성 인프라를 겸하는 구조다.

"제로 코스트 예외"는 이 테이블 방식을 가리킨다 — 정확히는 **"던지지 않는 한 제로"**다. throw가 나면 런타임이 복귀 주소로 테이블을 검색해 프레임을 하나씩 풀어야 하므로 예외 경로는 오히려 매우 비싸다(Itanium 계열은 핸들러 탐색과 정리 실행의 2단계 언와인드). 비용을 "항상 조금씩"에서 "예외 시에만 왕창"으로 옮긴 설계이고, 그래서 예외는 진짜 예외 상황에만 쓰라는 격언이 ABI 수준에서 근거를 갖는다. 참고로 MSVC의 C++ 예외는 Windows 구조적 예외 처리(SEH) 기계 위에 C++ 전용 핸들러를 얹은 구현이라, "Windows에서 C++ 예외 ABI = SEH 기반"이라고 말할 수 있다. 완전히 무비용도 아니다 — 테이블 자체가 바이너리 크기를 차지하고, 예외 가능성이 최적화(코드 이동)를 제약하는 간접 비용이 남는다.

### C++에 표준 ABI가 없다 — DLL 경계 규칙의 근거

C++ 표준은 소스 수준의 의미만 정의하고, mangling·레이아웃·vtable·예외 전파 방식은 전부 구현에 맡겼다. 이유는 설계의 자유 — 구현체가 레이아웃과 호출 방식을 계속 개선할 수 있게 하기 위해서다. 결과적으로 Linux 진영은 Itanium C++ ABI라는 사실상 표준으로 수렴했고, MSVC는 독자 ABI를 유지한다(VS 2015 이후 툴셋들끼리는 바이너리 호환을 유지하는 정책이지만, 이는 벤더의 약속이지 언어 표준이 아니다).

DLL 경계에서 STL을 넘기지 말라는 규칙이 여기서 도출된다. `std::string`을 경계로 넘기는 순간 세 겹의 암묵 계약이 생긴다: (1) **레이아웃** — 양쪽이 그 구현·설정에서 같은 멤버 배치를 볼 것(`_ITERATOR_DEBUG_LEVEL` 하나만 달라도 깨진다), (2) **할당자/힙** — 한쪽이 할당한 메모리를 다른 쪽이 해제해도 같은 힙일 것(/MT 정적 CRT면 모듈마다 힙이 달라 즉시 힙 손상), (3) **인라인 코드 일치** — 템플릿이라 양쪽 모듈에 각자 인스턴스화된 조작 코드가 동일할 것. 셋 다 언어가 보장하지 않고 **빌드 규율만이 보장**한다. 그래서 실무 선택지는 둘이다: 규율을 성문화하고 경계를 열거나(사내 모든 모듈 = 같은 컴파일러·같은 표준·같은 /MD — Winters의 선택, ④), 규율을 전제할 수 없으면 경계를 C ABI 수준(불투명 핸들, 호출자 버퍼, 순수 가상 인터페이스, pimpl)으로 내리거나. 예외도 마찬가지로 경계를 넘기면 안 된다 — 언와인드 방식과 예외 객체 표현이 모듈 간 일치한다는 보장이 없으므로, 경계에서는 에러 코드나 상태로 바꿔 넘긴다.

### CRT와 런타임 — main 이전과 이후

`main`은 프로그램의 시작이 아니다. PE 헤더의 엔트리 포인트는 CRT의 `mainCRTStartup`이고, 거기서:

```
mainCRTStartup
 ├─ 보안 쿠키 초기화 (/GS 스택 카나리 시드)
 ├─ CRT 내부 초기화 (힙, 표준 스트림, 로케일)
 ├─ C 이니셜라이저 실행 (.CRT$XI* 구간의 함수 포인터 배열 순회)
 ├─ C++ 동적 초기화 실행 (.CRT$XC* 구간 — 전역 객체 생성자들)
 │    · 이때 각 객체의 소멸자가 atexit로 등록됨
 ├─ main(argc, argv)
 └─ exit → atexit 핸들러 LIFO 실행 (= 정적 소멸자들, 생성의 역순)
```

전역 생성자 호출 메커니즘이 우아하다: 컴파일러는 동적 초기화가 필요한 전역마다 초기화 함수를 만들어 **그 함수의 포인터를 특정 이름의 섹션(.CRT$XC 계열)에 넣고**, 링커의 "같은 이름 섹션은 이름순으로 이어 붙인다"는 성질이 이 포인터들을 하나의 배열로 병합한다. CRT는 그 배열을 순회할 뿐이다 — 링커의 섹션 병합이 곧 초기화 목록 자료구조인 것이다.

문제는 순서다. **한 TU 안에서는 선언 순서, TU 사이에서는 순서 미보장** — 이것이 정적 초기화 순서 재앙(static initialization order fiasco)이다. 전역 A의 생성자가 다른 TU의 전역 B를 쓰면, B가 아직 생성 전일 수 있다. 소멸은 생성의 역순이므로 **소멸 순서 재앙도 대칭으로 존재한다** — 종료 시 싱글턴 A의 소멸자가 이미 죽은 싱글턴 B를 참조하는 사고. 표준적 처방이 함수 지역 정적(Meyers 싱글턴)이다: 첫 호출 시점에 초기화되므로 "사용 전 생성"이 자동 보장되고, C++11부터는 이 초기화가 스레드 안전(magic statics)하다. Winters의 `DECLARE_SINGLETON`이 정확히 이 형태다(④). DLL이 끼면 층이 하나 더 생긴다: 각 DLL의 전역 생성자는 그 DLL의 `DllMain(PROCESS_ATTACH)` 시점(로더 락 안)에 돌므로, EXE의 `main`보다도 먼저, 그리고 모듈별로 따로 논다 — "전역 초기화는 프로세스 전체에 한 줄로 서지 않는다"가 멀티 모듈의 현실이다.

## ③ 심화 — 꼬리질문 대비

**"컴파일은 됐는데 링크가 안 되는 것과, 링크는 됐는데 실행이 안 되는 것 — 각각 무엇이 깨진 건가?"**
전자는 심볼 해석 실패(선언은 있었지만 정의를 가진 .obj/.lib이 링크 입력에 없음 — LNK2019), 후자는 로드 타임 해석 실패(링크는 import lib의 약속만 보고 통과했는데, 실제 DLL이 없거나 그 DLL의 export 테이블에 심볼이 없음). 같은 "심볼을 못 찾음"이 정적 링킹에서는 빌드 에러로, 동적 링킹에서는 프로세스 기동 실패로 시점만 옮겨져 나타난다 — 동적 링킹이 "결정을 미룬 것"임을 보여주는 가장 좋은 예.

**"inline 함수의 ODR 위반은 왜 링커가 못 잡나?"**
inline/템플릿 심볼은 COMDAT(select-any)로 들어가고, 링커의 계약이 "여러 정의 중 하나를 골라 쓴다"이지 "모든 정의가 같은지 검사한다"가 아니기 때문이다. 검사하려면 TU마다 다른 최적화·매크로 환경에서 나온 기계어를 의미 수준에서 비교해야 하는데 이는 비현실적이라, 표준이 아예 "동일하지 않으면 미정의 동작, 진단 의무 없음"으로 못 박았다. 실무 방어는 기계가 아니라 규율이다: 헤더의 inline 코드가 빌드 설정 매크로에 의존하지 않게 하고, 설정이 갈리는 코드는 .cpp로 내린다.

**"strong/weak 심볼을 실무에서 언제 체감하나?"**
(1) 헤더에 non-inline 함수를 정의해 두 TU에 포함 → strong 중복 → LNK2005. (2) inline으로 바꾸면 COMDAT이 되어 통과 — 에러가 사라진 게 아니라 검사가 사라진 것. (3) ELF에서 라이브러리가 기본 구현을 weak로 내보내고 사용자가 strong으로 오버라이드하는 패턴(malloc 훅류). MSVC에는 사용자용 weak가 표준으로 노출되진 않지만 COMDAT과 `/ALTERNATENAME` 같은 링커 장치가 유사 역할을 한다 — "링커 수준에는 '대체 가능한 정의'라는 개념이 있고 C++의 inline이 그 위에 구현됐다"가 답의 뼈대.

**"정적 라이브러리에 넣은 자기 등록 팩토리가 동작하지 않는다. 왜인가?"**
링커는 .lib에서 미해결 심볼을 정의하는 .obj만 꺼낸다. 등록용 전역 객체만 있는 .obj는 아무도 그 심볼을 참조하지 않으므로 링크에 포함되지 않고, 전역 생성자도 실행되지 않는다. 해법: 그 .obj의 심볼을 명시 참조하거나, `/WHOLEARCHIVE`·`/INCLUDE`로 강제 포함하거나, 등록을 명시적 초기화 함수 호출로 바꾼다. "링커는 참조 그래프의 도달 가능한 것만 가져온다"는 모델을 아는지 묻는 단골 질문.

**"dllimport 없이 DLL 함수를 선언만 하고 부르면 어떻게 되나? 데이터는?"**
함수는 링커가 점프 썽크를 합성해 동작한다 — 직접 호출 명령이 썽크로, 썽크가 IAT로 점프하는 한 단계 손해. dllimport를 붙이면 처음부터 IAT 간접 호출로 컴파일되어 썽크가 없다. 데이터는 불가능하다 — 직접 주소 접근으로 컴파일된 명령을 링커가 간접 접근으로 재작성할 수 없기 때문(명령어 인코딩 자체가 다르다). 그래서 export되는 전역 데이터는 dllimport 선언이 필수이고, 애초에 데이터를 경계로 노출하지 말고 접근 함수로 감싸는 것이 더 나은 설계다.

**"가상 함수 호출이 느리다고들 하는데, 진짜 비용은 무엇인가?"**
간접 호출 명령 자체는 현대 CPU의 간접 분기 예측이 대부분 흡수한다. 실제 비용은 (1) 의존적 메모리 접근 2회(vptr 로드 → vtable 슬롯 로드)의 캐시 미스 가능성, (2) **인라인 불가** — 호출 대상이 런타임 결정이라 컴파일러가 본문을 펼칠 수 없고, 인라인이 막히면 후속 최적화가 연쇄로 막힌다, (3) 이종 타입이 섞인 컨테이너 순회에서 분기 예측 미스 증가. 그래서 "hot loop 안의 virtual"이 문제이지 프레임당 몇 번의 virtual은 문제가 아니다 — 비용의 위치를 정확히 말하는 것이 포인트.

**"vtable은 어느 모듈, 어느 섹션에 존재하나?"**
읽기 전용 데이터(.rdata)에 클래스당 한 벌이 원칙이지만, "어느 TU가 방출하나"는 구현 의존이다. Itanium ABI는 첫 non-inline 가상 함수(key function)를 정의한 TU가 방출한다는 규칙이 있고, MSVC는 필요한 TU마다 방출해 COMDAT으로 병합하는 방식이다. DLL 경계에서 실무적으로 중요한 귀결: vtable의 슬롯들은 절대 주소이므로 베이스 재배치 대상이고, 클래스를 dllexport하면 vtable도 그 DLL의 것이 쓰이도록 정리된다. "vtable이 두 모듈에 두 벌 생겨 포인터 비교·RTTI가 어긋날 수 있다"까지 말하면 mangling·COMDAT·DLL이 한 줄로 꿰인다.

**"shadow space가 없으면 무엇이 안 되나?"**
가변 인자 함수가 레지스터로 온 인자들을 연속 메모리로 볼 방법이 사라지고, 디버거·스택 워커가 인자를 메모리에서 읽을 고정 위치가 없어진다. System V는 shadow space 없이 가변 인자를 별도 저장 영역(register save area)을 피호출자가 만드는 방식으로 푼다 — 즉 "누가 저장 공간을 만드나"를 호출자(Windows)에 둘지 피호출자(System V)에 둘지의 설계 차이다. 이렇게 답하면 규약 암기가 아니라 문제-해법 구조로 이해했음을 보여준다.

**"예외가 '제로 코스트'라면서 게임 엔진은 왜 끄는 경우가 많나?"**
제로인 것은 정상 경로의 실행 명령 수뿐이다. 남는 비용: (1) 언와인드 테이블의 바이너리 크기, (2) "이 지점에서 예외가 날 수 있다"는 가정이 컴파일러의 코드 재배치·값 수명 최적화를 제약, (3) throw 경로 자체는 테이블 검색+프레임 해제로 매우 비싸고 소요 시간 예측 불가 — 프레임 예산이 밀리초인 게임에서 "가끔 수십 µs~ms급 스파이크"는 스터터다. 그리고 (4) DLL 경계·스레드 경계를 넘는 예외는 ABI 보장이 없다. 그래서 게임은 예외 대신 에러 코드/Result 타입을 쓰고 예외를 빌드에서 끄는 선택이 흔하다 — "제로 코스트의 '코스트'가 무엇의 비용인지"를 가르는 질문.

**"/MT와 /MD의 차이가 왜 DLL 경계 사고로 이어지나?"**
/MT는 CRT를 각 모듈에 정적 링크한다 — 모듈마다 자기 CRT, 자기 힙. A 모듈의 `new`로 만든 메모리를 B 모듈이 `delete`하면 다른 힙에 반납 시도 → 힙 손상. /MD는 모든 모듈이 하나의 CRT DLL(하나의 힙)을 공유해 이 축을 제거한다. 단 /MD로 통일해도 레이아웃 축(디버그/릴리스 혼합, `_ITERATOR_DEBUG_LEVEL` 불일치)은 남는다 — "힙 통일은 필요조건이지 충분조건이 아니다"까지 말해야 완결. 링커 경고 LNK4098(라이브러리 충돌)이 이 혼합의 조기 신호다.

**"ASLR은 링킹·로딩에 무슨 비용을 추가하나?"**
이미지가 임의 베이스에 로드되므로 절대 주소가 박힌 자리 전부(.reloc 목록 — vtable, 함수 포인터 전역, 점프 테이블 일부)를 로더가 보정해야 한다. 보정된 페이지는 copy-on-write로 프로세스 사유가 되어 공유 이점이 줄고 기동이 느려진다. x64가 RIP 상대 주소를 기본으로 삼은 덕에 코드 섹션의 재배치는 거의 사라졌다 — "보안(주소 예측 불가)을 위해 로드 비용과 메모리 공유를 지불한다"는 트레이드오프로 정리.

**"공개 헤더의 클래스에 멤버 변수 하나를 추가하고 DLL만 다시 빌드해 배포하면 무슨 일이 나나?"**
ABI 드리프트의 전형이다. 소비자 EXE는 옛 헤더 기준으로 `sizeof`·멤버 오프셋·(가상 함수를 바꿨다면) vtable 슬롯 번호를 코드에 구워 놨다. 새 DLL은 다른 레이아웃으로 객체를 만들므로, EXE가 어긋난 오프셋을 읽고 쓴다 — 링크도 로드도 통과하고 런타임에 조용히 깨진다. mangled name에는 시그니처만 있지 레이아웃 정보가 없어서 링커가 잡을 수 없다. 방어: 경계 타입 변경은 전 모듈 리빌드와 함께, 리빌드를 전제할 수 없으면 불투명 핸들/pimpl로 레이아웃을 경계에서 숨기고, 진짜 배포 SDK라면 인터페이스 버전 번호를 계약에 넣는다.

**"main 이전에 코드가 실행되는 경로를 아는 대로 말해보라."**
(1) 각 DLL의 `DllMain(PROCESS_ATTACH)`와 그 DLL의 전역 생성자 — 로더 락 안, main보다 먼저. (2) EXE의 CRT 초기화와 전역 생성자(.CRT 섹션 배열 순회). (3) TLS 콜백. 함정 질문의 의도는 대개 "전역 생성자에서 무거운 일/다른 모듈 의존을 하면 왜 위험한가" — TU 간 순서 미보장 + DLL 로더 락 + 아직 초기화 안 된 런타임 시설이라는 3중 지뢰밭이기 때문이다. 처방은 지연 초기화(함수 지역 정적)와 명시적 초기화 순서(엔진의 Init 단계 함수)로 옮기는 것.

**"정적 소멸 순서 재앙(fiasco)을 겪는다면 어떤 증상이고 어떻게 고치나?"**
프로그램 종료 중 크래시 — 싱글턴 A의 소멸자가 로그/리소스 매니저 B를 부르는데 B가 이미 소멸된 경우. 소멸은 atexit LIFO(생성의 역순)인데 TU 간 생성 순서가 미보장이므로 소멸 순서도 통제 밖이다. 해법 스펙트럼: 함수 지역 정적으로 "사용 시 생성"을 보장(생성 순서는 해결, 소멸 순서 의존이 남으면 여전히 위험) → 소멸자에서 다른 싱글턴을 참조하지 않는 규율 → 아예 엔진 수준의 명시적 `Init`/`Shutdown` 순서로 수명을 코드화(대형 엔진의 표준 답). "언어 기능으로 다 못 풀어서 아키텍처로 푼다"는 방향이 정답의 골격.

**"extern C로 감싼 함수 두 개를 같은 이름으로 오버로딩할 수 있나?"**
불가능하다. extern "C"는 mangling을 끄므로 두 오버로드가 같은 심볼 이름이 되어 중복 정의가 된다 — 오버로딩이 언어 의미론이 아니라 mangled name 위에 구현된 기능임을 역방향으로 증명하는 질문. C ABI 경계에서는 그래서 `Foo_i`/`Foo_f`처럼 이름에 타입을 수동 인코딩하게 된다 — mangling이 자동으로 해주던 일을 사람이 하는 것.

## ④ Winters에서의 적용

이 챕터의 개념들이 Winters에서는 "Engine을 DLL로 잘라 Client/Server/Tools가 소비한다"는 구조 결정 하나에 전부 걸려 있다.

### (1) 컴파일-링크-로드의 3단 분리가 프로젝트 구조에 그대로 있다

Engine은 DLL로 빌드된다(`Engine/Include/Engine.vcxproj:24`의 `<ConfigurationType>DynamicLibrary</ConfigurationType>`, `:54`에서 `WINTERS_ENGINE_EXPORTS` 정의). 소비자인 Client는 세 단계로 Engine을 만난다:

- **컴파일 타임 = 헤더(선언)**: `Client/Include/Client.vcxproj:57`이 `EngineSDK\inc`를 include 경로로 잡는다. EngineSDK/inc가 곧 Engine DLL의 **공개 계약서** — 여기 노출된 시그니처·구조체 레이아웃·vtable 배치가 전부 ABI 표면이다.
- **링크 타임 = import lib(약속)**: `Client/Include/Client.vcxproj:69`의 `WintersEngine.lib` — 실제 코드가 아니라 "이 심볼은 저 DLL에 있다"는 스텁 묶음이고, 링커는 이것만 보고 통과시킨다.
- **로드 타임 = DLL(정의)**: 로더가 Client.exe의 임포트 테이블에서 WintersEngine.dll을 찾아 IAT를 채운다.

`Engine/Include/WintersAPI.h:11-17`의 `WINTERS_ENGINE` 매크로가 이 경계의 스위치다 — Engine 빌드에서는 `dllexport`(export 테이블에 올림), 소비자에서는 `dllimport`(IAT 간접 호출로 컴파일), 정적 링크 툴에서는 빈 문자열. 한 헤더가 세 좌표에서 다른 코드로 컴파일된다는 것 자체가 "선언은 공유하되 정의의 소재는 빌드가 결정한다"는 이 챕터의 모델이다.

### (2) 앱 로컬 배포 — DLL 지옥의 실전 회피

`Client/Include/Client.vcxproj:74-75`의 빌드 후 이벤트가 `EngineSDK\bin\$(Configuration)\*.dll`을 Client 출력 폴더로 xcopy한다. 시스템 공유 경로에 DLL을 두지 않고 **EXE 옆에 자기 버전을 동봉** — DLL 지옥의 역사가 수렴한 현대적 해법을 그대로 쓰는 것이다. 로더의 DLL 검색 순서가 EXE 디렉터리를 우선하므로, 어떤 버전의 엔진과 함께 배포됐는지가 파일 시스템 수준에서 고정된다. 면접에서 "DLL 버전 관리 어떻게 하나"에 대한 실물 답변.

### (3) C4251 + /MD 통일 — "표준 ABI 부재"를 규율로 관리

`EngineSDK/inc/Engine_Defines.h:30`의 `#pragma warning(disable : 4251)`은 "dll-interface 클래스가 STL 멤버를 노출한다"는 경고를 끈 것인데, 이것은 방치가 아니라 **전제를 깐 결정**이다: Winters의 모든 모듈은 같은 VS 툴체인, 같은 구성 짝(Debug↔Debug), 같은 DLL CRT로 빌드된다 — `Client/Include/Client.vcxproj:60`(`MultiThreadedDebugDLL`)과 `:94`(`MultiThreadedDLL`)가 /MDd·/MD 계열 통일의 증거다. 모든 모듈이 CRT DLL의 단일 힙을 공유하므로 "A가 할당, B가 해제"가 안전해지고, 같은 툴체인이므로 STL 레이아웃·인라인 코드도 일치한다. 이 전제를 못 까는 순간(서드파티 배포 SDK) C4251은 다시 켜고 경계를 pimpl/C ABI로 내려야 한다 — "경고를 끈 게 아니라 경고의 전제 조건을 빌드 시스템으로 충족시켰다"고 말하는 것이 정확한 화법이다.

### (4) dllexport 클래스와 특수 멤버 — gotcha의 실물

MSVC는 클래스 단위 `dllexport`에서 모든 멤버(암묵 특수 멤버 포함)를 즉시 인스턴스화해 export하려 한다. `unique_ptr` 멤버가 있으면 암묵 복사 생성자 생성 시도가 deleted copy에 걸려, 복사를 아무도 안 써도 컴파일이 터진다 — 일반 클래스의 "지연 인스턴스화" 관용이 export 경계에서 깨지는 것이다. Winters의 처방이 코드에 있다: `Engine/Public/AI/BehaviorTree.h:36-45`의 `class WINTERS_ENGINE CBTNode`는 복사를 명시적으로 `= delete`하고 이동만 `= default`로 열었고, 그 하위 노드들이 `std::vector<std::unique_ptr<CBTNode>>` 멤버(`:77`)를 가진다. 명시적 delete 덕에 컴파일러가 복사 경로 생성 자체를 건너뛴다. `Engine/Public/Engine_Macro.h:52-55`의 `NO_COPY` 매크로가 이 규칙을 관용구로 굳힌 것이고, `:57-65`의 `DECLARE_SINGLETON`은 함수 지역 정적(Meyers 싱글턴)으로 초기화 순서 재앙을 회피하는 이 챕터 마지막 절의 처방과 정확히 같은 형태다.

### (5) extern "C" 두 실물 — "링커는 헤더가 아니라 심볼을 잇는다"

- `Client/Private/main.cpp:15-19`: `extern "C" { __declspec(dllexport) DWORD NvOptimusEnablement = 0x00000001; ... }`. GPU 드라이버는 실행 파일의 **export 테이블에서 이 이름 문자열을 그대로 검색**해 dGPU 사용 여부를 정한다. mangling이 켜져 있으면 이름이 달라져 드라이버가 못 찾는다 — "심볼 이름 = 외부 세계와의 계약"을 데이터 심볼 export로 보여주는 사례.
- `Shared/GameSim/Core/Debug/SimDebugOutput.h:22-23`: `Windows.h`를 include하지 않고 `extern "C" __declspec(dllimport) void __stdcall OutputDebugStringA(const char*)`를 **직접 재선언**한다. 헤더는 선언의 전달 수단일 뿐이고 링커는 심볼 이름으로만 잇기 때문에, 시그니처와 심볼 이름만 정확하면 오염 헤더(`using namespace`, dinput) 없이도 kernel32의 export에 링크된다. 덧붙일 세부: `__stdcall`은 x86-32의 호출 규약 지정이고 x64에서는 규약이 하나로 통일되어 사실상 무시된다 — 호출 규약이 "플랫폼 ABI의 조항"임을 아는지 확인하는 좋은 연결 고리다.

### 면접 화법으로 정리하면

"엔진을 DLL로 자르면서 링커·로더·ABI를 전부 실전으로 겪었다 — export/import 매크로(WintersAPI.h)로 심볼 가시성을 설계했고, import lib과 실제 DLL의 어긋남·dllexport 클래스의 특수 멤버 즉시 인스턴스화 같은 링크 계층 사고를 규칙(NO_COPY, 명시적 delete)으로 성문화했으며, STL을 경계에 노출하는 대신 '단일 툴체인 + /MD 통일'이라는 전제를 빌드 시스템에 박아 C++ 무표준 ABI를 관리했고, DLL은 앱 로컬로 배포해 버전 문제를 차단했다" — 이 네 문장이 이 챕터 전체의 요약이자 실무 증거다.
