**한 줄**: 언리얼의 데디케이티드 서버가 하는 일을 바닥부터 직접 만들며, LoL 구조의 MOBA를 서버 권위 멀티플레이로 구현한 1인 프로젝트.

**왜 자체 엔진·자체 서버인가** — 팀 프로젝트(아래 2번)에서 20TPS 스냅샷 동기화 서버를 만들며, 프레임워크가 정해준 구조 안에서는 틱 제어·시야 기반 복제 필터·치팅 검증 같은 서버 권위의 핵심을 마음대로 설계할 수 없다는 한계를 느꼈습니다. 그래서 상용 엔진이 대신 해주는 층을 직접 구현해 원리를 확보하는 쪽을 택했습니다.

**구조** — 모든 게임플레이 진실(truth)은 서버만 만듭니다:

```text
Client Input → GameCommand → Server GameSim (30Hz 고정 틱)
            → FlatBuffers Snapshot/Event → Client 보간·연출
```

- 네트워크용 identity(netId)와 시뮬 내부 identity(EntityHandle)를 분리해 경계 오염을 차단했고, **봇 AI조차 GameCommand 생산자**로 통일해 서버 진실을 직접 수정하는 경로를 없앴습니다.
- 클라이언트 예측과 서버 보정이 충돌하는 문제(우클릭 직후 캐릭터가 홱 도는 현상)는, 예측 값을 서버가 따라잡을 때까지 최대 12스냅샷 동안 보호하는 규약으로 풀었습니다.
- 지연 보상(lag compensation)을 위해 서버가 틱 히스토리를 기록합니다.

**계측으로 잡은 병목** — 프레임이 17.8ms로 떨어졌을 때 추측하지 않고 자체 CPU 프로파일러를 먼저 만들었습니다. 스키닝 갱신이 프레임의 90%(16ms)라는 것을 계측으로 확정한 뒤 불필요 갱신을 제거해 **9ms(~110fps)**로 복구했습니다.

**리팩터링을 증명하는 방법** — 시뮬레이션 결정성 하니스(SimLab)를 만들어, 108개 파일을 리팩터링한 전후에 300틱 시뮬 해시가 동일함(BB6A67502987351F)을 확인했습니다. 감이 아니라 해시로 무회귀를 증명합니다.

**에셋 파이프라인** — Assimp 기반 오프라인 컨버터로 FBX/GLB 27종을 자체 바이너리(.wmesh 등)로 전수 변환(FAIL 0). 대표 메시 60MB → 1.2MB(~50배), POD 고정 레이아웃이라 zero-copy로 GPU에 올립니다.

**부속 시스템** — 세대 핸들 sparse-set ECS + Phase 스케줄러 · DX11/DX12 선택형 RHI 추상화 · Go 마이크로서비스 백엔드(인증/매치메이킹/전적, PostgreSQL/Redis/JWT) · ImGui 툴(이펙트 튜너, 맵 에디터, F3 프로파일러) · 리플레이 기록.

**한계와 다음 단계 (정직 기재)** — 수송은 아직 TCP이며 UDP 이관은 설계 완료 상태입니다. Job 병렬화(work-stealing deque)는 구현했으나 동시성 race 수정 전까지 기본 비활성입니다. 전 코드베이스 감사로 HIGH 이슈 18건(크래시 리포팅 부재 등)을 확정해 로드맵으로 관리 중입니다.

- **명령 큐 아키텍처**: Order 구조체(타입+경로+타깃)의 deque와 Commandable 인터페이스로 유닛·건물이 같은 커맨드 계약을 공유합니다 (shift 예약 명령 지원).
- **셰이더 없는 합성**: SRCAND/SRCPAINT 비트연산 + AlphaBlend 3단으로 팀 컬러·클로킹 반투명을 구현하며 렌더링 합성의 원리를 바닥에서 확인했습니다.
- 인게임 맵 에디터에서 편집한 타일을 바이너리(Tile.dat)로 저장하고, 같은 데이터로 내비게이션 그리드를 재생성합니다.

## 6. 자기소개 <a id="6-자기소개"></a>

<!-- 질문-답 사슬: 각 문단이 독자의 질문을 열고 다음 문단이 닫는다. [ ] 부분 본인 확정 필요 -->

저는 문과 출신입니다. [게임 개발을 꿈꾸게 된 계기 1~2문장 CONFIRM] 처음에는 혼자서도 만들 수 있다고 믿었고, 실제로 혼자 배워 공포 게임 Liberation을 Steam에 출시했습니다.

시장의 답은 냉정했습니다. 완성과 출시는 해냈지만, 혼자의 속도와 품질로는 부족하다는 것을 판매 결과가 알려줬습니다. 그 실패가 저를 움직였습니다 — 제대로 배우고, 다른 사람들과 함께 만들어야 한다는 것을요. [군 복무 후] 쥬신 아카데미에 들어갔습니다.

아카데미에서는 바닥부터 다시 쌓았습니다. WinAPI만으로 스타크래프트의 코어 시스템을 구현하며 상용 엔진이 대신 해주던 것들의 원리를 확보했고, 5인 팀 프로젝트에서는 git 세팅부터 서버 개발까지 맡아 PR 135건을 머지하며 협업을 배웠습니다.

팀 프로젝트가 끝났을 때, 프레임워크가 정해준 구조의 한계가 보였습니다. 서버 권위, 시야 기반 복제, 결정성 검증 — 제가 만들고 싶은 것들은 구조 자체를 소유해야 가능했습니다. 그래서 언리얼·유니티 같은 상용 엔진과 객관적으로 비교하며 배울 수 있는 자체 엔진 Winters를 시작했고, 지금 LoL 구조의 서버 권위 멀티플레이가 그 위에서 돌아갑니다.

저의 장기 목표는 제 게임을 만드는 회사를 세우는 것입니다. 그래서 더더욱, 현업의 규모와 기준을 배우는 것이 지금 제게 가장 필요한 단계라고 생각합니다. 배운 것을 수치로 증명하는 방식으로 기여하겠습니다.
