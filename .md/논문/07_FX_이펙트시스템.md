# 07. FX·파티클 시스템 — 박사 연구 심화

> 이 문서는 `00_PHD_Paper_Guide.md`의 개념 틀을 전제로 한다.
> 특히 §1 "구현 vs 기여", §3 thesis statement 형식
> (`"[기법 X]를 쓰면 [제약 C] 하에서 [목표 Y]를 [SOTA 대비 개선 Z]로 달성한다"`),
> §4 표준 구조, §7 평가(파티클 throughput M particles/s, frame/sim time ms,
> 메모리 MB, 시각적 품질, 결정론 재현률)를 모든 세부주제에서 일관되게 적용한다.
> 읽는 이는 Winters 엔진(자체 WFX 이펙트 포맷 + Niagara식 데이터 주도 이펙트 툴,
> LoL식 MOBA + 오픈월드)을 직접 만든 숙련 C++ 엔진 개발자다.
> 따라서 "어떻게 구현하나"가 아니라 **"FX 분야에서 박사 기여는 어디서 나오나"**에 초점을 둔다.

## 0. 이 분야를 박사로 본다는 것

### 0.1 "FX는 대부분 엔지니어링이다" — 이 명제가 왜 옳고, 무엇이 그 예외인가

솔직하게 시작하자. **상용 FX 시스템의 90%는 연구가 아니라 엔지니어링이다.**
Niagara, Cascade, PopcornFX, Houdini의 POP solver는 천재적 소프트웨어이지만,
그 안의 대부분(emitter 계층, 모듈 스택, curve editor, 텍스처 atlas, 블렌드 모드,
spawn rate 곡선)은 "새로운 지식"이 아니라 "잘 정리된 알려진 기법"이다.
가이드 §1의 함정이 FX에서 특히 깊은 이유가 여기 있다. FX는 **시각적으로 화려해서**
"멋진 데모 = 기여"라는 착각을 가장 쉽게 일으킨다.

| 이건 구현이다 (석사·포트폴리오) | 이건 기여 후보다 (박사) |
|--------------------------------|--------------------------|
| "GPU 파티클을 compute로 백만 개 띄웠다" | "백만 파티클의 view-dependent 투명 정렬을, 정확 정렬 대비 SSIM 0.99를 유지하며 per-frame 정렬 비용을 O(n log n)→O(n)으로 줄이는 적응적 버킷 정렬을 제안한다" |
| "노드 그래프 에디터를 만들었다" | "이펙트 노드 그래프를 HLSL로 트랜슬레이션할 때, 저작자가 표현한 곡선·랜덤 의미를 보존하면서 dead-module 제거·상수 폴딩으로 VGPR 점유를 평균 22% 낮추는 IR-레벨 최적화를 제안한다" |
| "curl noise로 연기를 만들었다" | "절차적 force field와 depth-buffer 충돌을 결합할 때 생기는 시간적 깜빡임을, 충돌 응답을 sub-step velocity clamping으로 안정화해 flicker(temporal L2)를 1/4로 줄인다" |
| "리플레이에서 FX가 재생된다" | "GPU 병렬 파티클 시뮬에서 spawn 순서·warp 스케줄에 무관하게 비트 단위로 재현되는 결정론적 난수 스트림 분배를 제안하고, 1e6 파티클·1e4 틱에서 재현률 100%를 보인다" |

핵심: **"FX 기능을 구현"한 것은 비교 대상이 없다.** FX가 연구가 되려면
가이드 §5의 기여 유형 중 하나로 환원되어야 한다. FX 분야에서 실제로 박사가 나오는
지점은 좁고 분명하다.

```text
FX가 연구가 되는 네 가지 통로
  (1) 대규모 병렬 시뮬의 자료구조/알고리즘  → HPG/I3D (정렬·컬링·충돌·메모리)
  (2) 저작 그래프 → 효율 코드 컴파일        → 컴파일러/그래픽스 경계 (의미 보존 최적화)
  (3) 결정론·재현·네트워크 권위 분리        → Systems/네트워킹 경계 (GPU 난수 재현)
  (4) 지각 기반 LOD·budget                   → TVCG/I3D (perceptual metric으로 "더 싸게 동급")
```

이 네 통로가 본 문서 §1~§4의 골격이다. 나머지(emitter 계층, 블렌드, atlas)는
**testbed 인프라**일 뿐 기여가 아니다.

### 0.3 FX 박사의 공통 무기: throughput–quality–determinism 3축

렌더링 박사가 "ground truth vs frame budget" 2축에 산다면(→ `02_Graphics`),
FX 박사는 **세 축**에 산다. FX는 (a) 대규모(throughput), (b) 시각적 그럴듯함(quality는
정답이 없는 "perceptual"), (c) 인터랙티브·네트워크 게임이라 재현(determinism)이 필요하다는
세 제약이 동시에 걸린다.

```text
              throughput (M particles/s)              determinism
                  ↑                                    재현률(%) / 비트 일치
                  │  내 기법 ●                          ↑
   GPU sim ───────┤        ↗ "좌상단(빠르고 싼)으로"     │ 100% ●─── lockstep replay 요구선
                  │   ● naive                           │
   CPU sim ───────┤                                     │  98% ●  GPU warp race로 깨짐
                  └──────────────────→ 비용(ms/MB)      └────────────────→ 병렬도(#warps)
                       quality 축(perceptual: FLIP/SSIM/user study)은
                       이 평면에 수직으로 꽂힌다 — "같은 ms에 더 그럴듯한가"
```

박사 기여 = "이 3D Pareto 표면을 특정 제약(투명 정렬 / depth 충돌 / GPU 재현 / 저사양 LOD)
하에서 바깥으로 민다." §1~§4는 이 표면을 각 축에서 구체화한 것이다.

> Winters 강점(가이드 §169): `CLAUDE.md`의 "inspectable debug UI/overlay + bounded trace" 문화와
> 이미 보유한 `WfxEffectToolPanel`이 곧 **측정 인프라**다. FX 박사 평가의 절반은
> "throughput·flicker·재현률을 어떻게 측정했나"에서 나오며, Winters는 그 hook을 이미 갖췄다.

## 1. 파티클 시스템 아키텍처와 수명(Lifetime/Duration) 모델

### 1.1 핵심 원리

파티클 시스템의 추상 모델은 보편적이다. **System ⊃ Emitter ⊃ Particle**의 3계층과,
각 입자에 대한 **상태 적분(state integration over time)**.

한 입자 `p`의 상태는 시간에 대한 ODE다(전형적 explicit Euler/semi-implicit):

```text
state p = { x(위치), v(속도), age, lifetime, color, size, rotation, customK... }

매 틱 dt:
  age   += dt
  v     += (force(x, v, t) / m) · dt        # gravity, drag, curl noise, attractor...
  x     += v · dt
  (color, size, rotation, ...) = Module_k( normalizedAge )   # 곡선 평가
  if age >= lifetime: kill(p)
```

여기서 **수명 모델의 핵심은 두 시간 축의 분리**다.

```text
  emitter duration  : emitter가 입자를 "방출하는" 기간 (시스템 수준 lifecycle)
  particle lifetime : 방출된 각 입자가 "사는" 기간 (입자 수준)
  normalizedAge     : age / lifetime ∈ [0,1]  → 모든 곡선의 정의역
```

이 분리가 왜 본질적인가? **모듈(color-over-life, size-over-life, drag-over-life)은
거의 전부 `normalizedAge`의 함수**이기 때문이다. 즉 lifetime을 1로 정규화하면
모듈은 입자 수와 무관한 "재사용 가능한 곡선"이 되고, 시스템은 "곡선 × 입자 배열"의
SIMD/SoA 평가로 환원된다. 이것이 Niagara의 normalized-age 설계와
모든 data-oriented 파티클 시스템의 출발점이다.

**Open problem(여기서 박사가 나온다):** 모듈을 스택으로 합성(compose)할 때
**표현력 vs 성능**의 긴장. 모듈이 많을수록 저작 표현력은 커지지만, 각 모듈이
입자 배열을 한 번씩 훑으면(pass-per-module) 메모리 대역폭이 폭발한다.
반대로 모든 모듈을 한 커널로 융합(fusion)하면 빠르지만 모듈 독립성·재사용성이 깨진다.
"표현력을 유지하며 모듈을 자동 융합하는 합성 모델"이 §3의 컴파일 문제와 만난다.

### 1.2 대표 기존 연구/시스템

- **Reeves 1983**, *Particle Systems—A Technique for Modeling a Class of Fuzzy Objects* (ACM TOG).
  파티클 시스템의 원전. emitter→particle 계층, 확률적 spawn, per-particle attribute,
  age/lifetime 모델을 처음 정식화. "fuzzy object"(불·구름·물)를 점 집합의 확률적 진화로 표현.
- **Sims 1990**, *Particle Animation and Rendering Using Data Parallel Computation* (SIGGRAPH).
  파티클을 **데이터 병렬**로 본 선구. Connection Machine에서 입자를 SIMD로 적분 —
  오늘날 GPU compute 파티클의 개념적 조상.
- **Unreal Niagara** (Epic, GDC 2018~). 산업 SOTA. 핵심 설계 결정 4가지:
  (1) **System / Emitter / Module / Script** 계층, (2) 모든 곡선이 normalized-age 기반,
  (3) **Simulation Stages**(spawn→update를 여러 패스로 나눠 grid/iteration 표현),
  (4) 모듈 스택이 결국 **HLSL/VM으로 트랜슬레이션**(→ §3). Cascade(전작) 대비
  CPU/GPU 통합과 데이터 주도 저작이 기여점.
- **PopcornFX, Houdini POP**. 노드 기반 저작과 대규모 시뮬의 상용 성숙도. POP는
  오프라인 품질, PopcornFX는 런타임 통합·크로스플랫폼이 강점.

> 비판적 정리(가이드 §92): 위는 전부 **시스템(엔지니어링)**이다. 박사가 인용할 때는
> "이들이 무엇을 했나"가 아니라 **"이들이 명시적으로 미루거나 휴리스틱으로 때운 지점이
> 어디인가"**를 찾아 gap으로 삼아야 한다. 예: Niagara 문서가 "모듈 순서가 성능에
> 영향을 준다"고 말하면, 그 순서 최적화의 자동화·최적성이 빈칸이다.

### 1.3 자료구조/알고리즘 (의사코드)

핵심 자료구조는 **SoA(Structure of Arrays) 파티클 풀 + swap-back kill**.
AoS(입자 struct 배열)는 cache·SIMD에 적대적이므로, 속성별 열(column)로 쪼갠다.

```text
struct ParticlePool:          # SoA: 속성별 연속 배열
  pos[]      : Vec3
  velocity[] : Vec3
  color[]    : Vec4
  lifetime[] : f32
  age[]      : f32
  size[]     : f32
  custom[K]  : ...            # 그래프/DataInterface payload
  activeCount

  Spawn() -> idx:             # 끝에 추가, O(1)
    idx = activeCount; activeCount += 1; resetParticle(idx); return idx

  KillSwapBack(idx):          # 죽은 입자를 마지막 입자로 덮어쓰기, O(1), 순서 비보존
    last = activeCount - 1
    if idx != last: moveParticle(dst=idx, src=last)
    activeCount -= 1

  Tick(dt):                   # 한 패스에 SoA 선형 스캔 (cache 친화)
    for i in [0, activeCount):     # 가능하면 SIMD 8-wide
      age[i] += dt
      velocity[i] += force(i) * dt
      pos[i] += velocity[i] * dt
      t = clamp(age[i] / lifetime[i], 0, 1)
      color[i] = curveColor(t); size[i] = curveSize(t)
    i = 0                          # 죽은 입자 컴팩션 (역순 swap-back)
    while i < activeCount:
      if age[i] >= lifetime[i]: KillSwapBack(i)   # i 증가시키지 않음(덮인 입자 재검사)
      else: i += 1
```

수명 상태기계는 입자뿐 아니라 **이펙트 인스턴스 전체**에도 필요하다(스킬 종료 시
즉시 사라지지 않고 fade-out). 3-상태 lifecycle:

```text
Active ──(elapsed≥lifetime  또는  pendingDelete)──▶ Completing ──(fadeOut 끝)──▶ Complete
  │                                                     │
  fadeAlpha = clamp(elapsed/fadeIn,0,1)                 fadeAlpha *= 1 - completionElapsed/duration
```

> Winters 매핑: 위 SoA 풀은 `Engine/Public/FX/ParticlePool.h`의 `CParticlePool`과
> **1:1로 일치한다**(`m_vecPos/m_vecVelocity/m_vecColor/m_vecLifetime/m_vecAge/m_vecSize`,
> `m_arrCustomInt[4]`, `m_arrCustomVec4[8]`, `Spawn()`/`KillSwapBack()`). 3-상태 lifecycle은
> `Engine/Public/FX/FxLifeCycle.h`의 `FxAdvanceLifecycle()`·`eFxLifecycleState`에 그대로 있다.

### 1.4 박사급 novel 각도 (open problems)

1. **모듈 합성의 자동 융합(auto-fusion) + 표현력 보존.** 모듈 스택을
   의존성 그래프로 보고, 데이터 의존이 없는 모듈을 한 SIMD 패스로 융합하되
   저작자가 본 "모듈 단위 의미"는 보존하는 스케줄러. 기여 = "패스 수 vs 표현력"의
   Pareto를 자동으로 최적점에 놓는 것.
2. **시간적 일관성(temporal coherence)이 보장되는 spawn.** spawn rate를 dt로
   적분할 때 생기는 sub-frame jitter(프레임마다 입자가 뭉치는 stair-stepping)를,
   서브프레임 위치 보간으로 제거. perceptual flicker를 metric으로.
3. **continuous LOD로서의 lifetime 리타겟팅.** 카메라 거리/budget에 따라
   입자 수를 줄이되, 남은 입자의 lifetime·size를 재분배해 **에너지(적분된 alpha·area)를
   보존**함으로써 "팝핑 없는 입자 컬링"을 달성(→ §4와 연결).

### 1.5 Thesis statement 예시

> "파티클 모듈 스택을 데이터 의존 그래프로 보고 비충돌 모듈을 SIMD 패스로 자동 융합하면,
> 저작자가 지정한 모듈 단위 의미를 보존(round-trip 동등)한 채로, 모듈 수 N에 대해
> 시뮬 시간을 naive pass-per-module 대비 메모리 대역폭 기준 1/N→상수배로 줄이고,
> 결과 이미지를 FLIP<0.02로 유지할 수 있다."

### 1.6 평가 방법

- **성능:** sim time(ms) vs 입자 수·모듈 수 곡선; 메모리 대역폭(측정 또는 모델); throughput(M/s).
- **품질:** 융합 전/후 프레임의 FLIP/SSIM(동등성 증명); spawn jitter는 temporal L2 flicker.
- **Ablation:** 융합 끄기 / 모듈 재정렬 / SoA→AoS — 각 요소의 기여 분해.
- **Baseline:** naive pass-per-module, 그리고 수동 융합(전문가가 손으로 한 커널).
- **재현성:** 동일 .wfx 자산·시드·하드웨어 명시(가이드 §166).

## 2. GPU 파티클 시뮬레이션 (Compute 기반 대규모)

### 2.1 핵심 원리

CPU 파티클은 코어 수에 묶여 수만~수십만에서 멈춘다. **GPU compute**는 입자를
수십만 스레드로 동시 적분해 **수백만**까지 민다. 본질은 §1.3의 Tick 루프를
compute shader의 `[numthreads(64,1,1)]`로 옮기고, 데이터를 GPU 상주
`RWStructuredBuffer`에 두는 것이다. 단, 세 가지 새 문제가 생긴다.

```text
(1) 가변 개수 dispatch  : 살아있는 입자 수가 매 틱 바뀜 → CPU가 모르는 수를 GPU가 그림
                          → indirect dispatch / indirect draw (GPU가 인자 버퍼를 채움)
(2) 동시 alloc/free     : 여러 스레드가 동시에 spawn/kill → atomic 기반 dead/alive 리스트
(3) 투명 정렬           : 알파 블렌드는 back-to-front 정렬 필요, 매 틱 수백만 개 → GPU sort
```

**Dead/Alive 리스트 + indirect의 데이터 흐름:**

```text
  AliveList_A (이번 틱 입력) ──update CS──▶ AliveList_B (다음 틱)
        │  죽으면 DeadList에 free index push (atomic)
  DeadList ──spawn CS──▶ pop free index, 새 입자 초기화, AliveList_B에 push
  AliveCount(B) ──build-args CS──▶ IndirectDrawArgs (vertexCount = alive*verts)
  Renderer ──DrawIndexedIndirect(IndirectDrawArgs)──▶  (CPU는 개수 모름)
```

핵심 통찰: **CPU는 한 틱에 입자 개수를 모른다.** spawn/kill이 전부 GPU에서 일어나므로,
draw call의 인자조차 GPU가 compute로 채운 버퍼에서 읽는다(`ExecuteIndirect`). CPU↔GPU
왕복(readback)을 없애는 것이 대규모의 전제다.

**절차적 force field — curl noise.** 대규모 연기/마법의 그럴듯한 난류는 입자마다
물리 fluid를 풀지 않고 **divergence-free(비압축) 절차적 속도장**으로 흉내 낸다.

```text
curl noise (Bridson et al. 2007):
  potential field Ψ(x) = (noise(x), noise(x+o1), noise(x+o2))   # 3D 벡터 포텐셜
  v(x) = ∇ × Ψ(x)                                                # 회전(curl) → ∇·v = 0 보장
  → 비압축이라 입자가 뭉치거나 발산하지 않고 "소용돌이치는" 자연스런 흐름
```

`∇×Ψ`는 닫힌 형태(유한차분 3회 noise 샘플)로 입자당 O(1). 이것이 fluid solver 없이
대규모 난류를 GPU에서 싸게 얻는 표준 트릭이다.

**Depth-buffer collision.** 입자가 씬과 충돌하려면 정식 충돌 질의는 너무 비싸다.
대신 입자를 화면에 투영해 **depth buffer를 읽어 "표면 뒤로 갔는지"로 충돌을 근사**한다.

```text
collide(p):
  s = project(p.pos)                      # 클립→스크린
  sceneDepth = DepthBuffer.Load(s.xy)     # 화면 공간 깊이
  if p.depth > sceneDepth + thickness:    # 표면 뒤로 관통
     normal ≈ reconstructNormal(DepthBuffer, s.xy)   # 깊이 기울기로 법선 추정
     p.velocity = reflect(p.velocity, normal) * restitution
```

screen-space라 화면 밖·가려진 면은 못 막는 **본질적 한계**가 있고, 여기서 §2.4의 연구가 나온다.

### 2.2 대표 기존 연구/시스템

- **Bridson, Hourihan, Nordenstam 2007**, *Curl-Noise for Procedural Fluid Flow* (SIGGRAPH).
  비압축 절차적 속도장의 원전. 위 §2.1 force field의 근거.
- **Drone 2007 / NVIDIA·AMD GPU particle 강연들** (GDC/SIGGRAPH course).
  dead/alive 리스트, atomic append, indirect draw의 산업 정착.
- **Green 2008/2010** (NVIDIA), *Screen Space Fluid Rendering*, depth 기반 입자 처리.
  screen-space 충돌·렌더의 계보.
- **GPU 정렬:** Merrill & Grimshaw 2011 (radix sort), Satish et al. 2009.
  대규모 키 정렬의 GPU SOTA — 투명 입자 정렬의 baseline.
- **OIT(Order-Independent Transparency):** Enderton et al. 2010 (Stochastic Transparency),
  McGuire & Bavoil 2013 (Weighted Blended OIT). "정렬을 아예 피하는" 대안 계보.
- **Niagara GPU sim** (Epic, GDC). 노드 그래프를 GPU compute로 내리는 산업 통합(→ §3).

### 2.3 자료구조/알고리즘 (의사코드)

**컴퓨트 셰이더 개념 — update + spawn + build-args 3-패스:**

```text
// PASS 1: update (per alive particle, indirect dispatch over AliveCount_A)
[numthreads(64,1,1)] FxParticleUpdate_CS(tid):
  i = AliveList_A[tid]
  age[i] += dt
  v = VelocityLifetime[i].xyz + (curlNoise(Position[i].xyz, t) + gravity) * dt
  x = PositionAge[i].xyz + v * dt
  if collideDepth(x, v): v = reflect(...)             // §2.1 depth collision
  PositionAge[i] = float4(x, age[i]); VelocityLifetime[i].xyz = v
  if age[i] >= VelocityLifetime[i].w:                 // 죽음
     idx = DeadList.IncrementCounter(); DeadList[idx] = i   // atomic free
  else:
     idx = AliveList_B.IncrementCounter(); AliveList_B[idx] = i  // 생존 carry

// PASS 2: spawn (per requested new particle)
[numthreads(64,1,1)] FxParticleSpawn_CS(tid):
  if tid >= spawnThisFrame: return
  d = DeadList.DecrementCounter(); if d < 0: return   // 풀 고갈
  i = DeadList[d]
  seed = Hash(emitterSeed, baseSpawnId + tid)         // §4 결정론: 스레드 무관 시드
  initParticle(i, seed)
  idx = AliveList_B.IncrementCounter(); AliveList_B[idx] = i

// PASS 3: build indirect args (1 thread)
[numthreads(1,1,1)] FxBuildDrawArgs_CS():
  DrawArgs.IndexCountPerInstance = indicesPerParticle
  DrawArgs.InstanceCount         = AliveCount_B        // CPU가 모르는 값
```

**정렬(투명):** 정확 정렬은 GPU radix sort(키 = view-space depth). 본 문서가 미는
대안은 **적응적 버킷 정렬** — depth를 K개 버킷으로 양자화해 버킷 단위만 정렬:

```text
approximate bucket sort:
  bucket = quantize(viewDepth, K)        // O(n) scatter to K buckets (atomic)
  draw buckets back-to-front             // 버킷 내부는 비정렬 (perceptual error 허용)
  // K를 화면 깊이 분산에 적응 → 오차 vs 비용 trade-off
```

### 2.4 박사급 novel 각도 (open problems)

1. **대규모 투명의 지각 기반 적응 정렬.** 정확 정렬 O(n log n) vs OIT(정렬 회피, 품질↓)
   사이에서, **버킷 수 K를 장면 깊이 분포·카메라 운동에 적응**시켜
   "SSIM 0.99를 유지하는 최소 K"를 자동 결정. 기여 = perceptual budget 하의 정렬 비용 하한.
2. **Screen-space 충돌의 한계 극복.** 화면 밖/가려진 면 누락을, sparse한 월드공간
   거리장(SDF) 캐시 + screen-space의 하이브리드로 보완. 기여 = "정확 충돌 대비 X% 비용으로
   누락률을 Y% 제거".
3. **다른 시스템과의 상호작용(coupling).** GPU 파티클 ↔ fluid grid ↔ 천/강체의 양방향 결합을
   한 프레임 budget 안에서. SCA 영역. open: 결합의 안정성·시간 일관성.
4. **충돌 응답의 시간적 안정화.** depth collision은 카메라가 움직이면 충돌점이 깜빡인다.
   sub-step velocity clamping으로 flicker(temporal L2)를 정량 억제.

### 2.5 Thesis statement 예시

> "대규모 GPU 파티클의 투명 정렬에서, 버킷 수를 장면 깊이 분산에 적응시키는 근사 버킷
> 정렬을 쓰면, 정확 radix 정렬 대비 per-frame 정렬 시간을 60% 줄이면서 결과 이미지를
> SSIM≥0.99·FLIP≤0.03으로 유지하고, 100만 파티클에서 16.6ms@1080p budget을 만족한다."

### 2.6 평가 방법

- **성능:** throughput(M particles/s), sim+sort time(ms), GPU 메모리(MB), indirect 왕복 제거 효과.
- **품질:** 정확 정렬을 ground truth로 SSIM/FLIP; 충돌 깜빡임은 temporal L2(연속 프레임 차).
- **확장성:** 입자 수 1e4→1e6 강/약 scaling 곡선, async compute on/off.
- **Ablation:** 버킷 K 스윕, SDF 보완 on/off, sub-step clamp on/off.
- **Baseline:** GPU radix 정확 정렬, Weighted Blended OIT, CPU sim.
- **재현성:** GPU 모델·드라이버·해상도·시드 명시; 결정론 모드(§4)로 동일 입력 보장.

### 2.7 Winters 연결점

이펙트 툴 로드맵 **EFX-5(`07_EFX5_GPU_COMPUTE_DATA_INTERFACE_AND_ELDEN_PATH.md`)**가
GPU compute 경로를 명시적으로 규정한다: 1차 목표 8192 파티클·64 emitter·GPU tick ≤1.5ms,
2차 32768·indirect draw·**GPU sort 또는 approximate bucket sort**·async compute 검토 —
즉 §2.4-(1)의 적응적 버킷 정렬이 Winters 로드맵에 이미 후보로 적혀 있다.
GPU buffer layout도 SoA로 설계돼 있다(`PositionAge.xyzw`, `VelocityLifetime.xyzw`,
`ColorRandom`, `SizeRotation`, `Custom0..N`)와 `FxParticleUpdate_CS / FxParticleSpawn_CS /
FxBuildDrawArgs_CS` 셰이더 파일까지 예약돼 있다. RHI 쪽은 `IRHICommandList`에
`Dispatch`·`DrawIndexedIndirect`·`TransitionResource`가 이미 선언되어 있어
(상당수 DX12 구현은 아직 초기), §2의 기여를 Elden 보스 장판·6-way smoke·검기 같은
실제 대규모 워크로드(`Data/.../Elden/*.wfx` 예약)로 평가할 testbed가 깔려 있다.

## 3. 데이터 주도 저작(Data-Driven Authoring): 노드 그래프 → VM/셰이더

### 3.1 핵심 원리

아티스트가 코드 없이 이펙트를 만들려면 **노드 그래프**가 필요하다. 그러나 그래프는
저작 표현일 뿐, 런타임은 입자당 수백만 번 평가되어야 한다. 따라서 핵심은
**"그래프를 효율적 실행 코드로 컴파일"**하는 것이고, 두 갈래가 있다.

```text
노드 그래프 (저작)
   │
   ├─(A) 바이트코드 VM   : 그래프 → opcode 스트림 → 인터프리터가 입자당 실행
   │                        장점: 핫리로드 쉬움, CPU/플랫폼 이식 쉬움
   │                        단점: dispatch 오버헤드, GPU에서 불리
   │
   └─(B) HLSL 트랜슬레이션: 그래프 → HLSL 소스 생성 → DXC로 컴파일 → GPU compute
                            (Niagara의 길) 장점: GPU 최대 성능
                            단점: 컴파일 지연(핫리로드 어려움), 셰이더 순열 폭발
```

이것은 본질적으로 **컴파일러 문제**다. 그래프는 DAG(데이터 흐름), 노드는 연산,
엣지는 SSA 값. 따라서 그래프→IR→최적화→코드생성의 고전 파이프라인이 그대로 적용된다.

```text
Graph(DAG) → typed IR(SSA) → opt passes(상수폴딩·CSE·dead-module 제거) → {VM bytecode | HLSL}
```

**타입 시스템이 핵심.** 노드 포트는 타입(float/float3/curve/texture/...)을 갖고,
연결 합법성·암묵 변환·균일성(uniform vs per-particle)을 정적으로 검사해야 한다.
특히 **uniform/varying 분리**(상수 폴딩의 핵심): emitter 동안 안 변하는 값은
입자 루프 밖으로 호이스팅해야 입자당 비용이 줄어든다.

**Open problem(여기서 박사가 나온다):** 그래프→코드 컴파일에서
**(a) 저작자 의도 보존**(곡선의 모양·랜덤의 통계·모듈 경계의 의미를 최적화가 깨지 않을 것),
**(b) 효율**(VGPR/대역폭/셰이더 순열 최소화), **(c) 라이브 프리뷰 결정론**(에디터 미리보기와
런타임이 같은 결과)을 **동시에** 만족하는 것. 산업 시스템은 이 셋을 휴리스틱으로 절충할 뿐
"최적성"이나 "보존 보장"을 증명하지 않는다 — 빈칸이다.

### 3.2 대표 기존 연구/시스템

- **Niagara HLSL Translation** (Epic, GDC 2018~2021). 모듈 스택을 **HLSL로 트랜슬레이션**하고
  DXC로 컴파일하는 산업 최대 사례. uniform/varying 구분, DataInterface로 외부 자원 접근.
  본 문서가 미는 "의미 보존 최적화"의 직접 baseline.
- **셰이더 그래프 일반:** Unreal Material Editor, Unity Shader Graph — 노드→HLSL 생성의
  성숙 사례(파티클이 아니라 머티리얼이지만 컴파일 구조는 동형).
- **Karpenko/UI of node graphs**, **Houdini VEX/VOP** — VOP 노드 그래프를 VEX 코드로
  컴파일하는 오랜 사례. 오프라인이지만 그래프→스칼라/벡터 코드 생성의 풍부한 선행.
- **컴파일러 기반:** 일반 컴파일러 이론(SSA, Cytron et al. 1991; 상수 폴딩·CSE·DCE).
  FX 그래프 최적화는 이 고전을 도메인에 적용하는 것이며, 도메인 특화 불변식
  (곡선 단조성, 랜덤 스트림 보존)이 새 연구 여지.
- **셰이더 순열(permutation) 관리:** static branch vs uber-shader vs 순열 — FX 노드 그래프의
  feature 조합이 만드는 순열 폭발은 잘 알려진 산업 고통(GDC 다수).

### 3.3 자료구조/알고리즘 (의사코드)

**그래프 → typed SSA IR → 최적화 → HLSL 생성:**

```text
compileGraph(graph G) -> hlsl:
  // 1) 위상정렬 + 타입검사
  order = topoSort(G)                       # DAG → 평가 순서
  for node n in order:
    checkTypes(n)                           # 포트 타입·변환 합법성
    n.uniformity = all(inputs uniform) ? UNIFORM : VARYING   # uniform 전파

  // 2) IR 생성 (SSA): 노드 → 값
  for n in order: ir.emit(n.toSSA(inputsValues))

  // 3) 최적화 패스 (의미 보존)
  ir = constantFold(ir)                     # uniform 부분 식 폴딩
  ir = commonSubexprElim(ir)                # 동일 부분식 공유
  ir = deadModuleElim(ir)                   # 출력에 기여 안 하는 모듈 제거
  ir = hoistUniform(ir)                     # uniform 식을 입자 루프 밖으로

  // 4) 코드 생성
  hlsl = emitHLSL(ir, layout=SoA_buffers)
  // 불변식 검증: 곡선 샘플 동등성·랜덤 스트림 동등성 (의도 보존 체크)
  assert sampleEquiv(G, hlsl, randomSeeds, tolerance)
  return hlsl
```

**의도 보존(intent preservation) 체크의 핵심 — 곡선/랜덤 불변식:**

```text
sampleEquiv(G, compiled, seeds, eps):
  for t in dense_samples([0,1]):           # normalizedAge 샘플
     assert |evalGraph(G,t) - evalCompiled(compiled,t)| < eps   # 곡선 모양 보존
  for s in seeds:                          # 같은 시드 → 같은 난수 시퀀스
     assert randStream(G,s) == randStream(compiled,s)           # 통계 아닌 비트 보존
```

이 체크가 "최적화가 저작자 의도를 깨지 않았다"를 **검증 가능한 명제**로 만든다(가이드 §70 falsifiability).

### 3.4 박사급 novel 각도 (open problems)

1. **의미 보존 최적화의 형식화·증명.** dead-module 제거·상수 폴딩·융합이
   "출력 이미지 동등(또는 ε 이내)"임을 도메인 불변식(곡선 단조성, 랜덤 스트림 동등)으로
   증명하고, 그 위에서 VGPR/대역폭을 최소화. 기여 = "최적화 안전성 보장 + 정량 이득".
2. **셰이더 순열 폭발의 원리적 해결.** feature 조합을 static permutation 대신
   "비용-인지 동적 분기 vs 순열"의 자동 선택으로. 기여 = 컴파일 시간·바이너리 크기 vs
   런타임 비용의 Pareto.
3. **라이브 프리뷰 결정론.** 에디터 인터프리터(VM)와 런타임 컴파일 코드(HLSL)가
   **비트/ε 동등한 미리보기**를 주도록 하는 dual-path 일관성(→ §4 결정론과 직결).
4. **VM vs HLSL 자동 선택.** 한 이펙트 안에서 입자 수·플랫폼·핫리로드 요구에 따라
   모듈별로 VM/HLSL 백엔드를 선택하는 적응적 컴파일.

### 3.5 Thesis statement 예시

> "파티클 노드 그래프를 typed SSA IR로 낮춘 뒤 곡선·난수 불변식을 보존하는 최적화
> (dead-module 제거·uniform 호이스팅·CSE)를 적용하면, 생성된 HLSL의 결과를 저작 그래프와
> 곡선 ε<1e-4·난수 스트림 비트 동등으로 보존하면서, 순열당 VGPR 점유를 수동 작성 대비
> 평균 22% 낮추고 컴파일 순열 수를 절반으로 줄일 수 있다."

### 3.6 평가 방법

- **성능:** 생성 셰이더의 VGPR/SGPR 점유, 점유율(occupancy), sim time; 컴파일 시간·순열 수·바이너리 크기.
- **정확도(의도 보존):** 그래프 인터프리터 vs 컴파일 코드의 곡선 ΔE/ε, 난수 스트림 비트 일치율.
- **Ablation:** 각 최적화 패스 on/off로 이득의 원천 분해(가이드 §164).
- **Baseline:** 최적화 없는 직역(naive emit), 수동 작성 HLSL(전문가 상한), Niagara 트랜슬레이션(가능하면).
- **품질:** 최적화 전/후 렌더 FLIP(동등성).
- **재현성:** 그래프 자산·시드·컴파일러 버전 공개.

### 3.7 Winters 연결점

`FxAsset.h`에 이미 `FxNodeDesc { strType, inputs, outputs, bytecodeBlob }`가 있다 —
즉 **노드 그래프와 바이트코드 슬롯이 자료구조 수준에서 예약**돼 있다(§3.1-(A) VM 경로의 씨앗).
WFX 자산은 `WfxDocument.cpp`가 보여주듯 구조화 텍스트(JSON류, `EscapeJson`/`RenderTypeToString`
등 직렬화 함수)로 round-trip되며, 로드맵 **EFX-1(`03_EFX1_WFX_WMI_SCHEMA_AND_ROUNDTRIP.md`)**이
수동 string parser를 structured reader/writer로 교체하고 **canonical save/load 동등성**을
요구한다 — 이는 §3.3의 "의도 보존 round-trip"의 직접 전제다. 단, 로드맵의 정책 결정
(`27_..._MASTER_MATERIALS_BAKE.md`)은 **노드 그래프보다 Master Material 3종을 우선**한다.
따라서 Winters에서 §3 기여는 "그래프 에디터를 먼저 만든다"가 아니라
**"Master Material 파라미터/곡선 → DX12 HLSL 생성"의 의미 보존·최적 컴파일**로 프레이밍하는 것이
로드맵과 정합적이다(EFX-3 = `05_EFX3_DX12_MASTER_MATERIAL_RENDERER.md`).

## 4. 결정론·네트워크·LOD를 위한 FX (replay·서버 동기화)

### 4.1 핵심 원리

게임 FX는 단지 예쁘면 되는 게 아니다. 세 가지 시스템 제약이 걸린다.

```text
(1) 결정론(determinism) : 리플레이·관전·디버깅에서 "같은 입력 → 같은 화면" 재현
(2) 네트워크 권위(authority) : 무엇이 서버 동기 대상이고 무엇이 클라 전용인가
(3) LOD/budget          : 수백 이펙트가 동시에 떠도 프레임이 안 죽게
```

**(1) 결정론.** FX의 비결정성은 거의 전부 **난수**에서 온다(spawn 위치 jitter, 수명 분산,
색 변이). 결정론 FX의 원리는 단순하다: **시드 기반 PRNG + 입자 인덱스 기반 시드 분배**.

```text
입자 i의 모든 난수 = PRNG(Hash(emitterSeed, spawnId_i))
  → 시드가 같으면 입자 i의 위치·수명·색이 비트 단위로 동일
  → wall-clock·프레임레이트·실행 순서와 무관 (고정 dt 가정)
```

핵심 함정: **부동소수점 비결정성**(컴파일러·SIMD·FMA·플랫폼 차이로 마지막 비트가 다름).
리플레이가 같은 바이너리·플랫폼이면 보통 안전하지만, cross-platform 결정론은 정수화/고정소수점이
필요할 수 있다(→ `10_Server`의 lockstep과 동일한 문제).

**(2) 네트워크 권위 — FX의 이분법.** 이것이 FX 네트워킹의 핵심 통찰이다.

```text
  순수 시각 FX (피격 섬광, 발밑 먼지, 시전 오라)
     → 서버는 시뮬 안 함. 클라 전용. 각 클라가 독립 재생(약간 달라도 무방).
     → 대역폭 0. "이벤트(스킬 시전)"만 받으면 클라가 알아서 FX.

  게임플레이-영향 FX (투사체: 미니언/포탑/스킬샷)
     → 위치·충돌이 "맞았는가"를 결정 → 반드시 서버 권위.
     → 서버가 ProjectileSpawn/Hit 이벤트를 tick과 함께 송신, 클라는 시각만 보간.
     → FX는 "결과의 시각화"이지 FX가 게임 상태를 만들면 안 됨(치트 표면).
```

즉 **"FX가 게임 상태를 결정하면 안 된다"**가 철칙이다. 투사체의 *판정*은 서버에서 권위적으로
일어나고, 클라의 파티클은 그 권위적 결과를 그릴 뿐이다.

**(3) LOD/budget.** 화면에 이펙트가 폭증하면 throughput이 무너진다. FX LOD는
거리·중요도·화면 점유로 **입자 수·tick rate·emitter 수를 동적으로 줄인다**. 단순 컬링은
**팝핑**을 부르므로, "지각적으로 덜 중요한 것부터 부드럽게 줄이는" 것이 관건(→ §1.4-(3)).

### 4.2 대표 기존 연구/시스템

- **Bettner & Terrano 2001**, *1500 Archers on a 28.8: Network Programming in Age of Empires*
  (GDC). 결정론적 lockstep의 고전 — "상태가 아니라 입력만 보내고 모두가 같은 시뮬을
  재현". FX 결정론도 같은 원리(시드/입력 재현).
- **Valve / Bernier 2001**, *Latency Compensating Methods* (Source 엔진). 서버 권위 +
  클라 예측 + lag compensation. "무엇이 서버 권위인가"의 표준틀(→ `10_Server`).
- **Overwatch netcode** (Ford, GDC 2017). 결정론적 시뮬과 high-fidelity 재현 —
  게임플레이 영향 요소의 서버 권위와 시각 분리의 산업 사례.
- **부동소수점 결정론:** Glenn Fiedler, *Floating Point Determinism* (gafferongames).
  cross-platform 비트 재현의 어려움과 고정소수점 대안 — FX·물리 공통.
- **지각 LOD:** Luebke et al., *Level of Detail for 3D Graphics* (2003)의 perceptual LOD
  원리; FX 특화는 산업(Niagara Scalability, GDC)에 흩어져 있어 **학술 빈칸이 큼**.
- **GPU 결정론 난수:** counter-based RNG — Salmon et al. 2011, *Parallel Random Numbers:
  As Easy as 1, 2, 3* (SC, Philox/Threefry). **스레드·실행순서 무관 재현**의 핵심 도구
  (§4.4-(1)의 근거).

### 4.3 자료구조/알고리즘 (의사코드)

**결정론적 spawn — counter-based 시드(GPU에서도 재현):**

```text
// CPU/GPU 동일: 입자별 난수는 "위치(스레드)"가 아니라 "정체성(spawnId)"으로 시드
initParticle(i, emitterSeed, spawnId_i):
  // counter-based RNG: 상태 캐리 없이 (key, counter) → 난수 (Philox류)
  r0 = philox(key=emitterSeed, counter=spawnId_i*4 + 0)
  r1 = philox(key=emitterSeed, counter=spawnId_i*4 + 1)
  pos[i]      = emitterOrigin + unitSphere(r0) * spawnRadius
  lifetime[i] = lerp(lifeMin, lifeMax, asFloat(r1))
  // 어떤 warp/스레드가 이 입자를 spawn하든 spawnId가 같으면 결과 동일 → GPU 재현
```

> 왜 Xoroshiro류 "상태 캐리" PRNG가 아니라 counter-based인가? 상태 캐리 PRNG는
> "몇 번째로 뽑았는가"(호출 순서)에 결과가 의존한다. GPU에서 warp 스케줄은 비결정적이므로
> 호출 순서가 흔들리면 재현이 깨진다. counter-based는 `(key,counter)→난수`라
> **순서·병렬도와 무관**해 GPU 결정론의 정석이다. (CPU 단일스레드 경로는 Xoroshiro128로 충분.)

**네트워크 — 게임플레이 FX의 권위적 이벤트 흐름:**

```text
SERVER (권위):
  tick:
    advanceProjectile(p)            # 위치·충돌은 서버가 결정
    if hit: emit Event{ ProjectileHit, victimId, serverTick }   # FX 아님, "결과"
    if spawn: emit Event{ ProjectileSpawn, kind, origin, dir, serverTick }

CLIENT (시각화만):
  onEvent(ProjectileSpawn ev):
    if seen(cueKey(ev)): return     # 중복 방지 (재전송/리플레이 대비 멱등)
    visual = ResolveVisual(ev.kind) # kind → .wfx 비주얼 카탈로그
    playFxCue(visual, ev.origin, ev.dir)   # 순수 시각: 클라가 독립 재생
  // 주의: 클라 FX는 게임 상태를 만들지 않음. 판정은 전적으로 서버 이벤트.
```

**지각적 FX LOD (팝핑 없는 budget 관리):**

```text
chooseLOD(fxInstance, camera, frameBudgetMs):
  importance = screenCoverage * gameplayWeight / (1 + dist^2)
  lod = budgetScheduler.assign(importance, remainingBudget)   # 중요도순 budget 배분
  // 부드러운 전이: 입자 수를 깎되 남은 입자의 size/lifetime 재분배로 적분 alpha 보존(§1.4)
  spawnRate *= lod.scale; maxParticles *= lod.scale; tickRate = lod.tickHz
```

### 4.4 박사급 novel 각도 (open problems)

1. **결정론적 GPU 난수 + 비트 재현 보장.** counter-based RNG를 파티클 시스템에 통합해,
   warp 스케줄·spawn 순서·async compute와 무관하게 **1e6 파티클·1e4 틱에서 재현률 100%**를
   증명. 기여 = "GPU 병렬 FX의 결정론을 어떤 비용으로 사는가"의 정량화. (드문 학술 빈칸.)
2. **게임플레이-영향 FX의 권위 분리 형식화.** "어떤 FX 속성이 게임 상태에 영향을 주는가"를
   정적으로 분류하고, 영향 속성만 동기화하는 최소 대역폭 프로토콜. 기여 = 시각 충실도 vs
   bandwidth(KB/tick)의 Pareto + 치트 표면 축소.
3. **지각적 FX LOD.** "팝핑 없이 budget을 지키는" 연속 LOD를, 입자 적분 에너지 보존 +
   perceptual metric으로. 기여 = 동일 user-study 품질에서 frame time을 X% 절감.
   (산업엔 있으나 **학술적 정식화·측정이 비어 있다** — §4.2 참조.)
4. **리플레이 결정론과 라이브 프리뷰 결정론의 통일.** §3.4-(3)의 에디터/런타임 일관성과
   §4의 리플레이 재현을 한 결정론 모델로 묶기.

### 4.5 Thesis statement 예시

> "파티클 난수를 입자 정체성 기반 counter-based RNG(Philox)로 분배하면, GPU의 비결정적
> warp 스케줄·async compute·spawn 순서와 무관하게 100만 파티클·1만 틱 시뮬을 비트 단위로
> 재현(재현률 100%)하면서, per-particle 난수 비용을 상태캐리 PRNG 대비 1.3배 이내로 유지하고
> 리플레이를 추가 저장 없이(시드만으로) 복원할 수 있다."

### 4.6 평가 방법

- **결정론:** 동일 입력·다른 실행(다른 GPU 점유/순서/async)으로 **비트 재현률(%)**;
  cross-platform 시 마지막-비트 발산 분포.
- **네트워크:** 동기 FX의 대역폭(KB/tick), 이벤트 멱등성(중복/재전송 하 동일 결과), 권위 위반 0건.
- **LOD:** 동일 perceptual 품질(user study 또는 FLIP)에서 frame time(ms)·입자 수 절감; 팝핑은 temporal flicker.
- **Ablation:** counter-based vs 상태캐리, LOD 에너지 보존 on/off, 동기 속성 집합 축소.
- **Baseline:** 비결정 spawn, 순진한 거리 컬링(팝핑), 전체 FX 동기화(대역폭 상한).
- **재현성(가이드 §166):** 시드·tick 로그·하드웨어 매트릭스 공개.

### 4.7 Winters 연결점

결정론 도구가 이미 있다: `Engine/Public/FX/DeterministicRandom.h`의 `CXoroshiro128`
(`Seed/Next/NextFloat/NextRange`) — §4.3의 CPU 결정론 경로에 직결되고,
§4.4-(1)의 기여는 "이 Xoroshiro 경로를 GPU에서 재현 가능한 counter-based로 확장"하는
구체적 다음 단계가 된다(GPU buffer의 `ColorRandom`/`SizeRotation.w=materialRandom` 슬롯이
난수 payload 자리). 네트워크 권위 분리는 이미 코드로 구현돼 있다:
`Client/Private/Network/Client/EventApplier.cpp`가 서버의 `ProjectileSpawn`/`ProjectileHit`
이벤트(+`serverTick`)를 받아 `ProjectileVisualCatalog::Resolve(kind)`로 비주얼을 정하고
WFX cue를 재생하며, `m_seenProjectileCueKeys`로 **멱등 재생(중복 방지)**까지 한다 —
이것이 §4.1-(2) "게임플레이-영향 FX는 서버 권위, 시각은 클라"의 살아있는 사례다
(투사체 자산: `Data/LoL/FX/Object/Minion/ranged_projectile_*.wfx`,
`Data/LoL/FX/Object/Turret/projectile_red.wfx`). 따라서 §4 전체가 Winters에서
"이미 있는 결정론 RNG + 이미 있는 권위적 이벤트 경로"를 testbed로 즉시 평가 가능하며,
이 분야는 자연스럽게 `10_Server`(lag compensation·lockstep)와 맞물린다.

## 종합. 통합 학위논문 구조 예시

가이드 §4의 "Three Papers Make a Thesis" 모델로, FX 분야 **한 학위논문**을 엮는다.
세 기여 챕터는 §0.1의 "FX가 연구가 되는 통로" 중 인접한 셋을 고른다.

```text
제목(예): "결정론적·대규모 데이터 주도 파티클 시스템: 컴파일·재현·지각 LOD"

Ch1 서론
  - 동기: 산업 FX(Niagara 등)는 화려하나 (a) 그래프→코드 최적성, (b) GPU 결정론,
          (c) 지각 LOD를 휴리스틱으로 절충할 뿐 보장·측정하지 않는다.
  - Thesis statement(한 문장):
    "파티클 이펙트를 (i) 의미 보존 컴파일, (ii) 정체성 기반 결정론 난수,
     (iii) 에너지 보존 지각 LOD로 설계하면, 저작 의도를 보존한 채로 대규모 GPU 시뮬을
     비트 단위로 재현하면서 동일 지각 품질을 더 낮은 frame budget에 달성할 수 있다."
  - 기여 목록 3~5개 (아래 Ch3~5에 대응)

Ch2 배경·관련연구
  - Reeves83 / Sims90 / Bridson07(curl) / Niagara(GDC) / lockstep(GDC) /
    Salmon11(counter RNG) / perceptual LOD(Luebke03)
  - gap 명시: "그래프 컴파일의 의미 보존·GPU 결정론·지각 FX LOD는 학술적으로 비어 있다"

Ch3 [HPG/I3D] 의미 보존 그래프 컴파일 (= §3)
  - typed SSA IR + 곡선/난수 불변식 보존 최적화 → VGPR/순열 절감, 동등성 증명

  - dead/alive·indirect·적응 버킷 정렬 + counter-based 결정론 난수 → throughput·재현률

Ch5 [I3D/TVCG] 게임플레이 권위 분리 + 지각 LOD (= §4 네트워크·LOD)
  - 영향 속성만 동기화하는 최소 프로토콜 + 에너지 보존 연속 LOD → bandwidth·frame time

Ch6 종합 평가
  - 통합 워크로드(LoL 챔피언 FX + Elden 보스 장판·6-way smoke)에서 세 기여 동시 측정
  - throughput(M/s)·sim ms·VGPR·재현률(%)·KB/tick·FLIP/user-study·MB
  - baseline: Niagara식 직역 컴파일, GPU radix 정렬, 전체 동기화, 거리 컬링
  - ablation: 각 기여의 on/off; limitations(cross-platform 비트 발산, screen-space 충돌 누락)

Ch7 결론·향후연구
  - thesis 재진술; fluid coupling, cross-platform 고정소수점 결정론, 모바일 tile GPU 이식
```
