계획서는 아래 형식과 순서를 따른다. 사용자가 "계획서 작성", "계획서 보여줘", "plan"을 요청하면 이 문서를 먼저 읽고 그대로 적용한다.

## 출력 목표

계획서는 구현자가 바로 붙잡고 수정할 수 있는 코드 변경 지시서다.

설명문, 진행 요약, 파일 목록, anchor 목록을 늘어놓지 않는다. 최종 출력은 "어느 기존 코드를 어떤 코드로 바꾸는가"에 집중한다.

## 우선순위

계획서 출력에는 이 문서가 최우선이다.

코드베이스의 기존 스타일, CLAUDE.md, AGENTS.md, 일반 코딩 관례와 충돌하면 계획서 작성 방식은 이 문서를 따른다. 단, 기존 코드를 인용하거나 기존 API/override/signature를 그대로 호출해야 하는 경우에는 실제 코드와 정확히 맞아야 하므로 원문 이름을 유지한다.

계획서에서 새로 제안하는 함수, 메서드, 타입, helper 이름은 PascalCase를 사용한다. 새 이름에 snake_case 또는 임의의 밑줄 구분을 쓰지 않는다.

## 필수 시작 형식

계획서는 반드시 아래 한 줄로 시작한다.

```text
Session - [현재 달성 목표 한 줄]
```

예:

```text
Session - B 키 recall command를 client/server/shared pipeline에 연결한다.
```

## 필수 섹션 순서

계획서는 반드시 아래 순서만 사용한다.

```text
1. 반영해야 하는 코드
```

```text
2. 검증
```

아래 섹션은 출력하지 않는다.

```text
현재 전체 세션 흐름
현재까지 진행 상황
현재 세션 목표
추가해야 하는 파일
수정해야 하는 h / cpp 파일
정확한 위치 line 및 anchor
```

필요한 배경이 있으면 `Session - ...` 한 줄에만 압축한다. 이미 반영된 작업은 언급하지 않는다.

## 반영해야 하는 코드 형식

반드시 파일별로 쓴다.

파일 섹션 제목은 아래 형식을 따른다.

```text
1-1. C:/Users/user/Desktop/Winters/Client/Private/Scene/InGameCombatInputBridge.cpp
```

파일 섹션 안에서는 아래 세 가지 형식만 사용한다.

### 아래에 추가

`아래에 추가`는 반드시 기준 위치가 되는 `기존 코드` 바로 아래에 붙인다.

나쁜 예:

```text
아래에 추가:

cpp

void Save_CurrentNavGrid();
```

좋은 예:

```text
기존 코드:

cpp

void Load_CurrentStage();

아래에 추가:

cpp

void SaveCurrentNavGrid();
```

동일한 `기존 코드`가 여러 곳에 있을 수 있으면 파일 섹션 안에서 함수/클래스 범위를 함께 쓴다. 별도 anchor 섹션은 만들지 않는다.

```text
`class CScene_Editor final : public IScene`의 private 영역에서 아래 기존 코드 바로 아래에 추가:
```

```text
기존 코드:

cpp

if (!bImGuiKbd)
{
    if (in.IsKeyPressed('Q'))
    {
        ClearNetworkAttackIntent();
        scene.DispatchSkillInput(static_cast<uint8_t>(eSkillSlot::Q));
    }

아래에 추가:

cpp

if (in.IsKeyPressed('B') &&
    scene.m_bNetworkAuthoritativeGameplay &&
    scene.m_pCommandSerializer &&
    scene.m_pNetworkView &&
    scene.m_pNetworkView->IsConnected())
{
    ClearNetworkAttackIntent();
    scene.m_pCommandSerializer->SendRecall(*scene.m_pNetworkView);
}
```

### 아래로 교체

```text
기존 코드:

cpp

if (!bImGuiKbd)
{
    if (in.IsKeyPressed('Q'))
    {
        ClearNetworkAttackIntent();
        scene.DispatchSkillInput(static_cast<uint8_t>(eSkillSlot::Q));
    }

아래로 교체:

cpp

if (!bImGuiKbd)
{
    if (in.IsKeyPressed('B') &&
        scene.m_bNetworkAuthoritativeGameplay &&
        scene.m_pCommandSerializer &&
        scene.m_pNetworkView &&

    }
```

### 삭제

```text
삭제할 코드:

cpp

#include "AssetFormat/Mesh/WMeshLoader.h"
```

긴 블록 삭제는 시작/끝을 자연어로 쓴다.

```text
삭제할 범위:
`class CSceneMapSurfaceSamplerLegacy final` 줄부터
`CScene_InGame::CScene_InGame() = default;` 바로 위까지 삭제.
`CScene_InGame::CScene_InGame() = default;` 줄은 남긴다.
```

## 위치 작성 규칙

별도 anchor 섹션을 만들지 않는다.

`아래에 추가`, `아래로 교체`, `삭제할 코드`, `삭제할 범위`는 모두 적용 위치를 스스로 특정할 수 있어야 한다. 위치가 애매하면 같은 파일 섹션 안에서 함수명, 클래스명, private/public 영역, 주변 기존 코드 중 하나를 함께 적는다.

나쁜 예:

```text
정확한 위치 line 및 anchor

3-1. Scene legacy sampler 제거
파일: C:/Users/user/Desktop/Winters/Client/Private/Scene/Scene_InGame.cpp (line 113 근처)

anchor:
cpp
class CSceneMapSurfaceSamplerLegacy final

종료 anchor:
cpp
CScene_InGame::CScene_InGame() = default;
```

좋은 예:

```text
1-1. C:/Users/user/Desktop/Winters/Client/Private/Scene/Scene_InGame.cpp

삭제할 범위:
`class CSceneMapSurfaceSamplerLegacy final` 줄부터
`CScene_InGame::CScene_InGame() = default;` 바로 위까지 삭제.
`CScene_InGame::CScene_InGame() = default;` 줄은 남긴다.
```

line 번호는 꼭 필요할 때만 괄호 안에 참고로 붙인다.

```text
line 207 근처 `CMapSurfaceSampler::RasterizeTriangle` 안에서 아래 코드를:
```

line 번호만 쓰지 않는다.

## 새 파일 작성 규칙

새 파일이 있으면 파일 섹션에 전체 내용을 쓴다.

```text
1-2. C:/Users/user/Desktop/Winters/Shared/GameSim/Components/RecallComponent.h

새 파일:

cpp

#pragma once

struct RecallComponent
{
    ...
};
```

새 파일이 없으면 "추가 파일 없음"을 쓰지 않는다.

## 프로젝트 파일 작성 규칙

`.vcxproj`, `.vcxproj.filters`의 XML 항목 추가/삭제는 계획서에 기본으로 쓰지 않는다.

나쁜 예:

```text
4-10. Server.vcxproj

기존 코드:

    <ClCompile Include="..\..\Shared\GameSim\Systems\MoveSystem.cpp" />

아래에 추가:

    <ClCompile Include="..\..\Shared\GameSim\Systems\RecallSystem.cpp" />
```

새 `.h/.cpp` 파일 때문에 프로젝트 등록이 필요할 수 있으면 검증 섹션에만 아래처럼 적는다.

```text
확인 필요:
- 새로 추가한 파일이 빌드 프로젝트에 포함되는지 확인.
```

사용자가 명시적으로 `.vcxproj` 또는 `.filters` 수정 계획을 요구한 경우에만 XML 변경 지시를 쓴다.

## EngineSDK 작성 규칙

`EngineSDK/inc` 아래 파일은 `UpdateLib.bat`으로 동기화되는 산출물이다.

계획서의 `반영해야 하는 코드`에는 `EngineSDK/inc/...` 파일을 넣지 않는다.

Engine public header 변경 때문에 SDK 동기화가 필요하면 검증 섹션에만 아래처럼 적는다.

```text
후속 동기화:
- Engine public header 변경 후 `UpdateLib.bat` 실행 필요.
```

사용자가 명시적으로 EngineSDK 파일 직접 수정을 요구한 경우에만 예외로 한다.

## 이미 반영된 코드 작성 금지

이미 코드에 반영된 내용은 계획서에 쓰지 않는다.

나쁜 예:

```text
코드상 1차 패치 반영:
- CNavGrid에 SetAllWalkable, SegmentWalkable, CountWalkableCells가 이미 추가되어 있다.
- Client 클릭 이동 보정과 local segment 차단이 이미 연결되어 있다.
```

이미 반영된 내용은 검증 대상일 때만 검증 섹션에 짧게 쓴다.

```text
수동 확인:
- Client 클릭 이동 보정이 인게임에서 벽을 넘지 않는지 확인.
```

## 코드 블록 규칙

Markdown fenced code block을 쓸 때는 info string을 붙인다.

```text
...
```

```cpp
...
```

대화 출력에서 사용자가 plain text 스타일을 원하면, 섹션 제목 아래에 `text`, `cpp` 라벨을 별도 줄로 적는 형식도 허용한다.

## 준수 규칙

계획서는 코드 작성 규칙과 CLAUDE.md의 가이드를 준수한다.

단, 계획서의 이름 작성 방식과 출력 형식은 이 문서가 우선한다. 새로 제안하는 심볼은 PascalCase를 기본으로 쓰고, 기존 코드를 정확히 참조해야 할 때만 기존 이름을 그대로 쓴다.

불확실한 위치나 의미가 있으면 추측으로 단정하지 말고, "확인 필요"라고 표시한다.

장황한 배경 설명보다 구현자가 바로 적용할 수 있는 기존 코드/교체 코드/추가 코드/삭제 범위를 우선한다.
