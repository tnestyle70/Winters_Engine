# 1. Winters Engine — 자체 C++20/DX11 엔진 + 서버 권위 MOBA

[GIF: 클라 2분할 + 서버 콘솔 + 백엔드 docker — 서버 권위 동기화]

| | |
|---|---|
| 인원·역할 | 1인 (엔진·클라·서버·백엔드 전체) |
| 기간 | 2026.04 ~ 진행 중 |
| 기술 | C++20, DirectX 11 (자체 RHI + DX12 백엔드), IOCP, FlatBuffers, ImGui, FMOD, Go/PostgreSQL/Redis/Kafka/Docker |
| 링크 | https://github.com/tnestyle70/Winters_Engine · ▶ 영상 0:00~0:30 |

DX9 팀 프로젝트에서 TCP 서버와 20tick 동기화를 만들면서, 프레임워크가 정해준 구조 안에서는 틱 제어·복제 필터·치팅 검증 같은 핵심을 제 손으로 설계할 수 없다는 걸 알았습니다. 그래서 엔진의 본질을 "만드는 사람이 한 번의 빌드 안에서 반복할 수 있는 속도"로 정의하고, LoL 구조의 MOBA 하나에 최적화된 엔진을 바닥부터 만들었습니다. 지금은 로그인 → 계정(RP·전적·리플레이) → 상점 → 로비 → 밴픽 → 로딩 → 인게임(서버 권위 30Hz) → 결과·MMR 반영 → 리플레이 저장까지 한 바퀴가 전부 실제로 돌아갑니다. 그리고 이 엔진의 구현 사이클 자체를 AI 에이전트 하네스(§6)로 돌립니다 — 제가 문제를 정의하면 AI가 구현하고, 빌드·스모크·결정성 해시 게이트를 통과해야만 반영됩니다.

## 서버 권위 — 클라이언트를 신뢰하지 않는다

전작에서 드래곤 보스의 공격이 시간이 지날수록 여러 마리가 공격하는 것처럼 중첩되는 동기화 버그를 겪었고, 당시엔 원인을 닫지 못했습니다. 나중에 코드를 다시 열어 원인이 서버와 클라이언트 양쪽에 공격 연출의 권위가 있었던 것임을 확인했고, Winters는 그 사고가 불가능한 구조로 설계했습니다 — 모든 게임플레이 진실은 서버만 만들고, 클라이언트는 입력과 연출만 담당합니다. 커맨드는 단일 입구에서 시퀀스·유효성(CanMove/CanCast)을 검증받고, 봇 AI조차 같은 관문을 통과하는 GameCommand 생산자라서 진실을 직접 수정하는 경로 자체가 없습니다. 지금 챔피언 17종이 30Hz 서버 시뮬레이션 위에서 돌아갑니다.

즉시 반응(우클릭 회전)과 서버 보정이 충돌하는 문제는 예측 보호 규약(최대 12스냅샷)으로, 고핑 유저의 판정은 서버가 틱마다 기록하는 200ms 위치 히스토리 지연 보상으로 풀었습니다. 스냅샷은 수신자별로 개별 빌드되는데(ack 시퀀스·로컬 엔티티 개인화), 여기에 시야 기반 필터를 더해 "안 보이는 적은 애초에 보내지 않는" 복제로 확장하는 것이 다음 단계입니다.

## 결정성 — 검증을 기계에게 맡긴다

108개 파일을 고치는 리팩터링을 하고 나면 "동작이 안 변했다"를 사람이 확인할 방법이 없습니다. 그래서 시뮬레이션 300틱을 돌린 결과를 해시 하나로 요약해, 수술 전후 해시가 같으면 무회귀임을 기계가 증명하게 했습니다. 서버는 매 매치를 리플레이(.wrpl)로 기록하고 클라이언트는 네트워크 없이 그 스트림을 재생하는데, 이 리플레이가 버그 재현·성능 측정·관전의 공용 기반이 됩니다. 시뮬레이션을 90초까지 되감는 크로노 브레이크도 서버에 내장돼 있습니다.

## 성능 — 감이 아니라 계측으로

프레임이 느려졌을 때 코드 추론 대신 프로파일러부터 만들었습니다. 처음 Debug 빌드에서 잡은 병목 순위가 Release에서 뒤집히는 걸 직접 실측한 뒤로는, 최적화 빌드에 계측을 상시 탑재했습니다 — GPU 패스별 타임스탬프, 통합 드로우콜 카운터, 리플레이 기반 무인 캡처까지, 프로파일러 자기 오버헤드는 프레임의 1.5% 게이트 안에서. 이 파이프라인이 지목한 1위 병목은 화면 밖 캐릭터까지 매 프레임 애니메이션 포즈를 계산하는 것(프레임의 25%)이었고, 포즈 계산을 소비 지점으로 미루는 지연 평가로 바꿔 **Update 페이즈를 0.72ms에서 0.17ms(-76%)로** 줄였습니다. 같은 리플레이를 세 번 돌려 재현을 확인했고, GPU 시간과 드로우콜이 불변임을 계측으로 검증했습니다.

## 엔진 코어 — 같은 문제를 세 번 푼 이야기

Starcraft에서는 모든 객체가 CObj 상속이었습니다. 직관적이었지만 죽은 유닛을 가리키는 참조가 계속 사고를 냈습니다. 던전스에서는 COM 스타일 수동 AddRef/Release로 수명을 지켰는데, 이번엔 사람이 Release를 빼먹으면 끝이었습니다. Winters의 답은 세대(generation) 핸들입니다 — 파괴 후 재사용된 슬롯을 가리키는 낡은 참조가 사용 시점에 O(1)로 거절됩니다. 병렬화는 Chase-Lev work-stealing JobSystem으로, lock-free deque의 owner-push 규약을 위반한 race를 직접 규명해 고쳤고, Fiber 실행 모드는 10만 잡 벤치마크에서 워커 처리량 **+25%(439,389→550,902 jobs/s)** 를 실측했습니다. 렌더러는 IRHIDevice 핸들 경계 뒤에 DX11 기본 경로와 DX12 백엔드를 격리했습니다.

## 데이터와 툴 — 기획자·디자이너가 코드 없이 일하게

JSON은 무엇을 할 것인가를, C++은 그것을 어떻게 실행할 것인가를 담당합니다. 밸런스 수치는 JSON으로 저작하고 쿡 타임에 전수 검증(중복 키·범위·누락 단계 차단)해 C++ 팩으로 컴파일합니다 — 서버 런타임의 JSON 파싱은 0회이고, 85개 스킬을 "전체 계약이 유효한가"로 검사합니다. 에셋은 FBX/GLB 27종을 자체 .wmesh로 전수 변환(FAIL 0)해 60MB 모델이 1.2MB가 됐습니다. 디자이너용 이펙트 툴(FBX·텍스처 로드, 수치 조정, 인게임 프리뷰)과 챔피언·구조물 튜너는 전부 ImGui이고 Debug 게이트 뒤에 격리돼 있으며, 에디터는 클라이언트와 분리된 독립 실행 파일로 시작했습니다.

## 매치 밖 — 백엔드와 예외 경로

계정·RP·상점·전적은 전투와 소유자가 다릅니다. Go 마이크로서비스 3종(인증/프로필/상점)에 PostgreSQL·Redis·Kafka를 docker-compose로 묶어, 가입부터 챔피언 구매·매치·전적 저장까지 로컬 E2E를 완주했습니다. 그리고 행복 경로만 믿지 않기 위해 로비 이탈·픽 없이 시작·로딩 실패·서버 사망·악성 커맨드까지 예외 경로를 전수 감사했습니다 — 크래시 경로는 0이었고, 대신 침묵하는 실패들을 찾아 수복 순위를 매겼습니다. 클라이언트 게이트(UX)와 서버 게이트(보안)는 둘 다 있어야 한다는 게 이 감사의 결론입니다.

## 한계와 다음 단계

UDP 경로는 구현·검증까지 마쳤고 기본 전환 게이트가 남아 있습니다. 시스템 스케줄러의 병렬 배치 최적화가 계측이 지목한 다음 타깃이고, 데이터 버전 핸드셰이크는 로그 전용에서 접속 거부 게이트로 승격할 예정입니다.

### 구현

- **[구조] Base/Engine/Client/Server 분리** — `CBase` AddRef/Release 수동 reference count(COM 스타일)로 엔진 객체 생명주기 관리, GameObject/Component + `CProtoMgr`의 Prototype/Clone으로 반복 생성 비용 절감 (Unity Prefab·UE CDO에 대응하는 개념을 몸으로 먼저 배움)
- **[담당] TCP 소켓 서버(select) + 20 tick 스냅샷 동기화** — Shared `PacketDef`로 login/input/StateSnapshot/projectile/damage/DragonSync 등 패킷 계약 명시(구조체 크기 static_assert 검증), 틱마다 상태 브로드캐스트. 이때 select 모델의 한계를 체감한 것이 Winters에서 IOCP(워커 4+틱 1)로 넘어간 이유
- **[담당] 엔더드래곤 보스전** — 블럭 vertex 로컬 좌표 조작으로 날개 생성, IK solver + sin파 연결 움직임으로 물리 시뮬레이션풍 드래곤 구현, 서버 권위 브로드캐스트 동기화
- **[담당] 블럭 배치 렌더링 — 측정으로 선택한 최적화**: 배치 버퍼로 드로우콜을 묶는 방식과 쿼드트리 프러스텀 컬링을 **둘 다 구현해 비교**했고, 이 씬에서는 배치가 더 좋은 수치가 나와 배치를 채택하고 컬링은 제외했습니다 — "정석"이 아니라 측정이 결정
- 그 외: ImGui 맵 에디터(멀티 스테이지·트리거 스폰), 파티클, HLSL 포스트프로세스 6종, 파도 스테이지(Sin파/Gerstner 구현 — FFT는 시도 후 범위 조정을 인정)

### ▶ 더 깊이 — 트레이드오프와 뒷이야기 (토글)

- **협업의 실제**: 다섯 명이 한 레포에서 4주 — 충돌은 기술이 아니라 규약으로 줄었다. 브랜치 전략과 PR 리뷰 강제, 리소스 일정과 코드 일정의 분리, 그리고 팀원 간 갈등을 팀장이 조율하는 것이 마감을 지킨 이유였다.
- **설계 일화 — 버그의 디자인 승격**: 드래곤 돌진이 의도(1회+꼬리치기)와 달리 2연속 돌진으로 동작했는데, 플레이해 보니 더 위협적이라 채택했다. 구현 중 발견을 기획으로 승격한 경험.
- **미해결 버그의 사후 규명 — 드래곤 동기화 버그**: 플레이 시간이 지날수록 드래곤의 공격이 **여러 마리가 공격하는 것처럼 중첩되어 보이는** 동기화 버그가 있었고, 당시에는 원인을 닫지 못한 채 마감했습니다. 프로젝트가 끝난 뒤 실코드를 다시 열어 원인을 규명했습니다 — ①공격 연출의 권위가 서버와 클라이언트 양쪽에 있어 같은 공격이 두 경로로 발화하는 **이중 권위 에코** ②클라이언트 로컬 타이머가 서버 틱과 어긋나며 지난 패턴의 연출이 누적되는 **타이머 드리프트** ③스테이지 트리거의 재장전. Winters는 이 사고의 구조적 재발 방지로 설계됐습니다: **모든 전투 사실은 서버 이벤트/스냅샷으로만 생성**되고 클라이언트는 연출만 담당하며, 시퀀스 규율과 이벤트 적용 단일 지점(EventApplier)이 이중 발화를 원천 차단하고, 300틱 결정성 해시가 회귀를 감시합니다. "버그를 고쳤다"가 아니라 **"버그가 불가능한 구조로 다시 지었다"**는 것이 이 서사의 결론입니다.
- **수동 ref-count의 교훈**: Release 누락 한 번이 어디서 터질지 모르는 누수가 된다는 걸 팀 코드에서 겪었다 — Winters에서 RAII+세대 핸들로 같은 문제를 구조적으로 재해결한 출발점.
- **이 프로젝트가 Winters로 이어진 것**: 패킷 계약·스냅샷 동기화 → GameCommand 검증·FlatBuffers Snapshot/Event. 드래곤 이중 권위 사고 → 서버 단일 진실. 그리고 "프레임워크가 정해준 구조 안에서는 틱 제어·복제 필터를 설계할 수 없다"는 한계 인식이 자체 엔진의 출발점.

# 3. NYPC — 정답 없는 문제를 AI와 함께: 게임 봇 대회

[이미지: 대회 게임 화면 또는 self-play 승률 그래프]

| | |
|---|---|
| 대회 | NYPC [연도·부문·최종 성적 정식 표기 CONFIRM] — 게임 2종 (턴제 보드 / 경제 RTS) |
| 기술 | C++, Python, alpha-beta 탐색, self-play 인프라, 리플레이 분석 |
| 링크 | [GitHub — NYPC_Mushroom](https://github.com/tnestyle70/NYPC_Mushroom) |


게임 봇 대회는 "정답이 없는 문제"입니다 — 컴파일 성공이 아니라 상대를 이겨야 하고, 뭐가 좋은 수인지 아무도 알려주지 않습니다. 저는 봇을 잘 만드는 것보다 **봇이 좋아졌는지 판정하는 장치**(자기 자신과 수천 판 대전시키는 self-play 인프라)를 먼저 만들었고, 튜닝 값을 감이 아니라 대전 승률로 골랐습니다. 34일의 대회 운영을 기록으로 남겨 실패 원인까지 해부했습니다.

### 구현

- **버섯게임 (턴제 보드)**: Search Order(먼저 볼 후보 선별) + alpha-beta pruning + 2단 평가함수(수 평가 = ImmediateSwing+DenyGain / 리프 평가 = ScoreDiff+BestAll)를 설계하고, **self-play 인프라(Python)를 먼저 구축해 변수·가중치 튜닝을 주관이 아니라 대전 승률로 선별** — 성능 정체(플래토)가 오면 가중치를 더 만지는 대신 평가 구조 자체를 의심하는 법을 배움
- **NEXT NATION (경제 RTS)**: 3층 봇 구조 — GameState(사실) → Perception(사실의 가공: 이전 턴 상대 행동 분석, 이동 예상 경로와 ETA 산출) → Behavior(Attack/Move/Upgrade/Rush/Banking을 수학적 손익 계산으로 선택). self-play trace로 "봇이 왜 그 선택을 했는가"를 역추적 가능하게 설계 — 목표는 강한 AI가 아니라 **디버깅 가능한 AI**
- **정직한 회고**: 초반 이후 "이기기 위한 전략"보다 확실하게 닫히는 개선에 시간을 쓴 것이 정체의 원인 — 34일 행동 기록을 해부해 원인을 특정했고, 이후 모든 프로젝트의 시간 배분 규칙으로 반영

### ▶ 더 깊이 — 트레이드오프와 뒷이야기 (토글)

- **왜 평가 인프라가 봇보다 먼저인가**: 평가 장치가 없으면 모든 개선이 "느낌상 좋아짐"이 된다. 리그 평가(신버전 vs 구버전 N판)는 개선을 가설→실험→판정 루프로 바꾼다 — 이 사고방식이 이후 Winters의 결정성 해시·프로파일링 원장으로 그대로 이어졌다. "측정 장치 먼저"는 제 모든 프로젝트의 공통 패턴입니다.
- **탐색의 대가**: alpha-beta는 평가함수가 나쁘면 깊이가 독이 된다(나쁜 기준으로 더 멀리 본다). 평가함수 항 하나를 바꿀 때마다 리그를 다시 돌린 이유.
- **이 경험이 만든 것**: 결정론 시뮬 → self-play → trace 분석 → 리그 평가 방법론. 대회 종료 후 Winters의 봇 설계(봇도 GameCommand 생산자·AI 디버그 패널·시뮬 되감기)로 이식 중.

# 4. Starcraft 모작 — WinAPI 로우레벨 RTS 코어

[GIF: 유닛 군집 이동과 전장의 안개 (sc_flocking)]

| | |
|---|---|
| 인원·역할 | 1인 · 기간 2025.12 ~ 2026.01 · 기술 C++20, WinAPI/GDI, FMOD |
| 링크 | [GitHub](https://github.com/tnestyle70/Starcraft) · ▶ 영상 1:00~1:30 |


엔진도 그래픽 라이브러리도 없이, 윈도우 기본 API만으로 스타크래프트의 코어(유닛 이동·전투·생산·전장의 안개·미니맵)를 구현했습니다(약 5.4만 줄). 상용 엔진이 대신 해주던 것들 — 길찾기, 수십 유닛의 동시 이동, 반투명 그리기 — 을 바닥에서 직접 만들며 원리를 확보한 프로젝트입니다.

### 구현

- **[이동] Octile A\*(코너컷 방지·타일 비용) + Separation/Cohesion 스티어링 결합** — 전역 경로탐색과 지역 조향의 역할 분리로 수십 유닛 군집 이동
- **[안개] 3상태 전장의 안개(미탐사/탐사/시야)를 dirty-flag 캐시 DC로 최적화** — 매 프레임 전체 재계산 대신 변한 타일만 갱신
- **[렌더] GDI 비트연산(SRCAND/SRCPAINT)+AlphaBlend 3단 합성** — 셰이더 없이 팀컬러·반투명 구현
- **[객체] CObj 최상위 추상 + Object Manager, pending delete** — 죽은 객체를 즉시 delete하지 않고 selection·컨트롤 그룹·명령·투사체 타깃 참조를 일괄 정리 후 해제
- **[생산] 템플릿 팩토리 + deque 기반 FIFO 생산 큐** — 명령 입력과 결과 생성의 시간 분리
- **[UI] Commandable/CommandSlot 계약** — 유닛(32종)·건물(40종)이 자기 커맨드 카드를 정의하고 UI는 렌더만: UI가 게임 로직을 침범하지 않는 경계
- 인게임 맵 에디터 → 바이너리 저장 → 내비 그리드 재생성 파이프라인

### ▶ 더 깊이 — 트레이드오프와 뒷이야기 (토글)

- **가장 어려웠던 것 — 어택무브**: "이동 중 적을 만나면 교전하고, 끝나면 원래 목적지로 복귀"는 명령 큐 소비와 상태 전환이 충돌하는 문제다. Order에 데이터를 내장하고 유닛 FSM(9상태)과 책임을 분리해 풀었다 — 명령은 의도, 상태는 실행이라는 구분을 여기서 배웠다.
- **RTS에서 죽음은 화면에서 사라지는 것으로 끝나지 않는다**: 죽은 유닛을 선택 목록·컨트롤 그룹·투사체 타깃이 여전히 참조한다. pending delete는 이 문제의 수동 해법이었고, Winters의 세대 핸들은 같은 문제의 구조적 재해결이다 — 같은 문제를 두 번 푼 것이 아니라, 더 나은 언어로 다시 푼 것.
- **로우레벨의 가치**: GDI에는 알파 블렌딩이 사실상 없다 — 비트 마스크 연산 조합으로 만들었다. 이후 DX11 블렌드 스테이트를 쓸 때 "엔진이 뭘 대신해 주고 있는지"를 정확히 아는 상태가 됐다.

# 6. AI 에이전트 협업 방식 (마지막 섹션)


AI가 코드를 짜는 시대에 제 일은 두 가지로 이동했다고 생각합니다 — **문제를 정확한 언어로 정의하는 것**, 그리고 **AI의 출력이 통과해야 하는 검증 장치를 설계하는 것**. 두 개의 AI(Claude, Codex)를 병렬로 운영하며 자체 엔진을 개발하고 있고, 그 운영 방식 전부가 저장소에 실물로 남아 있습니다.

### 구현

- **컨벤션과 맥락의 문서화**: 코딩 가드레일(가정 금지·최소 수정·검증 가능한 목표), 아키텍처 컴퍼스, 계층 소유권 문서를 저장소에 두고 모든 에이전트 세션이 같은 규칙에서 출발
- **재발 방지 로그**: 같은 실수가 두 번 나오면 `gotchas.md`에 "실수 → 예방 규칙" 한 줄로 박제(30여 항목) — 에이전트가 매 세션 이 로그를 읽고 시작하므로 실수가 세대를 넘어 전파되지 않음
- **검증 게이트 강제**: 에이전트 출력은 빌드·스모크·SimLab 300틱 결정성 회귀를 통과해야 하고, 계획서는 근거 코드 경로·검증 명령·롤백 범위를 포함해야 하는 출력 규칙을 훅으로 강제 — "AI가 짰다"가 아니라 "게이트를 통과했다"가 머지 기준
- **레인 분리와 맥락 승계**: Claude/Codex가 각자 세션 문서 시퀀스를 남기며 서로의 산출물을 읽고 이어받음 — 최근 사례: 한쪽이 작성한 프로파일링 측정 계약 문서를 다른 쪽이 코드로 구현하고, 실측 결과로 원문서의 가설(Debug 캡처 병목 순위)을 반증·교정

### ▶ 더 깊이 — 트레이드오프와 뒷이야기 (토글)

AI 활용은 3단계로 진화했습니다: 질문-답변 프롬프트(학습기) → 파일 단위 수정 → **에이전트가 일하는 환경 자체의 설계**(현재). 핵심 발견은 프롬프트 기술이 아니라, 사람 팀과 똑같이 **컨벤션·리뷰·회귀 테스트·인수인계 문서가 AI 협업의 품질을 결정**한다는 것입니다. 한 사람이 다룰 수 있는 작업의 폭이 몇 배로 넓어진 만큼, 문제를 정의하고 검증 체계를 만드는 능력이 곧 생산성이 되었습니다 — 이 포트폴리오의 프로젝트들이 그 방식의 산출물입니다.
