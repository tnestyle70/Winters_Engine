## 1. Winters Engine — 자체 C++20/DX11 엔진 + 서버 권위 MOBA <a id="1"></a>

![클라 2개 + 서버 콘솔 동시 화면 — 서버 권위 동기화](media/winters_sync.gif)

| | |
|---|---|
| 인원·역할 | 1인 (엔진·클라·서버·백엔드 전체) |
| 기간 | 2026.04 ~ 진행 중 |
| 기술 | C++20, DirectX 11 (자체 RHI 추상화 + DX12 실험 경로), IOCP, FlatBuffers, ImGui, FMOD, Go/PostgreSQL/Redis |
| 링크 | [GitHub CONFIRM] · ▶ 영상 0:00~0:30 |

**[의도]** 팀 프로젝트(DX9 자체 엔진)에서 멀티스레드와 20TPS 스냅샷 동기화를 만들며, 프레임워크가 정해준 구조 안에서는 틱 제어·복제 필터·치팅 검증 같은 핵심을 설계할 수 없다는 한계를 느꼈습니다. 상용 엔진은 모든 장르를 담기 위해 범용으로 설계되어 있어, 특정 게임에 최적화된 구조와는 필연적으로 거리가 생깁니다. 그래서 저는 엔진의 본질을 **"만드는 사람이 한 번의 빌드 안에서 반복할 수 있는 속도"**로 정의하고, LoL 구조의 MOBA 하나에 최적화된 엔진을 바닥부터 만들기 시작했습니다.

**[핵심 설계 — 클라이언트를 신뢰하지 않는다]** 모든 게임플레이 진실은 서버만 만듭니다:

```text
Client Input → GameCommand → Server GameSim (30Hz 고정 틱)
            → FlatBuffers Snapshot/Event → Client 보간·연출
```

클라이언트가 메모리를 조작해 임의 값을 보내도, 서버는 커맨드의 시퀀스·유효성(`CanMove/CanCast`)을 검증하고 자기 시뮬레이션 결과만 배포합니다. 봇 AI조차 GameCommand 생산자로 통일해 서버 진실을 직접 수정하는 경로 자체를 없앴습니다. 클라 예측(우클릭 즉시 회전)과 서버 보정의 충돌은 예측 보호 규약(최대 12스냅샷)으로 풀었고, 지연 보상을 위해 서버가 200ms 위치 히스토리를 틱마다 기록합니다.

**[계측 문화 — 프로파일러 스토리]** 프레임이 17.8ms로 떨어졌을 때, 코드 추론 대신 자체 CPU 프로파일러를 먼저 만들었습니다. 클래스별 update/render에 계측 스코프를 심고 설정한 틱 단위로 JSON을 축적해, 특정 구간의 병목이 로직(update)인지 렌더인지부터 분리했습니다. 스키닝 갱신이 프레임의 90%(16ms)라는 것을 계측으로 확정한 뒤 불필요 갱신을 제거해 9ms(~110fps)로 복구했습니다. 계측은 Debug 구성에서만 활성화됩니다.

**[데이터 파이프라인 — 기획자 친화 구조]** 밸런스 수치는 JSON으로 저작하고, 쿡 타임에 전수 검증(중복 키·해시 충돌·도메인 검사)해 static const C++ 팩으로 컴파일합니다 — **서버 런타임의 JSON 파싱은 0회**이고, 산출물 스테일은 `--check` 게이트가 빌드 전에 차단합니다. 식별자는 3계층(안정 DefinitionKey / 팩-로컬 DefId / 프로세스-로컬 EntityHandle)으로 분리해 팩-로컬 id가 네트워크·저장 경계를 넘지 못하게 했습니다. 밸런스 변경 사이클은 JSON 수정 → 쿡 → SimLab 300틱 결정성 회귀가 무회귀 게이트입니다.

**[에셋 파이프라인]** Assimp 기반 오프라인 컨버터로 FBX/GLB 27종을 자체 바이너리(.wmesh 등)로 전수 변환(FAIL 0). 런타임에 Assimp 파싱 비용이 사라지고, 대표 메시 60MB → 1.2MB(~50배), POD 고정 레이아웃 zero-copy GPU 업로드입니다.

**[게임 봇 — 정답 없는 문제의 연장전, 진행 중]** NYPC 대회(아래 3번)가 끝난 뒤, 대회에서 만든 방법론(결정론 시뮬 → self-play → trace 분석 → 리그 평가)을 이 엔진의 봇 설계로 이식하고 있습니다. 현재 상태: 봇 공용 가치평가 스켈레톤 + 시뮬 되감기 인프라(명령 저널 기반, SimLab 검증)까지 구현됐고, Influence Map과 모방학습 파이프라인은 설계 단계입니다. 평가 지표는 대회와 동일하게 봇 vs 봇 리그 승률로 잡았습니다.

**[한계와 다음 단계 — 정직 기재]** 수송은 TCP이며 UDP 이관은 설계 완료 상태입니다. Job 병렬화(Chase-Lev work-stealing deque)는 구현했으나 동시성 race 수정 전까지 기본 비활성입니다. 데이터 버전 핸드셰이크는 현재 로그-only로, 접속 거부 게이트 승격이 다음 단계입니다. RHI는 DX11 런타임 + DX12 실험 경로이며, 추상화 계층은 백엔드 추가를 전제로 설계했습니다.

- **[담당] IOCP 서버 + 20TPS 스냅샷 동기화**: 커스텀 바이너리 패킷 20여 종을 정의하고 틱마다 상태를 브로드캐스트 — 팀원들과 같은 와이파이에 모여 한 스테이지에서 레이드·PVP를 실제 시연했습니다.
- **[담당] 엔더드래곤 보스전**: 블럭 vertex의 로컬 좌표를 조작해 날개 형태로 생성하고, IK solver + sin파로 이동에 따른 블럭 연결 움직임을 구현했습니다. 위치는 서버 권위 broadcast로 동기화. **설계 일화**: 돌진 공격이 구현 과정에서 의도(1회 돌진+꼬리치기)와 다르게 2연속 돌진으로 동작했는데, 실제로 더 위협적이고 재미있어서 그 움직임을 채택했습니다 — 구현 중 발견을 디자인으로 승격시킨 경험입니다.
- **[담당] 블럭 배치 렌더링 최적화**: 월드 블럭들을 배치 버퍼로 묶어 `DrawIndexedPrimitive` 호출을 대폭 축소(쿼드트리 프러스텀 컬링과 결합) — `CBlockBatchBuffer` 직접 구현.
- 그 외: ImGui 맵 에디터(멀티 스테이지·트리거 스폰), 파티클, HLSL 포스트프로세스 6종, 파도 스테이지(sin/거스트너 파 — FFT 파도는 시도 후 실패를 인정하고 범위 조정).
- **이 프로젝트가 Winters로 이어진 이유**: 여기서 만든 20TPS 스냅샷이 30Hz 서버 권위 풀 파이프라인(커맨드 검증·예측·랙컴펜·결정성 해시)으로 확장됐습니다.

## 3. NYPC — 정답 없는 문제를 AI와 함께: 게임 봇 대회 <a id="3"></a>

| | |
|---|---|
| 대회 | NYPC [연도·부문 정식 표기 CONFIRM] — 게임 2종 (턴제 보드 / 경제 RTS) |
| 기술 | C++, Python, alpha-beta 탐색, self-play 인프라, 리플레이 분석 |
| 링크 | [GitHub — NYPC_Mushroom](https://github.com/tnestyle70/NYPC_Mushroom) |

- **버섯게임**: alpha-beta pruning 탐색 + 평가함수(MoveEval = ImmediateSwing+DenyGain, LeafEval = ScoreDiff+BestAll)를 설계하고, **self-play 인프라를 먼저 구축해 튜닝 값을 주관이 아니라 대전 승률로 선별**했습니다.
- **NEXT NATION (경제 RTS)**: 상대 관측 정보를 Perception 구조체(GameState·ETA·상대 유닛 예상 경로)로 모델링하고, self-play trace를 추출해 "봇이 왜 그 선택을 했는가"를 역추적 가능한 구조로 설계했습니다 — 디버깅 가능한 AI가 목표였습니다.
- **정직한 회고**: 초반 이후 "이기기 위한 전략"보다 확실하게 닫히는 개선에 시간을 쓴 것이 정체의 원인이었습니다. 34일 행동 기록을 해부해 원인(승자 분석 지연, 프록시 지표 최적화)을 특정했고, 이후 모든 프로젝트에 시간 배분 규칙으로 반영했습니다.
- **이 경험이 만든 것**: "정답이 없는 문제를 AI와 함께 끝까지 푸는" 방법론 — 결정론 시뮬 → self-play → trace 분석 → 리그 평가. 대회 종료 후 이 파이프라인을 Winters 봇 설계로 이식 중입니다(1번 참조).

## 4. Starcraft 모작 — WinAPI 로우레벨 RTS 코어 <a id="4"></a>

![유닛 군집 이동과 전장의 안개](media/sc_flocking.gif)

| | |
|---|---|
| 인원·역할 | 1인 · 기간 2025.12 ~ 2026.01 · 기술 C++20, WinAPI/GDI, FMOD |
| 링크 | [GitHub](https://github.com/tnestyle70/Starcraft) · ▶ 영상 1:00~1:30 |

- **가장 어려웠던 구현 — 어택무브(A-move)**: 명령 큐(`deque<Order>`)가 하나씩 소비되는 구조에서, "웨이포인트로 이동하며 사거리 내 적을 순차 처치 후 복귀"라는 상태 전환이 큐 소비와 충돌했습니다. Order에 경로·타깃을 내장하고 유닛 FSM(9상태)과 명령 큐의 책임을 분리해 해결했습니다.
- **댕글링 타깃 크래시**: 타깃 유닛이 죽어 매니저에서 제거된 뒤 다음 프레임에 nullptr 접근으로 크래시 — 타깃 전용 관리(TargetMgr)와 사망 체크를 도입해 죽은 유닛 참조를 일괄 정리했습니다. (Winters에서 세대(generation) 핸들로 같은 문제를 구조적으로 재해결 — 같은 문제를 두 번, 더 좋게 푼 사례)
- Octile A\*(코너컷 방지) + 플로킹 결합 군집 이동, 3상태 전장의 안개(dirty-flag 캐시), GDI 비트연산 알파 합성, 인게임 맵 에디터→바이너리 저장→내비 그리드 재생성. 고스트 빌딩/미네랄/캐리어 등 원작 디테일 재현.

## 6. AI 에이전트 협업 방식 <a id="6"></a>

AI 활용은 3단계로 진화했습니다. 처음에는 질문-답변 프롬프트와 파일 단위 버그 수정으로 시작했고(Liberation 이후 학습기), 지금은 **에이전트 하네스를 설계해 두 개의 LLM(Claude Code, Codex)을 병렬 레인으로 운영**하며 NYPC 봇과 Winters Engine을 개발합니다.

핵심은 프롬프트가 아니라 **AI가 일하는 환경의 설계**라고 생각하며, 전부 실물로 보여드릴 수 있습니다:

- **컨벤션과 맥락의 문서화**: 코딩 가드레일(가정 금지·최소 수정·검증 가능한 목표 — Karpathy 원칙을 명문화), 아키텍처 컴퍼스, 계층 소유권 문서를 저장소에 두고 모든 에이전트 세션이 같은 규칙에서 출발하게 합니다.
- **재발 방지 로그**: 같은 실수가 두 번 나오면 `gotchas.md`에 "실수 → 예방 규칙" 한 줄로 박제합니다(현재 30여 항목). 에이전트가 매 세션 이 로그를 읽고 시작하므로 실수가 세대를 넘어 전파되지 않습니다.
- **검증 게이트 강제**: 에이전트의 출력은 빌드·스모크·SimLab 300틱 결정성 회귀를 통과해야 하고, 계획서는 근거 코드 경로·검증 명령·롤백 범위를 포함해야 하는 출력 규칙을 훅으로 강제합니다. "AI가 짰다"가 아니라 "게이트를 통과했다"가 머지 기준입니다.
- **레인 분리와 맥락 승계**: Claude/Codex가 각자 세션 문서 시퀀스(S001, S002…)를 남기며 서로의 산출물을 읽고 이어받습니다 — 34일 NYPC 대회 운영과 Winters 협업 파이프라인 문서가 그 기록입니다.

AI가 코드 작성과 빌드 검증까지 수행하는 시대에, 개발자의 시간은 **문제를 정의하고, 구조의 확장성·유지보수성을 설계하고, 검증 체계를 만드는 곳**으로 이동했다고 생각합니다. 한 사람이 다룰 수 있는 작업의 폭이 몇 배로 넓어진 만큼, 문제를 정확한 언어로 정의해 전달하는 능력이 곧 생산성이 되었습니다 — 이 이력서의 프로젝트들이 그 방식의 산출물입니다.

## 7. 자기소개 <a id="7"></a>

저는 문과 출신입니다. 혼자서도 만들 수 있다고 믿었고, 실제로 UE Blueprint를 독학해 공포 게임 Liberation을 1년에 걸쳐 Steam에 출시했습니다.

시장의 답은 냉정했습니다. 완성과 출시는 해냈지만, 혼자 만든 게임은 난이도조차 객관화할 수 없다는 것을 배웠습니다. 제대로 배우고 함께 만들기 위해 쥬신 아카데미에 들어갔고, WinAPI로 스타크래프트의 코어를 바닥부터 구현하며 상용 엔진이 대신 해주던 것들의 원리를 확보했습니다. 5인 팀 프로젝트에서는 팀장으로 git 세팅부터 서버 개발까지 맡아 PR 135건을 머지하며 협업을 배웠습니다.

팀 프로젝트가 끝났을 때 프레임워크의 한계가 보였습니다. 서버 권위, 시야 기반 복제, 결정성 검증 — 만들고 싶은 것들은 구조를 소유해야 가능했습니다. 그래서 자체 엔진 Winters를 시작했고, 지금 LoL 구조의 서버 권위 멀티플레이가 그 위에서 돌아갑니다.

NYPC 대회는 저에게 "정답이 없는 문제를 AI와 함께 끝까지 푸는" 훈련장이었습니다. AI 이전 시대에 책과 영상으로 배웠던 저와 지금의 저를 비교하면, 차이는 정보 획득과 구현의 속도입니다. AI가 코드를 작성하고 빌드하고 버그까지 수정하는 시대에 개발자의 본질적 능력은 **문제를 잘 정의하고, 그 정의를 정확한 언어로 전달하는 것**이라고 생각합니다. 저는 AI와의 협업을 위해 컨벤션 문서·재발 방지 로그·검증 게이트를 설계해 맥락을 잇는 방식으로 일하며, 이 방식 자체가 제 포트폴리오의 생산 속도를 만들었습니다.

장기 목표는 제 게임을 만드는 회사를 세우는 것입니다. 그래서 더더욱 현업의 규모와 기준을 배우는 것이 지금 가장 필요한 단계이고, 배운 것을 수치로 증명하는 방식으로 기여하겠습니다.
