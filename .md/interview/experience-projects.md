## 출제 경향 개요

면접관이 "해온 프로젝트 소개해 주세요"를 묻는 진짜 목적은 네 가지다.

1. **진위 검증** — 정말 본인이 만들었는가. 소개에서 나온 키워드 하나를 골라 3~5단계 꼬리질문으로 파고든다. ("A*를 썼다고요? open list는 어떤 자료구조였죠? 휴리스틱은요? 그게 실제 거리보다 크면 어떻게 되죠?")
2. **깊이 검증** — 돌아가게 만든 것을 넘어 **왜 그렇게 했는지**, 대안과 트레이드오프를 아는지.
3. **협업 신호** — 팀에서 어떻게 일하는 사람인가. 갈등/일정/리뷰 관련 행동면접 질문이 반드시 붙는다.
4. **성장 궤적** — 프로젝트 사이에 연결이 있는가. 이전 프로젝트의 한계를 다음 프로젝트에서 어떻게 넘었는가.

따라서 이 카테고리의 준비 전략은 (a) 각 프로젝트의 **1분 소개**를 준비하고, (b) 소개에 넣은 키워드마다 **꼬리질문 3단계**까지 방어선을 파고, (c) 행동면접 질문에는 **STAR 형식 + 실명 사례**로 답하는 것이다. 추상적인 "열심히 했다"는 탈락 답변이다.

## 핵심 개념 정리

### 1. WinAPI 메시지 루프 — GetMessage vs PeekMessage

**정의**: Windows GUI 프로그램은 이벤트 구동이다. OS가 입력/윈도우 이벤트를 스레드별 **메시지 큐**에 넣고, 프로그램은 루프를 돌며 꺼내 처리한다. `GetMessage`는 큐가 비면 **블로킹**(대기)하고, `PeekMessage`는 큐가 비어도 **즉시 반환**한다.

**동작 원리**: 일반 앱은 이벤트가 없으면 CPU를 쓸 이유가 없으니 `GetMessage`로 잔다. 게임은 입력이 없어도 매 프레임 시뮬레이션과 렌더를 돌려야 하므로 `PeekMessage`로 "메시지 있으면 처리, 없으면 게임 틱"을 반복한다.

**내 코드** (`스타크래프트\Starcraft\Starcraft\Starcraft.cpp`):

```cpp
while (msg.message != WM_QUIT)
{
    if (PeekMessage(&msg, nullptr, 0, 0, PM_REMOVE))
    {
        if (!TranslateAccelerator(msg.hwnd, hAccelTable, &msg))
        {
            TranslateMessage(&msg);
            DispatchMessage(&msg);
        }
    }
    else
    {
        CTimeMgr::Get_Instance()->BeginFrame();
        game.Update();
        game.Late_Update();
        game.Render();
    }
}
```

**세부 포인트**:
- 종료 판정이 `while (msg.message != WM_QUIT)`인 이유 — `PeekMessage`는 `GetMessage`처럼 WM_QUIT에서 0을 돌려주는 관례가 없어서, 메시지 자체를 검사한다. `WM_QUIT`은 보통 `WndProc`의 `WM_DESTROY`에서 `PostQuitMessage(0)`가 큐에 넣는다.
- `TranslateMessage`: `WM_KEYDOWN`(가상키)을 문자 입력 `WM_CHAR`로 변환해 큐에 추가.
- `DispatchMessage`: 메시지를 해당 윈도우의 `WndProc`으로 **콜백 호출**시킨다.
- `PM_REMOVE`: 큐에서 꺼내면서 제거 (PM_NOREMOVE는 엿보기만).

**게임 맥락**: 이 구조 때문에 "게임이 한 프레임에 하는 일 = 메시지 소진 + Update + Render"라는 프레임 개념이 성립한다. 메시지 하나만 처리하고 게임 틱으로 넘어가는 위 구조는 폭주 입력 시 입력 지연이 생길 수 있어, 상용 코드는 보통 `while(PeekMessage(...))`로 큐를 다 비운 뒤 틱을 돈다 — 꼬리질문으로 나오면 이 개선 여지를 먼저 말하면 좋다.

### 2. WndProc — 콜백 등록과 메시지 처리

**정의**: `WndProc`은 `LRESULT CALLBACK WndProc(HWND, UINT, WPARAM, LPARAM)` 시그니처의 **윈도우 프로시저**로, 윈도우 클래스 등록 시 함수 포인터로 OS에 넘긴다.

**동작 원리**: `RegisterClassEx`의 `WNDCLASSEX::lpfnWndProc`에 등록 → `CreateWindow`로 그 클래스의 윈도우 생성 → `DispatchMessage`가 호출되면 OS(user32)가 등록된 `WndProc`을 호출한다. 즉 **내가 호출하는 게 아니라 OS가 호출하는 역전(callback) 구조**다. 처리하지 않는 메시지는 반드시 `DefWindowProc`으로 넘긴다 (안 넘기면 창 이동/닫기 같은 기본 동작이 죽는다).

**내 코드**: `Starcraft.cpp`의 `MyRegisterClass`에서 `wcex.lpfnWndProc = WndProc;` 등록. Winters도 동일 계층이 `Engine/Private/Platform/CWin32Window.cpp`의 `WndProc`이고, ImGui 입력 후킹을 위해 `ImGui_ImplWin32_WndProcHandler`를 체인 앞에 끼웠다 (`.md/plan/engine/IMGUI_INTEGRATION_PLAN.md`).

**대표 메시지**: `WM_CREATE`(생성), `WM_PAINT`(그리기 요청 — 게임은 보통 무시하고 자체 루프에서 그림), `WM_KEYDOWN/UP`, `WM_LBUTTONDOWN/UP`, `WM_MOUSEMOVE`, `WM_SIZE`, `WM_DESTROY`(→ `PostQuitMessage`).

**흔한 함정**: (a) 큐 메시지(Post)와 논큐 메시지(Send — 큐를 거치지 않고 WndProc 직접 호출)의 구분, (b) WndProc은 메시지를 보낸 스레드가 아니라 **윈도우를 만든 스레드**에서 호출됨, (c) 게임 입력을 WndProc에서만 받으면 눌림 '상태'(held) 폴링이 어려워 `GetAsyncKeyState` 병행이 흔하다 — Winters의 `CInput`도 `m_Keys[256]`을 WndProc에서 채우되 마우스 좌클릭은 `GetAsyncKeyState(VK_LBUTTON) & 0x8000` 폴링으로 보완했다.

### 3. GDI 더블 버퍼링

**정의**: 화면 DC에 직접 "지우고 → 그리기"를 반복하면 지워진 중간 상태가 화면에 노출되어 깜빡인다(flicker). **메모리 DC(백버퍼)에 완성 화면을 만든 뒤 마지막에 한 번만 화면 DC로 전송**하는 기법.

**동작 원리 (전형 구현)**:

```cpp
// 준비 (1회)
HDC hMemDC = CreateCompatibleDC(hScreenDC);                 // 메모리 DC
HBITMAP hBackBmp = CreateCompatibleBitmap(hScreenDC, W, H); // 백버퍼 비트맵
SelectObject(hMemDC, hBackBmp);

// 매 프레임
PatBlt(hMemDC, 0, 0, W, H, BLACKNESS);   // 백버퍼 클리어
/* 모든 스프라이트를 hMemDC에 그림 */
BitBlt(hScreenDC, 0, 0, W, H, hMemDC, 0, 0, SRCCOPY); // 한 번에 present
```

**투명/합성 처리 — 내 코드**: `CBmpMgr.cpp`는 단순 복사 이상을 한다. 팀 컬러 틴트 합성을 위해 `CreateCompatibleDC`로 중간 DC 여러 장을 만들고, `BitBlt`를 `SRCCOPY`(복사) → `SRCAND`(마스크로 구멍 뚫기) → `SRCPAINT`(OR로 얹기) 순으로 조합한다 (`CBmpMgr.cpp:214~259`). 이것이 GDI 시절의 고전 마스킹 기법이다: **검정(0)과 AND하면 지워지고, 검정과 OR하면 원본이 남는다**는 비트 연산 성질을 이용한다. 이 외 대안이 `GdiTransparentBlt`(색상키 투명), `AlphaBlend`(per-pixel 알파, `BLENDFUNCTION`), GDI+(진짜 알파 채널) — 프로젝트에서 `CGdiPlusMgr`도 함께 운용했다.

**게임 맥락**: DX의 스왑체인 Present와 같은 문제의식(테어링/깜빡임 없는 화면 갱신)을 GDI 수준에서 손으로 구현한 것이다. "DX11에서는 이걸 스왑체인 백버퍼+Present가 대신하고, 저는 Winters에서 그 계층을 직접 관리합니다"로 연결하면 기본기→현재 실력 서사가 된다.

**함정**: 메모리 DC/비트맵/이전 SelectObject 핸들의 **누수** — GDI 핸들은 프로세스당 1만 개 제한이 있어, 매 프레임 Create/Delete를 반복하면 수 분 안에 그리기가 전부 실패한다. "GDI 리소스는 로딩 때 만들어 캐싱하고, 프레임 루프에서는 재사용만 했다"가 모범 답.

### 4. 프레임 타이밍 (deltaTime)

**정의**: 프레임마다 걸린 실제 시간(초)을 측정해 이동/애니메이션을 `속도 × dt`로 갱신하는 것. 프레임률이 달라도 게임 속도가 동일해진다.

**측정 수단 비교**:
- `GetTickCount()` — 해상도 ~15.6ms. 게임에 부적합.
- `timeGetTime()` — `timeBeginPeriod(1)`시 1ms. 구세대 게임 표준.
- `QueryPerformanceCounter/Frequency` — 마이크로초 이하. 현재 표준. **내 코드**: `CTimeMgr.cpp`가 생성 시 `QueryPerformanceFrequency(&m_liFreq)`를 캐싱하고 매 프레임 `QueryPerformanceCounter`로 dt를 계산한다 (`BeginFrame()`을 루프 선두에서 호출).

```cpp
dt = double(liNow.QuadPart - m_liPrev.QuadPart) / m_liFreq.QuadPart;
```

**고정 스텝 vs 가변 스텝**: 가변 스텝(dt 그대로)은 구현이 쉽지만 dt 폭주 시 터널링(벽 뚫기)·물리 불안정이 생긴다. 고정 스텝(예: 60Hz 어큐뮬레이터)은 시뮬레이션이 결정적이 되어 네트워크/리플레이에 유리하다. Winters 서버 시뮬레이션이 고정 틱 기반 서버 권위 구조인 이유와 그대로 연결된다. 최소 방어선: `dt = min(dt, 0.1)` 클램프 (디버거 브레이크 후 폭주 방지).

**함정 꼬리질문**: "QueryPerformanceCounter는 멀티코어에서 문제 없나요?" — 구형 하드웨어의 TSC 비동기 이슈가 있었으나 Vista 이후 OS가 invariant TSC/HPET로 보정한다. "그래도 음수 dt 방어는 넣는다"까지 말하면 좋다.

### 5. 유닛 드래그 선택 박스

**정의**: 마우스 눌림 지점→현재 지점의 사각형과 교차하는 아군 유닛을 선택 집합으로 만드는 RTS 기본 입력.

**동작 원리 (3단계)**:
1. `WM_LBUTTONDOWN`에서 시작점 저장, 드래그 상태 on → **내 코드**: `CSelectionMgr.h`의 `bool m_bDragging`.
2. 드래그 중 시작점/현재점으로 사각형을 만들되, 어느 방향으로 끌어도 성립하도록 **정규화** → `static RECT NormalizeRect(POINT a, POINT b)` (min/max로 left≤right, top≤bottom 보장). 이걸 빼먹으면 오른쪽 아래→왼쪽 위 드래그가 빈 사각형이 된다 — 실제로 처음 만들 때 누구나 한 번 밟는 함정.
3. `WM_LBUTTONUP`에서 확정: 카메라 스크롤이 있으므로 **스크린 좌표 사각형을 월드 좌표로 변환**(또는 유닛 월드 좌표를 스크린으로 변환)해 AABB 교차 검사. 클릭(드래그 거리 < 임계값)이면 단일 선택으로 분기, Shift면 집합에 추가.

**게임 맥락 꼬리질문**: "유닛 500마리면 매 프레임 전수 검사?" — 확정 시점 1회만 검사하면 전수도 충분하지만, 프레임마다 하이라이트하려면 공간 분할(타일 버킷)로 후보를 줄인다. "적군도 박스에 잡히나?" — 아군만 필터(내 코드의 `eTeamType`), 적은 단일 클릭만 허용이 스타크래프트 규칙.

### 6. 이동 명령과 타일 길찾기 (A*)

**정의**: 맵을 타일 그리드로 이산화하고, 시작 셀→목표 셀의 최소 비용 경로를 A*로 찾은 뒤 월드 좌표 웨이포인트로 되돌려 유닛을 따라가게 한다.

**내 코드 API** (`스타크래프트\Starcraft\Starcraft\CNavMgr.h`):

```cpp
void BuildFromTile();                                    // 타일 → walkable 그리드 구축
vector<Vec2> RequestPathWorld(Vec2& startW, Vec2& goalW);// 월드 좌표 in/out 상위 API
bool IsWalkable(int row, int col) const;
bool WorldToCell(Vec2 World, int& outR, int& outC);
bool SnapToNearestWalkable(int& ioR, int& ioC, int iMaxRadius); // 막힌 목표 보정
vector<int> AStarCells(int scrR, int scrC, int goalR, int goalC);
```

**A* 원리 (반드시 술술 나와야 함)**:
- 각 노드 비용 `f(n) = g(n) + h(n)`. g = 시작→n 실제 누적 비용, h = n→목표 추정(휴리스틱).
- **open list**(발견됐지만 미확정, f 최소를 꺼냄 — 보통 우선순위 큐/이진 힙)와 **closed**(확정) 관리. 꺼낸 노드의 이웃을 relax하며 부모 포인터 기록 → 목표 도달 시 부모 체인 역추적으로 경로 복원.
- 휴리스틱이 **admissible**(실제 잔여 비용을 절대 과대평가하지 않음)이면 최적 경로 보장. 4방향 그리드면 맨해튼 거리, 8방향이면 옥타일/체비셰프. h=0이면 다익스트라로 퇴화(느려지지만 최적), h 과대면 빨라지지만 최적성 상실(Weighted A* — 게임에선 종종 의도적으로 씀).

**전체 흐름으로 답하기**: 우클릭 → 스크린→월드 변환(카메라 오프셋) → `WorldToCell` → 목표가 막힌 셀이면 `SnapToNearestWalkable`로 반경 내 가장 가까운 통행 셀로 보정(스타에서 미네랄 뒤 클릭해도 유닛이 근처까지 가는 그 동작) → `AStarCells` → 셀 경로를 `CellToWorldCenter`로 웨이포인트화 → 유닛은 매 프레임 다음 웨이포인트로 `정규화(방향) × 속도 × dt` 이동, 도달 판정 반경 내면 다음 웨이포인트로.

**Winters 연결**: 같은 문제를 Winters에서는 서버 권위로 다시 풀었다 — `Engine/Private/Manager/Navigation/Pathfinder.cpp` + 클라 예측/보정, 그리고 "직접 걸어갈 수 있는 클릭은 raw 목표 유지, 막힌 목표만 pathfind"라는 정책을 gotchas로 박제했다 (`.claude/gotchas.md` 2026-05-20 Click movement 항목). "2D 타일 A*를 손으로 짜본 경험이 3D 냅그리드 디버깅(현재 셀/다음 웨이포인트/보정 방향 오버레이)으로 이어졌다"가 강한 어필.

**함정**: (a) 경로가 안 나올 때 무한 재요청 폭주 — 실패 캐싱/쿨다운 필요, (b) 유닛 간 겹침은 A*가 못 풀어서 별도 분리(steering/pushing)가 필요 — Winters에서 미니언 crowd collision을 Soft Separation으로 따로 계획했던 것과 동일 구조, (c) 다수 유닛이 동시에 요청 시 프레임 스파이크 — 요청 큐잉/분산 처리.

### 7. 매니저 아키텍처 — 싱글턴, 오브젝트 매니저, 씬, 팩토리

**내 구조**: 전 시스템이 `Get_Instance/Destroy_Instance` 싱글턴 매니저다 — `CObjMgr, CSceneMgr, CCollisionMgr, CCommandMgr, CCameraMgr, CCursorMgr, CAnimMgr, CBmpMgr, CTimeMgr, CNavMgr, CSelectionMgr` (+`CAbstractFactory.h`).

**CObjMgr 핵심** (`CObjMgr.h`):

```cpp
list<CObj*>   m_ObjList[OBJ_END];          // 논리 그룹별 (유닛/건물/탄환/UI...)
list<CObj*>   m_RenderList[RENDER_END];    // RENDER_BG, RENDER_WORLD, RENDER_UI, RENDER_UI_TOP
vector<CObj*> m_PendingDelete;             // 지연 삭제
```

- **렌더 순서**: 레이어 배열 순회 = painter's algorithm. 레이어 안에서는 `CObj::GetSortY()`(`return m_tRect.bottom`)로 **Y-sort** — 발밑 y가 큰(화면 아래) 오브젝트를 나중에 그려 2D 탑다운의 앞뒤 겹침을 해결한다. DX 없이 depth 개념을 손으로 구현한 것.
- **지연 삭제**: 순회 중 `delete`하면 이터레이터 무효화 + 같은 프레임 타 시스템의 댕글링 접근이 나므로, 죽음은 표시만 하고 프레임 끝에 `m_PendingDelete` 일괄 정리. 이 패턴은 Winters ECS의 지연 destroy와 정확히 같은 문제의식이다.
- **CObj 인터페이스**: `Initialize / Update / Late_Update / Render(HDC) / Release` 순수 가상 + `TakeDamage/Heal`. Update(의사결정·이동)와 Late_Update(카메라 반영, 상태 후처리)의 2단 분리는 엔진의 Update/LateUpdate 관례를 그대로 재현한 것.

**싱글턴 트레이드오프 (반드시 물어봄)**: 장점 — 전역 접근으로 배선 비용 제로, 소규모 1인 프로젝트에서 개발 속도 최대. 단점 — (a) 숨은 전역 상태라 의존 관계가 코드에 안 드러남, (b) 테스트에서 격리 불가, (c) 파괴 순서 제어 어려움, (d) 멀티스레드 시 lazy-init race. "당시엔 규모상 합리적이었지만, Winters에서는 매니저를 `CGameInstance` 게이트웨이 뒤로 숨기고 Client가 직접 접근 못 하게 배포 스크립트가 `*_Manager.h`를 SDK에서 제거하는 물리적 강제까지 넣었다"로 성장을 보여준다.

**추상 팩토리**: 유닛/건물 클래스가 수십 개(테란/프로토스/저그의 `CBattleCruiser, CCarrier, CDarkTemplar, CBarracks, ...`)라 생성을 `CAbstractFactory` 템플릿으로 일원화했다. "지금이라면?" — 클래스 폭발 자체를 데이터 주도로 없앤다. Winters가 실제로 그렇게 갔다: 챔피언은 클래스 상속이 아니라 `ChampionDef` 데이터 + 공용 시스템(`Client/Public/GameObject/ChampionDef.h`, `Shared/GameSim/Champions/*`) 구조다.

### 8. OOP GameObject vs ECS — 던전스에서 Winters로

**던전스 구조** (근거: `.md/TODO/05-12/INVENTORY_UI_DUNGEONS_TO_WINTERS_PLAN.md`): `CGameObject` 상속 트리, UI조차 GameObject로 렌더 큐에 들어가고, `CInventoryMgr` 같은 매니저가 UI 객체들을 직접 소유하며 `Update/LateUpdate/Render`를 수행한다. 데미지 폰트도 `CFontMgr`(DX9 `D3DXFONT`) + `CDamageMgr` 조합 (`.md/TODO/05-12/DAMAGE_FONT_DUNGEONS_COMPARISON_PLAN.md`).

**차이의 본질**:
- OOP: "객체가 자기 데이터와 행동을 소유" → 기능 추가 = 상속/멤버 추가. 직관적이지만 (a) 다중 역할에서 상속 다이아몬드, (b) 데이터가 객체별로 흩어져 캐시 비친화, (c) "누가 언제 뭘 바꾸는지"가 포인터 그물 속에 숨는다.
- ECS: 데이터(Component)와 로직(System)을 분리, 엔티티는 ID. 같은 컴포넌트끼리 연속 배열 → 캐시 친화 + 시스템 실행 순서가 곧 데이터 흐름이라 병렬화/네트워크 스냅샷 직렬화가 자연스럽다.

**실전 번역 경험**: 위 문서가 던전스 `CInventoryMgr` 하나가 하던 일을 Winters에서 "Shared 데이터 + CommandExecutor + UI draw layer" 3분할로 옮기는 계획이다. "던전스의 '객체 포인터 따라가기' 대신 '컴포넌트와 커맨드 흐름 따라가기'로 사고가 바뀌었다"(같은 문서 §8)가 면접용 핵심 문장.

### 9. TCP IOCP 서버 기초 (던전스 서버)

**실존 근거**: `SR_MinecraftDungeons\SR_Minecraft_Dungeons\Server\` — `CServer, CSession, CSessionMgr, CGameLoop, CStageMgr, IOCPDef.h`. 세션 메모리 기록에도 "이전 SR_MinecraftDungeons Server는 TCP IOCP, UDP는 Winters가 처음"으로 남아 있다.

**IOCP 한 단락 요약(면접용)**: Windows의 확장 가능한 비동기 I/O 모델. 소켓을 **완료 포트**에 연결하고 `WSARecv/WSASend`를 비동기로 걸면, 완료 시 OS가 완료 통지를 포트 큐에 넣고, 워커 스레드들이 `GetQueuedCompletionStatus`로 꺼내 처리한다. select/poll처럼 "준비됨"을 묻는 게 아니라 "완료됨"을 통지받는 **proactor** 모델이고, OS가 동시 실행 워커 수를 코어 수에 맞춰 조절해 컨텍스트 스위칭을 줄인다. 핵심 구성: 세션별 `OVERLAPPED` 확장 구조체로 어떤 I/O의 완료인지 식별, 수신 링버퍼에서 패킷 경계 조립(TCP는 스트림이라 메시지 경계가 없음 — length-prefix 프레이밍), 세션 수명은 참조 카운트로 보호(진행 중인 비동기 I/O가 있는 세션을 닫는 race가 최대 함정).

**Winters와의 대비**: 던전스는 TCP(신뢰성/순서 보장, 대신 head-of-line blocking), Winters는 실시간성이 우선이라 UDP + 자체 스냅샷/이벤트 복제 + FlatBuffers 검증 실패 시 bounded trace(`.claude/gotchas.md` 2026-07-09 Authoritative packets 항목)로 갔다. "왜 바꿨나"에 대한 답이 준비되어 있는 셈이다.

### 10. 출시 파이프라인 개념 (Liberation 방어용)

Liberation의 구체 사실이 확보되지 않았으므로, 아래는 어떤 출시 경험에도 통하는 표준 개념 — 본인 사례를 채워 넣는 뼈대다.

**빌드/QA 사이클**: 개발 빌드 → 데일리/CI 빌드(자동 빌드 + 스모크 테스트) → **코드 프리즈**(기능 동결, 버그 수정만) → RC(Release Candidate) → 골드. 버그는 심각도(크래시/진행불가/기능/외관) × 빈도로 우선순위를 매기고, 출시 게이트는 "P0/P1 제로"다. 프리즈 이후의 수정은 **수정 자체가 새 리스크**이므로 diff 최소화 + 회귀 범위 명시가 원칙.

**크래시 대응**: (a) 크래시 시 **minidump** 자동 수집(`SetUnhandledExceptionFilter` + `MiniDumpWriteDump`), (b) 빌드마다 **PDB 심볼 보관**(심볼 서버) — 심볼 없는 덤프는 주소 나열일 뿐, (c) 콜스택 시그니처로 **그룹핑**해 최다 발생부터, (d) 재현 스텝 확보 → 수정 → 해당 버전 이후 발생률로 검증. "출시 후 크래시율(crash-free sessions %)을 지표로 본다"까지 말하면 라이브 경험자 신호. (Winters 감사에서도 minidump 부재가 HIGH 항목이었다 — 자체 엔진에도 같은 체계가 필요함을 아는 것으로 연결.)

**스토어 심사**: 연령 등급(GRAC/IARC 등), 결제 정책(인앱 결제 수수료/외부 결제 금지 규정), 개인정보 처리방침 링크, 크래시/블로킹 버그, 메타데이터(스크린샷/설명) 일치 여부가 단골 리젝 사유. 리젝 → 수정 → 재제출 사이클이 며칠씩 걸리므로 출시일에 심사 버퍼를 넣는 게 일정 관리의 핵심.

**라이브 대응**: 장애 시 판단 순서 — (1) 영향 범위 파악(전체/일부, 데이터 훼손 여부), (2) **롤백 가능하면 롤백이 우선**(원인 규명보다 서비스 복구 먼저), (3) 데이터 마이그레이션이 껴서 롤백 불가면 핫픽스, (4) 사후 포스트모템으로 재발 방지를 프로세스에 박제. 지표는 DAU/리텐션/크래시율/서버 에러율. "박제" 습관은 Winters `.claude/gotchas.md` 운영으로 실증 가능.

### 11. 협업 프로세스 개념

**브랜치 전략**: Git Flow(develop/release/hotfix 브랜치 — 릴리즈 주기가 긴 팀), GitHub Flow(main + 단명 feature 브랜치 + PR — 소규모/빠른 배포), trunk-based(main 직커밋 + 피처 플래그 — CI 성숙한 팀). 게임팀 특수성: **바이너리 에셋은 merge 불가**라 락킹(Perforce 스타일) 또는 Git LFS + 소유권 규약이 필요하다. Winters에서 실제로 운영한 것: `.md/collab/GIT_SYNC_RULES.md`(브랜치 prefix `codex/<area>-<task>`, push 전 fetch→rebase→검증 하네스→diff --check 순서, dirty pull 금지), `.md/collab/OWNERSHIP_MATRIX.md`(영역별 owner + 충돌 위험 파일 Always-Lock), `.md/collab/ACTIVE_WORK_PACKETS.md`(작업 단위 상태 추적).

**코드리뷰**: 목적은 결함 발견 + 지식 공유 + 소유권 분산. 좋은 리뷰의 조건 — 작은 diff(400줄 이하), 리뷰어가 검증 가능한 설명(무엇을/왜/어떻게 검증했는지), 사람이 아니라 코드에 대한 코멘트. Winters에서는 작성자(Claude)와 리뷰어(Codex)를 **다른 모델로 분리**해 교차 리뷰를 돌렸고, 대표 사례로 병렬 미니언 AI 설계를 4회 리뷰에서 20건(entity 간 write race, Decision/Apply 2-pass 필요성 등) 지적받아 전수 반영했다 (`.md/interview/engine/13_collaboration_process.md` 결정 6). 리뷰 지적이 충돌하면 최종 판정은 코드를 직접 읽고 내가 내린다 — 이게 리뷰 문화 질문의 킬러 답변.

**STAR 기법**: 행동면접 답변 구조. **S**ituation(상황 1~2문장) → **T**ask(내 책임/목표 1문장) → **A**ction(**내가** 한 행동 3~4문장, 구체 기술/수치) → **R**esult(결과+수치+배운 것 1~2문장). 함정: A에서 "우리 팀이~"로 뭉개기(내 기여가 안 보임), R에서 교훈 생략(성장 신호 상실).

**갈등 해결 프레임**: (1) 감정과 사실 분리 — "사실-영향-요청" 화법, (2) 의견 충돌은 **데이터/실험으로 환원**("둘 다 프로토타입 만들어 프로파일러로 재자"), (3) 결정 후에는 반대했더라도 실행에 커밋(disagree and commit), (4) 사람 문제가 아니라 프로세스 문제로 치환해 재발 방지를 구조화. Winters의 "실수 → gotchas 한 줄 박제 → 반복되면 빌드 게이트/훅으로 기계 강제" 에스컬레이션이 (4)의 실물 증거다.

### Q. 지금까지 해온 프로젝트를 간단히 소개해 주세요.

**전략**: 시간순 나열이 아니라 **성장 서사**로 60~90초. 각 프로젝트당 "무엇을 + 내가 맡은 것 + 거기서 얻은 것" 한 호흡씩, 마지막에 Winters로 수렴.

**모범 골격**:
"WinAPI와 GDI만으로 스타크래프트를 모작하면서 게임 루프, 더블 버퍼링, 타일 A* 길찾기 같은 바닥 기본기를 익혔습니다. 이후 팀 프로젝트로 마인크래프트 던전스를 DX9 클라이언트 + TCP IOCP 서버 구조로 만들며 처음으로 협업과 네트워크 동기화를 경험했고 [CONFIRM_NEEDED: 던전스 팀 규모/기간/본인 역할 한 줄], Liberation이라는 게임을 실제로 출시하면서 QA 사이클과 라이브 대응의 무게를 배웠습니다 [CONFIRM_NEEDED: Liberation 장르/플랫폼/역할 한 줄]. 지금은 그 경험을 통합해 자체 DX11 엔진 Winters 위에 서버 권위 방식의 LoL 클론을 만들고 있습니다 — 클라 예측/스냅샷 보간, ECS, 자체 에셋 포맷과 툴까지 포함합니다."

**꼬리질문 대비**: 어떤 단어를 물고 늘어져도 3단계 방어가 되는 키워드만 소개에 넣는다. 설명 못 할 키워드는 빼는 게 이득.

### Q. 그중 기술적으로 가장 어려웠던 프로젝트는 무엇이었나요?

**전략**: "가장 어려웠다"는 곧 "가장 깊게 설명할 수 있다"여야 한다. 안전한 선택은 Winters(현재 진행형이라 디테일이 살아 있음) 또는 던전스(협업+네트워크 복합).

**모범 골격**: 문제를 하나로 좁혀 말한다. 예: "서버 권위 이동에서 클라 예측과 서버 스냅샷이 싸우는 문제였습니다. 클라 내비게이션 시스템이 스냅샷으로 갱신된 yaw를 매 프레임 덮어써서 캐릭터가 순간적으로 떨렸는데, 원인 분리를 위해 현재 셀/다음 웨이포인트/보정 방향을 화면 오버레이로 계측한 뒤에야 '복제된 챔피언에는 로컬 이동 시스템을 아예 돌리면 안 된다'는 구조적 결론에 도달했습니다. 증상 튜닝이 아니라 권위 소유권을 정리한 거죠." (근거: `.claude/gotchas.md` 2026-05-22 Network movement 항목)

**함정**: 어려움을 나열만 하고 해결 과정(계측→가설→검증)이 없으면 감점. "추측이 2회 빗나가면 즉시 계측으로 전환한다"는 원칙까지 말하면 시니어 신호.

### Q. 왜 Unity/Unreal 같은 상용 엔진을 안 쓰고 밑바닥부터 만들었나요?

**모범 골격**: (1) 목적이 달랐다 — "게임을 빨리 내는 것"이 아니라 "엔진과 서버가 어떻게 동작하는지 층층이 이해하는 것"이 목표였다. (2) 결과적으로 상용 엔진을 더 잘 쓰게 됐다 — 렌더 파이프라인/잡시스템/네트워크 복제를 직접 구현해봤기 때문에 Unreal의 같은 기능이 왜 그렇게 설계됐는지 읽힌다. (3) 균형 감각 — "회사에서는 검증된 엔진과 팀 컨벤션을 따르는 것이 맞고, 자체 엔진 경험은 그 아래층이 보이는 눈으로 기여하겠다"로 마무리. **함정**: 상용 엔진 폄하는 절대 금지 — NIH(Not Invented Here) 성향으로 읽히면 치명적.

### Q. 본인이 쓴 코드 중 하나만 화이트보드에 설명한다면 무엇을 고르겠어요?

**전략**: 그림으로 그릴 수 있고, 트레이드오프가 있고, 버그 스토리가 딸린 코드. 추천 1순위: 스타 모작의 **A* + SnapToNearestWalkable** (그리드 그림 + open/closed + 막힌 목표 보정까지 5분 완결). 추천 2순위: 던전스→Winters의 **지연 삭제 패턴**(순회 중 삭제 문제는 화이트보드로 보여주기 최적). 미리 한 번 손으로 그려보고 갈 것.

### 3-B. 스타크래프트 WinAPI 모작

### Q. WinAPI 게임의 메인 루프 구조를 설명해 주세요. GetMessage와 PeekMessage의 차이는?

**답**: 핵심 개념 1 그대로. "GetMessage는 큐가 비면 블로킹이라 이벤트 구동 앱용, 게임은 입력이 없어도 매 프레임 시뮬레이션을 돌려야 해서 PeekMessage로 논블로킹 폴링을 합니다. 제 코드는 `while (msg.message != WM_QUIT)` 루프에서 메시지가 있으면 Translate/Dispatch, 없으면 `CTimeMgr::BeginFrame()`으로 dt를 갱신하고 Update → Late_Update → Render를 돌립니다." **꼬리 대비**: "WM_QUIT은 어디서 오나?"(WM_DESTROY→PostQuitMessage), "PeekMessage를 if로 한 번만 처리하면 문제 없나?"(입력 폭주 시 지연 — while로 큐 소진 후 틱이 개선안), "CPU 100% 아닌가?"(맞다 — 프레임 리미터나 DwmFlush/Sleep(0) 보완 여지 인정).

### Q. WndProc은 누가, 언제 호출하나요?

**답**: "제가 아니라 OS가 호출합니다. RegisterClassEx에 함수 포인터로 등록하고, 메시지 루프의 DispatchMessage가 user32를 통해 해당 윈도우의 WndProc을 콜백합니다. SendMessage 계열은 큐를 안 거치고 직접 호출되기도 합니다. 처리 안 하는 메시지는 DefWindowProc으로 넘겨야 창의 기본 동작이 유지됩니다." **연결**: "Winters에서도 같은 자리에 ImGui 입력 후킹(`ImGui_ImplWin32_WndProcHandler`)을 체인 앞에 끼워, UI가 입력을 소비하면 게임 입력으로 안 내려가게 했습니다." **꼬리 대비**: "키 '눌려 있음' 상태는 WndProc만으로 불편하지 않나?" → 에지(눌림 순간)는 메시지, 상태(held)는 GetAsyncKeyState 폴링 병행.

### Q. 더블 버퍼링은 왜 필요하고 어떻게 구현했나요?

**답**: 핵심 개념 3. "화면 DC에 직접 지우고 그리면 지워진 중간 프레임이 노출되어 깜빡입니다. CreateCompatibleDC + CreateCompatibleBitmap으로 백버퍼를 만들어 완성 화면을 구성한 뒤 BitBlt 한 번으로 전송했습니다. 스프라이트 투명은 SRCAND(마스크로 구멍) + SRCPAINT(OR로 얹기) 비트 연산 조합이고, 팀 컬러 틴트도 같은 방식의 중간 DC 합성으로 처리했습니다(`CBmpMgr.cpp`)." **꼬리 대비**: "GDI 핸들 누수는?"(로딩 시 생성·캐싱, 루프 내 Create 금지, SelectObject 원복), "AlphaBlend와 차이는?"(마스킹은 1비트 투명, AlphaBlend는 per-pixel 알파), "DX라면?"(스왑체인 백버퍼+Present가 이 계층 — Winters에서 직접 관리).

### Q. deltaTime은 어떻게 측정했고, 왜 그 방법을 썼나요?

**답**: 핵심 개념 4. "QueryPerformanceCounter/Frequency 조합입니다. GetTickCount는 15.6ms 해상도라 60fps(16.6ms) 게임에서는 dt가 0 아니면 프레임 전체가 튀는 수준이고, timeGetTime은 1ms까지 가능하지만 QPC가 마이크로초 이하로 가장 정밀합니다. Frequency는 부팅 후 불변이라 생성자에서 한 번만 캐싱했습니다(`CTimeMgr.cpp`)." **꼬리 대비**: "고정 스텝 vs 가변 스텝?" — 가변은 간단하지만 물리/판정이 프레임률에 종속, 고정은 결정적이라 네트워크/리플레이에 필수. "Winters 서버가 고정 틱 시뮬레이션인 이유이기도 합니다." dt 클램프(브레이크포인트 복귀 시 폭주 방지)도 언급.

### Q. 유닛 드래그 선택은 어떻게 구현했나요?

**답**: 핵심 개념 5. LBUTTONDOWN에서 시작점 + `m_bDragging`, 드래그 중 `NormalizeRect(POINT a, POINT b)`로 방향 무관 정규화, LBUTTONUP에서 스크린→월드 변환 후 아군 유닛 AABB 교차로 선택 집합 확정 (`CSelectionMgr.h`). "정규화를 빼먹으면 왼쪽 위로 끄는 드래그가 빈 사각형이 되는데, 실제로 처음에 그 버그를 만들고 나서 min/max 정규화를 static 유틸로 분리했습니다." **꼬리 대비**: 카메라 스크롤 중 드래그(시작점도 월드 기준으로 저장해야 함), 클릭 vs 드래그 판별(이동 거리 임계값), 대규모 유닛 최적화(타일 버킷).

### Q. 우클릭으로 이동 명령을 내리면 유닛이 도착할 때까지, 코드 흐름을 처음부터 끝까지 설명해 보세요.

**답 (이 카테고리의 하이라이트 — 통째로 암기 수준으로)**: "① WndProc에서 우클릭 수신, 스크린 좌표에 카메라 오프셋을 더해 월드 좌표로. ② `CNavMgr::WorldToCell`로 목표 셀 산출. ③ 목표가 통행 불가 셀이면 `SnapToNearestWalkable`이 반경을 넓혀가며 가장 가까운 통행 셀로 보정 — 미네랄 뒤를 찍어도 유닛이 근처까지 가는 동작입니다. ④ `AStarCells`가 f=g+h 기반 A*로 셀 경로를 찾고, ⑤ 셀 중심 좌표 배열(웨이포인트)로 변환해 유닛에 전달. ⑥ 유닛은 매 프레임 다음 웨이포인트 방향 단위벡터 × 속도 × dt로 이동하고, 도달 반경 안이면 다음 웨이포인트로 넘어갑니다." **연결**: "Winters에서는 같은 문제를 서버 권위로 다시 풀면서 '직접 걸어갈 수 있는 클릭은 pathfind 없이 raw 목표 유지, 막힌 클릭만 pathfind'라는 정책 차이까지 다뤘습니다." **꼬리 대비**: 다음 질문들이 그대로 이어진다.

### Q. A*에서 휴리스틱이 실제 거리보다 크면 어떻게 되나요?

**답**: "admissible 조건(과대평가 금지)이 깨져서 **최적 경로 보장이 사라집니다**. 대신 목표 방향으로 탐색이 쏠려 확장 노드 수가 줄어 빨라지죠. 그래서 게임에서는 h에 가중치를 곱한 Weighted A*를 의도적으로 쓰기도 합니다 — 최단보다 '그럴듯하고 빠른' 경로가 나은 경우가 많으니까요. h=0이면 다익스트라로 퇴화합니다." **꼬리 대비**: "consistent(단조) 조건은?" — h(n) ≤ cost(n,n') + h(n')이면 closed 재방문이 불필요해진다. "open list 자료구조는?" — 이진 힙(우선순위 큐)이 표준, 소규모 그리드면 배열 선형 탐색도 실용적.

### Q. 목표 지점이 벽 안이면 어떻게 처리했나요?

**답**: "`SnapToNearestWalkable(ioR, ioC, iMaxRadius)`로 목표 셀부터 반경을 키워가며 나선/링 탐색해 가장 가까운 통행 셀로 스냅했습니다. 이걸 안 하면 A*가 전 맵을 다 뒤지고 실패하는 최악 케이스가 됩니다 — 도달 불가 목표는 A*의 최악 성능 시나리오라, 탐색 전에 목표를 유효화하는 게 성능 방어이기도 합니다." **꼬리 대비**: "그래도 경로가 없으면?"(실패 반환 → 상위에서 명령 무시 or 최근접 도달 — 무한 재요청 폭주 방지 쿨다운), "여러 유닛이 같은 목표로 몰리면?"(도착 지점 분산/포메이션 — A* 밖의 문제임을 인지하고 있음을 보여줄 것).

### Q. 오브젝트 매니저는 어떻게 설계했나요? 렌더 순서와 오브젝트 삭제는?

**답**: 핵심 개념 7. "논리 그룹 `m_ObjList[OBJ_END]`와 렌더 레이어 `m_RenderList[RENDER_END]`(BG→WORLD→UI→UI_TOP)를 분리했고, WORLD 레이어 안에서는 `GetSortY()` — 스프라이트 발밑 bottom 값 — 로 Y-sort해서 탑다운 앞뒤 겹침을 해결했습니다. 삭제는 순회 중 delete가 이터레이터 무효화와 댕글링을 만들기 때문에 `m_PendingDelete`에 모아 프레임 끝에 일괄 처리했습니다." **연결**: "이 지연 삭제 문제의식은 Winters ECS의 지연 destroy, 그리고 던전스에서 겪은 죽은 객체 참조 크래시와 하나로 이어집니다." **꼬리 대비**: "탄환이 대상 유닛 포인터를 들고 있는데 대상이 죽으면?" — 포인터 대신 ID/핸들 + 유효성 조회가 정답. 당시는 포인터+생존 플래그였고, Winters에서 엔티티 핸들 방식으로 교정했다고 답한다.

### Q. 매니저를 전부 싱글턴으로 했는데, 문제점은 없었나요?

**답**: "규모(1인, 소스 294파일)에서는 배선 비용 제로라는 실익이 컸지만, 단점을 알고 씁니다 — 숨은 전역 상태라 의존이 코드에 안 드러나고, 테스트 격리가 안 되고, Destroy 순서를 손으로 관리해야 하고, 멀티스레드면 lazy-init race가 있습니다." **연결**: "그래서 Winters에서는 매니저 직접 접근을 금지하고 `CGameInstance` 게이트웨이로만 열었으며, 배포 스크립트가 `*_Manager.h`를 SDK에서 물리적으로 제거해 규칙을 빌드 수준에서 강제합니다. '규칙은 문서가 아니라 기계가 강제해야 유지된다'는 걸 이 프로젝트들을 거치며 배웠습니다." — 이 답은 협업 질문으로 이어지는 다리가 된다.

### Q. 유닛 클래스가 종족별로 수십 개던데, 클래스 폭발은 어떻게 관리했나요? 지금이라면 어떻게 하겠어요?

**답**: "당시엔 `CObj → CUnit/CBuilding → 개별 유닛` 상속과 `CAbstractFactory` 템플릿으로 생성만 일원화했습니다. 클래스는 많지만 생성/등록 경로가 한 곳이라 관리는 됐죠. 지금이라면 클래스 폭발 자체를 없앱니다 — 유닛을 '데이터(스탯/애님/스킬 셋) + 공용 시스템'으로 정의하는 데이터 주도 설계로요. 실제로 Winters의 챔피언 구조가 그렇게 되어 있습니다: 챔피언별 클래스 상속이 아니라 `ChampionDef` 데이터와 GameSim 공용 로직 조합입니다. 150종으로 스케일해야 한다는 목표를 세우니 설계가 강제로 바뀌더군요." **꼬리 대비**: "데이터 주도의 단점은?" — 디버깅 시 흐름 추적이 어려워지고 데이터 검증 계층이 필수 — 툴/에디터가 함께 가야 한다.

### Q. 그 프로젝트에서 가장 오래 잡은 버그는 뭐였나요?

**답**: [CONFIRM_NEEDED: 스타 모작에서 실제로 가장 오래 걸린 버그 — 예: 드래그 정규화, 죽은 유닛 참조 크래시, 경로 재요청 폭주 중 실제 겪은 것]. 없으면 이 프레임으로: "증상 → 첫 가설(틀림) → 계측 추가 → 진짜 원인 → 수정 + 재발 방지"의 5단으로 어떤 버그든 서술. **주의**: 지어내지 말 것 — 꼬리질문 한 번에 무너진다. 실제 기억이 없으면 Winters의 버그(예: 미니언 stuck — 시스템 실행 순서 swap으로 stale velocity 제거)로 대체하고 "스타 모작 시절엔 이런 계측 습관이 없어서 오래 헤맸다"로 성장 서사화.

### Q. WinAPI 프로젝트가 지금 회사 업무와 무슨 상관이 있죠?

**답**: "세 가지입니다. ① 게임 루프/입력/타이밍은 엔진이 바뀌어도 동일한 계층이고, 저는 그걸 추상화 없이 만져봤습니다. ② 메시지 펌프와 윈도우 관리는 지금도 모든 PC 게임 클라이언트의 바닥입니다 — Winters의 DX11 윈도우/입력 계층을 짤 때 그대로 재사용된 지식입니다. ③ '엔진이 해주던 일'의 원가를 압니다. 프레임 하나가 공짜가 아니라는 감각은 최적화 업무의 출발점입니다." **함정**: 방어적으로 답하지 말 것. 이 질문은 공격이 아니라 연결 능력 테스트다.

### 3-C. 마인크래프트 던전스 — 팀 협업

### Q. 던전스 프로젝트를 소개해 주세요. 팀에서 어떤 역할이었나요?

**모범 골격**: "마인크래프트 던전스를 모작한 팀 프로젝트로, DX9 기반 OOP 클라이언트와 TCP IOCP 서버, 그리고 별도 UI 에디터 툴까지 갖춘 구조였습니다(Base/Client/Engine/Server/Shared 계층 분리). 저는 [CONFIRM_NEEDED: 던전스에서 본인 담당 시스템 — 예: 특정 보스/UI/서버 세션/렌더링 중 실제 맡은 것]을 맡았고, 팀은 [CONFIRM_NEEDED: 던전스 팀 인원 수와 역할 분담], 기간은 [CONFIRM_NEEDED: 던전스 개발 기간]이었습니다."

**실존 디테일(채우기 전에도 말할 수 있는 것)**: 클라이언트 코드에 팀원별 이니셜 프리픽스 흔적이 남아 있다 — `Client/Code/`의 `CAG*`(AncientGuardian 계열), `CCY*`(CCYCamera/CCYGuardian/CCYMonster) 같은 파일군. "담당자별 프리픽스로 파일 소유권을 구분하는 소박한 협업 규약이 있었다"는 사실 자체가 협업 스토리 소재다. **꼬리 대비**: "그 규약의 한계는?" — 소유권은 보이지만 인터페이스 경계가 없어서 서로의 매니저를 직접 호출하는 결합이 생겼다 → Winters에서 Ownership Matrix + 게이트웨이 접근으로 교정.

### Q. 팀원들과 코드 통합은 어떻게 했나요? 충돌이 났을 때는?

**답**: [CONFIRM_NEEDED: 던전스 당시 실제 버전 관리 방식 — Git 사용 여부/브랜치 규칙/USB·수동 병합 여부]. 사실이 확인되기 전 안전한 프레임: "당시의 통합 방식에는 한계가 있었고, 그 경험 때문에 지금은 통합 규칙을 문서가 아니라 절차로 강제합니다 — Winters에서는 push 전 fetch→rebase→검증 하네스(빌드+스모크)→diff 확인 순서를 `.md/collab/GIT_SYNC_RULES.md`로 못 박고, 충돌 위험이 높은 파일(씬 루트, vcxproj, SDK 헤더)은 Always-Lock으로 지정해 한 작업 단위에서만 수정합니다(`.md/collab/OWNERSHIP_MATRIX.md`). 병합 사고는 사람이 조심해서가 아니라 규칙이 좁혀줘야 안 난다는 게 결론입니다."

### Q. 던전스 서버는 어떤 구조였나요? 왜 IOCP였죠?

**답**: 핵심 개념 9. "TCP IOCP 서버로, `CServer/CSession/CSessionMgr` 세션 계층과 `CGameLoop/CStageMgr` 게임 루프 계층으로 나뉘어 있었습니다. IOCP는 Windows에서 소수의 워커 스레드로 다수 소켓의 완료 통지를 처리하는 proactor 모델이라, 접속자당 스레드를 만드는 방식보다 확장성이 좋습니다." **꼬리 대비 3연타**: (a) "TCP인데 패킷 경계는?" — 스트림이라 length-prefix로 프레이밍해 링버퍼에서 조립. (b) "세션이 끊기는 순간 진행 중인 WSARecv는?" — 완료 통지가 에러로 돌아오므로 세션 수명을 참조 카운트로 보호해야 댕글링이 없다. (c) "왜 Winters는 UDP로 갔나?" — 실시간 액션은 오래된 데이터의 재전송 대기(head-of-line blocking)가 손해라, 스냅샷 재전송 없이 최신 상태 덮어쓰기가 맞다. 오래된 프로젝트의 기술 선택을 현재 기준으로 재평가할 수 있음을 보여주는 최고의 지점.

### Q. 던전스의 OOP 구조를 지금 다시 만든다면 무엇을 바꾸겠어요?

**답**: "두 가지입니다. ① UI까지 GameObject 상속으로 렌더 큐에 넣고 `CInventoryMgr` 같은 매니저가 UI 객체를 직접 소유하던 구조를, 데이터(인벤토리 상태)/명령(장착·구매)/그리기(draw layer) 3분할로 바꿉니다 — 실제로 Winters로 이 번역을 설계한 문서가 있습니다(`.md/TODO/05-12/INVENTORY_UI_DUNGEONS_TO_WINTERS_PLAN.md`). ② 게임플레이 로직과 렌더/플랫폼 계층의 경계를 컴파일 수준에서 강제합니다 — 던전스는 경계가 관례뿐이라 서로 침범했는데, Winters에서는 Shared 시뮬레이션이 Engine 헤더를 include하면 PreBuild lint(`Tools/Harness/Check-SharedBoundary.ps1`)가 빌드를 실패시킵니다." **왜 강한가**: 과거 프로젝트 비판 + 현재 실행 증거가 한 답에 다 들어간다.

### Q. 팀원과 기술적 의견이 갈렸던 경험을 말해 주세요.

**답**: [CONFIRM_NEEDED: 던전스(또는 다른 팀 경험)에서 실제 의견 충돌 사례 — 무엇을 두고, 누구와, 어떻게 결론냈는지]. STAR 골격 제안 — S: 시스템 경계/구현 방식을 두고 팀원과 이견. T: 일정 안에 결론을 내야 하는 상황. A: 감정 아닌 기준 제시("성능이 걱정이면 둘 다 간단 프로토타입 만들어 재보자" 또는 "게임 규칙상 어느 쪽이 수정에 강한가"), 작은 실험으로 환원, 결정 후 커밋. R: 결과 + "결정보다 결정 방식이 팀 자산으로 남았다". **대체 실화(팀원이 사람이 아니어도 유효)**: "Winters에서 작성 모델과 리뷰 모델(Codex)의 분석이 3회 연속 어긋난 적이 있는데, 논쟁을 끊고 프로파일러 카운터 5분 계측으로 판정했습니다. 이후 '추측 2회 실패 시 즉시 계측'을 팀 규칙으로 박제했습니다."

### Q. 던전스 경험이 이후 프로젝트에 어떻게 이어졌나요?

**답**: "세 개의 직선이 있습니다. ① 네트워크: 던전스 TCP IOCP → Winters UDP 서버 권위 + 클라 예측. TCP의 순서 보장이 실시간 액션에서 왜 병목인지 몸으로 겪고 전환했습니다. ② 아키텍처: OOP 매니저 소유 구조 → ECS + 명령 흐름. 던전스 구조를 Winters로 번역한 설계 문서를 직접 썼습니다. ③ 협업: 이니셜 프리픽스 수준의 소유권 규약 → Ownership Matrix/Work Packet/기계 강제 게이트. 프로젝트마다 이전 프로젝트의 한계 하나씩을 구조적으로 갚아왔습니다."

### 3-D. Liberation — 출시 경험

> **주의**: 이 절의 프로젝트 사실은 레포/메모리에서 전혀 확인되지 않았다. 아래는 질문 프레임 + 표준 모범답변 골격이며, 슬롯을 채우기 전에는 구체 수치를 말하지 말 것.

### Q. Liberation은 어떤 게임이고, 본인은 무엇을 했나요?

**골격**: "[CONFIRM_NEEDED: Liberation 장르/플랫폼(모바일·PC·콘솔)/엔진·기술 스택] 게임으로 [CONFIRM_NEEDED: 출시 스토어와 출시 시기]에 출시했습니다. 팀은 [CONFIRM_NEEDED: Liberation 팀 규모/직군 구성]이었고 저는 [CONFIRM_NEEDED: Liberation에서 본인 역할 — 클라/서버/툴/전반]을 맡았습니다. 출시 후 [CONFIRM_NEEDED: 다운로드/유저 반응/운영 기간 등 결과 지표 한 개]." **전략**: 규모가 작아도 절대 축소하지 말 것 — 면접관이 원하는 건 규모가 아니라 "출시라는 마감을 통과해본 사람의 태도"다.

### Q. 출시 직전 마지막 2주는 어떻게 보냈나요?

**골격**: 핵심 개념 10의 어휘로 답한다. "코드 프리즈 후에는 기능 추가를 멈추고 버그 수정만 받았고, 수정 하나하나가 새 리스크라 diff를 최소화하며 회귀 범위를 명시했습니다. 버그는 심각도×빈도로 트리아지해서 '크래시/진행불가 제로'를 출시 게이트로 삼았습니다. [CONFIRM_NEEDED: Liberation 출시 직전 실제 있었던 마지막 이슈와 그 처리 — 이 한 방이 답변의 진짜 무게를 만든다]." **꼬리 대비**: "프리즈 중 기획이 '작은 기능 하나만'을 요구하면?" — 룰은 지키되 결정권자에게 리스크를 수치로 보여 판단을 위임한다(내가 몰래 넣지도, 감정적으로 거절하지도 않는다).

### Q. 출시 후 크래시 제보가 들어오면 어떻게 대응하나요? 실제로 겪었나요?

**골격**: "우선순위는 발생량 기준입니다. 크래시를 콜스택 시그니처로 그룹핑해 최다 발생부터 잡습니다. 재현 스텝이 없으면 덤프가 전부라서, 빌드마다 심볼(PDB)을 보관하고 minidump를 수집하는 체계가 전제 조건입니다. [CONFIRM_NEEDED: Liberation에서 실제 크래시/장애 대응 사례 — 증상/원인/수정/재발 방지]." **연결(사실이 없어도 성립)**: "이 경험 때문에 자체 엔진 Winters를 감사할 때도 minidump 파이프라인 부재를 최우선 리스크로 분류했습니다 — 출시를 해보면 '잡을 수 없는 크래시'가 최악이라는 걸 알게 됩니다." **꼬리 대비**: "Release 빌드에서만 나는 크래시는?" — 최적화로 초기화 누락/UB가 드러나는 전형 — 심볼 있는 Release(=RelWithDebInfo류) 빌드로 재현.

### Q. 스토어 심사에서 리젝당한 경험이 있나요?

**골격**: "[CONFIRM_NEEDED: 스토어 심사 리젝 여부와 사유 — 등급/결제/개인정보/버그 중 무엇이었는지]." 없거나 기억이 흐리면: "심사에서 배운 건 심사 기간이 일정의 변수라는 점입니다. 리젝 한 번에 며칠이 사라지므로 출시일 역산 시 심사 버퍼를 넣고, 등급/결제/개인정보 요건은 개발 중반부터 체크리스트로 관리해야 한다는 것이 결론이었습니다." **함정**: 심사 규정을 아는 척 세부 수치를 지어내지 말 것 — 스토어별 규정은 자주 바뀌어서 면접관이 더 잘 안다.

### Q. 라이브 중 서버(또는 게임)가 죽으면, 핫픽스와 롤백 중 무엇을 먼저 고려하나요?

**골격**: "복구가 원인 규명보다 먼저입니다. 판단 순서는 ① 영향 범위(전체/일부/데이터 훼손 여부) ② **롤백 가능하면 롤백** — 검증된 이전 상태로 돌리는 것이 가장 리스크가 낮습니다 ③ 데이터 마이그레이션이 껴서 롤백 불가면 최소 diff 핫픽스 ④ 복구 후 포스트모템으로 재발 방지를 프로세스에 박제. [CONFIRM_NEEDED: Liberation 라이브 중 실제 장애/점검 사례]." **연결**: "'박제'는 제 실제 습관입니다 — Winters에서 반복 실수를 `.claude/gotchas.md`에 날짜-영역-실수-방지책 한 줄 포맷으로 27건 이상 쌓았고, 반복되는 항목은 빌드 게이트로 승격시켰습니다."

### Q. 출시 경험에서 배운 가장 큰 것 하나만 꼽는다면?

**골격 (사실 무관하게 준비 가능한 안전 답변)**: "'완성'의 정의가 바뀌었다는 것입니다. 개발자의 완성은 '기능이 돌아간다'지만, 출시의 완성은 '모르는 사람이, 지원되지 않는 기기에서, 이상한 순서로 눌러도 무너지지 않는다'입니다. 그 차이가 방어 코드, 로깅, 크래시 수집, QA 사이클의 존재 이유라는 걸 머리가 아니라 일정으로 배웠습니다. [CONFIRM_NEEDED: Liberation에서 이 교훈을 상징하는 구체 에피소드 하나]."

### 3-E. 인성/컬처핏

### Q. 왜 게임 프로그래머가 되려고 하나요?

**전략**: "게임을 좋아해서"로 시작하면 평범. **만드는 행위의 어떤 지점**이 좋은지로 구체화한다.

**모범 골격**: "게임은 제가 아는 소프트웨어 중 가장 넓은 스택을 한 프레임 안에 요구합니다 — 16ms 안에 입력, 시뮬레이션, 네트워크, 렌더링이 전부 돌아야 하죠. 저는 그 제약이 주는 엔지니어링 밀도가 좋아서, WinAPI 모작부터 자체 DX11 엔진까지 같은 게임(RTS/MOBA)을 층을 바꿔가며 다시 만들어왔습니다. 취미가 아니라 커리어로 선택한 이유는, 이 일은 잘하는 것과 좋아하는 것이 제게 일치하는 유일한 분야이기 때문입니다." **주의**: 장기 목표(창업)는 먼저 꺼내지 말 것 — "몇 년 다니다 나갈 사람" 신호가 된다. 물으면 "언젠가 내 게임/팀을 이끌고 싶고, 그러려면 출시와 라이브를 여러 번 통과한 엔지니어가 먼저 되어야 한다"로 회사 기여와 정렬시켜 답한다.

### Q. 왜 우리 회사인가요?

**전략**: 회사별 커스터마이즈가 필수 — 범용 답변은 즉시 들킨다. 준비 공식: **(회사의 구체적 기술/게임 하나) × (내 경험 자산 하나) = 기여 문장**.

**골격 예시**: "이 회사의 [게임/기술 — 예: 대규모 동시접속 액션, 자체 엔진, 라이브 운영 역량]에 관심이 있습니다. 저는 서버 권위 이동/스킬 동기화를 자체 엔진에서 바닥부터 구현해봤고(Winters), TCP와 UDP 기반 게임 서버를 모두 만들어봤기 때문에, [해당 팀]에서 [구체 기여 — 예: 네트워크 동기화 품질, 툴링, 성능]에 바로 기여할 수 있다고 판단했습니다." **면접 전날 할 일**: 지원 회사의 대표작 1개를 플레이하고, 기술 블로그/발표 1개를 읽고, 그 안의 키워드 하나를 답변에 넣는다.

### Q. 가장 크게 실패한 경험은 무엇인가요?

**전략**: 실패 질문의 채점 기준은 실패의 크기가 아니라 **회복 절차의 품질**이다. 사소한 실패("시험을 못 봤다")도, 남 탓 실패도 감점.

**모범 골격 (실증 가능한 실화)**: "병렬 잡시스템을 도입하면서 워커 스레드의 데이터 race를 안일하게 봤다가, 미니언 AI 전체가 간헐적으로 멈추는 사고를 냈습니다. 첫 대응이 더 문제였는데 — 코드를 읽으며 추측만 세 번 반복했고 전부 빗나갔습니다. 결국 프로파일러 카운터를 심어 5분 만에 원인(시스템 실행 순서로 인한 stale 데이터)을 특정했죠. 이 실패에서 두 가지를 박제했습니다: '의사결정과 적용을 2-pass로 분리하는 설계 원칙'과 '추측이 2회 빗나가면 즉시 계측으로 전환하는 행동 규칙'. 이후 같은 계열 사고는 없습니다." (근거: 메모리 2026-04-28 미니언 전투/stuck 세션 기록, `.claude/gotchas.md`) [CONFIRM_NEEDED: 팀/대인 관계에서의 실패 사례도 1개 별도 준비 — 기술 실패만 있으면 협업 실패 질문에 재사용 불가]

### Q. 동료와 갈등이 생기면 어떻게 해결하나요?

**모범 골격**: "원칙은 세 가지입니다. ① 사람과 문제를 분리해 '사실-영향-요청' 순서로 말합니다 — '네 코드가 나쁘다'가 아니라 '이 함수가 프레임당 3ms를 쓰고 있고, 그래서 제 시스템 예산이 깨지는데, 캐싱을 검토해줄 수 있나'. ② 의견 충돌은 가능한 한 데이터로 환원합니다 — 프로파일링, 프로토타입, A/B. ③ 결정이 나면 반대했더라도 실행에 커밋하고, 대신 결정 과정에서 배운 것을 팀 규칙으로 남깁니다. [CONFIRM_NEEDED: 실제 갈등→해결 경험 1개 — 던전스 팀원/학원 동기/외주 등 실명 상황]" **꼬리 대비**: "그래도 상대가 감정적으로 나오면?" — 그 자리에서 이기려 하지 않고 시간을 분리한다(회의를 끊고 1:1로).

### Q. 일정이 부족할 때 품질과 일정 중 무엇을 택하나요?

**모범 골격**: "이분법을 거부하는 게 정답이라고 생각합니다. 실제 선택지는 '스코프'입니다. 일정이 고정이면 품질을 깎는 게 아니라 범위를 깎아야 하고, 그 결정은 제가 몰래 하는 게 아니라 우선순위를 아는 사람(리드/기획)에게 옵션으로 올립니다 — 'A안: 기능 3개 전부, 크래시 리스크 있음 / B안: 기능 2개, 안정'. 다만 깎으면 안 되는 바닥은 있습니다: 데이터 훼손 가능성과 크래시. 그건 일정과 교환 불가입니다. [CONFIRM_NEEDED: 실제 스코프 컷 경험 — Liberation 출시 직전 또는 팀 프로젝트 마감]"

### Q. 기획자/아티스트와 일할 때 프로그래머로서 중요하게 생각하는 것은?

**모범 골격**: "번역과 옵션 제시입니다. '안 됩니다' 대신 '그대로면 X 비용이 들고, 이렇게 바꾸면 80%를 절반 비용으로 얻습니다'로 답하는 것. 그리고 파이프라인을 코드보다 먼저 생각합니다 — 아티스트가 리소스를 넣고 결과를 보는 사이클이 느리면 게임 품질이 구조적으로 떨어지기 때문에, 저는 Winters에서도 에셋 변환 파이프라인(.wmesh/.wanim 컨버터)과 ImGui 튜너(이펙트 파라미터 실시간 조정) 같은 도구를 게임 코드와 같은 우선순위로 만들었습니다. [CONFIRM_NEEDED: 던전스/Liberation에서 실제 타 직군 협업 에피소드 1개]" **왜 강한가**: 툴 지향 답변은 신입에게서 거의 안 나온다.

### Q. 5년, 10년 뒤 어떤 개발자가 되고 싶나요?

**모범 골격**: "5년 안에는 출시와 라이브를 반복 경험한 네트워크/엔진 영역의 확실한 한 축이 되고 싶습니다 — 장애가 났을 때 '그 사람 부르면 된다'는 신뢰의 대상이요. 10년 뒤에는 시스템 설계와 팀의 개발 문화(리뷰/게이트/포스트모템)를 함께 설계하는 사람이 되고 싶습니다. 제가 1인 프로젝트에서도 소유권 매트릭스와 빌드 게이트 같은 팀 프로세스를 실험해온 이유이기도 합니다." **주의**: "창업하겠다"는 표현은 지원 동기 질문과 동일한 이유로 직접 언급 비추천.

### 행동면접 질문 뱅크 (STAR 템플릿 + 채우기 슬롯)

**공용 STAR 템플릿** — 모든 답변을 이 골격으로 60~90초에:

```text
S (Situation): 언제 / 어떤 프로젝트 / 어떤 상황이었나 — 1~2문장
T (Task):      그 상황에서 '내가' 책임진 목표는 무엇이었나 — 1문장
A (Action):    내가 한 행동. "우리"가 아니라 "내가". 구체 기술/수치 — 3~4문장
R (Result):    결과(수치/상태 변화) + 배워서 다음에 바꾼 것 — 1~2문장
```

**소재 은행 (실증 가능한 것들)** — 아래 질문들에 재사용:

#### 협업·코드리뷰

### Q. 코드리뷰에서 받은 가장 아픈 지적은 무엇이었고, 어떻게 반응했나요?
**의도**: 피드백 수용성. **프레임**: 소재 ⓐ — "제 병렬 AI 설계에서 entity 간 write race 가능성을 리뷰어가 지적했을 때, 처음엔 '발생 확률이 낮다'고 방어하고 싶었지만, 확률이 낮은 race는 낮아서 더 위험하다는 걸 인정하고 Decision/Apply 2-pass로 구조를 갈아엎었습니다. 20건 지적을 전수 반영했고, 그 리뷰어의 검토 패턴 자체(신설 전 기존 인프라 grep, 데이터 형태 우선)를 제 체크리스트로 흡수했습니다." [CONFIRM_NEEDED: 사람 리뷰어에게 받은 지적 사례가 있으면 그것으로 교체]

### Q. 반대로, 동료 코드에서 심각한 문제를 발견하면 어떻게 말하나요?
**의도**: 전달 방식. **프레임**: "코드에 대해, 근거와 함께, 대안을 붙여서. '이 부분 위험해요'가 아니라 '이 경로에서 세션이 닫히면 이 포인터가 댕글링이 됩니다. 참조 카운트나 핸들 방식이 대안인데 어떻게 생각하세요?' 상대가 고칠 수 있는 형태로 주는 게 리뷰의 예의라고 생각합니다." [CONFIRM_NEEDED: 실제로 타인 코드의 결함을 지적한 사례]

### Q. 리뷰 의견이 평행선일 때는 어떻게 결론을 내나요?
**의도**: 교착 해소. **프레임**: 소재 ⓑ — "논쟁의 수명을 정합니다. 왕복 2회를 넘기면 말이 아니라 실험으로 — 벤치마크, 프로토타입, 계측. 실제로 분석이 3회 어긋났을 때 논쟁을 끊고 카운터 계측 5분으로 판정한 뒤, 그걸 규칙으로 만들었습니다."

### Q. 팀의 컨벤션이 본인 취향과 다르면 어떻게 하나요?
**의도**: 팀 적응성. **프레임**: "따릅니다. 컨벤션의 가치는 내용이 아니라 일관성입니다. 실제로 제 프로젝트 규칙에도 '기존 스타일과 다르게 짜고 싶어도 파일의 기존 스타일을 따른다'를 명문화했습니다. 바꾸고 싶으면 몰래 다르게 짜는 게 아니라 규칙 자체의 개정을 제안합니다."

### Q. 협업 도구나 프로세스를 직접 개선한 경험이 있나요?
**의도**: 주도성. **프레임**: 소재 ⓒⓖ — "반복 실수를 한 줄 포맷으로 박제하는 gotchas 로그를 만들고, 문서로도 안 지켜지는 규칙은 훅과 PreBuild lint로 기계 강제화했습니다. '규칙은 있는데 강제가 없다'가 모든 팀의 공통 실패 패턴이라는 걸 1인 프로젝트에서 먼저 확인한 셈입니다."

### Q. 다른 사람이 짠 레거시 코드를 유지보수한 경험은?
**의도**: 실무 적응력. **프레임**: "던전스 팀 코드와 학원 수업 코드 베이스 위에서 작업하며, '고치기 전에 왜 이렇게 됐는지 먼저 찾는' 습관을 만들었습니다. 이해 없이 리팩터링하다 동작을 바꾸는 게 최악이라, 지금도 '요청과 무관한 인접 코드는 건드리지 않는다'를 작업 규칙으로 명문화해 둡니다." [CONFIRM_NEEDED: 레거시 유지보수 구체 사례]

#### 갈등

### Q. 팀원이 맡은 일정을 계속 어길 때 어떻게 했나요?
**의도**: 어려운 대화. **프레임**: S: 통합 일정에 병목 발생. T: 감정 상하지 않게 일정 복구. A: ① 비난 전 사실 확인(막힌 건지, 미룬 건지 — 대부분 막힌 것) ② 막혔다면 페어로 뚫기 ③ 구조 문제면 작업 단위를 더 작게 쪼개 진행이 보이게. R: [CONFIRM_NEEDED: 실제 결과]. "사람을 고치려 하지 않고 작업 크기를 고쳤더니 해결됐다"류의 결론이 강하다.

### Q. 리더/사수가 기술적으로 틀렸다고 확신하는 지시를 내리면?
**의도**: 순응 vs 소신의 균형. **프레임**: "한 번은 근거를 갖춰 반론합니다 — 가능하면 데이터로. 받아들여지지 않으면 실행하되, 제 우려가 현실화될 때 빨리 감지되도록 계측/로그를 심어둡니다. 맞고 틀리고보다 '틀렸을 때 빨리 아는 구조'가 팀에 더 이익이라고 생각합니다." [CONFIRM_NEEDED: 실제 사례]

### Q. 내 코드가 원인이라고 오해받은 적이 있나요?
**의도**: 방어 대신 검증. **프레임**: "누구 잘못이냐를 다투지 않고 재현/이등분(bisect)으로 원인을 좁힙니다. 원인이 저였으면 바로 인정하고 재발 방지를 붙이고, 아니었으면 데이터만 공유하고 감정은 얹지 않습니다." [CONFIRM_NEEDED: 실제 사례]

### Q. 팀원 간 갈등을 중재해 본 경험은?
**의도**: 리더십 씨앗. [CONFIRM_NEEDED: 중재 경험 — 없으면 이 질문은 '아직 그 위치에 있어본 적은 없지만 이렇게 접근하겠다'로 정직하게]. **프레임**: 각자 1:1로 먼저 듣기(공개 석상 중재는 체면 싸움이 됨) → 두 주장의 공통 목표 찾기 → 결정 기준 합의.

### Q. 감정이 상할 만큼 부딪힌 뒤 관계를 회복한 경험은?
**의도**: 성숙도. [CONFIRM_NEEDED: 실제 사례 — 학원 동기/팀원]. **프레임**: 시간을 두고 → 일 얘기가 아닌 자리에서 → "그때 내 전달 방식이 거칠었다"처럼 내 몫부터 인정. R에는 "그 사람과 이후 더 잘 일했다"가 반드시 있어야 한다.

#### 일정·압박

### Q. 마감 직전에 큰 버그가 발견되면 어떻게 하나요?
**프레임**: "① 영향 범위 판정(크래시/데이터 훼손이면 무조건 잡는다, 외관이면 알려진 이슈로 출시 가능) ② 수정의 리스크 평가 — 마감 직전 수정은 그 자체가 새 버그원이므로 diff 최소화 ③ 결정을 혼자 안 함 — 리스크 옵션을 정리해 결정권자에게. [CONFIRM_NEEDED: 실제 마감 직전 버그 사례 — Liberation 출시 또는 팀 프로젝트 시연 전날]"

### Q. 일정 산정을 크게 틀린 경험은?
**프레임**: "산정 실패의 원인은 대부분 '모르는 걸 모른다'입니다. [CONFIRM_NEEDED: 실제 산정 실패 사례 — 무엇을 몰랐는지]. 이후 바꾼 것: 착수 전 가장 불확실한 부분부터 스파이크(반나절 탐색)로 찔러 불확실성을 먼저 줄이고, 산정에 '모르는 것 목록'을 명시해 전달합니다."

### Q. 스코프를 잘라야 할 때 기준은?
**프레임**: "유저가 겪는 코어 루프 보호가 1순위, 되돌리기 쉬운 것부터 자르기가 2순위입니다. 기능을 반쯤 만들다 마는 것보다 완성도 있는 축소가 낫습니다 — '넓고 얕게'는 QA 비용을 폭발시킵니다." [CONFIRM_NEEDED: 실제 컷 결정 사례]

### Q. 크런치(장기 야근) 상황을 어떻게 관리했나요?
**프레임**: "단기 스퍼트는 받아들이되, 실수율이 올라가는 신호(리뷰 지적 증가, 어이없는 버그)를 크런치 중단 기준으로 삼습니다. 피곤할 때 저지른 실수의 수습 비용이 크런치로 번 시간보다 커지는 지점이 있습니다." [CONFIRM_NEEDED: 실제 크런치 경험]

#### 실패·실수

### Q. 본인 실수로 팀 전체가 피해를 본 경험은?
**프레임**: STAR + 소재 ⓔ 변형 가능 — "빌드/배포 스크립트가 특정 조건에서 SDK 트리를 반쯤 지워 다른 작업 라인의 컴파일을 깨뜨린 사고가 있었습니다. 수습 후, 위험한 purge 동작을 명시적 환경변수 뒤로 옮기고 사고 경위를 스크립트 주석으로 남겼습니다 — 코드에 사건 기록을 남기면 다음 사람이 같은 지뢰를 밟지 않으니까요." [CONFIRM_NEEDED: 다인 팀에서의 실제 민폐 사례가 있으면 우선]

### Q. 빌드나 배포를 망가뜨린 경험은?
**프레임**: 위와 동일 소재 사용 가능. 핵심 채점 포인트는 "숨기지 않고 즉시 알림 → 롤백/수습 → 재발 방지 장치"의 3박자.

### Q. 같은 실수를 반복하지 않기 위한 본인만의 장치가 있나요?
**프레임 (이 질문은 준비된 최강 답변)**: 소재 ⓒ — "실수를 날짜-영역-실수-방지책 한 줄로 박제하는 로그를 운영합니다. 지금 27건이 넘고, 이 로그는 매 작업 세션 시작 시 자동으로 읽힙니다. 더 중요한 건 에스컬레이션입니다 — 같은 계열 실수가 반복되면 그 규칙을 사람 기억에서 꺼내 빌드 게이트나 훅으로 승격시킵니다. 아키텍처 경계 위반은 이제 PreBuild lint가 빌드를 실패시키기 때문에 '조심'이 필요 없습니다."

### Q. 오랫동안 못 잡은 버그와 해결 과정을 이야기해 주세요.
**프레임**: 소재 ⓑ 풀버전 — 증상(미니언이 제자리 애니만 반복) → 추측 3회 실패(코드 리딩만으로 stale 분석 반복) → 전환점(프로파일러 카운터 계측) → 원인(시스템 실행 순서에 의한 stale velocity + 경로 실패의 silent fail) → 수정(Phase swap + fallback) → 박제(gotchas 2건). "silent fail은 버그가 아니라 관측 불가능성이 문제"라는 교훈으로 마무리.

#### 소통·타 직군

### Q. 기획자가 기술적으로 불가능한(비싼) 요구를 하면?
**프레임**: "'불가능'이라는 단어를 쓰기 전에 요구의 목적을 묻습니다. 대부분 요구는 수단이고 목적은 따로 있어서, 목적을 알면 80%를 1/10 비용으로 주는 안이 나옵니다. 그리고 비용을 체감 단위로 번역합니다 — '그건 프레임당 5ms인데, 우리 전체 예산이 16ms입니다'." [CONFIRM_NEEDED: 실제 기획 협상 사례]

### Q. 비개발자에게 복잡한 기술 문제를 설명해야 했던 경험은?
**프레임**: "비유 + 선택지 + 추천의 3단으로 말합니다. 설명의 목적은 이해가 아니라 의사결정이니까요." [CONFIRM_NEEDED: 실제 사례 — 시연/발표/가족에게라도]

### Q. 아트 리소스 파이프라인 문제로 부딪힌 경험은?
**프레임**: "리소스 규격 문제는 사람이 아니라 검증 도구로 풉니다. Winters에서 챔피언 메시마다 forward axis가 달라 생기는 회전 버그를 겪은 뒤, '에셋별 보정값을 코드에 산발적으로 넣지 말고 규격 함수 하나로 모은다'는 규칙을 만들었습니다. 아티스트와 일한다면 같은 원리로 '임포트 시점 자동 검증'을 먼저 만들 겁니다." [CONFIRM_NEEDED: 실제 아티스트 협업 사례]

#### 성장·주도성

### Q. 최근 스스로 학습한 기술과 학습 방법은?
**프레임**: "만들면서 배웁니다. IOCP를 책으로 읽는 대신 서버를 짰고, Fiber 기반 잡시스템을 논문/발표 자료 → 최소 구현 → 스트레스 테스트 순으로 익혔습니다. 학습의 완료 조건을 '이해했다'가 아니라 '돌아가고, 깨지는 지점을 안다'로 둡니다."

### Q. 아무도 시키지 않았는데 주도적으로 개선한 경험은?
**프레임**: 소재 ⓓ — "규칙 위반이 80파일까지 쌓인 아키텍처 경계를 전수 감사하고, 어댑터 계층을 만들어 79파일을 재라우팅한 뒤, 재발을 빌드 실패로 전환했습니다. 아무도 요구하지 않았지만, 이걸 안 하면 6개월 뒤의 제가 비용을 낸다는 걸 알았으니까요."

### Q. 피드백을 스스로 요청해서 받은 경험은?
**프레임**: 소재 ⓐ — "설계마다 교차 리뷰를 스스로 돌립니다. 지적이 나오라고 요청하는 리뷰라서, 지적 0건이면 오히려 리뷰 방식을 의심합니다." [CONFIRM_NEEDED: 사람에게 피드백을 요청한 사례]

### Q. 다른 사람을 가르치거나 지식을 공유한 경험은?
[CONFIRM_NEEDED: 멘토링/스터디/발표 경험 — 학원 동기 스터디, 팀 내 세미나 등]. **프레임**: 없으면 문서화로 답한다 — "제 프로젝트의 온보딩 문서 체계(읽기 순서 명시, 실수 로그, 아키텍처 나침반)는 '미래의 팀원'을 독자로 상정하고 씁니다. 실제로 새 세션(사람이든 에이전트든)이 같은 순서로 온보딩됩니다."

## 내 프로젝트 연결 포인트

어떤 질문이 와도 아래 문장들로 회수한다. (모두 실존 근거 있음)

1. **RTS 길찾기의 수직 성장**: "2D 타일 A*(`CNavMgr::AStarCells`)를 손으로 짜봤기 때문에, Winters의 3D 냅그리드에서 '현재 셀/다음 웨이포인트/보정 방향'을 오버레이로 계측하는 디버깅 문화를 만들 수 있었습니다."
2. **네트워크의 세대 교체를 직접 통과**: "던전스에서 TCP IOCP를, Winters에서 UDP 스냅샷 + 클라 예측 + 서버 권위를 만들었습니다. '왜 실시간 게임은 UDP인가'를 슬라이드가 아니라 두 프로젝트의 차이로 설명할 수 있습니다."
3. **OOP→ECS를 번역해 본 사람**: "던전스의 `CInventoryMgr` 구조를 Winters의 데이터+커맨드+드로우 3분할로 옮기는 설계 문서를 직접 썼습니다(`.md/TODO/05-12/INVENTORY_UI_DUNGEONS_TO_WINTERS_PLAN.md`). 아키텍처 전환은 개념이 아니라 구체 파일 단위 매핑으로 해봤습니다."
4. **협업을 시스템으로 설계**: "소유권 매트릭스, 작업 패킷, push 전 검증 하네스, 반복 실수의 빌드 게이트 승격까지 — 다인 팀 프로세스를 1인+에이전트 환경에서 먼저 구축해 운영했습니다(`.md/collab/*`, `Tools/Harness/*`). 팀에 합류하면 프로세스의 소비자가 아니라 기여자가 될 수 있습니다."
5. **리뷰 문화의 실증**: "작성자와 리뷰어를 분리한 교차 리뷰로 설계 결함 20건을 출시 전에 잡아 전수 반영한 사이클을 돌려봤고, 좋은 리뷰어의 패턴을 제 체크리스트로 흡수했습니다."
6. **출시 감각과 엔진 감각의 결합**: "출시를 해봤기 때문에 [CONFIRM_NEEDED: Liberation 사실 확정 후 이 문장 활성화] 자체 엔진을 감사할 때도 minidump/크래시 수집/버전 핸드셰이크 같은 라이브 인프라 부재를 최우선 리스크로 분류했습니다."
7. **클래스 폭발→데이터 주도**: "스타 모작에서 유닛 클래스 수십 개를 상속+팩토리로 버텼던 경험이, Winters에서 챔피언을 데이터 정의+공용 시스템으로 설계한(150종 스케일 목표) 직접적 이유입니다."
8. **같은 게임을 세 번 만든 집요함**: "RTS/MOBA라는 같은 장르를 WinAPI→DX9 팀 프로젝트→자체 DX11 엔진으로 세 번 다시 만들었습니다. 스택이 바뀌어도 변하지 않는 문제(선택, 이동, 판정, 동기화)와 스택마다 달라지는 해법을 모두 압니다."
