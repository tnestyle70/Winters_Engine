# 변환 파이프라인 수학 — Local에서 픽셀까지

> "정점 하나가 화면 픽셀이 되기까지의 전 과정을 말해보세요" — 렌더링 면접의 1순위 단골 질문에 수식 수준으로 답하는 챕터. 각 단계의 행렬이 **왜 그 모양인지** 유도하고, 규약 차이(DX/GL, row/column, LH/RH)가 만드는 함정까지 관통한다.
>
> 역할 분담: Winters 렌더 파이프라인의 **구조**(RHI·DX11·FX 계층)는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)가 담당한다. 이 챕터는 그 구조 위를 흐르는 **수학과 API 본질**만 다룬다.

---

## ① 한 줄 본질

> **"정점은 Local → World(S·R·T) → View(카메라 월드 변환의 역) → Clip(투영 행렬)까지 4×4 행렬 곱으로 이동하고, 클리핑 후 w로 나눠 NDC가 되고, 뷰포트 변환으로 픽셀 좌표가 된다. 이 전 과정을 '행렬 곱'이라는 단일 연산으로 통일해 주는 장치가 동차 좌표(homogeneous coordinates)의 4번째 성분 w다 — w는 평행이동을 행렬 안으로 넣어주고(아핀), 원근 나눗셈을 클리핑 뒤로 지연시킨다(투영). 버텍스 셰이더가 책임지는 구간은 Local→Clip까지이고, 나눗셈·클리핑·뷰포트는 고정 기능이다."**

---

## ② 기본 개념

### 전체 체인 조감도

```
 Local space   (모델 저작 좌표 — 아티스트가 원점 기준으로 만든 정점)
     │
     │  World = S · R · T            ← 개체를 월드에 배치 (row-vector 규약: v' = v·M)
     ▼
 World space   (씬 공통 좌표계)
     │
     │  View = (카메라 World 행렬)⁻¹  ← 카메라를 원점으로, 시선을 +Z로 (LH)
     ▼
 View space    (눈 좌표계 — 카메라가 원점)
     │
     │  Projection (fovY, aspect, near, far)
     ▼
 Clip space    (아직 4D 동차 좌표 — w' = view-space z)
     │
     ├── 클리핑: -w ≤ x ≤ w, -w ≤ y ≤ w, 0 ≤ z ≤ w   ← 나눗셈 "전"에 수행
     │
     │  ÷ w   (원근 나눗셈, perspective divide — 고정 기능)
     ▼
 NDC           (x,y ∈ [-1,1], z ∈ [0,1] — DirectX 규약)
     │
     │  Viewport 변환 (스케일 + 평행이동 + y 뒤집기)
     ▼
 픽셀 좌표      (래스터라이저 → 깊이 테스트 → 픽셀 셰이더)
```

책임 경계를 정확히 말할 수 있어야 한다: **버텍스 셰이더의 출력(SV_POSITION)은 clip space**다. 클리핑, w 나눗셈, 뷰포트 변환, 래스터화는 프로그래머블 셰이더 사이의 고정 기능(fixed-function) 단계다. "VS에서 NDC를 출력한다"고 말하면 감점.

### 동차 좌표 — w 하나가 두 가지 문제를 산다

3×3 행렬은 선형 변환(linear transform)이라 **원점을 항상 원점으로** 보낸다. 즉 회전·스케일은 되지만 평행이동(translation)은 수학적으로 불가능하다. `v' = v·M + t`처럼 덧셈을 따로 들고 다니면 변환 합성이 "행렬 곱 하나"로 안 묶인다. 해결책이 4번째 성분 w:

```
                     ┌ R₃ₓ₃   0 ┐
[x y z 1] · M,  M =  │           │   →  x·R행들 + 1·t  = 회전 + 평행이동
                     └ t      1 ┘
```

w=1인 점(point)은 마지막 행 t를 "1배"로 받아 평행이동되고, **w=0인 방향(direction)은 t행이 0배가 되어 평행이동을 무시한다**. 점과 방향의 구분이 데이터(w)에 인코딩되는 것이다. 이것이 아핀(affine) 측면.

두 번째 역할이 투영(projective) 측면이다. 원근 투영의 본질은 `x/z` — 즉 **z로 나누기**인데, 행렬은 선형이라 나눗셈을 표현할 수 없다. 그래서 투영 행렬은 나눗셈을 하는 대신 **w'에 z를 복사해 두고**, 나눗셈 자체는 클리핑 이후의 고정 기능 단계(÷w)로 지연시킨다. "행렬 곱만으로 파이프라인을 끝까지 밀고 가되, 비선형 연산(나눗셈)은 딱 한 번, 정해진 자리에서" — 이것이 동차 좌표 설계의 핵심 트레이드오프다.

### World 행렬 — S·R·T 합성과 순서가 중요한 이유

row-vector 규약(`v' = v·M`)에서 World 행렬은:

```
World = Scale · Rotation · Translation      (왼쪽이 먼저 적용됨)
```

행렬 곱은 교환법칙이 성립하지 않으므로 순서가 곧 의미다. **S→R→T 순서인 이유**: 스케일과 회전은 원점 기준 연산이라, 개체가 아직 원점에 있을 때(로컬 상태) 적용해야 "자기 자리에서 커지고 자기 축으로 도는" 결과가 나온다. 순서를 T·R로 바꾸면 "이동된 위치 자체가 회전"돼 개체가 원점 주위를 공전(orbit)한다. 또 R·S 순서(회전 후 스케일)는 월드 축 기준 스케일이 되어 개체가 기울어진 채 찌그러진다(전단처럼 보임).

계층(hierarchy)이 있으면:

```
World(child) = Local(child) · World(parent)     (row-vector 규약)
```

자식의 로컬 변환이 먼저 적용되고, 그 결과가 부모의 좌표계에 실려 간다. column-vector 교재라면 같은 식이 `World(parent) · Local(child)`로 표기된다 — 규약이 다를 뿐 같은 변환이다.

### View 행렬 — 왜 "카메라 월드 행렬의 역"인가

view space의 정의는 "카메라가 원점에 있고 시선이 +Z(LH 기준)인 좌표계"다. 카메라를 월드에 배치하는 행렬을 `W_cam`이라 하면, 월드의 점을 카메라 좌표계로 옮기는 변환은 정확히 그 **역행렬**이다:

```
View = W_cam⁻¹
```

직관: 카메라를 움직이는 것과 세상 전체를 반대로 움직이는 것은 같은 상대 배치를 만든다. 카메라가 오른쪽으로 이동 = 세상이 왼쪽으로 이동. GPU는 카메라를 모른다 — 항상 "원점의 관찰자"에게 세상을 끌어다 놓을 뿐이다.

**LookAt 유도** (LH, row-vector). eye에서 target을 바라보는 카메라의 기저(basis)를 만든다:

```
forward = normalize(target - eye)          // 시선 = +Z 기저
right   = normalize(cross(up, forward))    // LH: up × forward
up'     = cross(forward, right)            // 재직교화 (입력 up은 힌트일 뿐)
```

카메라의 월드 행렬은 이 기저를 **행(row)**으로 갖는다(row-vector 규약에서 행렬의 각 행 = 그 좌표계의 축이 월드에서 가리키는 방향):

```
         ┌ right    0 ┐
W_cam =  │ up'      0 │
         │ forward  0 │
         └ eye      1 ┘
```

이걸 뒤집어야 View인데, **일반 역행렬(가우스 소거)을 계산할 필요가 없다.** 회전 부분(위 3×3)은 정규직교(orthonormal)라서 `R⁻¹ = Rᵀ`이고, 평행이동은 `-eye`를 회전의 역으로 돌려주면 끝:

```
        ┌ right.x   up'.x   forward.x   0 ┐
View =  │ right.y   up'.y   forward.y   0 │
        │ right.z   up'.z   forward.z   0 │
        └ -eye·right  -eye·up'  -eye·forward  1 ┘
```

기저가 행에서 **열로 전치**되고, 마지막 행에 `-dot(eye, 축)` 세 개가 놓인다. "왜 역행렬을 싸게 구할 수 있나?"는 꼬리질문의 답이 "회전은 직교행렬이라 전치가 곧 역"이다.

### 원근 투영 행렬 — 원소 하나하나의 유도

#### (1) x, y 스케일 — 상사 삼각형(similar triangles)

view space의 점 `(x, y, z)`를 눈(원점)에서 본다. 시선축에서 거리 z만큼 떨어진 지점의 화면 반높이는 `z·tan(fovY/2)`다 — fovY는 위아래 **전체** 각이고 시선축 기준 대칭이므로 절반만 삼각형을 그리면 `tan(fovY/2) = 반높이/거리`. 이게 "왜 반각인가"의 답이다.

y를 [-1,1]로 정규화하려면 그 반높이로 나눈다:

```
y_ndc = y / (z · tan(fovY/2)) = (y · cot(fovY/2)) / z
```

x는 화면이 `aspect = width/height`배 더 넓으므로 반너비 `z·tan(fovY/2)·aspect`로 나눈다:

```
x_ndc = (x · cot(fovY/2) / aspect) / z
```

둘 다 "z로 나누기"만 빼면 **선형**(상수 곱)이다. 그래서 상수 부분만 행렬에 넣고 나눗셈은 w로 미룬다.

#### (2) z 매핑 — 경계 조건으로 두 미지수 풀기

동차 행렬이 만들 수 있는 z는 `z' = A·z + B, w' = z` 꼴뿐이므로, 나눗셈 후 깊이는 반드시 `z_ndc = A + B/z` 형태다. DirectX 규약(near→0, far→1)을 경계 조건으로 넣으면:

```
A + B/n = 0,   A + B/f = 1
→  A = f/(f-n),   B = -n·f/(f-n)
```

#### (3) w 트릭

`r[2][3] = 1`(row-vector 기준 3행 4열)로 **view-space z를 w'에 복사**한다. 행렬은 1/z를 만들 수 없으니, "나중에 나눌 값"을 w에 실어 보내고 고정 기능이 나눈다. 이 한 칸이 원근 투영의 전부라 해도 과언이 아니다.

#### 완성 — XMMatrixPerspectiveFovLH (DirectX LH, z∈[0,1], row-vector)

```
h = cot(fovY/2),  w0 = h/aspect,  A = f/(f-n),  B = -n·f/(f-n)

        ┌ w0   0    0    0 ┐
        │ 0    h    0    0 │        [x y z 1]·P = (x·w0,  y·h,  z·A + B,  z)
P =     │ 0    0    A    1 │
        └ 0    0    B    0 ┘        ÷w 후: ( x·w0/z,  y·h/z,  A + B/z )
```

검산: `z=n → A + B/n = 0`, `z=f → A + B/f = 1`. 정확히 [0,1].

#### 직교 투영(orthographic)과의 차이

```
        ┌ 2/W   0     0         0 ┐
        │ 0     2/H   0         0 │      w' = 1  →  나눗셈이 사실상 no-op
O =     │ 0     0     1/(f-n)   0 │      z_ndc = (z-n)/(f-n)  — z에 "선형"
        └ 0     0    -n/(f-n)   1 ┘
```

차이의 본질은 마지막 열이다. 원근은 `w'=z`(거리에 따라 축소), 직교는 `w'=1`(거리 무관). 그래서 직교는 깊이도 선형으로 분포한다 — 뒤의 깊이 정밀도 논의에서 대비가 된다. UI, 미니맵, 그림자맵(방향광)이 직교를 쓰는 이유는 "거리에 따른 크기 변화가 없어야 하는" 용도이기 때문이다.

### 클리핑 — 왜 clip space(나눗셈 전)에서 하는가

클리핑 조건은 NDC가 아니라 **동차 좌표의 부등식**으로 정의된다:

```
DX:  -w ≤ x ≤ w,   -w ≤ y ≤ w,   0 ≤ z ≤ w
```

나눗셈 전에 해야 하는 이유가 세 겹이다:

1. **w=0 나눗셈 폭발.** 눈 평면(z=0)에 걸친 프리미티브는 w'=0인 정점을 가진다. 먼저 나누면 무한대가 튄다.
2. **카메라 뒤 점의 반전.** w<0인 점(카메라 뒤)을 나누면 부호가 뒤집혀 화면 반대편에 "유령 프리미티브"가 생긴다. 클리핑이 이 점들을 나눗셈 전에 잘라낸다.
3. **선형성.** Local→Clip까지는 전부 행렬 곱, 즉 선형이다. 선형 공간에서는 평면과 선분의 교점을 매개변수 `t = d₀/(d₀-d₁)`로 구하고 위치·UV·색 등 **모든 어트리뷰트를 같은 t로 선형 보간**하면 정확하다. 나눗셈 후(NDC)는 비선형 공간이라 이 보간이 틀어진다.

부등식 자체도 우아하다: NDC 조건 `-1 ≤ x/w ≤ 1`의 양변에 w를 곱한 것뿐인데, 나눗셈 없이 곱셈·비교만으로 판정된다. (현대 GPU는 x/y는 가드밴드(guard-band)로 러프하게 처리하고 실제 클리핑은 주로 near 평면에서 일어난다 — 꼬리질문 대비 한 줄.)

### NDC 규약 차이 — DX vs GL, 이식의 함정

| | 깊이 z 범위 | NDC y | 픽셀 원점 |
|---|---|---|---|
| DirectX | `[0, 1]` | 위가 + | 좌상단 |
| OpenGL (전통) | `[-1, 1]` | 위가 + | 좌하단 |
| Vulkan | `[0, 1]` | **아래가 +** | 좌상단 |

이식 함정 두 가지를 짚을 수 있어야 한다:

- **GL용 투영 행렬을 DX에 그대로 쓰면**: z가 [-1,1]로 나오는데 DX는 [0,1]만 클리핑 통과 + 뷰포트가 [0,1]을 가정 → near 쪽 절반이 잘리거나 깊이 범위 절반이 낭비된다. 행렬의 z행/열만 다르고 겉보기엔 굴러가서 발견이 늦는 종류의 버그.
- **y 방향**: GL 텍스처 좌표 원점(좌하단)과 DX(좌상단) 차이로 렌더타겟을 텍스처로 읽을 때 상하 반전. Vulkan은 아예 NDC y가 아래로 자라 뷰포트 height를 음수로 주는 관용구가 생겼다.

### 뷰포트 변환 — NDC에서 픽셀로

마지막은 순수한 스케일+평행이동이다:

```
x_px = (x_ndc + 1) · Width/2  + TopLeftX
y_px = (1 - y_ndc) · Height/2 + TopLeftY      ← y 뒤집기: NDC는 위가 +, 픽셀은 아래가 +
depth = MinDepth + z_ndc · (MaxDepth - MinDepth)   (보통 0~1 그대로)
```

y에 `(1 - y_ndc)`가 들어가는 이유: 수학 좌표(위가 +)와 스크린 메모리 좌표(스캔라인이 위→아래)의 방향이 반대라서다. 그리고 픽셀 (x,y)의 **샘플 위치는 (x+0.5, y+0.5)** — 픽셀은 점이 아니라 면적이고 그 중심에서 커버리지를 판정한다. D3D9 시절에는 픽셀 중심 규약이 달라 "반 픽셀 오프셋" 보정이 필요했고, D3D10+에서 0.5 중심으로 통일됐다. 픽셀 셰이더의 SV_POSITION에 x.5가 들어오는 이유가 이것이다.

### 깊이 버퍼의 비선형성 — z-fighting의 수학적 원인

위에서 유도했듯 `z_ndc = A + B/z` — 깊이값은 view z가 아니라 **1/z에 선형**이다. 이 분포가 얼마나 극단적인지 숫자로 보자. `f ≫ n`일 때:

```
z_ndc(2n) = f/(f-n) · (1 - n/2n) ≈ 0.5
```

**near와 2·near 사이가 전체 깊이 범위의 절반을 소모한다.** near=1이면 카메라 앞 1~2 유닛에 [0, 0.5]를 다 쓰고, 2~far 전체가 나머지 [0.5, 1]에 압축된다. 먼 곳일수록 이웃한 z가 같은 깊이값으로 뭉개진다.

정량화: 거리 z에서 깊이 1비트가 구분할 수 있는 최소 간격은 (24bit UNORM, 양자 ≈ 2⁻²⁴):

```
Δz_min ≈ z² · (f-n) / (n·f·2²⁴)  ≈  z² / (n·2²⁴)     (f ≫ n일 때)
```

이 식에서 두 가지 면접 포인트가 나온다:

1. **far는 근사식에서 소거된다.** far를 10000→1000으로 내려도 정밀도는 거의 안 변한다. 반면 **near를 0.1→1로 올리면 정밀도가 10배** 좋아진다. "z-fighting 나면 뭘 조정하나?" → "far 내리기보다 near 올리기가 압도적으로 효과적, 분모에 n이 있기 때문"이 정답.
2. z²에 비례하므로 먼 곳일수록 제곱으로 나빠진다. near=1, far=1000일 때 z=500 지점의 Δz_min ≈ 0.015 유닛 — 이보다 가까운 두 표면(겹쳐 모델링된 벽, 데칼)은 프레임마다 승자가 바뀌며 지글거린다. 이것이 z-fighting이다.

**포맷의 차이**: `D24_UNORM`은 고정소수점이라 [0,1]을 균일하게 양자화한다 — 1/z 곡선의 비선형성을 그대로 얻어맞는다. `D32_FLOAT`는 부동소수점이라 **0 근처에 값이 조밀**한데, 표준 매핑(near→0)에서는 어차피 near 쪽에 정밀도가 몰리는 곳에 float의 조밀함까지 겹쳐 **이중 낭비**다.

**reversed-Z**가 이 둘을 맞물리게 한다: near→1, far→0으로 뒤집으면(투영 행렬의 A,B를 near↔far 스왑으로 재유도, 깊이 클리어 0, 비교 함수 GREATER), 1/z 곡선이 값을 잔뜩 몰아넣는 "먼 거리 구간"이 float 정밀도가 조밀한 "0 근처"에 배치된다. 두 비선형성이 서로 상쇄되어 전 구간 거의 균일한 정밀도가 나온다. 단, **UNORM에서는 균일 양자화라 뒤집어도 이득이 없다** — reversed-Z는 `D32_FLOAT + GREATER`와 세트일 때만 의미 있다.

### row-major vs column-major — 두 개의 독립된 문제

면접에서 가장 많이 뒤엉키는 주제. **저장 순서**와 **곱셈 규약**은 서로 독립이라는 것부터 분리해야 한다.

| 축 | 질문 | 선택지 |
|---|---|---|
| 저장 순서 (memory layout) | 메모리의 연속 float 4개가 행인가 열인가 | row-major / column-major |
| 곱셈 규약 (convention) | 벡터가 행인가 열인가 | `v·M` (row-vector) / `M·v` (column-vector) |

- **DirectXMath**: row-major 저장 + row-vector 규약. 그래서 코드가 `World * View * Proj` 순서로 곱한다(`v·W·V·P`, 왼쪽부터 적용).
- **교과서/GL 문헌**: column-vector 규약(`P·V·M·v`). **같은 변환인데 표기 순서가 거울상**이다. 이걸 설명 못 하면 규약을 이해 못 한 것.
- **HLSL cbuffer 기본 패킹은 column_major**다. CPU가 row-major 메모리를 그대로 올리면 셰이더는 그 바이트를 열 우선으로 재해석한다 — **공짜 전치**가 일어난다.

그래서 실무 관행이 두 갈래다:

1. **CPU에서 전치해 올리기** (D3DX 시절 표준): `Transpose(M)`을 올리면 column_major 해석이 전치를 한 번 더 해서 원래 M로 복원. `mul(v, M)` 그대로. 기본 패킹(column_major)에서 `mul(v, M)`은 출력 성분마다 "v와 열 하나의 내적" = 레지스터당 dp4 하나로 컴파일되는 게 역사적 이유였다.
2. **HLSL에 `row_major` 선언** (Winters의 선택): 메모리를 행 우선으로 해석하게 지시하고 전치 없이 raw 업로드. 현대 GPU에서 두 방식의 성능 차는 사실상 없다.

핵심 함정: **전치와 row_major를 "둘 다" 하거나 "둘 다 안" 하면 행렬이 전치된 채 들어간다.** 전치된 View 행렬은 평행이동이 4번째 열로 이동해 `w'`가 카메라 위치에 의존하는 항을 갖게 되고, 카메라가 움직일 때마다 화면이 비대칭으로 클리핑되거나 왜곡되는 기묘한 증상이 난다. 순수 회전만 있을 땐 `Rᵀ = R⁻¹`이라 "그럭저럭 굴러가서" 발견이 늦는다.

`mul(a, b)`는 오버로드다: `mul(v, M)`은 v를 행벡터로, `mul(M, v)`는 열벡터로 취급한다. `mul(M, v)`로 바꾸면 수학적으로 전치를 한 번 곱한 것과 같아, 잘못된 패킹과 이중 부정으로 "우연히 맞는" 코드가 만들어지기도 한다 — 우연히 맞는 코드가 제일 위험하다.

### LH vs RH 좌표계

- **왼손(LH)**: +X 오른쪽, +Y 위, **+Z 화면 안쪽**. DirectX 전통.
- **오른손(RH)**: +Z가 화면 바깥쪽. OpenGL/수학 전통.

차이가 새는 지점: forward의 부호(LH는 +Z가 시선, RH는 -Z), 외적의 방향(`cross(up, forward)` 결과가 규약에 따라 반대), 감기 방향(winding)과 백페이스 컬링(같은 메시가 LH/RH에서 앞뒤가 뒤집힘), 그리고 LookAt·Perspective 팩토리의 LH/RH 변형이 전부 이 부호 차이의 전파다. 엔진 안에서 **하나로 통일하고 절대 섞지 않는 것**이 유일한 정답이고, 외부 에셋(RH로 저작된 FBX 등)은 임포트 시점에 변환한다.

### 노멀은 왜 역전치(inverse transpose)인가

노멀은 "표면에 수직"이라는 **관계**를 나타내지, 표면 위의 방향이 아니다. 접선(tangent) t는 표면을 따라가므로 위치와 같은 행렬 M으로 변환돼야 하지만, 노멀 n에게 필요한 조건은 변환 후에도 `n' ⊥ t'`가 유지되는 것이다.

유도 (row-vector): `n' = n·G`라 두고 수직 조건을 강제한다.

```
n'·t'ᵀ = (n·G)(t·M)ᵀ = n · (G·Mᵀ) · tᵀ = 0   (모든 n·tᵀ=0인 t에 대해)
→  G·Mᵀ = c·I   →   G = c·(Mᵀ)⁻¹ = c·(M⁻¹)ᵀ
```

즉 `G = transpose(inverse(M))`. 스케일 상수 c는 정규화로 흡수되므로 무시한다.

**비균등 스케일에서 깨지는 직관적 예**: 45° 경사면의 노멀 `(-1,1)/√2`. x를 2배 늘리면(`diag(2,1)`) 경사는 완만해진다(기울기 1/2). 그런데 노멀을 같은 행렬로 변환하면 `(-2,1)` — 더 가팔라진 쪽으로 눕는다. 완만해진 면의 올바른 노멀은 `(-1,2)` 방향(더 수직에 가깝게). 역전치 `diag(1/2,1)`을 곱하면 `(-1/2,1) ∝ (-1,2)` — 정확히 맞는다. **면이 늘어난 축에서 노멀은 반대로 줄어야** 수직이 유지되고, 그게 역행렬의 전치가 하는 일이다.

따름 규칙들: (1) 회전만 있으면 `R⁻ᵀ = R`이라 그냥 월드 행렬을 써도 된다. (2) 균등 스케일은 방향이 보존되므로 정규화만 하면 된다. (3) 역전치는 길이를 보존하지 않으므로 셰이더에서 **항상 normalize**. (4) 노멀은 방향이므로 `w=0`으로 곱해 평행이동을 차단한다.

## ③ 심화 — 꼬리질문 대비

**"w=1과 w=0의 차이는 뭔가?"**
점(point)과 방향(direction)의 구분이다. 행렬 마지막 행(평행이동)이 w배로 더해지므로 w=1은 이동하고 w=0은 이동하지 않는다. 위치는 `TransformCoord`(w=1, 나눗셈 포함), 노멀·속도 같은 방향은 `TransformNormal`(w=0)로 API가 갈라져 있는 이유가 이것이다. "노멀에 실수로 w=1을 넣으면?" → 오브젝트가 원점에서 멀수록 노멀이 이동 벡터 쪽으로 쏠려 라이팅이 위치 의존적으로 틀어진다.

**"왜 굳이 4×4인가, 3×4로도 아핀은 되지 않나?"**
아핀만 필요하면 3×4(+암묵적 [0 0 0 1])로 충분하고 실제로 스키닝 본 팔레트는 3×4로 압축하기도 한다. 4×4가 필요한 건 **투영** 때문이다 — 마지막 열에 z를 w로 복사하는 원소가 있어야 원근 나눗셈을 인코딩할 수 있다. "모든 변환을 같은 타입(4×4)으로 합성 가능"이라는 균일성의 값이기도 하다.

**"클리핑을 나눗셈 후 NDC에서 하면 안 되나?"**
세 가지가 깨진다. w=0(눈 평면) 정점에서 0 나눗셈, w<0(카메라 뒤) 점이 나눗셈으로 부호가 뒤집혀 반대편에 유령으로 등장, 그리고 NDC는 비선형 공간이라 클리핑 교점에서의 어트리뷰트 보간이 부정확해진다. clip space는 나눗셈 전이라 선형이고, 조건도 `-w ≤ x ≤ w`처럼 곱셈·비교만으로 판정된다.

**"near plane을 0으로 두면 왜 안 되나?"**
z 매핑 유도에서 `B = -nf/(f-n)` — n=0이면 B=0이 되어 `z_ndc = A` 상수, 깊이 정보가 통째로 사라진다. 기하적으로는 눈 위치에서 투영 자체가 발산한다(`x/z, z→0`). 그래서 near는 "작지만 0이 아닌" 값이어야 하고, 작을수록 깊이 정밀도를 지수적으로 낭비하므로 **감당 가능한 한 크게** 잡는 게 정석이다.

**"깊이값이 view z에 비선형인데 깊이 테스트는 왜 올바른가?"**
`z_ndc = A + B/z`는 비선형이지만 z>0 구간에서 **단조증가**다(B<0이므로 1/z가 줄면 z_ndc가 증가). 대소 비교만 하는 깊이 테스트에는 단조성이면 충분하다. 문제는 순서가 아니라 **해상도** — 먼 곳에서 서로 다른 z가 같은 양자화 칸에 떨어지는 것이고, 그게 z-fighting이다.

**"reversed-Z가 왜 float 깊이에서만 의미 있는지 설명해보라."**
float는 지수 표현이라 0 근처에 표현 가능한 값이 기하급수적으로 조밀하다. 표준 매핑은 near→0이라 "1/z 곡선이 이미 정밀한 near 구간"에 float의 조밀함이 중복 투자된다. near→1, far→0으로 뒤집으면 1/z 곡선이 넓게 펴 놓는 far 구간이 float가 조밀한 0 근처로 가서 두 비선형성이 상쇄된다. UNORM은 균일 양자화라 뒤집어도 분포가 안 변한다. 구현은 투영 행렬 z행 재유도 + 클리어값 0 + 비교 GREATER + (있다면) depth bias 부호 반전까지 세트.

**"뷰 행렬 역행렬을 어떻게 싸게 구하나?"**
카메라 변환은 강체(rigid) — 회전+이동뿐이다. 회전 3×3은 정규직교라 역=전치이고, 이동은 `-eye`를 그 전치 회전으로 돌리면 된다. O(n³) 가우스 소거가 아니라 전치 한 번 + 내적 3번. 반대로 스케일이 섞인 일반 월드 행렬의 역은 이 지름길이 없다.

**"mul(v, M)과 mul(M, v)의 차이는?"**
HLSL `mul`은 인자 형태로 규약을 정한다. `mul(v, M)`은 v를 1×4 행벡터로(v·M), `mul(M, v)`는 4×1 열벡터로(M·v) 취급한다. 수학적으로 `mul(M, v) = mul(v, Mᵀ)`이므로, 패킹(row_major/column_major)과 mul 순서 중 **정확히 한 군데서만** 전치를 소비해야 한다. 두 군데면 이중 부정으로 우연히 맞고, 0군데면 틀린다.

**"왜 cbuffer에 올리기 전에 Transpose하는 관행이 있나?"**
HLSL 기본 패킹이 column_major라서다. CPU(DirectXMath)가 row-major로 만든 바이트를 GPU가 열 우선으로 재해석하면 전치된 행렬이 되므로, 미리 한 번 전치해 상쇄시키는 것. 대안은 셰이더에 `row_major` 지정자를 붙여 해석 자체를 바꾸는 것이고, 어느 쪽이든 프로젝트 전체가 한 가지로 통일돼야 한다.

**"픽셀 중심이 왜 (x+0.5, y+0.5)인가?"**
픽셀은 점이 아니라 [x, x+1)×[y, y+1) 면적이고, 래스터라이저는 그 **중심**이 삼각형 안에 있는지로 커버리지를 판정한다. 중심을 정수로 잡으면 인접 픽셀과의 경계가 모호해지고 텍셀 중심 규약과도 어긋난다. D3D9는 이 규약이 달라 풀스크린 쿼드에 반 픽셀 오프셋을 수동 보정해야 했던 역사가 있고, D3D10+에서 0.5 중심으로 정리됐다.

**"UV를 화면 공간에서 그냥 선형 보간하면 왜 안 되나?"**
원근 투영은 비선형이라, 3D에서 균일한 간격이 화면에서는 균일하지 않다(먼 쪽이 압축됨). 화면 선형 보간(affine mapping)을 쓰면 바닥 텍스처가 휘어 보인다 — PS1 텍스처 워블이 정확히 이 아티팩트다. 올바른 방법은 `attr/w`와 `1/w`를 화면 선형으로 보간한 뒤 픽셀에서 `(attr/w)/(1/w)`로 복원하는 원근 보정 보간(perspective-correct interpolation)이고, 현대 GPU는 이를 하드웨어로 한다(HLSL `noperspective` 지정자가 예외를 만드는 스위치).

**"FOV를 키우면 무슨 일이 생기나?"**
`cot(fovY/2)`가 줄어 같은 화면에 더 넓은 각이 들어온다. 90°를 넘어가면 가장자리 왜곡(직선 보존은 되지만 각도·면적 왜곡)이 급격해진다 — 평면 투영(gnomonic projection)의 본질적 한계다. 성능 관점으론 보이는 지오메트리가 늘어 드로우·오버드로우가 증가한다. 또 aspect를 fovY에서 유도하므로, 해상도 비율이 바뀌면 수평 시야가 함께 변한다(Hor+ 방식).

**"직교 투영에서도 z-fighting이 나나?"**
난다. 직교는 깊이가 선형이라 원거리 편중 문제는 없지만, **동일 평면(co-planar) 지오메트리**는 정밀도 분포와 무관하게 양자화·보간 오차로 싸운다. 데칼처럼 의도적으로 겹치는 표면은 정밀도 개선이 아니라 depth bias, 폴리곤 오프셋, 또는 아예 깊이 테스트를 다르게 하는 렌더 순서 설계로 푸는 문제다.

**"w-buffer라는 것도 있었다던데?"**
초기 하드웨어(D3D9 시절 일부)가 지원했던, 1/z 대신 view z(=w')를 선형 저장하는 방식. 정밀도 분포는 균일해서 좋지만, 화면 공간에서 **1/w는 선형 보간 가능해도 w 자체는 아니라서** 래스터라이저 구현이 비싸고, 계층적 Z 같은 최적화와도 안 맞아 도태됐다. 지금은 같은 목표를 reversed-Z + D32_FLOAT가 사실상 공짜로 달성한다.

**"GL 코드를 DX로 이식할 때 이 파이프라인에서 뭐가 깨지나?"**
(1) 투영 행렬의 z 매핑: GL은 [-1,1] 출력이라 DX에서 near 절반이 잘리거나 정밀도 절반 낭비. (2) 텍스처/화면 y 원점 차이로 렌더타겟 상하 반전. (3) 규약: GL 문헌은 column-vector·column-major라 곱 순서와 전치를 재검토해야 한다. (4) LH/RH가 다르면 winding이 뒤집혀 컬링이 반대로 먹는다. "행렬 하나만 이식해도 z행, y부호, 전치, winding 네 가지를 점검한다"고 답하면 실전 경험으로 들린다.

## ④ Winters에서의 적용

### World 단계 — TransformComponent와 S·R·T의 실물

`Engine/Public/ECS/Components/TransformComponent.h:15`의 컴포넌트가 Local 데이터(위치·오일러 회전·스케일)와 캐시 행렬을 들고, 주석이 규약을 성문화한다 — `m_LocalMatrix{}; //Scale Rot Translation`(22행), `m_WorldMatrix{}; //parent world * local`(23행). 실제 합성은 `Engine/Private/ECS/Systems/TransformSystem.cpp:82`에서: 오일러를 `XMQuaternionRotationRollPitchYaw`로 쿼터니언화(83행)한 뒤 `XMMatrixAffineTransformation(scale, zero, rot, pos)`(88~90행)로 S·R·T를 한 번에 만든다(회전 원점이 0이므로 정확히 `S·R·T`). 계층은 110행 `worldMat = local * parent` — row-vector 규약에서 "자식 로컬 먼저, 부모 나중"의 코드 그대로다. 더티 플래그(`m_bLocalDirty`/`m_bWorldDirty`)로 변경된 서브트리만 재계산하는 것도 "행렬 곱은 싸지 않다"는 실전 감각의 증거로 쓸 수 있다.

### View·Projection 단계 — LH 통일과 곱 순서

`Engine/Include/WintersMath.h:281`의 팩토리 세 개가 좌표계 규약을 한 곳에 못 박는다: `LookAt = XMMatrixLookAtLH`(283행), `Perspective = XMMatrixPerspectiveFovLH`(288행), `Orthographic = XMMatrixOrthographicLH`(293행) — 전부 LH, DirectX z∈[0,1] 계열이다. 카메라는 `Engine/Private/Renderer/CCamera.cpp:92`의 `RecalcView()`가 매 프레임 LookAt으로 뷰를 재구성하고, `GetViewProjection()`(40~43행)은 `m_ViewMatrix * m_ProjMatrix` — row-vector 규약이라 View가 왼쪽이다. 재미있는 대목은 53~63행: WASD 이동을 위해 **뷰 행렬의 역행렬(=카메라 월드 행렬)의 1행에서 right, 3행에서 look**을 뽑는다. "row-vector 규약에서 월드 행렬의 행 = 그 좌표계의 기저"라는 이 챕터의 명제가 코드로 검증되는 지점이다. 실전 파라미터는 `Client/Public/DynamicCamera.h:13`에 있다 — fovY 60°, aspect 1280/720, **near 1.0, far 1000**. near를 0.01 같은 값이 아니라 1로 잡은 것 자체가 깊이 정밀도 절의 결론과 일치하는 선택이다.

### cbuffer 규약 — 전치 대신 row_major

Winters는 "CPU 전치" 대신 "셰이더 `row_major` 선언" 노선이다. `Shaders/Mesh3D.hlsl:11`의 `row_major matrix g_matViewProj`, 25~26행의 `row_major matrix g_matWorld / g_matWorldInvTranspose`가 그 선언이고, 업로드 쪽 `Engine/Private/Renderer/ModelRenderer.cpp:314`는 `data.world = matWorld.m`으로 **전치 없이 raw 복사**한다 — 한쪽에서만 전치를 소비한다는 규칙의 실물. VS는 `Shaders/Mesh3D.hlsl:146`에서 `mul(float4(input.vPosition, 1.0f), g_matWorld)` → 147행 `mul(worldPos, g_matViewProj)` — `mul(행벡터, 행렬)` 순서로 규약이 끝까지 일관된다. 이 규약은 `.md/architecture/WINTERS_ENGINE_CONVENTIONS.md:276`("HLSL 행렬은 `row_major` 키워드 필수")에 팀 규칙으로 박제돼 있는데, 그렇게 된 이유가 있다 — 초기에 `row_major`를 빠뜨렸을 때 뷰 행렬이 전치돼 평행이동이 w' 계산에 섞였고, **카메라가 움직일 때마다 화면이 비대칭으로 클리핑되는** 증상으로 나타났다. "전치 버그는 회전만 있을 땐 숨어 있다가 이동에서 터진다"는 함정의 실사례다.

### 노멀 — 역전치의 실물

`Engine/Private/Renderer/ModelRenderer.cpp:316`이 오브젝트 상수를 채울 때 `XMMatrixTranspose(XMMatrixInverse(nullptr, worldMatrix))`로 정확히 `(M⁻¹)ᵀ`를 만들어 올리고, `Shaders/Mesh3D.hlsl:148`이 `normalize(mul(float4(input.vNormal, 0.0f), g_matWorldInvTranspose).xyz)`로 소비한다 — **w=0**(방향이므로 평행이동 차단)과 **normalize**(역전치는 길이 비보존)까지 유도의 따름 규칙 세 개가 한 줄에 다 들어 있다. CPU 쪽 방향 변환도 `Engine/Include/WintersMath.h:305`의 `TransformDirection`이 `XMVector3TransformNormal`(w=0 경로)을 쓰도록 타입이 갈라져 있다.

### 클리핑 부등식을 CPU에서 재사용 — 프러스텀 컬링

`Engine/Private/Resource/Model.cpp:433`의 `IsBoundsVisibleInClip`은 이 챕터의 클리핑 절을 CPU 코드로 옮긴 것이다. 로컬 AABB 8코너를 `matLocalToClip`으로 clip space에 보내고(461~463행), **나눗셈 없이** 동차 부등식으로 판정한다: `c.x >= -c.w / c.x <= c.w`(좌우, 469~470행), near/far는 DX 규약 그대로 `c.z >= 0.f / c.z <= c.w`(473~474행). `c.w <= 0.0001f`면 near 평면에 걸친 것으로 보고 보수적으로 "보임" 처리하며(467행, 488~489행), NDC 좌표가 필요한 미세 판정만 `c.w > 0.0001f`일 때 `invW`를 곱해 내려간다(476~480행) — "w≈0에서 나누지 마라, 클리핑 판정은 clip space에서"라는 원칙이 방어 코드로 구현된 형태다.

### 픽셀에서 거꾸로 — 피킹은 파이프라인의 역주행

LoL식 클릭 조작의 마우스 피킹이 이 챕터 전체의 **역함수**다. `Engine/Private/Platform/CInput.cpp:58`의 `GetMouseWorldRay`: 픽셀→NDC로 `ndcX = 2mx/W - 1`, **`ndcY = 1 - 2my/H`**(63~64행 — 뷰포트 y 뒤집기의 역), `invVP = (View·Proj)⁻¹`(69행)로 NDC의 **z=0(near)과 z=1(far)** 두 점을 월드로 언프로젝트(72~73행 — DX z∈[0,1] 규약이 여기서도 전제)해 레이를 만든다. `XMVector3TransformCoord`가 내부에서 w 나눗셈을 수행하므로 역방향의 원근 나눗셈까지 포함된 완전한 역변환이다. 이 레이가 `GetMouseGroundPos`(85~92행)의 y=0 평면 교차(이동 클릭), `Client/Private/Scene/GameplayQuery.cpp:18`의 `RayVsCylinder`(챔피언 호버 타겟팅)로 이어진다. "파이프라인을 정방향으로 유도할 수 있으면 역방향(피킹)은 공짜"라는 걸 본인 코드로 증명할 수 있는 지점.

### 깊이 버퍼 현황과 z-fighting 실사례 — 정직한 로드맵 답변

현재 Winters의 깊이 설정: 뷰포트는 `Engine/Private/RHI/DX11/CDX11Device.cpp:772`에서 `MinDepth 0 / MaxDepth 1`(DX 표준), 깊이 버퍼 포맷은 891행 `DXGI_FORMAT_D24_UNORM_S8_UINT`(24bit 고정소수점 + 8bit 스텐실), 비교 함수는 `Engine/Private/RHI/DX11/DX11Pipeline.cpp:44`의 `D3D11_COMPARISON_LESS` — 즉 **표준 Z + UNORM 조합**이다. 소환사의 협곡 맵을 처음 올릴 때 겹친 지오메트리에서 z-fighting을 실제로 겪었고, 당시 해법은 depth bias 튜닝이 아니라 **겹쳐 익스포트된 메시 데이터 정리 + near/far 정돈**이었다(상세는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md) §④-6). "동일 평면 중첩은 정밀도 문제가 아니라 데이터 문제"라는 이 챕터 결론과 일치하는 경험이다.

reversed-Z는 아직 도입 전인데, 이것을 그대로 면접 답으로 쓸 수 있다: *"현재는 D24_UNORM + LESS라 reversed-Z의 수혜가 없는 구성이다. 도입한다면 (1) 깊이 포맷을 D32_FLOAT(스텐실 필요 시 D32_FLOAT_S8X24)로, (2) 투영 행렬 z행을 near↔far 스왑으로 재유도, (3) 클리어값 1→0, (4) 비교 함수 LESS→GREATER, (5) 깊이를 읽는 모든 셰이더(소프트 파티클, 재구성)와 depth bias 부호를 일괄 점검 — 이 다섯이 한 커밋으로 움직여야 하는 세트다. far가 1000인 현 스케일에선 체감이 작지만, 오픈 필드로 far를 늘리는 순간 UNORM 표준 Z가 먼저 한계에 닿으므로 그 시점이 도입 타이밍이다."* — 개념 이해 + 자기 엔진의 현재 좌표 + 마이그레이션 비용까지 한 호흡에 담는 답변이 된다.
