# 직접 만들며 배우는 OS — 6주 실습 로드맵

> OS 지식은 책으로 읽으면 "용어의 지도"가 생기지만, 부트섹터부터 시스템 콜까지 직접 짜보면 "왜 그 설계여야만 했는가"가 몸에 남는다. 면접관이 꼬리질문으로 파고들 때 버티는 것은 지도가 아니라 몸에 남은 쪽이다.

---

## ① 목표와 얻는 것

### 완주 시 상태

6주 뒤 당신의 QEMU 창에는 **직접 짠 커널이 부팅되고, 키보드 입력을 받고, 타이머로 태스크를 선점 전환하고, 유저 모드 프로그램이 `int 0x80`으로 커널에 문자 출력을 요청하는** 미니 셸이 떠 있다. 코드 전체가 수천 줄을 넘지 않는다. 그런데 이 수천 줄이 면접에서 바꾸는 것은:

| 완주 전 | 완주 후 |
|---|---|
| "페이지 폴트는 페이지가 메모리에 없을 때 발생하는 예외입니다" (교과서 암송) | "제가 페이지 테이블을 직접 구성해서 페이징을 켜봤는데, CR3에 디렉터리 주소를 넣고 CR0의 PG 비트를 세우는 순간 그 다음 명령어 페치부터 주소 변환이 걸리기 때문에, 커널 영역을 미리 항등 매핑하지 않으면 즉시 폴트가 났습니다" (경험 서사) |
| "컨텍스트 스위치는 레지스터를 저장하고 복원하는 것입니다" | "직접 어셈블리로 짜보니 callee-saved 레지스터 4개와 스택 포인터만 바꾸면 되더라 — 진짜 비용은 레지스터 저장이 아니라 캐시/TLB 오염이라는 걸 그때 체감했습니다" |
| "시스템 콜은 커널 기능을 호출하는 인터페이스입니다" | "IDT 게이트의 DPL을 3으로 열어야 유저 모드에서 int를 걸 수 있고, 링 전환 때 TSS에 적어둔 커널 스택으로 갈아탄다는 것까지 직접 배선해봤습니다" |

왼쪽 답은 누구나 한다. 오른쪽 답은 "만들어본 사람"만 한다. **이 로드맵의 목표는 지식 총량이 아니라, OS 챕터 전체의 답변을 오른쪽 열의 문장으로 바꾸는 것이다.**

### 왜 "직접 만들기"인가 — 시간 대비 효율의 역설

6주는 짧지 않다. 그 시간에 OSTEP을 두 번 읽을 수도 있다. 그런데도 만들기를 권하는 이유는 **OS의 어려움이 개념이 아니라 "순서와 전제"에 있기 때문**이다. 페이징 자체는 2단계 배열 조회일 뿐이다. 어려운 것은 "페이징을 켜기 전에 무엇이 준비돼 있어야 하는가"(항등 매핑, 유효한 CR3), "인터럽트를 켜기 전에 무엇이 준비돼 있어야 하는가"(IDT, PIC 리매핑 — 안 하면 타이머 IRQ가 CPU 예외 벡터와 충돌한다)다. 책은 이 의존성 그래프를 순서대로 서술해주지만, 직접 만들면 **순서를 어겼을 때 시스템이 어떻게 죽는지**(트리플 폴트 → QEMU 무한 리부팅)를 보게 된다. 면접 꼬리질문은 대부분 이 의존성 그래프를 찌른다.

또 하나 — 당신은 이미 Winters에서 IOCP 서버와 JobSystem을 만들었다. 이 실습이 끝나면 그것들이 "커널이 해주던 일을 유저 공간에서 재현한 것"으로 재해석된다(⑤절). 별개의 프로젝트 두 개가 아니라 **하나의 이야기**가 된다.

## ② 환경 세팅 — Windows 11 기준

### 선택지와 트레이드오프

| 트랙 | 언어 | 장점 | 단점 |
|---|---|---|---|
| **C + NASM (본 로드맵)** | C, x86 어셈블리 | 면접 대상인 x86 개념(GDT/IDT/링)을 그대로 만짐. 자료 최다(OSDev Wiki) | 툴체인 준비가 번거로움 |
| **Rust 트랙** | Rust | 부트로더를 크레이트가 대신해줘 커널 본론 진입이 빠름. 빌드 시스템이 편함 | 부트/리얼 모드 경험이 생략됨. C++ 면접과의 언어 접점 약함 |

**권장: C + NASM 트랙.** 당신의 무기는 C++이고, 면접관도 C/x86 기준으로 묻는다. Rust 트랙(Philipp Oppermann의 *Writing an OS in Rust*, os.phil-opp.com)은 W1~W2를 건너뛰고 싶거나, 완주 후 2회차로 도는 용도로 추천한다 — 같은 커널을 두 언어로 지어보면 "언어가 가려주는 것"이 보인다.

### WSL2 + 크로스 컴파일러 + QEMU

호스트 Windows에 MSVC가 있지만 쓰지 않는다. **커널은 "운영체제 없이 도는 코드"라서, Windows용 실행 파일(PE, CRT 링크)을 만드는 기본 툴체인과 목적이 어긋난다.** 표준 경로는 ELF를 뱉는 크로스 컴파일러다.

```powershell
# 1) Windows 11 PowerShell (관리자) — WSL2 + Ubuntu
wsl --install -d Ubuntu
```

```bash
# 2) Ubuntu 안에서 — 기본 도구
sudo apt update
sudo apt install -y build-essential nasm qemu-system-x86 gdb curl xorriso grub-pc-bin
```

Windows 11의 WSLg 덕에 `qemu-system-x86_64`의 GUI 창이 별도 X 서버 설치 없이 그대로 뜬다. 이게 Windows 11에서 WSL2를 권하는 실질적 이유다.

**크로스 컴파일러 — 두 가지 길:**

```bash
# (A) 기성 i686-elf 툴체인 다운로드 — 빠른 길 (권장)
# OSDev Wiki "GCC Cross-Compiler" 문서의 prebuilt 링크에서 리눅스용 i686-elf 바이너리를 받아 PATH에 추가
# 확인:
i686-elf-gcc --version

# (B) 소스 빌드 — binutils + gcc 를 --target=i686-elf 로 직접 빌드 (30분~1시간)
# OSDev Wiki "GCC Cross-Compiler" 절차 그대로. 한 번 해두면 면접에서
# "크로스 컴파일러가 왜 필요한가"에 실경험으로 답할 수 있다
```

**왜 호스트 gcc에 `-m32 -ffreestanding -nostdlib`를 주는 것으로는 부족한가?** 돌아는 간다. 하지만 호스트 gcc는 리눅스 유저 공간을 전제로 한 기본값(PIE, 스택 보호기, libc 헤더)을 곳곳에 심어놔서, 플래그를 하나 빼먹을 때마다 정체불명의 링크 에러나 런타임 폭사를 만난다. 크로스 컴파일러는 "아무 OS도 전제하지 않는다"가 기본값이라 이 함정군이 통째로 사라진다. **freestanding 환경이라는 개념을 도구 수준에서 강제하는 것** — 이것 자체가 배움이다.

**빌드-실행 루프 확인:**

```bash
nasm -f bin boot.asm -o boot.bin
qemu-system-x86_64 -drive format=raw,file=boot.bin
```

이 두 줄이 앞으로 6주간의 "컴파일-실행" 루프다. 실기계 재부팅이 아니라 에뮬레이터라서 루프가 수 초로 짧다 — OS 개발이 과거보다 압도적으로 쉬워진 단 하나의 이유가 QEMU다.

**디버깅 준비(중요):** QEMU는 GDB 스텁을 내장한다. `qemu-system-x86_64 -s -S ...`로 띄우면 1234 포트에서 GDB 접속을 기다리며 정지 상태로 시작한다. `gdb`에서 `target remote :1234`로 붙어 브레이크포인트를 걸 수 있다. **W3부터는 printf가 없는 세계에서 이것이 유일한 디버거다.** 트리플 폴트로 무한 리부팅할 때는 `-d int -no-reboot` 옵션으로 인터럽트 로그를 뜯는다.

여기에 하나 더 — **시리얼 출력을 W2에서 미리 뚫어두면 이후 4주가 편해진다.** QEMU에 `-serial stdio`를 주고 커널에서 COM1 포트(0x3F8)에 바이트를 쓰면, 출력이 QEMU를 띄운 터미널에 그대로 흐른다. VGA 25줄에 갇히지 않는 로그 채널이 생기는 것이다. Winters에서 렌더 버그를 잡을 때 "증상 튜닝 전에 관측 수단부터"를 규율로 삼았던 것과 정확히 같은 원리다 — **관측 불가능한 시스템은 디버깅이 아니라 기도가 된다.**

### Rust 트랙을 택한다면 — 세팅만 요약

본 로드맵은 C 트랙 기준이지만, Rust 트랙의 진입 커맨드도 남겨둔다(완주 후 2회차용으로도 유효):

```bash
# WSL2 Ubuntu 안에서
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
rustup toolchain install nightly       # 커널 개발은 nightly 전용 기능이 필요
rustup component add rust-src llvm-tools-preview
cargo install bootimage                # 부트로더+커널을 부팅 디스크 이미지로 묶는 도구
# 이후는 os.phil-opp.com 의 "A Freestanding Rust Binary" 장부터 순서대로
```

Rust 트랙의 커리큘럼 대응: 블로그의 "A Freestanding Rust Binary"~"VGA Text Mode"가 본 로드맵의 W1~W2 축약판(부트로더는 크레이트가 대신), "CPU Exceptions"~"Hardware Interrupts"가 W3, "Introduction to Paging"~"Heap Allocation"이 W4, "Async/Await"는 W5의 대체물(선점 대신 협조적 태스크)이다. **W5의 "직접 짠 컨텍스트 스위치"와 W6의 링 전환 경험은 Rust 트랙에 없다** — C 트랙을 기본으로 권하는 결정적 이유다.

## ③ 주차별 커리큘럼

전체 여정의 지도. 각 주는 "CPU가 켜진 직후의 원시 상태"에서 "현대 OS의 최소 골격"까지 한 계단씩 오른다.

```
전원 ON
  │ BIOS (펌웨어가 하드웨어 초기화, 부팅 디스크 탐색)
  ▼
W1  부트섹터 512B @ 0x7C00        ← 16bit 리얼 모드, BIOS 서비스 사용 가능
  ▼
W2  GDT → 보호 모드 32bit         ← BIOS와 작별, 이제 모든 것을 직접
  ▼
W3  IDT + PIC + 키보드/타이머     ← 외부 세계(하드웨어 이벤트)와 연결
  ▼
W4  물리 프레임 + 페이징          ← 주소가 "가상"이 되는 순간
  ▼
W5  TCB + 컨텍스트 스위치 + RR    ← "동시에 여러 일"이라는 환상 제조
  ▼
W6  int 0x80 + 링3 + 미니 셸      ← 커널/유저의 경계선 완성 = "OS"
```

### 운영 규율 — 6주를 실제로 완주하는 방법

로드맵이 실패하는 원인은 난이도가 아니라 운영이다. 세 가지 규칙:

1. **verify 기준을 먼저 쓰고 코딩한다.** 각 주의 "무엇이 화면에 보이면 성공"을 그 주 첫날 커밋 메시지에 박아라. 기준이 없으면 "GDT를 더 이해하고 나서"라는 무한 준비 모드에 빠진다. Winters에서 체득한 목표 주도 실행(goal-driven execution)을 그대로 적용하는 것이다 — "보호 모드 공부" 대신 "초록 글자가 0xB8000 직접 쓰기로 뜬다".
2. **주 단위로 git 태그를 끊는다** (`w1-boot`, `w2-pm`, ...). 커널 개발은 "어제는 됐는데 오늘 안 된다"가 일상이고, 그때 이등분 탐색(bisect)이 가능한 히스토리가 유일한 구명줄이다. 각 태그 시점은 곧 면접 데모 포인트이기도 하다.
3. **막히면 2시간 룰.** 같은 증상에 2시간 이상 갇히면 그 주의 "막힐 때 볼 자료"의 완성 코드와 내 코드를 diff한다. 커널 초기화 버그의 대부분은 "순서 하나, 비트 하나"라서, 개념 재학습보다 대조가 빠르다. 베끼는 게 아니라 **차이를 찾은 뒤 왜 그 차이가 치명적인지 설명할 수 있으면** 학습은 완료된 것이다.

주당 예산은 10~15시간을 상정한다(평일 저녁 2시간 + 주말 반나절). W2와 W4가 체감 난이도의 두 봉우리이니, 일정이 밀린다면 W3(인터럽트)의 예외 핸들러를 최소화하는 쪽으로 조절하고 봉우리를 지켜라.

### W1 — 부트섹터: CPU가 켜진 직후의 세계

- **목표**: 부팅이라는 과정에서 펌웨어와 내 코드의 책임 경계를 몸으로 확인한다.
- **구현물**: (1) BIOS 텔레타이프 출력(INT 0x10)으로 문자열을 찍는 512바이트 부트섹터, (2) INT 0x13으로 디스크 2번째 섹터를 메모리에 읽어와 점프하는 2단계 로더.
- **verify 기준**: QEMU 검은 화면에 내가 넣은 문자열이 뜨면 1단계 성공. 2번째 섹터에 넣어둔 다른 문자열이 이어서 뜨면 2단계 성공(= "512바이트 감옥"에서 탈출했다는 증거).
- **막힐 때 볼 자료**: OSDev Wiki "Boot Sequence" · Nick Blundell *Writing a Simple Operating System from Scratch*(무료 PDF, W1~W2 최고의 안내서) · cfenollosa/os-tutorial(GitHub) 초반부 · BIOS 인터럽트 상세는 Ralf Brown's Interrupt List.

**왜 이 설계인가.** BIOS는 부팅 가능한 디스크의 **첫 512바이트를 메모리 0x7C00에 복사하고 점프한다**. 그게 펌웨어 책임의 끝이다. 왜 512바이트인가 — 초기 플로피 디스크의 섹터 크기이고, 마지막 2바이트가 시그니처(0x55, 0xAA)여야 "부팅 가능"으로 인정된다. 이 시절 호환성이 지금까지 살아있는 것이 레거시 BIOS 부팅이다(현대 기계는 UEFI로 넘어갔지만, 개념 학습에는 단순한 BIOS 경로가 낫다).

이 시점의 CPU는 8086과 호환되는 **16비트 리얼 모드(real mode)**다. 보호도 페이징도 없고, 주소는 `세그먼트×16+오프셋`으로 계산되며, 대신 BIOS 인터럽트라는 "펌웨어 제공 표준 라이브러리"를 쓸 수 있다.

```nasm
[bits 16]
[org 0x7c00]            ; BIOS가 우리를 여기 놓는다 — 어셈블러에게 알려줌
    xor ax, ax
    mov ds, ax          ; lodsb 는 DS:SI 기준 — BIOS는 DS=0 을 보장하지 않으므로 직접 세팅
    mov si, msg
.print:
    lodsb               ; AL = [SI++]
    or al, al
    jz .halt
    mov ah, 0x0e        ; BIOS 텔레타이프 출력
    int 0x10
    jmp .print
.halt:
    hlt
    jmp .halt
msg: db "WINTERS OS", 0
times 510-($-$$) db 0   ; 510바이트까지 0으로 패딩
dw 0xaa55               ; 부트 시그니처
```

핵심 함정: `[org 0x7c00]`을 빼먹으면 문자열 주소가 0 기준으로 계산되어 쓰레기가 출력된다. **"내 코드가 메모리 어디에 놓이는가"와 "어셈블러가 주소를 뭐라고 믿는가"가 별개라는 것** — 링커 스크립트와 재배치의 본질을 512바이트 안에서 만나는 순간이다.

2단계 로더가 필요한 이유는 자명하다: 커널은 512바이트에 안 들어간다. INT 0x13(AH=0x02)으로 디스크의 다음 섹터들을 원하는 주소에 읽어오고 그리로 점프한다 — **모든 부트로더(GRUB 포함)가 하는 일의 최소형**이다.

### W2 — 보호 모드: BIOS와 작별하고 32비트로

- **목표**: 리얼 모드 → 보호 모드(protected mode) 전환의 3요소(GDT, A20, 파 점프)를 직접 배선하고, "이제부터 아무도 도와주지 않는다"는 상태를 확인한다.
- **구현물**: GDT 정의 + A20 활성화 + CR0.PE 세트 + 32비트 코드로 파 점프 + VGA 텍스트 버퍼(0xB8000) 직접 쓰기 함수. 이후 C 커널(`kmain`)을 크로스 컴파일해 로더에서 진입.
- **verify 기준**: BIOS 없이 — 즉 `int 0x10` 없이 — 0xB8000에 직접 쓴 문자가 화면에 뜨면 성공. 색상 속성 바이트를 바꿔 초록 글자가 나오면 확실한 증거(BIOS 출력은 이렇게 색을 못 바꾼다).
- **막힐 때 볼 자료**: OSDev Wiki "GDT Tutorial", "Protected Mode", "A20 Line" · *The little book about OS development*(littleosbook) · os-tutorial 중반부.

**왜 이 설계인가.** 보호 모드는 80286이 도입하고 80386이 32비트로 확장한 "OS를 만들 수 있는 모드"다 — 특권 레벨(링)은 286부터, 32비트 주소와 나중의 페이징은 386부터 열린다. 그런데 전환 의식이 3단계나 되는 이유는 전부 **하위 호환성의 역사**다:

1. **GDT(Global Descriptor Table)**: 보호 모드에서 세그먼트 레지스터는 더 이상 "주소×16"이 아니라 **이 테이블의 인덱스**다. 각 엔트리가 세그먼트의 시작/크기/권한을 기술한다. 현대 OS는 세그멘테이션을 사실상 쓰지 않으므로, 우리도 "베이스 0, 리밋 최대"인 코드/데이터 세그먼트 두 개만 만든다(플랫 모델). **면접 포인트: "x86에 세그먼트가 있는데 왜 아무도 안 쓰나요" → 페이징이 더 유연한 보호/변환을 제공해서, 세그먼트는 '전부 0부터 끝까지'로 무력화하고 페이징만 쓴다.**
2. **A20 라인**: 8086은 주소가 1MB를 넘으면 되감겼고(wrap-around), 이를 이용한 프로그램들이 있어서 IBM은 21번째 주소선(A20)을 **키보드 컨트롤러를 거쳐** 끄고 켤 수 있게 만들었다. 40년 전 호환성 땜질이 지금도 부팅 코드에 화석으로 남아 있다. (QEMU에서는 이미 켜져 있는 경우가 많지만, 켜는 코드를 넣고 확인 루틴으로 검증하는 것까지가 실습이다.)
3. **CR0.PE 세트 + 파 점프(far jump)**: 제어 레지스터 CR0의 PE 비트를 올리는 순간 보호 모드다. 직후 `jmp CODE_SEG:label` 파 점프가 필수인 이유 — CS 레지스터를 새 GDT 기준 값으로 다시 로드하고, 리얼 모드 기준으로 미리 해석해둔 파이프라인을 비우기 위해서다.

전환하고 나면 BIOS 인터럽트를 잃는다(BIOS는 16비트 리얼 모드 코드다). 이제 화면 출력은 **VGA 텍스트 버퍼를 직접 쓰는 것**: 0xB8000부터 80×25칸, 한 칸 2바이트(문자, 속성).

```c
volatile unsigned short* vga = (unsigned short*)0xB8000;
vga[0] = (0x0A << 8) | 'W';   // 속성 0x0A = 검은 배경에 밝은 초록
```

"화면에 글자를 찍는다 = 특정 물리 주소에 바이트를 쓴다"— **메모리 맵 I/O(memory-mapped I/O)의 첫 체험**이다. Winters에서 GPU에 상수 버퍼를 밀어 넣던 감각과 같은 계열의 개념이 여기서 시작된다.

### W3 — 인터럽트: 하드웨어가 CPU에게 말을 거는 방법

- **목표**: "폴링이 아니라 인터럽트"라는 OS 설계의 근본 선택을 IDT/PIC/핸들러 배선으로 직접 구현한다.
- **구현물**: (1) IDT(Interrupt Descriptor Table) 256엔트리 + 로드, (2) CPU 예외 핸들러(0~31번, 최소한 divide error·GP fault·page fault는 번호와 에러 코드를 화면에 출력), (3) PIC 리매핑(IRQ0~15 → 벡터 0x20~0x2F), (4) 키보드 IRQ1 핸들러(스캔코드 → ASCII 화면 에코), (5) PIT 타이머 IRQ0(초당 N회 틱 카운터 증가, 화면 구석에 출력).
- **verify 기준**: ① 키보드를 누르면 그 문자가 화면에 찍힌다(폴링 루프 없이!). ② 화면 구석의 틱 숫자가 혼자 올라간다. ③ 일부러 `1/0`을 실행하면 리부팅이 아니라 "EXCEPTION 0" 메시지가 뜬다 — 예외를 "잡았다"는 증거.
- **막힐 때 볼 자료**: OSDev Wiki "Interrupts", "8259 PIC", "IDT" · littleosbook 인터럽트 장 · (주의: 검색에 자주 걸리는 JamesM 튜토리얼은 유명하지만 알려진 버그가 있다 — OSDev Wiki의 "James Molloy's Tutorial Known Bugs" 문서와 교차 확인할 것).

**왜 이 설계인가.** 키보드 입력을 받는 두 가지 방법: CPU가 계속 물어보거나(폴링), 장치가 CPU를 부르거나(인터럽트). 폴링은 CPU 시간을 태우고, 인터럽트는 배선이 복잡하다. OS가 인터럽트를 택한 이유는 **CPU 시간이 세상에서 가장 비싼 자원**이기 때문 — 그리고 이 선택이 W5의 선점 스케줄링과 W6의 시스템 콜까지 전부 지탱한다. **인터럽트는 이 커리큘럼 후반부 전체의 기반 시설이다.**

배선의 핵심 두 가지:

1. **IDT**: "N번 이벤트가 오면 이 주소의 핸들러로 점프하라"는 256칸 점프 테이블. 0~31번은 CPU가 예약한 예외(divide error, page fault 등)다.
2. **PIC(8259) 리매핑**: 구형 PIC의 기본 설정은 IRQ0~7을 벡터 8~15에 꽂는데, **이는 CPU 예외 번호와 정면 충돌한다**(예: 타이머 IRQ0이 더블 폴트 8번으로 들어옴). IBM PC 시절의 설계 실수가 표준이 되어버린 것으로, 모든 OS가 부팅 때 PIC에 초기화 커맨드를 보내 0x20~0x2F로 옮긴다. "왜 리매핑하나요"는 이 역사를 아는지 묻는 질문이다. 핸들러 끝에서 EOI(End Of Interrupt)를 PIC에 보내지 않으면 같은 IRQ가 다시는 오지 않는다 — **이 실수를 한 번 겪어보는 것 자체가 커리큘럼이다.**

```
[키 누름] → 키보드 컨트롤러 → PIC(IRQ1) → CPU INT 핀
   → CPU: 현재 명령 완료, EFLAGS/CS/EIP 스택에 push
   → IDT[0x21] 핸들러로 점프 (레지스터 저장은 핸들러 몫)
   → 포트 0x60에서 스캔코드 read → 화면 에코 → PIC에 EOI → iret
```

이 그림이 **모든** 인터럽트/예외/시스템 콜의 공통 골격이다. W6의 `int 0x80`도 같은 파이프라인을 소프트웨어가 의도적으로 발화하는 것뿐이다.

### W4 — 메모리: 주소가 "가상"이 되는 순간

- **목표**: 물리 메모리 관리(프레임 할당)와 가상 메모리(페이징)를 분리해 이해하고, 페이징이 켜지는 순간 세계가 어떻게 바뀌는지 확인한다.
- **구현물**: (1) 물리 프레임 비트맵 할당자(4KB 프레임당 1비트, alloc/free), (2) 페이지 디렉터리 + 페이지 테이블 구성(커널 영역 항등 매핑), (3) CR3 로드 + CR0.PG 세트로 페이징 활성화, (4) 페이지 폴트 핸들러(CR2의 폴트 주소와 에러 코드 출력), (5) 프레임 할당자 위에 얹은 간단한 kmalloc(범프 또는 free-list).
- **verify 기준**: ① 페이징을 켜도 커널이 아무렇지 않게 계속 돈다(항등 매핑 성공). ② 매핑 안 한 주소(예: 아주 높은 주소)를 일부러 읽으면 "PAGE FAULT at addr=..., err=..."가 내 핸들러에서 출력된다. ③ 그 주소에 새 프레임을 매핑해주고 리턴하면 그 접근이 성공한다 — **디맨드 페이징의 최소 재현.**
- **막힐 때 볼 자료**: OSDev Wiki "Paging", "Setting Up Paging" · OSTEP 13~20장(주소 공간·페이징·TLB) · 최종 심판은 Intel SDM Vol.3의 페이징 장.

**왜 이 설계인가.** 두 계층을 반드시 분리해서 만들어야 한다:

- **물리 프레임 할당자**: "어느 4KB 물리 조각이 비어 있나"만 답하는 장부. 비트맵을 쓰는 이유는 단순함과 밀도다 — 4GB 물리 메모리 전체를 관리하는 데 128KB면 된다(리눅스는 buddy 시스템으로 연속 할당을 최적화하지만, 그건 확장 주제로 남긴다).
- **페이지 테이블**: "가상 주소 X를 물리 프레임 Y로 번역하라"는 매핑. 32비트 x86은 2단계다 — CR3가 가리키는 **페이지 디렉터리**(1024엔트리)의 엔트리가 **페이지 테이블**(1024엔트리)을 가리키고, 그 엔트리가 4KB 프레임을 가리킨다. 왜 평평한 1단계 배열이 아닌가: 4GB/4KB = 100만 엔트리를 프로세스마다 통째로 갖는 건 낭비고, 2단계면 **안 쓰는 주소 영역의 테이블을 아예 안 만들 수 있다**. 희소한 주소 공간을 희소한 자료구조로 — 64비트의 4단계 테이블도 같은 원리의 연장이다.

가장 배움이 큰 순간은 **켜는 순간**이다. CR0.PG를 세운 직후의 명령어 페치부터 이미 주소 변환이 걸린다. 그러니 지금 실행 중인 커널 코드의 주소가 "가상==물리"로 매핑돼 있지 않으면(항등 매핑, identity mapping) 즉시 폴트 → 폴트 핸들러 주소도 번역 불가 → 더블 폴트 → 트리플 폴트 → 리셋. **QEMU가 말없이 무한 리부팅하는 것을 처음 보는 주가 이번 주다.** `-d int -no-reboot`로 로그를 뜯어 원인을 찾는 것까지가 실습이다.

페이지 폴트 핸들러에서 확인할 것: 폴트를 일으킨 가상 주소는 CR2에 담겨 오고, 에러 코드의 비트들이 "매핑 부재였나/권한 위반이었나, 읽기였나/쓰기였나, 유저였나/커널이었나"를 알려준다. **이 에러 코드를 해석해서 '매핑만 없었으면 프레임 하나 물려주고 재시도'를 구현하는 순간, 스왑·CoW·mmap이 전부 "페이지 폴트 핸들러의 정책 분기"라는 것이 보인다.** 면접에서 페이지 폴트 질문이 나오면 이 문장으로 시작하면 된다.

### W5 — 멀티태스킹: "동시에"라는 환상의 제조

- **목표**: 컨텍스트 스위치를 어셈블리로 직접 작성해 "태스크 전환의 실체 = 스택 포인터 교체"임을 확인하고, 타이머 인터럽트로 선점형 라운드 로빈(round-robin)을 완성한다.
- **구현물**: (1) TCB(Task Control Block) 구조체 — 최소 구성은 저장된 커널 스택 포인터 + 상태 + 다음 태스크 링크, (2) 새 태스크의 초기 스택을 "이미 한 번 스위치돼 나간 것처럼" 위조하는 태스크 생성 함수, (3) 컨텍스트 스위치 어셈블리, (4) 타이머 IRQ0 핸들러에서 스케줄러 호출 → RR 선점.
- **verify 기준**: 태스크 A는 화면 왼쪽에 'A' 카운터를, 태스크 B는 오른쪽에 'B' 카운터를 찍게 한다. ① 협조적 버전: 서로 yield를 호출할 때만 번갈아 증가. ② 선점 버전: **yield 호출을 전부 지웠는데도** 두 카운터가 함께 증가하면 성공 — 타이머가 강제로 뺏고 있다는 증거다.
- **막힐 때 볼 자료**: OSDev Wiki "Brendan's Multi-tasking Tutorial"(협조적→선점 순서로 쌓는 구성이 이 커리큘럼과 같다), "Context Switching" · OSTEP 6장(Limited Direct Execution)과 스케줄링 장들.

**왜 이 설계인가.** 컨텍스트 스위치의 신화를 깨는 주다. 직접 짜보면 이렇게 짧다:

```nasm
; void switch_context(uint32_t** old_sp, uint32_t* new_sp)  — cdecl
global switch_context
switch_context:
    mov eax, [esp+4]      ; old_sp 를 저장할 위치
    mov edx, [esp+8]      ; new_sp
    push ebp              ; callee-saved 4개만 저장
    push ebx
    push esi
    push edi
    mov [eax], esp        ; 현재 태스크: "내 스택 여기까지 썼음"
    mov esp, edx          ; ★ 이 한 줄이 태스크 전환의 실체
    pop edi
    pop esi
    pop ebx
    pop ebp
    ret                   ; 새 태스크가 예전에 push해둔 복귀 주소로
```

왜 레지스터 4개만 저장하나 — cdecl 호출 규약(calling convention)에서 EAX/ECX/EDX는 호출자가 이미 포기한 레지스터(caller-saved)라서, **함수 호출의 형태를 띠는 한** callee-saved만 지키면 된다. 컨텍스트 스위치가 "함수 호출인 척하는 스택 바꿔치기"라는 것, 그리고 `ret`가 "새 태스크의 스택에 적혀 있던 주소"로 돌아간다는 것 — 이 두 문장이 이해되면 이번 주는 끝났다. (인터럽트 문맥에서의 선점은 전체 레지스터가 이미 인터럽트 진입부에서 저장돼 있으므로 결이 같다.)

새 태스크 생성이 은근한 고비다: 아직 한 번도 실행된 적 없는 태스크는 "저장된 스택"이 없다. 그래서 **처음부터 위조한다** — 새 스택에 [시작 함수 주소, 가짜 callee-saved 4개]를 미리 push해두면, 첫 스위치의 `pop×4 + ret`가 자연스럽게 시작 함수로 "복귀"한다. 이 트릭을 스스로 이해하고 나면 Windows의 `CreateFiber`나 코루틴 라이브러리가 내부에서 뭘 하는지 감이 잡힌다.

선점(preemption)은 여기에 W3의 타이머를 꽂는 것이다: PIT 틱마다 스케줄러가 다음 태스크를 골라 스위치. **"멀티태스킹 = 컨텍스트 스위치(메커니즘) + 타이머 인터럽트(강제력) + 스케줄러(정책)"의 3분해**가 면접 답변의 뼈대가 된다. 그리고 직접 재보면 레지스터 저장/복원 자체는 수십 사이클 수준으로 저렴하다 — 실제 비용의 본체는 전환 후의 캐시/TLB 미스, 그리고 (유저 프로세스라면) 주소 공간 전환이라는 점을 경험으로 말할 수 있게 된다.

### W6 — 시스템 콜: 커널과 유저의 경계선 긋기

- **목표**: 특권 분리(링 0/링 3)를 실제로 배선해 "시스템 콜이 함수 호출과 무엇이 다른가"를 종결짓는다.
- **구현물**: (1) IDT에 `int 0x80` 게이트를 **DPL=3**으로 등록(유저 코드가 발화 가능하도록), (2) EAX=콜 번호 규약으로 `sys_write`(문자열 출력)·`sys_yield`(스케줄러 양보) 디스패치, (3) TSS(Task State Segment)에 커널 스택(esp0) 등록 + GDT에 유저 코드/데이터 세그먼트(DPL 3) 추가 + `iret` 트릭으로 링 3 진입, (4) 유저 모드에서 도는 미니 셸 — 키 입력을 받아 `write hello` 같은 내장 명령을 시스템 콜로 실행.
- **verify 기준**: ① 유저 모드 코드가 `int 0x80`으로 화면에 문자열을 찍는다. ② **결정적 증거**: 같은 유저 코드에서 VGA 버퍼(0xB8000)나 커널 데이터에 직접 쓰기를 시도하면 폴트가 나서 커널 핸들러에 잡힌다 — 보호가 진짜로 걸려 있다는 뜻. ③ 셸 프롬프트에서 명령이 실행된다.
- **막힐 때 볼 자료**: OSDev Wiki "System Calls", "Getting to Ring 3", "TSS" · xv6 북(트랩/시스템 콜 장) — RISC-V 기준이지만 개념 대응이 가장 깔끔한 교과서.

**왜 이 설계인가.** 지금까지 모든 코드는 링 0(최고 특권)에서 돌았다. OS가 OS인 이유는 **신뢰하지 않는 코드를 링 3에서 돌리면서도 서비스는 제공하는** 데 있다. 두 방향의 문이 필요하다:

- **내려가는 문(링0→링3)**: 낮은 특권으로 "점프"하는 명령은 없다. 대신 `iret`이 스택에서 [EIP, CS, EFLAGS, ESP, SS]를 꺼내 복원한다는 점을 역이용해, **유저 세그먼트 셀렉터(RPL=3)와 유저 스택/진입점을 스택에 위조해 쌓고 iret** — "인터럽트에서 돌아가는 척하며 유저 모드로 강등"하는 관용구다.
- **올라오는 문(링3→링0)**: 유저 코드가 `int 0x80`을 걸면 CPU가 IDT를 조회하는데, 게이트의 DPL이 3이어야 통과시킨다(아니면 GP 폴트 — 이게 유저가 임의 벡터를 못 찌르는 이유다). 통과하면 CPU는 **TSS에 적어둔 커널 스택(esp0)으로 자동으로 갈아타고** 커널 핸들러로 진입한다. 왜 스택을 갈아타나 — 유저가 자기 스택 포인터를 쓰레기로 만들어놓고 시스템 콜을 걸 수 있기 때문에, 커널은 유저 스택을 1바이트도 신뢰할 수 없다.

```
링3 유저 코드                          링0 커널
────────────                          ────────
mov eax, 1        ; sys_write 번호
mov ebx, msg      ; 인자
int 0x80  ────────────────────────▶  IDT[0x80] (DPL=3 게이트)
                                      CPU: TSS.esp0 커널 스택으로 전환
                                      레지스터 저장 → EAX로 디스패치
                                      sys_write 실행 (커널만 VGA 접근 가능)
          ◀────────────────────────  iret (유저 CS/EIP/ESP 복원)
다음 명령 계속
```

이 그림이 "시스템 콜은 왜 함수 호출보다 비싼가"의 답 그 자체다: 특권 전환 + 스택 전환 + 레지스터 저장/복원 + (파이프라인 관점 비용). 리눅스가 나중에 `sysenter`/`syscall` 전용 명령으로 이 경로를 단축한 이유까지 한 문장으로 잇는다면 그 주제의 꼬리질문은 끝난다.

미니 셸은 기술적으로 새로울 게 없지만 **데모로서 결정적**이다. 면접에서 "만들었다"는 말은 화면이 증명한다 — 노트북에서 QEMU를 띄워 셸 프롬프트를 보여주는 15초가 어떤 설명보다 강하다.

## ④ 완주 후 확장 경로

### 왜 다음이 xv6인가

내 커널은 "동작하는 최소"지 "좋은 설계"가 아니다. 다음 단계는 더 만들기보다 **잘 만든 작은 커널을 읽는 것**이 효율적이다. xv6-riscv(MIT 6.1810 교재용 유닉스 클론, 약 만 줄 미만)가 표준 답인 이유: 진짜 유닉스 구조(fork/exec/파일시스템)를 갖추고 있으면서, 전체를 한 사람이 다 읽을 수 있는 유일한 규모다. 직접 만들어본 뒤에 읽으면 "아 이걸 이렇게 푸는구나"가 페이지마다 터진다.

**리딩 순서와 OSTEP 병행 매핑** — 내가 만든 것과 겹치는 곳부터 읽어야 대조가 된다:

| 순서 | xv6 파일 | 내 6주 커널의 대응물 | 병행할 OSTEP 영역 | 대조 포인트 |
|---|---|---|---|---|
| 1 | `proc.c` | W5 TCB + RR | 프로세스·스케줄링 장 (Virtualization 전반부) | 내 "TCB+스위치"에 fork/exec/wait·수면/기상(sleep/wakeup)이 얹히면 뭐가 달라지나 |
| 2 | `vm.c` | W4 페이징 | 주소 공간·페이징·TLB 장 (Virtualization 후반부) | 커널/유저 페이지 테이블 분리, 프로세스별 주소 공간이라는 내가 안 한 반쪽 |
| 3 | `trap.c` | W3 IDT + W6 int 0x80 | Limited Direct Execution 장 | 예외·인터럽트·시스템 콜이 **하나의 트랩 경로**로 통합되는 설계 |
| 4 | `fs.c` | (없음 — 신영역) | Persistence 파트(파일시스템·크래시 일관성) | 6주 커널에 없던 유일한 대주제. 버퍼 캐시→로그→inode 계층을 처음부터 |

읽는 방법: 코드만 읽지 말고 **xv6 북(교재 PDF)의 해당 장 → 소스 → OSTEP 해당 장** 삼각측량. OSTEP은 "왜"를, xv6는 "어떻게"를 준다.

### 그 외 가지치기

- **Rust 2회차**: os.phil-opp.com을 따라 같은 커널을 다시 — 언어가 강제하는 안전성이 커널 코드에서 어디까지 통하고 어디서 `unsafe`로 뚫어야 하는지가 보인다.
- **64비트/UEFI**: 롱 모드 전환, 4단계 페이지 테이블. 현대 부팅의 실제 모습이 궁금할 때.
- **깊이 파기**: 특정 주제(스케줄러, 할당자)를 리눅스 소스에서 추적. 단, 리눅스는 "읽는" 대상이지 "완독하는" 대상이 아니다 — xv6에서 개념을 잡고 리눅스에서 그 개념의 실전 형태만 확인한다.

### Winters와 잇기 — "커널이 해주던 일을 유저 공간에서 재현했다"

이 로드맵의 마지막 조각은 6주 커널과 기존 포트폴리오를 **하나의 서사**로 묶는 것이다. Winters에는 이미 커널 메커니즘의 유저 공간 판이 세 개 있다:

1. **컨텍스트 스위치 ↔ Fiber**: W5에서 직접 짠 스택 바꿔치기가, Windows가 `CreateFiber`/`SwitchToFiber`로 제공하는 것의 본체다. Winters의 JobSystem은 파이버 실행 모드를 스위치 가능한 구조로 설계돼 있고(`Engine/Public/Core/Fiber/FiberTypes.h:8`의 `eJobExecutionMode::FiberShell`, `Engine/Public/Core/Fiber/Fiber.h:5`의 `CFiber` 핸들 래퍼), 헤더 주석부터 "Phase 5-B에서 내부가 Fiber로 교체되어도 public API 불변"을 계약으로 박아뒀다(`Engine/Public/Core/JobSystem.h:25`). 커널 스케줄러와의 차이는 단 하나 — **선점이 없다**(하드웨어 타이머 인터럽트를 유저 공간이 직접 받을 수 없고, Win32 파이버는 명시적 `SwitchToFiber` 호출로만 전환되므로 협조적 양보만 가능). W5를 완주하면 이 차이를 "빠진 부품" 수준으로 정확히 짚을 수 있다.
2. **스케줄러 ↔ JobSystem**: W5의 RR 스케줄러는 "누구를 다음에 돌릴까"의 커널 판이고, `Engine/Public/Core/JobSystem.h:27`의 `CJobSystem`은 같은 질문의 유저 공간 판이다 — 워커별 `CWorkStealingDeque`(`Engine/Public/Core/JobSystem/WorkStealingDeque.h:15`, Chase-Lev)에서 자기 것은 LIFO로 캐시를 지키고 남의 것은 FIFO로 훔친다. 커널 RR이 "공평"을 정책으로 삼는다면 워크 스틸링은 "캐시 지역성과 부하 균형"을 정책으로 삼는다 — **메커니즘(전환) 위의 정책(선택)이 목적에 따라 갈린다**는 OS 교과서 문장의 실물 대조다.
3. **인터럽트/블로킹 ↔ IOCP**: W3에서 "이벤트가 CPU를 깨운다"를 배선했다면, `Server/Public/Network/IOCPCore.h:33`의 `CIOCPCore`는 그 위 계층이다 — 워커 스레드가 `GetQueuedCompletionStatus(..., INFINITE)`로 잠들면(`Server/Private/Network/IOCPCore.cpp:216`), NIC 인터럽트 → 커널 완료 처리 → 완료 큐 → 스레드 기상이라는 사슬의 끝을 받는 것이다. W3를 완주하면 "비동기 I/O가 왜 스레드-당-연결보다 확장되는가"를 인터럽트 계층부터 설명할 수 있다.

이 재해석이 주는 면접 효과: "OS를 만들어봤다"와 "게임 서버/엔진을 만들었다"가 별개 항목이 아니라, **"커널 메커니즘을 이해했고, 그것이 부족한 지점(선점 불가, 커널 진입 비용)에서 유저 공간 재구현을 선택했다"는 하나의 판단 서사**가 된다.

- **T(과제)**: "6주 목표로 부트섹터부터 시스템 콜까지 — 유저 모드 미니 셸이 뜨는 것을 완료 조건으로 잡았습니다. 매주 '화면에 무엇이 보이면 성공'이라는 검증 기준을 먼저 정했습니다."
- **A(행동)**: "WSL2 + QEMU + i686-elf 크로스 컴파일러로 환경을 잡고, 부트섹터 → 보호 모드 → 인터럽트 → 페이징 → 선점 멀티태스킹 → int 0x80 순서로 쌓았습니다. 가장 막혔던 건 페이징을 켜는 순간의 트리플 폴트였는데, QEMU 인터럽트 로그로 항등 매핑 누락을 잡았습니다. printf가 없는 환경이라 GDB 원격 디버깅과 VGA 직접 출력으로 관측 수단부터 만드는 습관이 생겼습니다."
- **R(결과)**: "유저 모드 셸에서 시스템 콜로 문자를 찍는 데모가 돌고, 무엇보다 제 JobSystem과 IOCP 서버를 '커널이 해주던 일의 유저 공간 재현'으로 다시 설명할 수 있게 됐습니다. 파이버 전환이 왜 스레드 전환보다 싼지 — 커널 진입도, 전체 레지스터 저장도 없이 스택 포인터 교체가 본체라서 — 를 직접 짠 코드로 답할 수 있습니다."

한 가지 규율: **못 만든 부분을 부풀리지 말 것.** 파일시스템·SMP·스왑은 안 만들었으면 "거기까지는 xv6 소스 리딩으로 대체했다"고 경계를 정직하게 긋는 쪽이, 어설픈 과장보다 신뢰를 산다. 면접관은 경계선을 정확히 아는 사람을 신뢰한다.
