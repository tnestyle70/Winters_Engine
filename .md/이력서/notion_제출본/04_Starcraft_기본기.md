엔진도 그래픽 라이브러리도 없이, 윈도우 기본 API만으로 스타크래프트의 코어(유닛 이동·전투·생산·전장의 안개·미니맵)를 구현했습니다(약 5.4만 줄). 상용 엔진이 대신 해주던 것들 — 길찾기, 수십 유닛의 동시 이동, 반투명 그리기 — 을 바닥에서 직접 만들며 원리를 확보한 프로젝트입니다.

### 구현

- **[이동] Octile A\*(코너컷 방지·타일 비용) + Separation/Cohesion 스티어링 결합** — 전역 경로탐색과 지역 조향의 역할 분리로 수십 유닛 군집 이동
- **[안개] 3상태 전장의 안개(미탐사/탐사/시야)를 dirty-flag 캐시 DC로 최적화** — 매 프레임 전체 재계산 대신 변한 타일만 갱신
- **[렌더] GDI 비트연산(SRCAND/SRCPAINT)+AlphaBlend 3단 합성** — 셰이더 없이 팀컬러·반투명 구현
- **[객체] CObj 최상위 추상 + Object Manager, pending delete** — 죽은 객체를 즉시 delete하지 않고 selection·컨트롤 그룹·명령·투사체 타깃 참조를 일괄 정리 후 해제
- **[생산] 템플릿 팩토리 + deque 기반 FIFO 생산 큐** — 명령 입력과 결과 생성의 시간 분리
- **[UI] Commandable/CommandSlot 계약** — 유닛(32종)·건물(40종)이 자기 커맨드 카드를 정의하고 UI는 렌더만: UI가 게임 로직을 침범하지 않는 경계
- 인게임 맵 에디터 → 바이너리 저장 → 내비 그리드 재생성 파이프라인

### ▶ 더 깊이 — 트레이드오프와 뒷이야기 (토글)

- **가장 어려웠던 것 — 어택무브**: "이동 중 적을 만나면 교전하고, 끝나면 원래 목적지로 복귀"는 명령 큐 소비와 상태 전환이 충돌하는 문제다. Order에 데이터를 내장하고 유닛 FSM(9상태)과 책임을 분리해 풀었다 — 명령은 의도, 상태는 실행이라는 구분을 여기서 배웠다.
- **RTS에서 죽음은 화면에서 사라지는 것으로 끝나지 않는다**: 죽은 유닛을 선택 목록·컨트롤 그룹·투사체 타깃이 여전히 참조한다. pending delete는 이 문제의 수동 해법이었고, Winters의 세대 핸들은 같은 문제의 구조적 재해결이다 — 같은 문제를 두 번 푼 것이 아니라, 더 나은 언어로 다시 푼 것.
- **로우레벨의 가치**: GDI에는 알파 블렌딩이 사실상 없다 — 비트 마스크 연산 조합으로 만들었다. 이후 DX11 블렌드 스테이트를 쓸 때 "엔진이 뭘 대신해 주고 있는지"를 정확히 아는 상태가 됐다.
