# 렌더링 면접 질문 은행

> 이 챕터는 개념을 다시 설명하지 않는다. 개념 설명은 이 시리즈의 앞 챕터(01~07)에 있고, 여기는 **면접장에서 실제로 나오는 질문 → 한 줄 정답 → 파고들기(꼬리질문)**만 모은 질문 은행이다.
> 사용법: (A)로 기초를 훑고, 각 항목의 `▸` 를 스스로 답할 수 있는지 점검한다. 막히면 그게 공부할 지점이다. (B)는 꼬리질문 방어용, (C)는 "경험을 말해보라"에 STAR로 답하는 골격이다.
> 역할 분담: [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)는 **Winters 렌더러 구조**(RHI 계층, 핸들 모델, FX 이중 트랙) 중심, [engine/16_interview_qa_bank_engine.md](../engine/16_interview_qa_bank_engine.md)는 엔진 도메인 전반의 질문 은행이다. 이 챕터는 **그래픽스 이론 + 수학 + API 본질** 중심이며, 구조 설명이 필요하면 서술을 반복하지 않고 그쪽으로 크로스링크한다.
> 인용은 전부 실제 파일에서 확인한 `path:line` 이다. 수식은 LaTeX가 아니라 일반 텍스트/코드 표기로 쓴다.

## A. 기초 개념 질문 (한 줄 정답 + 파고들기)

### A-1. 파이프라인과 좌표 변환

1. **렌더링 파이프라인의 스테이지 순서는?**
   한 줄: IA(입력 조립기, Input Assembler) → VS(정점 셰이더) → [테셀레이션 3단: Hull/Tessellator/Domain] → [GS(지오메트리 셰이더)] → 래스터라이저(rasterizer) → PS(픽셀 셰이더) → OM(출력 병합기, Output Merger).
   ▸ "프로그래머블과 고정 기능을 나눠보라" → VS/HS/DS/GS/PS는 프로그래머블, IA·래스터라이저·OM은 상태로 설정하는 고정 기능(fixed function)이다. 클리핑·원근 나눗셈·뷰포트 변환·보간은 전부 **래스터라이저(고정 기능)** 소관 — "원근 나눗셈을 어느 셰이더가 하나요?"라는 함정 질문의 답은 "셰이더가 아니다"이다.

2. **정점 셰이더(vertex shader)가 하는 일은?**
   한 줄: 정점 1개당 1회 실행되어 위치를 로컬 공간에서 **클립 공간**(clip space)까지 변환하고(`SV_POSITION` 출력), 스키닝·정점 단위 속성 계산을 한다.
   ▸ "정점을 만들거나 없앨 수 있나?" → 못 한다. 1:1 변환만. 증폭/제거는 GS·테셀레이션의 영역이다.
   ▸ Winters FX 정점 셰이더의 최소형: 로컬 → 월드 → 뷰투영 두 번의 `mul` (`Shaders/FxMesh.hlsl:58`~`:59`).

3. **픽셀 셰이더(pixel shader)가 하는 일은?**
   한 줄: 래스터라이저가 만든 프래그먼트(fragment)마다 실행되어 최종 색을 계산한다 — 텍스처 샘플, 조명, `clip`/`discard`에 의한 픽셀 폐기까지.
   ▸ "픽셀 위치를 바꿀 수 있나?" → 못 바꾼다. 위치는 래스터화로 확정. 깊이는 `SV_Depth`로 덮어쓸 수 있지만 early-z를 죽인다 → (B)2.

4. **정점 하나가 픽셀이 되기까지 전 과정을 설명해보라. (단골 1순위)**
   한 줄: 로컬 → (월드 행렬) → 월드 → (뷰 행렬) → 뷰 → (투영 행렬) → 클립 → [클리핑] → (원근 나눗셈) → NDC → (뷰포트 변환) → 스크린 픽셀 → 래스터화 → PS → OM(깊이 테스트/블렌딩).
   각 단계가 존재하는 이유까지 붙여야 합격점이다:
   ```text
   로컬→월드   : 모델을 장면에 배치 (이동/회전/스케일)
   월드→뷰     : 카메라를 원점·-축 정렬로 옮겨 이후 계산을 단순화
   뷰→클립     : 시추대(frustum)를 정규 상자로 눌러 담음. w에 z를 복사
   클리핑      : 나눗셈 전에 -w<=x,y<=w, 0<=z<=w (D3D) 검사로 화면 밖 제거
   원근 나눗셈  : xyz /= w — "멀수록 작게"가 여기서 실제로 일어남
   NDC→뷰포트  : 해상도 독립 좌표를 실제 픽셀 격자에 매핑
   ```
   ▸ "왜 클리핑을 나눗셈 전에 하나?" → `w=0`(카메라 평면)에서 나눗셈이 발산하고, 카메라 뒤 정점은 나눗셈 후 부호가 뒤집혀 화면 안으로 "접혀 들어온다". 특이점을 지나기 전에 잘라야 한다.
   ▸ Winters의 행렬 팩토리: `Engine/Include/WintersMath.h:283`(LookAtLH) / `:288`(PerspectiveFovLH).

5. **동차 좌표(homogeneous coordinates)와 w의 역할은?**
   한 줄: `(x,y,z,w)` 4성분으로 확장하면 3D에서 선형 변환이 아닌 **이동(translation)**과 **원근 투영**까지 전부 4x4 행렬 곱 하나로 통일된다.
   ▸ "w=1과 w=0의 차이는?" → w=1은 점(이동의 영향을 받음), w=0은 방향 벡터(이동 무시). 노멀/방향을 `mul(float4(v, 0.f), M)`로 변환하는 근거가 이것이다 (`Shaders/FxMesh.hlsl:61`).
   ▸ "왜 이동이 3x3으론 안 되나?" → 이동은 원점을 보존하지 않으므로 선형 변환이 아니다. 차원을 하나 올려 아핀(affine) 변환을 한 차원 위의 선형 변환으로 승격시킨 것.

6. **뷰 행렬이 카메라 월드 행렬의 역행렬인 이유는?**
   한 줄: "카메라를 원점으로 되돌리는 변환"을 세상 전체에 적용하는 것 — 카메라를 움직이는 대신 세상을 반대로 움직인다.
   ▸ "역행렬을 실제로 계산하나?" → 안 한다. 카메라 변환은 회전+이동뿐이고 회전 행렬은 정규직교(orthonormal)라 **역 = 전치**. LookAt 함수가 right/up/forward 축을 전치로 배치하고 이동을 `-eye`와 각 축의 내적으로 채워 역행렬을 직접 조립한다 — 일반 역행렬(가우스 소거)보다 압도적으로 싸고 수치적으로 안정.
   ▸ Winters는 `XMMatrixLookAtLH` 사용 (`Engine/Include/WintersMath.h:283`).

7. **원근 나눗셈(perspective divide)은 왜 필요한가?**
   한 줄: 원근이란 `x' = x/z` 꼴의 **나눗셈**인데 나눗셈은 선형 연산이 아니라 행렬로 표현할 수 없으므로, 행렬은 w에 z를 복사해 두는 데까지만 하고 나눗셈은 래스터라이저의 고정 단계로 분리한 것이다.
   ▸ "투영 행렬의 어느 원소가 그 일을 하나?" → z를 w로 넘기는 열의 1 — 그래서 클립 공간의 `w`는 사실상 뷰 공간 z다. 직교 투영은 이 복사가 없어서 `w=1`, 나눗셈이 항등이 된다.

8. **NDC(정규화 장치 좌표, Normalized Device Coordinates)란?**
   한 줄: 원근 나눗셈 직후의 해상도 독립 좌표계 — D3D 기준 `x,y ∈ [-1,1]`, `z ∈ [0,1]`.
   ▸ "GL과 차이는?" → 전통적 GL은 z도 `[-1,1]`. D3D의 `[0,1]` 컨벤션은 깊이 정밀도와 reversed-Z 조합에 더 유리하다 → (B)1.

9. **뷰포트 변환(viewport transform)의 수식은?**
   한 줄: `x_screen = (x_ndc + 1) * width/2 + topLeftX`, `y_screen = (1 - y_ndc) * height/2 + topLeftY` — y가 뒤집히는 이유는 NDC는 위가 +인데 화면 픽셀 좌표는 아래로 증가하기 때문.
   ▸ "z는?" → `z_ndc`가 그대로(또는 MinDepth/MaxDepth 범위로) 깊이 버퍼 값이 된다. 뷰포트를 절반 크기로 잡으면 같은 장면이 화면 일부에만 그려진다 — 미니맵/분할 화면의 원리.

### A-2. 깊이 · 컬링

10. **깊이 버퍼(depth buffer)의 원리와 z-fighting 해결법은?**
    한 줄: 픽셀마다 "지금까지 가장 가까운 깊이"를 저장하고 새 프래그먼트와 비교(LESS)해 이기면 색·깊이를 갱신한다 — 그리는 순서와 무관하게 가림이 해결된다.
    z-fighting의 원인은 **깊이 값의 비선형 분포**다. 투영 후 깊이는 `z_ndc = f(z-n) / (z(f-n))` 꼴의 1/z 함수라 정밀도가 near 근처에 몰리고, 먼 곳의 겹친 두 면은 같은 값으로 양자화되어 프레임마다 승자가 바뀐다.
    해결 우선순위: (1) **near plane을 올린다** — 정밀도는 far보다 near에 지배적으로 민감하다. (2) 24/32bit 깊이 포맷. (3) depth bias(상수 + slope-scaled). (4) 애초에 **겹친 지오메트리를 데이터에서 제거** — Winters가 소환사의 협곡 초기에 택한 길 → (C)1. (5) reversed-Z → (B)1.
    ▸ "far를 줄이는 것과 near를 올리는 것 중 뭐가 효과적인가?" → near. `f/n` 비율이 정밀도를 결정하는데 분모 쪽 변화가 비율을 훨씬 크게 움직인다.

11. **백페이스 컬링(backface culling)은 어떻게 판정하나?**
    한 줄: 스크린 공간에서 삼각형 두 변의 2D 외적 — `(v1-v0) x (v2-v0)`의 z 부호 = 부호 있는 넓이(signed area) = 와인딩(winding) 방향으로 판정한다.
    ▸ "왜 하나?" → 닫힌 메시의 뒷면은 절대 보이지 않으므로 래스터라이저 단계에서 삼각형의 절반을 PS 실행 전에 버릴 수 있다.
    ▸ "D3D 기본 규약은?" → 시계 방향(CW)이 앞면(`FrontCounterClockwise = FALSE`), 뒷면 컬링. 미러 스케일(음수 스케일)을 먹이면 와인딩이 뒤집혀 메시가 "뒤집혀 보이는" 사고의 원인이 된다.

### A-3. 벡터 · 회전 수학

12. **내적과 외적의 그래픽스 용도는?**
    한 줄: 내적(dot product)은 "얼마나 같은 방향인가"(코사인) — 디퓨즈 `N·L`, 시야 판정, 투영 길이. 외적(cross product)은 "두 벡터에 수직인 벡터" — 노멀 생성, 카메라 기저(basis) 구축, 와인딩/좌우 판정.
    ▸ "내적으로 앞뒤를 어떻게 아나?" → `dot(forward, toTarget) > 0`이면 앞. 정규화 없이도 부호는 유효하다 — 비교만 할 거면 `sqrt`를 아낀다.
    ▸ Winters 실물: FX rim 계산의 `dot(normal, viewDir)` (`Shaders/FxMesh.hlsl:79`), 실린더 피킹의 반직선-원 판별식 → (C)2.

13. **오일러 각 vs 쿼터니언(quaternion) — 짐벌락(gimbal lock)이란?**
    한 줄: 오일러는 세 축 회전을 순서대로 합성하는데 가운데 축이 ±90°가 되면 첫 축과 셋째 축의 회전 평면이 겹쳐 **자유도 하나를 잃는** 것이 짐벌락 — 쿼터니언은 축·각을 4성분 `q = (axis * sin(theta/2), cos(theta/2))`으로 담아 이 문제가 없고 구면 보간(slerp)이 매끄럽다.
    ▸ "왜 반각(theta/2)인가?" → 회전이 `q * v * q^-1` 샌드위치 곱으로 적용되어 q가 **양쪽에서 한 번씩** 곱해지기 때문. 각각 theta/2씩 기여해 합이 theta가 된다.
    ▸ "그럼 오일러는 버려야 하나?" → 아니다. 제약된 회전엔 더 단순한 표현이 이긴다. Winters 챔피언은 y축 요(yaw) 하나만 도는 LoL 탑다운 특성상 **스칼라 yaw 하나**로 몸통 방향을 관리한다 — `yaw → forward`가 `(sin, 0, cos)` 한 줄 (`Client/Private/Network/Client/SnapshotApplier.cpp:105`~`:108`). 다만 각도는 mod 2pi 동치라 ±PI 경계 처리가 별도 문제로 남는다 → (C)6.

14. **노멀 변환에 역전치(inverse transpose) 행렬을 쓰는 이유는?**
    한 줄: 노멀은 "표면에 수직"이라는 **관계**가 보존돼야 하는데, 비균등 스케일에서 노멀을 같은 행렬로 변환하면 수직성이 깨지기 때문이다.
    유도: 접선 t는 `t' = M*t`로 변환된다. 변환 후에도 `dot(n', t') = 0`이어야 하므로 `n'^T * M * t = 0`이 모든 접선 t에 대해 성립해야 하고, 이는 `n' = (M^-1)^T * n`일 때다.
    ▸ "왜 대부분의 코드는 그냥 월드 행렬을 쓰나?" → 회전+균등 스케일이면 `(M^-1)^T`가 M과 (스케일 상수 배 차이 빼고) 같아서 정규화만 하면 된다. Winters FX 셰이더도 이 전제로 월드 행렬 + `normalize`를 쓴다 (`Shaders/FxMesh.hlsl:61`) — 비균등 스케일 메시가 들어오면 깨지는 걸 알고 쓰는 것과 모르고 쓰는 것의 차이를 면접에서 보여줘야 한다.

### A-4. 색 · 조명 · PBR

15. **감마 보정(gamma correction)이란? 왜 필요한가?**
    한 줄: 기원은 CRT의 비선형 응답(출력 ≈ 입력^2.2) 보정이지만, 존속하는 진짜 이유는 **인간 눈이 어두운 영역에 민감**해서 8bit를 지각 균등하게 배분하는 인코딩(sRGB)이 저장 효율상 이득이기 때문이다.
    ▸ "sRGB의 0.5는 실제 밝기 얼마인가?" → 선형 광량으로 약 21% — "절반 코드값 = 절반 에너지"가 아니라는 것이 요점이다.

16. **linear workflow란?**
    한 줄: 조명·블렌딩은 빛 에너지 계산이므로 **선형 공간**에서 하고, 저장·출력만 sRGB로 인코드하는 규율 — sRGB 텍스처는 샘플 시 디코드, 백버퍼는 기록 시 인코드.
    ▸ "감마 공간에서 라이팅하면 뭐가 깨지나?" → 덧셈이 깨진다. 광원 두 개를 합쳐도 에너지 합이 안 맞아 하이라이트가 타고 중간톤이 탁해진다.
    ▸ "모든 텍스처를 sRGB로 읽나?" → 아니다. 노멀맵·러프니스·마스크는 색이 아니라 **데이터**라 선형으로 읽어야 한다. `_SRGB` 포맷 지정 실수는 화면 톤이 통째로 어긋나는 고전 사고다.

17. **PBR이 Phong과 다른 점은?**
    한 줄: (1) 에너지 보존 — 반사 총량이 입사를 넘지 않음, (2) 미세면(microfacet) 이론 기반 스펙큘러, (3) albedo/roughness/metallic이라는 물리 대응 파라미터 — 조명 환경이 바뀌어도 머티리얼을 다시 튜닝하지 않는다.
    ▸ "Phong의 구체적 문제는?" → `pow(dot(R,V), shininess)`는 경험적 모양일 뿐 정규화가 없어서 shininess를 올리면 총 반사 에너지가 임의로 변한다. 아티스트가 라이트마다 강도를 다시 맞추는 비용이 곧 파이프라인 비용이었다.
    ▸ "디퓨즈에서 왜 pi로 나누나?" → Lambert BRDF는 `albedo / pi`다. 반구 전체로 `cos` 가중 적분하면 `∫ cos dw = pi`가 나오므로, 나눠주지 않으면 들어온 빛보다 pi배 많은 빛이 나가는 에너지 위반이 된다.

18. **프레넬(Fresnel) 효과란?**
    한 줄: 시선이 표면을 스치는 각(grazing angle)일수록 반사율이 올라가 모든 물질이 100%에 접근하는 현상 — 물웅덩이를 멀리서 보면 거울이 되는 이유.
    Schlick 근사: `F(cos) = F0 + (1 - F0) * (1 - cos)^5`, cos은 보통 `dot(h, v)`.
    ▸ "왜 5제곱인가?" → 물리 유도가 아니라 실측 프레넬 곡선에 대한 **수치 피팅**이다. 싸고 충분히 정확해서 표준이 됐다 — "근사의 출처를 아는가"를 시험하는 질문.
    ▸ Winters의 rim 라이트가 같은 수학 골격이다: `1 - saturate(dot(n, v))`를 pow로 올려 가장자리를 밝힌다 (`Shaders/FxMesh.hlsl:76`~`:81`) — 물리 프레넬이 아니라 연출용이지만 "각도가 눕을수록 세진다"는 구조가 같다.

19. **roughness / metallic의 의미는?**
    한 줄: roughness는 미세면 법선 분포의 폭 — 하이라이트의 크기와 번짐을 결정. metallic은 도체/부도체 보간 스위치 — 금속은 디퓨즈가 없고(투과광을 자유전자가 흡수) F0가 albedo 색, 비금속은 F0 ≈ 0.04 무채색.
    ▸ "metallic 0.5는 물리적으로 뭔가?" → 물리적으로는 없는 물질이다. 픽셀 안에 금속/비금속이 섞인 경우(먼지 낀 금속)의 텍셀 평균으로만 정당화된다.
    ▸ "F0 = 0.04의 근거는?" → (B)7.

20. **렌더링 방정식(rendering equation)이란?**
    한 줄: 한 점에서 한 방향으로 나가는 빛 = 자체 발광 + 반구의 모든 입사광을 BRDF로 가중 적분한 것:
    ```text
    Lo(x, wo) = Le(x, wo) + INT_hemisphere{ fr(x, wi, wo) * Li(x, wi) * dot(n, wi) } dwi
    ```
    ▸ "실시간 렌더링은 이 식의 무엇인가?" → 이 적분을 항별로 쪼갠 근사의 스택이다. 직접광 = 광원 몇 개의 합(적분 → 유한 합), 그림자 = Li 안의 가시성 항, IBL = 원거리 Li의 사전 적분 → (B)6. "우리가 뭘 버리고 있는지"를 아는 것이 이 식을 아는 실익이다.

### A-5. 드로우콜 · 렌더 경로

21. **드로우콜이 왜 비싼가?**
    한 줄: GPU가 아니라 **CPU 쪽 비용** — 드라이버가 드로우 시점마다 상태 조합을 검증·변환하고, DX11에서는 이 작업이 사실상 단일 스레드에 묶인다.
    ▸ "그래서 뭘 하나?" → 배칭(정적 병합), 인스턴싱, 상태 변경 최소화를 위한 정렬(sort by state), 그리고 API 차원 해법이 DX12/Vulkan의 선컴파일 PSO + 병렬 커맨드 기록 → (A)31.

22. **인스턴싱(instancing)이란?**
    한 줄: 같은 메시·머티리얼 N개를 드로우 1번으로 — 인스턴스별 데이터(월드 행렬 등)를 per-instance 정점 스트림이나 structured buffer로 주고 `SV_InstanceID`로 조회한다.
    ▸ "인스턴싱이 안 통하는 경우는?" → 머티리얼/셰이더가 다르거나 스키닝 본이 인스턴스마다 다른 경우. Winters는 본 팔레트를 cbuffer 대신 structured buffer SRV(t8, 1024본)로 올려 대형 rig 한계를 푼 이력이 있다 (`Engine/Private/Renderer/ModelRenderer.cpp:30`~`:44`) — 같은 "cbuffer 한계를 SRV로 우회" 계열 기법이다.

23. **포워드 vs 디퍼드(deferred) 렌더링?**
    한 줄: 포워드는 오브젝트를 그리면서 조명까지 — 비용이 대략 `오브젝트 x 라이트`. 디퍼드는 G-buffer에 표면 정보(albedo/normal/depth/roughness)만 기록한 뒤 스크린 공간에서 조명 — 비용이 `보이는 픽셀 x 라이트`로 바뀌어 다광원에 강하다.
    ▸ "디퍼드의 대가는?" → G-buffer 대역폭/메모리, 투명체는 못 다뤄 별도 포워드 패스, MSAA와 상성 최악 → (B)9, 머티리얼 다양성이 G-buffer 레이아웃에 갇힘.
    ▸ "절충은?" → Forward+(타일별 라이트 컬링) — 포워드의 유연함에 라이트 스케일링을 더한 현대 기본기.

### A-6. 그림자 · AO

24. **그림자 매핑(shadow mapping)의 원리와 acne 해결법은?**
    한 줄: 라이트 시점에서 깊이만 렌더해 두고, 본 패스에서 각 픽셀을 라이트 공간으로 투영해 "저장된 깊이보다 멀면 그림자"로 판정한다.
    acne(자기 그림자 줄무늬)의 원인은 섀도맵 텍셀 하나가 화면의 여러 픽셀을 커버하는 **해상도 불일치 + 깊이 양자화** — 표면이 자기 자신과의 비교에서 지고 이기기를 반복한다.
    ▸ "해결은?" → depth bias. 표면이 빛에 대해 기울수록 오차가 커지므로 상수 bias + 기울기 비례(slope-scaled) bias를 함께 쓴다. 과하면 그림자가 발에서 떨어지는 peter-panning — bias는 두 오차 사이의 트레이드오프 다이얼이다.
    ▸ "가장자리 계단은?" → PCF(주변 비교 결과 평균), 해상도 배분은 캐스케이드(CSM).

25. **SSAO(스크린 공간 주변광 차폐)의 원리는?**
    한 줄: 깊이/노멀 버퍼만으로 "이 픽셀 주변이 얼마나 막혀 있나"를 근사 — 픽셀 주변 반구에서 샘플을 뿌려 깊이 버퍼보다 안쪽에 묻힌 비율만큼 ambient를 깎는다.
    ▸ "노멀 반구를 쓰는 이유는?" → 구 전체로 뿌리면 평평한 벽도 절반이 "막혀" 있어 회색이 된다(초기 Crysis 구현의 특징). 노멀 방향 반구로 제한하는 것이 개선판.
    ▸ "한계는?" → 화면에 없는 것은 차폐를 못 만든다 — 화면 밖/뒤 물체 정보가 없는 스크린 스페이스 기법의 공통 한계. 샘플 수 부족 노이즈는 랜덤 회전 + 블러로 가린다.

### A-7. 텍스처 · 필터링

26. **밉맵(mipmap)이 왜 필요한가?**
    한 줄: 먼 표면은 픽셀 하나가 텍셀 수백 개를 커버하는데 그중 1개만 샘플하면 프레임마다 다른 텍셀이 걸려 반짝이는 앨리어싱이 생긴다 — 미리 절반씩 줄인 체인에서 픽셀:텍셀 ≈ 1:1인 레벨을 골라 "커버 영역의 평균"을 O(1)에 얻는 사전 계산이 밉맵이다.
    ▸ "메모리 비용은?" → 1/4씩 줄어드는 등비 합이라 +33%.
    ▸ "부수 효과는?" → 캐시 지역성. 먼 표면을 밉 없이 읽으면 샘플마다 메모리를 널뛰어 캐시가 무력화된다. 레벨 선택 메커니즘은 → (B)11.

27. **텍스처 필터링 종류는?**
    한 줄: point(최근접 1텍셀) → bilinear(4텍셀 가중평균) → trilinear(인접 밉 2레벨을 추가 블렌드 — 레벨 경계 팝 제거) → anisotropic(비스듬한 표면의 길쭉한 커버 영역을 여러 샘플로 — 최대 16x).
    ▸ "anisotropic이 해결하는 구체적 증상은?" → 바닥을 낮은 각도로 볼 때의 원거리 흐림. 등방 필터는 커버 영역을 정사각형으로 가정해 긴 축에 맞춰 과하게 흐린 밉을 고르기 때문이다.

### A-8. 투명도 · FX

28. **알파 블렌딩의 정렬 문제는?**
    한 줄: `src*a + dst*(1-a)`(over 연산자)는 **순서 의존** 연산이라 뒤에서 앞으로 그려야 하고, 반투명은 깊이 기록을 꺼야(depth write off) 뒤의 반투명이 잘리지 않는다.
    ▸ "정렬해도 안 되는 경우는?" → 서로 관통하는 메시 — 오브젝트 단위 정렬로는 픽셀 단위 순서를 못 맞춘다. 해법은 OIT(순서 무관 투명) 또는 회피: **가산(additive) 블렌드는 덧셈이라 순서 무관**이므로 글로우 계열 FX는 정렬 문제가 없다.
    ▸ Winters FX의 실물 계약: `blendMode = AlphaBlend` + `bDepthWrite = false` (`Client/Private/GameObject/Champion/Irelia/IreliaFxPresets.cpp:362`, `:368`).

29. **blend vs clip은 언제 나눠 쓰나?**
    한 줄: clip/discard는 픽셀을 이진으로 버림 — 깊이 기록 가능, 정렬 불요, 대신 가장자리 계단과 early-z 손해 → (B)2. blend는 부드럽지만 정렬 + depth write off 규율이 따라온다.
    ▸ Winters는 한 셰이더 안에서 용도별 병용: dissolve 소멸 연출은 노이즈 기반 `clip(dissolved + edgeWidth)` (`Shaders/FxMesh.hlsl:143`), erode 연출은 노이즈맵 `discard` (`Shaders/FxMesh.hlsl:171`~`:175`), 잔광은 알파 블렌드. "잘림이 룩의 일부인가"가 선택 기준이다.

30. **빌보드(billboard)란? 어떻게 구현하나?**
    한 줄: 항상 카메라를 향하는 쿼드 — 카메라의 right/up 벡터로 사각형을 전개한다: `pos = center + right * x * size + up * y * size`.
    ▸ "카메라 right/up은 어디서 얻나?" → 뷰 행렬의 역(=카메라 월드 행렬)의 축. 뷰 행렬이 정규직교라 전치에서 바로 읽을 수 있다. Winters FX가 정확히 이 구조다: 카메라에서 `GetRight`/`GetUp`을 꺼내 (`Client/Private/GameObject/FX/FxSystem.cpp:281`~`:282`) 쿼드 확장 벡터로 쓴다 (`:388`~`:389`).
    ▸ "종류는?" → 전방향(spherical)과 y축 고정(cylindrical — 나무/불기둥처럼 세워야 하는 것).

### A-9. API · 하드웨어

31. **DX11 vs DX12 vs Vulkan의 차이는?**
    한 줄: DX11은 드라이버가 해저드 추적·동기화·메모리를 암시적으로 처리해 주는 대신 CPU 오버헤드와 예측 불가능성(히칭)을 떠안고, DX12/Vulkan은 그 일을 앱에 명시적으로(explicit) 넘겨 — PSO 선컴파일, 커맨드리스트 병렬 기록, 수동 배리어/펜스 — CPU 병렬성과 예측 가능성을 산다.
    ▸ "그래서 DX12가 항상 빠른가?" → 아니다. 드라이버가 하던 일을 엔진이 잘 해낼 때만 빠르다. 못 하면 DX11 드라이버보다 못한 동기화를 짜게 된다 — "공짜 성능이 아니라 책임의 이동"이라고 답해야 한다.
    ▸ Winters는 이 차이를 인터페이스로 흡수했다: DX12 수준 계약(배리어/렌더패스)을 DX11 백엔드가 no-op으로 소화 — 구조 상세는 [engine/03 ③](../engine/03_rendering_pipeline.md) 참조.

32. **PSO(Pipeline State Object)란?**
    한 줄: 셰이더 + 블렌드/래스터/깊이 상태 + 입력 레이아웃 등 파이프라인 전체 상태를 하나의 **불변 객체로 선컴파일**한 것.
    ▸ "왜 이렇게 바꿨나?" → DX11의 조각난 상태 조합은 최종 GPU 코드가 드로우 시점에야 확정돼 드라이버가 그때 재컴파일(히칭)하는 구조였다. 전체 상태를 미리 알면 최종 ISA를 생성 시점에 한 번에 구울 수 있다 → 비용이 사라진 게 아니라 로딩 시점으로 **이동**한 것 → (B)13.
    ▸ Winters RHI의 대응물: `IRHIDevice::CreatePipeline` 순수 가상 계약 (`Engine/Public/RHI/IRHIDevice.h:88`).

33. **cbuffer packing 규칙은?**
    한 줄: HLSL 상수버퍼는 16바이트(float4) 레지스터 단위로 패킹되고 멤버는 레지스터 경계를 걸칠 수 없다 — `float3 + float`은 한 레지스터에 들어가지만 `float + float3`은 float3이 다음 레지스터로 밀린다.
    ▸ "틀리면 무� 일이 나나?" → 에러가 아니라 **조용히 어긋난 데이터** — C++ 구조체 오프셋과 셰이더 해석이 어긋난 채 그럴듯하게 그려져서 최악의 디버깅이 된다.
    ▸ Winters의 3중 방어: `static_assert(sizeof(T) % 16 == 0)` 컴파일타임 강제 (`Engine/Private/RHI/DX11/DX11ConstantBuffer.h:15`), 구조체에 명시적 `_padding` 필드 (`Engine/Public/Renderer/FxShaderConstants.h:13`, `:20`), 디바이스단 16바이트 올림. 런타임 디버깅 비용을 컴파일타임에 선지불한 사례.

## B. 심화 / 꼬리질문 (한 방에 무너지지 않기)

1. **reversed-Z는 왜 정밀도를 개선하나?**
   깊이 두 가지 비선형이 겹치기 때문이다. (1) 투영 깊이는 1/z 꼴이라 값들이 far 쪽에서 1 근처에 몰리고, (2) float 자체도 0 근처에서 지수부 덕에 촘촘하고 1 근처에서 성기다 — 두 조밀 구간이 **반대편**에 있어 서로 낭비한다. near=1, far=0으로 뒤집으면 "1/z가 몰아넣는 far 값들"이 "float가 촘촘한 0 근처"에 떨어져 분포가 준균등해진다. 요건: 비교 함수 GREATER, 클리어 0, 32bit float 깊이에서 극대화. D3D의 NDC z가 `[0,1]`이라 이 재배치가 깔끔하다(GL의 `[-1,1]`은 클립 제어 확장이 필요했던 이유).

2. **early-z와 discard의 관계는?**
   깊이 테스트는 명세상 PS 뒤(OM)지만, 하드웨어는 결과가 같다면 PS **앞**에서 미리 테스트해 가려진 픽셀의 PS를 통째로 건너뛴다(early-z). 그런데 PS가 `discard`/`clip`을 쓰면 "이 픽셀이 살지 죽을지"를 PS가 끝나야 알므로 **깊이 기록을 미리 확정할 수 없다** — 하드웨어는 depth write를 PS 이후로 미루고, 구현에 따라 early test 일부만 유지한다. `SV_Depth` 출력은 아예 early-z를 무효화한다. 알파테스트 식생(vegetation)이 비싼 구조적 이유고, Winters FX처럼 clip이 룩의 일부인 패스는 이 비용을 알고 지불하는 것이다 (`Shaders/FxMesh.hlsl:143`).

3. **perspective-correct 보간은 왜 필요하고 어떻게 하나?**
   래스터라이저는 스크린 공간에서 보간하는데, 원근 투영은 깊이에 따라 화면 밀도가 달라서 스크린 선형 == 월드 선형이 아니다(스크린 어파인 보간이 PS1 시절 텍스처 휘어짐의 원인). 수학적 사실: 원근 투영에서 스크린 공간 선형인 양은 `attr/w`와 `1/w`다. 그래서 하드웨어는 두 값을 선형 보간한 뒤 픽셀에서 나눈다:
   ```text
   attr(pixel) = lerp(attr/w) / lerp(1/w)
   ```
   `noperspective` 한정자가 이걸 끄는 스위치라는 것까지 알면 원리를 아는 것이다.

4. **HLSL의 row/column major와 전치 문제는?**
   두 개의 독립 축을 분리해야 안 헷갈린다: (1) **저장 순서** — 메모리에 행부터 깔리나 열부터 깔리나, (2) **곱셈 규약** — 벡터를 행벡터로 왼쪽에 두나(`mul(v, M)`) 열벡터로 오른쪽에 두나(`mul(M, v)`). HLSL의 기본 저장은 column-major고 DirectXMath는 row-major + 행벡터 규약이라, 아무 처리 없이 memcpy하면 전치가 끼어든다. 해결은 셋 중 하나로 **일관**시키는 것: CPU에서 전치해 올리거나, HLSL에 `row_major`를 선언하거나, 곱셈 순서를 뒤집어 전치를 흡수하거나. Winters는 cbuffer에 `row_major`를 명시하고 (`Shaders/FxMesh.hlsl:4`) `mul(vector, matrix)` 행벡터 규약을 쓴다 (`:58`~`:59`) — XMMATRIX 메모리를 전치 없이 그대로 올리는 선택. 기본이 column-major인 것은 행벡터 곱을 레지스터 단위 내적(dp4)으로 떨어뜨리기 유리했던 역사적 선택으로 알려져 있다.

5. **GGX가 Blinn-Phong NDF보다 나은 점은?**
   꼬리(tail)다. 실측 반사 데이터는 하이라이트 중심은 뾰족하되 주변으로 **천천히** 감쇠하는데, `pow(dot(n,h), p)` 꼴은 꼬리가 급격히 죽어 플라스틱처럼 보인다. GGX(Trowbridge-Reitz)는 유리 함수 꼴이라 긴 꼬리를 갖는다:
   ```text
   D_GGX(h) = a^2 / (pi * ((n.h)^2 * (a^2 - 1) + 1)^2),  a = roughness^2
   ```
   pi로 나누는 이유는 NDF 정규화 조건 — 미세면 법선 분포를 투영 입체각으로 적분하면 1이 되어야(`INT{ D(h) * (n.h) } dw = 1`) 미세면 총 면적이 보존된다. `a = roughness^2`는 지각 선형성을 위한 Disney 리매핑 — 슬라이더 중간이 시각적 중간이 되게 한 관례다.

6. **split-sum 근사란?**
   IBL 스펙큘러는 `INT{ Li(wi) * fr * (n.wi) } dwi`인데 매 픽셀 실시간 적분은 불가능하니, 적분을 **조명 항과 BRDF 항의 곱으로 분리**해 각각 사전 계산한다: (1) roughness별로 환경맵을 미리 블러한 prefiltered environment map, (2) BRDF 적분을 `(roughness, n.v)` 2D LUT로 구운 `F0 * scale + bias` 테이블. 런타임은 텍스처 두 번 조회로 끝난다. 대가는 두 항의 상관관계(가려진 방향의 조명과 BRDF 로브의 상호작용)를 버리는 것 — "왜 나눠도 대충 맞는가"까지 물으면 "로브가 좁을수록 Li가 로브 안에서 상수에 가깝다"로 답한다.

7. **F0 = 0.04의 근거는?**
   수직 입사 반사율은 굴절률(IOR)에서 나온다:
   ```text
   F0 = ((n1 - n2) / (n1 + n2))^2
   ```
   공기(1.0) → 유리/플라스틱류(약 1.5)를 넣으면 `(0.5 / 2.5)^2 = 0.04`. 대부분의 부도체가 IOR 1.3~1.6 범위(F0 0.02~0.05)에 몰려 있어 상수 하나로 뭉친 것이다. 물(1.33)은 0.02, 보석류는 더 높다 — "0.04는 측정값의 클러스터 중심이지 법칙이 아니다"까지 말하면 완성.

8. **TAA 고스팅의 원인과 대응은?**
   TAA는 과거 프레임 결과를 모션 벡터로 리프로젝션해 누적하는 시간축 슈퍼샘플링이다. 고스팅은 히스토리가 **거짓말**할 때 생긴다: (1) disocclusion — 이번 프레임에 새로 드러난 영역은 대응하는 과거가 없다, (2) 모션 벡터가 없는 움직임 — 투명체, 셰이더 내 UV 스크롤/버텍스 애니메이션. 표준 대응은 neighborhood clamping — 현재 프레임 주변 3x3 색 범위(AABB)로 히스토리를 클램프해 "지금 화면에 없는 색"을 기각한다. 클램프를 세게 걸면 고스팅 대신 앨리어싱/플리커가 돌아온다 — TAA 튜닝은 이 두 실패 사이의 다이얼이다.

9. **MSAA는 왜 디퍼드와 상성이 나쁜가?**
   MSAA의 절약 원리는 "커버리지/깊이는 샘플 단위, PS는 픽셀당 1회" — 삼각형 가장자리에서만 서브샘플이 갈라진다. 이 절약은 **래스터화 시점의 커버리지 정보**에 기대는데, 디퍼드는 조명을 G-buffer를 다시 읽는 별도 패스에서 하므로 그 시점엔 커버리지가 사라져 있다. 제대로 하려면 G-buffer 전체를 MSAA 타깃으로 만들고 조명을 샘플별로 돌려야 해서 메모리/대역폭이 배수로 뛰고 사실상 SSAA가 된다. 게다가 노멀·깊이 같은 데이터는 평균(resolve)하면 물리적으로 틀린 값이 된다. 그래서 디퍼드 엔진들은 포스트 프로세스 AA(FXAA → TAA)로 이동했다.

10. **워프 발산(warp divergence)이란?**
    GPU는 32~64개 스레드 묶음(warp/wavefront)이 같은 명령을 lockstep으로 실행한다(SIMT). warp 안에서 if 분기가 갈리면 하드웨어는 양쪽 경로를 **순차로 모두** 실행하며 해당 없는 스레드를 마스크한다 — 최악 경로 합만큼 느려진다. 대응: 분기가 화면에서 뭉치게(coherent) 설계하거나, `lerp`/마스크 산술로 분기를 데이터화한다. 단 "branch는 무조건 나쁨"은 과단정 — warp 전체가 같은 쪽으로 가면 분기는 거의 공짜고, 큰 계산을 통째로 건너뛰는 분기는 이득이다. Winters FX 셰이더의 styleMode 분기가 그 경우다 — 한 드로우의 전 픽셀이 같은 상수 값으로 분기하므로 발산이 없다 (`Shaders/FxMesh.hlsl:169`~`:179`).

11. **ddx/ddy는 무엇이고 밉 선택과 무슨 관계인가?**
    PS가 2x2 쿼드 단위로 실행되는 이유가 이것이다 — 이웃 픽셀과의 값 차이로 미분을 근사한다(`ddx`/`ddy`). 텍스처 샘플 시 하드웨어는 UV의 화면 미분으로 텍셀 발자국 크기를 재고 `lod = log2(footprint)`로 밉 레벨을 고른다. 함정: 동적 분기나 discard 이후의 `Sample`은 쿼드 이웃이 죽어 미분이 미정의가 된다 — 컴파일러가 gradient 경고를 내는 이유이고, `SampleLevel`/`SampleGrad`로 명시하는 것이 우회다. 알파테스트 잎사귀 가장자리의 밉 아티팩트도 같은 계열이다.

12. **리소스 배리어(barrier)는 왜 필요한가?**
    GPU 작업은 깊은 파이프라인에서 겹쳐 돌고 캐시는 자동 일관성이 없다. "렌더 타깃으로 쓰던 텍스처를 셰이더에서 읽기"로 용도를 바꾸는 순간 필요한 것이 세 가지다: (1) 이전 쓰기 완료 대기(실행 의존성), (2) 쓰기 캐시 플러시(메모리 가시성), (3) 압축 레이아웃 전환(RT 압축 포맷 → 샘플 가능 레이아웃). DX11은 드라이버가 바인딩을 추적해 암시적으로 처리했다 — 안전하지만 보수적 과동기화. DX12/Vulkan은 앱이 정확한 시점을 아니까 최소 동기화가 가능하지만, 틀리면 검은 화면이 아니라 **간헐적으로 깨진 데이터**가 나온다. Winters RHI가 배리어 API를 계약에 두고 DX11 백엔드에서 no-op으로 흡수하는 이유가 이 비대칭이다 ([engine/03 ③ 결정 3](../engine/03_rendering_pipeline.md)).

13. **드라이버 히칭과 PSO 프리워밍(prewarming)이란?**
    DX11에서 어떤 상태 조합이 **처음 등장하는 드로우** 시점에 드라이버가 셰이더를 최종 GPU ISA로 재컴파일하면 수십 ms 스파이크가 난다 — "전투에서 스킬을 처음 쓸 때 한 번 컥"의 전형적 원인. DX12는 이 비용을 PSO 생성으로 명시화했으니, 대응은 로딩 화면에서 쓸 조합을 전부 미리 생성(프리워밍)하고 파이프라인 캐시를 디스크에 저장하는 것. 요점은 "비용이 사라진 게 아니라 지불 시점의 통제권이 앱으로 왔다"는 것 — 반대로 조합 폭발을 방치하면 로딩이 분 단위가 되는 것이 DX12 시대의 새 문제다(셰이더 permutation 관리가 엔진 설계 이슈가 된 이유).

14. **오버드로우(overdraw)는 어떻게 측정하고 줄이나?**
    같은 픽셀이 여러 번 셰이딩되는 것. 측정: RenderDoc 등의 overdraw 오버레이, 또는 가산 블렌드로 드로우 횟수를 색으로 시각화. 주범은 거의 항상 파티클/FX — 큰 반투명 쿼드가 겹겹이 쌓인다. 대응: (1) 불투명은 앞→뒤 정렬로 early-z가 뒤를 죽이게, (2) 파티클은 픽셀 면적 자체를 줄임 — 알파 0 영역을 잘라낸 타이트한 지오메트리(8각형 스프라이트), (3) 저해상도 오프스크린 버퍼에 그려 업샘플. 반투명은 depth write가 꺼져 있어 early-z 도움을 못 받는다는 것이 문제의 뿌리임을 짚어야 한다 → (A)28.

## C. 경험 서술형 (STAR 골격 — 전부 Winters 실제 사건)

> 형식: **S**(상황) **T**(과제) **A**(행동) **R**(결과) + 이어질 꼬리질문. 구조/사건 서사는 [engine/03 ④](../engine/03_rendering_pipeline.md)와 중복되지 않게, 여기서는 **이론과의 연결 고리**를 붙인다.

### C-1. "Z-fighting을 해결해본 경험"

**주제: 소환사의 협곡 맵 초기 렌더링의 Z-fighting — 옵션 튜닝보다 데이터 정리**
사건 서사: [engine/03 ④-6](../engine/03_rendering_pipeline.md)

- **S**: LoL 맵을 처음 올렸을 때 겹친 지오메트리에서 Z-fighting이 났다.
- **T**: 어떤 층위에서 고칠지 — 렌더 상태(bias)인가, 데이터(메시)인가.
- **A**: 겹친 메시 자체를 정리하고 depth 설정을 손봤다. 전용 depth-bias 장치를 남기지 않은 것은 의도다 — bias는 acne/peter-panning처럼 **다른 오차와 교환하는 다이얼**이지 근본 해결이 아니고, 원인이 "같은 자리에 두 면이 실제로 존재"하는 데이터 문제였기 때문.
- **R**: 이후 Destroyed 넥서스 중첩 메시 사건에서도 같은 순서(데이터 먼저)로 진단해 재발을 빨리 잡았다.
- ▸ 꼬리질문: "이론적으로 왜 먼 곳에서만 났나?" → 투영 깊이가 `1/z` 분포라 far 쪽 정밀도가 급감하기 때문 — (A)10. "reversed-Z는 검토했나?" → 당시 파이프라인에선 근본 원인이 정밀도가 아니라 중복 지오메트리라 우선순위가 아니었다. 정밀도가 진짜 병목이 되면 GREATER 비교 + 클리어 0 + float 깊이로 도입한다 — (B)1로 원리를 설명할 수 있어야 한다.

### C-2. "3D 피킹을 직접 구현해본 경험"

**주제: 사일러스 호버 타겟팅 — 반직선 vs 실린더 해석적 교차**
`Client/Private/Scene/GameplayQuery.cpp:18`

- **S**: LoL식 조작에서 마우스 호버로 챔피언/미니언/구조물을 집어야 했다.
- **T**: 메시 정밀 교차 없이, 프레임마다 여러 유닛을 싸게 판정하기.
- **A**: 히트 볼륨을 수직 실린더로 모델링하고 해석적 교차를 짰다. 실린더는 축이 y축 평행이라 문제가 **XZ 평면의 반직선-원 2D 교차로 강하**된다: `(Ox + t*Dx)^2 + (Oz + t*Dz)^2 = r^2`을 전개하면 t의 2차식이고, half-b 판별식 `fB*fB - fLenSq*fC`로 근을 얻어 (`GameplayQuery.cpp:50`~`:52`) 교차점의 y가 실린더 높이 구간 안인지만 확인한다 (`:67`~`:68`). 거의 수직으로 내려꽂는 레이는 XZ 방향 성분이 0에 가까워 2차식이 퇴화하므로 캡 평면 교차로 분기했다 (`:26`~`:45`).
- **R**: 유닛당 O(1) 판정. 후보들 중 최소 t 선택으로 "가장 가까운 대상" 호버가 완성.
- ▸ 꼬리질문: "왜 캡슐/메시가 아니라 실린더인가?" → 탑다운 카메라에서 클릭 감각은 "세로로 긴 기둥"이 실제 실루엣보다 잘 맞고, 게임플레이 판정은 렌더 메시와 분리된 계약이어야 스킨/애니에 흔들리지 않는다. "스크린 좌표에서 레이는 어떻게 만드나?" → 뷰포트 역변환 → NDC → 투영 역 → 뷰 역의 언프로젝트 체인 — (A)4의 역방향을 말로 재생할 수 있어야 한다.

### C-3. "셰이더를 설계/분리해본 경험"

**주제: FxSprite / FxMesh 셰이더 분리 — 같은 파라미터 계약, 다른 수학**
`Shaders/FxSprite.hlsl:70` / `Shaders/FxMesh.hlsl:76`

- **S**: 이렐리아 FX를 만들며 빌보드 스프라이트와 3D 메시 이펙트가 같은 룩 언어(tint/erode/rim/dissolve)를 공유해야 했다.
- **T**: 셰이더 하나로 우겨넣을 것인가, 둘로 나눌 것인가.
- **A**: 상수버퍼 계약(`CBFxParams`)은 동일하게 두고 셰이더를 분리했다. 갈라야 했던 본질은 **rim의 수학**이다: 메시는 노멀이 있으니 프레넬형 `1 - dot(n, viewDir)`의 pow (`FxMesh.hlsl:76`~`:81`)가 성립하지만, 빌보드는 항상 카메라 정면이라 `dot(n, v)`가 상수에 가까워 같은 식이 무의미하다 — 스프라이트는 로컬 UV의 가장자리 거리 기반 가짜 rim으로 대체했다 (`FxSprite.hlsl:70`~`:74`). erode도 메시는 전용 노이즈맵 샘플(`FxMesh.hlsl:171`~`:175`), 스프라이트는 자기 밝기(luminance) 기반 clip(`FxSprite.hlsl:156`~`:160`)으로 지오메트리 특성에 맞게 갈랐다.
- **R**: 룩 파라미터는 한 계약으로 공유하면서, 각 지오메트리에서 수학적으로 유효한 식만 남았다.
- ▸ 꼬리질문: "왜 하나의 셰이더 + 분기로 안 했나?" → 두 경로는 입력 시맨틱부터 다르고(월드 노멀 유무), 성립하지 않는 수식을 분기로 덮으면 "왜 스프라이트에서 rim이 이상하죠" 류의 튜닝 미스터리가 된다. 분기는 같은 수학의 변형에만 쓴다 — styleMode 분기는 상수 기반이라 워프 발산도 없다 (B)10.

### C-4. "GPU 렌더링 버그를 디버깅해본 경험"

**주제: "호출은 됐는데 안 보임" — UV가 알파 0을 가리켜 전 픽셀이 clip으로 죽은 사건을 RenderDoc으로 확정**
`Engine/Private/Renderer/RHIFxSpriteRenderer.cpp:184`~`:191` / 사건 서사: [engine/03 ④-3](../engine/03_rendering_pipeline.md)

- **S**: 이렐리아 FX가 스폰 로그까지 다 찍히는데 화면에 아무것도 없었다.
- **T**: CPU 디버거로는 원인 후보를 좁힐 수 없었다 — 셰이더에서 `clip`으로 죽는 픽셀은 브레이크포인트가 없다.
- **A**: 코드 추론을 누적하는 대신 **데이터 계측**으로 전환했다. RenderDoc 캡처로 드로우가 실제 제출되고 픽셀이 PS까지 도달함을 확인 → 셰이더의 폐기 지점이 erode clip → alpha clip 이중 관문임을 읽고 (`RHIFxSpriteRenderer.cpp:184`~`:191`) → 텍스처의 알파 유효 영역(bbox)과 메시 UV 범위를 수치로 뽑아 대조 → 메시 UV가 텍스처의 **알파 0 영역**을 가리키고 있었다. 원인은 셰이더가 아니라 에셋: LoL 추출물의 `render/*.png`는 diffuse가 아니라 스프라이트 캡처였다.
- **R**: 진짜 머티리얼 텍스처(`*_texture.png`/`*_mult.png`)로 교체. clip 파이프라인은 룩의 핵심이라 그대로 유지 — 증상(클립)이 아니라 원인(입력 데이터)을 고쳤다. "안 보임 계열은 셰이더 Read + 데이터 계측 먼저"를 팀 디버깅 규칙으로 박제했다.
- ▸ 꼬리질문: "RenderDoc에서 구체적으로 뭘 보나?" → 이벤트 브라우저에서 드로우 존재 확인 → 파이프라인 상태에서 바인딩된 SRV/블렌드/깊이 확인 → 픽셀 히스토리로 해당 픽셀이 어느 단계(depth/scissor/shader discard)에서 죽었는지 — "GPU 버그는 어느 관문에서 죽는지부터 이분 탐색"이라고 답한다.

### C-5. "이펙트에 수학을 적용해본 경험"

**주제: 이렐리아 R 부채꼴 칼날 — 극좌표 파라미터화로 N개 배치**
`Client/Private/GameObject/Champion/Irelia/IreliaFxPresets.cpp:332`

- **S**: R 적중 지점에 칼날 여러 개가 부채꼴로 꽂히는 연출이 필요했다.
- **T**: 개수/퍼짐각/거리/모양(직사각 vs 삼각)을 데이터로 조절 가능한 배치 수학.
- **A**: 전방 벡터에서 `atan2f(forward.x, forward.z)`로 기준 yaw를 뽑고 (`:342` — 이 엔진은 +Z 전방 규약이라 인자 순서가 (x, z)다), i번째 칼날을 `t = i/(N-1)`로 [0,1] 정규화한 뒤 `angleOffset = -spread/2 + spread * t`로 좌우 대칭 부채를 만든다 (`:373`~`:379`). 각도 → 방향은 극좌표 변환 `dir = (sin(yaw), 0, cos(yaw))` (`:382`). 모양 제어는 `sideRatio = |2t - 1|` — 중앙 0, 가장자리 1인 삼각 프로파일로, 중앙 칼날은 앞으로 밀고(tipBoost) 가장자리는 축소(sideShrink)해 삼각형 전개를 만든다 (`:378`, `:386`~`:390`). 시작 위치와 최종 위치를 따로 두고 startDelay를 줘서 "펼쳐지는" 시간축까지 수학에 포함했다.
- **R**: 개수/각/거리/모양이 전부 파라미터인 재사용 가능한 배치 함수 — 다른 챔피언 부채꼴 스킬에 그대로 이식 가능.
- ▸ 꼬리질문: "N=1이면?" → `t = i/(N-1)`이 0으로 나누기가 되므로 `t = 0.5`(중앙) 특례를 뒀다 (`:375`~`:377`). 엣지 케이스를 수식 설계 단계에서 막은 것. "왜 yaw 스칼라 기반인가?" → 칼날은 지면에 꽂히는 연출이라 회전 자유도가 y축 하나뿐 — 쿼터니언은 과잉이다 (A)13과 같은 판단.

### C-6. "네트워크 동기화와 렌더링이 충돌한 경험"

**주제: 서버 권위 yaw가 로컬 예측 회전을 되감던 문제 — 보호 창과 각도 연속성 분리**
`Client/Private/Network/Client/SnapshotApplier.cpp:438`

- **S**: 우클릭 이동 시 클라가 예측으로 몸을 즉시 돌리는데, 아직 옛 yaw를 담은 서버 스냅샷이 도착해 덮어쓰면서 얼굴이 순간 되돌아가는 팝이 났다.
- **T**: 서버 권위를 유지하면서 로컬 회전 연출을 보호하기 — "권위"와 "표현"의 경계 긋기.
- **A**: 두 층으로 갈랐다. (1) **보호 창**: 클릭 시점의 예측 yaw를 등록하고(`ProtectLocalMoveYaw`, `:438`) 서버 yaw가 실제로 따라잡을 때까지(오차 0.20 rad 이내 판정) 최대 12스냅샷 동안 스냅샷 yaw 덮어쓰기를 보류한다(`:685`~`:693`). 단 서버가 반대 방향(반바퀴 대치)을 주장하면 즉시 서버 우선 — 권위 원칙 유지. (2) **각도 연속성**: 적용할 yaw를 그대로 쓰지 않고 `MakeChampionVisualYawNear`로 현재 각 근처의 동치 표현(`+2*pi*k`)을 골라 저장한다(`:694`) — 각도는 mod 2pi 동치류라서, wire의 정규화 값(-pi~pi)을 그대로 대입하면 ±pi 경계에서 반대 방향 최단 회전으로 보간되는 수학적 함정이 있기 때문.
- **R**: 클릭 반응성(즉시 회전)과 서버 권위(이내 수렴)가 공존. "정규화는 wire/로그 비교용, 연속 상태에는 near-표현" 규약을 팀 gotcha로 박제.
- ▸ 꼬리질문: "왜 각도 lerp가 위험한가?" → `lerp(179deg, -179deg)`는 수치상 358도를 거꾸로 도는 보간이 된다 — 각도는 선형 공간이 아니라 원(S1) 위의 값이라 최단 호 선택이 별도로 필요하다. 쿼터니언 slerp가 이 문제를 일반화 차원에서 푸는 도구라는 것까지 이으면 (A)13과 연결된다.

### C-7. "렌더러 추상화를 설계해본 경험"

**주제: IRHIDevice — 순수 가상 핵심 계약 + 점진 마이그레이션용 기본 구현의 절충**

- **S**: DX11 라이브를 유지한 채 DX12 백엔드를 붙여야 했다 — 전면 재작성은 라이브 리스크로 불가.
- **T**: 두 API의 실행 모델 차이(암시적 vs 명시적, (B)12·13)를 하나의 인터페이스로 덮되, 미완성 백엔드도 링크·실행 가능해야 한다.
- **A**: 인터페이스 수위를 이원화했다 — 파이프라인/렌더패스/바인드그룹 등 **아키텍처를 결정하는 계약은 순수 가상**으로 강제하고 (`IRHIDevice.h:88`의 `CreatePipeline = 0` 등), 그 외는 빈 핸들 기본 구현으로 점진 이행을 허용. 인터페이스 자체는 DX12 수준(배리어/렌더패스 존재)으로 잡고 DX11이 no-op으로 흡수하게 해서, 추상화가 "두 API의 교집합"으로 쪼그라드는 것을 피했다. 백엔드 구체 타입은 `GetNativeHandle`의 `void*` 뒤로만 노출 (`IRHIDevice.h:17`~`:18`).
- **R**: 같은 상위 코드가 두 백엔드에서 돌고, FX 신규 트랙 같은 실험은 DX12 가드 뒤에서 parity 검증 후 승격하는 운영이 가능해졌다. 핸들 모델/자원 테이블 등 구조 상세는 [engine/03 ③](../engine/03_rendering_pipeline.md)에 위임.
- ▸ 꼬리질문: "추상화를 교집합이 아니라 상위 API 기준으로 잡은 근거는?" → 하위(DX11)가 상위 개념을 no-op으로 무시하는 건 안전하지만, 상위(DX12)에 없는 암시적 동작을 흉내 내는 건 동기화 버그의 온상이다 — 추상화는 **더 명시적인 쪽**에 맞추는 게 원칙. "실패는 어떻게 드러나나?" → 빈 핸들 `IsValid()` 관례 + bounded 실패 로그로, 조용한 no-draw 대신 진단 가능한 실패로.
