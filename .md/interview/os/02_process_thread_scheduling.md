# 프로세스 · 스레드 · 스케줄링

> "프로세스와 스레드의 차이는 무엇인가요?", "컨텍스트 스위칭 비용이 왜 큰가요?", "OS 스케줄러는 어떻게 동작하나요?" — 신입 면접의 최빈출 3연타에 대한 답. 암기된 정의가 아니라 "왜 이런 추상화가 만들어졌고, 진짜 비용은 어디에 숨어 있는가"를 관통한다. 동기화 도구(뮤텍스·아토믹·메모리 모델)는 동시성 챕터의 몫이고, 여기서는 실행 단위와 그것을 CPU에 얹는 정책만 다룬다.

---

## ① 한 줄 본질

> **"프로세스는 '자원 소유와 보호의 단위'(가상 주소공간 + 자원 + PCB)이고, 스레드는 '실행과 스케줄링의 단위'(스택 + 레지스터 컨텍스트 + TCB)다. 컨텍스트 스위치의 진짜 비용은 레지스터 저장/복원(직접 비용, 마이크로초 안팎)이 아니라 TLB 플러시와 캐시 오염(간접 비용)이며, 그래서 같은 주소공간 안에서 커널을 거치지 않고 전환하는 파이버(fiber)가 빠르다. 스케줄러는 처리량·응답성·공정성이라는 서로 충돌하는 목표 사이의 정책 기계이고, FCFS→SJF→RR→MLFQ→CFS의 진화는 전부 '실행 시간을 미리 알 수 없다'는 한 가지 문제에 대한 답의 개선사다."**

이 문장에서 세 개의 축이 나온다.
1. **추상화의 분리** — 자원의 단위(프로세스)와 실행의 단위(스레드)는 다른 관심사라서 분리됐다.
2. **비용의 위치** — 전환 비용의 본체는 저장/복원이 아니라 "따뜻하던 캐시·TLB가 식는 것"이다.
3. **정책의 트레이드오프** — 스케줄링에 만능 정답은 없고, 무엇을 희생할지 고르는 문제다.

---

## ② 기본 개념

### 프로세스 — "혼자 기계를 다 쓰는 것 같은 환상"의 단위

왜 프로세스라는 추상화가 필요했나. CPU는 한 줌이고 물리 메모리는 한 장인데 프로그램은 여러 개를 동시에 돌리고 싶다. 그러려면 (1) CPU를 시분할로 **다중화(multiplexing)**해야 하고, (2) 한 프로그램의 버그가 다른 프로그램의 메모리를 짓밟지 못하게 **보호(protection)**해야 한다. OS의 해법은 각 프로그램에게 "너 혼자 CPU와 메모리 전체를 쓰는 것 같은 환상"을 주는 것이고, 그 환상의 단위가 프로세스다.

프로세스 = 세 가지의 묶음:

| 구성 요소 | 내용 | 주는 환상 |
|---|---|---|
| 가상 주소공간(virtual address space) | 코드·데이터·힙·스택이 놓이는 프로세스 전용 주소 체계 | "메모리 전체가 내 것" (상세는 가상 메모리 챕터) |
| 자원(resources) | 열린 파일·핸들, 소켓, 페이지 테이블 등 커널 객체 | "장치를 내가 소유" |
| PCB(process control block) | 커널이 이 프로세스에 대해 기록하는 장부 | 환상을 유지하는 실체 |

PCB에 들어가는 것: 프로세스 ID, 현재 상태, 저장된 레지스터 컨텍스트, 페이지 테이블 위치, 열린 파일 테이블, 스케줄링/회계 정보. Linux에서는 `task_struct`, Windows에서는 `EPROCESS`(+스케줄러용 `KPROCESS`)가 이 역할이다.

**핵심 관점: 프로세스는 물리적 실체가 아니라 커널 자료구조의 묶음이다.** "프로세스가 죽는다" = 커널이 그 장부를 파기하고 자원을 회수한다. "프로세스를 만든다" = 커널이 새 주소공간과 장부를 할당한다. 이 관점이 있어야 뒤의 모든 비용 이야기(생성 비용, 전환 비용)가 "어떤 자료구조를 만지는 작업인가"로 환원된다.

### 프로세스 상태 전이도 — 화살표마다 "누가, 왜"

```
                        디스패치(dispatch)
                       ┌──────────────────┐
  new ──admit──▶ ready │                  ▼ running ──exit──▶ terminated
                   ▲ ▲ └──────────────────┘   │ │
                   │ │                        │ │
                   │ └──── 선점(preempt) ◀────┘ │
                   │   타임슬라이스 소진,       │ 블로킹(block)
                   │   더 높은 우선순위 등장    │ I/O 요청, 락/이벤트 대기
                   │                            ▼
                   └────────── waiting ◀────────┘
                    I/O 완료·이벤트 시그널 → ready 로 복귀
```

외우는 게 아니라 화살표의 성격을 구분하는 게 요점이다.

- **running → waiting은 자발적이다.** 스레드가 스스로 블로킹 호출(I/O, 락 대기)을 했다. CPU 입장에서 "얘는 지금 CPU를 줘도 할 일이 없다"가 확정된 상태.
- **running → ready는 비자발적이다.** 할 일이 남았는데 스케줄러가 강제로 뺏었다(선점, preemption). 선점형 OS의 정의가 이 화살표의 존재다.
- **waiting → running 직행은 없다.** I/O가 끝나도 일단 ready로 가서 스케줄러의 심사를 다시 받는다. "깨어남"과 "실행됨"은 다른 사건이고, 이 간극이 나중에 우선순위 부스트(I/O 완료 부스트)가 존재하는 이유가 된다.

### 스레드 — 왜 "실행"만 따로 떼어냈나

초기 OS에서는 프로세스가 유일한 실행 단위였다. 두 가지가 아팠다. (1) 협력하는 작업 여러 개를 돌리려면 프로세스를 여러 개 만들어야 하는데, 생성도 전환도 비싸다(주소공간과 장부를 통째로 만들고 교체해야 하므로). (2) 그 작업들이 데이터를 공유하려면 IPC(프로세스 간 통신)라는 별도 통로를 뚫어야 한다 — 같은 메모리를 그냥 같이 보면 될 일인데.

여기서 관찰: **"자원을 소유한다"와 "실행 흐름이 흐른다"는 서로 다른 관심사다.** 이 둘을 분리한 것이 스레드다. 프로세스는 자원 소유의 단위로 남고, 스레드가 스케줄링(실행)의 단위가 된다. 한 프로세스 안에 실행 흐름 여러 개가 같은 주소공간을 공유하며 흐른다.

| | 스레드끼리 **공유** (프로세스 소유) | 스레드마다 **독립** (스레드 소유) |
|---|---|---|
| 메모리 | 코드(.text), 전역/정적 데이터, **힙** | **스택** (지역변수, 호출 이력) |
| 실행 상태 | — | **레지스터 컨텍스트** (PC, SP 포함) |
| 커널 장부 | 열린 파일/핸들 테이블, 소켓, 페이지 테이블 | **TCB(thread control block)**, 커널 스택, 스케줄링 상태·우선순위 |
| 언어 차원 | 전역 변수 | `thread_local`(TLS), errno |

```
┌────────────── 프로세스 가상 주소공간 (전부 공유) ──────────────┐
│  코드(.text)     전역/정적(.data/.bss)     힙(heap)            │
│  열린 파일 테이블 · 소켓 · 페이지 테이블  ← 프로세스가 소유    │
│                                                                │
│    스레드 A           스레드 B           스레드 C              │
│   ┌─────────┐        ┌─────────┐        ┌─────────┐            │
│   │ 스택 A  │        │ 스택 B  │        │ 스택 C  │ ← 각자     │
│   └─────────┘        └─────────┘        └─────────┘            │
│   레지스터·PC        레지스터·PC        레지스터·PC ← 각자     │
└────────────────────────────────────────────────────────────────┘
```

왜 스택은 공유할 수 없나 — 스택이 곧 **실행 흐름 그 자체**이기 때문이다. 스택에는 "지금 어떤 함수를 어떤 인자로 부르는 중이고, 리턴하면 어디로 돌아가는가"라는 호출 이력이 쌓여 있다. 실행 흐름이 둘이면 호출 이력도 둘이어야 한다. 반대로 힙과 전역은 "누가 실행 중인가"와 무관한 데이터라 공유가 자연스럽다.

이 설계의 대가가 바로 동시성 문제다. **힙·전역 공유가 기본값이므로 데이터 레이스(data race)도 기본값이다.** 스레드는 통신이 공짜인 대신 격리를 잃었다 — 프로세스와 정확히 반대의 트레이드오프. 이 대가를 어떻게 갚는지는 동시성 챕터의 주제다.

실무 감각 하나: Windows에서 스레드 스택은 기본적으로 1MB의 주소공간을 예약(reserve)한다. "클라이언트 연결 1개 = 스레드 1개" 모델로 1만 연결을 받으면 스택 예약만 10GB 규모가 된다. 스레드가 "가볍다"는 건 프로세스 대비 상대적인 말이지 공짜라는 뜻이 아니다 — 이 한계가 뒤에 나올 IOCP와 파이버의 존재 이유다.

### 컨텍스트 스위치 — 청구서는 두 장이다

컨텍스트 스위치(context switch)에서 실제로 일어나는 일:

```
[스레드 A 실행 중]
   │  타이머 인터럽트 / 시스템콜 / 블로킹 호출
   ▼
커널 모드 진입
   ▼
A의 레지스터·PC·SP 를 A의 커널 측 컨텍스트(TCB/커널 스택)에 저장
   ▼
스케줄러 실행: ready 큐에서 다음 스레드 B 선택
   ▼
(B가 다른 프로세스라면) 페이지 테이블 교체 → TLB 무효화
   ▼
B의 컨텍스트 복원 → 유저 모드 복귀
   ▼
[스레드 B 실행 중]  ← 그러나 캐시에는 아직 A의 데이터가 가득
```

**직접 비용** — 레지스터 저장/복원, 모드 전환, 스케줄러 실행. 이건 생각보다 싸다. 대략 마이크로초 안팎 수준으로, "레지스터 몇십 개를 메모리에 쓰고 읽는 일"이 본질이라 하드웨어가 빠르게 처리한다.

**간접 비용** — 진짜 청구서는 전환이 끝난 **뒤에** 날아온다.

1. **TLB 플러시(TLB flush).** TLB는 가상→물리 주소 변환 결과를 담는 캐시다. 프로세스가 바뀌면 주소공간이 바뀌므로 기존 변환 항목이 전부 무효가 된다. 태깅이 없는 단순한 구현에서는 페이지 테이블 교체 시 TLB를 통째로 비우고, 이후 한동안 메모리 접근마다 페이지 테이블을 다시 걷는(page walk) 비용을 낸다. 하드웨어의 완화책이 x86의 PCID, ARM의 ASID 같은 **주소공간 태그** — TLB 항목에 "어느 프로세스 것인지" 꼬리표를 달아 플러시 없이 공존시킨다.
2. **캐시 오염(cache pollution).** B가 실행되며 자기 워킹셋으로 캐시 라인을 채우면 A의 데이터가 밀려난다. A가 다시 스케줄되면 콜드 캐시에서 시작 — 미스를 연발하며 수 MB의 L2/L3를 다시 채울 때까지 성능이 깎인다. 이 간접 손실이 직접 비용보다 훨씬 클 수 있다.

여기서 전환 종류별 비용 서열이 나온다:

| 전환 종류 | 커널 경유 | 페이지 테이블 교체 | TLB | 상대 비용 |
|---|---|---|---|---|
| 프로세스 간 전환 | O | O | 플러시(태그 없으면) | 최고 |
| 같은 프로세스 스레드 간 전환 | O | X (주소공간 동일) | 생존 | 중간 |
| 파이버/코루틴 전환 | **X** | X | 생존 | 최저 |

같은 프로세스의 스레드 전환이 프로세스 전환보다 싼 이유가 이 표 한 줄이다 — **주소공간이 같아서 페이지 테이블을 갈아끼우지 않는다.**

게임 관점의 결론: 프레임 예산이 16.6ms인 세계에서, 코어 수보다 많은 러너블 스레드를 만드는 것(oversubscription)은 스위칭 직접 비용 + 캐시 오염 세금을 프레임마다 내겠다는 뜻이다. 그래서 게임 엔진의 워커 스레드 수는 "많을수록 좋다"가 아니라 **코어 수 근처**로 잡는다.

### 스레드는 누가 스케줄하나 — 1:1, N:1, M:N

"스레드"라는 말은 사실 두 층을 가리킬 수 있다. 커널이 아는 스레드(커널 스레드)와, 유저 공간 라이브러리가 만들어낸 실행 단위(유저 레벨 스레드). 둘의 매핑 방식이 모델을 가른다.

| 모델 | 스케줄 주체 | 전환 비용 | 블로킹 시스템콜 | 멀티코어 병렬 | 예 |
|---|---|---|---|---|---|
| 1:1 (커널 스레드) | 커널 | 커널 경유 | 그 스레드만 블록 | O | Windows 스레드, Linux NPTL(std::thread의 실체) |
| N:1 (유저 레벨) | 유저 라이브러리 | 함수 호출 수준 | **전체 프로세스 블록** | X | 초기 Java green threads |
| M:N (하이브리드) | 양쪽 협조 | 유저 단위는 초저가 | 커널 스레드 단위로 블록 | O | Go 고루틴, 게임 잡시스템+파이버 |

- **1:1**: 커널이 모든 스레드를 알고 직접 코어에 배치한다. 블로킹해도 다른 스레드가 돌고, 진짜 병렬이 된다. 대신 생성·전환마다 커널을 거친다.
- **N:1**: 커널은 스레드 1개만 안다. 라이브러리가 유저 모드에서 초고속 전환을 한다. 치명적 약점 — 유저 스레드 하나가 블로킹 시스템콜을 하면 커널은 "그 프로세스의 유일한 스레드가 잤다"고 보고 **전부** 멈춘다. 멀티코어도 못 쓴다. 그래서 순수 N:1은 도태됐다.
- **M:N**: 가벼운 실행 단위 M개(수천)를 커널 스레드 N개(≈코어 수) 위에 얹는다. 전환의 저비용과 멀티코어 병렬을 다 가지는 대신, 유저 스케줄러와 커널 스케줄러가 협조해야 하는 복잡성(블로킹 감지 등)을 떠안는다.

게임 엔진의 잡시스템(job system)이 M:N으로 수렴하는 이유: 한 프레임에 처리할 작업(잡)은 수백~수천 개인데, 이걸 커널 스레드로 만들면 생성·스위칭 세금이 프레임 예산을 먹는다. 그래서 **잡은 유저 모드 큐에 쌓고, 코어 수만큼의 워커 스레드가 커널 개입 없이 꺼내 실행**한다.

```
잡/파이버 (M, 프레임당 수천)   J1 J2 J3 J4 J5 ... J2000   ← 유저 모드 분배(잡시스템)
                                 \  |  /       \  |  /
워커 커널 스레드 (N ≈ 코어 수)     W0    W1  ...   W5      ← 커널 스케줄러가 코어에 배치
                                   │     │         │
물리 코어                          C0    C1  ...   C5
```

### 파이버 · 코루틴 — 커널 없는 전환, "협조"라는 계약

스케줄링에는 두 가지 체제가 있다. **선점형(preemptive)**은 OS가 타이머 인터럽트로 강제로 CPU를 뺏는다. **협조형(cooperative)**은 실행 중인 코드가 스스로 양보(yield)할 때만 전환이 일어난다.

역사가 교훈을 준다. Windows 3.x와 클래식 Mac OS는 협조형 멀티태스킹이었다 — 앱 하나가 양보를 안 하면 시스템 전체가 멈췄다. 그래서 범용 OS는 선점형으로 이동했다. **신뢰할 수 없는 코드들 사이에서는 협조가 성립하지 않는다.** 그런데 게임 서버는 사정이 다르다 — 한 프로세스 안, 전부 내가 짠 코드끼리다. 협조 계약을 지킬 수 있는 환경이라면, 협조형의 이점(커널 없는 전환, 전환 지점의 완전한 통제)만 취할 수 있다. 그 도구가 파이버(Windows `CreateFiber`/`SwitchToFiber`)와 코루틴(coroutine)이다.

**파이버 전환이 빠른 이유를 단계별로 분해하면:**

1. **커널 모드 진입이 없다.** `SwitchToFiber`는 유저 모드 함수다. 시스템콜 진입/복귀 비용 자체가 사라진다.
2. **스케줄러가 실행되지 않는다.** "다음에 누구를 돌릴지"를 코드가 직접 지정한다. 심사 비용 0.
3. **저장할 상태가 적다.** 함수 호출 규약상 휘발성 레지스터는 호출자가 이미 포기한 상태라, 비휘발성 레지스터 + 스택 포인터/명령 포인터 정도만 바꿔치우면 된다.
4. **주소공간이 그대로다.** 같은 프로세스 안이므로 TLB 무효화가 없고, 이어서 같은 데이터를 만지면 캐시도 따뜻하다.

결과적으로 전환이 "함수 호출 몇 개" 수준의 비용까지 내려간다.

Win32 파이버의 최소 형태는 이렇다:

```cpp
// 워커 스레드 자신을 파이버로 변환 (파이버 세계로 들어가는 관문)
LPVOID hMain = ConvertThreadToFiber(nullptr);

// 잡을 실행할 파이버 생성 (스택 크기 0 = 기본값)
LPVOID hJob = CreateFiber(0, [](void* p) {
    // ... 잡 실행. 도중에 SwitchToFiber(hMain) 하면
    //     이 스택이 통째로 얼어붙고, 나중에 이 지점에서 재개된다.
    SwitchToFiber(/*복귀 대상*/ static_cast<LPVOID>(p));
    // 주의: 파이버 함수가 return 하면 스레드 자체가 종료된다 (Win32 사양)
}, hMain);

SwitchToFiber(hJob);   // ← 여기서 제어가 hJob으로 넘어가고,
                       //    hJob이 양보/완료하면 이 다음 줄에서 재개
DeleteFiber(hJob);
```

커널 함수는 하나도 없다 — 전환의 실체는 유저 모드에서의 레지스터·스택 포인터 교체다. 그리고 마지막 주석이 협조형의 성격을 요약한다: 양보 규약을 어기는 순간(그냥 return) 안전망 없이 스레드가 통째로 사라진다.

**게임 서버가 파이버를 쓰는 이유** — 게임 로직은 시간에 걸쳐 있는 서사다: "스킬 시전 시작 → 0.5초 캐스팅 대기 → 판정 → 쿨다운". 이걸 콜백/상태머신으로 쪼개면 하나의 논리가 수십 조각으로 흩어진다. 스레드로 만들면 순차 코드로 쓸 수 있지만 "0.5초 대기 중"인 수천 개의 스레드가 스택과 스케줄링 비용을 태운다. 파이버는 제3의 길이다 — **대기 지점에서 스택을 통째로 얼려두고(suspend) 커널 스레드는 다른 파이버를 돌리다가, 시간이 되면 그 자리에서 재개(resume)한다.** 순차 코드의 가독성과 M:N의 효율을 동시에 가진다.

**파이버의 함정 3종** (면접 단골):
1. **블로킹 시스템콜.** 커널은 파이버의 존재를 모른다. 파이버 안에서 블로킹 I/O를 하면 그 밑의 커널 스레드가 통째로 잠들고, 그 스레드에 매달린 다른 파이버 전부가 멈춘다. 그래서 파이버는 비동기 I/O(IOCP 등)와 세트로 쓴다.
2. **TLS 배신.** `thread_local` 변수는 커널 스레드에 묶인다. 파이버가 스레드 X에서 잠들었다가 스레드 Y에서 재개되면 다른 TLS를 본다. 파이버를 스레드 간 이동시키는 설계라면 TLS 캐싱이 곧 버그다.
3. **양보 없는 루프.** 협조 계약 위반 — 한 파이버가 양보 없이 돌면 그 워커에 걸린 작업 전부가 굶는다. 선점이라는 안전망이 없다는 대가.

**stackful vs stackless** — 파이버는 스택풀(stackful): 진짜 스택을 통째로 가지므로 호출 깊이 어디서든 양보할 수 있지만, 파이버마다 스택 메모리를 예약한다. C++20 코루틴은 스택리스(stackless): 컴파일러가 함수를 상태머신으로 변환해 **살아남아야 할 지역변수만** 코루틴 프레임에 저장한다. 프레임이 수십~수백 바이트로 가벼운 대신, `co_await`는 코루틴 함수 본문 안에서만 가능하다(깊은 콜스택 중간에서 양보 불가). "무엇을 저장하는가"의 차이가 유연성과 비용의 차이를 만든다.

### 스케줄링 정책 — 세 목표는 동시에 가질 수 없다

| 목표 | 측정 | 극대화하면 희생되는 것 |
|---|---|---|
| 처리량(throughput) | 단위 시간당 완료 작업 수 | 응답성 (전환을 줄이면 대화형이 굶음) |
| 응답성(response time) | 요청→첫 반응 지연 | 처리량 (잦은 전환 = 스위칭 세금) |
| 공정성(fairness) | CPU 몫의 균등/가중 분배 | 둘 다 조금씩 (중요한 놈만 밀어주지 못함) |

배치 서버는 처리량, 데스크톱은 응답성, 다중 사용자 서버는 공정성이 우선이다. **만능 스케줄러는 없고, 알고리즘의 역사는 "무엇을 포기할지"를 다듬어온 역사다.** 각 단계가 이전 단계의 어떤 실패를 고쳤는지가 서사의 뼈대다.

**FCFS(first-come, first-served)** — 온 순서대로. 단순하지만 **호위 효과(convoy effect)**가 치명적: 긴 작업 하나 뒤에 짧은 작업들이 줄줄이 묶인다. 마트 계산대에서 카트 가득 채운 손님 뒤에 껌 하나 든 손님들이 늘어선 그림.

**SJF(shortest job first)** — 짧은 것부터. 평균 대기시간이 수학적으로 최적임이 증명된다(긴 작업과 짧은 작업의 순서를 바꾸면 항상 평균이 좋아진다는 교환 논증). 그러나 두 가지가 부순다: (1) **실행 시간을 미리 알 수 없다** — 미래 정보를 요구하는 알고리즘은 실전에 못 쓴다. (2) 짧은 작업이 계속 들어오면 긴 작업이 영원히 굶는 **기아(starvation)**.

**RR(round robin)** — 미래를 모르니 시간을 잘게 나눠 돌아가며 준다. 타임 퀀텀(time quantum)이 응답성을 보장한다. 퀀텀 크기가 그대로 트레이드오프다: 짧으면 스위칭 오버헤드가 비중을 먹고(퀀텀 1ms에 스위칭 간접비용까지 수십 μs면 세금이 수 %), 길면 FCFS로 퇴화한다. 교과서의 경험칙은 "CPU 버스트 대부분이 퀀텀 안에 끝나도록" 잡는 것.

**MLFQ(multi-level feedback queue)** — RR의 남은 문제: 모든 작업을 평등하게 취급해서, I/O 위주 대화형 작업(짧게 쓰고 양보)과 CPU 위주 작업을 구분 못 한다. MLFQ의 통찰: **미래는 모르지만 과거는 안다.** 우선순위 큐를 여러 단 두고, 퀀텀을 다 쓰는 놈은 강등(CPU 바운드로 판정), 퀀텀을 남기고 양보하는 놈은 우선순위를 유지(대화형으로 판정)한다. 과거 행동으로 미래를 예측하는 학습 스케줄러다. 보정 장치 두 개가 필수다 — 저층에서 굶는 걸 막는 **주기적 우선순위 부스트**, 그리고 "퀀텀 끝나기 직전에 양보해서 고층에 눌러앉는" 꼼수를 막는 **누적 사용시간 회계**.

**Linux CFS(completely fair scheduler)** — 발상을 바꾼다. 우선순위 큐 대신 "**이상적으로 공정한 CPU**(모두가 동시에 1/n씩 실행되는 가상의 기계)"를 근사한다. 각 태스크의 **vruntime(가상 실행시간)** — 실제 실행시간을 nice 가중치로 나눈 값 — 을 기록하고, 항상 vruntime이 가장 작은(= 지금까지 몫을 가장 못 받은) 태스크를 실행한다. 태스크들을 vruntime 순으로 **레드블랙 트리(red-black tree)**에 담아 최좌측 노드를 뽑는다 — 삽입/삭제 O(log n), 다음 태스크 선택은 최좌측 캐싱으로 O(1). nice 값은 우선순위 서열이 아니라 **몫의 가중치**로 작동한다. (참고: Linux 6.6부터 CFS는 지연 시간 요구까지 반영하는 EEVDF 스케줄러로 교체됐다 — "공정한 몫" 모델 자체는 계승된다.)

진화의 서사를 한 줄로 정리하면:

```
FCFS ──(호위 효과)──▶ SJF ──(미래를 모름 + 기아)──▶ RR
 RR ──(모두를 평등 취급, 대화형 구분 불가)──▶ MLFQ
MLFQ ──(우선순위 계단의 튜닝 지옥, 기아 보정의 땜질)──▶ CFS (공정한 몫 모델)
```

각 화살표의 괄호 — 즉 **이전 세대가 실패한 지점** — 를 말할 수 있으면 알고리즘 나열이 아니라 설계사를 이해한 답이 된다.

### 멀티코어 스케줄링 — 큐는 코어마다 있다

지금까지의 그림은 ready 큐가 하나인 단일 CPU 모델이었다. 멀티코어에서 전역 큐 하나를 모든 코어가 공유하면 두 가지가 무너진다. (1) **큐 자체가 락 경합 지점**이 된다 — 코어가 늘수록 스케줄러가 병목이 되는 자기모순. (2) 태스크가 매번 아무 코어에나 배치되므로 **캐시 온기가 계속 깨진다.**

그래서 현대 OS는 **코어마다 자기 실행 큐(per-CPU runqueue)**를 둔다. 각 코어는 평소 자기 큐만 보므로 경합이 없고, 태스크는 같은 코어에 머무는 경향이 생겨 캐시가 산다. 대신 새 문제가 생긴다 — 큐들 사이의 **불균형**. 한 코어에 일이 몰리고 옆 코어가 놀 수 있다. 해법이 부하 분산(load balancing): 유휴 코어가 바쁜 코어의 큐에서 태스크를 끌어온다(일종의 OS 레벨 워크스틸링).

여기에 본질적 긴장이 있다. **부하 균형을 위해 태스크를 옮기면 캐시 온기를 잃고, 캐시를 지키려고 안 옮기면 코어가 논다.** 그래서 스케줄러는 "얼마나 굶어야 옮길 가치가 있는가"를 저울질하며, 물리 코어/캐시 공유/소켓 경계 같은 하드웨어 위상(topology)을 반영해 가까운 곳에서 먼저 훔친다. CPU 친화도 API는 이 저울질에 프로그래머가 개입하는 손잡이이고, 게임 엔진의 잡시스템이 워커별 데크 + 스틸(뒤 ④절)을 쓰는 것도 정확히 같은 구조를 유저 모드에 복제한 것이다 — **로컬 큐로 경합과 캐시 오염을 피하고, 스틸로 불균형을 회복한다.**

### Windows 스케줄러 — 게임이 실제로 올라가는 무대

Winters의 클라이언트도 서버도 결국 이 위에서 돈다.

```
 31 ┐
    │  실시간 범위 (16~31): 부스트/감쇠 없음. OS 핵심 스레드와 경쟁.
 16 ┘
 15 ┐
    │  동적 범위 (1~15): 베이스 우선순위 + 일시 부스트, 퀀텀마다 1씩 감쇠
  1 ┘
  0     zero-page thread 전용 (유휴 시 빈 페이지를 0으로 채우는 커널 스레드)
```

- **0~31 우선순위, 엄격한 선점.** 스레드의 베이스 우선순위는 프로세스 우선순위 클래스(NORMAL_PRIORITY_CLASS 등)와 스레드 상대 우선순위의 조합으로 정해진다. 같은 레벨끼리는 RR로 돌고, **더 높은 레벨의 스레드가 ready가 되는 순간 즉시 선점한다.** 낮은 레벨은 높은 레벨이 전부 잘 때만 돈다.
- **퀀텀(quantum).** 클록 인터럽트 몇 회 분량으로 정의된다. 클라이언트 SKU는 짧은 퀀텀(응답성 우선), 서버 SKU는 긴 퀀텀(처리량·캐시 온기 우선)이 기본값이고, 클라이언트에서는 포그라운드 프로세스에 퀀텀을 우대한다 — 데스크톱과 서버의 목표 트레이드오프가 제품 설정으로 박제된 예.
- **우선순위 부스트(priority boost).** I/O 완료, 대기하던 이벤트/세마포어의 시그널, 포그라운드 GUI 활동 등이 일어나면 스레드 우선순위를 일시적으로 끌어올리고, 이후 퀀텀이 지날 때마다 1씩 베이스로 감쇠시킨다. 왜? — I/O를 마치고 막 깨어난 스레드를 빨리 돌려줘야 응답성이 살고 다음 I/O가 빨리 나간다(waiting→ready→running 간극 줄이기). 오래 굶은 ready 스레드를 주기적으로 크게 부스트하는 기아 방지 장치도 있다. 부스트는 동적 범위(16 미만)에서만 작동한다.
- **실시간 범위의 함정.** 16 이상은 부스트도 감쇠도 없는 진짜 고정 우선순위로, 시스템 스레드들과 경쟁한다. 게임이 REALTIME_PRIORITY_CLASS를 잡으면 빨라지는 게 아니라 입력·오디오·커널 작업을 굶겨 시스템이 불안정해진다. "우선순위를 올리면 빨라진다"는 직관이 배신당하는 지점.

**우선순위 역전(priority inversion)** — 고정 우선순위 체계의 구조적 함정. 낮은 우선순위 L이 락을 쥔 채, 중간 우선순위 M이 CPU를 점유하면, 락을 기다리는 높은 우선순위 H가 사실상 M보다 낮게 돈다(H는 L이 락을 놔야 진행하는데 L은 M에 밀려 실행을 못 하므로). 실제 사고: 1997년 화성 탐사선 Mars Pathfinder가 이 패턴으로 반복 리셋됐다 — 저순위 기상 데이터 태스크가 버스 뮤텍스를 쥔 사이 중간 순위 통신 태스크가 CPU를 먹어, 고순위 버스 관리 태스크가 데드라인을 놓치고 워치독이 시스템을 리셋했다. 지구에서 우선순위 상속 옵션을 원격으로 켜서 해결했다. 정석 해법이 그 **우선순위 상속(priority inheritance)** — 락을 쥔 스레드를 대기자 중 최고 우선순위로 일시 승격 — 과 **우선순위 실링(priority ceiling)**이다. Windows는 완전한 상속 대신 부스트 메커니즘(굶은 스레드 부스트 등)으로 완화하는 노선이다.

**CPU 친화도(affinity)** — 스레드를 특정 코어(집합)에 묶는 것. `SetThreadAffinityMask`는 강제(hard affinity), `SetThreadIdealProcessor`는 "가능하면 이 코어로"라는 힌트(soft)다. 목적은 **캐시 온기(cache warmth)** — 같은 스레드가 같은 코어에서 돌면 L1/L2에 남은 자기 데이터를 재활용한다. 게임 서버 튜닝의 방향: 틱 루프나 네트워크 워커처럼 지속적으로 뜨거운 스레드는 코어를 나눠 서로의 캐시를 오염시키지 않게 한다. 단, 하드 고정은 스케줄러의 손발을 묶는 양날의 검이다 — 고정된 코어가 바쁘면 다른 코어가 놀아도 옮겨가지 못한다. 그래서 확신이 없으면 ideal processor 힌트가 먼저다.

## ③ 심화 — 꼬리질문 대비

**"프로세스와 스레드의 차이를 커널 자료구조 관점에서 설명해 보라."**
프로세스를 만들면 커널은 새 가상 주소공간(페이지 테이블)과 프로세스 장부(PCB — Linux `task_struct`, Windows `EPROCESS`)를 만든다. 같은 프로세스에 스레드를 추가하면 주소공간·파일 테이블은 그대로 두고 스레드 장부(TCB)와 커널 스택, 유저 스택만 추가한다. "스레드가 가볍다"의 실체는 **만들고 교체해야 할 커널 자료구조가 적다**는 것이다. 전환 비용 차이도 같은 뿌리 — 프로세스 전환만 페이지 테이블을 갈아끼운다.

**"컨텍스트 스위치 때 정확히 무엇이 저장되나?"**
저장되는 것은 CPU 실행 상태 — 범용 레지스터, 프로그램 카운터, 스택 포인터, 플래그 — 이고 스레드의 커널 측 컨텍스트에 기록된다. 저장되지 **않는** 것이 요점이다: 캐시와 TLB의 내용은 저장/복원 대상이 아니라 그냥 **잃는다**. 그래서 직접 비용(레지스터)은 싸고, 돌아왔을 때 식어 있는 캐시·TLB를 다시 데우는 간접 비용이 본체다.

**"같은 프로세스 스레드 간 전환이 프로세스 간 전환보다 싼 이유는?"**
주소공간이 같아서 페이지 테이블을 교체하지 않기 때문이다. 페이지 테이블 교체는 TLB의 기존 변환을 무효로 만들므로(태그가 없다면 플러시), 프로세스 전환 후에는 메모리 접근마다 페이지 워크 비용을 다시 낸다. 스레드 전환은 이 단계 자체가 없다. 커널 모드 진입과 스케줄러 실행은 둘 다 동일하게 낸다 — 차이는 순수하게 주소공간 교체 여부다.

**"TLB 플러시를 피하는 하드웨어 기법을 아는가?"**
태그드 TLB — 항목마다 주소공간 식별자를 붙여 여러 프로세스의 변환이 공존하게 한다. x86의 PCID(process-context identifier), ARM의 ASID가 그것이다. 컨텍스트 스위치 때 통째로 비우는 대신 "현재 태그와 다른 항목은 무시"로 바꿔, 짧게 나갔다 돌아온 프로세스가 자기 TLB 항목을 재활용할 수 있게 한다.

**"컨텍스트 스위치 비용을 실제로 측정한다면?"**
고전적 방법은 파이프 두 개로 두 프로세스(또는 스레드)가 1바이트를 핑퐁시키며 왕복 시간을 재는 것 — 왕복 1회에 스위치 2회가 강제되므로 (왕복시간 - 순수 통신비용)/2 로 근사한다. 단 이 방식은 직접 비용 위주로 잡힌다. 간접 비용까지 보려면 각 스레드가 큰 워킹셋을 만지게 한 뒤 스위치 빈도를 바꿔가며 전체 처리량 저하를 봐야 한다 — "스위치 1회 = X ns"라는 단일 숫자가 없다는 것 자체가 정답의 일부다.

**"스레드를 코어 수보다 많이 만들면 정확히 무엇이 나빠지나?"**
러너블 스레드가 코어보다 많으면(oversubscription) 스케줄러가 강제로 시분할한다. (1) 스위칭 직접 비용이 빈도만큼 쌓이고, (2) 스레드들이 서로의 캐시를 밀어내 전부가 콜드 캐시로 돌며, (3) 스택 메모리가 스레드 수만큼 예약되고, (4) 락 홀더가 선점당하면 락 대기자들이 연쇄로 굶는다(락 홀더 선점 문제). 그래서 CPU 바운드 워커는 코어 수 근처, 대기가 많은 I/O 워커만 그보다 여유 있게 잡는 것이 정석이다.

**"파이버 전환이 스레드 전환보다 빠른 이유를 단계별로 분해해 보라."**
스레드 전환 비용 = 커널 모드 진입 + 컨텍스트 저장 + 스케줄러 실행 + (다른 프로세스면 주소공간 교체) + 복원 + 간접 비용. 파이버 전환은 이 중 **커널 진입과 스케줄러 실행이 통째로 없다** — 유저 모드 함수가 비휘발성 레지스터와 스택/명령 포인터만 바꿔치운다. 다음 실행 대상도 코드가 직접 지정하므로 심사 비용이 없다. 같은 주소공간이라 TLB도 그대로다. 남는 건 사실상 레지스터 교체뿐이라 함수 호출 몇 개 수준까지 내려간다.

**"파이버의 대표적 함정은?"**
(1) 커널이 파이버를 모른다 — 파이버 안 블로킹 시스템콜은 그 밑의 커널 스레드와 거기 매달린 파이버 전부를 멈춘다. 그래서 비동기 I/O와 세트다. (2) `thread_local`은 커널 스레드에 묶인다 — 파이버가 다른 스레드에서 재개되면 다른 TLS를 본다. (3) 선점이 없다 — 양보 안 하는 파이버 하나가 워커를 독점한다. 셋 다 "커널의 안전망을 포기하고 얻은 속도"라는 동일한 뿌리에서 나온다.

**"C++20 코루틴과 파이버의 차이는?"**
저장 단위가 다르다. 파이버는 스택풀 — 진짜 스택을 통째로 소유해서 호출 깊이 어디서든 양보할 수 있지만 파이버마다 스택 메모리를 예약한다. C++20 코루틴은 스택리스 — 컴파일러가 함수를 상태머신으로 변환해 중단점을 넘어 살아남을 지역변수만 프레임에 담는다. 프레임이 수십~수백 바이트라 수만 개를 띄워도 가볍지만, `co_await`는 코루틴 본문에서만 가능해 깊은 콜스택 중간에서는 양보할 수 없다. "일반 함수들을 그대로 두고 얼릴 수 있는가"가 실무적 분기점이다.

**"SJF가 평균 대기시간 최적인데 왜 실제 OS는 못 쓰나?"**
실행 시간이라는 **미래 정보**를 입력으로 요구하기 때문이다. OS는 작업이 얼마나 걸릴지 모른다. 그래서 실전 스케줄러는 전부 근사 전략이다 — RR은 "모르니 잘게 나눈다", MLFQ는 "과거 행동으로 추정한다"(지수 평균으로 다음 CPU 버스트를 예측하는 변형도 같은 정신). 추가로 SJF는 짧은 작업이 계속 유입되면 긴 작업이 영원히 밀리는 기아 문제도 있다 — 최적성과 공정성이 충돌하는 예.

**"MLFQ를 악용해 우선순위를 유지하려는 프로그램은 어떻게 막나?"**
꼼수: 퀀텀이 끝나기 직전에 잠깐 I/O나 양보를 해서 "대화형인 척" 고층에 눌러앉는 것. 방어: 강등 판정을 "퀀텀을 한 번에 다 썼는가"가 아니라 **그 층에서의 누적 CPU 사용시간**으로 바꾼다 — 잘게 나눠 써도 총량이 차면 강등된다. 여기에 주기적 전체 부스트로 저층 기아까지 막는다. "스케줄러는 적대적 입력을 가정해야 한다"는 관점을 보여주면 좋은 답이 된다.

**"CFS는 왜 레드블랙 트리인가?"**
필요한 연산이 (1) vruntime 순서 유지 삽입/삭제, (2) 최소 vruntime 태스크 조회, (3) 잠들었다 깬 태스크의 재삽입 — 즉 **정렬된 동적 집합**이다. 레드블랙 트리는 셋 다 O(log n)을 보장하고, 최좌측 노드를 캐싱하면 "다음 실행 태스크"는 O(1)로 뽑는다. 단순 이진 힙과의 차이는 임의 원소 갱신/삭제가 자연스럽다는 것, 정렬 리스트와의 차이는 삽입이 O(n)이 아니라는 것. "자료구조는 연산 요구사항의 그림자"라는 답변 프레임이 먹힌다.

**"우선순위 역전을 겪는다면 어떻게 진단하고 고치겠나?"**
증상은 "높은 우선순위 스레드가 이유 없이 밀린다". 진단은 프로파일러/ETW 등으로 그 스레드가 무엇을 대기하는지, 그 락의 홀더가 어떤 우선순위로 얼마나 실행됐는지를 본다. 해법은 (1) 우선순위 상속이 있는 동기화 프리미티브 사용, (2) 고순위-저순위가 같은 락을 공유하지 않게 설계 변경(락 분리, 락프리 큐로 전달), (3) 애초에 우선순위 차등을 줄이는 것. Mars Pathfinder 사례를 한 줄 곁들이면 원리와 실사고를 잇는 답이 된다.

**"게임 스레드의 우선순위를 올리는 게 좋은 선택인가?"**
대개 아니다. 동적 범위 안의 소폭 상향(예: 오디오 스레드 ABOVE_NORMAL)은 유효할 수 있지만, 실시간 범위(16+)로 올리면 입력·디스크·네트워크를 처리하는 시스템 스레드를 굶겨 오히려 히치가 생긴다. 근본적으로 우선순위는 "누가 먼저"만 정하지 총 연산량을 줄여주지 않는다 — 프레임이 밀리는 원인이 CPU 총량이라면 우선순위가 아니라 작업 분배(잡시스템)와 알고리즘이 답이다.

**"CPU 친화도 고정은 항상 이득인가?"**
아니다. 이득은 캐시 온기와 예측 가능성, 대가는 스케줄러의 자유 박탈이다. 고정한 코어에 다른 부하가 몰리면 옆 코어가 놀아도 못 옮겨간다. 그래서 순서는: 측정으로 캐시 미스/코어 이동이 문제임을 확인 → ideal processor 힌트 먼저 → 그래도 부족하고 배치를 완전히 통제할 수 있는 전용 서버 환경일 때만 하드 고정. "일반 PC에서 도는 클라이언트"는 사용자 환경이 제각각이라 하드 고정이 특히 위험하다.

## ④ Winters에서의 적용

이 챕터의 개념들이 레포에서 코드로 서 있는 지점들. 면접에서 "직접 만들어봤나"에 대한 실물 답변이다.

### (1) 잡시스템 = M:N 모델의 실물

`Engine/Public/Core/JobSystem.h:20`의 클래스 주석이 설계를 요약한다 — 워커 N개(hw_concurrency - 2, 최소 1), 워커마다 워크스틸링 데크 1개, 그리고 "Phase 5-B에서 내부가 Fiber로 교체되어도 public API 불변"(25행). 실행 단위(잡)는 프레임당 수백 개인데 커널 스레드는 코어 수 근처만 만든다 — M:N의 M/N 분리다. 워커 수 산정은 `Engine/Private/Core/JobSystem.cpp:47`에서 `hardware_concurrency`가 2보다 크면 `hc - 2`: 메인 스레드와 나머지 시스템(렌더 제출, OS, 오디오 등)이 쓸 코어를 남겨두는, oversubscription을 피하는 계산이다. 각 워커가 자기가 몇 번인지는 `JobSystem.cpp:18`의 `thread_local std::int32_t t_iWorkerIdx`로 안다 — 커널 스레드별 저장소(TLS)의 정석 사용례이자, 파이버가 스레드를 옮겨 다니게 되는 순간 재검토해야 할 지점(위 파이버 함정 2번)이기도 하다.

### (2) Chase-Lev 워크스틸링 데크 — 캐시를 아는 유저 모드 스케줄링

`Engine/Public/Core/JobSystem/WorkStealingDeque.h:8`의 헤더 주석이 규약을 명시한다: 주인(owner)은 bottom에서 Push/Pop — LIFO, 캐시 친화적. 도둑(thief)은 top에서 Steal — FIFO, 경쟁 최소. 이 비대칭이 스케줄링 관점에서 정확히 옳다. **주인이 LIFO인 이유**: 방금 push한 잡의 데이터가 아직 그 코어의 캐시에 따뜻하게 남아 있다 — 캐시 온기를 유저 모드 스케줄링 정책으로 활용하는 것. **도둑이 FIFO인 이유**: 반대쪽 끝을 건드리니 주인과의 경쟁이 최소이고, 가장 오래 묵은(대개 트리의 뿌리에 가까운 큰) 작업을 가져가 훔치는 빈도 자체를 줄인다. `WorkStealingDeque.h:91`의 `alignas(64)`로 bottom/top을 캐시 라인 단위로 분리한 것(false sharing 방지)까지, 이 자료구조 전체가 "컨텍스트 스위치의 진짜 비용은 캐시"라는 이 챕터의 명제를 유저 모드에서 실천한다.

### (3) Submit race 사고 — 락프리의 정확성은 호출 규약에 있다

Phase 5-A에서 겪은 실사고. Chase-Lev 데크는 "Push/Pop은 **주인 스레드만** 호출한다"가 알고리즘 정확성의 전제인데, 초기 Submit이 메인 스레드에서 라운드로빈으로 **남의 데크에 Push**해 규약을 위반했고, 잡시스템을 임시 비활성화해야 했다. 현재 코드가 그 수습이다 — `Engine/Private/Core/JobSystem.cpp:146`의 `EnqueueJob`은 호출자가 워커 자신일 때만(`t_iWorkerIdx >= 0`) 자기 데크에 Push하고, 메인/외부 스레드나 오버플로는 뮤텍스로 보호되는 글로벌 큐로 우회시킨다(156~160행). 워커는 자기 데크 → 글로벌 큐 → Steal 순으로 소비한다(`JobSystem.cpp:212` `TryExecuteOneJob`). 교훈을 면접 문장으로: **"락프리 자료구조의 정확성은 코드 안이 아니라 '누가 어떤 스레드에서 호출하는가'라는 규약에 있다. 규약은 컴파일러가 강제해주지 않으므로, 위반은 크래시가 아니라 간헐적 침묵 오염으로 나타난다."**

### (4) 유휴 워커와 대기 전략 — busy-wait / yield / 블로킹의 3단

일이 없는 워커를 어떻게 재울 것인가는 스케줄링 비용 감각의 시금석이다. `Engine/Private/Core/JobSystem.cpp:182`의 주석이 결정을 기록한다 — "yield 스핀은 외부 CPU 부하와 경쟁하며 코어를 태운다. 짧은 타임아웃 대기로 전환해 잡이 없을 땐 실제로 잠든다." 워커는 `condition_variable::wait_for` 1ms로 블로킹하며(185~186행), 깨우기 누락도 타임아웃으로 1ms 안에 회복한다. 반면 `JobSystem.cpp:306`의 `WaitForCounter`는 다른 전략이다 — 카운터가 끝나길 기다리는 스레드가 잠들지 않고 **자기도 잡을 훔쳐 실행**한다(help-stealing). "대기 중인 스레드는 낭비되는 코어"라는 관점에서, 기다림 자체를 일로 바꾼 것. busy-wait(지연 최소, CPU 소모) / yield(양보하지만 러너블 유지) / 블로킹(코어 반납, 깨우기 지연)의 3단 트레이드오프에서 상황별로 다른 단을 고른 실례다.

### (5) FiberShell — 협조적 스케줄링을 잡시스템 밑에 까는 중

파이버 통합은 현재진행형 마스터리 트랙이다. `Engine/Private/Core/JobSystem.cpp:173`에서 워커는 FiberShell 모드일 때 `ConvertThreadToFiber`로 자신을 파이버화하고, `JobSystem.cpp:272`의 `TryExecuteItemOnFiber`가 잡마다 `CreateFiber` → `SwitchToFiber` → 잡 실행 → 복귀 `SwitchToFiber`(`FiberShellEntry`, 294~303행) → `DeleteFiber` 사이클을 돈다. 지금 단계의 한계를 아는 것이 곧 이해의 증거다: (1) 파이버 풀이 없어 잡마다 생성/삭제 비용을 낸다 — 전환 자체는 유저 모드라 싸지만 생성은 스택 할당이라 싸지 않다. (2) `WaitForCounter`에 파이버 양보가 없다 — 파이버의 존재 이유인 "대기 지점에서 스택을 얼리고 다른 잡을 돌린다"가 아직 미완이다. 학습 로드맵과 단계별 목표는 `.md/guide/FIBER_LEARNING_GUIDE.md`에 정리돼 있다. 면접에서는 "SwitchToFiber가 왜 빠른지(커널 미경유, 스케줄러 없음)와, 그럼에도 풀링·양보 설계가 없으면 이점이 반감되는 이유"를 이 코드로 설명할 수 있다.

### (6) Decision/Apply 2-pass — 스케줄링 비결정성 밑에서 쓰기를 통제

병렬 실행의 근본 문제는 "워커들이 언제 어떤 순서로 돌지 스케줄러 마음"이라는 비결정성이다. `Client/Private/GamePlay/System/LocalUnitAISystem.cpp:61`의 미니언 AI는 이를 락이 아니라 **데이터 흐름 설계**로 푼다. 1패스(Decision): 후보가 16마리 이상이면(`kParallelThreshold`, 20행) 개체별 판단을 잡으로 뿌리는데(93~97행), 이 패스는 월드를 **읽기만** 하고 결정(`MinionDecision`)을 슬롯별 버퍼에 쌓는다. 슬롯 인덱스는 `Engine/Private/Core/JobSystem.cpp:35`의 `Get_WorkerSlot` — 메인 스레드는 0, 워커 i는 i+1 — 이고, 버퍼는 워커 수+1개로 준비된다(`LocalUnitAISystem.cpp:53`). 스레드마다 제 버퍼에만 쓰므로 락도 아토믹도 없이 레이스가 소거된다. 2패스(Apply): `WaitForCounter`로 전원 합류 후 **단일 스레드**가 버퍼를 순회하며 월드에 쓴다(`LocalUnitAISystem.cpp:239`). 이 구조는 이 챕터 개념의 응용문제다 — "스레드는 힙을 공유하므로 레이스가 기본값"(②절)이라는 전제를, 공유 쓰기 구간 자체를 스케줄링 비결정성 바깥(순차 구간)으로 밀어내는 방식으로 해소했다. 과거 이 시스템의 프로파일러가 thread_local 슬롯 처리를 잘못해 레이스를 낸 사고가 이 2-pass + 슬롯 구조 정착의 계기였다.

### (7) IOCP — 커널과 협조하는 서버 스레딩, 그리고 Fiber와의 접점

서버의 네트워크 계층은 "연결 1개 = 스레드 1개" 모델을 버리고 IOCP(I/O completion port)를 쓴다. `Server/Private/Network/IOCPCore.cpp:216`에서 워커들은 `GetQueuedCompletionStatus`로 완료 큐에 블로킹 대기한다 — 스레드 상태 전이(running→waiting)를 "소켓별"이 아니라 "완료 이벤트 큐"에 묶어, 수천 연결을 소수 워커로 처리한다. `IOCPCore.cpp:44`의 `CreateIoCompletionPort(..., m_workerCount)` 마지막 인자는 동시성 힌트로, 커널이 이 포트에서 **동시에 러너블로 풀어줄 스레드 수의 상한**이다 — 유저가 스레드를 몇 개 만들었든 커널이 러너블 수를 코어 수준으로 조절해 oversubscription을 막아주는, OS 스케줄러와 앱이 협조하는 드문 공식 창구다. 여기에 파이버를 얹는 것이 진행 중인 통합이다 — `Server/Public/Game/ServerEntry.h:15`의 `Initialize(u32_t uWorkerCount = 0, bool_t bEnableFiberShell = true)`가 그 진입점(구현은 아직 골격 단계). 방향: IOCP 완료를 받은 워커가 "패킷 처리 도중 다른 완료를 기다려야 하는" 로직을 만나면, 스레드를 블로킹하는 대신 파이버를 얼려두고 다음 완료를 처리한다 — N:1의 약점(블로킹 = 전체 정지)을 비동기 I/O로 소거한 M:N의 완성형이다.

## ⑤ 면접 Q&A

**Q1. 프로세스와 스레드의 차이를 설명해 보세요.**
- 핵심: 프로세스 = 자원 소유·보호의 단위(주소공간 + 자원 + PCB), 스레드 = 실행·스케줄링의 단위(스택 + 레지스터 + TCB). 스레드는 힙·전역·파일을 공유하고 스택·레지스터만 독립 — 통신이 공짜인 대신 격리를 잃는 트레이드오프.
- 꼬리 대비: "커널 자료구조로는?"(task_struct/EPROCESS vs TCB), "그래서 전환 비용이 왜 다른가?"(페이지 테이블 교체 여부)까지 이어갈 수 있어야 한다.
- Winters: 잡시스템 워커가 `thread_local`로 자기 인덱스를 식별(JobSystem.cpp:18) — 스레드별 독립 저장소의 실사용.

**Q2. 컨텍스트 스위칭이 왜 비싼가요?**
- 핵심: 직접 비용(레지스터 저장/복원 + 커널 진입 + 스케줄러)은 마이크로초 안팎으로 싸다. 진짜 비용은 간접 비용 — 주소공간 교체 시 TLB 무효화, 그리고 새 스레드 워킹셋이 캐시를 밀어내는 캐시 오염. 청구서는 전환 이후 미스의 형태로 온다.
- 꼬리 대비: "그럼 어떻게 줄이나?" → 스레드 수를 코어 근처로(oversubscription 회피), 태그드 TLB(PCID/ASID), 친화도로 캐시 온기 유지, 아예 커널을 안 거치는 파이버.
- Winters: 워커 수 = hw_concurrency - 2 (JobSystem.cpp:47), Chase-Lev 데크의 owner-LIFO(캐시 온기 활용).

**Q3. 파이버(코루틴)가 스레드보다 빠른 이유는?**
- 핵심: 커널 모드 진입이 없고, 스케줄러가 실행되지 않고(다음 대상을 코드가 지정), 비휘발성 레지스터 + 스택/명령 포인터만 교체하며, 같은 주소공간이라 TLB·캐시가 산다. 대가는 협조 계약 — 블로킹 콜 하나가 워커를 통째로 세우고, 선점이라는 안전망이 없다.
- 꼬리 대비: "그럼 왜 전부 파이버로 안 하나?" → 병렬성은 여전히 커널 스레드가 공급. 파이버는 동시성 도구.
- Winters: FiberShell 사이클(JobSystem.cpp:272~) + 현재 한계(파이버 풀 없음, WaitForCounter 양보 없음)를 아는 것까지가 답.

**Q4. OS 스케줄러 알고리즘의 발전 과정을 아는 대로 설명해 보세요.**
- 핵심: 전부 "실행 시간을 미리 모른다"에 대한 답의 개선사. FCFS(호위 효과) → SJF(평균 대기 최적이지만 미래 정보 필요 + 기아) → RR(시간을 잘게 나눔, 퀀텀 트레이드오프) → MLFQ(과거 행동으로 예측 + 부스트/회계) → CFS(공정한 몫: vruntime 최소 태스크, 레드블랙 트리).
- 꼬리 대비: "멀티코어에선?" → per-CPU 런큐 + 부하 분산, 캐시 온기와 균형의 긴장.
- Winters: 같은 긴장을 유저 모드에서 워커별 데크 + Steal로 복제(WorkStealingDeque.h).

**Q5. 우선순위 역전이 뭔가요?**
- 핵심: 저순위가 락을 쥔 채 중간 순위가 CPU를 점유하면, 락을 기다리는 고순위가 사실상 중간 순위보다 낮게 도는 현상. Mars Pathfinder 리셋 사고가 실례. 해법은 우선순위 상속/실링, 또는 고·저순위가 락을 공유하지 않는 설계.
- 꼬리 대비: "Windows는 어떻게 하나?" → 완전한 상속 대신 부스트 메커니즘(굶은 ready 스레드 부스트 등)으로 완화하는 노선.

**Q6. 게임 서버 스레드 구성을 어떻게 설계하겠습니까?**
- 핵심: CPU 바운드 워커는 코어 수 근처(oversubscription 회피), I/O는 블로킹 스레드 증식 대신 완료 기반 비동기(IOCP)로 소수 워커에 다중화, 시간에 걸친 게임 로직은 파이버/코루틴으로 대기 지점만 얼린다. 우선순위 상향과 하드 친화도는 측정 후 최후 수단.
- Winters: IOCP 동시성 힌트(IOCPCore.cpp:44) + GetQueuedCompletionStatus 대기(:216) + IOCP×Fiber 통합 방향(ServerEntry.h:15)으로 실물 연결.

## ⑥ 흔한 오답 / 함정

1. **"프로세스는 무겁고 스레드는 가볍다"에서 답이 끝남.**
   왜를 말해야 한다 — 프로세스는 주소공간(페이지 테이블)과 커널 장부를 통째로 만들고 교체하지만, 스레드는 스택과 TCB만 추가한다. 전환 시 페이지 테이블 교체 → TLB 무효화 여부가 비용 차이의 핵심이라는 데까지 가야 원리 수준의 답이다.

2. **컨텍스트 스위치 비용 = 레지스터 저장/복원이라고 답함.**
   직접 비용은 마이크로초 안팎으로 싸다. 진짜 비용은 TLB 플러시와 캐시 오염이라는 간접 비용이고, 이건 전환 "이후"에 미스의 형태로 청구된다. 이 구분이 없으면 "왜 스레드를 코어 수 근처로 잡는가"도 설명할 수 없다.

3. **"파이버/코루틴이 스레드보다 항상 좋다"는 과잉 일반화.**
   파이버는 커널의 안전망(선점, 블로킹 투명성)을 포기하고 속도를 산 것이다. 블로킹 시스템콜 하나가 워커를 통째로 세우고, TLS가 배신하고, 양보 없는 루프가 기아를 만든다. 그리고 병렬성 자체는 여전히 커널 스레드에서 나온다 — 파이버는 동시성(concurrency) 도구지 병렬성(parallelism) 도구가 아니다.

4. **"우선순위를 올리면 빨라진다"는 직관.**
   우선순위는 순서만 바꾸지 총량을 줄이지 않는다. 실시간 범위로 올리면 시스템 스레드를 굶겨 오히려 불안정해지고, 고정 우선순위 체계에서는 우선순위 역전이라는 구조적 함정이 생긴다. "빠르게"의 정답은 대개 우선순위가 아니라 작업 분배와 알고리즘이다.

5. **CFS를 "우선순위 큐 스케줄러"로 설명.**
   CFS의 발상은 우선순위 서열이 아니라 **공정한 몫**이다. nice 값은 "먼저 실행될 권리"가 아니라 vruntime이 쌓이는 속도의 가중치이고, 항상 몫을 가장 못 받은(vruntime 최소) 태스크가 선택된다. MLFQ류(Windows 포함)와 모델 자체가 다르다는 걸 구분해야 한다.

6. **상태 전이도에서 waiting → running 직행을 그리는 실수.**
   I/O가 끝난 스레드는 ready로 가서 심사를 다시 받는다. 이 간극을 알아야 "I/O 완료 부스트가 왜 존재하는가"(깨어난 스레드를 빨리 돌려 응답성과 I/O 파이프라인을 살리기 위해)가 설명된다.
