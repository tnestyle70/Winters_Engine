# 컴파일러의 본질과 전체 파이프라인

> "컴파일러가 뭐라고 생각하세요?", "소스 코드가 실행 파일이 되기까지 무슨 일이 일어나나요?", "인터프리터와 뭐가 다른가요?"에 대한 답. 각론(파싱 알고리즘·최적화 패스 상세·레지스터 할당)은 뒤 챕터로 미루고, 여기서는 왜 세상의 모든 컴파일러가 같은 뼈대를 갖게 됐는지를 관통한다.

---

## ① 한 줄 본질

> **"컴파일러는 프로그램의 의미(semantics)를 보존하면서 한 언어를 다른 언어로 번역하는 프로그램이다. '의미 보존'이 계약이고, 그 계약 안에서 코드를 최대한 빠르게 바꾸는 것(최적화)이 존재 이유다. 이 일을 감당 가능하게 만들기 위해 프론트엔드(언어 이해) → 미들엔드(IR 위 최적화) → 백엔드(타깃 코드 생성)의 3단으로 쪼갰고, 가운데의 중간 표현(IR) 덕분에 M개 언어 × N개 타깃이라는 조합 폭발이 M+N개의 모듈로 줄어든다."**

두 문장으로 압축하면:
1. 계약: 번역 전후로 프로그램의 **관찰 가능한 동작(observable behavior)**이 같아야 한다.
2. 존재 이유: 그 계약을 지키는 한, **사람이 쓴 코드보다 빠른 코드**를 내놓을 자유가 있다.

---

## ② 기본 개념

### 컴파일러란 무엇인가 — "의미를 보존하는 번역기"

컴파일러의 정의는 "고수준 언어 → 기계어 변환기"가 아니다. 그건 특수 사례다. 정확한 정의는 **언어 A로 쓰인 프로그램을, 의미가 같은 언어 B의 프로그램으로 바꾸는 프로그램**이다. 이 정의로 보면:

| 번역 | 소스 언어 | 타깃 언어 |
|---|---|---|
| 전통적 컴파일 | C++ | x64 기계어 |
| 트랜스파일 | TypeScript | JavaScript |
| 셰이더 컴파일 | HLSL | DXBC/DXIL (GPU IR) |
| 스키마 컴파일 | FlatBuffers `.fbs` | C++ 헤더 |
| JIT 컴파일 | JVM 바이트코드 | 실행 중인 CPU의 기계어 |

전부 같은 문제다. 그래서 뼈대도 같아진다 — 이 챕터의 결론을 미리 말하면, **"입력 언어를 이해하는 앞부분 + 의미를 다듬는 가운데 + 출력 언어를 만드는 뒷부분"**이라는 구조는 위 다섯 줄 모두에서 반복된다.

"고수준 → 저수준"이라는 방향성도 본질이 아니다. 디컴파일러는 낮은 데서 높은 데로 가고, 트랜스파일러는 같은 높이에서 옆으로 간다. 변하지 않는 것은 단 하나, **의미가 보존되어야 한다**는 계약이다.

### 정확성이 계약, 최적화가 존재 이유 — as-if 규칙

C++ 표준은 이 계약을 **as-if 규칙**으로 성문화한다: 컴파일러는 프로그램의 관찰 가능한 동작(volatile 접근, 파일 I/O, 프로그램의 최종 출력 등)만 같다면 **내부적으로 무슨 짓을 해도 된다**. 실물을 보자.

```cpp
int sum100() {
    int sum = 0;
    for (int i = 1; i <= 100; ++i) sum += i;
    return sum;
}
```

최적화를 켜면 주류 컴파일러는 루프를 아예 돌리지 않는다:

```asm
sum100:
    mov eax, 5050
    ret
```

루프가 사라졌지만 관찰 가능한 동작(반환값 5050)은 같으므로 계약 위반이 아니다. "컴파일러는 내가 쓴 코드를 번역한다"는 직관은 틀렸고, **"내 코드와 동작이 같은, 더 빠른 다른 프로그램을 지어낸다"**가 맞다.

이 계약에서 세 가지 중요한 따름정리가 나온다.

1. **정확성 > 성능.** 빠르지만 틀린 코드를 내놓는 컴파일러는 존재 가치가 없다. 모든 최적화 패스는 "이 변형이 의미를 보존하는가"를 증명 가능한 형태로 설계되고, 증명이 안 되면 변형을 포기한다(보수적 최적화). 최적화가 기대만큼 안 되는 코드는 대개 컴파일러가 멍청해서가 아니라 **안전을 증명할 수 없어서**다 — 두 포인터가 같은 곳을 가리킬 가능성(앨리어싱, aliasing)을 배제 못 하면 로드/스토어를 옮길 수 없는 것이 대표 사례.

2. **미정의 동작(undefined behavior)은 계약의 구멍이 아니라 계약의 일부다.** UB가 있는 프로그램은 "보존해야 할 의미" 자체가 정의되지 않으므로, 컴파일러는 그 경로가 실행되지 않는다고 가정하고 변형한다. "릴리스에서만 나는 버그"의 상당수가 여기서 나온다 — 뒤의 심화에서 다룬다.

3. **결과 비트를 바꾸는 변형은 기본 금지, 명시적 허가제.** 부동소수점 덧셈은 결합법칙이 성립하지 않아서 `(a+b)+c`를 `a+(b+c)`로 바꾸면 결과 비트가 달라질 수 있다. 그래서 이런 재결합(reassociation)은 `/fp:fast` 같은 스위치로 프로그래머가 계약 완화에 서명해야만 허용된다. 네트워크 게임의 결정론 문제가 정확히 이 지점에 걸린다(④에서 실물 연결).

### 왜 3단 구조인가 — M×N 문제와 IR

언어가 M개(C, C++, Rust, Swift...), 타깃 CPU가 N개(x64, ARM64, RISC-V, WASM...)라고 하자. 언어마다 타깃마다 전용 컴파일러를 만들면 M×N개가 필요하다. 각각이 수백만 줄짜리 소프트웨어라면 유지 불가능하다.

해법은 번역가들이 쓰는 것과 같다 — **중간 언어(pivot language)**를 하나 두는 것. 모든 언어를 하나의 중간 표현(IR, Intermediate Representation)으로 내리고, 모든 타깃을 그 IR에서 출발시키면 필요한 모듈이 M+N개로 준다.

```
        M×N 문제                          M+N 해법
  C ──────→ x64               C ──┐               ┌──→ x64
  C ──────→ ARM               C++ ┼──→ [ IR ] ────┼──→ ARM
  C++ ────→ x64               Rust┘      │        └──→ RISC-V
  C++ ────→ ARM                          │
  Rust ───→ x64                    최적화는 여기서
  ...  (M×N개 컴파일러)            한 번만 구현
```

이렇게 역할이 3단으로 갈린다:

| 단계 | 입력 → 출력 | 아는 것 | 모르는 것 |
|---|---|---|---|
| 프론트엔드(frontend) | 소스 → IR | 언어 문법·타입 규칙 | 타깃 CPU |
| 미들엔드(middle-end) | IR → 더 나은 IR | 데이터/제어 흐름 | 언어도 CPU도 (대체로) 모름 |
| 백엔드(backend) | IR → 기계어 | 레지스터·명령어·ABI | 원본 언어 |

이 분리의 진짜 이득은 코드 재사용을 넘어선다. **최적화를 언어 독립·타깃 독립인 한 곳에서 구현**할 수 있다는 것이다. LLVM에 새 최적화 패스를 하나 넣으면 C++, Rust, Swift가 전부 혜택을 본다. Rust가 신생 언어이면서 처음부터 세계 최고 수준의 최적화를 갖고 출발할 수 있었던 것은 LLVM 백엔드를 빌렸기 때문이다 — IR이 언어 생태계의 진입 장벽을 낮춘 실례.

IR 자체의 설계에도 이유가 있다. 소스처럼 높으면 CPU에 못 붙이고, 기계어처럼 낮으면 타입·제어 흐름 같은 의미 정보가 사라져 최적화를 못 한다. 그래서 IR은 "타입이 붙은, 레지스터가 무한한 어셈블리" 정도의 중간 고도에 놓인다. 그리고 현대 IR은 대부분 **SSA(Static Single Assignment) 형식**이다 — 모든 값이 정확히 한 번만 대입되는 형태(재대입은 `x1`, `x2`처럼 새 이름을 만들고, 분기 합류 지점에서는 φ 함수로 "어느 경로에서 왔는지에 따른 값"을 표현). 대입이 유일하면 "이 값의 정의는 정확히 저기 한 곳"이 구조적으로 보장돼, 데이터 흐름 분석이 극적으로 단순해진다. GCC의 GIMPLE도 LLVM IR도 SSA 기반이라는 사실이 이 형식의 승리를 증명한다.

### 전체 파이프라인 조감도

`int square(int x) { return x * x; }` 한 줄이 실행 파일이 되기까지. 단계마다 산출물이 뭔지가 핵심이다.

```
소스(.cpp)
  │  ① 어휘 분석 (lexer)
  ▼
토큰 스트림          int / square / ( / int / x / ) / { / return / x / * / x / ; / }
  │  ② 구문 분석 (parser)
  ▼
AST                 FunctionDecl square: int(int)
                    └─ ReturnStmt
                       └─ BinaryOperator '*'
                          ├─ DeclRefExpr x
                          └─ DeclRefExpr x
  │  ③ 의미 분석 (semantic analysis)
  ▼
장식된 AST           각 x가 매개변수 x의 선언을 가리킴, int*int → int 확정
  │  ④ IR 생성 (lowering)
  ▼
IR                  define i32 @square(i32 %x) { ... }        (아래 상세)
  │  ⑤ 최적화 패스들 (미들엔드)
  ▼
최적화된 IR          인라인·상수 접기·죽은 코드 제거를 거친 IR
  │  ⑥ 코드 생성 — 명령어 선택, 레지스터 할당, 스케줄링
  ▼
어셈블리             imul ecx, ecx / mov eax, ecx / ret        (x64, MSVC 규약)
  │  ⑦ 어셈블 (assembler)
  ▼
오브젝트(.obj)       기계어 + 심볼 테이블 + 재배치 정보. 주소는 미확정
  │  ⑧ 링크 (linker)
  ▼
실행 파일(.exe/.dll) OS 로더가 읽는 포맷(Windows PE)으로 코드·데이터 배치
```

단계 구분은 편의상의 분업이 아니라 **형식 언어 이론의 계층**을 그대로 따른 것이다:

- **어휘 분석**은 정규 언어(regular language)로 충분하다 → 유한 오토마톤 하나로 빠르고 단순하게.
- **구문 분석**은 중첩(괄호, 블록)을 세야 하므로 문맥 자유 문법(context-free grammar)이 필요하다 → 스택 기반 파서.
- **의미 분석**은 "이 이름이 선언됐는가", "타입이 맞는가" 같은 문맥 의존적 질문이라 문법으로는 못 다루고, 심볼 테이블을 든 코드로 검사한다.

각 단계가 **자기 문제를 푸는 데 충분한 가장 약한 도구**를 쓴다. 약한 도구일수록 빠르고, 검증 가능하고, 에러를 정확한 단계에 붙일 수 있다. 컴파일 에러가 "syntax error"(②), "undeclared identifier"(③), "unresolved external symbol"(⑧)로 갈라지는 이유가 이것이다 — **에러 메시지의 종류만 봐도 내 코드가 파이프라인 어디까지 통과했는지 읽을 수 있다.** 이 감각 하나가 빌드 에러 디버깅 속도를 바꾼다.

주의: C++에서는 ① 앞에 전처리기(preprocessor)가 따로 있고(#include 붙여넣기, 매크로 치환), ⑧ 링크가 번역 단위(TU)·ODR·DLL 경계라는 별도 문제 축을 만든다. 그 각론은 [cpp/02_compile_link_dll.md](../cpp/02_compile_link_dll.md)가 전담하므로 여기서는 파이프라인의 칸으로만 둔다.

### 단계별 해부 (1) 프론트엔드 — 토큰, AST, 의미 분석

**어휘 분석(lexing)**: 문자 스트림을 토큰(의미 있는 최소 단위)으로 자른다. `x*x`가 `식별자(x)`, `연산자(*)`, `식별자(x)` 세 조각이 된다. 파서와 분리하는 이유는 (a) 정규 언어라는 더 약한 형식으로 충분해 빠르고, (b) 공백·주석 처리 같은 지저분함을 한 층에서 격리하고, (c) 파서가 "문자"가 아니라 "토큰" 단위로 미리보기(lookahead)를 하게 해 문법 처리가 단순해지기 때문이다. 참고로 "소스가 어떤 인코딩의 바이트열인가"를 정하는 것도 이 단계 관할이라, MSVC의 `/utf-8` 같은 스위치는 파이프라인의 가장 첫 칸을 조종하는 옵션이다.

**구문 분석(parsing)**: 토큰 열을 문법 규칙에 따라 트리(AST, Abstract Syntax Tree)로 세운다. "abstract"인 이유는 괄호·세미콜론처럼 구조를 표시하기 위해서만 존재했던 토큰이 트리 구조 자체에 흡수되어 사라지기 때문이다. `a + b * c`가 `+(a, *(b, c))` 트리가 되는 순간 연산자 우선순위는 트리 모양으로 박제되고, 이후 단계는 우선순위를 다시 고민할 필요가 없다 — **각 단계는 앞 단계가 푼 문제를 다시 풀지 않는다**는 파이프라인 원칙의 예. 이론과 달리 실무 C/C++ 컴파일러(GCC, Clang)는 파서 생성기가 아니라 **손으로 짠 재귀 하강(recursive descent) 파서**를 쓰는데, C++ 문법의 문맥 의존성과 "좋은 에러 메시지·에러 복구"라는 실용 요구가 생성기의 표현력을 넘어서기 때문이다.

**의미 분석(semantic analysis)**: 트리에 의미를 입힌다 — 이름 결정(이 `x`는 어느 선언의 `x`인가, 스코프 규칙), 타입 검사(int*int는 int), 암시적 변환 삽입, C++라면 오버로드 해소와 템플릿 인스턴스화까지. 산출물은 "장식된(annotated) AST"다. 프로그래머가 체감하는 "컴파일러가 똑똑하다/까다롭다"의 대부분이 이 단계다. 정적 타입 언어의 본질적 이점이 여기서 확정된다: **연산의 의미가 컴파일 타임에 결정되므로, 이후 단계는 타입 검사 코드를 산출물에 심을 필요가 없다.** 동적 언어는 이 결정을 런타임으로 미루고, 그 대가를 실행 시마다 지불한다.

### 단계별 해부 (2) 미들엔드 — IR 위의 최적화 패스

프론트엔드가 뱉은 직역 IR은 순진하다. LLVM IR로 보면(-O0 수준의 개형):

```llvm
define i32 @square(i32 %x) {
entry:
  %x.addr = alloca i32              ; 지역 변수 슬롯을 스택에
  store i32 %x, ptr %x.addr         ; 인자를 일단 메모리에 저장
  %0 = load i32, ptr %x.addr        ; 다시 읽고
  %1 = load i32, ptr %x.addr        ; 또 읽어서
  %mul = mul nsw i32 %0, %1
  ret i32 %mul
}
```

최적화 패스(메모리 슬롯을 SSA 값으로 승격하는 mem2reg 등)를 거치면:

```llvm
define i32 @square(i32 %x) {
  %mul = mul nsw i32 %x, %x
  ret i32 %mul
}
```

미들엔드는 이런 **패스(pass)들의 파이프라인**이다. 각 패스는 "IR을 받아 의미가 같은 더 나은 IR을 돌려주는" 독립 변환이고, -O2 같은 레벨은 패스 수십 개의 실행 목록이다. 대표 패스와 논리:

| 패스 | 하는 일 | 성립 근거 |
|---|---|---|
| 인라인(inlining) | 호출을 본문으로 치환 | 호출 비용 제거보다, 치환 후 **다른 최적화가 문맥을 보게 되는 것**이 진짜 이득 |
| 상수 전파/접기 | 컴파일 타임에 아는 값은 미리 계산 | 위의 `mov eax, 5050`이 이 계열의 끝판 |
| 죽은 코드 제거(DCE) | 결과가 관찰되지 않는 코드 삭제 | SSA에서 "사용처 없는 정의"는 기계적으로 찾힌다 |
| 공통 부분식 제거(CSE) | 같은 계산 반복을 한 번으로 | 두 계산이 "같음"의 증명이 SSA에서 쉬워진다 |
| 루프 불변 코드 이동(LICM) | 루프 안에서 변하지 않는 계산을 루프 밖으로 | 반복 횟수만큼의 비용 절약 |

패스들이 서로를 돕는다는 것이 설계의 묘다 — 인라인이 상수를 노출하고, 상수 전파가 분기를 죽이고, 죽은 분기 제거가 또 인라인 기회를 연다. 그래서 패스 실행 **순서**가 결과 품질을 좌우하고, 이 순서 조율이 컴파일러 튜닝의 큰 주제다.

미들엔드가 3단 구조의 존재 이유라는 점을 다시 강조한다: 위 표의 패스 어느 것도 "C++이라서" 또는 "x64라서" 성립하는 것이 없다. **언어와 타깃에서 독립적이기 때문에 한 번 구현해 모든 조합에 재사용**된다.

### 단계별 해부 (3) 백엔드 — 기계에 내려앉기

백엔드에서 IR의 추상이 하나씩 현실과 충돌한다.

- **명령어 선택(instruction selection)**: IR 연산을 실제 명령어 패턴에 대응시킨다. 같은 의미에 여러 후보가 있고 비용이 다르다 — `x*9`를 곱셈 명령 대신 `lea eax, [rcx+rcx*8]`(주소 계산 유닛으로 x+8x) 한 방으로, 상수 나눗셈을 나눗셈 명령 대신 "마법 상수 곱셈+시프트"로 바꾸는 것이 이 단계다. 타깃마다 명령어가 다르므로 여기부터는 타깃 종속.
- **레지스터 할당(register allocation)**: IR의 무한 가상 레지스터를 유한한 물리 레지스터(x64 범용 16개)에 배정한다. 동시에 살아 있는 값이 레지스터 수를 넘으면 스택에 밀어내는데(spill), 이게 성능에 치명적이라 할당 품질이 백엔드 품질을 좌우한다. 최적해는 계산 불가능한 규모라 그래프 컬러링/선형 스캔 계열의 휴리스틱을 쓴다.
- **명령어 스케줄링(scheduling)**: CPU 파이프라인·실행 유닛을 놀리지 않도록 독립 명령어를 재배열한다.

산출물이 어셈블리이고, 어셈블러가 이를 기계어 인코딩으로 바꿔 **오브젝트 파일**에 담는다. 오브젝트 파일의 요점은 "기계어인데 주소가 미확정"이라는 것 — 다른 TU의 함수 주소를 모르므로, "여기는 나중에 `foo`의 주소를 박아 넣어라"라는 **재배치(relocation) 항목**과 심볼 테이블을 함께 담는다. 이 미완성분을 채우는 것이 링커다: 심볼 해소(모든 참조에 정의를 짝지음) + 재배치(주소 확정) + 실행 포맷(PE) 배치. MSVC에서 `unresolved external symbol`은 정확히 이 심볼 해소의 실패다. 링커 각론은 cpp/02로 넘긴다.

한 가지 현대적 반전: MSVC `/GL`(전체 프로그램 최적화)을 켜면 .obj에 기계어 대신 **백엔드 IL**이 담기고, 코드 생성이 링크 시점으로 미뤄진다(LTCG, Link-Time Code Generation). C++의 TU 분리 컴파일 모델에서는 백엔드가 한 번에 한 TU밖에 못 보므로 다른 .cpp의 함수를 인라인할 수 없는데, 링크 시점엔 프로그램 전체가 보이므로 TU 경계를 넘는 인라인·전역 분석이 가능해진다. LLVM의 LTO도 같은 아이디어다. **"정보가 모이는 지점으로 최적화 단계를 옮긴다"** — 파이프라인이 고정 불변이 아니라 정보 가용성에 따라 재배치되는 설계 공간임을 보여주는 사례.

### 실행 모델 스펙트럼 — "언제 번역하는가"의 트레이드오프

컴파일러 vs 인터프리터는 이분법이 아니라 **번역 시점의 스펙트럼**이다. 축은 하나: 번역 비용을 언제, 얼마나 지불하는가.

```
번역을 전부 미리          ◄────────────────────────►      번역을 전혀 안 함
(실행은 최고속)                                            (시작은 최고속)

 AOT 컴파일     ──   JIT 컴파일   ──  바이트코드 VM  ──  트리워크 인터프리터
 C++/Rust           JVM, V8          CPython, Lua        간단한 DSL, 셸
```

| 모델 | 동작 | 강점 | 약점 |
|---|---|---|---|
| AOT(Ahead-Of-Time) | 실행 전 기계어로 완전 번역 | 최고 실행 속도, 예측 가능성 | 빌드 느림, 타깃별 배포, 런타임 정보 활용 불가 |
| 트리워크 인터프리터 | AST를 직접 순회하며 실행 | 구현 최단순, 즉시 실행 | 노드마다 분기+포인터 추적 → 매우 느림 |
| 바이트코드 VM | 소스 → 가상 명령어로 컴파일, VM 루프가 실행 | 이식성, 시작 빠름, 트리워크보다 수 배 빠름 | 네이티브 대비 여전히 격차 |
| JIT(Just-In-Time) | 실행 중 뜨거운 코드만 기계어로 | 런타임 프로파일 기반 최적화 | 워밍업, 메모리, 구현 복잡도 |

트리워크가 왜 느린지는 구현을 상상하면 자명하다 — `eval(node)`가 노드 타입마다 switch를 타고 자식으로 재귀하니, `x*x` 하나에 함수 호출·분기·캐시 미스가 줄줄이 붙는다. 바이트코드 VM은 트리를 **선형 명령 배열**로 미리 눌러 놓아(`LOAD x / LOAD x / MUL / RET` 같은) 순회 비용을 없앤 것이다. 즉 VM도 내부에 컴파일러(소스→바이트코드)를 가진다 — "인터프리터 언어"라는 말이 부정확한 이유.

JIT는 여기서 한 발 더 간다. 대표 구현들의 공통 패턴이 **계층형 컴파일(tiered compilation)**이다:

```
실행 시작 → 인터프리터 (즉시 실행 + 프로파일 수집)
               │ 호출/루프 카운터가 임계 초과 (뜨거움 감지)
               ▼
          베이스라인 JIT (빠르게 컴파일, 얕은 최적화)
               │ 더 뜨겁고 타입 프로파일이 안정됨
               ▼
          최적화 JIT (추측 기반, 공격적 최적화)
               │ 추측이 깨짐
               ▼
          역최적화(deoptimization) → 인터프리터로 복귀
```

- **JVM HotSpot**: 인터프리터 + 빠르지만 얕은 컴파일러(C1)와 느리지만 공격적인 컴파일러(C2)의 계층. "번역 비용은 뜨거운 코드에만 지불한다"는 원칙의 구현.
- **V8(JavaScript)**: 바이트코드 인터프리터(Ignition)로 즉시 실행을 확보하고 뜨거운 함수를 최적화 JIT(TurboFan)로 올리는 다층 구조(중간 계층도 세대를 거치며 추가돼 왔다). 핵심 기법은 **추측 기반 최적화(speculative optimization)** — "이 지점엔 지금까지 항상 정수가 왔으니 정수라 가정하고 컴파일"하고, 가정이 깨지면 역최적화로 물러난다. 동적 타입 언어를 빠르게 만드는 유일한 현실적 방법.

**게임 스크립팅의 Lua는 바이트코드 VM 자리에 있다**(LuaJIT를 붙이면 JIT 자리까지 간다). 게임이 C++(AOT)과 Lua(VM)를 **한 프로세스에 동시에** 얹는 이유가 이 스펙트럼의 정수다: 프레임 예산을 먹는 시뮬레이션·렌더링은 AOT의 최고 속도가 필요하고, 퀘스트·UI·밸런스 로직은 성능보다 **재컴파일 없이 고치고 즉시 재실행하는 반복 속도**가 값지다. 엔진 재빌드는 분 단위, 스크립트 리로드는 밀리초. 실행 모델을 하나 고르는 게 아니라 **코드의 성격별로 다른 실행 모델을 배치하는 것**이 엔진 설계다.

### 트랜스파일러 — 타깃이 기계어가 아닐 뿐

트랜스파일러(transpiler, source-to-source compiler)는 백엔드가 기계어 대신 **다른 고수준 언어**를 뱉는 컴파일러다. TypeScript → JavaScript가 대표이고, 역사적으로 시사적인 예가 최초의 C++ 컴파일러 Cfront다 — C++ → C 트랜스파일러였다. 새 언어를 만들 때 기존 언어를 "이식성 있는 어셈블리"로 쓰면 그 언어가 가진 백엔드 N개를 공짜로 얻는다. M+N 문제의 또 다른 해법인 셈이다. 파이프라인은 동일하다: 렉싱 → 파싱 → 의미 분석까지 다 하고, 코드 생성의 출력 언어만 다르다. "컴파일러냐 아니냐"는 출력물이 아니라 **의미 보존 번역이라는 일**로 판정한다.

### 부트스트래핑과 셀프 호스팅 — 닭과 달걀

"C 컴파일러는 C로 짜여 있다"는 순환은 어떻게 시작됐을까. **부트스트래핑(bootstrapping)**의 표준 절차:

```
세대 0: 기존 언어(어셈블리, C 등)로 짠 언어 X의 미니 컴파일러  →  X0 바이너리
세대 1: X(의 부분집합)로 다시 짠 X 컴파일러 소스 ── X0으로 컴파일 →  X1
세대 2: X 전체를 쓰는 컴파일러 소스              ── X1로 컴파일 →  X2  ★셀프 호스팅
이후 :  Xn 소스                                  ── X(n-1)로 컴파일 → Xn
```

이 상태를 **셀프 호스팅(self-hosting)**이라 하고, 언어의 성인식으로 통한다. 컴파일러는 파서·그래프 알고리즘·대규모 자료구조·파일 I/O를 다 가진 대형 소프트웨어의 대표격이라, 그걸 자기 언어로 감당한다는 것이 언어+라이브러리+도구가 실전 규모에 도달했다는 통합 증명이기 때문이다. Go, Rust 모두 이 관문을 통과했다(Rust의 첫 컴파일러는 OCaml로 쓰였다). 부수 효과로 컴파일러 개발자가 자기 언어의 헤비 유저가 되어 언어의 불편이 즉시 피드백된다(dogfooding).

여기서 Ken Thompson의 **Trusting Trust**(1984년 튜링상 강연 "Reflections on Trusting Trust")가 나온다. 부트스트랩 순환의 어두운 면이다. 컴파일러 바이너리에 두 가지 조작을 심는다: (a) 특정 프로그램(예: login)을 컴파일할 때 백도어를 삽입하고, (b) **컴파일러 자신을 컴파일할 때는 이 조작 로직 전체를 다시 삽입**한다. 그러면 이후 컴파일러의 **소스에서 조작 코드를 완전히 지워도**, 오염된 바이너리로 새 컴파일러를 빌드하는 한 백도어가 바이너리 계보를 따라 영원히 재생산된다. 소스 코드를 아무리 감사해도 못 잡는다 — "자신이 만들지 않은 코드를 신뢰하는 것"의 한계에 대한 근본 논증. 대응책으로 계보가 다른 독립 컴파일러로 같은 소스를 교차 빌드해 산출물을 비교하는 기법(diverse double-compiling)과 재현 가능한 빌드(reproducible build)가 연구되어 왔다. 컴파일러 질문이 공급망 보안으로 확장될 때 꺼낼 수 있는 소재다.

### 실전 컴파일러 지도 — GCC / Clang+LLVM / MSVC

세 주류 컴파일러 모두 3단 구조인데, IR의 개수와 공개 수준이 다르고, 그 차이가 생태계 차이를 만들었다.

| | GCC | Clang + LLVM | MSVC |
|---|---|---|---|
| 프론트엔드 | 언어별 프론트 (C/C++/Fortran...) | Clang (C/C++/ObjC) | cl.exe가 구동 (C는 c1, C++는 c1xx 모듈) |
| IR | GENERIC → **GIMPLE**(SSA, 타깃 독립 최적화) → **RTL**(저수준, 타깃 인접) | **LLVM IR** 단일 — 메모리/비트코드(.bc)/텍스트(.ll) 3형태가 등가 | 내부 IL (비공개, 백엔드가 소비) |
| 백엔드 | RTL에서 코드젠 | 타깃별 LLVM 백엔드 | **c2 모듈** (코드젠+최적화) |
| 설계 성격 | 단일 프로젝트 강결합 | **라이브러리 집합** — IR이 문서화된 공개 API | 툴체인 수직 통합, 내부 비공개 |

- **GCC의 2단 IR**은 고도 분리다: 타깃 독립 최적화는 SSA인 GIMPLE에서, 명령어 선택에 가까운 일은 기계 인접 표현인 RTL에서. 한 IR에 두 고도의 일을 다 시키지 않겠다는 판단.
- **LLVM**은 "컴파일러를 라이브러리로" 재설계하고 IR을 공개 인터페이스로 삼았다. 그래서 Rust·Swift·Julia가 백엔드를 빌려 쓰고, 정적 분석기·새니타이저·JIT 프레임워크가 그 위에 자랐다. 아키텍처 결정(IR의 공개 여부)이 20년 뒤 생태계 지형을 결정한 사례.
- **MSVC**는 IR을 공개하지 않는 대신 Windows 툴체인(link.exe, 디버거, PDB 포맷)과의 수직 통합에 집중한다.

실무에서 알아둘 구조 두 가지:
- `cl.exe`는 컴파일러 본체가 아니라 **드라이버(driver)**다 — 프론트엔드와 백엔드 모듈을 순서대로 부르고, 기본 동작에선 link.exe까지 이어 부른다(gcc/clang 명령도 마찬가지로 드라이버다). 그래서 파이프라인 중간 산출물을 꺼내는 스위치가 다 있다: `/P`(전처리 결과를 파일로), `/FA`(어셈블리 리스팅), `/c`(링크 없이 .obj까지만). **컴파일러를 이해한다는 건 이 중간 산출물들을 직접 꺼내 볼 줄 안다는 것**이고, Compiler Explorer(godbolt.org)가 표준 도구가 된 이유다.
- 위에서 본 `/GL`+LTCG: .obj에 IL을 담아 코드젠을 링크로 미루는 단계 재배치.

### 게임 개발의 숨은 컴파일러들 — 전부 같은 뼈대

게임 프로그래머는 cl.exe 말고도 컴파일러를 매일 여러 개 돌린다. 정의("의미 보존 번역기")와 뼈대(프론트/IR/백엔드)를 알면 전부 같은 그림으로 보인다.

**1) 셰이더 컴파일러: HLSL → DXBC/DXIL → GPU 명령어**

```
HLSL 소스 ──fxc(D3DCompile)──→ DXBC   ──GPU 드라이버──→ 벤더별 GPU 명령어
          ──dxc───────────────→ DXIL  ──GPU 드라이버──→ (NVIDIA/AMD/Intel 각자)
                                 ↑                          ↑
                          여기까지가 "프론트+IR"        이게 진짜 "백엔드"
```

핵심 통찰: **DXBC/DXIL은 GPU의 기계어가 아니라 IR이다.** GPU는 벤더·세대마다 명령어 집합이 다르고 비공개인데, 게임은 어느 GPU에서든 돌아야 한다 — 정확히 M×N 문제다. 그래서 Microsoft가 중간 표현을 표준으로 박았다: 셰이더 모델 5.x까지는 fxc(라이브러리 형태로는 D3DCompile)가 뱉는 DXBC, 셰이더 모델 6부터는 LLVM 기반의 새 컴파일러 dxc가 뱉는 DXIL(LLVM IR에 기반한 포맷 — CPU 컴파일러 인프라가 GPU로 수출된 사례). **최종 코드 생성은 각 벤더의 드라이버가 유저 PC에서, 런타임에** 한다. 게임 중 셰이더 최초 사용 시 걸리는 히치(hitch), 드라이버 업데이트만으로 성능이 변하는 현상, 파이프라인 상태 오브젝트(PSO) 캐싱과 사전 워밍 — 전부 "백엔드가 유저 PC의 드라이버 안에 있다"는 구조에서 나오는 증상이다.

**2) 스키마 컴파일러: FlatBuffers flatc**

`.fbs` 스키마(선언형 미니 언어)를 파싱해 C++/C# 등의 접근자 코드를 생성한다. 소스 언어 = 스키마, 타깃 언어 = C++ — 트랜스파일러의 일종이다. 존재 이유는 zero-copy: 스키마를 컴파일 타임에 타입으로 굳혀 두므로 와이어 바이트를 **파싱 없이** 타입 있는 접근자로 읽는다. "정적으로 알 수 있는 것은 컴파일 타임에 해치운다"는 AOT 철학을 직렬화에 적용한 것.

**3) 셰이더 크로스 컴파일**

HLSL로 짠 셰이더를 Vulkan(SPIR-V), Metal(MSL)에서도 쓰려면 IR을 갈아타는 번역 계층이 필요하다. SPIR-V 역시 표준화된 IR이고, SPIRV-Cross 같은 도구는 SPIR-V → HLSL/MSL/GLSL로 가는 **역방향 트랜스파일러**다. 멀티플랫폼 엔진의 셰이더 파이프라인 운영은 사실상 소형 컴파일러 인프라 운영이다.

이 밖에도 애셋 쿠커(원본 FBX → 런타임 바이너리), 머티리얼 그래프 → 셰이더 코드 생성기, 비주얼 스크립팅 → 바이트코드가 전부 "의미 보존 번역 + 파이프라인" 구조다. **엔진을 만든다는 것은 작은 컴파일러들을 여러 개 만들고 운영한다는 것**이고, 이 관점이 있으면 각 도구의 "무엇을 빌드 타임에, 무엇을 런타임에 두는가"라는 설계 판단을 일관된 원리로 내릴 수 있다.

### 체득 실험 — 파이프라인의 중간 산출물을 직접 꺼내 보기

이 챕터를 지식이 아니라 감각으로 만드는 30분짜리 실험. 위의 `square` 함수 하나로:

```
cl /P  square.cpp        → square.i    전처리 결과 (include가 얼마나 불어나는지 목격)
cl /FA square.cpp        → square.asm  어셈블리 리스팅 (/Od와 /O2로 두 번 뽑아 비교)
cl /c  square.cpp        → square.obj  오브젝트까지만
dumpbin /symbols square.obj            .obj 안의 심볼 테이블 (맹글링된 이름 확인)
dumpbin /relocations square.obj        재배치 항목 ("주소 미확정"의 실물)
clang -S -emit-llvm square.cpp         LLVM IR 텍스트(.ll) — -O0과 -O2 비교
```

특히 해볼 것 세 가지: (1) `sum100`을 /O2로 뽑아 `mov eax, 5050`을 눈으로 확인 — as-if 규칙의 실물. (2) 같은 코드의 /Od와 /O2 어셈블리를 나란히 놓고 "변수가 메모리에서 레지스터로 승격되는 것"을 관찰 — O0/O2 꼬리질문의 근거가 생긴다. (3) 일부러 정의 없는 함수를 호출해 `unresolved external symbol`을 만들고, dumpbin으로 그 심볼이 .obj에 "미해결"로 남아 있음을 확인 — 컴파일 성공/링크 실패의 경계를 몸으로 가른다. Compiler Explorer(godbolt.org)는 이 실험의 웹 버전으로, 소스↔어셈블리 대응을 색으로 보여줘 반복이 빠르다.

## ③ 심화 — 꼬리질문 대비

**"컴파일러와 인터프리터의 본질적 차이는?"**
번역과 실행의 분리 여부다. 컴파일러는 번역만 하고 실행은 산출물이 하며, 인터프리터는 번역(이해)과 실행을 한 몸에서 한다. 단 이분법이 아니라 스펙트럼이다 — CPython은 바이트코드로 "컴파일"한 뒤 VM이 "인터프리트"하고, JIT는 인터프리터 안에 컴파일러가 산다. "언제 번역 비용을 지불하는가"의 축으로 답하는 것이 정확하다.

**"IR이 왜 필요한가? 프론트엔드에서 바로 기계어를 뽑으면 안 되나?"**
가능은 하지만 세 가지를 잃는다. (1) M×N 문제 — 언어×타깃 조합마다 코드젠을 다시 짜야 한다. (2) 최적화 재사용 — 언어 독립·타깃 독립 최적화를 놓을 자리가 없어진다. (3) 분석 용이성 — 소스는 너무 높고 기계어는 너무 낮으며, SSA 기반 IR이라는 중간 고도가 데이터 흐름 분석을 실용적으로 만든다. 초기 소형 컴파일러들은 실제로 IR 없이 직행했다 — IR은 규모가 강제한 설계다.

**"최적화가 프로그램을 바꿔버리면 의미 보존 위반 아닌가?"**
as-if 규칙이 답이다. 보존 대상은 코드의 형태가 아니라 **관찰 가능한 동작**이다. 결과가 같다면 루프를 상수로 접든 함수를 지우든 계약 위반이 아니다. 단 부동소수점 재결합처럼 **결과 비트가 달라지는 변형은 기본 금지**라 `/fp:fast` 같은 명시적 허가 스위치가 따로 존재하고, UB가 있으면 보존할 의미 자체가 정의되지 않아 변형이 예상 밖으로 간다.

**"SSA가 뭐고 왜 쓰나?"**
모든 값이 정확히 한 번만 대입되도록 IR을 변환한 형식이다(재대입은 새 이름, 분기 합류점은 φ 함수). 대입이 유일하면 "이 값의 정의는 정확히 저기 한 곳"이 구조적으로 보장돼, 상수 전파·죽은 코드 제거·공통 부분식 제거가 use-def 사슬을 따라가는 단순한 알고리즘이 된다. GIMPLE과 LLVM IR이 모두 SSA라는 사실이 이 형식의 표준 지위를 보여준다.

**"JIT가 AOT보다 빠를 수도 있다는데 왜인가?"**
AOT가 가질 수 없는 두 정보 때문이다. (1) **런타임 프로파일** — 실제로 어떤 분기가 뜨겁고 어떤 타입이 오는지 보고 추측 기반 인라인·탈가상화(devirtualization)를 한다. (2) **실행 중인 실제 CPU** — 배포 시점의 최소 사양이 아니라 지금 이 기계의 명령어 집합에 맞춰 코드젠한다. 대신 워밍업 시간, 컴파일 스레드의 CPU/메모리, 추측 실패 시 역최적화 비용을 낸다. 그래서 오래 도는 서버엔 JIT가 유리한 국면이 있고, 첫 프레임부터 예산이 있는 게임 클라이언트엔 AOT가 기본이다. AOT도 PGO(Profile-Guided Optimization)로 프로파일 이점 일부를 회수할 수 있다 — "두 정보 중 (1)은 AOT로 가져올 수 있지만 (2)와 '실행 중 재컴파일'은 못 가져온다"까지 말하면 완결된다.

**"동적 타입 언어는 왜 근본적으로 느리고, V8은 어떻게 빠른가?"**
`a + b`의 의미가 런타임 타입에 달려 있어 모든 연산에 타입 검사+분기가 붙기 때문이다. V8은 인라인 캐시로 "이 지점엔 지금까지 이 타입만 왔다"를 기록하고, 그 가정 하에 정적 타입 언어 수준의 기계어를 추측 생성하며, 가정이 깨지면 역최적화로 물러난다. 즉 동적 언어의 속도는 "동적임에도"가 아니라 **"실행 중 정적인 부분을 발견해서"** 얻는 것이다. 정적 타입 언어는 이 정보를 컴파일 타임에 공짜로 갖고 시작한다 — 타입 시스템의 성능적 의미.

**"게임에서 스크립트 언어를 쓰는 이유와 대가는?"**
이유: 반복 속도(재빌드 없이 리로드), 프로그래머가 아닌 직군의 로직 작성, 크래시 격리(스크립트 오류가 프로세스를 죽이지 않게 샌드박싱). 대가: VM 실행 속도, C++↔스크립트 경계 비용(바인딩·마샬링), 스크립트에 GC가 있다면 프레임 스파이크 관리. 그래서 핵심은 경계 설계다 — 프레임마다 도는 뜨거운 루프는 C++에, 이벤트성 로직은 스크립트에 두고, 경계 호출 횟수 자체를 줄이는 API를 짠다.

**"셀프 호스팅이 왜 언어의 이정표인가?"**
컴파일러는 대형 소프트웨어의 대표격이라(파서, 그래프 알고리즘, 대규모 자료구조, 도구 연동) 그걸 자기 언어로 짤 수 있다는 것이 언어+표준 라이브러리+도구가 실전 규모라는 통합 증명이다. 부수 효과로 컴파일러 팀이 언어의 헤비 유저가 되어 설계 결함이 빠르게 피드백된다.

**"Trusting Trust 공격을 설명해 보라. 방어는?"**
컴파일러 바이너리에 "특정 프로그램 컴파일 시 백도어 삽입 + 컴파일러 자신 컴파일 시 이 조작 재삽입"을 심으면, 소스에서 조작을 제거해도 오염된 바이너리로 빌드하는 한 백도어가 세대를 넘어 전파된다. 소스 감사가 무력해진다는 것이 요지. 방어는 계보가 다른 독립 컴파일러로 같은 소스를 교차 빌드해 산출물을 비교하는 diverse double-compiling, 그리고 바이너리를 소스에서 결정론적으로 재생성해 대조하는 재현 가능한 빌드. 공급망 보안 논의의 원형이다.

**"-O0과 -O2의 산출물은 구체적으로 뭐가 다른가?"**
파이프라인 앞 단계(렉싱~IR 생성)는 거의 같고, **미들엔드 패스 목록과 백엔드의 공격성**이 다르다. O0(/Od)은 소스 문장↔기계어 대응을 보존한다 — 변수는 스택 메모리에 상주하고(디버거가 언제든 읽게), 인라인을 억제한다. O2는 인라인·레지스터 승격·루프 변환·DCE로 그 대응을 허문다(변수가 레지스터에만 살거나 아예 증발). 그래서 릴리스 디버깅이 어렵고, "릴리스에서만 나는 버그"의 상당수는 최적화가 만든 게 아니라 **최적화가 드러낸 UB**다 — O0에서 우연히 견디던 미초기화 변수·댕글링이 변형을 못 견딘 것.

**"링크 타임 최적화(LTO/LTCG)는 왜 필요하고 무엇을 바꾸나?"**
C++은 TU 단위로 컴파일하므로 백엔드가 한 번에 한 TU밖에 못 본다 — 다른 .cpp의 작은 함수를 인라인할 수 없다. LTO는 .obj에 기계어 대신 IR을 담아 코드젠을 링크 시점으로 미루고, 그 시점엔 프로그램 전체가 보이므로 TU 경계를 넘는 인라인·전역 분석이 가능해진다. 대가는 링크 시간 폭증과 증분 빌드 이득 감소. "정보가 모이는 지점으로 최적화를 옮긴다"는 원리로 답하면 된다.

**"셰이더는 왜 게임 실행 중에도 컴파일되나?"**
GPU 명령어 집합이 벤더·세대별로 다르고 비공개라, 게임은 IR(DXBC/DXIL/SPIR-V)까지만 만들어 배포하고 최종 코드젠은 유저 PC의 드라이버가 한다 — 백엔드가 런타임에 있는 것이다. 그래서 첫 사용 시 컴파일 히치가 생기고, 엔진들은 PSO 사전 워밍, 파이프라인 캐시 저장, 셰이더 조합(permutation) 수 관리로 대응한다. 드라이버 업데이트만으로 프레임레이트가 변하는 이유도 백엔드가 통째로 교체되기 때문이다.

**"컴파일러가 최적화를 '못 하는' 대표적 이유는?"**
의미 보존을 증명할 수 없을 때다. 대표가 포인터 앨리어싱 — 두 포인터가 같은 메모리를 가리킬 가능성을 배제 못 하면 로드/스토어를 재배치·제거할 수 없다. 그 밖에 내부를 볼 수 없는 외부 함수 호출 너머로는 메모리 상태를 가정할 수 없고, 부동소수점은 재결합이 결과를 바꿔 기본 금지다. 역으로 **"컴파일러에게 증명 재료를 주는 것"**(const, restrict 계열 힌트, 값 의미론, 지역화)이 최적화 친화적 코드 작성법의 본질이다.

**"C++은 왜 파싱이 어렵기로 악명 높나?"**
문법이 문맥 자유에 깔끔히 떨어지지 않기 때문이다. `T * x;`는 T가 타입이면 포인터 선언, 변수면 곱셈식이라 **의미 분석 정보가 있어야 구문이 결정**된다 — 파서와 심볼 테이블이 상호 의존한다. 그 외에 선언과 함수 선언이 모호한 most vexing parse, C++11 이전 `>>`가 시프트로 렉싱되던 문제, 템플릿 의존 이름에 `typename`/`template` 키워드를 요구하는 규칙까지 — "단계를 깔끔히 분리한다"는 교과서 설계가 실제 언어의 역사적 짐 앞에서 어떻게 타협하는지 보여주는 사례군이다.

**"바이트코드와 IR은 뭐가 다른가? 둘 다 중간 표현 아닌가?"**
설계 목적이 다르다. 바이트코드는 **실행**을 위한 표현이라 촘촘하고(메모리 절약), 선형이고, VM 디스패치 루프가 빨리 해석할 수 있게 설계된다. IR은 **분석과 변환**을 위한 표현이라 SSA·명시적 타입·제어 흐름 그래프처럼 최적화 알고리즘이 다루기 좋은 성질을 우선한다. 같은 "중간에 있는 언어"라도 소비자가 인터프리터냐 최적화 패스냐에 따라 형태가 갈리는 것 — JVM이 바이트코드(실행용)를 받아 JIT 내부에서 다시 자체 IR(최적화용)로 바꾸는 것이 두 목적의 분리를 보여주는 증거다.

**"컴파일 속도는 어떻게 개선하나? 파이프라인 관점에서."**
병목이 어느 단계인가부터 갈라야 한다. C++의 전통적 병목은 프론트엔드다 — #include 붙여넣기 때문에 TU 하나가 수십만 줄로 불어 파싱·의미분석을 반복한다. 대응이 프리컴파일드 헤더(PCH: 공통 헤더를 파싱한 프론트엔드 상태를 직렬화해 재사용), 그리고 C++20 모듈(텍스트 붙여넣기 모델 자체를 컴파일된 인터페이스로 대체). 미들엔드/백엔드 병목은 최적화 레벨 조정과 증분 링크, LTO 끄기로 다룬다. "빌드가 느리다"에 단계 진단 없이 답하면 감점이다.

## ④ Winters에서의 적용 — 이 레포에서 돌아가는 컴파일러 세 계열

Winters는 컴파일러를 직접 구현하진 않지만, **세 계열의 컴파일러를 소비자로서 정확히 통제**하고 있다. 면접에서 "컴파일러를 아는가"는 구현 경험보다 이 통제 지점들을 원리로 설명할 수 있는가로 판가름난다.

### (1) MSVC 파이프라인 통제 — vcxproj가 곧 컴파일러 설정의 성문화

vcxproj의 ClCompile 항목들은 파이프라인의 각 단계에 대한 지시로 읽을 수 있다.

- **렉서(가장 첫 칸)**: `Engine/Include/Engine.vcxproj:57`의 `/utf-8` — 소스 바이트열을 어떤 인코딩으로 문자화할지 정한다. 한글 주석·문자열이 있는 코드베이스에서 이게 없으면 시스템 로캘에 따라 소스 해석 자체가 흔들린다. 파이프라인의 0번째 문제.
- **프론트엔드**: 같은 파일 56행 `<LanguageStandard>stdcpp20</LanguageStandard>` — c1xx가 어떤 문법·의미 규칙으로 해석할지. 55행 `<ConformanceMode>true</ConformanceMode>`는 MSVC의 역사적 비표준 확장 대신 표준 문법을 강제한다.
- **프론트엔드 비용 최적화**: 63~65행의 PCH 설정(`WintersPCH.h`를 프리컴파일하고 모든 TU에 강제 포함) — "C++ 빌드의 병목은 #include로 불어난 프론트엔드"라는 진단에 대한 표준 처방. 파싱·의미분석 결과를 캐싱하는 것이므로, 파이프라인 단계를 알아야 왜 헤더 정리가 빌드 시간을 줄이는지 설명할 수 있다.
- **미들엔드/백엔드**: 60행 `<Optimization>Disabled</Optimization>`(Debug, /Od)은 소스↔기계어 대응을 보존해 62행의 PDB(`ProgramDatabase`)와 함께 디버깅 가능성을 산다. 33행 `<WholeProgramOptimization>true</WholeProgramOptimization>`(/GL)은 코드 생성을 링크 시점으로 미루는 LTCG — 본문에서 설명한 "TU 경계 너머 최적화"의 스위치 그 자체다.
- **링커의 최적화**: Release 구성 `Engine/Include/Engine.vcxproj:98-99`의 `EnableCOMDATFolding`(동일 기계어 함수 병합)과 `OptimizeReferences`(미참조 섹션 제거) — 최적화가 컴파일러 한 단계가 아니라 컴파일→링크 전체 파이프라인의 속성임을 보여준다.

가장 교육적인 스위치는 `Shared/GameSim/Include/GameSim.vcxproj:54`의 `<FloatingPointModel>Precise</FloatingPointModel>`이다. `/fp:fast`는 부동소수 재결합 같은 **결과 비트를 바꾸는 변형**을 허가하는 계약 완화인데, Winters의 클라/서버 공유 결정론 시뮬레이션은 두 바이너리의 연산 결과가 비트 단위로 일치해야 하므로 이 완화를 거부한다. "as-if 규칙에서 무엇이 관찰 가능한 동작인가"라는 컴파일러 계약의 문제가 네트워크 게임의 정합성이라는 제품 요구로 직결되는 지점 — **컴파일러 옵션 한 줄이 게임플레이 동기화를 결정한다**고 말할 수 있는 실물이다.

### (2) 셰이더 컴파일러 — 백엔드가 드라이버에 있는 파이프라인의 실물

`Engine/Private/RHI/DX11/DX11Shader.cpp:99`의 `D3DCompileFromFile` 호출이 HLSL의 프론트엔드+미들엔드다. 24행에서 타깃을 `"vs_5_0"`로 지정하는데, 이것이 산출물을 DXBC(셰이더 모델 5 바이트코드)로 정한다 — 그리고 이 DXBC는 GPU 기계어가 아니라 IR이므로, 28행 `CreateVertexShader`로 넘긴 뒤 **드라이버가 런타임에 벤더별 최종 코드젠**을 수행한다. 같은 파일 89~93행은 CPU 컴파일러의 -O0/-O3에 정확히 대응하는 셰이더판 스위치다: Debug는 `D3DCOMPILE_DEBUG | D3DCOMPILE_SKIP_OPTIMIZATION`, Release는 `D3DCOMPILE_OPTIMIZATION_LEVEL3`. 디버깅 가능성 vs 성능이라는 트레이드오프가 CPU와 GPU 컴파일러에서 같은 형태로 반복됨을 한 파일에서 보여줄 수 있다.

`Engine/Private/RHI/RHIShaderCompiler.cpp:33`의 `D3DCompile`은 같은 컴파일러를 **파일이 아닌 메모리 소스 문자열**로 구동하는 RHI 계층 래퍼로, 46~55행에서 실패 시 에러 블롭을 호출자에게 문자열로 돌려준다 — 컴파일러의 진단(diagnostics) 출력을 도구 파이프라인의 데이터로 취급하는 구조다. 여담으로 같은 파일 8행의 `#pragma comment(lib, "d3dcompiler.lib")`는 컴파일러가 .obj 안에 **링커용 지시문**을 심어 두는 MSVC 메커니즘 — 파이프라인의 앞 단계가 뒤 단계에게 메시지를 남기는 통로가 소스 레벨에 노출된 예로, 컴파일→링크가 단절된 단계가 아니라 협업하는 파이프라인임을 보여준다.

런타임 컴파일을 택한 대가(로드 시 컴파일 비용)와 이득(셰이더 수정 후 엔진 재빌드 없이 재실행 — 반복 속도)은 실행 모델 스펙트럼의 트레이드오프를 셰이더에 적용한 판단으로 설명하면 된다. 상용 규모라면 오프라인 사전 컴파일+바이트코드 캐시로 옮겨 가는 것이 정석이고, "지금 구조에서 무엇을 어느 시점으로 옮길지"를 말할 수 있으면 로드맵 질문까지 커버된다.

### (3) flatc — 내 손으로 돌리는 스키마 컴파일러와 신뢰 경계

`Shared/Schemas/Snapshot.fbs:40`의 `table EntitySnapshot`은 소스 언어(FlatBuffers 스키마)로 쓴 선언이고, `Shared/Schemas/Generated/cpp/Snapshot_generated.h:1`의 첫 줄 주석 "automatically generated by the FlatBuffers compiler, do not modify"가 그 번역 산출물임을 못박는다. 소스=스키마, 타깃=C++인 트랜스파일러다. 이 파이프라인이 주는 것은 zero-copy 접근 — 와이어 바이트를 파싱 없이 타입 있는 접근자로 읽는다. "정적으로 알 수 있는 것(스키마)은 컴파일 타임에 굳힌다"는 AOT 철학의 직렬화 버전.

생성 헤더 11~14행의 `static_assert(FLATBUFFERS_VERSION_MAJOR == 25 && ...)`는 **생성기와 런타임 라이브러리의 버전 정합을 컴파일 타임에 강제**한다 — 컴파일러 산출물과 런타임의 계약을 빌드가 검증하는 패턴. 반면 컴파일 타임에 검증할 수 없는 것(네트워크로 들어온 바이트가 정말 스키마를 따르는가)은 런타임 검증으로 남는다: `Client/Private/Network/Client/SnapshotApplier.cpp:561-562`의 `flatbuffers::Verifier` + `VerifySnapshotBuffer`가 그 신뢰 경계이고, 564~571행은 검증 실패를 침묵시키지 않고 상한 있는(8회) 트레이스로 노출한다. "타입 검사를 컴파일 타임과 런타임 어디에 배치하는가"라는 이 챕터의 스펙트럼 질문에 대해, 한 기능 안에서 양쪽 답을 다 보여주는 구조다.

도구 자체도 레포가 소유한다 — `Tools/VerifyPullRepro.ps1:51`이 클론 재현 검증 대상에 `Tools/Bin/flatc.exe`를 포함시킨다. 스키마 컴파일러 바이너리를 저장소에 고정하는 것은 "생성 코드의 재현성 = 컴파일러 버전 고정"이라는, 부트스트래핑 절에서 본 도구 신뢰 문제의 실무적 축소판이다.

### 면접에서 묶어 말하는 법

"컴파일러를 만들어 본 적은 없지만, 제 엔진에는 세 계열의 컴파일러 파이프라인이 돌고 있고 각 통제 지점을 원리로 설명할 수 있습니다 — MSVC는 렉서(/utf-8)부터 프론트엔드(언어 표준, PCH), 백엔드(/Od vs /GL LTCG), 링커 최적화까지 vcxproj로 구성별 통제하고, 특히 /fp:precise는 클라/서버 결정론이라는 제품 요구를 as-if 규칙의 '관찰 가능 동작' 수준에서 지키는 선택입니다. HLSL은 D3DCompile로 DXBC라는 IR까지만 만들고 최종 코드젠은 드라이버가 하는, 백엔드가 런타임에 있는 파이프라인이고, FlatBuffers는 스키마를 C++로 트랜스파일해 파싱 비용을 컴파일 타임으로 옮기되 신뢰 경계에서만 런타임 검증을 남긴 구조입니다."

## ⑤ 흔한 오답 / 함정

1. **"컴파일러 = 고수준 언어를 기계어로 바꾸는 것"으로만 정의.**
   특수 사례를 정의로 착각한 것. 본질은 "의미를 보존하는 언어 간 번역"이고, 트랜스파일러·스키마 컴파일러·셰이더 컴파일러가 전부 포함된다. 좁게 정의하면 "게임 개발에서 만나는 컴파일러들" 같은 확장 질문에서 막힌다.

2. **"컴파일 언어 vs 인터프리터 언어"라는 언어 분류.**
   실행 모델은 언어가 아니라 **구현**의 속성이다. C 인터프리터도 존재하고, JavaScript는 V8 안에서 기계어로 컴파일된다. "Python은 인터프리터 언어"라고 단정하면 CPython(바이트코드 VM)과 PyPy(JIT)의 공존으로 반박당한다. "그 언어의 주류 구현이 어떤 실행 모델인가"로 말해야 정확하다.

3. **"O2는 항상 이득"이라는 단정.**
   최적화는 디버깅 가능성, 컴파일 시간, 코드 크기(인라인 폭주 시 명령어 캐시 압박), 부동소수 모드까지 걸린 트레이드오프다. 특히 결정론이 필요한 시뮬레이션에서 /fp:fast 계열 완화는 정합성 버그가 된다. "이 프로그램은 무엇을 보존해야 하는가"부터 말하는 것이 정답 구조.

4. **"릴리스 빌드에서만 나는 버그 = 컴파일러 버그"라는 반사.**
   대부분은 O0에서 우연히 견디던 UB(미초기화 변수, 댕글링, 앨리어싱 위반)가 최적화 변형을 못 견딘 것이다. 컴파일러는 계약(as-if)을 지켰고, 계약 바깥(UB)에 있던 건 내 코드다. 컴파일러 자체 버그는 실존하지만 가장 마지막에 의심할 항목.

5. **IR을 "그냥 어셈블리 비슷한 것"으로 치부.**
   IR의 요점은 형태가 아니라 **위치와 성질**이다 — 언어 독립·타깃 독립 최적화가 놓이는 자리이고, SSA라는 성질이 분석을 가능케 한다. M+N 논리와 SSA 한 문장을 못 붙이면 IR을 이해했다고 보기 어렵다.

6. **셰이더 바이트코드(DXBC/DXIL)를 GPU 기계어로 오해.**
   DXBC는 IR이고 최종 코드젠은 드라이버가 한다. 이걸 모르면 셰이더 컴파일 히치, PSO 캐싱, 드라이버 업데이트에 따른 성능 변화를 설명할 수 없다 — 그래픽스 면접과 컴파일러 면접이 만나는 지점이라 오답 시 양쪽에서 감점된다.

7. **파이프라인을 "외울 단계 목록"으로 암송.**
   단계 이름 나열은 하수다. 각 경계가 왜 거기 있는지(정규/문맥자유/문맥의존이라는 도구 강도, M+N, 정보 가용성에 따른 단계 재배치)와 각 단계의 산출물을 직접 꺼내 본 경험(/P, /FA, godbolt, 에러 메시지로 단계 판별)을 붙여야 "이해"로 인정받는다.
