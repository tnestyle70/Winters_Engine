# 셰이더의 본질과 라이팅 기초 — Lambert에서 Blinn-Phong까지

> "셰이더가 뭔지 설명해 보세요", "diffuse에 왜 N·L이 들어가는지 유도할 수 있나요?", "Blinn-Phong이 Phong보다 나은 이유는요?"에 답하는 챕터. 파이프라인 구조 자체(스테이지 순서, RHI 계층)는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)가 담당하고, 여기서는 그 파이프라인에 꽂히는 **프로그램(셰이더)의 언어적·수학적 본질**만 판다.

---

## ① 한 줄 본질

> **"셰이더는 GPU 파이프라인의 프로그래머블 스테이지에 꽂히는, 정점당/픽셀당 수천 개가 병렬 실행되는 작은 순수 함수다 — 입력 시맨틱을 받아 출력 시맨틱을 돌려주는 것 외의 부수효과가 (원칙적으로) 없기에 병렬화가 성립한다. 라이팅의 출발점은 '표면이 받는 빛은 입사각의 코사인에 비례한다'는 Lambert 코사인 법칙이고, `diffuse = albedo * saturate(dot(N,L))`은 그 법칙의 코드 번역이다. 스페큘러는 Phong의 반사 벡터 `dot(R,V)`에서 Blinn의 하프 벡터 `dot(N,H)`로 진화했는데, 이는 싸게 만들려던 우연이 물리(마이크로패싯 방향)와 맞아떨어진 사례이며 오늘날 PBR의 `D(N·H)` 항으로 직계 계승됐다."**

---

## ② 기본 개념

### 셰이더의 본질 — 파이프라인에 꽂히는 병렬 순수 함수

고정 기능(fixed function) 시대의 GPU는 "정점 변환과 라이팅을 **어떻게** 할지"가 하드웨어에 구워져 있었고, 프로그래머는 스위치(라이트 개수, 포그 on/off)만 만졌다. 프로그래머블 파이프라인은 그 하드웨어 블록 중 몇 개를 "여기에 **함수를 꽂아라**"로 바꾼 것이다:

```
[IA 정점 조립] → [VS ← 내 함수] → [래스터라이저(고정)] → [PS ← 내 함수] → [OM 블렌딩(고정)]
   고정              프로그래머블         고정                  프로그래머블        고정
```

핵심 성질 세 가지:

1. **순수 함수(pure function)에 가깝다.** VS는 정점 하나를 받아 변환된 정점 하나를 내놓는다. PS는 보간된 값 하나를 받아 색 하나를 내놓는다. 옆 정점/옆 픽셀의 결과를 읽을 수 없고, 전역 상태를 마음대로 쓸 수 없다(UAV라는 명시적 예외 제외). **부수효과가 없으니 실행 순서를 보장할 필요가 없고, 순서 보장이 없으니 수천 개를 아무렇게나 병렬로 돌릴 수 있다.** GPU가 빠른 이유는 클럭이 아니라 이 계약이다.
2. **입력/출력이 시맨틱(semantic)으로 규격화된다.** 함수 시그니처의 `: POSITION`, `: SV_Position` 주석이 "이 값이 파이프라인에서 어떤 역할인지"를 선언한다. 다음 절에서 상세.
3. **묶음(wave/warp) 단위 SIMD로 실행된다.** 하드웨어는 픽셀 32~64개를 한 묶음으로 잡고 같은 명령을 동시에 실행한다(SIMT). `if` 분기에서 묶음 내 픽셀들이 갈라지면 양쪽 경로를 **다 실행하고 마스크로 버리므로**, 발산(divergence)하는 분기가 비싸다는 통념이 여기서 나온다.

"작은"이 중요하다. 셰이더 하나는 수십~수백 줄이지만 프레임마다 수백만 번 호출된다. CPU 코드에서 "함수 하나의 상수 비용"이 관대해도, 셰이더에서는 곱셈 하나가 ×2,000,000이다.

### 시맨틱 — 스테이지 사이의 계약, 그리고 SV_ 접두사의 특별함

시맨틱은 변수 이름이 아니라 **역할 선언**이다. 두 부류가 있다:

| 부류 | 예 | 누가 해석하나 |
|---|---|---|
| 사용자 시맨틱 | `POSITION`, `NORMAL`, `TEXCOORD0`, `TANGENT` | 사람/입력 레이아웃. GPU에게는 그냥 "이름표" — VS 출력의 `TEXCOORD1`이 PS 입력의 `TEXCOORD1`로 배선되게 하는 매칭 키 |
| 시스템 값(system value) | `SV_Position`, `SV_Target`, `SV_VertexID`, `SV_Depth` | **고정 기능 하드웨어**. 파이프라인이 직접 읽거나 쓴다 |

`SV_` 접두사가 특별한 이유: 그 값은 셰이더끼리의 약속이 아니라 **셰이더와 고정 기능 스테이지 사이의 계약**이기 때문이다. 대표가 `SV_Position`이다:

```
VS가 SV_Position에 쓰는 것: 클립 공간(clip space) 4D 좌표 (x, y, z, w)
                     ↓ 래스터라이저(고정 기능)가 받아서:
   1. 원근 나눗셈(perspective divide): (x/w, y/w, z/w)  → NDC
   2. 뷰포트 변환: NDC → 화면 픽셀 좌표
   3. 삼각형 내부 픽셀 열거 + 나머지 속성들의 원근 보정 보간
                     ↓
PS가 SV_Position에서 읽는 것: 이미 변환 끝난 값 — xy = 픽셀 중심 좌표(+0.5), z = 깊이값
```

즉 **같은 시맨틱인데 VS 출력과 PS 입력의 의미가 다르다.** 중간에서 래스터라이저가 계약대로 가공했기 때문이다. Winters의 SSAO 합성이 이 계약을 실전으로 쓴다 — PS에서 받은 `SV_Position.xy`를 화면 크기로 나누면 그대로 스크린 UV가 된다 (`Shaders/Mesh3D.hlsl:61-66`의 `SampleScreenAO`).

사용자 시맨틱 쪽 계약도 있다. VS 입력 시맨틱은 C++의 입력 레이아웃(input layout) 선언과 문자열로 매칭된다 — 정점 버퍼의 44바이트 중 어느 오프셋이 `NORMAL`인지는 CPU 쪽 `D3D11_INPUT_ELEMENT_DESC`가 정하고, 셰이더는 시맨틱 이름으로만 받는다. 이름이 어긋나면 **생성 시점 에러**로 잡히지만, 오프셋/포맷이 어긋나면 조용히 쓰레기 값이 들어온다.

### cbuffer와 레지스터 공간 — b/t/s/u

셰이더의 "전역 입력"은 네 종류의 레지스터 공간에 슬롯 번호로 바인딩된다:

| 레지스터 | 것 | HLSL 선언 | DX11 슬롯 수(스테이지당) |
|---|---|---|---|
| `b#` | 상수 버퍼(constant buffer) | `cbuffer X : register(b0)` | 14 |
| `t#` | 셰이더 리소스 뷰(SRV) — 텍스처/버퍼 읽기 | `Texture2D g_Tex : register(t0)` | 128 |
| `s#` | 샘플러 상태 | `SamplerState g_S : register(s0)` | 16 |
| `u#` | UAV — 읽기/쓰기 | `RWTexture2D<float4> : register(u0)` | 8 (11.0 기준) |

왜 이렇게 나눴나? **갱신 빈도와 하드웨어 경로가 다르기 때문이다.** cbuffer는 작고(최대 64KB) 자주 바뀌며 모든 스레드가 같은 값을 읽는다(uniform) — 전용 상수 캐시를 탄다. 텍스처는 크고 스레드마다 다른 주소를 읽는다 — 텍스처 유닛과 샘플러 하드웨어를 탄다. UAV는 부수효과(쓰기)를 허용하는 유일한 통로라 별도 공간으로 격리된다 — "셰이더는 순수 함수"라는 기본 계약의 명시적 예외 지점.

슬롯 번호는 **갱신 빈도별 관례**로 배정하는 것이 설계 요령이다. Winters의 관례 (`Shaders/FxMesh.hlsl:2-31`):

```hlsl
cbuffer CBPerFrame  : register(b0)  // 프레임당 1회 — viewProj, 카메라, 라이트
cbuffer CBPerObject : register(b1)  // 드로우당 1회 — world 행렬
cbuffer CBFxParams  : register(b2)  // 머티리얼별 — tint, style, dissolve 파라미터
```

프레임당 한 번 쓰는 것과 드로우마다 쓰는 것을 한 버퍼에 섞으면, 드로우마다 프레임 상수까지 통째로 재업로드하게 된다. 버퍼 분리는 성능 규칙이기 이전에 "변경 이유가 다른 데이터는 분리한다"는 일반 설계 원칙의 GPU 판이다.

### cbuffer 패킹 규칙 — 16바이트 경계와 조용한 버그

cbuffer의 메모리 레이아웃은 C++ 구조체 규칙이 아니라 **HLSL 패킹 규칙**을 따른다. 단위는 16바이트 레지스터(float4 하나)다:

1. 변수는 16바이트 레지스터 경계를 **걸치지 못한다(no straddle)**. 걸치게 되면 다음 레지스터로 밀린다.
2. 경계만 안 걸치면 한 레지스터 안에 여러 변수가 **꽉 채워 패킹**된다.
3. 배열 원소는 각각 새 레지스터에서 시작한다 (`float arr[4]`는 16B가 아니라 64B — 원소당 12B 패딩).

```
float3 g_vCameraWorld;   ┐
float  g_fFxTime;        ┘ → 레지스터 1개에 패킹 (12B + 4B = 16B)  ← 관행적 짝짓기

float2 g_vUVScroll;      ┐
float  g_fAlphaClip;     │ → 레지스터 1개 (8 + 4 + 4 = 16B)
float  g_fErodeThreshold;┘

float3 a;                ┐
float3 b;                ┘ → 레지스터 2개! b는 offset 12에 못 앉고(경계 걸침) 16으로 밀림
```

"float3 뒤에 float를 붙이는" 관행은 미관이 아니라 **패딩 슬롯을 실데이터로 채우는 기법**이다. Winters의 `CBPerFrame`이 정확히 이 짝짓기다 (`Shaders/FxMesh.hlsl:4-8`: `g_vCameraWorld`+`g_fFxTime`, `g_vLightDirWorld`+`g_fLightIntensity`).

**조용한 버그의 정체**: CPU에서 `memcpy`로 밀어넣는 C++ 구조체가 이 규칙과 어긋나면, 컴파일 에러도 런타임 에러도 없이 **어긋난 오프셋의 값이 셰이더에 들어간다.** 라이트 강도에 카메라 z좌표가 들어가는 식이라, 증상은 "미묘하게 이상한 화면"이고 디버거는 아무것도 못 잡는다. 방어는 두 겹:

- CPU 미러 구조체에 **명시적 패딩 멤버**를 둔다. Winters `Engine/Public/Renderer/FxShaderConstants.h:9-22`의 `CBPerFrame`은 `XMFLOAT3 cameraWorld; f32_t _padding0;`로 HLSL 레이아웃을 손으로 복제한다.
- 크기 규칙은 컴파일 타임에 강제한다. `static_assert(sizeof(T) % 16 == 0)` — 구조체마다(`FxShaderConstants.h:22,31,49`) 그리고 상수 버퍼 템플릿 자체에(`Engine/Private/RHI/DX11/DX11ConstantBuffer.h:15`) 이중으로 박혀 있다. 크기 검사는 내부 오프셋 어긋남까지는 못 잡으므로 완전한 방어는 아니고, 완전하게 하려면 멤버별 `static_assert(offsetof(...))`까지 가야 한다 — 면접에서 "한계까지 아는" 답이 된다.

### 텍스처와 샘플러는 왜 분리됐나 (DX11)

DX9까지는 "텍스처 스테이지"에 텍스처와 필터링 설정이 한 덩어리였다. DX10/11이 `Texture2D`(데이터)와 `SamplerState`(읽는 방법)를 분리한 이유는 **직교하는 관심사이기 때문**이다:

- 텍스처는 수백~수천 장이지만, "읽는 방법"의 조합은 몇 개 안 된다(point/linear/aniso × wrap/clamp 정도).
- 분리하면 샘플러 몇 개를 만들어 캐싱해 두고 모든 텍스처가 공유한다. 같은 텍스처를 한 곳에서는 linear로, 다른 곳에서는 point로 읽는 것도 상태 복제 없이 된다.
- 하드웨어 실물과도 일치한다 — 샘플러는 텍스처 유닛의 **설정 레지스터**이지 텍스처 데이터의 속성이 아니다.

Winters는 이 구조를 그대로 프리셋 캐시로 굳혔다: point/linear/aniso × wrap/clamp = 6종을 기동 시 전부 만들어 두고 공유한다 (`Engine/Private/RHI/DX11/SamplerStateCache.cpp:29-36`, aniso는 `MaxAnisotropy=16`). 블렌드 상태도 같은 패턴이다(후술).

### 샘플러 상태 — 필터, 주소 모드, 밉맵

**필터(filter)** — "UV가 텍셀 정중앙이 아닐 때 무엇을 반환하나":

| 필터 | 동작 | 텍셀 읽기 수 |
|---|---|---|
| point | 가장 가까운 텍셀 1개 | 1 |
| linear (bilinear) | 주변 2×2 텍셀 가중 평균 | 4 |
| trilinear (`MIN_MAG_MIP_LINEAR`) | 인접 밉 2장에서 bilinear 후 밉 간 보간 | 8 |
| anisotropic | 픽셀이 텍스처 공간에 드리우는 발자국(footprint)의 긴 축을 따라 여러 번 탭 | 최대 MaxAnisotropy×trilinear |

point는 픽셀아트/데이터 텍스처(정확한 값이 필요한 룩업)용, linear는 기본값, aniso는 **비스듬히 보는 바닥/도로**용이다 — 기울어진 면에서 픽셀 하나의 발자국이 텍스처 공간에서 길쭉한 타원이 되는데, trilinear는 이를 원으로 근사해 긴 축 방향이 뭉개진다. aniso는 긴 축을 따라 샘플을 여러 번 찍어 해결한다.

**주소 모드(address mode)** — "UV가 [0,1] 밖일 때":

- `WRAP`: 소수부만 취해 반복 — 타일링 텍스처(바닥, 노이즈 스크롤). Winters FX의 UV 스크롤(`uv += g_vUVScroll`)은 wrap이 전제다.
- `CLAMP`: 경계 텍셀로 고정 — 아틀라스/UI/룩업 테이블. wrap이면 아틀라스 프레임 경계에서 반대편 프레임의 픽셀이 배어 나온다(bleeding).
- 그 외 `MIRROR`, `BORDER`(지정 색)도 있다.

**밉맵(mipmap)과 LOD**: 텍스처를 절반씩 줄인 피라미드(1024→512→…→1)를 미리 만들어 두고, 픽셀 발자국 크기에 맞는 레벨을 골라 읽는다. 존재 이유는 둘이다:

1. **축소(minification) 앨리어싱 제거** — 멀리 있는 체커보드는 픽셀 하나가 텍셀 수백 개를 덮는다. 최상위 밉만 point/linear로 찍으면 프레임마다 다른 텍셀이 걸려 반짝인다(shimmering). 밉은 "수백 텍셀의 평균"을 미리 계산해 둔 것이다.
2. **캐시 효율** — 멀리 있는 물체가 4K 텍스처의 띄엄띄엄한 텍셀을 건드리면 캐시 미스 폭탄이다. 작은 밉은 통째로 캐시에 들어간다.

어느 레벨을 읽을지는 하드웨어가 **UV의 화면 공간 도함수(ddx/ddy)**로 계산한다: `lod ≈ log2(픽셀당 텍셀 이동량)`. 이 도함수를 얻기 위해 PS는 항상 2×2 픽셀 쿼드 단위로 실행된다 — "픽셀 하나만 그려도 4개가 돈다"는 GPU 상식의 출처. 도함수가 정의되지 않는 상황(분기 안, 컴퓨트)에서는 `SampleLevel`로 LOD를 명시한다 — Winters의 스크린 AO 샘플이 그 예다(`Shaders/Mesh3D.hlsl:65`, 화면 공간 UV라 밉 선택이 무의미하므로 `SampleLevel(..., 0)`).

### 셰이더 컴파일 파이프라인 — HLSL → DXBC/DXIL → GPU ISA

셰이더도 컴파일러 파이프라인(프론트엔드 → 중간 표현 → 백엔드)을 그대로 따른다. 특이한 점은 **백엔드가 드라이버 안에 있다**는 것:

```
HLSL 소스
  → fxc (d3dcompiler, ~SM 5.x): DXBC — 가상 ISA용 바이트코드
  → dxc (LLVM/Clang 기반, SM 6.0+): DXIL — LLVM IR 파생 포맷
       ↓ (여기까지가 개발자 영역 — 오프라인 또는 런타임)
드라이버가 CreateVertexShader/PSO 생성 시점에 DXBC/DXIL → 해당 GPU의 네이티브 ISA로 최종 컴파일
```

왜 두 단계인가? GPU ISA는 벤더/세대마다 다르고 문서화도 안 된다. 가상 ISA(DXBC/DXIL)를 중간에 두면 **하나의 바이너리가 모든 GPU에서 돈다** — 컴파일러 챕터(compiler/01)의 "IR을 사이에 두면 프론트엔드 N × 백엔드 M이 N+M이 된다"는 논리가 그대로 적용된 것이다. 대가는 드라이버 컴파일이 런타임(주로 PSO 생성 시점)에 일어나 히칭의 원인이 된다는 것 — 현대 엔진들의 "PSO 캐시/사전 워밍" 문제가 여기서 나온다.

오프라인 vs 런타임 컴파일 트레이드오프:

| | 오프라인 (빌드 시 fxc → .cso) | 런타임 (`D3DCompileFromFile`) |
|---|---|---|
| 로딩 | 빠름 (바이트코드 로드만) | HLSL 파싱+최적화 비용 |
| 이터레이션 | 빌드 필요 | **파일 저장 → 재실행/핫리로드** |
| 배포 | 소스 비공개 | HLSL 원문 노출 |

Winters는 개발 이터레이션을 우선해 런타임 컴파일을 쓴다 (`Engine/Private/RHI/DX11/DX11Shader.cpp:99`). 이때도 컴파일러답게 플래그를 구성별로 가른다(:89-94): Debug는 `D3DCOMPILE_DEBUG | SKIP_OPTIMIZATION`(RenderDoc에서 HLSL 소스 라인 디버깅 가능), Release는 `OPTIMIZATION_LEVEL3`. 출시 빌드로 간다면 오프라인 컴파일 + 바이트코드 캐시가 정답이고, 이는 ④에서 개선 항목으로 다시 언급한다.

### 라이팅의 출발점 — Lambert 코사인 법칙

모든 디퓨즈 라이팅의 뿌리는 기하학 한 조각이다. **평행광이 표면에 비스듬히 들어오면, 같은 양의 빛이 더 넓은 면적에 퍼진다:**

```
수직 입사 (θ=0):            비스듬한 입사 (θ):
  ||||||                        \\\\\\
  ||||||                         \\\\\\
  ▔▔▔▔▔▔                      ▔▔▔▔▔▔▔▔▔▔
  폭 A에 다 떨어짐             같은 빛이 폭 A/cosθ 에 퍼짐
                               → 단위 면적당 빛 = cosθ 배로 감소
```

유도: 단면적 `A`인 빛 다발이 법선과 `θ`를 이루는 표면에 떨어지면, 덮는 면적은 `A / cos(θ)`가 된다. 단위 면적당 광속(irradiance)은:

```
E = Φ / (A / cosθ) = (Φ / A) * cosθ = E_perp * cosθ
```

그리고 `cosθ = dot(N, L)` (N = 단위 법선, L = 표면→광원 단위 벡터). 이것이 **Lambert 코사인 법칙(Lambert's cosine law)**이고, 코드 번역이:

```hlsl
float ndotl = saturate(dot(N, L));          // saturate: 뒤에서 오는 빛(cosθ<0)은 0
float3 diffuse = albedo * ndotl * lightColor * lightIntensity;
```

Winters의 가장 단순한 실물이 `Shaders/Default3D.hlsl:54-64` — `ndotl = max(dot(normal, lightDir), 0)`에 ambient 0.3 / diffuse 0.7을 섞는 교과서 형태다.

**왜 디퓨즈는 뷰 방향과 무관한가?** 디퓨즈는 표면 아래로 파고든 빛이 내부에서 산란을 거듭하다 **방향을 잃고** 아무 방향으로나 빠져나오는 현상의 근사다. 나가는 방향이 균등하니(Lambertian 가정) 어디서 보든 같은 밝기다. 그래서 식에 V가 없다. 반대로 스페큘러는 표면에서 **방향을 기억한 채** 튕기는 빛이라 V가 필수다 — 이 물리적 구분이 "왜 두 항으로 나누나"의 답이다.

**왜 π로 나누나 (radiometric 형태)**: 에너지 보존 형태로 쓰면 `f_diffuse = albedo / π`다. 반구 전체로 나가는 에너지를 적분하면 `∫ cosθ dω = π`이므로, BRDF에 π를 미리 나눠 둬야 "받은 빛 × albedo"만 반사되고 에너지가 불어나지 않는다. 고전 셰이더는 π를 라이트 강도에 접어 넣고 생략하지만(아티스트가 강도를 π배 크게 잡은 셈), PBR은 명시한다 — Winters의 PBR 트랙도 `diffuse = kD * albedo / PI`로 명시한다 (`Shaders/BRDF/BRDF_GGX.hlsli:60`).

### 스페큘러의 진화 — Phong에서 Blinn-Phong으로

**Phong (1975)**: 거울이라면 빛은 정확히 반사 벡터 `R` 방향으로만 튕긴다. 매끈하지만 완전 거울은 아닌 표면은 R 주변으로 좁게 퍼진다고 근사한다:

```
R = 2 * dot(N, L) * N - L          // L의 N 성분은 유지, 접선 성분만 뒤집기
spec = pow(saturate(dot(R, V)), shininess)
```

`R` 유도의 직관: `L`을 법선 성분 `(N·L)N`과 접선 성분 `L - (N·L)N`으로 분해하면, 반사는 접선 성분의 부호만 뒤집는 것이다. `R = (N·L)N - (L - (N·L)N) = 2(N·L)N - L`. (HLSL 내장 `reflect(i, n)`은 `i`가 표면을 **향하는** 방향이라 `reflect(-L, N)`으로 써야 한다 — 부호 실수 단골 지점.)

**Blinn-Phong (1977)**: 반사 벡터 대신 **하프 벡터(half vector)**를 쓴다:

```
H = normalize(L + V)               // 빛 방향과 뷰 방향의 정중앙
spec = pow(saturate(dot(N, H)), shininess)
```

직관: "이 표면이 L의 빛을 V로 반사해 주려면, 표면(의 미세면)이 정확히 어느 방향을 봐야 하나?"의 답이 H다. `dot(N, H)`는 **거시 법선이 그 요구 방향과 얼마나 정렬됐는지**를 잰다.

왜 Blinn이 이겼나 — 두 층위의 답:

1. **역사적/성능적 이유**: 디렉셔널 라이트 + 무한 원점 뷰어(V를 상수로 근사)라는 당시의 흔한 설정에서 H는 **라이트당 한 번만 계산하면 되는 상수**가 된다. Phong의 R은 픽셀(정점)마다 다시 계산해야 한다. 픽셀당으로 비교해도 reflect가 빠지는 만큼 싸다.
2. **물리적 이유 (살아남은 진짜 이유)**: 마이크로패싯(microfacet) 이론에서 "L을 V로 반사하는 미세면"은 정확히 법선이 H인 면이다. 즉 `dot(N,H)` 기반 로브는 "미세면 법선이 H 근처에 몰려 있을 확률"이라는 **물리적 해석이 존재**하고, 실측 BRDF도 half-vector 기반 모델이 더 잘 맞는다(특히 스치는 각도의 길쭉한 하이라이트). Phong의 `dot(R,V)`에는 대응하는 물리 해석이 없다. 이 계보가 그대로 현대 PBR의 법선 분포 함수 `D_GGX(N·H)`로 이어진다 — Winters의 PBR 트랙에서 `H = normalize(V + L)`(`Shaders/BRDF/BRDF_GGX.hlsli:45`)과 `D_GGX(NoH)`(:7-14)로 실물 확인 가능. **Blinn-Phong의 pow 항은 D 항의 조상**이다.

**shininess 지수의 의미**: 로브의 폭이다. `pow(cosα, n)`은 n이 클수록 α=0 근처만 남기고 급락한다 — n=8이면 넓은 무광 하이라이트, n=256이면 좁고 쨍한 하이라이트. 주의점 둘:

- 같은 시각적 로브 폭을 내려면 Blinn의 지수를 Phong보다 크게(경험적으로 약 4배) 잡아야 한다. H와 N의 각도는 R과 V 각도의 대략 절반이라("왜 반각인가"의 답 — L과 V 사이 정중앙에 있는 H는 N에서 그만큼 덜 벗어난다), 같은 편차에서 `dot(N,H)`가 `dot(R,V)`보다 1에 가깝기 때문이다.
- 지수만 키우면 하이라이트가 좁아지면서 **총 에너지도 줄어든다**. 에너지를 보존하려면 정규화 계수(대표적 근사 `(n+8)/(8π)`)를 곱한다 — "정규화 Blinn-Phong"이며, roughness로 D 항을 정규화하는 PBR로 가는 징검다리다.

### 앰비언트 — GI의 가장 싼 근사

`ambient = albedo * 0.1` 같은 상수항의 정체는 **간접광(global illumination)의 0차 근사**다. 현실에서 그늘이 완전한 검정이 아닌 이유는 벽/바닥에서 튕긴 빛이 도달하기 때문인데, 그 다중 반사를 전부 계산하는 대신 "사방에서 균일한 빛이 온다"고 뭉갠 것이 상수 앰비언트다. 근사의 사다리로 이해하면 정확하다:

```
상수항 (ambient = k)                          ← 방향 정보 0
→ 반구 그라디언트 (하늘색↑/땅색↓, N.y로 보간)   ← 방향 정보 1축
→ 앰비언트 큐브/SH(구면 조화)                  ← 저주파 전방향
→ IBL(환경맵 사전적분) → 라이트맵/프로브 → 실시간 GI
```

Winters는 이 사다리의 1~2번째 칸을 실전 배치했다: PBR 트랙은 상수항 `0.03 * albedo`에 AO를 곱하고(`Shaders/Mesh3D_PBR.hlsl:125`), 스타일라이즈드 트랙은 `N.y`로 `ambientLow/ambientHigh` 두 색을 보간하는 반구 앰비언트를 쓴다(`Shaders/Mesh3D.hlsl:111-119`). AO(ambient occlusion)를 앰비언트에만 곱하는 것도 원리가 있다 — AO는 "간접광이 들어오기 어려운 구석"의 근사이지 직사광 그림자가 아니기 때문이다.

### 감쇠 — 역제곱 법칙과 게임식 폴오프

점광원의 물리는 명확하다. 반지름 r인 구의 표면적은 `4πr²`이고 광원의 총 에너지는 일정하므로:

```
intensity(d) = I / d²        (역제곱 법칙, inverse square law)
```

그런데 게임은 이걸 그대로 못 쓴다. 이유 둘:

1. **0이 되지 않는다.** `1/d²`은 무한히 꼬리를 끈다 → "이 라이트가 영향을 주는 범위"를 자를 수 없어 라이트 컬링(어느 물체에 어느 라이트를 계산할지)이 불가능해진다.
2. **d→0에서 폭발한다.** 라이트에 붙은 표면은 무한대 밝기가 된다.

그래서 게임식 감쇠는 **유한 반경 r에서 정확히 0이 되는 스무스 폴오프**를 쓴다. Winters의 선택 (`Shaders/Mesh3D_PBR.hlsl:91-92`):

```hlsl
float attenuation = saturate(1.0f - distanceToLight / max(light.radius, 1e-4f));
attenuation *= attenuation;      // 선형 → 제곱: 역제곱의 볼록한 감쇠 곡선을 흉내
```

`(1 - d/r)²`은 r에서 값 0에 도달하고, 제곱 덕에 근거리에서 가파르고 원거리에서 완만한 — 역제곱과 닮은 — 모양이 된다. 더 물리에 충실하려면 역제곱에 창 함수를 곱하는 방식(`1/(d²+1) * saturate(1-(d/r)⁴)²` 계열)도 있다. 공통 본질은 하나다: **물리 법칙을 아티스트가 통제 가능한 반경 파라미터와 컬링 가능한 유한 지지(finite support)로 변형한 것.** 면접에서 "왜 물리대로 안 하나"가 나오면 이 두 요구사항(반경 통제 + 컬링)으로 답한다.

### 노멀 매핑 — 왜 탄젠트 공간에 저장하나

노멀맵은 "픽셀 단위 법선 디테일"을 텍스처에 굽는 기법이다. 저장 공간을 고를 수 있다: 월드 공간, 오브젝트 공간, 탄젠트 공간(tangent space). 실무 표준이 탄젠트 공간인 이유:

1. **메시 재사용/타일링** — 탄젠트 공간 노멀은 "표면 기준 상대값"이라 같은 벽돌 노멀맵을 어느 벽, 어느 방향의 면에도 붙일 수 있다. 오브젝트 공간 맵은 특정 메시의 특정 UV 전개에 못박힌다.
2. **애니메이션/스키닝** — 캐릭터가 팔을 굽히면 표면의 월드 방향이 바뀐다. 오브젝트/월드 공간 맵은 무효가 되지만, 탄젠트 공간 맵은 변형된 표면의 TBN을 따라가므로 그대로 유효하다.
3. 압축 친화 — z가 거의 +1이므로 두 채널(x,y)만 저장하고 z를 복원하는 BC5 압축이 성립한다.

**노멀맵이 파란 이유**: 탄젠트 공간에서 "변형 없음"은 `n = (0, 0, 1)` — 표면에서 수직으로 솟은 방향이다. 텍스처는 [0,1]만 저장하므로 [-1,1]을 `* 0.5 + 0.5`로 인코딩한다: `(0,0,1) → (0.5, 0.5, 1.0) → RGB(128, 128, 255)`. 평평한 영역이 전부 이 연보라-파랑이라 맵 전체가 파랗게 보인다. 디코딩은 역변환 `n = tex * 2 - 1`.

**TBN 행렬** — 탄젠트 공간 법선을 월드로 가져오는 기저 변환:

```hlsl
// T: 표면 위에서 U가 증가하는 방향(월드), B: V 증가 방향, N: 법선
float3 B = cross(N, T) * handedness;          // 손잡이(UV 미러링) 부호
float3x3 TBN = float3x3(T, B, N);             // 행 = 기저 벡터
float3 nTS = normalTex.xyz * 2.0f - 1.0f;
float3 nWS = normalize(mul(nTS, TBN));         // = nTS.x*T + nTS.y*B + nTS.z*N
```

`mul(행벡터, 행렬)`이 "행들의 선형 결합"이라는 사실을 알면 이 코드가 그냥 **기저 벡터의 가중합**임이 보인다 — 좌표 변환의 본질이다. T는 보통 임포트 시 UV 그라디언트로 계산해 정점에 싣는다. Winters의 정점 레이아웃에는 이미 `TANGENT` 시맨틱이 실려 있지만(`Shaders/Mesh3D.hlsl:40`, `FxSprite.hlsl:39`) 아직 노멀맵 샘플링은 없다 — ④에서 "다음에 추가할 것"으로 다룬다.

### 알파 처리 2종 — 블렌딩 vs clip/discard

반투명을 다루는 길은 둘이고, 파이프라인의 **다른 위치**에서 일어난다:

| | 알파 블렌딩 | clip / discard |
|---|---|---|
| 위치 | OM(출력 병합기) — **고정 기능** | PS 안 — 셰이더 코드 |
| 동작 | `src*SrcBlend OP dst*DestBlend` 산술 혼합 | 조건 만족 시 픽셀 자체를 죽임 (all-or-nothing) |
| 정렬 | **필요** (뒤→앞) — 혼합이 순서 의존적 | 불필요 — 살아남은 픽셀은 불투명 취급 |
| 깊이 쓰기 | 꺼야 함(투명 뒤 물체가 그려져야) | 켠 채 유지 가능 |
| early-z | 유지 | **깨짐** (아래 설명) |
| 결과 품질 | 부드러운 반투명 | 경계가 계단(이진 마스크) |

블렌딩이 정렬을 요구하는 이유: `src*a + dst*(1-a)`는 교환법칙이 성립하지 않는다. 빨간 유리 뒤 파란 유리와 그 반대는 다른 색이다. 그래서 반투명 오브젝트는 카메라 거리로 정렬해 뒤부터 그린다 — 그런데 교차하는 폴리곤은 정렬 자체가 불가능하다는 게 이 방식의 근본 한계다(해법은 ③에서).

clip이 early-z를 깨는 메커니즘: 하드웨어는 원래 PS 실행 **전에** 깊이 테스트를 해서 가려진 픽셀의 PS를 아예 생략한다(early-z — API 명세상 깊이 테스트는 PS 뒤지만, 결과가 같으면 앞당겨도 되는 최적화). 그런데 PS가 `discard`할 수 있으면 "이 픽셀이 깊이 버퍼에 기록될지"를 PS가 끝나야 안다 → 깊이 **쓰기**를 앞당길 수 없고, 관련 최적화(계층적 Z 갱신 등)가 비활성화된다. 그래서 alpha-tested 지오메트리(식물, 철망)는 불투명보다 비싸다.

Winters의 선택: FX는 **블렌딩과 clip을 조합**한다. 블렌드 모드는 4종 프리셋 캐시(Opaque / 스트레이트 AlphaBlend / Premultiplied / Additive — `Engine/Private/RHI/DX11/BlendStateCache.cpp:20-69`)에서 고르고, 셰이더 안에서는 erode 노이즈와 알파 임계값으로 이중 clip을 건다(`Shaders/FxSprite.hlsl:156-163`). clip은 여기서 반투명 표현이 아니라 **디졸브 연출**(임계값이 시간에 따라 올라가며 이펙트가 타들어 가는)의 도구다. 이 이중 clip이 일으킨 실사고(UV-알파 전픽셀 클립)는 ④에서.

### 빌보드 수학 — 카메라를 향하는 쿼드

스프라이트 FX(불꽃, 글로우, 링)는 3D 메시가 아니라 **항상 카메라를 정면으로 보는 평면 쿼드**다. 핵심 발상: "카메라를 향한다" = "쿼드의 가로/세로 축이 카메라의 right/up 축과 일치한다". 카메라 기저(basis)는 이미 있다 — 뷰 행렬이 월드→뷰 회전이므로 그 역(전치)의 축들이 카메라의 월드 공간 right/up/forward이고, 카메라 객체가 이를 직접 들고 있다면 그냥 꺼내면 된다.

월드 행렬(행벡터 규약)의 각 행은 "로컬 축이 월드에서 어디로 가는가"이므로, 행에 카메라 기저를 직접 심으면 회전 행렬이 완성된다:

```
row0 (로컬 X 의 상) = camRight * width
row1 (로컬 Y 의 상) = 쿼드 법선이 가야 할 방향
row2 (로컬 Z 의 상) = camUp * height
row3 (위치)         = worldPos
```

Winters 실물 (`Client/Private/GameObject/FX/FxSystem.cpp:280-397`): 카메라에서 `GetRight/GetUp/GetForward`를 뽑고(:281-283), 쿼드가 XZ 평면에 놓인 지오메트리라서 로컬 X→`Right*drawWidth`, 로컬 Z→`Up*drawHeight`, 로컬 Y(쿼드 법선)→`-Fwd`(카메라를 향해)로 행을 채운다(:388-396). 스케일을 별도 행렬로 곱하지 않고 기저 벡터에 직접 접어 넣은 것도 포인트 — 행렬 곱 하나를 아낀다. 비-빌보드 경로(바닥 데칼)는 평범한 `S*R(yaw)*T` 합성으로 남겨(:402-405) 두 의도를 코드에서 분리했다.

## ③ 심화 — 꼬리질문 대비

**"PS에서 SV_Position을 읽으면 뭐가 들어 있나?"**
VS가 쓴 클립 좌표가 아니다. 래스터라이저가 원근 나눗셈+뷰포트 변환을 끝낸 값 — xy는 픽셀 중심의 화면 좌표(정수+0.5, 왼쪽 위 원점), z는 깊이 버퍼에 기록될 값이다. 그래서 `SV_Position.xy / screenSize`가 곧 스크린 UV가 된다(Winters SSAO 합성이 이 용법).

**"cbuffer에 float 배열을 넣으면 크기가 어떻게 되나?"**
원소마다 16바이트 레지스터 하나씩 차지한다. `float arr[64]`는 256B가 아니라 1KB다. 그래서 스칼라 배열은 `float4 arr[16]`으로 묶어 접근 시 `arr[i/4][i%4]`로 꺼내는 패킹이 관례다. "cbuffer는 float4 레지스터의 배열"이라는 모델을 갖고 있으면 전부 유도된다.

**"cbuffer 레이아웃 버그를 컴파일 타임에 완전히 잡을 수 있나?"**
`sizeof % 16` 검사는 총 크기만 본다(Winters 현재 상태). 내부 오프셋까지 잡으려면 멤버별 `static_assert(offsetof(CB, member) == N)`을 나열하거나, 셰이더 리플렉션(`ID3D11ShaderReflection`)으로 런타임 1회 대조하는 방법이 있다. 완전 자동화는 코드 생성(셰이더와 C++ 헤더를 한 소스에서 생성)까지 가야 한다 — 대형 엔진들이 실제로 하는 방식.

**"DX12에서는 텍스처/샘플러 바인딩이 어떻게 달라지나?"**
슬롯 테이블 대신 디스크립터 힙(descriptor heap)에서 연속 범위를 루트 시그니처로 가리킨다. 샘플러는 static sampler로 루트 시그니처에 아예 구워 넣을 수도 있다. "자주 안 바뀌는 소수의 상태"라는 샘플러의 성격이 DX12에서 더 노골적으로 드러난 것 — DX11의 분리 설계가 그 방향의 중간 단계였다.

**"트라일리니어인데도 비스듬한 바닥이 뭉개지는 이유는?"**
LOD 선택이 발자국의 **긴 축** 기준으로 내려가면 짧은 축이 과블러되고, 짧은 축 기준이면 긴 축이 앨리어싱된다. 등방(isotropic) 필터의 구조적 한계다. anisotropic 필터는 긴 축을 따라 여러 탭을 찍어 두 축을 분리 대응한다 — 비용은 탭 수(MaxAnisotropy)에 비례.

**"diffuse를 왜 π로 나누나?"**
Lambertian BRDF가 상수 `f`라면 반구로 나가는 총 반사 에너지는 `f * ∫cosθ dω = f * π`. 이것이 albedo(반사율)와 같아야 하므로 `f = albedo/π`. 나누지 않으면 albedo 0.5짜리 표면이 받은 빛의 1.57배를 내놓는 에너지 위반이 된다. 고전 셰이더는 π를 라이트 강도에 흡수시켜 생략했을 뿐이다.

**"Blinn-Phong의 지수가 왜 Phong보다 커야 같은 로브가 되나?"**
기하학적으로 H는 L과 V의 각 이등분 방향이라, N-H 각도는 R-V 각도의 대략 절반이다. 같은 편차에서 `dot(N,H)`가 더 1에 가까우므로 로브가 넓어지고, 좁히려면 지수를 (경험적으로 약 4배) 키워야 한다. "왜 반각인가"의 답이 곧 이 질문의 답.

**"스페큘러 색은 왜 albedo가 아니라 라이트 색인가?"**
비금속(dielectric)의 스페큘러는 표면 계면에서의 프레넬 반사라 파장을 거의 가리지 않는다 — 그래서 하이라이트가 라이트 색(대개 흰색)이다. albedo 색이 나오는 것은 표면 아래로 들어가 색소에 흡수를 겪은 디퓨즈뿐. 금속은 예외로 F0 자체가 유색이라 하이라이트에 금색/구리색이 낀다 — PBR의 `F0 = lerp(0.04, albedo, metallic)`(Winters `BRDF_GGX.hlsli:51`)이 이 물리를 한 줄로 압축한 것이다.

**"shininess와 PBR의 roughness는 어떤 관계인가?"**
같은 것(로브 폭)의 다른 매개변수화다. shininess는 지수라 지각적으로 비선형(8→16과 128→256의 체감이 다름)이고 상한이 없다. roughness는 [0,1]로 정규화되고 Disney 관행(`a = roughness²`)으로 지각 선형성까지 보정한다. 개념 매핑은 대략 `shininess ↑ ≈ roughness ↓`. Blinn-Phong의 `pow(NoH, n)`과 `D_GGX(NoH, roughness)`는 같은 자리(NDF)의 구세대/신세대 선수다.

**"역제곱 감쇠를 그대로 쓰면 안 되는 두 가지 이유와 게임식 해법은?"**
(1) 무한 꼬리 — 영향 반경을 못 잘라 라이트 컬링 불가. (2) d→0 발산. 해법은 유한 반경에서 0에 도달하는 창 함수 — `saturate(1-d/r)²`(Winters), 또는 역제곱에 `saturate(1-(d/r)⁴)²`를 곱하고 분모를 `d²+1`로 바닥 처리하는 물리 근접형. 요점은 "물리 곡선의 모양은 유지하되 유한 지지와 특이점 제거를 산다"는 것.

**"노멀맵을 BC5로 압축하면 왜 두 채널만 저장하나?"**
탄젠트 공간 법선은 단위 벡터이고 z(표면 바깥 성분)는 항상 양수라, `z = sqrt(1 - x² - y²)`로 복원 가능하다. BC5는 두 채널을 독립 보간 블록으로 압축해 3채널 압축(BC1)보다 법선 품질이 좋다. 셰이더에서 복원 코드 한 줄이 추가 비용.

**"discard가 early-z를 깨는 이유를 하드웨어 관점에서?"**
early-z는 "PS 결과가 깊이 기록 여부를 바꾸지 않는다"는 전제로 깊이 테스트/쓰기를 PS 앞으로 당기는 최적화다. discard는 그 전제를 부순다 — 픽셀 생존이 PS 실행 후에 결정되므로 깊이 쓰기를 미뤄야 하고, 계층 Z 버퍼 갱신도 보수적으로 꺼진다. 그래서 식생/철망 같은 alpha-test 패스는 불투명 지오메트리와 분리해 나중에 그리는 게 관례다.

**"교차하는 반투명 폴리곤은 정렬이 불가능한데, 해법은?"**
(1) 순서 무관한 블렌드 모드로 도피 — additive는 교환법칙이 성립해 정렬 불필요(FX가 additive를 사랑하는 진짜 이유). (2) alpha-to-coverage — 알파를 MSAA 커버리지 마스크로 변환, 정렬 없이 근사 반투명. (3) OIT(order-independent transparency) — per-pixel linked list 등으로 프래그먼트를 모아 정렬. (4) 콘텐츠 규칙으로 회피 — 교차하는 반투명을 만들지 않는다. 대부분의 게임은 (1)+(4)로 산다.

**"프리멀티플라이드 알파는 왜 쓰나?"**
RGB에 알파를 미리 곱해 저장하고 `(One, InvSrcAlpha)`로 섞는다. 이점 둘: 텍스처 필터링 시 투명 텍셀의 RGB가 배어 나오는 검은 테두리 문제가 사라지고, 알파=0에 RGB>0을 넣으면 같은 블렌드 상태로 additive까지 표현돼 상태 전환이 준다. Winters도 4종 프리셋에 포함해 두었고, FX 셰이더의 마법진 경로는 `finalRGB *= alpha`로 셰이더 안에서 직접 프리멀티플라이한다(`Shaders/FxSprite.hlsl:146`).

**"라이팅 계산은 왜 리니어 공간에서 해야 하나?"**
모니터 출력은 sRGB(감마) 인코딩이라 텍스처의 0.5는 물리 밝기 0.5가 아니다. 감마 공간에서 덧셈/곱셈(라이팅, 블렌딩)을 하면 에너지 계산이 전부 왜곡된다 — 하이라이트가 타 버리고 중간톤이 탁해진다. 올바른 순서는 `sRGB→linear (입력) → 라이팅 → linear→sRGB (출력)`. Winters 스타일라이즈드 트랙이 `pow(c, 2.2)` 근사 쌍으로 이를 명시한다(`Shaders/Mesh3D.hlsl:51-59`) — 정석은 `_SRGB` 텍스처 포맷으로 하드웨어에 맡기는 것이고, 근사 pow는 그 전 단계의 실용 타협이다.

## ④ Winters에서의 적용

### (1) 시맨틱 계약의 실물 — 44바이트 정점과 SV_Position 재활용

FX 셰이더의 입출력 구조체(`Shaders/FxSprite.hlsl:34-47`)가 시맨틱 두 부류를 다 보여준다. `VS_INPUT`은 `POSITION/NORMAL/TEXCOORD0/TANGENT` — CPU 쪽 44바이트 정점 레이아웃과 문자열로 매칭되는 사용자 시맨틱이고(주석 "VTX Mesh 44 bytes와 호환!"이 그 계약의 문서화), `PS_INPUT`의 `SV_POSITION`은 래스터라이저와의 시스템 계약이다. 계약의 "가공된 값" 쪽 활용은 `Shaders/Mesh3D.hlsl:61-66` — PS의 `SV_Position.xy`를 `g_vScreenSize`로 나눠 스크린 공간 AO 텍스처의 UV로 쓴다. "VS가 쓴 값과 PS가 읽는 값이 다르다"를 이 두 파일로 증명할 수 있다.

### (2) cbuffer 패킹 — 셰이더/CPU 양쪽의 방어선

`Shaders/FxMesh.hlsl:2-9`의 `CBPerFrame`은 float3+float 짝짓기의 교본이다(`g_vCameraWorld`+`g_fFxTime`, `g_vLightDirWorld`+`g_fLightIntensity`). `CBFxParams`(:16-31)는 `float2 + float + float = 레지스터 1개` 패킹까지 보여준다. CPU 미러는 `Engine/Public/Renderer/FxShaderConstants.h:9-49` — `_padding0` 명시 멤버로 HLSL 레이아웃을 복제하고 구조체마다 `static_assert(sizeof % 16 == 0)`을 박았으며, 상수 버퍼 템플릿 자체도 같은 검사를 한 번 더 한다(`Engine/Private/RHI/DX11/DX11ConstantBuffer.h:15`). 갱신은 `Map(WRITE_DISCARD) + memcpy`(같은 파일 :54-65) — 이 memcpy가 바로 "레이아웃이 어긋나면 조용히 깨지는" 지점이라 검사가 그 앞을 지킨다. 한계(오프셋 검사 부재)와 다음 단계(offsetof assert/리플렉션 대조)까지 말하면 완결된 답이 된다.

### (3) 상태 객체는 프리셋 캐시로 — 샘플러 6종, 블렌드 4종

"샘플러/블렌드는 조합이 적고 공유된다"는 분리 설계의 논리를 코드 구조로 굳혔다. `Engine/Private/RHI/DX11/SamplerStateCache.cpp:29-36`은 point/linear/aniso × wrap/clamp 6종을 기동 시 일괄 생성(aniso는 `MaxAnisotropy=16`), `Engine/Private/RHI/DX11/BlendStateCache.cpp:20-69`는 Opaque/AlphaBlend/Premultiplied/Additive 4종이다. 드로우마다 상태 객체를 만들지 않고 enum으로 고른다 — DX11 상태 객체가 불변(immutable)이라는 API 설계와 정확히 맞물리는 사용 패턴이다.

### (4) 라이팅의 사다리가 레포 안에 있다 — Lambert → 스타일라이즈드 램프 → Cook-Torrance

세 셰이더가 라이팅 진화 단계를 하나씩 밟는다:

- **교과서 Lambert**: `Shaders/Default3D.hlsl:54-64` — `max(dot(N,L),0)`에 상수 앰비언트 0.3.
- **스타일라이즈드 변형**: `Shaders/Mesh3D.hlsl:68-74`의 `StylizedMapRamp`는 N·L을 그대로 쓰지 않고 wrap 라이팅(`(ndotl + 0.34) / 1.34`)으로 명암 경계를 밀어 올린 뒤 smoothstep 램프를 통과시킨다 — "Lambert는 물리, 램프는 아트 디렉션"이라는 관계를 한 함수에서 설명할 수 있다. 앰비언트는 상수가 아니라 `N.y` 기반 반구 2색 보간(:111-119), 포인트 라이트는 `saturate(1-d/r)²` 감쇠(:86-95). 라이팅 전후로 sRGB↔linear 근사 변환(:51-59)을 끼운 것까지, 이 챕터 개념 대부분이 이 한 파일에 있다.
- **PBR 트랙**: `Shaders/Mesh3D_PBR.hlsl:115-117`이 `BRDF_CookTorrance`로 직사광을 평가하고, `Shaders/BRDF/BRDF_GGX.hlsli:45`에 `H = normalize(V + L)` — **Blinn의 하프 벡터가 D_GGX의 입력으로 살아 있는 현장**이다. `diffuse = kD * albedo / PI`(:60)로 π 정규화도 명시한다.

주의할 정직한 사실: Winters에 **고전 Blinn-Phong 스페큘러 단계 자체는 없다** — unlit/Lambert에서 스타일라이즈드 디퓨즈로, 그리고 바로 Cook-Torrance로 건너뛰었다. 면접에서는 이걸 숨길 게 아니라 "Blinn-Phong을 건너뛴 대신 그 핵심(하프 벡터, NDF의 개념)은 GGX 트랙에서 원리로 답할 수 있다"로 프레이밍하는 편이 강하다.

### (5) clip 파이프라인과 UV-알파 전픽셀 클립 사고

FX 셰이더는 erode(휘도/노이즈 기반 디졸브)와 alpha 임계값의 이중 clip을 건다 — 스프라이트 경로는 `Shaders/FxSprite.hlsl:156-163`(휘도 erode → alpha clip), 메시 경로는 `Shaders/FxMesh.hlsl:171-176`(erode 노이즈맵 `discard`). 실사고: LoL 추출 에셋의 `render/*.png`는 메시 diffuse가 아니라 스프라이트 캡처였고, 메시 UV가 그 텍스처의 알파 0 영역을 가리켜 **전 픽셀이 clip으로 조용히 죽었다**. CPU 브레이크포인트로는 원인 규명이 불가능했고(호출은 전부 성공), RenderDoc으로 픽셀의 clip 지점을 확인하고 UV/알파 바운딩 박스를 데이터로 계측해서 잡았다. "clip은 실패를 조용히 만든다 — GPU 문제는 GPU 도구로"라는 교훈의 실체다. 사건 전말은 [engine/03 ④-3](../engine/03_rendering_pipeline.md)에, 재발 방지 규약은 debug-pipeline 스킬에 박제되어 있다.

### (6) 빌보드 — 카메라 기저를 행에 심는 실물

`Client/Private/GameObject/FX/FxSystem.cpp:280-284`에서 프레임당 한 번 카메라 basis(Right/Up/Forward/Eye)를 뽑고, 빌보드 분기(:385-397)에서 월드 행렬의 행에 직접 심는다: 쿼드 지오메트리가 XZ 평면이므로 row0=`Right*drawWidth`, row2=`Up*drawHeight`, row1(쿼드 법선)=`-Fwd`, row3=위치. 스케일을 기저 벡터에 접어 넣어 행렬 곱을 아끼는 것, 그리고 비-빌보드(바닥 데칼) 경로는 `S*R(yaw)*T` 합성으로 분리(:402-405)한 것이 "수학을 이해하고 쓴" 증거로 제시하기 좋다.

### (7) 없는 것 — 면접용 "다음에 추가할 것" 답변 3종

개념은 알지만 레포에 아직 없는 것들은 개선 로드맵으로 답하면 오히려 강해진다:

1. **노멀 매핑**: 정점 레이아웃에 `TANGENT`가 이미 실려 있고(모든 메시 셰이더의 `VS_INPUT`) 임포트 파이프라인도 탄젠트를 나른다. 그런데 노멀맵을 샘플하는 셰이더가 하나도 없다 — 추가 작업은 (a) VS에서 T를 월드로 변환해 PS로 넘기고, (b) PS에서 `B=cross(N,T)`, TBN 구성, `n=tex*2-1` 디코딩, (c) `.wmesh` 파이프라인에 노멀맵 슬롯(t1) 추가. 스타일라이즈드 룩에서는 노멀맵 디테일이 램프를 지저분하게 만들 수 있어 **의도적으로 보류했다**고 답할 수도 있다 — 그게 사실에 더 가깝고 트레이드오프 감각을 보여준다.
2. **밉맵 생성**: RHI 텍스처 로더는 WIC로 어떤 포맷이든 32bpp RGBA로 강제 변환해 올리지만 `mipLevels = 1`이다(`Engine/Private/RHI/RHITextureLoader.cpp:108-142`). 쿼터뷰 카메라라 축소 비율 변동이 작아 당장은 티가 덜 나지만, 원리대로면 밉 체인 생성(+`GenerateMips` 또는 오프라인 쿠킹)과 sRGB 포맷 지정이 다음 단계다. "왜 아직 문제가 안 됐는가(카메라 각도 고정)"까지 말하면 증상-원인 연결이 된다.
3. **오프라인 셰이더 컴파일**: 현재는 `D3DCompileFromFile` 런타임 컴파일(`Engine/Private/RHI/DX11/DX11Shader.cpp:99`)로 이터레이션을 샀다. 출시 형상은 빌드 단계 fxc/dxc → `.cso` 캐시 + 해시 무효화가 정답이고, 이는 컴파일러 챕터(compiler/01)의 "번역은 미리, 실행은 빠르게" 원칙의 셰이더 판이다.
