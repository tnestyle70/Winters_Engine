# GPU 파이프라인과 하드웨어 — 드로우콜 뒤에서 벌어지는 일

> "DrawIndexed를 호출하면 그 다음에 무슨 일이 벌어지나요?" "GPU는 왜 코어가 수천 개인가요? CPU와 뭐가 다른가요?" "Map에 WRITE_DISCARD는 왜 넣나요?" — 이 세 질문에 원리 수준으로 답하는 챕터다. Winters 파이프라인의 **구조**(RHI 계층, 핸들 모델, FX 트랙)는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)가 담당하고, 여기서는 그 구조가 올라타 있는 **GPU 논리 파이프라인과 하드웨어 실행 모델 자체**를 다룬다.

---

## ① 한 줄 본질

> **"드로우콜 하나는 CPU가 커맨드 큐에 쌓은 '주문서'이고, GPU는 1~3프레임 뒤에서 그 주문을 IA→VS→(테셀레이션/GS)→클리핑·원근 나눗셈→래스터라이저→PS→OM이라는 고정된 순서로 처리한다. 각 스테이지는 '정점당 1회, 프래그먼트당 1회'라는 데이터 병렬성을 갖고, 하드웨어는 이를 32/64개 스레드 묶음(warp/wavefront)의 락스텝 SIMT로 실행하며, 메모리 레이턴시는 캐시가 아니라 '수천 개의 대기 스레드로 갈아타기'로 숨긴다. 렌더링 최적화의 언어(early-z, 분기 발산, 오버드로우, 대역폭, 스톨)는 전부 이 두 층 — 논리 파이프라인의 순서 규약과 SIMT 하드웨어의 물리적 제약 — 에서 파생된다."**

---

## ② 기본 개념

### 전체 지도 — 드로우콜 하나의 여정

```
CPU 타임라인                     GPU 논리 파이프라인
─────────────                    ──────────────────────────────────────────
Map/Update cbuffer               [IA]  버텍스/인덱스 페치, 토폴로지 조립
Bind 셰이더/상태                    │   (정점 스트림 → 프리미티브)
DrawIndexed(...)  ──커맨드 큐──►    ▼
   │ (호출 즉시 리턴,            [VS]  정점당 1회 — 로컬→클립 공간 변환
   │  실행은 나중)                  │
   ▼                            ([HS → 테셀레이터 → DS])  (선택)
다음 오브젝트 준비...            ([GS])                     (선택, 기피 대상)
                                    ▼
                                 [클리핑 → 원근 나눗셈 → 뷰포트 변환]
                                    ▼
                                 [래스터라이저]  삼각형 → 2x2 쿼드 단위 프래그먼트
                                    ▼            (perspective-correct 보간)
                                 [PS]  프래그먼트당 1회 — 색 결정
                                    ▼
                                 [OM]  깊이/스텐실 테스트 → 블렌딩 → 렌더타겟
```

| 스테이지 | 실행 단위 | 성격 | 핵심 책임 |
|---|---|---|---|
| IA (Input Assembler) | 정점/인덱스 | 고정 기능 | 버퍼에서 정점 조립, 프리미티브 구성 |
| VS (Vertex Shader) | 정점당 1회 | 프로그래머블 | 공간 변환, 스키닝, SV_Position 출력 |
| HS/테셀레이터/DS | 패치/생성 정점 | 혼합 | 표면 세분화 (선택) |
| GS (Geometry Shader) | 프리미티브당 1회 | 프로그래머블 | 프리미티브 증폭/제거 (선택, 느림) |
| 클리핑/원근 나눗셈 | 프리미티브 | 고정 기능 | 절두체 클리핑, 클립→NDC |
| 래스터라이저 | 프리미티브→프래그먼트 | 고정 기능 | 커버리지 판정, 속성 보간 |
| PS (Pixel Shader) | 프래그먼트당 1회 | 프로그래머블 | 텍스처링, 라이팅, 색 출력 |
| OM (Output Merger) | 프래그먼트→픽셀 | 고정 기능(설정 가능) | 깊이/스텐실, 블렌딩, 쓰기 |

이 표에서 면접관이 원하는 관통 질문은 하나다 — **"왜 어떤 스테이지는 프로그래머블이고 어떤 스테이지는 고정 기능인가?"** 답: 래스터라이제이션과 블렌딩은 (1) 모든 렌더링이 똑같이 쓰는 연산이라 전용 하드웨어가 압도적으로 효율적이고, (2) **순서 보장**이 필요하다. 특히 블렌딩은 같은 픽셀에 대한 읽기-수정-쓰기(read-modify-write)를 드로우콜 제출 순서대로 보장해야 하는데, 이걸 수천 개 셰이더 스레드가 자유롭게 하게 두면 픽셀 단위 동기화 지옥이 된다. 그래서 ROP(Raster Operations)라는 전용 유닛이 담당한다.

### IA — 버텍스 페치, 인덱스, 토폴로지

IA는 "메모리의 바이트 덩어리를 정점으로 해석하는" 스테이지다. 해석 규칙이 입력 레이아웃(input layout)이고 — 시맨틱 이름, 포맷, 오프셋 — DX11에서는 VS 입력 시그니처와 짝을 맞춰 `CreateInputLayout`으로 미리 검증해 둔다(런타임 매 드로우마다 파싱하지 않기 위해).

**인덱스 버퍼가 존재하는 이유는 정점 중복 제거다.** 닫힌 메시에서 정점 하나는 평균 6개 삼각형에 공유된다. 인덱스 없이 삼각형 리스트를 펼치면 같은 정점을 6번 저장하고 6번 VS를 돌린다. 인덱스를 쓰면 저장은 1회가 되고, GPU는 **변환 후 정점 캐시**(post-transform vertex cache) — 최근 VS 결과를 인덱스 키로 재사용하는 작은 캐시 — 로 VS 실행도 1회에 가깝게 줄인다. 그래서 인덱스 순서를 캐시 친화적으로 재배열하는 메시 최적화(예: 스트립화, Forsyth/meshoptimizer 계열)가 존재한다.

토폴로지(topology)는 인덱스 스트림을 프리미티브로 묶는 규칙이다. 트라이앵글 스트립(strip)은 인덱스가 없던 시절 정점 공유를 강제하는 수단이었지만, 인덱스 버퍼+정점 캐시가 같은 효과를 더 유연하게 주면서 현대 파이프라인은 거의 **트라이앵글 리스트(triangle list)** 하나로 수렴했다. Winters도 모든 메시 드로우가 `D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST`다.

### VS — 정점당 1회, 변환의 무대

VS는 정점 하나를 받아 정점 하나를 내놓는다. **1:1이 강제라는 것 자체가 설계다** — 입출력 개수가 고정이라 GPU가 몇 개의 스레드를 언제 띄울지 미리 알 수 있고, 정점 간 통신이 없어 완전 병렬이 된다. (이 제약을 깨는 게 GS이고, 그래서 GS가 느리다 — 아래에서.)

VS의 계약은 `SV_Position`에 **클립 공간(clip space)** 좌표를 출력하는 것이다. 로컬→월드→뷰→투영까지의 행렬 수학은 변환 챕터의 몫이고, 이 챕터에서 중요한 건 "VS가 끝난 좌표는 아직 나눗셈 전"이라는 사실이다. `(x, y, z, w)` 동차좌표(homogeneous coordinates) 그대로 다음 스테이지로 넘어간다. 왜 VS에서 미리 w로 나눠서 넘기지 않는가? — 클리핑을 나눗셈 전에 해야 하기 때문이다(아래 클리핑 절).

스키닝(skinning)이 VS에 있는 이유도 실행 단위로 설명된다. 본 팔레트 행렬 블렌딩은 "정점당 독립 연산"이라 VS의 병렬 모델에 정확히 들어맞는다. 본 데이터는 정점마다 다르지 않고 드로우마다 공유되므로 상수 버퍼/구조화 버퍼(structured buffer)로 올린다.

### HS/DS 테셀레이션과 GS — 왜 실무에서 잘 안 쓰이나

테셀레이션은 "적은 정점(패치)을 GPU 안에서 세분화"하는 3단이다: HS(Hull Shader)가 패치당 세분화 계수를 정하고, 고정 기능 테셀레이터가 도메인 위치를 생성하고, DS(Domain Shader)가 생성된 각 위치를 실제 정점으로 평가한다. 이론상 매력(LOD를 GPU가 연속적으로) 대비 실무 채택이 낮았던 이유:

- **콘텐츠 파이프라인 비용**: 디스플레이스먼트 맵과 크랙 방지(패치 경계 세분화 계수 일치)까지 아트/툴 체인 전체가 대응해야 한다.
- **작은 삼각형 생성기**라는 근본 문제: 세분화 결과는 픽셀보다 작은 삼각형이 되기 쉽고, 이는 쿼드 오버셰이딩(아래 래스터라이저 절)으로 PS 효율을 무너뜨린다.
- 대안의 승리: 노멀맵/패럴랙스가 "실루엣 빼고 다" 해결했고, 실루엣이 필요한 지형은 CPU/컴퓨트 LOD가 더 통제 가능했다.

GS는 "프리미티브 하나를 받아 0~N개를 출력"할 수 있는 스테이지인데, 바로 그 **가변 출력이 병렬성의 적**이다. 출력 개수를 미리 알 수 없으니 GPU는 최악 크기의 중간 버퍼를 (일부 아키텍처에서는 온칩이 아니라 메모리에) 잡고, 프리미티브 순서 보장을 위해 출력을 직렬화해야 한다. "GS로 빌보드 확장" 같은 고전 용례는 이제 인스턴싱(instancing)이나 컴퓨트 셰이더로 정점 버퍼를 미리 만드는 쪽이 빠른 경우가 대부분이고, 방향성 자체가 메시 셰이더(mesh shader, DX12 Ultimate)로 대체되는 흐름이다. 면접 답변으로는 "GS를 안 쓰는 이유를 설명할 수 있는 것"이 GS를 써본 것보다 가치 있다.

### 클리핑과 원근 나눗셈 — 왜 나눗셈 '전에' 자르나

클립 공간에서 절두체 내부 조건은 나눗셈 없이 이렇게 쓸 수 있다 (DX 규약):

```
-w <= x <= w,   -w <= y <= w,   0 <= z <= w
```

**왜 여기서 자르는가? 원근 나눗셈(perspective divide)이 안전하지 않기 때문이다.** 카메라 뒤의 점은 `w <= 0`이다. 그대로 `x/w`를 하면 (1) `w = 0`에서 0 나눗셈, (2) `w < 0`에서 부호가 뒤집혀 화면 반대편으로 "래핑"된 좌표가 나온다 — 카메라 뒤를 지나는 삼각형이 화면을 가로지르는 거대한 쓰레기 삼각형으로 변하는 고전 버그. 클립 공간은 동차좌표라 클리핑이 **선형 보간만으로** 된다: 평면과 교차하는 변 위의 새 정점을 `lerp(v0, v1, t)`로 만들면 속성까지 올바르게 잘린다. 나눗셈 후에는 이 선형성이 깨진다.

그래서 순서가 고정이다: **클리핑(동차 공간) → 원근 나눗셈 `(x/w, y/w, z/w)` → NDC → 뷰포트 변환(픽셀 좌표)**. 참고로 현대 GPU는 좌우/상하 평면은 실제 기하 클리핑 대신 가드 밴드(guard-band) + 시저(scissor)로 때우는 경우가 많고, 진짜 기하 클리핑이 필수인 것은 near 평면(`z=0`, 즉 `w`가 0에 접근하는 쪽)이다 — 0 나눗셈만은 하드웨어도 피할 수 없기 때문이다.

`w`에는 물리적 의미가 있다: 투영 행렬을 통과하면 `w = 뷰 공간 깊이(z_view)`가 된다(표준 원근 투영 기준). "멀수록 크게 나누니 작아진다"는 원근법 그 자체가 `w` 나눗셈이다. 이 사실이 다음 절의 보간 수식을 만든다.

#### 원근 나눗셈의 후폭풍 — 깊이 정밀도와 reversed-Z

깊이 버퍼에 저장되는 값은 `z_ndc = z_clip / w = A + B / z_view` 꼴이다(A, B는 near/far로 정해지는 상수). 즉 **뷰 깊이의 역수에 선형**이다. 결과:

- NDC 깊이 구간의 대부분이 near 평면 근처에 몰린다. near=0.1, far=1000 같은 설정이면 `z_ndc`의 절반 이상이 카메라 앞 몇 미터 안에서 소진되고, 원거리는 촘촘히 뭉개진다.
- 원거리에서 서로 가까운 두 표면의 `z_ndc` 차이가 부동소수/고정소수 정밀도 아래로 떨어지면 프레임마다 승자가 바뀐다 — **z-fighting**의 수학적 원인.

왜 이런 값을 쓰나? — `z/w`는 화면 공간에서 선형이라 래스터라이저가 싸게 보간할 수 있고, 원근 보정 나눗셈 없이 프래그먼트마다 비교 가능하기 때문이다. 정밀도는 그 대가다.

완화책의 원리도 여기서 나온다: (1) near를 최대한 밀어 올리기 — B항 분포가 가장 민감한 변수가 near다. (2) **reversed-Z** — float의 정밀도는 0 근처에 몰려 있으므로, near=1/far=0으로 매핑을 뒤집으면 "1/z의 밀집(near 쪽)"과 "float의 밀집(0 쪽 = far 쪽)"이 서로 반대편을 보완해 전 구간이 고르게 좋아진다(비교 함수는 GREATER로 뒤집고, float 깊이 버퍼에서 효과가 큼). (3) 폴리곤 오프셋/깊이 바이어스는 공면(coplanar) 지오메트리용 국소 처방이다.

### 래스터라이저 — 삼각형이 프래그먼트가 되기까지

래스터라이저는 세 가지를 한다: (1) 어떤 픽셀이 삼각형에 덮이는지 판정(coverage), (2) 그 픽셀 위치에서 정점 속성을 보간, (3) 2x2 쿼드로 묶어 PS에 발급.

**커버리지 판정**은 픽셀 중심이 삼각형 내부인가를 에지 함수(edge function, 2D 외적 부호)로 검사한다. 두 삼각형이 공유하는 변 위의 픽셀이 두 번 그려지거나(블렌딩 시 이중 적용) 한 번도 안 그려지는(틈) 것을 막기 위해 **top-left rule** 같은 타이브레이크 규약이 API 스펙에 박혀 있다 — "공유 변은 정확히 한 번"이 결정론적으로 보장된다.

#### perspective-correct interpolation — 왜 필요하고 어떻게 되나

화면 공간에서 속성을 그냥 선형 보간하면 틀린다. 직관: 철길을 비스듬히 보면 침목 간격이 **세계에서는 균일하지만 화면에서는 멀수록 압축**된다. 화면에서 절반 지점은 세계에서 절반 지점이 아니다. PS1 세대 텍스처가 벽에서 "휘어 보이던" 것이 바로 화면 선형(affine) 보간의 아티팩트다.

수학적 근거는 한 문장이다: **원근 투영 아래에서, 속성 `a` 자체는 화면 좌표의 선형 함수가 아니지만 `a/w`와 `1/w`는 화면 좌표의 선형 함수다.** (투영은 사영 변환이고, 삼각형 평면 위에서 선형인 양을 `w`로 나눈 것들만 화면에서 다시 선형이 된다.) 그래서 래스터라이저는:

```
보간할 것: a0/w0, a1/w1, a2/w2  그리고  1/w0, 1/w1, 1/w2   (화면 무게중심좌표 b0,b1,b2로 선형 보간)

a_pixel = (b0*a0/w0 + b1*a1/w1 + b2*a2/w2) / (b0*1/w0 + b1*1/w1 + b2*1/w2)
```

분자는 "a/w의 화면 선형 보간", 분모는 "1/w의 화면 선형 보간"이고, 나누면 `a`가 복원된다. **픽셀마다 나눗셈 한 번**이 원근 보정의 비용이며, 이건 래스터라이저 하드웨어가 해 준다. 깊이 버퍼에 들어가는 `z_ndc = z/w`가 화면 선형이라 하드웨어 깊이 보간이 싸다는 것, 대신 깊이 정밀도가 near 쪽에 몰린다는 것도 같은 수학의 다른 얼굴이다.

HLSL에서 `noperspective` 한정자로 이 보정을 끌 수 있다는 사실은 "보정이 기본값으로 켜져 있는 별도 연산"임을 아는 증거로 쓸 수 있다.

#### 2x2 픽셀 쿼드와 ddx/ddy — 밉맵 LOD의 원천

PS는 픽셀 하나씩이 아니라 **2x2 쿼드(quad) 단위로 발급된다.** 이유는 미분이다. `ddx(v)`는 "쿼드 안 가로 이웃 픽셀의 `v`와 내 `v`의 차이", `ddy(v)`는 세로 이웃과의 차이로 구현된다 — 락스텝으로 같은 인스트럭션을 돌고 있으니 이웃 레인의 레지스터 값을 빼기만 하면 미분 근사가 공짜로 나온다.

```
쿼드:  [P00][P10]     ddx ≈ P10.v - P00.v
       [P01][P11]     ddy ≈ P01.v - P00.v      (coarse 기준)
```

이 미분이 필요한 첫 번째 고객이 **밉맵 LOD 선택**이다. `Sample`이 내부적으로 하는 일은 대략:

```hlsl
float2 dx = ddx(uv) * texSize;   // 화면 1픽셀 이동당 텍셀 이동량
float2 dy = ddy(uv) * texSize;
float rho = max(length(dx), length(dy));
float lod = log2(max(rho, 1.0));  // 텍셀:픽셀 = 1:1이 되는 밉 레벨
```

원리: 화면 픽셀 하나가 텍셀 `rho`개를 건너뛰며 샘플링하면, `rho`가 2를 넘는 순간 나이퀴스트(Nyquist) 기준을 어겨 앨리어싱(반짝임)이 난다. 밉 레벨을 하나 올릴 때마다 텍셀 밀도가 절반이 되므로 `log2(rho)` 레벨을 고르면 "미리 평균 내둔(pre-filtered) 텍스처"를 1:1 비율로 읽게 된다. 밉맵은 메모리 33% 추가로 (1) 앨리어싱 제거와 (2) **텍스처 캐시 지역성**(멀리 있는 표면을 밉 0으로 읽으면 샘플마다 캐시 미스)을 동시에 산다.

쿼드의 대가가 **헬퍼 레인(helper lane)/오버셰이딩**이다. 삼각형이 쿼드의 1픽셀만 덮어도 4레인이 전부 PS를 실행한다(3개는 미분 제공용, 쓰기는 안 함). 삼각형이 픽셀 크기에 가까워지면 PS 비용이 최대 4배로 뻥튀기된다 — "폴리곤을 무한정 늘리면 안 되는" 하드웨어적 이유이고, 테셀레이션과 나이브한 LOD 부재가 비싼 이유다.

### PS — 프래그먼트당 1회

PS는 보간된 속성을 입력으로 색(`SV_Target`)을 출력한다. 계약상 중요한 것 두 가지: (1) PS는 자신의 프래그먼트 좌표 밖 프레임버퍼를 읽을 수 없다(블렌딩은 OM의 몫 — 이 격리가 병렬성의 조건), (2) `clip()`/`discard`로 프래그먼트를 버리거나 `SV_Depth`로 깊이를 직접 쓸 수 있는데, **둘 다 아래 early-z를 깨는 스위치다.**

### OM — 깊이/스텐실 테스트, 블렌딩

API 스펙의 논리 순서는 "PS가 색을 만든 후" 깊이/스텐실 테스트 → 블렌딩이다.

깊이 테스트: `z_fragment (비교함수) z_buffer`가 참이면 통과, 통과 시 (쓰기 마스크가 켜져 있으면) 깊이를 갱신하고 색을 낸다. 비교 함수·쓰기 여부가 **분리된 스테이트**라는 게 핵심 설계다 — "테스트는 하되 쓰지 않기"(반투명), "항상 통과"(UI) 같은 조합이 나온다.

블렌딩 방정식은 고정된 틀에 계수만 꽂는다:

```
O = SrcColor * SrcBlend  (BlendOp)  DestColor * DestBlend

알파 블렌드:  O = src.rgb * src.a  +  dst.rgb * (1 - src.a)   ← "src.a만큼 덮어쓰기"
가산 블렌드:  O = src.rgb * src.a  +  dst.rgb                  ← "빛을 더하기" (FX 글로우)
```

알파 블렌드는 **순서 의존**이다: `A over B ≠ B over A`. 그래서 반투명은 뒤→앞 정렬이 필요하고, 가산 블렌드는 덧셈이라 교환법칙이 성립해 정렬이 필요 없다 — FX에 가산이 사랑받는 실무적 이유가 수학에서 나온다.

### early-z — 깊이 테스트를 PS 앞으로

논리 순서(PS 후 깊이 테스트)를 그대로 구현하면, 어차피 가려질 프래그먼트에 비싼 PS를 다 돌리고 버리게 된다. 그래서 하드웨어는 **결과가 동일하다는 조건 하에** 깊이 테스트를 래스터라이저 직후로 당긴다(early-z). 여기에 타일 단위 계층 깊이 버퍼(Hi-Z)로 프래그먼트 묶음을 통째로 기각하는 최적화가 얹힌다.

"결과가 동일하다"가 깨지는 조건이 면접 단골이다:

- **PS가 `SV_Depth`를 쓰면** — 테스트할 깊이값 자체가 PS를 돌려야 나온다. early-z 완전 불가.
- **PS가 `discard`/`clip()`을 하면** — 테스트는 미리 할 수 있어도 **깊이 쓰기**는 미리 못 한다(프래그먼트가 살아남을지 PS가 끝나야 안다). 미리 써버렸는데 discard되면 "존재하지 않는 표면의 깊이"가 버퍼에 남아 뒤 물체를 잘못 가린다. 하드웨어는 보수적으로 early 쓰기를 포기하거나 테스트만 early로 한다(re-Z).
- PS가 UAV에 쓰는 경우 등 부수효과가 있으면 역시 보수적으로 동작한다.

SM5 HLSL의 `[earlydepthstencil]` 속성은 "깨지는 조건이 있어도 강제로 early로 해라(부작용은 내 책임)"라는 명시적 계약이다. 실무 패턴으로는 **깊이 프리패스(depth pre-pass)** — 씬을 깊이만 먼저 깔고 본 패스는 `EQUAL` 비교로 그리면 오버드로우 PS 비용이 0에 수렴 — 가 early-z를 전제로 한 설계다.

### GPU 하드웨어 실행 모델 — SIMT

여기까지가 논리 파이프라인이라면, 이제 그것이 도는 물리 기계다.

#### 락스텝 SIMT — 워프 32 / 웨이브프론트 64

GPU 코어 수천 개는 CPU 코어 수천 개가 아니다. **SIMT(Single Instruction, Multiple Threads)**: 인스트럭션 디코더/스케줄러 하나가 32개(NVIDIA warp) 또는 64개(AMD GCN wavefront; RDNA는 wave32/64 선택) 레인을 **같은 인스트럭션으로 동시에** 구동한다. 제어 로직을 1/32로 아모타이즈하고 그 트랜지스터를 ALU에 쏟는 것 — "GPU 코어가 많은" 이유는 코어가 단순해서다. 분기 예측기, 비순차 실행, 큰 개인 캐시 같은 "단일 스레드를 빠르게 하는 장치"를 전부 버렸다.

VS의 "정점당 1회", PS의 "프래그먼트당 1회"라는 파이프라인 계약이 바로 이 하드웨어에 정확히 대응한다 — 서로 통신하지 않는 동일 연산 수백만 개는 락스텝 기계의 이상적인 먹이다.

#### 분기 발산(divergence) — 비용의 실체

락스텝의 대가: 한 워프 안에서 `if`의 참/거짓이 갈리면, 하드웨어는 **두 경로를 순차로 모두 실행**하고 실행 마스크(execution mask)로 각 레인의 쓰기만 걸러낸다.

```
if (cond) { A; }  else { B; }

워프 내 cond가 섞이면:  A 실행 (거짓 레인은 마스크 아웃)
                       B 실행 (참 레인은 마스크 아웃)
                       → 비용 = A + B  (둘 중 하나가 아니라 둘 다)
```

따라서 정확한 명제는 "GPU에서 분기는 나쁘다"가 아니라 **"발산하는 분기가 나쁘다"**이다. 워프 32레인 전원이 같은 방향이면(coherent) 반대 경로는 통째로 스킵되어 분기 비용이 거의 없다. 머티리얼별로 드로우를 나누고, 화면 타일 단위로 라이트를 컬링하는 실무 패턴들은 전부 "분기를 워프 단위로 정렬"하는 작업이다.

#### 레이턴시 숨기기 — 캐시 대신 스레드

VRAM 접근은 수백 사이클이다. CPU는 이걸 큰 캐시 계층과 비순차 실행으로 숨긴다. GPU의 답은 다르다: **워프가 메모리를 기다리면 스케줄러가 0 사이클에 다른 상주 워프로 갈아탄다.** 문맥 전환이 공짜인 이유는 각 워프의 레지스터가 거대한 레지스터 파일에 **상주한 채로** 있기 때문이다 — 저장/복원이 없다.

여기서 **occupancy(점유율)** 개념이 나온다: SM(Streaming Multiprocessor) 하나에 동시 상주 가능한 워프 수의 실제/최대 비율. 상주 워프가 많을수록 "갈아탈 상대"가 많아 레이턴시가 잘 숨는다. 상주를 제한하는 자원이:

- **레지스터**: 스레드당 레지스터를 많이 쓰는 무거운 셰이더는 레지스터 파일이 먼저 바닥나 상주 워프 수가 준다 — 이것이 **레지스터 압박(register pressure)**. "셰이더에 지역 변수/분기 경로를 늘렸더니 전체가 느려졌다"의 정체가 대개 이것이다.
- **공유 메모리**(컴퓨트의 groupshared): 그룹당 할당량이 SM 용량을 나눠 갖는다.
- 스레드/그룹 수 하드 리밋.

단, occupancy는 수단이지 목표가 아니다 — 숨길 레이턴시보다 상주가 이미 충분하면 더 올려도 이득이 없고, 레지스터를 줄이려 메모리에 흘리면(spill) 오히려 느려진다.

#### 메모리 계층 — 대역폭이 병목인 이유

| 계층 | 스코프 | 대략적 레이턴시 | 성격 |
|---|---|---|---|
| 레지스터 | 스레드 | ~0 (파이프라인 내) | 워프당 상주 할당 |
| 공유 메모리/LDS | 스레드 그룹 | 수십 사이클 | 프로그래머가 관리하는 스크래치 |
| L1/텍스처 캐시 | SM | 수십 사이클 | 2D 지역성 최적화(텍스처) |
| L2 | 칩 전체 | ~백 수십 사이클 | 모든 SM 공유 |
| VRAM (GDDR) | 디바이스 | 수백 사이클 | 수백 GB/s 대역폭 |

(수치는 아키텍처마다 다르므로 자릿수 감각으로만.)

**왜 병목이 연산이 아니라 대역폭인가.** 연산 처리량(FLOPS)은 트랜지스터를 더 박으면 늘지만, 대역폭은 핀 수·클럭·전력의 물리적 벽에 막혀 훨씬 느리게 는다. 4K 프레임 하나만 해도 색+깊이 읽기쓰기, G버퍼, 텍스처 샘플까지 픽셀당 수십~수백 바이트가 오간다. 그래서 렌더링 최적화의 절반은 "바이트를 아끼는 것"이다: 텍스처 압축(BC 계열은 하드웨어가 캐시 라인에서 직접 디코드), 밉맵(캐시 히트), 포맷 다이어트(R11G11B10 등), 그리고 아예 트래픽을 온칩에 가두는 TBDR(아래).

연산과 메모리의 비율을 **산술 강도(arithmetic intensity)** = FLOP/byte로 재고, 낮으면 대역폭 바운드라 ALU 최적화가 무의미하다는 루프라인(roofline) 사고방식이 면접에서 한 단계 깊은 답을 만든다.

### 컴퓨트 셰이더 — 그래픽스 파이프라인 밖의 GPU

컴퓨트 셰이더는 IA도 래스터라이저도 OM도 없이 **SIMT 기계만 직접 노출**한 것이다.

```hlsl
[numthreads(8, 8, 1)]                       // 스레드 그룹 = 8x8 = 64 스레드
void CSMain(uint3 dtid : SV_DispatchThreadID)
{
    // dtid로 "내가 담당하는 데이터"를 직접 계산; 출력은 UAV에 자유 쓰기
}
```

`Dispatch(gx, gy, gz)`가 그룹을 띄우고, 그룹 안에서만 `groupshared` 메모리와 배리어(`GroupMemoryBarrierWithGroupSync`)로 협력할 수 있다 — 그룹이 "한 SM에 함께 상주함"이 보장되는 단위이기 때문이다(그룹 간 협력은 보장 수단이 없다). 그룹 크기를 32/64의 배수로 잡는 관례는 워프/웨이브 단위 락스텝에 딱 맞춰 부분 워프 낭비를 없애기 위해서다.

그래픽스 파이프라인이 "정점→픽셀"이라는 고정 데이터 흐름을 강제하는 반면, 컴퓨트는 입출력 형태가 자유라서 스키닝, GPU 컬링, 파티클 시뮬레이션, 라이트 컬링, 포스트 프로세싱, 그리고 아예 소프트웨어 래스터라이저(UE5 Nanite)까지 — "래스터라이저가 필요 없는 GPU 일" 전부가 컴퓨트로 넘어가는 것이 지난 10년의 큰 흐름이다.

### CPU-GPU 비동기 — 커맨드는 큐에, 실행은 나중에

`DrawIndexed` 호출은 그림을 그리지 않는다. **드라이버의 커맨드 버퍼에 명령을 기록하고 즉시 리턴한다.** GPU는 그 버퍼를 뒤에서 소비하는 별개의 프로세서이고, 통상 CPU보다 1~3프레임 뒤를 실행한다(DXGI 기본 최대 프레임 지연 3).

```
CPU:  [프레임 N 기록.....][프레임 N+1 기록.....][프레임 N+2 기록...]
GPU:            [프레임 N-1 실행.....][프레임 N 실행.....]
                ↑ 이 간격이 버퍼 = 지연이자 방어막
```

이 간격은 양날이다: CPU/GPU 중 한쪽이 잠깐 튀어도 다른 쪽이 굶지 않게 하는 완충이면서, 입력→화면 지연(input latency)의 원천이다.

#### Map WRITE_DISCARD가 존재하는 이유 — 버퍼 리네이밍

문제: CPU가 프레임 N+2의 상수 버퍼를 쓰려는데, GPU는 아직 프레임 N에서 **같은 버퍼를 읽고 있다.** 그냥 쓰면 레이스, 기다리면 비동기성이 무너진다(파이프라인 스톨).

`D3D11_MAP_WRITE_DISCARD`는 "이전 내용은 버려도 된다"는 계약이다. 드라이버는 그 순간 **새 메모리 조각을 내주고(리네이밍, renaming)** 핸들만 거기에 다시 묶는다. GPU는 옛 조각을 계속 읽고, CPU는 새 조각에 쓴다 — 대기 0. 옛 조각은 GPU가 다 읽으면 드라이버가 회수한다. 즉 동적 버퍼 하나의 뒤에는 사실상 프레임 수만큼의 물리 사본이 돌고 있다. `WRITE_NO_OVERWRITE`는 반대로 "GPU가 읽는 중인 영역은 내가 안 건드린다고 약속"하고 같은 할당에 이어 쓰는 모드다(동적 정점 버퍼에 링 버퍼처럼 append할 때).

DISCARD 없이 `MAP_WRITE`나 즉시 readback(`GetData` 강제 플러시 등)을 하면 **CPU가 GPU를 기다리는 스톨**, 반대로 CPU가 명령을 늦게 공급하면 GPU가 노는 **버블(bubble)**이 생긴다. GPU 프로파일러 타임라인에서 빈 구간을 읽는 언어가 이 두 단어다.

#### Present, vsync, 백버퍼

그리는 대상은 화면이 아니라 **백버퍼(back buffer)**다. `Present`는 "이 백버퍼를 표시 큐에 넘겨라"는 명령이고, SyncInterval=1이면 수직 동기(vblank) 시점에만 교체돼 화면 위아래가 서로 다른 프레임이 되는 티어링(tearing)이 없다. 0이면 즉시 교체 — 지연은 최소지만 티어링을 감수한다. vsync on일 때 프레임이 16.6ms를 살짝 넘기면 다음 vblank까지 통째로 기다려 30fps로 반토막 나는 계단 현상, 그리고 그걸 푸는 가변 주사율(VRR)/트리플 버퍼링까지가 이 주제의 꼬리 세트다. 스왑 방식에는 교체 시 복사 기반의 구식 blt 모델(DXGI DISCARD)과 포인터 교환으로 합성기(DWM)와 직접 협상하는 FLIP 모델이 있고, FLIP이 지연/효율에서 낫지만 백버퍼 수명 규약이 더 엄격하다.

### 모바일 TBDR — 같은 파이프라인, 다른 물리 전략

데스크톱 GPU는 대체로 **IMR(Immediate Mode Rendering)**: 삼각형이 오는 순서대로 즉시 래스터→셰이딩→VRAM의 프레임버퍼에 쓴다. 모바일(Apple/ARM Mali/Qualcomm Adreno 계열)은 **TBDR(Tile-Based Deferred Rendering)**:

```
IMR :  삼각형 → 래스터 → PS → VRAM 프레임버퍼 (픽셀당 수차례 왕복)
TBDR:  [비닝 패스] 모든 지오메트리를 먼저 변환, 화면 타일(예: 16x16~32x32)별
       프리미티브 목록으로 분류 → [타일 패스] 타일 하나씩 온칩 타일 메모리에서
       래스터+깊이+블렌딩 전부 처리 → 완성된 타일만 VRAM에 1회 기록
```

이유는 전력이다. 모바일에서 DRAM 왕복은 연산보다 자릿수로 비싼 에너지를 먹는다. 깊이 버퍼와 블렌딩의 read-modify-write 트래픽을 전부 온칩 SRAM에 가두면 대역폭과 전력이 극적으로 준다 — 깊이 버퍼는 아예 VRAM에 존재조차 안 할 수 있다(타일 끝에서 버림). PowerVR 계열의 HSR(hidden surface removal)은 타일 안에서 불투명 픽셀의 최종 승자만 셰이딩해 오버드로우를 구조적으로 제거하기도 한다.

대가: (1) 비닝 패스가 지오메트리 전체를 한 번 소화해야 하므로 정점 폭주에 취약, (2) 렌더 패스 중간에 렌더타겟을 읽거나 바꾸면 타일 전체 flush가 강제되는 함정(모바일에서 "패스를 쪼개지 마라"는 격언의 근거), (3) discard/alpha test가 HSR 같은 타일 최적화를 방해. Vulkan의 renderpass/loadOp/storeOp 개념이 바로 TBDR에게 "이 어태치먼트는 타일 끝에 버려도 된다"를 알려주는 API 표면이다.

## ③ 심화 — 꼬리질문 대비

**"perspective-correct interpolation에서 왜 하필 1/w인가? 유도해 보라."**
원근 투영에서 화면 좌표는 `x_s = x/w` 꼴, 즉 사영 변환이다. 삼각형 평면 위에서 (월드 기준) 선형인 속성 `a`는 화면에서는 분수 함수가 된다. 그런데 `1/w`를 곱해 보면: 클립 공간에서 `x, y, w`가 정점 간 선형이므로 `x_s = x/w`로 나눈 화면에서는 `1/w` 자신과 `a/w`가 화면 좌표의 선형(정확히는 아핀) 함수가 된다. 그래서 "화면에서 선형 보간해도 되는 양"인 `a/w`, `1/w`를 보간한 뒤 나눠 `a`를 복원한다. 한 줄 답: **"화면 공간에서 선형인 것은 a가 아니라 a/w와 1/w이기 때문."**

**"ddx/ddy를 분기 안에서 쓰면 왜 위험한가?"**
미분은 2x2 쿼드 이웃 레인의 값 차이로 구현되는데, 발산 분기 안에서는 이웃 레인이 그 코드를 실행하지 않아(마스크 아웃) 비교할 값이 정의되지 않는다 — 결과는 미정의. 그래서 `Sample`(암시적 미분 사용)을 발산 분기 안에서 호출하는 것은 스펙상 미정의이고, 컴파일러가 경고한다. 해법: 미분/`Sample`을 분기 밖으로 호이스트하거나, `SampleLevel`(LOD 명시)이나 `SampleGrad`(미분 명시)로 미분 의존을 끊는다.

**"early-z가 깨지는 조건을 정확히 말해 보라."**
(1) PS가 `SV_Depth` 출력 — 테스트 대상 깊이가 PS 결과이므로 원천 불가. (2) PS에 `discard`/`clip` — 테스트는 일러도 되지만 깊이 **쓰기**를 미리 하면 discard된 프래그먼트의 깊이가 남는 오답이 되므로 쓰기가 늦춰지거나 전체가 보수적으로 late로 간다. (3) PS의 UAV 쓰기 같은 부수효과 — 깊이에서 탈락할 프래그먼트의 부수효과 실행 여부가 달라지므로 보수적 처리. 덧붙이면 좋은 것: Hi-Z(타일 단위 조기 기각)와 `[earlydepthstencil]` 강제 속성, 그리고 "그래서 알파 테스트 머티리얼은 불투명보다 비싸다"는 실무 결론.

**"반투명은 왜 뒤에서 앞으로 정렬하나? 정렬이 싫으면 대안은?"**
알파 블렌딩 `src*a + dst*(1-a)`는 비가환 — 겹침 순서가 결과 색을 바꾼다. 올바른 합성은 카메라에서 먼 것부터 over 연산자를 누적하는 것이므로 뒤→앞 정렬이 필요하다. 대안: (1) 가산 블렌드는 가환이라 정렬 불요(FX가 애용), (2) 순서 독립 투명(OIT) — per-pixel linked list나 weighted blended 근사, (3) 알파 테스트로 아예 불투명화, (4) 메시 단위 정렬의 근본 한계(상호 관통)는 인정하고 픽셀 단위가 필요하면 OIT뿐이라고 답한다.

**"드로우콜이 비싸다는데, 정확히 어디가 비싼가?"**
GPU가 아니라 주로 CPU다. DX11 기준 드로우콜마다 드라이버가 상태 조합 검증, 해저드 추적, 커맨드 버퍼 인코딩을 수행하고, 상태 변경(셰이더/텍스처/블렌드 교체)이 파이프라인 재구성을 유발한다. GPU 쪽도 드로우 사이 상태 전환에서 파이프라인이 부분적으로 비워질 수 있다. 그래서 해법이 전부 "호출 수 줄이기"다: 배칭, 인스턴싱(같은 메시 N개를 드로우 1회), 텍스처 아틀라스/배열로 상태 변경 제거, 그리고 DX12/Vulkan의 사전 컴파일 PSO + 커맨드 리스트 병렬 기록으로 드라이버 오버헤드 자체를 구조적으로 없애는 방향.

**"GPU에서 분기는 무조건 피해야 하나?"**
아니다. 비용은 발산에서만 나온다. 워프 전체가 같은 방향이면(예: 화면 타일 단위로 균일한 조건, 상수 버퍼 기반 분기) 분기는 거의 공짜고, 오히려 일을 스킵해 이득이다. 또 `cond ? a : b` 수준은 컴파일러가 분기 없이 select/predication으로 편다. 정확한 워딩: "분기를 피하라"가 아니라 **"분기 조건을 워프 경계와 정렬시켜라."**

**"삼각형을 아주 작게 쪼개면 왜 느려지나?"**
쿼드 오버셰이딩. PS는 2x2 쿼드 단위로 실행되므로 1픽셀짜리 삼각형도 4레인을 태우고, 인접 삼각형들이 같은 쿼드를 각자 다시 태운다. 픽셀 크기 삼각형 밀집 지대에서는 PS 인보케이션이 커버 픽셀 수의 몇 배로 폭증하고, 래스터라이저의 삼각형당 고정 비용(셋업)도 픽셀 수 대비 비대해진다. 이것이 LOD의 하드웨어적 근거이고, Nanite가 픽셀급 삼각형을 하드웨어 래스터 대신 컴퓨트 소프트웨어 래스터로 처리하는 이유다.

**"CPU는 캐시로 레이턴시를 숨기는데 GPU는 왜 스레드로 숨기나?"**
워크로드 성격이 다르다. CPU는 소수 스레드의 순차 로직 — 같은 데이터를 재사용할 확률이 높아 캐시가 맞다. GPU는 수백만 픽셀을 한 번씩 스치는 스트리밍 — 재사용이 적어 캐시 적중이 구조적으로 낮은 대신, 서로 독립인 일감이 무한하다. 그러면 기다리는 동안 딴 일을 하는 게 정답이고, 그러려면 문맥 전환이 공짜여야 하므로 모든 상주 워프의 레지스터를 거대한 레지스터 파일에 상시 보관한다. 트랜지스터 예산을 "캐시+예측기" 대신 "레지스터 파일+ALU"에 쓴 것 — 같은 실리콘, 다른 베팅.

**"occupancy 100%면 항상 최선인가?"**
아니다. occupancy는 레이턴시를 숨길 후보 워프의 수일 뿐, 숨길 레이턴시가 이미 다 숨었다면 더 올려도 성능이 늘지 않는다. 역효과도 있다: occupancy를 올리려 스레드당 레지스터를 줄이면 스필(spill)이 나 메모리 트래픽이 늘고, 상주 워프가 많아지면 워프당 L1/캐시 지분이 줄어 적중률이 떨어질 수 있다. 올바른 진술: "occupancy는 메모리 바운드 셰이더에서 부족하면 병목이지만, 충분함 이상은 무의미하다."

**"UpdateSubresource와 Map(WRITE_DISCARD)의 차이는?"**
둘 다 CPU→GPU 데이터 갱신이지만 경로가 다르다. `Map(WRITE_DISCARD)`는 DYNAMIC 리소스 전용으로, CPU가 쓰기 가능한 메모리를 직접 받아 쓰며 리네이밍으로 무대기를 보장한다 — 매 프레임 갱신에 적합. `UpdateSubresource`는 DEFAULT 리소스용으로, 드라이버가 데이터를 내부 스테이징에 복사해 두었다가 커맨드 스트림 순서에 맞춰 GPU가 리소스로 옮긴다 — 추가 복사가 생길 수 있어 빈번한 갱신엔 불리하고, 가끔 갱신하는 리소스에 적합. 한 줄: "매 프레임이면 DYNAMIC+DISCARD, 가끔이면 DEFAULT+UpdateSubresource."

**"vsync를 끄면 왜 티어링이 생기나?"**
디스플레이는 프레임버퍼를 위에서 아래로 순차 스캔아웃한다. 스캔 도중에 백버퍼가 교체되면 화면 위쪽은 이전 프레임, 아래쪽은 새 프레임 — 그 경계선이 티어링이다. vsync는 교체를 vblank(스캔 사이 공백)로 미뤄 이를 막지만, 프레임이 주기를 못 맞추면 다음 vblank까지 대기해 프레임레이트가 주사율의 정수 분수로 양자화된다(60→30). VRR(G-Sync/FreeSync)은 반대로 디스플레이가 GPU에 맞춰 스캔 시점을 조절해 양쪽 문제를 푼다.

**"GS는 왜 느린가?"**
출력 개수가 프리미티브마다 가변이라 (1) 최악 크기 기준의 중간 버퍼 할당이 필요하고 일부 아키텍처에서는 이것이 온칩이 아닌 메모리를 경유했으며, (2) API가 입력 프리미티브 순서대로의 출력 순서를 보장하므로 병렬 실행 결과를 직렬로 재정렬해야 한다. "1:1 고정"이라는 VS의 계약이 얼마나 하드웨어 친화적이었는지를 반대로 증명하는 스테이지. 현대의 대체재: 인스턴싱, 컴퓨트로 버퍼 사전 생성, 메시 셰이더.

**"컴퓨트 스레드 그룹 크기는 어떻게 정하나?"**
(1) 워프/웨이브(32/64)의 배수 — 아니면 마지막 부분 워프 레인이 낭비된다. (2) groupshared 사용량과 레지스터가 SM 상주 그룹 수를 결정하므로 occupancy와의 균형. (3) 데이터 접근 패턴 — 이미지 처리면 8x8/16x16 타일이 텍스처 캐시 2D 지역성과 맞물린다. "64나 256부터 시작해 프로파일링으로 조정"이 실무 답.

**"TBDR에서는 무엇이 싸지고 무엇이 비싸지나?"**
싸지는 것: 깊이/스텐실/블렌딩 — 전부 온칩 타일 메모리의 read-modify-write라 알파 블렌딩조차 대역폭 비용이 미미하다. MSAA도 타일 안에서 resolve하면 VRAM엔 resolve 결과만 나간다. 비싸지는 것: 지오메트리 양(비닝 패스가 전량 소화 + 타일별 목록 메모리), 렌더 패스 중간의 렌더타겟 전환/읽기(타일 flush 강제), 그리고 discard/알파 테스트(HSR 같은 타일 내 은면 제거 최적화를 무력화). 그래서 모바일 최적화 격언이 데스크톱과 다르다: "오버드로우보다 패스 분할과 정점 수를 먼저 봐라."

**"왜 블렌딩은 프로그래머블이 아닌가?"**
같은 픽셀에 닿는 프래그먼트들의 read-modify-write를 **드로우 제출 순서대로** 보장해야 하는데, PS에서 프레임버퍼를 읽게 하면 픽셀 단위 락이 필요해 병렬성이 붕괴한다. 그래서 순서 보장은 ROP 전용 하드웨어에 격리하고 PS는 프레임버퍼를 못 읽게 잘랐다. 이 절단선의 존재를 알면 파생 질문(프로그래머블 블렌딩이 필요한 OIT가 왜 UAV/linked-list 같은 우회로 가는지, 모바일 TBDR이 framebuffer_fetch 확장을 줄 수 있는 이유 — 타일 메모리라 순서 보장이 싸다)까지 연결된다.

## ④ Winters에서의 적용

### (1) 드로우 경로 실물 — 스테이지 바인딩이 곧 파이프라인 지도

`Engine/Private/Renderer/ModelRenderer.cpp:487-506`의 스키닝 드로우 경로는 이 챕터의 논리 파이프라인을 API 호출 순서로 그대로 보여준다: 셰이더/파이프라인(입력 레이아웃) 바인딩 → cbPerFrame(b0)/cbPerObject(b1) 바인딩 → 본 팔레트 SRV(t8, `:503`) → `IASetPrimitiveTopology(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST)`(`:505`) → 메시 드로우. "IA에 무엇을(토폴로지·레이아웃), VS에 무엇을(cbuffer·본), PS에 무엇을(머티리얼 SRV) 공급하는가"가 코드 블록 하나에 정렬돼 있어, 면접에서 화이트보드에 파이프라인을 그린 뒤 "제 엔진의 이 함수가 정확히 이 순서입니다"로 착지할 수 있다. 입력 레이아웃이 VS 시그니처와 사전 검증되는 지점은 `Engine/Private/RHI/DX11/DX11Pipeline.cpp:18`(`CreateInputLayout`), 바인딩은 `:108`이다. 실제 인덱스 드로우는 RHI 경유 시 `Engine/Private/RHI/DX11/CDX11Device.cpp:678`의 `DrawIndexedInstanced`로 떨어진다 — 인스턴스 수 1의 일반 드로우도 인스턴스드 엔트리로 통일한 것인데, "인스턴싱이 드로우콜 오버헤드 대책"이라는 ③의 논의와 접붙일 수 있는 지점이다.

### (2) Map WRITE_DISCARD — 버퍼 리네이밍의 실물과 이중 경로

매 프레임 갱신되는 상수 버퍼는 `Engine/Private/RHI/DX11/DX11ConstantBuffer.h:47`에서 `D3D11_USAGE_DYNAMIC`으로 만들고 `:59`에서 `Map(m_pBuffer, 0, D3D11_MAP_WRITE_DISCARD, 0, &mapped)`로 쓴다 — GPU가 1~3프레임 전 값을 읽는 동안 CPU가 대기 없이 새 값을 쓰는 리네이밍 계약의 정확한 사용례다. 같은 파일 `:15`의 `static_assert(sizeof(T) % 16 == 0, ...)`는 HLSL cbuffer 16바이트 정렬 규약을 컴파일 타임에 강제한다.

더 좋은 이야깃거리는 RHI 계층의 **이중 경로**다. `Engine/Private/RHI/DX11/CDX11Device.cpp:693-707`의 `UpdateBuffer`는 리소스가 dynamic이면 `Map(WRITE_DISCARD)`(`:696`), 아니면 `UpdateSubresource`(`:706`)로 분기한다 — ③의 "매 프레임이면 DISCARD, 가끔이면 UpdateSubresource"라는 원칙이 엔진 코드에 조건문 하나로 박제돼 있는 셈이다. 본 팔레트도 같은 패턴이다: `Engine/Private/Renderer/ModelRenderer.cpp:41`(DYNAMIC) + `:62`(WRITE_DISCARD Map).

### (3) cbuffer의 한계와 구조화 버퍼 — 왜 본 팔레트는 t8인가

`Engine/Private/Renderer/ModelRenderer.cpp:28-30`의 주석과 `kMaxGPUBones = 1024`는 실제 겪은 한계의 기록이다: cbuffer는 크기 제한(64KB)과 상수 레지스터 중심 접근에 묶여 256~512본 팔레트에서 벽을 만났고, 구조화 버퍼 SRV(`MiscFlags = D3D11_RESOURCE_MISC_BUFFER_STRUCTURED`, `:44`)로 옮겨 1024본 대형 리그를 드로우 1회로 스키닝한다. 면접 연결: "cbuffer는 워프 전체가 같은 값을 읽는 uniform 접근에 최적화된 경로이고, 정점마다 다른 인덱스로 읽는 본 행렬은 SRV 버퍼가 맞는 도구"라는 메모리 경로 구분까지 말하면 하드웨어 이해가 증명된다.

### (4) clip()과 early-z — 내 셰이더의 비용을 스스로 감사하기

Winters 셰이더에는 `clip()`이 광범위하게 있다: 불투명 캐릭터 경로인 `Shaders/Skinned3D_PBR.hlsl:120`의 `clip(albedoSample.a - 0.05f);`, FX 경로인 `Shaders/FxSprite.hlsl:142`/`:159`(알파 클립·erode 디졸브) 등. ②의 early-z 논의를 적용하면 이 한 줄들의 숨은 가격이 보인다 — **불투명 메시 전부가 early 깊이 쓰기를 보수적으로 포기당할 수 있는 조건**을 달고 있는 것이다. 면접 답변 각: "알파 컷이 실제로 필요한 머티리얼(잎사귀·머리카락류)과 항상 alpha=1인 머티리얼을 분리해, 후자는 clip 없는 변형 셰이더로 돌리는 것이 다음 최적화 후보다. 검증은 GPU 프로파일러에서 깊이 프리패스 유무/PS 인보케이션 카운트 비교로 한다." — 자기 코드의 트레이드오프를 인지하고 개선안·측정법까지 말하는 구도가 된다.

### (5) OM 스테이트의 의도적 조합 — FX의 "테스트는 하되 쓰지 않기"

`Engine/Private/Renderer/FxStaticMeshRenderer.cpp:535-538`은 깊이 스테이트 교과서다: `DepthEnable = TRUE` + `DepthWriteMask = D3D11_DEPTH_WRITE_MASK_ZERO` + `LESS_EQUAL` — 반투명 FX가 벽 뒤에서는 가려지되(테스트 on) 자기들끼리 깊이를 오염시키지 않는(쓰기 off) 조합이고, `:546-549`에는 깊이 무시 오버레이용 `DepthEnable = FALSE` 변형이 나란히 있다. 블렌딩은 `Engine/Private/RHI/DX11/BlendStateCache.cpp:20-69`가 Opaque/AlphaBlend/PremultipliedAlpha/Additive 4종 프리셋 캐시로 관리한다 — `:60-61`의 Additive(`SrcBlend = SRC_ALPHA, DestBlend = ONE`)가 ②에서 말한 "가환이라 정렬 불요"인 FX용 가산 합성의 실물이다. 프레임 안의 순서도 이 원칙대로다: `Client/Private/Scene/Scene_InGameRender.cpp:328`의 `OnRender`는 NormalPass/SSAO(`:359-419`) → 맵/구조물/미니언 등 불투명(`:559-567`) → FX 시스템들(`:615-619`)을 마지막에 그린다 — "불투명 먼저(early-z 활용), 반투명 나중(블렌딩 순서)"의 표준 프레임 구성.

### (6) 밉맵 — ddx/ddy의 소비자를 런타임에 만들어 주기

`Engine/Private/Resource/Texture.cpp:88`의 `EnsureMipsOnBind`는 1-mip으로 로드된 텍스처를 풀 밉 체인 텍스처로 교체하고 `GenerateMips`(`:149`)를 수행한다(`RENDER_TARGET | GENERATE_MIPS` 플래그로 하드웨어 다운샘플). ②의 LOD 수식과 이어 말하면: "샘플러가 ddx/ddy로 LOD를 골라도 밉이 1장이면 고를 것이 없다 — 원거리 표면이 밉 0을 읽으며 앨리어싱과 텍스처 캐시 미스를 동시에 낸다. 그래서 밉 체인 보장을 리소스 계층에서 강제했고, immediate context를 쓰므로 렌더 스레드에서만 호출되게 제한했다(`:87` 주석)."

### (7) CPU-GPU 비동기의 실물 — 쿼리 링 버퍼와 Present

GPU 타이밍 계측 코드가 비동기성의 가장 정직한 증거다. `Engine/Private/RHI/DX11/CDX11Device.cpp:990-1001`의 `EndFrame`은 타임스탬프 쿼리를 슬롯 링(`m_uGpuTimingWriteIndex` 순환)에 쌓기만 하고, 결과 회수는 `:923-941`의 `ReadGpuTimingResults`가 `GetData(..., D3D11_ASYNC_GETDATA_DONOTFLUSH)`로 시도하다 **아직 준비 안 됐으면 그냥 넘어간다** — "지금 당장 내놔"라고 플러시하면 파이프라인 스톨이므로, 몇 프레임 뒤에 도착하는 결과를 링으로 받는 구조다. GPU가 CPU보다 뒤에서 달린다는 사실을 코드가 전제하고 있다. 화면 제시는 `:1005`의 `Present(m_bVSync ? 1 : 0, 0)` — SyncInterval로 vsync를 토글하고, `DXGI_ERROR_DEVICE_REMOVED`를 명시적으로 처리한다. 스왑체인은 `:790-801`에서 `BufferCount = 1` + `DXGI_SWAP_EFFECT_DISCARD`의 보수적 blt 모델인데, 주석(`:788-789`)에 "FLIP 모델은 백버퍼 순환/재바인딩 정책을 더 엄격히 맞춰야 해 초기 안정화 단계에서는 DISCARD"라는 의사결정이 남아 있다 — "FLIP_DISCARD + BufferCount 2~3으로 가면 DWM 합성 경유 복사 없이 지연·대역폭이 개선된다, 마이그레이션 시 백버퍼 RTV 재획득 규약을 지켜야 한다"까지 말하면 트레이드오프 답변이 완성된다.

### (8) 없는 것 — 컴퓨트 셰이더는 스텁이다

`Engine/Private/RHI/DX11/CDX11Device.cpp:681`은 `void Dispatch(u32_t, u32_t, u32_t) override {}` — 컴퓨트 디스패치가 아직 no-op 스텁이다. 이걸 숨길 게 아니라 면접 답변으로 쓴다: "현재 SSAO는 풀스크린 PS 패스로 돌고 있는데, 컴퓨트로 옮기면 (1) groupshared에 깊이/노멀 타일을 캐시해 이웃 샘플 재사용으로 대역폭을 줄이고, (2) 래스터라이저/OM을 통과하지 않아 고정 오버헤드가 빠진다. 첫 이식 대상으로 SSAO와 스키닝(본 팔레트 곱을 컴퓨트로 사전 계산해 정점 버퍼에 기록 — VS 부담과 패스 간 재계산 제거)을 보고 있고, 스레드 그룹은 8x8에서 시작해 occupancy를 프로파일링으로 잡겠다." — 파이프라인 밖의 GPU를 이해하고 있으며 자기 엔진의 다음 단계를 하드웨어 언어로 설계할 수 있음을 보여주는 답이다.
