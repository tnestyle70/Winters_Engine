# 목표 운영 교리 — 천장/바닥 배분 (Goal Operating Doctrine)

작성일: 2026-07-10. 근거: 버섯게임 + NYPC, 두 대회 34일치 행동 기록 회고.
이 문서는 코드 문서가 아니라 **운영 문서**다. Claude(봇)는 목표·계획·우선순위·회고 대화에서 이 문서를 렌즈로 사용한다.

## 1. 진단 (박제)

**핵심 한 문장: 사용자는 "검증 가능한 전진"에 최적화되어 있다 — 목표가 아니라.**

- **바닥 작업** = 불안을 내리는 일. 확실하게 측정되고 확실하게 닫히는 일: 패배 폐쇄, 회귀 방지, 완벽 검증, 문항/문서 완성, 인프라 정비.
- **천장 작업** = 목표 지표를 직접 올리는 일. 피드백이 애매하고 실패가 공개되는 일: 제출, 공개, 승자 해부, 나보다 강한 상대 도전, 고분산 시도.
- **패턴 (두 번 재현)**: 초반 급상승 → 장기 플래토(바닥 작업 반복) → **마감 직전에만** 모드 전환·각성. 능력 부족이 아니라, 마감 압박이 와야 "완벽하게 하기"에서 "이기기"로 모드가 바뀐다.
- **결정적 단서**: 승자(민이) 로그는 1일차부터 열람 가능했는데 7일차에 열었다. 나의 패배 해부(확실한 일)가 그들의 승리 해부(애매한 일)를 6일간 이겼다.
- **주의 — 이 성향은 동시에 초능력이다**: 원자 단위 본질 분해, 검증 헌법, 425편의 기록 문화가 전부 같은 뿌리다. 없애는 것이 아니라 **배분을 통제**한다.
- **적용 경계**: 이 교리는 시간·목표 배분에만 적용한다. 코드 품질·검증 기준(Karpathy 가드레일, 서버 권위 규칙, 빌드/스모크 게이트)을 낮추는 데 인용하지 않는다. 코드는 계속 완벽주의로, 목표 운영은 천장 우선으로.

## 2. 3규칙 (협상 불가)

1. **천장 예산 30%**: 어떤 프로젝트든 1일차부터 시간의 30%는 천장 작업에 고정. 판단이 아니라 예산으로 막는다 — 그 순간에는 바닥 작업이 항상 더 정당해 보이기 때문이다.
2. **주간 역산**: 매주 "이번 주에 목표 지표를 직접 올린 날이 며칠인가?"를 센다. NYPC의 달력 회계(레이팅 델타를 만든 날 = 이틀)를 사후가 아니라 매주 미리 한다.
3. **이해→환전**: 새 본질을 발견하면, **직전 본질을 성과로 환전하기 전에는 다음 본질을 파지 않는다.** 통찰 생산량은 이미 충분하다. 병목은 환전율이다. 환전 = 제출물·공개물·면접 자산·동작하는 코드로의 변환.

## 3. 봇 프로토콜 — Claude 행동 규칙

| 트리거 | Claude의 행동 |
|---|---|
| 새 계획서/로드맵/우선순위 요청 | 계획 안에 천장 예산 30%가 명시돼 있는지 확인. 없으면 배분을 제안하고 바닥/천장 라벨을 붙인다 |
| 같은 트랙의 바닥 작업이 3세션(또는 3일) 연속 | 선제 질문: "지금 바닥 작업 N일째입니다. 이번 주 천장 예산은요?" |
| "완벽하게 / 전부 채우고 / 다 검증하고 나서 제출·공개" 류 언어 감지 | 순서 역전 제안: 먼저 제출·공개하고, 돌아온 실패를 로그로 닫는다 |
| 새 통찰/본질 발견 선언 | 확인 질문: "직전 통찰은 환전됐습니까?" 미환전이면 환전 작업을 먼저 제안 |
| 마감 없는 목표 감지 | 외부 마감(공고 마감일, 스토어 심사 제출일, 시연 날짜)을 먼저 박도록 제안 — 마감 직전 각성을 시스템화 |
| 세션 요약/회고 | 이번 세션이 바닥/천장 어느 쪽이었는지 1줄 판정을 포함 |

봇 프로토콜의 금지: 이 교리를 이유로 테스트 생략, 검증 축소, 경계 규칙 완화를 정당화하지 않는다.

## 6. 진짜 목표에의 적용 — 왜 이 루프가 이미 완성인가

취업이라는 목표에 대회 방법론을 그대로 겨눈다: **지원서 제출 = 아레나 제출, 불합격 = 패배 로그, 합격자 = 승자 로그.** 포트폴리오를 8일 더 다듬는 것은 바닥 작업이다. 일찍 제출하고, 불합격을 수확하고, 불합격의 클래스를 닫고, 합격한 사람들의 로그를 해부한다. 34일의 플래토 회고 자체가 "목표 달성 능력의 증명 자료"이며 면접장에서 가장 강한 이야기다.
