# 자료구조 & 알고리즘 — 기술면접 대비

> 대상: 자체 DX11 엔진(Winters) 제작 경험이 있는 게임 클라이언트/서버 프로그래머 지망자.
> 원칙: 모든 답변은 "정의 → 원리 → 트레이드오프 → 내 프로젝트 실전 연결" 순서로 말한다. 암기한 표를 읊지 말고, **내가 실제로 겪은 코드로 되돌아오는 답변**을 만든다.

## 출제 경향 개요

게임회사 기술면접에서 자료구조/알고리즘은 세 갈래로 나온다.

1. **기본기 검증** — 배열 vs 리스트, 해시 충돌, 힙, BFS/DFS, 정렬. "정의를 아는가"가 아니라 "왜 그걸 고르는가"를 묻는다. 특히 게임회사는 **캐시/메모리 레이아웃 관점의 상수항**을 집요하게 판다 (vector vs list, SoA vs AoS, sparse set).
2. **게임 도메인 응용** — A\*와 휴리스틱, 공간 분할(그리드/쿼드트리/BVH), 근접 탐색 O(N²) 최적화, 우선순위 큐 기반 이벤트/타이머. "LoL 같은 게임에서 유닛 500개의 시야 판정을 어떻게 하겠나" 류의 설계 질문으로 변형된다.
3. **저수준/동시성** — lock-free 개념, CAS, work-stealing, false sharing. 서버/엔진 포지션일수록 비중이 커진다.

나의 강점: **이 카테고리의 거의 모든 주제를 Winters에서 실제 코드로 갖고 있다.** A\*(`Engine/Private/Manager/Navigation/Pathfinder.cpp`), Chase-Lev work-stealing deque(`Engine/Public/Core/JobSystem/WorkStealingDeque.h`), sparse set ECS(`Engine/Public/ECS/ComponentStore.h`), 균일 그리드 공간 분할(`Engine/Public/ECS/SpatialIndex.h`). 일반론 대신 이 코드들을 인용하며 답하면 차별화된다.

## 핵심 개념 정리

### 1. 시간·공간 복잡도, 그리고 상수항의 실제 의미

- **정의**: Big-O는 입력 크기 N에 대한 연산 횟수의 점근적 상한. O(1) < O(log N) < O(N) < O(N log N) < O(N²) < O(2^N).
- **실무 감각 수치**: 현대 CPU는 대략 초당 10^8~10^9회의 단순 연산을 처리한다. 코딩테스트에서 N=10^5면 O(N log N)까지, N=10^3~10^4면 O(N²)까지, N≤20이면 O(2^N) 비트마스크까지 허용된다는 예산 계산이 먼저다.
- **상수항이 지배하는 구간**: Big-O가 같아도(심지어 더 나빠도) 실측이 뒤집히는 이유는 **메모리 계층** 때문이다.
  - L1 캐시 ~4 cycle, L2 ~12, L3 ~40, 메인 메모리 ~100~300 cycle. 캐시 미스 한 번이 산술 연산 수십~수백 개 값이다.
  - 캐시 라인은 64바이트. 연속 메모리를 순회하면 한 번의 미스로 이웃 데이터까지 딸려오고, 하드웨어 프리페처가 다음 라인을 미리 가져온다.
- **vector vs list 순회**: 이론상 둘 다 O(N)이지만, `std::vector`는 연속 메모리라 프리페치가 먹히고, `std::list`는 노드마다 포인터 추적(pointer chasing)으로 캐시 미스가 나서 실측 5~20배 느린 게 보통이다. 심지어 "중간 삽입/삭제가 많으면 list"라는 교과서 조언도, N이 수천 이하면 vector의 memmove가 리스트의 노드 탐색+할당보다 빠른 경우가 많다 (Stroustrup의 유명한 벤치마크).
- **게임 맥락**: 매 프레임 16.6ms 안에 수천 개 엔티티를 도는 엔진에서 자료구조 선택은 곧 캐시 전략이다. Winters ECS의 `CComponentStore`(`Engine/Public/ECS/ComponentStore.h`)가 컴포넌트를 `std::vector<T> m_vecData`에 **연속으로** 두는 이유가 정확히 이것이다. Winters에서 프레임 17.8ms→9ms 복구 작업을 할 때도 알고리즘 교체보다 데이터 레이아웃/불필요한 순회 제거가 주효했다.

### 2. 배열 / 동적 배열 (std::vector)

- **정의**: 같은 타입 원소를 연속 메모리에 배치. 인덱스 접근 O(1) — `주소 = base + index * sizeof(T)` 단일 산술.
- **동적 배열의 성장**: capacity 초과 시 더 큰 버퍼(보통 1.5~2배)를 할당하고 전체를 이동한다. 개별 push_back은 최악 O(N)이지만, 기하급수 성장 덕에 **분할상환(amortized) O(1)** — N번 push의 총 이동 횟수가 N + N/2 + N/4 + … < 2N.
- **함정**: 재할당 시 모든 iterator/pointer/reference 무효화. 엔진 코드에서 컴포넌트 벡터를 참조로 잡아둔 채 Add를 호출하면 댕글링 — ECS에서 흔한 사고 패턴.
- **swap-and-pop**: 순서가 중요 없을 때 중간 삭제를 O(1)로: 마지막 원소를 삭제 위치로 옮기고 pop_back. Winters `CComponentStore::Remove`가 정확히 이 패턴이다 (`m_vecDense[index] = lastEntity; m_vecData[index] = std::move(m_vecData[last]); ... pop_back()`).

### 3. 연결 리스트

- **정의**: 노드가 데이터 + 다음(및 이전) 노드 포인터를 가진 구조. 위치를 알고 있을 때 삽입/삭제 O(1), 탐색 O(N), 임의 접근 불가.
- **단일 vs 이중**: 이중 리스트는 prev 포인터로 임의 노드를 O(1) 삭제 가능 — LRU 캐시의 핵심 부품.
- **실무 위치**: 범용 컨테이너로는 캐시 때문에 거의 항상 vector에 진다. 살아남는 자리는 (1) **intrusive list** — 객체 안에 링크를 심어 할당 없이 연결 (커널, 엔진의 free list), (2) **LRU** 같은 O(1) 임의 삭제가 필수인 조합 구조, (3) 노드 주소 안정성이 필요한 경우.
- **게임 맥락**: 오브젝트 풀의 free list, 렌더 패스 체인 등. "리스트 쓰겠다"고 답하기 전에 항상 "vector + swap-and-pop으로 안 되나?"를 먼저 검토하는 태도를 보여주는 게 좋다.

### 4. 해시 테이블

- **정의**: 키를 해시 함수로 정수화하여 버킷 인덱스로 매핑. 평균 O(1) 삽입/탐색/삭제, 최악 O(N)(모든 키가 충돌).
- **충돌 해결 두 계열**:
  - **체이닝(separate chaining)**: 버킷마다 리스트/벡터를 매단다. 구현 단순, 부하율 1 초과 허용, 삭제 쉬움. 단점: 노드 할당 + 포인터 추적으로 캐시 비친화. `std::unordered_map`이 이 방식(표준이 버킷 인터페이스와 iterator 안정성을 요구해서 사실상 체이닝 강제).
  - **오픈 어드레싱(open addressing)**: 충돌 시 같은 배열의 다른 슬롯을 탐사(probing). 선형 탐사(캐시 최고, 군집화 clustering 문제), 이차 탐사, 로빈 후드, 그리고 SIMD로 메타데이터 16개를 한 번에 비교하는 swiss table(abseil `flat_hash_map`) 계열. 삭제 시 tombstone 필요. 게임/고성능 코드에서 선호.
- **부하율(load factor)**: `원소 수 / 버킷 수`. 체이닝은 1.0 근처, 오픈 어드레싱은 0.7~0.875를 넘기면 성능이 급락 → **리해싱**: 버킷 수를 늘리고(보통 2배) 전 원소를 재삽입. 리해싱은 O(N)이며, 그 순간 iterator가 무효화된다(unordered_map은 rehash 시 iterator 무효, reference는 유지 — 체이닝이라 노드가 안 움직이므로).
- **좋은 해시 함수**: 균등 분포 + 빠름. 나쁜 해시는 이론적 O(1)을 O(N)으로 붕괴시킨다 — 서버에서는 악의적 키로 충돌을 유도하는 hash-flooding DoS까지 이어져서, 런타임 랜덤 시드를 섞는 방어가 있다.
- **게임 맥락**: Winters `CSpatialIndex`(`Engine/Public/ECS/SpatialIndex.h`)는 `std::unordered_map<i64_t, std::vector<CellEntry>>`로 (cx,cz) 셀 좌표를 64비트 키로 합쳐 점유 셀만 저장한다 — 희소한 월드에서 밀집 2D 배열 대신 해시를 쓰는 전형적 선택. 반면 매 프레임 hot loop에서는 unordered_map의 노드 할당이 부담이라, ECS 본체는 해시가 아니라 sparse set(아래 §17)을 쓴다.

### 5. 트리 — BST, 균형 트리, B/B+tree

- **BST**: 왼쪽 서브트리 < 노드 < 오른쪽 서브트리. 평균 O(log N) 탐색/삽입/삭제, but 정렬된 입력이 들어오면 한쪽으로 치우쳐 O(N) 퇴화.
- **균형 트리**: 회전(rotation)으로 높이를 O(log N)으로 유지.
  - **AVL**: 좌우 높이차 ≤ 1로 엄격 균형. 탐색이 가장 빠르지만 삽입/삭제 시 회전이 잦다.
  - **레드-블랙 트리**: "루트~리프 검은 노드 수 동일 + 빨간 노드 연속 금지" 규칙으로 최장 경로 ≤ 최단 경로 × 2 라는 느슨한 균형. 삽입/삭제 시 회전이 평균 O(1)회로 적어 갱신 많은 워크로드에 유리. `std::map/set`의 사실상 표준 구현.
  - **비교 포인트**: 읽기 위주 → AVL, 쓰기 섞임 → RB. 단 둘 다 노드 기반이라 캐시에 약해서, "정렬 유지가 필요 없다면 해시, 벌크 조회라면 정렬된 vector + 이진 탐색"이 실무 우선순위.
- **B-tree / B+tree**: 노드 하나가 수백 개의 키를 갖는 다진 탐색 트리. 트리 높이를 3~4로 낮춰 **디스크/페이지 단위 I/O 횟수를 최소화**하는 게 목적. B+tree는 실제 데이터(레코드)를 리프에만 두고 리프들을 연결 리스트로 이어 **범위 스캔이 순차 I/O**가 된다. DB 인덱스(MySQL InnoDB), 파일시스템의 표준. 면접 포인트: "메모리 최적 구조(RB)와 디스크 최적 구조(B+)의 차이 = 접근 단위(캐시 라인 64B vs 페이지 4~16KB)에 맞춘 노드 크기".
- **게임 맥락**: 게임 서버의 랭킹/영속 데이터는 결국 DB의 B+tree 인덱스 위에 놓인다. 클라이언트 쪽에서는 트리는 주로 공간 분할(쿼드트리/BVH, §15)로 등장한다.

### 6. 힙 / 우선순위 큐

- **정의**: 완전 이진 트리이며 부모 ≥(max-heap) 또는 ≤(min-heap) 자식인 구조. 배열로 표현: 노드 i의 자식은 2i+1, 2i+2 — 포인터 없이 산술만으로 이동, 캐시 친화적.
- **연산**: top O(1), push O(log N)(sift-up), pop O(log N)(루트에 마지막 원소 놓고 sift-down). N개 배열을 힙으로 만드는 `make_heap`은 O(N)(리프 절반은 sift 불필요라는 관찰).
- **decrease-key 부재**: `std::priority_queue`는 내부 원소 갱신 API가 없다. 다익스트라/A\*에서 표준 해법은 **lazy deletion** — 더 좋은 경로가 나오면 그냥 새로 push하고, pop한 원소가 이미 closed면 버린다. Winters A\*가 정확히 이 방식이다: `Engine/Private/Manager/Navigation/Pathfinder.cpp`의 `FindPathInternal`에서 pop 직후 `if (tls_closed[curIdx]) continue;`로 stale 노드를 걸러낸다.
- **게임 맥락**: A\*/다익스트라 open list, 타이머/스킬 쿨다운 이벤트 스케줄러, top-K 위협 대상 선정. "가장 급한 것 하나만 자주 꺼낸다"면 정렬(O(N log N) 전체)이 아니라 힙이다.

### 7. 그래프 표현

- **인접 행렬**: N×N bool/가중치. 간선 존재 확인 O(1), 공간 O(N²). 밀집 그래프나 N이 작을 때(플로이드-워셜).
- **인접 리스트**: 정점마다 이웃 목록. 공간 O(V+E), 순회 O(degree). 희소 그래프의 기본값.
- **암시적 그래프(implicit graph)**: 게임에서 가장 흔한 형태. 그리드 맵은 간선을 저장하지 않고 "이웃 오프셋 배열"로 즉석 생성한다. Winters A\*도 간선 자료구조가 없다 — `constexpr int32_t kDX[8]/kDY[8]` 8방향 오프셋과 `CanStepToNeighbor` 술어(walkable 검사)가 그래프 그 자체다 (`Pathfinder.cpp` 상단).
- **정점 번호화**: 2D 셀 → 1D 인덱스 flatten (`CellIdx = cy * kCellCountX + cx`, `Pathfinder.cpp`). 방문 배열을 vector 하나로 두는 관용구.

### 8. 정렬 — 비교와 std::sort의 내부

- **비교 정렬 하한**: 비교 기반은 결정 트리 리프가 N!개 필요 → 최소 O(N log N).
- **주요 알고리즘 비교**:

  | 알고리즘 | 평균 | 최악 | 공간 | 안정성 | 비고 |
  |---|---|---|---|---|---|
  | 퀵 정렬 | N log N | **N²** | log N(스택) | X | 파티션, 캐시 우수 |
  | 병합 정렬 | N log N | N log N | **N** | O | 연결리스트/외부정렬 |
  | 힙 정렬 | N log N | N log N | 1 | X | 상수항 큼(캐시 나쁨) |
  | 삽입 정렬 | N² | N² | 1 | O | **N 작을 때 최강**, 거의 정렬된 입력 O(N) |
  | 기수/카운팅 | N·k / N+k | — | N+k | O | 비교 안 함, 정수 키 한정 |

- **std::sort = introsort**: 퀵 정렬로 시작 → 재귀 깊이가 2·log₂N을 넘으면(악성 피벗 패턴 감지) **힙 정렬로 전환**해 최악 O(N log N) 보장 → 구간이 작아지면(대략 16~32 미만) **삽입 정렬**로 마무리. "평균은 퀵, 최악은 힙, 작은 구간은 삽입"의 3단 하이브리드. `std::stable_sort`는 병합 정렬 계열(메모리 부족 시 in-place 변형, O(N log²N)).
- **게임 맥락**: 렌더 큐 정렬이 대표적 — 머티리얼/셰이더/깊이를 묶은 64비트 sort key를 만들어 draw call을 배칭한다. 반투명은 뒤→앞 깊이 정렬이 강제. 프레임마다 "거의 정렬된" 상태가 유지되므로 삽입 정렬 성질(적응성)이나 이전 프레임 순서 재활용이 실효 최적화가 된다.

### 9. 이진 탐색

- **정의**: 정렬된 배열에서 중간값과 비교해 탐색 범위를 절반씩 줄임. O(log N).
- **표준 관용구**: `lower_bound`(x 이상 첫 위치) / `upper_bound`(x 초과 첫 위치) — 직접 짤 때는 `[lo, hi)` 반열림 불변식을 유지하는 게 버그를 없앤다.
- **고전 버그**: `mid = (lo + hi) / 2`의 오버플로 → `mid = lo + (hi - lo) / 2`. 그리고 `<= vs <`, `mid ± 1` 경계 — 불변식("답은 항상 [lo, hi) 안에 있다")을 먼저 적고 코드를 맞춘다.
- **파라메트릭 서치**: "조건을 만족하는 최소/최대 값" — 답의 공간이 단조(monotonic)라면 답을 이진 탐색. 예: "이 사거리면 타격 가능한 최소 스탯". 코딩테스트 단골.
- **게임 맥락**: 애니메이션 키프레임 탐색(시간 t가 속한 구간 찾기), 정렬된 에셋 ID 테이블 조회, 확률 테이블(누적합 + upper_bound)의 가챠 추첨.

### 10. BFS / DFS

- **BFS**: 큐. 레벨(거리) 순서로 방문 → **무가중치 그래프 최단 경로**. O(V+E).
- **DFS**: 스택/재귀. 경로 존재성, 사이클 검출, 위상 정렬, 연결 요소, 백트래킹의 뼈대. O(V+E). 재귀 깊이 폭발 주의(그리드 10^6 셀이면 명시적 스택으로).
- **멀티 소스 BFS**: 시작점 여러 개를 큐에 동시에 넣으면 "가장 가까운 소스까지의 거리 필드"가 한 번에 나온다. Winters `Pathfinder.cpp`의 `BuildReachabilityCache`가 실전 예: 모든 walkable 셀을 depth 0으로 넣고 BFS를 돌려, **unwalkable 셀마다 '가장 가까운 walkable 셀'(`nearestCell`)과 거리(`nearestDistance`)를 사전 계산**한다. 벽 클릭 시 가장 가까운 이동 가능 지점을 O(1)로 찾기 위한 테이블이다.
- **flood fill = 연결 요소 라벨링**: 같은 함수에서 walkable 셀들을 BFS로 훑어 `components[idx]`에 연결 요소 번호를 매긴다. "출발지와 목적지가 같은 요소인가"를 O(1)로 판정해, 도달 불가능한 목표에 A\*를 헛돌리는(전 맵 탐색 후 실패) 최악 케이스를 차단한다.
- **방문 배열 초기화 트릭**: 매 탐색마다 O(전체 셀) clear 대신 **generation counter** — `visited[idx] == currentGeneration`이어야 방문으로 친다. 탐색 시작 시 generation만 +1. Winters `BeginSearchGeneration`(`Pathfinder.cpp`)이 이 패턴이고, uint32 wrap-around 시에만 전체 fill로 리셋한다. 면접에서 "매 프레임 수십 번 경로 탐색하는데 초기화 비용은?"의 모범답.

### 11. 다익스트라

- **정의**: 음수 간선이 없는 가중 그래프의 단일 출발 최단 경로. 우선순위 큐로 "현재 최단 거리 확정 후보"를 꺼내며 이웃을 relax. O((V+E) log V).
- **탐욕이 성립하는 이유**: 간선 가중치 ≥ 0이면, 큐에서 꺼낸 최소 거리 정점은 더 이상 짧아질 수 없다(다른 경로는 이미 그보다 긴 정점을 거친다). **음수 간선이 있으면 이 확정 논리가 깨진다** → 벨만-포드 O(VE)로.
- **A\*와의 관계**: 다익스트라 = A\*에서 h(n)=0인 특수 케이스. 목표가 하나뿐이면 휴리스틱으로 탐색을 목표 방향으로 조여주는 A\*가 항상 같거나 적은 노드를 방문한다.
- **게임 맥락**: 목표가 여러 개이거나 없는 경우(가장 가까운 아무 안전지대), 영향 맵(influence map) 생성 등에는 h를 만들 수 없어 다익스트라/BFS가 맞다.

### 12. A\* — f = g + h, 허용성, 그리고 Winters 구현

- **정의**: f(n) = g(n)(시작→n 실제 비용) + h(n)(n→목표 추정 비용)이 최소인 노드부터 확장하는 최적 우선 탐색.
- **허용성(admissibility)**: h가 실제 잔여 비용을 **절대 과대평가하지 않으면** A\*는 최적해를 보장한다. h=0이면 다익스트라(느리지만 최적), h가 실제와 같으면(perfect heuristic) 최단 경로만 곧장 확장.
- **일관성(consistency/monotonicity)**: `h(n) ≤ cost(n, n') + h(n')` (삼각 부등식). 일관되면 각 노드를 한 번 close하는 것으로 충분 — closed 노드 재개방이 필요 없어진다. 그리드 + octile 조합은 일관적이다.
- **휴리스틱 선택 (그리드)**: 4방향 = 맨해튼 `|dx|+|dy|`, 8방향 = **octile** `max(|dx|,|dy|) + (√2−1)·min(|dx|,|dy|)`, 자유 각도 = 유클리드. 이동 규칙과 안 맞는 휴리스틱(8방향에 맨해튼)은 과대평가 → 최적성 상실.
- **Winters 실제 구현** (`Engine/Private/Manager/Navigation/Pathfinder.cpp`):
  - 휴리스틱: `OctileDistance(dx, dy)` — `max(ax,ay) + kSqrt2m1 * min(ax,ay)`, `kSqrt2m1 = √2 − 1`. 8방향 이동(직선 1, 대각 √2 비용)과 정확히 일치하는 허용적·일관적 휴리스틱.
  - **open list**: `std::priority_queue<Node, std::vector<Node>, std::greater<Node>>` — `Node{cx, cy, fG, fF}`에 `operator>`가 `fF`를 비교하는 **min-heap**. decrease-key 없이 **중복 push + closed 체크(lazy deletion)**로 처리.
  - 탐색 상태: `tls_gScore / tls_parent / tls_closed / tls_generation` — 전부 `thread_local` 벡터 + generation counter로 O(1) 리셋. 스레드마다 독립이라 여러 스레드가 동시에 경로 탐색해도 안전.
  - **corner cutting 방지**: 대각 이동은 양쪽 직교 셀이 모두 walkable일 때만 허용 (`CanStepToNeighbor`의 `if (dx != 0 && dy != 0)` 분기). 유닛이 벽 모서리를 뚫고 비비는 버그의 정석 해법.
  - **에이전트 반경 대응**: `FindPathForRadius`는 walkable 술어를 `IsCellWalkableForRadius`로 바꿔 같은 A\* 본체를 재사용 — 알고리즘과 통과 규칙을 함수 포인터(`CellPredicate/StepPredicate`)로 분리했다.
  - **A\* 이전의 단락 최적화**: `TryBuildDirectCellPath`가 시작→목표 선분이 walkable하면 A\*를 아예 생략하고 2점 경로를 반환("AStar::DirectBypass" 프로파일 카운터). 열린 평지에서 대부분의 클릭이 여기서 끝난다.
  - **도달 불가 목표 처리**: `TryFindNearestReachableGoal` — 연결 요소 캐시(§10)로 같은 요소인지 확인하고, 아니면 사전 계산된 nearest map 또는 반경 제한 BFS로 가장 가까운 도달 가능 셀로 목표를 치환한 뒤 탐색.
  - 실패를 침묵시키지 않음: `ePathFindResult`(StartBlocked/GoalBlocked/NoRoute/BrokenPath)로 원인을 반환 — "path empty silent fail"로 미니언이 제자리에 서던 사고 이후 넣은 분류다.
- **게임 맥락 심화**: 계층적 경로 탐색(HPA\*), navmesh 위의 A\* + funnel 알고리즘, 가중치 A\*(h에 ε>1 곱해 속도↑/최적성 포기), JPS(균일 그리드 전용 대칭 경로 가지치기)까지 이어 말할 수 있으면 좋다.

### 13. 위상 정렬

- **정의**: DAG(유향 비순환 그래프)의 정점을 "모든 간선이 앞→뒤"가 되도록 나열. 사이클이 있으면 불가능(이것으로 사이클 검출).
- **두 구현**: Kahn — 진입차수 0인 정점을 큐에 넣고 제거하며 반복, O(V+E), 남은 정점이 있으면 사이클. DFS — 종료(finish) 순서의 역순.
- **게임 맥락**: 에셋 의존성 로딩 순서, 스킬/버프 의존 그래프, 빌드 시스템, 그리고 **ECS 시스템 스케줄링**. Winters에서 시스템들을 Phase로 나눠 실행하는데(SystemScheduler), NavigationSystem과 AISystem의 Phase 순서가 뒤집혀 stale velocity가 한 프레임 남는 사고를 겪었다 — "Phase 배열은 곧 데이터 의존성의 수동 위상 정렬"이라는 교훈. 시스템의 read/write 컴포넌트 집합으로 의존 그래프를 만들어 자동 위상 정렬하는 것이 Unity DOTS류의 접근이다.

### 14. 유니온 파인드 (Disjoint Set Union)

- **정의**: 서로소 집합들의 합치기(union)와 소속 찾기(find)를 지원. parent 배열 기반.
- **두 최적화**: **경로 압축**(find 중 만난 노드를 루트에 직접 연결) + **union by rank/size**(작은 트리를 큰 트리 밑에). 둘 다 적용 시 연산당 분할상환 O(α(N)) — 역아커만 함수, 사실상 상수.
- **용도**: 동적 연결성(간선이 추가만 될 때), 크루스칼 MST, 그리드에서 같은 영역 판정, 사이클 검출.
- **게임 맥락**: 맵의 연결 판정이 대표 용도인데, Winters는 간선이 정적(맵 로드 시 확정)이라 유니온 파인드 대신 **BFS flood fill 한 번으로 요소 라벨을 굽는** 방식을 택했다 (`BuildReachabilityCache`의 `components`). 면접 답변 포인트: "간선이 동적으로 추가되면 유니온 파인드, 정적이면 flood fill 사전 계산 — 파괴 가능 지형처럼 간선이 **삭제**되면 둘 다 곤란해서 재계산 또는 오프라인 재라벨링" — 자료구조 선택을 갱신 패턴으로 정당화하는 논리.

### 15. 공간 분할 — 균일 그리드 / 쿼드트리 / BVH

충돌·시야·근접 판정의 O(N²) (N개 유닛 쌍별 검사)을 깨는 세 가지 계열.

- **균일 그리드(uniform grid)**: 월드를 고정 크기 셀로 나누고 각 오브젝트를 자기 셀에 등록. 반경 질의는 겹치는 셀만 순회. 갱신 O(1), 구현 최소. 약점: 크기 편차가 큰 오브젝트, 밀도 편중(한 셀에 몰림), 빈 공간 낭비(해시 셀로 완화).
  - **Winters 실전**: `Engine/Public/ECS/SpatialIndex.h`의 `CSpatialIndex` — `cellSize 8`, `unordered_map<i64_t, vector<CellEntry>>` 해시 그리드, `QueryRadius`(kind/team 마스크 필터)와 `QueryClosest` 제공, 매 프레임 `Rebuild`. LoL류 게임은 유닛 크기가 비슷하고 분포가 레인에 몰려도 셀 수십 개 수준이라 그리드가 최적이다. 초기 MinionAI의 `FindClosestEnemy`가 O(N²) brute force였고, 이를 대체하는 구조가 이것.
- **쿼드트리/옥트리**: 공간을 재귀적으로 4/8분할, 밀한 곳만 깊게. 편중된 분포에 강함. 갱신(이동 시 재삽입)이 그리드보다 비쌈. loose quadtree(셀 경계를 겹치게)로 경계 걸침 문제 완화.
- **BVH(Bounding Volume Hierarchy)**: 공간이 아니라 **오브젝트를** AABB로 묶는 트리. 오브젝트가 트리 어디든 정확히 한 곳에만 속함(공간 분할의 중복 등록 없음). 레이캐스트/정밀 충돌의 표준 — 물리 엔진 broad-phase, 레이트레이싱 가속 구조. 정적 메시에 최적, 동적은 refit/재구축 비용.
- **선택 기준 요약**: 비슷한 크기 + 동적 다수 + 반경 질의 → 그리드. 분포 편중 + 크기 다양 → 쿼드트리/옥트리. 레이 vs 삼각형/정밀 형상 → BVH. kd-tree는 정적 점군 최근접 질의.
- **시야(FoW) 판정**: 그리드 셀 단위 visibility 마스크 + 시야 반경 질의 → 셀 단위로 계산하고 유닛은 자기 셀 값을 읽는 게 상수항을 지배적으로 줄인다. 부시(concealment) 같은 volume 판정도 인덱스로 분리 (Winters `ECS/ConcealmentVolumeIndex.h`가 그 자리).

### 16. LRU 캐시

- **정의**: 용량 초과 시 "가장 오래 안 쓰인(Least Recently Used)" 항목을 축출하는 캐시.
- **O(1) 설계 (면접 단골 구현 문제)**: **해시맵 + 이중 연결 리스트** 조합.
  - `unordered_map<Key, list<pair<Key,Value>>::iterator>` — 키로 노드 위치 O(1) 조회.
  - get: 맵에서 iterator 찾고 `list.splice(list.begin(), list, it)`로 맨 앞 이동(O(1), 노드 재할당 없음).
  - put: 존재하면 갱신+앞으로, 없으면 앞에 삽입, 용량 초과면 `list.back()`을 맵과 리스트 양쪽에서 제거.
  - 왜 이 조합인가: 해시는 임의 삭제 위치를 못 기억하고, 리스트는 검색이 O(N) — 서로의 약점을 상쇄. **이중** 리스트여야 중간 노드 O(1) 제거 가능.
- **게임 맥락**: 텍스처/사운드 스트리밍 캐시, 서버의 세션·프로필 캐시. 실무에서는 엄밀 LRU 대신 근사(CLOCK, 랜덤 2-choice)로 잠금 경합을 줄이기도 한다 — "멀티스레드에서 LRU 리스트 갱신이 병목이 되면?"이라는 꼬리질문의 답.

### 17. ECS 엔티티 조회 — sparse set

- **구조** (`Engine/Public/ECS/ComponentStore.h`): 배열 3개.
  - `m_vecSparse[entityId] = denseIdx` — 엔티티 ID로 O(1) 위치 조회 (없으면 INVALID).
  - `m_vecDense[denseIdx] = entityId` — 역방향.
  - `m_vecData[denseIdx] = T` — **컴포넌트 실 데이터가 빈틈없이 연속**.
- **연산**: Has/Get/Add O(1), Remove는 swap-and-pop O(1)(마지막 원소를 구멍으로 이동, sparse 두 곳 갱신). 순회는 dense 배열 선형 스캔 — 시스템이 매 프레임 도는 경로가 순수 연속 메모리.
- **vs unordered_map<EntityID, T>**: 해시도 O(1)이지만 버킷/노드 간접 참조와 순회 시 캐시 미스가 있다. sparse set은 조회가 인덱싱 두 번, 순회가 프리페치 친화 — "이론 복잡도 동률에서 상수항으로 승부하는" 대표 사례로 §1과 연결해 답한다.
- **비용**: sparse 배열이 최대 엔티티 ID만큼 커짐(메모리 ↔ 속도 트레이드), 순서 불안정(swap-remove), 다중 컴포넌트 조인은 가장 작은 store를 기준으로 나머지를 Has 검사.

### 18. 문자열 — KMP (개념 수준)

- **문제**: 텍스트 T(길이 N)에서 패턴 P(길이 M) 찾기. 소박한 방법은 O(NM).
- **KMP 핵심**: 패턴의 **실패 함수(failure function)** `fail[i]` = P[0..i]의 "접두사이면서 접미사인 최장 길이"를 O(M)에 전처리. 불일치가 나면 텍스트 포인터를 되돌리지 않고 패턴 포인터만 `fail`을 따라 점프 → 전체 O(N+M).
- **직관**: "지금까지 맞은 부분의 끝부분이 패턴의 앞부분과 겹친다면, 그 겹침을 재활용해 비교를 이어간다."
- **주변 지식 한 줄씩**: 라빈-카프 = 롤링 해시로 O(N+M) 기대(해시 충돌 검증 필요, 2D/다중 패턴에 유연). 트라이 = 다중 패턴 접두사 사전. 아호-코라식 = 트라이 + 실패 링크로 다중 패턴 동시 탐색 — 채팅 금칙어 필터의 표준(게임 서버 실무 연결 포인트).

### 19. Lock-free 자료구조와 Chase-Lev work-stealing deque

- **lock-free 정의**: 잠금 없이 원자 연산(CAS 등)으로 동시성을 처리하며, **어떤 스레드가 멈춰도 시스템 전체는 전진이 보장**되는 성질. (모든 스레드의 전진 보장은 더 강한 wait-free.)
- **CAS(compare-and-swap)**: "메모리가 기대값이면 새 값으로 교체"를 원자적으로. 실패하면 재시도 루프. **ABA 문제**: 값이 A→B→A로 돌아오면 CAS는 변화를 못 본다 — 태그(버전) 카운터나 epoch/hazard pointer로 방어.
- **memory ordering**: `relaxed`(원자성만) / `acquire·release`(이 쌍으로 선행 쓰기의 가시성 동기화) / `seq_cst`(전역 단일 순서). 약한 순서를 쓰는 이유는 x86보다 ARM 등 약한 메모리 모델에서 불필요한 배리어 비용을 빼기 위함이고, 필요한 곳에만 fence를 박는 게 설계의 전부다.
- **false sharing**: 서로 다른 스레드가 쓰는 변수들이 같은 64B 캐시 라인에 있으면, 논리적 공유가 없어도 라인 소유권이 코어 간 핑퐁(cache line ping-pong)하며 성능 급락. 해법은 `alignas(64)` 패딩.
- **Chase-Lev deque (2005)** — work-stealing 스케줄러의 표준 부품. Winters 구현: `Engine/Public/Core/JobSystem/WorkStealingDeque.h`의 `CWorkStealingDeque<T>`.
  - **소유권 규약이 알고리즘의 본체**: **owner만 `Push`/`Pop`(bottom 쪽, LIFO)**, **thief는 `Steal`(top 쪽, FIFO)만**. owner는 bottom에서 최근 작업을 꺼내(뜨거운 캐시 재사용) thief는 top에서 오래된 작업을 가져가(경합 최소, 큰 단위 작업 분산).
  - 구현 디테일: 고정 4096 슬롯 링버퍼(2의 거듭제곱 `static_assert` — 모듈로를 `& (kCapacity-1)`로), `alignas(64) atomic<int64> m_iBottom / m_iTop` 분리(false sharing 방지).
  - 동기화 요점: `Push`는 버퍼 쓰기 후 release fence → bottom 증가(thief가 bottom을 읽었다면 원소도 보이도록). `Pop`은 bottom 감소 후 **seq_cst fence** → top 읽기(마지막 1개를 두고 Steal과 경합할 때 서로의 감소/증가를 반드시 관측해야 하므로), 마지막 원소는 top에 대한 CAS로 승부를 가린다. `Steal`은 top CAS 실패 시 그냥 false(다른 thief가 이김).
  - **내가 겪은 실제 race**: Phase 5-A에서 `CJobSystem::PushToSomeDeque`가 **Main 스레드에서 Worker의 deque에 직접 Push** — owner-only 규약 위반. 두 스레드가 bottom을 동시에 조작하며 작업이 유실됐고, `WaitForCounter`의 카운터가 영원히 0이 안 되어 **첫 프레임 메인 스레드 hang**으로 나타났다 (엔티티 루트 36개가 병렬 임계값 16을 넘어 병렬 경로에 진입할 때만 재현 — "특정 규모에서만 터지는" 전형적 동시성 버그).
  - **수정** (`Engine/Private/Core/JobSystem.cpp`의 `EnqueueJob`): 호출자가 Worker 자신이면(`t_iWorkerIdx >= 0`) 자기 deque에 Push(규약 준수), **Main/외부 스레드이거나 deque가 가득 차면 mutex 보호 global `std::queue`로 우회**하는 하이브리드. Worker의 소비 순서는 자기 deque Pop → global queue → 랜덤 victim Steal(`PickVictim` — thread_local xorshift PRNG, 자기 자신 제외). `PushToSomeDeque`는 현재 `EnqueueJob` 래퍼로만 남아 있다.
  - 교훈으로 정리해 말할 것: "lock-free 구조는 코드가 아니라 **규약**이 자료구조다. 규약을 문서화하고, 위반을 컴파일/assert로 막을 수 없다면 API 경계에서 라우팅으로 막아야 한다."
- **보조 구조들**: 유휴 Worker는 스핀 대신 `condition_variable::wait_for(1ms)`로 잠들고 Push 시 `notify_one`(스핀의 CPU 소모 vs 웨이크 지연 트레이드). `WaitForCounter`는 블로킹 대신 **help-stealing** — 기다리는 스레드도 큐에서 일을 꺼내 처리한다(데드락 회피 + 처리량).

### 20. 코딩테스트 접근 전략 (문제 유형 분류)

1. **입력 크기로 복잡도 예산 산출** (§1의 수치): N ≤ 20 → 완전탐색/비트마스크/백트래킹, N ≤ 500 → O(N³) 플로이드류, N ≤ 5,000 → O(N²) DP, N ≤ 10^5~10^6 → O(N log N) 정렬/힙/이진탐색/투포인터, 그 이상 → O(N) 해시/누적합/수학.
2. **키워드 → 유형 매핑**: "최단/최소 횟수" + 무가중치 → BFS, 가중치 → 다익스트라 / "모든 경우" → 백트래킹 / "최댓값의 최솟값" → 파라메트릭 서치 / "구간 합/갱신" → 누적합·세그먼트 트리 / "연결/그룹" → 유니온 파인드·flood fill / "순서 제약" → 위상 정렬 / "겹치는 부분 문제 + 최적 부분 구조" → DP / "괄호/최근 것 먼저" → 스택.
3. **풀이 절차**: 문제를 그래프/배열 모델로 번역 → 소박한 풀이 먼저 명시("O(N²)면 이렇게 되는데 N=10^5라 안 됩니다"라고 말로 깔고) → 병목 연산을 자료구조로 교체 → 경계(빈 입력, 1개, 중복, 오버플로 — 합계는 int64) 검증.
4. **면접식 라이브 코딩에서는** 침묵 금지: 가정을 말하고, 반례를 스스로 들고, 복잡도를 마지막에 재확인. 이는 CLAUDE.md에 박아둔 "가정을 명시하고 확인한다"는 평소 작업 규율과 같은 근육이다.

## 예상 질문 & 모범답변

### Q. 배열과 연결 리스트의 차이를 설명하고, 실무에서 언제 무엇을 쓰는지 말해보세요.

- **정의/원리**: 배열은 연속 메모리 — 임의 접근 O(1), 중간 삽입/삭제 O(N). 연결 리스트는 노드+포인터 — 위치를 알면 삽입/삭제 O(1), 접근/탐색 O(N).
- **왜/트레이드오프**: 표만 보면 "삽입 많으면 리스트"지만 실측은 다르다. 리스트는 노드마다 힙 할당 + 포인터 추적으로 캐시 미스가 나서, 순회 실측이 vector 대비 수 배~수십 배 느리다. 삽입 위치를 **찾는** 비용 O(N)이 붙는 순간 리스트의 O(1) 삽입은 의미가 없어진다. 그래서 기본값은 항상 vector이고, 리스트는 (1) O(1) 임의 삭제가 구조적으로 필요한 조합(LRU), (2) intrusive list(할당 없음), (3) 노드 주소 안정성이 필요할 때만 쓴다.
- **내 프로젝트**: Winters ECS는 컴포넌트를 `std::vector`에 연속 배치하고(`Engine/Public/ECS/ComponentStore.h`), 순서가 필요 없으니 삭제는 swap-and-pop O(1)로 처리해 "배열의 약점(중간 삭제)"을 설계로 제거했다. 매 프레임 수천 엔티티를 도는 시스템 루프가 전부 연속 메모리 스캔이다.
- **함정/꼬리 대비**: "vector 중간 삭제도 O(N)인데요?"가 오면 swap-and-pop, 그리고 순서가 필요하면 삭제 마킹 후 일괄 compact를 답한다. "재할당 시 무효화"도 언급하면 좋다.

### Q. std::vector의 push_back이 O(1)이라는 말은 정확한가요?

- **정의/원리**: 분할상환 O(1)이다. capacity가 차면 보통 1.5~2배로 재할당 + 전체 이동 O(N)이 일어나지만, 기하 성장 덕에 N번 push의 총 복사가 2N 미만 — 평균 O(1).
- **왜/트레이드오프**: 성장 배수가 크면 메모리 낭비, 작으면 재할당 빈발. 2배가 아닌 1.5배를 쓰는 구현(MSVC)은 해제된 이전 블록들을 재사용할 수 있게 하려는 의도다.
- **내 프로젝트**: 엔진에서는 프레임 중 재할당 자체가 스파이크라서, 로딩 시 `reserve`로 상한을 잡는다. Winters JobSystem도 `Initialize`에서 `m_vecDeques.reserve(iWorkerCount)` 후 채운다 (`Engine/Private/Core/JobSystem.cpp`).
- **함정/꼬리**: 재할당 시 iterator/포인터/참조 전부 무효화. "move 생성자가 noexcept가 아니면 재할당 때 copy로 폴백된다"까지 말하면 가산점.

### Q. 해시 테이블의 충돌 해결 방법 두 가지를 비교해 주세요.

- **정의/원리**: 체이닝 — 버킷에 리스트를 매닮. 오픈 어드레싱 — 같은 배열의 다른 슬롯을 탐사(선형/이차/로빈후드).
- **왜/트레이드오프**: 체이닝은 구현 단순 + 삭제 쉬움 + 부하율 1 초과 허용이지만 노드 할당과 포인터 추적으로 캐시 비친화. 오픈 어드레싱은 배열 하나라 캐시 친화적이지만 부하율에 민감(0.7~0.875 상한), 삭제에 tombstone이 필요하고 선형 탐사는 군집화가 생긴다. `std::unordered_map`은 표준의 iterator/reference 안정성 요구 때문에 체이닝이고, 고성능이 필요하면 swiss table류(flat_hash_map)를 쓴다.
- **내 프로젝트**: hot path(매 프레임 컴포넌트 조회)는 해시 대신 sparse set으로 아예 해시를 제거했고, 콜드한 희소 데이터(점유된 그리드 셀)는 `unordered_map<i64_t, vector<CellEntry>>`로 관리한다 (`Engine/Public/ECS/SpatialIndex.h`). "해시냐 아니냐"보다 "접근 빈도와 키 밀도에 따라 구조를 가른다"가 내 기준이다.
- **함정/꼬리**: "최악 O(N)은 언제?" → 나쁜 해시/악의적 키. 서버라면 hash-flooding DoS와 랜덤 시드 방어까지.

### Q. 부하율(load factor)과 리해싱을 설명해 주세요. 리해싱 중에는 무슨 일이 벌어지나요?

- **정의/원리**: 부하율 = 원소 수/버킷 수. `unordered_map`은 `max_load_factor`(기본 1.0) 초과 시 버킷을 늘리고(대략 2배 이상의 소수/2^k) **모든 원소를 재해싱해 재배치** — O(N) 일시 정지.
- **왜/트레이드오프**: 부하율이 높으면 충돌 증가로 조회가 느려지고, 낮으면 메모리 낭비. 리해싱 순간의 스파이크가 문제라면 `reserve(n)`으로 선확보하거나, 점진적 리해싱(두 테이블 병행, Redis 방식)을 쓴다.
- **내 프로젝트**: 게임 루프에서 O(N) 스파이크는 곧 프레임 드랍이라, 크기를 아는 맵은 로딩 타임에 reserve하는 걸 규칙으로 삼는다.
- **함정/꼬리**: "리해싱 후 iterator는?" — 무효. 단 체이닝 특성상 원소 **reference/pointer는 유지**된다(노드가 이동하지 않으므로). 이 구분을 아는지 자주 확인한다.

### Q. BST가 O(N)으로 퇴화하는 경우와 그 해결책은?

- **정의/원리**: 정렬된(또는 거의 정렬된) 순서로 삽입하면 한쪽 사슬이 되어 높이가 N — 탐색/삽입 O(N). 해결은 자가 균형 트리: AVL(높이차 ≤ 1 엄격), 레드-블랙(최장 ≤ 최단×2 느슨).
- **왜/트레이드오프**: AVL은 더 평평해 탐색 빠름, 대신 갱신 시 회전 잦음. RB는 갱신 회전이 평균 O(1)이라 삽입/삭제 섞인 워크로드용 — `std::map/set`이 RB인 이유. 정렬 순회가 필요 없으면 해시가, 벌크 로드 후 조회만이면 정렬 vector + `lower_bound`가 더 낫다(캐시).
- **내 프로젝트**: 엔진 코드에서 `std::map`을 고를 때는 "정렬 순회 또는 범위 질의가 정말 필요한가"를 먼저 물었고, 대부분 unordered_map이나 vector로 내려갔다.
- **함정/꼬리**: "RB 트리 규칙 다섯 개 외워보세요"에는 규칙 나열보다 **불변식의 목적**(검은 높이 동일 + 빨강 연속 금지 → 높이 2배 이내 보장)으로 답하는 게 안전하다.

### Q. B-tree와 B+tree는 왜 DB와 파일시스템에서 쓰이나요?

- **정의/원리**: 노드 하나에 키 수백 개를 담는 다진 트리. 노드 크기를 디스크 페이지(4~16KB)에 맞춰 트리 높이를 3~4로 낮춘다 → 조회당 디스크 I/O 3~4회. B+tree는 데이터를 리프에만 두고 리프를 연결 리스트로 이어 범위 스캔이 순차 I/O.
- **왜/트레이드오프**: 메모리용 RB 트리는 노드당 키 1개 — 디스크에서 쓰면 높이 log₂N(수십)만큼 I/O가 난다. 접근 단위가 캐시 라인(64B)이냐 페이지(수 KB)냐가 노드 크기를 결정한다는 게 본질. 내부 노드에 키만 있으니 fan-out이 커지고, 전 데이터 순회가 리프 체인으로 끝나는 게 B+의 장점.
- **내 프로젝트**: 게임 서버의 랭킹/재화 조회가 결국 InnoDB B+tree 인덱스를 타므로, "인덱스 있는 컬럼으로 조건을 걸어야 하는 이유"를 이 구조로 설명할 수 있다.
- **함정/꼬리**: "그럼 SSD에서도 B+tree인가?" — 랜덤 읽기가 싸져도 쓰기 증폭/페이지 단위는 여전해 유효하고, 쓰기 최적화가 필요하면 LSM-tree(RocksDB)가 대안이라는 비교까지.

### Q. 힙(우선순위 큐)의 구조와 연산 복잡도를 설명해 주세요.

- **정의/원리**: 완전 이진 트리 + 힙 속성(부모 ≤ 자식, min-heap 기준). 배열로 표현해 i의 자식이 2i+1, 2i+2. push = 끝에 넣고 sift-up O(log N), pop = 루트에 마지막 원소를 놓고 sift-down O(log N), top O(1), 건설(make_heap)은 O(N).
- **왜/트레이드오프**: "최솟값 하나를 자주 꺼낸다"에 전체 정렬 O(N log N)은 과잉 — 힙이 정확한 도구. 완전 이진 트리라 배열에 빈틈없이 담겨 포인터가 없고 캐시도 무난하다. 임의 원소 탐색/갱신은 O(N)이라는 게 약점.
- **내 프로젝트**: Winters A\*의 open list가 `std::priority_queue<Node, vector<Node>, greater<Node>>` min-heap이다 (`Engine/Private/Manager/Navigation/Pathfinder.cpp`, `FindPathInternal`). f값이 최소인 노드를 O(log N)으로 꺼내며 탐색한다.
- **함정/꼬리**: "make_heap이 왜 O(N)?" — 높이 h인 노드의 sift 비용 h를 전 노드에 합치면 Σ N/2^(h+1)·h = O(N). 절반이 리프라 일이 없다는 직관으로 답한다.

### Q. std::priority_queue에는 decrease-key가 없는데, 다익스트라나 A\*는 어떻게 구현하나요?

- **정의/원리**: **lazy deletion** — 더 짧은 경로를 발견하면 기존 항목을 갱신하는 대신 새 항목을 push한다. pop했을 때 그 노드가 이미 확정(closed)이면 stale이므로 버린다.
- **왜/트레이드오프**: 큐에 중복이 쌓여 최대 O(E)개 — 메모리/log 항이 약간 늘지만, decrease-key를 지원하는 인덱스드 힙(위치 테이블 유지)이나 피보나치 힙의 구현 복잡도·상수항보다 실전에서 거의 항상 낫다.
- **내 프로젝트**: Winters `FindPathInternal`이 정확히 이 구조다 — relax 성공 시 무조건 `open.push({nx, ny, tentativeG, tentativeG + h})`, pop 직후 `if (tls_closed[curIdx]) continue;`로 stale 노드 폐기. octile 휴리스틱이 일관적이라 close 후 재개방이 필요 없다는 것까지 붙여 답한다.
- **함정/꼬리**: "stale인지 어떻게 아나?" — closed 플래그 또는 "pop한 g가 기록된 gScore보다 크면 stale" 비교. 두 방식 다 말할 수 있어야 한다.

### Q. std::sort는 내부적으로 어떤 알고리즘인가요?

- **정의/원리**: introsort — 퀵 정렬로 시작해서, 재귀 깊이가 2·log₂N을 넘으면 힙 정렬로 전환(최악 O(N log N) 보장), 구간이 작아지면(~16개) 삽입 정렬로 마무리.
- **왜/트레이드오프**: 퀵은 평균 최속(파티션이 순차 접근이라 캐시 우수)이지만 최악 O(N²)이 있고, 힙은 최악이 안전하지만 상수항이 크며(비지역적 접근), 삽입은 작은/거의 정렬된 구간에서 무적 — 셋의 약점을 서로 지우는 하이브리드다. 깊이 감시라는 "런타임 자기 진단"으로 알고리즘을 갈아타는 설계가 포인트.
- **내 프로젝트**: 렌더 큐/이벤트 정렬은 그냥 std::sort를 믿고, 대신 비교자를 64비트 sort key 정수 비교로 만들어 비교 비용 자체를 줄이는 쪽에 노력을 쓴다.
- **함정/꼬리**: "stable_sort와의 차이?" — 안정성 보장 + 병합 정렬 계열 + 추가 메모리. "정렬 없이 top-K만 필요하면?" — `nth_element`(평균 O(N), introselect) / `partial_sort`.

### Q. 퀵 정렬의 최악 케이스는 언제 발생하고 어떻게 방지하나요?

- **정의/원리**: 피벗이 매번 최소/최대가 되어 파티션이 1 : N−1로 쪼개질 때 O(N²). 고전적으로 "이미 정렬된 배열 + 첫 원소 피벗" 조합.
- **왜/트레이드오프**: 방지책 — median-of-three 피벗, 랜덤 피벗, 그리고 introsort처럼 깊이 초과 시 힙 정렬 폴백. 완전한 확률적 방어(랜덤)냐 결정적 보장(introsort)이냐의 차이.
- **내 프로젝트/실무 연결**: 외부 입력(유저 데이터)을 정렬하는 서버 코드는 최악 보장이 있는 표준 sort를 쓰고 직접 퀵을 구현하지 않는다 — 알고리즘 지식은 "직접 짜기 위해서"가 아니라 "라이브러리의 성질과 한계를 알기 위해" 필요하다는 태도로 답한다.
- **함정/꼬리**: "중복 원소가 아주 많으면?" — 3-way partition(네덜란드 국기)으로 등가 구간을 통째로 제외.

### Q. 안정 정렬(stable sort)이 필요한 실제 상황을 들어보세요.

- **정의/원리**: 같은 키의 원소들이 정렬 후에도 원래 상대 순서를 유지하는 성질. 병합/삽입/카운팅은 안정, 퀵/힙/선택은 불안정.
- **왜/트레이드오프**: 다중 키 정렬을 "부 키로 먼저, 주 키로 나중에" 두 번 정렬하는 기법이 안정성 위에서만 성립한다. 불안정 정렬로 매 프레임 재정렬하면 같은 키 원소들의 순서가 프레임마다 흔들린다.
- **내 프로젝트**: 렌더 큐에서 같은 sort key(같은 머티리얼·깊이 밴드)의 draw들이 프레임마다 순서가 바뀌면 z-fighting성 플리커가 생길 수 있다 — 안정 정렬을 쓰거나, 아예 엔티티 ID를 key 하위 비트에 넣어 전순서(total order)를 만들어 버리는 해법을 안다. 후자는 불안정 정렬로도 결정론을 얻는 트릭이라 서버 리플레이 결정론에도 통한다.
- **함정/꼬리**: "std::sort를 안정적으로 만들 수 있나?" — 비교자에 원래 인덱스를 tie-breaker로 추가하면 된다(메모리 추가 대신 키 확장).

### Q. 이진 탐색을 직접 구현할 때 자주 나는 버그는 무엇인가요?

- **정의/원리**: (1) `mid = (lo+hi)/2` 오버플로 → `lo + (hi-lo)/2`. (2) 경계 조건 — `<` vs `<=`, `hi = mid` vs `mid-1`이 뒤섞이면 무한 루프/off-by-one. (3) "같은 값 여러 개일 때 어느 것을 반환하는가" 미정의.
- **왜/트레이드오프**: 반열림 `[lo, hi)` + "답은 항상 구간 안" 불변식 하나로 통일하면 세 버그가 모두 사라진다. lower_bound 관용구: `while (lo < hi) { mid = lo + (hi-lo)/2; if (a[mid] < x) lo = mid+1; else hi = mid; }`.
- **내 프로젝트**: 애니메이션 키프레임 시간 탐색, 확률 누적합 테이블 조회처럼 "구간을 찾는" 코드는 전부 이 관용구 하나로 짠다 — 변형을 안 만드는 것이 버그를 안 만드는 방법.
- **함정/꼬리**: "정렬 안 된 배열에서 쓰면?" — 미정의 결과. "실수 이진 탐색은?" — 종료 조건을 반복 횟수(100회)로 고정하는 게 epsilon 비교보다 안전.

### Q. BFS와 DFS는 각각 언제 쓰나요?

- **정의/원리**: BFS는 큐로 거리 순 방문 — **무가중치 최단 경로**, 레벨 탐색, 최소 횟수 문제. DFS는 스택/재귀로 깊이 우선 — 경로 존재, 사이클 검출, 위상 정렬, 백트래킹, 연결 요소. 둘 다 O(V+E).
- **왜/트레이드오프**: "최단"이 붙으면 무조건 BFS(DFS는 먼저 찾은 경로가 최단이 아님). 메모리는 BFS가 최악 O(V) 프론티어, DFS는 깊이만큼 — 깊은 그래프에서 재귀 DFS는 스택 오버플로 위험이라 명시적 스택으로.
- **내 프로젝트**: Winters `Pathfinder.cpp`의 `BuildReachabilityCache`가 BFS 2연발이다 — flood fill로 연결 요소 라벨링(도달 가능성 O(1) 판정), 이어서 **멀티 소스 BFS**로 모든 unwalkable 셀에 대해 "가장 가까운 walkable 셀" 테이블을 사전 계산. 벽을 클릭해도 즉시 가장 가까운 이동 가능 지점으로 목표를 치환한다.
- **함정/꼬리**: "가중치가 있으면?" — 다익스트라. "0/1 가중치면?" — 0-1 BFS(덱, 0 간선은 앞으로). 방문 표시는 큐에 **넣을 때** 해야 중복 삽입이 안 생긴다는 디테일도 자주 찔러본다.

### Q. 다익스트라 알고리즘의 원리와, 음수 간선에서 실패하는 이유를 설명해 주세요.

- **정의/원리**: min-heap에서 "현재 알려진 최단 거리가 가장 짧은 정점"을 꺼내 확정하고 이웃을 relax. 간선 ≥ 0이면, 꺼낸 정점의 거리는 이후 어떤 경로로도 줄일 수 없다(다른 경로는 이미 더 먼 정점을 경유) — 이 탐욕적 확정이 정당성의 전부다. O((V+E) log V).
- **왜/트레이드오프**: 음수 간선은 "나중에 더 짧아질 수 있음"을 허용해 확정 논리를 깨뜨린다 → 벨만-포드 O(VE)(음수 사이클 검출 겸용). 모든 쌍이 필요하면 플로이드-워셜 O(V³).
- **내 프로젝트**: Winters A\*에서 h=0으로 두면 그대로 다익스트라다 — open list, lazy deletion, gScore relax 구조가 동일하다는 점을 코드로 설명할 수 있다 (`FindPathInternal`의 relax: `if (tentativeG < tls_gScore[nextIdx])`).
- **함정/꼬리**: "우선순위 큐 없이 하면?" — 매번 선형 스캔 O(V²), 밀집 그래프에선 오히려 그게 낫다. "간선 가중치가 전부 같으면?" — BFS로 충분.

### Q. A\*에서 f = g + h의 의미와, 휴리스틱의 허용성(admissibility)을 설명해 주세요.

- **정의/원리**: g(n) = 시작→n 확정 비용, h(n) = n→목표 추정 비용, f = 그 노드를 경유하는 경로의 총비용 추정. f 최소 노드부터 확장한다. **허용성**: h가 실제 잔여 비용을 과대평가하지 않으면(h ≤ h\*) 최적해 보장. **일관성**: h(n) ≤ c(n,n') + h(n')이면 closed 재개방 불필요.
- **왜/트레이드오프**: h=0이면 다익스트라(안전하지만 느림), h가 정확할수록 목표 방향으로만 탐색이 조여진다. h를 일부러 과대평가(가중 A\*, w·h)하면 훨씬 빨라지지만 최적성을 잃는다 — 게임에서는 "충분히 자연스러운 경로 + 빠른 응답"을 위해 의도적으로 쓰기도 한다는 트레이드오프까지 말한다.
- **내 프로젝트**: Winters는 8방향 그리드(직선 1, 대각 √2)라 **octile distance** `max(|dx|,|dy|) + (√2−1)·min(|dx|,|dy|)`을 쓴다 (`Pathfinder.cpp`의 `OctileDistance`, `kSqrt2m1 = √2−1`). 이동 규칙과 정확히 일치하므로 허용적이면서 최대한 팽팽한(tight) 휴리스틱 — 방문 노드 수를 프로파일 카운터("AStar::NodesVisited")로 계측하며 확인했다.
- **함정/꼬리**: "8방향 그리드에 맨해튼을 쓰면?" — 대각 이동을 2로 추정해 과대평가 → 최적성 붕괴. "유클리드는?" — 허용적이지만 octile보다 느슨해 노드를 더 방문. 휴리스틱은 "허용 범위 안에서 최대한 크게"가 원칙.

### Q. 실제 게임에서 A\* 구현할 때 교과서와 달라지는 부분은 무엇이었나요? (실전)

- **정의/원리**: 교과서 A\*는 "그래프 + 휴리스틱"이 끝이지만, 실제로는 (1) open list의 decrease-key 부재, (2) 방문 배열 초기화 비용, (3) 도달 불가능 목표, (4) 코너 커팅, (5) 에이전트 반경이 문제의 대부분이다.
- **내 프로젝트 (Winters `Engine/Private/Manager/Navigation/Pathfinder.cpp` 기준으로 답변)**:
  1. open list는 `std::priority_queue` + **lazy deletion**(중복 push, pop 후 closed 체크).
  2. gScore/parent/closed는 `thread_local` 벡터 + **generation counter**로 매 탐색 O(1) 리셋 — 수만 셀 맵에서 탐색마다 memset하지 않는다.
  3. **탐색 전 연결 요소 검사**: flood fill로 구운 component 라벨이 다르면 A\*를 아예 돌리지 않고, 사전 계산된 nearest map으로 목표를 가장 가까운 도달 가능 셀로 치환 — "도달 불가 목표에 전 맵을 태우는" 최악 케이스 제거.
  4. **대각 이동은 양쪽 직교 셀이 열려 있어야 허용**(corner cutting 방지) — 유닛이 벽 모서리에 끼이는 버그의 근본 처방.
  5. 시작→목표 **선분이 걸리는 것 없이 walkable하면 A\* 생략**(direct bypass) — 열린 지형에서 대부분의 이동 명령이 힙에 한 번도 안 들어가고 끝난다.
  6. 실패를 enum(`ePathFindResult`)으로 반환 — 빈 경로 silent fail로 미니언이 제자리 애니만 돌던 사고 후, "실패의 이유"를 데이터로 노출하는 걸 규칙으로 만들었다.
- **함정/꼬리**: "경로가 계단 모양인데?" — 셀 경로 후처리(string pulling / 시야 기반 waypoint 압축). "동적 장애물은?" — 지역 재탐색 또는 스티어링과 분리. 확장 주제로 JPS/HPA\*/navmesh+funnel을 예고해 두면 좋다.

### Q. 위상 정렬은 무엇이고, 게임 엔진에서 어디에 쓰이나요?

- **정의/원리**: DAG의 정점을 간선 방향이 모두 앞→뒤가 되게 나열. Kahn(진입차수 0 큐, O(V+E), 다 못 뽑으면 사이클) 또는 DFS 종료 역순.
- **왜/트레이드오프**: "의존성이 있는 작업들의 실행 순서"가 곧 위상 정렬이다. 사이클 검출이 공짜로 따라오는 것도 실무 가치(순환 의존 에셋/모듈을 로드 타임에 잡아냄).
- **내 프로젝트**: Winters의 시스템 실행 Phase가 사실상 수동 위상 정렬이다. NavigationSystem과 AISystem의 Phase 순서가 잘못돼 이전 프레임의 stale velocity로 미니언이 stuck되는 사고를 겪었고, "Phase 번호 = 데이터 흐름의 순서"라는 원칙으로 재배치해 해결했다. 시스템의 컴포넌트 read/write 집합으로 의존 그래프를 만들면 이 배치를 자동화(위상 정렬 + 독립 시스템 병렬화)할 수 있다는 발전 방향까지 말한다.
- **함정/꼬리**: "위상 정렬 결과는 유일한가?" — 아니다, 같은 레벨의 정점 순서는 자유 — 그래서 결정론이 필요한 서버 시뮬레이션에선 tie-break 규칙(ID 순)을 박아야 한다.

### Q. 유니온 파인드의 최적화 두 가지와 복잡도를 설명해 주세요.

- **정의/원리**: parent 배열로 서로소 집합 관리. **경로 압축**(find 경로의 노드를 루트 직결) + **union by rank/size**(낮은/작은 트리를 밑으로) — 두 최적화 동시 적용 시 연산당 분할상환 O(α(N)), 역아커만 함수로 사실상 상수.
- **왜/트레이드오프**: 간선이 **추가만** 되는 동적 연결성에 최적. 간선 삭제는 지원하지 않음(오프라인 역순 처리나 재계산으로 우회). 크루스칼 MST의 사이클 검사가 대표 용도.
- **내 프로젝트**: Winters 맵은 간선이 정적이라 유니온 파인드 대신 로드/개정 시 **BFS flood fill로 연결 요소를 한 번에 라벨링**했다 (`Pathfinder.cpp`의 `ReachabilityCache::components`, 그리드 revision 변경 시에만 재빌드). "갱신 패턴(정적/추가만/삭제 포함)이 자료구조를 결정한다"는 프레임으로 답하면 단순 암기와 차별화된다.
- **함정/꼬리**: "rank 없이 경로 압축만 하면?" — 분할상환 O(log N)로 여전히 실용적. 코딩테스트에선 압축만으로 충분한 경우가 많다는 실전 감각도.

### Q. 유닛 500명의 근접 판정이 O(N²)으로 느립니다. 어떻게 최적화하시겠어요?

- **정의/원리**: 쌍별 검사 12.5만 회/프레임이 문제의 본질. 공간 분할로 "가까울 가능성이 있는 후보"만 좁힌다 — broad-phase(후보 축소) / narrow-phase(정밀 검사) 분리.
- **왜/트레이드오프**: 유닛 크기가 비슷하고 이동이 잦으면 **균일 그리드**(갱신 O(1), 질의는 겹치는 셀만). 분포 편중·크기 다양이면 쿼드트리. 레이캐스트/정밀 형상은 BVH. 셀 크기는 "평균 질의 반경과 비슷하게" — 너무 작으면 질의가 다중 셀 순회, 너무 크면 셀 내 brute force로 회귀.
- **내 프로젝트**: Winters MinionAI의 `FindClosestEnemy`가 초기엔 O(N²) brute force였다. 이를 `CSpatialIndex`(`Engine/Public/ECS/SpatialIndex.h`) — cellSize 8의 해시 그리드(`unordered_map<i64, vector<CellEntry>>`), `QueryRadius`(kind/team 마스크)와 `QueryClosest` — 로 대체하는 구조를 갖췄다. 매 프레임 `Rebuild`하는 단순화(증분 갱신 대신)를 택했는데, N이 수백이면 전체 재구축 O(N)이 증분 관리의 복잡도보다 싸다는 판단이었다 — 이 판단 근거까지 말하는 게 포인트.
- **함정/꼬리**: "한 셀에 유닛이 몰리면?" — 그리드의 근본 약점, 셀 내부는 여전히 선형이므로 `maxEntitiesInCell` 같은 통계(Winters `DebugStats`에 실제로 있음)로 감시하고 필요 시 계층화. "질의 반경이 셀보다 크면?" — 겹치는 셀 범위를 계산해 다중 셀 순회.

### Q. 쿼드트리와 BVH의 차이를 설명해 주세요.

- **정의/원리**: 쿼드트리는 **공간을** 재귀적으로 4분할하고 오브젝트를 위치한 셀에 넣는다(공간 분할, space partitioning). BVH는 **오브젝트들을** AABB로 묶어 트리를 만든다(객체 계층, object hierarchy).
- **왜/트레이드오프**: 공간 분할은 경계에 걸친 오브젝트가 여러 노드에 중복 등록되거나 상위 노드에 얹히는 문제가 있고(loose quadtree로 완화), BVH는 오브젝트가 정확히 한 노드에만 속하는 대신 형제 노드의 볼륨이 겹칠 수 있어 탐색 시 양쪽을 다 내려가야 할 수 있다. 동적 이동은 그리드 > 쿼드트리 > BVH 순으로 싸고, 레이캐스트 품질은 그 역순.
- **내 프로젝트**: 동적 유닛(미니언/챔피언)은 균일 그리드, 정적 지형/메시 피킹은 BVH가 맞는 배치라는 기준으로 설계를 갈랐다. 마우스 피킹은 현재 실린더 근사 히트 판정으로 시작해, 정밀화가 필요해지는 시점에 메시 BVH를 넣는 단계적 계획이다.
- **함정/꼬리**: "BSP는?" — 임의 평면 분할, 고전 FPS 맵/포탈 렌더링 계열이며 요즘은 BVH가 대세라는 역사적 맥락 한 줄.

### Q. LRU 캐시를 O(1)으로 설계해 보세요.

- **정의/원리**: `unordered_map<Key, list::iterator>` + `std::list<pair<Key,Value>>`. get — 맵에서 iterator를 찾아 `list.splice(begin(), list, it)`로 맨 앞 이동 후 값 반환. put — 있으면 갱신+앞으로, 없으면 push_front + 맵 등록, 용량 초과면 back()을 리스트/맵 양쪽에서 제거. 모든 연산 O(1).
- **왜/트레이드오프**: 해시(임의 키 O(1) 접근)와 이중 연결 리스트(임의 노드 O(1) 제거 + 순서 유지)가 서로의 약점을 지운다. splice는 노드를 재할당 없이 이어붙여서 iterator가 살아 있다는 게 핵심 디테일. 단일 리스트면 prev를 몰라 O(1) 제거가 안 된다.
- **내 프로젝트/실무 연결**: 텍스처 스트리밍·서버 세션 캐시가 이 구조의 실제 자리. 멀티스레드에선 리스트 갱신이 직렬화 병목이 되므로 샤딩(키 해시로 캐시 N개 분할)이나 근사 LRU(CLOCK)로 완화한다고 이어 말한다.
- **함정/꼬리**: "LFU와의 차이?" — 빈도 기반, O(1) LFU는 빈도 버킷 리스트로 구현. "TTL이 섞이면?" — 만료는 lazy 삭제(접근 시 검사) + 주기적 sweep.

### Q. KMP 알고리즘의 핵심 아이디어를 설명해 주세요. (개념 수준)

- **정의/원리**: 패턴 P의 각 위치에 대해 "여기까지의 접두사이면서 접미사인 최장 길이" = 실패 함수를 O(M) 전처리. 텍스트와 비교 중 불일치가 나면 **텍스트 포인터는 절대 뒤로 가지 않고**, 패턴 포인터만 fail 값으로 점프해 이미 맞은 부분의 자기 겹침을 재활용한다. 전체 O(N+M).
- **왜/트레이드오프**: 소박한 O(NM)의 낭비는 "이미 확인한 텍스트를 다시 본다"는 것 — KMP는 그 정보를 패턴 쪽 상태로 보존한다. 오토마톤 관점(현재 상태 = 매칭된 접두사 길이)으로 이해하면 아호-코라식(다중 패턴)으로 자연스럽게 확장된다.
- **내 프로젝트/실무 연결**: 게임 서버 채팅 금칙어 필터가 다중 패턴 매칭의 실전 자리 — 단어별 KMP N회가 아니라 아호-코라식 한 방이 정답이라는 설계 판단까지 말한다.
- **함정/꼬리**: "fail[i] 계산도 KMP 자신과 같은 원리"(자기 자신에 대한 매칭)라는 점, 라빈-카프(롤링 해시, 기대 O(N+M), 충돌 검증 필요)와의 비교.

### Q. lock-free 자료구조란 무엇이고, 왜/언제 쓰나요?

- **정의/원리**: 뮤텍스 없이 CAS 등 원자 연산으로 동시 접근을 조율하며, 어떤 스레드가 지연/중단되어도 시스템 전체는 전진이 보장되는 구조. 뮤텍스의 문제(우선순위 역전, 컨텍스트 스위치, 홀더가 죽으면 전체 정지)를 피한다.
- **왜/트레이드오프**: 빨라서가 아니라 **꼬리 지연(tail latency)과 전진 보장** 때문에 쓴다 — 무경합 구간에선 뮤텍스도 충분히 빠르다. 대가는 설계 난이도: ABA 문제(버전 태그/hazard pointer로 방어), 메모리 재사용 시점 문제, memory ordering 추론. "정말 필요한 지점(스케줄러 큐, 프레임당 수만 회 경합)에만 제한적으로"가 실무 원칙.
- **내 프로젝트**: Winters JobSystem의 per-worker 큐가 lock-free Chase-Lev deque(`Engine/Public/Core/JobSystem/WorkStealingDeque.h`)이고, **제출 경로는 mutex 큐를 병용**하는 하이브리드다(`Engine/Private/Core/JobSystem.cpp`) — "hot path만 lock-free, cold path는 단순한 잠금"이라는 절충을 직접 설계했다.
- **함정/꼬리**: "lock-free면 항상 더 빠른가?" — 아니오, CAS 재시도 폭풍/캐시 라인 핑퐁으로 고경합에서 뮤텍스보다 느릴 수 있다. wait-free와의 구분(개별 스레드 전진 보장)도 준비.

### Q. Chase-Lev work-stealing deque의 구조를 설명하고, 직접 겪은 문제가 있다면 말해보세요.

- **정의/원리**: worker마다 자기 deque를 소유한다. **owner는 bottom 쪽에서 Push/Pop(LIFO)** — 방금 만든 작업을 이어서 실행해 캐시 지역성 극대화. **thief는 top 쪽에서 Steal(FIFO)** — 가장 오래된(대개 가장 큰 단위) 작업을 가져가고 owner와의 경합을 인덱스 반대편으로 격리. owner의 Push/Pop은 거의 원자 연산 하나, 경합은 "마지막 1개 남았을 때"의 top CAS로만 발생.
- **내 구현** (`Engine/Public/Core/JobSystem/WorkStealingDeque.h`): 4096 슬롯 고정 링버퍼(2^k `static_assert`, 모듈로를 `& (kCapacity-1)`로), `alignas(64)`로 top/bottom을 다른 캐시 라인에 분리(false sharing 방지). Pop은 bottom 감소 후 **seq_cst fence**를 치고 top을 읽어 마지막 원소 경합을 감지하고, 그 경우 top CAS로 Steal과 승부를 가른다.
- **직접 겪은 race (핵심 어필)**: Phase 5-A에서 Main 스레드가 `PushToSomeDeque`로 **Worker의 deque에 직접 Push했다 — owner-only 규약 위반**. bottom을 두 스레드가 조작하며 작업이 유실, JobCounter가 영원히 목표에 도달하지 못해 `WaitForCounter`가 무한 루프 → InGame 첫 프레임 메인 스레드 hang. 재현 조건조차 "엔티티 루트 36개가 병렬 임계값 16을 넘을 때만"이라 전형적인 규모 의존 동시성 버그였다.
- **수정**: `EnqueueJob`(`Engine/Private/Core/JobSystem.cpp`)에서 호출자를 라우팅 — worker 자신이면 자기 deque Push(규약 준수), main/외부 스레드거나 deque 만석이면 **mutex 보호 global queue**로. 소비는 자기 deque Pop → global → 랜덤 victim Steal(xorshift `PickVictim`) 순.
- **교훈 (마무리 멘트)**: "lock-free 구조의 정합성은 코드가 아니라 **호출 규약**에 있다. 규약을 타입/API 경계로 강제할 수 없으면, 제출 지점에서 스레드 식별(`thread_local` worker index)로 라우팅해 규약 위반이 물리적으로 불가능하게 만들어야 한다."
- **함정/꼬리**: "덱이 가득 차면?" — Winters는 Push가 false를 반환하고 global queue로 폴백(원 논문은 동적 성장 circular array). "왜 Steal은 CAS 실패 시 재시도 안 하나?" — 다른 thief가 이겼다는 뜻이므로 다른 victim을 고르는 게 낫다.

### Q. false sharing이 무엇이고 어떻게 방지하나요?

- **정의/원리**: 서로 다른 스레드가 쓰는 **서로 다른 변수**가 우연히 같은 64B 캐시 라인에 있으면, 캐시 일관성 프로토콜(MESI)이 라인 소유권을 코어 간에 계속 넘기며(핑퐁) 논리적 공유가 전혀 없는데도 성능이 수 배 떨어지는 현상.
- **왜/트레이드오프**: 방지는 `alignas(64)` / 패딩으로 라인 분리, 또는 스레드별 데이터를 아예 구조적으로 분리(per-thread 슬롯) 후 마지막에 병합. 대가는 메모리(변수당 최대 64B).
- **내 프로젝트**: 두 곳에서 실제 적용 — (1) `CWorkStealingDeque`의 `alignas(64) m_iBottom / m_iTop`(owner와 thief가 각각 두들기는 변수 분리), (2) Profiler 카운터에서 worker들이 같은 슬롯을 친 race를 main=0/worker=idx+1 슬롯 분리로 해결한 경험(worker-safety 작업). 증상이 "락도 없고 로직도 맞는데 코어를 늘릴수록 느려짐"으로 나타난다는 진단 포인트까지.
- **함정/꼬리**: "std::hardware_destructive_interference_size를 아나?" — C++17의 이식성 있는 라인 크기 상수. "진짜 공유(true sharing)와의 구분?" — 같은 변수 경합은 알고리즘 문제, false sharing은 레이아웃 문제.

### Q. 원자 연산의 memory ordering을 왜 신경 써야 하나요? relaxed/acquire-release/seq_cst 차이는?

- **정의/원리**: 컴파일러와 CPU는 단일 스레드 관점에서 결과가 같으면 메모리 접근을 재배열한다. `relaxed`는 원자성만 보장(순서 없음), `release` 쓰기 ↔ `acquire` 읽기가 짝이 되면 release 이전의 모든 쓰기가 acquire 이후에 보인다(happens-before), `seq_cst`는 모든 seq_cst 연산에 전역 단일 순서까지 부여.
- **왜/트레이드오프**: x86은 원래 강한 순서라 acquire/release가 거의 공짜지만 seq_cst 저장은 전체 배리어 비용이 든다. ARM 등 약한 모델에선 차이가 더 크다. "필요한 곳에만 강한 순서"가 성능과 정확성의 균형.
- **내 프로젝트**: `CWorkStealingDeque`가 교과서적 사례 — Push는 버퍼 쓰기 후 **release fence** + bottom relaxed 저장(thief에게 원소 가시성 보장), Pop은 bottom 감소와 top 읽기 사이에 **seq_cst fence**(마지막 1개를 두고 owner의 감소와 thief의 CAS가 서로를 반드시 관측해야 하는 지점 — acquire/release만으론 양방향 관측이 보장 안 됨), Steal의 top CAS는 seq_cst. "어디가 왜 seq_cst인가"를 라인 단위로 설명할 수 있는 게 이 질문의 승부처다.
- **함정/꼬리**: "volatile로 되지 않나?" — C++에서 volatile은 동시성 도구가 아니다(원자성/순서 보장 없음). "fence와 연산 ordering의 차이?" — fence는 앞뒤 모든 접근에 적용되는 독립 배리어.

### Q. ECS에서 엔티티로 컴포넌트를 조회하는 구조를 설명해 주세요. 왜 해시맵을 안 쓰나요?

- **정의/원리**: Winters는 **sparse set** (`Engine/Public/ECS/ComponentStore.h`) — `m_vecSparse[entityId] → denseIdx`, `m_vecDense[denseIdx] → entityId`, `m_vecData[denseIdx] → 컴포넌트`. Has/Get/Add O(1), Remove는 swap-and-pop O(1), 순회는 dense 배열 선형 스캔.
- **왜/트레이드오프**: unordered_map도 O(1)이지만 해시 계산 + 버킷 간접 참조 + 순회 시 노드 점프가 있다. sparse set은 조회가 배열 인덱싱 두 번, **순회가 완전 연속 메모리**라 시스템 루프(매 프레임 전 컴포넌트 스캔)에서 프리페처가 최대로 먹힌다. 비용은 sparse 배열이 최대 엔티티 ID 크기만큼 커지는 메모리, 그리고 swap-remove로 인한 순서 불안정.
- **내 프로젝트**: 이 구조 위에 TransformSystem 등이 매 프레임 도는 게 Winters의 기본 실행 모델이고, 프레임 최적화(17.8ms→9ms) 때도 "순회가 이미 연속 메모리"라는 전제가 병목 분석의 출발점이었다.
- **함정/꼬리**: "두 컴포넌트를 모두 가진 엔티티 순회는?" — 작은 store의 dense를 돌며 큰 store에 Has 질의. "archetype 방식과의 비교?" — archetype(같은 조합끼리 테이블 통합)은 다중 조인 순회가 더 빠르고 구조 변경(컴포넌트 추가/제거)이 비쌈, sparse set은 그 반대 — 갱신 패턴에 따른 선택이라고 답한다.

### Q. 게임 서버에서 스킬 쿨다운/버프 만료 같은 타이머 수천 개를 관리한다면?

- **정의/원리**: (1) min-heap — 만료 시각 기준 O(log N) 삽입, 매 틱 top만 확인해 만료분 pop. (2) timing wheel — 시간축을 원형 버킷으로 양자화, 삽입/삭제 O(1), 커널/네트워크 스택 방식. (3) N이 작으면 그냥 배열 스캔.
- **왜/트레이드오프**: 힙은 정확한 순서 + 단순, 취소(임의 삭제)가 O(N)이라 lazy 취소(플래그 마킹 후 pop 시 폐기 — A\* lazy deletion과 동일 패턴)로 보완. wheel은 O(1)이지만 해상도가 버킷 폭으로 양자화되고 장기 타이머에 계층(hierarchical wheel)이 필요.
- **내 프로젝트**: Winters 챔피언 시뮬레이션은 프레임 카운트 기반(캐스트/회복 프레임)이라 "매 틱 감소 검사"로 충분한 규모지만, 서버에서 수만 엔티티로 늘면 힙 스케줄러로 옮기는 기준(틱당 활성 타이머 수 × 검사 비용 > log N 삽입 비용)을 갖고 있다.
- **함정/꼬리**: "취소가 빈번하면?" — lazy 취소로 힙 오염이 심해지면 주기적 재건설, 또는 인덱스드 힙. "부하 스파이크(같은 틱 만료 몰림)는?" — 만료 처리 개수 상한 + 이월.

### Q. 대규모 데이터에서 상위 K개만 필요할 때 어떻게 하나요?

- **정의/원리**: (1) 크기 K의 min-heap 유지 — 스트림 O(N log K), 메모리 O(K). (2) `nth_element` — 평균 O(N) 파티션 기반, 전체가 메모리에 있을 때. (3) `partial_sort` — O(N log K) + 상위 K 정렬 상태.
- **왜/트레이드오프**: 스트리밍(전체를 못 담음)이면 힙이 유일, 배치면 nth_element가 최속(단 상위 K 내부 순서는 미보장). "전체 정렬 O(N log N)은 과잉"임을 먼저 지적하는 게 답의 골자.
- **내 프로젝트/실무 연결**: 랭킹 보드 상위 100, AI의 위협도 상위 K 타겟 선정, 프로파일러에서 가장 느린 상위 N 구간 추출 같은 자리에 그대로 쓰인다.
- **함정/꼬리**: "왜 max-heap이 아니라 min-heap K개?" — 힙의 top이 '현재 상위 K 중 최약체'라서 새 원소와 O(1) 비교로 진입 판정이 되기 때문. 방향을 헷갈리게 하는 단골 함정.

### Q. 재귀 DFS로 백만 셀 그리드를 flood fill하면 무슨 일이 생기나요?

- **정의/원리**: 최악 경로 깊이가 셀 수에 비례 — 스택 프레임 백만 개로 스택 오버플로(기본 스택 1~8MB). 해결: 명시적 스택(vector) 또는 BFS 큐로 변환.
- **왜/트레이드오프**: 재귀는 가독성이 좋지만 깊이가 입력에 비례하는 문제에선 금지 패턴. 꼬리 재귀 최적화는 C++에서 보장되지 않는다.
- **내 프로젝트**: Winters `BuildReachabilityCache`의 flood fill이 `std::queue` 기반 반복 BFS인 이유가 이것 — 수만 셀 navgrid에서 재귀는 선택지가 아니었다 (`Pathfinder.cpp`).
- **함정/꼬리**: "방문 표시 시점은?" — push할 때(pop 때 하면 같은 셀이 큐에 중복으로 쌓여 메모리 폭발). Winters `FindNearestCellInComponent`의 `Push` 람다도 generation 마킹을 push 시점에 한다.

### Q. 두 포인터/슬라이딩 윈도우는 어떤 문제에서 쓰나요? (코딩테스트)

- **정의/원리**: 정렬된 배열이나 "구간의 단조성"이 있을 때, 구간 [l, r]의 양 끝 포인터를 각각 단방향으로만 이동시켜 O(N²) 구간 탐색을 O(N)으로 줄인다. 예: 합이 S인 부분 연속 수열, 중복 없는 최장 부분 문자열.
- **왜/트레이드오프**: 성립 조건은 단조성 — "r을 늘리면 조건이 한 방향으로만 변한다"(합 증가 등). 음수가 섞인 합 문제는 단조성이 깨져 누적합+해시로 전환해야 한다 — 이 성립 조건 판별이 실력 척도.
- **실전 연결**: 시계열 프레임 데이터에서 "최근 W 프레임의 이동 평균/최댓값" 같은 계산이 슬라이딩 윈도우(+단조 덱) 그 자체다. 네트워크 코드의 수신 윈도우 처리도 같은 사고방식.
- **함정/꼬리**: "윈도우 내 최댓값을 O(1)에?" — 단조 감소 덱(monotonic deque) 유지, 전체 O(N).

### Q. 동적 계획법 문제인지 어떻게 판별하고 어떻게 접근하나요? (코딩테스트)

- **정의/원리**: 두 신호 — **최적 부분 구조**(큰 문제의 최적해가 작은 문제의 최적해로 조립됨) + **겹치는 부분 문제**(같은 하위 문제가 반복 등장). 접근 절차: 상태 정의(무엇을 알면 미래가 과거와 독립인가) → 점화식 → 기저 조건 → 계산 순서(top-down 메모이제이션 / bottom-up 테이블) → 복잡도 = 상태 수 × 전이 비용.
- **왜/트레이드오프**: top-down은 점화식을 그대로 옮겨 실수가 적고 필요한 상태만 계산, bottom-up은 재귀 오버헤드가 없고 공간 최적화(이전 행만 유지)가 쉽다.
- **실전 연결**: 게임에서는 데미지 콤보 최적화, 스킬 트리 최적 배분, MCTS/AI의 가치 평가 캐싱이 같은 구조다. "상태 정의가 곧 문제 풀이"라는 원칙은 리플레이/스냅샷 설계(무엇이 시뮬레이션의 완전한 상태인가)와 같은 사고방식이라 서버 개발과도 잇는다.
- **함정/꼬리**: "그리디와의 구분은?" — 그리디는 지역 최적 선택이 전역 최적을 보장하는 교환 논증이 성립할 때만. 증명 없는 그리디가 최다 오답 원인.

### Q. 해시맵의 키로 float 좌표를 쓰고 싶습니다. 문제 없나요?

- **정의/원리**: 부동소수점은 연산 경로에 따라 비트가 달라질 수 있어(0.1+0.2 ≠ 0.3, -0.0 vs 0.0, NaN ≠ NaN) 동등 비교 기반인 해시 키로 부적합하다. 정석은 **양자화** — 셀 크기로 나눠 정수 좌표로 변환 후 키 합성.
- **왜/트레이드오프**: 양자화 폭이 곧 공간 해상도 — 너무 크면 다른 위치가 같은 키, 너무 작으면 셀 수 폭발. 경계 근처 점이 이웃 셀로 갈라지는 문제는 질의 시 인접 셀 포함으로 해결.
- **내 프로젝트**: Winters `CSpatialIndex`가 정확히 이 패턴 — `CellX/CellZ`로 월드 float 좌표를 셀 정수로 양자화하고 `MakeKey`로 64비트 키 하나에 패킹해 `unordered_map<i64_t, ...>`의 키로 쓴다 (`Engine/Public/ECS/SpatialIndex.h`).
- **함정/꼬리**: "두 int를 하나의 키로 합칠 때 주의점?" — 단순 XOR은 (a,b)/(b,a) 충돌 — 시프트 결합(`(i64)cx << 32 | (u32)cz`)이나 좋은 믹싱 필요. 음수 좌표의 캐스팅 함정도 언급.

### Q. 정렬된 두 배열을 하나로 합치는 최적 방법과, 그 아이디어의 확장을 말해보세요.

- **정의/원리**: 투 포인터 병합 O(N+M) — 병합 정렬의 merge 단계. K개 배열이면 min-heap으로 K-way merge O(N log K).
- **왜/트레이드오프**: "정렬을 이미 갖고 있다면 재정렬 O((N+M) log(N+M))은 낭비"라는 감각의 문제. K-way merge는 외부 정렬(메모리보다 큰 데이터), LSM-tree 컴팩션, 로그 파일 시간순 병합의 뼈대다.
- **실전 연결**: 서버 여러 대의 시간순 로그 통합 분석, 리플레이 이벤트 스트림 병합 등 "이미 정렬된 스트림들"은 실무에 널려 있다.
- **함정/꼬리**: "in-place 병합은?" — O(1) 공간 병합은 어렵고 실용 구현은 회전 기반으로 복잡 — "안 하는 게 정답"인 경우가 많다는 실무 감각.

### Q. 코딩테스트에서 문제를 받으면 어떤 순서로 접근하나요?

- **절차**: (1) 입력 크기·제한 확인 → 복잡도 예산 산출(N=10^5면 O(N log N)까지, 10^8 연산/초 기준). (2) 문제를 아는 모델로 번역 — "최단/최소 횟수→BFS·다익스트라", "연결/그룹→유니온파인드·flood fill", "최댓값의 최솟값→파라메트릭", "순서 제약→위상 정렬", "구간→누적합·투포인터·힙". (3) 소박한 풀이를 먼저 말로 세우고 병목 연산을 자료구조로 교체. (4) 경계 — 빈 입력, 1개, 전부 동일, int 오버플로(합은 long long), 그래프면 자기 루프/중복 간선. (5) 제출 전 복잡도 재검산.
- **왜**: 유형 분류가 안 된 문제는 대부분 "두 유형의 합성"이므로, 제약 조건에서 역산하는 게(이 N이면 출제자가 의도한 복잡도는 무엇인가) 가장 빠른 분류다.
- **내 프로젝트 연결**: 이 사고 절차는 엔진 최적화 절차와 동형이다 — Winters에서도 "계측(프로파일 카운터) → 병목 특정 → 자료구조/알고리즘 교체 → 재계측"(A\* NodesVisited 카운터, DirectBypass 카운터)으로 일했다. 문제 풀이 근육과 실무 최적화 근육이 같다는 걸 어필.
- **함정/꼬리**: 라이브 코딩에선 침묵이 최악 — 가정을 소리 내어 확인하고, 반례를 스스로 제시하고, 다 짠 뒤 복잡도와 경계를 재점검하는 모습 자체가 평가 항목이다.

## 내 프로젝트 연결 포인트

면접에서 "실제로 해본 사람"임을 증명하는 문장들. 각 문장은 코드 위치까지 즉답 가능해야 한다.

1. **A\*를 교과서가 아니라 제품 수준으로 구현했다** — "8방향 그리드에 octile 휴리스틱(`max + (√2−1)·min`), `std::priority_queue` min-heap open list에 lazy deletion, thread_local 상태 배열 + generation counter로 탐색당 O(1) 리셋, 대각 코너 커팅 방지, 에이전트 반경별 walkable 술어 분리까지 넣었습니다." (`Engine/Private/Manager/Navigation/Pathfinder.cpp`)
2. **탐색을 안 하는 최적화가 가장 큰 최적화였다** — "시작→목표 선분이 깨끗하면 A\*를 생략하는 direct bypass, flood fill로 구운 연결 요소 라벨로 도달 불가 목표를 O(1) 거절, 멀티 소스 BFS로 '벽 클릭 → 가장 가까운 걸을 수 있는 셀' 테이블 사전 계산 — 힙에 들어가기 전에 끝나는 경로가 대부분입니다."
3. **lock-free 자료구조의 규약 위반 race를 직접 겪고 고쳤다** — "Chase-Lev deque는 owner만 Push/Pop, thief는 Steal이라는 규약이 자료구조의 본체인데, Main 스레드가 worker deque에 Push하는 규약 위반으로 첫 프레임 hang을 겪었습니다. 스레드 식별 기반 라우팅(worker는 자기 deque, 외부는 mutex global queue)으로 규약 위반이 불가능한 API로 재설계했습니다." (`Engine/Public/Core/JobSystem/WorkStealingDeque.h`, `Engine/Private/Core/JobSystem.cpp`)
4. **캐시 관점의 상수항을 설계 기준으로 쓴다** — "ECS 컴포넌트 저장을 해시맵이 아니라 sparse set으로 한 것, deque의 top/bottom을 alignas(64)로 분리한 것, 프레임 17.8ms→9ms 최적화 모두 'Big-O가 아니라 메모리 레이아웃'이 병목이었던 경험입니다." (`Engine/Public/ECS/ComponentStore.h`)
5. **O(N²)를 공간 분할로 깬 경험** — "미니언 AI의 최근접 적 탐색이 O(N²) brute force로 시작했고, cellSize 8의 해시 그리드 SpatialIndex(QueryRadius/QueryClosest, kind/team 마스크)로 대체하는 구조를 설계했습니다. 유닛 크기가 균일한 LoL류에는 쿼드트리보다 균일 그리드가 맞다는 선택 근거까지 설명할 수 있습니다." (`Engine/Public/ECS/SpatialIndex.h`)
6. **알고리즘 실패를 데이터로 노출하는 습관** — "경로 탐색 empty 반환 silent fail로 미니언이 제자리에 서던 사고 이후, `ePathFindResult` enum과 프로파일 카운터(NodesVisited, DirectBypass, NearestMapHit)를 붙여 '왜 실패했는지'를 계측 가능하게 만들었습니다 — 알고리즘 지식과 운영 가능성(observability)을 함께 본다는 어필."
7. **시스템 실행 순서 = 위상 정렬이라는 감각** — "Navigation과 AI 시스템의 Phase 순서가 뒤집혀 stale 데이터 한 프레임 지연으로 유닛이 stuck된 사고를 겪고, 'Phase 배열은 데이터 의존성의 수동 위상 정렬'이라는 원칙을 세웠습니다."
8. **서버 지향 어필** — "다익스트라 = A\* with h=0, 힙 lazy deletion = 타이머 lazy 취소, flood fill vs 유니온 파인드 = 정적 vs 증분 갱신 — 자료구조 선택을 항상 갱신 패턴과 접근 패턴으로 정당화하는 훈련이 되어 있습니다."

## 마지막 점검 체크리스트

- [ ] Big-O 감각: 10^8 연산/초, N=10^5 → O(N log N) 예산. 캐시 라인 64B, 메모리 ~100+ cycle.
- [ ] vector vs list: 이론 동률, 실측은 캐시로 vector 압승. 리스트가 사는 곳 = LRU/intrusive.
- [ ] vector push_back = 분할상환 O(1) (기하 성장, 총 이동 < 2N). 재할당 시 iterator 전멸.
- [ ] 해시: 체이닝(unordered_map, ref 안정) vs 오픈 어드레싱(캐시, tombstone). 부하율 → 리해싱 O(N).
- [ ] BST 퇴화 O(N) → AVL(엄격/읽기) vs RB(느슨/쓰기, std::map). B+tree = 페이지 단위 I/O + 리프 체인 범위 스캔.
- [ ] 힙: 배열 2i+1/2i+2, push/pop O(log N), make_heap O(N). decrease-key 없음 → lazy deletion.
- [ ] std::sort = introsort: 퀵 → (깊이 2logN 초과) 힙 → (작은 구간) 삽입. stable_sort = 병합.
- [ ] 이진 탐색: `lo + (hi-lo)/2`, 반열림 `[lo, hi)` 불변식. 파라메트릭 = 단조 조건의 답 공간 탐색.
- [ ] BFS = 무가중 최단(방문 표시는 push 시점) / DFS = 존재성·사이클·위상. 깊은 그래프 재귀 금지.
- [ ] 다익스트라: 음수 간선 불가(확정 논리 붕괴) → 벨만-포드. h=0인 A\*.
- [ ] A\*: f=g+h, 허용성(과대평가 금지)=최적성, 일관성=재개방 불필요. 8방향 = octile `max+(√2−1)min`.
- [ ] Winters A\* 즉답: priority_queue+greater min-heap, lazy deletion(closed 체크), generation 리셋, 코너 커팅 방지, direct bypass, 연결 요소 사전 계산. — `Pathfinder.cpp`
- [ ] 위상 정렬: Kahn 진입차수 큐 / DFS 역순. 사이클 검출 공짜. 시스템 Phase = 수동 위상 정렬.
- [ ] 유니온 파인드: 경로 압축 + rank = O(α(N)). 간선 정적이면 flood fill 사전 계산이 대안.
- [ ] 공간 분할: 균일 유닛·동적 → 그리드 / 편중·다양 → 쿼드트리 / 레이·정밀 → BVH. Winters = cellSize 8 해시 그리드.
- [ ] LRU: 해시맵 + 이중 리스트 + splice = 전 연산 O(1). 왜 '이중'인지(임의 노드 O(1) 제거).
- [ ] KMP: 실패 함수 = 접두사=접미사 최장, 텍스트 포인터 후퇴 없음, O(N+M). 다중 패턴 → 아호-코라식.
- [ ] lock-free: CAS + 전진 보장. ABA → 버전 태그. false sharing → alignas(64). volatile은 동시성 도구 아님.
- [ ] Chase-Lev: owner Push/Pop(bottom, LIFO) / thief Steal(top, FIFO), 마지막 1개는 top CAS 승부. **Main-push 규약 위반 race → hang → global queue 라우팅 수정** 스토리 1분 버전 준비.
- [ ] sparse set: sparse[id]→dense, dense 연속 순회, swap-and-pop. 해시 대비 승점 = 캐시.
- [ ] 코딩테스트: 제약에서 복잡도 역산 → 키워드 유형 매핑 → 소박한 풀이 먼저 → 병목만 교체 → 경계(빈/1개/오버플로) → 복잡도 재검산. 침묵 금지.
