# 3. NYPC — 정답 없는 문제를 AI와 함께: 게임 봇 대회

[이미지: 대회 게임 화면 또는 self-play 승률 그래프]

| | |
|---|---|
| 대회 | NYPC [연도·부문·최종 성적 정식 표기 CONFIRM] — 게임 2종 (턴제 보드 / 경제 RTS) |
| 기술 | C++, Python, alpha-beta 탐색, self-play 인프라, 리플레이 분석 |
| 링크 | [GitHub — NYPC_Mushroom](https://github.com/tnestyle70/NYPC_Mushroom) |


게임 봇 대회는 "정답이 없는 문제"입니다 — 컴파일 성공이 아니라 상대를 이겨야 하고, 뭐가 좋은 수인지 아무도 알려주지 않습니다. 저는 봇을 잘 만드는 것보다 **봇이 좋아졌는지 판정하는 장치**(자기 자신과 수천 판 대전시키는 self-play 인프라)를 먼저 만들었고, 튜닝 값을 감이 아니라 대전 승률로 골랐습니다. 34일의 대회 운영을 기록으로 남겨 실패 원인까지 해부했습니다.

### 구현

- **버섯게임 (턴제 보드)**: Search Order(먼저 볼 후보 선별) + alpha-beta pruning + 2단 평가함수(수 평가 = ImmediateSwing+DenyGain / 리프 평가 = ScoreDiff+BestAll)를 설계하고, **self-play 인프라(Python)를 먼저 구축해 변수·가중치 튜닝을 주관이 아니라 대전 승률로 선별** — 성능 정체(플래토)가 오면 가중치를 더 만지는 대신 평가 구조 자체를 의심하는 법을 배움
- **NEXT NATION (경제 RTS)**: 3층 봇 구조 — GameState(사실) → Perception(사실의 가공: 이전 턴 상대 행동 분석, 이동 예상 경로와 ETA 산출) → Behavior(Attack/Move/Upgrade/Rush/Banking을 수학적 손익 계산으로 선택). self-play trace로 "봇이 왜 그 선택을 했는가"를 역추적 가능하게 설계 — 목표는 강한 AI가 아니라 **디버깅 가능한 AI**
- **정직한 회고**: 초반 이후 "이기기 위한 전략"보다 확실하게 닫히는 개선에 시간을 쓴 것이 정체의 원인 — 34일 행동 기록을 해부해 원인을 특정했고, 이후 모든 프로젝트의 시간 배분 규칙으로 반영

### ▶ 더 깊이 — 트레이드오프와 뒷이야기 (토글)

- **왜 평가 인프라가 봇보다 먼저인가**: 평가 장치가 없으면 모든 개선이 "느낌상 좋아짐"이 된다. 리그 평가(신버전 vs 구버전 N판)는 개선을 가설→실험→판정 루프로 바꾼다 — 이 사고방식이 이후 Winters의 결정성 해시·프로파일링 원장으로 그대로 이어졌다. "측정 장치 먼저"는 제 모든 프로젝트의 공통 패턴입니다.
- **탐색의 대가**: alpha-beta는 평가함수가 나쁘면 깊이가 독이 된다(나쁜 기준으로 더 멀리 본다). 평가함수 항 하나를 바꿀 때마다 리그를 다시 돌린 이유.
- **이 경험이 만든 것**: 결정론 시뮬 → self-play → trace 분석 → 리그 평가 방법론. 대회 종료 후 Winters의 봇 설계(봇도 GameCommand 생산자·AI 디버그 패널·시뮬 되감기)로 이식 중.
