# Winters 면접 대비 문서 세트

게임회사 기술면접 대비를 위한 2축 지식 베이스. 모든 내용은 Winters 코드베이스의 실제 파일(`path:line`)을 근거로 인용하며, 근거 없는 일반론은 일반론임을 명시한다.

## 구성

```
.md/interview/
├── README.md                ← 이 문서
├── cpp/                     ← 세트 1: C++ 언어 면접 (13챕터 + INDEX)
│   ├── 00_INDEX.md          ← 목차 + 학습 로드맵
│   ├── 01~12_*.md           ← 본질/컴파일·DLL/메모리·RAII/포인터/클래스설계/
│   │                           다형성/템플릿/STL·캐시/동시성/에러처리/ECS/네트워크
│   └── 13_interview_qa_bank.md      ← 질문 은행 71문항
└── engine/                  ← 세트 2: 엔진 도메인/아키텍처 면접 (16챕터 + INDEX)
    ├── 00_INDEX.md          ← 목차 + 직군별 로드맵
    ├── 01~11_*.md           ← 개관·소개스크립트/레이어경계/렌더링/ECS/리소스/
    │                           씬·루프/네트워크/GameSim·결정론/AI·내비/JobSystem/UI·툴
    ├── 12~15_*.md           ← 관점 챕터: 구조적 에러 처리 / 협업·프로세스 /
    │                           war story 사례집 / 향후 로드맵·기술부채
    └── 16_interview_qa_bank_engine.md ← 질문 은행 64문항
```

## 두 세트의 역할 분담

| | cpp 세트 | engine 세트 |
|---|---|---|
| 증명하는 것 | **언어를 정확히 아는가** | **시스템을 설계하고 문제를 풀 수 있는가** |
| 질문 유형 | "vtable이 뭔가", "move 후 상태는", "shared_ptr 비용은" | "왜 서버 권위인가", "어려웠던 버그는", "다시 만든다면" |
| 답변 무기 | 개념 정의 + Winters 실코드 인용 | 설계 결정 서사(왜→대안→선택→비용) + STAR 경험담 |

같은 코드가 양쪽에서 다른 각도로 등장한다 (예: JobSystem은 cpp/09에서 memory ordering의 예제, engine/10에서 "병렬화를 끄는 의사결정"의 서사). 이 교차가 의도된 설계다 — 면접에서 어느 축으로 질문이 와도 같은 자산으로 답한다.

## 사용법 (권장 사이클)

1. **매일 학습**: 각 INDEX의 로드맵 순서로 1챕터씩. 챕터의 "한 줄 본질"을 먼저 암기하고 심화로.
2. **모의 면접**: 13/16 질문 은행에서 무작위 5문항을 뽑아 소리 내어 답변. 첫 문장이 바로 안 나오면 해당 챕터 재회독.
3. **면접 전날**: 질문 은행 2개 + engine/01 소개 스크립트 + engine/14 war story 2건 리허설.
4. **면접 후**: 실제로 받은 질문 중 이 세트에 없던 것을 해당 챕터 Q&A에 추가한다 (세트를 살아있는 문서로 유지).

## 유지보수 규칙

- 코드가 바뀌어 인용(`path:line`)이 어긋나면 인용을 갱신하거나 삭제한다. 검증 안 된 인용을 남기지 않는다.
- 새 사고/해결 경험이 생기면 engine/14(war story)와 `.claude/gotchas.md`에 같이 기록한다 — gotchas는 재발 방지 규칙, 14장은 면접 서사.
- 이 세트는 면접 대비 문서다. 아키텍처의 현행 진실은 `.md/architecture/WINTERS_CODEBASE_COMPASS.md`가 소유한다 — 충돌 시 compass가 이긴다.
