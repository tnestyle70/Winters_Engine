# 최적화와 미정의 동작(UB) — 컴파일러는 무엇을 약속하는가

> "Release에서만 깨지는 버그를 어떻게 잡나요?", "signed 오버플로가 왜 UB인지 아세요?", "컴파일러 최적화가 실제로 뭘 하는지 설명해 보세요" — 세 질문은 사실 하나다. 컴파일러와 프로그래머 사이의 **계약서**를 읽어본 적 있는가. 이 챕터는 그 계약의 본문(as-if rule)과 이면(UB), 그리고 그 계약 위에서 돌아가는 최적화 기법들을 관통한다.

---

## ① 한 줄 본질

> **"컴파일러는 '관찰 가능한 동작(observable behavior)만 보존하면 코드를 마음대로 바꿔도 된다'는 as-if rule 아래에서 최적화한다. 미정의 동작(undefined behavior, UB)은 그 계약의 이면으로, '이런 일은 절대 일어나지 않는다'고 프로그래머가 컴파일러에게 서명해준 가정이다. 컴파일러는 그 가정을 믿고 검사를 지우고 루프를 벡터화한다 — 그래서 UB가 있는 코드는 최적화 수준에 따라 다르게 동작하고, 이것이 'Debug에선 되는데 Release에서 깨지는' 버그의 정체다. 최적화와 UB는 별개 주제가 아니라 같은 계약의 양면이다."**

---

## ② 기본 개념

### 계약의 원문 — as-if rule과 "관찰 가능한 동작"

C++ 표준은 컴파일러에게 "소스에 쓴 대로 기계어를 만들라"고 요구하지 않는다. 요구하는 것은 단 하나 — 프로그램이 **추상 기계(abstract machine)** 위에서 실행됐을 때의 **관찰 가능한 동작**을 재현하라는 것뿐이다. 이것이 as-if rule("마치 ~인 것처럼" 규칙)이다.

관찰 가능한 동작으로 표준이 못박은 것은 놀랄 만큼 적다:

| 보존해야 하는 것 (관찰 가능) | 마음대로 바꿔도 되는 것 (재량) |
|---|---|
| `volatile` 객체 접근의 존재와 순서 | 지역 변수의 존재 자체 (레지스터로, 혹은 소멸) |
| 프로그램 종료 시점에 파일에 쓰인 데이터 | 연산의 실행 순서·횟수 (결과가 같다면) |
| 인터랙티브 입출력의 프롬프트/입력 순서 | 함수 호출의 존재 (인라인으로 소거) |
| | 메모리 접근의 존재·순서 (별칭 규칙 안에서) |
| | 심지어 실행 시점 (컴파일 타임으로 앞당김) |

`int x = a * b; int y = a * b;`에서 곱셈을 두 번 할 의무가 없고, 결과를 안 쓰면 한 번도 할 의무가 없다. **소스 코드는 명령서가 아니라 의미 명세서다.** 이 관점 전환이 이 챕터 전체의 열쇠다 — 최적화는 "명세가 허용하는 가장 싼 구현을 찾는 탐색"이다.

파이프라인에서 최적화가 일어나는 위치만 짚고 넘어가면 (앞 챕터에서 파이프라인 자체를 다뤘으므로):

```
소스 → [프론트엔드: 파싱→AST]
     → [미들엔드: IR(SSA 형태) 위에서 기계 독립 최적화]   ← 상수접기·CSE·DCE·LICM·인라이닝·벡터화
     → [백엔드: 기계 의존 최적화]                          ← 명령어 선택·레지스터 할당·스케줄링
     → 기계어
```

### 로컬 최적화 — 식과 블록 단위의 4형제

**상수 접기(constant folding)와 상수 전파(constant propagation)** — 컴파일 타임에 알 수 있는 값은 컴파일 타임에 계산한다.

```cpp
constexpr int kTicksPerSecond = 30;
float dt = 1.f / kTicksPerSecond;   // 산출물에는 나눗셈이 없다. 0.0333333f 리터럴만 남는다
```

접기(folding)는 상수끼리의 연산을 계산해버리는 것이고, 전파(propagation)는 "x가 여기서 5라는 사실"을 이후 코드로 흘려보내 더 많은 접기를 유발하는 것이다. 둘은 연쇄한다 — 전파가 접기를 낳고, 접힌 상수가 다시 전파된다.

**공통 부분식 제거(common subexpression elimination, CSE)** — 같은 값을 두 번 계산하지 않는다.

```cpp
d = (a * b) + c;
e = (a * b) - c;   // a*b는 한 번만 계산해 임시에 담는다
```

단, "같은 값"임을 증명할 수 있어야 한다. 사이에 함수 호출이나 포인터 쓰기가 끼면 `a`나 `b`가 바뀌었을지 모르므로 CSE가 막힌다 — 이것이 뒤에서 볼 "별칭(aliasing) 문제"의 첫 등장이다.

**죽은 코드 제거(dead code elimination, DCE)** — 결과가 관찰되지 않는 계산, 도달 불가능한 분기를 지운다. 유의할 점: DCE는 무해한 청소부지만, **UB 기반 추론과 결합하면 "내가 쓴 null 체크"를 지우는 도구**가 된다(후술).

**강도 감소(strength reduction)** — 비싼 연산을 싼 연산으로 바꾼다.

```asm
; x * 9 를 곱셈 명령 없이 — x64에서 흔한 형태
lea eax, [rcx + rcx*8]    ; 주소 계산기(AGU)로 rcx + rcx*8 = 9x
```

고전 교과서 예시는 "곱셈을 시프트로"(`x * 8` → `x << 3`)지만, 현대 CPU에서 정수 곱은 3~4사이클 정도라 진짜 이득은 **나눗셈**에서 난다. 정수 나눗셈은 수십 사이클이라, 상수로 나누는 `x / 7`은 "매직 넘버 곱셈 + 시프트"로 변환된다. 루프 안에서는 `i * stride` 같은 유도 변수(induction variable)를 매 반복 덧셈으로 바꾸는 것도 강도 감소의 일종이다.

### 루프 최적화 — 프레임 시간은 루프 안에 있다

게임에서 CPU 시간의 대부분은 엔티티 순회, 정점 처리, 파티클 갱신 같은 루프다. 그래서 루프 최적화의 조건을 아는 것이 "컴파일러 친화적 코드"의 실체다.

**루프 불변 코드 이동(loop-invariant code motion, LICM)** — 반복해도 값이 안 변하는 계산을 루프 밖으로 끌어올린다.

```cpp
for (int i = 0; i < n; ++i)
    out[i] = in[i] * (scale * 0.5f);   // scale*0.5f 는 루프 밖에서 한 번
```

끌어올리려면 "불변임"과 "끌어올려도 안전함"(예외를 던지지 않고, 루프가 0회 돌 때 부작용이 없음)을 증명해야 한다. 루프 안에 불투명한 함수 호출이 있으면 증명이 깨진다 — 인라이닝이 중요해지는 이유의 예고편.

**언롤링(unrolling)** — 루프 본문을 여러 번 복제해 분기 오버헤드를 줄이고, 반복 간 명령어 수준 병렬성(ILP)을 노출한다. 대가는 코드 크기 — 명령어 캐시(i-cache)를 밀어내므로 무조건 이득이 아니다. 컴파일러가 휴리스틱으로 저울질하는 대표적 트레이드오프.

**자동 벡터화(auto-vectorization)** — 스칼라 루프를 SIMD 명령(SSE/AVX)으로 바꿔 한 명령에 4~8개 원소를 처리한다. 가장 이득이 크고, 가장 잘 깨진다:

| 축 | 벡터화 되는 조건 | 깨지는 조건 |
|---|---|---|
| 반복 간 의존 | `c[i] = a[i] + b[i]` (독립) | `a[i] = a[i-1] * k` (루프 캐리 의존) |
| 메모리 접근 | 연속(stride 1) 접근 | 간접 인덱스 `a[idx[i]]` (gather 필요) |
| 별칭 | 겹치지 않음을 증명 또는 런타임 검사 | 출처 불명 포인터 둘 — 겹칠 수도 있음 |
| 본문 | 직선 코드, 마스크로 바꿀 수 있는 단순 분기 | 함수 호출, 예외, 조기 `break` |
| 반복 횟수 | 진입 시점에 계산 가능(countable) | 데이터에 따라 끝나는 `while` |
| 부동소수 | 원소별 독립 연산 | 합산 리덕션 — 순서 고정 모드에선 제약 (후술) |

여기서 별칭 조건이 실무에서 제일 자주 걸린다. `void Add(float* dst, const float* src, int n)`에서 컴파일러는 `dst`와 `src`가 겹치지 않는다고 **가정할 수 없다**(둘 다 float 포인터라 strict aliasing 규칙으로도 못 가른다). 그래서 겹침 런타임 검사를 삽입하거나 벡터화를 포기한다. `__restrict` 힌트, 또는 포인터 대신 별개 배열임이 자명한 구조(SoA 컨테이너)가 해법이다. 벡터화 여부는 추측하지 말고 컴파일러 리포트로 확인한다 — MSVC `/Qvec-report`, Clang `-Rpass=loop-vectorize` 계열.

### 인라이닝 — "최적화의 어머니"

인라이닝(inlining)은 함수 호출을 본문 복사로 대체한다. 호출/리턴 몇 사이클을 아끼는 게 본질이 아니다 — **호출 경계는 옵티마이저의 시야 경계**라는 게 본질이다.

```
인라인 전:  caller ── call ──> [불투명한 상자 f]     ← f 안에서 무슨 일이? 포인터를 저장? 전역을 수정?
                                                       → CSE 막힘, LICM 막힘, 상수 전파 끊김
인라인 후:  caller 의 문맥 안에 f 의 본문이 펼쳐짐   ← 인자가 상수면 f 내부에서 접기 연쇄
                                                       → 분기 하나가 통째로 DCE, 루프가 벡터화
```

인자가 상수로 확정되면 함수 내부의 `if`가 컴파일 타임에 결정되고, 죽은 가지가 DCE로 사라지고, 남은 루프가 벡터화된다 — 앞의 모든 최적화가 인라이닝 뒤에서야 발동하므로 "최적화의 어머니"라 부른다. 연쇄가 실제로 어떻게 일어나는지 한 번에 따라가 보자:

```cpp
float Scale(float v, bool half) {
    if (half) return v * 0.5f;
    return v;
}
void Job(float* out, const float* in, int n) {
    for (int i = 0; i < n; ++i)
        out[i] = Scale(in[i], false);      // half 가 리터럴 false
}
```

```
1) 인라이닝      : Scale 본문이 루프 안에 펼쳐짐 — if (false) return v*0.5f; return v;
2) 상수 전파     : half = false 가 조건식으로 흘러듦
3) 접기 + DCE    : if (false) 가지 전체 삭제 → 본문은 out[i] = in[i]
4) 루프 최적화   : 조건 분기 없는 순수 복사 루프 → 벡터화 or memcpy 패턴 인식
```

호출 하나를 없앤 게 아니라, **호출 뒤에 숨어 있던 최적화 4단계를 해금**한 것이다. 반대로 `Scale`이 다른 TU에 있고 LTO가 꺼져 있으면 이 연쇄는 0단계에서 멈춘다 — 정의 가시성이 성능 문제가 되는 이유. 대가는 역시 코드 크기와 i-cache다. 그래서 컴파일러는 함수 크기·호출 빈도·인자의 상수성 같은 휴리스틱으로 판단한다.

여기서 유명한 괴리 — **`inline` 키워드는 인라이닝 지시가 아니다.** 현대 컴파일러는 인라이닝 여부를 자체 휴리스틱으로 결정하며 키워드는 힌트로서 사실상 무시된다. `inline`의 실제 언어적 의미는 **ODR(One Definition Rule) 완화**다: "이 정의가 여러 번역 단위에 중복 등장해도 위반이 아니며, 링커가 하나로 접어라." 즉 헤더에 함수 본문을 넣기 위한 자격증이지 성능 키워드가 아니다(C++17의 `inline` 변수도 같은 의미). 진짜 강제 수단은 컴파일러 확장인 MSVC `__forceinline`, GCC/Clang `[[gnu::always_inline]]` 쪽이고, 그마저 남용하면 i-cache를 태워 느려진다.

### 백엔드 — IR을 진짜 기계에 앉히는 세 단계

**명령어 선택(instruction selection)** — IR 패턴을 대상 ISA의 명령으로 매핑한다. `a + b*4`를 x64에선 `lea` 하나로, 곱-덧셈 연속을 FMA 지원 칩에선 `vfmadd`로 묶는 식이다. 같은 IR이라도 타깃 CPU에 따라 다른 기계어가 나오는 지점.

**레지스터 할당(register allocation)** — 무한한 IR 가상 레지스터를 유한한 물리 레지스터(x64 정수 레지스터 16개, XMM 16개)에 배정한다. 핵심 자료는 각 값의 **생존 구간(live range)** — "정의된 시점부터 마지막 사용까지"다:

```
시간 →   t1: a = ...      a ██████████░░░░░░░░
         t2: b = ...      b     ████████░░░░░░      a·b·c 가 동시에 살아있는 구간이
         t3: c = a + b    c         ██████████      물리 레지스터 수를 넘으면 → spill
         t4: use(b, c)
         t5: use(a, c)
```

고전 알고리즘은 간섭 그래프(interference graph)에 대한 **그래프 컬러링**: 생존 구간이 겹치는 값끼리 간선을 긋고, 레지스터 수 = 색 수로 칠한다. NP-완전이라 휴리스틱으로 근사한다. 색이 모자라면 **스필(spill)** — 값을 스택에 내렸다 다시 올린다. JIT처럼 컴파일 시간이 아까운 환경은 그래프를 안 만들고 구간을 한 번 훑는 **linear scan**을 쓴다(품질을 조금 내주고 컴파일 속도를 얻는 트레이드오프 — 어디에 시간을 쓸지조차 트레이드오프다). 스필은 L1 적중이라도 로드에 수 사이클씩 새는 데다 루프 안에서 반복되므로, "핫 루프에서 살아있는 변수 수를 줄이는 것"이 사람이 해줄 수 있는 협조다.

**명령어 스케줄링(instruction scheduling)** — 의존성이 허용하는 범위에서 명령 순서를 재배치해 지연(latency)을 겹친다. 비순차 실행(out-of-order) CPU가 하드웨어에서 상당 부분 해주는 시대지만, 의존 사슬의 길이 자체는 하드웨어도 못 줄이므로 여전히 의미가 있다. 역사적 맥락: 7세대 콘솔(Xbox 360/PS3)의 PowerPC 코어는 순차(in-order) 실행이라 컴파일러 스케줄링과 수동 지연 숨기기가 성능을 좌우했다 — 게임 업계가 이 단계에 유난히 민감했던 이유.

### UB와 최적화의 공생 — 계약의 어두운 절반

이제 계약의 이면. 표준은 일부 연산의 결과를 **정의하지 않는다** — signed 정수 오버플로, null 역참조, 배열 범위 밖 접근, 해제된 메모리 사용, 데이터 레이스, 잘못된 타입으로의 접근 등. "정의하지 않는다"의 실무적 의미는 이렇다:

```
프로그램의 가능한 모든 실행
├─ 정의된 실행 → 표준이 결과를 보장. 컴파일러는 as-if 로만 변형 가능
└─ UB 를 포함한 실행 → 표준이 아무것도 요구하지 않음
                        → 컴파일러는 이런 실행을 "존재하지 않는 경우"로 가정하고 최적화
```

UB는 컴파일러의 악의가 아니라 **프로그래머가 준 가정**이다. "signed 오버플로는 일어나지 않는다"고 언어 차원에서 서약되어 있으므로, 컴파일러는 그 전제로 추론한다. 이 가정이 실제 최적화를 어떻게 사는지 보자.

**예 1 — signed 오버플로가 UB라서 가능한 루프 최적화.**

```cpp
// 64비트 타깃, int(32비트) 인덱스
for (int i = 0; i < n; ++i)
    sum += p[i];
```

`p[i]`의 주소 계산은 64비트다. 만약 `i`가 랩어라운드(wrap-around)할 수 있다면 매 반복 32→64비트 부호 확장을 다시 해야 하고 "i가 단조 증가한다"는 보장도 없다. signed 오버플로 = UB이므로 컴파일러는 `i`를 64비트 레지스터로 승격해 포인터 증가 하나로 바꾸고, 반복 횟수를 확정해 벡터화까지 간다. 같은 코드를 `unsigned i`로 쓰면 랩이 **정의된 동작**(모듈로 2^32)이라 컴파일러가 랩 가능성을 성실히 고려해야 하고, 일부 변형이 막힌다. 극단적 축약형:

```cpp
bool f(int x) { return x + 1 > x; }
// Clang -O2: 무조건 true 를 반환하는 코드로 접힌다.
// "x+1 이 오버플로하면 false 아닌가?" — 오버플로는 UB, 즉 '일어나지 않는 일'
```

**예 2 — strict aliasing: 타입이 곧 별칭 정보다.**

컴파일러는 "T 타입 객체는 T(또는 호환 타입, 그리고 char 계열)의 lvalue로만 접근된다"고 가정해도 된다. 이 규칙이 없으면 모든 포인터 쓰기가 모든 메모리를 오염시킬 수 있다고 봐야 해서 CSE/LICM/벡터화의 별칭 분석이 전멸한다.

```cpp
int GetBits(float f) {
    return *reinterpret_cast<int*>(&f);   // UB — float 객체를 int lvalue 로 접근
}
int GetBitsOk(float f) {
    int i;
    std::memcpy(&i, &f, sizeof i);        // OK — 최적화되면 mov 한 개로 접힌다
    return i;                              // (C++20 은 std::bit_cast)
}
```

`memcpy`는 "느린 우회"가 아니다 — 컴파일러가 이 관용구를 인식해 레지스터 이동 하나로 접는다. 의미는 합법으로, 비용은 0으로. 참고로 MSVC는 역사적으로 strict aliasing 위반을 공격적으로 이용하지 않는 관용적 컴파일러지만, GCC/Clang은 이용한다 — "MSVC에서 돌아가니 합법"이라는 추론은 이식성 지뢰다.

**예 3 — null 체크 제거 사고.** 실제로 일어난 유형의 사고(2009년 Linux 커널 tun 드라이버 취약점이 유명하다):

```cpp
void f(Widget* p) {
    int v = p->value;      // (1) p 역참조 — p 가 null 이면 여기가 UB
    if (!p) return;        // (2) 컴파일러: (1)에서 역참조했으니 p != null 은 증명된 사실
    use(v);                //     → (2)의 검사는 DCE 로 삭제
}
```

프로그래머 의도는 "혹시 null이면 방어"였지만, 검사보다 **먼저** 역참조를 뒀다. 컴파일러는 (1)을 "p는 null이 아니다"라는 증거로 채택하고 (2)를 지운다. 커널 사례에선 0번 페이지를 매핑할 수 있는 환경 탓에 (1)이 크래시조차 안 나서, 지워진 검사가 보안 취약점이 됐다. 교훈: UB 기반 추론은 "크래시"가 아니라 **조용한 코드 소거**로 나타날 수 있다.

### 시간여행, 그리고 Debug/Release 괴리의 정체

표준은 UB가 발생하는 실행에 대해 **UB 지점 이전의 연산조차** 보장하지 않는다. 컴파일러가 UB를 전제로 코드를 재배치·소거하므로, UB "이전"의 printf가 안 찍히거나 분기가 미리 뒤틀리는 현상이 관측된다 — 이를 반농담으로 "시간여행(time travel)"이라 부른다. 디버깅 시 함의가 크다: **로그가 안 찍힌 지점 이후에 문제가 있다는 추론이 UB 앞에서는 성립하지 않는다.**

"Debug에선 되는데 Release에서 깨진다"는 미신적 현상의 실체도 여기 있다. UB가 있는 프로그램은 애초에 정답이 없고, 최적화 수준은 그 무정의가 **어떻게 관측되는지**를 바꿀 뿐이다:

| | Debug (`/Od`) | Release (`/O2`) |
|---|---|---|
| 코드 형태 | 소스를 순진하게 직역 — 매번 메모리에 읽고 쓴다 | 레지스터 승격, 재배치, 소거 |
| 미초기화 스택 | MSVC가 0xCC 패턴으로 채움 — "일관된 쓰레기" | 진짜 쓰레기 — 이전 프레임 잔재, 실행마다 다름 |
| 힙 | 디버그 CRT가 0xCD(미초기화)/0xDD(해제됨)/0xFD(경계) 패턴 + 가드 | 패턴 없음, 범위 밖 쓰기가 조용히 이웃을 오염 |
| UB 의존 코드 | 우연히 "직역된" 동작이 기대와 일치하기 쉬움 | 컴파일러 추론이 기대를 배신 |

즉 Debug는 버그를 고쳐주는 게 아니라 **우연의 배치를 안정화**해줄 뿐이다. Release에서 깨졌다면 십중팔구 코드에 원래 있던 UB(미초기화 읽기, 범위 밖, 댕글링, 데이터 레이스)가 드러난 것이고, "컴파일러 버그"는 통계적으로 최후의 용의자다. 접근법은 ③에서 정리한다.

### volatile — 최적화 장벽이지, 동기화 도구가 아니다

as-if rule 표에서 봤듯 `volatile` 접근은 관찰 가능한 동작이다. 컴파일러는 volatile 읽기/쓰기를 소거·병합·상호 재배치하지 못한다. 그래서 하드웨어 레지스터가 메모리에 매핑된 MMIO, 시그널 핸들러와 공유하는 `volatile sig_atomic_t` 같은 "컴파일러 몰래 값이 변하는 메모리"에 쓰는 물건이다.

멀티스레드 동기화에 안 되는 이유는 세 겹이다:
1. **원자성이 없다.** 워드보다 큰 타입은 찢어진 읽기(torn read)가 난다.
2. **하드웨어 재배치를 막지 않는다.** 컴파일러 재배치만 막을 뿐, CPU의 저장 버퍼/비순차 실행은 그대로다. 메모리 배리어를 안 깐다.
3. **happens-before 관계를 만들지 않는다.** C++ 메모리 모델에서 스레드 간 가시성은 atomic의 acquire/release가 만든다. volatile 접근끼리는 아무 동기화 변(edge)도 형성하지 않으며, non-atomic 공유 접근은 그 자체로 데이터 레이스 = UB다.

최적화와의 연결 고리로 이 고전 사례가 좋다:

```cpp
bool done = false;               // 다른 스레드가 true 로 바꿔줄 "예정"
while (!done) { /* spin */ }
// 옵티마이저의 싱글스레드 추론: 루프 안에서 done 을 아무도 안 바꾼다
// → 읽기를 루프 밖으로 끌어올림(LICM!) → if (!done) { while (true); }
```

무한 루프의 범인은 "최적화 버그"가 아니라 LICM이 계약대로 일한 것이다. 해법은 `volatile`이 아니라 `std::atomic<bool>`(+적절한 memory_order)이다 — atomic 접근은 컴파일러가 소거·승격할 수 없고 하드웨어 순서까지 계약된다. 함정 하나: MSVC는 역사적으로 `/volatile:ms` 모드에서 volatile에 acquire/release 의미를 얹어줬다(ARM 타깃 기본은 ISO 의미인 `/volatile:iso`). 이 관용이 "volatile로 동기화되던데요"라는 나쁜 습관을 길렀다 — 표준의 보장이 아니라 특정 컴파일러 옵션의 보장이다. 메모리 모델 각론은 [cpp/09_concurrency.md](../cpp/09_concurrency.md)와 크로스링크.

### 링크 너머 — LTO와 PGO

**LTO(link-time optimization)** — 번역 단위(TU)는 서로를 모른 채 컴파일되므로, TU 경계는 곧 인라이닝 경계였다. LTO는 .obj에 기계어 대신 IR을 담아두고 **링크 시점에 프로그램 전체를 놓고 다시 최적화**한다. TU를 넘는 인라이닝, 전 프로그램 관점의 DCE("이 함수는 아무도 안 부른다"), 가상 호출의 탈가상화가 열린다. MSVC에서는 컴파일 `/GL` + 링크 `/LTCG`이고, vcxproj의 `<WholeProgramOptimization>`이 이 스위치다. 비용: 링크 시간과 메모리가 크게 늘고, 최적화 단위가 프로그램 전체가 되면서 디버깅 시 코드 대응이 흐려지며, TU별로 조용히 공존하던 ODR 위반이 드러나기도 한다. 그리고 중요한 한계 — **LTO도 DLL 경계는 못 넘는다.** 링크 단위가 다르므로, DLL export 함수는 여전히 불투명한 호출이다.

**PGO(profile-guided optimization)** — 휴리스틱 대신 실측을 쓴다. 계측 빌드로 대표 시나리오를 실행해 분기 확률·호출 빈도를 수집하고, 그 프로파일로 다시 최적화한다(MSVC 기준 계측 → 훈련 실행 → `/USEPROFILE` 재빌드의 3단계). 효과는 핫 경로를 분기 없이 떨어지게 배치(fall-through), 핫/콜드 코드 분리로 i-cache 절약, 프로파일 기반 인라이닝·간접 호출의 추측성 탈가상화 등. 게임에서의 관건은 **"대표 시나리오"의 정의**다 — 훈련 플레이가 실전 워크로드(한타, 대규모 스폰)를 안 덮으면 그 경로는 오히려 콜드 취급을 받는다. 그래서 PGO는 출시 직전, 콘텐츠와 워크로드가 안정된 뒤에 넣는 최적화로 취급된다.

### 부동소수 — /fp 모드는 "무엇이 관찰 가능한가"의 선언

부동소수 덧셈은 결합법칙이 성립하지 않는다:

```cpp
float a = 1e8f, b = -1e8f, c = 1.f;
(a + c) + b   // 1e8f + 1.f == 1e8f  (24비트 가수에 1이 안 들어감) → 결과 0.f
(a + b) + c   // 0.f + 1.f          → 결과 1.f
```

따라서 재결합(reassociation)은 **값을 바꾸는 변형**이고, as-if rule상 원칙적으로 금지다. `/fp` 스위치는 이 원칙을 얼마나 풀어줄지의 선언이다:

| | `/fp:precise` (MSVC 기본) | `/fp:fast` | `/fp:strict` |
|---|---|---|---|
| 재결합·분배 | 금지 — 소스의 연산 순서 유지 | 허용 | 금지 |
| 역수 곱(x/y → x·(1/y))·근사 | 금지 | 허용 | 금지 |
| FMA 축약 | 버전/옵션에 따름 (`/fp:contract`) | 적극 | 금지 |
| NaN/Inf/예외·FP 환경 | 기본 가정 | 무시 가능 | 엄격 존중 |
| 목적 | 재현 가능한 결과 | 속도 (SIMD 리덕션 등) | IEEE 준수 검증·수치해석 |

`fast`가 빨라지는 대표 지점이 벡터화 절에서 미뤄둔 **합산 리덕션**이다. `sum += a[i]`를 SIMD로 하려면 부분합 4~8개를 병렬로 쌓다가 마지막에 합쳐야 하는데, 이는 덧셈 순서 변경 = 재결합이다. `precise`에서는 이 변형이 금지라 리덕션 벡터화가 제약된다 — **속도와 재현성의 정면 트레이드오프**.

네트워크 게임에 이것이 사활인 이유: 서버와 클라이언트가 같은 시뮬레이션 코드를 돌려 결과를 맞추는 구조(록스텝이든 서버 권위+예측이든)에서, 같은 소스라도 **다른 링크 단위**에 들어가면 인라이닝 문맥이 달라진다. `fast` 아래에서는 문맥이 다르면 접히는 순서도 달라질 수 있어, 서버 exe와 클라 exe가 같은 함수에서 다른 비트를 낼 수 있다. `precise`는 소스에 쓴 순서를 관찰 가능한 것으로 승격시켜 이 드리프트를 봉쇄한다. 단 `precise`가 만능은 아니다 — 컴파일러/아키텍처가 다르면(x87 80비트 잔재, 라이브러리 `sin/cos`의 구현 차) 여전히 어긋날 수 있어서, 크로스 플랫폼 결정론은 정수 고정소수점이나 소프트웨어 float까지 가기도 한다. 같은 툴체인·같은 아키텍처로 통제할 수 있다면 `precise`가 실용적 답이다.

## ③ 심화 — 꼬리질문 대비

**"as-if rule에서 '관찰 가능한 동작'이 정확히 뭔가?"**
volatile 객체 접근(존재와 상호 순서), 프로그램 종료 시 파일에 기록된 데이터, 인터랙티브 장치 입출력의 순서. 이 셋만 추상 기계와 일치하면 된다. 지역 변수의 메모리 존재, 함수 호출의 발생, 연산 횟수는 전부 재량이다. "디버거로 보이는 것"은 관찰 가능한 동작이 아니라는 점이 함정 — 그래서 Release에서 변수가 "optimized away"로 뜬다.

**"왜 signed 오버플로만 UB고 unsigned는 정의돼 있나?"**
역사적으로는 signed 표현이 기계마다 달랐기 때문이다(2의 보수/1의 보수/부호-크기). 표준은 특정 랩 동작을 강제할 수 없어 미정의로 뒀다. C++20부터 signed는 2의 보수 **표현**으로 확정됐지만 오버플로는 여전히 UB로 남았는데, 이제는 이유가 최적화 가치다 — 루프 유도 변수 승격, `x+1>x` 접기, 반복 횟수 확정 같은 변형이 이 가정에 기대고 있어서 정의해버리면 기존 코드가 일괄 느려진다. unsigned는 처음부터 "모듈로 2^N 산술"이라는 수학적 정의가 자연스러워 정의된 동작이 됐다.

**"UB, unspecified, implementation-defined의 차이는?"**
| | 표준의 요구 | 예 |
|---|---|---|
| implementation-defined | 구현이 선택하되 **문서화** 의무 | `sizeof(int)`, `char`의 부호 |
| unspecified | 유한한 후보 중 하나, 문서화 의무 없음, 실행마다 달라도 됨 | 함수 인자 평가 순서 |
| undefined | 아무 요구 없음 — 프로그램 전체의 의미 상실 | signed 오버플로, 범위 밖 접근 |
면접 포인트: 앞의 둘은 "프로그램은 여전히 유효하고 결과만 이식성이 없다", UB는 "프로그램 자체가 무의미해진다"는 급의 차이.

**"inline 붙이면 빨라지나?"**
아니다. 인라이닝 여부는 컴파일러 휴리스틱이 결정하고 키워드는 사실상 무시된다. `inline`의 실제 의미는 ODR 완화 — 헤더에 정의를 둘 수 있는 자격이다. 강제하려면 `__forceinline` 같은 확장을 쓰지만, i-cache 압박 때문에 남용은 역효과다. 꼬리의 꼬리: "그럼 인라이닝을 돕는 실질적 방법은?" — 정의를 호출자가 보는 위치(헤더/같은 TU)에 두거나 LTO를 켜는 것. 정의가 안 보이면 휴리스틱 이전에 물리적으로 불가능하다.

**"컴파일러는 왜 UB를 '악용'하나? 방어적으로 컴파일하면 안 되나?"**
악용이 아니라 논리적 귀결이다. 옵티마이저는 "일어날 수 있는 모든 정의된 실행"에 대해 결과를 보존해야 하는데, UB 실행은 그 집합에 없으므로 고려 대상이 아니다. 방어적으로 컴파일한다는 것은 곧 UB에 의미를 부여(랩 정의, null 접근 트랩 정의)한다는 뜻이고, 그 순간 해당 가정에 기댄 최적화가 전부 꺼진다. 실제로 그런 다이얼이 존재한다 — GCC/Clang `-fwrapv`(signed 랩 정의), `-fno-strict-aliasing`(리눅스 커널이 사용). 즉 "안전 vs 속도"는 컴파일러 옵션 수준에서 선택 가능한 트레이드오프고, 표준 기본값은 속도 쪽에 서 있다.

**"strict aliasing 규칙은 왜 존재하나?"**
별칭 분석(alias analysis)의 비용 때문이다. 모든 포인터가 모든 객체를 가리킬 수 있다면, 임의의 포인터 쓰기 뒤에는 레지스터에 캐시한 모든 값을 다시 읽어야 한다. "타입이 다르면 같은 객체가 아니다"라는 규칙은 타입 시스템을 별칭 정보로 재활용해 이 분석을 값싸게 만든다. 합법적 타입 재해석 통로는 `memcpy`/`std::bit_cast`(값 복사 의미라 별칭 문제가 없고, 컴파일러가 mov로 접는다)와 `char*`/`std::byte*`(바이트 검사용 예외).

**"volatile과 std::atomic의 차이를 컴파일러 관점에서 설명하라."**
volatile은 컴파일러에게만 말한다 — "이 접근을 소거·병합·상호 재배치하지 마라". atomic은 컴파일러와 하드웨어 모두에게 말한다 — "원자적으로 접근하고, memory_order에 맞는 배리어를 깔고, 스레드 간 happens-before를 형성하라". 스핀 대기 예시(`while (!done)`)에서 volatile은 LICM만 막고 CPU 재배치와 레이스-UB는 남는다. atomic은 셋 다 해결한다. 덤: atomic이라고 다 비싼 게 아니다 — x64에서 `load(acquire)`는 보통 일반 mov와 같은 명령이고, 비싼 것은 컴파일러 재배치 금지라는 계약과 `seq_cst` 저장의 배리어다.

**"부동소수는 왜 재결합하면 안 되나? 정수는 되는데."**
정수 덧셈은 (오버플로가 없다는 가정 아래) 결합법칙이 성립하지만 float은 각 연산마다 반올림이 끼어서 `(a+b)+c ≠ a+(b+c)`다. 유효자릿수가 크게 다른 값을 섞으면 순서에 따라 작은 값이 통째로 증발한다(1e8f + 1.f == 1e8f). 그래서 재결합은 값이 바뀌는 변형이고, 기본 모드에서 금지, `/fp:fast`에서만 허용된다.

**"Release에서만 깨지는 버그, 실제로 어떻게 접근하나?"**
순서가 있다. (1) **재현 고정** — 입력·시드·틱을 고정해 결정적으로 재현되게 만든다(결정론 시뮬 구조면 이게 공짜다). (2) **Release + 디버그 정보** — MSVC는 `/O2`와 `/Zi`(PDB)가 공존하므로 Release를 심볼 붙여 디버깅한다. 변수가 optimized away로 보이는 건 감수. (3) **이분 탐색** — 파일/함수 단위로 `#pragma optimize("", off)`를 걸어 범인 코드를 좁힌다. 범인을 찾는 목적이지 그걸 끄고 출시하는 게 해결이 아니다. (4) **도구** — ASan(MSVC 지원)으로 범위 밖/해제 후 사용, clang-cl로 빌드해 UBSan을 돌리는 방법도 있다. (5) **용의자 순서** — 미초기화 변수, 범위 밖 쓰기, 댕글링, 데이터 레이스, 평가 순서 가정, float 정밀도 가정. 컴파일러 버그는 이 목록을 다 소거한 뒤의 마지막 용의자.

**"레지스터 할당에서 spill이 왜 그렇게 문제인가?"**
스필 하나는 스택 저장+로드로 몇 사이클이지만, 스필은 정의상 레지스터 압박이 심한 곳 = 핫 루프 안에서 발생하므로 반복 횟수만큼 곱해진다. 게다가 로드는 의존 사슬에 지연을 삽입해 비순차 실행의 겹치기 여지를 줄인다. 사람이 돕는 방법: 핫 루프에서 동시에 살아있는 변수 수를 줄이고, 큰 구조체를 값으로 들고 다니지 않고, 루프를 쪼갠다(loop fission).

**"LTO 켜면 무조건 이득인가?"**
아니다. 링크 시간·메모리가 크게 늘어 반복 빌드가 느려지고(개발 중 이터레이션 비용), 인라이닝 확대가 코드 크기를 키워 i-cache에 불리할 수 있으며, TU별로 조용히 지나가던 ODR 위반이 링크 타임에 드러나거나 미묘한 동작 변화로 나타난다. 그래서 보통 Release/출시 구성에만 켠다. 그리고 DLL 경계는 LTO로도 못 넘는다 — 모듈 분리는 곧 최적화 경계 선언이다.

**"PGO가 특히 잘 듣는 코드는?"**
분기가 데이터에 따라 크게 쏠리는 코드(파서, 인터프리터, 상태 머신), 가상 호출이 사실상 한 타입으로 수렴하는 코드(탈가상화), 거대한 코드에서 핫셋이 작은 경우(핫/콜드 분리로 i-cache 개선). 반대로 균일 분포 분기나 이미 작은 핫루프는 이득이 적다. 함정은 훈련 데이터 대표성 — 프로파일에 없는 경로는 콜드로 강등된다.

**"constexpr와 상수 접기는 뭐가 다른가?"**
상수 접기는 옵티마이저의 **재량**이다 — 해줄 수도, 안 해줄 수도 있고 관측할 방법도 없다. `constexpr`/`consteval`은 언어의 **보장**이다 — 상수 표현식 문맥에서 컴파일 타임 평가가 요구되고, 불가능하면 컴파일 에러가 난다. 즉 constexpr는 "접힐 수 있는 코드"를 "접힘이 계약된 코드"로 승격시키는 도구다. 배열 크기, static_assert 인자처럼 컴파일 타임 값이 필요한 자리가 그 계약의 소비처다.

**"자동 벡터화가 안 될 때 컴파일러에게 뭘 해줄 수 있나?"**
(1) 별칭 해소 — `__restrict` 또는 겹칠 수 없는 구조(SoA, 별개 컨테이너). (2) 접근을 연속으로 — AoS의 건너뛰기 접근을 SoA로. (3) 본문 단순화 — 루프 안 함수 호출을 인라인 가능하게, 조기 탈출 제거. (4) 리덕션이면 순서 완화를 명시적으로 허용하거나(해당 파일만 fast 계열) 부분합을 손으로 쪼갠다. (5) 확인 — 벡터화 리포트로 "됐다고 믿는" 게 아니라 "됐음을 본다". 그래도 안 되면 intrinsic으로 직접 쓰는 게 게임 업계의 전통적 최종 수단이다.

## ④ Winters에서의 적용

### (1) 결정론은 코드가 아니라 빌드 플래그에서 시작한다 — 전 타깃 `/fp:precise`

Winters는 서버와 클라이언트가 같은 `Shared/GameSim` 시뮬 코드를 각자의 실행 파일로 링크해 돌리는 서버 권위 구조다. 두 링크 단위의 인라이닝 문맥이 다르므로, `/fp:fast`였다면 같은 소스 함수가 서버 exe와 클라 exe에서 다른 순서로 접혀 시뮬 드리프트를 만들 수 있다. 그래서 `<FloatingPointModel>Precise</FloatingPointModel>`를 GameSim(`Shared/GameSim/Include/GameSim.vcxproj:54`(Debug), `:73`(Release))뿐 아니라 서버(`Server/Include/Server.vcxproj:56`, `:83`), 클라이언트(`Client/Include/Client.vcxproj:58`, `:91`), 엔진(`Engine/Include/Engine.vcxproj:59`, `:88`)까지 **전 타깃·전 구성에 통일**해 박았다. 시뮬만 precise면 되는 것 아니냐는 질문에 대한 답: 경계에서 값이 오가는 이상 한 타깃이라도 fast면 그 값이 오염 경로가 되고, 구성(Debug/Release)마다 다르면 "Debug에서만 재현되는 드리프트"라는 최악의 디버깅 지형이 생긴다. 플래그 통일은 그 조합 폭발을 계약으로 차단한 것이다. 면접에서는 "as-if rule상 재량이던 부동소수 연산 순서를, 빌드 설정으로 관찰 가능한 동작에 편입시켰다"고 요약할 수 있다.

### (2) 컴파일러 재량에 맡기지 않은 소스 수준 강도 감소 — DeterministicRng

`Shared/GameSim/Core/Determinism/DeterministicRng.h:15~22`의 `NextU64`는 xorshift64 — 시프트와 XOR만 쓰는 순수 정수 연산이라 어떤 최적화 수준·플랫폼에서도 비트 단위로 같다(정수 연산은 /fp 모드와 무관). `:29~32`의 `NextF01`은 `(NextU32() >> 8) * (1.f / 16777216.f)`로, 나눗셈 대신 **역수 곱을 소스에 직접** 박았다. `/fp:precise`에서 `x / 16777216.f → x * (1/16777216.f)` 변형은 컴파일러 재량에 속하는 문제인데(2의 거듭제곱이라 값은 같지만), 재량에 맡기는 대신 소스에서 확정했다 — `1.f / 16777216.f`는 상수 접기로 리터럴이 되고, 남는 것은 곱셈 하나다. "강도 감소를 컴파일러가 해주길 비는 것"과 "결정론이 걸린 자리라 손으로 못박는 것"의 구분을 보여주는 코드.

### (3) 컴파일 타임 계약 — constexpr 상수와 static_assert

`Shared/GameSim/Core/Determinism/DeterministicTime.h:9~10`의 `kFixedDt = 1.f / 30.f`, `kTicksPerSecond = 30`은 `constexpr`다 — 모든 TU, 서버/클라 양쪽에서 동일한 리터럴로 접힘이 **보장**된다(재량인 접기를 계약으로 승격, ③의 constexpr 문답 그대로). 그리고 `Client/Private/GamePlay/SharedGameSimSmoke.cpp:46~47`은 `static_assert(DeterministicTime::kTicksPerSecond == 30)`로 이 상수를 컴파일 타임에 검증한다 — 누가 틱레이트를 건드리면 스모크 테스트가 실행되기도 전에 빌드가 깨진다.

### (4) LTCG는 켰지만, DLL 경계는 최적화 경계로 남는다

전 타깃 Release 구성에 `<WholeProgramOptimization>true</WholeProgramOptimization>`(`Server/Include/Server.vcxproj:33`)이 켜져 있어 TU 경계는 링크 타임에 지워진다. 그러나 Winters의 Engine.dll ↔ Client/Server exe 경계는 LTCG로도 못 넘는 인라이닝 장벽이다 — 엔진을 DLL로 분리한 아키텍처 결정(모듈 교체·협업 빌드 분리)이 곧 "export 함수는 영원히 불투명한 호출"이라는 최적화 상한을 선언한 셈이다. 이 트레이드오프를 인지하고 감수했다고 말할 수 있어야 한다: 핫 루프는 경계를 프레임당 1회 수준으로 건너게 설계하고(프레임 루프 안에서 DLL 경계를 N번 왕복하지 않기), 진짜 핫한 코드는 헤더/같은 모듈 안에 둔다. 경계 각론은 [cpp/02_compile_link_dll.md](../cpp/02_compile_link_dll.md).

### (5) 1st-party 코드에 volatile 0건 — 동기화는 전부 atomic

`Engine/Client/Server/Shared` 전체를 grep하면 `volatile`은 외부 라이브러리(lua, imgui, tracy)에만 있고 Winters가 작성한 코드에는 0건이다. 스레드 간 공유는 `Engine/Public/Core/JobSystem/WorkStealingDeque.h:91~92`의 `alignas(64) std::atomic<std::int64_t>`처럼 atomic + 명시적 memory_order로만 다룬다(`:37`의 release 펜스, `:47`의 seq_cst 펜스). "volatile로 동기화하면 안 되는 이유"를 물으면 ②의 3겹 논리(원자성/하드웨어 재배치/happens-before)를 말하고, "그래서 우리 코드베이스엔 volatile이 한 건도 없다"로 마무리하면 원리와 실천이 잇닿는다.

### (6) Debug/Release 이원 구성 — 그리고 결정론이 주는 디버깅 이점

Debug 구성은 `<Optimization>Disabled</Optimization>`(`Server/Include/Server.vcxproj:59`)로 직역 코드 + 디버그 CRT 패턴의 세계, Release는 `/O2` 계열 + `<IntrinsicFunctions>`(`Shared/GameSim/Include/GameSim.vcxproj:76~77`) + LTCG의 세계다. Release-only 버그가 났을 때 Winters가 유리한 지점: 시뮬이 고정 틱 + 시드 기반 RNG(`DeterministicRng`)라서 "같은 입력 → 같은 실행"이 구조적으로 성립하고, ③에서 말한 접근법 1단계(재현 고정)가 공짜다. 재현이 결정적이면 이분 탐색(`#pragma optimize("", off)`)과 상태 해시 비교가 실질적인 무기가 된다 — 비결정적 레이스성 버그와 달리 "몇 틱째에 갈라지는가"를 좁힐 수 있기 때문이다.

## ⑤ 면접 Q&A

**Q1. 컴파일러 최적화를 한 문장으로 정의한다면?**
- 핵심: "as-if rule 아래에서, 관찰 가능한 동작을 보존하는 가장 싼 구현을 찾는 탐색." 소스는 명령서가 아니라 의미 명세서라는 관점 전환을 먼저 말한다.
- 꼬리 대비: "관찰 가능한 동작이 뭔가?" → volatile 접근, 종료 시 파일 데이터, 인터랙티브 I/O 순서 — 이 셋뿐.
- Winters: `/fp:precise`는 "재량이던 부동소수 연산 순서를 관찰 가능한 동작에 편입시킨 선언"이라고 잇는다.

**Q2. UB가 왜 존재하나? 표준이 전부 정의하면 되지 않나?**
- 핵심: UB는 "이 일은 일어나지 않는다"는 프로그래머의 서약이고, 컴파일러는 그 가정 위에서 검사를 지우고 루프를 변형한다. 전부 정의하면(랩 정의, null 트랩 정의) 그 가정에 기댄 최적화가 일괄 꺼진다. 안전 다이얼이 실제로 있다 — `-fwrapv`, `-fno-strict-aliasing`.
- 꼬리 대비: signed 오버플로 루프 승격 예시를 코드로 쓸 수 있어야 한다.

**Q3. Debug에선 되는데 Release에서 깨진다. 원인과 접근법은?**
- 핵심: 최적화가 버그를 만든 게 아니라 원래 있던 UB의 관측이 달라진 것. Debug는 직역 코드 + 0xCC/0xCD 패턴으로 우연을 안정화할 뿐이다.
- 접근 순서: 재현 고정 → Release+PDB 디버깅 → `#pragma optimize("", off)` 이분 탐색 → ASan/UBSan → 용의자 목록(미초기화/범위 밖/댕글링/레이스). 컴파일러 버그는 마지막.
- Winters: 고정 틱 + 시드 RNG 결정론이라 1단계(재현 고정)가 구조적으로 공짜 — "몇 틱째에 갈라지는가"로 좁힌다.

**Q4. 인라이닝이 왜 "최적화의 어머니"인가?**
- 핵심: 호출 경계 = 옵티마이저의 시야 경계. 인라인이 되어야 상수 전파→접기→DCE→벡터화 연쇄가 해금된다. 호출 비용 몇 사이클의 문제가 아니다.
- 꼬리 대비: "inline 키워드는?" → ODR 완화가 실제 의미, 힌트로는 무시. 진짜 지렛대는 정의 가시성(헤더/같은 TU/LTO).

**Q5. /fp:fast를 쓰면 안 되는 경우는?**
- 핵심: fast의 본질은 재현성 포기(재결합 허용 → 링크 문맥마다 다른 비트). 여러 바이너리가 같은 float 결과를 내야 하는 결정론 시뮬·리플레이·록스텝에서 금지.
- Winters: 서버 exe와 클라 exe가 같은 GameSim 소스를 다른 인라이닝 문맥으로 링크하므로, 전 타깃 `/fp:precise` 통일(vcxproj 전수)이 드리프트 봉쇄의 전제.

**Q6. volatile로 스레드 플래그를 만들면 왜 안 되나?**
- 핵심: 3겹 — 원자성 없음(torn read), 하드웨어 재배치 미제어(배리어 없음), happens-before 미형성(레이스 = UB). volatile은 MMIO/시그널용 최적화 장벽일 뿐.
- 연결: `while (!done)` 무한 루프는 LICM이 계약대로 일한 결과라는 걸 짚으면 최적화 이해와 동시성 이해를 한 번에 보여준다.
- Winters: 1st-party 코드 volatile 0건, WorkStealingDeque의 atomic + memory_order가 실천 증거.

**Q7. LTO와 PGO를 언제 켜나?**
- 핵심: LTO = TU 경계 소거(인라이닝 확대, 전 프로그램 DCE), 비용은 링크 시간·ODR 위반 노출 → Release 전용. PGO = 휴리스틱을 실측으로 대체, 관건은 훈련 시나리오 대표성 → 워크로드가 안정된 출시 직전.
- 꼬리 대비: "LTO의 한계는?" → DLL 경계는 못 넘는다. 모듈 분리 = 최적화 경계 선언이라는 트레이드오프까지 말하면 아키텍처 이해로 연결된다.

## ⑥ 흔한 오답 / 함정

1. **"UB는 크래시다"로 이해.**
   틀림. UB의 무서움은 크래시가 아니라 **조용한 코드 소거와 시간여행**이다. null 체크가 지워지고, UB 이전의 로그가 안 찍히고, 오늘은 되다가 컴파일러 업데이트 후 깨진다. "정의되지 않음"은 "무슨 일이든 일어날 수 있음"이지 "죽는다"가 아니다.

2. **"Release에서 깨지니 컴파일러 최적화 버그"라고 답하기.**
   면접 감점 1순위. 최적화는 UB 없는 코드의 관찰 가능한 동작을 보존한다 — Release에서만 깨지면 십중팔구 내 코드의 UB가 드러난 것이다. 용의자 목록(미초기화/범위 밖/댕글링/레이스)을 먼저 말하고, 컴파일러 버그는 소거법의 마지막에 두는 순서로 답해야 한다.

3. **"inline 키워드 = 인라이닝 최적화"라고 답하기.**
   inline의 언어적 의미는 ODR 완화(헤더 정의 자격)다. 인라이닝 여부는 컴파일러 휴리스틱과 정의 가시성(같은 TU 또는 LTO)이 결정한다. 이 괴리를 모르면 "성능 튜닝으로 inline을 붙였다"는 말이 바로 꼬리를 잡힌다.

4. **"volatile 붙이면 스레드 안전"이라는 습관.**
   volatile은 컴파일러 재배치·소거만 막는 최적화 장벽이다. 원자성도, 하드웨어 배리어도, happens-before도 없다. MSVC `/volatile:ms`의 관용 때문에 x64에서 우연히 동작해온 코드가 많다는 것까지 알고 있으면 오히려 가점.

5. **"/fp:fast는 정밀도를 낮추는 옵션"이라는 요약.**
   부정확하다. fast의 본질은 정밀도 하락이 아니라 **재현성 포기**다 — 재결합·근사를 허용해 같은 소스가 문맥(인라이닝, 벡터화 여부)에 따라 다른 비트를 내게 된다. 오히려 중간 계산이 더 정확해지는 경우(FMA)도 있다. 네트워크 결정론 관점에서 문제는 "덜 정확해서"가 아니라 "예측 불가능해서"다.
