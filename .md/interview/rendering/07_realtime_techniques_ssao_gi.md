# 실시간 렌더링 기법 — 그림자 · SSAO · GI · 디퍼드

> "그림자는 어떻게 구현하나요? SSAO가 뭔지 원리부터 설명해 보세요. 디퍼드 렌더링의 장단점은? 드로우콜은 왜 비싼가요?" — 이 챕터는 개별 기법의 나열이 아니라, 전부 하나의 문제("렌더링 방정식을 16ms 안에 풀 수 없다")에 대한 서로 다른 타협임을 관통한다. Winters 파이프라인의 구조 자체는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)가 담당하고, 여기서는 이론·수학·API 본질을 다룬다.

---

## ① 한 줄 본질

> **"실시간 렌더링은 렌더링 방정식이라는 무한 재귀 적분을 프레임당 16ms 안에 '푸는 척'하는 근사의 예술이다. 전략은 크게 셋 — (1) 적분을 오프라인에 미리 굽는다(베이킹: 라이트맵·프로브), (2) 이미 렌더한 화면 공간 정보를 씬의 대용으로 훔쳐 쓴다(screen-space: SSAO·SSR), (3) 한 프레임에 못 모으는 샘플을 여러 프레임에 걸쳐 분할 상환한다(temporal: TAA·RTGI 축적). 여기에 '기여 없는 일은 아예 안 한다'(컬링·배칭·인스턴싱)는 직교 축이 더해진다. 그림자 매핑부터 루멘까지 모든 실시간 기법은 이 좌표 위의 한 점이고, 각 기법의 아티팩트(acne, 고스팅, 화면 밖 정보 부재)는 그 타협이 지불한 영수증이다."**

렌더링 방정식(rendering equation)을 텍스트로 쓰면:

```
L_o(x, w_o) = L_e(x, w_o) + ∫_Ω f_r(x, w_i, w_o) * L_i(x, w_i) * (n·w_i) dw_i
```

문제는 두 가지다. 첫째, 반구 Ω 전체에 대한 적분이라 샘플이 무한히 필요하다. 둘째, `L_i`(들어오는 빛)는 다른 표면의 `L_o`(나가는 빛)라서 **재귀**다 — 빛은 무한히 튕긴다. 오프라인 렌더러는 시간을 무한정 쓰며 몬테카를로로 수렴시키지만, 게임은 16.6ms(60fps) 안에 이 적분을 수천만 픽셀에 대해 "끝내야" 한다. 그래서 실시간 렌더링의 역사는 "적분의 어느 부분을, 어디서, 언제 계산할 것인가"를 재배치해 온 역사다.

## ② 기본 개념

### 0. 프레임워크 — 적분을 훔치는 세 가지 전략 + 한 가지 직교 축

| 전략 | 아이디어 | 대가 | 이 장의 기법 |
|---|---|---|---|
| **A. 베이킹(baking)** | 시간축을 오프라인으로 옮긴다. 몇 시간 걸려 풀고 결과만 저장 | 정적인 것만 가능. 메모리·빌드 시간 | 라이트맵, 라이트 프로브/SH, 리플렉션 프로브 |
| **B. 스크린 스페이스(screen-space)** | 이미 렌더한 깊이/노멀 버퍼를 "씬의 근사"로 재활용 | 화면 밖·가려진 정보가 없다 | SSAO, SSR, SSGI, 스크린 스페이스 그림자 |
| **C. 시간 분할 상환(temporal)** | 프레임당 샘플 수를 줄이고 여러 프레임에 걸쳐 축적 | 고스팅, 움직임에 약함, 히스토리 관리 | TAA, RTGI/루멘의 노이즈 축적, temporal SSAO |
| **(직교) 일 줄이기** | 화면에 기여하지 않는 계산은 아예 하지 않는다 | 판정 자체의 비용, 보수성 | 프러스텀/오클루전 컬링, 배칭, 인스턴싱, 상태 정렬, LOD |

그림자 매핑은 어디인가? "매 프레임 광원 시점으로 다시 렌더"하는 브루트포스지만, 가시성 항 하나만 떼어 **저해상도 깊이 텍스처로 근사**한다는 점에서 스크린 스페이스의 사촌(광원의 화면 공간)이다. 정적 그림자를 라이트맵에 구우면 전략 A가 되고, 캐스케이드를 프레임에 나눠 갱신하면 전략 C가 섞인다. **기법은 전략의 조합이지 카탈로그가 아니다** — 면접에서 이 프레임으로 답하면 개별 기법 암기가 아니라 설계 감각을 보여줄 수 있다.

### 1. 그림자 매핑 — "빛의 시점에서 보이는가"를 깊이 비교로 묻는다

그림자는 조명이 아니라 **가시성**이다. 렌더링 방정식의 직접광 항에 곱해지는 `V(x, light) ∈ {0,1}` — "표면 x에서 광원까지 막힘 없이 보이는가". 레이트레이싱이라면 광원으로 레이를 쏘면 되지만, 래스터라이저에는 레이가 없다. 그림자 매핑(shadow mapping)의 통찰: **"광원에서 보인다" = "광원 시점으로 렌더했을 때 깊이 버퍼 최전면에 있다"**. 가시성 질의를 깊이 렌더 한 번으로 일괄 치환한다.

```
패스 1 (광원 시점)                  패스 2 (카메라 시점)
┌─────────────┐                   픽셀의 월드 좌표 x를
│ 광원 뷰·투영으로 │                   광원 클립 공간으로 변환
│ 씬 깊이만 렌더  │ ── shadow map ──▶  z_light  vs  shadowMap[uv]
└─────────────┘                   z_light > 저장값 + bias → 그림자
```

```hlsl
float3 sm = mul(float4(worldPos, 1.f), g_matLightViewProj).xyz; // 직교광이면 w=1
float2 uv = sm.xy * float2(0.5f, -0.5f) + 0.5f;
float lit = (sm.z <= g_ShadowMap.Sample(g_PointSampler, uv).r + bias) ? 1.f : 0.f;
```

**shadow acne — 왜 줄무늬가 생기나.** 섀도우맵 텍셀 하나는 표면의 유한한 면적을 깊이값 **하나**로 대표한다(양자화). 표면이 광원에 비스듬하면 그 텍셀이 덮는 면적 안에서 실제 깊이가 크게 변하는데, 저장된 값은 하나뿐이라 같은 표면의 절반은 "저장값보다 뒤" 판정을 받아 자기 자신의 그림자에 들어간다. 원인이 둘이므로 처방도 둘이다:
- **상수 bias**: 양자화 오차만큼 비교를 후하게. 
- **slope-scaled bias**: 경사가 클수록 텍셀 내 깊이 변화가 크므로 `tan(경사각)`에 비례해 추가. D3D11 래스터라이저가 하드웨어로 지원한다(`D3D11_RASTERIZER_DESC`의 `DepthBias` + `SlopeScaledDepthBias` — 깊이 기울기 `max(|dz/dx|, |dz/dy|)`에 곱해 더한다).

bias가 과하면 **peter-panning**: 그림자가 물체 발밑에서 떨어져 공중부양처럼 보인다. bias는 "그림자 시작점을 광원 쪽으로 민다"와 동의어이기 때문. 대안으로 섀도우 패스에서 **앞면 컬링**(뒷면 깊이 저장)을 쓰면 acne는 물체 두께만큼의 여유로 해결되지만, 얇은 물체(종이·깃발)에서 빛이 새는 다른 문제와 교환한다. **공짜 처방은 없다.**

**PCF(percentage closer filtering) — 경계 소프트닝.** 그림자 경계 계단을 없애려고 섀도우맵을 그냥 블러하거나 깊이를 평균하면 안 된다 — "평균 깊이와의 비교"는 "비교 결과의 평균"과 전혀 다른 질문이다(비교는 비선형 연산이라 평균과 교환되지 않는다). PCF는 **주변 텍셀 각각과 비교한 0/1 결과를 평균**한다. 하드웨어가 `SampleCmpLevelZero`(comparison sampler)로 4탭 비교+보간을 한 번에 해 준다.

**원근 앨리어싱과 CSM — 왜 로그 분할인가.** 카메라 근처는 화면 픽셀 밀도가 높은데 섀도우맵 텍셀 밀도는 씬 전체에 균일하다 → 가까운 그림자가 블록처럼 깨진다. 캐스케이드 섀도우 맵(cascaded shadow map)은 뷰 프러스텀을 깊이 방향으로 N조각 내고 조각마다 딱 맞는 섀도우맵을 씌운다. 분할 경계가 로그(등비)인 이유: 원근 투영에서 화면 픽셀 하나가 덮는 월드 면적은 깊이 z에 대략 비례해 커진다. "섀도우맵 텍셀 : 화면 픽셀" 비율을 캐스케이드마다 일정하게 유지하려면 각 조각이 덮는 깊이 범위도 같은 비율로 커져야 하고, 그 결과가 등비수열이다:

```
z_i = n * (f/n)^(i/N)        // 로그 분할 (n=near, f=far, N=캐스케이드 수)
실무: z_i = λ*로그분할 + (1-λ)*균등분할   // PSSM — 로그 순수형은 근거리에 과도하게 몰림
```

**포인트 라이트**는 전방향이라 2D 섀도우맵 하나로 안 된다 → 큐브맵 6면에 깊이(보통 광원까지의 거리)를 렌더하고, 샘플링은 "프래그먼트→광원 벡터"로 큐브맵을 조회해 거리를 비교한다. 6패스 비용 때문에 실시간 포인트 섀도우는 개수를 강하게 제한하는 게 보통이다.

### 2. AO의 의미 — GI의 가시성 항만 떼어낸 근사

코너와 틈이 어두운 물리적 이유: 간접광(주변에서 오는 빛)은 대체로 **모든 방향에서** 온다. 열린 평지의 한 점은 반구 전체에서 빛을 받지만, 코너의 한 점은 반구의 절반이 벽으로 막혀 있다. 앰비언트 오클루전(ambient occlusion)은 "들어오는 간접광이 방향에 무관하게 균일하다"고 가정하고, **반구의 몇 %가 막혔는가**만 계산한다:

```
가시율 A(x) = (1/pi) * ∫_Ω V(x, w) * (n·w) dw     // AO = 1 - A = 차폐율
```

`1/pi`로 나누는 이유: 반구에서 `∫ (n·w) dw = pi`이므로, 아무것도 안 막혔을 때 A=1이 되도록 하는 정규화 상수다. `(n·w)` 코사인 가중이 붙는 이유: 렌더링 방정식의 코사인 항 그대로 — 법선에 비스듬히 들어오는 빛은 기여가 작으므로 그 방향이 막혀도 덜 어두워져야 한다.

핵심 정체성: **AO는 렌더링 방정식에서 `L_i`를 상수로 고정하고 가시성 항만 남긴 것**, 즉 GI의 극단적 저예산 근사다. 따라서 물리적으로 올바른 적용처는 간접광(앰비언트) 항뿐이다 — 직접광의 차폐는 섀도우맵이 이미 담당하므로, AO를 직접광에까지 곱하면 이중 감쇠다. (실무 엔진은 미관상 전체에 곱하기도 한다 — Winters도 그렇다. ④에서.)

### 3. SSAO — 씬 대신 깊이 버퍼에게 물어본다

진짜 AO는 반구로 레이를 쏴야 하는데(씬 전체 지오메트리 필요), SSAO(screen-space ambient occlusion)는 **이미 렌더한 깊이 버퍼를 "카메라에서 본 씬의 높이맵"으로 취급**해 레이캐스트를 텍스처 조회로 바꾼다. 2007년 Crysis가 상용화한 이 발상이 전략 B의 원형이다. 교과서(법선 지향 반구) 알고리즘의 단계와 각 단계의 "왜":

```
[1] 커널 생성 (CPU, 1회)          [2] 픽셀마다 (PS/CS)              [3] 후처리
접선 공간 반구 안 샘플 N개        각 샘플 p_i = x + TBN*k_i*radius   4x4 노이즈 주기의
원점 근처에 밀집 분포       →     p_i를 투영 → 깊이버퍼와 비교   →   edge-preserving 블러
(가까운 차폐가 더 중요)           저장깊이가 샘플보다 앞 = 차폐       (노이즈 제거)
                                 + range check + bias
```

1. **뷰 공간 반구 커널**: 샘플을 구가 아닌 **법선 방향 반구**에 두는 이유 — 구를 쓰면 평평한 벽면도 자기 뒤쪽 절반이 "차폐"로 잡혀 전체가 회색이 된다(초기 Crysis 구현의 실제 결함). 샘플을 원점 근처에 밀집시키는 건 가까운 차폐물이 AO에 더 크게 기여한다는 코사인·거리 감쇠의 이산화다.
2. **깊이 비교**: 샘플 위치를 화면에 투영해, 그 자리의 깊이 버퍼 값이 샘플보다 카메라에 가까우면 "그 방향은 지오메트리에 막혔다"로 센다. **range check**(깊이 차이가 radius를 넘으면 무시)가 없으면 전경 캐릭터가 멀리 있는 배경에 검은 후광(halo)을 드리운다 — 화면상 인접이 공간상 인접이 아니라는 스크린 스페이스의 근본 결함을 덧대는 패치다.
3. **랜덤 회전 노이즈**: 샘플 커널이 모든 픽셀에서 같으면 차폐 추정 오차가 공간적으로 상관돼 **밴딩**(등고선 줄무늬)으로 보인다. 4x4 노이즈 텍스처로 픽셀마다 커널을 회전시키면 오차가 고주파 노이즈로 바뀐다 — 인간 시각은 저주파 밴딩보다 고주파 노이즈에 훨씬 관대하고, 고주파는 블러로 지울 수 있다. **구조적 오차를 노이즈로 교환하는 것** — 몬테카를로 렌더링 전반의 핵심 트릭이다.
4. **블러 패스**: 노이즈 주기(4x4)에 맞춘 블러로 노이즈를 정리한다. 단, 그냥 가우시안이면 AO가 물체 경계를 넘어 번지므로 깊이 차이로 가중치를 죽이는 **bilateral(edge-preserving) 블러**를 쓴다.

**파라미터의 의미**: `radius`는 "얼마나 넓은 차폐를 볼 것인가" — 크면 넓은 스케일의 음영을 잡지만 텍스처 캐시 미스가 늘고 후광이 심해진다. `bias`는 샘플 깊이 비교에 주는 여유 — 섀도우 acne와 정확히 같은 병(자기 표면과의 양자화된 비교)에 같은 약이다.

**HBAO/GTAO 계열**: 샘플-비교 대신 화면 방향 몇 개를 따라 **마칭하며 수평선 각도(horizon angle)를 추적**하고, 그 각도로 차폐 적분을 해석적으로 계산한다. 같은 샘플 수에서 분산(노이즈)이 작다 — "무작위 점 찍기"에서 "방향당 구조적 탐색"으로 바꾼 것. Winters의 구현이 이 계열이다(④).

**스크린 스페이스 공통 한계** (SSAO·SSR·SSGI 전부): 화면에 그려진 것만이 세계의 전부다. (1) 화면 밖 지오메트리는 차폐를 못 만든다 — 카메라를 돌리면 화면 가장자리의 AO가 스르륵 사라진다. (2) 전경에 가려진 뒷공간 정보가 없다 — 깊이 버퍼는 픽셀당 표면 1개만 기억한다. (3) 뒤돌면 사라진다 — AO가 뷰 의존이라는 것 자체가 물리적 모순(진짜 AO는 뷰 무관)이다. 그래서 결론은 항상 같다: **스크린 스페이스는 소반경 접촉 디테일용, 대규모 차폐는 베이크나 레이트레이싱으로.**

### 4. GI 스펙트럼 — "무엇을 미리 굽고 무엇을 실시간으로 하나"

간접광(빛의 2차 이상 반사)을 다루는 기법들은 전부 하나의 축 위에 있다: **왼쪽일수록 많이 굽고(정적, 싸고, 고품질), 오른쪽일수록 실시간(동적, 비싸고, 노이즈)**.

```
상수 앰비언트 → 라이트맵 → 라이트 프로브/SH → 리플렉션 프로브 → SSGI/SSR → 복셀 GI/RTGI/루멘
(전부 상수)    (전부 굽기)   (수신점만 굽기)    (반사만 굽기)      (아무것도   (씬의 3D 근사에
                                                              안 굽기,     실시간 트레이싱
                                                              화면만)      + temporal 축적)
```

- **상수 앰비언트**: `color += albedo * ambientColor`. 적분을 상수 하나로 치환한 0차 근사. 코너도 평지도 똑같이 밝다 — AO가 이걸 보정하는 1차 패치다.
- **라이트맵(lightmap)**: 정적 지오메트리 + 정적 광원이면, 오프라인에서 **렌더링 방정식을 실제로 푼다**(패스 트레이싱/라디오시티로 수렴할 때까지). 결과(간접광 도달량)를 전용 UV(UV2)에 텍스처로 저장하고 런타임엔 샘플 한 번. 런타임 비용 대비 품질이 지금도 최강이라 모바일과 정적 배경 위주 게임의 주력이다. 대가: 움직이면 끝(구운 그림자가 물체를 따라오지 않음), UV2 작업과 seam, 빌드 시간, 메모리.
- **라이트 프로브(light probe) + 구면 조화 함수(spherical harmonics, SH)**: 라이트맵은 정적 표면 위의 값이라 **동적 오브젝트**(캐릭터)는 못 받는다. 그래서 공간에 프로브를 격자로 심고, 각 프로브에 "이 지점에 방향별로 도달하는 간접광"을 구워 둔 뒤, 캐릭터 위치에서 인접 프로브를 보간한다. 방향별 함수를 통째로 저장하면 크니 SH로 압축한다 — 직관만: **SH는 구면 위의 푸리에 급수**다. 저주파(부드럽게 변하는) 함수는 앞쪽 계수 몇 개로 충분한데, 간접광은 본질적으로 저주파라서 L2(밴드 3, 채널당 9계수)면 시각적으로 충분하다. 평가도 싸다 — 법선 방향으로 다항식 몇 개 평가. 대가: 고주파(선명한 그림자·하이라이트)는 표현 불가, 급격한 함수를 우겨넣으면 링잉(음수 로브)이 생긴다.
- **리플렉션 프로브(reflection probe)**: 위 둘은 디퓨즈용이고 스페큘러(거울 반사)는 방향성이 강해 SH로 안 된다. 지점에서 큐브맵을 캡처해 두고, 러프니스별로 블러한 밉 체인을 만들어(프리필터) 런타임에 반사 벡터로 조회한다. 대가: 캡처 지점에서만 정확 — 시차(parallax) 오차.
- **SSR/SSGI**: 반사 벡터(또는 반구 방향)를 따라 **깊이 버퍼 위를 레이마칭**. 굽는 게 없어 완전 동적이지만 스크린 스페이스 한계를 그대로 물려받는다 — 화면 밖이 비치는 거울은 못 만든다. 그래서 실무는 "SSR 실패 시 리플렉션 프로브로 폴백"처럼 스펙트럼의 여러 지점을 **겹쳐** 쓴다.
- **복셀 GI / RTGI / 루멘 (아이디어 수준)**: 화면이 아니라 **씬의 저해상도 3D 근사**(복셀 그리드, SDF, 하드웨어 BVH)를 만들어 거기에 콘/레이를 쏜다. 화면 밖 정보 문제를 원리적으로 해결하지만, 프레임당 픽셀당 레이 1~2개 수준이라 노이즈투성이 → **temporal 축적 + 디노이저**(전략 C)가 필수 짝이다. 루멘의 고스팅·반응 지연은 이 축적의 영수증이다.

### 5. 포워드 vs 디퍼드 — 라이팅을 언제 계산하는가

**포워드(forward)**: 오브젝트를 그리는 그 자리(픽셀 셰이더)에서 라이팅까지 끝낸다. 비용 구조가 문제다:

```
비용 ≈ Σ_오브젝트 (래스터된 픽셀 수 × 그 픽셀에 영향 주는 라이트 수)
```

라이트 M개가 오브젝트 N개에 겹치면 셰이더가 M×N 조합으로 라이트를 순회해야 한다(고전 멀티패스 시절엔 문자 그대로 드로우가 M×N). 게다가 나중에 다른 삼각형에 덮일 픽셀(오버드로우)의 라이팅 계산은 전부 버려진다.

**디퍼드(deferred)**: 계산을 미룬다(defer). 지오메트리 패스에서는 라이팅을 안 하고 **머티리얼 속성만 G-buffer에 기록**, 이후 화면 공간에서 픽셀당 정확히 한 번 라이팅한다. 전형적 G-buffer:

| RT | 내용 | 포맷 예 |
|---|---|---|
| RT0 | albedo (+AO) | RGBA8 |
| RT1 | world/view normal | RGB10A2 또는 RGBA16F(인코딩 선택) |
| RT2 | roughness / metallic / specular | RGBA8 |
| depth | 하드웨어 깊이 (위치는 여기서 재구성) | D24S8 / D32F |

비용이 `화면 픽셀 수 × 라이트 영향 범위`로 바뀌어 라이트 수백 개가 가능해진다 — 2000년대 후반 디퍼드 유행의 이유. 대신 청구서가 온다:
- **대역폭**: 1080p에서 RGBA8 3장 + 깊이만 해도 프레임당 수십 MB를 쓰고 라이팅 패스에서 다시 읽는다. "계산을 아끼고 대역폭을 내는" 교환이라, 대역폭이 귀한 모바일(타일 메모리)에선 손해가 되기 쉽다.
- **반투명을 못 담는다**: G-buffer는 픽셀당 표면 **하나**만 기록한다. 반투명은 픽셀에 여러 표면이 겹치는 것 자체가 정의라서, 결국 반투명만 별도 포워드 패스로 그린다 — 모든 디퍼드 엔진에 포워드가 공존하는 이유.
- **MSAA 곤란**: 아래 9절에서.
- **머티리얼 다양성 제약**: 모든 머티리얼이 같은 G-buffer 레이아웃이라는 계약에 묶인다. 피부·천·머리카락 같은 특수 셰이딩은 셰이딩 모델 ID를 G-buffer에 심고 라이팅 패스에서 분기하는 식으로 우회한다.
- **역설적 이점 하나**: 디퍼드는 depth+normal이 공짜 부산물이라 SSAO/SSR 같은 스크린 스페이스 기법과 궁합이 좋다. 포워드는 이걸 위해 별도 프리패스를 돌려야 한다(Winters가 정확히 이 케이스 — ④).

**포워드+/클러스터드가 되돌아온 이유**: 디퍼드의 진짜 수확은 "라이팅을 미룬 것"이 아니라 "**어떤 픽셀에 어떤 라이트가 닿는지를 자료구조로 정리한 것**"이었다. 그렇다면 그 자료구조(화면 타일별/3D 클러스터별 라이트 목록)만 컴퓨트로 만들고, 셰이딩은 포워드로 하면? 라이트 M×N 문제는 풀리고, 반투명·MSAA·머티리얼 자유는 돌아온다. 이것이 Forward+(tiled)와 clustered forward이고, Doom(2016)류 엔진과 모바일/VR이 이 노선이다. **역사가 한 바퀴 돈 게 아니라, 디퍼드에서 배운 것(라이트 컬링)만 챙겨서 돌아온 것이다.**

### 6. 드로우콜은 왜 비싼가 — GPU가 아니라 CPU의 문제

흔한 오해: "드로우콜마다 GPU가 커널 전환 같은 걸 해서 비싸다". 실제 병목은 대부분 **CPU 쪽 드라이버 경로**다. `DrawIndexed` 한 번마다 드라이버는:
1. **상태 검증**: 바인딩된 리소스가 유효한가, 읽기/쓰기 해저드는 없는가, 셰이더 시그니처와 입력 레이아웃이 맞는가.
2. **상태 해시/변환**: API 상태 조합(블렌드×래스터×깊이×셰이더)을 하드웨어 네이티브 커맨드로 변환 — 조합이 처음이면 내부 파이프라인 컴파일까지.
3. **커맨드 버퍼 기록**: GPU가 읽을 실제 커맨드 생성. DX11은 이걸 immediate context **한 스레드**에서만 할 수 있어 멀티코어를 못 쓴다.

DX12/Vulkan이 "드로우콜이 싸다"는 말의 실체: 검증·컴파일을 **PSO(pipeline state object) 생성 시점으로 선불**시키고(그래서 셰이더 컴파일 스터터가 로딩/최초 사용 시점으로 이동), 커맨드 기록을 멀티스레드로 열었다. 비용이 사라진 게 아니라 **지불 시점이 이동**한 것이다.

대응 4종 — 전부 "드로우콜 수 또는 상태 변경 수를 줄인다"로 환원된다:
1. **배칭(batching)**: 같은 상태를 쓰는 메시를 하나의 버퍼로 합쳐 드로우 1번. 정적 배칭은 메모리(정점 복제)와 교환.
2. **인스턴싱(instancing)**: `DrawIndexedInstanced(indexCount, instanceCount, ...)` — 같은 메시를 트랜스폼만 바꿔 N개. 조건: **메시와 머티리얼(파이프라인 상태)이 동일**해야 하고, 인스턴스별 데이터(월드 행렬 등)를 두 번째 정점 스트림(`D3D11_INPUT_PER_INSTANCE_DATA`)이나 structured buffer + `SV_InstanceID`로 공급해야 한다. 미니언 무리·풀·나무가 교과서적 적용처.
3. **텍스처 아틀라스/배열**: 텍스처 바인딩 변경이 배칭을 끊는 주범이므로, 여러 텍스처를 한 장(아틀라스)이나 texture array로 합쳐 배칭 가능 구간을 늘린다.
4. **상태 정렬(state sorting)**: 드로우들을 (셰이더, 텍스처, 상수) 키로 정렬해 상태 변경 횟수를 최소화. 불투명은 여기에 front-to-back 깊이 정렬을 더해 early-Z 기각까지 얻는다 — 단 반투명은 back-to-front가 강제라 정렬 기준이 충돌한다(8절).

### 7. 컬링 — 안 보이는 것에 0ms 쓰기

**프러스텀 컬링(frustum culling)**의 고전형: 뷰 프러스텀을 월드 공간 평면 6개로 표현하고(뷰-투영 행렬의 행 조합으로 추출 — Gribb/Hartmann 방법), 오브젝트 바운딩과 판정한다. 평면 `n·p + d = 0`에 대해:

```
구(중심 c, 반지름 r):  dist = n·c + d;  dist < -r 이면 그 평면 완전 바깥 → 컬
AABB: p-vertex 트릭 — 평면 법선 방향으로 가장 먼 코너 하나만 검사
      p.x = (n.x >= 0) ? max.x : min.x  (y,z 동일)
      n·p + d < 0 이면 박스 전체가 바깥 → 컬
```

한 평면이라도 "완전 바깥"이면 컬. 어느 평면에서도 완전 바깥이 아니면 **보이는 걸로 간주**한다 — 이 판정은 보수적이라, 프러스텀 코너 근처에 대각선으로 걸친 큰 박스는 실제로 안 보여도 통과시킨다(false positive는 허용, false negative는 불허 — 잘못 컬면 물체가 사라지는 버그, 잘못 통과시키면 성능 낭비일 뿐).

대안 정식화가 **클립 공간 판정**이다: 바운딩 8코너를 `local→clip` 행렬로 변환해 클립 공간 부등식 `-w <= x <= w`, `-w <= y <= w`, `0 <= z <= w`(D3D 규약)로 검사한다. 8코너 전부가 같은 부등식 하나를 위반하면 컬. 평면 추출이 필요 없고 "월드 행렬 × 뷰투영"을 곱해 로컬 공간에서 바로 판정할 수 있다. Winters가 이 방식이다(④) — 단 `w <= 0`(카메라 뒤) 코너가 섞이면 부등식 해석이 깨지므로 near 평면을 걸치는 박스는 무조건 보이는 걸로 취급하는 보수 처리가 필요하다.

**오클루전 컬링(occlusion culling)** — 프러스텀 안이지만 다른 물체에 가려진 것 제거. 세 계보만:
- **GPU 오클루전 쿼리**: 바운딩 박스를 깊이 테스트만 하며 그려 보고 "몇 픽셀 통과?"를 GPU에 묻는다. 결과 회수에 1~2프레임 지연 → 코너를 돌 때 물체가 늦게 나타나는 팝핑.
- **Hi-Z(hierarchical Z)**: 깊이 버퍼의 밉 피라미드(각 밉 텍셀 = 하위 4텍셀의 max)를 만들어, 바운딩의 화면 사각형을 굵은 밉 한 번 조회로 보수 판정. GPU 주도 렌더링(compute 컬링 + indirect draw)의 표준 부품.
- **소프트웨어 래스터**: CPU에서 오클루더 단순 메시를 저해상도로 직접 래스터해 즉시 판정 — 지연이 없다.

### 8. 반투명 정렬 — 알파 블렌딩은 순서를 잊지 못한다

알파 블렌딩은 over 연산자다:

```
C_out = C_src * a_src + C_dst * (1 - a_src)
```

이 연산은 **비가환**이다 — 빨간 유리 뒤 파란 유리와 파란 유리 뒤 빨간 유리는 다른 색이다. 대입 순서가 곧 물리적 앞뒤 관계이므로 **back-to-front**(뒤부터) 그려야 맞는 그림이 나온다. 그래서 반투명 렌더링의 3계명: (1) 불투명을 전부 그린 뒤에, (2) 카메라 거리로 뒤→앞 정렬해서, (3) **깊이 쓰기는 끄고**(깊이 테스트는 켠 채) 그린다. 깊이를 쓰면 먼저 그린 반투명이 뒤에 올 반투명을 depth로 잘라 버린다.

한계: 오브젝트 단위 정렬은 **교차하는 반투명**을 못 푼다. 서로 관통하는 두 유리판은 픽셀별로 앞뒤가 다르므로 어떤 오브젝트 순서도 정답이 아니다 — 올바른 해는 프래그먼트 단위 정렬, 즉 OIT(order-independent transparency: depth peeling, per-pixel linked list, weighted blended)인데 비용 때문에 게임에선 드물다. 실무의 우회는 블렌드 모드 선택이다: **additive**(`C_out = C_dst + C_src`)는 덧셈이라 **가환 = 정렬 불필요**. 불꽃·마법·글로우 같은 "빛을 더하는" FX가 additive를 선호하는 건 미학이자 정렬 비용 회피다.

### 9. 안티앨리어싱 계보 — 같은 문제, 지불 방식만 다르다

앨리어싱의 원인: 삼각형 경계는 무한히 날카로운(무한 주파수) 신호인데 픽셀은 유한한 점 샘플이다 — 나이퀴스트 위반. 처방의 역사는 "추가 샘플 비용을 누가 내는가"의 역사다:

| 기법 | 무엇을 늘리나 | 비용 | 한계 |
|---|---|---|---|
| **SSAA** | 모든 것(해상도 N배 렌더 후 축소) | 셰이딩·대역폭 N배 | 무식하게 비쌈. 품질 기준점 |
| **MSAA** | 커버리지·깊이만 서브샘플 N개, **PS는 픽셀당 1회** | 경계 픽셀 대역폭만 | 지오메트리 경계만 해결. 셰이더 앨리어싱(스페큘러 반짝임)·알파테스트는 못 잡음 |
| **FXAA/SMAA** | 아무것도 안 늘림 — 완성 이미지에서 엣지 감지 후 방향성 블러 | 포스트 1패스 | 서브픽셀 정보가 없어 얇은 지오는 여전히 깜빡임 |
| **TAA** | 시간축 샘플 — 프레임마다 투영에 서브픽셀 지터, 이전 프레임을 재투영해 축적 | 히스토리 버퍼 + 재투영 | 고스팅, 뭉개짐(선명도 손실) |

**MSAA의 정확한 동작**: 래스터라이저가 픽셀당 N개 샘플 위치에서 커버리지와 깊이를 각각 판정하되, 픽셀 셰이더는 픽셀당 한 번만 실행하고 그 색을 커버된 샘플들에 복사한다. 마지막에 resolve(샘플 평균). 삼각형 **경계**에서만 실질 비용이 들게 설계된 하드웨어 특화 해법이다.

**왜 디퍼드와 상성이 나쁜가**: (1) G-buffer 전체를 MSAA 타깃으로 만들면 메모리·대역폭이 N배가 되고, (2) 라이팅을 정확히 하려면 경계 픽셀은 **샘플별로** 라이팅해야 해서 "PS 1회"라는 MSAA의 존재 이유가 무너지며, (3) resolve를 라이팅 전에 하면 노멀·깊이의 평균이라는 물리적으로 무의미한 표면(두 벽의 중간을 향하는 유령 노멀)을 라이팅하게 된다. 비선형 연산(라이팅) 앞에서 선형 평균(resolve)은 교환되지 않는다 — PCF에서 깊이를 평균하면 안 되는 것과 같은 수학이다.

**TAA 고스팅의 원인**: 재투영이 실패하는 곳 — 새로 드러난 영역(disocclusion), 모션 벡터가 없는 변화(그림자·반사·반투명의 움직임) — 에서 과거 색이 유효하다고 믿고 섞기 때문. 현재 프레임 주변 색 범위로 히스토리를 죄는 neighborhood clamp가 표준 완화책이지만, 죄면 죌수록 AA 품질이 떨어지는 트레이드오프의 연속이다.

### 10. 포스트 프로세스 체인 — 블룸과 순서의 문제

**블룸(bloom)**: 밝은 광원 주변이 번지는 현상(눈·렌즈의 산란)의 모사. 파이프라인: (1) threshold — 일정 밝기 이상 픽셀만 추출, (2) 다운샘플 체인을 내려가며 블러(절반 해상도씩 내려가는 dual filter가 큰 가우시안을 싸게 근사), (3) 업샘플하며 원본에 합성. threshold가 "1.0보다 밝다"는 의미를 가지려면 씬이 **HDR**(1.0 초과 값 보존)이어야 한다 — 블룸은 HDR 파이프라인의 존재 이유 중 하나다.

**순서가 본질이다**: `씬 렌더(HDR, linear) → 블룸 등 HDR 포스트 → 톤 매핑(HDR→LDR) → UI`. 블룸을 톤 매핑 **뒤에** 하면 하이라이트가 이미 1.0으로 뭉개진 뒤라 "무엇이 밝았는지" 정보가 사라져 번짐이 죽는다. UI를 톤 매핑 앞에 넣으면 UI 색이 톤 커브에 왜곡된다. 톤 매핑·색 공간의 각론은 06장(HDR·톤 매핑)으로.

## ③ 심화 — 꼬리질문 대비

**"shadow acne와 SSAO의 bias는 같은 문제인가?"**
같은 병의 두 발현이다. 둘 다 "양자화된 깊이 표현과 자기 자신을 비교"하는 데서 오는 자기 차폐(self-occlusion) 오판이고, 처방도 같다 — 비교에 여유(bias)를 준다. 차이는 과용의 부작용: 섀도우는 peter-panning, SSAO는 접촉부 음영이 사라지는 것.

**"peter-panning은 왜 생기고, bias 말고 다른 해법은?"**
bias는 그림자 판정 기준을 광원 쪽으로 미는 것이라, 물체와 바닥이 닿는 지점의 그림자가 물체에서 분리된다. 대안은 섀도우 패스에서 앞면을 컬링해 뒷면 깊이를 저장하는 것 — acne 여유가 "물체 두께"만큼 생겨 bias를 거의 안 써도 된다. 대신 닫히지 않은/얇은 지오메트리에서 빛샘이 생긴다. 둘을 섞어 최소 bias + 두께 여유로 타협하는 게 실무다.

**"PCF에서 깊이를 먼저 평균하면 왜 안 되나?"**
비교(step 함수)는 비선형이라 평균과 교환되지 않는다. `mean(compare(z, d_i))`는 "차폐된 샘플의 비율"(원하는 것)이고, `compare(z, mean(d_i))`는 "평균 깊이보다 앞인가"라는 전혀 다른 이진 질문이다. 계단과 부드러운 경사의 평균 깊이가 같아도 그림자 비율은 다르다. VSM(variance shadow map)은 평균과 분산을 저장해 체비셰프 부등식으로 "차폐 확률의 상한"을 추정함으로써 이 비선형성을 통계로 우회한다 — 대가는 겹친 오클루더에서의 라이트 블리딩.

**"CSM 캐스케이드 경계가 눈에 띄는 문제는 어떻게 처리하나?"**
경계 부근에서 두 캐스케이드를 모두 샘플해 블렌딩하는 게 기본이다. 또 하나의 고전 이슈는 카메라 이동 시 그림자 가장자리가 지글거리는 shimmering — 섀도우맵 텍셀 격자가 월드에 고정돼 있지 않아서다. 캐스케이드 투영 원점을 섀도우맵 텍셀 크기 단위로 스냅해 격자를 월드에 고정하면 잡힌다. "텍셀은 면적을 대표한다"는 acne와 같은 뿌리의 문제다.

**"SSAO가 카메라를 돌리면 사라지는 이유를 설명해 보라."**
차폐를 만들던 지오메트리가 화면 밖으로 나가거나 다른 물체에 가려지면, 깊이 버퍼에 존재하지 않으므로 차폐 계산에서 빠진다. 진짜 AO는 뷰 무관량인데 SSAO는 "이번 프레임 화면"이라는 씬의 부분 집합으로 계산하니 뷰 의존이 된다. 이건 버그가 아니라 스크린 스페이스라는 선택의 정의적 한계다.

**"SSAO를 절반 해상도로 돌리는 이유와 부작용은?"**
AO는 저주파 신호라 해상도를 낮춰도 시각 손실이 작은 반면 비용은 1/4로 준다. 부작용은 업샘플 시 물체 경계에서 AO가 번지는 것 — 깊이/노멀로 가중하는 bilateral upsample로 경계를 보존한다. "신호의 주파수에 맞는 해상도로 계산한다"는 원칙의 사례.

**"구면 조화 함수를 수식 없이 설명한다면?"**
구면 위 함수의 푸리에 급수다. 1D 주기 함수를 sin/cos 급수로 쓰듯, 구면 함수를 기저 함수(밴드 0=상수, 밴드 1=방향별 기울기 3개, 밴드 2=5개...)의 가중합으로 쓴다. 간접광은 부드럽게(저주파로) 변하므로 밴드 2까지 9계수 × RGB = 27 float면 충분하고, 평가도 법선 성분의 다항식이라 셰이더에서 몇 곱셈이면 된다. 한계도 푸리에와 같다 — 급격한 함수를 적은 항으로 자르면 링잉(음수 로브)이 생겨, 밝은 창문 반대편이 음수(검은) 빛을 받는 아티팩트가 난다.

**"라이트맵이 이렇게 제약이 많은데 왜 아직도 쓰이나?"**
런타임 비용 대비 품질이 여전히 압도적이기 때문이다. 오프라인에서 무한 바운스까지 수렴시킨 GI를 런타임에는 텍스처 샘플 하나로 얻는다 — 어떤 실시간 GI도 이 비용으로 이 품질을 못 낸다. 실시간 GI(루멘 등)의 의의는 품질이 아니라 "정적이어야 한다"는 제약의 해제다. 콘텐츠가 정적이라면 베이킹이 지금도 정답이다.

**"디퍼드 렌더러에서 반투명은 어떻게 처리하나?"**
G-buffer는 픽셀당 표면 1개라는 전제 위에 서 있으므로 반투명은 원리적으로 못 담는다. 그래서 불투명을 디퍼드로 끝낸 뒤 반투명만 별도 포워드 패스로 back-to-front 정렬해 그린다. 모든 디퍼드 엔진이 사실상 하이브리드인 이유이고, "디퍼드 vs 포워드"가 양자택일이 아니라는 증거다.

**"G-buffer에 월드 포지션을 저장하나?"**
안 한다. 위치는 깊이 버퍼와 역투영 행렬에서 재구성할 수 있으므로, RGBA16F 한 장(대역폭)을 통째로 아낀다: NDC를 만들어 `mul(float4(ndc, depth, 1), invViewProj)` 후 w 나눗셈. "저장 가능한 것과 유도 가능한 것을 구분해 대역폭을 아낀다"는 G-buffer 설계의 제1원칙이다. Winters의 SSAO 셰이더가 정확히 이 재구성을 한다(④).

**"인스턴싱이 안 먹거나 효과가 없는 경우는?"**
(1) 머티리얼/텍스처가 인스턴스마다 다르면 파이프라인 상태가 갈라져 배칭이 깨진다 — 텍스처 배열·bindless로 우회. (2) 스킨드 메시는 인스턴스마다 본 팔레트가 달라 단순 인스턴싱이 안 된다 — 전체 본 행렬을 structured buffer에 몰아넣고 인스턴스별 오프셋으로 읽게 하면 가능. (3) 인스턴스당 정점이 매우 적으면(파티클 쿼드) 드로우 오버헤드보다 정점 처리 비효율이 문제가 되기도 한다. "인스턴싱 = 상태 동일성의 확보"가 본질이지 API 호출 하나가 아니다.

**"clip()/discard를 쓰면 왜 성능 경고가 붙나?"**
early-Z(픽셀 셰이더 전에 깊이 기각) 최적화와 충돌한다. 셰이더가 픽셀을 죽일 수 있으면 하드웨어는 "이 픽셀의 깊이 쓰기가 실제로 일어날지"를 PS 실행 전에 확정할 수 없어, 깊이 쓰기가 켜진 드로우에서 early-Z를 보수적으로 꺼야 한다(깊이 쓰기가 꺼진 패스는 영향이 적다). 알파테스트 식생이 비싼 이유의 절반이 이것이다.

**"드로우콜 1000개와 폴리곤 100만 개 — 어느 쪽이 문제인가?"**
별개의 병목이라 "어느 쪽이 지금 병목인가"를 측정해야 한다. 폴리곤 100만은 현대 GPU 정점 처리로는 대수롭지 않지만, DX11에서 드로우콜 1000개×상태 변경은 CPU 렌더 스레드를 수 ms 태울 수 있다. 반대로 드로우 10개에 폴리곤 1억이면 GPU 바운드다. CPU 바운드인지 GPU 바운드인지(렌더 스레드 시간 vs GPU 타임스탬프)를 먼저 가르는 게 최적화의 첫 질문이다.

**"오클루전 쿼리 기반 컬링의 고전적 함정은?"**
결과 회수가 1~2프레임 늦다는 것. 지연을 무시하고 동기로 기다리면 파이프라인 스톨, 지난 프레임 결과를 쓰면 카메라가 코너를 도는 순간 "지난 프레임엔 가려졌던" 물체가 한 프레임 늦게 나타나는 팝핑이 생긴다. 그래서 실무는 보수적 재검증(이전에 가려졌던 것만 바운딩 박스로 재확인)이나 Hi-Z 같은 지연 없는 대안으로 이동했다.

**"TAA 고스팅을 줄이는 방법을 아는 대로."**
(1) 모션 벡터로 히스토리를 정확히 재투영 — 단 그림자·반사처럼 모션 벡터가 없는 변화는 원리적으로 못 잡는다. (2) neighborhood clamp — 현재 프레임 주변 3x3의 색 범위(AABB/분산)로 히스토리를 죈다. 유효하지 않은 과거는 현재와 색이 다르다는 휴리스틱. (3) disocclusion 검출(깊이/ID 비교) 시 히스토리 가중치를 리셋. 전부 "축적량과 반응성의 트레이드오프"를 조절하는 노브이지 근본 해결이 아니다 — temporal 전략의 태생적 청구서다.

## ④ Winters에서의 적용

### (1) 프레임 아키텍처 — "포워드 + prepass + 컴퓨트 SSAO" 하이브리드

Winters는 디퍼드가 아니라 **포워드**다. `Shaders/Mesh3D.hlsl:9`의 `CBPerFrame`이 방향광 1개 + 포인트 라이트 4개를 픽셀 셰이더에 직접 공급하고(155행 `PS`에서 그 자리 라이팅), 라이트 수가 상수로 묶여 있어 라이트×오브젝트 곱셈 문제가 애초에 발생하지 않는 규모다 — LoL식 탑다운에서 합리적인 선택이다. 대신 스크린 스페이스 기법(SSAO)이 필요로 하는 depth+normal은 디퍼드처럼 공짜로 나오지 않으므로, **전용 프리패스를 돌린다**: `Client/Private/Scene/Scene_InGameRender.cpp:359`에서 `m_pNormalPass->Begin` → 맵/챔피언을 normal 전용 셰이더로 렌더 → `Scene_InGameRender.cpp:408`에서 `m_pSSAOPass->Execute(depthSRV, normalSRV, viewProj)` → 출력 AO SRV를 이후 모든 포워드 드로우에 주입한다(`Scene_InGameRender.cpp:417`, 챔피언은 `:492`의 `SetAmbientOcclusionSRV`, 미니언은 `Client/Private/Manager/Minion_Manager.cpp:839`).

폴백 계약도 명시적이다: `Scene_InGameRender.cpp:347`에서 AO SRV의 기본값은 **1x1 흰색 텍스처** — SSAO가 꺼져도 셰이더는 항상 AO를 샘플하고, "AO 없음"은 "AO=1.0(무차폐)"으로 정의된다. 분기 대신 중립 데이터로 특수 케이스를 흡수하는 패턴.

프레임 전체 순서는 이 장의 이론 그대로다: **prepass → SSAO → 불투명(맵→챔피언→구조물/미니언) → 접촉 그림자 → 전장의 안개 오버레이 → FX(반투명) → UI**. FX 3계열이 `Scene_InGameRender.cpp:614`(mesh→beam→billboard)에서 불투명 이후에 그려지고, HP바를 포함한 UI 오버레이가 그 뒤 `Scene_InGameRender.cpp:621`의 `UI_Render_Overlay`다 — "반투명은 불투명 다음, UI는 마지막"이라는 8절 원칙의 실물이다.

### (2) SSAO 구현 — 교과서(반구 커널)가 아니라 GTAO식 수평선 마칭

`Engine/Private/Renderer/SSAOPass.cpp:188`이 `Shaders/SSAO/GTAO_CS.hlsl`을 `cs_5_0` **컴퓨트 셰이더**로 컴파일하고, 8x8 스레드 그룹으로 디스패치한다(`SSAOPass.cpp:258`). 알고리즘(`Shaders/SSAO/GTAO_CS.hlsl`)은 3절의 교과서 SSAO와 대비 지점이 명확하다:

- **위치 재구성**: `GTAO_CS.hlsl:20`의 `ReconstructWorldPos`가 깊이 버퍼와 `viewProjInv`로 월드 위치를 복원한다 — "G-buffer에 위치를 저장하지 않는다"는 원칙의 실물. NDC y 뒤집기와 `w` 나눗셈까지 D3D 규약 그대로다.
- **반구 커널 대신 방향 마칭**: `GTAO_CS.hlsl:47`에서 화면 방향 8개 × 스텝 4개 = 32샘플을 방사형으로 마칭하며, 샘플 방향과 법선의 코사인(`ndotDir`)·거리 감쇠로 차폐를 적산한다(`:81`의 `horizon` 계산). 무작위 점 대신 구조적 방향 탐색 — HBAO/GTAO 계열의 접근이다.
- **thickness heuristic**: `GTAO_CS.hlsl:83`의 `(ndotDir - g_fThicknessHeuristic) / (1 - g_fThicknessHeuristic)` — 법선에 거의 접하는 방향의 미세 차폐를 잘라내는 항으로, 교과서 SSAO의 bias와 같은 역할(자기 차폐 방지)을 한다. 파라미터 기본값은 `Engine/Public/Renderer/SSAOPass.h:52`에 radius 1.25 / intensity 1.5 / thickness 0.05로 노출돼 런타임 튜닝이 가능하다.
- **교과서와 다른 점 — 노이즈 텍스처가 없다**: 방향 8개가 모든 픽셀에서 동일 위상이므로 원리상 밴딩 위험이 있다. 이를 받치는 것이 **깊이 인지 블러**다: `Shaders/SSAO/GTAO_Blur_CS.hlsl:33`에서 `spatialWeight(가우시안) * depthWeight(exp(-|Δdepth|*80))`의 곱으로 5x5 bilateral 블러 — 물체 경계를 넘는 번짐을 깊이 가중치로 차단한다. 면접 답변 포인트: "다음 개선은 per-pixel 회전(interleaved gradient noise)과 temporal 축적으로 같은 샘플 예산에서 품질을 올리는 것" — 전략 B에 C를 얹는 정석 수순이다.
- **출력은 `R16_FLOAT` 단채널**(`SSAOPass.cpp:140`), 하늘(원거리) 픽셀은 `GTAO_CS.hlsl:37`에서 AO=1로 조기 종료. AO는 스칼라 가시율이라는 정의가 포맷 선택에 그대로 반영돼 있다.

### (3) AO 적용 위치 — 물리보다 룩을 택한 타협의 자백

2절의 이론대로면 AO는 간접광 항에만 곱해야 한다. Winters는 `Shaders/Mesh3D.hlsl:123`에서 `color *= lerp(0.52f, 1.f, contactAO)` — 키 라이트까지 포함한 **최종 컬러 전체**에 곱한다. 물리적으로는 직접광 이중 감쇠지만, (1) 스타일라이즈드 램프 셰이딩이라 애초에 물리 정합이 목표가 아니고, (2) 섀도우맵이 없는 파이프라인에서 AO가 접촉 음영의 유일한 공급원이라 접지감을 최대로 뽑는 선택이다. 바닥을 0이 아닌 0.52로 막은 것도 "AO는 보정이지 조명이 아니다"라는 태도다. 면접에서 "당신 엔진의 AO는 물리적으로 맞나?"가 들어오면 이 타협을 근거와 함께 자백하는 것이 정답이다.

### (4) 그림자 — 섀도우맵이 없다. 그리고 그것이 왜 합리적인가

Winters에는 섀도우 매핑이 없다. 대신 챔피언 발밑에 **접촉 그림자(blob shadow) 쿼드**를 깐다: `Scene_InGameRender.cpp:164`의 `BuildContactShadowWorld`가 캐릭터 위치에 살짝 띄운(y+0.055) 타원 스케일(x 1.22, z는 0.72배) 평면을 만들고, `Scene_InGameRender.cpp:582`에서 챔피언마다 렌더한다. 왜 합리적인가: (1) 고정 탑다운 카메라라 그림자의 정보 가치가 "캐릭터의 접지 위치 표시"에 집중되고, 블롭이 그 역할을 픽셀당 최저 비용으로 수행한다. (2) 스타일라이즈드 룩에서 정확한 그림자 실루엣의 미적 이득이 작다. (3) 맵의 대규모 음영은 텍스처에 이미 그려져 있다(사실상 손으로 구운 라이트맵 — 전략 A). 확장한다면: 방향광 1개 고정이므로 캐스케이드 1~2장의 CSM이 첫 단계고, 프러스텀이 항상 지면을 비스듬히 보는 고정 카메라라 캐스케이드 분할·shimmering 스냅이 일반 3인칭보다 훨씬 단순해진다 — "우리 게임의 카메라 제약이 기법 비용을 얼마나 깎아 주는가"까지 말하면 설계 감각의 증명이 된다.

### (5) 컬링 — 클립 공간 8코너 판정과 서브메시 마스크

Winters의 프러스텀 컬링은 7절의 "클립 공간 정식화"다. `Engine/Private/Resource/Model.cpp:433`의 `IsBoundsVisibleInClip`이 AABB 8코너를 `local→clip` 행렬로 변환해 `-w<=x<=w` 계열 부등식으로 판정하고, 8코너가 모두 한 부등식을 위반할 때만 컬한다. 보수 처리도 이론대로다: `Model.cpp:488`에서 near 평면을 걸치는(`w<=0` 코너 포함) 박스는 무조건 보이는 것으로 수용 — 카메라 뒤 코너의 부등식 해석 붕괴를 안전 쪽으로 회피한다. 위에는 두 층이 더 있다:
- **서브메시 마스크**: `Model.cpp:724`의 `BuildClipVisibilityMask`가 모델 전체 바운딩으로 1차 기각 후, 서브메시별 바운딩으로 비트마스크를 만들어 보이는 서브메시만 그린다(`Engine/Private/Renderer/ModelRenderer.cpp:452`에서 호출). 큰 맵 메시의 일부만 화면에 걸릴 때 효과적인 계층 컬링.
- **미세 기여 컬**: `Model.cpp:773` — 서브메시가 128개 이상일 때만, NDC 폭·높이가 모두 0.0035 미만(사실상 서브픽셀)인 서브메시를 버린다. "화면 기여가 없는 일은 안 한다"를 픽셀 크기 기준으로 확장한 것. 반대로 `Model.cpp:753`은 합쳐진 정적 메시가 서브메시 임계 이상이면 판정 비용 자체가 손해라 전부 보이는 걸로 바이패스한다 — **컬링 자체의 비용도 예산이라는 자각**이 코드에 있다.
- FX는 별도의 거리 컬을 쓴다: `Client/Private/GameObject/FX/FxSystem.cpp:32`의 near/max 거리 상수로 카메라에서 너무 먼 파티클을 스킵. 오클루전 컬링은 없다 — 탑다운 고정 카메라에선 대형 오클루더 상황 자체가 드물어 우선순위가 낮다는 답이 가능하다.

### (6) 드로우콜 — 인스턴싱 배관은 있고 사용자는 없다

RHI 커맨드 리스트에는 인스턴싱 배관이 이미 있다: `Engine/Private/RHI/DX11/CDX11Device.cpp:678`이 `DrawIndexedInstanced`로 내려간다. 그러나 콜사이트는 전부 인스턴스 1개다 — `Engine/Private/Renderer/RHISceneRenderer.cpp:480`의 `DrawIndexed(mesh.indexCount, 1, ...)`가 스냅샷 아이템을 **아이템당 파이프라인 세팅 + 드로우 1회** 루프(`:417`)로 소비하고, 미니언도 `Minion_Manager.cpp:824`의 ForEach에서 엔티티마다 개별 `RenderFrustumCulled`를 호출한다. 메시 데이터는 `shared_ptr<CModel>`로 공유되지만(메모리는 배칭됨) **드로우는 개별**이다. 면접용 개선 플랜: 미니언은 "같은 메시 × 다른 트랜스폼 × 수십 마리"라 인스턴싱의 교과서적 후보이고, 스킨드라는 걸림돌은 전 인스턴스의 본 팔레트를 structured buffer 하나에 쌓고 인스턴스별 오프셋을 주는 표준 우회(③의 인스턴싱 문답)로 푼다. 그 전 단계로 `RHISceneRenderer`의 아이템 루프에 (파이프라인, 텍스처) 키 상태 정렬만 넣어도 상태 변경 횟수를 줄일 수 있다 — "측정 먼저: 현재 CPU 바운드인지 GPU 타임스탬프와 렌더 스레드 시간으로 가른 다음"이라는 단서를 붙이면 완성이다.

### (7) 반투명/FX — additive가 정렬 부재를 버텨 주는 구조

블렌드는 `Engine/Public/Renderer/BlendTypes.h:6`의 4프리셋(Opaque/AlphaBlend/PremultipliedAlpha/Additive)으로 표준화돼 있고, FX마다 `FxSystem.cpp:378`에서 자기 blendMode를 적용한다. 정렬은? `FxSystem.cpp:289`의 ForEach 순서 그대로 그린다 — **카메라 거리 back-to-front 정렬이 없다**. 이것이 버티는 이유가 8절의 이론이다: 주력 FX(마법·검기·글로우)가 additive라 순서 무관이고, alpha blend FX끼리 겹치는 경우가 드물다. 대신 겹치면 프레임 간 순서 팝이 날 수 있는 알려진 부채다. 깊이 정책은 명시적이다: `FxSystem.cpp:444`에서 런타임 FX는 `OverlayNoDepth` — 깊이 쓰기뿐 아니라 깊이 테스트까지 끄고 오버레이로 그린다(주석: "Runtime FX should not be clipped by terrain depth"). 지형 요철에 이펙트가 잘리는 사고를 막는 대신 벽 너머 FX도 비쳐 보이는 트레이드오프 — LoL식 가독성 우선의 선택이다. 공유 상태 오염 방지로 드로우 후 `FxSystem.cpp:451`에서 블렌드를 AlphaBlend로 리셋하는 것까지, 8절의 3계명 중 (1) 순서(불투명 뒤)와 (3) 깊이 쓰기 금지는 지키고 (2) 정렬만 additive에 기대 생략한 구조라고 정리해 말할 수 있다.

### (8) AA — 현재 없음, 그리고 최소 비용 진입로

스왑체인과 깊이 버퍼 모두 `SampleDesc.Count = 1`(`Engine/Private/RHI/DX11/CDX11Device.cpp:798`) — **MSAA를 포함해 하드웨어 AA가 없다**. 스타일라이즈드 아트와 탑다운 시점(얇은 서브픽셀 지오가 적음)이라 체감 열화가 작아 우선순위가 밀린 상태다. 추가한다면 순서는 명확하다: 포스트 1패스인 FXAA가 최소 비용 진입로이고, MSAA는 포워드 렌더러라 디퍼드 같은 원리적 장벽은 없지만 prepass·SSAO 등 depth를 소비하는 모든 패스가 멀티샘플 대응을 해야 해서 공사 범위가 넓다. TAA는 모션 벡터 파이프라인이 선행 투자다 — "기법의 비용은 셰이더가 아니라 파이프라인 침습 범위"라는 답변 프레임이 여기서 나온다.

## ⑤ 흔한 오답 / 함정

1. **"SSAO는 그림자 기술이다."** 아니다. 그림자는 직접광의 가시성(섀도우맵), AO는 간접광의 가시성이다. 조명 항이 다르고, 물리적으로 올바른 곱셈 위치도 다르다.
2. **"디퍼드가 포워드의 상위 호환이다."** 반투명·MSAA·머티리얼 다양성·대역폭에서 디퍼드가 손해다. 라이트가 적으면 포워드가 그냥 이긴다 — Winters가 포워드인 것은 결함이 아니라 규모에 맞는 선택이다.
3. **"드로우콜이 비싼 건 GPU 부담 때문."** 병목은 대개 CPU 드라이버(검증·상태 변환·커맨드 기록)다. DX12가 싸게 만든 방법이 "검증의 선불화(PSO)"라는 사실이 그 증거다.
4. **"반투명도 깊이 버퍼가 알아서 정렬해 준다."** 깊이 버퍼는 기각(가림)이지 정렬(섞임)이 아니다. 블렌딩 결과는 드로우 순서가 결정하며, 교차 반투명은 오브젝트 정렬로도 못 푼다.
5. **"MSAA는 픽셀 셰이더를 샘플마다 돌린다."** 반대다 — 커버리지·깊이만 멀티샘플이고 PS는 픽셀당 1회. 그래서 싸고, 그래서 셰이더 앨리어싱은 못 잡고, 그래서 샘플별 셰이딩이 필요해지는 디퍼드와 어긋난다.
6. **"PCF 대신 섀도우맵을 블러하면 된다."** 깊이의 평균은 비교의 평균이 아니다(비선형). 같은 함정이 MSAA resolve를 라이팅 전에 하는 실수, SSAO를 일반 가우시안으로 블러하는 실수에서 반복된다 — **"비선형 연산 앞에서 평균을 취하지 마라"** 하나로 묶어 기억하라.
7. **"라이트맵은 구식이다."** 정적 콘텐츠 한정으로는 지금도 품질/비용 최강이다. 실시간 GI의 가치는 품질이 아니라 동적 제약의 해제다.
