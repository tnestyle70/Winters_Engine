DX9 팀 프로젝트에서 TCP 서버와 20tick 동기화를 만들면서, 프레임워크가 정해준 구조 안에서는 틱 제어·복제 필터·치팅 검증 같은 핵심을 제 손으로 설계할 수 없다는 걸 알았습니다. 그래서 엔진의 본질을 "만드는 사람이 한 번의 빌드 안에서 반복할 수 있는 속도"로 정의하고, LoL 구조의 MOBA 하나에 최적화된 엔진을 바닥부터 만들었습니다. 지금은 로그인 → 계정(RP·전적·리플레이) → 상점 → 로비 → 밴픽 → 로딩 → 인게임(서버 권위 30Hz) → 결과·MMR 반영 → 리플레이 저장까지 한 바퀴가 전부 실제로 돌아갑니다. 그리고 이 엔진의 구현 사이클 자체를 AI 에이전트 하네스(§6)로 돌립니다 — 제가 문제를 정의하면 AI가 구현하고, 빌드·스모크·결정성 해시 게이트를 통과해야만 반영됩니다.

## 서버 권위 — 클라이언트를 신뢰하지 않는다

전작에서 드래곤 보스의 공격이 시간이 지날수록 여러 마리가 공격하는 것처럼 중첩되는 동기화 버그를 겪었고, 당시엔 원인을 닫지 못했습니다. 나중에 코드를 다시 열어 원인이 서버와 클라이언트 양쪽에 공격 연출의 권위가 있었던 것임을 확인했고, Winters는 그 사고가 불가능한 구조로 설계했습니다 — 모든 게임플레이 진실은 서버만 만들고, 클라이언트는 입력과 연출만 담당합니다. 커맨드는 단일 입구에서 시퀀스·유효성(CanMove/CanCast)을 검증받고, 봇 AI조차 같은 관문을 통과하는 GameCommand 생산자라서 진실을 직접 수정하는 경로 자체가 없습니다. 지금 챔피언 17종이 30Hz 서버 시뮬레이션 위에서 돌아갑니다.

즉시 반응(우클릭 회전)과 서버 보정이 충돌하는 문제는 예측 보호 규약(최대 12스냅샷)으로, 고핑 유저의 판정은 서버가 틱마다 기록하는 200ms 위치 히스토리 지연 보상으로 풀었습니다. 스냅샷은 수신자별로 개별 빌드되는데(ack 시퀀스·로컬 엔티티 개인화), 여기에 시야 기반 필터를 더해 "안 보이는 적은 애초에 보내지 않는" 복제로 확장하는 것이 다음 단계입니다.

## 결정성 — 검증을 기계에게 맡긴다

108개 파일을 고치는 리팩터링을 하고 나면 "동작이 안 변했다"를 사람이 확인할 방법이 없습니다. 그래서 시뮬레이션 300틱을 돌린 결과를 해시 하나로 요약해, 수술 전후 해시가 같으면 무회귀임을 기계가 증명하게 했습니다. 서버는 매 매치를 리플레이(.wrpl)로 기록하고 클라이언트는 네트워크 없이 그 스트림을 재생하는데, 이 리플레이가 버그 재현·성능 측정·관전의 공용 기반이 됩니다. 시뮬레이션을 90초까지 되감는 크로노 브레이크도 서버에 내장돼 있습니다.

## 성능 — 감이 아니라 계측으로

프레임이 느려졌을 때 코드 추론 대신 프로파일러부터 만들었습니다. 처음 Debug 빌드에서 잡은 병목 순위가 Release에서 뒤집히는 걸 직접 실측한 뒤로는, 최적화 빌드에 계측을 상시 탑재했습니다 — GPU 패스별 타임스탬프, 통합 드로우콜 카운터, 리플레이 기반 무인 캡처까지, 프로파일러 자기 오버헤드는 프레임의 1.5% 게이트 안에서. 이 파이프라인이 지목한 1위 병목은 화면 밖 캐릭터까지 매 프레임 애니메이션 포즈를 계산하는 것(프레임의 25%)이었고, 포즈 계산을 소비 지점으로 미루는 지연 평가로 바꿔 **Update 페이즈를 0.72ms에서 0.17ms(-76%)로** 줄였습니다. 같은 리플레이를 세 번 돌려 재현을 확인했고, GPU 시간과 드로우콜이 불변임을 계측으로 검증했습니다.

## 엔진 코어 — 같은 문제를 세 번 푼 이야기

Starcraft에서는 모든 객체가 CObj 상속이었습니다. 직관적이었지만 죽은 유닛을 가리키는 참조가 계속 사고를 냈습니다. 던전스에서는 COM 스타일 수동 AddRef/Release로 수명을 지켰는데, 이번엔 사람이 Release를 빼먹으면 끝이었습니다. Winters의 답은 세대(generation) 핸들입니다 — 파괴 후 재사용된 슬롯을 가리키는 낡은 참조가 사용 시점에 O(1)로 거절됩니다. 병렬화는 Chase-Lev work-stealing JobSystem으로, lock-free deque의 owner-push 규약을 위반한 race를 직접 규명해 고쳤고, Fiber 실행 모드는 10만 잡 벤치마크에서 워커 처리량 **+25%(439,389→550,902 jobs/s)** 를 실측했습니다. 렌더러는 IRHIDevice 핸들 경계 뒤에 DX11 기본 경로와 DX12 백엔드를 격리했습니다.

## 데이터와 툴 — 기획자·디자이너가 코드 없이 일하게

JSON은 무엇을 할 것인가를, C++은 그것을 어떻게 실행할 것인가를 담당합니다. 밸런스 수치는 JSON으로 저작하고 쿡 타임에 전수 검증(중복 키·범위·누락 단계 차단)해 C++ 팩으로 컴파일합니다 — 서버 런타임의 JSON 파싱은 0회이고, 85개 스킬을 "전체 계약이 유효한가"로 검사합니다. 에셋은 FBX/GLB 27종을 자체 .wmesh로 전수 변환(FAIL 0)해 60MB 모델이 1.2MB가 됐습니다. 디자이너용 이펙트 툴(FBX·텍스처 로드, 수치 조정, 인게임 프리뷰)과 챔피언·구조물 튜너는 전부 ImGui이고 Debug 게이트 뒤에 격리돼 있으며, 에디터는 클라이언트와 분리된 독립 실행 파일로 시작했습니다.

## 매치 밖 — 백엔드와 예외 경로

계정·RP·상점·전적은 전투와 소유자가 다릅니다. Go 마이크로서비스 3종(인증/프로필/상점)에 PostgreSQL·Redis·Kafka를 docker-compose로 묶어, 가입부터 챔피언 구매·매치·전적 저장까지 로컬 E2E를 완주했습니다. 그리고 행복 경로만 믿지 않기 위해 로비 이탈·픽 없이 시작·로딩 실패·서버 사망·악성 커맨드까지 예외 경로를 전수 감사했습니다 — 크래시 경로는 0이었고, 대신 침묵하는 실패들을 찾아 수복 순위를 매겼습니다. 클라이언트 게이트(UX)와 서버 게이트(보안)는 둘 다 있어야 한다는 게 이 감사의 결론입니다.

## 한계와 다음 단계

UDP 경로는 구현·검증까지 마쳤고 기본 전환 게이트가 남아 있습니다. 시스템 스케줄러의 병렬 배치 최적화가 계측이 지목한 다음 타깃이고, 데이터 버전 핸드셰이크는 로그 전용에서 접속 거부 게이트로 승격할 예정입니다.
