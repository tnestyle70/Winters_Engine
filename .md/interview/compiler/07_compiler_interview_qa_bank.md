# 컴파일러 면접 질문 은행

> 이 챕터는 개념을 다시 설명하지 않는다. 개념 설명은 이 시리즈의 앞 챕터(01~06장)에 있고, 여기는 **면접장에서 실제로 나오는 질문 → 한 줄 정답 → 파고들기(꼬리질문)**만 모은 질문 은행이다.
> 사용법: (A)로 기초를 훑고, 각 항목의 `▸` 를 스스로 답할 수 있는지 점검한다. 막히면 그게 앞 챕터로 돌아갈 지점이다. (B)는 연쇄 꼬리질문 방어용, (C)는 "경험을 말해보라"에 STAR로 답하는 골격이다.
> `cpp/02_compile_link_dll.md`와의 분업: 그쪽은 **빌드 실무 관점**(vcxproj 설정, DLL 경계 사고, 링크 에러 읽기), 여기는 **컴파일러 내부 관점**(왜 그렇게 동작하도록 설계됐는가)이다. 같은 소재(링커, mangling)가 나와도 각도가 다르다.

## A. 기초 개념 질문 (한 줄 정답 + 파고들기)

### A-1. 컴파일 파이프라인

1. **소스 파일이 실행 파일이 되기까지 4단계는?**
   한 줄: 전처리(preprocess, 텍스트 치환) → 컴파일(compile, C++→어셈블리) → 어셈블(assemble, 어셈블리→오브젝트) → 링크(link, 오브젝트들→실행 파일).

   ```text
   .cpp ──전처리──→ 펼쳐진 TU ──컴파일──→ .asm ──어셈블──→ .obj ──링크──→ .exe / .dll
        (#include 복붙,          (파싱·최적화·          (기계어 인코딩,        (심볼 해결,
         매크로 치환)             코드 생성)              심볼은 미해결)         주소 재배치)
   ```

   ▸ "각 단계를 따로 볼 수 있나?" → MSVC는 `/P`(전처리 결과), `/FA`(어셈블리 리스팅), `/c`(오브젝트까지만), GCC는 `-E`/`-S`/`-c`. "cl 한 번이면 끝"인 것은 드라이버가 단계를 이어줄 뿐, 단계 자체는 항상 존재한다.
   ▸ "왜 단계를 나눴나?" → 각 단계의 산출물이 **재사용 경계**다. `.obj`를 캐시하면 바뀐 파일만 재컴파일하는 증분 빌드가 되고, 링크만 다시 하면 된다. 단계 분리는 성능 취향이 아니라 **분리 컴파일(separate compilation)**이라는 설계 목표의 결과다.

2. **오브젝트 파일(.obj)에는 무엇이 들어 있나?**
   한 줄: 기계어와 데이터를 담은 섹션들 + "내가 정의한 심볼 / 참조만 한 심볼" 목록(심볼 테이블) + "이 자리의 주소는 나중에 채워라"라는 재배치(relocation) 항목.
   ▸ "어셈블러가 하는 일은?" → 니모닉을 기계어 비트로 인코딩하고, 같은 파일 안의 라벨은 오프셋으로 해결하고, 외부 이름은 재배치 항목으로 남긴다. 4단계 중 가장 기계적인 단계.
   ▸ "왜 .obj 상태에선 실행이 안 되나?" → 외부 심볼의 주소가 비어 있고 최종 메모리 배치가 미정이라서. 그 두 가지를 채우는 것이 정확히 링커의 일(→ A-23).

3. **전처리기는 왜 C++ 문법도 모르는 '무지한' 단계로 남아 있나?**
   한 줄: 역사적 유산 — C 초기에 별도 프로그램(텍스트 필터)으로 설계된 것이 하위 호환 때문에 그대로 남았다. 스코프·타입·네임스페이스를 모르는 순수 토큰 치환기다.
   ▸ "그 무지의 결과는?" → `#define new`가 placement new 문법을 깨고, Windows.h의 `min`/`max`가 `std::max` 호출을 파괴하는 함정 전부 (실무 방어는 `cpp/02` ④). 언어가 진화하며 매크로의 역할을 `constexpr`·템플릿·모듈이 하나씩 회수하는 중이라는 큰 흐름까지 말하면 좋다.

4. **컴파일러 내부는 어떻게 나뉘나?**
   한 줄: 프런트엔드(frontend, 언어 이해: 렉서→파서→시맨틱 분석) → 미들엔드(middle-end, IR 위 최적화) → 백엔드(backend, 타깃 기계어 생성).

   ```text
   문자열 →[렉서]→ 토큰 →[파서]→ AST →[시맨틱 분석]→ IR →[최적화 패스 N회]→ IR →[코드 생성]→ 기계어
   └────────── 프런트엔드 ──────────┘   └────── 미들엔드 ──────┘   └─── 백엔드 ───┘
   ```

   ▸ "왜 이 3분할인가?" → 언어(M개)와 타깃(N개)을 직교로 만들기 위해. 프런트는 언어만 알고 백은 CPU만 알며, 중간에서 IR이 둘을 절연한다. → A-14.
   ▸ "1번의 4단계와는 무슨 관계인가?" → 4단계는 빌드 시스템에서 보이는 **바깥 파이프라인**, 3분할은 그중 '컴파일' 한 단계의 **내부 구조**다. 두 지도를 구분해서 답하면 정리된 인상을 준다.

5. **컴파일러와 인터프리터의 차이는?**
   한 줄: 컴파일러는 실행 **전에** 프로그램 전체를 목적 언어로 번역해 두고, 인터프리터는 번역된 결과물을 남기지 않고 프로그램을 **직접 실행**한다.
   ▸ "장단점은?" → 컴파일: 실행 시 번역 비용 0, 전역 최적화 가능 / 빌드 대기 시간, 타깃별 재빌드. 인터프리터: 즉시 실행·이식성 / 매 실행마다 해석 비용.
   ▸ "현실은 이분법인가?" → 아니다, 스펙트럼이다. CPython은 바이트코드로 컴파일한 뒤 그 바이트코드를 해석하고, V8/JVM은 인터프리터로 시작해 핫 코드만 JIT 컴파일한다.

6. **JIT(just-in-time) 컴파일이란?**
   한 줄: 실행 **도중에** 자주 도는 코드를 기계어로 컴파일하는 방식 — 런타임에만 알 수 있는 정보(실제 타입, 실제 분기 방향)를 최적화에 쓸 수 있다.
   ▸ "AOT(ahead-of-time)보다 나은 점은?" → 추측 최적화(speculative optimization). "이 함수 인자는 지금까지 전부 int였다"를 근거로 특화 코드를 만들고, 가정이 깨지면 탈최적화(deoptimization)로 되돌린다. AOT는 이 정보가 없어 PGO로 흉내 낸다(→ B-5).
   ▸ "비용은?" → 웜업(초기엔 느림), 컴파일 자체의 CPU/메모리, 탈최적화 오버헤드. 게임처럼 첫 프레임부터 예측 가능한 성능이 필요한 도메인에서 C++ AOT가 선호되는 이유 중 하나.

7. **"C++은 컴파일 언어다"라는 말은 정확한가?**
   한 줄: 엄밀히는 부정확 — 컴파일/인터프리트는 언어가 아니라 **구현의 속성**이다. 다만 C++은 전처리기·템플릿·정적 타입이 번역 시점 작업을 전제하도록 설계돼 있어 사실상 컴파일 구현만 실용적이다.
   ▸ 이 한 줄을 하면 "언어 명세와 구현을 구분할 줄 아는 사람"으로 보인다. C++ 인터프리터(cling 등)도 존재하지만 내부적으로는 증분 컴파일에 가깝다.

### A-2. 프런트엔드 — 렉서 · 파서 · AST · 타입

8. **렉서(lexer, 어휘 분석기)의 역할은?**
   한 줄: 문자 스트림을 토큰(token) 스트림으로 자른다 — `int x = 42;` → `[키워드 int][식별자 x][=][정수리터럴 42][;]`.
   ▸ "왜 렉서와 파서를 분리하나?" → (1) 토큰 수준은 정규 문법(regular grammar)으로 충분해 단순·고속으로 처리 가능 (2) 공백/주석을 초기에 버려 파서 문법이 깨끗해짐 (3) 파서의 미리보기(lookahead)가 문자가 아니라 토큰 단위가 되어 문법이 단순해진다.
   ▸ "렉서가 판단하기 애매한 경우는?" → 최장 일치(maximal munch) 규칙 — `a+++b`는 `a ++ + b`로 잘린다. 규칙이 기계적이라서 사람 직관과 어긋나는 사례가 존재한다는 것까지 알면 방어가 된다.

9. **파서(parser, 구문 분석기)의 역할은?**
   한 줄: 토큰 스트림이 문법(grammar)에 맞는지 검증하면서 계층 구조(트리)를 만든다 — "이 토큰들이 곱셈식인가, 선언문인가"를 결정하는 단계.
   ▸ "재귀 하강(recursive descent) 파서란?" → 문법 규칙 하나당 함수 하나를 두고 서로 재귀 호출하는 하향식 파서. 손으로 짜기 쉽고 에러 메시지 제어가 좋아서 GCC·Clang·MSVC 같은 산업용 C/C++ 컴파일러가 실제로 손으로 짠 재귀 하강 계열을 쓴다.
   ▸ "연산자 우선순위는 어떻게 처리하나?" → 우선순위 단계마다 문법 규칙을 나누거나(`expr → term → factor`), 우선순위 등반(precedence climbing)으로 이항 연산자만 표 기반 처리. 06장 실습에서 직접 짜 보는 지점이다.

10. **AST(abstract syntax tree, 추상 구문 트리)란? 파스 트리와 뭐가 다른가?**
    한 줄: 프로그램의 의미 구조만 남긴 트리 — 파스 트리(구체 구문 트리)가 괄호·세미콜론·문법 유도 과정까지 전부 담는 것과 달리, AST는 `(a + b)`와 `a + b`를 같은 노드로 접는다.
    ▸ "AST로 뭘 하나?" → 시맨틱 분석(타입 검사, 이름 해석)의 입력이자 IR 생성의 원천. 컴파일러 외에도 포매터·정적 분석기·리팩터링 도구가 전부 AST 위에서 돈다.
    ▸ "트리인 게 왜 중요한가?" → 연산자 우선순위·결합성이 **구조로 고정**되어 이후 단계가 다시는 우선순위를 고민하지 않는다. `1 + 2 * 3`의 AST에서 `*`가 `+`의 자식인 순간, 문법 문제는 끝났다.

11. **심볼 테이블(symbol table)이란?**
    한 줄: 이름 → 속성(타입, 스코프, 저장 위치)의 매핑 — 시맨틱 분석이 "이 `x`가 어느 선언의 `x`인가"를 해석하는 자료구조.
    ▸ "스코프는 어떻게 처리하나?" → 스코프 진입 시 테이블을 쌓고 이탈 시 걷는 **테이블 스택**. 안쪽에서 바깥으로 탐색하면 가려짐(shadowing)이 자연스럽게 구현된다.
    ▸ "링커의 심볼 테이블과 같은 건가?" → 다르다. 컴파일러 내부 심볼 테이블은 번역 중 이름 해석용으로 쓰고 버려지고, 링커가 보는 심볼 테이블은 `.obj`에 남은 "정의한 심볼 / 참조만 한 심볼" 목록이다(→ A-2). C++에선 오버로딩 때문에 이름 하나가 **후보 집합**으로 매핑되고, 최종 선택(오버로드 해석)까지 끝난 결과가 장식된 이름으로 `.obj`에 박힌다.

12. **정적 타입과 동적 타입의 차이는?**
    한 줄: 정적 타입은 컴파일 시점에 모든 식의 타입이 확정·검사되고, 동적 타입은 값이 런타임에 타입 정보를 들고 다니며 연산 시점에 검사된다.
    ▸ "정적 타입이 성능에 왜 유리한가?" → 타입이 확정되면 **메모리 레이아웃이 확정**된다. 멤버 접근이 해시 조회가 아니라 고정 오프셋 load 한 개로 컴파일된다. C++에서 `obj.member`가 `mov eax, [rcx+8]` 한 줄이 되는 근거.
    ▸ "C++ 타입은 런타임에 완전히 사라지나?" → 대부분 사라진다(타입 소거). 예외가 다형적 타입의 RTTI/vtable — 런타임에 타입 정보가 필요한 지점만 선택적으로 비용을 낸다. "쓴 만큼만 낸다(pay for what you use)" 철학의 실물.

13. **C++은 왜 파싱하기 어려운 언어로 유명한가?**
    한 줄: 문맥 자유가 아니기 때문 — 같은 토큰열이 앞선 선언에 따라 다르게 파싱된다. `a * b;`는 `a`가 타입이면 포인터 선언, 변수면 곱셈식이다.
    ▸ 유명 사례: 가장 성가신 파싱(most vexing parse) — `Widget w(Gadget());`은 객체 생성이 아니라 **함수 선언**으로 파싱된다(중괄호 초기화 `Widget w{...}`가 해결책). 중첩 템플릿의 `>>`도 C++11 전까지는 시프트 연산자로 렉싱되는 함정이었다.
    ▸ 함의: 파서가 심볼 테이블(시맨틱 정보)을 참조해야 해서, 교과서적 "파싱 끝나고 시맨틱" 분리가 C++에선 성립하지 않는다. C++ 컴파일이 구조적으로 느린 이유 중 하나이고, 06장에서 작은 언어를 직접 파싱해 보면 이 고통의 크기가 실감된다.

### A-3. 미들엔드 — IR과 최적화

14. **IR(intermediate representation, 중간 표현)을 쓰는 이유는?**
    한 줄: 언어 M개 × 타깃 N개를 M×N개의 번역기 대신 M+N개(프런트 M + 백 N)로 줄이고, 최적화를 언어/타깃 중립인 한 층에 모으기 위해.

    ```text
        IR 없음: M×N 조합              IR 있음: M+N
        C++ ───→ x64                  C++  ─┐        ┌─→ x64
        C++ ───→ ARM                  Rust ─┼─→ IR ──┼─→ ARM
        Rust ──→ x64 ...              Swift ─┘        └─→ RISC-V
    ```

    ▸ "IR은 어떻게 생겼나?" → LLVM IR 기준 무한 가상 레지스터 위의 타입 있는 어셈블리 비슷한 형태이고, SSA 형식(→ B-1)이다. 소스보다 낮고 기계어보다 높은, **분석하기 위해 설계된 언어**다.
    ▸ "한 층이면 충분한가?" → 실제 컴파일러는 여러 층을 쓴다(고수준 IR에서 언어 최적화, 저수준 IR에서 기계 최적화). "IR = 관심사별 표현을 갈아타는 것"이라고 일반화해 두면 과단정을 피한다.

15. **최적화는 컴파일러의 어느 단계에서, 어떤 형태로 일어나나?**
    한 줄: 대부분 미들엔드에서 — 같은 IR을 입력받아 같은 IR을 내놓는 **패스(pass)** 수십 개의 파이프라인으로 돌고, 백엔드에서 기계 의존 마무리(명령 선택, 핍홀)가 더해진다.
    ▸ "패스 순서가 왜 중요한가?" → 최적화끼리 서로 기회를 만들어 준다 — 인라인이 먼저 돼야 상수가 전파되고, 상수가 전파돼야 분기가 접히고, 분기가 접혀야 죽은 코드가 보인다. 어떤 순서가 최적인지는 일반해가 없는 문제(phase-ordering problem)라 컴파일러마다 튜닝된 시퀀스를 쓴다.
    ▸ "O0와 O2의 차이는?" → 패스를 얼마나 돌리느냐. O0도 컴파일러가 멍청한 게 아니라 **디버깅 가능성(소스와 1:1 대응)을 보존하기로 선택**한 것이다.

16. **컴파일러 최적화 예시를 3개 들면?**
    한 줄: 상수 접기(constant folding), 죽은 코드 제거(dead code elimination), 공통 부분식 제거(common subexpression elimination).

    ```cpp
    int f(int a) {
        int x = 4 * 16;          // 상수 접기: 컴파일 시 64로 계산
        int y = a * 2 + a * 2;   // CSE: a*2 를 한 번만 계산해 재사용
        if (false) Log();        // DCE: 도달 불가 — 코드 자체를 생성 안 함
        return x + y;
    }
    ```

    ▸ "하나 더?" → 루프 불변 코드 이동(loop-invariant code motion, 루프 안에서 안 변하는 계산을 루프 밖으로), 강도 저감(strength reduction, 곱셈을 시프트/증분으로).
    ▸ "이 모든 최적화의 합법성 근거는?" → as-if rule(→ A-18). 관찰 가능한 결과가 같으면 무엇이든 바꿔도 된다.

17. **인라이닝(inlining)이란? 왜 "최적화의 어머니"라 부르나?**
    한 줄: 호출 지점에 함수 본문을 복사해 넣는 것 — 진짜 가치는 call 오버헤드 제거가 아니라 **호출 경계가 사라지면서 다른 최적화가 호출자 문맥에서 연쇄 발동**하는 것.
    ▸ 예: `Min(x, 5)`를 인라인하면 상수 5가 본문에 전파되고, 분기가 접히고, 남은 죽은 코드가 제거된다. 경계 너머로는 이 어떤 것도 불가능하다.
    ▸ "단점은?" → 코드 팽창 → 명령어 캐시(i-cache) 압박. 큰 함수를 다 인라인하면 오히려 느려질 수 있어 컴파일러가 크기/횟수 휴리스틱으로 결정한다.
    ▸ "`inline` 키워드는 인라이닝 지시인가?" → 아니다. 현대 C++에서 `inline`의 실질 의미는 ODR 완화(TU마다 정의 허용)다. 인라인 확장 여부는 옵티마이저가 독자 판단한다 — 빌드 관점 상세는 `cpp/02` 참조.

18. **as-if rule이란?**
    한 줄: 컴파일러는 프로그램의 **관찰 가능한 동작**(observable behavior)만 보존하면 코드를 어떻게 바꿔도 된다는 규칙 — 모든 최적화의 헌법.
    ▸ "관찰 가능한 동작이란?" → volatile 접근의 순서·횟수, 파일 등 외부로 나가는 I/O. 지역 변수가 레지스터에만 살다 사라지거나, 계산 순서가 바뀌거나, 루프가 통째로 접혀도 관찰 결과가 같으면 합법.
    ▸ "그래서 디버그 빌드와 뭐가 다른가?" → 최적화 빌드에서 변수가 "없어지고" 실행 순서가 소스와 다르게 보이는 것은 버그가 아니라 as-if rule의 정상 작동이다.
    ▸ "멀티스레드에선?" → as-if의 "관찰"은 단일 스레드 기준이라, 다른 스레드가 보는 순서는 atomic/메모리 오더로 **따로 계약**해야 한다. 동시성 챕터(`cpp/09`)와 만나는 지점.

19. **UB(undefined behavior, 미정의 동작)란?**
    한 줄: 표준이 아무 요구도 하지 않는 동작 — 프로그래머가 "이런 일은 일어나지 않는다"고 서명한 계약이고, 컴파일러는 그 서명을 **최적화의 가정**으로 쓴다.
    ▸ "왜 표준이 UB를 남겼나?" → (1) 검사 비용을 강제하지 않기 위해(모든 역참조에 null 검사를 넣으면 C++이 아니다) (2) 하드웨어마다 다른 동작(시프트 초과, 트랩 표현)을 하나로 정의하지 않기 위해. UB는 결함이 아니라 **성능과 이식성을 산 대가**다.
    ▸ "구현 정의(implementation-defined)와의 차이는?" → 구현 정의는 "무슨 일이 나는지 문서화된 선택지 중 하나"(예: `int` 크기), 명세되지 않은 동작(unspecified)은 "선택지 중 하나인데 문서화 의무 없음"(예: 함수 인자 평가 순서), UB는 아무 보장 없음. 3단계를 구분하면 확실히 가산점.

20. **컴파일러가 UB를 실제로 어떻게 '이용'하나?**
    한 줄: "UB는 일어나지 않는다"에서 역방향으로 추론한다 — 역참조가 이미 있었으면 그 포인터는 null이 아니라고 가정하고 이후의 null 검사를 제거한다.

    ```cpp
    int Get(int* p) {
        int v = *p;                    // 여기서 p 역참조 — UB가 아니려면 p != nullptr
        if (p == nullptr) return 0;    // 컴파일러: 항상 거짓 → 분기 제거 가능
        return v;
    }
    ```

    ▸ 이게 무서운 이유: 검사가 "작동 안 하는" 게 아니라 **조용히 사라진다**. UB는 시간역행처럼 보이는 효과(앞 줄의 UB가 뒤 줄 검사를 지움)를 낸다. signed overflow 사례는 → B-4.
    ▸ "방어 수단은?" → 새니타이저(ASan/UBSan)로 디버그에서 UB를 런타임 검출하고, 경계 입력은 검증 코드를 UB보다 **앞에** 둔다(검사가 역참조 뒤에 있으면 제거될 수 있으므로).

### A-4. 백엔드 — 호출 규약 · 스택 · vtable

21. **호출 규약(calling convention)이란? 왜 필요한가?**
    한 줄: 인자를 어디로 전달하고(레지스터/스택), 반환값을 어디서 받고, 스택 정리를 누가 하는지에 대한 합의 — 분리 컴파일된 코드끼리(다른 TU, 다른 언어, DLL 경계) 합의 없이는 호출 자체가 불가능하다.
    ▸ x64 Windows: 처음 4개 정수/포인터 인자는 RCX·RDX·R8·R9, 부동소수는 XMM0~3, 반환은 RAX, 호출자가 32바이트 그림자 공간(shadow space)을 확보한다. 규약이 사실상 하나로 통일돼 있다.
    ▸ "cdecl/stdcall은?" → 32비트 x86 시절의 분화 — cdecl은 호출자가, stdcall(Win32 API)은 피호출자가 스택을 정리했다. 규약 불일치는 스택 붕괴 크래시였다. x64에서 이 구분이 사라진 것 자체가 "규약 난립의 비용"에 대한 업계의 답이다.
    ▸ "멤버 함수 호출은 뭐가 다른가?" → 숨은 첫 인자로 `this`가 전달된다(x64에선 RCX). "멤버 함수 = this를 받는 일반 함수"라는 관점이 vtable(→ A-24)과 델리게이트 구현 이해의 기초.

22. **스택 프레임(stack frame)은 어떻게 생겼나?**
    한 줄: 함수 호출 하나가 스택에 갖는 작업 공간 — 복귀 주소, 저장된 레지스터, 지역 변수가 층을 이루고, 호출마다 쌓이고 반환마다 걷힌다.

    ```text
    높은 주소
    │  호출자의 프레임          │
    │  [5번째 이후 인자]        │  스택 전달분
    │  [shadow space 32B]      │  x64 Win: 호출자가 확보
    │  복귀 주소                │  call 명령이 push
    ├──────────────────────────┤ ← 프레임 경계
    │  저장된 비휘발 레지스터    │  프롤로그가 push
    │  지역 변수                │
    │  ...                     │ ← RSP (스택 꼭대기)
    낮은 주소  (성장 방향 ↓)
    ```

    ▸ "프롤로그/에필로그란?" → 함수 진입 시 프레임을 여는 코드(레지스터 저장, RSP 감산)와 반환 전 닫는 코드. 컴파일러가 자동 생성한다.
    ▸ "스택 할당이 힙보다 빠른 이유는?" → 할당이 RSP 감산 하나(사실상 공짜)고, 해제는 에필로그에 포함되며, 방금 쓴 프레임이라 캐시에 뜨겁다. 자유 리스트 탐색·동기화가 필요한 힙과 구조적으로 다르다.
    ▸ "재귀가 스택 오버플로를 내는 이유는?" → 호출마다 프레임이 쌓이는데 스택 크기는 고정(Windows 기본 1MB 수준)이라서. 꼬리 호출 최적화가 되면 프레임 재사용으로 안 쌓일 수도 있지만 C++은 이를 보장하지 않는다.

23. **지역 변수는 항상 스택에 있나?**
    한 줄: 아니다 — as-if rule 아래에서 주소가 관찰되지 않는 지역 변수는 **레지스터로 승격**되거나 상수로 접혀 아예 존재하지 않을 수 있다.
    ▸ "그럼 언제 스택에 강제되나?" → 주소를 찍어 넘길 때(`&x`), 레지스터가 모자랄 때(스필, → B-2), 크기가 클 때(배열). "지역 변수 = 스택"은 개념 모델이고, 실제 배치는 옵티마이저의 결정이라고 답하면 원리를 아는 것으로 보인다.

24. **컴파일러는 `virtual` 호출을 어떤 코드로 만드나?**
    한 줄: vptr 로드 → vtable 인덱싱 → 간접 call의 3단 — `p->Foo()`가 대략 `mov rax,[rcx]` / `call qword ptr [rax+슬롯오프셋]`으로 컴파일된다.
    ▸ "직접 호출과의 비용 차이는?" → 간접 분기 예측 부담 + **인라인 불가**가 본질 비용이다(→ A-17과 연결: 인라인이 막히면 연쇄 최적화가 전부 막힌다).
    ▸ "컴파일러가 이 비용을 없앨 수 있나?" → 실제 타입이 증명되면 디버추얼라이제이션(devirtualization)으로 직접 호출로 바꾼다(`final` 키워드가 이 증명을 돕는다). vtable의 메모리 구조 자체는 `cpp/13` 은행 B-1 참조 — 여기서는 "무슨 코드가 나오는가"가 포인트.

### A-5. 링킹 — 심볼 · 라이브러리 · DLL

25. **링커(linker)가 하는 일은?**
    한 줄: (1) 심볼 해결 — 각 `.obj`의 미해결 참조를 다른 `.obj`/`.lib`의 정의와 짝짓고 (2) 재배치(relocation) — 코드/데이터에 박힌 자리표시 주소를 최종 배치 주소로 고치고 (3) 섹션을 배치해 실행 파일을 만든다.
    ▸ "재배치가 왜 필요한가?" → 컴파일 시점엔 `g_Config`가 최종적으로 어느 주소에 놓일지 아무도 모른다. `.obj`는 "여기에 g_Config 주소를 채워라"라는 재배치 항목을 남기고, 주소는 링커(그리고 ASLR 환경에선 로더)가 확정한다.
    ▸ 링크 에러(LNK2019/2005)를 읽는 실무 요령은 `cpp/02` ③에 표로 정리돼 있다 — 여기서 답할 것은 "왜 그 에러가 링크 단계에서만 날 수 있는가"(TU는 서로를 모르므로 컴파일러는 참조가 해결될지 알 수 없다).

26. **정적 링킹과 동적 링킹의 차이는?**
    한 줄: 정적은 라이브러리 코드를 링크 시점에 실행 파일 안으로 **복사**, 동적은 코드를 별도 모듈(.dll/.so)에 한 벌 두고 **로드 시점에 연결**.

    | | 정적 (.lib) | 동적 (.dll) |
    |---|---|---|
    | 연결 시점 | 링크 타임 | 프로세스 로드 (또는 LoadLibrary) |
    | 코드 사본 | 사용 모듈마다 복사 | 시스템에 한 벌 |
    | 업데이트 | 전체 재링크·재배포 | DLL 교체만 (단, ABI 유지 조건) |
    | 전역 상태 | 모듈마다 별도 사본 | 한 벌 공유 |
    | 최적화 | LTO로 경계 넘는 인라인 가능 | 경계에서 최적화 단절 |

    ▸ "동적 링킹의 숨은 비용은?" → 경계에서 인라인/전역 최적화가 끊기고, 호출이 간접 호출(IAT)이 되고, 버전 불일치(DLL hell)·ABI 문제(→ B-10)가 생긴다. 그런데도 쓰는 이유: 빌드 시간 분리, 모듈 교체, 전역 상태 단일화 — Winters가 Engine을 DLL로 두는 이유도 이 축이다(→ C-1).

27. **name mangling(이름 장식)이란?**
    한 줄: 오버로딩·네임스페이스·템플릿 때문에 같은 이름의 함수가 여럿 존재할 수 있으므로, 시그니처를 인코딩해 링커 수준에서 유일한 심볼 이름을 만드는 것.
    ▸ 예: `foo(int)`가 GCC/Clang(Itanium ABI)에선 `_Z3fooi`, MSVC에선 `?foo@@YAHH@Z` 꼴 — **표준화되어 있지 않고 컴파일러 계열마다 다르다**. 이게 C++ 바이너리 호환 문제의 첫 번째 층이다(→ B-10).
    ▸ "왜 표준이 mangling을 통일하지 않았나?" → mangling만 통일해도 레이아웃·예외 모델이 다르면 링크 성공 후 침묵 사고가 난다. "이름만 맞고 내용이 다른" 최악을 막으려면 차라리 링크가 실패하는 편이 낫다는, 비호환을 **조기에 드러내는** 설계로 이해할 수 있다.
    ▸ "장식된 이름을 사람이 읽으려면?" → MSVC `undname`, GNU `c++filt`. 링크 에러의 mangled name을 풀어 "선언과 정의의 시그니처가 미묘하게 다르구나"를 잡는 게 실전 용도.

28. **`extern "C"`는 무엇을 하나?**
    한 줄: 그 선언의 심볼을 C++ 장식 없이 C 언어 규칙(사실상 이름 그대로)으로 내보내게 한다 — C ABI로 심볼을 노출하는 스위치.
    ▸ "대가는?" → 장식이 없으니 **오버로딩 불가**(같은 이름 하나만), C에 없는 타입(참조, 클래스 값 전달)을 시그니처에 쓰면 경계 의미가 흐려진다.
    ▸ "언제 쓰나?" → C 라이브러리 헤더를 C++에서 소비할 때(`extern "C" { #include ... }`), 그리고 컴파일러가 달라도 링크되어야 하는 경계 — DLL의 `GetProcAddress` 대상 함수가 대표. Winters는 `Windows.h` 오염 없이 Win32 심볼만 쓰려고 `extern "C" __declspec(dllimport)` 재선언을 활용한다 (`Shared/GameSim/Core/Debug/SimDebugOutput.h:22`) — "선언은 헤더가 아니어도 되고, 심볼만 맞으면 링커가 잇는다"의 응용.

29. **DLL 함수 호출은 실제로 어떻게 일어나나? IAT란?**
    한 줄: 호출부는 IAT(import address table, 임포트 주소 테이블)의 슬롯을 통한 **간접 호출**로 컴파일되고, 로더가 프로세스 시작 시 DLL을 매핑한 뒤 실제 함수 주소를 그 슬롯에 써 넣는다.

    ```text
    Client.exe 코드
      call qword ptr [__imp_CreateEntity]   ; IAT 슬롯 간접 호출
            │
            ▼
      IAT (.idata 섹션)   ← 로더가 로드 시점에 실제 주소 기록
            │
            ▼
      WintersEngine.dll 안의 CreateEntity 본체
    ```

    ▸ "그래서 `__declspec(dllimport)`가 하는 일이?" → 컴파일러에게 "이 심볼은 IAT 너머에 있다"고 미리 알려 `__imp_` 간접 호출로 곧장 컴파일하게 한다. 링크가 되는 것은 DLL이 아니라 임포트 라이브러리(import lib)라는 실무 함정은 `cpp/02` ④ 참조.
    ▸ "정적 링크 함수 호출과의 차이 한 줄?" → 직접 call 한 개 vs 메모리에서 주소 읽는 간접 call — 링크 시점이 로드 타임으로 미뤄진 대가가 간접 참조 한 겹이다.

30. **암시적 로딩과 명시적 로딩(LoadLibrary)의 차이는?**
    한 줄: 암시적은 임포트 테이블에 기록돼 로더가 프로세스 시작 시 전부 연결(하나라도 없으면 실행 자체가 실패), 명시적은 런타임에 `LoadLibrary`+`GetProcAddress`로 원하는 시점에 연결.
    ▸ "명시적 로딩은 언제 쓰나?" → 플러그인 아키텍처, 선택적 기능(있으면 쓰고 없으면 폴백). 이때 `GetProcAddress`는 문자열로 심볼을 찾으므로 mangling 없는 `extern "C"` 이름이 사실상 강제된다 — A-27·28이 여기서 실무와 만난다.

## B. 심화 꼬리질문 방어 (연쇄 시나리오)

> 형식: A절 답변을 들은 면접관이 이어서 파고드는 연쇄를 그대로 시뮬레이션한다. 화살표(→)가 실제 면접의 턴 교대다. 한 방에 무너지지 않는 게 목표다.

1. **SSA와 phi — "IR이 SSA 형식이라고 했는데, 그게 뭔가?"**
   → SSA(static single assignment, 정적 단일 대입)는 **모든 변수가 정확히 한 번만 대입되는** IR 형식이다. `x = 1; x = x + 1;`은 `x1 = 1; x2 = x1 + 1;`로 버전이 갈린다. 이득: 어떤 값이 어디서 왔는지(def-use)가 이름만으로 확정되어 상수 전파·죽은 코드 제거 같은 분석이 극적으로 단순해진다.
   → *"if 양 갈래에서 다른 버전이 합류하면?"* — 그래서 합류 블록에 "어느 선행 블록에서 왔는지에 따라 값을 고르는" **phi 노드**를 둔다:

   ```llvm
   define i32 @max(i32 %a, i32 %b) {
   entry:
     %cmp = icmp sgt i32 %a, %b
     br i1 %cmp, label %then, label %else
   then:
     br label %merge
   else:
     br label %merge
   merge:
     %r = phi i32 [ %a, %then ], [ %b, %else ]
     ret i32 %r
   }
   ```

   → *"phi는 실제 CPU 명령인가?"* — 아니다. 기계어엔 phi가 없으므로 백엔드가 SSA를 해제하며 선행 블록 끝의 복사(mov)로 낮춘다. phi는 실행 장치가 아니라 **분석을 위한 표기 장치**다.

2. **레지스터 할당 — "무한 가상 레지스터를 진짜 레지스터에 어떻게 욱여넣나?"**
   → x64의 범용 정수 레지스터는 16개뿐이다. 핵심 통찰: **동시에 살아있는(live) 값들만** 서로 다른 레지스터가 필요하다. 생존 구간(live range)이 겹치는 변수끼리 간섭 그래프를 만들고 그래프 컬러링으로 배정하는 것이 고전(최적해는 NP-hard라 휴리스틱), 컴파일 속도가 생명인 JIT는 정렬된 생존 구간을 한 번 훑는 선형 스캔(linear scan)을 쓴다.
   → *"레지스터가 모자라면?"* — 일부 값을 스택으로 쫓아낸다(spill). 최적화 빌드에서도 스택 접근이 남는 이유이고, 핫 루프의 스필은 곧 성능 저하다.
   → *"프로그래머가 도울 방법은?"* — 핫 루프에서 동시에 살아있는 변수 수를 줄이는 것(거대 함수 쪼개기, 루프 안 임시 줄이기). `register` 키워드는 오래전에 의미를 잃었고, 신뢰할 것은 키워드가 아니라 생존 구간이다.

3. **strict aliasing — "그 규칙이 뭐고, 왜 존재하나?"**
   → 서로 다른 타입의 lvalue로 같은 메모리를 접근하면(몇 가지 예외 빼고) UB라는 규칙. 존재 이유는 **별칭 분석(alias analysis)의 비용**이다: `void f(float* a, int* b)`에서 규칙 덕에 컴파일러는 `*a`와 `*b`가 같은 메모리일 리 없다고 가정하고, `*b` 쓰기 후에도 `*a`를 레지스터에 캐시한 채 재로드를 생략할 수 있다. 규칙이 없으면 모든 쓰기마다 모든 캐시된 로드를 버려야 한다.

   ```cpp
   float f = 1.f;
   int i1 = *reinterpret_cast<int*>(&f);    // strict aliasing 위반 — UB
   int i2; std::memcpy(&i2, &f, sizeof i2); // 정공법 — 컴파일러가 mov 수준으로 최적화
   ```

   → *"예외는?"* — `char`/`unsigned char`/`std::byte`로의 접근(바이트 뷰는 항상 허용), 호환 타입. C++20 `std::bit_cast`가 memcpy 관용구의 공식화다.
   → *"그런데 왜 위반 코드가 잘 도나?"* — GCC/Clang은 이 규칙을 최적화에 적극 활용하지만 MSVC는 전통적으로 보수적이다. "내 컴파일러에서 됐다"가 규칙 준수의 증거가 아닌 대표 사례 — 규칙의 근거(표준)와 관찰(특정 구현)을 구분해 답하면 강하다.

4. **signed overflow — "오버플로가 UB인 게 최적화랑 무슨 상관인가?"**
   → `int`가 오버플로하지 않는다고 가정하면 `x + 1 > x`를 무조건 참으로 접을 수 있고, 루프의 반복 횟수를 대수적으로 계산(트립 카운트 추론)해 벡터화·언롤링을 할 수 있다. unsigned는 wrap-around가 **정의된 동작**이라 이 가정을 못 쓴다 — "unsigned가 더 안전하다"는 통념과 반대로, 옵티마이저에겐 signed가 더 강한 약속이다.
   → *"실제로 물리는 사례는?"* — `for (int i = 0; i <= n; ++i)`에서 `n == INT_MAX`면 수학적으로 종료 불가인데, 컴파일러는 "오버플로는 UB니까 일어나지 않는다 = 루프는 종료한다"로 추론할 수 있다. UB 기반 추론이 소스의 순진한 의도를 배신하는 순간.
   → *"그럼 wrap이 필요하면?"* — unsigned로 계산한다(해시, 시퀀스 번호). Winters의 네트워크 시퀀스 비교가 unsigned 뺄셈 관용구인 이유가 이것(`cpp/13` 은행 B-22).

5. **LTO/PGO — "링크 타임 최적화와 프로파일 기반 최적화를 설명해보라."**
   → **LTO**(link-time optimization): 컴파일 단계에서 기계어 대신 IR을 `.obj`에 담아 두고, 링크 시점에 프로그램 전체 IR을 놓고 최적화한다(MSVC `/GL` + `/LTCG`). 핵심 이득은 **TU 경계를 넘는 인라인** — 분리 컴파일이 만든 최적화 단절을 링크 타임에 복구한다. 비용은 링크 시간 폭증과 디버깅 난도.
   → *"PGO는?"* — 계측 빌드 → 대표 시나리오 실행으로 프로파일 수집 → 그 데이터로 재최적화. 분기 방향에 따른 코드 배치(핫 경로를 fall-through로), 핫/콜드 분리, 인라인 판단 개선, 가상 호출의 추측적 디버추얼라이제이션까지 — JIT가 공짜로 얻는 런타임 정보를 AOT가 오프라인으로 사는 방법이다(→ A-6과 대칭).
   → *"PGO의 함정은?"* — 프로파일이 실제 워크로드를 대표하지 않으면 엉뚱한 경로가 최적화된다. 게임이라면 "타이틀 화면만 돌린 프로파일"로 전투 코드를 배치하는 꼴. 대표 시나리오 선정이 기술이 아니라 도메인 지식 문제라는 것까지 말하면 좋다.

6. **템플릿과 컴파일 시간 — "템플릿이 왜 빌드를 느리게 하나?"**
   → 템플릿은 코드가 아니라 **코드 생성 레시피**고, 인스턴스화는 사용하는 TU에서 일어난다. `vector<int>`를 쓰는 TU가 100개면 같은 인스턴스화·시맨틱 검사를 100번 반복하고, 결과물 100벌은 링커가 COMDAT(중복 허용 섹션) 병합으로 하나만 남긴다 — 즉 **99번의 작업은 순수 낭비**인데, 분리 컴파일 모델상 각 TU는 서로의 작업을 모르므로 구조적으로 피할 수 없다.
   → *"완화책은?"* — `extern template`(이 TU에서 인스턴스화 금지 선언) + 한 TU에서 명시적 인스턴스화, PCH, C++20 모듈. 이것들이 전부 "반복 작업의 공유 캐시를 만든다"는 한 아이디어의 변형임을 짚으면 나열이 아니라 이해로 보인다.
   → *"그 낭비가 바이너리도 키우나?"* — 링커가 병합하므로 동일 인스턴스의 최종 중복은 제한적이나, 서로 다른 T마다 코드가 진짜로 늘어나는 코드 팽창은 별개 문제다. `vector<int>`와 `vector<float>`는 병합 대상이 아니다.

7. **헤더 온리 — "헤더 온리 라이브러리는 왜 편하고 왜 비싼가?"**
   → 편한 이유: 빌드 산출물 배포가 없다 — include만 하면 끝, 링크 설정 불필요. 비싼 이유: (1) 모든 정의가 포함되는 **모든 TU에서 파싱·인스턴스화**된다(→ B-6의 낭비가 라이브러리 전체 규모로) (2) 구현 한 줄 수정이 사용자 전체 재컴파일로 번진다(컴파일 방화벽 부재) (3) 전부 `inline`이라 ODR "동일 정의" 요구가 라이브러리 전체에 걸린다 — 매크로 상태가 TU마다 다르면 → B-8의 조용한 ODR 위반으로 직결.
   → *"그럼 왜들 헤더 온리로 배포하나?"* — 트레이드오프 한 줄: **배포 비용을 컴파일 시간과 ODR 위험으로 치환**한 것이다. 템플릿 라이브러리는 어차피 정의가 헤더에 있어야 하니 합리적 선택이고, non-template까지 헤더 온리로 가는 것은 편의의 선택. Winters가 nlohmann json(헤더 온리)을 쓰면서 매크로 격리로 방어하는 것이 이 비용의 실물이다(`cpp/02` ④).

8. **조용한 ODR 위반 — "ODR 위반이 링커에 안 잡히는 경우가 있나?"**
   → 있다, 그리고 그쪽이 더 무섭다. non-inline 심볼 중복은 LNK2005로 잡히지만, **inline/템플릿**은 정의가 TU마다 있는 게 합법이라 링커가 COMDAT 더미에서 "아무거나 하나" 고르고 나머지를 버린다. 이때 정의들이 사실 달랐다면(진단 의무 없음, ill-formed NDR) 조용히 잘못된 조합이 링크된다.
   → *"실제로 어떻게 달라지나?"* — 같은 헤더가 TU마다 다른 매크로 상태로 컴파일될 때: 한쪽은 `NDEBUG`, 한쪽은 아닐 때 assert 포함 인라인 함수가 두 벌. 더 나쁜 건 `#pragma pack` 상태가 다른 채 같은 구조체를 본 두 TU — **같은 타입인데 레이아웃이 다른** 최악의 형태가 된다.
   → *"방어는?"* — 구성 매크로를 vcxproj 수준에서 전역 통일, pack 지시는 push/pop으로 반드시 닫기, 경계 구조체는 `static_assert(sizeof...)`로 레이아웃을 못 박기. Winters wire 헤더의 pack+static_assert가 이 방어의 실물(`cpp/13` 은행 B-17).

9. **예외 비용 — "C++ 예외는 '제로 코스트'라던데 무슨 뜻인가?"**
   → 현대 모델(x64 포함)은 **테이블 기반**이다: "이 PC 범위에서 예외가 나면 어디로 풀어라"라는 언와인드 정보를 코드 옆 테이블에 두고, 정상 경로에는 예외 관련 명령을 넣지 않는다. 그래서 **안 던지면 명령어 수 비용이 거의 0**, 대신 던지는 순간은 테이블 탐색+스택 언와인딩으로 매우 비싸다 — 예외는 예외적일 때만 싼 모델이다.
   → *"'거의' 0이라고 했는데?"* — 정직한 유보 두 개: (1) 언와인드 테이블만큼 바이너리가 커진다 (2) "여기서 예외가 샐 수 있다"는 가능성 자체가 옵티마이저의 코드 이동/재배치를 제약한다. `noexcept`가 이 제약과 테이블을 줄여주는 최적화 계약인 이유(무버 `noexcept`와 vector 폴백은 `cpp/13` A-8).
   → *"항상 이 모델이었나?"* — 아니다. 과거 32비트 MSVC는 프레임마다 핸들러를 링크드 리스트로 등록하는 방식이라 정상 경로에도 비용이 있었다. "제로 코스트는 아키텍처/모델 의존 서술"임을 아는 게 이 질문의 진짜 방어선이다.

10. **C++ ABI — "컴파일러가 다르면 바이너리가 왜 안 섞이나?"**
    → ABI(application binary interface)는 겹겹의 합의다: ① name mangling 방식(MSVC vs Itanium — 심볼 이름부터 다르니 링크부터 실패) ② 클래스 레이아웃(vtable 배치, 가상 상속 구현, 멤버 정렬) ③ 호출 규약과 인자 전달 세부(어떤 구조체를 레지스터로 넘기나) ④ 예외 전파 모델 ⑤ 표준 라이브러리 구현(같은 `std::string`이라도 내부가 다름). 하나만 어긋나도 링크 실패면 다행이고, 링크되면 힙 손상 같은 침묵 사고가 된다.
    → *"그래서 표준은 왜 ABI를 정의 안 하나?"* — ABI를 못 박으면 구현 개선(더 나은 레이아웃, 더 빠른 string)이 영구히 봉인된다. C++ 위원회가 ABI 안정 vs 진화 사이에서 진화를 골라온 대가를 사용자가 "경계에선 C ABI로 내려가라"로 지불하는 구도다.
    → *"실무 답은?"* — 모듈 경계를 컴파일러 독립으로 만들려면 C ABI(extern "C" + POD)나 COM류 인터페이스로 내려간다. 같은 MSVC끼리는 VS2015 이후 v14x 툴셋 간 조건부 바이너리 호환이 문서로 보장되지만, 그 전이나 GCC/Clang↔MSVC 간엔 보장이 없다. Winters가 "전 모듈 동일 툴체인·동일 CRT" 규율 위에서만 STL을 DLL 경계로 넘기는 이유의 원리 층(`cpp/02` ③).

## C. 경험 서술형 (STAR 골격)

> 형식: **S**(상황) **T**(과제) **A**(행동) **R**(결과) + 이어질 꼬리질문. Winters 실제 코드와 06장 직접 만들기 실습이 스토리 소재다. 인용은 실제 파일에서 확인한 `path:line`이다.

### C-1. "모듈 경계를 설계해본 경험" — Winters DLL 경계

**주제: 엔진을 DLL로 분리하며 컴파일러/링커의 한계를 경계 설계로 흡수**
`Engine/Include/WintersAPI.h:11`

- **S**: 엔진(DLL)과 클라이언트/서버(EXE)를 분리하면서, "무엇이 경계를 넘을 수 있는가"를 컴파일러 수준에서 결정해야 했다.
- **T**: 같은 헤더를 DLL 제작자·소비자·정적 링크 툴이 공유하면서도 심볼/ABI가 안전한 표면 설계.
- **A**:
  1. export/import/static 3-way 매크로로 헤더 하나가 세 소비자에게 다르게 컴파일되게 함 (`Engine/Include/WintersAPI.h:11`~`:17`). 근거는 A-29의 원리 — dllimport를 미리 알려야 IAT 간접 호출로 곧장 컴파일된다.
  2. 템플릿은 DLL 빌드 시점에 인스턴스가 없어 export 불가라는 원리(→ B-6)에 따라, `CWorld`를 non-template 멤버만 export하고 템플릿 멤버는 헤더 본문으로 소비자 TU 인스턴스화로 갈랐다 (`Engine/Public/ECS/World.h:58`~`:60`, 템플릿 본문 `:77`).
  3. MSVC가 dllexport 클래스의 특수 멤버를 즉시 인스턴스화하다 `unique_ptr`의 deleted copy에 걸려 빌드가 깨진 사고를 겪고, 복사 명시 delete를 규약으로 박제 (`Engine/Public/ECS/SystemScheduler.h:17`~`:25`).
  4. 컴파일러가 강제하지 못하는 레이어 규칙(Shared는 Engine 헤더 include 금지)은 텍스트 lint 스크립트를 빌드에 걸어 기계 강제 (`Tools/Harness/Check-SharedBoundary.ps1:25`).
- **R**: 경계 규칙이 문서가 아니라 매크로·include 경로·lint로 강제되는 구조. "링커와 로더가 어떻게 동작하는지 알아야 경계를 설계할 수 있다"를 몸으로 배움.
- ▸ 꼬리질문: "왜 템플릿은 export가 안 되나?" → 인스턴스화 주체가 소비자 TU라서(→ B-6). "extern C는 어디 썼나?" → 헤더 오염 없이 Win32 심볼만 재선언하는 데 (`Shared/GameSim/Core/Debug/SimDebugOutput.h:22`) — 선언과 정의(링킹)의 분리를 무기로 쓴 사례. "STL을 경계로 넘겨도 되나?" → B-10의 전제(동일 툴체인·동일 CRT)를 먼저 말하고, 그 전제가 깨지는 순간의 침묵 사고까지 이어 말한다.

### C-2. "컴파일 옵션이 정합성을 결정한 경험" — /fp:precise

**주제: 서버 권위 시뮬레이션의 결정론을 위해 부동소수 모델을 전 프로젝트 통일**
`Server/Include/Server.vcxproj:56`

- **S**: 서버 권위(server-authoritative) + 클라이언트 예측 구조에서 서버와 클라이언트가 같은 시뮬 코드를 돌리는데, 부동소수 결과가 어긋나면 예측이 계속 틀려 보정 스냅이 생긴다.
- **T**: 같은 소스가 어느 프로젝트에서 컴파일되든 같은 부동소수 결과를 내게 만들기.
- **A**:
  1. `/fp:fast`는 as-if rule의 부동소수 예외를 여는 스위치임을 확인 — 재결합(reassociation)과 축약(contraction)을 허용하면 `(a+b)+c`가 `a+(b+c)`로 바뀔 수 있는데, 부동소수 덧셈은 결합법칙이 성립하지 않으므로 **연산 순서가 곧 결과**다. "최적화가 정답을 바꾼다"가 아니라 "fast 모드는 정답의 정의를 느슨하게 하는 데 동의한 것"이라고 정리했다.
  2. Server/Client/Engine/Shared 시뮬 경계 전부를 `<FloatingPointModel>Precise</FloatingPointModel>`로 강제하고 (`Server/Include/Server.vcxproj:56`), fast 회귀를 grep 가드로 차단하는 것을 마이그레이션 계획에 포함했다.
  3. 표준 `<random>`의 분포(distribution)는 구현 정의라 플랫폼/버전 간 결과가 다를 수 있어, 난수는 정수 xorshift를 직접 구현해 컴파일러/라이브러리 의존을 제거 (`Shared/GameSim/Core/Determinism/DeterministicRng.h:15`) — 정수 연산은 부동소수와 달리 이 계열 문제가 없다는 것이 선택 근거.
- **R**: "결정론은 알고리즘이 아니라 빌드 설정에서 시작한다"는 결론. 부동소수 드리프트라는 재현 불가 버그 계열을 옵션 수준에서 예방.
- ▸ 꼬리질문: "precise면 두 다른 CPU에서도 비트 동일한가?" → 아니다 — 같은 옵션이라도 컴파일러 버전·아키텍처·초월함수 라이브러리가 다르면 어긋날 수 있어, 내 보장 범위는 "동일 툴체인으로 빌드한 서버/클라이언트 간"으로 한정했다고 답해야 과단정을 피한다. 완전한 크로스 플랫폼 비트 결정론은 고정소수점이나 소프트 float의 영역. "fast로 얻을 성능을 포기한 건가?" → 시뮬 경로만 precise면 되고, 렌더/FX처럼 결과가 회신되지 않는 경로는 별도 판단이 가능하다 — 옵션을 프로젝트(모듈) 단위로 가른 것 자체가 설계였다고 답한다.

### C-3. "컴파일러 원리를 직접 확인해본 경험" — 06장 미니 컴파일러 실습

**주제: 렉서→파서→AST→평가/코드 생성을 직접 구현해 파이프라인을 체득**
(06장 실습을 마친 뒤 자기 결과물로 채워 말하는 골격 — 완주 전엔 이 스토리를 면접에서 쓰지 말 것)

- **S**: 엔진을 만들며 컴파일러를 매일 사용하지만, 파이프라인을 소비자로만 알았다. 원리 수준 답변을 위해 작은 언어를 직접 구현했다.
- **T**: 산술식+변수+분기 수준의 미니 언어로 4단계(토큰화→파싱→AST→실행)를 전부 손으로 짜기.
- **A**:
  1. 렉서 — 문자 스트림을 토큰으로: "왜 렉서를 분리하는가"(공백/주석 처리를 파서에서 치우기, → A-8)를 체감.
  2. 파서 — 재귀 하강으로 우선순위를 문법 계층(`expr → term → factor`)으로 인코딩: 우선순위가 별도 데이터가 아니라 **코드 구조 자체**가 되는 것을 확인(→ A-9).
  3. AST 평가기 — 트리 워킹 인터프리터를 먼저 만들어 의미(semantics)를 실행 가능하게 고정.
  4. 같은 AST에서 다른 출력(스택 머신 명령이든 어셈블리 텍스트든)을 뽑아 "프런트/백 분리와 IR의 의미"(→ A-14)를 확인 — 평가기와 코드 생성기가 같은 트리를 소비한다는 사실이 3분할 구조의 미니어처다.
- **R**: "컴파일 에러 메시지가 왜 그 지점에서 그렇게 나는지"를 만든 사람의 눈으로 설명 가능해짐. 특히 파서를 짜 보면 C++의 문맥 의존 파싱(→ A-13)이 왜 고통인지 실감된다.
- ▸ 꼬리질문 대비: "어디까지 만들었나?"에는 실제 완주 범위만 말한다 — 과장하면 "그럼 클로저는 어떻게 구현했나" 같은 후속에서 무너진다. 범위가 작아도 "작은 것을 끝까지"가 미완의 큰 것보다 면접에서 강하다. "왜 만들어봤나?"에는 취미가 아니라 **도구의 원리를 알아야 도구의 실패(링크 에러, 최적화 이상 동작)를 진단할 수 있어서**라고 실무 동기로 연결한다.

### C-4. "빌드 시간을 다뤄본 경험" — 컴파일 방화벽과 PCH 규율

**주제: 헤더 수정의 재컴파일 파급을 구조로 통제**
`Engine/Public/WintersPCH.h:2`

- **S**: 엔진/클라이언트/서버가 헤더를 넓게 공유하면서, 헤더 하나 수정이 대규모 재컴파일로 번지는 구조적 위험이 있었다.
- **T**: 컴파일 시간의 원인(모든 TU가 모든 헤더를 반복 파싱 + 템플릿 반복 인스턴스화, → B-6·7)을 구조로 통제하기.
- **A**:
  1. PCH에 "절대 변경되지 않는 외부 헤더만, 엔진 내부 헤더는 절대 금지"를 규율 주석으로 박제 (`Engine/Public/WintersPCH.h:2`~`:4`) — 내부 헤더가 PCH에 들어가는 순간 그 헤더 수정 = PCH 무효화 = 전체 리빌드라는 원리 때문.
  2. 경계 헤더는 전방 선언+포인터로 완전 타입 요구를 끊어(컴파일 방화벽) DX11/Windows 헤더가 소비자 TU로 전파되는 것을 차단.
  3. 배포 헤더는 PCH 주입 이름에 기대지 않는 자기 완결(self-contained)로 작성해 PCH 없는 소비 환경에서도 컴파일되게 유지.
- **R**: "빌드 시간은 툴 문제이기 전에 include 그래프 설계 문제"라는 관점 확립. 헤더 수정의 파급 반경이 의존 구조에서 읽히게 됨.
- ▸ 꼬리질문: "왜 다 헤더에 안 두나(헤더 온리)?" → B-7의 트레이드오프로 답한다 — 배포 편의를 컴파일 시간과 ODR 위험으로 치환하는 선택이고, 엔진처럼 매일 재빌드하는 코드에선 손해다. "모듈(C++20)은?" → 텍스트 include의 반복 파싱을 근본 해결하는 방향이지만, 빌드 시스템/도구 성숙을 지켜보는 중이라고 현재형으로 답한다. "측정은 했나?" → 측정 없는 최적화 주장은 면접에서 자충수다 — 측정한 것(리빌드 파급 범위)과 측정하지 않은 것(절대 시간 수치)을 구분해 말한다.
