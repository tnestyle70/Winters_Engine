# 직접 만들며 배우는 컴파일러 — 실습 로드맵

> 컴파일러 파이프라인을 "책으로 아는 사람"과 "각 단계를 직접 짜본 사람"의 차이는, 면접에서 꼬리질문 두 개면 드러난다. 이 챕터는 6주 안에 후자가 되기 위한 실행 계획이다 — 이론 챕터가 아니라 체크리스트다.

---

## ① 목표와 얻는 것

### 완주 시 도달하는 상태

앞 챕터들이 "컴파일러가 무엇을 하는가"를 설명했다면, 이 챕터의 목표는 그 지식의 **소유권을 바꾸는 것**이다. 완주하면 다음 질문들에 "책에서 봤다"가 아니라 "제가 짠 코드가 그걸 합니다"로 답하게 된다.

- "연산자 우선순위는 파서가 어떻게 처리하나요?" → 내가 짠 문법 규칙의 계층이 곧 우선순위였다.
- "지역 변수는 메모리 어디에 사나요?" → 내 코드젠이 RBP 상대 오프셋을 직접 배정했다.
- "함수 호출 시 스택에 무슨 일이 일어나나요?" → 호출 규약(calling convention)을 지키지 않아 크래시를 내봤고, 고쳐봤다.
- "if문은 기계어로 뭐가 되나요?" → 조건 분기와 라벨로 내가 직접 번역했다.

핵심은 **실패 경험의 확보**다. 스택 정렬을 어겨 `printf`가 죽는 것을 직접 목격한 사람의 답변에는, 교과서 요약에는 없는 디테일(어디서 죽는지, 왜 죽는지, 어떻게 알아챘는지)이 붙는다. 면접관이 검증하려는 게 바로 그 디테일이다.

### 2트랙 구성과 추천 순서

| | 트랙 A — Crafting Interpreters | 트랙 B — chibicc 방식 C 컴파일러 |
|---|---|---|
| 저자/출처 | Robert Nystrom, craftinginterpreters.com (전문 무료 공개) | Rui Ueyama, chibicc 저장소 + 온라인 입문서(compilerbook) |
| 만드는 것 | Lox 언어 인터프리터 → 바이트코드 VM | C 부분집합 → x86-64 어셈블리 컴파일러 |
| 배우는 것 | 파싱, 스코프, 클로저(closure), VM 설계, GC | 토크나이저, 재귀 하강, 스택 프레임, 호출 규약, 포인터의 실체 |
| 기간 | 파트당 2~3주 | 6주 |
| 도착지 | 언어 위쪽 (설계·런타임) | 기계 아래쪽 (어셈블리·ABI) |

**추천: 트랙 B부터.** 이유는 세 가지다.

1. **본업과의 거리.** 게임 개발자의 일상은 기계 쪽에 있다 — 크래시 덤프의 콜스택, 디스어셈블리 창, 캐시와 정렬. 트랙 B는 그 세계의 생성 과정을 보여주므로 배운 것이 즉시 본업 이해로 환류된다.
2. **면접 배당률.** 게임회사 기술면접의 단골("스택 프레임", "호출 규약", "포인터와 배열")은 전부 트랙 B의 산출물이다. 트랙 A의 산출물(클로저, GC)은 좋은 심화 답변이지만 1차 관문은 아니다.
3. **동기 설계.** chibicc 방식은 첫날부터 "진짜 실행 파일"이 나온다. 전통적 컴파일러 수업이 3주째 파서만 붙잡다 이탈자를 내는 것과 정반대의 커리큘럼 설계다(아래 ③에서 상술).

트랙 A는 B 완주 후에 진행한다. 특히 파트 후반(바이트코드 VM)은 스킬 스크립트 DSL(domain-specific language)이라는 게임 도구 방향과 직결된다(④에서 연결).

## ② 환경 세팅 — Windows 11 기준

### 왜 WSL인가 (그냥 MSVC로 하면 안 되나?)

chibicc 계열 튜토리얼은 **어셈블리 텍스트를 출력하고 gcc에게 어셈블+링크를 맡기는** 구조다. 이 구조가 성립하려면:

- GNU 어셈블러(as)가 받아주는 어셈블리 문법이어야 하고,
- 호출 규약이 System V AMD64(리눅스 표준)여야 하며,
- 테스트 스크립트(`test.sh`)가 셸에서 돌아야 한다.

MSVC 생태계는 세 가지 모두 다르다(ml64 문법, Windows x64 호출 규약, 배치/PowerShell). 튜토리얼과 싸우느라 컴파일러 공부가 밀리는 걸 피하려면 WSL(Windows Subsystem for Linux)이 가장 마찰이 적다. 그리고 이 "두 세계의 호출 규약이 다르다"는 사실 자체가 W5에서 면접 자산이 된다.

### 설치 절차

PowerShell(관리자)에서:

```powershell
wsl --install -d Ubuntu-24.04
```

재부팅 후 Ubuntu 초기 사용자 생성. WSL 셸 안에서:

```bash
sudo apt update
sudo apt install -y build-essential gdb git make
cc --version   # gcc 버전이 나오면 성공
```

에디터는 VS Code + WSL 확장(Remote 계열)을 붙이면 Windows에서 편집하고 리눅스에서 빌드하는 구조가 바로 잡힌다. 프로젝트는 WSL 파일시스템 쪽(`~/mycc` 등)에 두는 편이 빌드 I/O가 빠르다.

### 최초 스모크 테스트

환경이 됐는지 확인하는 가장 빠른 방법: **손으로 쓴 어셈블리를 실행 파일로 만들어본다.** 이것이 곧 W1에서 내 컴파일러가 출력해야 할 목표물이다.

```asm
# tmp.s — "42를 반환하는 프로그램"
.intel_syntax noprefix
.globl main
main:
        mov rax, 42
        ret
```

```bash
cc -o tmp tmp.s && ./tmp; echo $?   # 42가 찍히면 환경 완성
```

여기서 두 가지 설계 선택이 보인다. 첫째, **출력 채널이 종료 코드(exit code)다.** 표준 출력을 쓰려면 런타임 지원(문자열, 시스템 콜)이 필요하지만, 종료 코드는 OS가 공짜로 주는 8비트 채널이다 — 최소 비용으로 "계산 결과"를 관측할 수 있다. (8비트이므로 테스트 값은 0~255로 제한된다. 이걸 모르고 `assert 300 ...`을 쓰면 44가 나와서 한참 헤맨다.) 둘째, **기계어가 아니라 어셈블리 텍스트를 출력한다.** 기계어 인코딩과 실행 파일 포맷(ELF)을 as/ld에 위임하면, 컴파일러 본체는 "의미를 보존하는 텍스트 변환"에 집중할 수 있고 산출물을 눈으로 검사할 수 있다. gcc가 지금도 내부적으로 `.s` 단계를 거치는 것과 같은 이유다 — 경계가 텍스트면 디버깅이 가능하다.

### 테스트 스크립트 — 매 스텝의 verify 장치

chibicc 방식의 심장은 이 스크립트다. 저장소 루트에 `test.sh`:

```bash
#!/bin/bash
assert() {
  expected="$1"; input="$2"
  ./mycc "$input" > tmp.s || exit 1
  cc -o tmp tmp.s
  ./tmp
  actual="$?"
  if [ "$actual" = "$expected" ]; then
    echo "$input => $actual"
  else
    echo "$input => $expected expected, but got $actual"; exit 1
  fi
}

assert 0 '0'
assert 42 '42'
# 문장(';')이 생기는 W3부터는 테스트 입력도 '0;' 형태로 함께 갱신한다
# 주차가 진행될 때마다 여기에 한 줄씩 쌓인다
echo OK
```

`make test`가 이걸 돌리게 하고, **테스트가 전부 초록일 때만 git 커밋**한다. 커밋 히스토리 자체가 "매 단계 동작하는 컴파일러"의 증명이 된다 — 게임 프로젝트에서 "매 커밋 빌드 그린"을 지키는 것과 같은 규율이고, 면접에서 저장소를 보여줄 때 커밋 로그가 곧 학습 서사가 된다.

### 관측 도구 3종 — 막히기 전에 익혀둘 것

컴파일러 실습의 디버깅은 게임 렌더 디버깅과 같은 원칙을 따른다: 증상을 코드 추론으로 때려맞히지 말고 **관측 장치를 먼저 댄다.** 이 프로젝트의 관측 지점은 세 곳이고, 도구가 각각 있다.

| 관측 지점 | 도구 | 용도 |
|---|---|---|
| 내가 뱉은 어셈블리 | `cat tmp.s` (그냥 눈) | 산출물이 텍스트인 이유. 기대한 명령이 나왔는지 대조 |
| 실행 시점의 상태 | `gdb ./tmp` → `b main`, `si`, `info registers` | 명령 단위 실행, 레지스터/스택 실측 |
| 정답 컴파일러의 출력 | `gcc -O0 -S ref.c && cat ref.s` | 같은 C 조각을 gcc는 어떻게 낮추는지 답안지로 활용 |

세 번째가 특히 강력하다. 막힐 때마다 "gcc라면 어떻게 짰을까"를 `-S`로 열어보는 습관은, 이 6주가 끝난 뒤에도 남는 자산이다 — C++ 면접에서 "그건 컴파일러 익스플로러(godbolt)로 확인해봤는데요"라고 말하는 사람과 같은 근육을 로컬에서 기르는 것이다.

## ③ 주차별 커리큘럼 — 트랙 B, 6주

전체 파이프라인은 매주 동일하고, 다루는 언어 조각만 자란다:

```
"1+2*3"                     소스 문자열
   │  토크나이저(tokenizer)
   ▼
[1] [+] [2] [*] [3]          토큰열
   │  재귀 하강 파서(recursive descent parser)
   ▼
      (+)                    추상 구문 트리(AST)
     /   \
   (1)   (*)
        /   \
      (2)   (3)
   │  코드젠(codegen)
   ▼
push/pop 어셈블리 텍스트  →  cc가 어셈블+링크  →  실행, 종료 코드로 검증
```

chibicc의 교육적 발명은 이 파이프라인을 **첫날 전부 관통**시킨다는 데 있다. 전통 커리큘럼(드래곤북 순서)은 "렉서 완성 → 파서 완성 → …"으로 단계별 완성을 요구해서, 몇 주 동안 실행 가능한 산출물이 없다. chibicc는 "정수 하나 컴파일"이라는 극단적으로 작은 세로 조각(vertical slice)으로 시작해 **매 커밋이 동작하는 컴파일러**를 유지한다. 게임 개발의 "회색 박스로 일단 플레이 가능하게" 프로토타이핑과 같은 철학이다.

참고 자료의 지형: 원전인 온라인 입문서(compilerbook, sigbus.info)는 일본어지만 브라우저 번역으로 충분히 따라갈 수 있고, 포인터 단원까지는 서술이 완결돼 있다. 그 이후의 진짜 교과서는 **chibicc 저장소의 커밋 히스토리 자체**다 — 커밋 하나가 기능 하나이고, diff가 곧 구현 해설이다.

### W1 — 정수와 사칙연산

- **목표**: 토크나이저 → 재귀 하강 파서 → 스택 머신 코드젠의 뼈대를 완성한다. `5+20-4`, `1+2*3`, `(1+2)*3`이 옳게 계산된다.
- **구현물**: 스텝 1은 정수 하나를 읽어 위 스모크 테스트의 어셈블리를 그대로 출력하는 20줄짜리 프로그램. 스텝 2에서 `+`/`-`, 스텝 3에서 토크나이저 분리(공백 허용), 스텝 4에서 위치를 가리키는 에러 메시지, 스텝 5에서 `*`/`/`와 괄호를 문법 계층으로 추가.

문법은 EBNF로 먼저 적고, 규칙 하나 = 함수 하나로 옮긴다:

```text
expr    = mul ("+" mul | "-" mul)*
mul     = primary ("*" primary | "/" primary)*
primary = num | "(" expr ")"
```

여기서 이론 챕터의 명제 하나가 손으로 증명된다 — **연산자 우선순위는 별도 메커니즘이 아니라 문법의 계층 그 자체다.** `mul`이 `expr`보다 아래층에 있어서 `*`가 먼저 묶인다. 우선순위를 바꾸고 싶으면 표를 고치는 게 아니라 문법 계층을 옮긴다.

코드젠은 AST를 후위 순회하며 스택 머신 코드를 뱉는다. `1+2`라면:

```asm
        push 1
        push 2
        pop rdi
        pop rax
        add rax, rdi
        push rax
```

**왜 레지스터 할당이 아니라 스택 머신인가** — 레지스터 할당(register allocation)은 컴파일러 백엔드에서 가장 어려운 문제 축에 든다(그래프 컬러링). 스택 머신은 "AST 평가 순서 = push/pop 순서"라서 코드젠이 자명하게 옳다. 생성 코드의 속도를 버리고 구현의 단순함과 정확성을 산 것이다. JVM, CPython, 그리고 트랙 A의 clox가 전부 스택 VM인 것도 같은 트레이드오프다.

- **verify 기준**: `assert 47 '5+6*7'`, `assert 15 '5*(9-6)'`이 초록. 잘못된 입력(`1+`)에서 위치를 가리키는 에러 메시지가 나온다.
- **막힐 때**: compilerbook 스텝 1~5, chibicc 초기 커밋들. 파서가 꼬이면 AST를 트리 형태로 stderr에 덤프하는 함수를 먼저 짠다 — 게임 디버깅에서 "증상 튜닝 전에 관측 장치부터"와 같은 원칙이다.

### W2 — 비교 연산자와 단항

- **목표**: `==` `!=` `<` `<=` `>` `>=`와 단항 `-`를 추가한다. "조건"이 기계 수준에서 무엇인지 본다.
- **구현물**: 문법에 `equality`/`relational` 계층 추가, 단항은 `unary = ("+" | "-")? primary`로. `-x`는 `0-x`로 디슈가(desugar)해도 되고 `neg rax` 한 줄로 해도 된다 — "언어 기능을 기존 기능의 조합으로 낮추는" 디슈가링의 첫 경험.

비교의 핵심 발견: **CPU에는 불리언이 없다.** `cmp`는 결과를 레지스터가 아니라 플래그 레지스터(EFLAGS)에 남기고, 그걸 0/1 값으로 물질화하려면 별도 명령이 필요하다:

```asm
        cmp rax, rdi
        sete al            # 같으면 AL=1, 다르면 0
        movzx rax, al      # 8비트를 64비트로 제로 확장
```

`<`는 `setl`, `<=`는 `setle`. 그리고 `>`는 전용 명령(`setg`)도 있지만 **피연산자를 뒤집어 `<`로** 정규화해 처리한다 — 파서 단계에서 정규화하면 코드젠 가짓수가 준다. "프론트엔드에서 정규화할수록 백엔드가 단순해진다"는 컴파일러 전반의 원칙을 처음 몸으로 겪는 지점.

- **verify 기준**: `assert 1 '2<3'`, `assert 0 '2>3'`, `assert 10 '-10+20'` 초록.
- **막힐 때**: compilerbook 비교 연산 스텝. `movzx`를 빼먹으면 AL 위쪽 쓰레기 비트 때문에 간헐적으로만 틀리는 버그가 난다 — 좋은 학습 사고이므로 일부러 한 번 겪어봐도 된다.

### W3 — 지역 변수와 대입

- **목표**: `a=3; b=5*6; a+b;` 가 돈다. 스택 프레임(stack frame)을 직접 세운다.
- **구현물**: 문장(`stmt = expr ";"`) 단위 파싱, 한 글자 변수 a~z부터 시작(오프셋 = 알파벳 순번×8로 고정), 이후 다중 글자 식별자 + 변수 목록 자료구조로 확장. 함수 프롤로그/에필로그:

```asm
main:
        push rbp           # 이전 프레임 베이스 저장
        mov rbp, rsp       # 내 프레임 기준점 확정
        sub rsp, 208       # 지역 변수 공간 확보 (26 × 8)
        ...
        mov rsp, rbp       # 지역 공간 반납
        pop rbp
        ret
```

이 6줄이 "지역 변수는 어디에 사는가"의 답 전부다. 변수 `a`는 이름이 아니라 **`[rbp-8]`이라는 주소**가 된다 — 컴파일이 끝나면 이름은 존재하지 않는다는 사실, 그리고 C++ 지역 변수 접근이 왜 빠른지(포인터 추적이 아니라 상수 오프셋 덧셈 한 번)가 여기서 체감된다.

두 번째 발견은 **lvalue/rvalue 구분이 코드젠 분기라는 것.** `a = 5`의 좌변 `a`는 "값을 읽어라"가 아니라 "주소를 계산하라"다. 그래서 대입의 코드젠은 좌변에 대해 `lea rax, [rbp-8]`(주소), 우변에 대해 값을 만들고, `mov [rax], rdi`로 저장한다. C++ 면접의 lvalue/rvalue 질문이 문법 암기가 아니라 "주소 경로냐 값 경로냐"라는 코드젠의 문제였음을 알게 된다.

- **verify 기준**: `assert 6 'a=b=3; a+b;'` 초록. `5=a;` 같은 잘못된 lvalue에 에러를 낸다.
- **막힐 때**: compilerbook 지역 변수 스텝. 오프셋 계산이 의심되면 생성된 `.s`에서 `[rbp-N]`을 눈으로 대조 — 산출물이 텍스트라서 가능한 디버깅이다.

### W4 — 제어문: if / while / for

- **목표**: 구조적 제어문이 전부 **조건 분기 + 라벨(=goto)로 낮아지는** 것을 구현으로 확인한다.
- **구현물**: `return` 문, `if (...) ... else ...`, `while`, `for`, 블록 `{ ... }`. if의 코드젠 골격:

```asm
        cmp rax, 0
        je  .L.else.1      # 조건이 거짓이면 else로
        ...                # then 절
        jmp .L.end.1
.L.else.1:
        ...                # else 절
.L.end.1:
```

라벨 번호는 전역 카운터로 발급한다 — 중첩 if에서 라벨이 충돌하는 버그를 한 번 내보면 왜 유일 번호가 필요한지 잊지 않는다. `while`/`for`는 "뒤로 가는 jmp"가 추가될 뿐 같은 재료다. 이 주를 마치면 "고수준 언어의 제어문은 goto 위의 문법 설탕이며, 컴파일러가 그 goto를 대신 짜준다"를 실물로 말할 수 있다. 분기 예측(branch prediction) 이야기로 꼬리질문이 이어져도, "내 컴파일러가 뱉는 je/jmp가 예측 대상"이라는 지점부터 출발하면 된다.

- **verify 기준**: `assert 55 'i=0; j=0; for (i=0; i<=10; i=i+1) j=i+j; return j;'` 초록 (1~10 합).
- **막힐 때**: compilerbook 제어문 스텝. 생성된 `.s`의 라벨 흐름을 종이에 화살표로 그려본다 — 제어 흐름 그래프(CFG)를 수동으로 그리는 셈이고, 이후 최적화 공부의 밑그림이 된다.

### W5 — 함수: 호출 규약을 몸으로

- **목표**: 함수 정의와 호출. 6주 중 면접 배당이 가장 큰 주다 — "함수 호출 시 스택에 무슨 일이 일어나나"를 경험으로 답하게 된다.
- **구현물**: 먼저 **호출만** 구현해 gcc로 컴파일한 C 함수를 링크해 불러본다(런타임을 빌려 쓰는 요령). 다음에 정의를 구현한다. System V AMD64 규약의 최소 사실:
  - 정수/포인터 인자 앞 6개는 레지스터로: RDI, RSI, RDX, RCX, R8, R9. 반환값은 RAX.
  - `call` 명령이 리턴 주소를 스택에 push하고, `ret`이 그걸 pop해서 점프한다.
  - **`call` 직전 RSP는 16바이트 정렬**이어야 한다.

마지막 항목이 이 주의 하이라이트다. 정렬을 어겨도 내 함수끼리는 대개 멀쩡히 돌다가, `printf`처럼 SSE 명령(16바이트 정렬된 스택을 전제)을 쓰는 라이브러리 함수를 부르는 순간 세그폴트가 난다. **"규약은 어긴 지점이 아니라 한참 뒤에서 터진다"** — ABI(application binary interface)가 왜 '약속'이고, 왜 어기면 디버깅이 지옥인지를 이보다 잘 가르치는 사고는 없다. (가변 인자 함수 호출 전 `mov al, 0` — 벡터 레지스터 인자 개수 전달 — 같은 규약의 잔가지도 이때 만난다.)

Windows에서 일하는 사람으로서 챙길 대비: **Windows x64 규약은 다르다.** 인자 레지스터가 RCX, RDX, R8, R9 4개이고, 호출자가 32바이트 섀도 스페이스(shadow space)를 예약한다. "그래서 리눅스용 어셈블리를 Windows에 그대로 못 가져온다"까지 말하면, 크로스 플랫폼 감각이 있다는 신호가 된다.

- **verify 기준**: `assert 55 'fib(9);'` (fib를 내 언어로 정의, 재귀 호출) 초록. 인자 7개 이상은 이 단계에서 미지원으로 명시해도 된다 — 스택 전달 인자는 확장 과제.
- **막힐 때**: compilerbook 함수 스텝, chibicc의 함수 커밋. 크래시가 나면 `gdb ./tmp` → `run` → `bt`로 어디서 죽었는지부터 — 콜스택 읽기는 게임 크래시 덤프 분석과 같은 근육이다.

### W6 — 포인터와 배열: 주소 연산의 실체

- **목표**: `&`와 `*`, 포인터 산술, 배열을 구현해 "포인터는 신비한 무언가가 아니라 정수 주소이며, 컴파일러가 타입을 보고 산술을 스케일한다"를 확인한다.
- **구현물**: 단항 `&`(주소 = lea, W3의 lvalue 경로 재사용)와 `*`(역참조 = 한 번 더 load). 그다음이 본론 — **포인터 산술의 스케일링**. `p+1`이 주소+1이 아니라 주소+sizeof(대상 타입)이 되려면, 컴파일러에 타입 시스템의 씨앗(int인가, int*인가)이 필요해진다. 타입이 "검사 장치"이기 전에 **코드젠에 필요한 정보**라는 발견이다.

배열은 그 완성이다. `int a[3];`에서 `a[1]`은 `*(a+1)`로 디슈가되고, 그것은 "a의 시작 주소 + 1×4바이트에서 load"가 된다. C++ 면접 단골 "배열과 포인터의 차이"에 대해 — 배열은 크기를 가진 메모리 그 자체이고, 식에서 쓰일 때 첫 원소 주소로 붕괴(decay)하며, 그 붕괴를 **내 컴파일러의 어느 함수가 수행하는지** 가리킬 수 있게 된다.

- **verify 기준**: `assert 3 'x=3; y=&x; return *y;'`, 배열 버전 `a[0]+a[1]` 류 테스트 초록.
- **막힐 때**: compilerbook 포인터 단원, chibicc의 pointer arithmetic 커밋. 스케일이 틀리면 값이 아니라 **옆 변수의 값**이 읽히는 조용한 버그가 난다 — 메모리 손상 버그가 왜 발견이 늦는지의 축소판.

### 6주 이후의 상태

이 시점의 산출물은 "C 컴파일러"라 부르기엔 작지만(구조체·전역·문자열·전처리기 없음), **파이프라인 전 단계를 소유한 상태**다. chibicc 본체는 이 뒤로 구조체, 7개 이상 인자(스택 전달), 문자열 리터럴, 전처리기까지 커밋 단위로 이어지므로, 흥미가 남으면 커밋 히스토리를 따라 계속 확장하면 된다. 단, 면접 대비 관점의 한계 효용은 W6에서 이미 대부분 회수된다.

## ④ 완주 후 확장 경로

### 1. 트랙 A — Crafting Interpreters (2~3주 + 2~3주)

전문이 craftinginterpreters.com에 무료 공개돼 있다. 두 부분의 성격이 다르다:

- **전반: 트리워크 인터프리터 jlox** — Lox(동적 타입, 클래스·클로저·상속을 갖춘 작은 언어)를 스캐너 → 재귀 하강 파서 → AST 순회 해석기로 구현. 원서는 Java지만 C++로 포팅하며 진행하는 것을 권한다 — AST 노드는 클래스 계층+방문자(visitor) 또는 `std::variant`로, 노드 수명은 학습용이므로 `shared_ptr`로 타협(성능보다 완주가 목표라는 트레이드오프를 의식하고 선택하면 그 자체가 배움이다). 여기서 얻는 것: **스코프와 클로저의 구현 실체** — 환경(environment) 체인, 변수 캡처가 왜 어려운지.
- **후반: 바이트코드 VM clox** — C로 짠 스택 머신 VM. 단일 패스 컴파일러(Pratt 파싱)로 바이트코드를 뽑고, 값 표현·해시 테이블·클로저(upvalue)·**마크스윕 가비지 컬렉터(mark-sweep garbage collector)**까지 직접 짠다. 트랙 B가 "네이티브로 낮추기"였다면 여기는 "내가 실행 환경 자체를 만들기"다.

트랙 B를 먼저 한 사람에게 트랙 A의 재미는 대비에서 온다: 같은 `1+2*3`이 한쪽에선 push/pop 어셈블리로, 다른 쪽에선 `OP_CONSTANT`/`OP_ADD` 바이트코드로 낮아진다. **"컴파일 타깃은 선택이다"** — 네이티브냐 VM이냐는 이식성·배포·핫리로드 요구가 결정하며, 게임 스크립팅이 대부분 VM(Lua, Blueprint VM, C# IL)을 택하는 이유가 이 트레이드오프에서 나온다.

### 2. LLVM Kaleidoscope 튜토리얼 — 백엔드를 사면 무엇이 공짜인가

LLVM 공식 튜토리얼. 작은 절차형 언어(Kaleidoscope)의 프론트엔드만 짜고, 코드젠은 IRBuilder로 **LLVM 중간 표현(intermediate representation, IR)을 생성해 위임**한다. 직접 어셈블리를 짜본 뒤에 해야 가치가 보인다 — 최적화 패스와 JIT가 거의 공짜로 붙는 걸 보면서, "현대 언어(Rust, Swift, Julia)가 왜 백엔드를 안 만들고 LLVM을 쓰는가"에 답할 수 있게 된다. 동시에 대가(빌드 의존성의 무게, IR 규약 학습 비용)도 체감된다.

### 3. 내 언어에 최적화 패스 하나 — 상수 접기

트랙 B의 내 컴파일러에 상수 접기(constant folding)를 직접 추가한다: 파서와 코드젠 사이에 AST→AST 변환 패스를 끼워, 자식이 모두 정수 리터럴인 연산 노드를 결과 리터럴로 치환. 30분짜리 작업이지만 개념 수확이 크다 — **최적화란 "의미를 보존하는 재작성"이고, 패스 파이프라인이라는 구조로 조립된다.** `1+2*3`이 `.s`에서 `push 7` 하나로 줄어드는 것을 diff로 확인하는 순간, "-O2가 뭘 하나요"라는 질문의 답이 관념에서 경험으로 바뀐다.

### 4. 게임 접점 — 스킬 DSL과 어빌리티 타임라인

여기가 이 로드맵이 취업 포트폴리오와 만나는 지점이다.

Winters의 현재 스킬 구현을 보자. 챔피언마다 C++ 훅 함수를 손으로 짜서 레지스트리에 등록한다 — `Shared/GameSim/Champions/Yone/YoneGameSim.cpp:521`의 `RegisterHooks()`는 `MakeGameplayHookId(eChampion::YONE, GameplayHookVariant::Q_CastFrame)` 키로 `&OnQ`를 등록한다(527~534행). 즉 **(챔피언, 이벤트) → 함수 포인터** 디스패치 테이블인데, 이것은 구조적으로 VM의 옵코드 디스패치와 동형이다 — 키가 옵코드가 되고 함수가 핸들러가 되는. 지금은 스킬 하나 추가할 때마다 C++ 재컴파일이 필요하다.

챔피언 150종 스케일을 겨냥하면 이 지점이 병목이고, 업계 표준 해법이 **스킬 DSL → 바이트코드 컴파일 → 게임 내 VM 실행**이다. "시전 8프레임에 판정 스폰, 20프레임에 대시 종료" 같은 타임라인 기술을 데이터 파일로 쓰고, 트랙 A 후반에서 만든 것과 같은 컴파일러가 이를 바이트코드로 낮추고, 시뮬레이션 틱마다 VM이 몇 옵코드씩 실행한다. Winters의 어빌리티 타임라인 에디터 툴 방향(3엔진 툴 포트폴리오의 척추)이 정확히 이 그림의 프론트엔드다 — **타임라인 데이터는 곧 프로그램이고, 에디터는 그 언어의 IDE이며, clox에서 배운 기술이 그 백엔드다.**

레지스트리 관점의 연결도 있다. `Shared/GameSim/Definitions/LoLMatchContext.h:5`의 `enum class eChampion : std::uint8_t`는 명시적 값(`IRELIA = 1` … `LEESIN = 17`)을 고정하는데, 이는 바이트코드 포맷이 옵코드 번호를 절대 재배열하지 않는 것과 같은 이유다 — **직렬화되는 번호는 코드가 아니라 계약**이며, 재컴파일 양쪽이 동시에 바뀌지 않는 세계(네트워크, 저장 파일, 컴파일된 바이트코드)에서는 번호 안정성이 정확성이다. 그리고 `Shared/GameSim/Registries/ChampionStats/ChampionStatsRegistry.h:8`의 `Add/Find/Resolve` 런타임 테이블은 인터프리터의 심볼 테이블(symbol table)과 같은 자료구조 문제다 — "이름/ID를 언제 해석하는가(컴파일 타임 오프셋 vs 런타임 룩업)"라는 축에서, W3의 RBP 오프셋 배정(컴파일 타임 해석)과 이 레지스트리(런타임 해석)는 같은 스펙트럼의 양 끝이다.

### 읽기 순서 정리

```
compilerbook + chibicc 커밋   ← 지금 (트랙 B의 교재)
        ↓
Crafting Interpreters         ← 트랙 A (전문 무료)
        ↓
LLVM Kaleidoscope             ← IR 위임 체험
        ↓
Engineering a Compiler 등 정통 교과서  ← 이제야 사전처럼 읽힌다
```

정통 교과서(드래곤북 계열)를 마지막에 둔 것은 의도다. 실물 없이 읽으면 형식 언어 이론의 벽에서 끝나지만, 두 개를 만들어본 뒤에는 "내가 임기응변으로 처리한 그 문제의 정식 명칭과 일반해"를 찾는 사전이 된다.

## ⑤ 면접 스토리로 바꾸는 법

### 주차별 면접 질문 매핑

각 주의 완료가 어떤 단골 질문을 "경험 답변"으로 바꾸는지의 지도다. 답의 첫 문장을 "제가 컴파일러를 만들 때…"로 시작할 수 있는지가 체크 기준.

| 주차 | 경험으로 답하게 되는 질문 | 답변의 핵심 문장 |
|---|---|---|
| W1 | 연산자 우선순위는 어떻게 구현되나? / 재귀 하강 파싱이란? | 우선순위는 표가 아니라 문법 계층 자체다. 규칙 하나 = 함수 하나로 옮기면 파서가 된다. |
| W2 | `a < b`는 기계어로 뭐가 되나? | cmp는 플래그만 세팅한다. 0/1 값이 필요하면 sete/setl로 물질화한다 — CPU에 불리언은 없다. |
| W3 | 지역 변수는 어디에 사는가? / lvalue와 rvalue의 차이는? | RBP 상대 상수 오프셋이다. 컴파일 후 이름은 없다. lvalue는 "주소를 계산하는 코드젠 경로"다. |
| W4 | if/for는 기계어로 뭐가 되나? | 조건 분기 + 라벨, 즉 goto다. 구조적 제어문은 컴파일러가 대신 짜주는 goto다. |
| W5 | 함수 호출 시 스택에 무슨 일이 일어나나? | 인자 6개는 레지스터(RDI…), call이 리턴 주소 push, 프롤로그가 RBP 프레임 구축. call 직전 RSP 16바이트 정렬을 어기면 printf에서 죽는 걸 직접 겪었다. Windows x64는 규약이 다르다(RCX… + 섀도 스페이스). |
| W6 | 배열과 포인터의 차이는? / 포인터 산술의 실체는? | p+1은 주소+sizeof(T)로 스케일된다 — 타입은 코드젠에 필요한 정보다. a[i]는 *(a+i)로 디슈가되며 그 코드를 내가 짰다. |
| 트랙 A | 클로저는 어떻게 구현되나? / GC는 어떻게 동작하나? | 환경 체인과 upvalue 캡처를 직접 짰다. 마크스윕의 루트 집합·도달성·스윕을 구현해봤다. |

주의할 균형: 이 표의 답들은 **System V AMD64 + 내 구현 기준의 사실**이다. 면접에서는 "제가 만든 컴파일러 기준으로는"이라는 한정을 붙이는 게 오히려 신뢰를 산다 — 다른 ABI, 다른 최적화 수준에서는 세부가 달라진다는 걸 안다는 신호이기 때문이다. 예컨대 실제 최적화 컴파일러는 지역 변수를 스택이 아니라 레지스터에만 둘 수도 있고(-O2에서 프레임 포인터 생략 포함), 그 얘기까지 이어가면 대화가 깊어진다.

### STAR 골격 — 하나를 완성해서 소유할 것

> **S (상황)** — 네트워크 대전 게임 프로젝트에서 챔피언 스킬이 챔피언당 C++ 훅 함수로 하드코딩돼 있었다. 챔피언이 17종을 넘어가며 스킬 타이밍 수정마다 재컴파일과 코드 리뷰가 필요했고, 기획 데이터와 코드의 경계가 무너지고 있었다.
>
> **T (과제)** — 스킬 타임라인을 데이터로 내리는 DSL 방향을 검토해야 했는데, "컴파일러/VM을 책으로만 아는 상태"로는 설계 판단(파싱 전략, 바이트코드 포맷, 옵코드 안정성)을 할 수 없다고 판단했다.
>
> **A (행동)** — chibicc 방식으로 6주간 C 부분집합 컴파일러를 매 커밋 동작 상태로 완주했다(토크나이저→재귀하강→x86-64 코드젠, 테스트 스크립트로 매 스텝 검증). 이어서 Crafting Interpreters의 바이트코드 VM으로 스택 VM·디스패치·GC를 구현했다. 이 과정에서 스택 정렬 위반으로 라이브러리 호출이 죽는 사고, 라벨 충돌, 포인터 스케일 버그를 직접 겪고 고쳤다.
>
> **R (결과)** — 스킬 DSL의 설계 초안(타임라인 이벤트 = 옵코드 열, 명시적 번호로 직렬화 안정성 확보, 틱당 제한 실행)을 팀 도구 방향과 연결해 제시할 수 있게 됐고, 호출 규약·스택 프레임·파싱 질문에 구현 경험으로 답한다. 커밋 히스토리 자체가 학습 증빙이다.

이 골격의 힘은 **동기가 프로젝트에서 나왔다는 서사**다. "공부하려고 만들었다"보다 "이 병목을 풀 설계 판단력이 필요해서 만들었다"가 강하다. R의 마지막 문장을 실제로 만들려면 — 저장소를 public으로 두고, README 첫 줄에 `assert 55 'fib(9);'`가 도는 스크린샷 하나를 박아두라. 면접관이 확인하는 데 10초면 충분한 증거가 완성된다.
