# Winters Engine 해부기 1 - 왜 자체 엔진을 만들었나

게임 프로젝트를 여러 개 만들다 보면 처음에는 "기능을 어떻게 구현할까"가 가장 큰 문제처럼 보인다.

이동, 공격, 스킬, UI, 렌더링, 에셋 로딩을 하나씩 붙이면 게임은 점점 돌아가는 것처럼 보인다. 하지만 프로젝트가 커질수록 진짜 문제는 기능의 개수가 아니라 기능 사이의 경계가 된다.

Winters Engine은 이 지점을 문제로 정의한 프로젝트다.

## 문제 정의

처음에는 DX11 기반 학습용 엔진 위에서 기능을 빠르게 붙이는 방식으로도 충분해 보였다. 하지만 LoL형 MOBA를 만들고, 이후 Elden형 Action RPG까지 같은 기반 위에 올리려고 하자 구조적 한계가 명확해졌다.

문제는 이런 형태로 나타났다.

- Renderer가 DX11 concrete type에 강하게 묶인다.
- Client가 gameplay 판정과 visual presentation을 동시에 들고 있다.
- 챔피언 스킬 수치가 C++ 상수, JSON, 툴 출력물 사이에 흩어진다.
- LoL Client에 맞춘 코드가 Elden Client로 재사용되기 어렵다.
- 리소스 경로와 배포 단위가 코드, 로컬 폴더, Drive 공유 사이에 섞인다.
- 기능을 고칠 때 어떤 모듈의 책임인지 먼저 불분명해진다.

이 문제는 기능 하나를 더 붙인다고 해결되지 않는다.

오히려 기능이 늘어날수록 더 커진다.

## 핵심 질문

Winters Engine의 출발 질문은 이것이었다.

> 기능을 더 많이 붙이기 전에, 기능이 붙을수록 무너지는 구조 자체를 어떻게 나눌 것인가?

그래서 Winters는 "자체 엔진을 만들었다"가 아니라 "기존 구조의 한계를 문제로 정의하고, 그 한계를 해결하기 위한 경계를 만들었다"로 설명하는 편이 더 정확하다.

## Winters의 큰 구조

Winters가 목표로 하는 제품 구조는 다음과 같다.

```text
WintersEngine.dll
├─ WintersLOL.exe      // MOBA product client
└─ WintersElden.exe    // Action RPG product client
```

그리고 gameplay authority 흐름은 다음과 같다.

```text
Client Input
-> GameCommand
-> Server GameSim
-> Snapshot/Event
-> Client Visual
```

이 두 줄이 Winters Engine의 거의 모든 도메인을 관통한다.

Engine은 LoL이나 Elden의 게임 의미를 직접 알지 않아야 한다. Client는 presentation을 담당하지만 gameplay truth를 새로 만들지 않아야 한다. Shared/GameSim은 deterministic gameplay contract를 담당하고, Server는 command를 실행해 snapshot과 event를 보낸다. Tools는 데이터와 에셋을 검증 가능한 런타임 포맷으로 만든다.

## 왜 이것이 중요했나

학생 프로젝트나 학습용 엔진에서는 "일단 보이게 만들기"가 가장 빠른 길인 경우가 많다.

하지만 장르가 달라지고, 네트워크가 붙고, AI가 붙고, 데이터가 많아지고, 리소스가 커지면 "일단 보이게 만든 코드"가 다음 기능을 막는 병목이 된다.

예를 들어 Client에서 스킬 성공 판정을 먼저 만들면 처음에는 이펙트를 빨리 볼 수 있다. 하지만 서버 권위 구조로 넘어가면 그 판정은 곧 동기화 버그가 된다. DX11 SRV를 Client public API에 바로 노출하면 UI는 빨리 만들 수 있지만, RHI나 DX12 방향으로 넘어갈 때 public API 전체가 발목을 잡는다.

Winters Engine은 이런 지점을 "나중에 고칠 문제"가 아니라 "처음부터 의심해야 하는 구조 문제"로 봤다.

## 실제 코드에서 보이는 기준

이 방향은 프로젝트 규칙에도 명시되어 있다.

대표 원칙:

- Engine은 generic runtime/render/resource primitive를 소유한다.
- Engine은 LoL Client나 Server product code를 include하지 않는다.
- Shared/GameSim은 Engine, Client, Renderer, UI, ImGui, DX type에 의존하지 않는다.
- Server는 Client visual에 의존하지 않는다.
- Client는 Engine과 Shared를 사용할 수 있지만 gameplay truth를 만들지 않는다.

관련 문서:

- `AGENTS.md`
- `.md/architecture/WINTERS_CODEBASE_COMPASS.md`
- `Shared/GameSim/Systems/CommandExecutor/ICommandExecutor.h`
- `EngineSDK/inc/RHI/IRHIDevice.h`
- `EngineSDK/inc/Renderer/RenderWorldSnapshot.h`

## 이 글을 이력서 문장으로 압축하면

> 학습용 DX11 엔진의 단일 클라이언트/하드코딩/렌더러 고정 한계를 문제로 정의하고, Engine/Client/Shared GameSim/Server/Tools 경계를 분리한 C++ 자체 엔진 Winters를 설계했습니다.
