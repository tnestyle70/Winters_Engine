## ② 기본 개념

### 내적(dot product) — 곱 하나, 얼굴 다섯

정의는 두 가지고, 이 둘이 같다는 것이 내적의 전부다.

```
대수 정의:   a·b = a.x*b.x + a.y*b.y + a.z*b.z
기하 정의:   a·b = |a| * |b| * cos(theta)
```

왜 같은가 — 코사인 법칙(law of cosines)에서 나온다. 삼각형의 세 변이 `a`, `b`, `a-b`일 때:

```
|a-b|^2 = |a|^2 + |b|^2 - 2|a||b|cos(theta)     (코사인 법칙)
|a-b|^2 = (a-b)·(a-b) = |a|^2 - 2(a·b) + |b|^2   (대수 전개)
→ 양변 비교하면  a·b = |a||b|cos(theta)
```

이 등식 하나에서 게임에서 쓰는 다섯 가지 얼굴이 전부 파생된다.

| 얼굴 | 수식 | 조건 | 게임에서의 용도 |
|---|---|---|---|
| 투영 길이 | `a·n` (n은 단위벡터) | n 정규화 | 벡터를 축에 사영, 속도의 전진 성분 분리 |
| 각도 | `cos(theta) = a·b` | 둘 다 정규화 | 두 방향 사이 각 |
| 앞/뒤 판정 | `sign(forward·toTarget)` | 부호만 보면 정규화 불필요 | 타겟이 내 앞인가 뒤인가 |
| 시야각(FOV) 판정 | `forward·dirToTarget >= cos(halfFov)` | 둘 다 정규화 | 시야 원뿔 안에 있는가 |
| 라이팅 N·L | `max(dot(N, L), 0)` | 둘 다 정규화 | 디퓨즈 밝기 |

```
        n (단위벡터)
        ^
        |      a
        |    /
        |   /
        |  /
        | / theta
        +-------------->
        |--- a·n ---|      ← a를 n 위에 떨어뜨린 그림자 길이 = |a|cos(theta)
```

**투영이 근본이고 나머지는 특수화다.** `a·n`은 "a의 그림자가 n 위에 얼마나 길게 드리우나"다. 둘 다 단위벡터면 그림자 길이가 곧 `cos(theta)`가 되고, 그림자가 음수(뒤로 드리움)면 둔각 — 그래서 부호가 앞/뒤 판정이 된다.

**시야각 판정에서 왜 acos을 안 쓰나.** `acos(dot) <= halfFov` 대신 `dot >= cos(halfFov)`를 쓴다. cos은 `[0, pi]`에서 단조 감소라 부등호 방향만 뒤집으면 동치인데, `cos(halfFov)`는 상수로 미리 구워둘 수 있고 acos은 매 판정마다 비싼 역삼각함수 호출 + 부동소수 오차로 `dot`이 1.0001이 되면 NaN을 뱉는다. "각도를 비교하고 싶으면 각도로 바꾸지 말고 cos 공간에서 비교하라"는 게임 수학의 기본 수칙.

**N·L이 왜 cos인가 — Lambert 코사인 법칙의 예고.** 단면적이 일정한 빛 다발이 표면에 비스듬히(입사각 theta) 닿으면, 같은 양의 빛이 `1/cos(theta)`배 넓은 면적에 퍼진다. 단위 면적이 받는 빛 = `cos(theta)` 배. 표면 법선 `N`과 광원 방향 `L`이 둘 다 단위벡터면 그 `cos(theta)`가 정확히 `dot(N, L)`이다. 즉 N·L은 "그럴듯해 보이는 공식"이 아니라 **에너지 보존이 강제하는 기하학**이다. 이 위에 "왜 디퓨즈를 pi로 나누나"까지 얹는 라이팅 방정식 전체는 05장에서 다룬다.

### 외적(cross product) — 수직, 넓이, 그리고 방향

```
a × b = ( a.y*b.z - a.z*b.y,
          a.z*b.x - a.x*b.z,
          a.x*b.y - a.y*b.x )

|a × b| = |a| * |b| * sin(theta)   ← a, b가 만드는 평행사변형의 넓이
a × b ⊥ a,  a × b ⊥ b              ← 결과는 두 입력 모두에 수직
a × b = -(b × a)                    ← 반교환(anti-commutative)
```

내적이 "얼마나 같은 방향인가(cos)"라면 외적은 "얼마나 다른 방향인가(sin)"다. 평행하면 0(넓이 없는 평행사변형), 수직일 때 최대.

**용도 1 — 수직 벡터 제조기.** 삼각형 두 변 `ab = B-A`, `ac = C-A`의 외적이 삼각형의 법선(normal)이다. 노멀맵을 쓰는 파이프라인에서 접선 공간(TBN — Tangent/Bitangent/Normal) 기저를 만들 때도 두 벡터에서 세 번째 수직 축을 외적으로 뽑는다. 카메라 기저도 같은 패턴 — up과 forward에서 right를 외적으로 만든다.

**용도 2 — 좌/우 판정.** XZ 평면(탑다운 게임)에서 forward와 toTarget의 외적을 취하면 y 성분만 남고, 그 **부호**가 타겟이 왼쪽인지 오른쪽인지 알려준다. 2D로 축약하면 `cross2D(a, b) = a.x*b.z - a.z*b.x` 하나의 스칼라다. 조향(steering), 시계/반시계 판정, 폴리곤 와인딩 검사 전부 이 부호다.

**용도 3 — 넓이.** `|a × b| / 2` = 삼각형 넓이. 무게중심 좌표(barycentric coordinates) 계산, 레이-삼각형 교차의 내부 판정이 전부 이 넓이 비율이다.

**왼손/오른손 좌표계와 외적 방향.** 성분 공식 자체는 좌표계와 무관한 순수 대수다. 좌표계가 개입하는 건 **결과를 손으로 해석할 때**다: 오른손 좌표계에서는 오른손 법칙(a에서 b로 감아쥐면 엄지가 a×b), 왼손 좌표계에서는 왼손 법칙. DirectX 전통의 왼손 좌표계(+X 오른쪽, +Y 위, +Z 화면 안쪽)에서 `up × forward = (0,1,0) × (0,0,1) = (1,0,0) = right`가 성립한다. 면접에서 외적 방향을 외운 공식으로 답하면 컨벤션이 다른 순간 틀린다 — **아는 벡터 두 개(up, forward)를 넣어 결과를 검산하는 습관**이 정답이고, 실무에서도 그렇게 잡는다.

### 정규화(normalization) — 잊는 순간 생기는 버그의 목록

정규화는 `v / |v|` — 방향은 유지하고 길이를 1로. 필요한 순간은 명확하다: **내적/외적의 결과를 cos/sin으로 해석하고 싶을 때, 그리고 "방향"이라는 의미를 부여할 때.** 잊었을 때의 증상은 놀랄 만큼 다양해서, 증상에서 원인을 역추적할 수 있어야 한다.

| 정규화를 잊은 곳 | 증상 |
|---|---|
| 라이팅의 N (스케일된 모델) | 오브젝트를 키우면 어두워지고 줄이면 밝아짐 — 스케일이 법선 길이에 새어 들어감 |
| 보간된 법선 (버텍스 사이) | 폴리곤 중앙이 미묘하게 어두움 — 단위벡터 두 개의 lerp는 항상 길이가 1 이하 (현이 호보다 짧다) |
| 이동 방향 벡터 | 타겟이 멀수록 빨라지는 "유도탄" 이동 — `target - pos`를 그대로 속도로 사용 |
| acos에 넣는 dot | 간헐적 NaN — 부동소수 오차로 dot이 1.0000001 |
| 쿼터니언 (연산 누적 후) | 회전에 스케일이 섞임 — 유닛이 아닌 쿼터니언의 sandwich 곱은 `|q|^2`배 스케일을 낳음 |

거꾸로 **정규화가 불필요한 곳에서 습관적으로 하는 것도 비용**이다(sqrt 하나 + 나눗셈 셋). 부호만 보는 앞/뒤 판정, 거리 비교(제곱 거리끼리 비교), 넓이 비율 계산에서는 생략해도 된다. Winters의 `RayVsCylinder`가 제곱 거리(`fOx*fOx + fOz*fOz > fRadiusSq`)로만 비교하고 sqrt를 판정 확정 후 한 번만 부르는 것이 이 원칙이다.

### 행렬 속의 SRT — 어디에 무엇이 살고 어떻게 다시 꺼내나

DirectXMath는 행벡터(row-vector) 컨벤션 — `v' = v * M`이고, 변환 합성은 왼쪽에서 오른쪽으로 읽는다. `S * R * T`는 "스케일 먼저, 회전, 마지막 이동". 이 컨벤션에서 4x4 행렬의 해부도는:

```
        | R00*sx  R01*sx  R02*sx   0 |   ← 1행 = 스케일된 로컬 X축 (basis)
M=SRT = | R10*sy  R11*sy  R12*sy   0 |   ← 2행 = 스케일된 로컬 Y축
        | R20*sz  R21*sz  R22*sz   0 |   ← 3행 = 스케일된 로컬 Z축
        |   tx      ty      tz     1 |   ← 4행 = 위치(translation)
```

**분해(decomposition)는 이 구조를 그대로 거꾸로 읽는 것이다:**
1. 위치 = 4행 (`_41, _42, _43`) — 그냥 읽으면 끝.
2. 스케일 = 각 basis 행의 **길이**. `sx = |1행 xyz|`, `sy = |2행|`, `sz = |3행|`. 회전은 길이를 보존하므로(회전 행렬의 행은 단위벡터) 행 길이에 남는 것은 스케일뿐이다.
3. 회전 = 각 행을 자기 스케일로 나눈 것. 세 행이 다시 정규직교(orthonormal)가 되면 그게 순수 회전 행렬.

Winters의 `TransformComponent::GetWorldPosition()`이 `m._41, _42, _43`을 읽는 것(`Engine/Public/ECS/Components/TransformComponent.h:58`)이 1번의 실물이다. 주의점 둘: (a) 열벡터(column-vector) 컨벤션(OpenGL 수학책 다수)은 전치라서 basis가 **열**에 산다 — 어느 컨벤션인지 먼저 확인하지 않으면 전부 틀린다. (b) 음수 스케일(미러링)이 섞이면 행 길이만으로는 부호를 복원할 수 없다 — 행렬식(determinant)의 부호로 반사 여부를 판별해야 한다.

**수치 드리프트와 재직교화.** 회전 행렬을 매 프레임 조금씩 곱해 누적하면 부동소수 오차로 행들이 서서히 직교도 아니고 단위길이도 아니게 된다 — 모델이 미세하게 찌그러지는(shear) 증상. 치료는 재직교화(re-orthogonalization): 한 행을 정규화하고, 두 번째 행에서 첫 행 성분을 빼서(그람-슈미트) 정규화하고, 세 번째는 두 행의 외적으로 만든다. 쿼터니언의 대응 치료는 정규화 한 번(sqrt 1개)이라는 것 — 이 비용 차이가 아래 표의 한 칸이 된다.

### 회전 표현 3종 — 왜 세 개나 필요한가

세 표현이 공존하는 이유는 각자 잘하는 일이 다르고, **한 표현이 세 역할(편집·변환·보간)을 모두 잘할 수 없기 때문**이다.

| 축 | 오일러 각(Euler angles) | 회전 행렬(rotation matrix) | 쿼터니언(quaternion) |
|---|---|---|---|
| 저장 | 3 float | 9 float (3x3) | 4 float |
| 직관 | ◎ yaw/pitch/roll 사람 언어 | X 숫자 9개는 못 읽음 | X 축은 보이지만 반각이라 안 읽힘 |
| 점 변환 | 행렬로 바꿔야 함 | ◎ 곱 한 번 (GPU가 원하는 형태) | 변환식 있지만 보통 행렬로 변환 |
| 합성 | X 축 순서 의존, 사실상 불가 | ◎ 행렬곱 | ◎ 쿼터니언 곱 (더 쌈: 16곱 vs 27곱) |
| 보간 | △ 축별 lerp는 경로가 이상함 | X 행렬 lerp는 회전이 아님 | ◎ slerp/nlerp |
| 특이점 | **짐벌락** | 없음 | 없음 |
| 드리프트 치료 | 해당 없음 | 재직교화(비쌈) | 정규화(sqrt 1개) |

실무 분업: **편집/디버깅은 오일러**(애니메이터와 로그가 읽을 수 있는 유일한 형태), **보간/합성/저장은 쿼터니언**, **최종 GPU 제출 직전에 행렬**. Winters의 애니메이션 파이프라인이 정확히 이 분업이다 — 본 키는 쿼터니언으로 slerp하고, 마지막에 `XMMatrixRotationQuaternion`으로 행렬화해 스키닝에 넘긴다(④에서 코드).

**짐벌락(gimbal lock)의 기하학적 실체.** 오일러 각은 "세 개의 고리(짐벌)가 순서대로 매달린 기계 장치"다. yaw(바깥 고리) → pitch(중간 고리) → roll(안쪽 고리) 순서라 하자. 문제는 중간 고리가 ±90도가 되는 순간:

```
정상 상태 (pitch = 0):              짐벌락 (pitch = +90):
  yaw축(월드 Y)                        yaw축(월드 Y)
     |                                    |
     |    roll축(로컬 Z, 앞을 향함)        |  roll축이 pitch 90도에 딸려 올라와
     |   /                                |  yaw축과 같은 방향이 됨
     |  /                                 |
     +------ pitch축                      +------ pitch축
  세 축이 서로 다른 방향                yaw와 roll이 같은 축을 돌린다
  = 자유도 3                          = 자유도 2 (하나 상실)
```

pitch가 90도가 되면 안쪽 roll축이 통째로 들려 올라와 바깥 yaw축과 **물리적으로 평행**해진다. 이제 yaw를 돌리든 roll을 돌리든 같은 회전이고, 원래 roll이 담당하던 방향의 회전은 **어떤 조합으로도 만들 수 없다**. 이건 구현 버그가 아니라 "3개의 숫자를 고정된 축 순서로 읽는다"는 표현 방식 자체의 특이점(singularity)이다 — 3개 숫자만으로 모든 3D 회전을 특이점 없이 매끄럽게 덮을 수 없다는 것은 위상수학적으로 알려진 사실이고, 그래서 4번째 숫자가 필요해진다.

**쿼터니언의 최소 수학 — 면접에서 필요한 전부.**

```
q = (w, x, y, z) = w + xi + yj + zk,   i^2 = j^2 = k^2 = ijk = -1

단위축 n 둘레로 각도 t 회전:
    q = ( cos(t/2),  sin(t/2)*n.x,  sin(t/2)*n.y,  sin(t/2)*n.z )

벡터 v 회전 (sandwich 곱):   v' = q * v * q^-1     (v는 w=0인 순수 쿼터니언)
회전 합성:                    q_total = q2 * q1     (q1 먼저 적용, Hamilton 컨벤션)
역회전:                       단위 쿼터니언이면 q^-1 = 켤레(conjugate) = (w, -x, -y, -z)
```

**왜 반각(t/2)인가.** 회전이 `q * v * q^-1`로 v를 **양쪽에서 감싸며 두 번** 적용되기 때문이다. q 하나가 t/2씩 기여해서 합계 t가 된다. "곱 한 번으로 회전"이 아니라 "켤레 conjugation으로 회전"이라는 구조가 반각의 근원이고, 이 구조 덕에 q의 벡터 부분(축)은 회전에서 불변으로 남는다.

**이중 커버(double cover) — q와 -q는 같은 회전.** sandwich 곱에 q가 두 번 들어가므로 `(-q) v (-q)^-1 = q v q^-1` — 부호가 상쇄된다. 즉 회전 하나당 쿼터니언이 정확히 **두 개** 존재한다. 이게 순수 이론이 아니라 실전 버그인 이유: 애니메이션 키프레임 q0에서 q1로 보간할 때 `dot(q0, q1) < 0`이면 두 쿼터니언이 4D 구의 반대편 반구에 있다는 뜻이고, 그대로 보간하면 **같은 도착지를 향해 270도 먼 길로 도는** 경로가 나온다(캐릭터 팔이 한 프레임 사이에 휙 뒤집히는 고전 버그). 치료: 보간 전에 `if (dot < 0) q1 = -q1` — 같은 회전이므로 결과는 동일하고 경로만 최단이 된다.

**slerp vs nlerp.**

```
slerp(q0, q1; t) = ( sin((1-t)*W)*q0 + sin(t*W)*q1 ) / sin(W),   cos(W) = dot(q0, q1)
nlerp(q0, q1; t) = normalize( q0 + (q1 - q0)*t )
```

핵심 사실: 단위 쿼터니언 q0, q1을 잇는 **현(chord)을 정규화하면 정확히 두 점을 잇는 대원(great circle) 호 위의 점**이 된다. 즉 nlerp와 slerp는 **같은 경로**를 지나고 다른 것은 **속도 분포**뿐이다 — slerp는 등각속도(호 길이에 비례), nlerp는 중간에서 빨라진다(현의 가운데가 호의 가운데보다 각도상 멀리 간다). 비용은 slerp가 acos+sin 3회, nlerp가 lerp+정규화 1회.

**게임에서 nlerp로 충분한 경우** = 두 회전 사이 각이 작을 때. 30Hz 애니메이션 키 사이, 30Hz 네트워크 스냅샷 사이의 회전 차는 보통 몇 도 수준이라 속도 왜곡이 지각 불가능하다. slerp가 필요한 곳은 각이 큰 단발 보간 — 컷씬 카메라 90도 선회, 애니메이션 키가 성긴(sparse) 리타게팅. "항상 slerp"는 오답이고 "각이 크거나 등속이 보이는 곳만 slerp"가 엔진의 실제 선택이다.

### 보간이 게임에서 사는 곳 — 그리고 각도 래핑이라는 복병

보간이 실전에서 사는 곳은 두 군데다. (1) **네트워크 스냅샷 보간**: 서버가 30Hz로 보내는 위치/회전을 60~144Hz 렌더링이 그대로 찍으면 계단식 순간이동이 된다. 스냅샷 도착 시점의 현재 상태를 시작점으로, 새 스냅샷을 목표로 짧은 시간 창 동안 보간해 채운다. (2) **애니메이션 블렌딩**: 키프레임 사이 보간(intra-clip)과 클립 사이 블렌딩(walk→run) 모두 위치는 lerp, 회전은 slerp/nlerp.

위치 보간이 lerp로 끝나는 이유는 위치 공간이 평평한 유클리드 공간이라서다. 회전이 특별 취급인 이유는 회전 공간이 **휘어져 있고 둥글게 감기기(wrap) 때문** — 그리고 yaw 하나만 쓰는 탑다운 게임에서는 이 감김이 가장 단순한 형태로 나타난다: **-PI/+PI 경계**.

```
yaw를 [-pi, pi)로 항상 정규화해 저장한다고 하자.

현재 yaw = +3.1 rad (경계 바로 왼쪽)      목표 yaw = -3.1 rad (경계 바로 오른쪽)
실제 필요한 회전: +0.083 rad (경계를 살짝 넘는 2도)
순진한 lerp(3.1 → -3.1): -6.2 rad, 즉 355도를 반대로 도는 풀 스핀
```

치료는 두 단계다:

```
Normalize(r)            : r을 (-pi, pi] 대표값으로 — "정식(canonical) 표기"
Near(r, ref)            : ref + Normalize(r - ref)
                          — r과 2*pi*k 차이 나는 동치값 중 ref에 가장 가까운 것을 선택
보간:  yaw = from + Normalize(to - from) * t     — 차이를 최단 호로 만들고 나서 lerp
```

핵심 통찰은 **"각도의 정식 표기(wire용)와 연속 상태(보간/저장용)는 다른 물건"**이라는 것. 네트워크로 보내거나 로그로 찍거나 두 각을 비교할 때는 정규화된 대표값이 맞다. 하지만 매 틱 회전하는 몸통의 yaw를 매번 정규화해 저장하면, 경계 근처에서 회전할 때마다 값이 +3.14 ↔ -3.14로 널뛰고, 이를 소비하는 모든 보간·델타 계산이 풀 스핀으로 오염된다. Winters는 이 구분을 함수 이름으로 강제한다(④에서 상세) — gotchas에 박제된 "빠른 우클릭 연타 시 캐릭터가 한 바퀴 도는 버그"의 근본 원인이 정확히 이것이었다.

### 스크린 → 월드 역변환 — 피킹 레이의 전 과정

렌더링 파이프라인의 정방향은 `월드 → (뷰 행렬) → 뷰 → (투영 행렬) → 클립 → (원근 나눗셈 perspective divide) → NDC → (뷰포트 변환) → 픽셀`. 피킹(picking)은 이 사슬을 정확히 거꾸로 걷는다. 단, 역방향에는 본질적 문제가 하나 있다: **픽셀은 2D인데 월드는 3D** — 깊이 정보가 투영에서 소실됐다. 그래서 답은 점이 아니라 **레이(반직선)**가 된다.

**1단계: 픽셀 → NDC.** 뷰포트 변환의 역. 스크린 좌표는 좌상단 원점에 y가 아래로 증가, NDC는 중앙 원점에 y가 위로 증가 — 그래서 y에 뒤집기가 들어간다:

```
ndcX = 2 * mouseX / screenW - 1        // [0, W] → [-1, +1]
ndcY = 1 - 2 * mouseY / screenH        // [0, H] → [+1, -1]  (y축 뒤집기)
```

**2단계: NDC의 near/far 두 점을 역투영.** 깊이를 모르니 NDC 깊이의 양 끝을 다 뚫는다. D3D의 NDC z 범위는 `[0, 1]`이다(OpenGL 전통의 `[-1, 1]`과 다름 — 뷰포트/투영 행렬이 다르게 생긴 이유). `(ndcX, ndcY, 0)`과 `(ndcX, ndcY, 1)`에 `inverse(View * Projection)`을 적용한다. 여기서 중요한 함정: 투영은 동차 좌표(homogeneous coordinates)를 쓰므로 역행렬을 곱한 결과도 `w != 1`이고, **w로 나눠야(원근 나눗셈의 역) 진짜 월드 점**이 나온다. DirectXMath의 `XMVector3TransformCoord`가 이 나눗셈을 내장하고 있어서 이 함정을 API가 흡수해 준다 — "왜 TransformCoord를 쓰고 TransformNormal을 안 쓰나"는 면접 꼬리질문의 답이 바로 이 w 나눗셈이다.

**3단계: 레이 구성.** `origin = nearPoint`, `dir = normalize(farPoint - nearPoint)`. 왜 두 점이나 필요한가 — 원근 투영에서는 **픽셀마다 레이 방향이 다르기 때문**이다(모든 레이가 카메라 위치에서 부챗살로 퍼진다). 방향을 카메라 forward로 고정할 수 있는 것은 직교 투영뿐이다.

**4단계: 레이 vs 형상 교차.** 레이 `p(t) = o + t*d`를 형상 방정식에 대입해 t에 대해 푼다.

```
레이-평면:  평면 n·x = k 에 대입 →  t = (k - n·o) / (n·d)
            지면 y=0 (n=(0,1,0), k=0) 이면:  t = -o.y / d.y
            분모 ≈ 0 = 레이가 평면과 평행 → 교차 없음 (0 나눗셈 가드 필수)
            t < 0 = 교차점이 카메라 뒤 → 기각

레이-구:    |o + t*d - c|^2 = r^2,  m = o - c 로 치환
            → (d·d)t^2 + 2(m·d)t + (m·m - r^2) = 0     ← t의 이차방정식
            판별식 < 0 = 빗나감, 근 두 개 = 들어간 점/나온 점

레이-실린더(y축 정렬): 구와 같은 이차방정식을 XZ 평면에서만 세운다
            (x, z 성분만으로 "2D 원 교차"를 풀고)
            → 얻은 t의 교차점 y가 [base, base+height] 안인지 클램프
```

실린더가 구보다 캐릭터 피킹에 맞는 이유: 사람 형태는 세로로 길고, 마우스 오차는 주로 수평이다. 구는 발/머리에서 판정이 얇아지지만 실린더는 전 높이에서 균일한 반경을 준다. 그리고 여러 후보가 맞으면 **t가 가장 작은(카메라에 가장 가까운) 것을 선택** — 이게 z-buffer가 하는 일의 CPU 버전이다.

| 질의 | 형상 | 수학 | 게임 용도 |
|---|---|---|---|
| 클릭 이동 | 평면/높이맵 | 1차 방정식 (또는 반복 수렴) | 지면 목표점 |
| 유닛 호버/선택 | 실린더 | XZ 이차방정식 + 높이 클램프 | 챔피언/미니언 피킹 |
| 발사체 명중 | 구/캡슐 | 이차방정식 | 히트 판정 |
| 지형 정밀 클릭 | 높이맵(heightfield) | 고정점 반복(fixed-point iteration) | 경사면 클릭 |

## ③ 심화 — 꼬리질문 대비

**"dot으로 각도를 구하려다 NaN이 나왔다. 왜인가?"**
부동소수 오차로 정규화된 두 벡터의 내적이 1.0을 미세하게 초과할 수 있다(예: 1.0000001). `acos`의 정의역은 `[-1, 1]`이라 밖의 입력에 NaN을 반환한다. 치료는 `acos(clamp(dot, -1, 1))` — 하지만 더 나은 답은 "그 각도가 정말 필요한가"를 먼저 묻는 것이다. 비교가 목적이면 cos 공간에서 비교하고, 회전이 목적이면 `atan2`가 사분면까지 안전하게 준다. Winters의 `YawFromDirectionXZ`도 acos이 아니라 `atan2(dir.x, dir.z)`를 쓴다.

**"외적의 결과는 '진짜 벡터'가 아니라는 말이 있던데?"**
유사벡터(pseudovector)다. 공간을 거울 반사하면 보통 벡터는 그대로 따라 뒤집히지만 외적의 결과는 **추가로 한 번 더 부호가 뒤집힌다**. 실무 증상: 모델을 음수 스케일로 미러링하면 외적으로 만든 법선이 안팎이 뒤집혀 라이팅이 반전되고, 와인딩 순서가 뒤집혀 백페이스 컬링이 반대로 먹는다. 미러링된 메시는 법선 재계산 또는 컬링 모드 반전이 필요한 이유.

**"쿼터니언은 왜 하필 4개 숫자인가? 회전은 자유도 3인데."**
단위 길이 제약(`|q| = 1`)이 자유도 하나를 소거해서 실효 자유도는 3이다. 4개를 쓰는 대가로 얻는 것은 **특이점 없음** — 3개 숫자만으로 모든 3D 회전을 특이점 없이 연속적으로 표현하는 것은 불가능하다는 게 위상수학적으로 알려져 있다(오일러 각의 짐벌락이 그 증상). 여분의 1차원이 "회전 공간을 4D 단위 구(S^3)에 펼쳐놓는" 비용이고, 그 구가 회전 공간을 두 겹으로 덮는 것(이중 커버)까지가 세트다.

**"반각이 나오는 이유를 수식 없이 직관으로 설명하라."**
쿼터니언 회전은 `q * v * q^-1` — v를 양쪽에서 감싸는 구조라 q가 왼쪽에서 절반, 오른쪽에서 절반씩 돌린다. 그래서 q 자체에는 목표 각의 절반만 담는다. 부수 효과 두 가지가 여기서 파생된다: (1) 360도 회전의 쿼터니언은 `q = (cos 180, ...) = -1`이 되어 항등이 아니다 — q와 -q가 같은 회전인 이중 커버의 근원. (2) 켤레를 양쪽에 쓰므로 q의 스케일 오차가 제곱으로 증폭된다 — 쿼터니언 정규화가 필요한 이유.

**"slerp에서 dot < 0이면 부호를 뒤집는 이유는?"**
q와 -q는 같은 회전이지만 4D 구 위에서는 서로 반대편 점이다. dot < 0이면 두 쿼터니언이 90도 이상 떨어진 반구에 있다는 뜻이고, 그대로 보간하면 물리적으로는 "270도를 반대로 도는" 경로가 된다. 한쪽 부호를 뒤집으면 표현하는 회전은 그대로인 채 4D 상의 거리만 최단이 되어, 보간이 최단 회전 경로를 탄다. yaw 한 축에서 `-PI/+PI` 경계를 넘을 때 `Normalize(to - from)`으로 최단 델타를 잡는 것과 완전히 같은 문제의 고차원 버전이다.

**"쿼터니언을 쓰면 짐벌락이 완전히 사라지나?"**
표현으로서의 특이점은 사라진다. 하지만 **입력이 오일러면 문제는 입력 단계에서 재발한다** — 에디터 UI가 yaw/pitch/roll 슬라이더고 내부만 쿼터니언이면, 슬라이더 값을 오일러→쿼터니언으로 변환하는 그 지점에서 pitch 90도 부근의 축 정렬은 그대로 있다. 또 FPS 카메라처럼 "pitch를 ±89도로 클램프하는" 설계는 짐벌락 회피이면서 동시에 게임 디자인(뒤집힌 카메라 방지)이기도 하다. "쿼터니언 = 만능"이 아니라 "보간·합성·저장 구간의 특이점 제거"라고 답해야 정확하다.

**"회전 행렬이 드리프트하면 어떻게 고치나?"**
그람-슈미트(Gram-Schmidt) 재직교화: 1행을 정규화 → 2행에서 1행 성분(`(row2·row1)*row1`)을 빼고 정규화 → 3행은 `row1 × row2`로 아예 새로 만든다. 외적으로 세 번째 축을 재생성하는 게 핵심 — 남은 오차를 한 축에 몰아 없앤다. 쿼터니언은 같은 문제를 `q / |q|` 한 번으로 고치므로, "장시간 누적 합성"이 있는 곳(물리 강체, 궤도 카메라)일수록 쿼터니언 저장이 유리하다.

**"위치는 lerp로 되는데 회전은 왜 특별 취급인가?"**
공간의 모양이 다르다. 위치 공간은 평평한 유클리드 공간이라 두 점의 직선 보간이 그대로 유효한 경로다. 회전 공간(SO(3))은 휘어져 있고 감긴다 — 행렬을 성분별로 lerp하면 중간값은 회전 행렬이 아니고(직교성 붕괴, 스케일/시어 섞임), 각도를 lerp하면 ±PI 경계에서 먼 길을 돈다. 쿼터니언 slerp/nlerp는 "휘어진 공간 위에서의 직선(측지선 geodesic)"을 걷는 방법이다.

**"네트워크 스냅샷 보간에서 보간 시간 창은 어떻게 정하나?"**
스냅샷 주기보다 약간 길게. 30Hz면 33ms 주기인데 창을 딱 33ms로 잡으면 패킷 지터로 다음 스냅샷이 1ms만 늦어도 보간이 끝나고 멈칫한다. 창을 주기의 1.5~2배로 늘리면 부드러워지지만 표시 지연이 그만큼 늘어난다 — **부드러움 vs 반응성 트레이드오프**를 상수 하나가 조절한다. 추가 장치 둘: 이동량이 임계 이하면 보간 생략(제자리 미세 진동 방지), 임계 이상이면 보간 없이 스냅(텔레포트를 활강으로 오해석하지 않기 위해). Winters의 보간 상수가 정확히 이 3종 세트다(④).

**"레이-실린더에서 뚜껑(cap)은 어떻게 처리하나?"**
옆면 이차방정식의 근이 높이 범위를 벗어나면, 레이가 윗뚜껑/아랫뚜껑 평면(`y = base + h`, `y = base`)과 만나는 t를 구해 그 교차점의 XZ 거리가 반경 이내인지 본다. 다만 탑다운 카메라의 유닛 피킹처럼 레이가 항상 비스듬히 내려오는 상황에서는 옆면 판정만으로 거의 모든 클릭이 잡혀서, 뚜껑 판정을 생략하고 수직 레이만 특수 처리하는 단순화가 실용적이다 — 판정 누락의 기하학적 조건(실린더 중심 바로 위에서 수직에 가까운 레이)이 실제 카메라에서 거의 발생하지 않음을 알고 하는 생략이어야 한다.

**"w 나눗셈을 빼먹으면 피킹이 어떻게 깨지나?"**
`inverse(VP)`를 곱한 직후의 4D 점은 `w != 1`인 동차 좌표다. 나눗셈 없이 xyz를 그대로 쓰면 near/far 점이 전부 틀린 위치에 놓이고, 레이 방향이 화면 중앙에서 멀어질수록 크게 어긋난다 — "화면 중앙 클릭은 맞는데 가장자리 클릭이 밀린다"는 특징적 증상. 원근 나눗셈은 정방향 파이프라인에만 있는 게 아니라 **역변환에도 대칭적으로 존재**한다는 것이 요점.

**"직교 투영에서도 피킹 레이를 두 점으로 만들어야 하나?"**
같은 코드가 그대로 동작하긴 한다(그래서 범용 구현은 near/far 언프로젝트를 쓴다). 다만 직교 투영은 모든 픽셀의 레이 방향이 카메라 forward로 동일해서 origin만 픽셀별로 다르다 — 원근은 origin이 (거의) 한 점이고 방향이 부챗살. "왜 두 점인가"의 답은 "원근에서는 방향이 픽셀마다 달라 한 점으로는 레이를 정의할 수 없기 때문"이고, 직교는 그 특수한 예외다.

**"높이가 있는 지형에서 클릭 지점은 평면 교차로 안 되지 않나?"**
안 된다. 평면 `y=0` 교차는 초기 추정일 뿐이다. 실전 기법은 고정점 반복: `t0 = 평면 교차` → 그 지점의 지형 높이 `h(x, z)`를 샘플 → `t1 = (h - o.y) / d.y`로 갱신 → 수렴(델타가 임계 이하)까지 반복. 완만한 지형에서 수 회 안에 수렴한다. 대안은 레이마칭(고정 스텝 전진)이나 지형 쿼드트리 하강인데, 반복 수렴은 코드가 짧고 LoL류의 완만한 지형에 충분하다. Winters가 정확히 이 6회 반복 구현이다(④).

**"각도 정규화 함수를 fmod로 짤 때의 함정은?"**
C++ `fmod`는 피제수의 부호를 따라가므로 `fmod(-0.1 + pi, 2pi)`처럼 음수 입력에서 음수가 나올 수 있다 — 한 번의 fmod로 `[-pi, pi)`가 보장되지 않아 "음수면 2pi 더하기" 보정이 한 단계 필요하다. 또 입력이 NaN/inf면 fmod가 NaN을 전파해 이후 모든 회전이 오염되므로, 정규화 함수 입구에서 `isfinite` 가드로 차단하는 것이 방어적이다. Winters `NormalizeRadians`가 이 두 가드를 다 갖고 있다.

### (4) yaw 래핑 시스템 — gotchas에 박제된 버그가 만든 함수 계층

②의 "정식 표기 vs 연속 상태" 구분이 Winters에서는 함수 이름의 계약으로 강제된다. 원시 연산은 `Engine/Include/WintersMath.h:188`의 `NormalizeRadians`(fmod + 음수 보정 + isfinite 가드로 `[-pi, pi)` 정규화)와 199행의 `NearestEquivalentRadians`(`ref + Normalize(r - ref)` — ref에 가장 가까운 동치각 선택). 이를 챔피언 도메인으로 감싼 것이 `Shared/GameSim/Definitions/ChampionRuntimeDefaults.cpp:46~68`의 4함수다:

- `NormalizeChampionVisualYaw`(46행) — 와이어 값·로그·델타 비교용 **정식 표기**
- `ResolveChampionVisualYawFromDirection`(56행) — 방향 벡터→yaw(`atan2(x, z)` 기반 `YawFromDirectionXZ`, `WintersMath.h:178`) + 챔피언별 메시 forward 보정
- `MakeChampionVisualYawNear`(51행) / `ResolveChampionVisualYawNear`(63행) — Transform에 쓰는 **연속 상태**용

이 분리는 이론 취향이 아니라 사고 기록이다: 매 이동 틱마다 yaw를 정규화해 저장하던 초기 구현이 빠른 우클릭 연타 시 `-PI/+PI` 경계를 반복해서 넘나들며 캐릭터가 풀 스핀하는 버그를 냈고(gotchas 2026-05-20 "Server yaw storage" 항목), 치료가 "저장은 Near 계열, 정규화는 비교/와이어 전용"이라는 현재 규약이다. 면접에서 "각도 래핑 버그를 실제로 잡아 본 적 있나"에 대한 완결된 서사.

### (5) 네트워크 스냅샷 보간 — 트레이드오프 상수 3종 세트

`Client/Private/Scene/Scene_InGameNetwork.cpp:133~136`에 ③에서 말한 상수들이 그대로 있다: 보간 창 `kNetworkActorInterpDurationSec = 0.055f`(30Hz 스냅샷 주기 33ms의 약 1.7배 — 지터 흡수 vs 지연의 절충), 텔레포트 임계 `kNetworkActorInterpTeleportSq = 9.f`(3m 이상 점프는 보간 없이 스냅), 미세 이동 임계(보간 생략). 적용부 `ApplyNetworkActorInterpolation`(761행)은 `SmoothStep01`(138행, `t*t*(3-2t)` — 양 끝 속도 0으로 이징)을 통과시킨 t로 위치는 `LerpVec3`(144행), 회전은 `LerpRotationNear`(153행)를 부른다. `LerpRotationNear`의 yaw 처리(159~160행)가 ②의 최단 경로 보간 공식 그대로다: `targetYaw = MakeChampionVisualYawNear(to.y, from.y)` 후 `from + Normalize(delta) * t`.

스냅샷이 Transform에 꽂히는 지점인 `Client/Private/Network/Client/SnapshotApplier.cpp:694`도 같은 규약을 지킨다 — 서버 yaw(정식 표기로 수신)를 `MakeChampionVisualYawNear(sourceYaw, rot.y)`로 현재 연속 상태 근처 동치각으로 바꿔 쓰고, 반턴 오판정 검사(`IsYawHalfTurn`, 99행)는 반대로 `NormalizeChampionVisualYaw`로 정규화한 델타로 한다. "쓰기는 Near, 비교는 Normalize"가 한 파일 안에서 둘 다 보이는 지점.

### (6) 쿼터니언이 실제로 사는 곳 — 본 애니메이션

`Engine/Private/Resource/Animation.cpp:120`의 `InterpolateRotation`이 회전 키를 `XMQuaternionSlerp`(133행)로 보간하고, 위치/스케일 키는 `XMVectorLerp`(117·148행)로 보간한다 — "위치는 평평한 공간이라 lerp, 회전은 휘어진 공간이라 slerp"의 분업이 함수 세 개로 나뉘어 있다. 보간된 S/R/T는 99~101행에서 `XMMatrixScalingFromVector(scale) * XMMatrixRotationQuaternion(rot) * XMMatrixTranslationFromVector(pos)`로 재조립된다 — ②의 SRT 합성 순서(행벡터 컨벤션: 스케일→회전→이동) 그대로다. 즉 Winters에서 쿼터니언은 "보간 구간 전용 통화"이고, 저장(키프레임)과 GPU 제출(행렬) 사이에서만 산다.

한편 `Engine/Public/ECS/Components/TransformComponent.h:17~19`의 게임플레이 Transform은 오일러 Vec3 저장이다. 이것은 결함이 아니라 도메인 판단 — 탑다운 LoL에서 챔피언 몸통 회전은 yaw 1 자유도뿐이라 pitch/roll이 상시 0이고, 짐벌락의 전제 조건(중간 축 90도) 자체가 성립하지 않는다. float 하나의 래핑만 (4)의 규약으로 관리하면 되고, 네트워크로도 yaw 하나만 보내면 되어 대역폭에도 맞다. "쿼터니언을 왜 안 썼나"가 아니라 "자유도 1에 4 float 표현은 과잉이고, 진짜 3D 회전이 필요한 본 애니메이션에는 쿼터니언을 쓴다"가 답변이 된다.

### (7) 내적 — 셰이더에서 다섯 얼굴 중 세 개

- **N·L 디퓨즈**: `Client/Bin/Debug/Shaders/Default3D.hlsl:58`의 `float ndotl = max(dot(p.normal, lightDir), 0.0f);` — Lambert cos의 최소형. `max(_, 0)`은 "빛을 등진 면은 음수 빛을 받지 않는다"는 클램프.
- **PBR의 내적 4종**: `Client/Bin/Debug/Shaders/BRDF/BRDF_GGX.hlsli:46~49`가 `NoV/NoL/NoH/VoH`를 전부 `saturate(dot(...))`로 뽑는다. GGX 분포 `D_GGX`(7~14행)는 `a2 / (PI * ((n·h)^2*(a2-1)+1)^2)` — 이 식과 "왜 PI로 나누나"는 05장의 본론.
- **림 라이트 = 1 - N·V**: `Client/Bin/Debug/Shaders/Mesh3D.hlsl:109`의 `pow(1.0f - saturate(dot(N, V)), 4.0f)` — 시선과 법선이 수직에 가까운 실루엣 가장자리에서 `dot(N,V) → 0`이 되는 성질을 뒤집어 외곽광으로 쓴다. "내적의 기하학적 의미를 아느냐"를 스타일라이즈드 셰이딩으로 증명하는 예.

### (8) 외적 — 기저 생성과 경사 판정

- **카메라 right 축**: `Engine/Private/Renderer/CCamera.cpp:37`의 `Vec3::Cross(m_vUp, GetForward()).Normalized()` — 왼손 좌표계에서 `up × forward = right`. ②의 "아는 벡터로 검산" 관례의 실물.
- **삼각형 법선의 y로 경사 판정**: `Engine/Private/Manager/Navigation/MapSurfaceSampler.cpp:281`의 `std::fabs(Vec3::Cross(ab, ac).Normalized().y)` — 지형 삼각형 두 변의 외적으로 법선을 만들고, 그 y 성분(수직에 가까울수록 1)으로 "걸을 수 있는 평평한 면인가"를 셈한다. `fabs`는 와인딩 순서(법선이 위로 향하든 아래로 향하든)에 무관하게 만들기 위한 것 — 외적의 방향이 와인딩에 의존한다는 사실을 알고 흡수한 코드다.
- 원시 연산 자체는 `Engine/Include/WintersMath.h:60`(Dot)·65행(Cross)에 성분 공식 그대로 있다.

### (9) 없는 것 — 면접용 확장 답변

Winters에 slerp 기반 **네트워크 회전 보간**은 없다(yaw 1축이라 불필요). 만약 3D 회전 액터(비행 유닛, 랙돌 동기화)가 생긴다면: 스냅샷에 쿼터니언을 최소 3성분+부호(smallest-three) 압축으로 싣고, 수신 측에서 `dot < 0` 부호 정렬 후 nlerp(스냅샷 간 각이 작으므로 slerp 불필요)로 보간하는 확장이 (5)의 보간 창 구조에 그대로 얹힌다 — 기존 설계의 어디에 무엇을 더하면 되는지까지 말할 수 있으면 "구현해 본 사람의 로드맵"이 된다.

## ⑤ 면접 Q&A

**Q1. 내적과 외적의 기하학적 의미를 설명하라.**
- 핵심: 내적 = 투영(`|a||b|cos`) — 각도/앞뒤/시야각/N·L이 전부 파생. 외적 = 수직 벡터 + 평행사변형 넓이(`|a||b|sin`) — 법선/기저/좌우 판정이 파생. 코사인 법칙으로 두 정의의 동치를 유도할 수 있으면 상급.
- Winters: N·L(`Default3D.hlsl:58`), 림 `1-dot(N,V)`(`Mesh3D.hlsl:109`), 카메라 right(`CCamera.cpp:37`), 지형 경사(`MapSurfaceSampler.cpp:281`).

**Q2. 마우스 클릭으로 3D 유닛을 집는 과정을 처음부터 끝까지 말하라.**
- 핵심: 픽셀→NDC(y 뒤집기, D3D z∈[0,1]) → near/far 두 점을 inverse(VP)로 언프로젝트(w 나눗셈!) → 레이 구성 → 형상별 교차(지면=1차식, 유닛=실린더 이차식) → 최소 t 승자.
- Winters: `CInput.cpp:58` 언프로젝트, `GameplayQuery.cpp:18` 실린더, `GameplayQuery.cpp:78` 최소 t 선택, `Scene_InGameMapNav.cpp:470` 높이맵 반복 수렴.

**Q3. 쿼터니언을 왜 쓰나? 오일러/행렬 대비로.**
- 핵심: 편집=오일러(직관), 변환=행렬(GPU), 보간·합성·저장=쿼터니언(4 float, 특이점 없음, slerp). 짐벌락 = 중간 축 90도에서 바깥/안쪽 축이 물리적으로 정렬돼 자유도 1 상실. 반각 = sandwich 곱이 양쪽에서 절반씩 돌리기 때문. 이중 커버 = q와 -q가 같은 회전 → 보간 전 dot<0 부호 정렬.
- Winters: 본 키 slerp(`Animation.cpp:133`) + 게임플레이 Transform은 yaw 1축이라 오일러 저장(`TransformComponent.h:17`) — 도구를 자유도에 맞춘 선택.

**Q4. slerp와 nlerp의 차이, 언제 nlerp로 충분한가?**
- 핵심: 같은 대원 경로, 다른 속도 분포(slerp=등각속도, nlerp=중간 가속). 비용은 nlerp 압승. 키/스냅샷 간 각이 작은 30Hz 데이터에서는 왜곡이 지각 불가 → nlerp. 큰 각 단발 보간(컷씬 카메라)만 slerp.

**Q5. -PI/+PI 경계에서 회전 보간이 튀는 버그를 어떻게 잡나?**
- 핵심: 원인 = "정식 표기(비교/와이어용)"를 "연속 상태(저장/보간용)"에 오용. 치료 = 보간·저장 시 `ref + Normalize(target - ref)`로 참조각 최근접 동치각 선택.
- Winters: 실제 사고(우클릭 연타 풀 스핀, gotchas 2026-05-20) → `NormalizeChampionVisualYaw` vs `MakeChampionVisualYawNear` 함수 계약 분리(`ChampionRuntimeDefaults.cpp:46~68`), 보간은 `LerpRotationNear`(`Scene_InGameNetwork.cpp:153`).

**Q6. 서버가 30Hz로 주는 위치를 클라가 어떻게 부드럽게 보여주나?**
- 핵심: 스냅샷 도착 시 현재 상태를 시작점으로 캡처, 보간 창(주기의 1.5~2배) 동안 이징 보간. 텔레포트 임계로 스냅 분기, 미세 이동은 생략. 창 길이 = 부드러움 vs 지연 트레이드오프.
- Winters: `kNetworkActorInterpDurationSec=0.055f` 등 상수 3종 + SmoothStep(`Scene_InGameNetwork.cpp:133~162, 761~801`).

## ⑥ 흔한 오답 / 함정

1. **"내적은 각도 구할 때 쓴다"에서 멈추기.** 각도는 다섯 얼굴 중 하나일 뿐이고, 실무에서는 오히려 acos을 피해 cos 공간 비교·부호 판정·투영으로 쓰는 경우가 압도적이다. "acos은 언제 위험한가"(정의역 초과 NaN)까지 나와야 실전 경험으로 들린다.

2. **외적 방향을 "오른손 법칙"으로 단정.** 성분 공식은 좌표계 무관, 해석하는 손이 좌표계를 따라간다. DirectX 왼손 좌표계 프로젝트에서 오른손 법칙으로 답하면 바로 꼬리질문에 걸린다. "아는 벡터로 검산한다"가 안전하고 정직한 답.

3. **"쿼터니언 = 짐벌락 해결"만 외우기.** 왜(3 파라미터 표현의 위상적 특이점), 무엇을 대가로(4번째 성분 + 단위 제약 + 이중 커버), 어디서만(보간·합성 구간) 이득인지 없이 한 줄 암기는 "직관은 오일러, 변환은 행렬"이라는 반쪽을 놓친다. 그리고 yaw 1축 게임에서는 쿼터니언이 과잉이라는 판단(Winters Transform)도 유효한 설계다.

4. **행렬 lerp로 회전 보간.** 성분별 lerp의 중간 행렬은 직교성이 깨진 "회전 아닌 것"(스케일+시어 오염)이다. 회전 보간은 반드시 쿼터니언(또는 각도 최단 델타)을 거쳐야 한다.

5. **언프로젝트에서 w 나눗셈 생략.** `inverse(VP) * ndc`의 결과를 그대로 3D 점으로 쓰면 화면 가장자리 클릭이 밀린다. `XMVector3TransformCoord`가 나눗셈을 내장한다는 것, 그리고 D3D NDC z가 `[0,1]`(GL과 다름)이라는 것 — 컨벤션 두 개가 피킹 버그의 단골 원인.

6. **모든 벡터를 습관적으로 정규화 / 아무 데서도 안 하기.** 양극단 모두 오답. cos/sin 해석이 필요한 곳(라이팅, 각도, 방향)은 필수, 부호·제곱거리 비교는 불필요. "어디서 아끼고 어디서 반드시 하는가"를 증상 목록(스케일 시 밝기 변화, 거리 비례 이동 속도, 간헐 NaN)과 함께 말할 수 있어야 한다.
