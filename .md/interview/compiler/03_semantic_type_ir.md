# 의미 분석 · 타입 시스템 · 중간 표현(IR)

> "심볼 테이블이 뭔가요? 타입 검사와 타입 추론은 뭐가 다른가요? 컴파일러는 왜 AST를 바로 기계어로 안 바꾸고 IR을 거치나요? SSA가 왜 필요한가요?" — 파싱이 끝난 뒤부터 최적화가 시작되기 전까지, 컴파일러의 '한가운데'에서 벌어지는 일을 관통한다. C++의 오버로드 해석·auto·lvalue/rvalue가 전부 이 단계의 용어로 다시 설명된다.

---

## ① 한 줄 본질

> **"파서가 만든 AST는 '문법적으로 맞는 문장'일 뿐이다. 의미 분석(semantic analysis)은 심볼 테이블로 모든 이름을 선언에 연결하고, 타입 시스템으로 모든 연산이 말이 되는지 증명한다. 그 다음 컴파일러는 검증된 AST를 언어 독립적이고 최적화하기 좋은 정규형인 중간 표현(IR)으로 낮추는데, 3-주소 코드를 기본 블록과 제어 흐름 그래프(CFG)로 조직하고, SSA로 '모든 변수는 한 번만 대입된다'는 형태를 강제하면 정의-사용 관계가 명시적인 그래프가 되어 최적화가 그래프 알고리즘 문제로 바뀐다. C++의 오버로드 해석은 이 의미 분석 단계의 '최소 변환 비용 탐색'이고, lvalue/rvalue는 IR 수준에서 '메모리 주소가 있는 값인가, 레지스터에만 존재해도 되는 값인가'라는 질문이다."**

---

## ② 기본 개념

### 이 챕터의 위치 — 프런트엔드 후반부와 미들엔드의 입구

```
[소스] → 어휘 분석 → 파싱 → ★의미 분석★ → ★IR 생성★ → 최적화 → 코드 생성 → [기계어]
         (토큰)     (AST)   (검증된 AST     (3-주소/SSA)  (미들엔드)  (백엔드)
                            + 심볼/타입)
        └────────── 프런트엔드 ──────────┘ └──── 미들엔드 ────┘ └─ 백엔드 ─┘
```

파서까지는 "문법적으로 올바른가"만 봤다. `champion.Attack(target)`이 문법에 맞는 건 파서가 알지만, `champion`이 선언된 적 있는지, `Attack`이라는 멤버가 존재하는지, `target`의 타입이 매개변수와 맞는지는 모른다. 촘스키(Chomsky) 계층의 한계 때문이다 — "선언 후 사용" 같은 규칙은 문맥 자유 문법(context-free grammar)으로 표현할 수 없어서, **문법이 못 잡는 규칙을 잡는 별도 단계**가 필요하다. 그게 의미 분석이고, 그 핵심 자료구조가 심볼 테이블, 핵심 판정 규칙이 타입 시스템이다.

매일 보는 MSVC 에러를 단계별로 다시 읽으면 이 구분이 체감된다:

| 에러 | 잡는 단계 | 실체 |
|---|---|---|
| `C2059: syntax error` | 파싱 | 문법 규칙으로 유도 불가능한 토큰 열 |
| `C2065: undeclared identifier` | 의미 분석 | 심볼 테이블 룩업 실패 |
| `C2664: cannot convert argument` | 의미 분석 | 암묵적 변환 경로 탐색 실패 |
| `C2668: ambiguous call` | 의미 분석 | 오버로드 변환 비용 동률 |
| `LNK2019: unresolved external` | 링크 | 선언만 있고 정의 심볼이 어느 오브젝트에도 없음 |
| `LNK2005: already defined` | 링크 | ODR 위반 — 강한 심볼 중복 |

"에러 메시지의 접두어(C냐 LNK냐)가 어느 단계가 거부했는지를 말해준다"는 관점 하나로, 디버깅 경험 전체가 컴파일 파이프라인 지식으로 재활용된다.

의미 분석이 타입 검사만 하는 것도 아니다. C++에서 이 단계가 판정하는 것들의 목록을 몇 개만 나열해 보면 성격이 보인다:
- 접근 제어 — `private` 멤버 접근 거부 (문법이 아니라 **선언 정보**가 필요한 판정)
- const 정확성 — `const` 객체에 비-const 멤버 호출 거부
- 제어 흐름 규칙 — 루프 밖의 `break`, case를 건너뛰는 `goto`의 변수 수명 침범
- 오버라이드 검증 — `override`가 붙었는데 부모에 매칭 가상 함수가 없으면 에러

공통점: 전부 "이 노드 하나만 봐서는 판정 불가, **멀리 있는 선언·문맥과 대조해야** 하는 규칙"이다. 그 '멀리 있는 정보'를 모아두는 곳이 심볼 테이블이다.

### 의미 분석 (1) — 심볼 테이블: 맵의 스택

심볼 테이블(symbol table)은 "이름 → 선언 정보(타입, 스코프, 저장 위치, 링키지...)"의 사전이다. 문제는 같은 이름이 스코프마다 다른 것을 가리킬 수 있다는 것:

```cpp
int g = 1;                  // 전역 스코프
void f(int a) {             // 함수 스코프
    int x = 2;
    {                       // 블록 스코프
        int x = 3;          // 바깥 x를 가림(shadowing)
        g = x + a;          // x → 블록, a → 함수, g → 전역
    }
}
```

가장 고전적인 구현이 **맵의 스택(stack of maps)**이다. 스코프에 들어갈 때(`{`) 새 맵을 push, 나올 때(`}`) pop. 이름을 찾을 때는 **스택 꼭대기(가장 안쪽)부터 바깥으로** 훑는다:

```
   탐색 방향: 안 → 밖 (첫 히트에서 멈춤)
  ┌───────────────────────────────┐
  │ 블록 스코프    { x → int(=3) } │ ← top: 먼저 탐색
  ├───────────────────────────────┤
  │ 함수 스코프    { a → int,      │
  │                 x → int(=2) } │
  ├───────────────────────────────┤
  │ 전역 스코프    { g → int,      │
  │                 f → 함수 }     │ ← 마지막 탐색
  └───────────────────────────────┘
```

이 설계가 답인 이유:
- **섀도잉이 공짜로 구현된다.** 안쪽 맵에서 먼저 히트하므로 안쪽 `x`가 자연히 이긴다.
- **스코프 종료가 O(1)이다.** pop 한 번으로 그 스코프의 모든 이름이 사라진다. 이름 하나하나 지울 필요가 없다.
- **렉시컬 스코프(lexical scope) 규칙과 자료구조가 1:1 대응한다.** "코드의 중괄호 중첩 = 스택의 깊이"라서 구현이 곧 언어 명세의 증명이 된다.

실제 프로덕션 컴파일러는 성능 때문에 변형을 쓴다(예: 이름 하나당 선언 스택을 매다는 방식 — 탐색이 스코프 수가 아니라 O(1)에 가깝게). 그러나 "스코프 진입/탈출을 push/pop으로 모델링한다"는 뼈대는 같다. 면접에서는 스택 오브 맵을 그리고, "실제 구현은 룩업을 O(1)로 만드는 변형"이라고 한 줄 덧붙이면 충분하다.

### 의미 분석 (2) — 이름 해석: 선언 vs 정의, 그리고 링커로 이어지는 복선

이름 해석(name resolution)은 "이 식별자가 어느 선언을 가리키는가"를 확정하는 일이다. 여기서 C/C++의 핵심 구분이 나온다:

| | 선언(declaration) | 정의(definition) |
|---|---|---|
| 역할 | 이름과 타입을 알린다 | 실체(코드/저장 공간)를 만든다 |
| 예 | `int Attack(int dmg);` | `int Attack(int dmg) { ... }` |
| 컴파일러에게 | **타입 검사에는 이걸로 충분** | 코드 생성에 필요 |
| 개수 제한 | 같은 TU에 여러 번 가능 | ODR: 프로그램 전체에 (원칙적으로) 하나 |

이 분리가 C++ 컴파일 모델의 뿌리다. 각 번역 단위(translation unit, TU)는 **선언만 보고** 호출 코드를 만들 수 있다 — 시그니처만 알면 인자를 어느 레지스터/스택에 놓을지 결정할 수 있기 때문이다. 정의가 어디 있는지는 몰라도 되고, 그 자리는 심볼(symbol)이라는 '외상 장부'로 남는다. **이 외상을 갚는 게 링커다.** 즉 심볼 테이블은 컴파일러 안에서 끝나는 자료구조가 아니라, 오브젝트 파일의 심볼 테이블로 이어져 링크 단계까지 관통하는 이야기다(링커의 이름 해석·ODR 위반의 실제 증상은 [cpp/02_compile_link_dll.md](../cpp/02_compile_link_dll.md)에서).

```
 컴파일러의 심볼 테이블            오브젝트 파일의 심볼 테이블         링커
 (스코프 구조, 타입 정보 포함)  →  (이름+정의됨/미정의 플래그만 남음) → (미정의 심볼을
  컴파일 끝나면 소멸               .obj 안에 실존                     정의와 짝짓기)
```

참고로 C++의 이름 해석은 이 기본 모델 위에 층이 더 얹힌다 — 한정 이름 룩업(`A::b`는 A 안에서만), 비한정 이름 룩업(스코프 체인 바깥 방향), 인자 의존 룩업(ADL: 인자의 네임스페이스도 후보에 포함 — `std::cout << x`가 동작하는 이유), 클래스 멤버 룩업(상속 위계 탐색). 전부 "어느 맵들을 어떤 순서로 뒤지는가"의 변주라는 것만 잡아두면 각론은 필요할 때 찾으면 된다.

ODR(One Definition Rule)이 왜 "규칙"으로 존재하는지도 이 구조로 설명된다. TU들은 서로를 모른 채 각자 컴파일되므로, **컴파일러는 TU를 넘는 중복 정의를 볼 방법이 없다.** 그래서 언어가 규칙으로 못박고, 위반의 일부는 링커가 잡고(중복 심볼 에러), 일부는 아무도 못 잡는다(inline 함수의 정의가 TU마다 미묘하게 다른 경우 — ill-formed, no diagnostic required). "컴파일러가 왜 안 잡아줬냐"는 질문에 대한 답이 "잡을 수 있는 단계가 구조적으로 없다"인 드문 사례다.

### 타입 시스템 (1) — 정적/동적 × 강/약 매트릭스

타입 시스템(type system)은 "어떤 연산이 어떤 값에 허용되는가"를 규정해서, **잘못된 프로그램의 일부를 실행 전에(또는 실행 중에라도) 거부**하는 장치다. 두 축으로 분류한다:

- **정적(static) vs 동적(dynamic)**: 타입 검사를 **언제** 하는가 — 컴파일 타임 vs 런타임.
- **강(strong) vs 약(weak)**: 타입 규칙을 **얼마나 엄격하게** 지키는가 — 암묵적 변환/재해석을 얼마나 허용하는가. (주의: 강/약은 정적/동적과 달리 엄밀한 정의가 없는 관용적 분류다. 면접에서도 이 단서를 붙이는 게 정확하다.)

| | 강한 타입 | 약한 타입 |
|---|---|---|
| **정적** | Rust, Haskell, Java/C#, (대체로) C++ | C |
| **동적** | Python, Ruby | JavaScript, PHP |

각 사분면의 체감:
- **정적·강 (Rust)**: `let x: i32 = 1; let y: i64 = x;`조차 에러 — 명시적 캐스트 요구. 오류를 컴파일 타임에 최대한 몰아넣는 설계.
- **정적·약 (C)**: 컴파일 타임에 검사하지만 `int`↔포인터 캐스트, 암묵적 축소 변환이 헐겁다. "검사는 하되 탈출구가 사방에 있다."
- **동적·강 (Python)**: `"1" + 1`은 TypeError — 타입은 런타임에야 확인되지만, 확인되면 엄격하다.
- **동적·약 (JavaScript)**: `"1" + 1 === "11"` — 런타임에 조용히 강제 변환(coercion)한다. 오류가 값의 오염으로 바뀌어 멀리서 터진다.

**C++의 좌표**: 정적 타입이고 C보다 훨씬 강하지만(enum class, explicit, 참조의 cv 규칙...), C 호환성 때문에 약한 탈출구를 남겼다 — 암묵적 축소 변환, C 스타일 캐스트, `reinterpret_cast`. 그래서 "C++은 강타입입니까?"에는 "**기본은 강타입, 단 프로그래머가 명시적으로 열 수 있는 탈출구가 있는 언어**"가 정확한 답이다. 탈출구가 *명시적 문법*(reinterpret_cast)을 요구한다는 점이 C와의 차이다.

언어의 진화 방향도 이 축 위에서 읽힌다. C++11의 리스트 초기화 `int x{3.14};`는 축소 변환(narrowing)을 **에러로** 거부한다 — `int x = 3.14;`는 C 호환 때문에 여전히 통과하지만, 새 문법에서는 강타입 쪽으로 기본값을 옮긴 것이다. enum → enum class, NULL → nullptr, C 캐스트 → 4종 캐스트 분리까지, "모던 C++"의 상당 부분은 사실 **약타입 유산을 강타입 방향으로 되감는 작업**이라고 요약할 수 있다.

트레이드오프도 말할 수 있어야 한다. 정적 타입의 비용은 표현의 경직성(맞는 프로그램도 증명이 안 되면 거부)과 컴파일 시간이고, 이득은 오류의 조기 발견 + **최적화 정보**다 — 타입이 컴파일 타임에 확정되면 필드 오프셋, 호출 대상, 레지스터 배치가 전부 정적으로 결정돼 동적 디스패치 비용이 사라진다. 게임 엔진이 정적 타입 언어를 쓰는 이유의 절반은 안전이 아니라 이 성능이다.

### 타입 시스템 (2) — 타입 검사 vs 타입 추론

- **타입 검사(type checking)**: 프로그래머가 쓴 타입 주석이 규칙에 맞는지 **확인**한다. AST를 상향식으로 순회하며 각 노드에 타입을 붙이고, 부모 노드의 규칙("`+`의 두 피연산자는 산술 타입"...)과 대조한다.
- **타입 추론(type inference)**: 주석 없이도 타입을 **계산**해 낸다.

타입 검사가 AST에 무엇을 하는지 그림으로 보면 (`int a; double b, c;`에서 `double d = a + b * c;`):

```
            =  : double
           / \
          d   +  : double        ← double + double : OK
             / \
   ┌────────┐   \
   │(암묵 변환│    *  : double     ← double * double : OK
   │ int →   │   / \
   │ double) │  b   c
   └───┬────┘
       a : int                    ← int이 double과 만남
                                    → 컴파일러가 변환 노드를 '삽입'
```

리프에서 타입이 올라오고, 타입이 안 맞는 지점에서 컴파일러는 (a) 암묵적 변환 노드를 삽입해 맞추거나 (b) 에러를 낸다. **"검사"라고 부르지만 실제로는 AST를 고쳐 쓰는(변환 노드 삽입) 단계**라는 게 포인트다. `clang -Xclang -ast-dump`로 보면 `ImplicitCastExpr` 노드가 실존하는 걸 확인할 수 있다.

추론의 정점이 힌들리-밀너(Hindley-Milner) 계열이다 — ML/Haskell이 쓰는 방식으로, 함수 본문 전체에서 타입 방정식을 수집한 뒤 단일화(unification)로 연립방정식 풀듯 해를 구한다. 주석이 하나도 없어도 함수 전체의 가장 일반적인 타입(principal type)을 찾아낸다. **전역적(global) 추론**이다.

C++의 `auto`와 템플릿 인자 추론은 이것과 다르다 — **지역적(local) 추론**이다:

```cpp
auto it = m_Map.find(ID);   // find의 반환 타입에서만 추론. 이후 사용처는 안 본다.
```

`auto`는 **초기화 식 하나만 보고** 결정하며(규칙 자체가 함수 템플릿 인자 추론과 동일 — [cpp/07](../cpp/07_templates_generic.md) §2.4), 뒤에서 `it`을 어떻게 쓰는지는 추론에 반영되지 않는다. 반환 타입 `auto`도 return 문들에서만 추론한다. C++이 전역 추론을 안 하는(못 하는) 이유는 언어 설계에 있다: **오버로딩 + 암묵적 변환 + 서브타이핑**이 있으면 타입 방정식의 해가 유일하지 않아 단일화가 성립하지 않는다. HM이 되는 언어들은 거꾸로 이 기능들을 제한한 언어다. "auto는 타입을 생략하는 편의 문법이고, HM 같은 전역 추론과는 급이 다른 지역 추론"이라고 위치를 잡아 말하면 이 구분을 안다는 신호가 된다.

### 타입 시스템 (3) — 암묵적 변환과 오버로드 해석: 컴파일러 관점

C++에서 `f(3.14f)`를 컴파일할 때 컴파일러가 하는 일은 "이름 f의 후보들 중, 인자를 **가장 싼 변환**으로 받을 수 있는 하나를 고르는 탐색"이다:

```
이름 룩업(후보 함수 집합) → 실행 가능 함수(viable) 선별 → 변환 비용 비교 → 유일 최선 선택
                                                              └ 동률이면 모호성(ambiguity) 에러
```

변환 비용의 순위(암묵적 변환 순서(implicit conversion sequence)의 범주):

| 순위 | 범주 | 예 |
|---|---|---|
| 1 | 완전 일치(exact match) | 동일 타입, 배열→포인터, cv 한정 추가 |
| 2 | 승격(promotion) | `char`→`int`, `float`→`double` |
| 3 | 표준 변환(standard conversion) | `int`→`double`, `Derived*`→`Base*`, 포인터→`bool` |
| 4 | 사용자 정의 변환(user-defined conversion) | 변환 생성자, `operator T()` — **한 체인에 최대 1회** |
| 5 | 생략 부호(ellipsis) | `...` 매칭 |

순위표가 실제로 결정을 가르는 예:

```cpp
void f(int);
void f(double);

f(3.14f);   // float 인자.
            // float→double = 승격(2순위), float→int = 표준 변환(3순위)
            // → f(double) 선택. 순위가 같았다면 C2668 ambiguous.
```

이걸 컴파일러 관점으로 다시 말하면: **오버로드 해석(overload resolution)은 의미 분석 단계의 제약 충족 탐색이고, 암묵적 변환은 컴파일러가 AST에 몰래 삽입하는 변환 노드다.** "암묵적"은 소스에 안 보인다는 뜻이지, 표현에 없다는 뜻이 아니다 — 위에서 본 `ImplicitCastExpr`가 IR 생성 단계에서 `sitofp`(signed int to floating point) 같은 실제 변환 명령이 된다. C++의 `explicit` 키워드는 "내 생성자를 이 탐색의 4순위 후보에 넣지 말라"는 지시로 읽으면 정확하고, `= delete`는 "이 후보가 선택되면 그 자체로 에러를 내라"는 지시다 — 문법 조각들이 전부 이 탐색 알고리즘에 대한 개입으로 통일되게 읽힌다.

### 타입 시스템 (4) — 서브타이핑

서브타이핑(subtyping)은 "S가 T의 서브타입이면, T를 기대하는 모든 자리에 S를 넣어도 된다"는 관계다(리스코프 치환 원칙(Liskov substitution principle)의 타입 이론 버전). 판정 방식이 두 갈래:

- **이름 기반(nominal)**: 선언으로 관계를 맺는다. C++/Java의 상속 — `class Derived : public Base`라고 **써야만** 서브타입. C++에서 `Derived*`→`Base*`가 표준 변환(위 표의 3순위)에 들어있는 게 서브타이핑의 언어 내 실체다.
- **구조 기반(structural)**: 모양이 맞으면 서브타입. Go의 인터페이스, TypeScript. 선언적 관계가 필요 없다.

C++의 재미있는 위치: 런타임 다형성은 이름 기반이지만, **템플릿은 컴파일 타임 구조 기반**이다 — `T`가 `.Release()`만 갖고 있으면 어떤 타입이든 통과한다(컴파일 타임 덕 타이핑). C++20 concepts는 이 구조적 요구사항을 명시적 서명으로 끌어올린 것이다. 그리고 값 의미론 언어라 서브타이핑에 함정이 하나 있다 — `Base b = derived;`는 치환이 아니라 **슬라이싱(slicing)**이다. 서브타입 치환이 안전한 건 포인터/참조를 통할 때뿐이라는 것까지 말하면 완결된다.

서브타이핑이 타입 시스템 규칙으로 실제 문법에 새어나온 예가 공변 반환 타입(covariant return type)이다 — 가상 함수를 오버라이드할 때 반환 타입만은 `Base*`→`Derived*`로 좁히는 게 허용된다. "반환값은 서브타입으로 대체해도 호출자의 기대를 깨지 않는다"는 치환 원칙이 언어 규칙으로 승인된 몇 안 되는 지점이다(매개변수 쪽은 C++이 허용하지 않는다 — 매개변수를 바꾸면 오버라이드가 아니라 다른 함수 오버로드가 된다).

### AST에서 IR로 — 왜 또 하나의 표현이 필요한가

의미 분석까지 끝난 AST를 바로 기계어로 바꾸면 안 되나? 두 가지 이유로 안 한다.

**첫째, M×N 문제.** 언어가 M개, 타겟 CPU가 N개면 직접 번역기는 M×N개 필요하다. 공용 IR을 끼우면 프런트엔드 M개 + 백엔드 N개 = M+N개로 줄어든다 — 언어 3개 × 타겟 4개면 12개 대신 7개. LLVM이 정확히 이 구조다 — C++/Rust/Swift가 같은 LLVM IR로 내려오고, x86/ARM 백엔드를 공유한다. 새 언어를 만들면 LLVM IR까지만 내리면 모든 타겟과 수십 년치 최적화를 공짜로 얻는다(Rust/Swift/Julia가 실제로 그렇게 출발했다). GCC도 내부적으로 GIMPLE→RTL이라는 자체 IR 계층을 갖는다.

```
 C++ ──┐                       ┌── x86-64
 Rust ─┼──→ [ 공용 IR ] ──→ 최적화 ──┼── ARM64
 Swift ┘    (LLVM IR)          └── WASM
  (M개 프런트엔드)                 (N개 백엔드)
```

**둘째, 최적화하기 좋은 정규형(normal form).** AST는 소스 문법의 다양성을 그대로 안고 있다 — `for`/`while`/`do-while`/`goto`는 AST에서 다 다른 노드지만 본질은 같은 제어 흐름이다. 최적화를 AST 위에 짜면 문법 가짓수만큼 케이스가 늘어난다. IR은 이 다양성을 **아주 작은 명령 집합 + 명시적 제어 흐름**으로 뭉갠 정규형이라, 최적화 패스를 한 번만 짜면 모든 언어·모든 문법에 적용된다. "IR = 문법 설탕을 다 걷어낸, 최적화 알고리즘이 다루기 좋은 표준 자세"가 핵심 문장이다.

### 3-주소 코드(three-address code)

중간 수준 IR의 고전적 형태. 규칙은 하나 — **명령 하나에 연산자 하나, 피연산자 최대 3개(결과 1 + 입력 2)**:

```
소스:  d = a + b * c;

TAC:   t1 = b * c        ← 복합식이 임시 변수(temporary)로 분해된다
       t2 = a + t1
       d  = t2
```

왜 이렇게 쪼개는가: (1) 중첩 식이 사라져 **모든 중간 값에 이름이 생긴다** — 공통 부분식 제거(CSE) 같은 최적화가 "같은 우변을 가진 임시가 둘인가?"라는 단순 비교가 된다. (2) 명령 하나가 기계 명령 한두 개에 대응할 만큼 낮아서 백엔드로 내리기 쉽다. (3) 제어 흐름도 `goto L` / `ifFalse t goto L`의 두 가지로 정규화된다 — `for`/`while`/`do-while`의 차이가 여기서 소멸한다:

```
소스:  sum = 0;                    TAC:
       for (i = 0; i < n; ++i)          sum = 0
           sum += i;                    i = 0
                                  L1:   t1 = i < n
                                        ifFalse t1 goto L2
                                        t2 = sum + i
                                        sum = t2
                                        t3 = i + 1
                                        i = t3
                                        goto L1
                                  L2:   ...
```

### 기본 블록(basic block)과 제어 흐름 그래프(CFG)

3-주소 코드의 나열에 구조를 입히는 단위가 기본 블록이다: **중간 진입도 중간 탈출도 없는 최대 길이의 직선 코드** — 첫 명령으로 들어오면 마지막 명령까지 무조건 실행된다. 나누는 기준은 기계적이다: (1) 첫 명령, (2) 점프의 목적지(레이블), (3) 분기 바로 다음 명령이 각 블록의 시작(리더, leader)이 되고, 다음 리더 직전까지가 한 블록이다. 블록의 끝은 항상 분기/점프/리턴(터미네이터)이고, 블록들을 분기 간선으로 이으면 제어 흐름 그래프(control flow graph, CFG)가 된다. 위의 TAC를 자르면:

```
            ┌──────────────────┐
            │ B0 (entry)       │  sum = 0
            │                  │  i = 0
            └────────┬─────────┘
                     ▼
            ┌──────────────────┐
     ┌─────►│ B1 (루프 헤더)    │  t1 = i < n
     │      └───┬──────────┬───┘  branch t1 → B2, B3
     │     true ▼          ▼ false
     │  ┌───────────┐  ┌───────────┐
     │  │ B2 (몸체)  │  │ B3 (탈출)  │
     │  │ sum += i  │  └───────────┘
     │  │ i += 1    │
     │  └─────┬─────┘
     └────────┘ (back edge)
```

CFG가 있어야 나오는 개념들: 도달 불가능 코드 = 그래프에서 entry로부터 못 가는 노드. 루프 = 역방향 간선(back edge)이 있는 사이클. 데이터플로 분석(liveness, 도달 정의) = CFG 위의 고정점 계산. **"루프를 문법(for)이 아니라 그래프의 사이클로 정의한다"**는 관점 전환이 미들엔드의 정체성이다 — 그래서 goto로 짠 루프도 똑같이 최적화된다.

### SSA(static single assignment) — 모든 변수는 한 번만 태어난다

3-주소 코드의 남은 문제: 같은 변수에 여러 번 대입하면 "이 `x`는 어느 대입의 `x`인가"를 알기 위해 코드를 거슬러 올라가야 한다. SSA는 이 문제를 형태로 풀어버린다 — **모든 변수가 프로그램 전체에서 정확히 한 번만 대입되도록 이름을 갈라 붙인다**:

```
원본:   x = 1          SSA:   x1 = 1
        x = x + 2             x2 = x1 + 2
        y = x * 3             y1 = x2 * 3
```

효과가 극적이다. `x2`의 정의는 유일하므로, **"이 값이 어디서 왔는가"(use→def)와 "이 값이 어디서 쓰이는가"(def→use)가 이름만으로 명시화**된다. 그러면:

- **상수 전파(constant propagation)**: `x1 = 1`이면 `x1`의 모든 사용처는 무조건 1 — 흐름 분석 없이 치환 가능.
- **죽은 코드 제거(dead code elimination)**: 사용처가 0인 정의는 무조건 죽은 코드 — 사용 리스트가 비었는지만 보면 된다.
- 값 번호 매기기(GVN), 조건 상수 전파(SCCP) 등이 전부 **정의-사용 그래프 위의 그래프 순회/워크리스트 알고리즘**이 된다.

즉 SSA의 본질은 "데이터 흐름 분석의 결과를 IR의 이름 규칙 안에 미리 구워 넣은 것"이다. 최적화가 '분석 후 변환'에서 '그래프 문제'로 바뀐다.

이름의 "static"도 오해하기 쉬운데, **텍스트상(정적으로) 대입문이 하나**라는 뜻이지 실행 횟수가 한 번이라는 뜻이 아니다 — 루프 안의 `x2 = x1 + 1`은 동적으로는 수만 번 실행될 수 있다. "소스 텍스트에서 각 이름의 정의 지점이 유일하다"가 정확한 진술이다.

### phi 함수 — 합류점의 필연

SSA에는 구조적 구멍이 하나 있다. 제어 흐름이 **합류(join)**하면?

```
        entry
       /      \
  x1 = 1     x2 = 2
       \      /
        join:
        y = x???        ← x1인가 x2인가? 어느 길로 왔는지에 달렸다
```

컴파일 타임에는 어느 경로로 올지 모른다. 그래서 SSA는 특수 명령 phi(φ)를 도입한다:

```
        join:
        x3 = φ(x1 from 왼쪽블록, x2 from 오른쪽블록)
        y1 = x3
```

φ는 "**어느 선행 블록에서 왔느냐에 따라 값을 고르는**" 명령이다. 실제 CPU에 φ 명령은 없다 — SSA라는 표현을 성립시키기 위한 표기 장치이고, 코드 생성 직전에 SSA를 해제(out-of-SSA)하면서 각 선행 블록 끝에 복사(`x3 = x1` / `x3 = x2`)로 풀려 사라진다(레지스터 할당기가 이 복사 대부분을 다시 없앤다). φ를 어느 합류점에 놓을지는 지배 경계(dominance frontier) 계산으로 최소화하는 게 표준 알고리즘이다(개념만 알면 충분: "이 정의의 영향권이 끝나는 경계에만 φ를 놓는다").

루프에서는 헤더 블록의 φ가 "초기값 또는 이전 반복의 값"을 고른다. 아까의 sum 루프를 SSA LLVM IR로 내리면:

```llvm
define i32 @sum_to(i32 %n) {
entry:
  br label %loop

loop:                                            ; 루프 헤더 = 합류점
  %i   = phi i32 [ 0, %entry ], [ %i.next, %body ]
  %sum = phi i32 [ 0, %entry ], [ %sum.next, %body ]
  %cmp = icmp slt i32 %i, %n
  br i1 %cmp, label %body, label %exit

body:
  %sum.next = add i32 %sum, %i
  %i.next   = add i32 %i, 1
  br label %loop                                 ; back edge

exit:
  ret i32 %sum
}
```

헤더의 φ 두 개가 "처음 진입이면 0, 몸체에서 돌아온 거면 이전 반복의 값"을 고른다. **루프 변수란 곧 '루프 헤더에 φ가 있는 값'**이라는 게 SSA의 정의고, 귀납 변수 분석·루프 불변 코드 이동·벡터화가 전부 이 φ를 출발점으로 삼는다. `mutable` 변수가 하나도 없는데 루프가 돈다는 것 — 함수형 언어의 재귀와 사실상 같은 형태가 된다는 것도 SSA의 흥미로운 성질이다.

### 실제 LLVM IR 읽기 — max 함수

```c
int mymax(int a, int b) {
    if (a > b) return a;
    return b;
}
```

이것을 SSA 형태의 LLVM IR로 손으로 내리면(φ가 보이도록 분기 형태 유지):

```llvm
define i32 @mymax(i32 %a, i32 %b) {
entry:
  %cmp = icmp sgt i32 %a, %b
  br i1 %cmp, label %then, label %else

then:
  br label %ret

else:
  br label %ret

ret:
  %r = phi i32 [ %a, %then ], [ %b, %else ]
  ret i32 %r
}
```

줄 단위로:
- `define i32 @mymax(i32 %a, i32 %b)` — 전역 심볼은 `@`, 지역 값(SSA 값)은 `%`. 타입이 모든 값에 명시된다(`i32` = 32비트 정수). **타입이 붙은 3-주소 코드**임이 바로 보인다.
- `entry:` / `then:` / `ret:` — 기본 블록 레이블. 각 블록의 마지막은 반드시 터미네이터(`br`/`ret`).
- `%cmp = icmp sgt i32 %a, %b` — 정수 비교(integer compare), `sgt` = signed greater than. 결과는 `i1`(1비트 불리언). C의 "int가 곧 bool" 같은 애매함이 IR에는 없다 — 정규형이니까.
- `br i1 %cmp, label %then, label %else` — 조건 분기. CFG의 간선이 소스가 아니라 명령에 명시돼 있다.
- `%r = phi i32 [ %a, %then ], [ %b, %else ]` — 합류점의 φ. "`%then`에서 왔으면 `%a`, `%else`에서 왔으면 `%b`". 이 함수의 반환값이 제어 흐름의 함수라는 사실이 데이터로 표현된 것.
- `ret i32 %r` — 반환.

참고로 실제 clang -O1은 이 단순한 패턴을 φ 대신 `select i1 %cmp, i32 %a, i32 %b`(조건 선택 명령) 하나로 접어버린다 — 분기 자체를 없애는 if-conversion의 가장 작은 예다. 직접 확인하는 법: `clang -S -emit-llvm -O1 max.c -o -`. 면접에서 "IR을 실제로 읽어본 적 있나"는 이 한 번의 실험으로 답이 갈린다.

### 컴파일러 관점의 값 카테고리 — lvalue/rvalue는 "주소가 있는가"

C++의 lvalue/rvalue 구분은 문법 규칙처럼 배우지만, 기원은 IR 수준의 질문이다: **이 표현식의 결과는 메모리 위치(location)인가, 값(value) 그 자체인가.**

- **lvalue** = 위치가 있는 표현식. IR에서는 **주소(포인터)**로 표현된다 — `store`의 목적지, `load`의 출처가 될 수 있다.
- **rvalue(prvalue)** = 순수한 값. IR에서는 **SSA 값**(사실상 가상 레지스터)으로만 존재해도 된다 — 메모리에 있을 필요가 없다.

`x = x + 1`에서 좌변 `x`는 주소(store 목적지), 우변 `x`는 값(load 결과)이다. 같은 이름이 문맥에 따라 다른 것으로 쓰이는 걸 lvalue-to-rvalue 변환이라 부르는데, IR로 보면 정체가 그냥 `load`다:

```llvm
%t = load i32, ptr %x.addr      ; lvalue-to-rvalue 변환의 실체
%s = add i32 %t, 1
store i32 %s, ptr %x.addr       ; 좌변 lvalue의 실체
```

이 관점이면 규칙들이 유도된다. **`&3`이 안 되는 이유**: 3은 레지스터에만 존재해도 되는 값이라 주소라는 개념 자체가 없다. **참조가 lvalue를 요구하는 이유**: 참조는 IR 수준에서 주소 전달이므로 주소 있는 대상이 필요하다. **최적화와의 연결**: clang -O0은 모든 지역 변수를 `alloca`(스택 슬롯)로 만들어 전부 lvalue답게 메모리에 두지만, mem2reg 패스가 "주소를 취한 적 없는 변수"를 SSA 값으로 승격시킨다 — **주소가 필요 없다고 증명된 lvalue는 메모리 존재 자체가 사라진다.**

```c
int three(void) { int x = 3; return x; }
```

```llvm
; -O0: x가 성실하게 '주소 있는 lvalue'로 존재
define i32 @three() {
  %x = alloca i32              ; 스택 슬롯 확보
  store i32 3, ptr %x          ; x = 3
  %t = load i32, ptr %x        ; return을 위한 lvalue→rvalue 변환
  ret i32 %t
}

; -O1 (mem2reg + 상수 전파 후): x의 메모리 실체가 소멸
define i32 @three() {
  ret i32 3
}
```

`&x`를 한 번이라도 쓰면 이 승격이 막히니(주소가 어디로 흘러갔는지 추적 못 하면 메모리에 남겨야 한다), "지역 변수의 주소 취하기는 최적화를 방해할 수 있다"는 실무 감각도 여기서 나온다.

C++11의 3분류는 이 이분법의 정제다:

| 카테고리 | identity(주소) | 이동 가능 | 예 |
|---|---|---|---|
| lvalue | 있음 | 아니오 | 변수 이름, `*p`, 배열 원소 |
| xvalue | 있음 | **예** | `std::move(x)`, rvalue 참조 반환 |
| prvalue | 없음 | (재료 자체) | 리터럴 `3`, `a+b`, 임시 객체 식 |

glvalue = {lvalue, xvalue}(주소 있음), rvalue = {xvalue, prvalue}(이동 가능). 이동 의미론은 "주소는 있지만 곧 죽을 값의 내용물은 훔쳐도 된다"는 규칙이므로, xvalue는 "lvalue의 몸에 rvalue의 권리"라고 요약할 수 있다. `std::move`가 IR에 아무 코드도 만들지 않는 캐스트(카테고리 딱지 변경)일 뿐이라는 사실도 이 표에서 유도된다 — 실제 '이동'은 그 딱지를 보고 오버로드 해석이 이동 생성자를 고르는 순간 일어난다.

## ③ 심화 — 꼬리질문 대비

**"심볼 테이블을 스택 오브 맵 말고 다르게 구현할 수도 있나?"**
있다. 대표 대안이 "이름 하나당 선언 스택"을 매다는 방식이다 — 전역 해시맵 하나에 `x → [블록의 x, 함수의 x]`처럼 같은 이름의 선언들을 스택으로 쌓고, 스코프 탈출 시 그 스코프에서 밀어넣은 것들만 pop한다. 룩업이 스코프 깊이와 무관하게 해시 한 번이라 빠르다. 다만 어느 구현이든 의미는 같다: 렉시컬 스코프 = 진입/탈출의 push/pop 규율. C++은 여기에 네임스페이스, 클래스 스코프, ADL(인자 의존 룩업), 템플릿 2단계 이름 룩업이 겹쳐서 이름 해석이 언어 전체에서 가장 복잡한 축에 든다 — "C++ 파싱/의미 분석이 어렵다"는 통설의 실체가 이 룩업 규칙 더미다.

**"선언만으로 타입 검사가 되는데 왜 정의가 필요한가?"**
컴파일러 단계별로 필요한 정보가 다르기 때문이다. 호출 코드를 만드는 데는 시그니처(선언)면 충분하다 — 인자 배치와 반환 타입이 결정되니까. 그러나 그 함수의 본체 기계어를 만들려면 정의가 필요하고, 인라이닝하려면 호출자 쪽에서 정의가 보여야 한다(그래서 헤더에 정의가 있는 인라인/템플릿 함수가 인라이닝에 유리하고, LTO(link-time optimization)는 이 한계를 링크 시점 IR로 뚫는 기술이다). "선언 = 인터페이스 계약, 정의 = 구현 실체, 그 사이를 잇는 게 심볼과 링커"로 답하면 된다.

**"ODR 위반은 왜 어떤 건 링크 에러고 어떤 건 조용히 지나가나?"**
비인라인 함수의 중복 정의는 두 오브젝트 파일이 같은 강한(strong) 심볼을 내놓으므로 링커가 충돌을 본다 — 에러. 반면 inline 함수/템플릿 인스턴스는 설계상 여러 TU에 중복 정의되는 게 정상이라, 링커는 "같겠거니" 하고 하나만 골라 병합한다(MSVC의 COMDAT 선택). 이때 정의들이 실제로는 달랐다면(예: 헤더가 매크로 설정에 따라 다르게 컴파일됨) 링커는 다른 걸 비교 없이 버리고, 프로그램은 ill-formed지만 진단 의무가 없다(NDR). "검사 주체가 구조적으로 없는 규칙"의 예로 이만한 게 없다.

**"C++은 강타입 언어인가?"**
축부터 분리해 답한다: 정적 타입인 것은 명확하고, 강/약은 스펙트럼이다. C++은 기본 규칙은 엄격하지만(참조 바인딩, cv, enum class) C 유산인 암묵적 축소 변환·산술 승격과 명시적 탈출구(reinterpret_cast, C 캐스트)를 갖는다. 핵심 차별점은 탈출이 **명시적 문법을 요구**한다는 것 — 코드 리뷰에서 grep 가능한 형태로 위험이 표시된다. "강타입이지만 탈출구를 문법으로 노출한 언어"가 한 줄 답.

**"타입 검사는 AST의 어떤 순회로 구현되나?"**
기본은 상향식(bottom-up) 합성이다: 리프(리터럴, 식별자→심볼 테이블 룩업)에서 타입이 정해지고, 연산자 노드가 자식들의 타입으로 자기 타입을 계산하며 올라간다. 단 하향식 문맥이 필요한 지점이 있다 — C++의 오버로드된 함수 이름이나 `{1,2}` 같은 초기화 리스트는 자기 혼자서는 타입이 없고 "기대되는 타입"이 내려와야 정해진다. 그래서 실제 구현은 합성(synthesized)과 상속(inherited) 속성이 섞인 순회다. 이 어휘(속성 문법)를 알고 있으면 컴파일러 수업을 제대로 들었다는 신호.

**"Hindley-Milner를 아는 만큼만 설명해 보라."**
함수 본문 전체에서 "이 값과 저 값의 타입은 같아야 한다"는 방정식들을 모으고, 단일화라는 절차로 방정식을 동시에 만족하는 가장 일반적인 해(principal type)를 구하는 전역 추론 체계다. ML/Haskell 계열이 채택했고, 주석 없이도 다형적 타입까지 찾아준다. 대신 언어가 오버로딩·암묵 변환·서브타이핑을 제한해야 성립한다. C++/Rust의 지역 추론(auto, let)은 이것과 다른, 초기화 식 단위의 추론이다 — 이 위치 구분만 정확히 말해도 충분하고, 알고리즘 세부(W 알고리즘)까지 파고들 필요는 없다.

**"오버로드 해석이 '탐색'이라면 최악의 경우는?"**
후보가 많고 인자마다 변환 경로가 여럿이면 조합적으로 비싸진다. 여기에 템플릿이 끼면 후보 생성 자체가 인스턴스화 시도(치환)라서, SFINAE — 치환 실패는 에러가 아니라 후보 탈락 — 규칙까지 얽힌다. 실무 증상은 두 가지: (1) 컴파일 시간 폭증, (2) "ambiguous call" 에러 — 변환 비용이 동률인 후보 둘. 후자는 컴파일러가 임의로 골라주지 않는 게 설계 의도다: 조용히 하나를 고르면 오버로드 추가가 기존 호출의 의미를 소리 없이 바꿀 수 있기 때문에, 동률은 프로그래머에게 반환한다.

**"IR이 하나가 아니라 여러 계층인 이유는?"**
최적화마다 편한 추상화 수준이 다르기 때문이다. 언어 고유 최적화(예: C++ 임시 객체 제거)는 소스 개념이 남아있는 고수준 IR이 편하고, 상수 전파·루프 최적화는 SSA 중수준 IR, 명령 스케줄링·레지스터 할당은 기계 명령에 가까운 저수준 IR이 필요하다. 그래서 실물 컴파일러는 IR 사다리를 내려간다 — GCC는 GENERIC→GIMPLE→RTL, LLVM 계열은 (언어별 AST/MLIR)→LLVM IR→MachineIR, Rust는 HIR→MIR→LLVM IR. "점진적 낮춤(progressive lowering)"이라는 단어를 쓰면 정확하다.

**"3-주소 코드와 JVM 바이트코드 같은 스택 머신 코드의 차이는?"**
둘 다 IR이지만 값 전달 모델이 다르다. 3-주소 코드는 모든 중간 값에 이름(가상 레지스터)을 붙이는 레지스터 머신 모델이고, JVM 바이트코드는 피연산자 스택에 push/pop하는 스택 머신 모델이다(`a+b*c` → `load a, load b, load c, mul, add`). 스택 코드는 조밀하고 생성이 쉬워 배포 포맷에 유리하고(그래서 JVM/WASM), 이름 있는 3-주소/SSA는 정의-사용 추적이 쉬워 최적화에 유리하다. 그래서 JIT도 내부에서는 바이트코드를 SSA로 다시 올려서 최적화한다 — 포맷은 스택, 최적화는 SSA라는 분업.

**"phi 함수는 실제 CPU에 없는데 최종 코드에서 어떻게 되나?"**
SSA 해제 단계에서 각 선행 블록의 끝에 복사 명령으로 풀린다 — `x3 = φ(x1, x2)`면 왼쪽 블록 끝에 `x3 ← x1`, 오른쪽 블록 끝에 `x3 ← x2`. 그 복사들도 레지스터 할당기가 두 값을 같은 레지스터에 배정(coalescing)하면 사라진다. 즉 φ는 "표현을 위한 가상 명령이고, 비용은 대부분 0으로 수렴한다". 병렬 복사 순서 문제(swap 문제) 같은 세부가 있다는 것 정도만 알아두면 꼬리질문 방어가 된다.

**"SSA가 구체적으로 어떤 최적화를 어떻게 쉽게 만드나, 예를 하나만."**
조건 상수 전파(SCCP)가 좋은 예다. `x1 = 1`처럼 정의가 유일하므로 "x1은 상수 1"이라는 사실이 조건 없이 성립하고, x1의 사용처 리스트를 따라가며 치환하면 된다. 치환으로 `br i1 true`가 되면 한쪽 간선이 죽고, 죽은 간선을 CFG에서 지우면 그 블록에서 오던 φ 입력도 줄어 또 상수가 될 수 있다 — 사실이 그래프를 타고 전파되는 워크리스트 알고리즘. SSA가 아니면 같은 결론을 얻는 데 "이 지점에 도달하는 정의 집합"을 CFG 전체 고정점 계산으로 구해야 한다(도달 정의 분석). SSA는 그 계산 결과를 이름에 새겨둔 것이다.

**"lvalue/rvalue를 IR 관점으로 설명해 보라."**
lvalue는 주소로 표현되는 표현식(load/store의 대상), rvalue는 SSA 값·레지스터로만 존재해도 되는 표현식. lvalue-to-rvalue 변환 = load. `&3` 불가 = 레지스터 값에는 주소가 없다. 그리고 -O0의 alloca 투성이 IR이 mem2reg로 SSA 값이 되는 것은 "주소가 안 필요한 lvalue는 메모리 실체를 잃는다"는 뜻이다. 이 답에 C++11 3분류(glvalue/prvalue/xvalue)와 "xvalue = 주소는 있으나 곧 죽어서 자원을 훔쳐도 되는 값"을 얹으면 이동 의미론까지 한 호흡에 연결된다.

**"템플릿 인스턴스화는 이 파이프라인 어디에서 일어나나?"**
의미 분석 단계다. 템플릿 정의 자체는 코드가 아니라 설계도로 저장되고(2단계 이름 룩업의 1단계 — 비의존 이름만 미리 검사), 구체 타입으로 사용되는 순간 치환→새 AST 생성→그 AST에 대한 의미 분석/타입 검사→IR 생성이 타입별로 반복된다. 그래서 템플릿 에러가 사용 지점에서 터지고, 인스턴스 수만큼 IR이 늘어난다(코드 팽창). "컴파일 타임 코드 생성기가 프런트엔드 안에서 돈다"는 그림이 잡혀 있으면 된다. 각론은 [cpp/07](../cpp/07_templates_generic.md).

**"C++ 캐스트 4종은 컴파일러에게 각각 무엇인가?"**
`static_cast` = 타입 시스템이 아는 변환 경로(승격/표준 변환/상속 위계)를 명시적으로 요청 — IR에 실제 변환 명령(`sitofp`, 포인터 오프셋 조정 등)이 생길 수 있다. `reinterpret_cast` = "비트는 그대로 두고 타입 딱지만 바꿔라" — 대개 IR 수준에서 명령이 거의 없다(타입 시스템에 대한 선언일 뿐). `const_cast` = 타입 주석(cv)만 제거, 코드 생성 없음. `dynamic_cast` = 유일하게 런타임 코드가 삽입되는 캐스트 — RTTI를 조회하는 함수 호출이 IR에 생긴다. "네 캐스트의 차이 = 컴파일러가 삽입하는 코드의 차이"로 답하면 문법 암기가 아니라 원리 이해로 들린다.

**"심볼 테이블은 선언보다 앞선 사용(전방 참조)을 어떻게 처리하나?"**
언어 규칙에 달렸다. C/C++의 함수/변수는 "선언이 먼저"가 원칙이라 테이블 미스 = 즉시 에러다. 그런데 클래스 멤버는 예외다 — 멤버 함수 본문 안에서는 **그보다 뒤에 선언된 멤버도 보인다.** 컴파일러가 클래스 정의를 끝까지 읽어 멤버를 전부 테이블에 넣은 뒤에 인라인 멤버 함수 본문을 분석하기 때문이다(지연 파싱). "왜 멤버끼리는 순서가 자유로운가"라는 흔한 의문의 답이 이 처리 순서다. Java/C#처럼 전방 참조가 전면 허용되는 언어는 아예 2패스(선언 수집 → 본문 분석)가 기본 구조다.

**"구조적 타이핑이 더 유연한데 왜 C++/Java의 런타임 다형성은 이름 기반인가?"**
우연한 일치(accidental conformance)를 계약으로 착각하는 걸 막기 위해서다 — `Draw()`를 가진 카우보이와 `Draw()`를 가진 도형이 구조적으로는 같은 타입이 된다. 이름 기반은 "이 관계는 의도된 것"이라는 선언을 요구해 의미 실수를 차단하고, 구현 관점에서도 vtable 레이아웃을 선언 시점에 고정할 수 있어 디스패치가 오프셋 접근 한 번이 된다. 구조 기반은 유연하지만 매칭 검사나 런타임 메타데이터가 필요하다. C++은 그래서 이원화했다 — 런타임은 이름 기반(vtable), 컴파일 타임은 구조 기반(템플릿/concepts)으로 양쪽 장점을 시점을 나눠 취한다.

**"동적 타입 언어에는 이 챕터의 단계들이 없나?"**
있다 — 시점이 런타임으로 밀릴 뿐이다. Python의 `a + b`는 실행할 때마다 a의 타입을 조회해 디스패치하고(그게 느린 이유), 심볼 테이블은 딕셔너리(`__dict__`, 스코프 체인)로 런타임에 실존한다. 고성능 JS 엔진(V8 등)은 여기서 더 나아가 **런타임에 타입을 관찰(type feedback)해서 "이 지점은 늘 정수였다"는 가정으로 JIT 컴파일**하고, 가정이 깨지면 탈최적화(deoptimization)로 되돌린다. 즉 "정적 vs 동적"은 검사를 하냐 마냐가 아니라 **언제, 몇 번 하느냐**의 문제고, JIT는 동적 언어에 정적 컴파일의 이득을 사후적으로 끼워 넣는 기술이다.

**"예외(exception)는 CFG를 어떻게 복잡하게 만드나?"**
예외를 던질 수 있는 모든 호출이 잠재적 탈출 간선이 된다 — "이 블록은 끝까지 실행된다"는 기본 블록의 전제가 흔들린다. 그래서 LLVM은 예외 가능 호출을 일반 `call`이 아니라 `invoke`(정상 계속 블록 + 예외 착지 블록의 두 후속을 가진 터미네이터)로 표현해, 예외 흐름도 CFG의 명시적 간선으로 만든다. 예외가 최적화를 방해한다는 통설의 절반은 이 간선 폭증(코드 이동 제약) 이야기고, `noexcept`가 성능 힌트가 되는 이유도 "이 호출에서는 탈출 간선을 안 그려도 된다"는 CFG 단순화로 설명된다.

**"최적화를 IR에서 하지 왜 어셈블리 수준에서 안 하나?"**
정보의 비대칭 때문이다. IR에는 타입, 정의-사용 관계, 별칭 정보, 함수 경계가 살아 있지만, 어셈블리는 이미 레지스터가 배정되고 명령이 선택된 뒤라 "이 두 값이 같은 변수였다", "이 포인터는 저 배열을 절대 안 가리킨다" 같은 사실이 소실됐다. 잃어버린 정보를 역공학하는 것보다 정보가 살아있을 때 변환하는 게 압도적으로 쉽다. 물론 백엔드에도 최적화는 있다(명령 스케줄링, 핍홀(peephole)) — 기계 세부가 필요한 것만 뒤로 미룬다는 분업이다.

## ④ Winters에서의 적용

이 챕터는 OS/컴파일러를 수업으로만 배운 사람이 가장 밀리는 영역이라, "직접 만든 것"과의 연결이 특히 중요하다. Winters에는 컴파일러는 없지만, **컴파일러가 쓰는 자료구조와 판정을 런타임/컴파일 타임에 그대로 재현한 지점**들이 있다.

### (1) eChampion + Registry = 런타임에 직접 만든 '심볼 테이블'

`Shared/GameSim/Definitions/LoLMatchContext.h:5`의 `enum class eChampion : std::uint8_t`(IRELIA=1 … LEESIN=17, 명시값 고정)가 '이름'이고, `Shared/GameSim/Registries/ChampionStats/ChampionStatsRegistry.h:8`의 `CChampionStatsRegistry`가 그 이름을 정의로 해석하는 테이블이다 — 내부는 `std::unordered_map<eChampion, ChampionStatsDef>`(같은 파일 24행), 인터페이스는 `Add`(등록)와 `Find`(룩업).

심볼 테이블 비유가 구조적으로 정확한 지점은 **폴백 체인**이다. `Shared/GameSim/Registries/ChampionStats/ChampionStatsRegistry.cpp:24`의 `Resolve`는 자기 맵에서 `Find`가 실패하면 `ChampionGameDataDB::ResolveStats(ID)`로 내려간다 — 안쪽 스코프에서 못 찾으면 바깥 스코프로 나가는 스코프 체인 탐색과 같은 모양이고, "명시 등록이 기본값을 가린다"는 점에서 섀도잉과도 같다. 면접에서 "심볼 테이블을 이해했나"는 질문에 "제 프로젝트의 챔피언 레지스트리가 정확히 그 구조입니다 — 키로 룩업하고, 미스면 바깥 계층으로 폴백하고, 안쪽 등록이 바깥 기본값을 가립니다"라고 실물로 답할 수 있다.

차이도 말할 수 있어야 완결된다: 컴파일러의 심볼 테이블은 컴파일 타임에 소멸하는 자료구조이고, Winters의 레지스트리는 런타임에 살아있는 디스패치 테이블이다. 즉 같은 자료구조 패턴을 "언제 해석하느냐"만 달리 쓴 것 — 정적 해석(컴파일러)과 동적 해석(레지스트리)의 트레이드오프(유연성 vs 조기 검증)를 스스로 선택해 본 경험이 된다.

### (2) 템플릿 인스턴스화 = 내가 매일 쓰는 '컴파일 타임 코드 생성'

`Engine/Public/ECS/ComponentStore.h:6`의 `template<typename T> class CComponentStore`는 소스에 한 번 쓰여 있지만, `TransformComponent`용, `HealthComponent`용… 사용된 컴포넌트 타입 수만큼 **프런트엔드가 별개의 클래스를 찍어내고 각각 의미 분석·IR 생성을 거친다**(단형화, monomorphization). Java 제네릭이 타입 소거로 클래스 하나를 공유하는 것과 반대 전략이고, 그 대가(코드 팽창)와 이득(타입별 최적화, 간접 호출 없음)을 이 챕터의 어휘로 설명할 수 있다: 인스턴스마다 IR이 따로 생기므로, `T`가 POD면 복사가 `memcpy`급 코드로, 인라이닝까지 타입별로 최적화된다. 각론은 [cpp/07_templates_generic.md](../cpp/07_templates_generic.md).

### (3) static_assert = 프로그래머가 확장하는 의미 분석

`Engine/Private/RHI/DX11/DX11ConstantBuffer.h:15`의 `static_assert(sizeof(T) % 16 == 0, ...)`는 "HLSL 상수 버퍼는 16바이트 정렬"이라는 **도메인 규칙을 의미 분석 단계의 판정으로 승격**시킨 것이다. 컴파일러의 타입 검사가 "int에 포인터를 더하지 마라"를 거부하듯, 이 한 줄은 "정렬 안 맞는 구조체로 cbuffer를 만들지 마라"를 같은 시점(컴파일 타임)에 거부한다. 타입 시스템의 본질이 "잘못된 프로그램을 실행 전에 거부하는 판정 규칙"이라는 걸 이해했다면, static_assert/concepts는 그 판정 규칙을 사용자가 추가하는 문이라는 관점이 자연히 나온다.

### (4) auto의 '지역 추론'을 실물로

`Shared/GameSim/Registries/ChampionStats/ChampionStatsRegistry.cpp:20`의 `auto it = m_Map.find(ID);`가 지역 추론의 전형이다 — `find`의 반환 타입(이 경우 컨테이너의 const_iterator)에서만 추론이 일어나고, 이후 `it`의 사용처는 추론에 관여하지 않는다. 만약 C++이 HM식 전역 추론 언어였다면 `it != m_Map.end()` 같은 뒷줄들도 방정식에 들어갔을 것이다 — 이 대비를 자기 코드 한 줄로 말할 수 있으면 "auto를 쓴다"에서 "추론의 범위를 안다"로 답의 급이 올라간다.

### (5) enum class의 underlying type = 타입 시스템으로 못박은 wire 계약

`eChampion : std::uint8_t`의 명시적 기반 타입과 고정 명시값은, 타입 시스템 관점에서 "이 이름 집합의 메모리 표현은 1바이트이고 값은 재배치되지 않는다"는 계약을 컴파일러에게 검증시키는 것이다. 일반 enum이 약타입스럽게 int로 새어나가는 것과 달리 enum class는 암묵 변환이 차단된다 — 이 챕터의 언어로 말하면 "암묵적 변환 경로를 타입 정의에서 끊은 것". 직렬화 관점의 각론은 [cpp/12_network_serialization.md](../cpp/12_network_serialization.md)에 있다.

### 개념 ↔ Winters 대응표

| 이 챕터의 개념 | Winters의 실물 | 관계 |
|---|---|---|
| 심볼 테이블 룩업 | `CChampionStatsRegistry::Find` | 이름(enum)→정의(StatsDef) 사전 |
| 스코프 체인 폴백 | `Resolve`의 Find 실패 → GameDataDB | 안쪽 미스 → 바깥 계층 탐색 |
| 섀도잉 | 명시 `Add` 등록이 DB 기본값을 가림 | 안쪽 정의가 이김 |
| 의미 분석 판정의 확장 | `static_assert(sizeof(T) % 16 == 0)` | 도메인 규칙을 컴파일 타임 거부로 |
| 단형화(타입별 코드 생성) | `CComponentStore<T>` 인스턴스들 | 타입 수만큼 별도 클래스/IR |
| 지역 타입 추론 | `auto it = m_Map.find(ID)` | 초기화 식 단독 추론 |
| 타입 시스템의 변환 차단 | `enum class eChampion : uint8_t` | 암묵 int 변환 봉쇄 + 표현 고정 |

### 이 개념을 경험으로 말하는 법 (컴파일러를 안 만들어봤을 때)

정직한 프레이밍이 가장 강하다: "컴파일러를 직접 만들어 본 적은 없지만, 컴파일러가 쓰는 구조를 게임 코드에서 재발명해 봤다 — 이름→정의 룩업과 폴백 체인(레지스트리), 컴파일 타임 판정의 확장(static_assert), 타입별 코드 생성의 소비자(템플릿 ECS). 그리고 `clang -emit-llvm`으로 제 코드가 어떤 IR이 되는지 읽어봤다"까지 말하면, 수업 지식이 아니라 자기 코드에 접지된 지식으로 들린다. 마지막 문장은 실제로 해보고 말하자 — max 함수 하나면 10분이다.

한 단계 더 나가려면, 위 대응표의 역방향 질문에도 답을 준비한다: "레지스트리가 심볼 테이블과 **다른** 점은?" — 컴파일러의 테이블은 검증이 끝나면 버려지는 임시 구조라 미등록 이름이 곧 컴파일 에러지만, 런타임 레지스트리는 미등록 키가 런타임에야 발견된다. Winters가 `Resolve`에 폴백을 두고 `eChampion::END` 센티널을 두는 이유가 바로 이 '늦은 발견'의 방어다. 정적 해석을 포기하면 안전망을 런타임에 직접 짜야 한다는 것 — 이 트레이드오프를 자기 코드로 겪었다는 게 이 챕터에서 가져갈 수 있는 가장 좋은 면접 재료다.
