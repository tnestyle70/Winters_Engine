# PBR · 감마 보정 · HDR — 물리 기반 렌더링의 수학

> "PBR이 뭔가요? Cook-Torrance 수식을 설명해 보세요. 감마 보정은 왜 하나요? HDR 파이프라인은 어떻게 구성하나요?" — 렌더링 면접의 단골 3연타에 대한 답. 파이프라인 구조(RHI/DX11/FX 시스템)는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)가 담당하고, 이 챕터는 그 파이프라인 위에서 픽셀 셰이더가 계산하는 **빛의 수학**을 담당한다.

---

## ① 한 줄 본질

> **"PBR은 '보기 좋을 때까지 튜닝하는 임의 파라미터(shininess)'를 '물리량을 뜻하는 파라미터(roughness/metallic)'로 바꿔서, 어떤 조명 환경에서도 일관된 결과와 에너지 보존을 보장하는 프레임워크다. 그 수학적 뼈대는 렌더링 방정식 — 반구 위 모든 입사 방향의 radiance를 BRDF와 코사인으로 가중해 적분하는 것 — 이고, 실시간 렌더링 전체는 이 적분을 프레임 예산 안에서 근사하는 게임이다. 감마 보정은 '계산은 물리(linear), 저장·표시는 지각(sRGB)'이라는 공간 분리 규칙이고, HDR은 '씬의 빛은 [0,1]에 안 담긴다'는 사실을 인정하고 넓은 포맷으로 계산한 뒤 톤 매핑으로 디스플레이에 욱여넣는 파이프라인이다."**

세 주제는 별개가 아니라 한 문장으로 이어진다: **물리적으로 옳은 BRDF(PBR)를 → 물리적으로 옳은 공간(linear)에서 → 물리적으로 옳은 범위(HDR)로 계산한다.** 하나라도 빠지면 나머지 둘의 정확도가 무의미해진다.

---

## ② 기본 개념

### 왜 PBR인가 — 임의 파라미터의 죽음

전통(Phong/Blinn-Phong) 셰이딩의 파라미터를 보자.

| 축 | Blinn-Phong (레거시) | PBR (metallic/roughness) |
|---|---|---|
| 하이라이트 폭 | `shininess` 지수 (1~수천, 단위 없음) | `roughness` (0~1, 표면 미세 거칠기) |
| 스페큘러 색/강도 | `specularColor` 임의 지정 | 금속 여부(`metallic`)와 물리 상수(F0)에서 유도 |
| 재질 색 | diffuse 텍스처 (라이팅 구움 흔함) | `albedo` (라이팅 없는 순수 반사율) |
| 에너지 보존 | 없음 — diffuse+specular가 입사광을 초과 가능 | 구조적으로 보장 (`kd = 1 - ks`) |
| 조명 이식성 | 씬 바꾸면 재튜닝 | 어떤 조명에서도 일관 |

`shininess = 87` 같은 값은 물리량이 아니다. 아티스트가 특정 씬의 특정 조명 아래에서 보기 좋을 때까지 돌린 수치다. 그래서 그 에셋을 다른 조명의 씬에 넣으면 깨지고, 낮/밤 전환이 있는 게임에선 재질이 시간대마다 다르게 "잘못" 보인다. PBR의 세 가지 승리 이유:

1. **조명 불변성** — 파라미터가 표면의 물리 속성이므로, 빛이 바뀌어도 재질 정의는 그대로다. 동적 조명·TOD(time of day)·IBL 환경 어디서든 한 벌의 텍스처로 일관된다.
2. **워크플로 표준화** — Substance, Marmoset, UE, Unity, glTF가 전부 같은 metallic/roughness 언어를 쓴다. 아티스트가 툴을 옮겨도, 에셋이 엔진을 옮겨도 의미가 보존된다.
3. **에너지 보존** — 반사된 빛의 총량이 입사한 빛을 넘지 않도록 수식이 강제한다. 이게 없으면 GI(전역 조명)에서 빛이 반사될 때마다 증폭되어 씬이 폭발하고, 아티스트는 그걸 어둡게 하는 역방향 튜닝으로 싸우게 된다.

핵심 관점: PBR은 "더 예쁜 셰이더"가 아니라 **재질 파라미터의 의미론(semantics)을 표준화한 계약**이다.

### 방사측정(radiometry) 최소 세트 — 픽셀이 계산하는 단위는 무엇인가

PBR 수식을 읽으려면 네 가지 물리량의 사다리를 타야 한다. 각 단계는 "무엇으로 나눴는가"의 차이다.

| 물리량 | 정의 | 단위 | 직관 |
|---|---|---|---|
| 방사속(radiant flux) `Φ` | 초당 방출 에너지 | `W` | 전구가 뿜는 총 파워. "60W 전구" |
| 방사강도(radiant intensity) `I = dΦ/dω` | 방향(입체각)당 플럭스 | `W/sr` | 같은 60W라도 레이저는 한 방향에 몰빵 |
| 조도(irradiance) `E = dΦ/dA` | 면적당 받는 플럭스 밀도 | `W/m²` | 책상 위 한 점이 **모든 방향에서 합산해** 받는 양 |
| 휘도(radiance) `L = d²Φ/(dω · dA⊥)` | 방향 **그리고** 면적당 | `W/(sr·m²)` | "그 방향에서 그 점으로 오는 빛" — **픽셀이 계산하는 값** |

왜 radiance가 최종 단위인가:

- **위치와 방향을 둘 다 갖는다.** 카메라 픽셀 하나는 "이 지점에서 이 방향으로 나오는 빛"을 측정하는 장치다. 그게 정확히 radiance의 정의다.
- **광선을 따라 불변**이다(진공에서). 표면에서 출발한 radiance는 카메라에 닿을 때까지 값이 유지되므로, 표면에서 계산한 `Lo`를 그대로 픽셀 색으로 쓸 수 있다.
- irradiance는 radiance의 적분이다: `E = ∫_Ω Li(wi) · (n·wi) dwi`. "받는 총량"은 "각 방향에서 오는 것"을 코사인 가중으로 모은 것.

```
              n (법선)
              │      wi (입사 방향)
              │     /
              │    / θi
              │   /
      ────────┴──────────  표면 p
         ↘ wo (관찰 방향)

  반구 Ω = 법선 위쪽의 모든 방향 집합
  픽셀 = Lo(p, wo) 하나를 구하는 문제
```

### 렌더링 방정식 — 전부의 원점

Kajiya(1986)의 렌더링 방정식:

```
Lo(p, wo) = Le(p, wo) + ∫_Ω  fr(p, wi, wo) · Li(p, wi) · (n·wi) dwi
```

항 하나씩:

| 항 | 이름 | 의미 |
|---|---|---|
| `Lo(p, wo)` | 출력 radiance | 지점 p에서 관찰 방향 wo로 나가는 빛 = 픽셀의 답 |
| `Le(p, wo)` | 자발광(emission) | 표면 스스로 내는 빛. 네온, 용암, 스킬 이펙트 |
| `fr(p, wi, wo)` | BRDF | "wi에서 들어온 빛 중 얼마가 wo로 반사되나"의 재질 함수 |
| `Li(p, wi)` | 입사 radiance | 방향 wi에서 p로 도착하는 빛 (다른 표면의 Lo일 수도 있다 — 재귀!) |
| `(n·wi)` | 코사인 항 | 램버트 코사인 법칙 — 비스듬히 오는 빛은 같은 플럭스가 더 넓은 면적에 퍼지므로 밀도가 `cosθ`배로 줄어든다 |
| `∫_Ω ... dwi` | 반구 적분 | 빛은 한 방향이 아니라 **반구 전체**에서 온다 |

코사인 항의 직관: 손전등을 바닥에 수직으로 쏘면 동그란 밝은 원, 비스듬히 쏘면 길쭉하고 어두운 타원. 같은 에너지가 `1/cosθ`배 넓은 면적에 퍼지니 단위 면적당 받는 양은 `cosθ`배다. 겨울이 추운 이유와 같은 수학이다.

이 방정식이 무서운 이유는 두 가지다. (1) `Li`가 다른 표면의 `Lo`라서 **재귀적**이고(빛은 무한히 튕긴다), (2) 반구 **적분**은 닫힌 형태 해가 없다. 오프라인 렌더러(패스 트레이싱)는 몬테카를로 샘플링으로 수천 샘플을 쏘아 이걸 푼다. 실시간은 16ms 안에 픽셀 수백만 개를 처리해야 하므로:

- **직접광**: 광원이 점/방향 광원 몇 개라면 적분이 이산 합으로 붕괴한다 — `Σ fr · Li · (n·l)` (Winters의 forward 셰이더가 하는 일).
- **환경광**: 적분을 오프라인에 미리 구워둔다 (IBL — 아래에서).
- **가시성/그림자, AO, 반사**: 각각 별도의 근사 기법으로 쪼갠다.

**실시간 렌더링 전체가 "이 적분의 어느 조각을 어떤 예산으로 근사하나"의 게임이다.** 섀도 맵, SSAO, SSR, 라이트맵, 프로브 — 전부 이 적분의 부분항 근사다 (이 시리즈 07장에서 그 근사 기법들을 다룬다). 지도를 그려두면:

| 방정식의 조각 | 실시간 근사 | 담당 챕터 |
|---|---|---|
| `Σ fr·Li·(n·l)` (이산 광원) | 포워드/디퍼드 라이팅 루프 | 이 챕터 |
| `fr` (재질) | Cook-Torrance BRDF | 이 챕터 |
| `∫` 환경광 (diffuse/specular) | IBL — irradiance map / split-sum | 이 챕터 (개요) |
| `Li`의 가시성 (그림자) | 섀도 맵 | 07장 |
| 근거리 차폐 (ambient의 V항) | SSAO/GTAO | 07장 |
| `Li`가 다른 표면의 `Lo`인 재귀 | 라이트맵, 프로브, SSR, RTGI | 07장 |

### BRDF — 정의를 정확히

BRDF(bidirectional reflectance distribution function)의 정의:

```
fr(wi, wo) = dLo(wo) / dE(wi) = dLo(wo) / ( Li(wi) · (n·wi) · dwi )
```

**입사 조도(irradiance) 대비 반사 radiance의 비율**이다. 단위는 `1/sr`. 여기서 흔한 오해 하나를 미리 끊자 — BRDF는 "반사율(0~1)"이 아니다. radiance/irradiance라는 **밀도의 비**라서 값이 1을 훨씬 넘을 수 있다(거울에 가까운 표면의 하이라이트 중심에선 수백~수천). 에너지 보존은 개별 값이 아니라 적분에 걸린다.

물리적으로 유효한 BRDF의 성질 두 가지:

1. **헬름홀츠 상반성(Helmholtz reciprocity)**: `fr(wi, wo) = fr(wo, wi)`. 빛의 경로를 거꾸로 돌려도 반사율이 같다. 이 대칭성 덕분에 카메라에서 광원으로 역방향 추적(패스 트레이싱)이 정당화되고, BRDF 모델이 물리적으로 말이 되는지 검사하는 리트머스지가 된다.
2. **에너지 보존**: 모든 wo에 대해 `∫_Ω fr(wi, wo) · (n·wi) dwi ≤ 1`. 들어온 것보다 많이 반사할 수 없다.

### Cook-Torrance — 마이크로패싯 스페큘러의 해부

실시간 PBR의 표준 BRDF는 diffuse + specular 두 로브(lobe)다:

```
f = kd · albedo/pi  +  D(h) · F(v,h) · G(l,v,h) / ( 4 · (n·v) · (n·l) )
```

스페큘러 항의 출발점은 **마이크로패싯 이론(microfacet theory)**: 거시적으로 한 픽셀에 들어오는 표면은 사실 수많은 미세한 완전 거울 조각들의 통계적 집합이다. 거울 반사의 법칙상, 빛 `l`을 시선 `v`로 튕겨줄 수 있는 조각은 **법선이 정확히 반각 벡터(half vector) `h = normalize(l + v)`인 조각뿐**이다.

```
        l         h = normalize(l+v)         v
         ↘        │        ↗
           ↘      │      ↗
             ↘    │    ↗
   ─╲/╲__/╲─╲──[m=h]──/╲_/─╲/─   ← 미세면들 (roughness가 법선의 산포를 결정)
   ──────────────────────────    ← 거시 표면 (법선 n)

   기여하는 조각 = 미세 법선 m이 h와 일치하는 조각뿐
   D: 그런 조각의 밀도 / F: 그 조각의 반사율 / G: 이웃에 안 가려진 비율
```

그래서 스페큘러의 세 항은 각각 이렇게 읽힌다:

- `D`: 미세 법선이 `h`를 향하는 조각이 **얼마나 많은가** (분포)
- `F`: 그 조각들이 빛을 **얼마나 반사하는가** (프레넬)
- `G`: 그 조각들 중 몇 %가 **가려지지 않고 실제로 기여하는가** (기하 감쇠)

분모의 `4·(n·v)·(n·l)`는 미세면 공간과 거시면 공간 사이의 정규화 상수다 — 반각 벡터로 변수를 바꿀 때의 야코비안(Jacobian) `dωh/dωo = 1/(4·(v·h))`과 거시면 투영 항이 결합해 나온다. "왜 4인가?"는 유도가 길지만, "반각 파라미터화의 변수 치환 비용"이라고 답하면 정확하다.

#### D — GGX / Trowbridge-Reitz NDF

```
a = roughness^2                                    (Disney 재매핑 관행)
D_GGX(h) = a^2 / ( pi * ( (n.h)^2 * (a^2 - 1) + 1 )^2 )
```

NDF(normal distribution function)는 미세 법선의 확률 밀도다. 정규화 조건은 `∫ D(h) · (n·h) dωh = 1` — 미세면들의 투영 면적 합이 거시면 면적과 같아야 한다는 물리 제약이다.

**왜 하필 GGX가 이겼나 (Beckmann, Blinn-Phong을 제치고):**

- **긴 꼬리(long tail).** Beckmann은 가우시안 계열이라 하이라이트가 중심에서 급격히 꺼진다. 실측 재질 데이터(MERL 데이터베이스)를 보면 실제 하이라이트는 "밝은 코어 + 넓게 은은한 헤일로" 구조인데, GGX의 유리 분수 형태가 이 꼬리를 자연스럽게 재현한다. 눈으로 봐도 GGX 하이라이트가 "사진 같다".
- **싸다.** `exp()` 없이 곱셈·나눗셈만으로 계산된다.
- **궁합 좋은 Smith G가 존재**해서 세트로 쓰기 좋다.

`a = roughness^2` 재매핑은 Disney(Burley 2012)가 제안한 관행이다. 이유는 **지각 선형성**: roughness를 원시 `a`로 그대로 쓰면 시각적 변화가 슬라이더의 아래쪽 구간에 몰려서, 0.0~0.3 사이에서 거의 다 변하고 나머지는 밋밋하다. 제곱 매핑을 거치면 아티스트의 슬라이더가 체감상 고르게 움직인다. 수학이 아니라 **인간공학적 재매핑**이다.

#### F — Fresnel-Schlick

```
F(v,h) = F0 + (1 - F0) * (1 - (v.h))^5
```

프레넬 효과: **모든 표면은 스치는 각도에서 거울이 된다.** 호수를 위에서 내려다보면 바닥이 보이지만 멀리 수평으로 보면 하늘이 비친다. 정면 입사 시의 반사율이 `F0`이고, 입사각이 90°에 가까워지면 재질과 무관하게 반사율이 1로 수렴한다. Schlick(1994)의 5제곱 근사는 정확한 프레넬 방정식(복소 굴절률 포함)을 두 끝점 보간으로 뭉갠 것인데, 오차가 시각적으로 무시 가능해 실시간 표준이 됐다. `(n·v)`가 아니라 `(v·h)`를 쓰는 이유: 프레넬은 **빛이 닿는 그 미세면**에서 일어나는데, 반사에 기여하는 미세면의 법선이 `h`이기 때문이다.

`F0`의 유도 — 정면 입사 프레넬은 굴절률(IOR)로 결정된다:

```
F0 = ( (n1 - n2) / (n1 + n2) )^2
유리(n=1.5) 기준: F0 = (0.5 / 2.5)^2 = 0.04
```

비금속(dielectric)의 IOR은 대부분 1.3~1.8에 몰려 있어 F0가 0.02~0.08 범위다. 그래서 실시간 PBR은 **비금속 F0 ≈ 0.04를 상수로 박는다** — 텍스처 채널 하나를 아끼는 실용적 결단이다.

**금속(conductor)은 근본적으로 다르다.** 자유 전자 때문에 (1) 굴절되어 표면 안으로 들어간 빛이 즉시 흡수된다 → **diffuse가 없다** (diffuse는 표면 아래로 들어갔다 산란되어 나오는 빛인데, 그 경로 자체가 사라짐), (2) F0가 0.5~1.0으로 높고 **파장별로 다르다** → 금이 노랗고 구리가 붉은 이유는 F0가 색을 갖기 때문이다. 그래서 metallic 워크플로에서 **금속의 albedo 텍스처는 diffuse 색이 아니라 F0**로 재해석된다. 이것이 금속/비금속의 본질적 차이이고, `metallic` 파라미터 하나로 두 세계를 lerp하는 근거다:

```hlsl
float3 F0 = lerp(float3(0.04, 0.04, 0.04), albedo, metallic);
```

#### G — Smith 기하 감쇠

미세면들은 서로를 가린다. 빛이 미세면에 닿기 전에 이웃 조각에 막히는 **자기 그림자(shadowing)**, 반사된 빛이 나가다 막히는 **마스킹(masking)**. 이 손실이 없으면 스치는 각도에서 `D / (4·(n·v)·(n·l))`의 분모가 0으로 가며 스페큘러가 발산한다 — G가 그 발산을 물리적으로 눌러주는 항이다.

Smith 모델은 "빛 방향의 가림과 시선 방향의 가림은 통계적으로 독립"이라 가정하고 곱으로 분리한다:

```
k = (roughness + 1)^2 / 8              (UE4/Karis의 직접광용 재매핑)
G1(x) = (n.x) / ( (n.x) * (1 - k) + k )
G_Smith(l, v) = G1(l) * G1(v)
```

Schlick-GGX의 `k`는 GGX의 정확한 Smith 항을 싸게 근사하기 위한 매핑 상수다. Karis(UE4, 2013)는 직접광에서 `k = (roughness+1)^2 / 8`, IBL에서 `k = a^2 / 2`로 다르게 재매핑했다 — 정확한 해가 아니라 "레퍼런스와 눈으로 맞춘" 실용 근사라는 걸 알고 쓰는 게 중요하다.

#### diffuse = albedo / pi — 왜 pi로 나누나

Lambertian diffuse는 "모든 방향으로 균일하게 반사"이므로 BRDF가 상수 `c`다. 에너지 보존을 적용하면:

```
반사된 총 에너지 비율 = ∫_Ω c · (n·wi) dwi = c · ∫_Ω cosθ dwi

반구 코사인 적분:
∫_Ω cosθ dω = ∫[0,2pi] dφ ∫[0,pi/2] cosθ · sinθ dθ = 2pi · (1/2) = pi

∴ c · pi = albedo  →  c = albedo / pi
```

즉 `pi`는 마법의 수가 아니라 **"반구 전체에 균일하게 뿌렸을 때 코사인 가중 총합이 pi"라는 기하학적 사실에서 온 정규화 상수**다. 이걸 빼먹으면 diffuse가 pi배(≈3.14배) 밝아지고, 그 위에 광원 강도를 어둡게 튜닝해서 상쇄하는 "이중 오류" 코드가 태어난다 — 레거시 엔진에서 흔한 화석이다.

### metallic/roughness 텍스처 워크플로

| 텍스처 | 내용 | 색 공간 | 이유 |
|---|---|---|---|
| albedo (baseColor) | 비금속=diffuse 반사율, 금속=F0 | **sRGB** (디코드 필요) | 아티스트가 "색"으로 그린 데이터 |
| normal | 접선공간 법선 벡터 | **linear** | 색이 아니라 (x,y,z) 좌표 데이터 |
| roughness | 미세 거칠기 0~1 | **linear** | 스칼라 물리 파라미터 |
| metallic | 금속 여부 0/1 | **linear** | 스칼라 (중간값은 먼지 낀 금속 등 혼합 상태) |
| AO | 사전 계산 차폐율 | **linear** | 스칼라 가중치 |

albedo의 **이중 역할**이 이 워크플로의 압축 트릭이다: 텍스처 한 장이 metallic에 따라 diffuse 색으로도, F0로도 읽힌다. 그리고 에너지 보존의 마지막 조각:

```hlsl
float3 kD = (1.0 - F) * (1.0 - metallic);
float3 diffuse = kD * albedo / PI;
```

`(1-F)`: 프레넬로 반사된 비율을 뺀 나머지(굴절해 들어간 빛)만 diffuse가 될 수 있다. `(1-metallic)`: 금속은 굴절광을 즉시 흡수하므로 diffuse가 0이 된다. 두 곱이 "스페큘러+디퓨즈 ≤ 입사광"을 구조적으로 보장한다.

### IBL 개요 — 적분을 미리 굽는 두 갈래

환경 전체(스카이박스 큐브맵)가 광원일 때, 렌더링 방정식의 반구 적분을 픽셀마다 실시간으로 풀 수는 없다. IBL(image-based lighting)은 이 적분을 **오프라인 사전 계산**으로 쪼갠다. 왜 두 갈래로 쪼개냐 — diffuse와 specular의 수학적 성질이 달라서다.

**Diffuse — irradiance map.** Lambert BRDF는 상수라 적분 밖으로 나온다:

```
Lo_diffuse = (albedo/pi) · ∫_Ω Li(wi) · (n·wi) dwi
                            └────────── n에만 의존 ──────────┘
```

적분이 **법선 방향 n만의 함수**가 됐다. 그러면 "모든 n에 대한 답"을 큐브맵 한 장(irradiance map)으로 미리 구워둘 수 있다 — 환경맵을 코사인 가중으로 컨볼루션(blur)한 결과다. 런타임은 `albedo * irradianceMap.Sample(n)` 한 줄.

**Specular — split-sum 근사 (Karis, UE4).** 스페큘러 BRDF는 v, roughness, F0에 다 의존해서 차원이 너무 많다. 그래서 적분을 둘로 쪼갠다:

```
∫ Li · fr · cosθ dwi  ≈  ( ∫ Li · D가중 dwi ) × ( ∫ fr · cosθ dwi )
                          └ 프리필터 환경맵 ┘   └ BRDF LUT ┘
```

- **프리필터 환경맵**: roughness 단계별로 환경맵을 GGX 로브 모양으로 블러해서 **밉 체인**에 저장. roughness 0 = 원본(거울), roughness 1 = 거의 균일 블러. 런타임은 반사 벡터로 roughness에 맞는 밉을 샘플.
- **BRDF LUT**: 남은 적분은 F0를 Schlick 형태로 인수분해하면 `(n·v), roughness` 두 변수만 남는다 → 2D 텍스처에 (scale, bias) 두 값으로 구움. 런타임은 `F0 * LUT.x + LUT.y`.

왜 이렇게까지 쪼개나: **차원 축소**가 목적이다. 원래 적분은 (n, v, roughness, F0)의 고차원 함수라 통째로 구울 수 없다. 조명과 BRDF의 상관을 무시하는 근사(두 적분의 곱 ≠ 곱의 적분)와 등방성 가정을 대가로, "큐브맵 밉 체인 + 2D LUT 한 장"이라는 저장 가능한 형태로 내려온다. 대가는 스치는 각도의 길게 늘어지는(anisotropic streak) 하이라이트가 사라지는 것 — 알고 쓰는 근사다.

### 감마 보정 — sRGB가 존재하는 두 가지 이유

sRGB는 취향이 아니라 두 가지 제약의 교집합이다.

1. **인간의 밝기 지각은 비선형이다.** 우리는 어두운 영역의 차이에 민감하고 밝은 영역의 차이에 둔감하다(물리 휘도 2배 ≠ 체감 2배 밝음). 지각적으로 균등한 밝기 스케일은 물리 휘도의 거듭제곱(power law)에 가깝다.
2. **8bit는 256단계뿐이다.** 물리 휘도를 선형으로 256등분하면, 인간이 민감한 어두운 구간에 코드가 부족해 그라데이션에 밴딩(banding)이 생기고, 둔감한 밝은 구간에 코드가 낭비된다. sRGB 인코딩(≈1/2.2 제곱)은 **비트를 어두운 쪽에 더 배분하는 지각적 압축 코덱**이다.

역사적 우연도 한몫했다: CRT 모니터의 전압-휘도 응답이 우연히 감마 ~2.2 곡선이어서, "지각 압축"과 "디스플레이 특성"이 맞아떨어진 채 표준이 굳었다.

정확한 sRGB 곡선과 2.2 근사는 다르다:

```
정확한 sRGB 인코딩 (linear → sRGB):
  C_srgb = 12.92 * C_lin                      , C_lin ≤ 0.0031308
  C_srgb = 1.055 * C_lin^(1/2.4) - 0.055      , C_lin > 0.0031308
```

지수는 2.4지만 0 근처의 **선형 토(linear toe)** 구간(순수 거듭제곱은 0에서 미분이 무한대라 저휘도가 불안정해지는 걸 방지) 때문에 곡선 전체의 평균 기울기가 감마 ~2.2에 근접한다. `pow(x, 1/2.2)`는 이 곡선의 근사이고, 어두운 구간에서 정확 곡선과 오차가 가장 크다.

**감마 공간에서 라이팅하면 무엇이 깨지나 — 숫자로.** 빛은 물리 세계에서 **선형으로 더해진다**. 라이트 A와 B가 각각 linear 0.2의 기여를 하면 합은 0.4다:

```
올바른 순서 (linear 계산):   0.2 + 0.2 = 0.4  → 인코드 → 0.4^(1/2.2) ≈ 0.66
잘못된 순서 (감마 공간 덧셈): enc(0.2) + enc(0.2) ≈ 0.48 + 0.48 = 0.96  (거의 흰색!)
```

인코딩이 비선형이라 `enc(a+b) ≠ enc(a)+enc(b)` — 덧셈이 왜곡된다. 증상: 중간톤이 과하게 밝아지고, 라이트가 겹치는 곳이 금방 하얗게 타고, 색이 과포화되고, 거리 감쇠가 너무 급하게 꺼진다. 아티스트는 이걸 광원을 어둡게 튜닝해 가리는데, 그 순간 물리 파라미터의 의미가 전부 무너진다. **PBR을 하면서 linear workflow를 안 하면 BRDF 수식의 정확도가 통째로 무의미해지는 이유다.**

### Linear workflow — 규칙 세 줄

```
[입력]  albedo 텍스처(sRGB로 저장됨) ──sRGB 디코드──┐
        normal/rough/metal/AO(linear 데이터) ──그대로──┼→ [계산: 전부 linear 공간] → [출력: sRGB 인코드] → 8bit 백버퍼
        라이트 색/강도(linear로 정의) ──────그대로──┘
```

1. **입력**: 색인 것(albedo, 이미시브 텍스처)만 sRGB 디코드. 데이터인 것(normal/roughness/metallic/AO)은 애초에 linear로 저장돼 있으므로 손대지 않는다 — 노말맵을 sRGB로 디코드하면 벡터가 비균일하게 휘어 라이팅이 미묘하게 틀어진다.
2. **계산**: 라이팅, 블렌딩, 밉 필터링, 포스트 프로세싱 전부 linear.
3. **출력**: 디스플레이 직전 한 번만 sRGB 인코드.

하드웨어가 이 규칙을 공짜로 해주는 장치가 `_SRGB` 텍스처 포맷이다 (`DXGI_FORMAT_R8G8B8A8_UNORM_SRGB`):

- **샘플링 시**: 텍셀을 필터링(bilinear 보간) **하기 전에** 텍셀 단위로 linear로 디코드한다. 셰이더는 항상 linear 값을 받는다. 순서가 중요하다 — 감마 값끼리 보간하면 보간 자체가 왜곡되는데, 하드웨어는 이를 텍셀별 디코드로 회피한다.
- **RT 쓰기 시**: 셰이더가 내보낸 linear 값을 인코드해서 저장한다. 알파 블렌딩이 필요하면 **대상 픽셀을 디코드해 linear에서 블렌딩한 뒤 결과를 인코드**한다 — 블렌딩까지 물리적으로 옳게 처리된다.

셰이더에서 `pow`로 직접 하는 수동 변환(shader-local) 대비 하드웨어 방식의 우위가 바로 이 두 지점 — **필터링과 블렌딩** — 이다. 수동 변환은 "샘플된 (감마 값들이 이미 보간된) 결과"를 디코드하고, 블렌딩은 PS 밖에서 일어나므로 손댈 수 없다.

### HDR 파이프라인 — 씬은 넓고 디스플레이는 좁다

현실의 휘도 범위: 태양 원반 ~10^9 cd/m², 맑은 낮 풍경 ~10^4, 실내 ~10^2, 달빛 아래 ~10^-1 이하. 십수 자릿수의 동적 범위(dynamic range)다. 8bit UNORM은 [0,1]을 256단계로 담을 뿐이다 — 태양이 보이는 창가와 실내 그늘을 **한 프레임에** 담으려면, 창밖을 1.0으로 클램프하는 순간 태양과 흰 벽이 같은 색이 되고, 실내에 노출을 맞추면 창밖이 전부 탄다. LDR로는 물리 단위의 라이팅이 원천적으로 불가능하다.

해법은 공간 분리다:

```
[씬 렌더: R16G16B16A16_FLOAT RT]     ← radiance를 물리 스케일로, 1.0 넘게 자유롭게 누적
        │  (여기서 bloom, 이미시브, 다중 광원 누적이 전부 자연스럽게 성립)
        ▼
[노출(exposure) 적용]                 ← color *= exposure. 카메라 조리개 역할. 자동 노출 = 평균 로그 휘도 기반
        ▼
[톤 매핑(tone mapping)]              ← HDR → [0,1] 압축. "필름/눈의 응답 곡선" 모사
        ▼
[sRGB 인코드]                        ← linear → 감마. 마지막 한 번
        ▼
[8bit 백버퍼 → 디스플레이]
```

16F(half float)를 쓰는 이유: 최대 ~65504까지 표현하면서 부동소수라 어두운 영역의 상대 정밀도가 유지된다. 32F는 대역폭 2배를 낼 가치가 (씬 컬러에는) 없다.

**톤 매핑 3종의 트레이드오프:**

| 연산자 | 수식 | 특성 | 문제 |
|---|---|---|---|
| Reinhard | `x / (1 + x)` | 가장 단순, 절대 안 탐 | 채널별 적용 시 밝은 부분이 회색으로 **탈색**(washed out) — 어떤 채널도 끝까지 못 가서 대비·채도가 죽음 |
| Filmic (Hable) | 토+숄더 유리함수 | 필름 응답 모사: 어두운 토(toe)가 블랙을 눌러 대비를 살리고, 숄더가 하이라이트를 부드럽게 말아 올림 | 파라미터 튜닝 필요 |
| ACES 근사 (Narkowicz fit) | `(x(2.51x+0.03)) / (x(2.43x+0.59)+0.14)` | 영화 산업 표준 ACES RRT+ODT의 저비용 곡선 근사, 채도 유지가 좋고 사실상 업계 기본값 | 살짝 강한 콘트라스트/채도 시프트 |

Reinhard의 탈색이 왜 생기나: `x/(1+x)`는 x가 커질수록 1에 점근하지만 **절대 도달하지 않는다**. (10, 2, 2) 같은 강한 주황 빛도 (0.91, 0.67, 0.67)로 압축되며 채널 간 비율이 뭉개져 회색빛이 된다. 필믹 계열의 숄더는 밝은 영역에서 세 채널을 함께 흰색으로 수렴시키면서도 중간톤의 채도를 지킨다 — "밝은 것은 하얗게 탄다"는 필름 사진의 직관을 코드로 옮긴 것이다.

## ③ 심화 — 꼬리질문 대비

**"BRDF 값이 1을 넘으면 에너지 보존 위반 아닌가?"**
아니다. BRDF는 에너지 비율이 아니라 radiance/irradiance라는 **밀도의 비**(단위 1/sr)다. 매끈한 표면은 반사 에너지를 아주 좁은 입체각에 몰아넣으므로 그 방향의 밀도는 수백을 넘을 수 있다. 에너지 보존은 `∫ fr·cosθ dω ≤ 1`이라는 적분 조건이다. 이 구분을 못 하면 "스페큘러가 1 넘으면 clamp해야 하나요" 같은 잘못된 코드가 나온다.

**"왜 반각 벡터 h를 쓰나? (n·l)이나 (n·v)로는 안 되나?"**
마이크로패싯 이론에서 각 미세면은 완전 거울이다. 거울이 빛 l을 시선 v로 반사하려면 그 거울의 법선이 정확히 l과 v의 이등분 방향, 즉 `h = normalize(l+v)`여야 한다. 그래서 "반사에 기여하는 미세면의 양"을 세는 D와, 그 면에서의 반사율 F는 h 기준으로 평가된다. h는 편의가 아니라 반사 기하의 필연이다.

**"Cook-Torrance 분모의 4는 어디서 오나?"**
반각 벡터 파라미터화의 변수 치환에서 온다. 적분 변수를 wo에서 h로 바꿀 때 야코비안이 `dωh/dωo = 1/(4·(v·h))`이고, 미세면-거시면 투영 항들과 결합하면 분모 `4·(n·v)·(n·l)`로 정리된다. "정규화 상수이며, 유도는 변수 치환의 야코비안"이라고 답하면 충분하다.

**"a = roughness^2 재매핑은 왜 하나?"**
지각 선형성. 원시 alpha를 그대로 노출하면 시각 변화가 슬라이더 저구간에 몰려 아티스트가 0.0~0.2 사이에서만 씨름하게 된다. Disney가 제곱 매핑으로 슬라이더 체감을 균등화했고 업계 표준이 됐다. 물리가 아니라 UX 결정이라는 점을 짚으면 이해도가 드러난다.

**"금속의 albedo 텍스처는 무슨 의미인가?"**
F0다. 금속은 자유 전자가 굴절광을 즉시 흡수해 diffuse가 없고, 정면 반사율 F0가 높으며 파장 의존적(금=노랑, 구리=빨강)이다. 그래서 같은 텍스처가 metallic=0이면 diffuse 반사율, metallic=1이면 F0로 읽힌다. `F0 = lerp(0.04, albedo, metallic)` 한 줄이 이 이중 의미의 구현이다.

**"비금속 F0 = 0.04는 어디서 온 수인가?"**
정면 입사 프레넬 `F0 = ((n1-n2)/(n1+n2))^2`에 유리의 굴절률 1.5를 넣으면 `(0.5/2.5)^2 = 0.04`. 비금속 대부분이 IOR 1.3~1.8에 몰려 F0 0.02~0.08이라, 텍스처 채널을 아끼려 0.04 상수로 통일한 실용적 관행이다. 보석류(다이아몬드 n≈2.4)는 이 관행을 벗어나므로 specular 채널을 따로 두는 확장(글로시 워크플로/specular 오버라이드)이 존재한다.

**"diffuse에 왜 (1-F)를 곱하나?"**
에너지 분기 때문이다. 표면에 닿은 빛은 프레넬 비율 F만큼 즉시 반사(스페큘러)되고, 나머지 (1-F)만 표면 안으로 굴절해 들어간다. diffuse는 그 굴절광이 내부 산란 후 재방출된 것이므로 (1-F)가 상한이다. 이걸 빼먹으면 스치는 각도에서 스페큘러+디퓨즈가 입사광을 초과한다.

**"roughness를 0까지 허용하면 무슨 일이 나나?"**
`a=0`이면 D_GGX가 델타 함수로 퇴화한다 — 분모가 `(n·h)^2·(-1)+1` 형태가 되어 n=h에서 0, 수치적으로 발산한다. 화면상으로는 픽셀 단위로 반짝이는 파이어플라이(firefly)와 스페큘러 앨리어싱이 생긴다. 그래서 실전 코드는 roughness에 하한(0.03~0.05)을 clamp한다. Winters도 0.04로 clamp한다(아래 ④).

**"노말맵을 sRGB 포맷으로 잘못 읽으면 어떤 버그가 보이나?"**
텍셀 값이 감마 곡선으로 휘어 디코드되므로 (0.5,0.5,1)로 인코딩된 '평평함'이 평평하지 않게 읽힌다. 라이팅이 전체적으로 뒤틀리고 표면이 부풀거나 파인 듯 보이는데, normalize를 해도 **방향 자체가 비균일하게 왜곡**된 상태라 복구되지 않는다. "albedo만 sRGB, 데이터 텍스처는 linear"라는 규칙의 존재 이유다.

**"_SRGB 하드웨어 변환은 정확히 언제 일어나나?"**
샘플링에서는 필터링(보간) **전에** 텍셀 단위로 디코드된다 — 감마 값끼리 보간하는 왜곡을 피하기 위해서다. RT 쓰기에서는 블렌딩이 있으면 대상 픽셀을 디코드해 **linear에서 블렌딩**한 뒤 결과를 인코드해 저장한다(D3D 스펙 요구사항). 즉 하드웨어 경로는 필터링·블렌딩까지 옳게 처리하고, 셰이더 수동 pow 방식은 이 두 지점을 놓친다.

**"정확한 sRGB 곡선과 pow 2.2는 뭐가 다른가?"**
정확 곡선은 0 근처의 선형 구간(기울기 12.92, 경계 0.0031308)과 지수 2.4의 거듭제곱 구간으로 이뤄진 조각 함수다. 선형 토는 순수 거듭제곱의 "0에서 미분 무한대" 문제를 피해 저휘도를 안정화한다. `pow(1/2.2)`는 전체 곡선의 근사이며 어두운 구간에서 오차가 가장 크다. 8bit 왕복 변환을 반복하면 이 오차가 누적되므로, 왕복이 잦은 파이프라인일수록 하드웨어 정확 변환이 유리하다.

**"Reinhard의 탈색 문제를 구체적으로 설명하라."**
`x/(1+x)`를 채널별로 적용하면 큰 값일수록 압축률이 높아 채널 간 비율이 뭉개진다. (10,2,2)의 강한 주황이 (0.91,0.67,0.67)의 물빠진 살구색이 된다 — 밝은 영역이 전부 회색빛으로 수렴하고 어떤 채널도 1.0에 도달하지 못해 화면 전체의 대비가 죽는다. 휘도 기반 Reinhard는 색은 지키지만 채널 클리핑이 되살아난다. 필믹/ACES는 토와 숄더로 "밝은 건 희게 태우되 중간톤 채도는 유지"를 곡선 설계로 해결한다.

**"split-sum 근사는 어디서 부정확해지나?"**
두 군데다. (1) `∫(A·B) ≈ ∫A · ∫B`는 조명과 BRDF 가중의 상관을 무시한다 — 밝은 광원이 로브의 가장자리에 있을 때 오차가 커진다. (2) 프리필터 시 시선=법선=반사 방향이라는 등방 가정을 깔아서, 실제로는 스치는 각도에서 길게 늘어져야 할 하이라이트(길바닥 젖은 반사의 세로 줄무늬 같은)가 둥글게 뭉개진다. 그래도 "큐브맵 밉 + 2D LUT"라는 저장 가능한 형태의 대가로는 싸다는 게 업계의 합의다.

**"에너지 보존이 안 되는 재질은 실제로 뭐가 보이나?"**
한 번의 셰이딩에선 "약간 과하게 밝은" 정도지만, GI처럼 빛이 여러 번 튕기는 시스템에서 반사마다 에너지가 증폭되어 씬이 하얗게 발산한다. 정적 라이트맵 베이크가 수렴하지 않거나, bloom이 임계를 넘어 화면이 타는 형태로 나타난다. 아티스트가 광원을 어둡게 낮춰 가리기 시작하면 물리 파라미터 전체의 신뢰가 무너진다 — PBR 도입의 근거를 되짚는 좋은 답변 소재다.

**"HDR 씬 RT 없이 톤 매핑을 포워드 PS 안에서 하면 뭘 잃나?"**
톤 매핑은 비선형 압축이므로 **그 이후의 어떤 덧셈도 물리적으로 틀린다**. (1) 알파 블렌딩되는 FX/파티클이 압축된 디스플레이 공간에서 섞인다. (2) bloom을 뽑을 "1.0 초과 값"이 이미 소멸해 있다. (3) 노출을 화면 전체에 일관 적용할 지점이 없다. 그래서 톤 매핑은 모든 radiance 누적이 끝난 뒤 풀스크린 포스트 패스 한 곳에서 해야 한다 — Winters의 현재 구조가 정확히 이 개선 지점에 있다(④에서).

## ④ Winters에서의 적용

Winters는 스타일라이즈드 트랙(Mesh3D/Skinned3D)과 PBR 트랙(Mesh3D_PBR/Skinned3D_PBR)을 병행한다. 이 챕터의 수식이 코드로 어디에 실존하고, 무엇이 비어 있는지 — "비어 있음을 진단하고 개선 계획을 말할 수 있는 것"까지가 면접 무기다.

### (1) Cook-Torrance 전체 구현 실물 — BRDF_GGX.hlsli

`Shaders/BRDF/BRDF_GGX.hlsli:7`에 이 챕터의 스페큘러 수식이 그대로 있다.

- **D**: `D_GGX`(7~14행)는 `a = roughness * roughness`(9행)로 Disney 재매핑을 적용한 뒤 `a2 / (PI * denom * denom)` 형태의 Trowbridge-Reitz를 계산한다. 주석에도 "Disney roughness convention: a = roughness^2"라고 관행의 출처를 박아뒀다.
- **G**: `G_Smith`(22~27행)는 `k = (roughness+1)^2 / 8`(24~25행) — Karis의 직접광용 재매핑 — 을 쓰는 Schlick-GGX 두 개의 곱이다. 주석이 "Karis approximation for direct lighting"이라고 명시해서, IBL용 `k = a^2/2`와 다르다는 걸 아는 상태의 구현임을 보여줄 수 있다.
- **F**: `F_Schlick`(30~34행)은 `F0 + (1-F0)·(1-(v·h))^5` 그대로다.
- **조립**: `BRDF_CookTorrance`(37~63행)에서 `F0 = lerp(0.04, albedo, metallic)`(51행 — 금속/비금속 이중 의미의 한 줄), `specular = D*F*G / (4·NoV·NoL + 1e-5)`(57행), `kD = (1-F)·(1-metallic)`(59행), `diffuse = kD·albedo/PI`(60행 — pi 정규화), 마지막에 `* NoL`(62행 — 렌더링 방정식의 코사인 항).

수치 방어도 읽을 거리다: `NoV`에 `+1e-5`(46행), 스페큘러 분모에 `+1e-5`(57행) — 스치는 각도에서 분모가 0으로 가는 발산(G 항 설명 참조)의 0-나눗셈 가드다. 그리고 `Shaders/Mesh3D_PBR.hlsl:106`은 `roughness = clamp(g_fRoughness, 0.04f, 1.0f)`로 하한을 강제한다 — a→0에서 D가 델타 함수로 퇴화하는 파이어플라이 문제(③ 참조)의 예방선이다.

재질 파라미터는 `Shaders/Mesh3D_PBR.hlsl:33`의 `PerMaterial` cbuffer(b3)에 metallic/roughness/AO/emissive로 정의되고, 이 셰이더 쌍은 `Engine/Private/Framework/CEngineApp.cpp:603`(Mesh3D_PBR)과 615행(Skinned3D_PBR)에서 엔진에 와이어링된다.

### (2) 감마 전략 — "shader-local sRGB"라는 명시적 계약

Winters의 감마 처리는 하드웨어 `_SRGB` 포맷이 아니라 **셰이더 수동 변환**이고, 그 계약이 타입으로 문서화돼 있다. `Engine/Public/Resource/Texture.h:10`의 `eTexColorSpace { Auto, ShaderLocalSRGB, IgnoreSRGB }` — `ShaderLocalSRGB`라는 이름 자체가 "로더는 변환하지 말 것, 셰이더가 직접 한다"는 약속이다. 로더 쪽 구현은 `Engine/Private/Resource/Texture.cpp:177`에서 이 enum을 보고 `DDS_LOADER_IGNORE_SRGB`(193행) / `WIC_LOADER_IGNORE_SRGB`(208행) 플래그로 UNORM 뷰를 강제하고, 소비자 쪽은 `Engine/Private/Renderer/ModelRenderer.cpp:710`의 `CTexture::Create(..., eTexColorSpace::ShaderLocalSRGB)`(714행)로 계약에 서명한다.

셰이더 쪽 상대 코드는 `Shaders/Mesh3D.hlsl:51`의 `SrgbToLinearApprox`(pow 2.2)와 56행의 `LinearToSrgbApprox`(pow 1/2.2)이고, 스타일라이즈드 PS는 **디코드 → linear 라이팅 → 인코드** 샌드위치를 정확히 지킨다: 163행에서 `baseLinear = SrgbToLinearApprox(texColor.rgb)`로 들어가 168행에서 `LinearToSrgbApprox(outlinedLinear)`로 나온다 (`Shaders/Skinned3D.hlsl:183~188`도 동일 구조). "linear workflow의 3규칙"이 이 두 줄 사이에 구현돼 있다.

왜 셰이더가 인코드까지 떠맡았나 — 백버퍼가 `DXGI_FORMAT_R8G8B8A8_UNORM`(`Engine/Private/RHI/DX11/CDX11Device.cpp:793`)이라 하드웨어가 쓰기 시 인코드해주지 않기 때문이다. 이 선택의 트레이드오프를 면접 답변으로 정리하면:

- **얻은 것**: 파이프라인 어디서도 암묵 변환이 없다. 모든 색 공간 전이가 셰이더 코드에 보이고, enum이 텍스처별 정책을 강제한다.
- **잃은 것 1 — 필터링**: 샘플러가 감마 값들을 보간한 **후에** 디코드하므로, 텍셀 단위 디코드 후 보간하는 하드웨어 경로와 결과가 미세하게 다르다(고대비 경계에서 어두워지는 쪽으로 치우침).
- **잃은 것 2 — 블렌딩**: PS가 감마 인코드된 값을 내보내므로 알파 블렌딩이 감마 공간에서 일어난다. `FxSprite.hlsl`/`FxMesh.hlsl` 계열 FX는 색 공간 변환 없이 그대로 블렌드되는데, 스타일라이즈드 이펙트라 아트 판단으로 흡수 가능하지만 "물리적으로는 덧셈이 왜곡된 공간"임을 알고 있어야 한다.
- **잃은 것 3 — 정확도**: pow 2.2는 근사다. 정확 sRGB 곡선(선형 토 포함)과 어두운 구간에서 어긋난다.

### (3) 진단 — RHI에 _SRGB 포맷이 없다

`Engine/Public/RHI/RHITypes.h:39`의 `eRHIFormat` 열거를 보면 `R8G8B8A8_UNorm`, `R16G16B16A16_Float` 등은 있지만 **`_SRGB` 변형이 하나도 없다**. 즉 하드웨어 감마 경로는 취향으로 안 쓴 게 아니라 **RHI 추상화 차원에서 아직 뚫려 있지 않다**. 이걸 "우리 엔진의 색 공간 처리는 RHI 포맷 열거 수준에서 결정돼 있고, 개선 1순위는 포맷 추가"라고 말할 수 있으면, 감마를 셰이더 트릭이 아니라 파이프라인 설계 문제로 이해하고 있다는 신호가 된다.

또 하나의 확인 포인트: PBR 경로의 albedo 입력이다. `Shaders/Mesh3D_PBR.hlsl:101`은 albedo를 샘플해 104행에서 바로 라이팅에 쓰는데, 스타일라이즈드 경로와 달리 **셰이더 내 sRGB 디코드가 없다**. 이 셰이더가 옳으려면 입력 SRV가 이미 linear여야 하고, 그건 로더의 `Auto` 정책(DirectXTK가 이미지의 sRGB 메타데이터에 따라 `_SRGB` 포맷을 만들 수도, UNORM으로 만들 수도 있다)에 달려 있다. 소스 텍스처가 UNORM으로 로드되면 PBR 라이팅이 감마 공간 albedo 위에서 돌게 된다 — ②의 "0.48+0.48=0.96" 왜곡이 정확히 여기서 재현된다. "PBR 경로의 albedo 색 공간을 로더-셰이더 양단에서 계약으로 못 박는 것"이 진단 결과이자 수정 계획이다. RenderDoc으로 SRV 포맷을 열어보면 한 번에 판정된다.

### (4) 진단 — 톤 매핑이 포워드 PS 인라인이고, HDR 씬 RT가 없다

`Shaders/Mesh3D_PBR.hlsl:129~130`:

```hlsl
color = color / (color + float3(1.0f, 1.0f, 1.0f));   // Reinhard 톤 매핑
color = pow(saturate(color), 1.0f / 2.2f);             // 감마 인코드
```

(`Shaders/Skinned3D_PBR.hlsl:146~148`도 동일.) 라이팅 누적은 물리 스케일로 옳게 하고 있지만, Reinhard 압축과 감마 인코드가 **오브젝트별 포워드 PS 안에서** 일어난다. ③의 마지막 질문 그대로의 상태다: 이 지점 이후에 블렌딩되는 FX는 압축된 공간에서 섞이고, bloom을 뽑을 1.0 초과 radiance는 PS를 나서는 순간 소멸하며, 화면 전체에 일관된 노출을 걸 자리가 없다. 엔진 코드 전체에 톤 매핑 포스트 패스와 노출 시스템은 아직 존재하지 않는다(서드파티 헤더 제외).

**개선 계획 — 면접 답변 형태로:**

> "현재 제 엔진은 Cook-Torrance BRDF까지는 표준 구현이 돼 있지만, 톤 매핑이 포워드 PS 인라인이라 HDR 파이프라인이 완성되지 않았습니다. 개선은 4단계로 봅니다. (1) `eRHIFormat`에 `R8G8B8A8_UNorm_SRGB`를 추가하고 DX11 포맷 매핑에 연결합니다 — `R16G16B16A16_Float`는 이미 열거와 매핑(`CDX11Device.cpp:133`)에 있어서 HDR RT 쪽 포맷은 뚫려 있습니다. (2) 씬을 R16G16B16A16_FLOAT 오프스크린 RT에 그리도록 하고, PS의 Reinhard+pow 두 줄을 제거해 셰이더는 linear radiance만 내보내게 합니다. (3) 노출 → 톤 매핑(ACES 근사) → sRGB 인코드를 수행하는 풀스크린 포스트 패스를 추가합니다 — 인코드는 백버퍼 RTV를 _SRGB로 만들면 하드웨어에 넘길 수 있습니다. 현재 DISCARD 스왑체인은 _SRGB 백버퍼를 직접 만들 수 있고, FLIP 모델로 옮기면 백버퍼는 UNORM으로 두고 RTV만 _SRGB로 만드는 표준 기법을 쓰면 됩니다. (4) 이 구조가 서면 bloom은 HDR RT에서 1.0 초과 값을 추출하는 패스로 거의 공짜로 얹힙니다. 순서를 이렇게 잡은 이유는, 색 공간과 범위라는 '데이터 계약'을 먼저 바로잡아야 그 위의 모든 포스트 이펙트가 물리적으로 옳은 입력을 받기 때문입니다."

"있다/없다"를 표로 요약하면:

| 구성 요소 | Winters 현재 | 근거 |
|---|---|---|
| Cook-Torrance (D_GGX/F_Schlick/G_Smith) | **있음** — 표준 구현 | `BRDF_GGX.hlsli:7~63` |
| roughness 하한 clamp | 있음 (0.04) | `Mesh3D_PBR.hlsl:106` |
| linear workflow (스타일라이즈드) | 있음 — 셰이더 수동 디코드/인코드 | `Mesh3D.hlsl:163,168` |
| 하드웨어 _SRGB 경로 | **없음** — RHI 포맷 열거에 부재 | `RHITypes.h:39~54`, 백버퍼 UNORM `CDX11Device.cpp:793` |
| PBR 경로 albedo 디코드 | **확인 필요** — 로더 Auto 정책 의존 | `Mesh3D_PBR.hlsl:101~104` |
| HDR 씬 RT | 없음 — 포맷만 존재, RT 미사용 | `RHITypes.h:44`, `CDX11Device.cpp:133` |
| 톤 매핑 | PS 인라인 Reinhard (포스트 패스 없음) | `Mesh3D_PBR.hlsl:129` |
| 노출 / 자동 노출 | 없음 | — |
| IBL (irradiance/prefilter/LUT) | 없음 — 상수 ambient가 그 자리 | `Mesh3D_PBR.hlsl:125`의 `float3(0.03, ...) * albedo * AO` |

마지막 행이 좋은 마무리 소재다: `Mesh3D_PBR.hlsl:125`의 `ambient = 0.03 * albedo * AO`는 LearnOpenGL 계열 튜토리얼의 관습 상수인데, **이 상수가 정확히 diffuse irradiance map이 대체할 자리**다. "환경광 적분을 상수 하나로 근사한 상태이고, 다음 단계가 코사인 컨볼루션을 구운 irradiance 큐브맵"이라고 잇는 순간, 렌더링 방정식 → IBL → 내 코드의 공백이 한 줄로 꿰어진다.
