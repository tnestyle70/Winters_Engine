# OS의 본질과 커널 설계

> "운영체제가 뭐라고 생각하세요?", "시스템 콜이 함수 호출보다 왜 느린가요?", "모놀리식 커널과 마이크로커널의 차이는?"에 대한 답.
> 프로세스·메모리·파일 각론은 뒤 챕터로 미루고, 여기서는 그 모든 각론이 걸려 있는 하나의 뼈대 —
> "커널은 무엇을 지키고, 무엇을 추상화하고, 무엇을 문 안쪽에 둘 것인가" — 를 관통한다.

---

## ① 한 줄 본질

> **"OS는 두 개의 얼굴을 가진 소프트웨어다. 하나는 심판(자원 관리자) — 유한한 CPU·메모리·디바이스를 서로 믿을 수 없는 프로그램들에게 공정하고 안전하게 나눠 준다. 다른 하나는 마술사(가상화 제공자) — 지저분한 실제 하드웨어를 '프로세스·주소 공간·파일'이라는 깨끗한 가상 자원으로 바꿔 보여 준다. 이 두 역할을 강제할 수 있는 근거가 CPU의 이중 모드(dual mode)이고, 유저 프로그램이 그 경계를 합법적으로 넘는 유일한 문이 시스템 콜(system call)이다. 커널 설계 논쟁(모놀리식 vs 마이크로커널)은 결국 '그 문 안쪽에 무엇을 둘 것인가'를 두고 안전성과 경계 통과 비용을 저울질해 온 역사다."**

---

## ② 기본 개념 — 커널은 왜 이렇게 생겼는가

### OS의 두 얼굴 — 심판이자 마술사

교과서 정의("하드웨어와 응용 사이의 계층")는 맞지만 아무것도 설명하지 못한다.
설계를 이해하려면 OS가 풀려는 문제 두 개에서 출발해야 한다.

**문제 1 — 자원은 유한한데 프로그램은 여럿이고, 서로 믿을 수 없다.**
CPU 코어는 8개인데 스레드는 수천 개다. RAM은 16GB인데 모든 프로세스가 "내가 다 쓰고 싶다"고 한다.
게다가 버그 있는 프로그램, 악의적인 프로그램이 섞여 있다.
그래서 OS는 **자원 관리자(resource manager)**여야 한다:

- **스케줄링** — 누가 CPU를, 언제, 얼마나 쓰나.
- **할당** — 누가 메모리·디스크·대역폭을 얼마나 받나.
- **보호(protection)** — 남의 것을 읽지도 쓰지도 못하게.
- **회수** — 죽은 프로세스의 자원을 되찾아 재분배.

**문제 2 — 실제 하드웨어는 지저분하고, 기종마다 다르고, 프로그래밍하기 끔찍하다.**
디스크에 한 바이트 쓰려면 컨트롤러 레지스터에 명령을 채우고 DMA를 걸고 인터럽트를 기다려야 한다.
이걸 모든 응용이 직접 하면 (a) 아무도 생산적인 코드를 못 짜고 (b) 응용끼리 디바이스를 두고 충돌한다.
그래서 OS는 **가상화(virtualization)/추상화 제공자**여야 한다:

- 물리 CPU → "각자 CPU를 독점한 것 같은 착각" = **프로세스(process)**
- 물리 RAM → "0번지부터 시작하는 나만의 연속 메모리" = **주소 공간(address space)**
- 디스크 섹터 → "이름 있는, 크기가 자라는 바이트 스트림" = **파일(file)**

두 얼굴은 동전의 양면이다.
**나눠 주려면(관리) 먼저 착각을 만들어야 하고(가상화), 착각을 유지하려면 진짜 하드웨어 접근을 독점해야 한다(보호).**
이 "독점"을 소프트웨어 약속이 아니라 하드웨어로 강제하는 장치가 이중 모드다.
즉 이 챕터의 나머지 전부는 "착각을 어떻게 하드웨어로 지키는가"의 각론이다.

### 커널이 개입하는 세 개의 문 — 인터럽트·예외·트랩

이중 모드로 가기 전에 한 가지 그림을 바로잡아야 한다.
**커널은 항상 돌고 있는 상주 프로그램이 아니다.**
평소 CPU는 유저 코드를 전속력으로 직접 실행하고, 커널 코드는 세 종류의 사건이 일어날 때만 실행된다:

| 종류 | 동기/비동기 | 발생 원인 | 예 |
|---|---|---|---|
| 인터럽트(interrupt) | 비동기 (지금 실행 중인 명령과 무관) | 하드웨어 | 타이머 만료, NIC 패킷 도착, 디스크 완료 |
| 예외(exception) | 동기 (명령 실행의 부산물) | 실수/상태 | 페이지 폴트(page fault), 0으로 나누기, 특권 위반(#GP) |
| 트랩(trap) | 동기 (명령으로 의도적 발생) | 요청 | 시스템 콜, 디버그 브레이크포인트 |

셋 다 공통 구조를 가진다:
CPU가 유저 실행을 중단하고 → 특권을 커널 모드로 올리고 → **커널이 부팅 때 미리 등록해 둔 핸들러**로 점프한다.
커널은 이 사건들에만 반응하는 **이벤트 구동(event-driven) 프로그램**이다.
(부팅 직후 잠깐만 "커널이 주인공"이고 — 하드웨어 초기화, 핸들러 등록, 첫 유저 프로세스 기동 — 그 뒤로는 무대 뒤로 물러난다.)

이 구조가 OS 설계의 1원리를 만든다 — **제한된 직접 실행(limited direct execution)**:
성능을 위해 평소엔 하드웨어가 유저 코드를 직접 실행하게 두되(커널이 명령을 하나하나 해석하면 수십 배 느려진다),
착각이 깨질 순간(타이머, 폴트, 특권 명령, 서비스 요청)에는 반드시 커널로 제어가 돌아오게 하드웨어 장치를 심어 둔다.
"빠르게 하려면 커널이 비켜서 있어야 하고, 안전하려면 언제든 다시 개입할 수 있어야 한다."

특히 **타이머 인터럽트**는 심판 역할의 최후 보루다.
유저 프로그램이 무한 루프를 돌아도 타이머가 주기적으로 커널을 깨우므로 스케줄러는 CPU를 되찾을 수 있다.
이게 없던 시절의 협조적 멀티태스킹(cooperative multitasking, Windows 3.x·초기 Mac)에서는 앱 하나가 양보를 안 하면 시스템 전체가 멈췄다.
선점(preemption)이 하드웨어 타이머 위에 서 있다는 사실은 뒤의 스케줄링 챕터의 전제가 된다.

### 이중 모드(dual mode) — CPU가 두 개의 신분을 갖는 이유

보호를 소프트웨어만으로는 구현할 수 없다.
"이 함수는 호출하지 마세요"라는 약속은 기계어 한 줄이면 깨진다.
그래서 **CPU 자체가 현재 실행 중인 코드의 신분을 추적**한다.
x86에서는 특권 수준(privilege level)을 링(ring) 0~3으로 부르고, 실제 범용 OS는 두 개만 쓴다 —
링 0 = 커널 모드(kernel mode), 링 3 = 유저 모드(user mode).
(링 1·2는 사실상 사장됐고, 가상화는 링을 더 쪼개는 대신 VMX 같은 별도 실행 모드를 추가하는 쪽으로 갔다.)

| | 유저 모드 (ring 3) | 커널 모드 (ring 0) |
|---|---|---|
| 실행 주체 | 응용 프로그램 | 커널·드라이버 |
| 특권 명령 | 실행 시 예외(#GP) 발생 | 실행 가능 |
| 접근 가능한 메모리 | 자기 주소 공간의 유저 영역 | 전부 (커널 영역 포함) |
| I/O 디바이스 | 직접 접근 불가 | 직접 접근 |
| 잘못되면 | 그 프로세스만 죽음 | 시스템 전체가 죽음 (커널 패닉/BSOD) |

**특권 명령(privileged instruction)**이 이 구분의 실체다.
유저 모드에서 실행하면 CPU가 일반 보호 예외(general protection fault)를 일으키는 명령들:

- `cli` / `sti` — 인터럽트 끄기/켜기. 유저가 하면 타이머 인터럽트가 막혀 **스케줄러 무력화**.
- `mov cr3, ...` — 페이지 테이블 루트 교체. 유저가 하면 **메모리 보호 무력화**.
- `lgdt` / `lidt` — 디스크립터 테이블 로드. 유저가 하면 **신분 체계와 핸들러 등록 자체가 무력화**.
- `in` / `out` — I/O 포트 접근. 유저가 하면 **디바이스 독점 무력화**.
- `hlt` — CPU 정지, `wrmsr` — 모델 특화 레지스터(MSR) 쓰기.

패턴이 보인다: **특권 명령 = "착각을 만드는 장치 자체를 조작하는 명령"**이다.
가상화의 기반(페이지 테이블, 인터럽트, 디바이스, 진입점 등록)을 건드리는 명령을 유저에게서 뺏어야 가상화가 유지된다.

메모리 보호도 같은 원리로 두 겹이다:

1. 주소 변환 자체 — 유저는 가상 주소밖에 못 쓰고, 매핑(페이지 테이블)은 커널만 만진다. 매핑에 없는 물리 메모리는 **표현할 방법조차 없다**.
2. 페이지 테이블 엔트리의 U/S(user/supervisor) 비트 — 같은 주소 공간에 매핑된 커널 영역도 유저 모드 접근이면 하드웨어가 폴트를 낸다.

여기서 놓치기 쉬운 배치 하나: **커널은 어디에 "사는가"?**
전통적으로 커널은 별도 주소 공간이 아니라 **모든 프로세스의 주소 공간 상위 절반에 똑같이 매핑**되어 있다.

```
각 프로세스의 가상 주소 공간 (x86-64 개념도)
┌──────────────────────────┐ 상위 절반
│  커널 영역 (모든 프로세스 공통,  │  ← U/S 비트로 유저 접근만 차단
│  시스템 콜 때 여기 코드 실행)    │
├──────────────────────────┤
│  유저 영역 (프로세스마다 다름)   │  ← 코드·힙·스택
└──────────────────────────┘ 0번지
```

이 배치 덕에 시스템 콜은 **주소 공간 교체 없이** 커널 코드로 들어갈 수 있었다 — 모드 전환이 문맥 교환보다 싼 구조적 이유.
그리고 2018년 Meltdown이 바로 이 "같은 공간에 있되 비트로만 가린" 배치를 투기 실행으로 우회했고,
KPTI(kernel page-table isolation)가 유저 모드일 때 커널 매핑 대부분을 치우면서 시스템 콜 단가가 올라갔다.
배치의 이점과 그 대가가 한 세트로 움직인 사건이라 면접 소재로 좋다.

그런데 유저 프로그램도 결국 파일을 읽고 패킷을 보내야 한다.
특권이 필요한 일을 특권 없는 코드가 **요청**하는 통로 — 그것이 시스템 콜이다.

### 시스템 콜 — 유일한 합법적 국경 통과

핵심 설계 요구: 유저 코드가 커널 코드를 실행하게 하되, **어디로 들어올지는 커널이 정해야 한다.**
유저가 임의 커널 주소로 점프할 수 있으면 인자 검증을 건너뛰는 뒷문이 생긴다.
그래서 시스템 콜은 "함수 호출"이 아니라 **제어된 트랩**이다:
유저는 "몇 번 서비스"인지 번호만 제시하고, 진입 지점은 커널이 부팅 때 등록해 둔 단 한 곳이다.
함수 포인터를 넘기는 게 아니라 **메뉴판에서 번호를 고르는** 구조 — 이 비대칭이 보호의 핵심이다.

x86-64에서의 전체 경로 (`ReadFile` 호출을 예로):

```
[유저 코드]   ReadFile(...)                          ← 일반 함수 호출 (ring 3)
     │
[래퍼 계층]   kernel32.dll → ntdll.dll의 NtReadFile 스텁:
     │            mov r10, rcx          ; syscall이 rcx를 덮어쓰므로 대피
     │            mov eax, <syscall 번호>
     │            syscall
     ▼ ═══════════════ CPU 특권 전환: ring 3 → ring 0 ═══════════════
[커널 진입]   syscall 명령이 하드웨어로 하는 일:
     │          - 복귀 주소(RIP)를 RCX에, RFLAGS를 R11에 보존
     │          - LSTAR MSR에 부팅 때 등록해 둔 커널 진입점으로 점프
     │        진입 코드가 소프트웨어로 하는 일:
     │          - swapgs로 per-CPU 커널 데이터 접근 → 커널 스택으로 전환
     │            (syscall은 스택조차 자동으로 안 바꿔 준다)
     │          - 유저 레지스터를 커널 스택에 보존
[디스패치]    RAX의 번호로 시스템 콜 디스패치 테이블 인덱싱
     │        (리눅스 sys_call_table, Windows SSDT 계열)
[커널 함수]   본체 실행: 인자 검증(이 포인터가 진짜 유저 영역인가?)
     │        → I/O 관리자 → 파일시스템/드라이버 → 결과를 RAX에
[복귀]        레지스터 복원 → sysret ═══ ring 0 → ring 3 ═══ 다음 줄부터 재개
```

리눅스에서 래퍼 없이 직접 쓰면 이 정도로 작다 (NASM 문법):

```asm
; Linux x86-64: write(1, msg, 14)
mov     rax, 1              ; 시스템 콜 번호: write
mov     rdi, 1              ; 인자1: fd = stdout
lea     rsi, [rel msg]      ; 인자2: 버퍼 주소
mov     rdx, 14             ; 인자3: 길이
syscall                     ; ← 이 한 줄이 국경
; 반환값은 rax (음수면 -errno)
```

두 OS에서 같은 구조가 이름만 다르게 반복된다:

| 계층 | Windows | Linux |
|---|---|---|
| 응용이 부르는 API | Win32 (`ReadFile`, `WSARecv`, ...) | POSIX/libc (`read`, `recv`, ...) |
| 유저 모드 최하단 스텁 | ntdll.dll의 `Nt*` 함수 | glibc 래퍼 (또는 직접 `syscall`) |
| 국경 | `syscall` 명령 | `syscall` 명령 |
| 커널 쪽 접수처 | 시스템 서비스 디스패처 | `sys_call_table` 디스패치 |

진입 방식 자체에도 역사가 쌓여 있다:

| 진입 방식 | 시대 | 동작 | 왜 대체됐나 |
|---|---|---|---|
| `int 0x80` (소프트웨어 인터럽트) | 전통 x86 | IDT를 메모리에서 조회, 게이트 특권 검사, 스택 정보 로드 | 범용 인터럽트 경로라 느림 |
| `sysenter`/`sysexit` | 32비트 후기 (Intel) | 진입점을 MSR에 고정, 테이블 조회 제거 | 32비트와 함께 퇴장 |
| `syscall`/`sysret` | x86-64 표준 (AMD 기원) | LSTAR MSR로 즉시 점프, 최소 상태만 하드웨어가 처리 | 현행 |

교훈은 하나다:
**시스템 콜은 너무 자주 일어나는 이벤트라서, CPU 제조사가 전용 명령을 만들 정도로 경로를 특수화했다.**
"자주 지나는 국경엔 전용 게이트를 뚫는다."

실무 디테일 두 가지가 면접 포인트다:

- **호출 규약**: 번호는 RAX, 인자는 레지스터(리눅스: rdi, rsi, rdx, r10, r8, r9).
  네 번째 인자가 함수 호출 규약의 rcx가 아니라 **r10**인 이유 — syscall 명령이 rcx를 복귀 주소 저장에 써 버리기 때문.
  Windows ntdll 스텁의 `mov r10, rcx`도 같은 이유다. 하드웨어 제약이 소프트웨어 규약을 바꾼 흔적.
- **안정 ABI의 위치가 다르다**: 리눅스는 syscall 번호가 공개·안정 ABI라 정적 바이너리가 직접 `syscall`해도 되고,
  Windows는 번호가 버전/빌드마다 바뀌는 비공개 세부라 **반드시 ntdll.dll을 거쳐야 한다.**
  "리눅스의 공개 경계는 커널 인터페이스, Windows의 공개 경계는 시스템 DLL" — 두 OS의 호환성 전략 차이가 이 한 줄에 있다.

### 시스템 콜이 함수 호출보다 비싼 이유

함수 호출은 `call`/`ret` — 나노초 수준이고 컴파일러가 인라인으로 없애 버릴 수도 있다.
시스템 콜 왕복은 **같은 스레드, 같은 주소 공간인데도** 수십~수백 나노초로 한두 자릿수 비싸다.
비용을 층위로 나눠 답할 수 있어야 한다.

**직접 비용 — 경계 통과 자체:**

1. 특권 전환 + 레지스터 보존·복원 + swapgs + 커널 스택 전환. 전부 함수 호출엔 없는 일이다.
2. 비순차 실행(out-of-order)·투기 실행(speculation)이 특권 경계를 자유롭게 넘지 못한다 — 파이프라인 관점에서 실행 흐름이 한 번 끊긴다.
3. Meltdown(2018) 이후 KPTI가 켜진 시스템은 진입/복귀마다 페이지 테이블(CR3)을 교체한다 —
   **보안 완화 조치가 시스템 콜 단가를 더 올린** 대표 사례.
   "시스템 콜 비용은 고정 상수가 아니라 보안 환경에 따라 변해 온 값"이라고 말할 수 있으면 좋다.

**간접 비용 — 돌아온 뒤에 내는 값:**

4. 커널 코드·데이터가 훑고 지나가며 유저 프로그램의 캐시 라인과 TLB 엔트리를 밀어낸다.
   시스템 콜 직후 유저 코드의 IPC(사이클당 명령 수)가 한동안 떨어진다는 측정 연구들이 있다 —
   **청구서는 경계에서만 오지 않는다.**
5. 컴파일러 최적화의 벽: 경계 너머는 인라인도 값 전파도 불가능하고, 컴파일러는 콜 이후 메모리 상태를 전부 모른다고 가정해야 한다.

이 비용 구조가 OS API의 형태를 통째로 결정해 왔다.
**"경계는 비싸니, 넘는 횟수를 줄이거나 한 번에 더 많이 실어라"**:

- **배칭(batching)** — `writev`/`readv` 같은 벡터 I/O, `sendmmsg`: 한 번 진입에 N개 처리.
- **vDSO** — `gettimeofday`처럼 빈번한 읽기 전용 콜은 커널이 데이터를 유저 공간에 매핑해 두고 **커널 진입 없이** 처리.
- **공유 링 버퍼** — 리눅스 `io_uring`, Windows Registered I/O: 제출 큐/완료 큐를 유저·커널이 공유해 정상 경로의 트랩을 최소화.
- **완료 통보 집약** — IOCP: 수천 소켓의 완료를 소수 스레드가 큐 하나에서 수거. ④절에서 Winters 서버가 정확히 이 원리 위에 서 있음을 본다.

### 커널 설계 스펙트럼 — 무엇을 문 안쪽에 둘 것인가

이중 모드가 정해지면 다음 질문이 자동으로 나온다:
**파일시스템, 드라이버, 네트워크 스택은 링 0에 둘까, 링 3에 둘까?**

- 안(링 0)에 두면: 서브시스템끼리 **함수 호출**로 통신 — 빠르다. 대신 그 코드의 버그가 시스템 전체를 죽인다.
- 밖(링 3)에 두면: 버그가 프로세스 하나로 격리된다. 대신 통신할 때마다 **경계를 넘어야** 한다 — 느리다.

이 트레이드오프의 답이 커널 설계 스펙트럼이다.

```
 모놀리식                      하이브리드                    마이크로커널
 (Linux)                      (Windows NT, XNU)             (Mach 3, L4, QNX, seL4)

┌──────────────┐   ring 3    ┌──────────────┐              ┌──────┐┌──────┐┌──────┐
│    응용       │             │ 응용, 서브시스템 │              │ 응용  ││ FS   ││드라이버│
├──────────────┤ ═══════════ ├──────────────┤ ═══════════  │      ││서버   ││서버   │
│  커널          │   ring 0   │ Executive     │              └──┬───┘└──┬───┘└──┬───┘
│  ┌─────────┐  │            │ (객체·메모리·   │              ═══╪═══════╪═══════╪═══
│  │FS 드라이버 │  │            │  I/O 관리자)   │              ┌──┴───────┴───────┴──┐
│  │네트워크스택│  │            │ + 커널 + HAL   │              │ 마이크로커널:          │
│  │스케줄러    │  │            │ + win32k      │              │ IPC·스케줄링·주소공간만 │
│  └─────────┘  │            └──────────────┘              └─────────────────────┘
 전부 한 주소공간,              커널 모드지만 내부는            서비스끼리 메시지 패싱
 함수 호출로 통신               모듈화된 계층 구조              (호출마다 경계 통과)
```

| | 모놀리식 | 마이크로커널 | 하이브리드 |
|---|---|---|---|
| 커널 안의 코드 | FS·드라이버·네트워크 전부 | IPC·스케줄링·주소 공간 최소만 | 성능 민감부는 커널에, 구조는 모듈로 |
| 내부 통신 | 함수 호출 (거의 공짜) | 메시지 패싱(message passing) | 커널 내부는 함수 호출 |
| 드라이버 버그의 반경 | 커널 패닉 | 서버 프로세스 재시작 | 대부분 크래시 (일부 유저 모드 드라이버) |
| 대표 | Linux, 전통 UNIX | Mach 3, L4, QNX, MINIX, seL4 | Windows NT, macOS XNU |
| 이긴 영역 | 범용 서버/데스크톱 성능 | 차량·항공·고신뢰 임베디드 | 범용 데스크톱의 실용 균형 |

**모놀리식 — Linux.**
전부 링 0, 한 주소 공간. 서브시스템 간 통신이 함수 호출이라 빠르고, 자료구조를 포인터로 그냥 공유한다.
대가: 수천만 줄 어디의 널 포인터 역참조든 커널 패닉이 될 수 있다.
주의할 구분 — 리눅스의 로더블 모듈(loadable kernel module)은 **빌드·배포의 유연성**이지 격리가 아니다.
모듈도 로드되면 링 0에서 뛴다. 이 구분을 정확히 말하면 좋은 신호다.
1992년의 Tanenbaum–Torvalds 논쟁("모놀리식은 1970년대로의 퇴보다" vs "지금 돌아가는 물건이 이긴다")이 이 지점의 고전이다 —
이론적 우아함과 당대 하드웨어에서의 실효 성능이 충돌했고, 범용 OS 시장에서는 후자가 이겼다.

**마이크로커널 — Mach와 L4.**
1980년대 CMU의 Mach가 대표 주자: 커널엔 IPC(프로세스 간 통신)·스케줄링·주소 공간만 남기고 파일시스템·드라이버를 유저 모드 서버로 밀어냈다.
이론은 아름다웠지만 **모든 서비스 요청이 '트랩 + 메시지 복사 + 권한 검사 + 서버 간 문맥 전환'의 연쇄**가 되면서 성능이 무너졌다 —
파일 하나 읽는 데 유저→커널→FS서버→커널→드라이버서버→... 경계를 여러 번 넘는다.
1990년대 Jochen Liedtke의 L4가 반격했다: IPC 경로를 극단적으로 다듬어
(동기 랑데부, 짧은 메시지는 레지스터로 전달, 커널을 캐시에 들어갈 만큼 작게)
**"마이크로커널이 느린 게 아니라 Mach의 구현이 느렸다"**를 측정으로 증명했다.
그리고 이 계보가 실전에서 이긴 영역이 따로 있다 —
QNX는 차량·임베디드에서 표준급 지위를 얻었고, seL4는 커널이 작았기에 **기능 정확성을 수학적으로 형식 검증**하는 데 성공했다.
격리와 검증 가능성이 성능보다 비싼 도메인에서는 마이크로커널이 답이다.

**하이브리드 — Windows NT.**
Dave Cutler(DEC VMS 출신)가 설계한 NT는 "구조는 마이크로커널의 영향, 배치는 실용주의"다.
커널 모드 안이 계층화되어 있다:

```
링 3:  응용 │ 서브시스템 프로세스(csrss 등) │ 유저 모드 드라이버
═══════════════════════ 시스템 콜 경계 ═══════════════════════
링 0:  Executive  — 객체 관리자 · 메모리 관리자 · I/O 관리자 · 보안 참조 모니터
       Kernel     — 스레드 디스패칭 · 인터럽트 처리 · 저수준 동기화
       HAL        — 인터럽트 컨트롤러·타이머 등 플랫폼 차이 흡수
       (+ win32k.sys — NT 4.0부터 커널에 들어온 GUI)
```

HAL(hardware abstraction layer) 덕에 초기 NT는 x86·MIPS·Alpha·PowerPC에 이식됐다 —
"바뀌는 것(플랫폼)과 안 바뀌는 것(Executive 로직)의 경계에 계층을 세운다"는 원칙의 실물.
유명한 역사적 타협도 NT에 있다:
초기 NT는 GUI(GDI/윈도 관리)를 유저 모드 CSRSS 프로세스에 뒀는데, 그래픽 호출마다 프로세스 간 왕복이 발생해 느렸고,
**NT 4.0(1996)에서 win32k.sys로 커널에 내렸다.** "격리의 이상이 경계 통과 비용에 패배한" 교과서적 사례다.
반대 방향의 흐름도 있다 — 이후 Windows는 크래시가 잦은 부류의 드라이버(USB 주변기기 등)를 유저 모드 드라이버 프레임워크로 다시 올렸다.
**경계 배치는 고정된 정답이 아니라 비용과 사고율에 따라 계속 재협상된다.**

**macOS XNU**도 하이브리드다:
Mach(IPC·스케줄링·가상 메모리) + BSD(POSIX·네트워크·파일시스템) + IOKit(드라이버 프레임워크)을 **한 커널 주소 공간**에 합쳤다.
Mach 메시지라는 마이크로커널의 언어를 여전히 쓰지만, 서비스들이 격리 경계 없이 붙어 있으므로 마이크로커널이 아니다.

관통하는 결론:
**커널 설계 논쟁은 이념 싸움이 아니라 '경계 통과 비용 × 통과 빈도 vs 버그 폭발 반경'의 산수다.**

- 하드웨어가 빨라져 경계 비용이 내려가면 → 마이크로커널 쪽으로.
- 보안·신뢰 요구가 올라가면 → 마이크로커널 쪽으로.
- 처리량·지연이 아쉬우면 → 모놀리식 쪽으로.
- 같은 산수를 다른 상수로 풀면 다른 답이 나오는 것뿐이다.

### 정책(policy)과 메커니즘(mechanism)의 분리

커널 내부를 관통하는 설계 원칙 하나를 더 잡아야 한다.
1970년대 CMU의 Hydra 커널에서 정식화된 원칙: **"어떻게 하는가(메커니즘)"와 "무엇을 할 것인가(정책)"를 분리하라.**

| 영역 | 메커니즘 (안정적) | 정책 (자주 바뀜) |
|---|---|---|
| 스케줄링 | 문맥 교환, 런 큐 조작, 타이머 | 다음에 누구를 얼마나: 우선순위, 공정 분배 |
| 가상 메모리 | 페이지 테이블 조작, 폴트 처리 | 어느 페이지를 내쫓을까: 교체 알고리즘 |
| I/O | 요청 큐잉, DMA, 완료 통보 | 어떤 순서로 디스크에 보낼까: I/O 스케줄러 |

분리가 왜 중요한가:
**정책은 워크로드마다 정답이 다르고 세월에 따라 바뀌지만, 메커니즘은 안정적이다.**
리눅스가 스케줄링 정책을 수차례 갈아 끼우면서도(O(1) → CFS → 그 후속) 문맥 교환 메커니즘은 거의 그대로인 것이 증거다.
바뀌는 것과 안 바뀌는 것을 분리해 두면 바뀌는 쪽만 교체할 수 있다.

이 원칙을 극단까지 밀면 MIT의 엑소커널(exokernel)이 나온다:
커널은 하드웨어를 안전하게 다중화(메커니즘)만 하고, 추상화(정책)는 전부 응용에 링크되는 라이브러리 OS로 —
"커널이 강요하는 범용 추상화 자체가 특정 응용에겐 성능 세금"이라는 문제의식이다.
범용 OS로 이기진 못했지만, 이 사상은 유저 스페이스 네트워킹(DPDK)이나 유니커널로 살아 있다.

그리고 이것은 커널만의 지혜가 아니라 게임 엔진 설계 원칙이기도 하다 —
렌더링 백엔드(메커니즘)와 렌더 패스 구성(정책)의 분리, 그대로 ④절의 RHI로 이어진다.

### OS가 제공하는 3대 가상화 — 이후 챕터의 지도

"마술사" 얼굴을 정리하면 OS 과목 전체의 목차가 된다. 물리 자원 하나마다 가상 자원 하나:

| 물리 자원 | 가상 자원 | 만드는 착각 | 핵심 메커니즘 | 각론 |
|---|---|---|---|---|
| CPU (소수의 코어) | 프로세스/스레드 | "CPU를 나 혼자 쓴다" | 문맥 교환 + 스케줄링 + 타이머 인터럽트 | 프로세스·스레드·스케줄링 챕터 |
| RAM (하나의 물리 메모리) | 주소 공간 | "0번지부터 연속된 메모리가 내 것" | 페이징 + MMU/TLB + 페이지 폴트 | 가상 메모리 챕터 |
| 디스크 (섹터 덩어리) | 파일 | "이름 있는, 크기가 자라는 바이트 스트림" | 파일시스템 + 버퍼 캐시 | 파일시스템·I/O 챕터 |

세 줄이 같은 문법이라는 게 요점이다:
**평소엔 직접 실행/직접 접근으로 빠르게, 착각이 깨질 순간에만 트랩으로 커널이 개입.**
CPU 가상화에서 그 순간은 타이머 인터럽트고, 메모리 가상화에서는 페이지 폴트고, 파일에서는 시스템 콜이다.
각론에 들어가서 길을 잃으면 이 표로 돌아오면 된다 — 모든 각론은 "착각을 유지하는 비용을 얼마나 줄이는가"의 이야기다.

### 커널과 그 변주들 — VM·컨테이너·유니커널

이 챕터의 어휘로 세 가지 격리 기술을 정리할 수 있다 (상세는 별도 주제, 여기서는 커널과의 관계만).

```
        VM                          컨테이너                     유니커널
┌─────────┐┌─────────┐      ┌────────┐┌────────┐         ┌───────────────┐
│ 앱       ││ 앱       │      │ 앱      ││ 앱      │         │ 앱+라이브러리 OS │
│ 게스트커널││ 게스트커널│      └───┬────┘└───┬────┘         │ (한 주소 공간,   │
└────┬────┘└────┬────┘      ════╪══════════╪════          │  모드 전환 없음)  │
═════╪══════════╪═════       ┌──┴──────────┴───┐          └───────┬───────┘
┌────┴──────────┴────┐       │ 호스트 커널 하나    │          ┌───────┴───────┐
│ 하이퍼바이저          │       │ (네임스페이스+cgroup)│          │ 하이퍼바이저     │
└────────────────────┘       └─────────────────┘          └───────────────┘
 커널이 N개                    커널이 1개, 시야만 분리          내부 커널 경계 0개
```

- **가상머신(virtual machine)**: 이중 모드 아이디어를 한 단 더 쌓은 것.
  하이퍼바이저(hypervisor)가 링 0보다 특권이 높은 실행 모드(x86 VMX 등)에서 돌며 **게스트 커널의 특권 동작까지** 가상화한다.
  게스트마다 커널이 통째로 하나씩 — 격리 최강, 비용 최대.
- **컨테이너(container)**: 커널은 하나를 공유하고, 네임스페이스(pid·net·mnt 등)로 "보이는 세계"를, cgroup으로 자원 할당량을 분리한다.
  프로세스 생성만큼 가볍다. 대가는 격리 강도 — **격리 경계가 곧 시스템 콜 경계**라서 커널 취약점 하나면 모든 컨테이너가 뚫린다.
  그래서 seccomp로 허용 syscall 목록을 줄인다.
- **유니커널(unikernel)**: 앱 하나를 라이브러리 OS와 링크해 단일 주소 공간 이미지로 만들고 VM 게스트로 띄운다.
  내부에 유저/커널 경계가 없어 **시스템 콜이 함수 호출이 된다** —
  이 챕터의 비용 논의를 뒤집어, 격리는 하이퍼바이저에 외주 주고 경계 비용을 소거한 극단이다.

셋 다 같은 질문의 변주다: "격리 경계를 어디에, 몇 겹으로 세울 것인가" — 커널 설계 스펙트럼의 연장선이다.

## ③ 심화 — 꼬리질문 대비

**"모드 전환(mode switch)과 문맥 교환(context switch)의 차이는?"**
모드 전환은 **같은 스레드**가 유저↔커널 특권만 오가는 것 — 주소 공간 그대로, 시스템 콜마다 일어난다. 문맥 교환은 **다른 스레드/프로세스로** CPU를 넘기는 것 — 레지스터 상태 교체, (프로세스가 다르면) 페이지 테이블 교체와 TLB 영향까지 따라온다. 시스템 콜이 반드시 문맥 교환을 일으키는 건 아니다. 다만 블로킹 콜이면 커널이 그 스레드를 재우고 다른 스레드로 교환하므로 둘이 이어질 수 있다. 이 구분을 섞어 말하면 바로 감점 포인트다.

**"커널은 항상 실행 중인가?"**
아니다. 커널은 상주 데몬이 아니라 **메모리에 상주하는 코드 + 이벤트 핸들러의 집합**이다. CPU가 유저 코드를 도는 동안 커널 코드는 한 줄도 실행되지 않는다. 인터럽트·예외·트랩이 올 때만 실행되고, 처리 후 유저로 복귀한다. 백그라운드 회수 작업 같은 커널 스레드가 있긴 하지만 기본 그림은 이벤트 구동이다. 이 그림이 서 있어야 "시스템 콜 비용" 논의가 정확해진다 — 커널에게 메시지를 보내는 게 아니라, 내 스레드가 신분을 바꿔 커널 코드를 실행하는 것이다.

**"유저 모드에서 특권 명령을 실행하면 정확히 무슨 일이 일어나나?"**
CPU가 실행 단계에서 현재 특권 수준을 검사해 일반 보호 예외(#GP)를 발생시킨다. 제어는 IDT에 등록된 커널 핸들러로 트랩되고, 커널이 "복구 불가"로 판단하면 그 프로세스에 예외/신호를 전달해 종료시킨다. 핵심 두 가지: 검사가 소프트웨어 규약이 아니라 **명령 파이프라인 안의 하드웨어 검사**라는 점, 그리고 죽는 것은 프로세스이지 시스템이 아니라는 점.

**"syscall 명령이 int 0x80보다 빠른 이유는?"**
`int`는 범용 인터럽트 경로다 — IDT 엔트리를 메모리에서 읽고, 게이트 종류·특권을 검사하고, 스택 전환 정보를 찾는다. `syscall`은 시스템 콜 전용으로, 진입점·세그먼트 설정을 MSR에 부팅 때 한 번 박아 두고 실행 시엔 테이블 조회 없이 레지스터 몇 개만 조작하고 점프한다. 범용 경로의 특수화다. 대신 스택 전환 같은 일을 안 해 주므로 커널 진입 코드가 더 조심스러워야 한다 — swapgs 처리 실수가 실제 보안 취약점의 소재가 됐을 만큼, 빠른 경로는 안전 마진을 소프트웨어에 넘긴 경로다.

**"시스템 콜 한 번이 실제로 몇 ns인가?"**
숫자 하나로 단정하면 안 되는 질문이다. 최소 콜(getpid류)의 왕복은 현대 CPU에서 수십~수백 ns 오더로, 함수 호출(약 1~2ns)과 두 자릿수 차이가 난다. 그러나 (1) KPTI 같은 보안 완화가 켜졌는지, (2) 콜 본체가 안에서 얼마나 일하는지, (3) 돌아온 뒤의 캐시/TLB 오염을 어디까지 계상하는지에 따라 크게 달라진다. 정확한 답변 태도: 오더와 비용의 구조(직접 전환 비용 + 간접 오염 비용)로 답하고, "그래서 경계 통과 횟수를 설계 변수로 삼는다"로 마무리하는 것.

**"OS는 왜 일부 시스템 콜을 커널 진입 없이 처리하나? (vDSO)"**
시간 조회처럼 (a) 극도로 빈번하고 (b) 읽기 전용이고 (c) 쓰기 특권이 필요 없는 콜은, 커널이 그 데이터를 담은 페이지와 읽기 코드를 유저 주소 공간에 매핑해 둔다(리눅스 vDSO, Windows의 공유 데이터 페이지). 호출은 그냥 유저 모드 함수 호출이 된다. 원리: **특권이 필요했던 건 '갱신'이었지 '조회'가 아니었다** — 경계의 목적을 다시 물어서 경계 자체를 제거한 사례. "보호가 무엇을 지키는지 정확히 알면 비용을 없앨 수 있다"는 이 챕터 전체의 축소판이다.

**"모놀리식인 리눅스가 왜 이렇게 안정적으로 돌아가나?"**
설계가 아니라 공정(process)으로 메꾼 것이다 — 대규모 리뷰 체계, 정적 분석, 커버리지 퍼징(syzkaller), 엄격한 서브시스템 관리자 구조. 그러나 원리적 한계는 그대로다: 서드파티 드라이버의 널 포인터 하나가 여전히 커널 패닉이다. 그래서 리눅스도 경계를 부분 도입해 왔다 — FUSE(유저 공간 파일시스템), eBPF(정적 검증기를 통과한 코드만 커널 안에서 실행 — "격리 대신 증명"이라는 제3의 답), 커널 일부의 Rust 도입(메모리 안전을 언어 차원에서). "모놀리식의 약점을 인정하고 다른 층위에서 보완해 온 역사"로 답하면 입체적이다.

**"마이크로커널 IPC는 왜 느렸고, L4는 어떻게 빨리 만들었나?"**
Mach의 메시지는 범용이었다 — 가변 크기, 포트 권한 검사, 메모리 복사, 비동기 큐잉. 서비스 하나 호출이 '트랩 + 복사 + 스케줄링 + 반대 방향 반복'이 된다. L4의 처방: IPC를 커널의 제1시민으로 놓고 극단적으로 다이어트 — 동기 랑데부(송신자와 수신자가 만나는 순간 커널이 바로 제어를 넘겨 스케줄링 왕복 제거), 짧은 메시지는 레지스터로 전달(복사 제거), 커널 자체를 캐시에 들어갈 만큼 작게(오염 최소화). 교훈: **"추상화가 느린 게 아니라 범용성이 느리다. 뜨거운 경로는 특수화하라."** syscall 명령이 int 0x80을 대체한 것과 같은 문장이다.

**"Windows NT가 GUI를 커널에 넣은 결정을 어떻게 평가하나?"**
NT 3.x는 GDI/윈도 관리를 유저 모드 CSRSS에 뒀고, 그래픽 호출마다 프로세스 간 왕복이 발생했다. 당시 하드웨어에서 이 비용이 체감 성능을 지배했고, NT 4.0(1996)이 win32k.sys로 커널에 내렸다. 평가는 양면으로: 당시엔 합리적 트레이드오프였지만, 이후 win32k는 Windows 커널 취약점의 주요 공급원이 됐다 — 폰트 파싱 같은 복잡한 코드가 링 0에 들어갔기 때문. **격리를 포기하면 성능을 즉시 받고 보안 비용을 할부로 낸다.** 이후 Windows가 일부 드라이버를 유저 모드로 되돌린 흐름까지 붙이면 "경계는 재협상된다"는 관점이 완성된다.

**"디바이스 드라이버는 유저 모드인가 커널 모드인가?"**
설계 선택의 문제고 실제로 갈린다. 전통적으로 커널 모드(성능, 인터럽트 직접 처리)지만, Windows는 크래시 통계가 나쁜 부류(USB 주변기기 등)를 유저 모드 드라이버 프레임워크로 올렸다. GPU가 가장 좋은 예다: 그리기 명령 생성은 **유저 모드 드라이버(UMD)** — 게임 프로세스에 로드되는 DLL — 가 하고, 하드웨어 제출·메모리·스케줄링만 **커널 모드 드라이버(KMD)** 가 한다. 이유는 이 챕터 그 자체: **드로우 콜마다 커널에 들어가면 게임이 안 돌아간다.** 유저 모드에서 명령 버퍼를 쌓고 가끔 한 번 커널로 제출한다.

**"안티치트는 왜 커널 드라이버로 내려가나?"**
이중 모드의 규칙 때문이다. 유저 모드 안티치트는 링 3의 존재라서, 같은 링 3의 치트는 볼 수 있어도 **링 0에 자리 잡은 치트(악성 드라이버)는 원리적으로 관측할 수 없다** — 낮은 특권은 높은 특권을 검사할 수 없다. 그래서 상용 안티치트가 커널 드라이버로 내려가 같은 특권 수준에서 메모리 변조·코드 인젝션을 감시한다. 대가도 이 챕터의 문법 그대로: 안티치트 자체가 링 0 코드가 되므로 그 버그는 BSOD가 되고, 보안 사고의 폭발 반경도 시스템 전체가 된다. "권한의 대칭성" 문제로 답하면 원리 이해로 읽힌다.

**"커널 패닉/BSOD는 왜 복구하지 않고 그냥 멈추나?"**
커널 코드가 자신의 불변식이 깨진 것을 확인한 순간, 커널이 관리하는 모든 자료구조(페이지 테이블, 파일시스템 캐시, 스케줄러 큐)의 무결성을 더 이상 신뢰할 수 없기 때문이다. 계속 실행하면 오염된 상태로 디스크에 쓰기가 나가 **영구 데이터 손상**으로 번질 수 있다. 그래서 fail-stop(즉시 정지 + 메모리 덤프)이 계속 실행보다 안전하다는 판단이 표준이 됐다. 유저 프로세스는 죽여도 커널이 자원을 회수해 주지만, 커널 자신이 죽으면 회수해 줄 상위 존재가 없다는 비대칭도 같이 말하면 좋다.

**"시스템 콜 횟수 자체를 줄이는 현대적 I/O 설계를 아는 대로 말해 보라."**
(1) **배칭**: `writev`, `sendmmsg` — 한 번 진입에 N개. (2) **공유 메모리 큐**: 리눅스 `io_uring`, Windows Registered I/O — 제출/완료 큐를 유저·커널이 공유해 정상 경로에서 트랩 없이 요청·완료를 교환. (3) **완료 통보 집약**: IOCP — 수천 소켓의 완료를 소수 스레드가 하나의 큐에서 수거. (4) **극단**: DPDK류 유저 스페이스 네트워킹 — NIC 큐를 유저 공간에 매핑해 커널을 데이터 경로에서 완전히 배제(대가: 폴링으로 코어를 태움). 전부 한 문장으로 묶인다: **경계 비용이 고정이라면, 통과 1회당 일의 양을 키우거나 통과를 없애라.**

**"malloc과 VirtualAlloc(또는 brk/mmap)의 관계는? 어느 쪽이 시스템 콜인가?"**
`malloc`/`new`는 유저 모드 라이브러리(CRT 힙)다. 대부분의 할당은 이미 받아 둔 페이지를 잘라 주는 것으로 **커널 진입 없이** 끝난다. 라이브러리 풀이 바닥날 때만 `VirtualAlloc`(리눅스는 `brk`/`mmap`) 시스템 콜로 주소 공간을 더 받아 온다. 즉 힙 관리자는 시스템 콜 비용을 상쇄하는 **유저 모드 캐시 계층**이다. 같은 구분이 printf(라이브러리+버퍼링) vs write(콜), std::thread(라이브러리) vs 커널 스레드 생성 콜에도 적용된다 — "이 함수는 국경 안인가 밖인가"를 습관적으로 묻는 것이 시스템 프로그래머의 감각이다.

**"컨테이너가 VM보다 가벼운 이유를 커널 관점에서 설명하라."**
VM은 게스트마다 커널을 통째로 하나씩 돌린다 — 부팅할 커널, 유지할 페이지 테이블, 이중으로 내는 특권 동작 가상화 비용. 컨테이너는 커널이 하나고, 프로세스들이 보는 **이름 공간**(pid·mount·network)과 **자원 한도**(cgroup)만 분리한다 — 시작이 프로세스 생성만큼 가볍다. 대가는 격리 강도: 모든 컨테이너가 같은 커널로 시스템 콜을 던지므로 커널 취약점이 공통 탈출구다. 그래서 실무는 seccomp로 syscall 표면을 줄이거나, 아예 경량 VM(컨테이너마다 최소 커널)으로 회귀하기도 한다 — 스펙트럼 위에서 격리와 비용을 다시 저울질하는 것.

**"seL4의 형식 검증은 왜 마이크로커널이라서 가능했나?"**
검증 비용은 코드 크기와 상태 공간에 대해 폭발적으로 는다. seL4는 커널이 만 줄 오더로 작았기에 "C 구현이 명세와 일치한다"는 기능 정확성 증명을 완수할 수 있었다. 수천만 줄의 모놀리식 커널엔 같은 접근이 사실상 불가능하다. 즉 마이크로커널의 '작음'은 성능을 내주고 산 속성이지만, 그 대가로 **도달 가능한 신뢰성의 상한**이 달라진다 — 항공·차량·군용에서 마이크로커널 계열이 선택되는 이유다.

**"게임 클라이언트/서버 프로그래머가 커널 설계를 알아야 하는 이유는?"**
프레임 예산과 틱 예산이 밀리초 단위이기 때문이다. 시스템 콜·문맥 교환·페이지 폴트는 전부 마이크로초~밀리초를 먹을 수 있는 커널 개입 이벤트고, 이를 모르면 "가끔 프레임이 튀는" 문제를 영원히 앱 코드 안에서만 찾게 된다. 구체적으로: 메인 루프 안의 무심한 블로킹 콜 한 줄, 매 프레임 힙이 커널에 페이지를 반납/재요청하게 만드는 할당 패턴, 스레드 수 ≫ 코어 수로 인한 문맥 교환 증가 — 전부 커널 경계에 대한 무지에서 나온다. 반대로 IOCP·드로우 콜 배칭·프레임 초입의 선할당은 전부 경계 비용을 아는 사람의 설계다.

## ④ Winters에서의 적용 — 내 코드는 어디서 커널과 만나는가

Winters는 커널을 만든 프로젝트가 아니라 커널 **위에서** 밀리초를 다투는 프로젝트다.
그래서 이 챕터의 개념은 "내 코드의 어느 줄이 국경이고, 왜 그 국경 배치를 택했나"로 연결하는 것이 정직하고, 면접에서도 그렇게 말하는 편이 세다.

### (1) 시스템 콜 경계 지도 — 서버의 모든 능력은 커널에서 빌려 온다

Winters 서버가 유저 모드에서 스스로 할 수 없어 커널에 요청하는 지점들이 코드에 그대로 있다:

| Winters 코드 | 커널에 요청하는 것 |
|---|---|
| `Server/Private/Network/IOCPCore.cpp:44` `CreateIoCompletionPort(INVALID_HANDLE_VALUE, ...)` | 완료 큐라는 커널 객체 생성 |
| `Server/Private/Network/IOCPCore.cpp:51` `WSASocketW(..., WSA_FLAG_OVERLAPPED)` | 비동기 가능 소켓 — 커널 네트워크 스택의 끝점 — 생성 |
| `Server/Private/Network/IOCPCore.cpp:121` `m_workers.emplace_back(...)` (`std::thread`) | 스레드 생성 — 내부적으로 CreateThread 계열 콜을 거쳐 커널 스케줄러에 등록 |
| `Server/Private/Network/Session.cpp:62` `WSARecv(..., &overlapped, nullptr)` | "이 소켓에 데이터가 오면 이 버퍼에 채워 달라"는 수신 요청 등록 |
| `Server/Private/Network/Session.cpp:114` `WSASend(...)` | 송신을 커널 프로토콜 스택에 위임 |
| `Server/Private/Network/IOCPCore.cpp:216` `GetQueuedCompletionStatus(..., INFINITE)` | 완료가 올 때까지 이 스레드를 커널 대기 상태로 |

이 표를 말로 풀면:
**스레드 생성도, 메모리 커밋도, 패킷 송수신도 유저 모드 코드가 직접 할 수 있는 일이 아니다.**
NIC 레지스터를 만지는 것도, 스케줄러 런 큐에 스레드를 넣는 것도 특권 작업이다.
`WSARecv`/`WSASend`도 Winsock 라이브러리 계층을 지나 결국 ntdll의 Nt* 시스템 콜로 커널 소켓 드라이버에 도달한다.
서버 코드의 본질은 "커널에 무엇을, 언제, 몇 번 요청할 것인가"의 설계다.

### (2) IOCP — "경계는 비싸다"를 아키텍처로 만든 것

이 챕터의 핵심 명제(경계 통과는 비싸고, 블로킹은 문맥 교환을 낳는다)를 그대로 뒤집으면 IOCP 서버 구조가 나온다.

고전적 대안(스레드-퍼-세션)은 세션마다 스레드가 recv에서 **블로킹**한다:
세션 수만큼 스레드 → 커널 대기 객체 수천 개 → 패킷이 올 때마다 깨우고 재우는 문맥 교환의 연쇄 → 스케줄링과 경계 비용이 세션 수에 비례해 자란다.

Winters의 구조는 반대다:

```
[세션 코드]  WSARecv(overlapped) ──syscall──▶ 커널에 수신 요청만 등록
   Session.cpp:62                              │  즉시 반환 (블로킹 없음)
   WSA_IO_PENDING이면 정상 경로 (Session.cpp:65)  │
                                               │  ... NIC 인터럽트, 커널이 버퍼 채움 ...
                                               ▼
[워커 스레드]  GetQueuedCompletionStatus ◀── 완료 패킷을 IOCP 큐에 게시
   IOCPCore.cpp:216 (INFINITE 대기)            워커는 완료가 있을 때만 깨어남
        │
        ▼  CONTAINING_RECORD로 OVERLAPPED* → IOContext 복원 (IOCPCore.cpp:225)
   Recv 완료 → 프레임 파싱/디스패치 → 다시 WSARecv 등록 → 다음 완료 수거
```

설계 판단이 챕터 개념과 겹치는 지점을 짚으면:

- **비동기 등록 + 완료 수거 = 경계당 일의 양 키우기.**
  `WSARecv`는 '기다림'을 '요청'에서 분리한다. 스레드는 소켓 수와 무관하게 소수(워커 풀)이고,
  커널 왕복은 완료 이벤트가 실제로 있을 때만 의미 있게 일어난다.
  `Session.cpp:65`에서 `WSA_IO_PENDING`을 에러가 아닌 정상으로 처리하는 코드가 이 모델의 문법이다 —
  "지금 안 됐음"이 아니라 "커널이 접수했음". Accept 경로도 같은 문법으로, `IOCPCore.cpp:189`가 `ERROR_IO_PENDING`을 정상 접수로 구분한다.
- **완료 큐의 동시성 힌트 = 정책의 위탁.**
  `IOCPCore.cpp:44`는 완료 포트를 만들 때 동시 실행 스레드 수를 `m_workerCount`로 지정한다.
  IOCP는 활성 상태로 깨우는 스레드 수를 이 값 근처로 조절하는 스로틀을 커널 안에 가진다 —
  "실행 가능한 스레드 수 ≈ 코어 수"라는 스케줄링 **정책**을 응용이 커널 **메커니즘**에 위탁하는 것.
  정책/메커니즘 분리를 소비자 입장에서 활용한 실물이다.
- **`INFINITE` 대기는 낭비가 아니라 설계다.**
  `IOCPCore.cpp:216`의 워커는 완료가 없으면 커널 대기 상태로 들어가 CPU를 전혀 소모하지 않는다.
  폴링(반복 시스템 콜로 계속 물어보기)과 인터럽트 구동(커널이 깨워 줌)의 트레이드오프에서 후자를 택한 것이고,
  이 선택이 가능한 이유 자체가 "스레드를 재우고 깨우는 일은 커널만 할 수 있는 특권 작업"이기 때문이다.
- **종료조차 커널 큐를 통해 한다.**
  `IOCPCore.cpp:143`은 셧다운 시 `PostQueuedCompletionStatus`로 널 완료를 워커 수만큼 밀어 넣어 `INFINITE` 대기를 깨운다.
  커널 안에서 블로킹된 스레드를 밖에서 안전하게 풀 수 있는 통로 역시 커널 객체뿐이라는 사실의 반영이다.

면접 문장으로 압축하면:
**"IOCP를 쓴 이유는 API 취향이 아니라, 시스템 콜과 문맥 교환이 비싸다는 사실에서 역산한 구조 선택입니다. 대기는 커널에 맡기고, 스레드는 코어 수 근처로 유지하고, 경계는 완료가 있을 때만 넘습니다."**

### (3) RHI와 D3D11 — 엔진 안의 HAL, 그리고 그래픽 경로의 경계 배치

Windows NT가 HAL로 플랫폼 차이를 흡수해 커널 상부를 이식 가능하게 만들었듯,
Winters 엔진은 `Engine/Public/RHI/IRHIDevice.h:12`의 `IRHIDevice` 가상 인터페이스로 그래픽 백엔드 차이를 흡수한다 —
렌더러 상부는 "버퍼를 만들고 커맨드를 제출한다"는 계약(메커니즘)만 알고, 그것이 DX11로 어떻게 구현되는지는 아래층(`CDX11Device`)의 사정이다.
규모는 전혀 다르지만 원리는 같은 문장이다:
**바뀔 수 있는 것(백엔드/플랫폼)과 바뀌지 않아야 하는 것(상부 로직)의 경계에 추상화 계층을 세운다.**

그리고 그 아래에서 D3D11 스택 자체가 이 챕터의 비용 논리로 설계돼 있다.
Draw/상태 변경 호출은 대부분 **유저 모드**(D3D 런타임 + 벤더 UMD)에서 커맨드 버퍼에 쌓이기만 하고,
커널 그래픽 서브시스템과 KMD로의 제출은 묶여서 훨씬 드물게 일어난다.
프레임당 수천 드로우 콜이 가능한 이유가 "드로우마다 시스템 콜"이 아니기 때문이다.
클라이언트는 그래픽 경로에서도 이미 '경계 통과 최소화' 설계의 수혜자다 — GPU 드라이버의 UMD/KMD 분리를 물으면 이 지점으로 답하면 된다.

### (4) 이 개념을 내 프로젝트 경험으로 말하는 법

"OS를 직접 설계해 본 적은 없다"를 약점으로 두지 않는 화법:
**커널의 소비자로서, 경계 비용을 근거로 구조를 선택해 본 경험**을 말한다.

1. **서버** — 블로킹 recv + 스레드-퍼-세션 대신 IOCP를 택한 이유를 모드 전환/문맥 교환/대기 비용으로 설명한다.
   "커널이 제공하는 완료 큐에 대기를 위탁했다"는 문장이 나오면 원리 이해로 읽힌다.
2. **클라이언트** — 드로우 콜이 시스템 콜이 아니라는 것, UMD/KMD 분리와 배칭 위에 렌더러가 서 있다는 것을 설명한다.
3. **공용** — std::thread·소켓·malloc이 전부 시스템 콜 위의 유저 모드 편의 계층임을 구분해 설명한다.
   "이 함수는 국경 안인가 밖인가"를 습관적으로 묻는다는 인상을 준다.

세 갈래가 전부 "경계가 어디이고 왜 비싼가"라는 하나의 원리에서 나온다는 걸 보여 주면,
각론 암기가 아니라 원리에서 각론을 유도하는 사람으로 읽힌다.

## ⑤ 면접 Q&A

**Q1. 운영체제란 무엇인가?**
- 핵심: 두 얼굴로 답한다 — 자원 관리자(스케줄링·할당·보호·회수) + 가상화 제공자(CPU→프로세스, RAM→주소 공간, 디스크→파일). 둘은 동전의 양면: 나눠 주려면 착각을 만들어야 하고, 착각을 지키려면 하드웨어 접근을 독점해야 한다.
- 꼬리 대비: "그 독점은 뭘로 강제되나?" → 이중 모드와 특권 명령으로 이어 갈 준비.

**Q2. 유저 모드/커널 모드는 왜 필요한가?**
- 핵심: 보호는 소프트웨어 약속으로 불가능하다. CPU가 특권 수준을 추적하고, 가상화의 기반을 조작하는 명령(cr3, cli, in/out...)을 하드웨어가 차단해야 착각이 유지된다.
- 꼬리 대비: "관리자 권한과는 뭐가 다른가?" → 계정 권한은 유저 모드 안의 OS 개념, 링은 CPU 개념 — 층이 다르다.

**Q3. 시스템 콜의 과정을 설명해 보라.**
- 핵심: API 래퍼 → 시스템 콜 스텁(번호를 RAX에) → `syscall` 명령(RIP/RFLAGS 보존, LSTAR 진입점 점프) → swapgs·커널 스택 전환 → 번호로 디스패치 테이블 인덱싱 → 인자 검증 후 본체 → `sysret` 복귀.
- 꼬리 대비: "왜 유저가 커널 함수 주소로 직접 점프하면 안 되나?" → 검증을 우회하는 뒷문 방지, 진입점은 커널이 정한 단 한 곳.
- Winters: WSARecv 한 줄이 이 경로 전체를 탄다(`Session.cpp:62`).

**Q4. 시스템 콜이 함수 호출보다 왜 비싼가?**
- 핵심: 직접 비용(특권 전환, 레지스터/스택 처리, 투기 실행 단절, KPTI 시 CR3 교체) + 간접 비용(캐시/TLB 오염, 최적화 경계). 오더로는 함수 호출 대비 수십~수백 배.
- 꼬리 대비: "그래서 뭘 하나?" → 배칭·vDSO·공유 링 버퍼·완료 집약(IOCP) — 횟수를 줄이거나 1회당 일을 키운다.
- Winters: IOCP 채택 이유가 이 문장의 역산(`IOCPCore.cpp:216`).

**Q5. 모놀리식 vs 마이크로커널을 비교하라.**
- 핵심: 문 안(링 0)에 두면 함수 호출로 빠르지만 버그 반경이 시스템 전체, 밖에 두면 격리되지만 호출마다 경계 통과. Linux vs Mach/L4/QNX/seL4, 그리고 하이브리드(NT, XNU)가 중간.
- 꼬리 대비: "마이크로커널은 왜 느렸나?" → Mach IPC의 범용성 비용, L4의 특수화 반격까지.
- 마무리: "이념이 아니라 '경계 비용 × 빈도 vs 폭발 반경'의 산수이고, 도메인마다 답이 다르다."

**Q6. Windows 커널 구조를 아는 대로 말해 보라.**
- 핵심: 하이브리드 — 링 0 안에 HAL(플랫폼 차이 흡수) / Kernel(디스패칭·인터럽트) / Executive(객체·메모리·I/O 관리자) 계층, 링 3에 서브시스템 프로세스와 유저 모드 드라이버. NT 4.0의 win32k 커널 이동(성능 타협)과 이후 유저 모드 드라이버 회귀(안정성 재협상)를 역사로 붙인다.
- 꼬리 대비: "그럼 Windows는 마이크로커널인가?" → 아니다, 격리 경계가 실제로 어디 있는지가 기준.

**Q7. IOCP가 왜 효율적인지 OS 관점에서 설명하라.**
- 핵심: 스레드-퍼-세션의 비용(스레드 수천, 블로킹 대기, 문맥 교환 연쇄)을 뒤집어 — 요청은 비동기로 등록하고(WSARecv+OVERLAPPED), 대기는 커널 완료 큐에 위탁하고(GQCS), 워커는 코어 수 근처로 유지한다(완료 포트 동시성 힌트).
- Winters: `IOCPCore.cpp:44`(동시성 힌트), `Session.cpp:62`(비동기 등록), `IOCPCore.cpp:216`(완료 수거), `IOCPCore.cpp:143`(셧다운 웨이크업)으로 전 과정을 코드로 짚을 수 있다.

**Q8. 정책과 메커니즘의 분리란?**
- 핵심: "어떻게"(문맥 교환, 페이지 조작)와 "무엇을"(누굴 스케줄할지, 뭘 내쫓을지)의 분리. 정책은 바뀌고 메커니즘은 안정적이므로, 분리해 두면 정책만 교체 가능 — 리눅스 스케줄러 교체사가 증거.
- Winters: RHI가 같은 원리 — 백엔드(메커니즘)와 렌더 구성(정책)의 경계에 `IRHIDevice`를 세웠다.

## ⑥ 흔한 오답 / 함정

1. **"커널 = 다른 곳에서 항상 돌고 있는 별도 프로그램"이라는 그림.**
   시스템 콜 동안 실행 주체는 **같은 스레드**다. 신분(특권)만 바꿔 커널 코드를 실행하고 돌아온다.
   "커널 프로세스에게 메시지를 보낸다"는 그림이 맞는 쪽은 마이크로커널의 유저 모드 서버들이고, 모놀리식/하이브리드의 시스템 콜에는 틀린 그림이다.

2. **모드 전환과 문맥 교환을 섞어 쓰기.**
   시스템 콜 ≠ 문맥 교환. 논블로킹 콜은 같은 스레드로 즉시 돌아온다.
   "시스템 콜은 컨텍스트 스위칭 때문에 느리다"는 정확히 반 틀린 답이다 —
   비용의 주범은 특권 전환 + 캐시/TLB 오염이고, 문맥 교환은 블로킹했을 때 따라오는 별개 항목이다.

3. **"마이크로커널은 실패한 아이디어"라는 단정.**
   범용 데스크톱/서버 시장에서 모놀리식·하이브리드가 이긴 건 맞지만,
   QNX(차량), seL4(고신뢰 임베디드), 하이퍼바이저(최소 신뢰 기반이라는 같은 문제의식)처럼 마이크로커널이 이긴 영역이 뚜렷하다.
   "워크로드와 요구 격리 수준에 따라 답이 다르다"가 정답의 형태다.

4. **라이브러리 함수와 시스템 콜을 구분하지 않기.**
   malloc, printf, std::thread 생성자는 시스템 콜이 아니다 — 시스템 콜 위에 세워진 유저 모드 계층이다.
   "malloc은 시스템 콜이라 느리다"는 두 겹으로 틀린 문장이다(대부분 커널 진입 없음 + 힙 비용의 주원인은 힙 관리자 자체).

5. **시스템 콜 비용을 구체 숫자 하나로 단정.**
   "300ns입니다"처럼 답하면 KPTI 여부·CPU 세대·측정 방식을 되물었을 때 무너진다.
   오더(함수 호출 대비 수십~수백 배)와 비용의 구조(직접 전환 + 간접 오염)로 답하는 것이 방어 가능한 형태다.

6. **"Windows는 마이크로커널"이라고 답하기.**
   NT는 내부가 계층·모듈화된 **하이브리드**고, 파일시스템·드라이버·GUI(win32k)가 커널 모드에서 돈다.
   "구조는 마이크로커널의 영향, 배치는 성능을 위해 커널 안"이 정확한 표현.
   macOS도 같은 함정 — Mach가 들어 있다고 마이크로커널이 되는 게 아니다. 격리 경계가 실제로 어디 있는지가 기준이다.

7. **이중 모드를 "관리자 권한"과 혼동.**
   관리자/루트는 **OS가 관리하는 계정 권한**(유저 모드 안의 구분)이고, 커널 모드는 **CPU가 강제하는 특권 수준**이다.
   관리자 프로세스도 링 3에서 돌며, 특권 명령을 실행하면 똑같이 죽는다.
   층이 다른 두 보안 개념을 한 층에 놓고 말하면 바로 티가 난다.

8. **"비동기 I/O = 논블로킹 I/O"로 뭉뚱그리기.**
   논블로킹은 "준비 안 됐으면 즉시 실패로 돌아옴"(readiness 모델, select/epoll 계열)이고,
   비동기(overlapped)는 "커널이 완료까지 해 주고 나중에 알려 줌"(completion 모델, IOCP/io_uring 계열)이다.
   커널이 해 주는 일의 양이 다르고, 버퍼 소유권 규칙도 다르다 — IOCP를 썼다면 이 구분은 반드시 물어 온다.
