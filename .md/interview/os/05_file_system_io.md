# 파일 시스템과 I/O 모델

> "write()가 성공하면 데이터는 디스크에 있는 건가요?", "select와 epoll, IOCP는 뭐가 다른가요?", "게임 서버는 왜 IOCP를 쓰나요?"에 대한 답. 파일이라는 추상화가 어떻게 만들어지는지(디스크립터 → 파일시스템 자료구조 → 저널링 → 캐시)를 아래에서 위로 쌓은 뒤, 그 위에서 I/O 모델 5종이 왜 그 순서로 진화했는지를 관통한다.

---

## ① 한 줄 본질

> **"파일 시스템은 '섹터의 바다'인 디스크를 '이름 붙은 바이트 스트림 + 메타데이터'로 추상화하는 계층이고, 그 사이의 모든 설계(inode, 저널링, 페이지 캐시)는 단 두 개의 적을 상대한다 — 디스크는 느리고, 전원은 아무 때나 나간다. I/O 모델의 진화(blocking → non-blocking → multiplexing → completion)는 '느린 장치를 기다리는 동안 스레드를 어디에 세워둘 것인가'라는 한 가지 질문에 대한 답이 정교해진 역사다. 준비 통지(readiness) 모델인 epoll은 '이제 읽어도 안 막힌다'까지만 알려주고 읽기는 앱이 하지만, 완료 통지(completion) 모델인 IOCP는 버퍼를 먼저 제출하면 커널이 '다 읽어놨다'를 큐로 알려준다. 게임 서버가 IOCP를 쓰는 이유는 스레드 수를 코어 수 규모로 고정한 채 수천 세션의 완료 이벤트를 커널 큐 하나로 받을 수 있기 때문이다."**

## ② 기본 개념

### 파일이라는 추상화 — 바이트 스트림 + 메타데이터

디스크가 하드웨어로서 제공하는 것은 "번호 붙은 고정 크기 블록(섹터)을 읽고 써라"가 전부다. 이름도, 크기도, 소유자도, "어디까지가 한 파일인지"도 없다. OS의 파일 시스템 계층이 그 위에 두 가지를 발명한다.

- **바이트 스트림(byte stream)**: 파일은 0번부터 시작하는 바이트 열이고, 오프셋으로 읽고 쓴다. 블록 경계는 사용자에게 보이지 않는다.
- **메타데이터(metadata)**: 크기, 소유자, 타임스탬프, 그리고 가장 중요한 "이 파일의 N번째 블록이 디스크 어디에 있는가"라는 매핑.

UNIX가 여기서 한 발 더 나간 것이 "모든 것은 파일이다"라는 결정이다. 디바이스, 파이프, 소켓까지 같은 `read`/`write` 인터페이스로 통일했다. 이유는 조합성이다 — 인터페이스가 하나면 프로그램들을 파이프로 이어붙일 수 있고, I/O 다중화 같은 메커니즘을 파일·소켓·파이프에 공통으로 적용할 수 있다. 뒤에 나올 I/O 모델 논의가 소켓과 파일을 넘나드는 것도 이 통일 덕분이다.

주의할 점 하나: 파일 내용에 구조를 부여하는 것은 OS가 아니라 **포맷의 소유자**다. OS 입장에서 `.txt`도 `.dat`도 `.wmesh`도 그냥 바이트 열이다. 매직 넘버, 버전, 레코드 배치는 전부 애플리케이션의 계약이다(→ ④의 Stage1.dat).

### 파일 디스크립터 — 왜 테이블이 3단인가

`open()`이 돌려주는 파일 디스크립터(file descriptor, fd)는 그냥 작은 정수다. 이 정수가 실제 파일에 닿기까지 테이블 세 개를 거친다.

```
  프로세스별                시스템 전역                     파일 실체
  fd 테이블            오픈 파일 테이블(OFT)              inode 테이블
+------------+      +----------------------+          +--------------+
| 0 (stdin)  |      | offset=0    mode=r   |          | inode #123   |
| 1 (stdout) |      +----------------------+     +--> | size, perm,  |
| 3  --------+----> | offset=4096 mode=rw  | ----+    | 블록 매핑     |
| 4  --------+--+   +----------------------+     |    +--------------+
+------------+  +-> | offset=0    mode=r   | ----+
                    +----------------------+
   (프로세스마다)       (open() 1회당 1개,          (디스크 위 파일당 1개,
                       오프셋이 여기 산다)           하드링크가 여럿 가리킴)
```

3단인 이유는 각 단계가 서로 다른 "공유 질문"에 답하기 때문이다.

| 계층 | 공유 단위 | 답하는 질문 |
|---|---|---|
| fd 테이블 | 프로세스 | "이 프로세스의 3번은 무엇인가" — 리다이렉션(`dup2`)이 여기서 성립 |
| 오픈 파일 테이블 | `open()` 호출 | "오프셋과 모드를 누가 공유하는가" — `fork()`한 부모/자식은 같은 항목을 가리켜 오프셋을 공유 |
| inode | 디스크 위 파일 | "파일의 실체는 무엇인가" — 같은 파일을 두 번 `open()`하면 OFT 항목은 둘, inode는 하나 |

오프셋을 fd에 두지 않고 중간 테이블에 둔 것이 핵심 설계다. 셸이 `프로그램 > log.txt`로 자식을 띄울 때 부모가 만든 오프셋 상태를 자식이 그대로 이어받아야 하고, 두 프로세스가 같은 로그에 이어 쓸 때 서로의 위치를 덮어쓰지 않아야 한다. "무엇을 공유할지"의 단위를 쪼개려면 테이블도 쪼개져야 한다.

Windows는 fd 대신 핸들(HANDLE)을 쓴다. 프로세스별 핸들 테이블이 커널 오브젝트(파일, 소켓, 이벤트, IOCP 포트까지)를 참조 카운트로 가리키는 구조로, "프로세스 지역 색인 → 시스템 전역 객체"라는 골격은 같다. 소켓과 파일과 완료 포트가 모두 같은 핸들 체계라는 점이 뒤의 IOCP 설계를 자연스럽게 만든다.

### 디스크 위 자료구조 — "N번째 블록은 어디에 있는가"

파일 시스템 설계의 중심 문제는 매핑이다: **파일 오프셋 → 디스크 블록 번호**. 이 매핑을 어떤 자료구조로 저장하느냐가 파일 시스템의 계보를 가른다.

| 방식 | 자료구조 | 랜덤 접근 | 강점 | 약점 |
|---|---|---|---|---|
| FAT | 클러스터 체인 (연결 리스트를 전역 테이블로 뽑아냄) | 체인을 앞에서부터 추적 | 단순함, 구현 쉬움 | 큰 파일 랜덤 접근 불리, 테이블 손상 = 전체 손상 |
| ext 계열 | inode + 직접/간접 블록, ext4는 익스텐트 트리 | 트리 하강 | 작은 파일 즉시 접근, 큰 파일도 완만 | 간접 블록 방식은 큰 연속 파일에 메타데이터 낭비 |
| NTFS | MFT 레코드 + 런 리스트 | 런 리스트 탐색 | 작은 파일은 레코드 안에 상주, 모든 것이 파일 | MFT 자체의 단편화 관리 필요 |

**FAT(File Allocation Table)**: 디스크 앞쪽에 클러스터 개수만큼의 항목을 가진 테이블을 두고, 각 항목에 "이 파일의 다음 클러스터 번호"를 적는다. 즉 연결 리스트인데, next 포인터를 데이터 옆이 아니라 별도 테이블에 모아둔 형태다. 디렉터리 엔트리는 시작 클러스터만 가진다. 파일 중간으로 점프하려면 체인을 처음부터 따라가야 한다(테이블을 메모리에 올려두면 견딜 만하지만 원리적으로 O(체인 길이)). 단순함의 대가로 확장성을 포기한 설계고, 그래서 지금도 USB 메모리·SD카드 같은 "단순함과 호환성이 최우선"인 곳에 산다.

**ext의 inode**: inode 안에 블록 포인터를 직접 두되, 작은 파일과 큰 파일의 비용을 차등화한다. 직접 블록 포인터 여러 개(ext2 기준 12개) 뒤에 단일 간접(포인터 블록을 가리킴), 이중 간접, 삼중 간접 포인터를 둔다. 결과적으로 불균형 트리가 되는데, 이 불균형이 의도다 — 대부분의 파일은 작으므로 작은 파일은 간접 참조 0회로 닿고, 드문 큰 파일만 트리를 내려간다. ext4는 여기서 한 번 더 진화해 **익스텐트(extent)** — "(시작 블록, 길이)"로 연속 구간을 통째로 기술 — 를 쓴다. 1GB 연속 파일을 간접 블록 방식으로 기술하면 블록 포인터 수십만 개가 필요하지만 익스텐트로는 몇 개면 된다.

**NTFS의 MFT(Master File Table)**: 모든 파일이 MFT라는 테이블의 고정 크기 레코드(통상 1KB) 하나씩을 갖고, 파일의 모든 것(이름, 타임스탬프, 데이터까지)을 "속성(attribute)"으로 그 레코드에 담는다. 작은 파일은 데이터 속성이 레코드 안에 그대로 들어가(resident) 디스크 추가 접근 없이 읽힌다. 큰 파일은 데이터가 밖으로 나가고 레코드에는 런 리스트(run list) — "(시작 클러스터, 길이)" 연속 구간 목록 — 가 남는다. 익스텐트와 같은 아이디어다.

세 계보가 서로 다른 출발점에서 왔는데 결론이 수렴한다는 점이 중요하다. **현대 파일 시스템은 전부 "연속 범위(extent/run)" 기술로 수렴했다.** 이유는 뒤의 디스크 특성 절에서 확인된다 — 순차 I/O가 이기기 때문에, 파일 시스템은 애초에 블록을 연속으로 할당하려 애쓰고, 연속 할당에는 연속 구간 표기가 가장 싸다.

### 디렉터리 = 이름 → inode 매핑

디렉터리는 특별한 것이 아니라 **내용이 (이름, inode 번호) 엔트리 목록인 파일**이다. 경로 해석 `/a/b/c.txt`는 루트 inode에서 시작해 "a를 찾아 inode 획득 → 그 디렉터리에서 b를 찾아 → c.txt를 찾아"를 반복하는 조회 연쇄다.

이 구조에서 두 종류의 링크가 자연스럽게 나온다.

```
하드링크: 엔트리 두 개가 같은 inode를 가리킴          심볼릭 링크: 경로 문자열이 담긴 별도 파일
 dir1/foo ──┐                                       dir1/foo ──> inode A (내용: "/dir2/bar")
            ├──> inode #123 (링크 카운트=2)                          │ (경로를 다시 해석)
 dir2/bar ──┘                                       dir2/bar ──> inode B (실제 데이터)
```

- **하드링크(hard link)**: 같은 inode를 가리키는 디렉터리 엔트리가 여러 개. "파일 삭제"의 실체가 여기서 드러난다 — `unlink`는 엔트리 하나를 지우고 링크 카운트를 줄일 뿐, 데이터 해제는 카운트가 0이 되고 여는 프로세스도 없을 때다.
- **심볼릭 링크(symbolic link)**: 경로 문자열을 담은 파일. 대상이 지워지면 깨진 링크가 되고, 파일 시스템 경계를 넘을 수 있다. 간접 참조를 이름 수준에서 하느냐(심볼릭) 실체 수준에서 하느냐(하드)의 차이다.

엔트리를 선형 목록으로 두면 파일 만 개짜리 디렉터리에서 조회가 O(N)이 되므로, 현대 파일 시스템은 큰 디렉터리를 트리/해시로 색인한다(ext4의 HTree, NTFS의 B-트리 인덱스). "디렉터리도 자료구조 문제"라는 것.

### 크래시 일관성과 저널링 — 전원은 아무 때나 나간다

파일 하나를 새로 만드는 단순한 작업도 디스크에는 여러 곳의 갱신이다: 블록 할당 비트맵, 새 inode, 데이터 블록, 부모 디렉터리 엔트리. 문제는 이 갱신들이 **원자적이지 않다**는 것이다. 절반만 쓴 시점에 전원이 나가면 "할당됐다고 표시됐지만 아무도 안 가리키는 블록", "디렉터리에는 있지만 inode가 쓰레기인 파일" 같은 불변식 붕괴가 남는다.

고전 해법은 fsck — 부팅 때 디스크 전체를 스캔해 모순을 찾아 고치는 것이었다. 디스크가 커질수록 스캔이 수십 분~수 시간으로 늘어나며 파산했다. 검사 비용이 디스크 크기에 비례하는 설계는 확장 불가능하다.

저널링(journaling)의 아이디어는 방향을 뒤집는다. **바뀔 내용을 최종 위치에 쓰기 전에, "무엇을 바꿀 것인가"를 저널(로그) 영역에 먼저 순차 기록하고 커밋 표시를 남긴다.** 그 다음에야 실제 위치에 반영한다. 크래시가 나면:

- 커밋 표시가 있는 트랜잭션 → 로그를 재생(redo)해 마저 반영. 
- 커밋 표시가 없는 트랜잭션 → 통째로 버림. 시작 전 상태 유지.

어느 쪽이든 "절반 상태"가 사라진다. 복구 비용은 디스크 크기가 아니라 **로그 길이**에 비례한다. 이것이 미리 쓰기 로그(write-ahead log, WAL)이고, 데이터베이스의 WAL·게임 서버의 이벤트 소싱과 정확히 같은 원리다 — "상태를 직접 바꾸기 전에 의도를 순차 로그로 먼저 박아라. 순차 쓰기는 싸고, 로그는 재생 가능하다."

트레이드오프도 로그답다. 모든 데이터를 저널에도 쓰면(데이터 저널링) 모든 쓰기가 두 번이 된다. 그래서 ext4의 기본은 **메타데이터만 저널링**하고 데이터 블록은 메타데이터 커밋 전에 먼저 쓰는 절충(ordered 모드)이다. 파일 시스템 구조의 일관성은 보장하되, 파일 "내용"의 원자성까지는 보장하지 않는다 — 이 구멍이 뒤의 fsync 논의로 이어진다.

### VFS — 정책/메커니즘 분리의 실례

ext4, NTFS, FAT32, 네트워크 파일시스템(NFS), 가상 파일(procfs)이 전부 다른 자료구조인데 애플리케이션은 같은 `open`/`read`/`write`를 쓴다. 이것이 가능한 이유가 가상 파일 시스템(Virtual File System, VFS) 계층이다.

```
        사용자: open("/mnt/usb/a.txt")
                    │
        ┌───────────▼───────────┐
        │   VFS (공통 오브젝트:    │   ← "파일이란 무엇인가"의 정의 (정책 인터페이스)
        │  superblock, inode,   │
        │   dentry, file)       │
        └──┬────────┬────────┬──┘
           │        │        │      ← 함수 포인터 테이블(연산 집합)로 디스패치
         ext4     FAT32     NFS     ← "그걸 어떻게 하는가" (메커니즘 구현)
```

VFS는 "inode를 읽어라, 이름을 조회하라" 같은 연산의 **인터페이스만 정의**하고, 각 파일 시스템이 자기 자료구조로 구현해 함수 포인터 테이블로 등록한다. C로 구현한 런타임 다형성 — C++로 치면 순수 가상 인터페이스다. 마운트는 "이 경로 아래부터는 저 구현체" 라는 디스패치 규칙 등록이고, "모든 것은 파일" 철학은 VFS 덕에 소켓·파이프·디바이스까지 확장된다.

이것이 교과서적 **정책/메커니즘 분리(policy/mechanism separation)**다: 상위 계층은 "무엇을"만 정의하고 "어떻게"는 교체 가능한 하위 구현에 위임한다. Winters에서 RHI가 `IRHIDevice` 인터페이스 뒤에 DX11 구현을 숨긴 것과 같은 층 구조이며, 면접에서 "OS 설계 원칙을 자기 코드로 설명하라"는 질문의 좋은 다리다.

### 페이지 캐시와 write-back — write() 성공은 디스크 보장이 아니다

`write()`가 성공을 반환하는 시점에 데이터는 대개 디스크가 아니라 **페이지 캐시(page cache)** — 커널이 RAM에 유지하는 디스크 블록 캐시 — 에 있다. 커널은 해당 페이지를 더티(dirty)로 표시만 하고 반환하며, 실제 디스크 기록은 백그라운드 플러시가 나중에 한다. 이것이 지연 쓰기(write-back) 전략이다.

왜 이렇게 하는가. (1) 디스크는 RAM보다 자릿수로 느리므로 호출자를 기다리게 할 이유가 없고, (2) 같은 블록을 여러 번 고치면 마지막 한 번만 쓰면 되고(쓰기 흡수), (3) 모아서 쓰면 순차·인접 배치로 스케줄링할 수 있다. 성능상 압도적으로 유리한 대신, **"성공 반환"과 "영속화" 사이에 시간 창이 생긴다.**

데이터가 통과하는 층을 전부 세면 이렇다:

```
 fwrite() ─> [1] stdio 버퍼 (사용자 공간, FILE*)     ← fflush()가 여기를 비움
    write() ─> [2] 커널 페이지 캐시 (dirty)          ← fsync()/FlushFileBuffers()가 여기를 밀어냄
       플러시 ─> [3] 디스크 컨트롤러 캐시              ← 배리어/플러시 명령이 필요할 수 있음
          기록 ─> [4] 플래터/NAND (진짜 영속)
```

**데이터 유실 시나리오**: 유저가 세이브 버튼을 누르고, `fwrite` + `fclose`가 성공하고, "저장 완료" UI까지 띄운 뒤 3초 후 정전. `fclose`는 [1]→[2]까지만 보장하므로 더티 페이지가 아직 디스크에 안 갔다면 세이브는 사라진다. 더 나쁜 경우는 여러 블록 중 일부만 내려간 상태 — 파일 크기는 갱신됐는데 내용이 옛 것인, 반쯤 섞인 파일이 남는다(메타데이터 저널링은 파일시스템 구조만 지켜주지 파일 내용은 안 지켜준다).

그래서 "성공했다고 말하기 전에 영속화가 필요한" 소프트웨어(DB 커밋, 유저 세이브)는 `fsync()`(Windows `FlushFileBuffers`)로 [2]→[3/4]를 강제한 뒤에야 성공을 보고한다. fsync는 디스크 왕복을 기다리므로 비싸다 — DB가 커밋 지연의 대부분을 fsync에 쓰는 이유이고, 아무 데나 fsync를 뿌리면 안 되는 이유다.

덮어쓰기 자체를 안전하게 만드는 관용구도 알아둘 가치가 있다: **임시 파일에 전체를 쓰고 → fsync → `rename()`으로 원본 위치에 교체**. rename은 디렉터리 엔트리 교체라 (같은 파일시스템 안에서) 원자적이므로, 크래시가 나도 "옛 파일 전체" 아니면 "새 파일 전체"만 남는다. 절반 파일이 불가능해진다.

### I/O 모델 5종 진화 — 스레드를 어디에 세워둘 것인가

디스크·네트워크는 CPU보다 자릿수로 느리다. I/O 모델의 역사는 "기다리는 동안 스레드로 무엇을 할 것인가"에 대한 답의 진화다.

| 세대 | 모델 | 기다림의 형태 | 본질적 한계 |
|---|---|---|---|
| 1 | blocking | 스레드가 커널에서 잠듦 | 연결 1개 = 스레드 1개 |
| 2 | non-blocking + polling | 스레드가 루프 돌며 물어봄 | CPU를 태워서 기다림 |
| 3 | I/O 다중화 (select/poll/epoll) | 스레드 1개가 fd N개의 "준비"를 대기 | 준비 통지까지만; 읽기는 여전히 앱 몫 |
| 4 | 완료 통지 (IOCP, POSIX AIO) | 작업을 제출하고 "완료"를 큐로 수령 | 프로그래밍 모델 복잡(버퍼 수명) |
| 5 | io_uring | 제출/완료 큐를 커널과 공유 링버퍼로 | (시스템콜 비용 자체를 상환) |

**1세대 — blocking**: `read()`를 부르면 데이터가 올 때까지 스레드가 커널에서 잔다. 코드는 순차적이라 가장 읽기 쉽다. 문제는 서버다 — 연결 1만 개면 스레드 1만 개가 필요하고, 스레드마다 스택(기본 수 MB 예약)과 스케줄링 비용이 붙는다. 1999년경 "C10K 문제"라는 이름으로 정식화된 한계다.

**2세대 — non-blocking polling**: 소켓을 논블로킹으로 바꾸면 `read()`가 데이터 없을 때 기다리는 대신 즉시 "지금은 없음(EWOULDBLOCK)"을 반환한다. 스레드 하나가 소켓 N개를 돌며 계속 물어볼 수 있다. 대신 준비된 소켓이 하나도 없어도 루프는 돈다 — **기다림을 CPU 소모로 바꿨을 뿐**이다. "물어보는 주체"가 앱인 한 이 낭비는 구조적이다.

**3세대 — I/O 다중화(multiplexing)**: 물어보는 방향을 뒤집는다. "이 fd 집합 중 하나라도 준비되면 깨워달라"고 커널에 위임하고 잠든다.

- `select`/`poll`: 호출할 때마다 fd 집합 전체를 커널에 건네고, 커널은 전체를 스캔해서 준비된 것을 표시해 돌려준다. 매 호출 O(전체 fd 수)이고, select는 집합 크기 상한(FD_SETSIZE)까지 있다. 연결 1만 개 중 이번에 준비된 게 3개여도 1만 개를 훑는다.
- `epoll`(Linux): 관심 fd 집합을 **커널 안에 상주 등록**(`epoll_ctl`)해 두고, `epoll_wait`는 준비된 fd 목록만 돌려받는다. 비용이 O(전체)에서 O(준비된 것)으로 떨어진다. 자료구조를 호출 인자에서 커널 상주 상태로 옮긴 것이 혁신의 전부이며, 그거면 충분했다.

3세대의 본질은 **준비 통지(readiness) 모델**이라는 것이다. 커널이 알려주는 것은 "이제 read를 불러도 안 막힌다"까지다. 데이터를 커널 버퍼에서 내 버퍼로 복사하는 read 호출은 여전히 앱이 직접 한다.

**4세대 — 완료 통지(completion) 모델**: 순서를 뒤집는다. **버퍼를 먼저 제출한다.** "이 버퍼로 읽어라"라는 작업 요청을 커널에 걸어두면, 커널이 데이터 도착 시 복사까지 끝내고 "완료됐다, N바이트다"를 완료 큐에 넣는다. 앱 스레드는 완료 큐에서 결과를 꺼내 소비만 한다.

```
 readiness (epoll):                       completion (IOCP):
   앱: "준비되면 알려줘" ──> 커널             앱: "이 버퍼에 읽어놔" ──> 커널 (작업 제출)
   커널: "3번 소켓 준비됨"                    커널: (도착 → 버퍼로 복사까지 수행)
   앱: read(3, buf) ← 복사는 지금, 앱이       커널: 완료 큐에 push "3번, 128바이트"
   앱: 데이터 처리                            앱: 큐에서 pop → 데이터 처리 (복사 이미 끝남)
```

Windows IOCP(I/O Completion Port)가 이 모델의 대표다. 구조는 세 조각이다:

1. **완료 포트** = 커널이 관리하는 완료 이벤트 큐. 소켓/파일 핸들을 포트에 연결(associate)해 두면 그 핸들의 비동기 I/O 완료가 전부 이 큐로 모인다.
2. **오버랩드(OVERLAPPED) I/O 제출**: `WSARecv`/`WSASend`/`ReadFile`에 OVERLAPPED 구조체를 붙여 호출하면 즉시 반환되고(진행 중이면 `WSA_IO_PENDING`), 작업은 커널에서 진행된다.
3. **워커 스레드 풀**: 스레드 몇 개가 `GetQueuedCompletionStatus`(GQCS)로 큐를 대기한다. 완료가 들어오면 커널이 대기 스레드 중 하나를 깨운다.

게임 서버가 IOCP를 쓰는 이유를 정리하면:

- **스레드 수가 연결 수가 아니라 O(코어 수)**: 세션 5천 개를 워커 4~8개가 처리한다. 스택 메모리와 컨텍스트 스위치가 연결 수와 무관해진다.
- **커널 큐가 곧 작업 분배기**: 완료 이벤트가 큐 하나로 모이고 커널이 대기 워커에게 나눠준다. 앱이 직접 짜야 할 디스패처/락 계층이 줄어든다.
- **동시성 제어가 내장**: 포트 생성 시 "동시에 실행 상태로 둘 스레드 수"를 지정하면, 커널이 실행 중 워커 수를 그 값 근처로 유지한다(하나가 블록되면 다음 대기자를 깨워 보충). 코어 수만큼만 돌리라는 정책을 커널이 집행한다.
- **Windows에는 epoll급 readiness 확장이 표준으로 없었다**: select는 상한이 작고, 이벤트 기반 대안들은 확장성이 부족했다. Windows에서 고성능 서버의 정답이 IOCP로 수렴한 것은 모델의 우수성과 플랫폼 사정이 함께 작용한 결과다.

**5세대 — io_uring(Linux)**: IOCP와 같은 완료 모델을 리눅스에 가져오되 한 발 더 간다. 제출 큐(SQ)와 완료 큐(CQ)를 **커널과 사용자 공간이 공유하는 링버퍼**로 만들어, 작업 제출과 완료 수거에서 시스템콜 호출 횟수 자체를 상환(amortize)한다 — 여러 요청을 링에 써놓고 한 번에 알리거나, 폴링 모드에서는 시스템콜 없이도 진행된다. "시스템콜 경계 넘기 자체가 비용"이라는 최신 인식(멜트다운 이후 완화 비용 증가)의 산물이다. 또한 readiness 모델이 처리하지 못하던 일반 파일의 진짜 비동기 읽기를 처리한다는 점도 중요하다(→ ③).

### 무복사(zero-copy) — 사용자 공간에 볼일 없는 데이터는 올리지 마라

파일을 소켓으로 보내는 고전 경로를 세어 보면:

```
 read(file, buf):   디스크 ─DMA→ 페이지 캐시 ─복사→ 사용자 버퍼        (시스템콜 1)
 send(sock, buf):   사용자 버퍼 ─복사→ 소켓 버퍼 ─DMA→ NIC            (시스템콜 2)
```

데이터가 사용자 버퍼에 들렀다 가지만, 앱은 그 내용을 건드리지도 않았다. 복사 두 번과 시스템콜 두 번이 순수 낭비다. `sendfile()`(Windows `TransmitFile`)은 이 왕복을 없앤다 — 커널이 페이지 캐시에서 소켓 쪽으로 내부 전달하고, NIC가 분산 수집(scatter-gather) DMA를 지원하면 커널 안에서조차 실제 바이트 복사 없이 디스크립터만 넘긴다. 정적 파일 서버·스트리밍이 이것으로 산다.

원리를 일반화하면 "**데이터가 사용자 공간에서 변형될 일이 없으면 사용자 공간으로 올려보내지 마라**"다. 게임 쪽에서는 패킷 직렬화 버퍼를 제출 버퍼로 그대로 쓰는 설계(중간 조립 버퍼 제거), GPU 업로드 시 스테이징 복사 최소화 같은 형태로 같은 원리가 반복된다.

### 디스크 특성 — 순차 I/O가 여전히 이기는 이유

**HDD**: 헤드를 트랙까지 물리적으로 옮기는 탐색(seek)에 수 ms, 플래터가 원하는 섹터까지 돌아오는 회전 지연에 또 수 ms. 랜덤 4KB 읽기는 매번 이 비용을 내므로 초당 수백 회 수준(수 MB/s)에 그치지만, 순차 읽기는 탐색 한 번 후 연속으로 읽으므로 초당 100MB대가 나온다. **랜덤과 순차의 격차가 수십~수백 배**인 장치다. 이 격차가 위에서 본 설계들의 공통 원인이다 — 저널이 순차 로그인 이유, 파일 시스템이 익스텐트(연속 할당)로 수렴한 이유, 엘리베이터 스케줄러가 요청을 디스크 주소순으로 재정렬하는 이유.

**SSD**: 이동 부품이 없어 탐색이 사라지고, 내부적으로 여러 NAND 채널·다이가 병렬로 동작해 랜덤 접근도 자릿수로 빨라졌다. 그럼에도 순차 I/O가 여전히 이긴다. 이유가 층층이 있다:

- SSD 내부의 FTL(Flash Translation Layer)이 논리 주소 → 물리 위치 매핑 테이블을 유지하는데, 순차 접근은 매핑·프리페치·내부 병렬화가 모두 규칙적으로 걸린다.
- NAND는 덮어쓰기가 불가능해 "지우고 다시 쓰기"를 큰 블록 단위로 해야 하는데, 랜덤 쓰기는 유효 데이터 이사(가비지 컬렉션)를 유발해 실제 기록량이 요청량보다 커진다 — 쓰기 증폭(write amplification). 순차 쓰기는 블록을 통째로 채우므로 증폭이 적다.
- OS·컨트롤러의 요청 병합과 프리페치가 순차 패턴에서만 작동한다.

그래서 "SSD니까 접근 패턴은 상관없다"는 오답이다. 격차가 HDD보다 줄었을 뿐 순위는 안 바뀌었고, 대용량 처리 시스템(LSM-tree 기반 DB, 게임의 pak/청크 패키징, 에셋 스트리밍 순서 최적화)은 여전히 순차 지향으로 설계된다.

## ③ 심화 — 꼬리질문 대비

**"fd와 FILE*는 뭐가 다른가?"**
fd는 커널 자원의 색인(정수)이고, `FILE*`는 그 위에 C 런타임이 얹은 **사용자 공간 버퍼 계층**이다. `fwrite`는 stdio 버퍼에 쌓다가 버퍼가 차거나 `fflush`/`fclose` 때 `write` 시스템콜로 내려보낸다. 시스템콜 횟수를 줄이는 최적화지만, 층이 하나 늘어난 만큼 "어디까지 갔는가"의 단계도 하나 는다: fflush는 stdio→커널, fsync는 커널→디스크. 로그가 크래시 때 잘려 있는 흔한 이유가 stdio 버퍼에 있던 채로 죽어서다.

**"fork() 후 부모와 자식이 파일 오프셋을 공유하는 이유는?"**
fork는 fd 테이블을 복사하지만 fd가 가리키는 **오픈 파일 테이블 항목은 공유**된다. 오프셋은 그 항목에 있으므로 둘이 같은 위치를 진행시킨다. 이는 버그가 아니라 설계다 — 셸 리다이렉션에서 부모가 열어준 로그 파일에 자식들이 이어 쓰려면 오프셋이 공유돼야 한다. 반대로 같은 파일을 각자 `open`하면 항목이 분리돼 오프셋도 독립이다. "무엇이 어느 테이블에 사는가"를 아는지 확인하는 단골 질문.

**"select와 epoll의 차이를 자료구조 관점에서 설명하라."**
select는 관심 집합을 **매 호출 인자로** 전달하고 커널이 전체를 스캔한다 — 상태가 호출에 있으므로 O(전체). epoll은 관심 집합을 `epoll_ctl`로 **커널 상주 자료구조에 등록**해 두고, 이벤트 발생 시 커널이 준비 목록에 넣어두면 `epoll_wait`는 그 목록만 가져온다 — 상태가 커널에 있으므로 O(준비된 것). 알고리즘 개선이 아니라 상태의 위치를 옮긴 것이 본질이다.

**"epoll의 레벨 트리거와 엣지 트리거 차이는?"**
레벨 트리거(level-triggered)는 "읽을 게 남아 있는 한" 계속 알려주고, 엣지 트리거(edge-triggered)는 "없다가 생기는 순간"만 알려준다. ET는 통지 횟수가 줄어 효율적이지만, 통지 한 번에 버퍼가 빌 때까지(EWOULDBLOCK까지) 다 읽어야 한다는 계약이 생긴다 — 덜 읽고 다음 통지를 기다리면 영원히 안 온다. 성능과 프로그래밍 계약의 엄격함을 맞바꾸는 선택.

**"epoll이 있는데 왜 io_uring이 필요했나? 일반 파일에 epoll을 쓰면?"**
readiness 모델의 맹점이다. 일반 파일(regular file)은 커널 입장에서 "항상 준비됨"이다 — 소켓처럼 '데이터가 아직 안 온' 상태가 없고, read하면 (페이지 캐시 미스 시) 그냥 디스크를 기다리며 **블록**한다. 그래서 select/poll은 일반 파일을 항상 ready로 보고하고, epoll은 아예 등록부터 거부한다(`epoll_ctl`이 EPERM 반환). 즉 readiness 모델은 디스크 I/O를 비동기화하지 못한다. 완료 모델은 "이 읽기를 수행하고 끝나면 알려달라"이므로 파일에도 적용된다. io_uring이 리눅스에서 파일 비동기 I/O의 실질적 첫 정답이 된 이유고, IOCP는 처음부터 소켓과 파일을 같은 포트로 다뤘다.

**"readiness와 completion 중 뭐가 더 좋은가?"**
장치와 요구에 따라 다르다. completion은 복사 완료까지 커널이 해주고 디스크에도 적용되지만, **제출한 버퍼의 수명을 완료까지 앱이 보장**해야 한다(댕글링 버퍼 = 커널이 해제된 메모리에 쓰는 사고). readiness는 버퍼를 read 호출 순간에만 빌려주므로 수명 관리가 단순하다. 또 readiness는 "준비됐지만 안 읽는" 백프레셔 제어가 자연스럽다. 리눅스 생태계가 오래 epoll로 충분했던 이유와, 그럼에도 io_uring으로 이동 중인 이유를 함께 말할 수 있으면 좋다.

**"IOCP의 NumberOfConcurrentThreads는 워커 스레드 수와 뭐가 다른가?"**
워커 수는 GQCS를 대기하는 스레드의 총수, concurrency 값은 커널이 **동시에 '실행 중' 상태로 허용하는 수**다. 워커 하나가 완료 처리 중 블록되면(파일 접근 등) 커널이 대기 중인 다른 워커를 깨워 실행 수를 보충한다. 그래서 관례가 "concurrency = 코어 수, 워커 = 그보다 약간 많이"다 — 블록에 대비한 여유 인력. 스레드 풀 정책의 일부를 커널이 집행해 주는, IOCP만의 독특한 설계다.

**"fsync는 왜 비싸고, 어디에만 써야 하나?"**
write-back의 이득(쓰기 흡수, 배치, 즉시 반환)을 전부 포기하고 디스크 왕복(경우에 따라 디스크 캐시 플러시까지)을 기다리는 호출이라서다. 회전 디스크에서는 ms 단위, SSD에서도 캐시 플러시가 걸리면 상당하다. 규칙은 "**성공을 외부에 약속하는 지점**에만" — DB 커밋 응답 직전, 유저 세이브 완료 UI 직전. 프레임마다·로그마다 fsync하는 코드는 I/O를 이해 못 했다는 신호다.

**"파일 시스템이 저널링을 하는데 DB는 왜 자체 WAL을 또 쓰나?"**
보호 대상이 다르다. FS 저널은 **파일 시스템 자신의 불변식**(비트맵·inode·디렉터리의 정합)을 지킬 뿐, "계좌 A 차감과 B 증가가 함께 반영된다" 같은 **애플리케이션 트랜잭션**은 모른다. DB는 자기 의미론의 원자성/내구성을 자기 WAL로 지켜야 하고, 오히려 FS 캐시와의 이중 버퍼링을 피하려 O_DIRECT로 페이지 캐시를 우회하기도 한다. "각 계층은 자기 계층의 불변식만 지킨다"는 층 원리의 좋은 예.

**"mmap으로 파일을 읽는 것과 read의 차이는?"**
mmap은 파일의 페이지 캐시를 내 주소 공간에 **직접 매핑**한다. read처럼 캐시→사용자 버퍼 복사가 없고, 접근은 페이지 폴트로 온디맨드 로드된다. 대신 폴트 비용이 접근 시점에 흩어지고, I/O 에러가 반환값이 아니라 시그널(SIGBUS)로 오며, 매핑 갱신의 디스크 반영 시점 제어가 어렵다. "복사 절약 vs 제어력"의 트레이드오프로, 큰 읽기 전용 에셋(팩 파일)에 유리하고 신중한 에러 처리가 필요한 쓰기 경로에는 불리하다.

**"논블로킹 소켓에서 write가 일부만 성공하는 경우는 왜 생기고 어떻게 다루나?"**
소켓 송신 버퍼에 공간이 모자라면 커널은 들어가는 만큼만 받고 그 바이트 수를 반환한다. 앱은 남은 조각을 보관했다가 '쓰기 가능' 통지에 이어 보내야 한다 — 즉 readiness 모델에서는 **부분 쓰기 재개 상태 기계**가 앱 책임이다. completion 모델에서도 순서 보장을 위해 송신을 큐로 직렬화하는 같은 문제가 나타난다(→ ④의 Session 송신 큐). "소켓 코드에서 상태 기계를 짜봤는가"를 가르는 질문.

**"순차 읽기가 빠른 데는 디스크만이 아니라 OS도 기여한다는데, 어떻게?"**
미리 읽기(read-ahead)다. 커널이 접근 패턴이 순차임을 감지하면 아직 요청되지 않은 다음 블록들을 미리 페이지 캐시에 올려둔다. 그래서 순차 read의 대부분은 실제로는 캐시 히트이고, 디스크 대기가 앱의 read 호출과 겹쳐 숨는다. 랜덤 접근은 다음 위치를 예측할 수 없어 이 이득이 사라진다 — "순차가 이긴다"는 디스크 물리 + FS 연속 할당 + OS 프리페치의 3중 합작이다.

**"sendfile 같은 zero-copy에도 한계가 있나?"**
데이터를 사용자 공간에서 변형해야 하는 순간 무너진다. 압축, 암호화(TLS), 프로토콜 프레이밍이 끼면 결국 사용자 버퍼로 올려야 하므로, zero-copy는 "저장된 그대로 내보내는" 경로(정적 파일, 미리 직렬화된 패킷)에서만 성립한다. 커널 내 TLS(kTLS) 같은 확장이 이 한계를 커널 쪽에서 메우려는 시도라는 것까지 말하면 충분하다.

**"디렉터리에서 파일 삭제가 rm인데 왜 시스템콜 이름은 unlink인가?"**
삭제의 실체가 "데이터 제거"가 아니라 "이름→inode 연결(link) 하나 제거"이기 때문이다. 링크 카운트가 0이 되고 열고 있는 프로세스가 없어야 실체가 해제된다. 그래서 열려 있는 파일을 unlink해도 그 fd로는 계속 읽고 쓸 수 있다 — 임시 파일을 만들자마자 unlink해서 "프로세스 죽으면 자동 소멸"을 얻는 고전 기법이 여기서 나온다.

**"게임 클라이언트의 에셋 로딩에 이 챕터가 어떻게 적용되나?"**
(1) 작은 파일 수천 개는 open마다 경로 해석·메타데이터 접근이 붙으므로 팩 파일 하나로 묶어 **순차 대량 읽기**로 바꾼다. (2) 로딩 순서를 디스크 배치 순서와 맞춰 seek/랜덤성을 줄인다. (3) 로딩 스레드에서 blocking read를 쓰되 메인 스레드는 절대 I/O로 막지 않는다 — 프레임 예산 안에서 디스크 ms는 감당 불가. (4) 두 번째 실행이 빠른 것은 페이지 캐시 덕분이므로, 콜드 스타트 성능은 캐시를 비우고 측정해야 한다.

## ④ Winters에서의 적용

이 챕터의 세 축(완료 모델, 바이트 스트림 위의 자기 포맷, "비동기처럼 보이는 동기")이 전부 이 레포에 실물로 있다.

### (1) IOCP 서버 코어 — completion 모델의 교과서 구현

`Server/Private/Network/IOCPCore.cpp:44`가 완료 포트의 뿌리다:

```cpp
m_hIOCP = CreateIoCompletionPort(INVALID_HANDLE_VALUE, nullptr, 0, m_workerCount);
```

마지막 인자가 위 ③에서 말한 **동시 실행 스레드 수 제한**이고, 워커는 `IOCPCore.cpp:120~121`에서 그 수만큼만 생성된다(`Server/Private/main.cpp:63`에서 워커 4개로 기동 — 세션 수가 아니라 코어 규모다). 리슨 소켓도 accept부터 비동기다 — `IOCPCore.cpp:183~189`의 `AcceptEx`는 즉시 반환하고 `ERROR_IO_PENDING`이면 진행 중으로 처리한다. blocking `accept()` 전용 스레드조차 없다.

워커 루프 `IOCPCore.cpp:216`이 완료 큐 소비의 전형이다:

```cpp
const BOOL ok = GetQueuedCompletionStatus(m_hIOCP, &bytes, &key, &pOverlapped, INFINITE);
```

여기서 받은 `OVERLAPPED*`를 어떻게 "어느 작업의 완료인지"로 되돌리는지가 IOCP 설계의 관용구다. `Server/Public/Network/IOCPCore.h:22~31`의 `IOContext`는 **OVERLAPPED를 첫 멤버로 내장**한 컨텍스트 구조체이고, `IOCPCore.cpp:225`가 `CONTAINING_RECORD(pOverlapped, IOContext, overlapped)`로 멤버 주소에서 소유 구조체를 역산한다. "커널은 OVERLAPPED 포인터만 돌려준다 → 그 옆에 내 상태(op 종류, 세션 id, 버퍼)를 붙여두면 완료 시점에 전부 복원된다"는, 완료 모델에서 컨텍스트를 나르는 표준 기법이다.

종료 처리도 완료 큐답게 한다 — `IOCPCore.cpp:143`은 `PostQueuedCompletionStatus(..., nullptr)`로 **가짜 완료(널 센티널)를 워커 수만큼 큐에 주입**해 INFINITE 대기 중인 워커들을 깨워 내보낸다. 큐 기반 시스템의 종료는 큐로 알린다.

### (2) Session — 제출 버퍼 수명과 송신 직렬화라는 completion 모델의 비용

`Server/Private/Network/Session.cpp:62~65`의 수신 제출:

```cpp
const int result = WSARecv(m_socket, &m_recvContext.wsaBuf, 1,
    &bytes, &flags, &m_recvContext.overlapped, nullptr);
if (result == SOCKET_ERROR && WSAGetLastError() != WSA_IO_PENDING)
```

"에러이되 IO_PENDING이면 정상 진행"이라는 판정이 오버랩드 I/O의 계약이고, 제출된 `m_recvContext`(버퍼 8KB 포함)는 완료가 올 때까지 세션이 수명을 보장해야 한다 — ③에서 말한 completion 모델의 대가가 `AddPendingIo`/`CompletePendingIo` 카운팅으로 관리된다.

송신은 더 흥미롭다. `Session.cpp:100~115`는 보낼 패킷을 큐에 넣되 **WSASend를 동시에 하나만 pending으로 유지**한다(`m_bSendPending` 플래그, 완료 시 `Session.cpp:149~157`에서 다음 것을 이어 제출). 오버랩드 송신을 여러 개 겹쳐 걸면 각 버퍼 수명 관리가 복잡해지고 완료 순서 처리가 꼬이기 때문에, "큐 + 단일 비행(single in-flight)"으로 직렬화한 것이다. ③의 부분 쓰기/재개 상태 기계 질문에 대한 이 레포의 답이기도 하다.

### (3) Stage1.dat — "파일은 그냥 바이트"이므로 구조는 내가 계약한다

에디터가 저장하는 스테이지 데이터가 바이트 스트림 추상화 위에 자기 포맷을 세운 예다. `Client/Private/Map/MapDataIO.cpp:99~102`는 저장 시 매직 넘버와 버전을 담은 헤더를 먼저 쓴다:

```cpp
WMap::StageHeader header{};
header.magic   = WMap::STAGE_MAGIC;
header.version = WMap::STAGE_VERSION;
fwrite(&header, sizeof(WMap::StageHeader), 1, pFile);
```

로드 쪽 `MapDataIO.cpp:144`는 매직 불일치를, `:150`은 버전 범위 밖(호환 최소~현재)을 거부하고, `:161~178`은 **버전별 분기**로 구버전 파일에서 없는 섹션을 기본값으로 부트스트랩한다. OS는 이 파일이 스테이지 데이터인지 모른다 — 식별(매직), 진화(버전), 하위호환(분기) 전부 포맷 소유자의 계약이라는 것을 그대로 구현한 코드다.

이 챕터의 렌즈로 한 가지 더 짚을 수 있다: `MapDataIO.cpp:109`의 `fclose`는 stdio 버퍼를 커널 페이지 캐시까지만 내려보낸다. `FlushFileBuffers` 호출이 없으므로 저장 직후 정전이면 파일이 유실/반영 전일 수 있다 — 에디터 작업 데이터라 재저장 가능하니 fsync 비용을 안 내는 것이 합리적 선택이고, 만약 이것이 유저 세이브였다면 "임시 파일 + 플러시 + rename" 패턴이 필요했을 것이라고 트레이드오프를 말할 수 있으면 면접에서 강하다.

### (4) CHttpClient — "비동기처럼 보이는 동기"의 실사고

`std::async`가 돌려주는 `std::future`는 특별한 규칙이 있다: **그 소멸자가 태스크 완료까지 블록한다.** 반환값을 받지 않으면 임시 future가 그 자리에서 소멸하며 대기하므로, `std::async(std::launch::async, ...);` 한 줄은 문법상 비동기, 의미상 **완전한 동기 호출**이 된다. 이 함정을 실제로 밟은 기록이 `Client/Private/Network/Backend/CHttpClient.cpp:90~92`에 주석으로 박제돼 있다:

```cpp
//과거 버그: async(launch::async, ...)의 반환 future를 버리면 임시 future 소멸자가
//작업 완료까지 대기해 사실상 동기 호출이 됐다 (gotcha 2026-07-09 async lifetime).
//지금은 m_PendingRequests가 future를 소유하고, 소멸자가 전부 드레인한다.
```

수정 후 구조는 이 챕터 관점에서 정확하다. `CHttpClient.cpp:148~156`은 future를 `m_PendingRequests`(`Client/Public/Network/Backend/CHttpClient.h:71`)로 소유해 호출 지점을 진짜 비동기로 만들고, 소멸자 `CHttpClient.cpp:103~117`가 전부를 `wait()`로 드레인해 **블로킹을 파괴 시점 한 곳으로 한정**한다(요청 람다가 this의 뮤텍스·큐를 만지므로, 객체가 먼저 죽으면 안 된다는 수명 계약). 완료 결과는 `ProcessCallbacks`가 메인 스레드에서 큐로 소비한다 — 워커가 완료를 큐에 넣고 소비 스레드가 꺼내는, 작은 completion 모델을 사용자 공간에 재구성한 셈이다.

면접 서사로 정리하면: "I/O 모델의 핵심은 '기다림이 어디서 발생하는가'를 정확히 아는 것이다. 서버에서는 IOCP로 기다림 자체를 커널 큐로 옮겼고, 클라이언트 HTTP에서는 future 수명 규칙을 놓쳐 비동기가 동기로 퇴화한 버그를 겪고 나서, 대기가 허용되는 지점(소멸자)을 명시한 계약으로 고쳤다."

## ⑤ 면접 Q&A

**Q1. write()가 성공을 반환했다. 데이터는 어디에 있는가?**
- 핵심: 대개 커널 페이지 캐시의 더티 페이지에 있다. 디스크 도달은 백그라운드 write-back 이후이고, 그 사이 정전이면 유실된다. 영속이 필요한 지점에서만 fsync/FlushFileBuffers로 강제한다.
- 꼬리 대비: fwrite라면 층이 하나 더 있다(stdio 버퍼) — fflush는 stdio→커널, fsync는 커널→디스크. "성공 반환 ≠ 영속"을 층 4개(stdio/페이지 캐시/디스크 캐시/매체)로 그릴 수 있어야 한다.
- Winters: `MapDataIO.cpp:109`의 fclose는 페이지 캐시까지만 — 에디터 데이터라 fsync 비용을 안 내는 선택이고, 유저 세이브라면 임시 파일+플러시+rename이 필요했다고 트레이드오프로 답한다.

**Q2. select, epoll, IOCP를 한 줄씩으로 구분하라.**
- 핵심: select = 관심 집합을 매번 건네고 커널이 전체 스캔(O(N), 크기 상한). epoll = 관심 집합을 커널에 상주 등록하고 준비된 것만 수령(O(ready), readiness). IOCP = 버퍼를 먼저 제출하면 커널이 복사까지 끝내고 완료를 큐로 통지(completion).
- 꼬리 대비: readiness는 "읽어도 안 막힘"까지, completion은 "다 읽어놨음"까지 — read 호출과 복사가 누구 몫인지가 경계선이다.
- Winters: `IOCPCore.cpp:216`의 GQCS 워커 루프 + `Session.cpp:62`의 WSARecv 제출이 completion 모델의 양 끝.

**Q3. 게임 서버가 IOCP를 쓰는 이유는?**
- 핵심: (1) 스레드 수가 연결 수가 아니라 코어 수 규모로 고정된다. (2) 수천 세션의 완료 이벤트가 커널 큐 하나로 모여 워커 풀에 분배된다. (3) 동시 실행 스레드 수 제한을 커널이 집행한다(블록되면 다음 워커를 깨워 보충). (4) 소켓과 파일이 같은 포트로 다뤄진다.
- 꼬리 대비: "연결당 스레드는 왜 안 되나" → 스택 메모리와 컨텍스트 스위치가 연결 수에 비례(C10K). "epoll로는 안 되나" → Windows에 epoll급 표준 readiness 확장이 없었고, 완료 모델이 복사까지 커널에 맡긴다.
- Winters: `main.cpp:63`에서 워커 4개로 기동, `IOCPCore.cpp:44`의 concurrency 인자, `AcceptEx`로 accept까지 비동기(`IOCPCore.cpp:183~189`).

**Q4. 저널링 파일 시스템의 동작 원리를 설명하라.**
- 핵심: 여러 블록에 걸친 갱신을 최종 위치에 쓰기 전에 "무엇을 바꿀지"를 저널 영역에 순차 기록하고 커밋 표시를 남긴다. 크래시 시 커밋된 트랜잭션은 재생, 미커밋은 폐기 — 절반 상태가 사라지고 복구가 로그 길이에 비례한다.
- 꼬리 대비: "fsck와 뭐가 다른가" → 검사 비용이 디스크 크기 비례에서 로그 길이 비례로. "DB WAL과의 관계" → 같은 원리, 다른 계층 — FS 저널은 FS 불변식만, 앱 트랜잭션은 앱 WAL이 지킨다. "데이터도 저널링하나" → 기본은 메타데이터만(모든 쓰기 2배를 피하는 절충).
- Winters: 직접 대응 코드는 없지만, 서버의 스냅샷+이벤트 복제 구조가 "상태 직접 변경 대신 순차 로그 재생"이라는 같은 원리임을 연결해 말할 수 있다.

**Q5. FAT, ext(inode), NTFS(MFT)의 차이를 "N번째 블록 찾기" 관점에서 말하라.**
- 핵심: FAT는 전역 테이블에 뽑아낸 클러스터 연결 리스트라 체인 추적이 필요하고, ext는 직접/간접 블록의 불균형 트리(작은 파일 즉시, 큰 파일 완만) 또는 ext4 익스텐트, NTFS는 파일당 MFT 레코드에 작은 파일은 상주·큰 파일은 런 리스트다.
- 꼬리 대비: "왜 다 익스텐트로 수렴했나" → 순차 I/O가 이기므로 FS가 연속 할당을 지향하고, 연속 구간은 (시작, 길이) 표기가 가장 싸다. "디렉터리는?" → 내용이 (이름, inode) 엔트리인 파일이고, 큰 디렉터리는 트리/해시로 색인.

**Q6. std::async를 호출했는데 성능이 동기 호출과 같았다. 원인은?**
- 핵심: 반환된 future를 버리면 임시 future의 소멸자가 그 자리에서 태스크 완료를 대기한다(std::async가 만든 공유 상태의 특칙). 문법상 비동기, 의미상 동기.
- 꼬리 대비: "고치는 법" → future를 멤버로 소유해 수명을 넘기고, 대기가 허용되는 지점(소멸자·조인 지점)을 명시한다. 이때 태스크가 this를 잡으므로 객체 수명 계약도 함께 고쳐야 한다.
- Winters: `CHttpClient.cpp:90~92` 주석에 실사고로 박제, `:148~156` future 소유 + `:103~117` 소멸자 드레인이 수정형.

## ⑥ 흔한 오답 / 함정

1. **"epoll은 비동기 I/O다"라고 답하기.**
   epoll은 준비 통지(readiness)일 뿐 I/O 자체는 여전히 앱이 동기 read/write로 수행한다. 비동기(완료 통지)는 IOCP/io_uring이다. 이 구분을 못 하면 "일반 파일에 epoll을 쓰면 왜 소용없나"(`epoll_ctl`이 EPERM으로 등록을 거부하고, select/poll은 항상 ready로 보고해서)에서 바로 무너진다.

2. **"write()가 성공하면 디스크에 있다"는 전제로 코드/답변 짜기.**
   write-back 캐시 때문에 성공 반환과 영속 사이에 창이 있다. 반대 극단인 "그럼 모든 쓰기에 fsync"도 오답 — write-back의 이득을 전부 버리는 것이므로, 외부에 성공을 약속하는 커밋 지점에만 쓴다.

3. **"SSD니까 랜덤/순차는 이제 상관없다."**
   격차가 줄었을 뿐 순위는 안 바뀌었다. FTL 매핑·프리페치·요청 병합이 순차에 유리하고, 랜덤 쓰기는 쓰기 증폭을 부른다. LSM-tree, pak 파일, 저널이 여전히 순차 지향인 이유를 대면 된다.

4. **IOCP 워커 수를 "동접 수만큼" 잡거나 concurrency 값과 혼동.**
   완료 모델의 요점이 스레드 수 O(코어)다. 워커는 GQCS 대기 스레드 총수, concurrency는 커널이 유지하는 동시 실행 수 상한이고, 관례는 "concurrency = 코어 수, 워커는 블록 대비 약간 여유".

5. **제출한 오버랩드 버퍼를 지역 변수로 두기.**
   completion 모델은 제출~완료 사이 버퍼/OVERLAPPED의 수명을 앱이 보장해야 한다. 스택 버퍼로 WSARecv를 걸고 함수를 빠져나가면 커널이 해제된 스택에 쓴다. Winters가 `IOContext`를 세션 멤버로 두고 pending 카운트로 수명을 지키는 이유(`Session.cpp:61~67`).

6. **"파일 삭제 = 데이터 삭제"로 이해.**
   삭제는 이름→inode 링크 제거(unlink)이고, 실체 해제는 링크 카운트 0 + 여는 프로세스 없음일 때다. 열린 파일을 지워도 fd로 계속 읽히는 것, 하드링크가 남으면 데이터가 사는 것이 여기서 나온다.
