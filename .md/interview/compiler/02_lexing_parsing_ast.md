# 어휘 분석과 구문 분석 — 텍스트에서 트리로

> "컴파일러가 소스 코드를 어떻게 이해하나요?", "렉서와 파서는 왜 나뉘어 있나요?", "C++은 왜 파싱하기 어려운 언어라고 하나요?"에 대한 답. 컴파일러 프런트엔드의 앞 절반 — 문자의 나열이 어떻게 의미 구조를 담은 트리가 되는가 — 를 원리 수준에서 관통한다.

---

## ① 한 줄 본질

> **"컴파일러 프런트엔드는 언어를 계층으로 쪼개서 정복한다. 토큰 수준은 정규 언어(regular language)라서 유한 오토마타(finite automaton) — 즉 상태 말고는 메모리가 없는 기계 — 로 O(n)에 스캔하고, 중첩 구조는 정규로는 불가능하므로 문맥 자유 문법(context-free grammar)과 스택을 가진 파서에 맡긴다. 파서의 출력은 파스 트리가 아니라 AST — 괄호·세미콜론 같은 구체 문법의 잡음을 버리고 의미 구조만 남긴 트리 — 이고, 실무 컴파일러(Clang/GCC/V8)는 이론적으로 더 강력한 LR 생성기 대신 손으로 짠 재귀 하강을 쓴다. 에러 메시지 품질과 문맥 의존 처리가 이론적 우아함보다 값지기 때문이다."**

핵심 설계 원리는 하나다: **각 단계에 딱 필요한 만큼의 계산 능력만 준다.** 토큰 인식에 스택은 과잉이고, 중첩 괄호에 유한 상태는 부족하다. 이 "능력의 계층"이 렉서/파서 분리의 이유이자, 이 챕터 전체를 관통하는 질문의 답이다.

---

## ② 기본 개념

### 프런트엔드 지도 — 문자에서 트리까지

```
"int x = a + b * 2;"                     ← 문자의 1차원 나열 (구조 없음)
        │
        ▼  어휘 분석 (lexing) — 정규 언어 / 유한 오토마타
[int] [x] [=] [a] [+] [b] [*] [2] [;]    ← 토큰 스트림 (단어 단위, 여전히 1차원)
        │
        ▼  구문 분석 (parsing) — 문맥 자유 문법 / 스택
        Decl                              ← AST (계층 구조 = 의미의 뼈대)
       /  |  \
     int  x   BinOp(+)
             /        \
            a         BinOp(*)
                     /        \
                    b          2
        │
        ▼  의미 분석 (semantic analysis) — 타입 검사, 심볼 해석 (다음 챕터)
```

포인트: `b * 2`가 `a + b`보다 먼저 묶인 것 — **연산자 우선순위가 트리의 깊이로 인코딩됐다.** 텍스트에는 없던 계층이 트리에는 있다. 이 변환이 프런트엔드 앞 절반의 전부다.

C++은 한 겹이 더 있다 — 전처리기(preprocessor)가 매크로 치환을 먼저 하고, 렉서는 전처리 토큰 위에서 돈다. 그 파이프라인 전체는 컴파일 단계 챕터의 몫이고, 여기서는 "렉서가 보는 것은 전처리가 끝난 텍스트"라는 사실만 기억하면 된다.

### 어휘 분석 1 — 토큰이란 무엇인가

토큰(token)은 "언어의 최소 의미 단위"다. 렉서는 문자 스트림을 토큰 스트림으로 바꾸면서 세 가지를 붙인다.

| 구성 요소 | 예 (`count = 42`의 `42`) | 왜 필요한가 |
|---|---|---|
| 종류(kind) | `TK_INT_LITERAL` | 파서가 분기하는 기준. 파서는 종류만 본다 |
| 렉심(lexeme) | 문자열 `"42"` | 원문 그대로. 값 계산·에러 출력에 사용 |
| 소스 위치(source location) | `main.cpp:17:9` | 에러 메시지·디버그 정보의 뿌리. 여기서 버리면 영영 복구 불가 |

파서 입장에서 `42`든 `1337`이든 똑같은 `TK_INT_LITERAL`이다. **렉서의 본질은 "파서가 신경 쓸 필요 없는 차이를 지우는 것"** — 공백, 주석, 리터럴의 구체 철자가 여기서 사라진다. 소스 위치만은 예외로 끝까지 들고 간다. 좋은 컴파일러 에러 메시지는 전부 이 위치 정보의 투자에서 나온다.

### 어휘 분석 2 — 왜 렉서를 파서에서 분리하는가

이론적으로는 파서가 문자를 직접 먹어도 된다. 그런데도 전 세계 컴파일러가 렉서를 분리하는 이유는 세 겹이다.

1. **계산 능력의 계층이 다르다.** 식별자·숫자·연산자의 모양은 정규 언어다 — 유한 상태로 충분하다. 이걸 문맥 자유 문법에 섞으면 문법이 공백·주석 처리 규칙으로 오염되고, 모든 규칙 사이에 "여기 공백 허용"을 끼워야 한다. 렉서가 공백을 먹어치우면 파서 문법이 극적으로 단순해진다.
2. **성능.** DFA 기반 스캔은 문자당 상태 전이 한 번, O(n)이고 분기 예측도 잘 된다. 컴파일러가 읽는 문자 수는 토큰 수보다 훨씬 많으므로(공백·주석·긴 식별자), 가장 뜨거운 안쪽 루프를 가장 값싼 기계에 맡기는 배치다.
3. **관심사 분리.** 인코딩 처리, 라인/컬럼 추적, 리터럴 이스케이프 해석은 전부 렉서에 격리된다. 파서는 "토큰 종류의 나열"이라는 깨끗한 세계에서 산다.

### 어휘 분석 3 — 정규 언어와 유한 오토마타: 렉서 제너레이터의 원리

토큰의 모양은 정규 표현식(regular expression)으로 정의된다: 식별자 = `[a-zA-Z_][a-zA-Z0-9_]*`, 정수 = `[0-9]+`. 여기서 핵심 정리는 **정규식 ↔ NFA ↔ DFA가 표현력이 완전히 같다**는 것(클레이니 정리). 이 동치가 렉서 제너레이터(lex/flex)의 원리 그 자체다.

```
토큰별 정규식들
   │  Thompson 구성 (기계적 변환: 정규식 연산자 하나 = 작은 NFA 조각)
   ▼
NFA (비결정적 — 한 입력에 여러 다음 상태 허용, ε-전이 허용)
   │  부분집합 구성 (subset construction: "NFA가 동시에 있을 수 있는
   ▼   상태들의 집합" 하나를 DFA 상태 하나로)
DFA (결정적 — 문자당 전이 딱 하나 → 표 참조 한 번으로 실행)
   │  상태 최소화 (Hopcroft)
   ▼
표 기반 스캐너 코드 생성
```

왜 NFA를 거치는가? **정규식→NFA는 기계적이고(합성 가능), DFA는 실행이 빠르기** 때문이다. 사람이 쓰기 좋은 표현(정규식)과 기계가 돌기 좋은 표현(DFA) 사이의 다리가 NFA다. 대가는 상태 수 — 부분집합 구성은 최악의 경우 상태가 지수적으로 늘 수 있다(2^n). 토큰 문법은 실제로는 작아서 문제가 안 되지만, "정규식 한 방 엔진"이 위험한 이유를 물으면 이 지수 폭발과 (백트래킹 엔진의) ReDoS를 답하면 된다.

손으로 짠 렉서도 결국 같은 물건이다 — `switch(state)` 안의 `switch(ch)`는 전이표를 코드로 편 것뿐. 실무 컴파일러 렉서(Clang 등)가 손코딩인 이유는 생성기가 못해서가 아니라, 인코딩·전처리기 얽힘·에러 위치 같은 주변 요구가 표 기반 틀보다 커서다.

### 어휘 분석 4 — 최장 일치(maximal munch)

`>=`를 만난 렉서는 `>`와 `=`로 자를 수도, `>=` 하나로 읽을 수도 있다. 규칙은 **"가능한 가장 긴 토큰을 만든다"** — 최장 일치다. 구현은 "마지막으로 수락 상태를 지난 위치를 기억해 두고, 더 못 가면 거기로 되감기".

이 규칙은 결정을 단순하게 만들지만 **문법을 아는 척하지 않는 규칙**이라 반례가 생긴다. 고전 예:

```cpp
a+++++b;   // 렉서: [a][++][++][+][b] — 최장 일치의 기계적 결과
           // 파서: (a++)++ 는 lvalue가 아니므로 컴파일 오류
a++ + ++b; // 사람이 공백으로 직접 끊어줘야 한다
```

렉서는 "여기서 `+`를 하나만 먹으면 파싱이 되는데"를 모른다 — 알려면 파서의 지식이 필요하고, 그러면 계층 분리가 깨진다. **최장 일치는 '대체로 옳고 예측 가능한' 결정을 '항상 옳은' 결정보다 우선한 설계**다. 이 트레이드오프가 가장 아프게 터진 곳이 아래 `>>` 사건이다.

### 어휘 분석 5 — 키워드 vs 식별자: 예약어 테이블

`if`는 키워드고 `iff`는 식별자다. 순진한 방법은 키워드마다 정규식을 주고 DFA에 태우는 것 — 되긴 하지만 상태 수가 불고, "키워드 규칙과 식별자 규칙이 동시에 매치되면 누가 이기나"라는 우선순위 문제가 생긴다.

실무 표준은 반대로 간다: **일단 전부 식별자로 스캔한 뒤, 예약어 테이블(해시 테이블)을 한 번 찾아 재분류한다.**

```
"while" 스캔 → 식별자 후보 → 예약어 테이블 lookup → 있음 → TK_KW_WHILE
"whale" 스캔 → 식별자 후보 → 예약어 테이블 lookup → 없음 → TK_IDENTIFIER
```

이 설계가 좋은 이유: DFA는 식별자 규칙 하나만 알면 되니 작고, 키워드 추가가 테이블 한 줄이고, 문맥 의존 키워드(C++11의 `final`/`override`는 특정 위치에서만 키워드)를 "테이블에 있지만 파서가 문맥 보고 결정"으로 자연스럽게 처리할 수 있다. 예약어가 식별자와 같은 모양인 것은 우연이 아니라 이 처리를 위한 언어 설계다.

### 어휘 분석 6 — C++ 렉싱 함정: `>>` 사건

```cpp
std::vector<std::vector<int>> v;
//                          ^^ C++03: 오른쪽 시프트 연산자 토큰 하나!
```

최장 일치에 충실한 렉서는 `>>`를 시프트 연산자로 먹는다. C++03에서는 이게 문법 오류라서 프로그래머가 `> >`로 띄어 써야 했다 — 언어가 렉서의 무지를 사용자에게 전가한 사례다.

C++11의 해법은 렉서를 고친 게 아니다. 렉서는 여전히 `>>` 하나를 만들고, **파서가 템플릿 인자 목록을 파싱 중일 때 `>>` 토큰을 `>` 두 개로 쪼갠다.** "토큰은 문맥 없이 만들고, 문맥이 있는 쪽(파서)이 예외적으로 재해석한다" — 계층 분리를 최대한 지키면서 구멍만 메운 설계다(Java는 2004년 제네릭 도입 때부터 같은 처리를 했다).

공짜는 아니었다. 유효했던 C++03 코드의 의미가 바뀐다:

```cpp
A<(1 >> 2)> a;  // C++11: 진짜 시프트는 이제 괄호가 필요하다
```

표준위원회는 "호환성 깨짐 < 매일 수백만 명이 겪는 불편"으로 저울질했다. 언어 진화에서 렉싱 규칙 하나가 어떤 파급을 갖는지 보여주는 최고의 사례라 면접 소재로도 좋다. 같은 계열의 마이너 사례로, `std::vector<::MyType>`의 `<:`가 다이그래프(digraph) `[`로 읽히던 문제도 C++11이 "`<::` 뒤가 `:`나 `>`가 아니면 `<`를 단독 토큰으로" 규칙을 넣어 풀었다.

### 구문 분석 1 — 문맥 자유 문법: 왜 정규 언어로는 안 되는가

토큰 스트림 위의 구조 — 중첩 괄호, 블록 안의 블록, 식 안의 식 — 는 정규 언어가 아니다. 직관 증명이 펌핑 렘마(pumping lemma)다:

```
언어 L = { (^n )^n }  — 여는 괄호 n개 + 닫는 괄호 n개

DFA는 상태가 유한하다(= 유한한 메모리).
상태가 k개인 DFA에 '(' 를 k+1개 먹이면, 비둘기집 원리로
어떤 상태를 두 번 지난다:

  ( ( ( ( ( ( ( ( ...
  ─┬─ ─┬─
   │    └ 이 구간(y)을 돌 때 같은 상태로 돌아옴
   │
   → y를 0번 돌든 100번 돌든 DFA는 구별 못 함
   → "(((" 와 "(((((" 를 같은 상태로 받아들임
   → 닫는 괄호 수를 맞출 방법이 없음
```

**유한 오토마타는 셀 수 없다.** 괄호 깊이는 무한히 깊어질 수 있으므로 무한한 카운터가 필요하고, 그게 스택이다. DFA에 스택 하나를 붙인 기계가 푸시다운 오토마타(pushdown automaton)고, 그것이 인식하는 언어 클래스가 문맥 자유 언어다. "HTML을 정규식으로 파싱하지 마라"는 밈의 수학적 근거가 정확히 이것이다.

문맥 자유 문법(CFG)은 생성 규칙(production)의 집합이다:

```
Expr → Expr '+' Term | Term        (비단말 Expr은 이 형태들로 다시 쓸 수 있다)
Term → Term '*' Factor | Factor
Factor → NUMBER | '(' Expr ')'     ← 규칙이 자기 자신(Expr)을 다시 부른다 = 재귀 = 중첩
```

시작 기호에서 규칙을 반복 적용해 토큰열을 만들어내는 과정이 유도(derivation)다. **정규식과의 본질 차이는 딱 하나 — 규칙이 재귀할 수 있다.** `Factor → '(' Expr ')'`의 재귀가 임의 깊이의 중첩을 만든다. 그리고 위 문법처럼 `*`를 `+`보다 안쪽 계층에 두면, **우선순위가 문법 구조 자체에 인코딩된다.**

### 구문 분석 2 — 파스 트리 vs AST

파스 트리(parse tree, 구체 구문 트리 CST)는 유도 과정을 그대로 그린 트리다. 모든 비단말, 모든 괄호, 모든 세미콜론이 노드로 남는다. `(a)`를 파싱하면 `Factor → '(' Expr ')'` 노드 아래에 괄호 두 개가 자식으로 매달린다.

AST(abstract syntax tree)는 거기서 **의미에 기여하지 않는 것을 전부 버린 트리**다:

```
"(a + b)"의 파스 트리                  같은 입력의 AST

        Factor
       /  |   \
     '('  Expr ')'                        BinOp(+)
          /  |  \                         /      \
       Expr '+' Term          →          a        b
        |        |
       Term    Factor
        |        |
      Factor     b
        |
        a
```

버리는 것: 괄호(우선순위는 이미 트리 모양에 있다), 세미콜론(문 경계도 트리에 있다), `Expr→Term→Factor` 같은 우선순위용 사다리 비단말. 남기는 것: 연산의 종류와 피연산자, 그리고 소스 위치. **"구체 문법은 사람이 타이핑하기 위한 표기법이고, AST는 컴파일러가 생각하기 위한 자료구조"** — 같은 의미의 두 표현이며, 이후 모든 단계(타입 검사, 최적화, 코드 생성)는 AST만 본다. 실무 파서는 파스 트리를 실제로 만들지 않고 파싱하면서 곧바로 AST를 조립한다.

### 구문 분석 3 — 재귀 하강: 문법 규칙 하나 = 함수 하나

톱다운 파싱의 대표가 재귀 하강(recursive descent)이다. 아이디어가 극도로 단순해서 강력하다: **비단말마다 함수를 하나 만들고, 규칙의 오른쪽을 코드로 그대로 옮긴다.** 비단말 참조는 함수 호출로, 단말은 토큰 매치로.

```cpp
// expr   := term  (('+'|'-') term)*
// term   := factor(('*'|'/') factor)*
// factor := NUMBER | '(' expr ')'

Node* ParseFactor() {
    if (Peek() == TK_NUMBER)  return MakeNumber(Next());
    Expect(TK_LPAREN);
    Node* inner = ParseExpr();     // 재귀 — 문법의 재귀가 콜 스택의 재귀로
    Expect(TK_RPAREN);
    return inner;                  // 괄호 노드는 만들지 않는다 = AST
}

Node* ParseExpr() {
    Node* lhs = ParseTerm();
    while (Peek() == TK_PLUS || Peek() == TK_MINUS) {
        Token op = Next();
        lhs = MakeBinary(op, lhs, ParseTerm());  // 왼쪽부터 접음 = 좌결합
    }
    return lhs;
}
```

CFG에 필요하다던 "스택"이 어디 갔나? **콜 스택이 그 스택이다.** 중첩 괄호를 만나면 `ParseExpr → ParseFactor → ParseExpr → ...`로 재귀가 쌓이고, 그 깊이가 괄호 깊이다. 이론의 푸시다운 오토마타를 언어 런타임이 공짜로 제공하는 셈이다.

이 기법이 실무 표준이다 — **Clang, GCC(2000년대 중반 bison 기반을 버리고 손파서로 교체, C++는 3.4부터), V8 전부 손으로 짠 재귀 하강**이다. 이유는 뒤의 트레이드오프 절에서 정리한다.

### 구문 분석 4 — LL(1)과 FIRST/FOLLOW 직관

재귀 하강이 성립하려면 각 함수가 **다음 토큰 하나만 보고 어느 규칙으로 갈지 결정**할 수 있어야 한다. 이 성질을 가진 문법이 LL(1)이다 — 왼쪽에서 읽고(Left-to-right), 좌측 최우선 유도(Leftmost derivation)를, 1토큰 미리보기(lookahead)로.

결정 도구가 FIRST/FOLLOW 집합이다:

- **FIRST(α)** = α에서 유도되는 문자열이 시작할 수 있는 토큰들. "이 규칙을 고르면 첫 토큰으로 뭐가 올 수 있나?"
- **FOLLOW(A)** = 비단말 A 바로 뒤에 올 수 있는 토큰들. "A가 비어 있을(ε) 수도 있다면, 지금 토큰이 A '다음' 것인지 어떻게 아나?"

직관: `Factor → NUMBER | '(' Expr ')'`에서 FIRST(첫 대안) = {NUMBER}, FIRST(둘째 대안) = {'('} — 겹치지 않으므로 다음 토큰만 보면 분기가 확정된다. 이게 위 코드의 `if (Peek() == TK_NUMBER)`다. 두 대안의 FIRST가 겹치면 LL(1)이 아니고, 그때는 룩어헤드를 늘리거나(LL(k)), 문법을 고치거나(좌인수분해), 잠정 파싱(tentative parsing)으로 물러선다. **FIRST/FOLLOW는 이론 장식이 아니라 "내 if문이 모든 경우를 정확히 가르는가"의 체크리스트다.**

### 구문 분석 5 — 좌재귀: 제거와 반복문 변환

문법의 자연스러운 좌결합 표기는 재귀 하강을 죽인다:

```
Expr → Expr '+' Term | Term

Node* ParseExpr() {
    Node* lhs = ParseExpr();   // 토큰을 하나도 안 먹고 자기 호출 → 무한 재귀
    ...
```

교과서 해법은 좌재귀 제거 — `Expr → Term Expr'`, `Expr' → '+' Term Expr' | ε`로 다시 쓰기. 그러나 실전 해법은 더 단순하다: **좌재귀는 반복이다.** `Expr → Term ('+' Term)*`로 읽고 while 루프를 돌리면 된다 — 위 `ParseExpr`의 루프가 정확히 그것이고, 루프 안에서 `lhs = MakeBinary(op, lhs, rhs)`로 접기 때문에 좌결합 트리가 자연히 나온다. "문법의 재귀 방향"과 "결합 방향"과 "코드의 루프"가 하나로 이어지는 지점 — 여기를 이해하면 파서 손코딩이 두렵지 않다.

### 구문 분석 6 — 연산자 우선순위: precedence climbing / Pratt parsing

우선순위 계층마다 함수를 만드는 방식(`ParseExpr → ParseTerm → ParseFactor`)은 C++처럼 우선순위가 15단을 넘는 언어에서 함수 15개 사다리가 된다 — 동작은 하지만 토큰 하나에 콜 스택 15겹은 낭비고, 우선순위 추가가 함수 추가가 된다.

실무 표준 해법이 precedence climbing(Pratt parsing과 본질적으로 같은 기법의 두 이름)이다. **우선순위 테이블 하나 + 함수 하나**로 모든 이항 연산을 처리한다:

```cpp
// Prec: '+'→10, '*'→20, ... 테이블 하나가 문법 15단을 대체
Node* ParseBinary(int minPrec) {
    Node* lhs = ParsePrimary();
    while (IsBinOp(Peek()) && Prec(Peek()) >= minPrec) {
        Token op = Next();
        // 좌결합: 나보다 '강한' 것만 오른쪽 하위식에 흡수 → +1
        // 우결합(대입, 거듭제곱): 같은 우선순위도 흡수 → +0
        int next = Prec(op) + (IsLeftAssoc(op) ? 1 : 0);
        lhs = MakeBinary(op, lhs, ParseBinary(next));
    }
    return lhs;
}
```

동작 직관: `a + b * c`에서 `+`를 처리하는 중 `*`(더 높음)를 만나면 재귀가 `b * c`를 먼저 묶어 돌려주고, `a * b + c`에서는 `+`(더 낮음)가 `minPrec` 검사에서 걸려 루프가 끝나 `a * b`가 먼저 확정된다. **우선순위와 결합성이 문법 구조에서 정수 비교로 옮겨온 것** — Clang의 이항 식 파서도 이 구조다. 면접에서 "파서 짜봤나"류 질문에 화이트보드로 재현할 수 있어야 하는 대표 코드다.

### 구문 분석 7 — 바텀업: LR/LALR의 아이디어

톱다운이 "시작 기호에서 내려가며 예측"이라면, LR은 반대로 **토큰을 스택에 쌓다가(시프트, shift) 규칙의 오른쪽이 완성되는 순간 왼쪽 비단말로 접는다(리듀스, reduce).** 잎에서 뿌리로, 트리를 아래에서 위로 짓는다.

```
입력: a + b * c        (Expr→Expr+Term, Term→Term*F, ... 문법)

스택               남은 입력      행동
──────────────────────────────────────────────
                   a + b * c     shift a
a                  + b * c       reduce (a→F→Term→Expr)
Expr               + b * c       shift +
Expr +             b * c         shift b
Expr + b           * c           reduce (b→F→Term)  ← 여기가 갈림길:
Expr + Term        * c           '*'가 보이므로 reduce(Expr+Term→Expr) 말고 shift!
Expr + Term *      c             shift c
Expr + Term * F    (끝)          reduce (Term*F→Term), reduce (Expr+Term→Expr)
```

갈림길("지금 접을까, 더 쌓을까")을 결정하는 것이 LR 오토마타 — 문법에서 기계적으로 만들어지는 상태 기계 + 룩어헤드 테이블이다. 이 표가 사람 손으로 만들 물건이 아니라서 LR은 태생부터 **파서 제너레이터(yacc/bison)의 기술**이다. LALR은 LR(1) 표의 상태 중 핵이 같은 것들을 병합해 표 크기를 줄인 실용 변종으로, yacc/bison이 만드는 것이 이것이다. (ANTLR은 이름이 같이 묶여 다니지만 LL 계열 — 강화된 톱다운 — 이라는 점은 구분해서 말해야 한다.)

LR의 강점은 표현력이다: 좌재귀를 그대로 먹고(오히려 좌재귀가 자연스럽다), LL(1)이 못 다루는 문법을 다수 수용한다. 약점은 **충돌(conflict)** — 같은 상태에서 시프트도 리듀스도 가능하면(시프트-리듀스 충돌) 생성기가 경고를 뱉고, 개발자는 오토마타 상태 덤프를 읽으며 문법을 디버깅해야 한다. 이 디버깅 경험의 난해함이 LR의 악명 절반을 차지한다.

### 구문 분석 8 — 모호성: dangling else

문법이 한 입력에 두 파스 트리를 허용하면 모호(ambiguous)하다. 가장 유명한 예가 매달린 else다:

```cpp
if (a) if (b) f(); else g();

해석 1 (안쪽 결합)           해석 2 (바깥 결합)
if (a)                       if (a)
    if (b) f();                  if (b) f();
    else   g();              else g();
```

문법 `Stmt → if (E) Stmt | if (E) Stmt else Stmt | ...`로는 둘 다 유도 가능 — LR 표에서는 `else`를 본 순간 "안쪽 if를 지금 리듀스할까(해석 2), else를 시프트할까(해석 1)"의 시프트-리듀스 충돌로 나타난다. 모든 주류 언어의 해법은 **"else는 가장 가까운 if에 붙는다"** = 시프트 우선이고, bison의 기본 충돌 해소(시프트-리듀스면 시프트)가 우연히도 정확히 이 관례라 실무에서는 경고 억제로 처리한다. 근본 해법은 언어 설계 쪽에 있다 — 블록을 강제하는 문법(Rust의 `if`는 중괄호 필수)은 이 모호성이 아예 생기지 않는다.

### 구문 분석 9 — 손파서 vs 파서 제너레이터

| 축 | 손 재귀 하강 (Clang/GCC/V8) | 생성기 (yacc/bison/ANTLR) |
|---|---|---|
| 문법 정확성 | 사람이 보장 (문법-코드 불일치 위험) | 문법 파일이 곧 명세, 충돌을 기계가 검출 |
| 에러 메시지 | 임의 지점에 임의 코드 — "여기서 `;` 빠뜨렸죠?" 가능 | 표가 "syntax error"라 하는 지점에 개입 창구가 좁음 |
| 에러 복구 | 파서 상태를 알고 있으니 정교한 복구 가능 | 정형화된 복구(error 토큰)에 의존 |
| 문맥 의존 처리 | if문 하나로 심볼 테이블 참조 (C++에 필수) | 액션 코드로 우회 — 어색하고 취약 |
| 유지보수 | 파서 코드가 수만 줄 (Clang) | 문법 파일은 짧고 선언적 |
| 적합한 곳 | 에러 품질이 제품인 컴파일러, 문맥 의존 언어 | DSL, 설정 언어, 문법이 안정된 포맷 |

한 줄 요약: **파서의 행복 조건(문법이 깨끗하고 입력이 대체로 올바름)과 컴파일러의 현실(문법이 더럽고 입력이 대체로 틀림)이 반대라서, 이론의 승자(LR)와 실무의 승자(손 재귀 하강)가 갈렸다.** 컴파일러는 하루 종일 틀린 코드를 읽는 프로그램이다 — 에러 메시지와 복구가 본업이고, 그 본업에서 손파서가 압도적으로 유리하다. IDE 시대에 이 격차는 더 벌어졌다: 타이핑 중인(항상 불완전한) 코드에서도 최대한 파싱을 복구해 자동완성을 줘야 하기 때문이다.

### 구문 분석 10 — 에러 복구: 패닉 모드와 동기화 토큰

첫 문법 오류에서 파싱을 멈추면 사용자는 컴파일 한 번에 오류 하나씩만 본다. 그래서 파서는 오류 후 **복구**해서 계속 간다. 기본기가 패닉 모드(panic mode)다:

```
오류 지점부터 토큰을 버리며 전진 → 동기화 토큰(synchronizing token)을
만나면 멈추고 → 파서 상태를 그 지점의 문법 위치로 리셋 → 재개

동기화 토큰 = "여기부터는 새 출발"이라 믿을 만한 경계:
  ';'  문의 끝        '}'  블록의 끝        'if'/'for'/... 문의 시작
```

단순 무식하지만 강력하다 — 오류의 파급을 문 하나로 격리한다. 대신 부작용이 연쇄 오류(cascading error)다: 진짜 오류 하나가 파서를 잘못된 상태로 밀어 유령 오류 수십 개를 낳는 것. 좋은 컴파일러는 패닉 모드 위에 국소 복구(빠진 `;` 삽입해 보기, 오타 교정 제안)를 얹고, 복구로 만든 노드에 "오류 산물" 표시를 남겨 의미 분석이 이중 보고하지 않게 한다. 에러 복구야말로 손파서 진영이 생성기를 이긴 주전장이다.

### 구문 분석 11 — C++이 파싱하기 어려운 이유

C++은 "파서가 의미 분석과 얽히는 언어"의 대표다. 구문만 봐서는 무슨 구조인지 결정할 수 없는 지점이 문법에 박혀 있다.

**(1) 선언 vs 표현식 — 문맥(심볼 테이블) 의존.**

```cpp
T * a;    // T가 타입이면: 포인터 a의 선언
          // T가 변수면:  T 곱하기 a라는 표현식 문
a b(c);   // a가 타입이면: c로 초기화된 b 선언 / 함수 선언
```

토큰열이 완전히 같은데 **`T`가 무엇인지(= 심볼 테이블)를 알아야 파스 트리가 갈린다.** 문맥 자유 문법의 전제 — 파싱은 이름의 의미를 몰라도 된다 — 가 여기서 무너진다. C는 이 문제를 렉서 해크(lexer hack)로 풀었다: typedef된 이름을 렉서가 심볼 테이블을 훔쳐보고 `TYPE_NAME`이라는 다른 토큰 종류로 뱉는 것. C++은 얽힘이 더 깊어(템플릿, 스코프, 오버로드) Clang은 잠정 파싱 — 선언으로 시도해 보고 안 되면 되감아 표현식으로 재시도 — 까지 동원하며, 표준은 이 부류에 "선언으로 파싱될 수 있으면 선언이다"라는 중재 규칙을 둔다.

**(2) most vexing parse — 그 중재 규칙의 대가.**

```cpp
Widget w(Gadget());
// 의도: Gadget 임시 객체로 Widget w를 생성
// 실제: "Gadget을 반환하는 함수"를 인자로 받아 Widget을 반환하는 '함수 w'의 선언!
// ("선언 가능하면 선언" 규칙이 기계적으로 적용된 결과)

Widget w{Gadget{}};   // C++11 중괄호 초기화 — {}는 선언으로 읽힐 길이 없어 모호성 소멸
```

문법 모호성 → 표준의 기계적 중재 → 인간 직관과의 충돌 → 새 문법(uniform initialization)으로 우회. 언어가 파싱 문제를 어떻게 '설계로' 푸는지의 전체 사이클이 이 한 줄에 들어 있다.

**(3) 템플릿 — 파서가 항복하고 프로그래머에게 묻는 지점.**

```cpp
template <typename T>
void f() {
    T::type * x;          // T::type이 타입? → 포인터 선언. 값? → 곱셈
    typename T::type * x; // 'typename'으로 프로그래머가 파서에게 답을 줌
    t.template get<int>(); // 'template' 없으면 get < int 로 파싱될 수 있음
}
```

템플릿 본문을 파싱하는 시점에는 `T`가 미정이라 심볼 테이블조차 답을 못 준다. 그래서 C++은 `typename`/`template`이라는 소거 키워드(disambiguator)를 문법에 넣어 **파싱 결정을 프로그래머에게 위임**했다. 여기에 `>>` 꺾쇠 문제(위 렉싱 절), 그리고 템플릿 인스턴스화가 튜링 완전하다는 사실까지 겹치면 — C++ 파싱은 일반적으로 의미 분석과 분리 불가능하다. **C++ 전용 정확한 파서가 사실상 컴파일러 프런트엔드 전체(Clang libTooling)인 이유, IntelliSense류 도구가 결국 진짜 컴파일러를 내장하게 된 이유가 이것이다.**

## ③ 심화 — 꼬리질문 대비

**"렉서와 파서를 하나로 합치면 안 되나?"**
된다 — 스캐너리스 파싱(scannerless parsing)이라는 분야도 있다. 하지만 문법이 공백/주석 규칙으로 오염되고, 가장 뜨거운 루프(문자 처리)가 가장 비싼 기계(파서)에 얹힌다. 합치는 게 이득인 곳은 토큰 경계 자체가 문맥 의존인 언어(마크다운 같은) 정도다. 핵심 답: "정규로 충분한 일에 스택을 쓰지 않는다 — 능력의 계층 분리가 성능과 단순성을 동시에 준다."

**"정규식으로 HTML/코드를 파싱하면 왜 안 되나?"**
중첩 구조 때문이다. 정규 언어의 인식 기계는 유한 상태 = 유한 메모리라 괄호/태그 깊이를 셀 수 없다(펌핑 렘마). 단, 정확히 말하면 "중첩 깊이가 유계라면" 정규로도 가능하다 — 실무 답은 "깊이 제한을 언어 명세에 박을 수 없으니 CFG로 간다"까지 말하면 완결된다.

**"NFA와 DFA의 차이는? 왜 굳이 NFA를 거치나?"**
NFA는 한 입력에 다음 상태가 여러 개일 수 있고 ε-전이를 허용한다. 표현력은 DFA와 동일하지만(부분집합 구성으로 변환 가능) 역할이 다르다 — 정규식→NFA(Thompson)는 기계적·합성적이라 만들기 쉽고, DFA는 문자당 전이 하나라 돌리기 빠르다. "쓰기 좋은 표현과 돌리기 좋은 표현 사이의 컴파일"이며, 대가는 부분집합 구성의 최악 지수적 상태 폭발이다.

**"최장 일치가 틀리는 경우를 아는가?"**
`a+++++b`(`[a][++][++][+][b]`로 갈려 파싱 불가 — `a++ + ++b`가 의도라도)와 C++03의 `vector<vector<int>>`. 최장 일치는 "파서를 모른 채 내리는 지역적으로 탐욕적인 결정"이라 전역적으로 틀릴 수 있다. C++11은 이걸 렉서가 아니라 파서 쪽에서(템플릿 문맥에서 `>>` 분할) 고쳤다는 것까지 말하면 좋다.

**"LL과 LR의 본질적 차이 한 줄로?"**
LL은 예측(내려가며 "이 규칙일 것이다"를 먼저 결정), LR은 확인(쌓아 놓고 "이 규칙이 완성됐다"를 나중에 결정). 결정을 늦출수록 정보가 많으므로 LR이 더 넓은 문법을 받는다 — 좌재귀도 LR에선 문제가 아니다. 대신 LL은 파싱 중 어느 순간에도 "지금 무슨 규칙 안에 있는지"를 알아서(콜 스택이 곧 문맥) 에러 메시지와 의미 액션 삽입이 쉽다.

**"이론상 LR이 더 강력한데 왜 Clang/GCC는 재귀 하강인가?"**
컴파일러의 본업이 "올바른 코드를 빨리 받아들이기"가 아니라 "틀린 코드에 좋은 진단 내기"이기 때문. 손파서는 임의 지점에서 임의 코드(심볼 테이블 조회, 맞춤 에러, 복구 휴리스틱)를 실행할 수 있다. 게다가 C++은 파싱이 의미 분석과 얽혀 있어서 순수 CFG 도구에 안 들어간다. GCC가 실제로 bison 파서를 버리고 손파서로 갈아탄 역사(C++는 3.4)가 이 논거의 실증이다.

**"LL(1)의 1은 무엇이고, k를 늘리면 다 해결되나?"**
룩어헤드 토큰 수. k를 늘리면 문법 클래스는 넓어지지만 표가 지수적으로 커지고, 그래도 C++의 선언/표현식 모호성 같은 것은 못 푼다 — `A a(b1, b2, ..., bn);`류는 유계 룩어헤드로 결정 불가능한 형태가 있어 결국 잠정 파싱(무제한 백트래킹)이나 의미 정보가 필요하다. "k의 문제가 아니라 문맥 자유성의 문제"로 넘어가는 게 정답 동선.

**"좌재귀는 왜 재귀 하강에서 무한 루프인가? 어떻게 푸나?"**
`Expr → Expr + Term`을 그대로 옮기면 `ParseExpr`이 토큰을 소비하기 전에 자신을 호출한다 — 진행 없는 재귀. 교과서 답은 좌재귀 제거(`Expr → Term Expr'`), 실전 답은 반복문 변환(`Expr → Term ('+' Term)*`) + 루프 안에서 왼쪽으로 접어 좌결합 트리 유지. "문법 변환이 아니라 관점 변환(재귀=반복)"이라 답하면 이해가 드러난다.

**"Pratt parsing이 뭔지 30초로 설명하라."**
우선순위 계층마다 문법 규칙/함수를 두는 대신, 연산자→우선순위 정수 테이블 하나와 함수 하나로 이항 식 전체를 파싱하는 기법. `ParseBinary(minPrec)`가 "minPrec보다 약한 연산자를 만나면 멈춘다"는 불변식을 유지하고, 좌결합이면 재귀에 `prec+1`, 우결합이면 `prec`을 넘긴다. 우선순위 추가가 테이블 한 줄이 되는 게 실무 가치다.

**"시프트-리듀스 충돌이 뭔가? 예를 들라."**
LR 파서의 한 상태에서 룩어헤드 토큰에 대해 시프트(더 쌓기)와 리듀스(지금 접기)가 모두 유효한 상황. 대표 예가 dangling else — `if (a) if (b) S` 뒤의 `else`에서 안쪽 if를 리듀스하면 else가 바깥에, 시프트하면 안쪽에 붙는다. 언어 관례(가장 가까운 if)와 bison의 기본(시프트 우선)이 일치해서 실무에선 그냥 둔다. 리듀스-리듀스 충돌은 대개 문법 설계 버그라는 대비도 언급하면 좋다.

**"AST에서 버리는 것과 절대 버리면 안 되는 것은?"**
버리는 것: 괄호, 구분자(세미콜론/콤마), 우선순위 사다리용 중간 비단말 — 전부 트리 구조 자체에 이미 인코딩된 정보다. 못 버리는 것: 소스 위치. 이후 모든 단계의 에러 메시지("이 타입 오류는 17행 9열")와 디버그 정보(-g)가 AST 노드에 붙은 위치에서 나온다. "구체 문법은 입력 표기법, AST는 사고용 자료구조"가 요약.

**"C++은 문맥 자유 언어인가?"**
파싱 관점에서 실질적으로 아니다. `T * a;`가 선언인지 곱셈인지는 `T`의 의미(심볼 테이블)에 달렸고, 템플릿 의존 이름은 `typename`/`template` 키워드로 프로그래머가 직접 소거해줘야 한다. C는 렉서 해크(typedef명을 별도 토큰으로)로 봉합했고, C++은 잠정 파싱 + "선언 가능하면 선언" 중재 규칙 + 소거 키워드의 3중 대응이 필요했다. "그래서 C++ 도구는 정확하려면 컴파일러 프런트엔드 전체를 내장해야 한다"까지가 완성된 답.

**"most vexing parse를 코드로 써 보라."**
`Widget w(Gadget());` — w는 객체가 아니라 함수 선언("Gadget을 반환하는 함수를 인자로 받고 Widget을 반환"). 원인은 "선언으로 파싱될 수 있으면 선언" 규칙. 해법은 C++11 중괄호 초기화 `Widget w{Gadget{}};` — 중괄호는 선언으로 읽힐 문법 경로가 없어 모호성이 원천 소멸. "문법 모호성을 문법 확장으로 푼 사례"로 정리.

**"파서의 에러 복구 전략을 아는 대로."**
(1) 패닉 모드: 동기화 토큰(`;`, `}`)까지 버리고 재개 — 단순, 견고, 대신 오류 사이 코드는 검사 누락. (2) 국소 교정: 토큰 삽입/삭제 가정("`;` 빠진 것 같다") — 메시지는 친절하나 잘못 짚으면 연쇄 오류. (3) 오류 산물 노드: 복구로 만든 AST에 표식을 남겨 후속 단계의 유령 진단을 억제. IDE 시대엔 복구 품질이 곧 자동완성 품질이라는 점이 현대적 포인트다.

**"렉서 제너레이터와 정규식 엔진(PCRE류)의 차이는?"**
렉서 제너레이터는 토큰 규칙 전체를 하나의 DFA로 미리 컴파일해 입력을 한 번 훑으며 최장 일치 + 규칙 우선순위로 토큰을 확정한다 — 백트래킹 없음, 문자당 O(1). PCRE류 백트래킹 엔진은 역참조 같은 비정규 기능을 주는 대신 최악 지수 시간(ReDoS)이 가능하다. "같은 '정규식'이라도 밑의 기계가 다르다"가 요지.

## ④ Winters에서의 적용 — 파서를 '안 만든' 결정들을 설명하기

Winters에는 텍스트 파서를 직접 짠 코드가 없다. 그런데 그게 이 챕터와 무관하다는 뜻이 아니라 — **파싱 비용을 어디서 낼지(런타임 텍스트 파싱 / 빌드 타임 코드 생성 / 파싱 제로 바이너리)를 세 지점에서 각각 다르게 결정한 프로젝트**다. 면접에서는 "파서를 만들어 봤다" 대신 "파서를 둘 곳과 없앨 곳을 판단해 봤다"로 말할 수 있다.

### (1) Stage1.dat — 파싱이 필요 없도록 설계한 포맷

맵 스테이지 데이터는 텍스트가 아니라 바이너리다. `Client/Private/Map/MapDataIO.cpp:126`의 `Load_Stage`는 렉서도 파서도 없다 — `fread(&header, sizeof(StageHeader), 1, pFile)`로 헤더 구조체를 통째로 읽는다. **텍스트 파싱의 본질이 "구조가 없는 문자열에서 구조를 복원하는 것"이라면, 바이너리 포맷은 구조체 레이아웃 자체가 문법이다.** 토큰 경계를 찾을 필요가 없고(필드 크기가 곧 경계), 유도할 문법이 없다(오프셋이 곧 구조). 오토마타가 하던 일을 `sizeof`가 한다.

그 대신 텍스트 파서가 공짜로 주던 것 — "이 입력이 이 언어의 문장인가"라는 검증 — 을 손으로 보충해야 한다. 그래서 `MapDataIO.cpp:144`가 매직 넘버를 검사하고(엉뚱한 파일 거부), `MapDataIO.cpp:150`이 버전 범위를 검사한다(스키마 드리프트 거부). 이후 버전 분기(v4 미만이면 웨이포인트 기본값 부트스트랩)는 문법으로 치면 "구버전 방언의 하위호환 규칙"에 해당한다. 면접 문장: "텍스트 포맷은 파서가 곧 검증기지만, 바이너리 포맷은 검증을 매직/버전/길이 검사로 직접 재구축해야 한다 — 그 차이를 알고 헤더를 설계했다."

### (2) FlatBuffers — 나는 파서 제너레이터의 소비자다

네트워크 스키마는 `.fbs`라는 텍스트 DSL로 쓴다. `Shared/Schemas/Command.fbs:23`의 `table CommandPacket { kind:CommandKind; ... }`, `Shared/Schemas/Snapshot.fbs:40`의 `table EntitySnapshot { ... }` — 이 파일들이 **이 프로젝트에서 렉싱/파싱되는 진짜 대상**이다. `flatc` 컴파일러가 `.fbs`를 렉싱(식별자/콜론/중괄호 토큰화)하고 파싱(테이블/필드 문법)해서 C++ 헤더를 생성한다. yacc가 문법 파일에서 파서를 생성하듯, flatc는 스키마 파일에서 직렬화 코드를 생성한다 — "작은 언어 + 코드 생성기" 패턴의 동일한 사례다.

여기서 설계 결정이 두 겹이다. 첫째, **파싱을 빌드 타임으로 밀어냈다.** JSON을 골랐다면 매 패킷마다 런타임 텍스트 파싱(렉서+파서 실행)을 했겠지만, FlatBuffers는 스키마 파싱을 빌드에서 한 번 하고 런타임 와이어 포맷은 오프셋 기반 바이너리라 역직렬화 파싱이 0이다. 둘째, 파싱이 없어진 자리에 검증이 남는다 — `Shared/Schemas/Generated/cpp/Snapshot_generated.h:1827`의 `VerifySnapshotBuffer`가 오프셋 무결성을 검사하는 것이 텍스트 세계의 "문법 오류 검출"에 대응하고, `Client/Private/GamePlay/SchemaSmoke.cpp:42`처럼 `flatbuffers::Verifier`를 실제로 돌린다. 이 프로젝트의 에러 정책(복제 경계의 verify 실패는 침묵 금지 — 스키마 드리프트가 네트워크 정지처럼 보이는 사고 방지)은 "파서의 에러 보고 품질" 논의의 바이너리 버전인 셈이다.

### (3) HLSL 런타임 컴파일 — 남의 프런트엔드를 부르고 에러 메시지를 소비한다

셰이더는 진짜 컴파일러 프런트엔드를 런타임에 호출한다. `Engine/Private/RHI/RHIShaderCompiler.cpp:33`의 `D3DCompile` 호출이 HLSL 소스를 렉싱→파싱→의미분석→코드생성까지 태우고, 실패 시 에러 블롭을 받아 `pOutErrors`로 끌어올린다(46~53행). **이 챕터에서 배운 "에러 메시지 품질이 파서 설계의 제1 요구"를 소비자 입장에서 체감하는 지점** — 에러 블롭을 버리는 코드였다면 셰이더 오타 하나가 "화면에 안 나옴"이라는 무언의 증상이 됐을 것이다. 파서를 만들 때 좋은 진단이 왜 본업인지, 파서를 쓸 때 진단을 왜 반드시 표면화해야 하는지가 같은 원리의 양면이다.

### 면접에서 이 경험을 말하는 프레임

"컴파일러를 직접 만든 적은 없지만, 제 엔진에는 파싱에 대한 결정이 세 번 있었습니다. 자주 읽는 대용량 데이터(맵)는 파싱 자체를 없앤 바이너리로 하되 매직/버전 검증을 직접 설계했고, 스키마가 자주 바뀌는 네트워크 포맷은 flatc라는 파서+코드 생성기에 맡겨 파싱을 빌드 타임으로 밀어냈고, 셰이더는 D3DCompile이라는 남의 프런트엔드를 부르되 에러 메시지를 반드시 표면화했습니다. 파서를 배우고 나니 이 셋이 전부 '렉싱/파싱 비용과 검증 책임을 어디에 둘 것인가'라는 같은 질문의 다른 답이라는 걸 알게 됐습니다." — 원리(이 챕터)와 실물(레포)이 붙는 답변이다.
