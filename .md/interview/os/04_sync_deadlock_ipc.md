# 동기화 · 데드락 · IPC — OS 관점

> "뮤텍스와 세마포어의 차이는?", "데드락 4조건을 말하고 실전에서는 어떻게 막나?", "스핀락은 언제 쓰나?", "IPC 수단들을 비교하라" — 전부 이 챕터의 질문이다. 암기 목록이 아니라 "하드웨어 원자 명령 위에 기다리는 방법을 어떻게 쌓아 올렸는가"라는 하나의 설계 이야기로 답한다.

---

## ① 한 줄 본질

> **"모든 동기화 프리미티브는 하드웨어 원자 명령(CAS·exchange·LL/SC) 위에 '기다리는 정책'을 다르게 얹은 층위다. 스핀락은 CPU를 태우며 기다리고, 블로킹 락은 커널에 스레드를 재워 달라고 부탁한다. 예상 대기가 컨텍스트 스위치 2회 비용보다 짧으면 스핀이 이득이라, 현대의 락(CRITICAL_SECTION의 spin count, Linux futex)은 전부 '짧게 스핀하다 잠드는' 하이브리드다. 뮤텍스의 본질은 소유권, 세마포어는 계수, 조건 변수는 상태 대기이고 — 소유권의 유무가 우선순위 상속 가능 여부를 가른다. 데드락은 4가지 필요조건의 동시 성립이며 현업의 대응은 락 순서 규약으로 순환 대기를 원천 차단하는 예방이다. IPC는 '커널을 몇 번 거치고 동기화를 누가 책임지는가'의 스펙트럼이고, 게임 서버의 실전 원칙은 락을 잘게 쥐고, 상태마다 라이터를 하나로 만들고, 데이터 대신 소유권을 넘기는 것이다."**

## ② 기본 개념

### 경쟁 조건의 두 근원 — 인터리빙과 메모리 가시성

경쟁 조건(race condition)은 한 가지 병이 아니라 근원이 다른 두 가지 병이다.

**근원 1: 인터리빙(interleaving).** `counter++`는 소스에서 한 줄이지만 기계어로는 세 단계다.

```
스레드 A:  load r1,[cnt]        inc r1              store [cnt],r1
스레드 B:          load r2,[cnt]        inc r2               store [cnt],r2
시간 ──────────────────────────────────────────────────────────────▶
결과: +2가 아니라 +1.  B의 load가 A의 store 전에 끼어들어
     두 스레드가 같은 옛 값을 읽고 같은 새 값을 썼다.
```

스케줄러는 명령 사이 어디서든 스레드를 끊을 수 있으므로, "읽고-고치고-쓰는" 시퀀스는 원자적(atomic)이어야만 안전하다. 이 문제는 단일 코어 시절부터 있었다 — 그때는 커널이 인터럽트를 잠깐 끄는 것만으로도 해결됐다.

**근원 2: 메모리 가시성(memory visibility).** 멀티코어가 추가한 문제다. 각 코어는 쓰기를 곧바로 캐시에 내리지 않고 스토어 버퍼(store buffer)에 쌓아두며, CPU와 컴파일러는 성능을 위해 명령을 재배열(reordering)한다.

```
코어 0                              코어 1
 │ store x=1 ──▶ [스토어 버퍼]       │ store y=1 ──▶ [스토어 버퍼]
 │ load  y   → 0 (!)                │ load  x   → 0 (!)
 ▼                                  ▼
   L1 ◀───── 캐시 일관성(MESI 계열) ─────▶ L1
```

캐시 일관성(cache coherence) 프로토콜은 "값이 결국 모든 코어에 전파된다"까지만 보장한다. **"내 쓰기가 언제, 어떤 순서로 다른 코어에 보이는가"는 메모리 배리어(memory barrier) 없이는 보장되지 않는다.** 그래서 x=1과 y=1을 각자 쓴 뒤 서로를 읽으면 둘 다 0을 보는, 단일 스레드 직관으로는 불가능한 결과가 실제로 난다.

락이 답이 되는 이유는 이 둘을 동시에 해결하기 때문이다. 상호 배제로 인터리빙을 차단하고, 락 획득이 acquire 배리어·해제가 release 배리어로 동작해 "락 안에서 쓴 것은 다음 획득자에게 반드시 보인다"를 보장한다. **락은 스케줄러 장치이면서 동시에 메모리 배리어 장치다.**

| | 인터리빙 문제 | 가시성 문제 |
|---|---|---|
| 원인 | 스케줄러의 임의 선점 | 스토어 버퍼·재배열·캐시 전파 지연 |
| 등장 시기 | 단일 코어 멀티스레딩부터 | 멀티코어부터 본격화 |
| 증상 | lost update, 반쪽 상태 관측 | 썼는데 안 보임, 순서 뒤집혀 보임 |
| 해법 | 상호 배제(원자성) | 메모리 배리어(순서·전파 강제) |

### 층위 0 — 하드웨어 원자 명령: 모든 동기화의 지반

소프트웨어만으로 상호 배제를 만들려던 고전 알고리즘(데커, 피터슨)은 "메모리 접근이 프로그램 순서대로 보인다"를 전제하는데, 현대 CPU는 그 전제를 깨므로 배리어 없이는 그대로 성립하지 않는다. 그래서 하드웨어가 "읽고-비교하고-쓰기"를 한 덩어리로 보장하는 명령을 제공한다.

```cpp
// CAS(compare-and-swap)의 의미론 — 이 전체가 나눠질 수 없는 한 단위
bool CAS(int* addr, int expected, int desired) {
    if (*addr != expected) return false;   // 다른 놈이 먼저 바꿨다 → 실패
    *addr = desired;       return true;    // 내가 이겼다
}
```

| 명령 | 대표 구현 | 의미 |
|---|---|---|
| CAS | x86 `lock cmpxchg`, C++ `compare_exchange` | "예상값이 맞을 때만 교체" — 낙관적 갱신의 기본기 |
| exchange | x86 `xchg`, C++ `exchange` | "무조건 바꾸고 옛 값을 돌려받기" — 스핀락의 최소 재료 |
| LL/SC | ARM `LDREX/STREX` 계열 | "감시하며 읽기 → 그 사이 아무도 안 건드렸으면 쓰기" |

CAS와 LL/SC는 같은 목적의 두 유파다. CAS는 값만 비교하므로 A→B→A로 되돌아온 변화를 감지 못하는 ABA 문제가 있고, LL/SC는 주소에 대한 개입 자체를 감시하므로 ABA에 면역이지만 허위 실패가 가능하다. 어느 쪽이든 **"경쟁에서 정확히 한 명만 이긴다"를 하드웨어가 보증**하고, 그 위에 모든 것이 쌓인다.

```
[응용 계층]   뮤텍스 · 세마포어 · 조건 변수 · 이벤트      ← 기다리는 '정책'
     ▲
[하이브리드]  CRITICAL_SECTION(spin count) · futex        ← 짧게 스핀, 길면 잠들기
     ▲
[커널 블로킹] 대기 큐 + 스케줄러                          ← 스레드를 '재운다'
     ▲
[유저 스핀락] TAS / TTAS + backoff                        ← CPU를 '태운다'
     ▲
[하드웨어]    CAS(lock cmpxchg) · exchange(xchg) · LL/SC  ← 원자성의 지반
```

### 층위 1 — 스핀락: CPU를 태우며 기다린다

exchange 하나면 락이 된다. 이것이 TAS(test-and-set) 스핀락(spinlock)이다.

```cpp
std::atomic<int> lock{0};
void Lock()   { while (lock.exchange(1, std::memory_order_acquire)) { /* 스핀 */ } }
void Unlock() { lock.store(0, std::memory_order_release); }
```

문제는 경합 시 성능이다. `exchange`는 쓰기이므로 캐시 라인의 배타 소유권을 요구하고, 대기자 N명이 스핀하면 캐시 라인이 코어 사이를 핑퐁하며 버스를 포화시킨다. 그래서 실전 스핀락은 **TTAS(test-and-test-and-set)** — 먼저 일반 읽기로 풀리기를 구경하다가(읽기는 캐시 라인을 공유 상태로 두므로 조용하다), 풀린 것을 본 순간에만 exchange를 시도한다. 여기에 재시도 간격을 늘리는 백오프(backoff), x86 `pause` 힌트가 더해진다.

스핀의 치명적 약점은 **락 보유자가 스케줄 아웃됐을 때**다. 대기자는 CPU를 태우며 기다리는데 정작 풀어 줄 스레드는 CPU를 못 받는다. 최악은 같은 코어에서 보유자를 선점한 스레드가 스핀하는 경우 — 자기가 기다리는 상대를 자기가 굶긴다. 그래서 유저 공간에서 "순수 스핀락"은 거의 항상 잘못된 선택이고, 스핀은 커널(선점을 끌 수 있는 곳)이나 하이브리드의 앞단에서만 정당하다.

### 층위 2 — 블로킹 락: 커널에 재워 달라고 부탁한다

블로킹 락(blocking lock)은 획득 실패 시 시스템 콜로 커널에 진입해, 자신을 해당 락의 대기 큐(wait queue)에 걸고 스케줄러에서 빠진다. 해제자가 대기 큐에서 하나를 깨운다. CPU를 안 태우는 대신 **컨텍스트 스위치(context switch) 비용**을 낸다 — 커널 진입, 레지스터 저장/복원, 스케줄러 실행, 그리고 더 아픈 간접 비용인 캐시·TLB 오염.

**손익분기가 "컨텍스트 스위치 2회"인 이유:** 잠들면 스위치 1회(남에게 CPU를 넘김), 깨어나면 또 1회(내게 돌아옴) — 블로킹의 고정 비용이 최소 2회다. 락이 그보다 빨리 풀릴 거라면 그냥 스핀하며 기다리는 편이 싸다. 그래서 결론은 양자택일이 아니라 하이브리드다.

| | 스핀락 | 블로킹 락 | 하이브리드 |
|---|---|---|---|
| 대기 방식 | CPU 점유하며 재시도 | 커널 대기 큐에서 수면 | 짧게 스핀 → 잠들기 |
| 대기 비용 | 코어 1개 소모 | 스위치 2회 + 캐시 오염 | 상황에 따라 최소 쪽 |
| 유리한 구간 | 임계 구역이 극히 짧고 보유자가 실행 중 | 대기가 길거나 예측 불가 | 일반 목적 |
| 대표 | 커널 내부 스핀락 | 커널 뮤텍스 객체 | CRITICAL_SECTION, futex |

**하이브리드의 두 실물:**
- **Windows CRITICAL_SECTION** — `InitializeCriticalSectionAndSpinCount`로 스핀 횟수를 지정하면, 멀티코어에서 그만큼 스핀해 보고 안 되면 커널 대기로 전환한다. 커널 객체는 경합이 실제로 발생할 때만 동원되므로, 비경합 획득/해제는 유저 공간 원자 연산으로 끝난다.
- **Linux futex(fast userspace mutex)** — 락 워드는 유저 공간의 평범한 정수다. 비경합이면 CAS 한 번으로 잠그고 풀며 **커널 진입이 0회**다. 경합이 나면 그때만 `futex` 시스템 콜로 "이 주소 값이 이 값인 동안 재워 달라"고 부탁한다. glibc 뮤텍스·조건 변수가 전부 이 위에 서 있다. "빠른 경로는 유저 공간, 느린 경로만 커널"이라는 이 구조가 현대 동기화의 표준 설계다.

### 커널 동기화 객체 — 각각의 '본질' 하나씩

세부 API는 잊어도 된다. 객체마다 본질 단어 하나를 쥐고 있으면 나머지 성질이 전부 유도된다.

**뮤텍스(mutex) = 소유권(ownership).** 잠근 스레드만 풀 수 있다. 소유자가 명확하니까 가능한 것들 — 재귀 잠금 검출, 소유 스레드가 죽었을 때의 감지(Windows abandoned mutex), 그리고 **우선순위 상속(priority inheritance): "누구를 승격시킬지"를 알 수 있다.** Windows에서 뮤텍스는 이름을 붙여 프로세스 간에도 쓰는 커널 객체이고, 프로세스 내부용 경량 락(CRITICAL_SECTION)과 구분된다.

**세마포어(semaphore) = 계수(counting).** "자원이 몇 개 남았나"라는 정수와, 0이면 기다리는 규칙(다익스트라의 P/V)이 전부다. **소유자가 없다** — 획득한 스레드와 다른 스레드가 신호를 올려도 정당하다(생산자가 올리고 소비자가 내리는 생산자-소비자가 바로 그 용법). 소유자가 없으므로 우선순위 상속이 원리적으로 불가능하다 — 누구를 승격해야 할지 시스템이 알 수 없다.

| | 뮤텍스 | 세마포어(이진) |
|---|---|---|
| 본질 | 임계 구역의 소유권 | 신호/자원의 계수 |
| 해제 주체 | 소유 스레드만 | 아무 스레드나 |
| 우선순위 상속 | 가능(소유자가 명확) | 불가능(소유자 개념 없음) |
| 오용 감지 | 재귀·비소유 해제·유기 감지 가능 | 감지 근거 자체가 없음 |
| 용도 | 상호 배제 | 자원 계수, 스레드 간 신호 |

값이 0/1로 같아 보여도 **의미론이 다르다**는 것이 면접 포인트다. "이진 세마포어 = 뮤텍스"라고 답하면 소유권과 상속을 모른다는 신호다.

**조건 변수(condition variable) = 상태 대기.** "큐가 비어 있지 않을 때까지"처럼 임의의 술어(predicate)를 기다린다. 계약이 정교하다: 뮤텍스를 잡은 채 `wait`를 부르면 **unlock과 잠들기가 원자적으로** 일어나고, 깨어날 때 뮤텍스를 다시 잡고 돌아온다. 이 원자성이 없으면 "unlock한 직후, 잠들기 직전"에 신호가 지나가 버리는 lost wakeup이 난다.

그리고 **허위 각성(spurious wakeup)** — 신호 없이도 깨어날 수 있음이 표준(POSIX, C++)에 명세돼 있다. 왜 이런 불편을 허용하나? (1) 구현 자유도: futex 대기는 시그널 등으로 조기 리턴할 수 있고, 이를 완벽히 숨기려면 구현이 비싸진다. (2) 어차피 재검사는 필수다: 깨어나서 뮤텍스를 다시 잡기까지의 틈에 다른 스레드가 조건을 먼저 소비할 수 있다(wake 경쟁). 그래서 표준은 "깨어남 = 조건 성립"이라는 보장을 아예 포기하고, **while 재검사를 사용자 계약으로** 만들었다.

```cpp
std::unique_lock lk(m);
cv.wait(lk, []{ return ready; });   // 내부적으로: while (!ready) wait(lk);
// if (!ready) wait(lk); 는 허위 각성/wake 경쟁 한 방에 깨진다
```

조건 변수의 온전한 모습은 항상 3인조다 — **상태(술어) + 그 상태를 지키는 뮤텍스 + 조건 변수.** 생산자-소비자로 보면:

```cpp
// 소비자                                  // 생산자
std::unique_lock lk(m);                    {
cv.wait(lk, []{ return !q.empty(); });         std::lock_guard lk(m);
auto item = std::move(q.front());              q.push(std::move(item));   // 상태 갱신은
q.pop();                                   }                              // 반드시 락 안에서
lk.unlock();                               cv.notify_one();               // 신호는 락 밖도 가능
// item 처리는 락 밖에서
```

조건 변수 자체는 상태를 하나도 저장하지 않는다는 점이 자주 나오는 함정이다 — 기다리는 스레드가 없을 때의 `notify`는 허공에 사라진다(세마포어는 카운트로 신호가 축적되는 것과 대조적이다). 그래서 술어가 진실의 원본이고, 조건 변수는 "술어를 다시 확인할 타이밍"만 알려 주는 장치다.

**Windows 이벤트(event)**도 상태 대기 계열이다(manual-reset은 신호가 유지되고, auto-reset은 대기자 하나를 통과시키며 꺼진다). 다만 이벤트는 술어(실제 상태)가 객체 밖에 따로 살아서, "상태 갱신과 신호"를 원자적으로 묶는 책임이 온전히 사용자에게 온다. 조건 변수가 뮤텍스와 한 몸으로 설계된 이유가 바로 이 실수 여지의 제거다.

### 데드락 — 4가지 필요조건, 그리고 현업의 선택

데드락(deadlock)은 Coffman의 4조건이 **동시에 전부** 성립할 때만 발생한다. 뒤집으면, 하나만 깨면 원천 차단된다.

| 조건 | 뜻 | 깨는 방법 |
|---|---|---|
| 상호 배제(mutual exclusion) | 자원을 한 번에 하나만 사용 | 락 없는 설계(불변 데이터, 원자 연산) |
| 점유 대기(hold and wait) | 하나 쥔 채 다음을 기다림 | 필요한 락을 한 번에 전부 획득 |
| 비선점(no preemption) | 뺏을 수 없음 | try_lock 실패 시 쥔 것을 놓고 재시도 |
| 순환 대기(circular wait) | 대기 그래프에 사이클 | **락 전역 순서 규약** |

```
   T1 ──(A 보유)──▶ B를 기다림
    ▲                   │
    │                   ▼
   A를 기다림 ◀──(B 보유)── T2      ← 사이클 완성 = 영원한 정지
```

대응은 교과서적으로 세 갈래인데, 실전 배분이 명확하다.

**예방(prevention) — 현업 표준.** 모든 코드가 락을 같은 전역 순서로만 잡게 하면(항상 A→B, 절대 B→A 없음) 대기 그래프에 사이클이 생길 수 없다. 비용이 0이고, 코드 리뷰에서 지역적으로 검증 가능하다는 것이 결정적 장점이다. 순서를 정하는 실용 기법은 여러 가지다 — 락에 계층 번호를 부여하고 "낮은 번호를 쥔 채 높은 번호만 잡을 수 있다"를 규약(또는 디버그 빌드의 assert)으로 강제하거나, 동종 객체 두 개를 잡아야 하면 주소나 ID가 작은 쪽부터 잡는다. C++ `std::scoped_lock(a, b)`는 보완 장치다 — 내부의 `std::lock` 알고리즘이 try-lock과 양보를 섞어, 순서 규약 없이도 두 락 동시 획득에서 교착하지 않는다.

번외로, 데드락의 가장 흔한 첫 경험은 두 스레드가 아니라 **한 스레드짜리 자기 교착(self-deadlock)**이다 — 락을 쥔 함수가 같은 락을 잡는 함수를 호출하는 경우. 재귀 뮤텍스로 덮을 수도 있지만, 대개는 "락을 잡는 공개 함수 / 락을 전제하는 내부 함수"로 계층을 나누는 편이 설계를 덜 흐린다.

**회피(avoidance) — 개념만.** 은행가 알고리즘(Banker's algorithm)은 "이 할당을 해줘도 모두가 완주 가능한 안전 상태(safe state)가 유지되는가"를 매번 검사한다. 실전에서 못 쓰는 이유: 모든 스레드의 **최대 자원 요구량을 사전에 선언**해야 하고, 할당마다 검사 비용을 내며, 자원 종류가 동적으로 변하는 실제 시스템과 맞지 않는다. 면접에서는 "왜 안 쓰는지"까지 말해야 아는 것이다.

**탐지/회복(detection & recovery) — DB의 영역.** 대기 그래프(wait-for graph)의 사이클을 주기적으로 탐지하고, 발견 시 희생자(victim)를 골라 되돌린다. DBMS가 이걸 실제로 하는 이유는 **트랜잭션이라는 롤백 단위**가 있어서다. 일반 앱의 락에는 롤백이 없다 — 탐지해도 회복 수단이 없으므로, 앱은 예방으로 간다. "예방은 앱, 탐지는 DB"라는 배분 자체가 설계 논리다.

### 우선순위 역전 — Mars Pathfinder가 남긴 교훈

우선순위 역전(priority inversion): 낮은 우선순위 스레드가 쥔 락을 높은 우선순위 스레드가 기다리는 동안, **중간 우선순위 스레드가 낮은 놈을 선점해 버리면** 높은 놈이 중간 놈에게 사실상 밀린다.

1997년 화성 탐사선 Mars Pathfinder에서 실제로 터졌다.

```
우선순위 高  버스 관리 태스크 ── 정보 버스 뮤텍스 대기 ──────── (블록)
        中  통신 태스크 ─────── 오래 실행되며 CPU 점유 ───────▶
        低  기상 데이터 태스크 ─ 뮤텍스 보유, 中에 선점돼 CPU 못 받음
결과: 高가 기한 내 실행 못 함 → 워치독 타임아웃 → 시스템 전체 리셋 반복
```

수정은 코드 재설계가 아니라 **VxWorks 뮤텍스의 우선순위 상속 옵션을 켜는 것**이었고, 그 패치가 지구에서 화성으로 전송됐다. 우선순위 상속은 락 보유자를 "대기 중인 최고 우선순위"로 임시 승격시켜 中이 低를 선점하지 못하게 한다. 변형으로 우선순위 천장(priority ceiling) — 락마다 천장 우선순위를 미리 정해 획득 즉시 그 수준으로 올리는 방식 — 이 있으며, 실시간 OS에서는 이 프로토콜들이 표준 장비다. 범용 데스크톱 OS는 엄밀한 상속 대신 오래 굶은 스레드의 우선순위를 일시적으로 올려 주는 부스팅 계열 휴리스틱으로 완화하는 경향이 있다 — 데드라인 보장이 아니라 평균적 완화라는 점이 실시간 OS와의 철학 차이다.

이 사례가 곧 "뮤텍스 vs 세마포어"의 마지막 퍼즐이다. **상속은 소유자가 있어야 가능하다.** 세마포어였다면 시스템은 누구를 승격시킬지 알 수 없었다.

### IPC — 커널을 몇 번 거치고, 동기화는 누가 책임지는가

프로세스는 가상 주소 공간으로 **격리가 기본값**이다(스레드는 반대로 공유가 기본값이라, 스레드의 문제가 '동기화'라면 프로세스의 문제는 '통로'다). 격리를 뚫는 통로는 전부 커널이 중개하며, 프로세스 간 통신(inter-process communication, IPC) 수단의 차이는 결국 세 질문으로 정리된다: 데이터가 몇 번 복사되는가, 메시지 경계가 보존되는가, 동기화를 누가 책임지는가.

| 수단 | 복사 횟수 | 경계 | 동기화 책임 | 도달 범위 |
|---|---|---|---|---|
| 파이프(pipe) | 2회 (유저→커널→유저) | 바이트 스트림 | 커널(버퍼가 흐름 제어) | 같은 머신, 익명은 부모-자식 |
| 공유 메모리(shared memory) | **0회** (매핑 후 직접 접근) | 없음 — 그냥 메모리 | **전적으로 사용자** | 같은 머신 |
| 소켓(socket) | 2회+ (프로토콜 스택 경유) | TCP=스트림 / UDP=데이터그램 | 커널 + 프로토콜 | **네트워크 경계를 넘는 유일한 수단** |
| 메시지 큐(message queue) | 2회 | **메시지 단위 보존** | 커널 | 같은 머신 |

```
파이프:      P1 유저 버퍼 ──copy──▶ [커널 파이프 버퍼] ──copy──▶ P2 유저 버퍼
공유 메모리:  P1 ──▶ [ 같은 물리 페이지를 양쪽에 매핑 ] ◀── P2      (copy 0회.
             대신 "P2가 언제 읽어도 되는가"를 세마포어 등으로 직접 풀어야 함)
```

관통하는 트레이드오프는 하나다. **복사를 줄일수록 동기화 책임이 사용자에게 넘어온다.** 파이프·메시지 큐는 복사 2회를 내는 대신 커널이 생산자-소비자 동기화(버퍼 참/빔 대기)를 대신 해 준다. 공유 메모리는 복사가 0회지만 경쟁 조건 방지를 사용자가 프로세스 간 뮤텍스/세마포어로 전부 책임진다 — 스레드 동기화의 모든 문제가 더 어려운 형태(프로세스가 죽으면 락은 누가 푸나?)로 돌아온다. 경계 보존도 같은 논리다: 스트림(파이프, TCP)은 "어디까지가 한 메시지인가"를 사용자가 길이 프리픽스 등으로 다시 만들어야 하고, 메시지 큐는 커널이 보존해 주는 대신 그 관리 비용을 낸다.

선택 기준을 결정 트리로 접으면:

```
머신 경계를 넘는가? ── 예 ─▶ 소켓 (선택의 여지 없음)
        │
        아니오
        ▼
대용량을 초저지연으로 반복 전달? ── 예 ─▶ 공유 메모리 (+동기화 직접 설계)
        │
        아니오
        ▼
메시지 경계 보존이 필요한가? ── 예 ─▶ 메시지 큐   │ 아니오 ─▶ 파이프
```

게임 서버가 이 스펙트럼에서 서 있는 자리는 명확하다. 클라이언트-서버는 머신 경계를 넘어야 하므로 **소켓이 유일한 선택**이고, 스트림(TCP) 위에서 패킷 경계를 길이 프리픽스로 재구성한다. 서버 내부는 프로세스를 쪼개지 않고 **단일 프로세스 멀티스레드** — 주소 공간을 공유하니 IPC 자체가 필요 없고, 대신 동기화 설계가 전부가 된다. 스레드 간 통신도 IPC의 축으로 읽을 수 있다는 것이 이 관점의 소득이다: 뒤에 나올 "락으로 공유" 대 "큐로 소유권 이전"은 프로세스 세계의 "공유 메모리 대 메시지 큐"와 정확히 같은 트레이드오프의 스레드판이다.

### 게임 서버 실전 원칙 — 결국 "공유 가변 상태를 줄여라"의 세 표현

1. **락 구간 최소화.** 락 안에서는 포인터 교환·스왑 같은 O(1) 작업만 하고, 정렬·직렬화·할당 같은 무거운 일은 락 밖에서 한다. 락 보유 시간이 곧 다른 스레드의 대기 시간이다.
2. **싱글 라이터(single writer).** 상태 조각마다 쓰는 스레드를 하나로 정하면 쓰기-쓰기 경합이 설계 차원에서 사라지고, 남는 문제는 "라이터→리더 전달"이라는 훨씬 쉬운 문제로 줄어든다.
3. **메시지 패싱으로 소유권 이전.** 데이터를 공유한 채 락으로 보호하지 말고, 소유권째 넘겨라(C++ move가 이 관용구의 언어 지원이다). 넘긴 쪽이 더는 접근하지 않으면 그 데이터에는 락이 필요 없다.

셋은 사실 같은 문장이다: **동기화를 잘하는 최고의 방법은 동기화할 대상을 없애는 것이다.**

## ③ 심화 — 꼬리질문 대비

**"CAS 하나로 뮤텍스·세마포어를 전부 만들 수 있는 근거는?"**
CAS는 "여러 스레드 중 정확히 한 명이 이긴다"는 합의(consensus)를 하드웨어로 보증한다(Herlihy의 합의 수 이론에서 CAS는 임의 개수 스레드의 합의가 가능한 만능 프리미티브다). 락 상태 전이(0→1)를 CAS로 만들고, 그 위에 '지면 어떻게 기다리는가'(스핀/수면)와 '풀 때 누구를 깨우는가'만 정책으로 얹으면 뮤텍스·세마포어·조건 변수가 전부 유도된다.

**"TAS보다 TTAS가 빠른 이유를 캐시로 설명하라."**
`exchange`는 쓰기라서 캐시 라인의 배타(exclusive) 소유권을 매번 요구하고, 대기자들이 이를 반복하면 라인이 코어 사이를 핑퐁한다. TTAS는 먼저 일반 읽기로 스핀한다 — 읽기는 라인을 공유(shared) 상태로 각자 캐시에 두므로 버스가 조용하고, 해제(쓰기)가 발생해 무효화됐을 때만 exchange를 시도한다.

**"스핀 손익분기가 왜 컨텍스트 스위치 '2회'인가?"**
블로킹의 고정 비용이 최소 2회이기 때문이다 — 잠들 때 남에게 넘어가는 스위치 1회, 깨어날 때 내게 돌아오는 스위치 1회. 락이 그 합보다 빨리 풀릴 것 같으면 스핀이 이득이다. 미래 대기 시간은 알 수 없으므로 실전은 "스위치 2회 비용어치만 스핀하고 잠들기"라는 하이브리드로 수렴한다.

**"spurious wakeup은 왜 명세에 존재하나? if로 기다리면 무슨 일이 생기나?"**
구현 자유도(대기 시스템 콜의 조기 리턴을 완벽히 숨기는 비용 제거)와, 어차피 wake 경쟁 때문에 재검사가 필수라는 현실을 표준이 인정한 것이다. 깨어나 뮤텍스를 재획득하기까지의 틈에 다른 스레드가 조건을 소비할 수 있으므로 "깨어남 = 조건 성립"은 원래 보장이 안 된다. `if`로 기다리면 허위 각성이나 wake 경쟁 한 번에 조건 불성립 상태로 진행해 버린다 — 빈 큐에서 pop하는 식의 붕괴다. 반드시 `while`(또는 술어 오버로드)이다.

**"뮤텍스와 이진 세마포어는 값이 0/1로 같은데 정확히 뭐가 다른가?"**
소유권이다. 뮤텍스는 잠근 스레드만 풀 수 있고, 그 소유자 정보가 재귀 검출·유기 감지·우선순위 상속을 가능케 한다. 세마포어는 소유자가 없어 아무나 signal할 수 있고(그래서 스레드 간 신호 전달에 쓰이고), 같은 이유로 우선순위 상속이 원리적으로 불가능하다. "상호 배제 의도면 뮤텍스, 계수/신호 의도면 세마포어"가 정답 프레임이다.

**"락을 쥔 스레드가 죽으면 어떻게 되나?"**
뮤텍스는 소유자가 있으므로 감지 장치가 존재한다 — Windows 커널 뮤텍스는 다음 대기자에게 유기(abandoned) 상태를 알려 주고, POSIX에도 robust mutex 계열 장치가 있다. 다만 "락은 풀렸지만 보호되던 데이터가 반쪽 상태"라는 문제는 남아, 감지 후 일관성 복구는 여전히 사용자 몫이다. 세마포어는 소유자 개념이 없어 감지 근거 자체가 없다 — 카운트 하나가 영원히 증발한다.

**"락프리(lock-free)가 락보다 항상 빠른가?"**
아니다. 락프리는 성능 약속이 아니라 진행 보장(progress guarantee) — "누군가는 반드시 전진한다, 락 보유자가 스케줄 아웃돼도 전체가 멈추지 않는다"는 약속이다. 고경합에서는 CAS 실패-재시도 폭풍이 락보다 느릴 수 있고, 메모리 회수(누가 언제 노드를 지워도 되는가) 문제가 추가된다. 짧은 임계 구역의 잘 설계된 락이 이기는 경우가 흔하다.

**"데드락·라이브락·기아를 구분하라."**
데드락은 서로를 기다리며 **아무도 상태를 바꾸지 않는** 영구 정지. 라이브락(livelock)은 서로 양보/재시도를 반복하며 **상태는 계속 바뀌는데 전진이 없는** 상태(try_lock 실패 시 모두가 놓고 동시에 재시도하는 패턴 — 무작위 백오프로 깬다). 기아(starvation)는 시스템 전체는 전진하는데 **특정 스레드만** 계속 밀리는 것(우선순위 역전이 대표 사례)이다. 진단이 다르니 처방도 다르다: 데드락은 순서 규약, 라이브락은 무작위성, 기아는 공정성(fairness)/상속.

**"은행가 알고리즘을 현업에서 안 쓰는 이유는?"**
모든 스레드의 최대 자원 요구를 사전에 선언해야 하는데 일반 프로그램은 그걸 모르고, 할당 요청마다 안전 상태 검사를 돌리는 비용이 들며, 자원의 종류·수량이 동적으로 변하는 실제 시스템 모델과 맞지 않는다. 가치는 "안전 상태"라는 사고 틀이고, 실전 자리는 락 순서 규약(예방)이 가져갔다.

**"공유 메모리가 복사 0회로 제일 빠른데, 왜 다들 파이프·소켓을 쓰나?"**
복사와 함께 커널의 동기화 서비스도 사라지기 때문이다. "언제 읽어도 되는가"를 프로세스 간 세마포어 등으로 직접 설계해야 하고, 상대 프로세스가 죽었을 때의 락 정리·반쪽 쓰기 복구까지 떠안는다. 처리량이 그 복잡도를 정당화하는 좁은 구간(대용량 프레임 전달 등)에서만 이기는 선택이고, 대부분의 통신은 커널이 경계와 동기화를 대신 해 주는 수단이 총비용에서 싸다. 그리고 머신 경계를 넘어야 하면 애초에 소켓 외의 선택지가 없다.

**"C++ volatile로 스레드 동기화가 되나?"**
안 된다. C++ `volatile`은 "컴파일러가 이 접근을 최적화로 제거·캐싱하지 말라"는 지시일 뿐, 원자성도 스레드 간 순서 보장도 없다(하드웨어 재배열을 막지 않는다). 용도는 메모리 맵 I/O 같은 곳이고, 스레드 간에는 `std::atomic`과 memory_order가 답이다. Java/C#의 volatile은 가시성·순서 의미가 있는 다른 물건이라 혼동이 잦다 — 언어를 명시해서 답해야 한다.

**"락이 메모리 가시성까지 해결하는 원리는?"**
락 획득이 acquire, 해제가 release 의미론을 갖기 때문이다. release는 "이 앞의 모든 쓰기를 이 지점 앞으로 완료시켜 공개"하고, acquire는 "이 뒤의 모든 읽기가 이 지점 뒤에서 수행됨"을 보장한다. 그래서 A가 락 안에서 쓰고 풀면, 같은 락을 잡은 B는 그 쓰기를 반드시 본다 — 상호 배제와 가시성이 한 쌍으로 묶인 계약이 락이다.

**"futex가 '혁신'이라 불리는 이유를 한 문장으로?"**
동기화의 흔한 경우(비경합)에서 커널을 완전히 배제했기 때문이다 — 락 워드를 유저 공간에 두고 CAS로 잠그며, 커널은 경합이라는 드문 경우에만 "이 주소로 재워/깨워 달라"는 대기 큐 서비스로 등장한다. "fast path는 유저 공간, slow path만 커널"이라는 이 배치가 이후 모든 현대 락 설계의 원형이 됐다.

**"비동기 I/O(IOCP) 모델을 쓰면 동기화가 필요 없어지지 않나?"**
반대다 — 완료 통지가 스레드 풀의 **아무 스레드에서나** 도착하므로, 세션 하나의 상태(송신 큐, 수신 파서)를 여러 완료 스레드가 만질 수 있다. 비동기가 없애 주는 것은 '블로킹 대기'이지 '공유 상태'가 아니다. 그래서 실전 설계는 세션별 락이나 "이 세션의 이벤트는 항상 같은 실행 단위로 직렬화"하는 장치(스트랜드 개념)를 얹는다. 게다가 미완료 비동기 연산이 참조하는 버퍼의 수명 관리라는, 동기 I/O에는 없던 문제가 추가된다.

**"메시지 패싱이 락 공유보다 검증하기 쉬운 이유는?"**
불변식이 지역적이기 때문이다. 락 공유는 "이 데이터를 만지는 **모든** 코드가 같은 락을 잡는가"라는 전역 성질을 검증해야 하고, 한 군데의 누락이 전체를 깨뜨린다. 소유권 이전은 "넘긴 뒤에는 접근하지 않는다"라는 규칙만 지키면 되고, 이는 큐 경계 한 곳만 보면 검증된다. C++에서는 move 이후 사용을 코드 리뷰·정적 분석이 잡기도 쉽다. Rust가 이 규칙을 아예 타입 시스템(소유권 이동)으로 강제한 것도 같은 통찰의 언어화다.

## ④ Winters에서의 적용

이 프로젝트의 동시성 설계는 위 원리의 실물 배치다. 층위 순서대로 짚는다.

### (1) 하드웨어 층위 — Chase-Lev 워크스틸링 데크와 '소유권 규약'

`Engine/Public/Core/JobSystem/WorkStealingDeque.h:9`의 헤더 주석이 이 자료구조의 계약을 선언한다 — 주인(owner)만 bottom에서 Push/Pop(LIFO), 도둑(thief)은 top에서 Steal(FIFO). `Push`(30행)의 "owner 만 호출" 주석은 장식이 아니라 알고리즘의 전제다: bottom을 만지는 스레드가 하나뿐이라는 가정 위에서만 Push가 락도 CAS도 없이 성립한다.

경합이 실제로 발생하는 유일한 지점 — 데크에 항목이 1개 남아 owner의 Pop과 thief의 Steal이 충돌하는 순간 — 만 CAS로 판정한다(`WorkStealingDeque.h:57~63`, `compare_exchange_strong`으로 top을 t→t+1). "정확히 한 명만 이긴다"는 하드웨어 보증을 알고리즘의 마지막 심급으로만 쓰는, 층위 0의 정확한 소비다. `Pop`의 `atomic_thread_fence(seq_cst)`(47행)는 ②절의 가시성 문제 그 자체다 — "bottom을 내렸다"는 내 쓰기가 상대에게 공개되기 전에 top을 읽어 버리면(스토어 버퍼) owner와 thief가 같은 항목을 둘 다 가져갈 수 있어, 쓰기→읽기 순서를 펜스로 강제한다. `alignas(64)`로 bottom/top을 캐시 라인 분리(91~92행)한 것은 TTAS 절에서 본 캐시 라인 경합의 예방이다.

소유권 규약은 주석이 아니라 코드 경로로 강제된다. `Engine/Private/Core/JobSystem.cpp:146~158`의 EnqueueJob은 현재 스레드가 워커 자신일 때만(`t_iWorkerIdx` 판정) 자기 데크에 Push하고, 외부 스레드나 오버플로는 뮤텍스로 보호되는 전역 큐로 우회시킨다. 초기 버전에서 외부 Submit이 워커 데크에 직접 push해 owner 규약을 깨는 경합이 있었고, 이 라우팅이 그 사후 수정이다 — "락프리 자료구조의 안전은 알고리즘이 아니라 **호출 규약**에 있다"를 몸으로 배운 지점.

### (2) 하이브리드 대기 — 스핀 대신 '짧은 타임아웃 수면'

워커 루프(`JobSystem.cpp:178~188`)는 잡이 없을 때 yield 스핀 대신 `wait_for(1ms)`로 잠든다. 코드 주석(182~184행)이 트레이드오프를 그대로 적고 있다: yield 스핀은 외부 CPU 부하와 경쟁하며 코어를 태우고, 짧은 타임아웃 대기는 깨우기 신호 누락이 있어도 1ms 안에 회복된다. 이는 조건 변수 정석(술어+while)과 다른 의도적 선택이다 — 술어를 뮤텍스로 감싸는 대신, lost wakeup을 **허용하고 타임아웃으로 상한을 거는** 쪽을 택했다. 정확성의 짐을 시간 상한으로 바꾼 하이브리드 설계로, "대기 정책은 정답이 하나가 아니라 비용 배치"라는 ②절의 논지를 실증한다. 일감 탐색 순서(자기 데크 Pop → 전역 큐 → 무작위 victim Steal, `JobSystem.cpp:212~247`)도 "락 없는 빠른 경로 먼저, 뮤텍스 경로는 다음"이라는 futex식 배치다.

### (3) 락 구간 최소화 + 소유권 이전 — CommandIngress

서버의 명령 수신 큐가 ②절 실전 원칙 1·3의 교과서 구현이다. IOCP 수신 스레드들(생산자 여럿)은 `Server/Private/Game/CommandIngress.cpp:64`의 뮤텍스 아래에서 pending 벡터에 명령을 적재하고, 같은 락 안에서 동일 세션의 옛 Move를 최신 Move로 치환하는 병합(74~85행)까지만 한다. 소비자인 틱 스레드의 `DrainSorted`(90~96행)는 **락 안에서 `swap` 한 번만** 수행하고, 정렬(비용이 큰 작업)은 락 밖에서 한다 — 락 보유 시간이 명령 수와 무관한 O(1)이다.

swap으로 빠져나온 벡터는 소유권째 틱 스레드로 넘어온다. 이후 정렬·실행 어디에도 락이 없다 — 생산자들은 이미 새 벡터를 보고 있으므로 이 데이터에 접근할 스레드가 하나뿐이기 때문이다. "데이터 공유 대신 소유권 이전"이 뮤텍스 한 개 규모에서도 성립함을 보여 준다.

### (4) 싱글 라이터 — 30Hz 틱 스레드가 월드의 유일한 라이터

`Server/Private/Game/GameRoomTick.cpp:68~80`의 TickThread는 33,333µs(30Hz) 고정 주기로 돌고, `Tick()`(82~111행)이 명령 드레인 → 봇 AI → 명령 실행 → 시뮬레이션 → 스냅샷 브로드캐스트를 한 스레드에서 순서대로 수행한다. 게임 월드(ECS 컴포넌트)를 **쓰는 스레드는 이 틱 스레드 하나뿐**이고, IOCP 스레드들은 위 (3)의 ingress 큐까지만 접근한다. 스레드 간 경계가 "큐 하나"로 좁혀졌으므로, 월드 내부 로직 전체가 락 없이 단일 스레드 코드처럼 작성된다. `m_stateMutex`(84행)는 월드 데이터 보호가 아니라 로비→인게임 같은 방 수명 상태 전이와 틱의 상호 배제라는 좁은 역할만 갖는다.

### (5) IOCP 송신 직렬화 — "outstanding WSASend는 항상 1개" 불변식

`Server/Private/Network/Session.cpp:100~105`의 Send는 `m_sendMutex`(선언은 `Server/Public/Network/Session.h:75`)로 송신 큐를 보호하되, `m_bSendPending` 플래그로 **미완료 WSASend가 세션당 최대 1개**라는 불변식을 지킨다 — 이미 송신 중이면 큐에만 넣고 리턴하고, 완료 콜백 `OnSendComplete`(134~147행)가 큐의 다음 패킷을 이어서 발행한다. 비동기 송신을 아무 스레드나 중첩 발행하면 (a) 완료 순서가 뒤섞여 TCP 스트림이 오염되고 (b) WSABUF가 가리키는 버퍼 수명 관리가 불가능해진다. 뮤텍스가 큐(데이터)를 지키고, 플래그가 발행(프로토콜 순서)을 직렬화하는 — 동기화 대상이 데이터만이 아니라 **순서 불변식**이기도 하다는 예다.

### (6) 락 없는 교착 — 콤보 AI 데드락과 진행 보장

최근 커밋 3847f3f("fix combo deadlock")가 고친 버그는 뮤텍스가 하나도 없는 데드락이다. 봇의 콤보 스텝이 사거리 안에 있는데 게이트 조건(쿨다운, 스킬 강탈로 인한 override 등)이 영원히 충족되지 않으면, AI는 그 스텝을 무한 대기하며 정지했다. 구조적으로 Coffman 조건의 논리 버전 — "현재 스텝을 점유한 채(점유 대기) 풀리지 않는 조건을 기다림(비선점)" — 이다. 수정(`Shared/GameSim/Systems/ChampionAI/ChampionAISystem.cpp:1693~1703`)은 주석 그대로 "정지 대신 다음 step으로 진행해 콤보가 교착되지 않게 한다" — 대기에 **진행 보장을 부여**해 조건 하나를 깬 것이다. 데드락 4조건 프레임이 락 자료구조 밖의 상태 기계에도 그대로 적용된다는 실전 사례로, 면접에서 "데드락을 실제로 겪고 고쳐 봤는가"에 답할 수 있는 지점이다.

### (7) 결정론 — 동기화 문제를 '없애는' 최상위 전략

Shared/GameSim의 결정론 시뮬레이션은 동기화의 관점에서 보면 극단적 선택이다: 시뮬 틱을 단일 스레드로 고정해 인터리빙이라는 변수 자체를 제거했다. 결정론은 "같은 입력 → 비트 단위로 같은 결과"를 요구하는데, 멀티스레드에서는 스케줄러의 인터리빙이 실행마다 달라 이 보장이 무너지기 쉽다. 나아가 순회 순서까지 통제한다 — `Shared/GameSim/Systems/DeterministicEntityIterator/DeterministicEntityIterator.h:14~25`의 `CollectSorted`는 엔티티를 모아 ID로 정렬한 뒤 순회해, 내부 컨테이너의 순서 우연성마저 배제한다. **"락이 없는 코드"가 아니라 "공유 가변 상태가 없는 코드"**라는 점이 요지이고, (4)의 싱글 라이터와 합쳐 보면 이 서버의 동시성 전략은 일관된다: 경합이 필요한 좁은 경계(ingress 큐, 송신 큐, 잡 데크)에만 동기화를 배치하고, 게임 로직 본체에서는 동기화가 등장할 이유를 설계로 제거한다.

## ⑤ 면접 Q&A

**Q1. 뮤텍스와 세마포어의 차이를 설명하라.**
- 핵심: 값의 범위(1 vs N)가 아니라 **소유권**이 본질이다. 뮤텍스는 잠근 스레드만 풀 수 있는 임계 구역의 소유권이고, 세마포어는 소유자 없는 계수/신호다. 소유권이 있어야 재귀 검출·유기 감지·우선순위 상속이 가능하다.
- 꼬리 대비: "그럼 이진 세마포어로 뮤텍스를 대신하면?" → 동작은 흉내 내도 오용 감지와 상속을 전부 잃는다. 의도(배제 vs 신호)에 맞는 물건을 골라야 한다.
- Winters: 잡 시스템의 워커 깨우기는 신호 계열(조건 변수), ingress 큐 보호는 배제 계열(뮤텍스) — 역할별로 다른 프리미티브를 쓴다.

**Q2. 스핀락과 뮤텍스 중 무엇을 언제 쓰나?**
- 핵심: 예상 대기가 컨텍스트 스위치 2회(잠들기+깨어나기) 비용보다 짧으면 스핀, 길거나 예측 불가면 블로킹. 미래를 모르므로 실전은 하이브리드(CRITICAL_SECTION spin count, futex) — 짧게 스핀하다 잠든다.
- 꼬리 대비: "유저 공간 순수 스핀락의 위험은?" → 락 보유자가 스케줄 아웃되면 대기자가 CPU만 태운다. 스핀이 정당한 곳은 선점을 통제할 수 있는 커널이나 하이브리드의 앞단.
- Winters: 워커 유휴 대기를 yield 스핀에서 `wait_for(1ms)`로 바꾼 결정(JobSystem.cpp:182~186의 주석)이 정확히 이 트레이드오프의 기록이다.

**Q3. 데드락 4조건을 말하고, 실전에서 어떻게 막는지 답하라.**
- 핵심: 상호 배제·점유 대기·비선점·순환 대기가 **동시에** 성립해야 발생 → 하나만 깨면 예방. 현업 표준은 락 전역 순서 규약으로 순환 대기를 차단하는 것 — 비용 0, 리뷰로 검증 가능.
- 꼬리 대비: "은행가 알고리즘은 왜 안 쓰나?" → 최대 요구량 사전 선언 불가 + 할당마다 검사 비용. "탐지/회복은 누가 쓰나?" → 롤백 단위(트랜잭션)가 있는 DB.
- Winters: 락 데드락은 아니지만 콤보 AI의 논리 교착(3847f3f)을 "진행 보장 부여"로 고친 경험 — 4조건 프레임이 상태 기계에도 적용된다는 실례로 답한다.

**Q4. spurious wakeup이 뭐고 왜 while로 기다려야 하나?**
- 핵심: 조건 변수는 신호 없이도 깨어날 수 있음이 표준에 명세돼 있고(구현 자유도), 신호가 있었어도 뮤텍스 재획득 전에 다른 스레드가 조건을 소비할 수 있다(wake 경쟁). 그래서 "깨어남 = 조건 성립"이 아니며 술어 재검사가 사용자 계약이다.
- 꼬리 대비: "조건 변수에 상태가 저장되나?" → 아니다. 대기자 없을 때의 notify는 사라진다. 술어가 진실의 원본이고 반드시 락 안에서 갱신해야 한다.

**Q5. 우선순위 역전을 실사례로 설명하라.**
- 핵심: Mars Pathfinder(1997) — 저우선순위가 뮤텍스를 쥔 채 중간 우선순위에 선점당해, 고우선순위가 무기한 블록 → 워치독 리셋 반복. 수정은 VxWorks 뮤텍스의 우선순위 상속 활성화 패치를 지구에서 전송.
- 꼬리 대비: "상속이 세마포어에 안 되는 이유?" → 소유자가 없어 누구를 승격할지 모른다. "상속 vs 천장?" → 상속은 대기자 발생 시 동적 승격, 천장은 획득 즉시 사전 지정 수준으로 승격.

**Q6. 파이프·공유 메모리·소켓·메시지 큐를 비교하라.**
- 핵심: 축은 세 개 — 복사 횟수(파이프/큐/소켓 2회+, 공유 메모리 0회), 경계(스트림 vs 메시지), 동기화 책임(커널 vs 사용자). 관통 원리는 "복사를 줄일수록 동기화 책임이 사용자에게 온다". 머신 경계를 넘는 건 소켓뿐.
- 꼬리 대비: "TCP로 패킷 경계는 어떻게?" → 스트림이라 경계가 없다. 길이 프리픽스로 애플리케이션이 재구성한다.
- Winters: 클라-서버는 TCP 소켓 + 길이 기반 프레이밍, 서버 내부는 단일 프로세스 멀티스레드 + 큐로 소유권 이전.

**Q7. 멀티스레드 게임 서버에서 락 경합을 줄이는 설계 원칙은?**
- 핵심: (1) 락 안에서는 swap/포인터 교환만 하고 무거운 일은 밖에서, (2) 상태마다 라이터를 하나로, (3) 데이터 공유 대신 소유권을 move로 이전. 셋 다 "공유 가변 상태 축소"의 표현.
- Winters: CommandIngress의 swap-후-정렬(락 구간 O(1)), 30Hz 틱 스레드가 월드의 유일한 라이터, drain된 벡터의 소유권 이전 — 세 원칙이 한 경로에 다 있다.

**Q8. 락프리 자료구조를 실제로 다뤄 봤나?**
- 핵심: Chase-Lev 워크스틸링 데크 — owner만 Push/Pop, thief는 Steal이라는 호출 규약 위에서 마지막 1개 경합만 CAS로 판정하고, StoreLoad 재배열을 seq_cst 펜스로 막는다. 락프리의 안전은 알고리즘 코드만이 아니라 **호출 규약**에 있다.
- 꼬리 대비: "락프리가 항상 빠른가?" → 아니다, 진행 보장이지 성능 약속이 아니다. 고경합 CAS 재시도는 락보다 느릴 수 있다.
- Winters: 외부 스레드 Submit이 owner 규약을 깨던 경합을 "owner면 자기 데크, 아니면 뮤텍스 전역 큐" 라우팅으로 고친 경험(JobSystem.cpp:146~158)까지 말하면 실전성이 산다.

## ⑥ 흔한 오답 / 함정

1. **"뮤텍스는 1, 세마포어는 N"이라고만 답하기.**
   값의 범위는 표면이다. 본질은 소유권 유무이고, 그로부터 해제 주체·오용 감지·우선순위 상속 가능성이 갈린다. 이걸 빠뜨리면 Mars Pathfinder 꼬리질문에서 무너진다.

2. **"스핀락 = 빠른 락, 뮤텍스 = 느린 락"이라는 이분법.**
   대기 시간 분포에 따라 승자가 바뀌고, 현대 락은 애초에 하이브리드라 양자택일 프레임 자체가 낡았다. "비경합이면 futex/CRITICAL_SECTION도 커널 진입 없이 원자 연산 몇 개"라고 말할 수 있어야 한다.

3. **조건 변수를 이벤트처럼 쓰기 — `if`로 기다리거나, 술어 갱신을 락 밖에서 하기.**
   허위 각성·wake 경쟁·lost wakeup이 전부 이 틈으로 들어온다. 3인조(술어+뮤텍스+CV)와 while 재검사는 스타일이 아니라 정확성 조건이다.

4. **데드락 4조건을 암송만 하고 "그래서 뭘 하는가"가 없는 답.**
   면접관이 원하는 건 "순환 대기를 락 순서 규약으로 깬다, 이유는 비용 0에 지역 검증 가능"이라는 실전 연결이다. 은행가 알고리즘을 실무 해법처럼 말하는 것도 감점 — 왜 못 쓰는지가 아는 사람의 표식이다.

5. **"공유 메모리가 제일 빠르니 최선"이라는 답.**
   복사 0회의 대가로 동기화·수명·장애 복구가 전부 사용자 책임이 된다는 트레이드오프를 말하지 못하면 반쪽 답이다. IPC 비교는 속도 순위표가 아니라 책임 배치의 스펙트럼이다.

6. **volatile을 동기화 도구로 언급하기 (C++ 면접 기준 즉시 감점).**
   C++ volatile은 원자성도 순서 보장도 없다. `std::atomic` + memory_order가 답이고, Java volatile과 다른 물건임을 언어 명시와 함께 구분해야 한다.

7. **경쟁 조건을 인터리빙 하나로만 설명하기.**
   멀티코어에서는 상호 배제를 지켜도 배리어가 없으면 "썼는데 안 보임"이 남는다. 인터리빙(스케줄러)과 가시성(스토어 버퍼·재배열)을 구분하고, 락이 두 문제를 한 번에 푸는 이중 장치(상호 배제 + acquire/release)임을 말해야 완결된 답이다.
