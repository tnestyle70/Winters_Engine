# OS 면접 질문 은행

> 이 챕터는 개념을 다시 설명하지 않는다. 개념 설명은 앞 챕터들(01~07)에 있고, 여기는 **면접장에서 실제로 나오는 질문 → 한 줄 정답 → 파고들기(꼬리질문)**만 모은 질문 은행이다.
> 사용법: (A)로 기초를 훑고, 각 항목의 `▸` 를 스스로 답할 수 있는지 점검한다. 막히면 그게 공부할 지점이다. (B)는 꼬리질문 연쇄 방어용, (C)는 "경험을 말해보라"에 STAR로 답하는 골격이다.
> 모든 `path:line` 인용은 실제 파일에서 확인한 것이다. 면접장에서 "이거 제 서버/엔진 코드입니다"라고 말할 수 있어야 한다.

## A. 기초 개념 질문 (한 줄 정답 + 파고들기)

### A-1. 프로세스 · 스레드

1. **프로세스와 스레드의 차이는?**
   한 줄: 프로세스(process)는 **자원 소유의 단위**(독립 주소 공간 + 핸들/파일 테이블), 스레드(thread)는 **CPU 스케줄링의 단위**(자기 스택 + 레지스터 문맥)다.
   ▸ "왜 이렇게 나눴나?" → 역사적으로 프로세스 하나가 둘 다였는데, 주소 공간 생성/전환이 너무 비싸서 "자원은 공유하고 실행 흐름만 여럿"이라는 경량 단위를 분리해낸 것. 협력에는 공유가 싸고, 격리에는 프로세스가 안전하다는 트레이드오프.
   ▸ "브라우저는 왜 탭마다 프로세스인가?" → 한 탭의 크래시/침해가 다른 탭을 못 건드리게 — 격리가 공유 비용보다 비싼 대가를 치를 만큼 중요한 도메인이라서.

2. **한 프로세스의 스레드들이 공유하는 것과 각자 갖는 것은?**
   한 줄: 공유 — 코드·전역 데이터·힙·열린 파일/핸들. 각자 — 스택, 레지스터(PC/SP), 스레드 로컬 저장소(TLS).
   ▸ "힙이 공유라서 생기는 문제는?" → 동기화 없는 동시 접근이 곧 데이터 레이스. Winters가 워커 신원·프로파일러 스택·A* 스크래치를 전부 `thread_local`로 뺀 이유가 이것 (`Engine/Private/Core/JobSystem.cpp:18`, `Engine/Private/Core/Profiler/CPUProfiler.cpp:29`, `Engine/Private/Manager/Navigation/Pathfinder.cpp:463`).
   ▸ "스택은 공유가 정말 불가능한가?" → 다른 스레드 스택의 주소를 넘기면 접근은 된다(같은 주소 공간이니까). 안 하는 건 하드웨어 금지가 아니라 **수명 규약** — 그 프레임이 pop되는 순간 댕글링이다.

3. **프로세스의 상태 전이를 설명하면?**
   한 줄: 생성 → 준비(ready) → 실행(running) → (선점되면 준비로 / I·O 등 대기면 blocked로) → 대기 해제 시 준비로 복귀 → 종료.

   ```text
   [new] → [ready] ⇄ [running] → [terminated]
              ▲          │ 블로킹 시스템 콜
              └─[blocked]◀┘   (I/O 완료 인터럽트가 ready로 되돌림)
   ```
   ▸ "blocked와 ready의 차이가 왜 중요한가?" → 스케줄러는 ready 큐만 본다. blocked는 CPU 후보조차 아니고, **깨우는 주체가 스케줄러가 아니라 이벤트(인터럽트/신호)**라는 게 I/O 모델 전체의 뿌리다.

4. **컨텍스트 스위치(context switch)란? 무엇을 저장하고 복원하나?**
   한 줄: 커널이 현재 실행 흐름의 레지스터 문맥(PC·SP·범용 레지스터)을 커널 자료구조(PCB/TCB)에 저장하고 다음 흐름의 것을 복원하는 일.
   ▸ "언제 일어나나?" → 타이머 인터럽트(선점), 블로킹 시스템 콜(자발적 양보), 더 높은 우선순위 스레드의 준비. 셋 다 커널 모드 진입이 선행된다.
   ▸ "유저 코드는 스위치를 감지할 수 있나?" → 직접은 못 한다. 그래서 "내 스레드는 언제든 두 명령 사이에서 중단될 수 있다"가 동시성 코드의 기본 전제가 된다.

5. **컨텍스트 스위치의 비용은? 스레드 전환과 프로세스 전환의 차이는?**
   한 줄: 직접 비용(레지스터 저장/복원 + 커널 진입)은 마이크로초 급이지만, **간접 비용(캐시·TLB 오염)이 더 클 수 있다**. 프로세스 전환은 주소 공간(페이지 테이블 루트) 교체가 추가돼 TLB 무효화 비용이 얹힌다.
   ▸ "그 수치 감각이 설계에 어디 쓰였나?" → Winters 잡 시스템 워커를 `hardware_concurrency - 2`개로 고정해(`JobSystem.cpp:50`) 코어보다 많은 스레드가 서로를 선점하며 스위치를 낭비하는 상황 자체를 없앴다. → TLB 상세는 (B)1.

6. **fork/exec 2단 모델은 왜 그렇게 생겼나? Windows와의 차이는?**
   한 줄: Unix는 "자신의 복제(fork)"와 "다른 프로그램으로 교체(exec)"를 분리해 그 **사이**에 자식이 fd 재배선·권한 조정을 자기 코드로 할 수 있게 했고, Windows `CreateProcess`는 한 호출에 옵션 파라미터로 몰아넣은 일체형이다.
   ▸ "fork가 주소 공간 전체 복사인데 안 비싼가?" → copy-on-write 덕분에 복사는 쓰는 페이지만 일어난다 (28번). fork 직후 exec하는 전형 시나리오에선 거의 공짜.
   ▸ "쉘의 리다이렉션(`>`)은 어디서 구현되나?" → fork와 exec **사이**에서 자식이 자기 fd를 파일로 바꿔치기 — 2단 분리가 만든 설계 공간의 대표 사례.

7. **좀비 프로세스와 고아 프로세스는?**
   한 줄: 좀비(zombie)는 종료했지만 부모가 종료 상태를 회수(wait)하지 않아 커널에 항목만 남은 프로세스, 고아(orphan)는 부모가 먼저 죽어 init 계열 프로세스에 입양된 프로세스다.
   ▸ "왜 좀비를 바로 못 없애나?" → 부모가 exit code를 읽을 권리가 있어서. 종료 상태는 누군가 수거할 때까지 커널이 보관해야 한다. Windows도 같은 원리 — 프로세스 커널 오브젝트는 열린 핸들이 전부 닫혀야 해제된다.
   ▸ "좀비가 쌓이면 뭐가 고갈되나?" → 메모리보다 먼저 프로세스 테이블 항목(PID 공간). "리소스 누수는 바이트만이 아니다"의 예.

8. **서버를 멀티프로세스로 짤 때와 멀티스레드로 짤 때의 기준은?**
   한 줄: 격리(한 워커의 크래시가 전체를 못 죽임)가 중요하면 프로세스, 상태 공유·통신 비용이 지배적이면 스레드.
   ▸ "게임 서버는 왜 대부분 멀티스레드인가?" → 월드 상태라는 거대한 공유 가변 상태를 모든 워커가 초당 수십 번 읽고 써야 해서 IPC 비용을 감당할 수 없다. Winters 서버도 IOCP 워커 스레드들이 세션/월드를 공유하는 단일 프로세스다 (`Server/Private/Network/IOCPCore.cpp:120`).

9. **스레드 안전(thread safety)이란?**
   한 줄: 여러 스레드가 동시에 호출해도 자료구조의 불변식과 결과의 정합성이 깨지지 않는 성질.
   ▸ "달성하는 3가지 축은?" → (1) 공유를 없앤다(값 복사·`thread_local`) (2) 동기화한다(뮤텍스·atomic) (3) 시간을 나눈다(phase 분리). Winters는 셋 다 실전 사례가 있다 — 값 스냅샷(`Client/Private/Network/Backend/CHttpClient.cpp:147`), 뮤텍스 병합(`CPUProfiler.cpp:93`), Decision/Apply 2-pass(`Client/Private/GamePlay/System/LocalUnitAISystem.cpp:79`).
   ▸ "재진입 가능(reentrant)과 같은 말인가?" → 다르다. 재진입성은 같은 스레드가 (시그널/재귀로) 중첩 진입해도 되는가의 문제 — 스레드 안전해도 재진입 불가일 수 있고 그 역도 있다.

### A-2. 커널/유저 모드 · 시스템 콜 · 인터럽트

10. **커널 모드와 유저 모드는 왜 나뉘어 있나?**
    한 줄: CPU 특권 레벨(x86의 ring 0/3)로 하드웨어 접근·페이지 테이블 조작·인터럽트 제어 같은 특권 명령을 커널만 실행하게 막아, 잘못된(또는 악의적인) 프로그램으로부터 시스템 전체를 보호하기 위해서다.
    ▸ "유저 모드 코드가 화면에 글자를 찍으려면?" → 직접 못 한다. 시스템 콜로 커널에 위임하는 것만 가능 — 이 강제 통로가 곧 보안 경계다.
    ▸ "보호가 소프트웨어 검사인가?" → 아니다. mode bit는 CPU 하드웨어가 매 명령·매 접근에서 강제한다. OS는 그 하드웨어 기능 위의 정책이다.

11. **시스템 콜(system call)은 어떻게 동작하나?**
    한 줄: 라이브러리 스텁이 시스템 콜 번호를 레지스터에 싣고 트랩 명령(x86-64 `syscall`)을 실행 → CPU가 커널 모드로 전환해 커널 진입점으로 점프 → 번호로 디스패치 → 처리 후 유저 모드 복귀.

    ```text
    유저 코드 ──> API 스텁(예: ntdll의 Nt* / libc의 read)
                     │  번호를 레지스터에 적재
                     ▼
               syscall 명령 (트랩) ── CPU 모드 전환: ring3 → ring0
                     ▼
               커널 진입점 → 시스템 콜 테이블[번호] → 커널 함수 실행
                     ▼
               복귀 명령 ── ring0 → ring3, 반환값 레지스터로
    ```
    ▸ "함수 호출과 뭐가 다른가?" → 함수 호출은 같은 특권 레벨의 점프, 시스템 콜은 **특권 전환 + 스택 교체(커널 스택) + 인자 검증**이 낀 통제된 진입이다.
    ▸ "커널은 유저 포인터를 그냥 믿고 역참조하나?" → 안 된다. 검증 없이 쓰면 유저가 커널 메모리를 읽고 쓰는 통로가 된다 — "신뢰 경계에서 입력 검증"은 커널과 네트워크 코드가 공유하는 원칙이다.

12. **시스템 콜은 왜 비싸고, 어떻게 줄이나?**
    한 줄: 모드 전환 자체 + 커널 코드/데이터로의 캐시 오염 때문이며, 줄이는 정석은 **배칭(한 번에 많이)과 유저 모드 완결(커널에 안 가기)**이다.
    ▸ 실례: 뮤텍스가 무경합이면 커널에 안 가는 futex 계열 설계 → (B)2. I/O를 모아 수확하는 IOCP/io_uring → (B)7.
    ▸ Winters 잡 시스템의 잡 전환은 함수 호출 아니면 파이버 전환 — 둘 다 시스템 콜이 없다 → (C)3.

13. **인터럽트(interrupt)란? 처리 흐름은?**
    한 줄: 장치·타이머가 CPU에 비동기로 거는 신호로, CPU가 현재 문맥을 중단하고 커널의 인터럽트 핸들러로 점프하게 만든다.
    ▸ "핸들러는 왜 짧아야 하나?" → 핸들러 실행 중엔 같은(또는 하위) 인터럽트가 밀리므로, 급한 확인만 하고 나머지는 지연 처리(bottom half 계열)로 넘기는 게 범용 설계다.
    ▸ "게임 루프와의 접점은?" → 타이머 인터럽트가 곧 선점 스케줄링의 심장. 우리가 짠 스레드가 "언제든 중단될 수 있다"는 전제가 여기서 나온다.

14. **인터럽트 방식과 폴링(polling) 방식의 트레이드오프는?**
    한 줄: 인터럽트는 유휴 시 CPU를 안 태우고 지연이 낮지만, 초당 이벤트가 매우 많으면 인터럽트 처리 자체가 병목(인터럽트 폭풍)이라 고부하에선 폴링/하이브리드가 이긴다.
    ▸ Winters의 두 대기 전략이 정확히 이 축 위에 있다: IOCP 워커는 이벤트 드리븐 무한 대기(`GetQueuedCompletionStatus(..., INFINITE)`, `IOCPCore.cpp:216`), 잡 워커는 1ms 타임아웃 하이브리드(`JobSystem.cpp:186`) — 깨우기 유실을 허용하되 1ms 안에 회복.
    ▸ "고성능 NIC 드라이버는?" → 부하가 오르면 인터럽트를 끄고 폴링으로 전환하는 하이브리드가 일반적 — 방식의 우열이 아니라 **부하 구간별 스위칭**이 정답이라는 사례.

15. **DMA(direct memory access)란?**
    한 줄: 장치가 CPU를 거치지 않고 메모리와 직접 데이터를 주고받고, 완료만 인터럽트로 알리는 방식 — CPU가 바이트 복사 노동에서 해방된다.
    ▸ "그럼 CPU는 뭘 하나?" → 전송 설정과 완료 처리만. 디스크/NIC 대량 전송이 전부 이 위에 있고, overlapped I/O가 "요청만 걸고 다른 일 하기"일 수 있는 물리적 근거이기도 하다.
    ▸ "overlapped recv 버퍼를 완료 전에 건드리면 왜 안 되나?" → 그 버퍼는 커널/장치가 쓰는 중인 영역이다. Winters 세션이 recv 완료 처리 전까지 컨텍스트 버퍼를 소유 이전하는 이유 (`Server/Private/Network/Session.cpp:54`~`:63`).

### A-3. 스케줄링

16. **스케줄러는 무엇을 최적화하나?**
    한 줄: 처리량(throughput)·평균 대기/응답 시간·공정성·상호작용 지연 — 서로 상충하는 지표들 사이의 정책 선택이다.
    ▸ "왜 '최고의 알고리즘'이 없나?" → 배치 서버는 처리량, 데스크톱은 응답성, 실시간은 마감 보장이 우선이라 목적함수 자체가 다르다. 알고리즘 암기보다 이 문장이 먼저다.

17. **FCFS / SJF / RR을 비교하면?**

    | 알고리즘 | 아이디어 | 강점 | 약점 |
    |---|---|---|---|
    | FCFS | 도착 순서대로 | 단순·공정해 보임 | 호위 효과(convoy) — 긴 잡 뒤 전부 대기 |
    | SJF/SRTF | 짧은 잡 먼저 | 평균 대기시간 최소(증명됨) | 실행 시간을 미리 알아야 함 + 긴 잡 기아 |
    | RR | 타임 퀀텀 순환 | 응답성 보장 | 퀀텀 선택 딜레마 (18번) |

    ▸ "SJF가 최적인데 왜 못 쓰나?" → 미래를 모르니까. 그래서 실제 OS는 **과거 행동으로 미래를 추정**하는 근사(SJF 흉내)를 쓴다 → MLFQ (19번).

18. **RR의 타임 퀀텀은 어떻게 정하나?**
    한 줄: 퀀텀이 짧으면 응답성은 좋지만 컨텍스트 스위치 오버헤드 비중이 커지고, 길면 FCFS에 수렴한다 — 스위치 비용 대비 충분히 크게(통상 수 ms~수십 ms 급)가 원칙이다.
    ▸ "퀀텀 = 스위치 비용이면?" → CPU 절반이 전환 노동에 쓰인다. "오버헤드가 상수면 단위 작업을 키워 상쇄한다"는 원칙은 잡 시스템 granularity 설계와 같은 사고다 (`LocalUnitAISystem.cpp:79`의 직렬 폴백 임계치).

19. **MLFQ(multi-level feedback queue)의 아이디어는?**
    한 줄: 여러 우선순위 큐를 두고 **CPU를 오래 쓰면 강등, 짧게 쓰고 양보하면 우대** — 실행 시간을 모른 채 과거 행동으로 SJF를 근사한다.
    ▸ "왜 I/O 위주 작업을 우대하나?" → 상호작용(키 입력, 네트워크)의 체감 지연을 줄이고, 어차피 CPU를 조금만 쓰고 다시 잠들기 때문에 우대 비용이 싸다.
    ▸ 기아·조작 문제와 부스트는 → (B)9.

20. **게임 엔진 워커 스레드 수를 왜 코어 수에 맞추나?**
    한 줄: 계산 위주 워커는 블로킹이 거의 없어, 코어보다 많으면 서로를 선점하며 스위치·캐시 오염만 낳기 때문 — "스케줄링을 OS에 맡기지 말고 코어당 1개로 고정"이 데이터 병렬 엔진의 정석이다.
    ▸ Winters: `hardware_concurrency - 2` (`JobSystem.cpp:50`) — 메인 스레드와 OS/드라이버 몫을 빼놓는다. IOCP 쪽은 반대로 동시성 값을 포트에 명시해 커널이 활성 스레드 수를 조절하게 한다 (`IOCPCore.cpp:44`) → (C)1.
    ▸ "I/O 워커는 왜 코어 수보다 많아도 되나?" → 대부분 잠들어 있으니까. "블로킹 비율이 스레드 수 공식을 바꾼다"가 만점 포인트.

### A-4. 가상 메모리 · 페이징

21. **가상 메모리(virtual memory)는 왜 필요한가?**
    한 줄: 각 프로세스에 "메모리 전체를 혼자 쓰는" 환상을 줘서 (1) 프로세스 간 보호 (2) 연속 주소라는 편한 프로그래밍 모델 (3) 물리 메모리 크기·배치로부터의 독립 (4) 공유와 COW 같은 트릭을 한 메커니즘으로 해결한다.
    ▸ "포인터에 들어있는 주소는 뭔가?" → 전부 가상 주소. 물리 주소는 유저 코드가 볼 수도 없다.
    ▸ "그럼 두 프로세스의 같은 주소 0x400000은?" → 서로 다른 물리 프레임(또는 공유 라이브러리라면 같은 프레임의 공유 매핑) — 주소는 프로세스 좌표계다.

22. **프로세스 주소 공간의 레이아웃은?**
    한 줄: 코드(текст)·정적 데이터가 아래쪽, 힙이 위로 자라고, 스택이 높은 주소에서 아래로 자라며, 중간에 매핑 영역(DLL/공유 라이브러리, 파일 매핑)이 놓이는 게 전통적 구도다 — 현대 OS는 여기에 무작위 배치(ASLR)를 얹는다.
    ▸ "힙과 스택이 충돌하면?" → 가상 주소 공간이 64비트로 커지면서 "마주 보고 자라다 충돌"은 사실상 옛날 이야기가 됐고, 실제 한계는 커밋 가능한 메모리와 스택 예약 크기다 (30번).
    ▸ "ASLR은 왜 하나?" → 공격자가 코드/스택 주소를 예측해 뛰어드는 익스플로잇을 확률적으로 방해 — 보호를 "비용 없는 배치 정책"으로 얻는 사례.

23. **페이징(paging)이란? 페이지 테이블은 왜 다단계인가?**
    한 줄: 주소 공간을 고정 크기 페이지(통상 4KB)로 잘라 페이지 단위로 물리 프레임에 매핑하는 것 — 외부 단편화가 사라진다. 단일 테이블이면 64비트 주소 공간에서 테이블 자체가 거대해지므로, **쓰는 영역만 테이블을 만드는** 다단계(x86-64는 4단계 급) 트리로 희소하게 표현한다.

    ```text
    가상 주소 ─┬─ 상위 비트들 ── 페이지 테이블 트리 워크 (다단계 인덱스)
               │                     │
               │                     ▼
               │              물리 프레임 번호(PFN)
               └─ 하위 비트  ── 페이지 내 오프셋 (그대로 사용)
    ```
    ▸ "매 메모리 접근마다 트리를 타면 너무 느리지 않나?" → 그래서 TLB가 있다 (24번).
    ▸ "내부 단편화는?" → 남는다 — 페이지 마지막 자투리. 외부 단편화(연속 공간 부족)를 없애는 대가로 내부 단편화(고정 단위 낭비)를 받아들인 교환이다.

24. **TLB(translation lookaside buffer)란?**
    한 줄: MMU 안의 "최근 페이지 변환 결과" 캐시 — 적중하면 테이블 워크 없이 한 번에 변환하고, 미스면 페이지 테이블을 걸어 채운다(x86은 하드웨어가 워크).
    ▸ "TLB가 왜 성능의 급소인가?" → 모든 메모리 접근이 변환을 거치므로 TLB 미스율이 곧 전역 세금. 데이터 지향 설계가 "연속 메모리 순회"를 고집하는 이유엔 캐시뿐 아니라 TLB 지역성도 있다. 컨텍스트 스위치와의 관계는 → (B)1.

25. **페이지 폴트(page fault)의 종류는?**
    한 줄: (1) minor — 유효한 매핑인데 아직 물리 프레임이 안 붙었거나 이미 메모리에 있는 페이지를 연결만 하면 되는 경우 (2) major — 디스크 I/O가 필요한 경우 (3) invalid — 매핑 자체가 없거나 권한 위반, 즉 세그폴트/access violation.
    ▸ "페이지 폴트는 버그인가?" → 아니다. 앞의 둘은 요구 페이징(demand paging)·COW의 **정상 동작 메커니즘**이고, 커널이 트랩을 받아 조용히 해결한다. invalid만 프로그램 오류다.
    ▸ "폴트가 트랩이어야 하는 이유는?" → 유저 명령 하나를 중단하고 커널이 개입한 뒤 **그 명령을 재실행**해야 하므로 — 하드웨어가 정확한 중단점(precise fault)을 보장해줘야 성립한다.

26. **malloc이 성공하면 물리 메모리를 받은 것인가?**
    한 줄: 아니다 — 일반적으로 가상 주소 공간의 예약/커밋일 뿐이고, 물리 프레임은 **첫 접근의 페이지 폴트에서** 붙는다(요구 페이징).
    ▸ "그럼 대량 할당 직후 첫 순회가 느린 이유는?" → 페이지마다 폴트 비용을 그때 내는 것. 게임에서 로딩 시점에 미리 터치(pre-touch)하는 관행의 근거. overcommit과 OOM은 → (B)10.

27. **페이지 교체 알고리즘과 스래싱(thrashing)은?**
    한 줄: 물리 프레임이 부족하면 희생 페이지를 골라 내보내는데, 이상적 정답(가장 늦게 다시 쓸 페이지)은 미래를 알아야 하므로 실제로는 LRU를 참조 비트로 싸게 근사(clock/second-chance 계열)한다. 스래싱은 워킹셋이 물리 메모리를 초과해 **실행보다 페이지 교체에 시간을 다 쓰는** 붕괴 상태다.
    ▸ "LRU를 정확히 구현하면 왜 안 되나?" → 매 접근마다 타임스탬프/리스트 갱신 비용이 모든 메모리 접근에 얹힌다 — 하드웨어가 주는 건 참조 비트 하나뿐이라 근사가 강제된다.
    ▸ "스래싱의 처방은?" → 페이지 교체를 더 잘하는 게 아니라 **동시 실행 프로세스를 줄여** 워킹셋 합을 물리 안으로 되돌리는 것 — 부하 조절이 정답인 드문 문제.

28. **copy-on-write(COW)란?**
    한 줄: 복사가 필요한 페이지를 일단 읽기 전용으로 공유 매핑해두고, 어느 쪽이든 **쓰는 순간의 페이지 폴트에서** 진짜 복사하는 지연 전략.
    ▸ "왜 이게 이득인가?" → fork 직후 exec처럼 "복사본 대부분을 안 건드리는" 워크로드에서 복사 비용이 0에 수렴. 하드웨어 보호 비트 + 폴트 핸들러 조합으로 "게으름"을 구현한 대표 사례다.
    ▸ "쓰기 폭주 워크로드면?" → 어차피 다 복사하면서 폴트 비용만 추가로 낸다 — 지연 전략은 "안 쓸 확률"에 거는 베팅이다.

29. **스택과 힙의 차이는?**
    한 줄: 스택은 함수 호출 프레임의 자동 수명 영역(SP 이동만으로 할당·해제, 연속·캐시 친화, 크기 제한 — MSVC 기본 1MB), 힙은 할당자를 거치는 동적 수명 영역(유연하지만 할당자 비용·단편화·다중 스레드 동기화).
    ▸ "스레드마다 뭐가 따로인가?" → 스택은 스레드마다 하나씩, 힙은 프로세스 공유. 그래서 스택 지역 변수는 태생적으로 레이스가 없다.
    ▸ "힙 할당이 시스템 콜인가?" → 대개 아니다. 할당자(CRT/jemalloc류)가 OS에서 큰 덩어리를 받아두고 유저 모드에서 쪼개 준다 — 시스템 콜은 덩어리가 바닥날 때만.

30. **스택 오버플로는 OS가 어떻게 감지하나?**
    한 줄: 스택 끝에 가드 페이지(guard page)를 두고, 침범 접근이 페이지 폴트를 일으키면 스택 확장 또는 예외(스택 오버플로)로 처리한다 — "접근 위반을 기능으로 쓰는" 가상 메모리 응용이다.
    ▸ "재귀가 너무 깊으면 어떻게 죽나?" → 가드 페이지를 치는 순간 예외. 한 페이지를 훌쩍 뛰어넘는 거대 지역 배열은 감지를 건너뛸 수 있어 컴파일러가 스택 프로브를 심는다.

31. **캐시 지역성(locality)이란? OS와 무슨 관계인가?**
    한 줄: 방금 만진 것(시간 지역성)과 그 근처(공간 지역성)를 다시 만질 확률이 높다는 경험칙으로, 캐시·TLB·페이지 캐시·프리페치까지 메모리 계층 전체가 이 가정 위에 설계돼 있다.
    ▸ "OS 층위의 예는?" → 파일 순차 읽기의 미리 읽기(readahead), 페이지 단위 로딩 자체. 유저 코드 층위의 실전은 C++ 챕터의 ECS/SoA 논의로 → [../cpp/08_stl_containers_cache.md](../cpp/08_stl_containers_cache.md).

### A-5. 동기화 · 데드락

32. **경쟁 상태(race condition)와 데이터 레이스(data race)의 차이는?**
    한 줄: 데이터 레이스는 "동기화 없는 동시 접근(하나 이상 쓰기)"이라는 좁은 정의(C++에선 UB), 경쟁 상태는 원자적 접근이라도 **타이밍에 따라 결과가 달라지는 논리 결함** 전반이다.
    ▸ "레이스 없는 경쟁 상태 예시는?" → check-then-act: `if (map.contains(k)) map.at(k)` 사이에 다른 스레드가 지우는 경우. 접근 각각이 안전해도 시퀀스는 깨진다.

33. **뮤텍스와 세마포어의 차이는?**
    한 줄: 뮤텍스(mutex)는 **소유권 있는 상호배제**(잠근 스레드만 풀 수 있다), 세마포어(semaphore)는 소유권 없는 **카운터 기반 신호/자원 계수**(다른 스레드가 signal해도 된다).
    ▸ "이진 세마포어 = 뮤텍스 아닌가?" → 기능은 비슷해도 계약이 다르다. 뮤텍스의 소유권은 "안 잠근 스레드의 해제" 같은 오류를 정의상 배제하고, 우선순위 상속 같은 커널 협조도 소유자를 알아야 가능하다.
    ▸ "세마포어가 자연스러운 문제는?" → 개수 제한 자원(커넥션 풀 N개)과 신호 전달(생산자→소비자, 39번) — "누가 잠갔는가"가 없는 문제들.

34. **뮤텍스와 스핀락(spinlock)은 언제 갈라 쓰나?**
    한 줄: 임계 구역이 컨텍스트 스위치 비용보다 확실히 짧고 홀더가 다른 코어에서 돌고 있을 때만 스핀이 이긴다 — 아니면 재우는 뮤텍스가 정답.
    ▸ 정확한 3조건과 싱글 코어의 함정은 → (B)8. 실제 뮤텍스는 "짧게 스핀하다 재우는" 하이브리드가 많다 → (B)2.

35. **읽기-쓰기 락(reader-writer lock)은 언제 쓰나?**
    한 줄: 읽기가 압도적으로 많고 읽기끼리는 충돌하지 않는 자료에 "다수 reader 동시 / writer 단독"을 허용해 병렬성을 회수하는 락이다.
    ▸ "함정은?" → (1) reader가 끊이지 않으면 writer 기아 — 공정성 정책이 구현마다 다르다 (2) 임계 구역이 짧으면 일반 뮤텍스보다 오히려 비싸다(상태 관리가 더 복잡해서). "읽기 많음 = 무조건 RW락"은 성급한 결론.
    ▸ "더 급진적 대안은?" → 읽기 쪽 비용을 0으로 만드는 스냅샷/더블 버퍼링. Winters 프로파일러가 프레임 경계 `swap`으로 reader(UI)와 writer(수집)를 시간으로 분리한 게 그 노선 (`CPUProfiler.cpp:47`~`:53`).

36. **조건 변수(condition variable)는 왜 필요하고, spurious wakeup은 뭔가?**
    한 줄: "조건이 참이 될 때까지 재워달라"를 뮤텍스와 결합해 표현하는 도구이고, 커널/구현 사정상 조건 없이 깨어날 수 있어(spurious wakeup) **깨어나면 반드시 조건을 재검사**(predicate 루프)해야 한다.
    ▸ "notify를 놓치면(lost wakeup)?" → 락 밖에서 조건 변경+notify를 하면 wait 진입과 겹쳐 유실 가능. Winters 잡 워커는 정석 대신 `wait_for(1ms)`로 **유실을 허용하되 회복 상한**을 두는 실용 절충을 골랐고 그 근거가 주석에 있다 (`JobSystem.cpp:182`~`:186`).
    ▸ "왜 조건 변수는 항상 뮤텍스와 세트인가?" → "조건 검사"와 "잠들기"가 원자적이어야 그 틈의 notify 유실을 막는다 — wait가 뮤텍스를 받아 **풀면서 잠드는** 이유.

37. **데드락(deadlock)의 4가지 필요조건은?**
    한 줄: 상호 배제, 점유 대기(hold and wait), 비선점(자원을 뺏을 수 없음), 순환 대기 — 넷이 **동시에** 성립해야 데드락이 가능하다.
    ▸ "하나만 깨면 예방된다는 뜻인가?" → 그렇다. 실무에서 가장 싸게 깨는 건 순환 대기 — 모든 스레드가 같은 순서로 락을 잡는 전역 락 순서 규약.
    ▸ "락 없는 데드락도 있나?" → 있다 — 서로의 완료를 기다리는 future/join 사이클, 꽉 찬 큐에 서로 push 대기 등. 조건 4개는 "자원"을 락으로 한정하지 않는다.

38. **데드락 대응 전략 4가지는?**
    한 줄: 예방(조건을 설계로 제거) / 회피(은행원 알고리즘처럼 안전 상태만 허용) / 탐지 후 회복 / 무시(ostrich) — 범용 OS는 대부분 무시+개발자 규약에 맡긴다.
    ▸ "왜 OS가 은행원 알고리즘을 안 쓰나?" → 모든 프로세스의 최대 자원 요구를 미리 알아야 하고 매 할당마다 검사 비용이 든다. 이론적으론 아름답지만 전제 자체가 범용 환경에서 성립하지 않는다.
    ▸ "그럼 DB는?" → 트랜잭션은 롤백이 가능해서 탐지 후 회복(희생자 abort)이 실용적이다 — 같은 문제도 "되돌릴 수 있는가"가 전략을 바꾼다.

39. **세마포어로 생산자-소비자를 구현하면?**
    한 줄: 빈 슬롯 세마포어(empty=N)·찬 슬롯 세마포어(full=0)·버퍼 뮤텍스 셋의 조합이다.

    ```text
    producer:                     consumer:
      wait(empty)                   wait(full)
      lock(m); push(x); unlock(m)   lock(m); x = pop(); unlock(m)
      signal(full)                  signal(empty)
    ```
    ▸ "wait(empty)와 lock(m)의 순서를 바꾸면?" → 버퍼가 가득할 때 **뮤텍스를 쥔 채** 잠들어 소비자가 pop을 못 하는 데드락. "자원 대기는 락 바깥에서"가 일반 원칙이다.
    ▸ "세마포어 둘은 왜 필요한가?" → empty는 생산자를, full은 소비자를 재우는 서로 다른 대기 조건이라서. 뮤텍스 하나로 합치면 "가득참"과 "비었음"을 구분해 잘 수 없다.
    ▸ "실코드에선?" → Winters의 워커→메인 콜백 큐가 같은 구조의 뮤텍스+swap 변형 (`CHttpClient.cpp:159`~`:171`) — 소비자가 큐를 통째로 바꿔치기해 락 구간을 상수로 줄인다.

40. **atomic 연산은 커널이 제공하는 것인가?**
    한 줄: 아니다 — CPU의 원자 명령(lock 접두 RMW, CAS 계열)으로 **유저 모드에서 커널 진입 없이** 실행된다. 커널이 제공하는 건 "재우고 깨우기"이고, atomic은 그 앞단의 무커널 경로다.
    ▸ "그럼 뮤텍스와 atomic의 관계는?" → 현대 뮤텍스는 atomic으로 무경합을 처리하고 경합 시에만 커널로 넘어가는 2단 구조 → (B)2. 메모리 순서 세부는 C++ 챕터 몫 → [../cpp/09_concurrency.md](../cpp/09_concurrency.md).

### A-6. IPC · 파일 시스템

41. **IPC(inter-process communication) 종류와 선택 기준은?**
    한 줄: 파이프(부모자식 단방향 스트림) / 명명 파이프 / 메시지 큐(구조화된 단위) / 공유 메모리(최대 대역폭) / 소켓(원격 투명) / 시그널(비동기 통지) — **관계(혈연/무관/원격), 데이터 형태(스트림/메시지), 대역폭 요구**로 고른다.
    ▸ "게임에서의 예는?" → 에디터↔엔진 같은 로컬 도구 통신은 명명 파이프나 로컬 소켓, 클라↔서버는 당연히 소켓. Winters 클라/서버도 TCP 소켓 위 자체 프레이밍이다.
    ▸ "스트림과 메시지의 차이가 왜 중요한가?" → TCP/파이프는 경계 없는 바이트 흐름이라 수신 측이 직접 프레이밍해야 한다 — Winters 세션이 수신 바이트를 파서에 누적하고 완성 프레임만 꺼내는 이유 (`Session.cpp:84`~`:85`).

42. **공유 메모리가 가장 빠른 IPC인 이유는?**
    한 줄: 한 번 매핑해두면 이후 통신에 **커널 진입도 데이터 복사도 없기** 때문 — 다른 IPC는 매 전송이 시스템 콜 + 커널 버퍼 경유 복사다.
    ▸ "공짜가 아닌 부분은?" → 동기화를 커널이 안 해준다. 프로세스 간 뮤텍스/이벤트를 따로 얹어야 하고, 그 순간 스레드 공유 메모리와 똑같은 레이스 문제를 상속한다.

43. **파일 디스크립터(file descriptor)란?**
    한 줄: 프로세스별 테이블의 정수 인덱스로, 커널의 열린 파일 항목(오프셋·플래그)을 가리키고 그 항목이 다시 실제 파일(inode)을 가리키는 3단 간접 참조다.

    ```text
    fd(정수) ─> 프로세스별 fd 테이블 ─> 시스템 열린파일 테이블 ─> inode
                (프로세스 소유)          (오프셋·플래그 공유 지점)    (파일 실체)
    ```
    ▸ "소켓도 왜 fd인가?" → "커널 자원에 대한 프로세스별 핸들"이라는 추상화를 파일·소켓·파이프가 공유해서 read/write/다중화를 한 인터페이스로 쓸 수 있다. Windows의 HANDLE이 같은 자리 — 소켓 HANDLE을 IOCP에 연결하는 Winters 코드가 그 증거 (`IOCPCore.cpp:202`).
    ▸ "fork하면 fd는?" → 테이블이 복제되지만 열린 파일 항목은 공유 — 부모/자식이 오프셋을 공유한다. "무엇이 복제되고 무엇이 공유되는가"는 3단 구조를 알아야 답할 수 있다.

44. **inode란?**
    한 줄: 파일의 메타데이터(크기·권한·타임스탬프)와 데이터 블록 위치를 담는 파일시스템의 실체 노드 — **파일 이름은 inode에 없다**. 이름은 디렉터리(이름→inode 번호 매핑 테이블)의 소유물이다.
    ▸ "이름과 실체를 왜 분리했나?" → 하나의 실체에 여러 이름(하드 링크)을 허용하고, 이름 변경/이동을 데이터 이동 없이 디렉터리 항목 수정만으로 끝내기 위해.
    ▸ "열려 있는 파일을 지우면?" → 디렉터리 항목만 사라지고, inode는 링크 수 0 + 열린 참조 0이 될 때 해제 — 삭제가 "이름 제거"라는 것을 보여주는 고전 질문.

45. **하드 링크와 심볼릭 링크의 차이는?**
    한 줄: 하드 링크는 같은 inode를 가리키는 또 다른 디렉터리 항목(링크 카운트가 0이 될 때 실체 삭제), 심볼릭 링크는 **경로 문자열을 담은 별도 파일**(대상이 지워지면 깨진 링크).
    ▸ "왜 하드 링크는 파일시스템을 못 넘나?" → inode 번호가 파일시스템 내부에서만 유효한 식별자라서.

46. **read()를 호출하면 매번 디스크까지 가나?**
    한 줄: 아니다 — 커널의 페이지 캐시(page cache)가 디스크 블록을 메모리에 캐싱해서 재읽기는 메모리 복사로 끝나고, 쓰기도 캐시에 쌓았다가 나중에 내려쓴다(write-back).
    ▸ "그럼 전원이 나가면?" → 캐시에만 있던 쓰기는 유실. 그래서 flush/fsync 계열이 존재하고, DB가 그 비용을 정면으로 다룬다. "빠름의 대가는 내구성 계약의 복잡화"라는 전형적 트레이드오프.

### A-7. I/O 모델 — blocking에서 IOCP까지

47. **blocking / non-blocking / async I/O의 차이는?**
    한 줄: blocking은 완료까지 호출 스레드가 잠들고, non-blocking은 준비 안 됐으면 즉시 에러 반환(준비 감시는 내 몫), async는 **요청만 걸어두면 커널이 수행하고 완료를 통지**한다.

    | 모델 | 누가 기다리나 | 누가 데이터를 옮기나 | 알림 단위 |
    |---|---|---|---|
    | blocking | 내 스레드가 잠듦 | 커널(깨어날 때 완료) | — |
    | non-blocking + 다중화 | 다중화 콜에서 대기 | 내가 read/write 호출 | **준비됨**(readiness) |
    | async (IOCP/io_uring) | 완료 큐에서 대기 | 커널이 백그라운드 수행 | **완료됨**(completion) |

    ▸ "async처럼 보이는데 blocking인 사고는?" → `std::async`의 future를 버리면 임시 future 소멸자가 동기 대기 — Winters가 실제로 겪고 주석으로 박제한 버그다 (`CHttpClient.cpp:90`~`:92`) → 상세 STAR는 C++ 은행의 C-2.

48. **select의 한계는?**
    한 줄: fd 집합 크기 상한(전통적으로 1024), 매 호출마다 전체 집합을 커널로 복사하고 커널이 전부 스캔하는 O(전체 fd) 비용, 반환 후에도 어떤 fd가 준비됐는지 전체 순회로 찾아야 하는 구조.
    ▸ "왜 이렇게 설계됐었나?" → fd 수십 개 시절엔 충분히 단순하고 좋았다. C10K(동시 연결 1만) 시대가 오면서 "매 호출 O(n)"이 구조적 병목으로 드러난 것 — 한계는 결함이 아니라 시대의 가정이 만료된 것이다.
    ▸ "poll은 뭘 고쳤나?" → fd 상한과 비트마스크 재작성 문제만. 매 호출 전체 전달·전체 스캔 O(n)은 그대로라 근본 해법이 아니다 — 그래서 epoll이 필요했다.

49. **epoll은 select 대비 무엇을 고쳤나?**
    한 줄: 관심 목록을 커널에 **한 번 등록**(epoll_ctl)해두고, 대기 호출은 준비된 fd 목록만 돌려받는다 — 비용이 전체 fd 수가 아니라 **준비된 이벤트 수에 비례**한다.
    ▸ "여전히 readiness 모델인 건 왜 중요한가?" → "준비됐다"를 받은 뒤 read/write는 내가 해야 한다. 완료 모델과의 본질 차이고, LT/ET 함정은 → (B)6.

50. **IOCP(I/O completion port)란?**
    한 줄: Windows의 완료 기반 비동기 I/O 허브 — overlapped 요청을 걸어두면 커널이 수행하고, 완료 패킷을 포트 큐에 넣어주며, 워커 스레드 풀이 `GetQueuedCompletionStatus`로 수확한다. 큐이면서 동시에 **스레드 풀 동시성 조절기**다.
    ▸ "readiness 모델과 뭐가 다른가?" → epoll은 "이제 read해도 안 막힘"을 알려주고, IOCP는 "read가 **이미 끝났고** 버퍼에 데이터가 있다"를 알려준다. 데이터 복사 시점과 주체가 다르다.
    ▸ 내 서버 구조로 설명하는 골격은 → (C)1.

51. **왜 게임 서버에 IOCP를 골랐나?**
    한 줄: Windows 네이티브의 완료 기반 모델이라 연결당 스레드 없이 수백~수천 세션을 워커 몇 개로 감당하고, 포트의 동시성 값으로 "동시에 깨어있는 워커 수 ≈ 코어 수"를 커널과 협조해 유지할 수 있어서다.
    ▸ "connection-per-thread는 왜 안 되나?" → 세션 수만큼 스택 메모리(기본 1MB급)와 컨텍스트 스위치가 늘어 코어가 전환 노동에 잠식된다. I/O 대기는 스레드가 아니라 **커널 큐**에 맡기는 게 확장의 핵심.
    ▸ "Linux로 이식한다면?" → epoll(readiness)로 구조를 뒤집거나 io_uring(completion)으로 모델을 유지 — 두 경로의 차이를 말할 수 있으면 이식 질문 방어 완료 → (B)7.

## B. 심화 꼬리질문 방어 (연쇄를 한 방에 끊기)

1. **"컨텍스트 스위치가 비싸다" → "정확히 TLB에는 무슨 일이 생기나?"**
   프로세스 전환은 페이지 테이블 루트 교체라 기존 TLB 엔트리가 무효가 된다 — 태깅이 없으면 전체 플러시, 이후 워킹셋을 미스로 다시 채우는 간접 비용이 직접 비용보다 클 수 있다.
   ▸ 꼬리 1 "그럼 스레드 전환은?" → 같은 주소 공간이라 TLB가 산다 — "스레드 전환이 프로세스 전환보다 싸다"의 주된 실체가 레지스터 수가 아니라 이것.
   ▸ 꼬리 2 "플러시를 피하는 하드웨어 장치는?" → 주소공간 태그(x86 PCID, ARM ASID)로 엔트리에 주인을 표시해 플러시 없이 공존시킨다. 커널 매핑은 전 프로세스 공통이라 전통적으로 플러시 면제(global)였다.
   ▸ 꼬리 3 "그게 보안과 얽힌 적이 있나?" → Meltdown 이후 커널/유저 페이지 테이블을 분리(KPTI)하면서 시스템 콜·스위치 비용이 올라갔다 — 보안 패치가 성능 상식을 바꾼 사례로 마무리하면 강하다.

2. **"`std::mutex::lock()`은 시스템 콜인가?" (futex 연쇄)**
   무경합이면 아니다. futex(fast userspace mutex) 계열 설계는 유저 모드 atomic CAS로 잠금을 시도하고, **경합일 때만** 커널의 wait 큐로 진입한다.
   ▸ 꼬리 1 "잠들기 직전에 unlock이 오면 영원히 자나?" → futex_wait는 "주소의 값이 아직 기대값이면 재워라"라는 **검증 동반 잠들기**라 그 틈을 커널이 닫는다 — lost wakeup 방어가 API 계약에 내장된 것.
   ▸ 꼬리 2 "Windows에도 있나?" → 같은 정신의 프리미티브(SRWLOCK, WaitOnAddress)가 있다. "무경합 락 = atomic 두 번 값, 경합부터 커널 값"으로 비용을 층위별로 말하면 끝.
   ▸ 꼬리 3 "그럼 락이 느리다는 통념은?" → 경합이 느린 것이다. 병목 처방도 "락 제거"가 아니라 **경합 제거**(샤딩, 락 구간 축소, 공유 제거)가 먼저다.

3. **"priority inversion(우선순위 역전)을 설명해보라" (사고 사례 연쇄)**
   낮은 우선순위 스레드가 락을 쥔 채 중간 우선순위 스레드에게 선점당하면, 그 락을 기다리는 높은 우선순위 스레드가 중간 것보다 뒤로 밀리는 역전.
   ▸ 꼬리 1 "실제 사고는?" → 1997년 화성 탐사선 Mars Pathfinder가 이걸로 반복 리셋됐고, 우선순위 상속(inheritance — 락 홀더를 대기자 우선순위로 임시 승격)을 켜서 해결한 게 교과서 사례. 대안으로 우선순위 실링(ceiling)도 있다.
   ▸ 꼬리 2 "게임 실무 번역은?" → 오디오/렌더 같은 높은 우선순위 스레드와 일반 스레드가 같은 락을 공유하게 두지 말라 — 락 공유 자체가 우선순위 설계를 무효화한다.
   ▸ 꼬리 3 "락 없이 피하는 법은?" → 스레드 간 전달을 SPSC 큐/더블 버퍼 같은 무락 구조로 — Winters 프로파일러의 스레드별 수집+프레임 경계 swap이 그 방향 (`CPUProfiler.cpp:29`, `:47`~`:53`).

4. **"false sharing은 OS 문제인가 하드웨어 문제인가?"**
   캐시 코히런시(coherency) 프로토콜이 **캐시라인(통상 64B) 단위**로 소유권을 주고받는 하드웨어 동작이 원인 — 서로 다른 변수라도 같은 라인이면 두 코어가 라인을 핑퐁한다.
   ▸ 꼬리 1 "증상은 어떻게 보이나?" → 락도 없는데 코어를 늘려도 확장이 안 됨. 프로파일러 없이 코드만 봐서는 절대 안 보이는 부류다.
   ▸ 꼬리 2 "해법은?" → 스레드별로 쓰는 데이터를 `alignas(64)`로 라인 분리. Winters work-stealing deque가 owner 전용 `m_iBottom`과 thief 전용 `m_iTop`을 각각 분리한 것이 정확히 이 방어 (`Engine/Public/Core/JobSystem/WorkStealingDeque.h:91`~`:92`).
   ▸ 꼬리 3 "OS는 무관한가?" → 원인은 아니지만 스레드를 어느 코어에 놓는지(어피니티)가 증상 크기를 좌우한다 — 층위를 정확히 가르는 답이 만점.

5. **"락을 쓰면 메모리 배리어를 신경 안 써도 되나?"**
   된다 — 락 획득/해제에 acquire/release 의미론이 내장돼 있고, 커널의 컨텍스트 스위치 역시 완전한 동기화 지점이라 락으로 보호되는 코드는 배리어를 직접 쓸 일이 없다.
   ▸ 꼬리 1 "그럼 배리어는 언제 직접 만지나?" → lock-free를 짤 때뿐. 그때도 acquire/release로 안 막히는 유일한 순서(store 후 **다른 위치** load)가 있어 Winters Chase-Lev deque의 `Pop`이 `seq_cst` fence를 박는다 (`WorkStealingDeque.h:47`).
   ▸ 꼬리 2 "왜 CPU는 순서를 지키지 않나?" → 메모리 지연을 숨기는 스토어 버퍼/비순차 실행이 단일 스레드 관점만 보존하기 때문 — 오더링 이론 전체는 C++ 챕터에 위임하고, OS 면접에선 "락 = 배리어 포함 계약"이라는 층위 구분만 정확히.

6. **"epoll의 level-triggered와 edge-triggered 차이는?" (함정 연쇄)**
   LT는 "조건이 유지되는 한" 매 대기마다 다시 알려주고(놓쳐도 다음에 또 옴 — 관대함), ET는 "상태가 변한 순간" 한 번만 알려준다.
   ▸ 꼬리 1 "ET의 계약은?" → fd를 non-blocking으로 두고 **EAGAIN이 나올 때까지 드레인** — 버퍼에 데이터를 남기면 새 도착(엣지)이 없는 한 알림이 다시 안 와 그 연결이 영원히 굶는다.
   ▸ 꼬리 2 "그 대가로 뭘 얻나?" → 깨우기 횟수 감소 — 고연결 서버에서 유효한 최적화. "관대함 vs 성능"의 교환이다.
   ▸ 꼬리 3 "IOCP엔 왜 이 구분이 없나?" → 완료 모델은 이벤트를 감시하는 게 아니라 **작업 완료를 큐잉**하므로 트리거 방식이라는 개념 자체가 성립하지 않는다 — 모델 차이를 아는지 확인하는 마지막 관문.

7. **"io_uring과 IOCP를 비교해보라"**
   둘 다 완료 기반이지만 제출·통지 경로가 다르다. IOCP(1990년대 NT부터)는 제출이 개별 시스템 콜(WSARecv 등)이고 수확이 `GetQueuedCompletionStatus`(Ex 계열로 배치 수확 가능). io_uring(2019, Linux)은 제출 큐(SQ)/완료 큐(CQ) 링버퍼를 **유저-커널 공유 메모리**에 두어 제출을 배칭하고, 폴링 모드에선 시스템 콜 자체를 0에 수렴시킬 수 있다.
   ▸ 꼬리 1 "왜 Linux는 늦었나?" → readiness(select→epoll) 전통이 강했고 완료 모델(aio)이 오래 미완이었다 — io_uring이 그 공백을 메운 것. "Windows가 먼저 갔던 길을 Linux가 더 급진적 인터페이스로 따라왔다"고 정리하면 좌표가 선다.
   ▸ 꼬리 2 "공유 링버퍼의 본질 이득은?" → 시스템 콜이라는 고정 비용을 통신 단위에서 떼어내 **상수→상각**으로 바꾼 것 — A-2의 "배칭과 유저 모드 완결" 원칙의 극단형.

8. **"스핀락이 블로킹 락을 이기는 조건을 정확히"**
   (1) 임계 구역이 컨텍스트 스위치 왕복 비용보다 확실히 짧고 (2) 락 홀더가 **다른 코어에서 실행 중**(선점당해 잠들어 있지 않음)이며 (3) 스핀할 CPU가 남아돌 때.
   ▸ 꼬리 1 "싱글 코어에서는?" → 항상 진다 — 홀더가 진행하려면 내가 코어를 내놔야 하는데 스핀은 그걸 막는다. 커널 내부 스핀락이 선점 비활성화와 결합되는 이유.
   ▸ 꼬리 2 "그래서 실제 락은?" → "짧게 스핀 후 잠들기"(adaptive) 하이브리드가 표준 — 두 세계의 손익분기를 런타임에 더듬는다.
   ▸ 꼬리 3 "스핀보다 나은 대기가 있나?" → 기다리는 동안 일을 하는 것. Winters `WaitForCounter`는 스핀 대신 **남의 잡을 훔쳐 실행**(help-stealing)한다 (`JobSystem.cpp:306`~`:318`) — 대기 시간을 처리량으로 환전하는 설계.

9. **"MLFQ는 기아가 생기지 않나?"**
   생긴다 — 짧은 잡이 계속 유입되면 바닥 큐의 CPU-bound 잡이 영원히 굶고, 반대로 퀀텀이 끝나기 직전 I/O를 살짝 걸어 상위 큐에 눌러앉는 조작(gaming)도 가능하다.
   ▸ 꼬리 1 "처방은?" → (1) **주기적 부스트** — 일정 주기마다 전부 최상위 큐로 올려 기아를 리셋 (2) **누적 사용 시간 기반 강등** — 양보 타이밍이 아니라 총 사용량으로 강등해 조작을 무력화.
   ▸ 꼬리 2 "부스트 주기는 어떻게 정하나?" → 짧으면 MLFQ가 RR로 퇴화하고, 길면 기아가 길어진다 — 여기도 은탄환 없는 튜닝 파라미터("부두 상수")임을 인정하는 게 정직한 답.
   ▸ 꼬리 3 "이 질문의 출제 의도는?" → 알고리즘 암기가 아니라 "정책에는 항상 구멍이 있고 패치의 역사가 있다"를 아는지 — 실제 OS들의 상호작용 부스트/공정 배분도 같은 문제의식의 변형이다.

10. **"malloc이 성공했는데 프로세스가 나중에 죽을 수 있는 이유는?" (overcommit 연쇄)**
    Linux는 기본적으로 오버커밋 — malloc은 가상 공간 예약일 뿐이고 물리 프레임은 첫 터치의 페이지 폴트에서 붙는다(26번). 시스템 전체 약속이 물리+스왑을 초과한 채 다들 진짜로 터치하면, 커널 OOM killer가 휴리스틱으로 프로세스를 골라 죽인다 — **할당 성공과 사용 가능이 분리**된 세계다.
    ▸ 꼬리 1 "Windows는?" → 커밋 차지(commit charge) 모델 — 커밋 시점에 RAM+페이지파일 한도를 검사해 초과분은 그 자리에서 실패를 돌려준다. `VirtualAlloc`의 MEM_RESERVE/MEM_COMMIT 구분이 이 모델의 표면.
    ▸ 꼬리 2 "왜 Linux는 오버커밋을 기본으로 하나?" → fork+COW처럼 "약속은 크게, 실사용은 작게"인 워크로드가 흔해서 — 엄격 모드면 fork가 걸핏하면 실패한다. 정책은 지배적 워크로드의 반영이다.
    ▸ 꼬리 3 "게임 개발 함의는?" → "malloc 성공 검사"만으로는 부족한 OS가 있다는 것, 그리고 대량 버퍼는 예약/커밋/터치 시점을 구분해 로딩 단계에서 비용을 치르게 설계하라는 것.

## C. 경험 서술형 (STAR 골격 — 전부 Winters/실습 실제 소재)

> 형식: **S**(상황) **T**(과제) **A**(행동) **R**(결과) + 이어질 꼬리질문. 이 뼈대에 판단 근거를 붙여 말한다.

### C-1. "IOCP 서버를 만들었다는데 구조를 설명해보세요"

**주제: AcceptEx 선게시 + 완료 포트 + 워커 풀 + CONTAINING_RECORD 디스패치**
`Server/Private/Network/IOCPCore.cpp:42`

```text
[초기화]  CreateIoCompletionPort(root, 동시성=workerCount)      IOCPCore.cpp:44
          listen 소켓을 포트에 연결                              :87
          AcceptEx 4건 선게시(연결 오기 전에 미리 요청)           :123~:129
          워커 스레드 N개 기동                                   :120~:121

[워커 루프]  GetQueuedCompletionStatus(INFINITE)                :216
             OVERLAPPED* → CONTAINING_RECORD → IOContext*       :225
             op 분기: Accept → 새 소켓을 포트에 바인드(:202) → WSARecv 게시
                      Recv   → 파서 append → 즉시 다음 WSARecv 재게시
                               (Session.cpp:84~:88)

[종료]  PostQueuedCompletionStatus(null) × 워커 수 = 종료 센티널  :143
```

- **S**: 포트폴리오 서버에 연결당 스레드 없이 다수 세션을 받는 네트워크 코어가 필요했다.
- **T**: Windows 네이티브 비동기 모델로 accept부터 종료까지 전체 수명을 직접 설계.
- **A**: 완료 포트 하나에 listen/세션 소켓을 전부 연결하고, 워커는 완료 패킷만 수확하는 순수 소비자로 뒀다. 요청은 항상 **선게시** — AcceptEx를 미리 걸어 연결 폭주 시에도 수락 공백이 없고, recv 완료 처리 끝에 곧바로 다음 recv를 재게시해 소켓당 정확히 1건의 outstanding recv를 유지한다. send는 세션별 큐+단일 outstanding send로 직렬화해 순서를 보장한다 (`Session.cpp:100`~`:115`).
- **R**: 세션 수와 무관한 고정 워커 수로 동작하는 완료 기반 코어. 종료도 "큐에 null 패킷을 워커 수만큼" — 큐 자체를 제어 채널로 쓰는 IOCP 관용구로 처리.
- ▸ 꼬리질문 대비:
  - "동시성 값(4번째 인자)의 의미는?" → 동시에 실행 상태로 둘 워커 수 힌트. 한 워커가 블로킹되면 커널이 다른 워커를 깨워 활성 수를 유지한다 — 큐이자 스케줄링 협조 장치.
  - "`WSA_IO_PENDING`은 에러인가?" → 아니다. "요청이 걸렸고 완료는 나중에 온다"는 정상 경로 (`Session.cpp:65`). 진짜 에러와의 분기야말로 overlapped 코드의 급소.
  - "OVERLAPPED 하나로 어떻게 컨텍스트를 복원하나?" → OVERLAPPED를 품은 확장 구조체를 게시하고, 완료 시 `CONTAINING_RECORD`로 바깥 구조체를 역산 (`IOCPCore.cpp:225`) — 커널에 포인터를 "맡겨두는" 패턴이라 그 수명 관리가 곧 안전성이다.
  - "send를 왜 큐로 직렬화했나?" → 같은 소켓에 outstanding send를 여러 건 걸면 완료 순서·부분 전송 관리가 복잡해진다. 큐 1건 규약이 정합성을 코드 구조로 보장한다.

### C-2. "멀티스레딩 버그를 잡아본 경험은?"

**주제: Chase-Lev deque의 owner-only-push 규약 위반 — 커널은 유저 규약을 지켜주지 않는다**
`Engine/Private/Core/JobSystem.cpp:18`

- **S**: 병렬 잡 제출 후 간헐적으로 잡이 유실되는 비결정적 사고. 크래시도 에러도 없이 결과만 틀렸다.
- **T**: lock-free 자료구조 자체 결함인지, 사용 규약 위반인지 구분.
- **A**: OS 관점의 핵심은 **스레드 신원**이었다. Chase-Lev deque는 "bottom은 owner 스레드만 만진다"가 정합성의 전제인데, 어떤 스레드든 아무 deque에나 push하고 있었다. `thread_local`로 각 스레드에 워커 인덱스를 부여하고(`:18`), 제출 경로를 신원으로 분기 — 워커 자신이면 자기 deque(owner 경로), 메인/외부 스레드거나 오버플로면 뮤텍스 보호 전역 큐(`:146`~`:158`)로 가는 하이브리드로 재설계했다.
- **R**: race 소멸. "lock-free 사고는 대개 자료구조가 아니라 규약 위반"이며, 그 규약은 컴파일러도 커널도 강제해주지 않아 **타입/TLS로 스스로 강제**해야 한다는 교훈.
- ▸ 꼬리질문 대비:
  - "커널 스레드 ID로 하면 안 되나?" → 되지만 매번 ID→역할 매핑 조회가 필요하다. `thread_local`은 그 매핑을 스레드 자신의 저장소에 상수 시간으로 박아두는 것.
  - "어떻게 원인을 확신했나?" → 코드 추론 대신 스레드별 카운터를 먼저 붙여 어느 스레드가 어느 큐를 만지는지 계측했다 — 비결정 버그일수록 관측 우선.
  - "왜 전역 큐는 뮤텍스로 뒀나?" → 외부 제출은 빈도가 낮아 경합이 작고, 낮은 경합 구간에 lock-free를 또 짜는 건 복잡도 대비 이득이 없다 — 동기화 강도도 예산 배분이다.

### C-3. "파이버(fiber)는 왜 도입했나?"

**주제: 커널 스케줄링을 우회하는 협조적 실행 단위 — WaitForCounter가 워커를 재우면 안 된다**
`Engine/Private/Core/JobSystem.cpp:173`

- **S**: 잡 안에서 다른 잡 묶음을 기다리는(WaitForCounter) 순간, 스레드를 재우면 코어 수에 맞춰둔 워커가 하나 통째로 사라진다 — 스레드는 커널 스케줄링 단위라 "이 잡만 보류"가 불가능하다.
- **T**: 잡 문맥만 보류하고 워커 스레드는 계속 다른 잡을 돌리는 실행 모델.
- **A**: 파이버는 커널이 모르는 유저 모드 실행 문맥(자기 스택+레지스터 저장소)으로, `SwitchToFiber` 호출로만 전환된다 — 전환에 커널 진입이 없어 스레드 스위치보다 훨씬 싸고, 무엇보다 **언제 전환할지를 내가 정한다**(협조적 스케줄링). Winters는 워커를 `ConvertThreadToFiber`로 파이버화하고(`:173`~`:175`) 잡을 별도 파이버에서 실행하는 FiberShell 모드를 구현했다(`:285`~`:290`, 진입/복귀 규약 `:294`~`:303`).
- **R**: public API(Submit/WaitForCounter)를 바꾸지 않고 내부 실행 모델을 교체할 수 있는 토대. 다만 현 단계의 한계를 먼저 말한다 — 매 잡 `CreateFiber/DeleteFiber`라 풀링 전 단계이고, yield 기반 대기는 미완이라 현재 WaitForCounter는 help-stealing(`:306`)이 담당한다. 미완을 스스로 짚으면 과장으로 안 들린다.
- ▸ 꼬리질문 대비:
  - "파이버의 함정은?" → (1) 커널이 파이버를 모르므로 파이버 안에서 블로킹 시스템 콜을 하면 **스레드가 통째로** 막힌다 (2) `thread_local`은 스레드에 붙어 있어, 파이버가 대기 후 다른 스레드에서 재개되는 설계면 TLS 캐싱이 시한폭탄이 된다.
  - "코루틴과의 관계는?" → 둘 다 협조적 문맥 전환. 파이버는 스택 통째 보존(stackful), C++20 코루틴은 컴파일러가 프레임을 힙 객체로 변환(stackless)이라는 구현 축이 다르다.
  - "선점이 없는데 한 잡이 무한 루프면?" → 그 워커는 영영 안 돌아온다 — 협조적 모델은 "모든 참여자가 양보한다"는 신뢰 위에 서고, 그래서 잡은 짧고 유한해야 한다는 규약이 함께 간다.

### C-4. "OS를 직접 설계해본 적 있나요?" — 실습(07장)으로 뒤집기

**주제: "수업으로 배웠다"에서 끝내지 않고 최소 커널 실습으로 메커니즘을 손으로 확인한 스토리**

- **S**: OS를 수업으로만 배워, 페이지 폴트나 컨텍스트 스위치를 "그림으로 아는" 수준과 "코드로 아는" 수준의 차이가 불안했다.
- **T**: 상용 OS를 만드는 게 아니라, 핵심 메커니즘 각각을 **최소 재현**으로 한 번씩 직접 통과하기 — 부팅, 특권 모드 전환, 페이지 테이블 구성, 타이머 인터럽트 기반 문맥 전환.
- **A**: 07장의 실습 순서대로 답한다 — 부트 단계에서 하드웨어가 아무 보호 없이 코드를 실행한다는 것, 페이징을 켜는 순간부터 모든 주소가 내가 만든 테이블을 통과한다는 것, 타이머 인터럽트 핸들러에서 레지스터를 저장하고 다른 태스크 것을 복원하면 그게 곧 컨텍스트 스위치의 전부라는 것을 각각 코드로 확인했다. (면접에서는 07장에서 실제로 만든 범위만 말한다 — 안 해본 단계를 얹지 않는 게 이 카드의 신뢰 조건이다.)
- **R**: "페이지 폴트는 왜 트랩인가", "스위치 때 정확히 무엇을 저장하나" 같은 질문에 정의가 아니라 **구현 순서**로 답할 수 있게 됐다. 그리고 그 감각이 실전 코드 판단으로 이어졌다 — 잡 워커 수 고정(`JobSystem.cpp:50`), 파이버 전환의 비용 우위(C-3), A* 스크래치의 `thread_local`화(`Pathfinder.cpp:463`)가 전부 "커널이 무엇을 해주고 무엇을 안 해주는지"를 근거로 한 선택이다.
- ▸ 꼬리질문 대비:
  - "그 실습이 실무에 무슨 의미가 있나?" → 추상화가 새는 지점(성능, 동시성, 크래시)에서 아래 층을 아는 사람만 원인을 향해 내려갈 수 있다 — C-1~C-3이 전부 그 실례.
  - "그럼 스케줄러도 만들 수 있겠네?" → 라운드로빈 수준의 원리 재현과 상용 스케줄러 설계는 다른 문제라고 선을 긋고, 원리(큐·퀀텀·선점 지점)는 설명 가능하다고 답한다 — 과장하지 않는 답이 제일 안전하다.
  - "가장 놀랐던 지점은?" → "OS가 해주는 줄 알았던 것의 대부분이 하드웨어 기능(모드 비트, MMU, 트랩) 위의 **정책**이더라"는 층위 감각 — 이 한 문장이 실습의 존재 이유를 요약한다.
