# 그래픽스 API의 본질 — DX11 · DX12 · Vulkan

> "DX11과 DX12의 차이가 뭔가요?", "왜 명시적 API로 넘어갔죠?", "본인 엔진은 왜 DX11인가요?"에 대한 답. 이 챕터는 API 이론과 설계 철학을 다룬다. Winters 렌더 파이프라인의 구조 자체(계층·프레임 흐름·FX)는 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md)가 담당하므로 여기서는 중복하지 않고 크로스링크한다.

---

## ① 한 줄 본질

> **"그래픽스 API의 본질은 '상태 설정 + 리소스 바인딩 + 드로우콜'을 GPU가 소비할 커맨드 스트림(command stream)으로 번역하는 것이다. 역사는 이 번역을 누가 책임지느냐의 이동사(移動史)다 — 고정 기능 시대엔 하드웨어가, DX11까지는 드라이버가 해저드 추적·리소스 수명·셰이더 최종 컴파일을 몰래 대행했고, DX12/Vulkan은 그 마법을 걷어내고 PSO·디스크립터·배리어·펜스로 앱에게 책임을 넘겼다. 대가는 코드량과 실수 가능성, 보상은 예측 가능한 프레임 타임과 멀티코어 커맨드 기록이다."**

이 한 문단에서 면접관이 파고들 수 있는 모든 가지(드로우콜 비용, PSO, 디스크립터, 배리어, 트레이드오프)가 나온다. 아래에서 하나씩 편다.

---

## ② 기본 개념

### API가 하는 일의 본질 — GPU와의 대화는 전부 "번역"이다

GPU는 CPU와 **비동기로 도는 별개의 프로세서**다. CPU가 GPU 함수를 직접 호출하는 게 아니라, 메모리의 커맨드 버퍼(command buffer)에 "파이프라인 상태를 이걸로 바꿔라", "이 주소의 버퍼를 정점 입력으로 써라", "삼각형 5,000개 그려라" 같은 **패킷을 쌓아 두면**, GPU의 커맨드 프로세서(command processor)가 링 버퍼(ring buffer)를 통해 그것을 자기 속도로 꺼내 소비한다.

```
[앱 코드]                [드라이버]                    [GPU]
 Draw(...)  ──번역──▶  커맨드 패킷 인코딩  ──제출──▶  커맨드 프로세서가
 SetState(...)          (유저 모드 드라이버)           링 버퍼에서 소비
                              │                          │
                        커널 모드 드라이버가         셰이더 코어·래스터라이저
                        OS(WDDM) 통해 스케줄         ·ROP 유닛 가동
```

그러므로 모든 그래픽스 API는 결국 세 가지 동사만 제공한다.

| 동사 | 의미 | DX11 예 | DX12/Vulkan 예 |
|---|---|---|---|
| 상태 설정 | 파이프라인을 어떤 모드로 굳힐지 | `OMSetBlendState` | PSO 바인딩 |
| 리소스 바인딩 | 셰이더가 읽고 쓸 메모리가 어디인지 | `PSSetShaderResources(t0)` | 디스크립터 테이블 세팅 |
| 실행 | 그려라 / 계산해라 | `DrawIndexed` | `DrawIndexedInstanced` + 커맨드 리스트 제출 |

API 세대 차이는 이 세 동사의 의미가 아니라, **번역 비용을 누가 언제 내는가**의 차이다. 이 관점 하나로 이후 모든 절이 정리된다.

### 진화사 — 번역 책임의 이동

**1기. 고정 기능(fixed-function) / 즉시 모드(immediate mode) — 하드웨어가 정해 준 대로.**
OpenGL 1.x의 `glBegin(GL_TRIANGLES); glVertex3f(...); glEnd();`는 정점 하나마다 함수 호출로 CPU가 드라이버에 데이터를 밀어 넣었다. 변환과 라이팅(T&L)은 하드웨어의 고정 회로였고, 앱이 할 수 있는 건 스위치를 켜고 끄는 것뿐이었다. 표현력의 한계(고정된 라이팅 모델)와 CPU 오버헤드(정점당 함수 호출)가 명백했다.

**2기. 상태 머신(state machine) + 프로그래머블 셰이더 — OpenGL / DX9.**
셰이더가 등장하면서 표현력 문제는 풀렸지만, API 모델은 여전히 **거대한 전역 상태 머신**이었다. `glEnable(GL_BLEND)`를 어디선가 켜 놓으면 화면 반대편 드로우까지 영향을 받는다. 드라이버는 드로우 시점마다 "지금 전역 상태 조합이 유효한가, 하드웨어로 어떻게 번역되나"를 검증해야 했고, 상태 조합은 폭발적이라 검증 비용이 드로우콜마다 붙었다. "드로우콜은 비싸다, 배칭하라(batch)"는 격언이 이 시대에 나왔다.

**3기. DX11(2009) — 상태의 객체화, 그러나 마법은 드라이버에.**
DX11은 상태를 **불변 상태 객체**(rasterizer/blend/depth-stencil state object)로 묶어 미리 만들고 검증해 두게 했고, 생성(device)과 실행(context)을 분리했다. 전역 상태 머신의 혼돈은 크게 줄었다. 그러나 해저드 추적, 리소스 수명, 셰이더 최종 컴파일, 메모리 레지던시는 여전히 드라이버 몫이었다 — 편했지만, 드라이버가 언제 무엇을 하는지 앱이 알 수 없었다.

**4기. 명시적(explicit) API — Mantle(2013) → Metal(2014) → DX12(2015) → Vulkan(2016).**
AMD Mantle이 "드라이버 마법을 걷어내면 드로우콜 비용이 한 자릿수 마이크로초 아래로 내려가고 멀티코어 기록이 가능하다"를 실증했고, 이 설계가 DX12와 Vulkan의 원형이 됐다. 핵심 이관 목록: 파이프라인 상태 조합 선언(PSO), 리소스 바인딩 레이아웃(디스크립터), GPU 해저드(배리어), CPU-GPU 동기화(펜스), 메모리 할당·수명. 전부 앱 책임이 됐다.

```
번역 책임의 이동:
고정기능    상태머신      DX11           DX12/Vulkan
[하드웨어] → [드라이버가 → [드라이버가   → [앱이 선언하고
             매 드로우      객체 검증 +      드라이버는 얇은
             검증]          런타임 마법]     번역만]
```

### DX11 모델 해부 (1) — device와 immediate context는 왜 분리됐나

DX11의 첫 설계 결정은 "만드는 일"과 "그리는 일"의 분리다.

| | `ID3D11Device` | `ID3D11DeviceContext` (immediate) |
|---|---|---|
| 역할 | 리소스/셰이더/상태 객체 **생성** | 상태 바인딩 + 드로우 **기록·제출** |
| 스레드 안전성 | 프리 스레디드(free-threaded) — 여러 스레드에서 동시 호출 가능 | 아님 — 한 시점에 한 스레드만 |
| 빈도 | 로딩 타임 위주 | 매 프레임 수천~수만 번 |

왜 이렇게 나눴나. **생성은 검증이 무겁지만 순서가 없고, 실행은 가볍지만 순서가 전부이기 때문이다.** 텍스처 생성은 언제 어느 스레드에서 해도 되므로 내부 락으로 보호해 병렬을 허용하고, 드로우 스트림은 순서가 곧 의미(나중 상태가 이전 드로우에 영향을 주면 안 됨)라서 직렬을 강제했다. 이 분리 덕에 DX11에서도 **로딩 스레드에서 리소스를 만들면서 메인 스레드가 그리는** 구조는 가능하다. 안 되는 것은 "여러 스레드가 동시에 드로우를 기록"하는 것뿐이다.

DX11에는 지연 컨텍스트(deferred context)라는 멀티스레드 기록 장치가 있긴 했다. 그러나 드라이버가 네이티브 커맨드 리스트를 지원하지 않으면 런타임이 소프트웨어로 재생(re-play)해 immediate context에서 다시 직렬 실행했고, 벤더별 지원 편차가 커서 실효가 제한적이었다. "API에 병렬 기록을 나중에 끼워 넣는 것"의 한계가 DX12 설계의 반면교사가 된다.

### DX11 모델 해부 (2) — 슬롯 기반 바인딩

DX11의 리소스 바인딩은 **스테이지별 고정 슬롯 배열**이다. HLSL 레지스터 네임스페이스와 1:1 대응한다.

| HLSL 레지스터 | 의미 | 바인딩 API 예 | 슬롯 수(FL 11.0 기준) |
|---|---|---|---|
| `b#` | 상수 버퍼(constant buffer) | `VSSetConstantBuffers` | 스테이지당 14 |
| `t#` | 셰이더 리소스 뷰(SRV) — 읽기 | `PSSetShaderResources` | 스테이지당 128 |
| `s#` | 샘플러(sampler) | `PSSetSamplers` | 스테이지당 16 |
| `u#` | 순서 없는 접근 뷰(UAV) — 읽기/쓰기 | `OMSetRenderTargetsAndUnorderedAccessViews` | 8~64 |

```hlsl
// HLSL 쪽 선언 — 슬롯 번호가 곧 계약
cbuffer PerObject : register(b0) { float4x4 gWorld; };
Texture2D    gDiffuse : register(t0);
SamplerState gLinear  : register(s0);
```

슬롯 테이블은 셰이더 스테이지마다 독립이다. 컨텍스트를 하나의 표로 그리면:

```
ID3D11DeviceContext (immediate)
├─ IA : 정점 버퍼 슬롯[32] · 인덱스 버퍼 · 입력 레이아웃 · 토폴로지
├─ VS : b[14] · t[128] · s[16] · 셰이더 객체
├─ PS : b[14] · t[128] · s[16] · 셰이더 객체
├─ CS : b[14] · t[128] · s[16] · u[...]
├─ RS : 래스터라이저 상태 · 뷰포트 · 시저
└─ OM : RTV[8] · DSV · 블렌드/뎁스스텐실 상태
```

이 모델의 본질은 **컨텍스트가 들고 있는 슬롯 테이블이라는 (작아진) 전역 상태**다. 상태 머신 시대보다 정돈됐지만, 드로우마다 슬롯을 갈아 끼우는 API 호출이 여러 번 발생하고, 드라이버는 그때마다 슬롯 테이블을 하드웨어가 실제로 쓰는 형태(GPU 메모리 위의 디스크립터)로 다시 번역해야 한다. 이 반복 번역이 뒤에서 볼 디스크립터 테이블 설계의 동기가 된다.

전역 상태의 고전적 함정도 그대로 남는다 — **상태 누수(state leakage)**. 앞선 패스가 t3에 걸어 둔 SRV를 다음 패스가 해제하지 않으면, 다음 패스의 셰이더가 t3을 안 읽는 동안은 멀쩡하다가 셰이더가 바뀌는 순간 엉뚱한 텍스처가 샘플링된다. 그래서 DX11 렌더러는 패스 경계에서 슬롯을 명시적으로 null로 밀거나, 상태 캐시 계층을 둬서 "실제로 바뀐 슬롯만 API를 호출"하는 관용구를 쓴다. 이 관용구가 결국 "바인딩을 묶음으로 선언하자"는 바인드 그룹 발상의 예고편이다.

### DX11 모델 해부 (3) — 드라이버가 몰래 해주는 네 가지 마법

DX11이 "생산적인" 이유이자 "예측 불가능한" 이유. 면접에서 DX12 얘기를 하려면 이 목록을 먼저 말할 수 있어야 한다 — **DX12가 앱에 넘긴 것이 정확히 이 목록이기 때문이다.**

**마법 1. 해저드 추적(hazard tracking).**
같은 텍스처에 렌더 타겟으로 쓴 직후 셰이더 리소스로 읽으면, 쓰기가 끝나기 전에 읽기가 시작되면 안 된다(write-after-read/read-after-write 해저드). DX11 드라이버는 리소스별로 "누가 언제 썼나"를 추적해 필요한 지점에 파이프라인 플러시·캐시 무효화·압축 해제를 **자동 삽입**한다. 심지어 런타임은 RTV로 바인딩된 리소스가 SRV 슬롯에 남아 있으면 자동으로 언바인딩까지 해 준다(디버그 레이어 경고와 함께). 앱은 해저드의 존재조차 모른 채 코딩할 수 있다.

**마법 2. 리소스 수명(lifetime) 관리.**
드로우가 아직 GPU에서 실행 중인 버퍼를 앱이 `Release()`해도 크래시가 나지 않는다. 드라이버가 내부 참조 카운트로 "GPU가 이 리소스를 다 쓸 때까지" 실제 해제를 미룬다. `Map(WRITE_DISCARD)`도 같은 계열의 마법이다 — GPU가 아직 읽고 있는 버퍼에 CPU가 쓰겠다고 하면, 드라이버가 **새 메모리를 몰래 발급하고 이름만 유지**하는 버퍼 리네이밍(buffer renaming)으로 스톨을 피한다.

**마법 3. 셰이더 최종 컴파일과 상태 패칭.**
HLSL을 `fxc`로 컴파일한 DXBC는 하드웨어 명령어가 아니라 중간 표현이다. 벤더 드라이버가 이를 자기 GPU의 네이티브 ISA로 다시 컴파일하는데, 문제는 **최종 코드가 파이프라인 상태 조합에 의존**한다는 것이다 — 정점 포맷이 셰이더 프롤로그에, 블렌드 모드가 에필로그에 영향을 줄 수 있다. DX11은 조합을 미리 알 수 없으므로 드라이버는 "그 조합이 처음 드로우에 쓰이는 순간" 재컴파일/패치를 한다. 이것이 게임에서 새 이펙트가 처음 나올 때 프레임이 튀는 **셰이더 히칭(hitching)의 정체**다. 앱은 언제 히칭이 날지 제어할 수 없다.

**마법 4. 메모리 레지던시(residency)와 멀티스레드 뒷처리.**
VRAM이 부족하면 OS(WDDM)와 드라이버가 리소스를 시스템 메모리로 페이징하고 필요할 때 되올린다. 또 일부 드라이버는 앱 몰래 워커 스레드를 띄워 커맨드 번역을 병렬화한다. 전부 앱이 통제할 수 없는 지점에서 성능이 출렁이는 원인이 된다.

### 화면에 내보내기 — 스왑체인도 API 이론의 일부다

렌더링의 마지막 단계인 표시(present)도 API 설계 문제다. DXGI 스왑체인(swap chain)의 스왑 이펙트는 크게 두 계열이다.

| | BitBlt 모델 (`DISCARD`/`SEQUENTIAL`) | Flip 모델 (`FLIP_DISCARD`/`FLIP_SEQUENTIAL`) |
|---|---|---|
| Present 방식 | 백버퍼 내용을 OS 합성기(DWM) 쪽 표면으로 **복사** | 백버퍼 **자체를** DWM과 공유해 포인터만 뒤집음(flip) |
| 비용 | 프레임마다 풀스크린 복사 1회 | 복사 없음 — 대역폭·지연 절약 |
| 규율 | 느슨함 — Present 후에도 같은 백버퍼를 계속 써도 됨 | 엄격함 — `BufferCount >= 2`, Present 후 백버퍼 소유권이 넘어가므로 매 프레임 새 백버퍼를 얻어 RTV를 다시 다뤄야 함 |

Flip 모델이 명백히 우월하지만(Win10 이후 권장), "백버퍼를 누가 언제 소유하나"라는 수명 규율을 앱에 요구한다 — 작은 규모의 명시성 이동이 스왑체인에서도 반복되는 것이다. 수직동기(vsync)와 지연 시간(latency)도 여기 걸린다: Present의 sync interval, 큐에 쌓이는 프레임 수(maximum frame latency)가 입력 지연을 결정하므로, 스왑체인 설정은 성능 튜닝이 아니라 **체감 반응성 튜닝**이다.

### 왜 진화해야 했나 — 드로우콜 CPU 오버헤드의 정체

"드로우콜이 비싸다"는 말을 분해하면 GPU가 아니라 **CPU 비용**이다. DX11에서 드로우 하나가 CPU에서 유발하는 일:

1. 슬롯 테이블·상태 객체 조합의 유효성 검증
2. 해저드 추적 자료구조 갱신 (마법 1의 비용)
3. 슬롯 테이블 → 하드웨어 디스크립터 번역 (매번)
4. 필요 시 셰이더 패치 확인 (마법 3의 비용)
5. 커맨드 패킷 인코딩 + 드라이버 내부 락

드로우당 비용은 상태 변화량에 따라 수 마이크로초 수준까지 올라가는데, 문제는 이것이 **한 스레드에서만** 일어난다는 것이다. 대략적 모델로 보면:

```
프레임 제출 CPU 시간 ≒ 드로우콜 수 × 드로우당 CPU 비용 / 제출 스레드 수

DX11:  10,000 draw × ~수 us × 1 thread  → 수십 ms — 16.6ms 예산 초과
DX12:  10,000 draw × ~1 us  × 4 thread  → 수 ms   — 예산 안
```

(수치는 하드웨어·드라이버에 따라 크게 변하는 자릿수 감각용 모델이다. 면접에서도 "정확한 수치"가 아니라 "왜 스레드 수가 분모에 들어갈 수 있게 됐는가"를 말하는 게 맞다.)

2010년대 게임은 오브젝트 수가 폭증하는데 CPU는 코어 수로 성장했다. 렌더 제출이 싱글 스레드에 묶여 있으면 **코어가 몇 개든 렌더링은 한 코어 성능에 갇힌다.** 드라이버 마법(1~4번 비용)을 앱이 미리 계산해 둔 정보로 대체하면 드로우당 비용 자체가 내려가고, 기록을 여러 스레드로 나누면 분모가 생긴다. 이것이 명시적 API의 존재 이유 전부다.

### DX12/Vulkan 모델 해부 (1) — 커맨드 리스트: 기록과 제출의 분리

```
[스레드 A] CommandList A 기록 ┐
[스레드 B] CommandList B 기록 ├─▶ [메인] Queue에 순서대로 제출 ─▶ GPU
[스레드 C] CommandList C 기록 ┘
```

DX12의 `ID3D12GraphicsCommandList`(Vulkan의 `VkCommandBuffer`)는 **드로우 명령을 담는 값 객체**다. 기록(record)은 어느 스레드에서든 병렬로 하고, 제출(submit)만 큐에서 직렬화한다. DX11 immediate context가 "기록=제출"이었던 것과 대비된다.

기록된 명령의 메모리는 커맨드 할당자(command allocator / `VkCommandPool`)가 소유하는데, 여기서 명시적 API의 성격이 바로 드러난다: **GPU가 그 커맨드를 다 소비하기 전에는 할당자를 리셋하면 안 되고, 그걸 확인하는 건 앱 책임이다.** 그래서 인플라이트 프레임(frames in flight) 수만큼 할당자를 두고, 펜스로 "N프레임 전 커맨드를 GPU가 끝냈는지" 확인한 뒤에야 그 프레임의 할당자를 재사용한다. DX11에서는 드라이버 마법 2가 해 주던 일이다.

### DX12/Vulkan 모델 해부 (2) — PSO: 왜 파이프라인 상태를 하나의 불변 객체로 미리 굽나

파이프라인 상태 객체(PSO, pipeline state object / `VkPipeline`)는 셰이더 바이트코드 + 입력 레이아웃 + 블렌드 + 래스터라이저 + 뎁스스텐실 + 렌더 타겟 포맷을 **한 덩어리의 불변 객체**로 생성 시점에 묶는다.

왜 이렇게까지 묶나. 마법 3에서 본 대로 **하드웨어에서 이 상태들은 서로 독립적이지 않기 때문이다.** 최종 네이티브 셰이더 코드는 상태 조합의 함수다:

```
네이티브 ISA = f(셰이더 바이트코드, 정점 포맷, 블렌드 모드, RT 포맷, ...)
```

DX11은 f의 입력이 드로우 직전에야 확정되므로 드라이버가 런타임에 f를 평가했고(→ 히칭), DX12는 "앱이 조합을 미리 선언하라, 그러면 f를 **로딩 타임에** 평가해 주겠다"로 계약을 바꾼 것이다. PSO 생성은 수십 ms가 걸려도 되는 로딩 작업이 되고, 드로우 시점에는 이미 구워진 네이티브 코드를 포인터 하나로 바인딩만 한다. 드로우가 빨라지고 히칭 시점이 예측 가능해진다.

대가는 **조합 폭발**이다. 블렌드 모드 하나만 바꿔도 별개의 PSO가 필요하므로, 실전 엔진은 수천~수만 개 PSO를 관리하고, 디스크에 PSO 캐시를 저장하며, 어떤 조합이 쓰일지 사전 수집(pre-warming)한다. 이걸 놓치면 런타임 PSO 생성이 일어나 **DX11보다 심한 스터터**가 난다 — UE 계열 게임들의 악명 높은 "셰이더 컴파일 스터터"가 바로 이 실패 모드다. 마법을 없앤 게 아니라 마법의 비용을 낼 시점을 앱이 선택하게 된 것이고, 선택을 안 하면 최악의 시점(첫 사용 프레임)에 낸다.

### DX12/Vulkan 모델 해부 (3) — 디스크립터: 슬롯 → 테이블 → 바인드리스

하드웨어 관점에서 "바인딩된 텍스처"의 실체는 GPU 메모리에 있는 작은 구조체(주소·포맷·크기 등을 담은 디스크립터)다. 셰이더는 그 구조체를 읽어 리소스에 접근한다. 그렇다면 바인딩 API의 진화는 "이 구조체 배열을 누가 어떻게 채우나"의 진화다.

```
DX11 슬롯:      드로우마다 슬롯 API 호출 → 드라이버가 매번 디스크립터로 번역
DX12 테이블:    앱이 디스크립터 힙(GPU 메모리)에 미리 써 두고,
                드로우 때는 "테이블 시작 포인터" 하나만 루트에 세팅
바인드리스:     아예 거대한 힙 하나를 상시 바인딩하고,
                셰이더에 인덱스(그냥 정수)만 넘겨 원하는 리소스를 집게 함
```

DX12의 디스크립터 힙(descriptor heap) + 루트 시그니처(root signature), Vulkan의 디스크립터 셋(descriptor set) + 파이프라인 레이아웃(pipeline layout)이 가운데 단계다. 루트 시그니처/파이프라인 레이아웃은 "셰이더가 어떤 형태의 바인딩을 기대하는가"라는 **함수 시그니처 선언**이고, 디스크립터 테이블/셋은 그 시그니처에 넘기는 **인자 묶음**이다. 묶음 단위로 갈아 끼우므로 드로우당 바인딩 비용이 호출 N번에서 포인터 세팅 1번으로 줄어든다.

함수 시그니처 비유를 그대로 코드로 옮기면:

```
// "이 파이프라인은 이런 인자를 받는다"는 선언 (개념 의사코드)
RootSignature = {
    [0] 32비트 상수 몇 개        // 초저비용, 매 드로우 바뀌는 값 (오브젝트 인덱스 등)
    [1] CBV 루트 디스크립터      // 주소 하나 — 자주 바뀌는 상수 버퍼
    [2] 디스크립터 테이블(t0~t7) // SRV 8개 묶음의 힙 오프셋
}
// 드로우 루프에서는:
SetGraphicsRootSignature(rs);
SetGraphicsRoot32BitConstant(0, objectIndex, 0);
SetGraphicsRootDescriptorTable(2, heapOffsetOfMaterial);  // 포인터 1개
DrawIndexedInstanced(...);
```

루트에 직접 넣는 인자는 접근이 가장 빠르지만 루트 시그니처 전체 크기가 제한되어 있어(64 DWORD), "자주 바뀌고 작은 것은 루트에, 크고 묶이는 것은 테이블로"라는 배치 설계가 성능 항목이 된다. 갱신 빈도별로 셋을 나누는 관용구(per-frame / per-pass / per-material / per-draw)도 여기서 나온다 — 바인딩이 API 호출에서 **데이터 레이아웃 설계 문제**로 바뀌었다는 뜻이다.

끝점이 바인드리스(bindless)다 — SM6.6의 `ResourceDescriptorHeap[index]`, Vulkan의 디스크립터 인덱싱(descriptor indexing, 1.2 코어). 리소스 식별이 "바인딩"이라는 CPU 행위에서 "인덱스"라는 데이터로 바뀌면, GPU 주도 렌더링(GPU-driven rendering)처럼 **그릴 목록 자체를 GPU가 컴퓨트로 만드는** 구조가 열린다. 방향성은 일관된다: 바인딩을 CPU 작업에서 데이터로 격하시키는 것.

### DX12/Vulkan 모델 해부 (4) — 명시적 동기화: 마법 1의 청구서

드라이버 해저드 추적을 없앴으므로 앱이 세 종류의 동기화 도구를 직접 다룬다.

| 도구 | 동기화 대상 | 예 |
|---|---|---|
| 펜스(fence) | CPU ↔ GPU 타임라인 | "3프레임 전 커맨드 다 끝났으면 그 할당자 재사용" |
| 리소스 배리어(barrier) / 이미지 레이아웃 전환 | GPU 내부, 리소스 상태 전환 | "이 텍스처, 렌더 타겟 쓰기 → 셰이더 읽기로 전환" |
| 세마포어(semaphore, Vulkan) | GPU 큐 ↔ 큐 | "그래픽스 큐는 스왑체인 이미지 획득 후에 시작" |

배리어가 하드웨어에서 실제로 하는 일은 세 가지다: (1) 앞선 쓰기가 끝날 때까지 대기(실행 의존성), (2) 캐시 플러시/무효화(GPU 캐시는 CPU처럼 자동 일관성이 없다), (3) 필요 시 데이터 재배치 — 렌더 타겟은 압축 포맷으로, 샘플링은 다른 레이아웃으로 저장되는 경우가 있어 전환 시 압축 해제(decompress)가 일어날 수 있다. DX11 드라이버는 이걸 보수적으로(최악 케이스 가정) 자동 삽입했고, DX12는 앱이 정확한 지점에 정확한 전환만 선언해 낭비를 없앤다 — 단, 빠뜨리면 검증 레이어 없는 릴리스에서 GPU 벤더별로 다르게 깨지는 최악의 버그가 된다.

전형적인 그림자 맵 프레임을 배리어 관점에서 쓰면:

```
[그림자 패스]  depthTex: DEPTH_WRITE 상태로 렌더
   │
   ├── Barrier(depthTex, DEPTH_WRITE → PIXEL_SHADER_RESOURCE)
   │      = "깊이 쓰기 전부 끝내고, 깊이 캐시 플러시하고,
   │         (하드웨어에 따라) 압축 데이터를 샘플링 가능 형태로"
   │
[메인 패스]    depthTex를 t# 슬롯에서 샘플링
   │
   └── 프레임 끝: Barrier(backBuffer, RENDER_TARGET → PRESENT)
```

펜스 쪽의 대표 패턴은 프레임 파이프라이닝이다. CPU가 GPU보다 1~2프레임 앞서 기록하도록 허용하되, 같은 슬롯의 리소스(커맨드 할당자·동적 버퍼)를 재사용하기 직전에만 펜스로 대기한다:

```
프레임 N   기록 → 제출 → Signal(fence, N)
프레임 N+1 기록 → 제출 → Signal(fence, N+1)
프레임 N+2 기록 전: Wait(fence >= N)   // 슬롯 0을 재사용하기 위한 최소 대기
```

이 구조를 잘못 잡는 두 극단이 모두 흔한 사고다 — 매 프레임 GPU 완료를 기다리면(`WaitForGpu` 남발) 파이프라이닝이 죽어 CPU/GPU가 교대로 노는 직렬 실행이 되고, 대기 없이 재사용하면 GPU가 읽는 중인 메모리를 덮어쓴다.

### DX12/Vulkan 모델 해부 (5) — 명시적 메모리

DX11의 "리소스 생성 = 메모리 할당"이 분리된다. DX12는 힙을 만들고 그 위에 배치 리소스(placed resource)를 서브할당할 수 있고, 레지던시도 `MakeResident/Evict`와 메모리 예산 질의로 앱이 관리한다. Vulkan은 메모리 타입(device-local, host-visible 등)을 질의해 골라 `vkAllocateMemory` 후 리소스에 바인딩한다. 실무에서는 이 계층이 너무 원시적이어서 D3D12MA / VMA(Vulkan Memory Allocator) 같은 할당자 라이브러리가 사실상 표준이다. 요점: **GPU 메모리 할당자도 이제 엔진의 자료구조**가 됐다는 것 — 프레임 단위 링 할당, 서브할당, 조각화 관리가 전부 앱 설계 문제다.

### DX12 vs Vulkan — 같은 사상, 다른 정치

둘은 같은 세대(Mantle 후예)라 개념이 거의 1:1 대응하지만, 차이가 면접 단골이다.

| 축 | DX12 | Vulkan |
|---|---|---|
| 플랫폼 | Windows 10+ / Xbox | Windows·Linux·Android·Switch, macOS는 MoltenVK 번역 계층 |
| 거버넌스 | Microsoft 단일 주도, Agility SDK로 신기능 배포 | Khronos 표준 + 벤더 확장(extension) 모델 |
| 셰이더 IR | DXIL | SPIR-V |
| 바인딩 | 루트 시그니처 + 디스크립터 힙 | 파이프라인 레이아웃 + 디스크립터 셋 |
| 큐 모델 | Direct/Compute/Copy 타입 고정 | 큐 패밀리(queue family)를 하드웨어별로 질의 |
| 렌더패스 | 필수 아님 (선택적 렌더패스 API 존재) | 1.0에서는 렌더패스/서브패스가 1급 개념, 1.3 dynamic rendering으로 완화 |

렌더패스 차이는 이유가 재미있다. Vulkan은 모바일까지 커버해야 했고, 모바일 GPU는 타일 기반(tile-based) 렌더링이라 **"이 어태치먼트를 언제 로드/스토어하고 패스 간 데이터가 온칩 타일 메모리에 머물 수 있는가"를 미리 선언**받으면 대역폭을 크게 아낄 수 있다. 렌더패스/서브패스는 그 선언 장치였다. 데스크톱에서는 이 정보의 이득이 작아 보일러플레이트로만 느껴졌고, 결국 1.3에서 dynamic rendering으로 "그냥 그리기 시작"이 허용됐다. **API 개념 하나에도 타깃 하드웨어의 정치가 들어 있다**는 좋은 사례. 확장 모델도 마찬가지다 — Vulkan의 확장 지옥은 파편화된 하드웨어 생태계의 반영이고, DX12의 깔끔함은 플랫폼 독점의 반영이다.

셰이더 툴체인은 수렴 중이다 — 최신 HLSL 컴파일러 DXC는 같은 HLSL 소스에서 DXIL(DX12)과 SPIR-V(Vulkan)를 모두 뽑을 수 있어, 실무에서는 HLSL을 단일 소스로 두고 백엔드별 IR로 내리는 구성이 흔하다. Winters의 RHI에 셰이더가 "스테이지 + 바이트코드 블롭"으로 추상화된 것도 같은 전제다(바이트코드의 정체는 백엔드가 해석).

Metal은 같은 명시적 세대지만 애플 단일 하드웨어 대상이라 좀 더 높은 추상 수준을 유지한다(자동화 구간이 상대적으로 많음). "명시성의 정도"는 이분법이 아니라 스펙트럼이고, 각 API가 스펙트럼 위 어디에 설지는 커버해야 하는 하드웨어의 다양성이 결정한다 — 단일 벤더일수록 API가 대신해 줄 수 있는 것이 많다.

### 냉정한 트레이드오프 — DX12는 빠른 API가 아니라 "제어를 주는" API다

| | DX11 | DX12/Vulkan |
|---|---|---|
| 코드량(삼각형 하나까지) | 수백 줄 | 천 줄 안팎 |
| 드라이버 최적화 | 벤더가 수십 년 튜닝한 마법 | 얇음 — 앱 책임 |
| 프레임 타임 | 예측 불가 스파이크 가능 | 잘 쓰면 예측 가능 |
| 멀티스레드 기록 | 사실상 불가 | 설계의 중심 |
| 잘못 썼을 때 | 드라이버가 어느 정도 구제 | DX11보다 느리거나, 벤더별로 다르게 깨짐 |

강조할 것: **명시적 API로 옮기기만 하면 빨라진다는 건 거짓이다.** 배리어를 보수적으로 남발하면 DX11 드라이버보다 못한 직렬화가 생기고, PSO 사전 준비를 안 하면 히칭이 더 심하고, 디스크립터 관리를 매 드로우 재구성하면 슬롯 모델보다 느리다. DX11 드라이버는 각 벤더 최고 전문가들이 튜닝한 코드다 — 그걸 이기려면 앱이 드라이버보다 자기 워크로드를 잘 안다는 전제를 실제 코드로 증명해야 한다. 그래서 실무 선택 기준은:

- **드로우콜 수천 개 이하 + 소규모 팀** → DX11(또는 그 수준의 추상화)로 충분. 병목이 제출이 아닌 경우가 대부분.
- **드로우콜 수만 개, GPU 주도 렌더링, 멀티스레드 렌더러** → 명시적 API가 아니면 구조적으로 불가능.
- **크로스 플랫폼 출시** → 어차피 플랫폼별 API가 갈리므로(PC=DX12/Vulkan, PS=GNM, Switch=Vulkan/NVN, 모바일=Vulkan/Metal) 다음 절의 RHI가 전제가 된다.

### RHI — 추상화 계층이 존재하는 이유와 "어느 세대로 추상화하나"

RHI(render hardware interface)는 엔진 렌더 코드와 플랫폼 API 사이의 인터페이스 계층이다. 존재 이유는 세 가지다.

1. **멀티 백엔드** — 위 표처럼 플랫폼마다 API가 다르다. 렌더러를 API 수만큼 다시 쓸 수는 없다.
2. **교체와 검증** — 같은 프레임을 두 백엔드로 그려 비교하면 "엔진 버그인가 API 사용 버그인가"가 분리된다. 널(null) 백엔드는 테스트를 가능케 한다.
3. **개념의 고정** — 렌더 코드가 의존할 개념(커맨드 리스트, 패스, 파이프라인, 바인딩 그룹)을 API 유행과 분리해 고정한다.

핵심 설계 질문은 **"어느 세대의 개념으로 인터페이스를 자르는가"**다. 정답은 업계에서 수렴됐다: **최신 세대(명시적 API)의 개념으로 자르고, 구세대 백엔드가 그걸 흡수한다.** 이유는 비대칭성이다 — DX12형 개념(PSO, 바인드 그룹, 배리어)을 DX11로 번역하는 건 쉽다(배리어는 무시하면 되고, PSO는 상태 객체 묶음으로, 바인드 그룹은 슬롯 콜 여러 번으로 풀면 된다). 반대로 DX11형 개념(슬롯, 암시적 해저드)을 DX12로 올리는 건 드라이버 마법을 앱에서 재구현하는 일이라 지옥이다. 언리얼 RHI가 DX12 세대에 맞춰 인터페이스를 재편(RHI 커맨드 리스트, PSO 캐시)한 것, WebGPU가 아예 "명시적 API들의 공통 분모"를 표준으로 굳힌 것이 같은 결론이다.

## ③ 심화 — 꼬리질문 대비

**"드로우콜이 비싸다는데, 정확히 어디서 비용이 발생하나?"**
GPU가 아니라 CPU다. 상태 조합 검증, 해저드 추적 갱신, 슬롯→디스크립터 번역, (필요 시) 셰이더 패치, 커맨드 패킷 인코딩, 드라이버 락 — 전부 드로우당 CPU 작업이고 DX11에서는 한 스레드에 몰린다. 그래서 해법도 두 갈래다: 드로우당 비용 절감(PSO·디스크립터로 사전 계산)과 병렬화(커맨드 리스트 멀티스레드 기록). 인스턴싱/배칭은 드로우콜 수 자체를 줄이는 직교적인 해법.

**"DX11 device는 스레드 세이프한데 context는 왜 아닌가?"**
생성은 순서 무관·저빈도라 내부 락 비용을 감수하고 병렬을 허용했고, 드로우 스트림은 순서가 곧 의미인 데다 매 프레임 수만 번 호출이라 락을 걸면 그 자체가 병목이 된다. "순서가 필요한 것은 직렬로, 순서가 없는 것은 병렬로"라는 일반 원칙의 API 표현이다.

**"DX11에도 deferred context가 있는데 왜 멀티스레드 문제가 해결 안 됐나?"**
기록은 나눠도 드라이버 번역·검증(진짜 비용)이 immediate context 제출 시점에 직렬로 남았기 때문이다. 드라이버 네이티브 커맨드 리스트 지원이 벤더별로 갈렸고, 미지원 드라이버에서는 런타임이 소프트웨어로 재생해 오히려 오버헤드가 붙었다. 해저드 추적·리소스 수명 같은 마법이 API 계약에 남아 있는 한 드라이버는 전역 검증을 포기할 수 없다 — 계약 자체를 바꾼 것이 DX12다.

**"PSO는 왜 불변인가? 상태 하나만 바꾸고 싶으면 어떡하나?"**
최종 네이티브 셰이더 코드가 상태 조합의 함수라서다. 가변이면 드라이버가 변경 시점마다 재컴파일 가능성을 검사해야 해서 DX11의 히칭 문제로 회귀한다. 불변이므로 조합별로 PSO를 따로 만들고, 그래서 PSO 캐싱·해싱·사전 수집이 엔진 서브시스템이 된다. 단 모든 상태가 PSO에 굳는 건 아니다 — 뷰포트·시저·스텐실 참조값·블렌드 팩터처럼 하드웨어 레지스터 몇 개로 끝나는 것들은 동적 상태로 남겨 뒀다(Vulkan은 dynamic state 확장으로 이 목록을 계속 늘려 왔다). "컴파일에 영향을 주는가"가 PSO에 들어가느냐의 기준선이라고 이해하면 된다.

**"명시적 API로 갔는데 셰이더 컴파일 스터터는 왜 오히려 유명해졌나?"**
DX11에서는 드라이버가 재컴파일을 부분 패치 등으로 완화하며 몰래 분산시켰지만, DX12/Vulkan에서 런타임 PSO 생성은 풀 컴파일이라 미스 한 번의 비용이 더 크고 온전히 앱 탓이다. 콘텐츠가 방대한 게임은 모든 상태 조합을 사전에 알기 어려워 첫 조우 시점에 컴파일이 터진다. 해법은 조합 수집 파이프라인, 디스크 PSO 캐시, 비동기 컴파일 + 폴백 PSO(일단 비슷한 걸로 그리고 뒤에서 굽기). "책임 이동"의 비용을 보여주는 가장 좋은 사례라 면접에서 트레이드오프 논증으로 쓰기 좋다.

**"리소스 배리어가 하드웨어에서 실제로 하는 일은?"**
세 가지: (1) 실행 의존성 — 앞선 쓰기 완료까지 후속 작업 대기, (2) 캐시 일관성 — GPU의 각 캐시(색상 쓰기, 텍스처 읽기 경로 등)는 자동 일관성이 없어 플러시/무효화가 필요, (3) 레이아웃/압축 전환 — 렌더 타겟용 압축 데이터를 샘플링 가능한 형태로 재배치. DX11은 드라이버가 최악 케이스를 가정해 자동 삽입했고, 명시적 API는 앱이 필요한 지점에만 선언해 과잉 동기화를 없앤다. 반대로 앱이 과잉 선언하면 DX11보다 느려진다.

**"fence와 barrier와 semaphore의 차이를 정리해 보라."**
축이 다르다. 펜스는 CPU와 GPU 타임라인 사이(제출한 작업이 어디까지 끝났는지 CPU가 확인/대기), 배리어는 한 커맨드 스트림 안에서 리소스 상태 전환(GPU 내부 해저드), 세마포어는 GPU 큐와 큐 사이(Vulkan에서 스왑체인 획득→렌더→표시 체인). "누가 누구를 기다리는가"로 답하면 깔끔하다: CPU가 GPU를 = 펜스, GPU 명령이 앞선 명령을 = 배리어, 큐가 큐를 = 세마포어.

**"인플라이트 프레임 수만큼 command allocator가 왜 필요한가?"**
할당자는 기록된 커맨드의 실제 메모리를 소유하는데, GPU가 그 커맨드를 아직 소비 중일 때 리셋하면 실행 중인 명령 메모리를 덮어쓰게 된다. CPU가 프레임 N+1, N+2를 미리 기록하려면(파이프라이닝) 프레임별 할당자를 링으로 돌리고, 펜스 값으로 "이 슬롯의 이전 커맨드를 GPU가 끝냈는지" 확인한 뒤 리셋해야 한다. DX11에서는 드라이버 수명 관리가 대신하던 일이 자료구조로 드러난 것.

**"디스크립터 테이블이 슬롯보다 빠른 이유는? 바인드리스는 뭐가 더 좋나?"**
슬롯 모델은 드로우마다 API 호출 N번 + 드라이버의 슬롯→하드웨어 디스크립터 번역이 반복된다. 테이블 모델은 디스크립터를 GPU 메모리에 미리 써 두고 드로우 때 테이블 포인터 하나만 바꾼다 — 하드웨어의 실제 동작(디스크립터는 GPU 메모리의 작은 구조체)에 API를 맞춘 것. 바인드리스는 한 발 더 나가 리소스 식별을 셰이더에 넘기는 정수 인덱스로 만들어, 머티리얼 데이터에 텍스처 인덱스를 박아 두거나 GPU 컴퓨트가 그릴 목록을 직접 만드는(GPU-driven) 구조를 연다. 대가는 검증 난이도 — 잘못된 인덱스는 CPU에서 잡을 수 없다.

**"DX12로 포팅하면 무조건 빨라지나?"**
아니다. 이득은 (1) CPU 제출이 병목이고 (2) 멀티스레드 기록·사전 PSO·정확한 배리어를 실제로 구현했을 때만 나온다. 배리어 남발, 매 드로우 디스크립터 재구성, PSO 런타임 생성이면 수십 년 튜닝된 DX11 드라이버보다 느려진다. GPU 바운드(픽셀 비용이 병목)인 게임은 제출 모델을 바꿔도 거의 안 빨라진다. "명시적 API는 성능을 주는 게 아니라 성능을 낼 권한과 책임을 준다"가 정확한 표현.

**"Vulkan 렌더패스는 왜 있었고 왜 dynamic rendering으로 완화됐나?"**
타일 기반 모바일 GPU를 위해서다. 패스와 어태치먼트의 로드/스토어, 서브패스 간 데이터 흐름을 미리 선언하면 드라이버가 중간 결과를 DRAM에 내리지 않고 온칩 타일 메모리에 유지하도록 스케줄할 수 있다 — 모바일에서 대역폭은 곧 전력이다. 데스크톱 즉시 렌더링(immediate-mode) GPU에서는 이 선언의 이득이 작아 보일러플레이트만 남았고, 1.3에서 dynamic rendering으로 렌더패스 객체 없이 그리기가 허용됐다. API 개념이 특정 하드웨어 계열의 요구에서 태어나고 하드웨어 지형이 바뀌면 물러난다는 사례.

**"DX11 Map(WRITE_DISCARD)을 호출하면 내부에서 무슨 일이 일어나나?"**
GPU가 그 버퍼를 아직 읽는 중일 수 있으므로, 드라이버는 CPU를 세우는 대신 새 메모리 조각을 발급해 포인터를 돌려주고 이후 드로우는 새 조각을 참조하게 한다(버퍼 리네이밍/버저닝). 옛 조각은 GPU 소비가 끝나면 회수. 그래서 동적 버퍼는 DISCARD로 열어야 하고, NO_OVERWRITE는 "내가 GPU가 읽는 영역을 안 건드린다고 약속"해 리네이밍을 생략하는 모드다. 명시적 API에서는 이 리네이밍을 링 버퍼 할당자로 앱이 직접 짠다 — 마법이 자료구조로 바뀌는 또 하나의 사례.

**"RHI를 설계한다면 인터페이스를 어느 세대 개념으로 자르겠나?"**
명시적 세대(커맨드 리스트, PSO, 바인드 그룹, 배리어)로 자른다. 신세대 개념→구세대 백엔드 번역은 하향이라 쉽다: 배리어는 no-op(DX11 드라이버가 어차피 추적), PSO는 상태 객체 묶음 캐시로, 바인드 그룹은 슬롯 콜 언롤로. 반대 방향은 드라이버 마법의 재구현이라 사실상 불가능하다. 언리얼 RHI의 DX12 세대 재편과 WebGPU의 표준화가 같은 결론의 증거.

**"커맨드 버퍼는 어디서 만들어지고 어디서 제출되나 — 유저 모드와 커널 모드의 역할은?"**
API 호출을 커맨드 패킷으로 번역하는 것은 유저 모드 드라이버(UMD, 프로세스마다 로드되는 DLL)이고, 완성된 버퍼를 OS 그래픽 커널(WDDM)을 거쳐 하드웨어 큐에 스케줄하는 것이 커널 모드 드라이버(KMD)다. 유저→커널 전환은 비싸므로 드로우마다가 아니라 버퍼 단위로 배치 제출된다. 이 구조를 알면 "드로우콜마다 커널 진입해서 느리다"는 흔한 오답을 피할 수 있다 — 비용의 주범은 커널 전환이 아니라 유저 모드 드라이버의 검증·번역·마법이다.

## ④ Winters에서의 적용

Winters는 이 챕터의 결론 — "인터페이스는 명시적 세대 개념으로 자르고, DX11 백엔드가 흡수한다" — 를 그대로 구현한 실물이다. 구조 전체 그림은 [engine/03_rendering_pipeline.md](../engine/03_rendering_pipeline.md) ②·③절에 있으므로, 여기서는 API 이론과 맞닿는 지점만 짚는다.

### RHI 계약이 DX12/Vulkan 세대 개념으로 잘려 있다

`Engine/Public/RHI/IRHIDevice.h:12`의 `IRHIDevice`는 `CreateCommandList`(33행)·`CreatePipeline`(88행)·`CreateRenderPass`(91행)·`CreateBindGroupLayout/CreateBindGroup`(94~98행)을 계약으로 갖는다. 커맨드 리스트, 파이프라인, 렌더패스, 바인드 그룹 — 전부 명시적 API 세대의 어휘다. `Engine/Public/RHI/IRHICommandList.h:7`의 `IRHICommandList`도 `Begin/End`(12~13행), `BeginRenderPass`(15행), `SetPipeline`(18행), `SetBindGroup`(19행), `TransitionResource`(34~35행)로, WebGPU/Vulkan의 커맨드 버퍼 계약과 같은 골격이다. 리소스 상태 열거도 `Engine/Public/RHI/RHITypes.h:82`의 `eRHIResourceState`(Common/RenderTarget/DepthWrite/ShaderResource/Present...)로 DX12 리소스 스테이트 모델을 그대로 반영한다.

면접 답변 형태로: "DX11로 시작했지만 인터페이스를 DX11 슬롯 모델로 자르지 않고 커맨드 리스트·PSO·바인드 그룹·배리어 세대로 잘랐습니다. 하향 번역은 쉽고 상향 번역은 드라이버 마법의 재구현이기 때문입니다."

### PSO 발상 — RHIPipelineDesc 하나가 두 백엔드에서 다르게 구워진다

`Engine/Public/RHI/RHIDescriptors.h:102`의 `RHIPipelineDesc`는 VS/PS 핸들 + 바인드 그룹 레이아웃 + 입력 요소 + 토폴로지/블렌드/뎁스/컬 + RT·DSV 포맷(104~122행)을 **한 구조체**로 묶는다. 즉 "최종 셰이더 코드에 영향을 주는 상태 조합의 선언"이라는 PSO의 정의를 데이터로 옮긴 것이다. 이 desc가 DX12 백엔드에서는 `Engine/Private/RHI/DX12/DX12Device.cpp:1555~1574`에서 `D3D12_GRAPHICS_PIPELINE_STATE_DESC`로 1:1 채워져 `CreateGraphicsPipelineState`(1574행)로 진짜 PSO가 되고, DX11 백엔드에서는 상태 객체 묶음으로 풀린다. 같은 선언, 두 번역 — "PSO는 왜 미리 굽나"를 본인 코드로 증명할 수 있는 지점이다.

### 바인드 그룹 → 슬롯 언롤 — 하향 번역의 실물

`Engine/Public/RHI/RHIDescriptors.h:154~176`의 `RHIBindGroupLayoutDesc/RHIBindGroupResource/RHIBindGroupDesc`는 디스크립터 셋 발상(슬롯·타입·가시성의 묶음 선언)이다. DX11 백엔드의 `SetBindGroup` 구현(`Engine/Private/RHI/DX11/CDX11Device.cpp:576~629`)은 이 묶음을 순회하며 `VSSetConstantBuffers`(593행)·`PSSetShaderResources`(609행)·`PSSetSamplers`(623행)로 **슬롯 콜로 언롤**한다. "테이블 포인터 1번 세팅"이 "슬롯 콜 N번"으로 번역되는 것 — 하향 번역이 왜 쉬운지를 코드 한 함수로 보여줄 수 있다.

### 배리어 no-op — 드라이버 마법의 증거

같은 파일 `Engine/Private/RHI/DX11/CDX11Device.cpp:710~711`에서 `TransitionResource` 두 오버로드는 빈 몸통이다. 계약에는 배리어 자리가 있지만 DX11에서는 드라이버가 해저드를 추적하므로 할 일이 없다. "DX11 드라이버가 해저드 추적을 대행한다"는 이 챕터의 명제가 no-op 함수 두 줄로 물화된 지점이라, 면접에서 마법 목록을 말한 뒤 "그래서 제 DX11 백엔드의 배리어는 의도적으로 no-op입니다"로 잇기 좋다.

### DX12 백엔드 — 명시적 동기화를 실제로 짠 경험

`Engine/Private/RHI/DX12/DX12Device.cpp`에는 이 챕터 (4)절의 도구들이 전부 실물로 있다.
- 펜스 생성과 이벤트(`CreateFence`, 2048~2055행), CPU 대기(`WaitForFenceValue`, 2128~2138행 — `GetCompletedValue` 확인 후 `SetEventOnCompletion`+`WaitForSingleObject`).
- 프레임별 커맨드 할당자 링: `BeginFrame`(2057행)이 `WaitForFrame(m_iFrameIndex)`(2062행)로 **그 슬롯의 이전 프레임 커맨드를 GPU가 끝냈는지 확인한 뒤에야** 할당자를 `Reset`(2064~2065행)한다 — "왜 인플라이트 수만큼 할당자인가"의 답이 코드 순서 그 자체다.
- 리소스 배리어: 백버퍼를 `PRESENT→RENDER_TARGET`(2075~2079행)으로 열고 `RENDER_TARGET→PRESENT`(2106~2110행)로 닫은 뒤 `ExecuteCommandLists`+`Signal`(2114~2119행)로 제출과 펜스 값 기록.
- 디스크립터 힙: `CreateDescriptorHeaps`(1815행)에서 SRV/샘플러 힙(+스테이징 힙)을 만들고 프레임 시작 시 `SetDescriptorHeaps`(2071~2072행)로 바인딩.

즉 "DX12 해봤나"는 질문에 개념 암송이 아니라 "펜스-할당자-배리어-힙을 직접 배선했고, 같은 RHI 계약 아래에서 DX11과 병행 구동된다"로 답할 수 있다.

### 명시적 수명 시대의 안전망 — 제너레이셔널 핸들

`Engine/Public/RHI/RHIHandles.h:6~47`의 `RHIHandle<TTag>`는 u64 하나에 인덱스(하위 32비트)+세대(상위 32비트)를 패킹한다(16~24행). 드라이버 참조 카운트 마법(마법 2)을 쓰지 않는 계층에서 리소스를 정수 핸들로 다루면 해제 후 사용(use-after-free)이 위험해지는데, 슬롯 재사용 시 세대를 올리면 낡은 핸들의 `Lookup`이 세대 불일치로 실패해 **댕글링 접근이 조용한 오염 대신 탐지 가능한 실패**가 된다. 명시적 API 세대의 수명 관리 문제의식을 핸들 설계로 받은 것.

### 디바이스 초기화의 현실 — 이론이 안 가르쳐 주는 부분

`Engine/Private/RHI/DX11/CDX11Device.cpp:785~865`의 `CreateDeviceAndSwapChain`은 API 초기화가 실전에서 왜 폴백 체인이 되는지 보여준다: 기능 수준(feature level) 11.1→11.0 배열 폴백(809~812행), 디버그 레이어 미설치 시 플래그를 빼고 재시도(841~849행), 고성능 어댑터(`EnumAdapterByGpuPreference`로 선택, 50~86행) 실패 시 OS 기본 어댑터로 재시도(851~857행). 스왑 이펙트는 FLIP_DISCARD를 목표로 뒀다가 백버퍼 순환/재바인딩 규율 문제로 DISCARD로 의도적 후퇴한 주석이 788~801행에 남아 있다(상세 결정 배경은 03 챕터 ③절 결정 4). "다양한 PC에서 초기화가 실패하는 경우들"은 면접에서 실무 경험을 증명하기 좋은 소재다.

### 만약 더 나아간다면 — 정직한 결핍 목록

면접에서 "부족한 점"을 물으면 이렇게 답할 수 있다. 현재 DX11 경로는 immediate context 단일 스레드 제출이고(deferred context 미사용 — 이유는 본문 ③절의 한계 그대로), 바인드리스·GPU 주도 렌더링은 없다. 다음 단계로 가치 있는 것은 (1) PSO 디스크 캐시와 사전 수집(현재는 생성 시점 관리만), (2) DX12 경로의 배리어 자동 검증 헬퍼(상태 추적 디버그 레이어), (3) 프레임 링 할당자 기반의 동적 버퍼 경로 통일이다. 결핍을 아는 것 자체가 API 모델을 이해한다는 증거가 된다.
