# 글쓰기 사고방식 — 초안 작성 전 로드하는 문서

> 목적: 자기소개서·포트폴리오 초안을 쓰기 전에 읽는 사고 프레임 (2026-07-12 세션 박제).
> 연결: `포트폴리오_MASTER_설계.md`(5칸 구조·3단 소비)와 `승자해부/` 3종의 원칙 위에서 동작한다.

## 0. 글의 전체 그래프

글은 문장의 나열이 아니라 트리다. 루트에 결론 한 문장, 아래로 주장 → 증거 → 경계.

```text
                    [결론 한 문장]                ← 30초 스캔
          "이 사람은 뽑아도 실패하지 않는다"
           /              |              \
  [깊이 — Winters] [협업 — MC던전스] [완성 — Liberation]   ← 주장 3개
   서버 권위를        PR 135건 ·        Steam 상용 출시
   바닥부터           커밋 64%
           \              |              /
  ┌──────────────────────────────────────────────┐
  │ 증거층 — 모든 주장은 여기로 역추적된다              │  ← 딥다이브
  │  수치 17.8ms→9ms · 코드/해시 300틱 무회귀         │
  │  git 실측 커밋 220/342 · 영상 클라 2분할+서버      │
  └──────────────────────────────────────────────┘
                          |
        [한계 · 역질문 — 모르는 것의 경계를 스스로 긋는다]   ← 면접 대비
```

트리의 성질 두 개가 구성의 전부:

- **역피라미드**: 독자가 어느 깊이에서 멈춰도 완결된 이해를 갖고 떠난다. 30초 = 루트, 5분 = 주장+증거, 면접 = 경계선. (= 면접관 3단 소비 구조)
- **두괄 사슬(topic sentence chain)**: 각 문단의 첫 문장만 이어 읽어도 글 전체 논리가 성립해야 한다. 검사법: 문단 첫 문장만 복사해 이어 붙였을 때 글이 되는가.

## 1. 글쓰기의 본질 — 상태 동기화

**글쓰기는 내 생각의 직렬화가 아니라, 독자 머릿속의 상태 동기화다.** 스냅샷 복제와 같은 문제: 서버(나)의 전체 상태를 그대로 쏘면 대역폭이 터지고 클라(독자)는 해석하지 못한다. 잘 쓴 글 = 수신자가 장면을 재구성하는 데 필요한 **최소·충분 정보를, 수신자의 디코더(배경지식)가 처리할 수 있는 포맷으로** 보내는 것.

채용 문서가 일으켜야 할 상태 변화는 단 하나: **"이 사람은 뽑아도 실패하지 않겠다"는 확신.** 모든 문장은 이 상태 변화에 기여하거나, 삭제되거나 둘 중 하나다.

## 2. 왜 글을 못 쓰는가 — 진단 5가지

"못 쓴다"는 느낌은 문장력 문제가 아니라 아래 다섯 개의 신호다.

1. **지식의 저주** (가장 큰 원인): 깊이 판 사람일수록 "모르는 상태"를 시뮬레이션하지 못한다. 나에게 "300틱 결정성 해시 무회귀"는 자명하지만 독자에겐 암호 — "리팩터링 전후로 시뮬레이션 결과가 1비트도 달라지지 않았음을 증명했다"라는 다리를 먼저 놓아야 한다. 못 쓰는 게 아니라 **독자 모델이 로드되어 있지 않은 것**.
2. **글쓰기 = 받아적기라는 오해**: 생각이 완성된 뒤 쓰는 게 아니라 **쓰면서 생각이 완성된다**. 문장이 안 나오는 지점 = 그 지점의 생각이 아직 흐릿하다는 계측 신호 (프로파일러가 병목을 표시하듯). 초안이 고통스러운 건 정상.
3. **개발자 순서(시간순)로 씀**: 겪은 순서(시도→실패→성공)로 쓰지, 독자가 필요한 순서(결론→근거)로 쓰지 않는다. 단 5칸 구조의 "문제 발견→해석→해결"은 의도적으로 선택한 시간순 — 차이는 **선택한 시간순인가, 관성의 시간순인가**.
4. **퇴고 부재**: 초안은 나를 위한 것, 퇴고가 독자를 위한 것. 초안을 완성본으로 착각하면 독자를 위한 작업이 통째로 빠진다. 글은 쓰는 게 아니라 고치는 것.
5. **평가 함수 부재**: 코드에는 "계측 없으면 주장 없음"이 있는데 글에는 판정 기준이 없어 개선 루프가 안 돈다. → §3의 루프가 평가 함수.

## 3. 쓰는 동안 계속 돌리는 루프 3개

- **문장 루프 — "그래서?(So what?)"**: 이 문장이 독자의 확신을 1이라도 움직이는가. 안 움직이면 삭제. "꽤/상당히/열심히" 같은 주관 부사는 전부 여기서 죽는다.
- **문단 루프 — 삭제 테스트**: 문단을 통째로 지웠을 때 글이 무너지는가. 안 무너지면 지운다. (108파일 리팩터링을 해시로 검증하듯, 문단 삭제를 "논리 해시"로 검증)
- **전체 루프 — 30초 테스트**: 30초만 읽은 사람 머리에 남는 한 문장이 내가 의도한 루트 문장과 일치하는가. 불일치 = 배치가 틀린 것.

배경 의식: **면접관은 수백 장을 읽는다.** 모든 문장은 자기를 읽는 데 든 비용을 즉시 상환해야 한다.

## 4. 자기소개서 vs 포트폴리오 — 두 사고방식

- **포트폴리오 = 검사(檢事)의 사고 — 증거의 체계**: 기소장처럼 주장 → 증거 → 수치. 형용사가 아니라 명사와 숫자로. "최적화를 잘했다" ❌ → "17.8ms→9ms" ⭕. 5칸 구조([의도]→[문제 발견]→[해석]→[해결]→[결과])가 이 사고를 강제하므로 포폴은 구조에 부어 넣기만 하면 된다.
- **자기소개서 = 인과(因果)의 사고 — 변화의 기록**: 상태 서술("나는 성실한 사람") ❌ → 인과 서술("이 사건이 나를 이렇게 바꿨다") ⭕. Before → 사건 → After → 그래서 지금의 선택. 보유한 사슬: **문과 → Liberation 1인 출시(성공 믿음 → 시장의 실패 피드백) → "같이 만들어야 한다" 각성 → 아카데미 → 팀 협업(git 세팅부터 갈등 조율) → 자체 엔진.** 검사법: 문장 사이가 "그래서/왜냐하면"으로 연결되면 서사, "그리고"로만 연결되면 나열.
- **두 문서의 관계 = 인덱스와 데이터**: 자소서의 모든 주장은 포폴의 증거로 하이퍼링크된다고 생각하고 쓴다. (김태호 포인트 3 "자소서의 객관적 근거가 되도록 구성" = FAULT 없는 제출봇의 "모든 주장은 weapons/ 카드로 역추적 가능")

## 5. 초안 작성 파이프라인 (이 순서대로만)

1. **루트 문장 확정** — 쓰기 전에 "독자가 기억할 단 한 문장"부터. 이게 없으면 아래 전부가 표류한다.
2. **뼈대** — 각 문단의 첫 문장만 먼저 쓴다 → 첫 문장만 이어 읽고 논리 성립 검증.
3. **초안** — 검열 없이 빠르게. 초안 품질에 자책 금지, 초안은 원재료.
4. **퇴고 3패스** — ① 구조(문단 순서·삭제 테스트) → ② 문장(한 문장 한 생각, 주어-서술어 붙이기) → ③ 단어(주관 형용사·부사 제거, 수치로 치환).
5. **검증** — 소리 내어 읽기(걸리는 곳 = 문장 병목) · 하루 묵히기(지식의 저주 완화) · 타인 30초 테스트.

---

**한 줄 요약**: 글쓰기는 재능이 아니라 스냅샷 동기화 문제다 — 독자의 디코더에 맞춰 최소·충분 정보를 결론부터 보낸다. "못 쓴다"는 느낌은 독자 모델 부재 + 퇴고 부재 + 평가 함수 부재의 신호이지 문장력의 문제가 아니다. 포폴은 검사처럼(증거), 자소서는 인과로(서사), 둘은 하이퍼링크로 묶인다.
