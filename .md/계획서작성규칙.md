# 계획서 작성 규칙 v2

계획서는 두 가지를 동시에 한다.

1. 구현자가 바로 붙잡고 적용하는 **코드 변경 지시서** (v1의 역할, 유지)
2. 그 변경의 **결정 기록** — 문제·제약 → 순진한 해법의 실패 → 메커니즘 → 대조 → 대가 (v2 신설)

v1은 1번만 남기고 "왜"를 금지해서, 결정이 증발하고 나중에 좌표(면접 방어) 작성 때 재발굴해야 했다. v2는 결정을 5단계로 원천 기록하되, **분량 상한**으로 v1의 반(反)장황 원칙을 유지한다. 계획서의 ①~⑤는 좌표 답안집의 초고가 된다.

사용자가 "계획서 작성", "계획서 보여줘", "plan"을 요청하면 이 문서를 먼저 읽고 그대로 적용한다. 출력 형식은 이 문서가 CLAUDE.md·일반 관례보다 우선한다. 기존 코드 인용·기존 API 호출은 원문 이름 그대로, 새로 제안하는 심볼은 PascalCase.

## 필수 시작 — 머리 3줄

```text
Session - [현재 달성 목표 한 줄]
좌표: [연결 좌표 id (W12, S07 ...) | 신규 좌표 후보 | 없음] · 축: [C1~C8 중 1~2개]
관련: [선행 PLAN/RESULT 문서명, 없으면 이 줄 생략]
```

축 참조 (⑤ 대가의 형태를 축이 결정한다):

```text
C1 기준계        어떤 연산 ↔ 다른 연산
C2 이동>계산     지역성 ↔ 유연성
C3 공유는 비싸다  동기화 ↔ 중복·불균형
C4 수명은 선언된다 예측가능성 ↔ 편의
C5 이산화와 오차  정확도 ↔ 예산
C6 가지치기      안전 ↔ 속도
C7 권위와 정합성  정합성 ↔ 지연·확장
C8 검증이 병목    변경 속도 ↔ 회귀 안전
```

## 1. 결정 기록 — 항목당 1~3줄, 전체 15줄 이내

```text
① 문제·제약: [제약·숫자] 안에서 [목표]. 순진하게 하면 [측정값/증상].
② 순진한 해법의 실패: 가장 단순한 대안과 그것이 안 되는 이유 (기록·실측 근거).
③ 메커니즘: 선택한 방법 한 줄. 상세는 §2 코드가 말한다 — 산문으로 코드를 반복하지 않는다.
④ 대조: 검토한 대안 / UE·업계·원본은 어떻게 푸는가, 왜 여기선 다른가.
⑤ 대가: 이 선택으로 잃는 것 + 언제 이 선택이 틀리게 되는가.
```

규칙:

- 숫자가 없으면 문제가 아니라 취향이다. ①에는 숫자(상수·측정값·예산)를 쓴다. 측정이 없으면 `(실측 없음)`으로 정직하게 표기한다.
- ⑤가 안 나오면 ⑤를 짜내지 말고 ①로 돌아간다 — 예산 초과로 정의된 문제만 대가를 낳는다.
- 스케일 규칙: ①⑤는 항상 쓴다. ②③④는 설계 선택이 실제로 있었을 때만 쓴다. 오타 수정·단순 배선은 ① 한 줄 + ⑤ "없음"으로 충분하다.
- §1이 §2보다 길면 실패다. 배경 서술·진행 요약·이미 반영된 작업 언급은 여전히 금지 — 필요한 배경은 ①~⑤ 안에만 존재한다.

## 2. 반영해야 하는 코드

반드시 파일별로 쓴다. 파일 섹션 제목:

```text
2-1. C:/Users/user/Desktop/Winters/Client/Private/Scene/InGameCombatInputBridge.cpp
```

파일 섹션 안에서는 아래 세 형식만 사용한다. 모든 지시는 적용 위치를 스스로 특정해야 한다 — 별도 anchor 섹션을 만들지 않고, 애매하면 함수명/클래스명/public·private 영역을 함께 적는다. line 번호는 꼭 필요할 때만 괄호 참고로.

### 아래에 추가

기준이 되는 `기존 코드`를 먼저, 그 바로 아래에 신규 코드를 붙인다.

```text
기존 코드:

cpp

void Load_CurrentStage();

아래에 추가:

cpp

void SaveCurrentNavGrid();
```

### 아래로 교체

`기존 코드` 블록 전체를 인용하고 교체본을 쓴다. **"X를 X로 교체"(no-op 앵커) 금지** — 변화가 없으면 그 파일 섹션 자체를 쓰지 않는다.

### 삭제

```text
삭제할 코드:

cpp

#include "AssetFormat/Mesh/WMeshLoader.h"
```

긴 블록은 시작/끝을 자연어로: `...줄부터 ...바로 위까지 삭제. ...줄은 남긴다.`

### 새 파일

새 파일은 **전체 내용**을 쓴다. 요약·플레이스홀더 금지 — 전문을 쓸 수 없으면 `확인 필요`로 표기하고 그 이유를 남긴다.

### 프로젝트/산출물 파일

- `.vcxproj`/`.filters` XML 지시는 기본적으로 쓰지 않는다. 새 파일 등록이 필요하면 §3에 `확인 필요: 새 파일의 빌드 프로젝트 포함 확인`으로만 적는다. (사용자가 명시 요구 시에만 예외)
- `EngineSDK/inc`는 UpdateLib.bat 산출물 — 반영 코드에 넣지 않는다. Engine public header 변경 시 §3에 `후속 동기화: UpdateLib.bat 실행 필요`.
- 이미 코드에 반영된 내용은 쓰지 않는다. 검증 대상일 때만 §3에 한 줄.

## 3. 검증 — 예측을 먼저 쓴다

검증은 "무엇을 돌린다"가 아니라 "**돌리면 무엇이 관측될 것이다**"부터 쓴다. 예측이 틀리는 지점이 가장 값진 데이터다.

```text
예측:
- [적용 후 관측: SimLab 골든 해시 불변 | 의도 변동, 기대 로그 문자열, 시각/동작 결과]
- [이 변경이 깨뜨릴 수 있는 것과 그것을 잡을 게이트. 잡을 게이트가 없으면 "게이트 없음"이라고 쓴다 — 그게 발견이다]

검증 명령:
- [빌드 / SimLab / 스모크 — 복사해서 실행 가능한 구체 명령]

미검증:
- [계획 시점에 검증하지 못한 것]

확인 필요:
- [불확실 지점 — 추측으로 단정하지 않는다]
```

서버 권위·GameSim·Bot AI 접촉 계획은 예측 항목에 다음 경계를 포함한다: **Bot AI는 GameCommand 생산자이며 게임플레이 진실을 직접 변경하지 않는다.**

## RESULT 문서 — 적용 후 짝 문서

계획서가 적용되면 같은 이름의 `_RESULT` 문서에 세 가지만 쓴다.

```text
1. 예측 vs 실측: 적중/빗나감과 이유 (빗나간 예측이 최우선 기록)
2. 판결: 계획 유지 | 수정 반영 | 롤백 — 근거 한 줄
3. ⑤ 갱신: 실측 후 달라진 대가·"언제 틀리나"
```

좌표 id가 붙은 계획서의 ①~⑤와 RESULT의 갱신분은 좌표 답안집(05) 갱신 원료다 — 면접 방어는 여기서 자동으로 자란다.
