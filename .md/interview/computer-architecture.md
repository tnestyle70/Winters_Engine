# 컴퓨터 구조 — 기술면접 대비

> 대상: 자체 DX11 엔진(Winters) 제작 경험이 있는 게임 클라이언트/서버 프로그래머 지원자.
> 이 문서의 모든 프로젝트 인용은 실제 레포 경로 기준이다. 면접에서 "직접 보여줄 수 있는" 근거만 인용했다.

## 출제 경향 개요

게임회사 기술면접에서 컴퓨터 구조는 **단독 과목으로 나오기보다 C++/최적화/멀티스레딩 질문의 바닥층**으로 출제된다. 전형적인 흐름:

- "왜 ECS가 빠른가요?" → 캐시라인 → 지역성 → 메모리 계층 → 레이턴시 수치까지 내려간다.
- "멀티스레딩 해보셨어요?" → atomic → CPU에서 atomic이 어떻게 구현되는가 → MESI → false sharing.
- "서버에서 float 써도 되나요?" → IEEE 754 → 결정론 → 컴파일러 부동소수점 모델.
- "SIMD 써보셨어요?" → DirectXMath → 정렬 → 레지스터와 메모리 타입의 분리.

즉 **정의 암기가 아니라 "내 코드의 어떤 결정이 하드웨어의 어떤 성질 때문인가"를 설명하는 능력**을 본다. 자체 엔진 제작자는 이 연결고리를 실코드로 보여줄 수 있다는 것이 최대 무기다. 아래 정리는 전부 그 연결을 전제로 쓴다.

## 핵심 개념 정리

### 1. CPU 파이프라인

**정의.** 명령어 실행을 여러 단계로 쪼개 각 단계를 서로 다른 명령어가 동시에 점유하게 하는 기법. 교과서 기준 5단계(고전 RISC/MIPS):

1. **IF (Instruction Fetch)** — PC가 가리키는 명령어를 I-캐시에서 읽는다.
2. **ID (Instruction Decode)** — 명령어 해석 + 레지스터 파일 읽기.
3. **EX (Execute)** — ALU 연산, 주소 계산, 분기 조건 판정.
4. **MEM (Memory Access)** — 로드/스토어가 D-캐시에 접근.
5. **WB (Write Back)** — 결과를 레지스터에 기록.

**동작 원리.** 파이프라인이 없으면 명령어 하나가 5사이클을 통째로 쓴다. 파이프라인이 차면 이상적으로는 **매 사이클 1개 명령어가 완료**된다(레이턴시는 그대로, 처리량이 5배). 실제 x86 코어는 5단계가 아니라 15~20단계 내외의 깊은 파이프라인이고, 디코드 후 μop으로 쪼개 실행한다.

**해저드(파이프라인이 멈추는 3가지 이유).**

- **구조적 해저드**: 두 명령어가 같은 하드웨어 자원을 동시에 요구. 해결 — 자원 복제(하버드식 I/D 캐시 분리, 다중 포트 레지스터 파일).
- **데이터 해저드**: 앞 명령어의 결과를 뒤 명령어가 아직 준비되기 전에 요구.
  - RAW(진짜 의존) — 포워딩/바이패스(EX 출력을 다음 EX 입력으로 직접 배선)로 대부분 해결, 로드-사용(load-use)만은 1사이클 스톨 불가피.
  - WAR/WAW(이름 의존) — **레지스터 리네이밍**으로 제거(물리 레지스터를 논리 레지스터보다 훨씬 많이 두고 매핑).
- **제어 해저드**: 분기 결과가 나올 때까지 다음 fetch 주소를 모른다. 해결 — 분기 예측 + 투기 실행.

**분기 예측.** 파이프라인이 깊고 폭이 넓을수록(슈퍼스칼라) 분기 하나를 기다리는 비용이 커지므로, CPU는 분기 방향과 목적지를 **예측하고 그 경로를 미리 실행**한다.

- 구현: 2비트 포화 카운터 → 지역/전역 히스토리(gshare) → 현대 CPU는 TAGE류 + 퍼셉트론 하이브리드. 목적지는 BTB, 함수 리턴은 RSB(Return Stack Buffer)가 담당.
- 예측 실패 비용: 파이프라인 플러시, 최신 x86에서 **약 15~20사이클**. 분당 수백만 번 실행되는 게임 루프 내부라면 무시 못 할 비용.
- 코드 함의: **예측 가능한 분기는 거의 공짜, 무작위 분기는 비싸다.** 데이터를 정렬하거나, 분기를 산술로 치환(cmov, 마스크 연산)하거나, 타입별로 배열을 분리해 분기 자체를 제거하는 이유.

**슈퍼스칼라 / 비순차 실행(OoO).**

- **슈퍼스칼라**: 한 사이클에 여러 명령어를 발행(issue). 현대 x86은 4~6-wide 디코드, 실행 포트 8~12개.
- **비순차 실행**: 프로그램 순서가 아니라 **피연산자가 준비된 순서**로 실행. 구성요소 — 레지스터 리네이밍, 예약 스테이션(스케줄러), ROB(재정렬 버퍼, 200~500+ 엔트리). 결과 커밋(retire)은 프로그램 순서대로 하여 예외/인터럽트의 정확성을 보존.
- 존재 이유: **메모리 레이턴시 은폐**. 캐시 미스로 300사이클을 기다리는 동안 의존성 없는 다른 명령어를 계속 실행한다. 뒤집어 말하면, **의존 체인이 길고 포인터 체이싱(다음 주소가 이전 로드 결과에 의존)인 코드는 OoO가 은폐해줄 수 없다** — 연결 리스트 순회가 배열 순회보다 근본적으로 느린 이유이자 데이터 지향 설계의 하드웨어적 근거.

**게임 개발 맥락.** 게임 루프의 핫패스(수천 엔티티 업데이트)는 "분기가 예측 가능한가, 의존 체인이 짧은가, 다음 메모리 주소를 하드웨어가 미리 알 수 있는가"로 성능이 갈린다. ECS의 타입별 시스템 분리는 분기 제거 + 프리페치 가능한 선형 접근을 동시에 얻는 구조다.

### 2. 캐시 계층

**정의.** CPU와 DRAM 사이의 SRAM 기반 완충 계층. 전형적 구성(코어당):

| 계층 | 크기 | 레이턴시 | 비고 |
|---|---|---|---|
| L1I / L1D | 각 32~48KB | 4~5 사이클 (~1ns) | 코어 전용, I/D 분리 |
| L2 | 256KB~1.25MB | 12~14 사이클 (~4ns) | 코어 전용 |
| L3 (LLC) | 코어당 2~3MB 공유 | 30~50 사이클 (~15ns) | 전 코어 공유 |
| DRAM | GB 단위 | 200~300 사이클 (60~100ns) | |

**캐시라인.** 캐시와 메모리 사이 전송 단위는 **64바이트**(x86 기준; Apple M 계열은 128B). 1바이트만 읽어도 64B가 통째로 올라온다. 함의 세 가지:

1. 인접 데이터는 공짜로 딸려 온다 → **공간적 지역성**을 활용하는 배치가 이득.
2. 자주 함께 쓰는 필드는 같은 라인에, 스레드별로 따로 쓰는 필드는 **다른 라인에**(false sharing 방지).
3. `float` 하나(4B) 때문에 64B를 낭비하면 실효 대역폭이 1/16 — 핫루프에서 안 쓰는 필드가 섞인 구조체(AoS)가 느린 산술적 이유.

**연관도(associativity).** 어떤 메모리 주소가 캐시의 어느 슬롯에 들어갈 수 있는가.

- **직접 매핑**: 주소당 슬롯 1개. 빠르고 싸지만 서로 다른 주소가 같은 슬롯을 두고 충돌(conflict miss).
- **완전 연관**: 아무 슬롯이나. 충돌 없음, 하드웨어 비쌈 (TLB 정도에만 사용).
- **집합 연관(N-way)**: 절충. L1은 보통 8-way, L3는 12~16-way. 주소의 인덱스 비트로 set을 고르고 set 안의 N개 way 중 하나에 배치.
- 함정: **2의 거듭제곱 스트라이드**(예: 4096바이트 간격)로 접근하면 모두 같은 set에 몰려 8-way라도 순식간에 쫓아낸다 — 텍스처/행렬 처리에서 pitch를 2^n에서 살짝 어긋나게 잡는 이유.

**미스의 3C 분류.** Compulsory(첫 접근) / Capacity(용량 초과) / Conflict(연관도 부족). 최적화 대상이 무엇인지 구분하는 어휘로 쓰인다.

**지역성.**

- **시간적 지역성**: 한 번 쓴 데이터는 곧 다시 쓴다 → 캐시에 남아 있는 동안 다시 쓰도록 작업을 묶어라(loop fusion, 프레임 내 같은 데이터에 대한 패스 통합).
- **공간적 지역성**: 쓴 데이터의 이웃도 곧 쓴다 → 선형 배열 + 순차 순회. 하드웨어 프리페처는 **일정한 스트라이드의 순차 접근**을 감지해 다음 라인을 미리 당겨온다. 포인터 체이싱은 프리페처가 무력화된다.

**false sharing.** 서로 다른 코어가 **논리적으로 무관한 변수**를 쓰는데 그 변수들이 **같은 캐시라인**에 있으면, 라인 소유권이 코어 간을 핑퐁하며 진짜 공유만큼 느려지는 현상. 증상은 "락도 없고 경합도 없는데 스레드를 늘리면 느려진다".

Winters 실증 — Chase-Lev 워크스틸링 데크에서 owner가 만지는 `bottom`과 thief가 만지는 `top`을 서로 다른 라인에 강제 분리했다 (`Engine/Public/Core/JobSystem/WorkStealingDeque.h`):

```cpp
// - alignas(64) 로 top/bottom false-sharing 방지.
alignas(64) std::atomic<std::int64_t> m_iBottom{ 0 };
alignas(64) std::atomic<std::int64_t> m_iTop{ 0 };
```

**MESI 일관성 프로토콜.** 코어마다 사본이 있는데 어떻게 하나의 메모리처럼 보이는가. 각 캐시라인은 코어별로 4상태 중 하나:

- **M**odified — 나만 갖고 있고 메모리보다 최신(dirty).
- **E**xclusive — 나만 갖고 있고 메모리와 동일(clean).
- **S**hared — 여러 코어가 clean 사본 보유. 읽기만 가능.
- **I**nvalid — 무효.

동작: 쓰기를 하려면 라인을 M/E로 가져야 한다. S 상태에서 쓰려면 **RFO(Read For Ownership)**를 브로드캐스트해 다른 코어의 사본을 전부 I로 만든 뒤 쓴다. 다른 코어가 M 상태 라인을 읽으려 하면 캐시-투-캐시 전송이 일어난다. **false sharing의 정체가 바로 이 RFO/무효화의 핑퐁**이고, atomic 연산 경합 비용의 본질도 "라인 소유권 이동"이다. (실제 인텔은 MESIF, AMD는 MOESI 변형.)

### 3. 메모리 계층과 레이턴시 수치 감각

면접에서 "대략 몇 ns인지 감 있으세요?"는 실무 감각 검증 단골. 오더만 정확하면 된다:

| 연산 | 시간 | 사이클 (3GHz 기준) |
|---|---|---|
| 레지스터 | ~0.3ns | 1 |
| L1 히트 | ~1ns | 4~5 |
| L2 히트 | ~4ns | 12~14 |
| L3 히트 | ~15ns | 30~50 |
| DRAM | 60~100ns | 200~300 |
| 경합 없는 atomic RMW | ~7~15ns | 20~40 |
| 분기 예측 실패 | ~5ns | 15~20 |
| 뮤텍스 lock/unlock (비경합) | ~25ns | |
| 스레드 컨텍스트 스위치 | 1~10µs | |
| NVMe SSD 랜덤 읽기 | 20~100µs | |
| HDD 시크 | 5~10ms | |
| 동일 IDC 내 RTT | ~0.5ms | |
| 한국–미서부 RTT | 100~150ms | |

**해석 훈련(면접에서 그대로 쓸 수 있는 문장들).**

- "L1과 DRAM은 약 **100배** 차이. 캐시 미스 한 번 = 산술 명령 수백 개 손실."
- "60fps 프레임 예산 16.6ms = 3GHz에서 약 5천만 사이클. DRAM 미스(100ns) 기준 프레임당 16만 번의 미스면 예산의 10%가 순수 대기."
- "디스크와 메모리는 1,000배, 네트워크 RTT와 메모리는 100만 배 — 그래서 게임 서버 틱 루프 안에서 동기 I/O는 곧 서버 사망이고, 로딩은 스트리밍/비동기로 뺀다."

**NUMA(참고).** 멀티소켓 서버에서 다른 소켓의 메모리는 로컬보다 1.5~2배 느리다. 게임 서버를 소켓 단위로 파티셔닝하는 이유.

### 4. 가상 주소와 TLB

**가상 메모리의 목적.** (1) 프로세스별 독립 주소 공간(보호·격리), (2) 물리 메모리 단편화 은폐, (3) 지연 할당·페이지 파일·메모리 맵 파일 같은 트릭의 기반.

**주소 변환.** x64는 48비트(확장 57비트) 가상 주소를 4단계(5단계) 페이지 테이블로 변환. 페이지 크기는 기본 4KB, 대형 페이지 2MB/1GB. 변환을 매번 테이블 워크(메모리 접근 4회)로 하면 재앙이므로 **TLB(Translation Lookaside Buffer)** — 변환 결과 캐시 — 를 둔다.

- L1 dTLB ~64엔트리, L2 TLB ~1500~2000엔트리. 4KB 페이지 기준 L1 dTLB가 커버하는 범위는 **고작 256KB**.
- **TLB 미스** → 하드웨어 페이지 워커가 페이지 테이블을 순회(수십~100+ 사이클, 테이블 자체도 캐시 미스 가능).
- **페이지 폴트** → 매핑 자체가 없으면 OS 트랩. minor fault(물리 페이지만 연결)도 µs 단위, major fault(디스크에서 로드)는 ms 단위.

**게임 개발 맥락.**

- 거대한 자산을 넓게 흩어 접근하면 캐시 미스 이전에 TLB 미스부터 맞는다. 데이터를 페이지 단위로도 뭉쳐두는 것(풀 할당자, 아레나)이 TLB에도 이득.
- 대형 페이지(2MB)는 TLB 커버리지를 512배 늘린다 — 대규모 서버 프로세스에서 고려 대상.
- "release 빌드인데 첫 프레임만 느려요" 종류의 증상 중 하나가 첫 접근 시 페이지 폴트(지연 커밋) — 로딩 단계에서 미리 touch하는 워밍업이 존재하는 이유.

### 5. SIMD — SSE/AVX와 DirectXMath

**정의.** Single Instruction Multiple Data. 한 명령어로 벡터 레지스터 안의 여러 데이터에 같은 연산을 적용.

- **SSE/SSE2**: 128비트 XMM 레지스터. float×4 또는 double×2. x64에서는 SSE2가 ABI 기본(스칼라 float 연산조차 x87이 아닌 SSE로 컴파일된다).
- **AVX/AVX2**: 256비트 YMM. float×8. AVX2가 정수 256비트와 FMA 동반.
- **AVX-512**: 512비트 ZMM + 마스크 레지스터. 소비자용 CPU에선 제한적.
- 정렬 요구: 전통적으로 `movaps`(aligned load)는 16B 정렬 필수. 비정렬 로드(`movups`)는 현대 CPU에서 패널티가 거의 없지만 **캐시라인 경계를 걸치면 여전히 느리고**, 정렬 명령에 비정렬 주소를 주면 크래시(#GP)한다.

**DirectXMath가 SIMD를 쓰는 방식.** 핵심 설계는 **"연산 타입과 저장 타입의 분리"**:

- `XMVECTOR` = 사실상 `__m128`의 typedef — **XMM 레지스터에 매핑되는 연산용 타입**. 16B 정렬 요구. 함수 인자로 넘길 때 레지스터 전달을 위해 `FXMVECTOR/GXMVECTOR` + `XM_CALLCONV`(vectorcall) 규약 사용.
- `XMFLOAT3/XMFLOAT4X4` = 정렬 요구 없는 **저장용 평범한 구조체**.
- 경계에서 `XMLoadFloat3` → XM 연산들 → `XMStoreFloat4x4`로 오간다. 로드/스토어는 명시적 비용이므로 **한 번 로드해서 연산을 몰아 하고 한 번 스토어**하는 것이 관용구.

Winters는 이 규칙을 수학 헤더 차원에서 강제한다 (`Engine/Include/WintersMath.h` 주석 원문):

```cpp
//  Vec3/Vec4/Mat4(XMFLOAT*)는 저장용 타입.
//  XMVECTOR/XMMATRIX는 함수 내부 연산용 임시 타입.
//  XMVECTOR/XMMATRIX를 멤버/배열/패킷에 저장하지 말고
//  XMLoad* -> XM* 연산 -> XMStore* 경계에서만 사용한다.
```

실사용 예 — 트랜스폼 시스템의 로컬/월드 행렬 재계산 (`Engine/Private/ECS/Systems/TransformSystem.cpp`):

```cpp
XMVECTOR scale = XMLoadFloat3(reinterpret_cast<const XMFLOAT3*>(&t.m_LocalScale));
XMVECTOR rot   = XMQuaternionRotationRollPitchYaw(...);
XMVECTOR pos   = XMLoadFloat3(reinterpret_cast<const XMFLOAT3*>(&t.m_LocalPosition));
XMMATRIX srt   = XMMatrixAffineTransformation(scale, XMVectorZero(), rot, pos);
XMStoreFloat4x4(reinterpret_cast<XMFLOAT4X4*>(&t.m_LocalMatrix), srt);
```

**왜 XMVECTOR를 멤버로 저장하면 안 되는가.** (1) 16B 정렬을 컨테이너/힙/패킹된 구조체에서 보장하기 어렵다(특히 x86 32비트, pre-C++17 `new`, `#pragma pack` 환경에서 크래시). (2) 저장하는 순간 레지스터 상주라는 이점이 사라지고 레이아웃만 경직된다. (3) `memcpy`·직렬화·네트워크 전송에 대한 안전성이 깨진다. Winters `Vec3`가 평범한 `float x,y,z` 구조체인 이유가 이것이며, `TransformComponent`(`Engine/Public/ECS/Components/TransformComponent.h`)도 저장 타입만 멤버로 갖는다.

**자동 벡터화 vs 명시적 SIMD.** 컴파일러 자동 벡터화는 앨리어싱/의존성 증명이 안 되면 쉽게 포기한다. 게임 수학처럼 4-wide가 자연스러운 도메인은 DirectXMath 같은 intrinsic 래퍼가 예측 가능하다. 더 나가면 SoA로 재배치해 8/16-wide AVX를 먹이는 방식(파티클, 스키닝)이 있다.

### 6. IEEE 754 부동소수점

**비트 구조.**

- float(binary32): 부호 1 + 지수 8(bias 127) + 가수 23(암묵적 선행 1 포함 유효 24비트) → 십진 유효자리 약 7.2자리.
- double(binary64): 1 + 11(bias 1023) + 52 → 약 15.9자리.
- 값 = (-1)^s × 1.M × 2^(E-bias). 특수값: E=0 → 비정규수(denormal, 연산이 매우 느릴 수 있음), E=전부1 → ±Inf / NaN. **NaN은 자기 자신과도 같지 않다** (`x != x`가 NaN 판별 관용구).

**핵심 성질과 함정.**

- 0.1, 0.2는 2진 유한소수가 아니다 → `0.1 + 0.2 == 0.3`은 false (double에서 0.30000000000000004).
- 머신 엡실론: float 2⁻²³ ≈ 1.19e-7. **"1.0 근처에서의" 상대 간격**이지 절대 오차 한계가 아니다 — 값이 커질수록 표현 간격이 커진다. float은 2²⁴=16,777,216까지의 정수만 연속으로 정확 표현: 그 너머에선 `16777216.0f + 1.0f == 16777216.0f`.
- 게임 실전 사례: 월드 좌표가 원점에서 수만 유닛 떨어지면 float 간격이 밀리미터를 넘어 애니메이션/물리가 덜덜 떨린다(대형 월드의 origin shifting이 존재하는 이유). 누적 타이머를 float 초로 오래 돌리면 dt가 간격보다 작아져 시간이 멈춘다 — **누적은 double이나 정수 틱으로**.
- 덧셈은 **결합법칙이 성립하지 않는다**: `(a+b)+c != a+(b+c)`. 이것이 컴파일러 최적화·멀티스레드 합산 순서가 결과를 바꾸는 근본 원인.

**비교 방법.**

```cpp
// 절대 epsilon: 큰 값에서 무의미 (1e6 근처의 float 간격은 0.0625)
fabs(a-b) < 1e-6f
// 상대 epsilon: 0 근처에서 무의미
fabs(a-b) <= eps * max(fabs(a), fabs(b))
// 실무: 절대+상대 결합, 또는 도메인 지식으로 튜닝된 임계값
```

Winters도 정규화 실패 판정을 도메인 임계값으로 처리한다 — `WintersMath::kEpsilon`(1e-6)과 길이제곱 비교, 실패 시 fallback 방향 반환 (`Engine/Include/WintersMath.h`의 `Normalize3D/NormalizeXZ`). "epsilon 하나로 만능 비교"가 아니라 **호출부가 실패 시 의미(fallback)를 정의**하게 만든 것이 포인트.

**게임 서버 결정론(determinism)과 /fp:precise.**

같은 소스라도 float 결과가 달라질 수 있는 원인: ① 컴파일러의 재결합(reassociation)·FMA 축약·역수 근사 등 최적화, ② x87(80비트 중간 정밀도) vs SSE 경로, ③ `sin/cos` 같은 수학 라이브러리 구현 차이, ④ 스레드/합산 순서, ⑤ CPU별 근사 명령(rsqrtss) 결과 차이.

Winters는 MSVC `/fp:precise`를 **Server만이 아니라 모든 프로젝트에 명시적으로 고정**했다:

- `Server/Include/Server.vcxproj` — `<FloatingPointModel>Precise</FloatingPointModel>` (Debug/Release 양쪽)
- 동일 설정이 `Client/Include/Client.vcxproj`, `Engine/Include/Engine.vcxproj`, `Shared/GameSim/Include/GameSim.vcxproj`, `Tools/SimLab/SimLab.vcxproj`, `EldenRingClient/Include/EldenRingClient.vcxproj`에 걸려 있다.

이유: Winters는 챔피언 시뮬레이션(`Shared/GameSim`)을 **클라이언트와 서버가 같은 코드로 컴파일해 공유**한다(서버 권위 + 클라 예측/검증 구조). `/fp:precise`는 소스에 쓴 연산 순서·반올림을 컴파일러가 재배치하지 못하게 하므로, **같은 툴체인·같은 x64/SSE2 타깃에서 클라와 서버의 시뮬 산출이 비트 단위로 일치**하도록 하는 최소 조건이 된다. 어느 한 프로젝트만 `/fp:fast`로 바뀌면 공유 심의 결과가 갈라져 예측 미스/스냅샷 불일치가 "간헐적 밀림"으로 나타나기 때문에, 프로젝트 전체에 일괄 고정해 드리프트 자체를 막았다.

정직한 한계(꼬리질문 대비): `/fp:precise`는 **동일 플랫폼·동일 툴체인 내 재현성**을 주는 것이지 크로스 플랫폼 결정론을 보장하지 않는다. 이기종(콘솔/모바일/타 컴파일러) 록스텝이 필요하면 고정소수점, 소프트웨어 float, 결정론 수학 라이브러리, 트랜센덴털 함수 자체 구현까지 가야 한다.

**/fp:fast vs /fp:precise 트레이드오프.** fast는 재결합·FMA·근사를 허용해 벡터화가 잘 걸리고 수 % ~ 수십 % 빨라질 수 있으나, 결과가 최적화 수준·컴파일러 버전에 따라 흔들리고 NaN/Inf 처리 가정도 약해진다. 렌더링 전용 코드라면 fast도 합리적이지만, **게임플레이 판정·네트워크 검증에 쓰이는 수치는 precise로 못 박는 것**이 서버 권위 구조의 정석.

### 7. 메모리 정렬과 패딩

**규칙.** 각 타입은 자연 정렬(대개 자기 크기)을 요구하고, 구조체 정렬은 멤버 중 최대 정렬, 구조체 크기는 그 정렬의 배수가 되도록 꼬리 패딩이 붙는다.

```cpp
struct A { char c; int i; char c2; };  // 1 +3pad +4 +1 +3pad = 12B
struct B { int i; char c; char c2; };  // 4 +1 +1 +2pad      = 8B
```

멤버를 **정렬 요구 큰 순서로** 배치하면 패딩이 최소화된다. 대량 배열(컴포넌트, 파티클, 네트워크 스냅샷)에서 패딩 33%는 곧 캐시 대역폭 33% 손실이다.

**왜 정렬이 필요한가.** 비정렬 접근은 x86에선 대개 동작하지만 캐시라인/페이지 경계를 걸치면 2회 접근으로 쪼개져 느려지고, SIMD 정렬 로드·일부 아키텍처·**atomic은 비정렬이면 원자성 자체가 깨질 수 있다**(라인 두 개에 걸친 lock 연산은 "split lock"으로 버스 전체를 세우는 대참사).

**실무 도구.** `alignas(N)`(false sharing 방지·SIMD 정렬), `static_assert(sizeof(T)==N)`(와이어 포맷 고정), `#pragma pack`(네트워크/파일 헤더 — 대신 비정렬 접근 비용과 UB 위험 감수), C++17 정렬 `new`. Winters의 `alignas(64)` 사용례는 §2 false sharing 참조.

**패킷/파일 포맷 맥락.** 구조체를 그대로 `memcpy`해 보내는 설계는 패딩·정렬·엔디안·컴파일러 차이에 전부 노출된다. Winters가 스냅샷/명령에 FlatBuffers를 쓰는 것(서버 `.vcxproj`의 `FlatBuffers/Inc` 포함 경로, `Server/Private/Game/SnapshotBuilder.cpp`)은 이 레이아웃 문제를 스키마 계층에서 격리하는 선택이다.

### 8. 엔디안

**정의.** 다바이트 값을 메모리에 놓는 바이트 순서. 리틀 엔디안 = 최하위 바이트가 낮은 주소(x86/x64, ARM 대부분의 OS 설정), 빅 엔디안 = 최상위 바이트부터(전통적 네트워크 바이트 오더).

```
0x12345678 (uint32_t)
LE: 78 56 34 12   |   BE: 12 34 56 78
```

**어디서 문제가 되나.** 같은 머신 안에서는 절대 안 보인다. **바이트열이 경계를 넘을 때** — 네트워크 전송, 파일 포맷, 타 플랫폼과의 데이터 교환 — 만 문제가 된다. BSD 소켓 API의 포트/주소는 빅 엔디안이라 `htons/ntohl` 계열을 쓴다(Winters 서버도 Winsock `ws2_32.lib` 기반이므로 이 경계는 존재).

**실무 감각.** 요즘 게임의 주요 타깃(x64, ARM64)이 전부 리틀 엔디안이라, FlatBuffers·Protobuf 같은 직렬화 계층은 **와이어 포맷을 리틀 엔디안으로 고정**해 x86에서 스왑 비용을 0으로 만든다. Winters의 FlatBuffers 채택은 엔디안·패딩·버전 드리프트를 한 계층에서 함께 봉인하는 결정. "엔디안 스왑 코드를 손으로 짠다"보다 "포맷을 고정하고 접근자를 생성한다"가 현대적 답이다.

### 9. 캐시 친화적 설계 — ECS / 데이터 지향 설계 (Winters 실증)

**OOP 게임 오브젝트의 하드웨어적 문제.** `GameObject*`의 배열을 돌며 `virtual Update()`를 호출하는 구조는 ① 객체가 힙에 흩어져 있어 매 객체가 캐시 미스, ② 가상 호출로 분기 예측·인라인 불가, ③ 객체 안에 지금 안 쓰는 필드가 섞여 캐시라인을 낭비, ④ 다음 주소를 예측할 수 없어 프리페처 무력화 — 4중고를 만든다.

**ECS/DOD의 답.** 같은 타입의 컴포넌트를 **연속 배열**에 모아두고, 시스템이 그 배열을 선형 순회한다. Winters의 컴포넌트 저장소는 sparse set 구조다 (`Engine/Public/ECS/ComponentStore.h`):

```cpp
std::vector<uint32_t> m_vecSparse;  // EntityID -> dense index
std::vector<EntityID> m_vecDense;   // dense index -> EntityID
std::vector<T>        m_vecData;    // 연속된 메모리 사용!  (원 주석)
```

- **조회** `Get(entity)` = sparse→dense 2단계 인덱싱, O(1).
- **순회** = `m_vecData`를 처음부터 끝까지 — 컴포넌트 T가 빈틈없이 연속이므로 캐시라인·프리페처·(가능하면) SIMD까지 전부 하드웨어 편에 선다.
- **삭제** `Remove()` = swap-and-pop: 마지막 원소를 빈자리로 옮겨 **밀도를 유지**한다. 구멍 난 배열(순회 중 분기 + 라인 낭비)을 원천 차단하는 결정이고, 그 대가로 dense 순서 안정성을 포기한다(트레이드오프를 물으면 이걸 말하면 된다).

컴포넌트 자체도 캐시를 의식한 평범한 데이터다 — `TransformComponent`는 저장 타입(Vec3/Mat4) + 더티 플래그만 가진 plain struct이고(§5 인용), 계산은 시스템(`TransformSystem.cpp`)이 SIMD 경계에서 수행한다. 데이터(컴포넌트)와 변환(시스템)의 분리가 곧 DOD.

**AoS vs SoA.**

- AoS(Array of Structures): `struct P{pos,vel,hp} p[N]` — 한 개체의 전 필드를 함께 쓰는 로직에 유리.
- SoA(Structure of Arrays): `pos[N], vel[N], hp[N]` — 한 필드만 대량 처리(SIMD 8/16-wide)에 유리.
- Winters는 **월드 레벨에서 SoA**(컴포넌트 타입별 배열 분리), **컴포넌트 내부는 AoS**(Transform의 pos/rot/scale은 항상 같이 쓰이므로) — 접근 패턴에 맞춘 절충이다.

정직한 한계도 말할 수 있어야 한다: `TransformComponent`는 두 개의 Mat4(각 64B)와 `std::vector<EntityID> m_vecChildren`을 품어 컴포넌트 하나가 캐시라인 여러 개에 걸치고, 계층 재귀는 포인터 체이싱이다. "핫패스가 어디냐에 따라 행렬을 별도 배열로 분리(hot/cold split)하는 개선 여지가 있다"는 자기 비판이 오히려 점수가 된다.

### 10. 원자적 연산 — CPU 관점의 구현과 비용

**정의.** 다른 코어가 중간 상태를 관측할 수 없는 읽기-수정-쓰기(RMW). C++ `std::atomic`의 x86 구현:

- 로드/스토어(정렬된 ≤8B)는 원래 원자적. `seq_cst` 스토어만 `xchg`/`mfence`가 붙는다.
- RMW(`fetch_add`, `exchange`, CAS)는 `lock` 접두 명령(`lock xadd`, `lock cmpxchg`) — 해당 **캐시라인을 M 상태로 독점**하고 연산이 끝날 때까지 다른 코어의 접근을 배제한다. 옛날처럼 버스 전체를 잠그지 않고 라인 단위로 잠근다(단, 라인 경계에 걸치면 split lock).

**비용 감각.** 비경합 atomic RMW ≈ 20~40사이클(일반 add의 20배). **경합 시** 비용의 본질은 명령이 아니라 **캐시라인 소유권의 코어 간 이동**(MESI RFO 핑퐁) — 코어 수가 늘수록 선형 이상으로 나빠진다. "lock-free = 빠르다"가 아니라 "lock-free = 커널 진입·우선순위 역전이 없다"이며, **경합 자체를 줄이는 설계(샤딩, 스레드 로컬 집계 후 병합)가 atomic 튜닝보다 상위 해법**이다.

**CAS와 ABA.** `compare_exchange`: 기대값과 같으면 교체, 다르면 현재값을 돌려준다 — lock-free 알고리즘의 원자 부품. ABA 문제: 값이 A→B→A로 돌아오면 CAS는 "변화 없음"으로 착각한다(노드 재활용되는 lock-free 스택의 고전 함정). 대응: 세대 카운터를 함께 비교(태그드 포인터/이중폭 CAS), epoch/hazard pointer 회수 기법. **인덱스가 단조 증가**하도록 설계하면 ABA가 원천 차단되는데, Chase-Lev 데크의 top/bottom이 정확히 그 사례다.

**메모리 순서(memory_order)와 x86.** x86은 TSO(강한 순서) — 유일하게 보이는 재배치가 **StoreLoad**(내 스토어가 남의 로드보다 늦게 보임)다. 따라서 acquire/release는 x86에서 사실상 공짜(컴파일러 재배치 방지용)지만, **StoreLoad 차단이 필요한 지점은 seq_cst 펜스(mfence)가 실제 비용을 낸다**. Winters의 Chase-Lev 구현이 정확히 이 지점을 구분해서 쓴다 (`Engine/Public/Core/JobSystem/WorkStealingDeque.h`):

```cpp
// Pop (owner): bottom 감소 '스토어'가 top '로드'보다 먼저 전역에 보여야
// thief 와 마지막 1개를 두고 안전하게 경합한다 — StoreLoad 차단 필수
m_iBottom.store(b, std::memory_order_relaxed);
std::atomic_thread_fence(std::memory_order_seq_cst);
std::int64_t t = m_iTop.load(std::memory_order_relaxed);
...
// 마지막 1개 — Steal 과 경합
bool ok = m_iTop.compare_exchange_strong(t, t + 1,
    std::memory_order_seq_cst, std::memory_order_relaxed);
```

Push는 release 펜스(데이터 쓰기가 bottom 증가보다 먼저 보이게), Steal은 acquire 로드 + CAS. owner의 Push/Pop은 LIFO(방금 만든 잡 = 캐시가 뜨거운 잡부터), thief의 Steal은 FIFO(반대쪽 끝을 훔쳐 경합 최소화) — 스케줄링 정책 자체가 캐시 지역성 논리로 설계돼 있다는 것까지 말하면 완성.

## 예상 질문 & 모범답변

### Q. CPU가 명령어 하나를 실행하는 과정을 파이프라인으로 설명해보세요.

**답.** 고전 5단계로 나누면 IF(명령어 인출) → ID(디코드+레지스터 읽기) → EX(연산/주소계산) → MEM(메모리 접근) → WB(레지스터 기록)입니다. 파이프라인의 요점은 각 단계를 서로 다른 명령어가 동시에 점유해서, 명령어 하나의 레이턴시는 그대로 두고 **처리량을 단계 수만큼** 끌어올리는 것입니다. 실제 x86은 15~20단계 이상으로 깊고, 디코드 후 μop으로 쪼개 실행합니다.

**왜/트레이드오프.** 깊게 쪼갤수록 클럭은 올라가지만 분기 예측 실패 시 버려야 할 작업(플러시 비용)도 커집니다. 펜티엄4가 30단계 이상으로 갔다가 실패한 게 그 극단 사례입니다.

**꼬리질문 대비.** "그럼 파이프라인이 항상 꽉 차 있나요?" → 아니요, 해저드 때문에 스톨이 생기고 그걸 줄이는 게 포워딩·분기예측·OoO입니다 — 로 자연스럽게 다음 주제로 넘어가면 됩니다.

---

### Q. 파이프라인 해저드의 종류와 해결 방법을 말해보세요.

**답.** 세 종류입니다. ① **구조적** — 자원 경합, 자원 복제(I/D 캐시 분리)로 해결. ② **데이터** — RAW는 진짜 의존이라 포워딩으로 대부분 흡수하되 load-use는 1스톨 불가피, WAR/WAW는 이름 의존이라 레지스터 리네이밍으로 제거. ③ **제어** — 분기 결과 대기, 분기 예측+투기 실행으로 해결.

**함정.** "포워딩이 뭐냐"까지 파고들면: EX 단계 출력이 레지스터 파일(WB)을 거치기 전에 다음 명령어의 EX 입력으로 직접 배선되는 것. "리네이밍이 왜 WAR/WAW를 없애냐": 논리 레지스터보다 훨씬 많은 물리 레지스터에 매핑을 새로 주면 '같은 이름을 재사용해서 생긴 가짜 의존'이 사라지기 때문.

---

### Q. 분기 예측은 왜 필요하고, 실패하면 어떤 비용이 발생하나요?

**답.** 파이프라인이 깊고 넓은 현대 CPU에서 분기마다 결과를 기다리면 매 분기당 수십 μop 슬롯이 비게 됩니다. 그래서 방향(taken 여부)과 목적지(BTB)를 예측해 그 경로를 투기 실행합니다. 실패하면 잘못 실행한 것을 전부 플러시하고 다시 인출 — 최신 x86 기준 **약 15~20사이클** 손실입니다.

**프로젝트 연결.** 수천 엔티티를 도는 게임 루프에서 "엔티티 타입별 switch"는 무작위 분기가 되기 쉽습니다. Winters에서 시스템을 컴포넌트 타입별로 분리한 것(ECS)은 루프 안의 타입 분기 자체를 없애 분기 예측기를 도울 필요조차 없게 만드는 구조입니다.

**꼬리질문.** "예측 잘 되는 분기와 안 되는 분기의 예?" → 루프 종료 조건(거의 항상 taken)은 잘 되고, 무작위 데이터 기반 조건(체력이 50% 이하인가)은 안 됩니다. 후자는 정렬, 파티셔닝, 무분기(branchless) 변환으로 대응합니다.

---

### Q. "정렬된 배열이 정렬 안 된 배열보다 if문 처리에서 빠르다"는 유명한 현상, 왜 그런가요?

**답.** `if (data[i] > 128)` 같은 분기에서 데이터가 정렬돼 있으면 분기 결과가 "쭉 false → 쭉 true"라 예측기가 거의 100% 맞추고, 무작위면 50%를 틀려 매번 ~15~20사이클 플러시를 냅니다. 같은 연산량인데 수 배 차이가 나는 이유가 순수하게 분기 예측입니다.

**꼬리질문 대비.** "그럼 어떻게 고치나요?" → ① 정렬/파티셔닝, ② 무분기 변환(`sum += data[i] * (data[i] > 128)` 또는 cmov), ③ 컴파일러가 벡터화하도록 마스크 연산 형태로 재작성. 단 무분기 변환은 예측이 잘 되는 분기에선 오히려 손해일 수 있다는 것까지 말하면 좋습니다(양쪽을 항상 계산하므로).

---

### Q. 슈퍼스칼라와 비순차 실행(Out-of-Order)을 구분해서 설명해보세요.

**답.** 슈퍼스칼라는 **폭** — 한 사이클에 여러 명령어를 발행하는 능력(현대 x86은 4~6-wide). OoO는 **순서** — 프로그램 순서가 아니라 피연산자가 준비된 순서로 실행하는 능력입니다. 리네이밍으로 가짜 의존을 제거하고, 스케줄러가 준비된 μop을 실행 포트로 보내고, ROB(수백 엔트리)가 프로그램 순서로 커밋해 예외 정확성을 지킵니다.

**왜 중요한가.** OoO의 실전 의미는 **캐시 미스 은폐**입니다. 로드가 300사이클 걸려도 의존 없는 명령을 계속 실행합니다. 뒤집으면 — **포인터 체이싱처럼 다음 로드가 이전 로드 결과에 의존하는 코드는 OoO가 못 구해줍니다.** 연결 리스트 vs 배열 순회의 격차가 벤치마크에서 10배 이상 나는 근본 이유고, 제가 ECS에서 연속 배열 순회를 고집하는 하드웨어적 근거입니다.

---

### Q. 캐시는 왜 존재하나요? 계층 구조와 함께 설명해보세요.

**답.** CPU 코어와 DRAM의 속도 격차가 ~100배(1ns vs 60~100ns)라서, 그 사이를 SRAM 계층으로 메웁니다. L1(32~48KB, 4~5사이클, 코어 전용, I/D 분리) → L2(~1MB, 12~14사이클) → L3(코어당 2~3MB 공유, 30~50사이클) → DRAM(200~300사이클). 프로그램의 지역성(쓴 것 또 쓰고, 이웃 것 곧 쓴다)이 성립하는 한 대부분의 접근이 상위 계층에서 해결됩니다.

**게임 맥락.** 60fps = 16.6ms = 3GHz에서 5천만 사이클. 프레임당 DRAM 미스가 수십만 번이면 예산의 상당분이 순수 대기입니다. 그래서 게임 최적화의 절반은 "연산을 줄이는 것"이 아니라 "메모리 접근 패턴을 고치는 것"입니다.

---

### Q. 캐시라인이 무엇이고, 64바이트라는 크기가 코드 작성에 왜 중요한가요?

**답.** 캐시와 메모리 간 전송 단위이며 x86에서 64B입니다. 1바이트를 읽어도 64B가 올라옵니다. 세 가지 실전 함의: ① 순차 접근하면 이후 15개의 4바이트 값이 공짜 — 배열 순회가 빠른 이유. ② 핫루프에서 안 쓰는 필드가 구조체에 섞여 있으면 그만큼 대역폭 낭비 — hot/cold 분리와 SoA의 근거. ③ 스레드마다 따로 쓰는 변수가 같은 라인에 있으면 false sharing.

**프로젝트 연결.** Winters의 워크스틸링 데크에서 owner용 `bottom`과 thief용 `top`을 `alignas(64)`로 라인 분리했습니다(`Engine/Public/Core/JobSystem/WorkStealingDeque.h`). 두 변수 합쳐 16바이트라 자연 배치하면 반드시 같은 라인에 들어가고, 그러면 owner의 push/pop마다 thief 코어와 라인을 핑퐁하게 됩니다.

---

### Q. 직접 매핑, 집합 연관, 완전 연관 캐시를 비교해보세요.

**답.** 주소가 들어갈 수 있는 슬롯 수의 차이입니다. 직접 매핑은 1곳(빠르고 싸지만 conflict miss), 완전 연관은 아무 곳(충돌 없지만 비교기가 비싸 TLB 정도만), N-way 집합 연관은 절충으로 L1이 보통 8-way입니다. 주소의 인덱스 비트로 set을 고르고 set 내 N개 way 중 LRU 근사로 교체합니다.

**함정/꼬리질문.** "연관도가 있어도 충돌 미스가 나는 경우는?" → 2^n 스트라이드 접근. 예컨대 4096B 간격으로 9개 이상의 주소를 반복 접근하면 8-way라도 같은 set에서 서로를 쫓아냅니다. 이미지/행렬 pitch를 2^n에서 살짝 어긋나게(패딩 추가) 잡는 관행의 이유입니다.

---

### Q. 시간적 지역성과 공간적 지역성을 코드 예시로 설명해보세요.

**답.** 시간적 — 방금 쓴 데이터를 곧 다시 쓴다(루프 변수, 반복 참조되는 룩업 테이블). 공간적 — 쓴 주소의 이웃을 곧 쓴다(배열 순차 순회). 하드웨어는 시간적 지역성을 LRU 교체로, 공간적 지역성을 64B 라인 + 스트라이드 프리페처로 지원합니다.

```cpp
for (auto& t : transforms) Update(t);   // 공간적: 연속 배열
// 같은 데이터에 2개 패스를 나눠 돌리기보다 한 패스로 합치면(loop fusion)
// 시간적 지역성으로 두 번째 패스의 미스가 사라진다
```

**프로젝트 연결.** Winters `CComponentStore`가 dense `std::vector<T>`를 유지하고 swap-and-pop으로 구멍을 안 남기는 것(`Engine/Public/ECS/ComponentStore.h`)이 공간적 지역성의 구조적 보장입니다. 시스템 순회는 항상 이 연속 배열 위에서 돕니다.

---

### Q. 2차원 배열을 행 우선/열 우선으로 순회할 때 성능 차이가 나는 이유는?

**답.** C/C++의 2차원 배열은 행 우선(row-major)으로 연속 배치됩니다. 행 우선 순회는 주소가 4바이트씩 증가해 캐시라인당 16개 float을 다 쓰고 프리페처도 완벽히 동작합니다. 열 우선 순회는 매 접근이 `열수×4` 바이트씩 점프해 라인당 1개 값만 쓰고 버리므로, 큰 행렬에서 5~10배 이상 느려집니다. 행렬이 캐시보다 크면 TLB 미스까지 겹칩니다.

**꼬리질문.** "그럼 전치가 꼭 필요한 연산(행렬곱)은?" → 블로킹/타일링: 캐시에 들어가는 타일 단위로 나눠 타일 안에서 재사용을 극대화. GEMM 라이브러리의 표준 기법입니다.

---

### Q. 주요 연산의 레이턴시를 대략적인 수치로 말해보세요.

**답.** L1 ~1ns(4~5사이클), L2 ~4ns, L3 ~15ns, DRAM 60~100ns, 비경합 atomic RMW 10ns 내외, 분기 미스 ~5ns, 뮤텍스 비경합 ~25ns, 컨텍스트 스위치 µs 단위, NVMe 랜덤 읽기 수십 µs, HDD 시크 ms 단위, 동일 IDC RTT ~0.5ms, 한–미 RTT 100ms+.

**해석까지 붙이기.** 숫자 자체보다 비율이 중요합니다 — L1:DRAM = 1:100이므로 "캐시 미스 한 번 = 산술 수백 개", 메모리:네트워크 = 1:100만이므로 "서버 틱 루프 내 동기 I/O 금지". 이 두 문장이 게임서버 아키텍처 결정 대부분의 근거입니다.

---

### Q. 가상 메모리는 왜 필요하고, 주소 변환은 어떻게 이루어지나요?

**답.** 목적은 ① 프로세스 격리·보호, ② 물리 메모리 단편화 은폐, ③ 지연 할당·mmap·페이지 파일 같은 기법의 기반. x64는 48비트 가상 주소를 4단계 페이지 테이블로 4KB 페이지 단위 변환하고, 이 변환 결과를 TLB에 캐시합니다. 매핑이 없으면 페이지 폴트로 OS가 개입합니다.

**게임 맥락.** "할당은 성공했는데 첫 접근이 느리다"(지연 커밋으로 첫 touch 시 폴트), "로딩 후 첫 프레임 스파이크"의 원인 중 하나가 이것이라, 로딩 단계에서 미리 페이지를 touch하는 워밍업을 둡니다. 서버에서 거대한 월드 상태를 다룰 때 대형 페이지(2MB)로 TLB 커버리지를 512배 늘리는 선택지도 있습니다.

---

### Q. TLB가 무엇이고, 미스가 나면 무슨 일이 벌어지나요?

**답.** 가상→물리 변환 결과의 캐시입니다. L1 dTLB가 ~64엔트리인데 4KB 페이지 기준 커버리지가 고작 256KB — 캐시(L2 1MB+)보다 먼저 바닥날 수 있습니다. 미스가 나면 하드웨어 페이지 워커가 4단계 테이블을 순회하는데, 테이블 항목 자체도 메모리라 수십~백여 사이클이 듭니다.

**꼬리질문 대비.** "캐시 히트인데 느린 경우가 있나요?" → 있습니다, TLB 미스가 겹치는 경우. 데이터가 넓은 주소 공간에 흩어져 있으면(작은 할당 남발) 캐시 워킹셋은 작아도 페이지 워킹셋이 커서 TLB가 병목이 됩니다. 풀/아레나 할당자로 데이터를 페이지 단위로도 뭉치는 이유입니다.

---

### Q. 구조체 패딩은 왜 생기고, sizeof가 예상과 달라지는 경우를 설명해보세요.

**답.** 각 멤버가 자연 정렬 주소에 놓이도록 컴파일러가 사이·꼬리 패딩을 넣기 때문입니다. `char,int,char` 순서면 12바이트, `int,char,char`로 재배치하면 8바이트. 구조체 정렬은 최대 멤버 정렬을 따르고 크기는 그 배수가 됩니다(배열에서 모든 원소가 정렬을 유지해야 하므로).

**왜 중요한가.** 컴포넌트·파티클·스냅샷처럼 수만 개 배열이 되는 타입에서 패딩은 곧 캐시 대역폭 손실입니다. 또 와이어/파일에 구조체를 그대로 쓰면 컴파일러·플랫폼별 패딩 차이로 포맷이 깨집니다 — Winters가 네트워크에 FlatBuffers를 두고 구조체 memcpy 전송을 피하는 이유 중 하나입니다. 검증엔 `static_assert(sizeof(T)==...)`를 습관화합니다.

---

### Q. 엔디안이 무엇이고 네트워크 프로그래밍에서 왜 신경 써야 하나요?

**답.** 다바이트 값의 메모리 바이트 순서입니다. x86/ARM64(주요 게임 타깃)는 리틀 엔디안, 전통적 네트워크 바이트 오더는 빅 엔디안이라 소켓 API의 포트/주소엔 `htons/ntohl`을 씁니다. 같은 머신 안에선 절대 문제가 안 되고, **바이트열이 머신 경계를 넘을 때만** 문제가 됩니다.

**실무 감각.** 페이로드 차원에서는 FlatBuffers/Protobuf가 와이어 포맷을 리틀 엔디안으로 고정해 x86에서 스왑 비용이 0입니다. Winters도 스냅샷/이벤트를 FlatBuffers로 직렬화하므로(Server.vcxproj의 FlatBuffers 포함 경로, `Server/Private/Game/SnapshotBuilder.cpp`) 엔디안·패딩·스키마 버전을 한 계층에서 처리합니다. "손으로 스왑"이 아니라 "포맷 고정 + 생성된 접근자"가 현대적 답이라고 생각합니다.

---

### Q. SIMD가 무엇이고, SSE와 AVX는 어떻게 다른가요?

**답.** 한 명령어로 벡터 레지스터 내 여러 데이터에 동일 연산을 적용하는 데이터 병렬성입니다. SSE는 128비트 XMM(float×4) — x64에선 SSE2가 ABI 기본이라 스칼라 float도 SSE로 컴파일됩니다. AVX/AVX2는 256비트 YMM(float×8)+FMA, AVX-512는 512비트+마스크 레지스터입니다.

**트레이드오프.** 폭이 넓다고 항상 이득은 아닙니다 — 데이터가 SoA로 놓여 있지 않으면 gather/셔플 비용이 이득을 잠식하고, (구형 CPU에선) AVX-512가 클럭 다운을 유발하기도 했습니다. **SIMD 최적화의 8할은 명령어가 아니라 데이터 레이아웃**이라는 말로 정리할 수 있습니다.

---

### Q. DirectXMath는 SIMD를 어떻게 활용하나요? XMVECTOR와 XMFLOAT3의 차이는?

**답.** DirectXMath의 핵심 설계는 **연산 타입과 저장 타입의 분리**입니다. `XMVECTOR`는 사실상 `__m128` — XMM 레지스터에 매핑되는 16B 정렬 연산 타입이고, `XMFLOAT3/XMFLOAT4X4`는 정렬 요구 없는 저장용 구조체입니다. `XMLoadFloat3`로 레지스터에 올리고, XM 연산을 몰아서 한 뒤, `XMStoreFloat4x4`로 내립니다. 함수 인자는 `FXMVECTOR`+`XM_CALLCONV`로 레지스터 전달을 보장합니다.

**프로젝트 연결.** Winters의 TransformSystem이 정확히 이 관용구입니다 — 컴포넌트에는 Vec3/Mat4(저장 타입)만 있고, 시스템이 `XMLoadFloat3 → XMMatrixAffineTransformation → XMStoreFloat4x4`로 경계를 지킵니다(`Engine/Private/ECS/Systems/TransformSystem.cpp`). 그리고 이 규칙을 리뷰에 맡기지 않고 수학 헤더(`Engine/Include/WintersMath.h`) 주석과 타입 설계로 강제했습니다 — Vec3/Mat4는 XMFLOAT 기반 평범한 구조체라 memcpy·배열·네트워크 전송이 전부 안전합니다.

---

### Q. XMVECTOR를 클래스 멤버로 저장하면 왜 위험한가요?

**답.** 세 가지입니다. ① **정렬**: `__m128`은 16B 정렬이 필요한데, 컨테이너·pre-C++17 힙 할당·`#pragma pack`된 구조체 안에서는 보장이 깨질 수 있고, 정렬 위반 시 aligned load에서 크래시합니다(특히 x86 32비트에서 실제로 터지는 고전 버그). ② **이점 소멸**: 저장하는 순간 "레지스터 상주"라는 XMVECTOR의 존재 이유가 사라지고 레이아웃 제약만 남습니다. ③ **직렬화 불능**: 패킷/파일에 넣는 순간 레이아웃·정렬 가정이 포맷에 새어 나갑니다.

**프로젝트 연결.** 그래서 Winters는 "XMVECTOR/XMMATRIX를 멤버/배열/패킷에 저장하지 말고 Load→연산→Store 경계에서만 사용한다"를 `WintersMath.h` 헤더 주석으로 박아 팀 규칙화했습니다. "힙이 위험한 게 아니라 SIMD 저장 타입의 정렬/레이아웃 조건을 전역 데이터 구조에서 놓치기 쉽다"는 문구 그대로입니다.

---

### Q. IEEE 754 float의 비트 구조를 설명해보세요.

**답.** 부호 1비트, 지수 8비트(bias 127), 가수 23비트(암묵적 선행 1로 유효 24비트). 값은 (-1)^s × 1.M × 2^(E-127)이고 십진 유효자리는 약 7자리입니다. 지수 전부 0이면 비정규수(성능 함정 — 연산이 마이크로코드로 떨어져 수십 배 느려질 수 있음), 전부 1이면 Inf/NaN입니다. NaN은 자기 자신과도 같지 않아 `x != x`가 NaN 판별식이 됩니다.

**게임 맥락 한 줄.** "float은 0 근처가 조밀하고 커질수록 성깁니다 — 그래서 좌표가 커지면 정밀도 문제가 생기고, 물리량 스케일 설계(미터 단위, 원점 근처 유지)가 컴퓨터 구조 문제와 직결됩니다."

---

### Q. 0.1f + 0.2f == 0.3f 이 false인 이유는?

**답.** 0.1과 0.2는 이진 유한소수가 아니라 가장 가까운 표현 가능 값으로 반올림돼 저장되고, 덧셈 결과도 다시 반올림됩니다. 그 결과가 "0.3의 가장 가까운 표현"과 일치할 보장이 없습니다(double에선 0.30000000000000004). 즉 버그가 아니라 **유한 비트 이진 표현의 필연**입니다.

**꼬리질문 대비.** "그럼 어떤 수는 정확한가요?" → 2의 거듭제곱 조합(0.5, 0.25, 0.75)과 2²⁴까지의 정수. "돈 계산은요?" → 이진 부동소수점 대신 정수(최소 단위 기반)나 십진 타입. 게임 재화도 서버에선 정수로 다루는 것이 정석입니다.

---

### Q. 부동소수점 비교는 어떻게 해야 하나요? epsilon 비교의 함정은?

**답.** `==` 직접 비교는 연산 경로가 다르면 깨집니다. 절대 epsilon(`fabs(a-b)<1e-6`)은 큰 값에서 무의미하고(1e6 근처 float 간격은 0.0625), 상대 epsilon은 0 근처에서 무의미합니다. 실무 답은 절대+상대 결합이거나, **도메인 지식으로 임계값의 의미를 정의**하는 것입니다.

**프로젝트 연결.** Winters 수학 유틸은 정규화 실패를 `LengthSq <= kEpsilon`으로 판정하고 **fallback 방향을 호출자가 지정**하게 했습니다(`WintersMath::Normalize3D/NormalizeXZ`, `Engine/Include/WintersMath.h`). "epsilon 비교" 자체보다 "실패했을 때 무엇을 반환할지가 정의돼 있는가"가 게임플레이 코드 안정성의 핵심이라는 관점입니다 — 방향 벡터가 0이 되는 순간(제자리 클릭 등)은 반드시 오기 때문입니다.

---

### Q. float은 정수를 어디까지 정확히 표현하나요? 게임에서 문제 되는 순간은?

**답.** 2²⁴ = 16,777,216까지는 연속으로 정확하고, 그 위에선 간격이 2 이상이 되어 `16777216.0f + 1.0f == 16777216.0f`가 됩니다. 게임에서의 발현: ① 프레임 카운트/경과 시간을 float 초로 누적하면 장시간 세션에서 dt가 표현 간격보다 작아져 **시간이 멈춥니다** — 누적은 double 또는 정수 틱으로. ② 대형 월드에서 원점에서 먼 좌표의 정밀도 저하(지터) — origin rebasing으로 대응. ③ 점수/재화 같은 이산 값을 float에 담는 설계 실수.

---

### Q. 게임 서버에서 부동소수점 결정론이 왜 문제가 되고, 어떻게 대응하나요?

**답.** float 연산은 결합법칙이 없어서 **연산 순서가 바뀌면 결과 비트가 바뀝니다.** 순서를 바꾸는 주범은 컴파일러 최적화(재결합, FMA 축약, 근사 명령), 빌드 설정 차이, 스레드 합산 순서, 수학 라이브러리 구현 차이입니다. 서버 권위 + 클라 예측 구조에서 클라와 서버가 같은 시뮬 코드를 돌리는데 결과가 어긋나면, 예측 미스가 누적돼 지속적인 위치 스냅/되감기로 나타납니다.

**프로젝트 연결(핵심 어필).** Winters는 챔피언 시뮬레이션을 `Shared/GameSim`으로 분리해 클라·서버가 같은 코드를 컴파일합니다. 그래서 `/fp:precise`(`<FloatingPointModel>Precise</FloatingPointModel>`)를 Server.vcxproj에만이 아니라 **Client, Engine, GameSim, SimLab까지 전 프로젝트에 명시적으로 고정**했습니다. 한 프로젝트만 `/fp:fast`로 드리프트하면 공유 심 결과가 갈라지는 것을 빌드 설정 차원에서 봉쇄한 것입니다. 동시에 한계도 인지하고 있습니다 — 이것은 동일 툴체인·동일 x64/SSE2 타깃 안에서의 재현성이고, 크로스 플랫폼 록스텝이 필요하면 고정소수점이나 결정론 수학 라이브러리 수준까지 가야 합니다.

**꼬리질문 대비.** "그럼 sin/cos는요?" → 같은 CRT로 링크되는 동일 툴체인 안에선 일치하지만, 플랫폼이 갈라지면 라이브러리 구현 차이가 첫 번째 비결정 요인이 됩니다. 록스텝 RTS들이 트랜센덴털 함수를 자체 테이블/다항식으로 구현했던 이유입니다.

---

### Q. /fp:precise와 /fp:fast의 차이는 무엇이고, 어떤 기준으로 선택하나요?

**답.** `/fp:precise`(MSVC 기본)는 소스에 쓴 연산 순서와 IEEE 반올림을 존중해 재결합·근사 최적화를 금지합니다. `/fp:fast`는 결합법칙을 가정한 재배열, FMA 축약, 역수/rsqrt 근사를 허용해 벡터화가 잘 걸리고 빨라질 수 있지만, 결과가 컴파일러 버전·최적화 수준·인라인 여부에 따라 흔들립니다.

**선택 기준.** "이 수치가 두 곳에서 계산돼 비교되는가?"입니다. 렌더링 전용(한 프레임 살고 사라지는 값)은 fast도 합리적이지만, 게임플레이 판정·네트워크로 검증되는 값은 precise로 못 박아야 합니다. Winters는 시뮬 코드가 Shared로 공유되는 구조라 경계를 나누는 대신 전 프로젝트 precise를 택했습니다 — 빠른 최적화보다 "클라=서버 비트 일치"가 아키텍처의 전제 조건이기 때문입니다.

---

### Q. OOP 게임 오브젝트 방식과 ECS의 성능 차이를 하드웨어 관점에서 설명해보세요.

**답.** `vector<GameObject*>` + `virtual Update()`는 ① 객체가 힙에 산재해 객체마다 캐시 미스, ② 다음 주소가 포인터 역참조 결과라 프리페처·OoO가 무력, ③ vtable 간접 호출로 분기 예측 부담 + 인라인 불가, ④ 지금 안 쓰는 필드까지 캐시라인에 실려 대역폭 낭비. ECS는 같은 타입 컴포넌트를 연속 배열로 모으고 시스템이 선형 순회 — 네 가지 문제가 전부 반전됩니다. 이건 취향이 아니라 §레이턴시 표(1:100)에서 도출되는 결론입니다.

**프로젝트 연결.** Winters `CComponentStore<T>`는 sparse set입니다(`Engine/Public/ECS/ComponentStore.h`): sparse(EntityID→dense index), dense(역방향), data(`std::vector<T>` — 원 주석 그대로 "연속된 메모리 사용!"). 조회는 O(1) 2단계 인덱싱, 순회는 dense 배열 선형 스캔, 삭제는 swap-and-pop으로 **배열에 구멍을 남기지 않아** 순회 밀도를 항상 100%로 유지합니다.

**꼬리질문 대비.** "swap-and-pop의 대가는?" → dense 내 순서 안정성 포기(순서 의존 로직은 별도 정렬 필요), 삭제 시 마지막 원소 이동 비용. "ECS가 항상 이기나?" → 아니요, 엔티티 수가 적고 로직이 무거우면 이점이 사라지고, 컴포넌트 간 랜덤 조인이 많으면 sparse 조회가 곧 랜덤 액세스입니다. 접근 패턴이 왕입니다.

---

### Q. AoS와 SoA의 차이, 각각 언제 유리한가요?

**답.** AoS는 개체 단위 묶음(`struct P{pos,vel,hp} p[N]`), SoA는 필드 단위 배열(`pos[N]; vel[N]; hp[N]`). 한 개체의 여러 필드를 함께 쓰는 로직(개별 유닛 AI)은 AoS가, 한 필드를 전 개체에 대량 적용(이동 적분, 스키닝, 파티클)은 SoA가 유리합니다 — SoA는 캐시라인에 유효 데이터만 실리고 SIMD 8/16-wide에 그대로 먹일 수 있습니다.

**프로젝트 연결.** Winters는 월드 레벨에선 SoA(컴포넌트 타입별 저장소 분리 — Transform만 도는 시스템은 Transform 배열만 만짐), 컴포넌트 내부는 AoS(pos/rot/scale은 항상 같이 쓰므로 `TransformComponent`에 동거)입니다. "AoS냐 SoA냐"가 아니라 **접근 패턴 단위로 계층마다 다르게 답하는 것**이 실전 설계라고 생각합니다.

---

### Q. false sharing이 무엇인지, 실제로 어떻게 발견하고 고치는지 말해보세요.

**답.** 서로 다른 코어가 **논리적으로 무관한** 변수들을 쓰는데 그것들이 같은 64B 캐시라인에 있어, MESI 소유권(RFO)이 코어 간을 핑퐁하는 현상입니다. 증상이 특징적입니다 — 락도 데이터 경합도 없는데 스레드를 늘리면 오히려 느려집니다. 발견은 VTune의 contested access/HITM 이벤트, Linux `perf c2c`, 또는 "의심 변수에 alignas(64) 패딩을 넣어보고 성능이 뛰는지" 실험으로 합니다.

**프로젝트 연결.** Winters Chase-Lev 데크에서 owner 전용 `m_iBottom`과 thief 전용 `m_iTop`을 `alignas(64)`로 각자 라인에 격리했습니다(`Engine/Public/Core/JobSystem/WorkStealingDeque.h`, 주석에 의도 명기). 이 둘은 알고리즘상 다른 스레드가 만지는 게 정상이라, 같은 라인에 두면 워크스틸링의 존재 이유(경합 최소화)가 무효화됩니다.

**꼬리질문.** "패딩의 비용은?" → 메모리. 배열 원소마다 64B 패딩을 넣으면 캐시 용량을 깎아 먹으니, **여러 스레드가 각자 쓰는 것이 확실한 변수에만** 선별 적용해야 합니다.

---

### Q. MESI 프로토콜을 설명해보세요.

**답.** 코어별 캐시 사본의 일관성을 지키는 상태 기계입니다. 라인마다 Modified(나만, dirty) / Exclusive(나만, clean) / Shared(여럿, 읽기 전용) / Invalid. 핵심 전이는 쓰기입니다 — S 상태에서 쓰려면 RFO를 내보내 다른 사본을 전부 무효화한 뒤 M으로 승격하고, 남이 내 M 라인을 읽으면 캐시-투-캐시로 데이터를 넘기며 S로 강등됩니다.

**왜 중요한가.** ① false sharing = 이 무효화 핑퐁의 낭비 사례, ② atomic 경합 비용 = 라인 소유권 이동 비용, ③ "쓰기가 읽기보다 비싸다"(공유 데이터에서)의 근거. 멀티스레드 설계에서 "공유 쓰기를 줄이고 스레드 로컬로 모았다 병합"하는 패턴이 전부 MESI에서 도출됩니다. 실제 인텔은 MESIF, AMD는 MOESI 변형이라는 것도 덧붙일 수 있습니다.

---

### Q. atomic 연산은 CPU에서 어떻게 구현되고, 비용은 어느 정도인가요?

**답.** x86에서 정렬된 8B 이하 로드/스토어는 원래 원자적이고, RMW(fetch_add, CAS)는 `lock` 접두 명령으로 구현됩니다 — 해당 캐시라인을 M 상태로 독점한 채 연산을 완료하며, 요즘은 버스가 아니라 **라인 단위 잠금**입니다(라인 경계에 걸치면 split lock으로 시스템 전체 급정지 — 정렬이 중요한 또 하나의 이유). 비용은 비경합 ~20-40사이클, 경합 시엔 명령이 아니라 **라인 소유권의 코어 간 이동**이 지배해 코어 수에 따라 악화됩니다.

**설계 함의.** "atomic이라 싸다"는 착각을 경계합니다. 진짜 해법은 경합 자체의 제거 — 스레드 로컬 집계 후 병합, 샤딩, 그리고 Winters JobSystem처럼 **각 워커가 자기 데크에서 일하고 빌 때만 남의 것을 훔치는** 구조로 공유 지점을 알고리즘 차원에서 최소화하는 것입니다.

---

### Q. CAS가 무엇이고 ABA 문제는 무엇인가요?

**답.** CAS(compare-and-swap)는 "현재 값이 기대값과 같으면 새 값으로 교체"를 원자적으로 수행하는 명령(`lock cmpxchg`)으로, lock-free 알고리즘의 기본 부품입니다. ABA 문제는 값이 A→B→A로 돌아온 경우 CAS가 "변화 없음"으로 오판하는 것 — 대표적으로 노드를 재활용하는 lock-free 스택에서 이미 해제·재할당된 노드를 살아있다고 믿고 연결을 조작해 터집니다.

**대응.** 세대 카운터 동반 비교(태그드 포인터, 이중폭 CAS), epoch 기반 회수, hazard pointer. 또는 **비교 대상이 단조 증가하도록 설계**해 원천 차단 — Winters Chase-Lev 데크의 `top`이 그 사례입니다. `top`은 훔칠 때마다 증가만 하므로(`compare_exchange`로 t→t+1) 같은 값으로 되돌아올 수 없어 ABA가 성립하지 않습니다.

---

### Q. memory_order(acquire/release/seq_cst)는 CPU 관점에서 무엇을 제어하나요? x86에서도 필요한가요?

**답.** 컴파일러와 CPU 양쪽의 메모리 재배치를 제어합니다. release 스토어는 "그 이전의 쓰기들이 먼저 보임", acquire 로드는 "그 이후의 읽기들이 나중에 수행됨"을 보장해 쌍으로 happens-before를 만듭니다. x86은 TSO라 하드웨어가 이미 acquire/release를 거의 공짜로 만족합니다 — 하지만 두 가지 이유로 여전히 필요합니다. ① 컴파일러 재배치는 아키텍처와 무관하게 막아야 하고, ② TSO에서도 **StoreLoad 재배치**(내 스토어가 상대 로드보다 늦게 보임)는 일어나므로 그걸 막으려면 seq_cst 펜스(mfence)라는 실비용을 내야 합니다.

**프로젝트 연결.** Winters Chase-Lev 데크의 `Pop`이 정확히 StoreLoad 지점입니다 — owner가 `bottom`을 내려 쓴 것이 thief의 눈에 보인 **다음에** `top`을 읽어야 마지막 잡 하나를 두고 벌어지는 경합이 안전합니다. 그래서 그 사이에만 `atomic_thread_fence(seq_cst)`를 두고, 나머지는 relaxed/acquire/release로 최소화했습니다(`Engine/Public/Core/JobSystem/WorkStealingDeque.h`). "전부 seq_cst로 도배"는 정확성은 지키지만 이유를 설명 못 하는 코드가 된다고 생각합니다.

---

### Q. 워크스틸링 스케줄러에서 owner는 LIFO, thief는 FIFO로 가져가는 이유가 뭔가요?

**답.** 캐시 지역성과 경합 분리, 두 가지가 동시에 이유입니다. owner가 방금 push한 잡은 그 데이터가 **owner 코어의 L1/L2에 아직 뜨겁습니다** — LIFO로 즉시 꺼내면 캐시를 그대로 재사용합니다. thief는 반대쪽 끝(가장 오래된 잡)을 가져가는데, ① 오래된 잡은 어차피 owner 캐시에서 식었고, ② 큰 작업 트리의 뿌리 쪽일 확률이 높아 한 번 훔쳐서 오래 일하고, ③ owner와 만지는 끝단이 달라 원자 연산 경합이 마지막 1개 남았을 때로 국한됩니다.

**프로젝트 연결.** Winters `CWorkStealingDeque` 헤더 주석에 이 설계 의도가 그대로 있습니다 — "주인이 Push/Pop (bottom) — LIFO, 캐시 친화적 / 타인이 Steal (top) — FIFO, 캐시 경쟁 최소". 그리고 `WaitForCounter`는 블로킹 대신 help-stealing(메인 스레드도 훔쳐서 일함)으로 대기를 일로 바꿉니다(`Engine/Public/Core/JobSystem.h`).

---

### Q. 컴파일러가 하는 최적화와 CPU가 하는 최적화(OoO)는 어떻게 다르고, 프로그래머 몫은 뭔가요?

**답.** 컴파일러는 정적으로 — 인라인, 루프 언롤, 벡터화, 명령 스케줄링. CPU는 동적으로 — 분기 예측, OoO, 프리페치. 둘 다 못 하는 것이 있는데, **메모리 접근 패턴과 자료구조 선택**입니다. 컴파일러는 `vector<GameObject*>`를 연속 배열로 바꿔주지 못하고, CPU는 포인터 체이싱의 다음 주소를 예언하지 못합니다. 그래서 프로그래머의 최우선 레버는 알고리즘 복잡도 다음에 **데이터 레이아웃**이고, ECS·SoA·풀 할당자가 전부 그 레버에 속합니다.

**꼬리질문 대비.** "restrict/anti-aliasing 얘기 아세요?" → 컴파일러가 벡터화를 포기하는 주원인이 포인터 앨리어싱 가능성입니다. `__restrict`, 값 복사, 로컬 변수로의 캐싱이 컴파일러 증명을 돕는 기법입니다.

---

### Q. 게임 엔진에서 캐시/메모리 병목을 실제로 어떻게 측정하고 개선하나요?

**답.** 원칙은 "추측하지 말고 계측"입니다. 절차: ① 프레임 캡처 프로파일러로 어느 시스템이 예산을 먹는지 좁히고, ② 그 구간에서 IPC(사이클당 명령 수)와 캐시 미스율을 봅니다 — IPC가 1 미만이면 대개 메모리 바운드, ③ 접근 패턴을 고치고(레이아웃 변경, 패스 통합) 전후를 같은 카운터로 비교합니다. VTune, `perf stat`(cache-misses, LLC-load-misses), Tracy 같은 도구를 씁니다.

**프로젝트 연결.** Winters에는 Tracy가 통합돼 있고(`Engine/External/tracy`), 핫패스에 `WINTERS_PROFILE_SCOPE/COUNT` 마커를 심어 둡니다(TransformSystem의 LocalRecompute/WorldRecompute 구간 등). 실제로 미니언 AI 스톨 이슈에서 코드 추론을 거듭하는 것보다 프로파일러 카운터 5분이 정답이었던 경험 이후, "증상 튜닝 전에 계측 먼저"를 팀 규칙(디버그 파이프라인)으로 문서화했습니다.

---

### Q. 게임 서버(수백 접속)에서 컴퓨터 구조 지식이 실제로 어디에 쓰이나요?

**답.** 네 군데가 즉답입니다. ① **틱 루프의 메모리 바운드화 방지** — 월드 상태를 엔티티별 힙 객체가 아니라 연속 배열(ECS)로 유지해 틱당 순회를 캐시 친화적으로. ② **스냅샷 직렬화** — 패딩·엔디안에 중립인 포맷(FlatBuffers)으로 와이어를 고정하고, 스냅샷 빌드가 순차 쓰기가 되도록 버퍼 설계. ③ **멀티스레드 경합 관리** — 세션/룸 단위 샤딩으로 공유 쓰기를 줄이고, 불가피한 공유 지점(카운터, 큐)만 atomic + false sharing 격리. ④ **부동소수점 결정론** — 판정 수치는 /fp:precise 고정, 클라-서버 공유 심의 비트 일치 유지. Winters 서버가 이 네 가지를 전부 실제 구조로 갖고 있어, 각각을 코드로 보여드릴 수 있습니다.

---

### Q. 비정규수(denormal)가 게임에서 성능 문제를 일으킨 사례를 아세요?

**답.** 지수부가 0인 극소값 영역인데, 많은 CPU에서 비정규수 연산이 마이크로코드 보조로 떨어져 수십 배 느려집니다. 게임에서의 전형적 발현은 **지수 감쇠**입니다 — 사운드 페이드아웃, 파티클 속도 감쇠, 스프링 댐핑에서 값이 0으로 수렴하며 비정규 영역에 머무는 몇천 프레임 동안 해당 루프만 느려집니다. 대응은 FTZ/DAZ 플래그(flush-to-zero — 단, IEEE 준수를 깨므로 결정론 코드엔 신중히) 또는 임계값 이하를 명시적으로 0으로 스냅하는 것입니다. 오디오 DSP 쪽에선 아주 유명한 함정입니다.

---

### Q. 하이퍼스레딩(SMT)은 왜 존재하고, 게임에서 항상 이득인가요?

**답.** 한 물리 코어의 실행 자원을 두 하드웨어 스레드가 공유하는 기술로, 존재 이유는 **한 스레드가 캐시 미스로 멈춘 사이 다른 스레드가 실행 포트를 채우는 것**입니다. 즉 SMT의 이득은 워크로드가 메모리 스톨이 많을수록 크고, 두 스레드가 모두 연산 집약적이면 서로 L1/L2와 실행 포트를 뺏어 오히려 손해일 수 있습니다. 게임 엔진 잡시스템에서 워커 수를 논리 코어가 아니라 물리 코어 기준으로 잡거나, 레이턴시 민감 스레드(메인/렌더)를 SMT 짝에서 격리하는 튜닝이 나오는 이유입니다. "코어 수 = 스레드 수"가 아니라 워크로드 특성으로 결정해야 한다는 답이 핵심입니다.

## 내 프로젝트 연결 포인트

면접에서 컴퓨터 구조 질문이 나왔을 때 자연스럽게 꺼낼 수 있는 "실증 카드" 목록. 각 문장은 그대로 말해도 되는 형태로 썼다.

1. **ECS 캐시 설계** — "제 엔진의 컴포넌트 저장소는 sparse set입니다. 조회는 sparse→dense 2단계 O(1), 순회는 dense `std::vector<T>` 선형 스캔이고, 삭제를 swap-and-pop으로 처리해 배열에 구멍이 안 남게 했습니다. 순회 밀도를 100%로 유지하는 대신 순서 안정성을 버리는 트레이드오프를 의식적으로 택했습니다." (`Engine/Public/ECS/ComponentStore.h`)
2. **SIMD 경계 규율** — "DirectXMath를 쓰되 XMVECTOR는 레지스터 타입, XMFLOAT은 저장 타입이라는 경계를 수학 헤더 주석과 타입 설계로 팀 규칙화했습니다. 컴포넌트·패킷에는 절대 XMVECTOR를 넣지 않고 Load→연산→Store로만 씁니다." (`Engine/Include/WintersMath.h`, `Engine/Private/ECS/Systems/TransformSystem.cpp`)
3. **부동소수점 결정론** — "클라와 서버가 같은 시뮬 코드(Shared/GameSim)를 컴파일하는 구조라, /fp:precise를 서버만이 아니라 Client·Engine·GameSim·SimLab 전 프로젝트에 고정해 빌드 설정 드리프트로 인한 심 분기를 봉쇄했습니다. 동시에 이것이 크로스 플랫폼 결정론까지는 아니라는 한계도 압니다." (`Server/Include/Server.vcxproj` 외 전 vcxproj)
4. **false sharing 실전 처리** — "Chase-Lev 워크스틸링 데크에서 owner의 bottom과 thief의 top을 alignas(64)로 라인 분리했습니다. 두 인덱스는 서로 다른 스레드가 만지는 게 정상 동작이라, 같은 라인이면 워크스틸링의 목적 자체가 무효화됩니다." (`Engine/Public/Core/JobSystem/WorkStealingDeque.h`)
5. **메모리 순서의 최소 비용 적용** — "x86 TSO에서 유일하게 비싼 StoreLoad 차단이 필요한 지점(Pop의 bottom 쓰기→top 읽기)에만 seq_cst 펜스를 두고 나머지는 relaxed/acquire/release로 내렸습니다. 왜 그 지점인지 알고리즘 수준에서 설명할 수 있습니다."
6. **ABA 회피 설계** — "데크의 top은 단조 증가만 하도록 CAS를 설계해 ABA가 원천적으로 성립하지 않습니다."
7. **계측 우선 문화** — "미니언 AI 스톨을 코드 추론으로 사흘 쫓다가 프로파일러 카운터로 5분 만에 잡은 경험 이후, '증상 튜닝 전에 debug UI/트레이스/캡처부터'를 팀 문서 규칙으로 박았습니다. Tracy 통합과 WINTERS_PROFILE 마커가 그 인프라입니다."
8. **직렬화와 레이아웃 격리** — "스냅샷·이벤트를 FlatBuffers로 고정해 구조체 패딩·엔디안·스키마 드리프트를 와이어에서 격리했고, 검증 실패는 침묵 드롭 대신 바운디드 트레이스를 남기는 정책을 문서화했습니다." (`Server/Private/Game/SnapshotBuilder.cpp`)
9. **워커 캐시 지역성** — "잡 스케줄링 자체를 캐시 논리로 설계했습니다 — owner LIFO(뜨거운 잡 먼저), thief FIFO(식은 잡·경합 반대편), 대기는 블로킹 대신 help-stealing."
10. **자기 비판 카드(신뢰 확보용)** — "TransformComponent는 Mat4 두 개와 children 벡터를 품어 컴포넌트가 캐시라인 여러 개에 걸칩니다. 핫패스 프로파일 결과에 따라 행렬을 별도 배열로 hot/cold 분리하는 개선 여지가 있다고 보고 있습니다."

## 마지막 점검 체크리스트

- [ ] 5단계 파이프라인: IF-ID-EX-MEM-WB, 해저드 3종(구조/데이터/제어)과 해결(복제/포워딩·리네이밍/예측) 즉답.
- [ ] 분기 미스 ~15-20사이클, 정렬된 데이터 분기 벤치 현상 설명 가능.
- [ ] 슈퍼스칼라=폭, OoO=순서, OoO의 존재 이유=미스 은폐, 한계=포인터 체이싱.
- [ ] 레이턴시: L1 1ns / L2 4ns / L3 15ns / DRAM 100ns / 디스크 µs~ms / 네트워크 ms. 비율(1:100, 1:100만)로 말하기.
- [ ] 캐시라인 64B → 순차접근 공짜, hot/cold 분리, false sharing 세 함의.
- [ ] 연관도: L1 8-way, 2^n 스트라이드 conflict miss 함정.
- [ ] MESI 4상태 + RFO. false sharing = 무효화 핑퐁. Winters alignas(64) 카드.
- [ ] TLB: L1 dTLB 64엔트리 ≈ 256KB 커버, 미스=페이지 워크, 대형 페이지 카드.
- [ ] SIMD: SSE 128b/AVX 256b, x64는 SSE2 기본. DirectXMath = 저장/연산 타입 분리, Load→연산→Store. WintersMath.h 카드.
- [ ] IEEE 754: 1+8+23(bias 127), 유효 7자리, 2²⁴ 정수 한계, NaN != NaN, 결합법칙 불성립.
- [ ] 비교: 절대 eps는 큰 값에서, 상대 eps는 0 근처에서 실패. Winters Normalize fallback 카드.
- [ ] 결정론: 비결정 5원인(컴파일러/x87 vs SSE/라이브러리/스레드 순서/CPU 근사) + /fp:precise 전 프로젝트 고정 카드 + "크로스 플랫폼은 별개" 한계.
- [ ] 패딩: char-int-char 12B vs int-char-char 8B. 큰 정렬 순 배치. static_assert 습관.
- [ ] 엔디안: x86=LE, 네트워크=BE(htons), FlatBuffers는 LE 고정이라 스왑 0.
- [ ] ECS: sparse set 3배열 구조, swap-and-pop, 월드=SoA/컴포넌트=AoS 절충. ComponentStore.h 카드.
- [ ] atomic: lock 접두=라인 독점, 비경합 20-40사이클, 경합 비용=라인 이동. split lock 함정.
- [ ] CAS/ABA: 세대 카운터·단조 증가 설계. Chase-Lev top 카드.
- [ ] memory_order: x86 TSO에서도 StoreLoad는 재배치됨 → seq_cst 펜스 필요 지점을 설명. Pop의 펜스 카드.
- [ ] 워크스틸링: owner LIFO=캐시 온기, thief FIFO=경합 분리, help-stealing.
- [ ] 계측: IPC<1 = 메모리 바운드 의심, VTune/perf/Tracy. "추측 말고 계측" 경험담 준비.
- [ ] 자기 비판 카드 1개(Transform hot/cold 분리 여지)를 반드시 준비 — 완벽 주장보다 신뢰를 산다.
