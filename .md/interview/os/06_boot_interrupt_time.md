# 부팅 · 인터럽트 · 시간 · 디바이스

> "전원을 켜면 main()까지 무슨 일이 일어나나요?", "인터럽트가 뭐고 왜 필요한가요?", "QueryPerformanceCounter는 내부적으로 뭘 읽나요?" — 게임 프로그래머에게 이 질문들은 전부 하나로 이어진다. **내 게임 루프가 딛고 선 발판(시간·디바이스·크래시 경로)이 어디서 오는가**를 원리 수준으로 답하는 챕터다.

---

## ① 한 줄 본질

> **"컴퓨터는 전원이 켜지는 순간 1978년 8086과 호환되는 16비트 리얼 모드에서 깨어나, 펌웨어 → 부트로더 → 커널로 '신뢰와 능력을 한 단계씩 넓히는 사다리'를 밟아 64비트 보호된 세계를 스스로 조립한다(부트스트랩). 그 뒤의 모든 것 — 키 입력, 패킷 도착, 시간의 흐름, GPU의 완료 통지, 심지어 크래시까지 — 는 전부 인터럽트(interrupt)라는 단일 메커니즘으로 CPU에 전달되고, 게임의 QueryPerformanceCounter 한 줄과 Present 한 줄은 이 하드웨어-펌웨어-커널-드라이버 스택 전체 위에 얹혀 있다."**

부팅은 "커널조차 처음엔 없다"는 닭-달걀 문제를 단계적 로딩으로 푸는 과정이고, 인터럽트는 "CPU가 세상을 기다리며 낭비되지 않게" 하는 비동기 통지 설계이며, 시간과 디바이스는 그 인터럽트 위에 세워진 서비스다.

---

## ② 기본 개념

### 전체 조감도 — 전원 버튼에서 게임 main()까지

```
[전원 인가]
   │  CPU 리셋 → 리얼 모드(16bit), 리셋 벡터(물리 주소 최상단 부근 0xFFFFFFF0)로 점프
   │  그 주소엔 펌웨어 ROM이 매핑되어 있다 — "RAM은 아직 텅 비었으므로 첫 코드는 ROM에서"
   ▼
[펌웨어: BIOS/UEFI]  POST(하드웨어 자가 진단) → 메모리/필수 디바이스 초기화 → 부팅 디바이스 탐색
   │  BIOS: 디스크 첫 섹터(MBR 512B)를 0x7C00에 적재 후 점프
   │  UEFI: GPT의 ESP 파티션에서 .efi 실행 파일(PE 포맷)을 로드
   ▼
[부트로더]  커널 이미지를 메모리에 적재, 부팅 파라미터(메모리 맵 등) 준비
   ▼
[모드 전환]  리얼 모드(16bit) → 보호 모드(32bit, GDT+세그먼트) → 롱 모드(64bit, 페이징 필수)
   ▼
[커널 초기화]  메모리 관리자 → IDT/인터럽트 → 스케줄러/타이머 → 디바이스 열거·드라이버 로드
   ▼
[첫 유저 프로세스]  Linux: init(PID 1) / Windows: smss.exe(세션 관리자) → 로그인 → 셸
   ▼
[사용자가 게임 더블클릭]  CreateProcess → 로더가 EXE+DLL 매핑 → CRT 시작 코드 → main()
```

핵심 통찰: **각 단계는 "다음 단계를 로드할 수 있을 만큼만" 안다.** 펌웨어는 (BIOS 기준) 파일시스템을 모르고 섹터만 읽을 줄 알고, 부트로더는 커널 파일 하나를 찾을 줄 알고, 커널이 비로소 프로세스·가상 메모리·드라이버라는 완전한 세계를 만든다. 이 계단식 설계가 부트스트랩(bootstrap) — "자기 신발끈을 잡아당겨 자신을 들어올린다"는 이름의 유래다. 왜 한 번에 못 올라가나? **커널을 로드하려면 디스크를 읽어야 하고, 디스크를 읽으려면 드라이버가 필요하고, 드라이버는 커널 안에 있다** — 이 순환을 끊는 유일한 방법이 "아주 작은 코드가 조금 더 큰 코드를 로드하는" 반복이기 때문이다.

### 왜 리얼 모드에서 시작하나 — 하위 호환의 화석

x86 CPU는 전원이 켜지면 아무리 최신 CPU라도 **리얼 모드(real mode)** 로 깨어난다. 주소는 `세그먼트 × 16 + 오프셋`으로 계산되어 1MB 공간만 접근할 수 있고, 보호(권한 검사·메모리 격리)가 전혀 없다 — 1978년 8086의 동작 그대로다.

```
리얼 모드 주소 계산:  물리 주소 = 세그먼트 << 4 + 오프셋
예) CS=0x07C0, IP=0x0000  →  0x07C0 << 4 + 0 = 0x7C00   (MBR이 적재되는 그 주소)
```

이유는 순전히 **하위 호환성(backward compatibility)** 이다. x86 생태계는 "옛 소프트웨어가 새 하드웨어에서 그대로 돈다"를 팔아온 역사이고, 부팅 규약(리얼 모드 시작, 0x7C00, MBR 시그니처)을 바꾸면 기존 펌웨어·부트로더·OS가 전부 깨진다. 그래서 40년 넘게 "16비트로 깨어나 소프트웨어가 스스로 64비트로 올라가는" 의식을 반복해 왔다.

이 호환 집착의 대표 화석이 **A20 게이트**다. 8086은 주소선이 20개라 1MB를 넘는 계산 결과가 0으로 되감겼는데(wraparound), 일부 프로그램이 이 되감김에 의존했다. 주소선이 늘어난 80286에서 그 프로그램들이 깨지자, IBM은 21번째 주소선(A20)을 강제로 끌 수 있는 스위치를 달아 옛 동작을 재현했다 — 이후 수십 년간 부트로더는 "A20을 켜는 것"부터 해야 했다. **"호환성은 공짜가 아니라 아키텍처에 지층처럼 쌓이는 부채"** 라는 교훈의 표본이다. 대조군으로, 하위 호환 부담이 없는 ARM 계열은 처음부터 자기 세대의 모드로 부팅한다. 그리고 x86도 UEFI 이후로는 펌웨어가 모드 전환을 끝내 주므로, UEFI 부트로더는 처음부터 보호/롱 모드에서 시작한다 — 화석이 서서히 지층에 묻히는 중이다.

### 펌웨어 — BIOS vs UEFI, 무엇이 문제여서 교체됐나

| 축 | BIOS (레거시) | UEFI |
|---|---|---|
| 실행 환경 | 16bit 리얼 모드 | 32/64bit (모드 전환 완료 상태) |
| 부팅 대상 | 디스크 첫 섹터 512B (MBR) | 파티션의 실행 파일 (.efi, PE 포맷) |
| 파티션 테이블 | MBR — 주 파티션 4개, 2TiB 한계 | GPT — 128개+, 사실상 무제한 |
| 파일시스템 이해 | 없음 (섹터 단위 읽기만) | FAT 계열 이해 (ESP 파티션) |
| 확장 | 인터럽트 벡터 기반 서비스(int 13h 등) | 표준화된 부트/런타임 서비스 API |
| 보안 | 없음 | Secure Boot (서명 검증 체인) |

BIOS의 근본 한계는 **"512바이트 안에 부팅 코드를 욱여넣고, 파일시스템도 모른 채, 16비트로 실행한다"** 는 것이다. MBR의 2TiB 한계(32비트 섹터 번호 × 512B)가 디스크 대형화로 실질 문제가 됐고, OS마다 부트로더를 첫 섹터에 밀어 넣는 방식은 멀티부팅·보안 검증과 상성이 나빴다. UEFI는 부팅을 "섹터 마법"에서 "파일시스템의 실행 파일을 로드하는 표준 절차"로 바꾼 것이다.

그 표준화가 열어 준 것이 **Secure Boot** 다: 펌웨어가 부트로더의 서명을 검증하고, 부트로더가 커널의 서명을 검증하고, 커널이 드라이버 서명을 검증한다 — 부팅 사다리가 그대로 **신뢰 사슬(chain of trust)** 이 된다. 부트킷(부팅 과정에 끼어드는 악성코드)이 OS보다 먼저 실행되면 OS의 어떤 보안도 무력해지므로, 검증은 반드시 사다리의 첫 칸부터 걸어야 한다. "각 단계는 다음 단계를 로드한다"는 구조가 "각 단계는 다음 단계를 검증한다"로 자연 확장된 설계다.

POST(Power-On Self Test)는 펌웨어가 부팅 전에 수행하는 하드웨어 자가 진단이다 — 메모리 존재 확인, 필수 디바이스 초기화. "본체에서 삑 소리가 나고 화면이 켜지기까지"가 대략 이 구간이다.

### 부트로더 — 512바이트의 공학

BIOS 경로에서 부트로더의 첫 조각은 MBR(Master Boot Record) 512바이트에 산다:

```
오프셋 0     ~ 445 : 부트 코드 (약 446B)
오프셋 446   ~ 509 : 파티션 테이블 (16B × 4)
오프셋 510, 511    : 시그니처 0x55 0xAA  ← 이게 없으면 펌웨어가 부팅 디스크로 안 침
```

펌웨어는 이 512바이트를 물리 주소 **0x7C00** 에 적재하고 점프한다. 446바이트로는 커널은커녕 파일시스템 파서도 못 넣는다. 그래서 실전 부트로더는 **체인 로딩(chain loading)** — 1단계(512B)는 "더 큰 2단계를 디스크의 알려진 위치에서 읽어오는 것"만 하고, 2단계가 파일시스템을 이해해 커널 파일을 찾아 적재한다. GRUB 같은 부트로더가 다단계인 이유다. 이것도 "각 단계는 다음 단계를 로드할 만큼만 안다"는 사다리 원리의 반복이다.

UEFI 경로는 이 곡예가 필요 없다 — 펌웨어가 FAT 파티션(ESP)에서 PE 실행 파일을 통째로 로드해 주므로, 부트로더는 처음부터 수 MB짜리 정상적인 프로그램으로 시작한다. Windows로 구체화하면 대략: UEFI 펌웨어 → Windows 부트 매니저(.efi) → OS 로더(winload)가 커널(ntoskrnl.exe)과 부트 필수 드라이버를 적재 → 커널 초기화로 이어진다. "부트 필수 드라이버를 커널보다 먼저 로더가 적재한다"는 대목이 닭-달걀 문제의 마지막 해법이다 — 디스크 드라이버 없이 디스크를 읽어야 하는 순환을, 로더가 펌웨어의 읽기 서비스로 미리 끊어 준다.

### 모드 전환 — 16 → 32 → 64, 그리고 GDT가 왜 필요한가

리얼 모드에서 보호 모드(protected mode)로 가는 관문이 **GDT(Global Descriptor Table)** 다. 보호 모드의 세그먼트 레지스터는 더 이상 "주소 × 16"이 아니라 **GDT 항목의 인덱스(셀렉터)** 가 되고, GDT 항목(디스크립터)이 그 세그먼트의 시작 주소·크기 한계·권한 링(ring 0~3)·코드/데이터 구분을 정의한다. 즉 GDT는 "보호"라는 단어의 실체다 — **주소 접근에 권한과 한계라는 메타데이터를 붙이는 표.** 전환 시퀀스는 순서가 강제된다:

```asm
lgdt [gdt_descriptor]      ; 1. 만들어 둔 GDT를 CPU에 등록
mov  eax, cr0
or   eax, 1                ; 2. CR0의 PE(Protection Enable) 비트 세트
mov  cr0, eax
jmp  CODE_SEL:pm_entry     ; 3. 파 점프 — CS를 셀렉터로 갱신하고 파이프라인 비움
```

표가 먼저다. 표 없이 PE를 켜면 다음 명령 페치부터 "셀렉터가 가리키는 디스크립터"가 정의 불능이기 때문이다.

롱 모드(long mode, 64bit)로 가려면 추가로 **페이징이 필수**다 — 페이지 테이블을 만들어 CR3에 걸고, PAE를 켜고, EFER 레지스터의 LME 비트를 세트한 뒤 페이징을 활성화해야 진입할 수 있다. 여기엔 설계 판단이 들어 있다: 64비트 모드는 세그먼트 기반 보호를 사실상 버리고(코드/데이터 세그먼트의 base/limit가 대부분 무시됨) **페이지 테이블 기반 보호로 일원화**했다. 세그먼트는 가변 크기라 단편화와 하드웨어 복잡도를 만들고, 페이징은 고정 크기 페이지로 같은 격리를 더 깔끔하게 제공한다. 32비트 시절 이미 모든 주류 OS가 "세그먼트는 전체 공간으로 펼쳐 두고(flat model) 페이징만 실제로 쓰는" 편법을 썼고, AMD64는 그 현실을 ISA에 반영했다 — **"두 개의 중복된 보호 메커니즘 중 실무가 선택한 쪽만 남긴다"** 는, 하위 호환의 제국 x86에서 보기 드문 정리 결정이다.

### 커널 초기화 → 첫 유저 프로세스 → 게임의 main()

커널이 적재되면 자기 세계를 조립한다: 물리 메모리 맵 파악(펌웨어/로더가 넘겨 준 정보) → 커널 자신의 페이지 테이블 → IDT(인터럽트 테이블) 설치 → 스케줄러·타이머 가동 → 디바이스 열거와 드라이버 로드. 이 시점부터 "인터럽트가 도착하면 처리하고, 아니면 다른 일을 하는" 정상 운영 체제가 된다.

그리고 커널은 **첫 유저 모드 프로세스**를 만든다. Linux는 init(PID 1, 현대엔 systemd)이고, Windows는 대략 이런 트리가 자란다:

```
ntoskrnl (커널)
 └─ smss.exe (세션 관리자, 첫 유저 모드 프로세스)
     ├─ csrss.exe (Win32 서브시스템)
     └─ wininit.exe ─ services.exe (서비스 호스트들), lsass.exe (보안)
        winlogon.exe ─ 로그인 → explorer.exe (셸)
```

여기서 중요한 관점 전환: **부팅이 끝난 뒤 OS 커널은 "돌고 있는 프로그램"이 아니라 "이벤트(인터럽트·시스템 콜)가 올 때만 실행되는 핸들러 집합+라이브러리"에 가깝다.** CPU 시간의 주인은 유저 프로세스들이고, 커널은 그 사이사이에 개입한다. "OS가 CPU를 계속 쓰고 있는 것 아닌가"라는 흔한 오해를 교정하는 문장이다.

사용자가 게임을 더블클릭하면: 셸이 CreateProcess → 커널이 주소 공간을 만들고 로더가 EXE와 DLL들을 매핑 → 진입점은 main()이 아니라 CRT 시작 코드(mainCRTStartup)로, 전역 객체 생성자 등을 돌린 뒤 비로소 main()을 호출한다(이 유저 공간 쪽 각론은 [cpp/02_compile_link_dll.md](../cpp/02_compile_link_dll.md)). 전원 버튼부터 여기까지가 "한 번도 마법 없이" 이어지는 경로다.

### 인터럽트 — 왜 폴링이 아닌가

CPU가 디바이스 상태를 아는 방법은 근본적으로 두 가지다:

- **폴링(polling)**: CPU가 반복해서 "다 됐어?"라고 레지스터를 읽는다. 구현은 단순하지만 기다리는 동안 CPU를 태운다.
- **인터럽트(interrupt)**: 디바이스가 "다 됐어!"라고 CPU에 신호를 보내면, CPU가 하던 일을 중단하고 핸들러를 실행한 뒤 복귀한다.

디스크 한 번 읽기가 밀리초, CPU 사이클이 나노초 단위인 세계에서 폴링은 수백만 사이클을 낭비한다. 그래서 범용 OS는 인터럽트가 기본이다. 다만 절대 법칙이 아니라 트레이드오프다 — 인터럽트는 건당 고정 비용(파이프라인 플러시, 컨텍스트 저장/복원, 캐시 오염)이 있어서, **이벤트가 초당 수십만~수백만 건으로 폭주하면 오히려 폴링이 이긴다.** 고성능 NIC가 인터럽트 병합(coalescing, 여러 패킷을 모아 한 번만 통지)이나 "인터럽트로 깨어난 뒤 한동안 폴링으로 긁는" 하이브리드를 쓰는 이유고, 게임 스레드의 스핀 대기도 같은 논리의 축소판이다. **"드물고 느린 이벤트 = 인터럽트, 극도로 잦은 이벤트 = 폴링"** 이 설계 원칙이다.

### IDT와 이벤트의 3분류 — fault / trap / abort + 하드웨어 인터럽트

CPU에 도착하는 "예정에 없던 제어 이전"은 전부 **IDT(Interrupt Descriptor Table)** 로 디스패치된다. 256개 벡터(vector) 각각에 핸들러 주소가 있고, 벡터 0~31은 CPU 예외용으로 예약되어 있다. 분류가 면접 단골이다:

| 분류 | 발생 원인 | 복귀 주소 | 예 |
|---|---|---|---|
| 폴트(fault) | 명령 실행 중 문제, **고치면 재실행 가능** | 문제를 일으킨 **그 명령** | 페이지 폴트(page fault, #PF=14), 0 나누기(#DE=0), 일반 보호 위반(#GP=13) |
| 트랩(trap) | 의도된 제어 이전 | **다음** 명령 | 브레이크포인트(int3, #BP=3), 시스템 콜류 소프트웨어 트랩 |
| 어보트(abort) | 복구 불능 하드웨어/일관성 손상 | 정의 안 됨 | 더블 폴트(#DF=8), 머신 체크 |
| 하드웨어 인터럽트 | 디바이스 신호, **비동기** | 중단된 지점 | 타이머, 키보드, NIC, GPU 완료 |

복귀 주소 차이가 이 분류의 본질이다. **폴트는 "고친 뒤 같은 명령을 다시 실행한다"는 계약** — 페이지 폴트 핸들러가 페이지를 채워 넣고 복귀하면 아까 실패한 메모리 접근이 이번엔 성공한다. 가상 메모리·디맨드 페이징·copy-on-write·메모리 맵 파일이 전부 이 재실행 계약 위에 서 있다. 트랩은 "이미 의도대로 실행됐으니 다음으로", 어보트는 "상태를 신뢰할 수 없으니 종료"다.

인터럽트가 도착하는 순간 CPU가 하는 일도 짚어 두자(하드웨어가 자동 수행):

```
1. (유저→커널 권한 전환이면) TSS에 등록된 커널 스택으로 스택 교체
   — 유저 스택은 신뢰할 수 없으므로 (일부러 망가뜨린 스택으로 커널을 속일 수 있다)
2. 복귀 프레임 푸시: (SS:RSP,) RFLAGS, CS:RIP (+ 일부 예외는 에러 코드)
3. IDT[벡터]의 핸들러 주소로 점프 (인터럽트 게이트면 IF 클리어 — 중첩 억제)
4. 핸들러 종료 시 iretq — 푸시했던 프레임을 복원하며 원래 지점으로 복귀
```

"유저 스택을 못 믿어서 커널 스택으로 갈아탄다"는 1번이 보안 관점의 핵심 설계다.

마지막으로 동기 vs 비동기 축: 예외는 "내 명령이 원인"이라 재현 가능하고, 하드웨어 인터럽트는 "바깥세상이 원인"이라 타이밍이 비결정적이다. **결정론 시뮬레이션이 "입력만 같으면 결과가 같다"를 지킬 수 있는 것도, 비동기 이벤트(패킷 도착 시점)를 그대로 쓰지 않고 틱 입력으로 정규화해 격리했기 때문**이라는 연결이 여기서 나온다.

### IRQ와 APIC — 인터럽트 배선의 진화

디바이스의 인터럽트 요청선이 **IRQ(Interrupt Request)** 다. 고전 PC는 8259 PIC(Programmable Interrupt Controller) 두 개를 종속 연결해 15개 IRQ를 제공했다 — 단일 CPU 시대의 설계라 모든 인터럽트가 한 CPU로만 가고, 벡터 수도 부족했다. 멀티코어가 오자 이 구조가 병목이 됐고 **APIC(Advanced PIC)** 로 교체됐다:

```
디바이스 ──> I/O APIC ──(라우팅: 어느 코어? 어느 벡터?)──> 코어0 [Local APIC]
디바이스 ──>    │                                          코어1 [Local APIC]
                └──────────────────────────────────────>   코어N [Local APIC]
                          코어들끼리: IPI(Inter-Processor Interrupt)
```

- **Local APIC**: 코어마다 하나. 인터럽트 수신·우선순위 처리, 그리고 코어별 독립 타이머 보유(스케줄러 틱의 발생원).
- **I/O APIC**: 디바이스 인터럽트를 받아 "어느 코어의 어느 벡터로 보낼지" 라우팅. 인터럽트 부하 분산과 특정 코어 격리(게임 스레드 전용 코어에 인터럽트 안 보내기)가 여기서 가능해진다.
- **IPI**: 코어가 다른 코어에 보내는 인터럽트. TLB 격추(shootdown), 원격 코어의 재스케줄 요청 등 멀티코어 OS의 필수 배관이다 — "코어 간에도 대화 수단은 인터럽트뿐"이라는 사실이 중요하다.

현대 PCIe 디바이스는 한발 더 나가 **MSI(Message Signaled Interrupt)** 를 쓴다 — 물리 배선 대신 "특정 주소에 특정 값을 쓴다"는 메모리 쓰기 자체가 인터럽트 전달이 된다. 배선 공유(IRQ 충돌) 문제가 사라지고, 디바이스 하나가 여러 벡터를 가질 수 있어 "NVMe 큐마다, NIC 큐마다 전용 인터럽트 → 각기 다른 코어로 분산"이 가능해졌다. 인터럽트의 역사는 곧 **"한 줄의 전선 → 라우팅 가능한 메시지"** 로의 진화다.

### 인터럽트 핸들러의 제약 — top half / bottom half (Windows: ISR / DPC)

인터럽트 핸들러(ISR, Interrupt Service Routine)는 특수한 문맥에서 돈다: **임의의 스레드가 실행되던 도중, 그 스레드의 문맥을 빌려** 실행되며, 실행 중엔 같은(또는 낮은) 우선순위 인터럽트가 막혀 있다. 그래서 두 가지 제약이 따른다:

1. **잠들 수 없다(스케줄 불가).** 핸들러는 특정 스레드가 아니므로 "블록했다가 나중에 깨어난다"는 개념이 성립하지 않는다. 페이지 폴트를 일으킬 수 있는 페이징 가능 메모리 접근, 블로킹 락, 일반 할당이 전부 금지 대상이다.
2. **짧아야 한다.** 핸들러가 도는 동안 다른 인터럽트가 지연된다. 오디오 버퍼 인터럽트가 밀리면 소리가 끊기고, 타이머가 밀리면 스케줄링이 밀린다.

이 제약이 낳은 보편 패턴이 **작업의 이분할**이다. Linux 용어로 top half(긴급 최소 작업: 하드웨어 ACK, 데이터 스냅)와 bottom half(나머지 처리: 나중에 안전한 문맥에서), Windows 용어로 **ISR과 DPC(Deferred Procedure Call)** 다. Windows는 이를 IRQL(Interrupt Request Level)이라는 우선순위 사다리로 체계화했다:

```
HIGH / IPI / CLOCK      ← 최상위 (클럭·코어 간 통신)
DIRQL (디바이스별)       ← ISR이 도는 레벨: 하드웨어 ACK + DPC 큐잉만
DISPATCH_LEVEL (2)      ← DPC와 스케줄러가 도는 레벨: 실제 처리, 단 여전히 잠들기 금지
APC_LEVEL (1)
PASSIVE_LEVEL (0)       ← 일반 스레드 코드: 여기서만 블로킹 가능
```

게이머가 겪는 "DPC 지연(DPC latency)으로 인한 오디오 튐/프레임 스파이크"가 이 계층의 병목 증상이다 — 잘못 짠 드라이버 하나가 DPC를 길게 잡으면, 그 코어의 DISPATCH_LEVEL 아래 전부(즉 모든 일반 스레드)가 밀린다. **"우선순위가 높다 = 짧아야 할 의무가 크다"** 는 역설적 계약이 이 사다리의 설계 원리다.

### 디바이스와의 대화 (1) — 포트 I/O vs MMIO

CPU가 디바이스 레지스터를 읽고 쓰는 방법은 두 계열이다:

| 축 | 포트 I/O (PIO) | MMIO (Memory-Mapped I/O) |
|---|---|---|
| 주소 공간 | 별도 I/O 공간 (64KB) | 일반 물리 주소 공간에 매핑 |
| 명령 | 전용 `in`/`out` | 일반 load/store (`mov`) |
| 접근 폭 | 좁음 | 임의 폭, 넓은 영역 매핑 가능 |
| 캐시/순서 | 캐시 안 됨 | **캐시·재배열 금지를 명시적으로 설정해야 함** |
| 현재 | x86 레거시 (구형 디바이스) | 현대 표준 (PCIe BAR) |

포트 I/O는 주소 공간이 좁던 시절 "메모리 주소를 아끼려" 만든 별도 공간이라는 역사적 산물이다. 64비트 주소 공간이 남아도는 현재는 MMIO가 표준이다 — 디바이스 레지스터가 그냥 주소가 되므로 특별한 명령이 필요 없고, PCIe가 디바이스마다 주소 영역(BAR)을 할당하는 모델과 자연스럽게 맞는다.

단 함정이 있다: **디바이스 레지스터는 메모리가 아니다.** 읽기에 부수효과가 있을 수 있고(읽으면 FIFO에서 한 항목이 빠짐), 같은 주소를 두 번 읽으면 다른 값이 나오며, 쓰기 순서가 프로토콜이다. 그래서 MMIO 영역은 캐시 불가로 매핑하고, 컴파일러/CPU 재배열을 막아야 한다 — **`volatile`이 원래 태어난 자리가 바로 이 디바이스 레지스터 접근이다.** "volatile은 스레드 동기화 도구가 아니다"라는 C++ 챕터의 명제가, 여기 오면 "volatile의 진짜 용도는 이것이었다"로 완성된다.

### 디바이스와의 대화 (2) — DMA, CPU 개입 없는 전송

디스크에서 1GB를 읽는데 CPU가 레지스터에서 한 워드씩 퍼 나른다면(PIO 방식), 그 시간 동안 CPU는 복사 노동자다. **DMA(Direct Memory Access)** 는 이 노동을 디바이스에 위임한다:

```
CPU: "이 메모리 주소에, 이 크기만큼, 디스크의 이 위치를 갖다 놔"
     (전송 기술서 작성 — 물리적으로 흩어진 페이지들이면 scatter-gather 목록으로)
 │
 ▼            CPU는 이 동안 다른 스레드 실행 (게임 로직, 렌더링...)
디바이스가 버스 마스터(bus master)로서 스스로 메모리에 데이터를 씀
 │
 ▼
완료 인터럽트 → 커널이 대기 중이던 스레드/완료 포트를 깨움
```

DMA가 **필수**인 이유는 규모다: 현대 NVMe SSD는 초당 수 GB, NIC는 수십 Gbps를 옮긴다. 이걸 CPU 복사로 하면 코어 하나가 통째로 I/O에 소모된다. DMA + 완료 인터럽트 조합이 있어야 "CPU는 계산, 디바이스는 전송"이라는 병렬성이 성립하고, **비동기 I/O(IOCP가 대표적)가 "요청 걸어두고 다른 일 한다"를 실현하는 물리적 기반이 바로 이것이다** — IOCP의 '완료 통지'는 결국 DMA 완료 인터럽트 → ISR → DPC → 완료 포트 큐잉이라는 배달 경로의 끝이다.

대가도 있다. 디바이스가 메모리에 직접 쓰므로 (1) **CPU 캐시와의 일관성** 문제 — 캐시에 낡은 사본이 남지 않도록 하드웨어 스눕이나 명시적 관리가 필요하고, (2) **보안** 문제 — "디바이스가 아무 물리 메모리나 쓸 수 있다"는 것은 악성 주변기기의 공격 벡터다. 후자를 막는 장치가 IOMMU — 디바이스용 MMU로, 각 디바이스가 DMA할 수 있는 주소 범위를 페이지 테이블로 제한한다. "CPU에 MMU가 필요했던 이유가 그대로 디바이스에도 반복된다"는 대칭이다.

### 디바이스와의 대화 (3) — 드라이버는 번역가다

디바이스 드라이버(device driver)의 본질은 **"OS의 표준 인터페이스"와 "특정 하드웨어의 레지스터 프로토콜" 사이의 번역**이다. 애플리케이션은 ReadFile/WriteFile/DeviceIoControl 같은 통일된 동사로 말하고, 커널 I/O 관리자가 이를 요청 패킷(Windows에선 IRP, I/O Request Packet)으로 만들어 드라이버 스택에 흘리면, 드라이버가 자기 하드웨어의 MMIO 레지스터 시퀀스와 DMA 기술서로 번역한다. 반대 방향으로는 ISR/DPC로 하드웨어 이벤트를 받아 OS 이벤트(IRP 완료, 완료 포트 큐잉)로 번역한다. 이 표준화 덕분에 게임은 "이 SSD가 삼성인지"를 몰라도 되고, 새 하드웨어는 OS를 고치지 않고 드라이버만 얹으면 된다 — **인터페이스와 구현의 분리라는 소프트웨어 설계 원칙이 OS-하드웨어 경계에서 실행되는 것.**

드라이버가 커널 모드에 사는 이유는 특권 명령·물리 메모리·인터럽트 등록이 필요해서다. 그 대가로 **드라이버 버그는 프로세스가 아니라 시스템을 죽인다**(블루스크린의 대부분이 드라이버 소행). 이 위험 때문에 현대 OS는 가능한 부분을 유저 모드로 밀어내는 방향으로 진화했다 — 아래 GPU 스택의 UMD/KMD 분리가 그 전형이다.

### 시간 — PIT / HPET / TSC, 그리고 QueryPerformanceCounter의 정체

하드웨어 타이머는 세대별로 목적이 다르다:

| 타이머 | 정체 | 해상도/성질 | 역할 |
|---|---|---|---|
| PIT (8254) | 약 1.19MHz 고정 발진 카운터 | 낮음, 접근 느림 | 고전 주기 틱 발생기 (레거시) |
| RTC | 배터리 백업 시계 | 초 단위 | 전원 꺼져도 벽시계(wall clock) 유지 |
| HPET | 표준화된 고해상도 카운터+비교기 | 10MHz 이상, 접근이 MMIO라 상대적으로 느림 | PIT 대체, 폴백 소스 |
| **TSC** | **CPU 코어의 사이클 카운터 (`rdtsc`)** | **최고 해상도, 접근이 명령 하나** | **현대의 주력 시간 소스** |
| APIC 타이머 | 코어별 로컬 타이머 | 코어마다 독립 | 스케줄러 틱/타이머 인터럽트 발생 |

먼저 두 역할을 구분해야 한다: **"몇 시인가"를 세는 카운터**(TSC/HPET — 읽기 전용, 인터럽트 없음)와 **"이 시점에 알려 달라"는 알람**(PIT/APIC 타이머 — 인터럽트 발생기). 스케줄러 틱과 Sleep 만료는 알람 쪽이고, 게임의 델타 타임 측정은 카운터 쪽이다. "타이머"라는 한 단어가 두 장치를 가리키는 게 혼동의 뿌리다.

TSC는 원래 함정투성이였다 — 초기엔 코어 클럭을 그대로 세서 절전/터보로 속도가 변하면 시간도 같이 늘고 줄었고, 코어마다 값이 어긋나 스레드가 코어를 옮기면 시간이 뒤로 가는 것처럼 보였다. 현대 CPU는 **불변 TSC(invariant TSC)**: 실제 코어 클럭과 무관한 **고정 레이트**로 돌고(그래서 "사이클 카운터"라는 이름과 달리 이제는 시간 카운터에 가깝다), 패키지 내 코어 간 동기화된다. 그래서 지금은 가장 싸고(명령 하나) 가장 정밀한 시간 소스다.

**QueryPerformanceCounter(QPC)의 정체**는 이 하드웨어 사정의 추상화 계층이다. Windows가 부팅 시 가장 좋은 소스를 골라(현대 하드웨어에선 사실상 불변 TSC, 신뢰 불가 시 HPET 등으로 폴백) **단조 증가(monotonic)·프로세서 간 일관성**을 보장해 주는 카운터로 포장한 것. `QueryPerformanceFrequency`가 주는 주파수는 **부팅 시 결정되어 이후 변하지 않는다.** 게임이 `rdtsc`를 직접 읽지 않고 QPC를 쓰는 이유가 이것이다 — "이 기계의 TSC를 믿어도 되는가"라는 판정을 OS에 위임한다. 참고로 벽시계(GetSystemTime 계열)는 NTP 동기화로 **뒤로 갈 수 있으므로** 프레임 델타에 절대 쓰면 안 된다 — 게임 내 시간 측정은 단조 시계가 철칙이다.

### 타이머 인터럽트, 스케줄러 틱, 그리고 게임 루프가 기대는 것

주기적 타이머 인터럽트(틱)는 선점형 스케줄링의 심장이다 — 틱마다 커널이 개입해 타임슬라이스 만료를 확인하고 재스케줄한다. Windows의 기본 틱은 약 15.6ms(초당 64회)로, 이것이 **Sleep과 대기 함수들의 기본 입자(granularity)** 가 된다. `Sleep(1)`이 1ms가 아니라 수 ms~15ms를 재우는 이유: Sleep은 "최소 이만큼 재우고, 그 뒤 스케줄러가 나를 다시 볼 때 깨워 달라"는 요청이지 정밀 지연이 아니다.

그래서 게임/미디어 앱은 timeBeginPeriod로 타이머 해상도를 1ms로 올려 왔다 — 대가는 전력 소모 증가(틱이 잦아짐)이고, 최신 Windows에선 이 요청의 적용 방식이 프로세스 상태에 따라 달라지는 변화가 있었다는 정도까지 알아두면 과단정을 피할 수 있다. 커널 쪽도 같은 이유(전력)로 "유휴 코어엔 틱을 안 보내는" 동적/틱리스(tickless) 방향으로 진화해 왔다 — 주기 틱은 편리하지만 잠든 노트북을 깨우는 비용이기 때문이다.

정리하면 게임 루프의 타이밍은 세 층에 기댄다:

```
측정   : QPC(=TSC) — 프레임 델타 계산. 정밀, 인터럽트와 무관
대기   : Sleep/이벤트 — 스케줄러 틱 입자에 종속. 부정밀 (±수 ms)
페이싱 : VSync(Present 블록) — 디스플레이 주사에 동기. 사실상 가장 안정적인 프레임 페이서
```

"측정은 정밀한데 대기는 부정밀하다"는 비대칭이 프레임 페이싱 문제의 뿌리다. 실전 골격은:

```cpp
// 목표 시각까지: Sleep은 거칠고, 스핀은 정밀하나 CPU를 태운다 — 그래서 이단 대기
while (TimeLeft() > 2ms) Sleep(1);          // 대략 대기 (틱 입자에 종속, 저전력)
while (Now() < target)   _mm_pause();       // 마지막 구간만 스핀 (정밀, 고비용)
```

또는 아예 VSync를 페이서로 삼고(Present가 블록) 델타 측정만 QPC로 한다 — 고정 틱 시뮬레이션과 가변 렌더 프레임을 분리하는 구조의 하드웨어적 배경이 이것이다.

### GPU 드라이버 스택 — UMD/KMD와 Present의 여정 (WDDM)

Windows의 GPU 드라이버는 WDDM(Windows Display Driver Model) 아래 두 조각으로 산다:

```
게임 프로세스 (유저 모드)
 ├─ D3D11 런타임 (Microsoft)          ← API 검증, 상태 관리
 ├─ UMD: 벤더 유저 모드 드라이버        ← HLSL 바이트코드→GPU 네이티브 컴파일,
 │                                       GPU 커맨드 버퍼 빌드 (전부 유저 모드에서!)
─────────── 커널 경계 ───────────
 ├─ dxgkrnl (GPU 스케줄러, 비디오 메모리 관리)   ← 여러 프로세스의 커맨드 버퍼를 중재
 └─ KMD: 벤더 커널 모드 드라이버        ← 하드웨어 레지스터/링버퍼에 실제 제출
 ▼
GPU (비동기 실행 → 완료는 인터럽트로 통지 → ISR/DPC → 대기자 깨움)
```

설계 원리는 두 가지다. 첫째, **비싼 작업(셰이더 컴파일, 커맨드 기록)을 유저 모드에 둬서 커널 전환 비용을 피하고, 드라이버 크래시를 프로세스 단위로 격리**한다 — UMD가 죽으면 게임만 죽지만 KMD가 죽으면 시스템이 죽는다. XP 시절 디스플레이 드라이버가 커널에 통째로 살던 구조에서 블루스크린의 주범이었던 역사가 이 분리의 직접 동기다. 둘째, **GPU는 공유 자원이므로 커널(dxgkrnl)이 스케줄링과 메모리를 중재**한다 — 게임, 브라우저, 합성기(DWM)가 동시에 GPU를 쓰는 시대의 필연이다.

Draw 호출 하나하나는 커널로 가지 않는다 — UMD가 유저 모드 버퍼에 커맨드를 쌓을 뿐이다. **Present가 배치 제출의 트리거**가 되어 커널 경계를 넘고, dxgkrnl 스케줄러를 거쳐 KMD가 GPU에 제출한다. "Draw는 싸고 Present 근처에서 목이 졸린다"는 프로파일링 직감의 구조적 이유다. VSync가 켜져 있으면 Present는 제출 큐가 찼을 때 주사 신호를 기다리며 블록한다 — 위 시간 절에서 "VSync가 사실상의 프레임 페이서"라 한 이유다.

그리고 GPU가 응답 불능이 되면 커널은 일정 시간(기본 약 2초) 후 **TDR(Timeout Detection and Recovery)** 로 드라이버를 리셋한다 — 앱에는 `DXGI_ERROR_DEVICE_REMOVED`로 통지되고, 그 디바이스 위의 모든 리소스가 무효가 된다. 무한 루프 셰이더 한 방에 화면이 2초 얼었다 깜빡이며 살아나는 그 경험의 정체다. "디스플레이 드라이버가 응답을 중지했지만 복구되었습니다"라는 옛 풍선 알림이 TDR의 유저 대면 문장이었다.

### 크래시의 해부 — CPU 예외에서 minidump까지

널 포인터 역참조의 전체 경로를 따라가 보자:

```
mov rax, [rbx]   ; rbx = 0
 → CPU: 페이지 폴트(#PF) 발생, IDT 통해 커널 진입
 → 커널: 0번 페이지 부근은 어떤 유효 매핑에도 없음(일부러 매핑 안 함 — 널 참조를 잡기 위해)
         → 정당한 폴트 아님 → 유저 모드로 예외 디스패치 (EXCEPTION_ACCESS_VIOLATION)
 → SEH(Structured Exception Handling) 탐색:
     x64에선 스택의 등록 체인이 아니라 PE에 박힌 언와인드 테이블 기반으로
     __try/__except 프레임을 탐색 (예외 없는 경로 비용 0 — 제로 오버헤드 방식)
 → 아무도 처리 안 함 → 최종 필터(UnhandledExceptionFilter)
 → SetUnhandledExceptionFilter로 등록해 둔 커스텀 핸들러가 있다면:
     MiniDumpWriteDump(DbgHelp) → .dmp 파일 기록 → (선택) 서버 업로드
 → 없다면: WER(Windows Error Reporting) → "프로그램이 작동을 중지했습니다"
```

세 가지가 면접 포인트다. 첫째, **"널 참조가 잡히는 것 자체가 설계"다** — OS가 주소 공간의 첫 영역을 일부러 매핑하지 않기 때문에 널 근처 접근이 페이지 폴트가 된다. 하드웨어의 폴트 메커니즘을 버그 검출기로 재활용한 것.

둘째, **C++ 예외와 SEH는 다른 층이다** — SEH는 OS가 제공하는 예외 배달 메커니즘이고(액세스 위반 같은 하드웨어 예외 포함), C++ 예외는 그 위(MSVC에선 SEH를 타고)에 구현된 언어 기능이다. 액세스 위반은 C++ `catch(...)`의 대상이 아니라 SEH의 영역이다. "try/catch로 크래시를 잡겠다"는 발상이 왜 틀렸는지가 여기서 나온다.

셋째, **minidump는 "사후 부검 킷"이다** — 크래시 순간의 스레드 스택·레지스터·모듈 목록을 담고(옵션으로 힙까지 — 덤프 크기와의 트레이드오프), 빌드 시점의 PDB와 조합하면 사용자 기계에서 난 크래시를 개발 기계에서 콜스택으로 연다. PDB 매칭은 파일 이름이 아니라 PE에 박힌 GUID로 하므로, **출시한 정확히 그 빌드의 PDB를 보관하는 것**이 크래시 리포팅 인프라의 절반이다. 등록 코드 수십 줄로 "재현 불가 크래시 제보"가 "정확한 파일:라인"으로 바뀌는, 비용 대비 효과가 가장 큰 인프라 중 하나다.

## ③ 심화 — 꼬리질문 대비

**"왜 최신 CPU가 아직도 16비트 리얼 모드로 부팅하나?"**
하위 호환 계약 때문이다. x86 생태계는 부팅 규약(리얼 모드 시작, MBR, 0x7C00)을 고정해 두고 그 위의 소프트웨어 자산을 보전해 왔다. 규약을 바꾸는 대신 "소프트웨어가 스스로 모드를 올린다"는 의식을 추가하는 쪽을 택했다. A20 게이트가 이 집착의 화석 표본이고, UEFI가 사실상의 규약 교체다 — UEFI 부팅에서 OS 부트로더는 이미 전환이 끝난 모드에서 시작한다.

**"MBR 512바이트에 부트로더가 다 안 들어갈 텐데?"**
안 들어간다. 그래서 체인 로딩이다 — 1단계(446B 코드)는 더 큰 2단계를 알려진 섹터에서 읽어 오는 일만 하고, 2단계가 파일시스템을 이해해 커널을 적재한다. "각 단계는 다음 단계를 로드할 만큼만 안다"는 부트스트랩 원리가 부트로더 내부에서도 반복되는 것이다.

**"보호 모드 진입에 GDT가 왜 선행 조건인가?"**
보호 모드의 '보호'가 GDT로 정의되기 때문이다. 세그먼트 레지스터가 디스크립터 인덱스로 바뀌는 순간, 각 접근의 한계·권한(ring)·종류를 알려 줄 표가 이미 존재해야 한다. 표 없이 PE 비트를 켜면 다음 명령 페치부터 정의 불능이다. 그래서 순서가 강제된다: GDT 작성 → `lgdt` → CR0.PE → 파 점프.

**"fault와 trap의 복귀 주소 차이가 왜 그렇게 중요한가?"**
가상 메모리가 그 차이 위에 서 있기 때문이다. 페이지 폴트가 '그 명령 재실행'이 아니라 '다음 명령'으로 복귀한다면, 폴트를 일으킨 메모리 접근은 영원히 완료되지 않는다. "고치고 재시도"라는 폴트의 계약이 있어야 디맨드 페이징·copy-on-write·메모리 맵 파일이 투명하게 동작한다. 반대로 트랩(int3, 시스템 콜)이 같은 명령으로 복귀하면 무한 루프다.

**"인터럽트 진입 때 왜 커널 스택으로 갈아타나?"**
유저 스택을 신뢰할 수 없기 때문이다. 악의적/버그 있는 프로그램이 스택 포인터를 임의 주소(커널 데이터 등)로 만들어 두면, 커널이 그 스택에 복귀 프레임을 푸시하는 순간 커널 메모리가 오염된다. 그래서 권한 전환이 있는 인터럽트는 하드웨어가 TSS에 등록된 커널 스택으로 자동 교체한 뒤 프레임을 쌓는다 — "경계를 넘을 때는 상대의 자원을 믿지 않는다"는 원칙의 하드웨어 구현.

**"인터럽트 핸들러에서 왜 잠들면 안 되나?"**
핸들러는 스레드가 아니라 '빌린 문맥'에서 돌기 때문이다. 잠든다는 건 스케줄러가 나중에 나를 깨워 준다는 뜻인데, 핸들러는 스케줄링 단위가 아니므로 깨워 줄 대상이 없다. 게다가 핸들러가 도는 동안 동급 인터럽트가 막혀 있어, 여기서 락을 기다리면 그 락을 풀 코드 자체가 실행 기회를 못 얻는 데드락이 된다. 그래서 ISR은 최소 작업 후 DPC(bottom half)로 미룬다.

**"인터럽트가 항상 폴링보다 나은가?"**
아니다. 인터럽트는 건당 고정 비용(파이프라인 플러시, 문맥 저장, 캐시 오염)이 있어서 이벤트가 극도로 잦으면 배보다 배꼽이 커진다. 초당 수십만 패킷 이상을 받는 서버 NIC는 폴링 또는 인터럽트+폴링 하이브리드(인터럽트로 깨어나 한동안 폴링)가 이기고, 수 마이크로초 내 풀릴 락은 스핀이 이긴다. "이벤트 빈도 × 건당 비용"으로 손익분기점이 갈리는 트레이드오프다.

**"rdtsc를 직접 읽지 않고 QueryPerformanceCounter를 쓰는 이유는?"**
TSC를 믿어도 되는지가 하드웨어마다 다르기 때문이다. 구형 CPU는 절전으로 TSC 레이트가 변했고 코어 간 값이 어긋났다. QPC는 부팅 시 OS가 "이 기계에서 신뢰 가능한 최선의 소스"(현대에선 대개 불변 TSC, 아니면 HPET 등)를 골라 단조 증가와 프로세서 간 일관성을 보장해 주는 추상화다. 직접 rdtsc를 쓰면 그 판정 책임을 앱이 지게 된다.

**"Sleep(1)이 1ms를 보장하지 못하는 이유는?"**
Sleep은 정밀 지연이 아니라 "최소 이 시간 이후, 스케줄러가 나를 다시 선택할 때 깨어난다"는 요청이다. 깨어나는 시점은 타이머 인터럽트 입자(기본 약 15.6ms)와 그 시점의 런큐 상황에 종속된다. timeBeginPeriod로 해상도를 1ms로 올려도 '1ms 입자'가 될 뿐 정확한 1ms가 아니다. 그래서 프레임 페이싱은 "Sleep으로 대충 + 마지막 스핀" 또는 VSync 의존이 실전 패턴이다.

**"DMA가 있는데 CPU 복사(memcpy)는 왜 여전히 존재하나?"**
DMA는 설정 비용(기술서 작성, 캐시 일관성 처리, 완료 통지)이 있어서 작은 전송엔 CPU 복사가 더 싸다. 또 DMA는 디바이스↔메모리 전송 장치이지 메모리↔메모리 범용 복사기가 아니다(그런 엔진도 있지만 범용적이지 않다). "큰 스트리밍 전송 = DMA, 작은 메모리 내 이동 = CPU"로 역할이 갈린다.

**"시스템 콜도 인터럽트인가?"**
분류상 소프트웨어 트랩이고, 고전 구현은 실제로 인터럽트 명령(`int 0x80` 등)을 썼다. 하지만 IDT 디스패치는 범용이라 비싸서, 현대 x86-64는 전용 명령(syscall/sysret)으로 IDT를 우회해 고정된 진입점으로 바로 뛴다 — 검사와 간접 참조를 줄인 전용 고속 경로다. "유저→커널 특권 전환"이라는 본질은 같고, 진입 배관만 최적화된 것.

**"더블 폴트, 트리플 폴트가 뭔가?"**
예외를 처리하러 가는 도중 또 예외가 나는 상황이다(예: 페이지 폴트 핸들러 진입에 필요한 스택이 못 쓰는 상태). 더블 폴트(#DF)는 전용 핸들러로 마지막 수습을 시도하고, 그 처리마저 실패하면 트리플 폴트 — CPU는 포기하고 리셋한다. 커널 스택 오염 같은 치명 버그의 최종 증상이며, "핸들러 자신이 의존하는 자원은 항상 유효해야 한다"는 원칙의 극단적 사례다.

**"DXGI_ERROR_DEVICE_REMOVED는 왜 발생하고 앱은 뭘 해야 하나?"**
GPU가 일정 시간(기본 약 2초) 응답하지 않으면 커널이 TDR로 드라이버를 리셋하고, 그 GPU 위에 있던 모든 디바이스 객체·리소스는 무효가 된다. 원인은 무한 루프 셰이더, 드라이버 버그, 과도한 단일 제출, 하드웨어 불안정 등. 앱의 정석 대응은 디바이스·스왑체인·전체 GPU 리소스의 재생성이다 — 사실상 렌더러 재부팅이라 비용이 커서, 상용 엔진들이 '디바이스 로스트 복구'를 별도 설계 항목으로 다룬다.

**"크래시 순간 스택이 깨져 있으면 minidump가 소용 있나?"**
부분적으로 그렇다. minidump에는 레지스터와 모듈 목록이 함께 담기므로, 스택 워킹이 실패해도 "어느 모듈의 어느 오프셋에서 죽었는가"(RIP 기준)는 대부분 복원된다. 스택 오염 자체가 단서(버퍼 오버런 의심)이기도 하다. 덤프 옵션으로 담는 메모리 범위를 조절할 수 있어 힙 일부를 포함하면 부검 가능성이 올라간다 — 덤프 크기·수집 비용과의 트레이드오프다.

**"게임이 주기적으로 프레임 스파이크를 일으키는데 내 코드가 아니라면, OS 계층에서 뭘 의심하나?"**
인터럽트/DPC 계층이다. 잘못 짠 드라이버의 ISR/DPC가 길게 돌면 그 코어에서 내 스레드가 밀린다 — 전형적으로 네트워크·오디오·스토리지 드라이버가 범인이고, DPC 지연 측정 도구로 확인한다. 그 외 후보는 타이머 해상도 변경(다른 프로세스의 timeBeginPeriod), 전원 관리의 클럭 전환, 그리고 GPU 쪽이면 셰이더 지연 컴파일이다. "내 프로파일러에 안 잡히는 시간은 커널/드라이버가 쓴 시간"이라는 감각이 요점이다.

## ④ Winters에서의 적용

### (1) 게임 루프의 시간 — CTimer는 QPC 스택의 소비자

`Engine/Private/Core/CTimer.cpp:5`의 `Initialize`는 `QueryPerformanceCounter` 3회와 `QueryPerformanceFrequency`(9행)로 시작하고, `Update_Timer`는 24행에서 `(m_FrameTime.QuadPart - m_LastTime.QuadPart) / (f32_t)m_CpuTick.QuadPart`로 프레임 델타를 계산한다. 위 원리와 연결하면: 이 두 API가 결국 불변 TSC(현대 하드웨어 기준)를 OS가 검증·포장해 준 카운터이고, 주파수는 부팅 시 고정된다. 그래서 18~22행에서 1초마다 `QueryPerformanceFrequency`를 재조회하는 블록은 사실 불필요한 레거시 패턴이다(값이 변하지 않으므로 해롭지는 않다) — "QPF는 부팅 시 결정되고 불변"이라는 원리를 알면 자기 코드의 관습을 비판적으로 볼 수 있다는 예로 면접에서 쓸 수 있다.

26~31행의 주석은 시간 계층의 실사고 기록이다: 과거 델타를 16.7ms로 하드 클램프했다가 60fps 경계에서 게임 시간이 실제 시간보다 느려지는 슬로모션 스터터를 겪었고, 지금은 스파이크 상한(0.1s)만 둔다. "측정(QPC)은 정밀하지만 프레임 간격은 스케줄러·VSync·드라이버 사정으로 출렁인다"는 비대칭을 상한 클램프라는 정책으로 흡수한 것이다. 스파이크의 공급원이 바로 이 챕터의 계층들(DPC 지연, 스케줄러 틱, TDR)이라는 점에서, 이 한 줄 클램프는 OS 계층에 대한 방어다.

### (2) 디바이스 생성 — 드라이버 스택과 악수하는 지점

`Engine/Private/RHI/DX11/CDX11Device.cpp:824`의 `D3D11CreateDeviceAndSwapChain` 호출이 이 챕터의 GPU 스택 그림과 만나는 지점이다. 이 한 호출 뒤에서 D3D11 런타임이 벤더 UMD를 게임 프로세스에 로드하고, 커널의 그래픽스 서브시스템과 어댑터를 연결한다. 826행의 `D3D_DRIVER_TYPE_HARDWARE`(어댑터 명시 시 `UNKNOWN`) 선택과 855~856행의 어댑터 폴백 경로는 "어떤 물리 GPU의 KMD 스택에 붙을 것인가"라는 결정이다.

Present 쪽은 1005행 `m_pSwapChain->Present(m_bVSync ? 1 : 0, 0)`이 커맨드 버퍼 배치 제출의 트리거이자 VSync 페이싱 지점이고, 바로 다음 1006행이 `DXGI_ERROR_DEVICE_REMOVED || DXGI_ERROR_DEVICE_RESET` 검사다 — 위에서 설명한 TDR의 도착 지점을 코드가 실제로 받고 있다. 현재는 로그만 남기고 복구(디바이스/리소스 전체 재생성)는 없으므로, "TDR가 무엇이고 정석 복구가 무엇인지 알지만, 복구 경로는 비용 대비 우선순위로 미뤘다"고 말하는 것이 정직한 답변이다 — 모르는 것과 알고 미룬 것의 차이를 면접관은 구분한다.

### (3) 크래시 경계 — minidump 부재가 HIGH로 잡힌 이유

UE 5.7 대비 감사에서 minidump 부재는 HIGH 항목이었다(`.md/engines/02_THREE_ENGINES_CHARACTERISTICS.md:245`). 이 챕터의 크래시 해부와 연결하면 이유가 선명하다: Winters는 현재 액세스 위반이 나면 SEH 체인에 아무 필터가 등록돼 있지 않아 WER 기본 처리로 끝난다 — 크래시 순간의 스택·레지스터가 증발한다. `SetUnhandledExceptionFilter` + `MiniDumpWriteDump` 등록은 코드 수십 줄인데, 그 수십 줄이 "재현 불가 제보"를 "PDB로 여는 콜스택"으로 바꾼다. 에러 복원력 챕터의 로드맵(`.md/interview/engine/12_error_resilience.md:214`)에도 "크래시 경계(minidump)"가 다음 단계로 명시돼 있다. 면접 화법: "예외가 CPU→커널→SEH→최종 필터로 배달되는 경로를 알기 때문에, 어느 지점에 후크를 걸어야 하고 비용이 얼마인지(예외 없는 경로엔 x64 테이블 기반이라 0) 설명할 수 있고, 그래서 이 과제가 감사에서 HIGH인 이유 — 수십 줄로 관측 불가능이 관측 가능으로 바뀐다 — 도 설명할 수 있다."

### (4) 이 개념을 프로젝트 경험으로 말하는 법

부팅·인터럽트 자체를 구현한 경험은 없으므로, 정직한 연결은 "내 코드가 소비하는 계층의 원리를 안다"는 방향이다:

- **비동기 I/O의 물리적 기반**: 서버의 IOCP 모델("요청 걸어두고 완료 통지를 받는다")이 성립하는 이유가 DMA + 완료 인터럽트라는 하드웨어 계층임을 설명할 수 있다 — CPU가 복사 노동에서 해방되어 있기에 워커 스레드가 그동안 다른 세션을 처리한다. IOCP의 완료 통지는 "DMA 완료 → 인터럽트 → ISR → DPC → 완료 포트 큐잉"이라는 배달 경로의 종점이다.
- **결정론과 비동기의 격리**: 하드웨어 인터럽트는 타이밍이 비결정적이므로, 공유 시뮬레이션은 벽시계가 아닌 정수 틱을 진리로 삼고 네트워크 도착 같은 비동기 이벤트를 틱 입력으로 정규화한다 — "비동기 이벤트 vs 동기 예외"라는 이 챕터의 분류가 아키텍처 결정의 언어가 된다.
- **프레임 스파이크 디버깅의 어휘**: 내 프로파일러 구간에 안 잡히는 시간이 보이면 DPC 지연·타이머 해상도·TDR을 후보로 올릴 수 있다 — 증상을 코드 안에서만 찾지 않는 계층 감각. CTimer의 스파이크 클램프가 이 어휘의 방어적 실물이다.

## ⑤ 면접 Q&A

**Q1. 전원 버튼을 누르면 게임 main()까지 무슨 일이 일어나는지 설명해 보라.**
- 핵심: 리셋 벡터(ROM) → 펌웨어 POST → 부트로더(MBR 체인 로딩 또는 UEFI .efi) → 커널 적재 → 모드 전환(리얼→보호→롱, GDT/페이징) → 커널 초기화 → 첫 유저 프로세스 → 로그인/셸 → CreateProcess → 로더 → CRT → main(). 관통 원리는 "각 단계는 다음 단계를 로드할 만큼만 안다"는 부트스트랩 사다리.
- 꼬리 대비: "왜 리얼 모드로 시작?"(하위 호환), "MBR 512바이트로 어떻게?"(체인 로딩), "GDT는 왜 먼저?"(보호의 정의가 표에 있음).

**Q2. 인터럽트와 예외의 차이는?**
- 핵심: 축이 두 개다. 동기(내 명령이 원인 = 예외) vs 비동기(디바이스가 원인 = 하드웨어 인터럽트), 그리고 예외 내부에서 복귀 주소 계약으로 fault(그 명령 재실행)/trap(다음 명령)/abort(복귀 없음)가 갈린다.
- 꼬리 대비: "페이지 폴트가 fault인 게 왜 중요?"(재실행 계약 위에 가상 메모리 전체가 선다).
- Winters: 결정론 시뮬이 비동기 이벤트를 틱 입력으로 정규화하는 이유를 이 분류로 설명.

**Q3. QueryPerformanceCounter는 내부적으로 무엇을 읽나?**
- 핵심: 부팅 시 OS가 고른 최선의 하드웨어 카운터의 추상화 — 현대 하드웨어에선 사실상 불변 TSC(rdtsc), 신뢰 불가 시 HPET 등 폴백. 단조 증가·코어 간 일관성 보장, 주파수는 부팅 시 고정.
- 꼬리 대비: "그럼 rdtsc 직접 읽으면?"(TSC 신뢰성 판정 책임을 앱이 짐), "GetSystemTime으로 델타 재면?"(NTP로 뒤로 갈 수 있음 — 단조 시계 철칙).
- Winters: `CTimer.cpp:5`의 QPC 델타 계산 + QPF 매초 재조회가 불필요한 레거시임을 원리로 지적 가능.

**Q4. Sleep(16)으로 60fps 프레임 페이싱을 하면 안 되는 이유는?**
- 핵심: Sleep은 스케줄러 틱 입자(기본 약 15.6ms)에 종속된 "최소 지연 요청"이지 정밀 지연이 아니다. 깨어나는 시점이 ±수 ms 출렁여 프레임 시간이 요동친다.
- 꼬리 대비: "그럼 실전은?"(timeBeginPeriod + Sleep 대략 대기 + 마지막 스핀, 또는 VSync 페이싱), "왜 OS는 틱을 잘게 안 쪼개나?"(전력 — 틱리스 방향으로 진화).

**Q5. Present를 호출하면 무슨 일이 일어나나?**
- 핵심: Draw들은 UMD가 유저 모드 커맨드 버퍼에 쌓아 둘 뿐이고, Present가 배치 제출 트리거로 커널 경계를 넘는다 → dxgkrnl 스케줄러 중재 → KMD가 GPU에 제출 → GPU 완료는 인터럽트로 통지. VSync 시 Present가 주사 대기 블록 = 프레임 페이서.
- 꼬리 대비: "UMD/KMD를 왜 나눴나?"(커널 전환 비용 회피 + 크래시 격리 + GPU는 공유 자원이라 커널 중재 필요), "DEVICE_REMOVED는?"(TDR 약 2초 → 디바이스 리셋 → 리소스 전멸, 정석은 전체 재생성).
- Winters: `CDX11Device.cpp:824`(디바이스 생성), 1005~1006행(Present와 DEVICE_REMOVED 검사)으로 실물 연결.

**Q6. 크래시 리포팅 시스템을 만든다면 어떻게 설계하겠나?**
- 핵심: 예외 배달 경로(CPU 예외 → 커널 → SEH → 최종 필터)의 끝에 `SetUnhandledExceptionFilter`로 후크, `MiniDumpWriteDump`로 스택·레지스터·모듈 기록, 빌드별 PDB 보관(GUID 매칭), 업로드·심볼화 파이프라인. 예외 없는 경로 비용은 x64 테이블 기반이라 0.
- 꼬리 대비: "C++ try/catch로는 왜 안 되나?"(액세스 위반은 SEH 층), "덤프 크기는?"(스택만 vs 힙 포함 트레이드오프).
- Winters: minidump 부재가 UE5.7 대비 감사 HIGH 항목(`.md/engines/02_THREE_ENGINES_CHARACTERISTICS.md:245`) — "알고 있고 로드맵에 있다"로 답변.

**Q7. IOCP가 "요청 걸어두고 다른 일 한다"를 실현할 수 있는 물리적 근거는?**
- 핵심: DMA — 디바이스가 버스 마스터로서 CPU 개입 없이 메모리에 데이터를 옮기고, 완료를 인터럽트로 통지한다. CPU가 전송 노동에서 해방되어 있으므로 워커 스레드가 그동안 다른 작업을 실행할 수 있다. 완료 통지는 인터럽트 → ISR → DPC → 완료 포트 큐잉의 종점.
- 꼬리 대비: "DMA의 대가는?"(캐시 일관성, 보안 → IOMMU), "왜 폴링 안 하나?"(빈도-비용 트레이드오프, 고성능 NIC는 하이브리드).

## ⑥ 흔한 오답 / 함정

1. **"부팅은 BIOS가 OS를 실행하는 것"이라고 뭉개기.**
   펌웨어는 512바이트(또는 .efi 하나)를 로드할 뿐이다. 커널 적재·모드 전환·초기화라는 다단계 사다리를 설명 못 하면 "전원 → 마법 → 윈도우"가 된다. 각 단계가 "다음 단계를 로드할 만큼만 안다"는 구조와, 그 구조가 닭-달걀 문제(드라이버 없이 디스크 읽기)의 해법임을 말할 수 있어야 한다.

2. **예외와 인터럽트를 동의어로 쓰기.**
   축이 두 개다: 동기(내 명령이 원인 = 예외) vs 비동기(디바이스가 원인 = 하드웨어 인터럽트), 그리고 예외 안에서 fault/trap/abort(복귀 주소 계약)가 갈린다. 페이지 폴트를 "인터럽트의 일종"으로만 말하면 재실행 계약을 모른다는 신호다.

3. **"QueryPerformanceCounter는 OS가 시간을 재 주는 함수" 수준의 답.**
   정체는 하드웨어 카운터(현대에선 대개 불변 TSC)의 OS 추상화이고, 주파수는 부팅 시 고정이며, 단조 증가를 보장한다는 것까지가 원리 수준 답이다. 반대로 "그냥 rdtsc다"라고 단정하는 것도 폴백 소스의 존재를 모르는 과단정이다.

4. **Sleep을 정밀 지연으로 취급.**
   Sleep은 스케줄러 틱 입자에 종속된 "최소 지연 요청"이다. 프레임 페이싱을 Sleep(16)으로 짜면 안 되는 이유를 타이머 인터럽트 계층에서 설명할 수 있어야 한다.

5. **드라이버를 "설치하는 프로그램" 정도로 이해.**
   본질은 표준 인터페이스↔하드웨어 레지스터의 번역가이고, 커널 모드라서 버그가 시스템 전체를 죽인다는 위험, 그래서 UMD/KMD처럼 유저 모드로 밀어내는 진화가 있었다는 것까지가 답이다.

6. **C++ try/catch로 액세스 위반을 잡으려는 발상.**
   하드웨어 예외는 SEH의 영역이고 C++ 예외는 그 위의 언어 기능이다. 크래시 수집은 catch가 아니라 SEH 최종 필터 + minidump로 설계한다.

7. **volatile을 스레드 동기화 도구로 아는 것.**
   volatile의 본가는 MMIO 디바이스 레지스터(부수효과 있는 읽기, 순서가 프로토콜인 쓰기)다. 스레드 간 가시성/순서는 atomics와 메모리 모델의 영역 — 두 세계를 섞으면 둘 다 틀린다.
