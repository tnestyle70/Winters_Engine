# 이력서 & 포트폴리오 전략 — 기술면접 대비

> 대상: 자체 DX11 엔진 **Winters** 제작자, 신입/주니어 게임 클라이언트·서버 프로그래머 지원자.
> 근거: 이 문서의 모든 Winters 수치·파일 경로는 레포(`C:/Users/user/Desktop/Winters`)와 세션 메모리(2026-04~07 개발 기록)에서 실측 확인한 것만 사용했다. 확인 못 한 항목은 `CONFIRM_NEEDED`로 표시했다.

## 출제 경향 개요

한국 게임사 채용에서 이력서/포트폴리오는 "면접 질문의 출제 범위"다. 서류에 쓴 모든 줄이 질문으로 돌아온다는 전제로 작성해야 한다.

- **서류 심사관은 프로그래머다.** 인사팀은 형식만 거르고, 기술 검증은 현업 리드/시니어가 한다. 즉 "열심히 했다"류 서술은 0점, "무엇을 어떤 구조로 만들었고 결과가 수치로 어땠는가"만 점수가 된다.
- **자체 엔진 포트폴리오는 양날의 검.** "왜 상용 엔진이 아닌가"라는 질문이 100% 나오고, 답이 준비되어 있으면 최상급 차별화 요소, 준비가 안 되어 있으면 "고집 세고 협업 안 되는 사람" 프레임으로 소모된다.
- **면접관이 파고드는 순서**: 이력서의 가장 구체적인 수치 → 그 수치의 측정 방법 → 그 문제를 만난 과정 → 대안과 트레이드오프 → "그때로 돌아가면 다르게 할 것". 이 5단 콤보를 모든 항목에 대해 준비해야 한다.
- **신입에게 기대하는 것**: 완성도보다 (1) 문제를 스스로 정의하고 (2) 측정하고 (3) 고치고 (4) 검증한 **사이클의 증거**. Winters 개발 기록(프로파일러로 병목 확정 → 수정 → 프레임 복구 확인, SimLab 해시로 리팩터링 무변경 증명)은 정확히 이 사이클이므로 이력서 서술도 이 구조를 따른다.

## 핵심 개념 정리

### 1. 한국 게임사 서류 3종 구조와 심사 관점

**정의**: 한국 게임사 신입 공채/수시 채용 서류는 통상 (1) 이력서(경력·기술 요약, 1장), (2) 자기소개서(문항별 답변), (3) 포트폴리오(영상 + 문서 + 코드 저장소)로 구성된다. 프로그래머 직군은 여기에 코딩테스트(넥슨/크래프톤 등) 또는 사전 과제가 붙는 경우가 많다.

**동작 원리 (심사 흐름)**:
1. 인사팀 필터 — 지원 자격, 형식, 표절 검사. 여기서 보는 건 "읽을 수 있는 문서인가".
2. 현업 프로그래머 서류 리뷰 — 1인당 수 분. **이력서의 프로젝트 요약과 포트폴리오 영상 첫 30초**가 사실상 전부를 결정한다.
3. 통과 시 이력서가 그대로 면접 질문지가 된다. 면접관은 이력서에 형광펜을 치며 "이거 직접 했어요?"를 항목마다 확인한다.

**예시 (심사관의 실제 시선 이동)**:
```
이력서 스캔 순서:
① 기술 스택 줄 (C++? 버전은? 네트워크 경험?)
② 프로젝트 제목+한 줄 요약 (자체 엔진? 규모는?)
③ 수치가 박힌 불릿 (17.8ms→9ms 같은 것에 눈이 멈춤)
④ 포트폴리오 링크 클릭 → 영상 30초 시청 → 판단
```

**게임 개발 맥락 연결**: Winters는 "자체 DX11 엔진 + 서버 권위 멀티플레이 + 자체 에셋 파이프라인 + 툴"이라는, 신입 포트폴리오에서 극히 드문 4축을 모두 가진다. 서류의 목표는 이 4축을 **각 1줄의 수치 불릿**으로 압축해 ③단계에서 눈을 멈추게 하는 것이다.

### 2. 결과 수치 중심 기술 서술법 (STAR/CAR 패턴)

**정의**:
- **STAR** = Situation(상황) → Task(과제) → Action(행동) → Result(결과). 자소서/면접 답변용 서사 구조.
- **CAR** = Challenge(도전) → Action(행동) → Result(결과). 이력서 불릿용 압축 구조. 이력서에는 CAR을 한 줄로 접고, 면접에서 STAR로 펼친다.

**동작 원리**: 심사관은 "Action만 있는 불릿"(~을 구현했다)을 신뢰하지 않는다. Result(수치)가 있어야 Action이 검증 가능해지고, Challenge가 있어야 난이도가 전달된다. 순서는 **Result를 앞으로** 빼는 것이 스캔에 유리하다: `[결과 수치] — [무엇을] [어떻게]`.

**예시 (Winters 실측 수치로 Before/After)**:

나쁜 서술 (Action만):
```
- 게임 최적화를 진행했습니다.
- 에셋 로딩 시스템을 개발했습니다.
```

좋은 서술 (CAR + 수치, 전부 실측):
```
- 프레임 타임 17.8ms → 9ms (약 2배) 개선 — 자체 CPU 프로파일러(스코프 트리+카운터)를
  먼저 구축해 스키닝 애니메이션 갱신이 프레임의 90%(16ms)임을 계측으로 확정한 뒤,
  정적 엔티티 bAnimated 플래그 도입 + 메인 루프 중복 Update/Render 호출 제거로 해결.
- FBX/GLB 27종 → 자체 바이너리 포맷 .wmesh 전수 변환 파이프라인 구축 —
  Irelia FBX 60MB → 1.2MB(약 50배 압축), POD 레이아웃 static_assert 고정으로
  파싱 없이 zero-copy GPU 업로드.
```

**게임 개발 맥락 연결**: 게임 프로그래머 이력서에서 수치는 ms/fps/MB/개수/처리량이 표준 단위다. "느낌상 빨라졌다"는 서술은 프로파일링을 안 해봤다는 자백으로 읽힌다. Winters는 프로파일러를 직접 만들어 병목을 계측한 기록(`Engine/Private/Manager/Profiler/` 관련 코드, 2026-04-24 세션)이 있으므로 "측정 → 수정 → 재측정" 서사를 실화로 말할 수 있다 — 이것이 신입 중 상위 1%의 서술이다.

### 3. Winters 프로젝트 요약문 (한 줄 / 3줄 버전)

**정의**: 요약문은 이력서 프로젝트 항목의 첫 줄이자, 면접 "프로젝트 소개해주세요"의 첫 문장이자, 포트폴리오 영상의 자막이다. 세 곳에서 동일한 문장을 쓰면 기억에 남는다.

**한 줄 버전 (이력서 헤더용)**:
```
C++20 자체 DX11 엔진 위에 LoL 구조의 MOBA를 서버 권위(authoritative) 멀티플레이로
구현한 1인 프로젝트 — ECS 시뮬레이션, IOCP 서버, 자체 에셋 포맷/툴 체인 포함.
```

**3줄 버전 (포트폴리오 표지/자소서용)**:
```
1. 엔진: C++20/DX11 기반 자체 엔진(Engine DLL / Client / Server / Shared 4계층).
   ECS, work-stealing JobSystem, RHI 추상화, 자체 CPU 프로파일러, ImGui 툴링.
2. 게임플레이/네트워크: 클라 입력 → GameCommand → 서버 GameSim → FlatBuffers
   Snapshot/Event → 클라 시각화의 서버 권위 파이프라인. 챔피언 15종의 스킬/투사체/
   미니언 AI를 서버에서 시뮬레이션하고, 300틱 결정성 해시 테스트로 회귀 검증.
3. 파이프라인: FBX/GLB 27종 → .wmesh 자체 포맷 변환기, 데이터 드리븐 FX 시스템과
   ImGui 이펙트 튜너, Go 마이크로서비스 백엔드(인증/매치메이킹/리더보드) 연동.
```

**동작 원리**: 한 줄 버전은 "자체 엔진 + 서버 권위 + 1인"이라는 희소성 3요소를 앞에 배치한다. 3줄 버전은 심사관 직군별 관심사(엔진/네트워크/파이프라인)를 한 줄씩 분배해 어느 직군 심사관이 읽어도 자기 분야 키워드에 걸리게 한다.

**근거 파일**: 4계층 구조(`Engine/`, `Client/`, `Server/`, `Shared/` + `EngineSDK/`), 챔피언 15종(`Shared/GameSim/Champions/` 하위 Annie·Ashe·Fiora·Irelia·Jax·Kalista·Kindred·LeeSin·MasterYi·Riven·Sylas·Viego·Yasuo·Yone·Zed 15개 폴더), IOCP 서버(`Server/Private/Network/IOCPCore.cpp`), 스냅샷 파이프라인(`Server/Private/Game/SnapshotBuilder.cpp` → `Client/Private/Network/Client/SnapshotApplier.cpp`), 결정성 테스트(`Tools/SimLab/main.cpp`), Go 백엔드(`Services/` — go.mod, cmd/, internal/).

### 5. 서류 3종 각각의 역할 분담

**정의와 원리**: 세 문서는 같은 내용의 반복이 아니라 줌 레벨이 다른 렌즈다.
- **이력서** = 위성 사진. 수치 불릿만. 1~2장.
- **자기소개서** = 항공 사진. 문항별 STAR 서사. 협업·성장·문제해결 스토리.
- **포트폴리오** = 지상 답사. 영상 + 기술 문서 + 코드. "직접 확인 가능"한 증거.

**흔한 실패**: 이력서에 서사를 쓰고(길어서 안 읽힘), 자소서에 기술 나열을 하고(감동 없음), 포트폴리오에 설명 없는 코드 덤프를 던지는 것(안 열어봄). 반대로 배치해야 한다.

**게임 개발 맥락 연결**: Winters처럼 범위가 큰 프로젝트는 "포트폴리오 문서"에 아키텍처 다이어그램 1장(Engine/Client/Server/Shared 의존 방향 + 네트워크 파이프라인)을 반드시 넣는다. 이미 `.md/architecture/WINTERS_CODEBASE_COMPASS.md`와 `WINTERS_DEPENDENCY_MAP.md`에 실측 기반 자료가 있으므로 이를 1장짜리 그림으로 재가공하면 된다.

### 6. 포트폴리오 영상 구성 (1분 데모 + 기술 하이라이트)

**정의**: 심사관이 코드를 열어보게 만드는 관문. 통상 (a) 1분 종합 데모와 (b) 3~5분 기술 하이라이트 2개를 준비하고, 이력서에는 (a) 링크를, 포트폴리오 문서에 (b) 링크를 건다.

**1분 데모 콘티 (Winters 기준)**:
```
0:00-0:05  타이틀: "Winters — C++20 자체 엔진, 서버 권위 MOBA (1인 개발)"
0:05-0:20  인게임: 챔피언 이동/QWER 스킬/투사체/미니언 전투 (가장 그림 좋은 챔피언 2종)
0:20-0:30  화면 분할: 클라 2개 + 서버 콘솔 — 한 클라의 입력이 서버를 거쳐
           다른 클라에 복제되는 장면 (서버 권위의 시각 증명)
0:30-0:40  F3 프로파일러 오버레이 켜기 → 스코프 트리/카운터 → "17.8ms→9ms" 자막
0:40-0:50  ImGui 이펙트 튜너로 FX 실시간 편집 + 에셋 변환기 콘솔(27종 변환) 스냅컷
0:50-1:00  아키텍처 다이어그램 1장 + GitHub/문서 링크 + 연락처
```

**기술 하이라이트 영상 원리**: 1편 = 네트워크(스냅샷 적용, 예측, 재접속 — `SnapshotApplier.cpp` 화면과 함께 코드 워크스루), 1편 = 엔진/파이프라인(.wmesh zero-copy, JobSystem, 프로파일러). **말로 설명하며 코드와 실행 화면을 교차**시키는 것이 핵심 — "본인이 직접 만들었는가" 검증을 영상 단계에서 끝내준다.

**흔한 함정**: 배경음악만 깔린 무언 영상(설명 능력 검증 불가), 10분 넘는 영상(안 봄), 렌더링 잘 나온 장면만 이어붙인 영상(기술 증거 0).

### 8. 회사군별 맞춤 전략

**정의**: 같은 이력서를 모든 회사에 내지 않는다. 프로젝트는 같아도 **불릿의 순서와 강조**를 회사군별로 바꾼다.

**대형 (넥슨/크래프톤/엔씨/스마일게이트 등) — 서버·엔진 기초 역량 강조**:
- 원리: 대형사는 분업이 명확하고, 신입에게 CS 기초(자료구조/OS/네트워크)와 깊이를 본다. 코딩테스트 통과가 전제.
- 불릿 순서: ① 서버 권위 네트워크 파이프라인 ② IOCP/멀티스레딩(JobSystem, race 해결 경험) ③ 결정성 검증(SimLab) ④ 프레임 최적화. "혼자 다 했다"보다 "각 계층의 경계를 어떻게 설계했다"(`Shared`가 Engine에 의존하지 않도록 어댑터+빌드 lint로 강제 — `Tools/Harness/Check-SharedBoundary.ps1`)를 강조.
- 자소서 키워드: 규모 확장(150챔피언 아키텍처), 코드 리뷰 수용(외부 리뷰 20건 전수 반영 경험), 라이브 안정성 관점(크래시 덤프/버전 핸드셰이크 도입 로드맵).
- 크래프톤 지원 시: 정글 게임테크랩 커리큘럼(개인 C++ 엔진 → 엔진 내부 학습)과 Winters의 경로가 일치함을 활용 — UE5.7 소스와 자체 엔진을 7개 차원에서 비교 감사한 경험(2026-07-10)을 직접 언급.

**중소/인디 — 풀스택 + 완주 능력 강조**:
- 원리: 소규모 팀은 "혼자서도 기능 하나를 끝까지 닫는 사람"을 원한다. 렌더링부터 서버, 툴, 백엔드까지 전 파이프라인을 다뤄본 경험이 최대 무기.
- 불릿 순서: ① 1인 풀사이클(엔진+게임+서버+툴+Go백엔드) ② 에셋 파이프라인/툴(팀 생산성 기여 가능) ③ FX·UI 등 눈에 보이는 결과물 ④ 빠른 챔피언 추가 파이프라인(콘텐츠 생산 속도).
- 자소서 키워드: 출시 지향(데모 패키징/공개 릴리스 계획), 툴로 기획·아트 협업 지원(ImGui 이펙트 튜너, 맵 에디터, EldenRingEditor의 Undo 트랜잭션).

**공통 주의**: 어느 쪽이든 "자체 엔진 = 상용 엔진 거부"로 읽히지 않게, 이력서 기술 스택 줄에 상용 엔진 학습 경험 또는 이식 계획(UE 툴 매핑 문서 `.md/architecture/` 계열 작업, Winters↔UE 등가물 매핑)을 한 줄 넣는다.

### 9. 자소서 단골 문항별 답변 전략

**문항 1: "가장 어려웠던 기술적 문제와 해결 과정" (출제율 최상)**
- 전략: 디버깅 서사 중 계측이 반전 역할을 하는 것을 고른다.
- 추천 소재 A (클라): 프레임 저하 — "추측으로 코드를 고치지 않고 프로파일러를 먼저 만들었다. 범인으로 의심한 내비게이션은 0.003ms로 무죄, 진범은 16ms짜리 스키닝 갱신이었다" → 계측 문화 어필.
- 추천 소재 B (서버/멀티스레딩): JobSystem 행(hang) — Chase-Lev deque의 소유자 전용 push 규약을 메인 스레드가 위반 → 메모리 오더링 race → WaitForCounter 무한 대기. 규약 위반을 규명하고 메인 전용 제출 경로로 수정. "lock-free 자료구조는 논문의 전제조건까지가 스펙"이라는 교훈으로 마무리.

**문항 2: "협업 경험" (1인 개발자의 최대 약점 문항)**
- 전략: (1) 팀 프로젝트가 있으면 그것을 메인으로 (마크 던전스/Liberation — CONFIRM_NEEDED), (2) Winters에서는 "협업 가능한 상태 만들기"를 협업 역량의 증거로 치환: 코딩 컨벤션 문서화(`.md/architecture/WINTERS_ENGINE_CONVENTIONS.md`), 외부 코드 리뷰(4회, 지적 20건)를 전수 검증 후 반영한 경험, 온보딩 가이드(`WINTERS_HANDOFF_GUIDE.md`) 작성, 빌드 시 자동 경계 검사로 규칙을 사람이 아닌 기계가 지키게 만든 것.
- 함정: "혼자가 편하다" 뉘앙스 절대 금지.

**문항 3: "지원 동기 / 왜 이 회사인가"**
- 전략: 회사의 실제 기술 과제와 내 경험을 접합. 예: 라이브 MOBA/MMO 보유사 → "서버 권위와 치팅 방지 구조를 밑바닥부터 구현하며 라이브 게임의 신뢰성 문제에 관심을 갖게 되었다" + 그 회사 게임의 네트워크 특성 언급.

**문항 4: "성장 과정 / 개발자가 된 계기"**
- 전략: 추상적 열정 서사 금지. "LoL을 하며 '이 스킬 판정은 서버가 하나 클라가 하나'가 궁금했고, 결국 직접 서버 권위 구조로 재구현했다"처럼 호기심→실행의 구체 서사로.

**문항 5: "입사 후 포부"**
- 전략: 자체 엔진 경험을 회사 기술 스택으로 번역하는 계획을 쓴다. "자체 엔진에서 익힌 렌더 파이프라인/넷코드 원리를 UE 소스 레벨 이해로 확장" — 실제로 UE5.7 소스 비교 감사를 해봤다는 사실이 뒷받침.

### 11. 이전 프로젝트 골격 (마크 던전스 / 스타 WinAPI / Liberation)

**정의**: Winters 이전의 학습 프로젝트들. 이력서에서는 "성장 궤적"(WinAPI 2D → 팀 3D+서버 → 자체 엔진+권위 서버)을 보여주는 역할. 세부 사실은 확인 전이므로 골격만 제시한다.

- **스타크래프트 WinAPI 모작**: WinAPI 메시지 루프/GDI(또는 더블 버퍼링) 기반 2D RTS 모작으로 추정. 어필 포인트 후보: 게임 루프를 프레임워크 없이 밑바닥부터, 다수 유닛 선택/명령 구조. `CONFIRM_NEEDED`: 기간, 구현 범위(길찾기? 포그? 유닛 AI?), 코드 존재 여부.
- **마크(마인크래프트) 던전스 모작**: 팀 프로젝트로 추정. 확인된 사실 1건 — **서버가 TCP IOCP였다**(2026-04-30 세션 기록에 "이전 SR_MinecraftDungeons Server는 TCP IOCP, UDP는 Winters가 처음"이라고 명시). 따라서 "IOCP를 두 프로젝트에서 연속 사용, Winters에서 구조를 개선"이라는 연결 서사 가능. `CONFIRM_NEEDED`: 기간, 팀 규모, 본인 담당(클라/서버), 엔진(DX9? DX11? 팀 프레임워크?), 핵심 구현.
- **Liberation**: 정체 미확인. `CONFIRM_NEEDED`: 팀/개인 여부, 장르, 사용 기술, 본인 기여, 수상/전시 여부.

**서사 원리**: 세 프로젝트는 개별 자랑이 아니라 "같은 문제(게임 루프, 네트워크)를 점점 더 어려운 조건에서 다시 푼" 반복 학습의 증거로 배치한다. 면접 답변: "IOCP는 마크 던전스에서 처음 썼고, 그때는 [한계 — CONFIRM_NEEDED], Winters에서는 워커/틱 스레드 분리와 커맨드 큐 구조로 재설계했습니다."

C++20 자체 DX11 엔진 위에 MOBA를 서버 권위 구조로 구현했고, 클라 입력이 서버 ECS 시뮬레이션을 거쳐 FlatBuffers 스냅샷으로 복제되는 파이프라인 전체를 혼자 완성했습니다. 이 과정에서 프로파일러를 직접 만들어 프레임을 17.8ms에서 9ms로 줄였고, 300틱 결정성 해시 테스트로 리팩터링 회귀를 검증하는 습관을 얻었습니다. 이 경험으로 [회사]의 [직군]에서 측정하고 검증하는 개발을 하고 싶습니다."

### Q. 왜 언리얼/유니티를 안 쓰고 자체 엔진을 만들었나요?
- **정의/원리**: 이 질문의 실제 의도는 "상용 엔진을 무시하는가, 팀에 와서 우리 스택을 배울 것인가"의 태도 검증이다.
- **모범답변**: "상용 엔진이 더 낫다는 걸 부정하려는 게 아니라, 상용 엔진이 숨겨주는 계층(렌더 파이프라인, 넷코드, 에셋 쿠킹)을 한 번은 직접 소유해봐야 어떤 엔진에서든 원리로 문제를 풀 수 있다고 판단했습니다. 실제로 UE 5.7 소스를 받아 제 엔진과 빌드 강제, 넷코드, 에셋 파이프라인 등 7개 차원에서 비교 감사를 했고, 제 엔진의 약점(크래시 덤프 부재, CI 부재, 버전 핸드셰이크)을 UE의 해법 기준으로 로드맵화했습니다. 자체 엔진은 목적이 아니라 상용 엔진을 소스 레벨로 읽기 위한 훈련 과정이었습니다."
- **트레이드오프 인정**: "혼자 엔진을 만드느라 콘텐츠 완성도와 팀 협업 경험은 상대적으로 부족합니다. 그래서 컨벤션 문서화와 빌드 레벨 규칙 강제로 '협업 가능한 코드베이스' 상태를 유지하는 데 별도로 투자했습니다."
- **꼬리질문 대비**: "그럼 UE에서 뭐가 제일 인상 깊었나" → 규칙을 문서가 아니라 빌드시스템/매크로가 강제하는 문화(모듈 경계, ensure 계열)라고 답하고 내 lint 도입(`Tools/Harness/Check-SharedBoundary.ps1`) 사례로 연결.

### Q. 이력서의 '17.8ms → 9ms'는 어떻게 한 건가요?
- **답변 구조 (STAR 풀버전)**: "미니언 다수 스폰 후 프레임이 17.8ms까지 떨어졌습니다(S). 추측 수정 대신 원인 계측을 과제로 잡았습니다(T). 엔진에 스코프 트리+카운터 방식 CPU 프로파일러를 먼저 구현하고 내비게이션/AI/애니메이션에 계측 지점을 심었습니다. 의심했던 A* 길찾기는 호출당 0.003ms로 무죄였고, 진범은 스키닝 본 갱신 16ms — 프레임의 90%였습니다. 구조물·맵 같은 정적 엔티티까지 매 프레임 본을 갱신하고 있었고, 추가로 메인 루프에서 Update/Render가 중복 호출되는 버그도 발견했습니다(A). 정적 엔티티에 bAnimated 플래그를 도입해 스킵하고 중복 호출을 제거해 9ms(~110fps)로 복구, 프로파일러 카운터로 애니 갱신 호출 수 감소를 재확인했습니다(R)."
- **핵심 교훈 한 줄**: "이 경험 이후 '계측 없는 최적화는 하지 않는다'가 원칙이 됐습니다."
- **꼬리질문 대비**: "왜 VTune 안 썼나" → 아래 문항. "다음 병목은?" → 스키닝 자체의 병렬화(JobSystem)와 시야 밖 엔티티 컬링이 후속 과제였다고 답변.

### Q. 프로파일러를 직접 만들었다고요? Tracy나 VTune을 쓰면 되지 않나요?
- **원리**: 외부 프로파일러는 범용 계측에 강하지만, 게임 도메인 카운터(길찾기 재탐색 횟수, 애니 갱신 호출 수)와 인게임 오버레이 상시 노출은 자체 구현이 빠르다.
- **모범답변**: "외부 툴을 배제한 게 아니라 역할을 나눴습니다. 상시 켜두는 인게임 F3 오버레이와 게임 로직 카운터(A* 방문 노드 수, 애니 갱신/스킵 수)는 자체 구현이 통합 비용이 낮았습니다. 구현하며 매크로의 `__LINE__` 지연 확장 문제(2단계 CONCAT 필요), 멀티스레드에서 스코프 스택의 thread_local 분리 같은 실전 함정도 배웠습니다. GPU 계측은 자체 툴 영역이 아니라서 RenderDoc을 씁니다."
- **꼬리질문 대비**: "thread_local 분리는 왜?" → 워커 스레드들이 단일 vector 스택을 공유하면 push_back 재할당 중 race로 크래시 — 실제로 겪고 고쳤다고 답변.

### Q. .wmesh 같은 자체 에셋 포맷은 왜 만들었나요? FBX를 그대로 로드하면 되잖아요.
- **정의**: 런타임 포맷과 교환(interchange) 포맷의 분리는 상용 엔진 공통 패턴이다(UE의 .uasset 쿠킹, 유니티의 임포트 라이브러리).
- **모범답변**: "FBX는 교환 포맷이라 런타임 로드 시 Assimp 파싱+후처리 비용이 크고 메모리 레이아웃도 GPU와 다릅니다. 그래서 오프라인 변환기(`Tools/WintersAssetConverter`)로 GPU 업로드 가능한 레이아웃 그대로 저장하는 .wmesh를 설계했습니다. 정점/서브메시 구조체를 POD로 고정하고 크기를 static_assert로 박아, 로드는 헤더 검증 후 포인터를 그대로 버퍼 업로드하는 zero-copy입니다. 27종 에셋을 전수 변환했고 Irelia 기준 60MB FBX가 1.2MB가 됐습니다."
- **트레이드오프**: "버전 관리가 약점입니다 — 초기 포맷에 버전 필드를 안 넣어서, 포맷 진화 시 분기 로딩이 불가능한 상태였고 이후 감사에서 HIGH 이슈로 잡아 로드맵에 올렸습니다. 포맷 설계는 첫 필드부터 버전을 넣어야 한다는 걸 배웠습니다."
- **꼬리질문 대비**: "엔디안/정렬 문제는?" → 단일 플랫폼(Windows x64) 전제를 명시했고, static_assert가 컴파일러 간 레이아웃 드리프트를 잡는 1차 방어선이라고 답변.

### Q. 서버 권위(server authority) 구조를 설명해보세요.
- **정의**: 게임 상태의 유일한 진실 원본(source of truth)을 서버가 소유하고, 클라이언트는 입력을 보내고 결과를 시각화만 하는 구조. 치팅 방지와 클라 간 일관성의 기반.
- **모범답변**: "제 구조는 5단계입니다. ①클라가 입력을 GameCommand로 직렬화해 전송 ②서버가 커맨드를 큐잉·검증 ③서버 ECS 시뮬레이션이 고정 틱으로 이동/스킬/투사체/데미지를 판정 ④결과를 FlatBuffers Snapshot(상태)+Event(순간 사건)로 브로드캐스트 ⑤클라가 적용·보간해 시각화(`Server/Private/Game/SnapshotBuilder.cpp` → `Client/Private/Network/Client/SnapshotApplier.cpp`, `EventApplier.cpp`). 클라는 데미지 계산 코드 자체가 없어서 메모리 조작으로도 판정을 바꿀 수 없습니다. 서버 봇 AI도 특권 경로가 아니라 플레이어와 동일한 GameCommand 생산자로 만들어 판정 경로를 단일화했습니다."
- **왜/트레이드오프**: "권위 서버의 비용은 반응성입니다. 그래서 클라 예측을 얹었고, 그게 다음 난제였습니다."
- **꼬리질문 대비**: "스냅샷 전체를 매번 보내나?" → 현재는 룸 단위 스냅샷이고, 델타 압축과 AOI(관심 영역)가 UDP 넷스택 설계에 포함된 다음 단계라고 솔직히 답변.

### Q. 클라이언트 예측은 어떻게 구현했고, 서버와 어긋나면 어떻게 하나요?
- **정의**: 입력 즉시 로컬에서 결과를 가정 렌더링하고, 이후 도착한 서버 상태로 조정(reconciliation)하는 기법.
- **모범답변**: "이동과 회전에 예측을 적용했습니다. 우클릭 즉시 로컬에서 이동·회전을 시작하고, 서버 스냅샷이 도착하면 서버 값으로 수렴시킵니다. 여기서 배운 게 많은데, 대표적으로 회전(yaw) 떨림 문제가 있었습니다. 서버 스냅샷이 아직 예전 회전값인 상태에서 로컬 예측값을 덮어써 캐릭터가 홱 돌아가는 증상 — 커맨드 ack 시퀀스가 진행됐다고 예측 보호를 풀면 안 되고, 서버 yaw가 실제로 따라잡거나 스킬 같은 액션 락이 잡을 때까지 예측을 보호해야 했습니다. 또 ±π 경계에서 매 틱 정규화하면 급속 클릭 시 경계를 재교차하며 떨려서, 저장은 연속각으로 하고 정규화는 전송·비교 시점에만 하도록 규칙을 세웠습니다."
- **함정/꼬리질문 대비**: "미스프리딕션 시 스냅?" → 현재 수렴 방식이며, 되감기(rewind-replay) 방식과의 비교와 그 비용(입력 버퍼 보관, 재시뮬레이션)을 설명할 수 있게 준비. "왜 이동만 예측?" → 스킬은 서버 판정 대기(밸런스 민감), 이동은 지연 체감이 가장 큰 곳이라 선택했다고 답변.

### Q. TCP를 썼던데, 게임 서버면 UDP 아닌가요?
- **원리**: TCP는 순서·신뢰 보장이 공짜지만 head-of-line blocking으로 최신성이 죽는다. UDP는 최신성이 좋지만 신뢰/순서/조각화를 직접 만들어야 한다.
- **모범답변**: "의도적 단계 설계였습니다. 먼저 TCP로 권위 파이프라인(커맨드→시뮬레이션→스냅샷)의 정합성을 완성해 '게임플레이 로직'과 '전송 계층' 문제를 분리했고, 전송 교체가 가능하도록 transport-aware한 코드와 transport-중립 코드의 경계를 설계 문서에 명시했습니다. UDP 넷스택은 신뢰성(ack/재전송), 조각화, 스냅샷 엔벨로프, 델타·AOI까지 6개 마일스톤으로 설계를 완료한 상태입니다. TCP의 한계로는 스냅샷 유실 시 오래된 데이터까지 순서대로 다 받아야 하는 head-of-line blocking을 꼽겠습니다."
- **꼬리질문 대비**: "UDP에서 ack는 어떻게 설계했나" → 최신 시퀀스 기준 ack + 비트필드 방식, 스냅샷은 재전송보다 최신본 대체가 원칙이라고 답변 준비.

### Q. IOCP가 뭐고 왜 썼나요? select/epoll과 비교하면?
- **정의**: IOCP(I/O Completion Port)는 Windows의 비동기 I/O 완료 통지 커널 큐. 소수의 워커 스레드가 다수 소켓의 완료 이벤트를 큐에서 꺼내 처리하는 proactor 모델.
- **모범답변**: "select는 O(n) 폴링에 소켓 수 제한이 있고, epoll은 리눅스의 readiness 모델입니다. IOCP는 '완료' 모델이라 WSARecv를 미리 걸어두면 커널이 완료 시점에 워커를 깨워줍니다. 제 서버(`Server/Private/Network/IOCPCore.cpp`)는 워커 4개가 수신 완료를 처리해 패킷을 프레이밍(`FrameParser.cpp`)·큐잉하고, 게임 틱 스레드 1개가 커맨드를 소비해 시뮬레이션합니다. 이렇게 나눈 이유는 게임 로직을 단일 스레드로 유지해 시뮬레이션 결정성과 락 최소화를 지키기 위해서입니다."
- **꼬리질문 대비**: "워커에서 게임 상태 직접 만지면 안 되나?" → 락 지옥+결정성 파괴. 완료→큐→틱 소비의 단방향 흐름이 정답이라고 설계 근거 답변. "buffer 수명은?" → OVERLAPPED와 수신 버퍼는 세션이 소유, 완료 전 해제 금지(세션 종료 시 outstanding I/O 드레인) 이슈 인지를 보여줄 것.

### Q. Fiber를 도입했다는데, 스레드와 뭐가 다르고 왜 필요한가요?
- **정의**: Fiber는 유저 모드에서 스택을 교체하는 협력적(cooperative) 실행 단위. 커널 스케줄러를 거치지 않아 전환 비용이 스레드 컨텍스트 스위치(~1μs급)보다 훨씬 작다(~수십 ns급).
- **모범답변**: "JobSystem에서 잡이 다른 잡의 완료를 기다릴 때, 스레드를 블록하면 워커가 죽고, 스핀하면 CPU를 태웁니다. Fiber면 대기 시점에 스택째 잠들고 워커는 다른 잡을 집으면 됩니다. Naughty Dog GDC 발표로 알려진 구조고, 제 엔진의 JobSystem(`Engine/Private/Core/JobSystem.cpp`)에 Fiber 실행 셸을 배선했습니다. 서버 통합에서는 IOCP 워커까지 fiber화하지 않고 '완료→큐→틱 fiber 소비'로 경계를 그었는데, I/O 스레드와 시뮬레이션의 결정성을 분리하기 위해서입니다."
- **함정 대비**: "fiber에서 mutex 잡고 yield하면?" → 같은 락을 다른 fiber가 같은 스레드에서 기다리는 데드락/소유권 위반 — 락 보유 중 yield 금지 규칙과 TLS 사용 주의(fiber가 스레드를 옮기면 TLS가 배신)까지 답하면 상급.

### Q. work-stealing JobSystem 구조와, 거기서 겪은 race를 설명해보세요.
- **정의**: 워커마다 Chase-Lev deque를 두고, 자기 큐는 owner가 LIFO로 push/pop, 남의 큐는 thief가 FIFO로 steal하는 부하 분산 구조.
- **모범답변**: "구현 후 특정 엔티티 수(36 루트)에서 메인 스레드가 행에 걸렸습니다. 원인은 제가 Chase-Lev의 전제를 위반한 것 — 이 자료구조는 push가 owner 스레드 전용인데, 메인 스레드가 워커의 deque에 직접 push하고 있었습니다. 메모리 오더링이 깨져 워커가 잡을 못 보고, 카운터가 0이 안 되어 WaitForCounter가 무한 대기했습니다. 메인은 자기 전용 제출 경로로 넣고 워커가 steal해 가도록 수정했습니다. 교훈은 'lock-free 자료구조는 논문의 전제조건까지가 스펙'이라는 것입니다."
- **추가 탄약**: ECS 병렬화에서 워커가 다른 엔티티 컴포넌트를 쓰는 race는 Decision/Apply 2-pass(워커=읽기+슬롯 버퍼 기록, 메인=단일 스레드 reduce)로 구조적으로 제거했고, 메인=슬롯0/워커=idx+1 규칙으로 help-stealing 중 버퍼 충돌도 막았다고 이어서 설명 가능.
- **꼬리질문 대비**: "ABA 문제는?" / "memory_order 뭐 썼나" → Chase-Lev의 top CAS와 acquire/release 의미를 복습해둘 것.

### Q. 시뮬레이션 결정성은 왜 필요하고 어떻게 검증했나요?
- **정의**: 같은 입력 시퀀스가 항상 같은 상태를 만드는 성질. 리플레이, 서버 검증, 클라 예측 정합, 회귀 테스트의 기반.
- **모범답변**: "SimLab이라는 헤드리스 하니스(`Tools/SimLab/main.cpp`)를 만들어 고정 입력으로 300틱을 돌리고 최종 상태를 해시로 접습니다. 실용 효과가 컸는데, 에러 경계 리팩터링으로 108개 파일을 고친 뒤 해시가 리팩터링 전과 동일(BB6A67502987351F)함을 확인해 '동작 무변경'을 증명했습니다. diff 리뷰로는 불가능한 수준의 회귀 검증입니다. 부동소수점 결정성을 위해 서버 프로젝트에 /fp:precise를 강제한 것도 같은 맥락입니다."
- **꼬리질문 대비**: "크로스 플랫폼 결정성은?" → 단일 플랫폼 전제라 회피한 문제이며, 크로스면 고정소수점 또는 소프트 float가 필요하다고 한계 인지 답변.

### Q. FlatBuffers는 왜 선택했나요? Protobuf나 직접 직렬화와 비교하면?
- **원리**: Protobuf는 파싱(디코드) 단계가 필요하고, FlatBuffers는 버퍼를 그대로 랜덤 액세스(zero-copy read)한다. 스냅샷처럼 크고 자주 오는 데이터에 유리.
- **모범답변**: "초당 수십 회 오는 스냅샷을 매번 힙 할당+디코드하는 비용을 피하려고 FlatBuffers를 골랐습니다. 스키마 진화(필드 추가 하위호환)도 데이터 팩 해시 핸드셰이크 같은 프로토콜 확장에 활용했습니다. 직접 바이너리 직렬화 대비로는 버전 호환과 검증기(verifier)가 공짜라는 점이 큽니다. 다만 verify 실패를 조용히 버리면 스키마 드리프트가 '네트워크 멈춤'처럼 보이는 사고가 나서, 복제 경계의 verify 실패는 반드시 바운디드 로그를 남긴다는 정책을 세웠습니다."
- **꼬리질문 대비**: "verify 비용은?" → 신뢰 경계(외부 입력)에서만 verify, 내부 재사용 버퍼는 생략하는 기준을 답변.

### Q. '150챔피언 스케일 아키텍처'가 무슨 뜻인가요? 챔피언은 15종이라면서요.
- **정의**: 콘텐츠 n번째 추가 비용이 상수에 수렴하도록 만드는 구조 설계. 중앙 테이블/스위치문을 제거하고 self-registration으로 대체.
- **모범답변**: "구현된 챔피언은 15종이 맞고(`Shared/GameSim/Champions/` 폴더 15개), '150'은 목표 규모로 잡은 설계 제약입니다. 초기엔 챔피언 추가마다 중앙 테이블, 씬 코드, UI를 전부 수정해야 했습니다. 이를 Champion/Skill/SkillHook 3종 Registry의 self-register로 바꾸고, 스킬 훅을 4단계(키스왑/시전수락/시전프레임/후딜)로 분할하고, 밴픽 UI를 Registry 순회로 자동 생성하게 해서, 신규 챔피언 1종 = 폴더 1개+파일 6개+프로젝트 등록이면 되는 15~20분 파이프라인을 목표로 리팩터링했습니다. enum ID는 네트워크/DB 호환을 위해 명시값으로 박고 재사용 금지 규칙을 세웠습니다."
- **왜 이게 어필인가**: 라이브 게임 회사의 실제 고민(콘텐츠 추가 비용)을 신입이 스스로 정의하고 풀었다는 증거.
- **꼬리질문 대비**: "static 초기화 순서 문제는?" → self-register의 고전 함정(SIOF, 링커의 미참조 오브젝트 제거)을 인지하고 명시적 등록 호출 지점을 뒀는지 설명 준비.

### Q. MCTS AI를 게임에 어떻게 적용했나요?
- **정의**: Monte Carlo Tree Search — 선택(UCB)→확장→시뮬레이션(플레이아웃)→역전파를 반복해 행동 가치를 추정하는 탐색.
- **모범답변**: "봇 AI의 의사결정 계층으로 MCTS 시스템을 엔진 ECS에 두고(`Engine/Public/ECS/Systems/MCTSSystem.h`), 봇이 특권 코드가 아니라 플레이어와 같은 GameCommand를 생산하게 했습니다. MOBA에 MCTS를 쓰려면 상태 공간 축약이 관건이라, 행동 후보를 추상화된 선택지(교전/파밍/후퇴 류)로 줄이고 짧은 horizon의 시뮬레이션으로 평가하는 방향입니다. 여기서 시뮬레이션 결정성이 다시 중요해집니다 — 플레이아웃이 비결정적이면 가치 추정이 오염되니, 결정성 하니스가 AI 품질의 기반이기도 합니다."
- **정직성 주의**: 완성도를 과장하지 말 것. "탐색 프레임과 공통 가치평가 스켈레톤까지 구축, 플레이아웃 품질 개선이 진행 중"처럼 현 상태를 정확히 말하는 게 안전하다.
- **꼬리질문 대비**: "왜 행동트리(BT)가 아니라?" → BT는 반응적 규칙에, MCTS는 수읽기에 강함. 실제로는 BT로 기본기를 깔고 MCTS를 상위 결정에 얹는 하이브리드가 현실적이라고 답변.

### Q. FX 시스템을 직접 만들었는데, Niagara 같은 상용 툴과 비교해 설명해보세요.
- **모범답변**: "핵심 구조는 스프라이트/메시 이원화입니다. 스프라이트 계열은 아틀라스 애니메이션·페이드·UV 스크롤·블렌드 프리셋을 상수버퍼 하나로 제어하고, 메시 계열은 erode(알파 클립 침식) 같은 재질 파라미터를 별도 셰이더로 처리합니다. 스킬 코드와 이펙트를 분리하려고 FX 큐(cue) 재생기(`Client/Private/GameObject/FX/FxCuePlayer.cpp`)와 챔피언별 프리셋 카탈로그로 호출부를 데이터화했고, ImGui 이펙트 튜너로 런타임에서 색·타이밍을 조정해 코드 재빌드 없이 이터레이션했습니다. Niagara와의 차이는 그래프 기반 저작 툴의 유무인데, 노드 그래프 이펙트 툴은 설계 문서까지 완료된 다음 단계입니다. 그리고 이 계열 디버깅에서 배운 것 — '호출은 됐는데 안 보임' 류 버그는 코드 추론보다 UV/알파 데이터 계측과 RenderDoc이 먼저라는 원칙을 얻었습니다."
- **꼬리질문 대비**: "오버드로 관리?" → additive 이펙트 다중첩 시 fill-rate 비용, depth-write off/블렌드 상태 백업·복원을 실제로 다뤘음을 언급.

### Q. ECS를 채택한 이유와 OOP 대비 트레이드오프는?
- **모범답변**: "두 가지 이유였습니다. 첫째 데이터 지향 — 미니언 수백 개의 이동/AI는 컴포넌트 배열 순회가 캐시 친화적입니다. 둘째 서버-클라 코드 공유 — 시뮬레이션 상태를 POD 컴포넌트로 두면 스냅샷 직렬화와 서버/클라 공유(`Shared/GameSim/`)가 자연스럽습니다. 트레이드오프는 (1) 개별 개체의 복잡한 상태 머신(챔피언 스킬)은 순수 ECS보다 훅/레지스트리 하이브리드가 나았고 실제로 그렇게 갔습니다 (2) 시스템 간 실행 순서가 암묵 의존이 되기 쉬워서, 페이즈 규약과 접근 계약(exclusive 기본값)을 스케줄러에 명시했습니다. 상속 깊은 OOP였다면 서버 권위 분리(렌더 없는 서버 빌드)가 훨씬 고통스러웠을 겁니다."
- **꼬리질문 대비**: "아키타입 vs 스파스셋?" → 자기 구현이 어느 쪽인지, 반복 성능/추가삭제 성능 트레이드오프 복습해둘 것.

### Q. 혼자 개발했는데, 협업은 할 수 있나요?
- **원리**: 이 질문은 사실 확인이 아니라 태도 확인. 방어적으로 굴면 진다.
- **모범답변**: "팀 경험이 상대적으로 부족한 건 사실이라, '협업 가능한 상태'를 만드는 데 의식적으로 투자했습니다. 구체적으로 ①코딩 컨벤션과 아키텍처 규칙을 문서화하고(`.md/architecture/` 컨벤션·컴퍼스 문서) ②규칙을 사람이 아니라 기계가 지키게 — 계층 경계 위반 시 빌드가 실패하는 lint를 프리빌드에 걸었고(`Tools/Harness/Check-SharedBoundary.ps1`) ③외부 리뷰를 상시 받았습니다. AI 리뷰 에이전트에게 설계·구현 리뷰를 받고 지적 20건을 실측 검증 후 전수 반영한 사이클이 여러 번 있는데, 지적을 그대로 믿지 않고 코드로 검증한 뒤 선별 수용하는 습관은 사람 코드 리뷰에도 그대로 통한다고 생각합니다. ④인수인계 문서(`WINTERS_HANDOFF_GUIDE.md`)도 '내가 내일 팀에 합류시킬 사람'을 가정하고 썼습니다."
- **보강**: 팀 프로젝트(마크 던전스 — CONFIRM_NEEDED) 경험이 확인되면 그걸 먼저 말하고 위 내용을 후속으로.

### Q. 이 프로젝트에서 가장 어려웠던 버그는 무엇이었나요?
- **선택지 3개 (직군별)**:
  - **서버/동시성 지원**: Chase-Lev owner-push 위반 race (위 JobSystem 문항 참조). "재현이 엔티티 수에 의존하는 행이라 계측·최소재현 만들기부터가 일이었다"는 디버깅 과정 강조.
  - **클라/네트워크 지원**: yaw 예측 떨림 — "클라 내비게이션 시스템이 스냅샷 적용과 렌더 동기화 사이에서 서버 회전값을 덮어쓰는 3자 경합이었습니다. 서버 권위 게임에서 클라의 로컬 이동 시스템을 복제 챔피언에 돌리면 안 된다는 소유권 원칙으로 정리했고, 이후 움직임이 계단식이 되면 로컬 내비를 되살리는 게 아니라 보간을 고치는 게 맞다는 규칙까지 문서화했습니다."
  - **렌더/에셋 지원**: FX 텍스처 '보이지 않는 이펙트' — "render용 스프라이트 캡처 PNG를 머티리얼로 잘못 써서 UV가 알파 0 영역을 가리켜 전 픽셀이 clip으로 버려진 사례입니다. CPU 디버거로는 영원히 못 잡는 패턴이라, 이후 '셰이더부터 읽고 UV/알파 바운딩박스를 데이터로 계측'하는 GPU 디버깅 원칙을 세웠습니다."
- **공통 마무리**: 버그 자체보다 "그 뒤로 어떤 원칙/도구가 생겼나"로 끝낼 것.

### Q. 본인 프로젝트의 약점이나 기술 부채를 말해보세요.
- **원리**: 자기 객관화 검증. "없다"가 최악, 사소한 것만 대면 차악.
- **모범답변**: "직접 전수 감사를 두 번 했고 결과를 문서로 남겼습니다. 2026-07 에러 경계 감사에서 65건, HIGH 18건을 확정했는데 대표적으로 ①실패 경로가 무음이던 곳들 — FlatBuffers verify 실패가 bare return이라 스키마 드리프트가 네트워크 멈춤처럼 보이는 문제 ②std::future를 버려서 사실상 동기가 된 가짜 async HTTP 클라이언트 ③크래시 미니덤프 부재, CI 부재, 프로토콜 버전 핸드셰이크 부재였습니다. UE5.7 소스와 비교 감사했을 때의 메타 결론은 '규칙은 있는데 기계 강제가 없다'였고, 그래서 개선을 문서가 아니라 강제 장치(빌드 lint, 프리빌드 검사, 해시 핸드셰이크)로 넣는 방향으로 로드맵을 잡았습니다. 일부는 이미 적용했고(경계 lint, dataBuildHash 핸드셰이크), 미니덤프·CI가 다음 순서입니다."
- **효과**: 약점 질문을 감사 능력 어필로 반전시키는 답변.

### Q. 출시나 라이브 서비스 경험이 없는데 어떻게 보완했나요?
- **모범답변**: "실서비스는 없지만, 라이브에서 터지는 문제를 개발 단계 장치로 앞당겨 겪으려 했습니다. ①버전 드리프트 — 접속 시 서버·클라 데이터 빌드 해시를 교환해 불일치를 감지하는 핸드셰이크를 넣었고 ②관측 가능성 — 실패 경로 무음 금지 정책과 바운디드 로그 규칙을 세웠고 ③회귀 안전망 — 결정성 해시 스모크를 리팩터링마다 돌립니다. 또 게임 서버 밖 운영 계층(인증/매치메이킹/리더보드)을 Go 마이크로서비스로 만들어 게임과 플랫폼의 경계도 경험했습니다. 그래도 실트래픽·실유저 대응은 입사 후 배워야 할 영역이라고 인정합니다."

### Q. 포트폴리오 영상에는 뭘 담았나요? (또는: 지금 3분 안에 프로젝트를 보여준다면?)
- **모범답변**: "1분 데모는 '서버 권위가 진짜다'를 증명하는 데 씁니다 — 클라 2개+서버 콘솔 분할 화면에서 한 클라의 입력이 서버를 거쳐 복제되는 장면이 핵심 컷입니다. 이어서 F3 프로파일러 오버레이와 이펙트 튜너로 '만든 사람만 보여줄 수 있는 화면'을 넣었습니다. 기술 하이라이트는 코드와 실행 화면을 교차하며 제 목소리로 설명해, 대필 의심을 영상 단계에서 차단하는 구성입니다."
- **원리**: "영상 구성을 물었는데 기술 증명 전략으로 답한다" — 커뮤니케이션 설계 능력까지 같이 전달된다.

### Q. 클라이언트와 서버 중 왜 그 직군인가요? (지망 직군 답변)
- **원리**: 풀스택 포트폴리오의 역설 — "다 해봤다"는 "전문성이 없다"로 읽힐 수 있다. 1지망을 명확히 하고 나머지를 보조 역량으로 재배치. (1지망 확정 — CONFIRM_NEEDED)
- **서버 지망 버전**: "클라부터 서버까지 다 만들어보고 나서, 제가 가장 오래 몰입한 문제가 권위·결정성·동시성이었습니다. 예측/보간 같은 클라 쪽 경험은 서버 프로그래머로서 '클라가 뭘 필요로 하는지 아는' 자산이 된다고 생각합니다."
- **클라 지망 버전**: "렌더 파이프라인과 FX, 최적화에서 결과가 눈에 보이는 순간이 제일 즐거웠습니다. 서버를 직접 만들어본 경험은 넷코드 붙은 클라 기능(예측, 스냅샷 적용)을 서버 관점까지 이해하고 구현하는 자산입니다."

### Q. 이력서에 수치가 많은데, 그 수치들 어떻게 믿죠?
- **모범답변**: "전부 재현 경로가 있습니다. 17.8ms→9ms는 인게임 프로파일러 오버레이로 시연 가능하고, 27종 변환은 변환 배치 스크립트를 다시 돌리면 되고, 결정성 해시는 SimLab을 실행하면 나옵니다. 원하시면 지금 노트북으로 보여드리겠습니다." — 실제로 면접장에서 시연 가능한 상태를 유지하는 것 자체가 전략.
- **원리**: 수치의 신뢰는 '측정의 재현성'에서 온다. 이 답변이 가능하려면 GitHub/로컬 빌드가 항상 실행 가능해야 한다(면접 전날 체크리스트 항목).

### Q. 자체 엔진 경험이 상용 엔진(UE) 실무에 어떻게 도움이 되나요?
- **모범답변**: "매핑이 이미 머릿속에 있습니다. 제 RHI 추상화↔UE RHI, ECS 시뮬레이션↔게임플레이 프레임워크, .wmesh 쿠킹↔uasset 쿠킹, FX 시스템↔Niagara, 스냅샷 복제↔리플리케이션. 실제로 UE5.7 소스를 받아 7개 차원에서 제 엔진과 비교 감사를 했고, 제 툴을 UE 커스텀 에셋 에디터로 이식하는 설계 문서까지 썼습니다. 엔진 사용자가 아니라 '왜 이렇게 만들었는지 아는 사용자'로 시작할 수 있다는 게 자체 엔진 경험의 실무 가치라고 생각합니다."

### Q. Go로 백엔드도 만들었던데, 게임 서버와 뭐가 다른가요?
- **모범답변**: "실시간 시뮬레이션 서버와 플랫폼 서버는 요구가 다릅니다. 게임 서버는 상태ful·틱 기반·지연 민감이라 C++/IOCP로, 인증·매치메이킹·리더보드·프로필 같은 플랫폼 기능은 상태less·요청-응답·수평 확장이 중요해 Go HTTP 서비스(`Services/` — PostgreSQL/Redis/JWT)로 분리했습니다. 이 경계를 직접 그어본 게 수확입니다 — 예를 들어 매치메이킹 결과로 게임 룸에 입장하는 토큰 흐름처럼 두 세계가 만나는 지점을 설계해봤습니다."
- **꼬리질문 대비**: "왜 Go?" → goroutine 기반 동시성으로 I/O bound 서비스 생산성이 높고, 게임업계 플랫폼 서버에서 실제 채택이 많다는 것.

### Q. 개발 중 AI 도구를 썼다고 했는데, 그럼 이 코드는 본인 실력인가요?
- **원리**: 2026년 채용의 신규 단골 질문. 숨기면 위험, 당당하되 검증 프로세스를 보여야 한다.
- **모범답변**: "네, 적극적으로 썼고 그걸 엔지니어링 프로세스로 만들었습니다. 설계 결정과 아키텍처 규칙은 제가 세우고 문서로 박제하며, AI 리뷰의 지적은 그대로 수용하지 않고 코드 실측으로 검증한 뒤 선별 반영합니다 — 리뷰 지적 20건을 전수 검증해 반영한 사이클, 계획서의 불확실 지점을 CONFIRM_NEEDED로 표시해 검증 전 구현을 막는 규칙 같은 게 그 예입니다. 결과적으로 모든 코드의 '왜'를 제가 설명할 수 있고, 그건 이 면접에서 얼마든지 검증하셔도 됩니다. AI를 쓰되 검증 책임을 지는 방식이 앞으로의 실무 표준이라고 생각합니다."

### Q. 이 프로젝트를 지금 다시 시작한다면 뭘 다르게 하겠어요?
- **모범답변**: "세 가지입니다. ①포맷·프로토콜에 버전 필드를 첫날부터 — .wmesh에 버전이 없어 진화가 막히는 걸 나중에 감사로 발견했습니다. ②CI를 첫 주에 — 수동 빌드 검증은 규모가 커지며 반드시 구멍이 납니다. ③계층 경계를 문서가 아니라 빌드 강제로 먼저 — Shared가 Engine 헤더를 직접 include하는 위반이 80파일까지 퍼진 뒤에야 어댑터+lint로 절단했는데, 처음부터 include 경로 화이트리스트였으면 발생 자체가 없었습니다. 공통 교훈은 '규칙은 만들 때 강제 장치도 같이 만든다'입니다."
- **원리**: 회고 능력 = 성장 가능성 신호. 전부 실제 감사에서 나온 항목이라 후속 질문에도 무너지지 않는다.

### Q. 미니언 수백 마리의 AI와 이동은 어떻게 처리했나요?
- **모범답변**: "미니언은 감지→추격→공격→사망의 상태 기계를 ECS 시스템으로 돌리고, 공격은 선딜(windup)/타격/후딜 프레임 모델입니다. 병렬화가 문제였는데, 워커가 다른 엔티티를 직접 쓰면 race라서 Decision(워커, 읽기 전용+슬롯 버퍼)/Apply(메인, 일괄 반영) 2-pass로 구조화했습니다. 길찾기는 A*이고 자체 카운터로 방문 노드·재탐색 횟수를 상시 계측하며, 서버 쪽 웨이브 이동은 플로우필드 기반 경로(`Server/Private/Game/ServerMinionFlowField.cpp`)를 씁니다. 길찾기 실패가 무음이면 미니언이 제자리 걸음하는 유령 버그가 되기 때문에, 실패 사유 enum(시작점 막힘/목표 막힘/경로 없음 등)을 시스템 밖으로 노출해 디버깅 가능하게 만들었습니다(`Engine/Private/Manager/Navigation/Pathfinder.cpp`)."

### Q. 스킬 시스템 구조를 설명해보세요. 챔피언마다 스킬이 다 다른데 어떻게 일반화했나요?
- **모범답변**: "공통 골격(쿨다운/랭크/자원/시전 프레임)은 데이터로, 챔피언 고유 로직은 훅으로 분리했습니다. 훅은 시전 시점별로 4개(키 입력 스왑/시전 수락/시전 프레임/후딜)로 쪼갰는데, 처음에 하나로 뭉쳤더니 '리븐 Q 스택 갱신'과 '이즈리얼 E 순간이동'처럼 발동 시점이 다른 로직이 한 훅에 섞이는 문제가 났기 때문입니다. 판정은 공통 데미지 파이프라인(DamageRequest→ApplyDamage)으로 수렴시켜 씬 코드 의존을 제거했고, 투사체도 챔피언별 enum이 아니라 일반 투사체 종류로 통합했습니다. 이 구조 덕에 챔피언 15종이 같은 파이프라인 위에 있고, 새 챔피언은 폴더 하나로 추가됩니다."

### Q. 재접속(reconnect)은 어떻게 처리하나요?
- **모범답변**: "세션과 플레이어 정체성을 분리하는 게 핵심입니다. 소켓이 끊겨도 서버의 게임 내 엔티티와 로비 권위 상태는 유지되고, 재접속 시 정체성을 다시 바인딩한 뒤 현재 상태의 전체 스냅샷으로 클라를 복구시키는 뼈대를 만들었습니다(`Server/Private/Game/LobbyAuthority.cpp`, `SessionBinding.cpp`). 상태 전체가 서버에 있는 권위 구조라서 가능한 설계입니다 — 클라가 상태를 소유했다면 재접속 복구는 훨씬 어려웠을 겁니다."
- **꼬리질문 대비**: "재접속 중 그 플레이어 챔피언은?" → 현재 정책(방치/AI 대체 등)을 솔직히 말하고, 라이브 게임들의 처리(LoL의 /remake 등)와 비교할 준비.

### Q. 리플레이 시스템은 왜 만들었고 어떤 구조인가요?
- **모범답변**: "권위 서버가 이미 모든 상태 스냅샷을 만들고 있으니, 그걸 기록하면 리플레이가 거의 공짜라는 판단이었습니다(`Server/Private/Game/ReplayRecorder.cpp`, `Shared/Replay/`). 구조는 스냅샷 스트림 컨테이너를 저장하고 오프라인 재생하는 방식이고, 용도는 세 가지 — 버그 재현(비결정 이슈를 파일로 고정), 관전, 포트폴리오 데모입니다. 결정성 시뮬레이션과 묶으면 '입력만 기록'하는 초경량 리플레이도 가능한데, 스냅샷 방식은 용량을 내주는 대신 임의 시점 탐색이 쉽다는 트레이드오프로 이해하고 있습니다."

### Q. 마지막으로 하고 싶은 말 있나요? (역질문 포함)
- **원리**: 마무리 인상 결정. 준비된 역질문 2~3개는 지원 진정성의 증거.
- **역질문 예시**: "신입 프로그래머가 첫 6개월에 맡는 업무 범위와, 코드 리뷰 문화가 어떻게 되는지 궁금합니다." / "라이브 서비스에서 최근 겪은 기술적 도전 중 신입이 기여할 수 있는 영역이 있다면 어떤 것인가요?" / (서버 직군) "서버 아키텍처에서 권위/검증 경계를 어디까지 두시는지, 제 프로젝트 설계와 비교해보고 싶습니다."
- **마무리 멘트 골격**: "혼자서 엔진-게임-서버를 완주하며 배운 가장 큰 것은 기술이 아니라 '측정하고 검증하는 습관'이었습니다. 팀에서는 그 습관에 리뷰와 협업을 더해 더 빨리 성장하겠습니다."

## 내 프로젝트 연결 포인트

면접·자소서 어디서든 꺼내 쓸 수 있는 "카테고리 → Winters 증거" 접합 문장들.

- **측정 문화**: "추측으로 고치지 않습니다. 프레임 문제에서 프로파일러를 먼저 만들어 병목(스키닝 16ms/프레임 90%)을 확정한 뒤 고쳤고, 17.8ms→9ms로 복구했습니다."
- **검증 문화**: "리팩터링의 안전은 diff 리뷰가 아니라 해시로 증명합니다 — 108파일 변경 후 300틱 시뮬레이션 해시가 이전과 동일함을 확인했습니다(`Tools/SimLab/`)."
- **네트워크 권위**: "치팅 방지의 출발은 '클라에 판정 코드가 없는 것'입니다. 제 클라이언트는 데미지를 계산하지 않습니다 — 서버 GameSim이 판정하고 스냅샷/이벤트로 통보합니다."
- **동시성 실전**: "lock-free는 논문의 전제조건까지가 스펙이라는 걸 Chase-Lev owner-push 규약 위반 race로 배웠습니다."
- **스케일 설계**: "콘텐츠 추가 비용을 상수로 만드는 설계를 해봤습니다 — 챔피언 추가가 중앙 테이블 수정 없이 폴더 하나로 끝나는 Registry 구조입니다."
- **파이프라인/툴**: "아티스트·기획 없이 혼자였기에 오히려 툴을 먼저 만들었습니다 — 에셋 변환기(27종 전수), ImGui 이펙트 튜너, 맵 에디터, 인게임 프로파일러."
- **자기 객관화**: "제 코드의 약점 목록은 제가 제일 깁니다 — 전수 감사 2회, HIGH 18건, 개선 로드맵과 적용 이력이 문서로 있습니다."
- **규칙의 기계 강제**: "좋은 규칙은 문서가 아니라 빌드가 지킵니다 — 계층 경계 위반은 프리빌드 lint가 빌드를 실패시킵니다."
- **상용 엔진 연결**: "자체 엔진은 UE를 소스로 읽기 위한 훈련이었습니다 — 실제로 UE5.7 소스와 7개 차원 비교 감사를 수행했습니다."
- **다른 장르 확장**: "MOBA 외에 액션 장르 클라이언트/에디터(`EldenRingClient/`, `EldenRingEditor/` — Undo 트랜잭션 포함)도 같은 엔진 위에 올려 엔진의 범용성을 검증했습니다."
- **이전 프로젝트 연결(확인 후 사용)**: "IOCP는 팀 프로젝트(마크 던전스 모작)에서 처음 썼고, Winters에서 워커/틱 분리 구조로 재설계했습니다." (세부 CONFIRM_NEEDED)
