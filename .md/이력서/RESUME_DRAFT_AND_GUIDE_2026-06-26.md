## 2. 포트폴리오 — 프로젝트 상세 (포트폴리오 본체)

> 포트폴리오 중심 전략의 핵심. 각 프로젝트를 **문제 → 접근 → 검증 → 회고** 4박자로. Winters는 도메인 하이라이트 5개로 깊게, 나머지 3개는 1박자씩.

### 2-1. Winters Engine (메인 무기)

#### 메타 서사 (포트폴리오 첫 화면 — 가장 중요)

> **문제**: 동료들은 학원에서 배운 엔진(단일 스레드, DX11 고정, 챔프별 하드코딩)을 *전제*로 게임을 만들었다. 나는 그 엔진이 60fps 경계에서 슬로모션 스터터를 내고 멀티코어를 못 쓰는 것을 보고, *"엔진 자체가 문제 아닌가?"*를 의심했다. Unreal/Unity 같은 회사들이 어떻게 최신 기술로 자체 엔진을 만드는지를 질문의 출발점으로 삼았다.
>
> **접근**: 기능을 더 얹기 전에, (1) 무엇이 느린지 측정할 프로파일러를 먼저 만들고, (2) DX12/Vulkan의 명시적 모델을 기준으로 RHI를 추상화하고, (3) 데이터 접근 기반 병렬 스케줄러를 가진 ECS와 서버권위 결정론 시뮬레이션을 직접 설계했다.
>
> **검증**: 자체 프로파일러 JSON 캡처, 결정론 골든 테스트(SimLab), 의존성 경계 audit 하니스를 게이트로 두고, 각 도메인이 "어디까지 동작하고 어디부터 계획인지"를 스스로 선을 그었다.
>
> **회고**: 측정 인프라가 오히려 *내 코드의 한계*(병렬 배치가 실제로는 한 번도 발화하지 않음)를 드러냈다. 나는 그것을 숨기지 않고 다음 최적화 단계의 정의로 삼는다. 자체 엔진은 "다 만들었다"가 아니라 "어디까지 측정하고 검증했나"가 핵심이라고 본다.

#### 도메인 하이라이트 5 (각각 4박자, 정직한 경계 포함)

**① 측정·검증 인프라 ⭐ (가장 강한 차별화)**
- 문제: 측정 없이 최적화하면 성공/실패 판정이 불가능하다. 프로파일러 카운터가 DLL 경계에서 중복되고, GPU 시간을 못 봐 CPU/GPU 바운드 판별이 안 됐다.
- 접근: thread-local scope stack 계층형 CPU 프로파일러 + DX11 disjoint/timestamp 4슬롯 링 GPU 타이머 + 버전드 JSON 캡처. DLL 카운터 중복 버그를 strcmp 병합으로, 60fps 슬로모션 스터터를 dt 클램프 재설계로 해결.
- 검증: F4 JSON 골든 캡처(트렁케이션 플래그 포함), Debug/Release 양쪽 상시 계측. "작은 씬임에도 완전 CPU 바운드(드로우콜 ~94)"를 측정으로 확정.
- 회고: 측정이 ECS 병렬 스케줄러가 실게임에서 발화하지 않음(MaxBatchSize=1)을 드러냈다 → 다음 최적화 타깃을 데이터로 정의.

**② 멀티백엔드 RHI (DX11 production + DX12)**
- 문제: DX11 concrete 타입이 렌더 코드 30~40곳에 결합돼 백엔드 추가 시 전체를 헤집어야 했고, 엘든링급 오픈월드는 DX11 단일스레드 command 병목이 명확했다.
- 접근: DX12/Vulkan의 명시적 모델(PSO·BindGroup·explicit barrier·frame-in-flight) 기준으로 `IRHIDevice` 추상화, DX11을 emulation 백엔드로. DLL CRT 충돌을 피하려 자원을 64bit handle(index+generation)로 반환하는 `CRHIResourceTable`(use-after-free를 generation mismatch로 차단).
- 검증: `--rhi-scene-only` 플래그로 legacy draw를 끄고 RHI 스냅샷만 렌더해 비교하는 parity 게이트 + runtime smoke 하니스.
- 회고(정직한 경계): DX12는 scene-parity 검증 트랙이고 production은 DX11. Vulkan/Console은 경계 설계만.

**③ 서버권위 결정론 MOBA 시뮬레이션**
- 문제: 게임플레이가 클라 Scene에 묶여 서버가 같은 코드를 못 돌렸고, PvP는 서버권위·결정론이 필수다.
- 접근: IOCP 위에 ECS 시뮬 코어를 올려 30Hz 고정-dt 권위 루프. Input→GameCommand→서버 GameSim→Snapshot/Event→Client 시각까지 닫고, 명령 issuer를 sessionId→entity로 서버 결정(스푸핑 차단), DeterministicRng/정렬 순회로 결정성 강제.
- 검증: SimLab 헤드리스 러너가 동일 seed의 per-tick 상태 해시 일치 + seed 민감도를 이중 검증(exit 0/1).
- 회고(정직한 경계): 넥서스 승패·매치 종료는 미구현(1판 완결은 다음 단계), 랙보정은 기록만/적용 미연결.

**④ 챔피언 스킬 시스템 (GAS 통찰의 ECS 재해석)**
- 문제: 150챔프×4스킬을 if-체인으로 짜면 유지불가, 동시에 데미지/쿨다운/CC를 클라가 확정하면 안 된다.
- 접근: UE5 GAS의 통찰만 차용(1급 Tag/Effect 객체는 의도적으로 안 만듦). champ×variant 함수포인터 디스패치 테이블 + 단일 캐스트 게이트로 cooldown/rank/CC/타겟·사거리를 서버에서만 확정, 스킬 파라미터를 JSON 데이터팩으로 분리.
- 검증: 15챔피언 QWER 서버 배선(Viego 빙의·Sylas 궁 탈취 포함), SimLab 결정론 골든.
- 회고(정직한 경계): 완성도 편차(Zed/Yasuo/Viego 풀, Riven 절반, MasterYi 스텁), 데이터 cutover에 fallback 상수 잔존.

**⑤ 자체 바이너리 에셋 파이프라인**
- 문제: FBX 런타임 파싱이 챔프 1체당 50~200ms, 버전관리·런타임 슬림화 불가.
- 접근: 16B 공통 헤더 기반 .wmesh/.wskel/.wanim/.wmat 포맷 + Assimp 컨버터 CLI + zero-copy 로더. skel_hash·본카운트 교차검증, 불일치 시 Assimp fallback, 정점 stride를 입력 레이아웃과 static_assert로 ABI 계약화.
- 검증: info 헤더 게이트 + F5 인게임 Idle/Run 재생 체크리스트, 디스크 산출물 바이트가 writer 코드와 일치.
- 회고(정직한 경계): 무결성(SHA256)·압축(LZ4)·.wtex/번들은 설계만.

> 추가로 어필 가능한(2박자) 도메인: **데이터주도 ECS/정의 파이프라인**(JSON→codegen 불변 pack), **work-stealing JobSystem**, **Niagara식 WFX 데이터주도 FX**, **DX11 배치 UI 렌더러+Atlas 매니페스트**, **Go 백엔드 6서비스+WinHTTP 실연동**, **다중머신 협업 검증 하니스**. 지원 직무에 맞춰 5+α를 골라 배치.

#### 직무별 강조 스위치 (같은 프로젝트, 다른 1순위)

- **엔진/시스템**: ①측정인프라 → ②RHI → ECS/스케줄러 → 에셋포맷
- **서버/네트워크**: ③서버권위 → IOCP/스냅샷 → 결정론/SimLab → Go 백엔드
- **그래픽스**: ②RHI → SSAO/PBR/스키닝 → ⑤에셋포맷 → WFX FX
- **게임플레이/클라**: ④챔피언스킬 → AI 봇 → UI 파이프라인 → FX

### 2-2. Liberation (출시·수익 — "끝까지 가본 사람" 증명)
- 문제: 만드는 것과 *출시하는 것*은 다르다. 빌드·스토어 심사·가격·정산·유저 피드백까지 가본 적이 없었다.
- 접근: UE5 Blueprint로 공포 게임을 `[[기간]]` 동안 `[[단독/팀 역할]]` 개발, Steam 출시 파이프라인(빌드 패키징·스토어 페이지·심사·패치)을 직접 수행.
- 검증: **실제 Steam 출시 + 매출 $91 발생** `[[판매 N건]]`, 유저 리뷰 `[[수/평가]]`.
- 회고: 기술 완성도 ≠ 제품 완성도. 출시는 마지막 10%가 90%의 노력임을 체감 → 이후 Winters에서 "검증 게이트·정직한 구현상태 표"를 습관화한 계기.

### 2-3. SR_MinecraftDungeons (팀 협업 — "혼자만의 코드 아님" 증명)
- 문제: 자체 엔진은 "팀에서 남의 코드와 협업되나?"라는 의심을 부른다. 그 전에 팀 작업을 경험했다.
- 접근: 학원 SR 프레임워크 위에 Client/Engine 도메인을 분리하고 서버를 얹어, `[[N인 팀]]`에서 `[[내 담당]]`을 맡아 `[[협업 도구]]`로 통합.
- 검증: `[[완성/제출/시연 결과]]`.
- 회고: 도메인 경계와 소유권 분담의 중요성 → Winters의 Client/Engine/Shared 의존성 규칙·ownership matrix·검증 하니스로 발전.

## 4. 기술 블로그 전략 (velog 권장)

**결론: 쓴다. 플랫폼은 velog 1순위(개발 커뮤니티 노출·마크다운·코드블록). 잡글 50개보다 딥다이브 5~8개.**

각 글 = **문제 → 접근 → 검증 → 회고** + 코드/측정 스크린샷. 글 끝에 "면접에서 이걸 물어보세요" 식 자신감.

### 글감 (우선순위 순, 전부 정직성 지도 근거)
1. **"단일 스레드 학습용 엔진이 왜 한계인가 — 프로파일러를 직접 만들어 측정해봤다"** (★1번 글. 당신 서사의 출발점 + 측정주도 개발 증명. dt 클램프 슬로모션·CPU 바운드 진단·DLL 카운터 버그를 실측 스크린샷으로.)
2. "DX11 위에 RHI 추상화를 올리며: 왜 DX12의 명시적 모델을 기준으로 설계했나" (handle/generation, DLL CRT 충돌 회피)
3. "서버권위 MOBA를 직접 짜기: Client Input→GameCommand→Snapshot 루프와 결정론" (SimLab 골든 테스트로 결정성을 어떻게 자동 검증했나)
4. "Chase-Lev work-stealing JobSystem을 만들고, 측정이 내 병렬화가 안 돈다는 걸 알려줬다" (정직한 회고형 — 가장 인상적)
5. "LoL 챔피언 스킬을 데이터 주도로: hard-code 600개 if에서 JSON 정의팩으로" (GAS 통찰의 ECS 재해석)
6. "자체 바이너리 에셋 포맷(.wmesh): FBX 런타임 파싱을 zero-copy 로드로 바꾸기"
7. "EldenRing 원본 에셋을 자체 엔진으로: WitchyBND→Blender→자체 컨버터 5000줄 파이프라인" (데이터주도 복원, 저작권 경계 포함)
8. "스팀에 공포게임을 실제로 출시하며 배운 비기술적인 것들" (Liberation — 출시/심사/정산. 기술 외 차별화)

### 블로그 운영 팁
- 1번 글을 가장 공들여 쓰고 이력서/포트폴리오/README가 전부 1번을 가리키게.
- 글마다 GitHub 해당 코드 링크 + 측정 스크린샷(프로파일러 오버레이, SimLab exit code, JSON 캡처).
- "회고"에서 한계를 스스로 적는 글(4번 같은)이 역설적으로 신뢰를 가장 크게 올린다.
