# 가상 메모리와 메모리 관리

> "malloc하면 무슨 일이 일어나나요?", "페이지 폴트가 뭔가요?", "가상 메모리는 왜 필요한가요?"에 대한 답. 포인터가 들고 있는 주소가 사실 거짓말이라는 것, 그 거짓말을 하드웨어(MMU·TLB)와 커널(폴트 핸들러)이 합작해서 유지한다는 것, 그리고 그 위에 malloc과 게임의 커스텀 할당자가 어떻게 얹히는지를 관통한다.

---

## ① 한 줄 본질

> **"가상 메모리는 '모든 프로세스에게 자기만의 연속된 주소 공간을 가진 것처럼 속이는' 시스템이다. CPU의 MMU가 페이지(4KB) 단위로 가상 주소를 물리 주소로 변환하고, 변환 정보가 없으면 페이지 폴트를 일으켜 커널이 그 자리에서 채워 넣는다. 이 '접근하는 순간에만 실체를 만든다'는 지연(lazy) 원칙 하나에서 보호·격리, 요구 페이징, copy-on-write, 과할당, 스택 자동 성장, 메모리 매핑 파일이 전부 파생된다. malloc은 시스템 콜이 아니라 이 페이지 시스템(VirtualAlloc/mmap) 위에 얹힌 유저 공간 라이브러리이고, 게임이 풀/아레나 할당자를 쓰는 것은 그 라이브러리의 비용(락·단편화·비결정성)마저 프레임 예산에서 밀어내기 위해서다."**

면접에서 이 챕터의 질문들은 전부 하나로 환원된다: **"주소에 간접(indirection) 한 층을 넣으면 무엇을 살 수 있고, 무슨 비용이 드는가."** 사는 것 = 격리·재배치·지연·공유. 비용 = 변환 오버헤드(그래서 TLB), 테이블 메모리(그래서 다단계 트리), 예측 불가능한 폴트 지연(그래서 게임은 폴트를 로딩 시점으로 몰아넣는다).

## ② 기본 개념

### 물리 주소를 직접 쓰면 무엇이 무너지는가 — 가상 메모리의 4가지 존재 이유

역사적 순서로 보면 가상 메모리는 "메모리가 부족해서" 만든 게 아니라 **"여러 프로그램을 한 기계에서 동시에, 서로 망가뜨리지 않고 돌리기 위해"** 만들어졌다. 물리 주소를 직접 쓰던 시절의 문제와 해법을 짝지으면:

| 물리 주소 직접 사용의 문제 | 가상 메모리의 해법 |
|---|---|
| 프로세스 A의 버그가 B의 메모리를 덮어씀 | **보호/격리(protection/isolation)** — 프로세스마다 독립된 페이지 테이블 |
| 프로그램을 어느 물리 주소에 올릴지 로드마다 달라짐 | **재배치(relocation)** — 모두가 같은 가상 배치를 믿고, 물리 배치는 테이블이 흡수 |
| 물리 RAM 총량 = 실행 가능한 프로그램 총량 | **과할당(overcommit)** — 접근 전까지 물리 페이지를 안 주므로 RAM보다 큰 약속 가능 |
| 같은 라이브러리를 프로세스 수만큼 복사 | **공유(sharing)** — 같은 물리 페이지를 여러 테이블이 가리켜 시스템 전체에 한 벌 |

각각을 한 단계 더 파면:

- **격리의 강도**: 다른 프로세스의 메모리는 "접근하면 느린" 것도 "권한 검사에 걸리는" 것도 아니다. 내 페이지 테이블에 **그 물리 페이지로 가는 매핑 자체가 없다.** 주소를 아무리 조작해도 내 주소 공간 안을 도는 것뿐이다. 이게 프로세스 격리가 스레드 격리(같은 주소 공간, 규율만 존재)와 근본적으로 다른 이유이고, 브라우저가 탭을 프로세스로 쪼개는 이유다.
- **재배치의 결과**: 모든 프로세스에서 스택은 높은 주소, 코드는 낮은 주소 같은 표준 배치를 쓸 수 있다. 링커는 물리 메모리 사정을 몰라도 되고, 로더는 빈 물리 프레임 아무 데나 페이지를 흩뿌려도 된다 — 가상 공간의 "연속"은 물리의 연속을 요구하지 않는다.
- **과할당의 본질**: "할당"이라는 말이 장부 기록(가상)과 실물 지급(물리)으로 분리된다. 이 분리가 아래 나올 demand paging·lazy allocation·COW의 공통 뿌리다.
- **공유의 조건**: 공유가 안전하려면 읽기 전용이거나, 쓰는 순간 사본이 생겨야 한다(COW). 그래서 공유와 보호 비트는 한 세트다.

핵심 통찰: 네 가지가 별개 기능이 아니라 **"가상 → 물리 매핑을 페이지 단위 간접 테이블로 관리한다"는 단일 설계의 네 가지 결과**다. 대가는 모든 메모리 접근에 변환이 낀다는 것 — 그래서 TLB가 필수가 된다.

### 프로세스 주소 공간의 지도 — 이 챕터 전체의 무대

뒤 절들을 배치할 무대를 먼저 그린다. x64 Windows 유저 프로세스의 전형적 그림:

```
0xFFFF800000000000 ~   ┌────────────────────────────┐
                       │ 커널 공간 (모든 프로세스 공통, │  ← U/S 비트로 유저 접근 차단.
                       │  페이지 테이블 상위 절반 공유)  │     시스템 콜은 '이사'가 아니라 '모드 전환'
0x00007FFF...          ├────────────────────────────┤
                       │ DLL들 (매핑 파일, ASLR 배치)  │  ← 코드 페이지는 프로세스 간 공유
                       │ 스레드 스택들 (예약↓, 가드)    │  ← 아래로 성장
                       │        …(거대한 빈 구멍)…     │  ← 트리 상위 'not present'로 공짜 표현
                       │ 힙 (malloc이 관리하는 커밋 영역)│  ← 필요 시 위로 확장
                       │ .data/.bss (전역, COW/제로필) │
                       │ .text (exe 코드, RO+X 매핑)   │
0x0000000000000000     └── 널 페이지 (의도적 미매핑) ──┘  ← 널 역참조가 폴트로 잡히는 이유
```

읽어 낼 것 세 가지:

- **널 포인터가 크래시 나는 것도 설계다.** 주소 0 근처를 일부러 매핑하지 않아서, `nullptr` 역참조가 protection fault로 즉시 잡힌다. 하드웨어에 "널 검사" 기능이 있는 게 아니라 **매핑을 안 만든 것뿐**이다.
- **커널이 모든 프로세스의 상위 절반에 '같이' 산다.** 시스템 콜 때 주소 공간을 통째로 바꾸면 TLB가 전멸하므로, 커널 매핑을 전 프로세스가 공유하고 U/S 비트로만 막는다. (이 가정을 흔든 것이 Meltdown 부류의 하드웨어 취약점이고, 커널/유저 테이블을 더 분리하는 완화가 도입됐다 — 세부는 과단정하지 말고 이 정도로.)
- **빈 구멍이 공짜라는 것.** 스택과 힙 사이 수십 TB의 미사용 공간은 페이지 테이블 트리에서 상위 엔트리 몇 개의 'not present'일 뿐이다. 64비트에서 주소 공간을 사치스럽게 쓰는 기법들(영역 예약, 레드존)의 전제.

### 주소 변환의 실체 — MMU와 x86-64 4단계 페이지 테이블 워크

변환은 소프트웨어가 아니라 **CPU 안의 MMU(Memory Management Unit) 하드웨어**가 한다. 유저 코드의 `mov rax, [rbx]` 한 줄마다 커널이 개입한다면 게임은 존재할 수 없다. 커널의 역할은 테이블을 **만들어 두는 것**까지고, 평상시 변환은 하드웨어가 커널 개입 없이 수행한다. 이 역할 분담(정책 = 커널, 메커니즘 = 하드웨어)이 가상 메모리 전체의 성능을 지탱한다.

x86-64(4단계 페이징)의 가상 주소는 하위 48비트만 쓰고, 이렇게 쪼개진다:

```
63          48 47       39 38       30 29       21 20       12 11          0
┌─────────────┬───────────┬───────────┬───────────┬───────────┬────────────┐
│ 부호 확장     │ PML4 인덱스│ PDPT 인덱스│ PD 인덱스  │ PT 인덱스  │ 페이지 오프셋│
│ (canonical)  │   9비트    │   9비트    │   9비트    │   9비트    │   12비트    │
└─────────────┴───────────┴───────────┴───────────┴───────────┴────────────┘
```

숫자들이 서로 맞물려 있다:

- 오프셋 12비트 = 2^12 = **4KB 페이지** 안에서의 위치. 변환의 최소 단위가 페이지인 이유가 여기 있다 — 바이트 단위로 매핑하면 테이블이 메모리 자체보다 커진다.
- 인덱스 9비트 = 테이블 하나에 2^9 = **512개 엔트리**. 엔트리 하나가 8바이트라 테이블 하나가 정확히 4KB — **페이지 테이블 자체도 페이지 하나에 딱 들어가게** 설계됐다.
- 9×4 + 12 = 48비트 → 프로세스당 256TB 가상 공간. 상위 16비트는 47번 비트의 부호 확장이어야 하며(canonical 주소), 아니면 폴트다. 최신 CPU는 5단계 페이징(PML5)으로 57비트까지 확장 가능하지만 원리는 동일하다.

TLB 미스가 났을 때 MMU가 걷는 경로(페이지 테이블 워크):

```
CR3 레지스터 (현재 프로세스의 PML4 물리 주소)
 │
 ▼  47:39         38:30         29:21         20:12          11:0
PML4[i] ──▶ PDPT[i] ──▶ PD[i] ──▶ PT[i] ──▶ 물리 프레임 번호 ＋ 오프셋
(512개)      (512개)     (512개)     (512개)        │
                                                   ▼
                                              실제 물리 주소
```

구체 주소 하나로 손 계산을 보여줄 수 있으면 면접에서 강하다. `VA = 0x00007FFAC0011234`라면:

```
PML4 인덱스 = (VA >> 39) & 0x1FF = 0x0FF (255)
PDPT 인덱스 = (VA >> 30) & 0x1FF = 0x1EB (491)
PD   인덱스 = (VA >> 21) & 0x1FF = 0x000 (0)
PT   인덱스 = (VA >> 12) & 0x1FF = 0x011 (17)
오프셋      =  VA        & 0xFFF = 0x234
→ PML4[255] → PDPT[491] → PD[0] → PT[17] 이 가리키는 물리 프레임 + 0x234
```

**왜 평면 배열이 아니라 4단계 트리(radix tree)인가?** 48비트 공간을 4KB 페이지로 나누면 2^36개 페이지 — 평면 테이블이면 프로세스당 테이블만 512GB다. 그런데 실제 프로세스는 이 공간의 극히 일부만 쓴다(코드 몇 MB + 힙 + 스택, 사이는 전부 빈 구멍). 트리는 **비어 있는 영역의 상위 엔트리 하나만 'not present'로 두면 하위 테이블 512개를 통째로 안 만들어도 된다.** 즉 다단계 구조는 속도 최적화가 아니라 희소(sparse) 주소 공간을 표현하기 위한 **공간** 최적화다. 대가로 미스 한 번에 메모리 접근이 최대 4회 추가되는데, CPU는 상위 단계 엔트리를 캐싱하는 paging-structure cache로 이를 완화한다.

컨텍스트 스위치에서 "주소 공간 교체"의 실체는 **CR3에 다른 프로세스의 PML4 물리 주소를 쓰는 것 한 줄**이다. 테이블을 복사하는 게 아니라 트리의 루트 포인터만 바꾼다 — 격리 전환이 O(1)인 이유.

### PTE — 페이지 하나의 신분증

말단 테이블 엔트리(PTE, Page Table Entry) 8바이트에는 물리 프레임 번호와 함께 제어 비트들이 박혀 있다. 주요 비트와 **각 비트가 존재하는 이유**:

| 비트 | 이름 | 왜 존재하는가 |
|---|---|---|
| P (present) | 존재 | 0이면 접근 시 페이지 폴트. **이 비트 하나가 요구 페이징·스왑·lazy allocation 전체의 트리거** |
| R/W | 쓰기 허용 | 0인 페이지에 쓰기 → 폴트. 코드 페이지 보호와 copy-on-write의 재료 |
| U/S | 유저/수퍼바이저 | 0이면 커널 모드에서만 접근 가능. 유저 코드가 커널 메모리를 못 읽는 하드웨어적 근거 |
| A (accessed) | 접근됨 | MMU가 읽기/쓰기 시 자동으로 1 세팅. **커널이 "최근에 쓰인 페이지"를 알 수 있는 유일한 하드웨어 힌트** → 페이지 교체의 재료 |
| D (dirty) | 수정됨 | 쓰기 시 자동 세팅. 교체 시 "디스크에 다시 써야 하는가" 판단 — clean 페이지는 그냥 버려도 된다 |
| NX/XD (bit 63) | 실행 금지 | 1이면 이 페이지의 바이트를 명령으로 실행 불가. 스택/힙에 주입된 셸코드 실행을 하드웨어에서 차단 — Windows DEP의 실체 |

관통하는 원리: **P/RW/US는 커널이 정책을 선언하는 비트**(위반하면 폴트로 커널에게 통보되는 함정)이고, **A/D는 하드웨어가 커널에게 보고하는 비트**다. 이 양방향 채널 덕분에 커널은 모든 접근을 감시하지 않고도 메모리를 관리한다. "커널은 함정을 파 두고, 하드웨어는 발자국을 남긴다" — 이 한 문장이 폴트 기반 메커니즘 전부의 요약이다.

P=0인 엔트리의 나머지 63비트는 하드웨어가 해석하지 않으므로 커널이 마음대로 쓴다 — 스왑된 페이지가 디스크 어디에 있는지 같은 정보를 여기 숨긴다. "present가 아니면 나머지는 커널의 메모장"이라는 것도 좋은 디테일이다.

### TLB — 변환에도 캐시가 필요하다

매 메모리 접근마다 4단계 워크(메모리 접근 4회)를 한다면 메모리가 실질 5배 느려진다. 그래서 MMU 안에 **TLB(Translation Lookaside Buffer)** — "가상 페이지 번호 → 물리 프레임 번호 + 권한 비트" 변환 결과의 캐시 — 가 있다.

- **규모 감각**: L1 TLB는 수십 엔트리, L2 공유 TLB는 천 단위 엔트리 수준(마이크로아키텍처마다 다름). 4KB 페이지 기준 L1 TLB가 커버하는 메모리는 수백 KB 정도 — **랜덤 접근 워킹셋이 이걸 넘는 순간 TLB 미스가 일상이 된다.** 데이터 지향 설계가 "연속 배열"을 강조하는 숨은 이유 중 하나다.
- **미스 비용**: x86는 하드웨어 페이지 워커가 처리하므로 커널 진입은 없지만, 워크 자체가 추가 메모리 접근이다(최대 4회, paging-structure cache가 흡수 못 한 만큼). 데이터 캐시 미스와 겹치면 접근 한 번이 수백 사이클이 될 수 있다.
- **컨텍스트 스위치와의 관계**: CR3를 바꾸면 이전 프로세스의 변환 항목이 전부 무효다. 순진한 구현은 TLB 전체 플러시 → 스위치 직후 모든 접근이 미스. 이를 줄이려고 TLB 엔트리에 주소 공간 태그를 붙인 것이 x86의 **PCID(Process Context Identifier)**, ARM의 **ASID**다. 태그가 다르면 히트로 안 치므로 플러시 없이 여러 프로세스의 변환이 공존한다. "컨텍스트 스위치가 비싸다"의 진짜 비용 대부분이 레지스터 저장이 아니라 이 TLB·캐시 오염이라는 점은 스케줄링 챕터와 이어지는 핵심이다.
- **멀티코어의 골칫거리 — TLB shootdown**: 페이지 테이블은 메모리에 있지만 TLB는 코어마다 있다. 커널이 매핑을 회수/축소하면(unmap, 권한 강화) 다른 코어의 TLB에 남은 낡은 엔트리를 지워야 하는데, x86에는 원격 코어 TLB를 지우는 명령이 없어 **IPI(inter-processor interrupt)로 각 코어를 깨워 직접 지우게 한다.** munmap·COW 마킹이 스레드 많은 프로세스에서 비싸지는 이유이고, "매핑 추가는 싸고 회수는 비싸다"는 비대칭의 근원이다.

### 페이지 폴트 — 에러가 아니라 핵심 메커니즘이다

페이지 폴트(page fault)는 "MMU가 변환에 실패하거나 권한 위반을 감지해 CPU가 커널을 호출하는 예외"다. 중요한 건 **폴트가 크래시가 아니라 가상 메모리의 정상 동작 경로**라는 것. 세 종류로 분류된다:

| 종류 | 상황 | 디스크 I/O | 결과 |
|---|---|---|---|
| minor (soft) | 유효한 주소인데 PTE만 아직 없음 — 첫 터치(제로필), COW 쓰기, 이미 메모리(페이지 캐시/스탠바이)에 있는 페이지 재연결 | 없음 | 마이크로초 수준 해소, 유저 코드는 인지 못함 |
| major (hard) | 내용이 디스크에 있음 — 스왑 아웃된 페이지, 아직 안 읽은 매핑 파일/실행 파일 부분 | **있음** | 밀리초 수준 — 16.6ms 프레임 예산에서는 재앙. "인게임 스터터"의 흔한 정체 |
| protection (invalid) | 매핑 자체가 없거나 권한 위반 — 읽기 전용에 쓰기, NX 페이지 실행, 널 포인터 역참조 | — | 복구 불가 → Linux는 SIGSEGV, Windows는 Access Violation(0xC0000005) |

처리 경로:

```
mov rax, [rbx]        ← 변환 실패 or 권한 위반
  │
  ▼  CPU가 #PF 예외 발생: CR2 ← 폴트 주소, 에러코드(읽기/쓰기·유저/커널·P비트 여부) 전달
커널 폴트 핸들러
  │   프로세스의 메모리 영역 기술 구조(Windows VAD 트리 / Linux vm_area) 검색:
  │   "이 주소, 내가 약속해 준 영역인가? 어떤 권한으로?"
  ├── 유효 → 물리 페이지 확보(제로필 / 디스크 읽기 / COW 복사) → PTE 기록 → 복귀
  └── 무효/권한 위반 → 시그널/예외를 프로세스에 전달 (여기서 처음으로 '에러'가 된다)
  ▼
폴트 난 명령을 처음부터 재실행 — 유저 코드는 폴트가 있었는지조차 모른다
```

마지막 줄이 설계의 정수다. **"폴트 → 채움 → 같은 명령 재실행"이 완전히 투명하기 때문에**, 커널은 "약속(가상 주소 예약)"과 "이행(물리 페이지 배정)"을 마음대로 분리할 수 있다. 폴트 핸들러가 참조하는 것은 페이지 테이블이 아니라 그 위의 논리 장부(VAD/vm_area — "이 범위는 읽기 전용 파일 매핑", "이 범위는 커밋된 힙")라는 것도 짚어 둘 것: **페이지 테이블은 캐시된 결론이고, 진리의 원본은 커널의 영역 장부다.**

또 하나: 새로 배정되는 익명 페이지는 항상 **0으로 채워져** 온다. 편의가 아니라 보안 요구다 — 직전에 그 물리 페이지를 쓰던 다른 프로세스의 데이터(비밀번호, 키)가 새 프로세스에 노출되면 안 되기 때문. Windows가 유휴 시간에 빈 페이지를 미리 0으로 밀어 두는(zero page list) 이유이기도 하다.

### 요구 페이징과 lazy allocation — "할당"이라는 말의 해체

위 폴트 메커니즘에서 두 가지 표준 기법이 파생된다.

**요구 페이징(demand paging)**: 실행 파일을 시작할 때 통째로 읽지 않는다. 코드/데이터 페이지를 파일에 매핑만 해 두고, 실행 흐름이 실제로 닿는 페이지만 major fault로 읽어 온다. 수백 MB짜리 프로그램이 수십 ms에 "시작"되는 이유 — 사실은 아직 거의 안 읽은 것이다.

**lazy allocation**: 메모리 "할당"은 대개 장부 기록일 뿐이다.

```cpp
// 64MB를 커밋했지만, 이 시점에 물리 페이지는 거의 배정되지 않는다
char* p = static_cast<char*>(VirtualAlloc(nullptr, 64ull << 20,
                             MEM_RESERVE | MEM_COMMIT, PAGE_READWRITE));
p[0] = 1;   // 첫 터치 → minor fault → 이 4KB짜리 한 페이지만 물리 프레임 획득
```

Windows는 이 분리를 API로 노출한다 — `MEM_RESERVE`(주소 범위만 확보, 커밋 한도 소비 없음)와 `MEM_COMMIT`(커밋 한도 소비, 물리 배정은 여전히 첫 터치까지 지연)의 2단계. "예약 → 커밋 → 터치"의 3단 사다리에서 물리 메모리는 마지막 단에서만 나간다.

**과할당 정책의 OS별 차이** — 같은 메커니즘, 다른 철학:

| | Linux (기본 heuristic overcommit) | Windows (commit charge) |
|---|---|---|
| 약속 시점 | 물리+스왑을 초과해도 대개 성공 | 커밋 총량이 RAM+페이지파일 한도 안일 때만 성공 |
| 실패 시점 | 첫 터치 때 물리가 정말 바닥나면 **OOM killer**가 프로세스 선정 | `VirtualAlloc(MEM_COMMIT)`이 그 자리에서 실패 반환 |
| 결과 | malloc 성공 ≠ 사용 가능. 할당 지점이 아닌 접근 지점에서, 남의 프로세스가 죽을 수도 | 실패가 조기·국소적. 대신 실제로 안 쓸 메모리도 한도를 소비 |

"언제 거짓말이 들통나게 할 것인가"라는 정책 차이일 뿐, lazy 배정이라는 메커니즘은 동일하다.

### copy-on-write — "복사"를 "약속"으로 바꾸기

COW(copy-on-write)는 폴트 메커니즘의 가장 우아한 응용이다. 원리: **복사해야 할 페이지를 실제로 복사하지 않고, 양쪽 페이지 테이블이 같은 물리 페이지를 읽기 전용으로 가리키게 한다.** 어느 쪽이든 쓰기를 시도하면 R/W 위반 폴트가 나고, 핸들러가 장부를 보고 "이건 에러가 아니라 COW다"라고 판정한 뒤 그 순간에만 진짜 복사본을 만들어 쓴 쪽의 PTE를 바꿔 준다.

```
fork() 직후                          자식이 페이지 X에 쓰기 후
부모 PTE ──┐                         부모 PTE ──▶ [물리 X]  (R/W 복원)
           ├──▶ [물리 X] (RO 마킹)
자식 PTE ──┘                         자식 PTE ──▶ [물리 X'] (복사본, R/W)
```

- **POSIX fork가 실용적인 이유**: 프로세스 복제가 "페이지 테이블 복사 + 전 페이지 RO 마킹"으로 끝난다. 수 GB 프로세스도 밀리초에 fork되고, 실제 복사는 이후 쓰는 페이지만큼만 발생한다. Windows에는 fork가 없지만(CreateProcess는 새 이미지 로드) COW 자체는 파일 매핑과 아래 DLL 공유에서 똑같이 쓰인다.
- **DLL/공유 라이브러리 코드 페이지**: 코드는 애초에 읽기 전용이라 모든 프로세스가 영원히 한 벌을 공유한다. 쓰기가 필요한 페이지 — 전역 데이터, 그리고 로드 주소가 어긋나 재배치(relocation) 패치를 받은 페이지 — 만 COW로 프로세스 사유가 된다. **DLL이 어긋난 주소에 로드될수록 공유가 깨지고 메모리가 늘어나는** 이유. x64가 RIP 상대 주소 지정으로 재배치 필요 자체를 줄인 것도 이 공유를 지키는 방향의 진화다.
- **제로 페이지도 같은 패턴**: 커밋만 하고 읽기만 한 페이지는 시스템 공용 제로 페이지를 RO로 가리키게 해 둘 수 있고, 첫 쓰기에서야 사유 페이지를 받는다. "0으로 가득 찬 1GB"는 물리적으로 거의 공짜다.

관통 원리: **COW는 "복사를 없앤" 게 아니라 "쓰기 비율이 낮다는 데 베팅해 복사를 지연"한 것**이다. 베팅이 빗나가면(전부 쓰면) 복사 비용에 폴트 오버헤드만 얹힌다.

### 페이지 교체 — 진짜 LRU는 만들 수 없다

물리 메모리가 차면 누군가를 내보내야 한다. 이론 최적은 "가장 오래 안 쓸 페이지"(미래 지식 필요 — 불가능)고, 실용 목표는 LRU(least recently used)다. 그런데 **진짜 LRU는 못 만든다** — 매 접근마다 타임스탬프를 갱신하려면 모든 메모리 접근에 소프트웨어가 개입해야 하는데, 그 접근은 MMU가 커널 몰래 처리하기 때문이다. 커널이 가진 건 PTE의 accessed 비트 하나뿐.

그래서 **clock(second chance) 알고리즘** 같은 LRU 근사를 쓴다. 페이지들을 원형으로 놓고 시곗바늘이 돌면서:

```
     ┌─▶ [A=1] 최근 접근됨 → A를 0으로 지우고 한 번 봐줌(second chance), 다음으로
바늘 ─┤
     └─▶ [A=0] 바늘이 한 바퀴 도는 동안 아무도 안 씀 → 희생자로 선정
```

접근이 잦은 페이지는 바늘이 올 때마다 A=1이라 계속 살아남는다 — "정확한 접근 순서"는 포기하고 **"최근 한 바퀴 안에 쓰였는가"라는 1비트 정보만으로 LRU를 근사**한 것. dirty 페이지는 내보내기 전 디스크 기록이 필요하므로 clean 페이지가 우선 희생된다(D 비트의 존재 이유가 여기서 회수된다).

실제 OS는 이 위에 층을 더 쌓는다. Windows의 모델로 보면:

```
프로세스 워킹셋 ──(트리밍)──▶ modified 리스트 ──(디스크 기록)──▶ standby 리스트 ──▶ free/zero
     ▲                                                              │
     └────────────── 재접근 시 minor fault로 즉시 복귀 ◀─────────────┘
```

워킹셋에서 뺀 페이지를 바로 버리지 않고 스탠바이에 보관해서, 다시 접근하면 **디스크 없이 minor fault로 되살린다** — 교체 결정에 실수해도 싸게 무를 수 있는 완충 지대다. "빈 RAM은 낭비"라는 철학으로 남는 물리 메모리는 전부 이런 캐시(스탠바이 + 파일 캐시)로 쓰인다. 작업 관리자의 "사용 가능" 메모리가 "빈" 메모리가 아닌 이유.

### 스택 — 가드 페이지 위에서 자란다

스레드 스택은 "예약은 크게, 커밋은 조금"의 대표 사례다. Windows 기준 기본 예약 1MB(링커 `/STACK`으로 조정), 커밋은 소량으로 시작하고, 커밋 영역 끝에 **가드 페이지(guard page)** — PAGE_GUARD 속성이 박힌 함정 — 를 둔다:

```
높은 주소
┌──────────────────┐
│ 커밋된 스택        │ ← rsp가 아래로 자람
├──────────────────┤
│ 가드 페이지        │ ← 터치 → 폴트 → 커널이 커밋 확장 + 가드를 한 칸 아래로 이동
├──────────────────┤
│ 예약만 됨(미커밋)   │
└──────────────────┘ ← 예약 한계까지 소진 → 스택 오버플로 예외
낮은 주소
```

- 스택이 "자동으로 자라는" 것처럼 보이는 이유 = 가드 페이지 폴트를 커널이 성장 신호로 해석하기 때문. 스택 성장은 문법이 아니라 폴트 핸들러의 정책이다.
- **스택 오버플로가 즉시, 정확한 지점에서 잡히는 이유** = 더 자랄 예약 공간이 없을 때의 가드 폴트를 복구 불가로 판정하기 때문. 힙 버퍼 오버런이 대개 조용히 이웃을 오염시키는 것과 대비된다 — 힙에는 오버런 지점에 보호 페이지가 없다(같은 커밋 페이지 안이면 하드웨어는 아무것도 모른다).
- 함정 하나: 지역 배열이 4KB(한 페이지)보다 크면 함수 진입 시 rsp가 가드 페이지를 **건너뛰어** 그 아래 미커밋 영역을 바로 찍을 수 있다. 그래서 MSVC는 큰 프레임에 `__chkstk`를 삽입해 페이지를 한 장씩 순서대로 터치(스택 프로빙)한다 — 가드 메커니즘이 "순차 성장" 가정 위에 서 있음을 보여주는 디테일. 재귀 폭주는 자연히 순차라 프로빙 없이도 잡힌다.

### 힙 — malloc은 시스템 콜이 아니다

계층을 정확히 그리면:

```
new / std::vector / std::string           ← C++ 레이어 (생성자·타입)
        │
malloc / free (CRT 힙·jemalloc·rpmalloc…) ← 유저 공간 라이브러리:
        │                                    큰 덩어리를 받아 버킷/빈으로 쪼개 재분배
VirtualAlloc (Windows) / mmap·brk (Linux)  ← 시스템 콜: 페이지(4KB) 단위 커밋
        │
커널 폴트 핸들러 + 물리 프레임 관리          ← 첫 터치 시 실제 배정
```

malloc이 존재하는 이유: 시스템 콜은 비싸고 최소 단위가 4KB인데, 프로그램은 24바이트짜리 노드를 초당 수만 번 할당한다. 그래서 **malloc은 OS에서 큰 덩어리를 도매로 받아 소매로 파는 중간상**이다. `free`가 대개 OS 반환이 아니라 중간상의 재고 환입인 이유도 이 구조에서 자명하다.

이 구조에서 두 가지 단편화가 나온다:

| | 내부 단편화(internal) | 외부 단편화(external) |
|---|---|---|
| 정의 | 할당 단위 반올림으로 블록 **안**에서 낭비 | 총 여유는 충분한데 **연속** 블록이 없음 |
| 예 | 33바이트 요청 → 48바이트 버킷 배정 | 여유 10MB가 512KB 조각으로 흩어져 1MB 연속 할당 실패 |
| 누구의 문제 | 할당자의 버킷 설계 | 할당·해제 패턴(수명이 뒤섞일수록 악화) |
| 완화 | 버킷 크기 세분화 | 크기별 분리, 풀/아레나로 수명·크기 균질화 |

참고로 가상 메모리는 **물리** 단편화 문제를 이미 페이지 매핑으로 풀었다(연속 가상 = 불연속 물리 허용). 남는 것은 **가상 주소 공간 안에서의** 단편화이고, 이건 유저 공간 할당자의 몫이다.

**게임이 풀(pool)/아레나(arena) 커스텀 할당자를 쓰는 이유**는 범용 malloc의 세 가지 비용이 프레임 예산과 충돌하기 때문이다:

1. **비결정적 지연** — malloc은 평소 수십 ns지만, 프리리스트가 비면 시스템 콜 + 폴트 경로를 타며 마이크로초 이상으로 튄다. 16.6ms 예산에서 "가끔 느림"은 곧 프레임 드랍이고, 프로파일러에 안 잡히는 간헐 스파이크가 가장 악질이다.
2. **멀티스레드 락/동기화** — 전역 힙은 공유 자원이다. 잡시스템 워커들이 동시에 malloc하면 경합한다(현대 할당자는 스레드 로컬 캐시로 완화하지만 비용이 0은 아니다).
3. **장기 단편화와 캐시 산개** — 몇 시간 플레이 후 힙은 조각나고, 개별 할당된 객체들은 물리적으로 흩어져 캐시·TLB 지역성이 무너진다.

풀은 "같은 크기 객체 N개를 선할당하고 인덱스만 돌려쓰기" — 할당 = 프리리스트 pop, O(1), 설계에 따라 락 프리. 아레나(선형 할당자)는 "프레임 시작에 포인터 하나 리셋, 할당 = 포인터 증가, 개별 해제 없음". 둘 다 **유연성(임의 크기·임의 수명)을 버리고 결정성과 지역성을 사는 거래**다. "커스텀 할당자 = 빠른 malloc"이 아니라 "제약을 받아들인 대가로 보장을 얻는 계약"이라고 말해야 정확하다.

### 메모리 매핑 파일 — read( )에서 복사 한 벌을 빼기

파일을 읽는 두 경로:

```
read() 경로:  디스크 ──▶ 커널 페이지 캐시 ──(memcpy)──▶ 유저 버퍼    : 데이터가 RAM에 2벌
mmap  경로:  디스크 ──▶ 커널 페이지 캐시 ◀──(매핑)──── 유저 가상주소  : 1벌, 복사 0회
```

`CreateFileMapping`/`MapViewOfFile`(Windows), `mmap`(POSIX)은 파일을 주소 공간에 매핑해서 **파일 읽기를 페이지 폴트로 바꾼다.**

- 포인터로 건드리는 부분만 요구 페이징으로 올라온다 — 거대 에셋의 일부만 쓸 때 I/O가 접근량에 비례.
- 페이지 캐시 → 유저 버퍼 복사가 사라진다 — 대용량에서 CPU·메모리 절약.
- 같은 파일을 매핑한 프로세스끼리 물리 페이지를 공유한다 — 실행 파일·DLL 로딩이 정확히 이 메커니즘이다. "exe를 실행한다" = "exe 파일을 매핑하고 엔트리 포인트로 점프한다."

트레이드오프도 명확하다: 미리 다 읽는 read()는 이후 접근이 폴트 없이 균일하지만, mmap은 접근 도중 major fault(디스크 지연)가 **임의 시점에** 낄 수 있다 — 게임플레이 중 스터터로 나타난다. 에러 처리도 달라진다(디스크 I/O 실패가 반환값이 아니라 접근 지점의 예외로 온다). 그래서 게임은 "로딩 화면에서 통째 읽기"와 "스트리밍 매핑" 사이에서 에셋 성격별로 고른다.

### ASLR — 재배치 능력을 보안 무기로 재활용

가상 메모리가 어차피 "어느 가상 주소에든 올릴 수 있는" 재배치 능력을 갖고 있으므로, 이를 보안에 쓴 것이 **ASLR(Address Space Layout Randomization)**이다: 실행 파일·DLL·스택·힙의 기준 주소를 무작위화해서, 공격자가 "이 함수는 항상 주소 X에 있다"는 가정으로 익스플로잇을 짜지 못하게 한다.

- NX(실행 금지)가 "주입한 코드 실행"을 막으면, 공격은 기존 코드 조각을 이어붙이는 ROP로 진화했다. ASLR은 그 조각들의 **주소를 숨겨서** ROP를 어렵게 만든다 — 두 방어는 서로의 우회로를 막는 한 세트다.
- 링커 옵션(`/DYNAMICBASE`, 64비트 고엔트로피는 `/HIGHENTROPYVA`)으로 opt-in하는 것도 재배치 정보가 있어야 무작위화가 가능하기 때문이다.
- **공유와의 긴장**: DLL 주소를 프로세스마다 다르게 뽑으면 재배치 패치 때문에 COW로 코드 페이지 공유가 깨진다. 그래서 Windows는 DLL의 무작위 기준 주소를 **부팅당 한 번** 뽑아 전 프로세스가 공유하게 한다 — 보안(무작위화)과 메모리 효율(공유)의 절충이 명시적으로 설계에 박혀 있다.

### 거대 페이지(large page) — TLB 커버리지와의 거래

페이지 테이블 워크에서 PD 단계 엔트리에 크기 플래그(PS)를 세우면 하위 PT를 생략하고 **2MB 페이지**로 매핑할 수 있다(한 단계 위에서는 1GB도 가능). 효과는 TLB 산수로 바로 나온다:

- 같은 TLB 엔트리 수로 커버하는 메모리가 **512배** — 수 GB 힙을 랜덤 접근하는 워크로드(DB, 대형 시뮬레이션, 서버)에서 TLB 미스가 극적으로 준다.
- 워크가 한 단계 짧아져 미스 비용 자체도 준다.

공짜가 아니다: 2MB **연속 물리** 메모리가 필요해 부팅 후 시간이 지나면(물리 단편화) 확보가 어렵고, 할당 단위가 커져 내부 단편화가 늘며, Windows의 large page는 페이징 불가(항상 물리 고정) + `SeLockMemoryPrivilege` 권한이 필요하다. "기본은 4KB, 큰 힙을 아는 소프트웨어만 opt-in"이 현재의 합의다.

### 캐시라인과 false sharing — 페이지 아래층의 단위

주소 공간의 관리 단위가 페이지(4KB)라면, CPU 캐시의 관리 단위는 **캐시라인(cache line, 통상 64바이트)**이다. 서로 다른 스레드가 쓰는 두 변수가 우연히 같은 캐시라인에 있으면, 논리적으로 무관한데도 코어 간 캐시 일관성 프로토콜이 라인 소유권을 계속 주고받는 **false sharing**이 난다. 해법은 `alignas(64)`로 라인을 분리하는 것.

층을 정확히 구분해 답하면 강하다: **페이지는 OS·MMU의 단위(보호·매핑·스왑), 캐시라인은 코어 간 일관성의 단위(MESI), 관할이 다르다.** 상세는 C++ 시리즈의 캐시/동시성 챕터([cpp/08](../cpp/08_stl_containers_cache.md), [cpp/09](../cpp/09_concurrency.md))에서 다룬다.

## ③ 심화 — 꼬리질문 대비

**"페이지 테이블은 왜 다단계 트리인가? 1단계 평면 배열이면 더 빠르지 않나?"**
빠르긴 하다 — 미스 시 메모리 접근이 1회니까. 문제는 크기다. 48비트/4KB면 엔트리 2^36개 × 8B = 프로세스당 테이블만 512GB다. 실제 프로세스는 코드·힙·스택 몇 군데만 쓰는 희소 공간이므로, 트리는 빈 영역을 상위 엔트리 하나의 'not present'로 접어 버린다. 다단계는 속도를 희생해 희소성을 표현하는 공간 절약 설계고, 그 속도 희생을 TLB와 paging-structure cache가 도로 메운다.

**"페이지 테이블 자체는 어디에 있나? 그것도 페이징되나?"**
커널이 관리하는 물리 메모리 페이지들에 있고, CR3와 각 엔트리는 **물리 주소**로 다음 단계를 가리킨다 — 변환 하드웨어가 변환 없이 따라가야 하므로. 원리적으로 테이블도 스왑 대상이 될 수 있지만 워크 도중 폴트가 나는 재귀 문제가 있어, 최소한 현재 활성 매핑의 상위 테이블은 상주한다. 구현 세부는 OS마다 다르므로 면접에서는 "엔트리는 물리 주소 체인이고 커널 관리 메모리에 산다"까지가 안전한 답이다.

**"TLB 미스와 캐시 미스는 뭐가 다른가?"**
캐시 미스는 "데이터가 SRAM에 없어서 DRAM에서 가져온다" — **데이터의 위치** 문제. TLB 미스는 "주소를 번역할 수 없어서 페이지 테이블을 걸어야 한다" — **주소의 번역** 문제. 별개 축이라 조합이 4가지 다 나온다. 최악은 TLB 미스 + 워크로 읽는 테이블 엔트리도 캐시 미스 + 최종 데이터도 캐시 미스인 경우로, 접근 한 번이 DRAM 왕복 여러 번이 된다. 링크드 리스트 순회(랜덤 포인터 추적)가 느린 이유가 이 조합이다.

**"컨텍스트 스위치가 비싸다는데, 레지스터 저장 비용인가?"**
아니다. 레지스터 저장/복원은 수백 ns 수준의 직접 비용이고, 진짜 비용은 간접 비용 — CR3 교체로 인한 TLB 무효화와 캐시에서 이전 프로세스 데이터가 밀려나는 오염이다. 스위치 직후 한동안 새 프로세스는 미스 폭풍 속에서 돈다. PCID/ASID가 TLB 쪽을 완화하지만 캐시 오염은 남는다. 같은 프로세스 내 스레드 전환이 프로세스 전환보다 싼 핵심 이유도 CR3를 안 바꾸기 때문이다.

**"malloc(1GB)이 성공했는데 나중에 프로그램이 죽을 수 있는 이유는?"**
Linux 기본 설정에서 malloc 성공은 "가상 주소 약속"일 뿐이다. 물리 배정은 첫 터치 폴트에서 일어나는데, 그 시점에 물리+스왑이 바닥이면 OOM killer가 희생 프로세스를 고른다 — 할당 지점이 아니라 접근 지점에서, 심지어 다른 프로세스가 죽을 수도 있다. Windows는 커밋 시점에 commit charge 한도를 검사해 실패를 반환하는 보수적 정책이라 이 패턴이 덜하다. "언제 실패를 알려줄 것인가"라는 정책 차이일 뿐 lazy 배정 메커니즘은 같다.

**"free()하면 물리 메모리가 OS로 돌아가나?"**
대개 즉시는 아니다. free는 malloc의 유저 공간 프리리스트에 반납될 뿐이고, 할당자가 판단해서(큰 블록, 힙 꼬리 정리 등) 페이지 단위로 `VirtualFree`/`munmap`/`madvise`를 부를 때만 실제 반환된다. 그래서 작업 관리자 수치는 free 후에도 안 줄 수 있다 — 누수를 의심하기 전에 할당자의 보유 정책부터 볼 것. 한편 프로세스가 종료되면 커널이 페이지 테이블과 물리 페이지를 통째로 회수한다. "OS 관점의 누수는 없다, 프로세스 수명 안에서의 낭비가 있을 뿐"이 정확한 표현이다.

**"스택 오버플로는 즉시 잡히는데 힙 버퍼 오버런은 왜 조용히 지나가나?"**
잡힘의 단위가 페이지라서다. 스택 끝에는 커널이 의도적으로 가드 페이지를 깔아 놨으니 넘는 순간 폴트가 난다. 힙에서 `buf[n]`을 넘어 쓴 곳은 대개 같은 커밋 페이지 안의 이웃 할당이라 MMU 관점에서 완전히 합법이다 — 오염이 한참 뒤 엉뚱한 곳에서 터진다. 그래서 디버깅 도구(Windows PageHeap, ASan)는 할당 뒤에 보호 페이지를 붙이거나 레드존+검사를 깔아 오버런을 즉시 위반으로 승격시킨다 — 페이지 보호 메커니즘을 디버깅 도구로 재활용하는 것.

**"use-after-free가 어떨 때는 크래시 나고 어떨 때는 조용한 이유는?"**
free된 메모리의 페이지가 아직 프로세스에 커밋돼 있으면(대부분 — 할당자가 재사용하려고 들고 있음) 접근이 하드웨어적으로 유효해서 크래시가 안 난다. 쓰레기 값 또는 재할당된 남의 데이터를 읽을 뿐이다. 크래시는 할당자가 그 페이지를 OS에 반환해 매핑이 사라진 뒤 접근했을 때만 난다. 즉 UB가 "잡히느냐"는 코드가 아니라 할당자와 페이지 상태라는 우연이 결정한다 — "크래시 안 남 ≠ 버그 없음"의 하드웨어적 근거.

**"fork가 빠른 이유와 COW의 숨은 비용은?"**
빠른 이유: 페이지 복사 대신 페이지 테이블 복사 + 전 페이지 RO 마킹으로 끝나서. 숨은 비용: (1) 이후 어느 쪽이든 첫 쓰기마다 폴트+페이지 복사로 쓰기 지연이 튄다, (2) RO 마킹 자체가 멀티코어에서 TLB shootdown을 유발한다, (3) 쓰기 비율이 높으면 결국 다 복사하면서 폴트 오버헤드만 얹는다. "복사를 없앤 게 아니라 지연한 것"이고, 쓰기가 드물다는 베팅이 전제다.

**"NX/DEP가 막는 공격과 못 막는 공격은?"**
막는 것: 스택/힙에 주입한 셸코드의 직접 실행 — 데이터 페이지에 NX가 서 있으니 실행 시도가 곧 폴트. 못 막는 것: 이미 실행 가능한 기존 코드 조각을 리턴 주소 체인으로 이어붙이는 ROP. 그래서 ASLR로 조각들의 주소를 숨기고, 최신 하드웨어는 섀도 스택 같은 제어 흐름 보호(Intel CET 계열)를 더 얹는다. "방어는 한 방이 아니라 계층"이라는 서사로 답한다. 덤: JIT 컴파일러는 데이터로 쓴 코드를 실행해야 하므로 W^X(쓰기와 실행을 동시에 안 줌) 규율로 페이지 권한을 단계 전환한다.

**"왜 페이지가 하필 4KB인가?"**
트레이드오프의 균형점이다. 작으면 테이블·엔트리 수가 폭증하고 TLB 커버리지가 준다. 크면 내부 단편화(1바이트 써도 페이지 통째)와 I/O·COW·스왑 단위가 굵어진다. 4KB는 수십 년 전 균형점이 호환성 관성으로 굳은 것에 가깝고, 메모리가 커진 현대에는 작다는 평가가 많다 — 2MB 거대 페이지 opt-in, 일부 아키텍처(Apple Silicon의 16KB)의 기본 페이지 확대가 그 방증이다. "고정 정답이 아니라 시대별 균형"이라고 답하는 게 정확하다.

**"게임 로딩/스터터를 가상 메모리 관점에서 최적화한다면?"**
(1) major fault를 로딩 화면으로 몰기 — 플레이 직전 필요한 에셋을 순차 읽기(디스크 최적 패턴)로 워밍업해 인게임 첫 터치가 디스크를 안 치게 한다. (2) 할당을 프레임 루프에서 추방 — 풀/아레나 선할당으로 폴트·malloc 지연을 초기화 시점으로 이동. (3) 데이터 배치 — 함께 쓰는 데이터를 같은 페이지/캐시라인에 모아 TLB·캐시 미스를 줄인다(SoA, 연속 배열). (4) 대형 상주 데이터는 거대 페이지 검토. 전부 "비용을 없애는 게 아니라 **프레임 밖으로 옮기는**" 전략이라는 공통 원리로 묶는다.

**"스왑(페이지 파일)을 꺼버리면 더 빠르지 않나?"**
절반만 맞는 직관이다. 스왑은 '느려지는 기능'이 아니라 '안 쓰는 dirty 익명 페이지를 치워 파일 캐시 공간을 버는 수단'이다. 끄면 그 페이지들을 치울 곳이 없어 커밋 한도가 사실상 RAM으로 고정되고(Windows면 할당 실패가 빨라지고, Linux면 OOM kill이 빨라진다), 파일 캐시로 쓸 물리 메모리가 준다. RAM이 넉넉하면 차이가 없고, 부족하면 더 험하게 죽는다 — "성능 스위치가 아니라 실패 모드의 선택"이라고 답한다.

**"32비트 프로그램은 왜 큰 메모리를 못 쓰나? 64비트로 오면서 뭐가 달라졌나?"**
32비트 가상 주소는 4GB 공간이고, 관례상 절반 안팎을 커널 영역이 가져가 유저는 2~3GB에서 산다 — 물리 RAM이 아무리 커도 **주소를 표현할 비트가 없다.** (PAE로 물리 주소만 36비트+로 늘린 시절도 있었지만 프로세스당 가상 공간 한계는 그대로였다.) 64비트는 가상 공간이 사실상 무한(256TB)이라 주소 공간 자체가 자원에서 제외됐고, 그 여유가 ASLR 고엔트로피·거대 매핑·주소 공간 낭비형 기법(레드존, 예약 후 부분 커밋)을 가능케 했다.

## ④ Winters에서의 적용

### (1) .wmesh 로딩 — "통째 읽기" 경로의 실체와 mmap 전환 지점

Winters의 에셋 바이너리 로딩은 read() 계열 경로다. `Engine/Private/AssetFormat/Common/BinaryReader.cpp:28`의 `LoadFileToMemory`가 `std::ifstream`으로 파일 전체를 `std::vector<uint8_t>` 힙 버퍼에 읽는다 — ② 절의 그림 그대로 "디스크 → 페이지 캐시 → (복사) → 유저 버퍼"의 2벌 경로이고, `resize` 직후 버퍼를 처음 채우는 순간들이 전부 demand-zero minor fault다. `Engine/Private/AssetFormat/Mesh/WMeshLoader.cpp:34`의 `Load`는 이 버퍼(`m_vRawFile`)를 파싱 결과의 수명 동안 들고 있고, 주석("m_vRawFile 수명 유지 → blob 포인터가 가리킴")대로 정점 blob 포인터들이 버퍼 내부를 직접 가리키는 **in-place 파싱** 구조다.

이 구조가 면접 재료로 좋은 이유가 두 가지다:

- `WMeshLoader.cpp:41`의 `LoadFromMemory(const void*, size_t)`가 이미 "메모리 이미지 파싱" 인터페이스라서, **`MapViewOfFile`로 얻은 매핑 포인터를 그대로 꽂으면 복사 한 벌이 사라지는 mmap 전환점이 코드에 준비돼 있다.** 동시에 트레이드오프를 안다고 말할 수 있다 — 로딩 시점에 다 읽는 현재 방식은 인게임 major fault가 없다는 보장이 있고, 매핑 방식은 거대 에셋 부분 접근에 유리한 대신 플레이 중 디스크 폴트 스터터 위험을 진다.
- `WMeshLoader.cpp:9`의 `MAX_VERTICES` 등 상한 검증("악의적 파일 방어" 주석)은 "읽었든 매핑했든 파일 이미지는 신뢰 불가 입력"이라는 원칙의 실물이다. 매핑으로 바꿔도 이 검증 계층은 그대로 필요하다는 점까지 이으면 보안 감각도 함께 보여 줄 수 있다.

### (2) 리소스 캐시 — OS의 '공유'를 유저 공간에서 재현한 설계

OS가 같은 DLL 코드 페이지를 프로세스들끼리 공유하듯, `Engine/Private/Resource/ResourceCache.cpp:47`의 `LoadModel`은 정규화된 경로 키(대소문자·슬래시 통일)로 캐시를 조회해 히트면 기존 `shared_ptr<CModel>`을 돌려준다 — 같은 모델을 쓰는 유닛이 몇이든 정점 데이터는 메모리에 한 벌이다. 텍스처 쪽(`ResourceCache.cpp:14`)은 캐시가 `unique_ptr`로 소유하고 소비자에게 raw 포인터만 빌려준다. "누가 소유하고(캐시) 누가 관찰하는가(렌더러)"를 타입으로 선언한 것으로, 커널이 공유 페이지에서 하는 "한 벌 + 참조 + 소유권 규율" 관리를 애플리케이션 레이어에서 반복한 구조라고 설명할 수 있다. 중복 로드 방지 = 물리 페이지 공유의 유저 공간 아날로그.

### (3) DLL 경계 — 재배치·공유·ASLR이 내 빌드 시스템에 닿는 곳

`Engine/Include/WintersAPI.h:11`의 `WINTERS_ENGINE` 매크로(정적 빌드/export/import 3분기)는 컴파일 타임 이야기지만, 그 산출물인 Engine DLL이 런타임에 겪는 일이 정확히 이 챕터다:

- DLL 이미지는 **메모리 매핑 파일**로 로드된다 — 로드가 빠른 이유는 요구 페이징이지 전체 읽기가 아니다.
- 코드 페이지는 읽기 전용이라 같은 DLL을 로드한 프로세스들이 물리 페이지를 공유하고, 재배치 패치가 필요해진 페이지만 COW로 사유화된다.
- ASLR이 부팅당 한 번 뽑은 기준 주소 덕에 무작위화와 공유가 공존한다.
- `dllimport` 함수 호출이 IAT(Import Address Table)를 거치는 간접 호출이라는 것까지 이으면, "링커 매크로 → 로더 → 페이지 공유"를 한 줄기로 답할 수 있다. C++ 시리즈의 [컴파일·링크·DLL 챕터](../cpp/02_compile_link_dll.md)와 이 챕터가 만나는 지점이다.

### (4) 풀 할당자 실물 — 프레임 루프에서 malloc 추방

`Engine/Public/FX/ParticlePool.h:12`의 `CParticlePool`이 ② 절 "게임이 풀을 쓰는 이유"의 증거물이다. `Initialize(uMaxParticles)`(21행)에서 용량만큼 SoA 컬럼(`m_vecPos`, `m_vecVelocity` … 58~66행)을 선할당하고, 런타임 `Spawn`/`KillSwapBack`(24~25행)은 인덱스 증감과 swap-back뿐이라 **파티클이 초당 수천 개 태어나고 죽어도 프레임 중 힙 할당이 0회**다. 덤으로 SoA 연속 배열이라 시뮬레이션 순회가 캐시·TLB 친화적이다 — "할당 비용 제거"와 "지역성 확보"가 한 설계에서 같이 나오는 것을 보여 준다.

서드파티 쪽 대조군도 리포 안에 있다. tracy가 포함한 rpmalloc(`Engine/External/tracy/client/tracy_rpmalloc.cpp:880`)은 `VirtualAlloc(0, size, ... | MEM_RESERVE | MEM_COMMIT, PAGE_READWRITE)` 위에 자체 힙을 얹고, 옵션으로 `MEM_LARGE_PAGES`까지 켠다 — "malloc은 페이지 시스템 위의 유저 공간 라이브러리"라는 계층 그림과 거대 페이지 opt-in의 실물이 프로젝트 안에 있는 셈이다.

### (5) 캐시라인 경계 — false sharing 방지의 실전 코드

`Engine/Public/Core/JobSystem/WorkStealingDeque.h:91`의 `alignas(64) std::atomic<std::int64_t> m_iBottom` / `m_iTop`(91~92행)은 훔치는 쪽 스레드(top)와 소유 스레드(bottom)가 각각 두드리는 두 카운터를 서로 다른 캐시라인에 격리한다. "페이지(4KB)는 OS·MMU의 단위, 캐시라인(64B)은 코어 간 일관성의 단위 — 관할이 다르다"는 ② 절의 층 구분을 잡시스템 코드 두 줄로 시연할 수 있다.

### 이 개념을 내 프로젝트 경험으로 말하는 법

"가상 메모리를 직접 구현해 봤나"는 질문에는 솔직하게 아니라고 하되, 이렇게 잇는다: **"커널이 페이지로 푸는 문제(선할당·지연·공유·단편화 회피)를 저는 엔진 레이어에서 반복해서 풀었습니다."** — 파티클 풀(선할당 + O(1) 재사용), 리소스 캐시(한 벌 공유 + 소유권 분리), 에셋 통째 로드(major fault를 로딩 시점으로 몰기), 잡시스템 캐시라인 격리. 그리고 개선 여지를 아는 것까지가 이해의 증거다: "wmesh 로더는 인터페이스가 이미 (포인터, 크기)라 메모리 매핑 파일로 바꾸면 복사 한 벌이 사라지는데, 인게임 폴트 스터터와의 트레이드오프가 있어 에셋 성격별로 골라야 한다"까지 말하면 원리와 실무가 연결된다.

## ⑤ 면접 Q&A

**Q1. 가상 메모리가 왜 필요한가?**
- 핵심: 주소에 간접 한 층(페이지 테이블)을 넣어 격리·재배치·과할당·공유 네 가지를 동시에 산다. 특히 격리는 "남의 메모리로 가는 매핑 자체가 없다"는 하드웨어 수준 보장.
- 꼬리 대비: "비용은?" → 변환 오버헤드(TLB로 상쇄), 테이블 메모리(다단계 트리로 상쇄), 폴트 지연(게임은 로딩 시점으로 몰아 상쇄).
- Winters: 리소스 캐시의 한 벌 공유(ResourceCache.cpp:47)를 "공유 개념의 유저 공간 재현"으로 연결.

**Q2. malloc하면 무슨 일이 일어나나?**
- 핵심: 3층으로 답한다 — (1) malloc은 유저 공간 중간상: 프리리스트에 있으면 시스템 콜 없이 반환. (2) 부족하면 VirtualAlloc/mmap으로 페이지 도매 확보 — 이때도 장부 기록뿐. (3) 첫 터치에서 minor fault → 그 페이지만 물리 배정 + 제로필.
- 꼬리 대비: "free하면 OS로 돌아가나?" → 대개 프리리스트 환입, 반환은 할당자 재량.
- Winters: rpmalloc이 VirtualAlloc 위에 얹힌 실물(tracy_rpmalloc.cpp:880).

**Q3. 페이지 폴트의 종류를 구분해 보라.**
- 핵심: minor(물리 재연결만, 디스크 없음 — 첫 터치/COW/스탠바이 복귀), major(디스크 I/O — 스왑·매핑 파일), protection(권한 위반/무효 — SIGSEGV·Access Violation). 앞의 둘은 정상 동작, 마지막만 에러.
- 꼬리 대비: "처리 경로는?" → #PF 예외 → CR2의 폴트 주소로 커널이 영역 장부(VAD/vm_area) 검색 → 채우고 같은 명령 재실행. "재실행이 투명하다"가 요점.
- 게임 관점: major fault가 프레임 도중 오면 스터터 — 그래서 로딩 화면에서 워밍업.

**Q4. 컨텍스트 스위치와 TLB의 관계는?**
- 핵심: 주소 공간 전환 = CR3 교체 한 줄이지만, 그 순간 TLB의 기존 변환이 무효가 되는 게 문제. PCID/ASID 태그로 플러시를 피하는 게 현대의 답.
- 꼬리 대비: "그럼 스위치의 진짜 비용은?" → 레지스터가 아니라 TLB·캐시 오염이라는 간접 비용.
- 연결: 멀티코어에서 매핑 회수는 TLB shootdown(IPI)까지 필요 — "추가는 싸고 회수는 비싸다".

**Q5. 게임에서 커스텀 할당자를 왜 쓰나?**
- 핵심: 범용 malloc의 세 비용 — 비결정적 지연 스파이크, 멀티스레드 경합, 장기 단편화+캐시 산개 — 이 프레임 예산과 충돌해서. 풀 = 동일 크기 선할당 + O(1) 재사용, 아레나 = 프레임 단위 포인터 리셋. 유연성을 버리고 결정성·지역성을 사는 계약.
- Winters: CParticlePool(ParticlePool.h:12) — Initialize 선할당, Spawn/KillSwapBack, SoA 컬럼. 프레임 중 할당 0회.

**Q6. 스택 오버플로는 어떻게 감지되나?**
- 핵심: 커밋된 스택 끝의 가드 페이지. 터치하면 폴트 → 커널이 커밋 확장(정상 성장) 또는 예약 소진이면 스택 오버플로 예외. 성장도 감지도 전부 페이지 폴트의 응용.
- 꼬리 대비: "큰 지역 배열은?" → 가드를 건너뛸 수 있어 컴파일러가 __chkstk로 페이지를 순서대로 프로빙.
- 대비 포인트: 힙 오버런이 조용한 이유(같은 페이지 안은 하드웨어가 모름)와 짝으로 답하면 이해가 드러난다.

**Q7. 메모리 매핑 파일과 read()의 차이는?**
- 핵심: read()는 페이지 캐시 → 유저 버퍼로 한 번 더 복사(데이터 2벌), mmap은 페이지 캐시를 주소 공간에 직접 매핑(1벌, 접근한 부분만 폴트로 로드). 실행 파일·DLL 로딩이 곧 mmap이다.
- 꼬리 대비: "단점은?" → 접근 도중 임의 시점의 major fault(스터터), 에러가 반환값이 아니라 접근 지점 예외로 옴.
- Winters: BinaryReader.cpp:28은 현재 read() 경로, LoadFromMemory(WMeshLoader.cpp:41)가 이미 (포인터, 크기) 인터페이스라 전환점이 준비돼 있음.

## ⑥ 흔한 오답 / 함정

1. **"가상 메모리 = 스왑(디스크로 메모리 확장)"으로 답하기.**
   스왑은 파생 기능 중 하나일 뿐이다. 본질은 주소 간접화로 얻는 격리·재배치·지연·공유이고, 스왑 없는 시스템(스왑 끈 PC, 일부 콘솔/임베디드)에도 가상 메모리는 있다. 이 오답 하나로 이해 깊이가 갈린다.

2. **"malloc이 메모리를 준다 = OS가 물리 메모리를 준다"로 뭉개기.**
   세 층(라이브러리 프리리스트 / 시스템 콜의 가상 약속 / 첫 터치의 물리 배정)이 분리돼 있다. "malloc 직후 물리 사용량은 거의 안 늘고, memset하는 순간 는다"를 말할 수 있어야 한다.

3. **"페이지 폴트 = 버그/크래시"라는 등식.**
   minor/major fault는 요구 페이징·COW·스택 성장을 굴리는 정상 메커니즘이다. 크래시는 protection 위반 한 종류뿐. "폴트는 커널이 파 둔 함정이 작동한 것"이라는 프레임으로 답하면 정확하다.

4. **"TLB는 그냥 캐시 중 하나"로 넘기기.**
   데이터 캐시와 축이 다르다(번역 vs 데이터). 컨텍스트 스위치 비용, 거대 페이지의 존재 이유, 랜덤 접근이 느린 이유가 전부 TLB에서 나오므로, TLB를 빼고는 이 챕터의 절반이 설명되지 않는다.

5. **"커스텀 할당자 = malloc보다 빠르게 만든 것"이라는 단순화.**
   핵심은 속도가 아니라 **보장**이다 — 상한이 있는 지연, 경합 없음, 단편화 없음. 그 대가로 임의 크기·임의 수명을 포기한다. "일반성을 좁힌 만큼 보장이 생긴다"는 계약 관점으로 답해야 설계를 아는 사람으로 들린다.

6. **스택/힙을 "언어의 기능"으로 설명하기.**
   스택 성장은 커널의 가드 페이지 정책이고, 힙은 페이지 시스템 위의 라이브러리다. C++ 문법(자동 변수, new)은 이 OS 메커니즘 위의 표기법일 뿐이다. 언어 답변과 OS 답변을 층으로 구분해서 말하는 것 자체가 점수다.
